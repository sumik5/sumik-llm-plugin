# OOUI 理論的背景

オブジェクト指向UI（OOUI）の哲学的根拠・歴史・モードレス原則の参照資料。
OOUI設計の「なぜ」を理解するためのリファレンス。実践手順は `OOUI-METHODOLOGY.md` を参照。

---

## 1. オブジェクトの原義

**object** の語源は `ob-`（toward/against）＋ `-ject`（throw）。
「その方へ投げかけられた客体」というニュアンスを持つ。

- 主体（subject）が知覚する前に、客体（object）がそこに**先行して存在する**
- OOPもOOUIも、設計の焦点を「主体がすること」ではなく「客体として存在するもの」に置く
- 日本語に訳すなら「対象」「物体」「目的」——それらすべてのニュアンスを内包

### プラトンのイデア論との対応

OOPの「クラス→インスタンス」生成モデルはプラトンのイデア論に対応する:

| 概念 | OOP | GUI |
|------|-----|-----|
| イデア（一般・本質） | クラス定義 | コンポーネント型 |
| 個体（具象・現象） | インスタンス | 画面上のUI要素 |
| 観念の共有 | インタフェース | デザインシステム |

人間は「椅子そのもの」を見なくても「椅子らしさ」という抽象概念を持てる。
GUIはこの人間の抽象化能力を活用し、コンピューター内の概念を知覚可能な形で対象化する。

## 2. OOPとOOUIの統一的パラダイム

### 構文論的転回（Syntactic Turn）

OOPとOOUIは、どちらも同じ「Object → Verb」構文を採用する:

```
// 手続き型（Verb → Object）
strtoupper('hello')

// オブジェクト指向型（Object → Verb）
'hello'.toUpperCase()
```

UIの操作構文でも同じ転回が起きる:

| 操作モデル | 構文 | 例 |
|-----------|------|----|
| CLI（コマンド指向） | Verb → Object | `open helloworld.txt` |
| OOUI（オブジェクト指向） | Object → Verb | ファイル選択 → メニューから「開く」 |
| タスク指向UI | Verb → Object | 「再生機能」選択 → データ選択 |

### OOPとOOUIの関係

- GUIを実現するためにOOP環境が整えられ、OOPが発展することでGUIが普及した
- GUIはUIをオブジェクト指向にしたものとして生まれた表現
- 見た目がグラフィカルでも、操作設計がオブジェクト指向でなければGUIらしいとは言えない

## 3. GUIの歴史

### 年表（要約）

| 年代 | 出来事 | 意義 |
|------|--------|------|
| 1940年代 | Whirlwind（MIT） | CRTで情報を視覚化、ライトガンによる直接操作の原型 |
| 1950年代 | SAGE（防空システム） | 飛行物体オブジェクトをリアルタイム表示・直接操作 |
| 1963年 | Sketchpad（MIT） | 非手続き型操作、ウィンドウ・アイコンの原型 |
| 1968年 | NLS（エンゲルバート） | マウス・ハイパーリンク・共同作業の実装 |
| 1970年代 | Smalltalk（Xerox PARC） | OOP+GUIの統合、モードレス操作体系の確立 |
| 1981年 | Xerox Star | GUI設計ガイドライン、デスクトップメタファー |
| 1984年 | Macintosh | 商業的に成功した最初のGUI、Human Interface Guidelines |
| 2007年 | iPhone | タッチ操作による直接操作感の強化、スマートフォンUI標準化 |

### 設計思想の変遷

- **計算機から知的道具へ**: 単なる計算処理ではなく、人の知性・記憶を拡張する道具として
- **コマンドから直接操作へ**: 「コンピューターへの命令」から「オブジェクトへの直接アプローチ」へ
- **UIからメタメディアへ**: コンピューターの本質はシミュレーション能力であり、あらゆるメディアになれる

### ユーザーイリュージョン

> "目のまえにある知覚できるものが、その人にとってのコンピューターである"

- UIは操作盤ではなく「ユーザーと目当てを直接つなぐ空間」
- 画面上のオブジェクトがリアルな仕事の対象として知覚されるとき、人とコンピューターの境界は溶ける
- この擬似世界（イリュージョン）を作るためにOOUIがあり、それを実装するためにOOPがある

## 4. ソフトウェアデザインの3レイヤー

OOUIを構造的に分解すると3層になる:

```
┌─────────────────────────────────────────┐
│  プレゼンテーション層（Look）              │
│  グラフィック・レイアウト・アニメーション   │
├─────────────────────────────────────────┤
│  インタラクション層（Feel）               │
│  ビュー種別・ナビゲーション・CRUD・ビジネスロジック │
├─────────────────────────────────────────┤
│  モデル層（設計全体の約60%）              │
│  ユーザーのメンタルモデルを反映したオブジェクト構成 │
└─────────────────────────────────────────┘
```

### 各層の役割

**モデル層**
- ユーザーが意識する概念（エンティティ）を定義
- プログラム実装上の内部オブジェクトは含めない
- ユーザー調査・タスク分析が前提。不十分だとシステム全体が誤方向に向かう

**インタラクション層**
- コレクション/シングルのビュー割り当て
- ビュー間のナビゲーション（呼び出し関係）
- CRUD（作成・閲覧・更新・削除）の設計

**プレゼンテーション層**
- レイアウトパターン適用
- オブジェクトの視覚的表象（アイコン・色・形）
- モデル・インタラクションとの整合を取りながら並行設計

### デザインの非線型性

設計は「抽象→具象」の線型プロセスではない。
優れた実践者は「形から入り、後からロジックを見出す」（アブダクション）。
モデル・インタラクション・プレゼンテーションを行き来しながら全体性を高める。

## 5. タスク指向UIが生まれる背景

### 3つの原因

1. **業務分析がタスクをまとめる**: 業務＝仕事の流れ＝タスクであり、ユーザー要求も「やること」としてまとめられる。そのままUIに反映するとタスク指向になる

2. **利用手続きをデザインしようとする**: UXの向上を「ユーザーの利用手続きのデザイン」として捉えると、タスク指向のUIになる

3. **GUIの設計原則が周知されていない**: オブジェクト指向UIの設計手法が明文化・普及していないため、設計者が意図せずタスク指向を選択する

### 例外的にタスク指向が許容されるケース

- オブジェクトが限定的で選択の必要がない場合（例: ATM ← カードがオブジェクト選択を代行）
- 自己完結的な定型入力フローのみを提供する場合
- ユーザーの行動をあえて制限して合目的性を高めたい場合

## 6. モードレスデザインの原則

### モードとは何か

モードとは「同じジェスチャが異なる意味を持つ状態」。

- モードがあると: ユーザーは作業の途中で考えを変えた際、まずモードから抜けるための操作が必要になる
- モードレスだと: いつでも考えを変えて別な作業を始められる。操作の順序が自由になる

**モードエラー**: UIがモードを持つとき、「インタフェースの現在の状態がユーザーの注意の所在となっておらず、同じジェスチャに複数の異なった応答が実行される」ことで生じる操作ミス。

### タスク指向とモードの関係

「動詞→名詞」のタスク指向操作はモーダル（モードがある）:

```
タスク選択（動詞） → モードに入る → オブジェクト指定（名詞） → 処理完了でモード解除
```

「名詞→動詞」のオブジェクト指向操作はモードレス:

```
オブジェクト選択（名詞） → いつでも変更可 → アクション選択（動詞） → 即時完了
```

### モードレスにする方法

| 手法 | 説明 | 例 |
|------|------|-----|
| 名詞→動詞構文 | アプリ全体の操作を Object→Verb に統一 | ファイル選択→メニューから「開く」 |
| ダブルクリック/タップ | 対象選択とアクション実行を1ジェスチャに統合 | デスクトップのファイルをダブルクリック |
| パラメーターをオブジェクト化 | 保存済み検索条件・設定をオブジェクトとして提示 | プレイリスト、スマートフォルダ |
| アクションを分割 | 複雑な手続きを「1ストローク即時実行」の組み合わせに | カット&ペースト（テスラー） |
| サブミットボタン廃止 | 入力変更をリアルタイムにオブジェクトへ反映 | データバインド・モードレスエディットパターン |
| モードレスコンファーム | 削除確認をモードを作らずに実現 | iPhoneアラームの2段階削除 |
| デフォルトテンプレート | 新規作成時にデフォルト値でインスタンス生成し、後から編集 | Keynoteの図形追加（ガッツパターン） |

### 道具の抽象性とモードレスの関係

| 道具 | 抽象度 | モード |
|------|--------|--------|
| りんご皮むき機 | 低（専用） | モーダル（特定用途に固定） |
| 果物ナイフ | 高（汎用） | モードレス（様々な使い方が可能） |
| タスク指向UI | 低 | モーダル（手順に従うだけ） |
| オブジェクト指向UI | 高 | モードレス（ユーザーが自由に使いこなす） |

> 「良いデザインとは、ユーザーに合わせたものではなく、ユーザーが自らを合わせられるものである」

抽象度の高い道具はモードレスであり、ユーザーの習熟とともに意味性が広がる。
これが、タスク指向（低抽象・モーダル）ではなくオブジェクト指向（高抽象・モードレス）を採用する根本的な理由。

## 7. ユーザー、タスク、オブジェクトの関係

### タスク指向の構造

```
ユーザー → タスク選択 → オブジェクト操作
```

- タスクの数は通常オブジェクトより多いため、認知負荷が高い
- 操作開始時にユーザーの関心対象（オブジェクト）が見えていない
- 画面数が多くなりやすい（タスク指向をオブジェクト指向に変えると画面数が5〜20分の1になることも）

### オブジェクト指向の構造

```
ユーザー → オブジェクト選択 → タスク（アクション）実行
```

- ユーザーの関心対象が最初から見えている
- ひとつのオブジェクトにタスクがまとめられるため設計がシンプル
- 操作手順を考える認知負荷が小さい

### 設計上の原則

アプリケーション発想時にタスクが先行した場合でも:
1. そのタスクの対象となっているオブジェクトを抽出する
2. オブジェクトをUIの出発点にする
3. タスクはオブジェクトに対するアクションとして位置づける

タスクを画面フローとしてそのまま実装するのではなく、**タスクに必要な情報オブジェクトを定義し、ユーザーがそれらに自由な順序で働きかけながら目的を達成できるようにする**。

## 8. デザインプロセスのミッシングリンク

ISO 9241-210（人間中心設計）などのUXデザインプロセスは、「利用状況の把握→ユーザー要求の特定→デザイン試作→評価」というサイクルを提示する。しかし、**「特定されたユーザー要求をどうUIに落とし込むか」というロジックが抜け落ちている**。

OOUIの設計メソッドがこのミッシングリンクを埋める:

```
[前段: リサーチ]        [ミッシングリンク]        [後段: 品質保証]
ユーザー要求の特定  →  オブジェクトモデリング  →  デザイン評価
                       ↓
                  OOUI設計3ステップ
                  Step1: オブジェクト抽出
                  Step2: ビュー/ナビゲーション検討
                  Step3: レイアウトパターン適用
```

### オブジェクトとして認識できる概念の条件

ある概念がUIのオブジェクトとして適切かの判断基準:

| 条件 | 説明 | 例（メールクライアント） |
|------|------|----------------------|
| 数えられる名詞 | 「1件の〇〇」と表現できる | 1件のメッセージ、1つのメールボックス |
| 同種の集合として管理 | 複数が束になって扱われる | メッセージ一覧、メールボックス一覧 |
| 共通のアクションを持つ | 同種に同じ操作が適用できる | 各メッセージに削除・移動・フラグ |

**オブジェクトではない概念の例**:
- 「受信」— 動詞的にも名詞的にも取れ、タスク（動詞）として扱う方が適切
- 「フラグ」— メッセージの属性であり、それ自体を独立したオブジェクトとして管理しない
- 内部処理のロジック用クラス — UIに現れる必要がない実装上の概念

### ビューの基本形式

オブジェクトは必ず「コレクション」か「シングル」のビューで表象される:

| ビュー種別 | 説明 | 表示情報量 | 典型的な役割 |
|-----------|------|-----------|------------|
| コレクションビュー | 同種オブジェクトを複数並べて表示 | 主要属性のみ | 一覧・探索・フィルタリング |
| シングルビュー | オブジェクト1件を詳細表示 | 全属性 + アクション | 閲覧・編集・個別操作 |

コレクション→シングルへの遷移が「オブジェクトを選んでから詳細を見る」という「名詞→動詞」構文を支える。

### デバイス間の一貫したUIモデル

同じオブジェクトモデル（コレクション+シングル）でも、デバイスによってレイアウトを変えられる:

```
[デスクトップ: 1画面内にペインとして並置]
┌──────────┬──────────────┬────────────────────┐
│ MBX一覧  │ MSG一覧       │ MSG詳細             │
└──────────┴──────────────┴────────────────────┘

[モバイル: 画面遷移で同じ構造を実現]
MBX一覧 → MSG一覧 → MSG詳細
```

UIモデルが正しければ、表現形式はデバイスに合わせて柔軟に変えられる。

## 9. OOUIと現代のUI設計

### タスク指向UIの典型パターン（アンチパターン）

| パターン | 症状 | 影響 |
|---------|------|------|
| タスク別入口 | トップ画面にボタンが「やること」単位で並ぶ | 目的不明確、ユーザーが迷う |
| 情報なき合意 | やることを選ばないとオブジェクトが見えない | インタラクションコスト高 |
| 機能重複 | 同じオブジェクトへの操作が複数の入口に散在 | 設計が恣意的で複雑 |
| 線形強制フロー | 電話/紙の手順をそのままデジタル化 | 自由度なし、GUIの利点を活かせない |

### OOUIによる改善効果

タスク指向→オブジェクト指向への転換で得られる効果:

- **画面数削減**: 5〜20分の1に減ることも（重複・派生画面が統合される）
- **操作手順の圧縮**: 同じ目的を達成するための操作ステップが大幅に減る
- **設計のシンプル化**: 機能重複がなくなり開発工数・保守コストが下がる
- **学習効率向上**: ナビゲーションが「名詞形」になり、どこに何があるか直感的にわかる
- **拡張性向上**: 新しい要求が来ても既存オブジェクトへのアクション追加で対応できる

### デジタルトランスフォーメーションとの接点

既存業務の手順をそのままデジタル化するとタスク指向になる。
OOUIの実践は「デジタルのネイチャーに合わせて業務を設計し直す」という視点変換を促す。
これが本来のDX（デジタルトランスフォーメーション）の意味に対応している。
