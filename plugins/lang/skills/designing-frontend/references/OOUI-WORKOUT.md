# OOUI 設計判断パターン集

18のワークアウトから抽出したオブジェクト指向UI設計の実践的判断パターン。

---

## 1. オブジェクト抽出の判断パターン

### 1.1 名詞分析による基本抽出

タスク記述・画面内のラベル・ナビゲーション項目から**名詞**を収集する。
既存のタスク指向UIからリバースエンジニアリングする場合も同様に名詞を拾う。

```
タスク記述例: 「仕入れで購入した全商品の商品名・単価・個数を確認する」
     ↓
抽出名詞: 仕入れ、商品、単価、個数
     ↓
メインオブジェクト候補: 仕入れ、商品（の種類）
```

### 1.2 暗黙的オブジェクトの3パターン

#### パターンA：同一語の2意味（インスタンス vs. 型）

同じ言葉が「物理的な実体」と「種類・カテゴリー」の両方の意味で使われている場合、
それぞれ別のオブジェクトとして分離する。

| 文脈 | オブジェクト |
|------|------------|
| 「公園に設置されたブランコ」 | 設備（インスタンス） |
| 「ブランコのある公園を探す」 | 設備の種類（型） |

#### パターンB：2オブジェクト間の関係が新オブジェクトになる（明細パターン）

2つのオブジェクトが多対多で関係するとき、その間に**明細オブジェクト**が潜む。

```
仕入れ ×（多）商品の種類 → 仕入れの明細（商品の種類 + 数量のセット）
棚卸し ×（多）商品の種類 → 棚卸しの明細
レシート ×（多）商品 → 注文明細
```

一般的な名称がないオブジェクトほど見落としやすい。
「明細」「ログ」「割り当て」などの概念を積極的に疑う。

#### パターンC：タスク群の共通対象（隠れたコアオブジェクト）

複数のタスク名（動詞）が同一の概念を対象にしている場合、
その共通対象がメインオブジェクトとして隠れている。

```
申請する・精算する・承認する → すべての対象 = 「出張」
新規申請・変更申請・解約申請・承認 → すべての対象 = 「契約」
登録・照会・更新・削除（CRUD）→ すべての対象 = 「アセット」
```

### 1.3 オブジェクト名の汎化・特化の判断

**汎化すべきケース**：将来的な拡張が見込まれる場合
- 例: 会議室 → スペース（将来、屋外エリアも管理する可能性）

**汎化を抑えるケース**：ユーザーへの表示名が難解になる場合
- 例: 「遊具の種類」は概念として正確でも、ユーザーには単に「遊具」と表示する

### 1.4 手順の入出力からオブジェクトを抽出

ウィザードの各ステップの入出力情報を分析し、共通項からオブジェクトを見つける。

```
入力: 計算元通貨の種類・金額 / 出力: 計算先通貨の種類・金額
→ 共通項: 「通貨」オブジェクト（種類・金額のプロパティ）
```

### 1.5 アクションのオブジェクト化

「〇〇を計算する」アクションは、計算結果を**プロパティ**として持つオブジェクトに変換する。

- BMIを計算するボタン → 生徒オブジェクトのBMIプロパティ（身長・体重から自動算出）
- 売上原価を計算するボタン → レポートオブジェクトの売上原価プロパティ

## 2. ドメイン別のメインオブジェクト選定パターン

### 管理系（CRUD中心）

- 管理対象そのものがオブジェクト（アセット管理 → アセット）
- CRUD操作すべてが1つのオブジェクトを対象 → コレクション+シングルに集約
- 10画面 → 3画面への圧縮例（照会/更新/削除を統合）

### 予約系

| 主要オブジェクト | コレクション表示形式 |
|---------------|------------------|
| 予約 | カレンダー（日時×リソースの2軸マッピング） |
| スペース（会議室） | リスト or グリッド |

- 「予約をしたユーザー」はタスク記述に現れなくても実装上必要なケースがある

### 申請・ワークフロー系

```
アンチパターン: 申請フロー・精算フロー・承認フローを別々の画面として作る
                         ↓
OOUIパターン: 「出張」「契約」等のコアオブジェクトを抽出し、
              ステータス（申請中/精算済/承認待ち）をプロパティとして持たせる
```

- ステータスをコレクションのフィルタープリセットとして実装
- ロール別の表示切り替えはオブジェクトのプロパティ（承認可能性フラグ等）で制御

### 集計・レポート系

- 集計データを表示するための「レポート」オブジェクトを定義
- 集計結果はプロパティ（動的に算出される表示値）として扱う
- 動的レポートか静的スナップショットかでモデルが変わる

### 営業・顧客管理系

- 顧客を「企業」と「人」に分離（人は企業に所属）
- 主要オブジェクト（顧客・案件・訪問予定・資料）をルートナビゲーションに配置

## 3. ビューとナビゲーションの設計パターン

### コレクションビューの形式選択

| 特性 | 推奨形式 |
|------|---------|
| 一般的なリスト | 1次元リスト（固定高 or 可変高） |
| 空間的配置が重要 | 地図マッピング |
| 時間軸が重要 | カレンダー / ガントチャート |
| 少数アイテム（視覚的） | グリッド（タイル） |

### コレクションビューのプリセット（業務コンテキスト対応）

同一オブジェクトに複数の業務文脈がある場合、タスク別フローではなく**プリセット**で対応する。

```
タスク指向: 満期管理画面 / 保険料収納画面 / 契約照会画面（それぞれ別入口）
         ↓
OOUI: 「契約」コレクションビューにプリセットを用意
      - すべての契約（デフォルト）
      - 満期が近い契約（業務コンテキストA）
      - 保険料未納の契約（業務コンテキストB）
      - 自分が担当する契約（個人コンテキスト）
```

### シングルビューの設計判断

| 状況 | 判断 |
|------|------|
| 情報量が多い・種類が異なる | タブで複数シングルビューに分割（例: 設定/内容） |
| 閲覧のみの用途がある | 閲覧用と編集用を分離 |
| 情報量が少ない | 省略し、コレクション行への展開で対応 |
| 親オブジェクトのシングルに一体化 | 子オブジェクトのコレクションを内包 |

### ビューの省略判断

- 情報量が少ないオブジェクトのシングルビューは省略可
- フィルタリング目的のみで存在するオブジェクトはビュー不要（例: カテゴリー、お気に入り）
- タスクの中でユーザーが操作の手掛かりとして意識しないオブジェクトはビュー不要

### ルートナビゲーション選定基準

✅ **配置する**：ユーザーの思考の起点になるオブジェクト
❌ **配置しない**：上位オブジェクト経由でのみアクセスするオブジェクト

同一オブジェクトでもコンテキストにより表示形式が変わる（全店舗リスト vs. 会場内地図）。

オブジェクト同士がループする場合、2ペインレイアウトは避けフルページ遷移を採用する。

## 4. タスク指向 → オブジェクト指向への変換パターン

### タスク指向UIの症状チェックリスト

- [ ] ナビゲーション/メニューの大項目が「やること」（動詞・行為）になっている
- [ ] 同じ検索・一覧画面がタスク別に複数存在する
- [ ] 「動詞を選ぶ → 対象を選ぶ」という順序（先にタスクを選ぶ）
- [ ] 線形ウィザード形式の多画面フロー
- [ ] 管理対象のオブジェクトが画面上に視覚的に現れていない

### 変換の5ステップ

```
Step 1: 現行UIのメニュー・ラベルから名詞を全収集
Step 2: 動詞タスクが共通して対象にしている名詞を特定
Step 3: その名詞をメインオブジェクトとして定義・モデル化
Step 4: コレクションビュー + シングルビューを構成
Step 5: タスクの条件 → ステータス/フィルタープリセット/アクションに変換
```

### タスクの変換対応表

| タスク指向 | オブジェクト指向への変換 |
|-----------|----------------------|
| タスク別の入口（新規・更新・削除） | 単一コレクション + シングルビュー |
| 重複する検索フォーム | フィルタリング付きコレクションビュー |
| ステータス別の画面 | オブジェクトのステータスプロパティ + フィルタープリセット |
| ウィザードの手順 | 入出力からオブジェクトを抽出しビューを直接操作 |
| 計算・集計アクション | オブジェクトの算出プロパティ |
| ロール別の画面 | 権限プロパティによる表示切り替え |

## 5. アンチパターン集

### オブジェクト抽出の失敗パターン

| アンチパターン | 正しいアプローチ |
|-------------|---------------|
| タスク名（動詞）をオブジェクトと誤認 | 動詞を除去し、その対象となる名詞を探す |
| 関係テーブル的な概念（明細）の見落とし | 2オブジェクト間の多対多関係を疑う |
| 同義語の混在（会議室とスペース） | 一貫した名前でオブジェクトを定義する |
| 汎化しすぎて意味が失われる | ユーザーの語彙に合わせた名称を選ぶ |
| ユーザーが意識しないオブジェクトをUIに出す | 操作の手掛かりにならないオブジェクトはビュー省略 |

### ビュー設計の失敗パターン

| アンチパターン | 正しいアプローチ |
|-------------|---------------|
| CRUDそれぞれに別の入口を作る | 単一コレクション+シングルビューに集約 |
| 業務フロー別に同じオブジェクトの画面を複製 | コレクションプリセットで対応 |
| 全オブジェクトをルートナビゲーションに並べる | 思考起点のオブジェクトのみを選定 |
| 横スクロールが発生する表形式 | ペイン分割またはシングルビューで詳細表示 |

## 6. 複雑度別の設計アプローチ

### 単純（メインオブジェクト1種）

```
構成: コレクションビュー → シングルビュー
レイアウト: デスクトップ=2ペイン、モバイル=ページ分離
適用例: メモアプリ、アセット管理
```

### 中程度（メインオブジェクト2-3種）

```
構成: 各オブジェクトにコレクション+シングル
      シングルビューに関連オブジェクトのコレクションを一体化
ナビ: 思考起点の2-3オブジェクトをルートナビゲーションに配置
ループ注意: ループがある場合は2ペイン採用を見送る
適用例: 社員名簿（社員+チーム）、予約管理（予約+スペース）
```

### 複雑（業務アプリ・多数オブジェクト）

```
追加パターン:
  - 業務タスクをコレクションプリセットとして実装
  - ビューを持たないモデル専用オブジェクトが存在する
  - ロール別の表示プロパティを設ける
  - 段階的拡張: 共通オブジェクト（商品）を維持しながら新機能（棚卸し・レポート）を追加
適用例: 保険契約管理、出張申請・精算、商品管理+仕入れ+棚卸し
```
