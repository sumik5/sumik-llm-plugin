## 1. オブジェクト所有権

### 1.1 基本原則

PostgreSQLでは **オブジェクトの所有者（または スーパーユーザー）のみ** がオブジェクトを変更・削除できる。他のロールが `CREATE` 権限を持っていても、自分が所有していないオブジェクトは変更不可。

```sql
-- オブジェクトの所有者を変更
ALTER TABLE public.rental OWNER TO new_owner;
ALTER FUNCTION f_calculate_total OWNER TO new_owner;
ALTER SCHEMA api OWNER TO new_owner;

-- スキーマ内のすべてのオブジェクトの所有権を一括移譲
REASSIGN OWNED BY old_owner TO new_owner;

-- 特定のデータベース内で所有権移譲する場合（セッションごとに実行）
-- \c target_database
-- REASSIGN OWNED BY old_role TO new_role;
```

### 1.2 オブジェクト作成に必要な権限

| オブジェクト | 必要な権限 |
|------------|----------|
| データベース | `CREATEDB` ロール属性 |
| スキーマ | データベースへの `CREATE` 権限 |
| テーブル | スキーマへの `USAGE` + `CREATE` 権限 |
| インデックス | テーブルの所有者であること |
| ビュー | スキーマへの `USAGE` + `CREATE` 権限 |
| マテリアライズドビュー | スキーマへの `USAGE` + `CREATE` 権限 |
| 外部テーブル | スキーマへの `USAGE` + `CREATE` 権限 |
| シーケンス | スキーマへの `USAGE` + `CREATE` 権限 |
| 型 | データベースへの `CREATE` 権限 |
| ドメイン | 基底型への `USAGE` 権限 |
| トリガー | スキーマへの `USAGE` + `CREATE` 権限 |

## 2. スキーマ

### 2.1 スキーマの役割

スキーマはデータベース内のオブジェクトをグループ化する名前空間。

**推奨設計パターン:**
```sql
-- 内部実装スキーマ（外部アクセス禁止）
CREATE SCHEMA customer;
CREATE SCHEMA sales;
CREATE SCHEMA inventory;

-- 公開APIスキーマ（外部からアクセス可能な唯一の入口）
CREATE SCHEMA api;
```

### 2.2 publicスキーマのセキュリティ注意事項

- PostgreSQL 15以前: `PUBLIC` ロールがデフォルトでpublicスキーマに `CREATE` 可能（セキュリティリスク）
- PostgreSQL 15以降: デフォルトで `CREATE` 権限が剥奪済み
- **推奨**: アプリケーションオブジェクトは明示的に命名したスキーマに配置

```sql
-- PostgreSQL 14以前での対応
REVOKE CREATE ON SCHEMA public FROM PUBLIC;
```

## 3. テーブル

### 3.1 基本テーブル作成

```sql
CREATE TABLE public.rental (
  rental_id     bigint PRIMARY KEY NOT NULL,
  rental_period tstzrange,
  inventory_id  integer NOT NULL
    REFERENCES inventory (inventory_id) ON DELETE RESTRICT,
  customer_id   integer NOT NULL
    REFERENCES customer (customer_id) ON DELETE RESTRICT,
  staff_id      integer NOT NULL,
  last_update   timestamp with time zone DEFAULT now() NOT NULL
);
```

### 3.2 IDENTITYカラム

PostgreSQL 11以降推奨。古い `SERIAL` / `BIGSERIAL` より意図が明確。

**`GENERATED BY DEFAULT AS IDENTITY`（値の明示的INSERT許可）:**
```sql
CREATE TABLE public.rental (
  rental_id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY NOT NULL,
  -- ...
);
```

**`GENERATED ALWAYS AS IDENTITY`（明示的INSERTを禁止）:**
```sql
CREATE TABLE public.rental (
  rental_id bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY NOT NULL,
  -- ...
);
```

> ⚠️ `GENERATED ALWAYS` でも明示的な値を挿入したい場合は `OVERRIDING SYSTEM VALUE` を追加する（SQL Serverの `SET INSERT_IDENTITY=ON` 相当は存在しない）:
> ```sql
> INSERT INTO public.rental (rental_id, ...) OVERRIDING SYSTEM VALUE VALUES (999, ...);
> ```

| 方式 | 明示INSERT | 意図の明確さ | 推奨度 |
|------|-----------|------------|--------|
| SERIAL | ✅ 可能 | △ 曖昧 | ❌ 非推奨（PostgreSQL 11以降） |
| GENERATED BY DEFAULT | ✅ 可能 | ✅ 明確 | ✅ |
| GENERATED ALWAYS | ❌ 不可（OVERRIDINGで可能） | ✅ 最も明確 | ✅ SQL Server移行者向け |

### 3.3 テーブルスキーマのコピー（CREATE TABLE ... LIKE）

開発・テスト環境やデータ処理での一時テーブル作成に有効。

```sql
-- すべての属性（制約・統計・インデックス等）をコピー
CREATE TABLE bluebox.rental_new
  (LIKE bluebox.rental INCLUDING ALL);

-- インデックスを除いてコピー
CREATE TABLE bluebox.rental_new
  (LIKE bluebox.rental INCLUDING ALL EXCLUDING INDEXES);
```

**コピー対象の制御オプション:**

| オプション | 内容 |
|-----------|------|
| `INCLUDING ALL` | すべての属性をコピー |
| `INCLUDING CONSTRAINTS` | CHECK制約・NOT NULL等 |
| `INCLUDING DEFAULTS` | デフォルト値 |
| `INCLUDING INDEXES` | インデックス |
| `INCLUDING STATISTICS` | 統計情報 |
| `EXCLUDING INDEXES` | インデックスを除外 |

### 3.4 パーティショニング

PostgreSQL 10以降のネイティブパーティショニング。3種類をサポート。

**⚠️ 重要**: 既存テーブルは直接パーティション化できない。"expand/contract"パターン（新テーブル作成→データ移行→テーブルリネーム）が必要。

**RANGE パーティショニング（日付範囲の例）:**
```sql
-- 親テーブル（データを保持しないシェル）の作成
CREATE TABLE bluebox.payment (
  payment_id   serial4 NOT NULL,
  customer_id  int4 NOT NULL,
  rental_id    int4 NOT NULL,
  amount       numeric(5, 2) NOT NULL,
  payment_date timestamptz NOT NULL,
  CONSTRAINT payment_bak_pkey PRIMARY KEY (payment_date, payment_id)  -- パーティションキーを主キーに含める必須
) PARTITION BY RANGE (payment_date);

-- 月次パーティションの作成
CREATE TABLE bluebox.payment_y2024m01
  PARTITION OF bluebox.payment FOR VALUES
  FROM ('2024-01-01') TO ('2024-02-01');

CREATE TABLE bluebox.payment_y2024m02
  PARTITION OF bluebox.payment FOR VALUES
  FROM ('2024-02-01') TO ('2024-03-01');

CREATE TABLE bluebox.payment_y2024m03
  PARTITION OF bluebox.payment FOR VALUES
  FROM ('2024-03-01') TO ('2024-04-01');
```

**LIST パーティショニング（地域コードの例）:**
```sql
CREATE TABLE orders (
  order_id  bigint NOT NULL,
  region    text NOT NULL,
  amount    numeric(10, 2)
) PARTITION BY LIST (region);

CREATE TABLE orders_japan   PARTITION OF orders FOR VALUES IN ('JP');
CREATE TABLE orders_us      PARTITION OF orders FOR VALUES IN ('US');
CREATE TABLE orders_default PARTITION OF orders DEFAULT;  -- その他のすべての値
```

**HASH パーティショニング（均等分散の例）:**
```sql
CREATE TABLE user_events (
  event_id bigint NOT NULL,
  user_id  bigint NOT NULL,
  payload  jsonb
) PARTITION BY HASH (user_id);

CREATE TABLE user_events_p0 PARTITION OF user_events FOR VALUES WITH (MODULUS 4, REMAINDER 0);
CREATE TABLE user_events_p1 PARTITION OF user_events FOR VALUES WITH (MODULUS 4, REMAINDER 1);
CREATE TABLE user_events_p2 PARTITION OF user_events FOR VALUES WITH (MODULUS 4, REMAINDER 2);
CREATE TABLE user_events_p3 PARTITION OF user_events FOR VALUES WITH (MODULUS 4, REMAINDER 3);
```

| パーティション種類 | 適するユースケース | キーの制約 |
|-----------------|-----------------|----------|
| **RANGE** | 時系列データ・日付範囲 | パーティションキーを主キーに含める必須 |
| **LIST** | 列挙可能な区分値（地域・カテゴリ等） | 同上 |
| **HASH** | 均等分散・特定の順序がない大量データ | 同上 |

## 4. シーケンス

### 4.1 基本動作

シーケンスはユニークな整数を順番に返す自動採番オブジェクト。内部的には1行テーブルとして実装されている。

```sql
-- シーケンスの手動作成
CREATE SEQUENCE public.order_id_seq
  START WITH 1
  INCREMENT BY 1
  NO MINVALUE
  NO MAXVALUE
  CACHE 1;

-- テーブルカラムへのシーケンス紐付け（IDENTITYカラムでは自動作成）
ALTER SEQUENCE public.order_id_seq OWNED BY public.orders.order_id;
```

### 4.2 シーケンス操作関数

```sql
-- 次の値を取得（消費）
SELECT nextval('public.order_id_seq');

-- 現在の値を参照（同セッション内で nextval 後のみ有効）
SELECT currval('public.order_id_seq');

-- 値を任意の数に設定
SELECT setval('public.order_id_seq', 1000);         -- 次は1001
SELECT setval('public.order_id_seq', 1000, false);  -- 次は1000（is_calledをfalseに）
```

### 4.3 SERIAL vs IDENTITY

```sql
-- 古いSERIALパターン（非推奨）
CREATE TABLE old_style (id BIGSERIAL PRIMARY KEY);

-- 現代的なIDENTITYパターン（推奨）
CREATE TABLE new_style (id bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY);
```

## 5. インデックス

PostgreSQLは6種類のインデックスをネイティブサポート。B-Tree・GiST・GIN・BRINは複数カラムのインデックスをサポート。B-Tree・GiST・SP-GiSTはカバリングインデックス（`INCLUDE`）をサポート。

**重要な特徴**: `CREATE INDEX CONCURRENTLY` でテーブルロックなしにインデックスを作成可能（商用DBでは有償機能相当）。

### 5.1 インデックス種類の選択ガイド

| インデックス | 最適なユースケース | 対応演算子 | 備考 |
|------------|----------------|----------|------|
| **B-Tree** | 等値・範囲検索（デフォルト） | `=`, `<`, `>`, `<=`, `>=`, `BETWEEN`, `IN` | PostgreSQL 13+でデデュプリケーション対応 |
| **HASH** | 等値検索のみ | `=` のみ | B-Treeより小さいサイズ・高速等値検索 |
| **GiST** | 地理空間・範囲型・全文検索 | `&&`, `@>`, `<@`, `<<`, `>>` | 拡張可能なフレームワーク |
| **SP-GiST** | 非均一データ・電話番号・IP | `=`, `<`, `>` | スペース分割木（非平衡が特徴） |
| **GIN** | 配列・JSONB・全文検索 | `@>`, `<@`, `&&`, `@@` | 複合値の転置インデックス |
| **BRIN** | 大規模テーブルの自然順序データ | 範囲の包含 | 非常に小さい・時系列データ向け |

### 5.2 B-Tree（デフォルト）

```sql
-- 基本的なB-Treeインデックス
CREATE INDEX rental_inventory_id_idx
  ON bluebox.rental USING btree (inventory_id);

-- 複合インデックス（左端のカラムから順番に使用される）
CREATE INDEX idx_customer_email_status
  ON public.customer (email, status);

-- カバリングインデックス（INCLUDE句でクエリの追加カラムをカバー）
CREATE INDEX idx_rental_covering
  ON bluebox.rental (inventory_id)
  INCLUDE (customer_id, rental_period);

-- CONCURRENTLY（テーブルロックなし）
CREATE INDEX CONCURRENTLY idx_payment_date
  ON bluebox.payment (payment_date);
```

**デデュプリケーション（PostgreSQL 13+）**: 同じインデックス値を持つ複数行を配列でまとめる最適化。デフォルト有効。低カーディナリティのカラムでストレージ効率が向上。

### 5.3 HASH

```sql
-- HASHインデックス（等値検索専用）
CREATE INDEX rental_inventory_id_hash_idx
  ON bluebox.rental USING hash (inventory_id);
```

> **選択基準**: キー値がB-Treeより小さいハッシュ値に変換されるため、インデックスサイズが小さくなる。等値比較のみが必要な場合に検討する。

### 5.4 GiST（Generalized Search Tree）

```sql
-- 範囲型（tstzrange等）へのGiSTインデックス
CREATE INDEX rental_rental_period_idx
  ON bluebox.rental USING gist (rental_period);

-- 地理空間データ（PostGIS使用）
CREATE INDEX idx_location_gist
  ON store USING gist (location);  -- geometry型
```

**対応データ型（標準）**: geometry、range/multirange、inet、tsvector（全文検索）

**GiSTで可能な範囲型クエリ:**
```sql
-- 特定日時を含む予約期間を検索
SELECT * FROM rental WHERE rental_period @> '2024-06-15'::timestamptz;

-- 重複する期間を検索
SELECT * FROM rental WHERE rental_period && '[2024-06-01, 2024-06-30]'::tstzrange;
```

### 5.5 SP-GiST（Space-Partitioned GiST）

```sql
-- 電話番号等の非均一データ
CREATE INDEX customer_phone_idx
  ON bluebox.customer USING spgist (phone);
```

> **特徴**: ツリーが非平衡になることが設計上の特徴。データが上位ノードに偏在する（クラスタリングがある）場合に効率的。

### 5.6 GIN（Generalized Inverted Index）

```sql
-- 全文検索（tsvector型）
CREATE INDEX film_fulltext_idx
  ON bluebox.film USING gin (fulltext);

-- JSONB列の検索
CREATE INDEX idx_product_attributes_gin
  ON product USING gin (attributes);

-- 配列型の検索
CREATE INDEX idx_genre_ids_gin
  ON film USING gin (genre_ids);

-- pg_trgm拡張でのトライグラム検索（部分一致）
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE INDEX idx_film_title_trgm
  ON film USING gin (title gin_trgm_ops);
```

**GINの特徴**: 各キー値がインデックスに1回のみ格納され、そのキーを含む行へのポインタリストを保持する（転置インデックス）。

### 5.7 BRIN（Block Range Index）

```sql
-- 時系列データへのBRINインデックス
CREATE INDEX idx_rental_rental_date_brin
  ON bluebox.rental USING brin (lower(rental_period));

-- カスタムページ範囲サイズ（デフォルト128ページ）
CREATE INDEX idx_payment_brin
  ON payment USING brin (payment_date)
  WITH (pages_per_range = 64);
```

**動作原理**: 指定ページ数の範囲ごとに最小値・最大値を記録。クエリ値がその範囲に含まれない場合、128ページを丸ごとスキップ。

**BRIN適用条件:**
- ✅ テーブルが非常に大きい（数GB以上）
- ✅ カラムの値がディスク上の物理順序と相関している（時系列・採番等）
- ✅ インデックスサイズの大幅削減が必要
- ❌ ランダムアクセスパターン（相関がない場合は効果薄）

## 6. ビューとマテリアライズドビュー

### 6.1 ビュー（View）

ビューは定義されたクエリを参照のたびに実行する仮想テーブル。

```sql
-- ビューの作成
CREATE VIEW api.active_customers AS
SELECT
  c.customer_id,
  c.full_name,
  c.email,
  COUNT(r.rental_id) AS total_rentals
FROM customer c
LEFT JOIN rental r ON r.customer_id = c.customer_id
WHERE c.active = true
GROUP BY c.customer_id, c.full_name, c.email;

-- ビューの更新（パラメータ変更不可）
CREATE OR REPLACE VIEW api.active_customers AS
-- ...更新されたクエリ

-- ビューの削除
DROP VIEW api.active_customers;
```

> **権限**: ビューの所有者またはスーパーユーザーのみが変更可能。参照ロールがテーブルに直接アクセス権を持たなくても、ビュー経由でデータを見られる（`SECURITY DEFINER` パターンに有用）。

### 6.2 マテリアライズドビュー（Materialized View）

クエリ結果をディスクに物理化して保存。SQL Serverの「インデックス付きビュー」に相当。

```sql
-- マテリアライズドビューの作成
CREATE MATERIALIZED VIEW sales_summary AS
SELECT
  date_trunc('month', payment_date) AS month,
  SUM(amount) AS total_revenue,
  COUNT(*) AS transaction_count
FROM payment
GROUP BY date_trunc('month', payment_date);

-- 通常インデックスも追加可能
CREATE INDEX idx_sales_summary_month
  ON sales_summary (month);

-- 手動リフレッシュ（テーブルロックあり）
REFRESH MATERIALIZED VIEW sales_summary;

-- 同時リフレッシュ（ロックなし・既存データ参照可能）
REFRESH MATERIALIZED VIEW CONCURRENTLY sales_summary;
-- ※ CONCURRENTLY にはユニークインデックスが必要
```

**ビュー vs マテリアライズドビューの選択基準:**

| 観点 | ビュー | マテリアライズドビュー |
|-----|--------|---------------------|
| データ鮮度 | 常に最新 | リフレッシュ時点のスナップショット |
| クエリ速度 | ベーステーブルに依存 | 事前計算済みで高速 |
| ストレージ | なし | 必要 |
| 適するケース | 常に最新が必要・軽量クエリ | 重い集計クエリ・DWH系 |

## 7. 外部テーブル（Foreign Table）

### 7.1 概念

Foreign Data Wrapper (FDW) を使って、PostgreSQL外部のデータソースをSQLで操作できる仮想テーブル。

```
外部データソース（CSV、別DB、S3等）
    ↓ FDW（Foreign Data Wrapper）
PostgreSQL外部テーブル（Foreign Table）
    ↓ 通常のSQLで操作
アプリケーション
```

### 7.2 postgres_fdwを使った別PostgreSQLへの接続

```sql
-- 1. 拡張機能のインストール
CREATE EXTENSION IF NOT EXISTS postgres_fdw;

-- 2. 外部サーバーの定義
CREATE SERVER remote_db_server
  FOREIGN DATA WRAPPER postgres_fdw
  OPTIONS (host 'remote.example.com', port '5432', dbname 'source_db');

-- 3. ユーザーマッピング（接続認証情報）
CREATE USER MAPPING FOR current_user
  SERVER remote_db_server
  OPTIONS (user 'remote_user', password 'remote_password');

-- 4. 外部テーブルの作成
CREATE FOREIGN TABLE remote_orders (
  order_id   integer NOT NULL,
  customer_id integer NOT NULL,
  amount     numeric(10, 2),
  order_date timestamptz
)
SERVER remote_db_server
OPTIONS (schema_name 'public', table_name 'orders');

-- 5. 通常のSQLで操作
SELECT * FROM remote_orders WHERE order_date >= '2024-01-01';
```

### 7.3 既存スキーマの一括インポート

```sql
-- 外部スキーマ全体を一括インポート
IMPORT FOREIGN SCHEMA public
  FROM SERVER remote_db_server
  INTO local_foreign_schema;
```

> **注意**: 大規模な外部テーブルの統計情報をローカルで維持することはボトルネックになる可能性がある。`ANALYZE` の頻度を調整すること。

## 8. トリガー

### 8.1 PostgreSQLトリガーの設計思想

PostgreSQLでは **トリガーは関数を呼び出す指示書** であり、関数自体はどのテーブル・どのDML操作にも再利用可能。1つのトリガー関数を複数のテーブルで共有できる。

```
トリガー（いつ・何をきっかけに）
    ↓ 実行
トリガー関数（何をするか）← テーブル横断で再利用可能
```

### 8.2 トリガーの種類

| 種類 | 説明 | `NEW`/`OLD` |
|------|------|------------|
| **BEFORE 行レベル** | 各行の変更前に実行。`NEW`の値を変更可能 | 両方利用可能 |
| **AFTER 行レベル** | 各行の変更後に実行 | 両方利用可能 |
| **BEFORE 文レベル** | SQL文の実行前に1回実行 | 利用不可 |
| **AFTER 文レベル** | SQL文の実行後に1回実行 | 利用不可 |
| **INSTEAD OF** | ビューへのDMLの代わりに実行 | 両方利用可能 |

### 8.3 実装例

**例1: 更新日時・更新者の自動記録（BEFORE 行レベル）**
```sql
-- トリガー関数（複数テーブルで再利用可能）
CREATE OR REPLACE FUNCTION tr_set_audit_fields()
RETURNS TRIGGER
AS $$
BEGIN
  NEW.updated_at = NOW();
  NEW.updated_by = CURRENT_USER;
  RETURN NEW;  -- 変更後のNEWを返す（BEFORE行レベルトリガーでは必須）
END
$$ LANGUAGE PLPGSQL;

-- テーブルへのアタッチ
CREATE OR REPLACE TRIGGER tr_audit_orders
  BEFORE INSERT OR UPDATE
  ON public.orders
  FOR EACH ROW
  EXECUTE FUNCTION tr_set_audit_fields();
```

**例2: 変更履歴の記録（AFTER 行レベル）**
```sql
CREATE OR REPLACE FUNCTION tr_log_changes()
RETURNS TRIGGER
AS $$
BEGIN
  INSERT INTO audit_log (
    table_name,
    operation,
    old_data,
    new_data,
    changed_at,
    changed_by
  ) VALUES (
    TG_TABLE_NAME,
    TG_OP,
    TO_JSONB(OLD),
    TO_JSONB(NEW),
    NOW(),
    CURRENT_USER
  );
  RETURN NULL;  -- AFTER行レベルトリガーは戻り値を無視
END
$$ LANGUAGE PLPGSQL;

CREATE OR REPLACE TRIGGER tr_audit_log_orders
  AFTER INSERT OR UPDATE OR DELETE
  ON public.orders
  FOR EACH ROW
  EXECUTE FUNCTION tr_log_changes();
```

**例3: 条件フィルター付きトリガー**
```sql
-- 特定条件（高額注文のみ）でのみ起動
CREATE OR REPLACE TRIGGER tr_notify_large_order
  AFTER INSERT
  ON public.orders
  FOR EACH ROW
  WHEN (NEW.amount > 100000)  -- ← 条件フィルター
  EXECUTE FUNCTION tr_notify_finance_team();
```

### 8.4 トリガー変数（TG_*）

| 変数 | 内容 |
|------|------|
| `TG_NAME` | トリガー名 |
| `TG_OP` | 操作種類（`INSERT` / `UPDATE` / `DELETE` / `TRUNCATE`） |
| `TG_WHEN` | タイミング（`BEFORE` / `AFTER` / `INSTEAD OF`） |
| `TG_LEVEL` | レベル（`ROW` / `STATEMENT`） |
| `TG_TABLE_NAME` | 対象テーブル名 |
| `TG_TABLE_SCHEMA` | 対象テーブルのスキーマ名 |
| `NEW` | INSERT/UPDATE後の行データ（行レベルのみ） |
| `OLD` | UPDATE/DELETE前の行データ（行レベルのみ） |

## 9. ユーザー定義型とドメイン

### 9.1 ユーザー定義型（CREATE TYPE）

PostgreSQLは5種類のカスタム型をサポート。

#### 列挙型（Enumerated Type）

```sql
-- 映画レーティングの列挙型
CREATE TYPE mpaa_rating AS ENUM ('G', 'PG', 'PG-13', 'R', 'NC-17');

-- テーブルで使用
CREATE TABLE film (
  film_id integer PRIMARY KEY,
  title   text NOT NULL,
  rating  mpaa_rating
);

-- 列挙値を順序を考慮して比較可能
SELECT * FROM film WHERE rating <= 'PG-13';

-- 後から値を追加
ALTER TYPE mpaa_rating ADD VALUE 'TV-MA' AFTER 'NC-17';
```

#### 複合型（Composite Type）

```sql
-- 住所を表す複合型
CREATE TYPE address_type AS (
  street      text,
  city        text,
  postal_code text,
  country     text
);

-- テーブルで使用
CREATE TABLE customer (
  customer_id integer PRIMARY KEY,
  full_name   text,
  address     address_type
);

-- INSERT
INSERT INTO customer VALUES (1, 'Alice', ROW('123 Main St', 'Tokyo', '100-0001', 'JP'));

-- アクセス
SELECT (address).city FROM customer WHERE customer_id = 1;
```

#### 範囲型（Range Type）

```sql
-- カスタム数値範囲型の例
CREATE TYPE float8range AS RANGE (subtype = float8);

-- 使用例
SELECT * FROM measurements WHERE value_range && '[0.5, 1.5]'::float8range;
```

### 9.2 ドメイン（CREATE DOMAIN）

既存のデータ型に制約を追加したユーザー定義型。複数の場所で同じ制約を再利用できる。

```sql
-- メールアドレスドメイン
CREATE DOMAIN email_address AS text
  NOT NULL
  CHECK (VALUE ~ '^[a-zA-Z0-9.!#$%&*+/=?^_`{|}~-]+@[a-zA-Z0-9-]+(?:\.[a-zA-Z0-9-]+)*$');

-- 郵便番号ドメイン（日本）
CREATE DOMAIN jp_postal_code AS char(7)
  CHECK (VALUE ~ '^\d{7}$');

-- 正の数値ドメイン
CREATE DOMAIN positive_numeric AS numeric
  CHECK (VALUE > 0);

-- テーブルで使用
CREATE TABLE customer (
  customer_id integer PRIMARY KEY,
  email       email_address,  -- ← ドメインを型として使用
  postal_code jp_postal_code,
  balance     positive_numeric
);

-- ドメインの制約を後から変更
ALTER DOMAIN email_address ADD CONSTRAINT email_length CHECK (length(VALUE) <= 254);
ALTER DOMAIN email_address DROP CONSTRAINT email_length;
```

**型 vs ドメインの選択基準:**

| 観点 | カスタム型（TYPE） | ドメイン（DOMAIN） |
|-----|-----------------|-----------------|
| 用途 | 新しいデータ構造の定義 | 既存型への制約の追加 |
| 検証 | 型構造による整合性 | CHECK制約による値検証 |
| 複数箇所での再利用 | ✅ | ✅ |
| 一貫した制約の強制 | △ | ✅ |
| 典型例 | 列挙型・複合型 | メールアドレス・郵便番号・正数 |
