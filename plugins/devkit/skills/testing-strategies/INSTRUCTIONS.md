# フルスタックテスト戦略ガイド

Web / モバイルアプリケーションを対象に、品質を「全レイヤー × 全品質次元」で捉えるフルスタックテスティングの実践ガイド。テスト戦略を構成する 10 次元（各専門リファレンスへのルーティング表）、shift-left テスティングの日常プラクティス、AI テストツールの分類と導入判断、時代やツールが変わっても揺るがないテストの第一原理を、判断基準・チェックリスト・表として整理する。

- TDD / ユニットテスト方法論（AAA・Four Pillars・モックパターン）は `testing-code` スキルを使う
- Playwright E2E の実装詳細は `web:testing-e2e-with-playwright` を使う
- コードレベルのセキュリティ実装は `securing-code` を使う

---

## 1. 概要: フルスタック品質とは

### 1.1 現代のソフトウェア品質の定義

「欠陥がないこと」だけでは品質の定義として不十分。ステークホルダーごとに品質の意味が異なり、どれか一つの欠落でも全体の体験を損なう。

| 視点 | 品質を構成する要素（例） |
|------|--------------------------|
| エンドユーザー | 使いやすさ、直感的なデザイン、データプライバシー、高速なパフォーマンス、24/7 の可用性 |
| ビジネス | ROI、リアルタイム分析、ゼロダウンタイム、ベンダーロックイン回避、スケーラブルなインフラ、データセキュリティ、法令遵守 |

- 品質の失敗はビジネスの財務・ブランド評判に直結する（大規模セール時のサイトダウン、音声 AI の暴走注文、広範な IT 障害などの公知の事例が示すとおり）。
- 「time-to-market を優先して品質を後回しにする」判断は、競合に突かれる前に返済すべき負債を作る行為だと認識する。

### 1.2 フルスタックテスティングの定義

**全レイヤー（UI / サービス / データベース）ごとに、機能要件と非機能（cross-functional）要件の両方を検証し、さらにアプリケーション全体としても検証する**実践。

| 検証の粒度 | 対象 | 例 |
|-----------|------|-----|
| micro レベル | クラスの各メソッド、入力値、ログメッセージ、エラーコード | 注文金額計算を境界値（負の価格・桁あふれの小数）で検証 |
| macro レベル | フィーチャー、フィーチャー間の統合、E2E ワークフロー | 注文作成フロー、サードパーティ統合、UI フロー、失敗系 |
| 横断的品質 | セキュリティ、パフォーマンス、アクセシビリティ、ユーザビリティ | 各レイヤー・全体で個別ツールにより検証 |

**原則: 開発とテストは不可分**。小さなコード（例: 注文合計の計算）を書いた時点で「正しく計算するか」と「セキュアか」を同時に検証しないと、欠陥のある土台の上に開発を積み重ねることになり、統合不良と機能劣化が連鎖する。

## 2. テスト戦略の 10 次元とルーティング表

テストスキルを「手動 / 自動」の 2 分類で捉えるのは粗すぎる。フルスタック品質の達成には次の 10 次元が必要。各次元の詳細手順・ツール・チェックリストは対応するリファレンスを読み込むこと。

| # | 次元 | 何を検証するか / 要点 | 詳細リファレンス |
|---|------|----------------------|-----------------|
| 1 | 手動探索的テスト | ユーザーストーリーに書かれていない現実のシナリオを発想し、テスト環境で再現して挙動を観察する。テストケース設計フレームワーク 8 種（同値分割・境界値・状態遷移・デシジョンテーブル等）、機能探索の 4 観点、AI 併用探索 | [references/EXPLORATORY-TESTING.md](references/EXPLORATORY-TESTING.md) |
| 2 | 自動機能テスト | 機能要件をコードで自動検証。micro/macro のテストタイプ分類、テストピラミッド戦略、レイヤー別フレームワーク構築（Selenium / Playwright / REST Assured / JUnit / Pact）、コントラクトテスト、agentic test authoring、アンチパターン検出 | [references/FUNCTIONAL-AUTOMATION.md](references/FUNCTIONAL-AUTOMATION.md) |
| 3 | 継続的テスト | 常時リリース可能な状態を自動チェックで維持。CI/CT/CD の全体像、CI の原則とエチケット、テスト振り分け戦略、CI パイプラインへの AI エージェント統合、DORA メトリクス | [references/CONTINUOUS-TESTING.md](references/CONTINUOUS-TESTING.md) |
| 4 | データテスト | データ喪失・誤表示はユーザーの信頼を直撃する。RDB・キャッシュ・バッチ処理・イベントストリームそれぞれの特性に応じたテスト戦略、SQL / JDBC / data contract / Testcontainers / ストリームテスト | [references/DATA-TESTING.md](references/DATA-TESTING.md) |
| 5 | ビジュアルテスト | 見た目の品質はブランド価値に直結。機能テストが取りこぼす視覚属性（サイズ・色・レイアウト）の検証、スクリーンショット比較、ベースライン管理とフレーク対策 | [references/VISUAL-TESTING.md](references/VISUAL-TESTING.md) |
| 6 | セキュリティテスト | 専門ペネトレーションテスターへの丸投げをやめ、チームの日常業務に基礎的なセキュリティテストを組み込む。脅威モデリング（STRIDE）、攻撃者視点の思考、OWASP 系自動スキャナ、CI 組込み | [references/SECURITY-TESTING.md](references/SECURITY-TESTING.md) |
| 7 | パフォーマンステスト | わずかな性能低下も財務・評判の損失につながる。性能 KPI、テストタイプと負荷パターン、JMeter / Gatling の使い分け、RAIL モデルと Core Web Vitals、フロントエンド計測 | [references/PERFORMANCE-TESTING.md](references/PERFORMANCE-TESTING.md) |
| 8 | アクセシビリティテスト | 多くの国で法的義務であり、ビジネス機会でもある。WCAG の原則とレベル、支援技術エコシステム、Lighthouse / axe-core / Pa11y CI などの監査ツールの使い分け | [references/ACCESSIBILITY-TESTING.md](references/ACCESSIBILITY-TESTING.md) |
| 9 | CFR（横断的要求）テスト | 可用性・スケーラビリティ・保守性・可観測性などの cross-functional requirements。FURPS モデル、カオスエンジニアリング、アーキテクチャテスト、インフラテスト、コンプライアンステスト | [references/CFR-TESTING.md](references/CFR-TESTING.md) |
| 10 | モバイルテスト | デバイス断片化・アプリ種別・ネットワーク変動というモバイル固有の 3 変数。デバイス選定、テスト次元別ツール選択、モバイルテストピラミッド | [references/MOBILE-TESTING.md](references/MOBILE-TESTING.md) |

**AI は 11 番目の次元ではない**。プログラミング言語の習得と同じく、10 次元すべての土台となる基盤スキルとして扱う（§4 参照）。

### ルーティングの判断基準

| 状況 | 読むべきリファレンス |
|------|---------------------|
| テスト戦略全体を設計する / どのテスト種別を採用するか決める | 本ファイル §2〜§5 を通読し、該当次元へ |
| ユニットテストしかないプロジェクトのテスト計画を拡充する | FUNCTIONAL-AUTOMATION → CONTINUOUS-TESTING の順 |
| 「バグは無いのに見た目が崩れる」「リリース後にレイアウトが壊れた」 | VISUAL-TESTING |
| DB・キャッシュ・イベント基盤が絡む欠陥、データ不整合の再発防止 | DATA-TESTING |
| 性能目標・負荷テスト計画・Core Web Vitals 改善 | PERFORMANCE-TESTING |
| セキュリティを開発チーム内でテストしたい、脅威モデリングをしたい | SECURITY-TESTING |
| WCAG 適合・アクセシビリティ監査 | ACCESSIBILITY-TESTING |
| 可用性・スケーラビリティ・カオス実験・アーキテクチャ適合性 | CFR-TESTING |
| モバイルアプリ（ネイティブ / ハイブリッド / PWA）のテスト計画 | MOBILE-TESTING |
| 受入基準の消化で終わらない手動テストを設計する | EXPLORATORY-TESTING |

## 3. Shift-Left テスティングの要点

### 3.1 定義と狙い

伝統的な SDLC（要件分析 → 設計 → コーディング → テスト → リリース）ではテストが末尾に置かれる。**shift-left テスティングはテスト活動をライフサイクルの先頭側（計画段階）へ移動させ、開発と並行してデリバリーサイクル全体で継続する**考え方。

- 建築に例えると「全棟完成後に初めて検査する」のではなく「壁を 1 枚建てるごとに検査し、その場で直す」方式。
- micro / macro 両レベルの自動化と、変更の小さな塊（コミット）ごとに CI サーバー上でテストを回すことに強く依存する。手動テストを毎回繰り返すよりはるかに低コストで多面的な品質検証を継続できる。

### 3.2 開発「前」の品質チェック（要件の多段検証）

| プラクティス | 内容 | 効果 |
|-------------|------|------|
| Three amigos プロセス | 分析フェーズでビジネス代表・開発者・テスターが集まりフィーチャーを多角的に検討。AI を「4 人目」として加え追加のエッジケースを出させることも可能 | 統合ポイント・エッジケース・業務要件の見落とし防止 |
| デザイン検証 | ビジネス代表と UX デザイナーがアプリケーションデザインを検証・改善 | UI/UX 起因の手戻り防止 |
| イテレーション計画ミーティング（IPM） | イテレーション冒頭にユーザーストーリーを全員で詳細検討 | 要件の集団的な再検証 |
| ストーリーキックオフ | 開発着手直前に当該ストーリーの要件・エッジケースを深掘り（three amigos のミニ版）。テスターがテストケースをここで提示・議論する | 着手時点で要件検証が完了している状態を作る |

### 3.3 開発「中」の品質チェック（多段フィードバック）

| プラクティス | フィードバック段階 | 内容 |
|-------------|------------------|------|
| ユニットテスト作成 | 継続 | 開発者が（LLM 支援も含め）ストーリーごとに作成。リンター・静的解析を CI に統合 |
| 機能テスト作成 | 継続 | UI 駆動の機能テストをストーリー開発の一部として（または直後に）作成し CI に統合 |
| ローカル自動テスト | 第 1 段階 | コミット前に開発者マシンで自動テスト一式を実行 |
| CI 自動テスト | 第 2 段階 | コミットごとに CI でユニット / サービス / UI テストのスイートを実行 |
| dev-box テスト | 第 3 段階 | テスター・ビジネス代表が開発者マシン上で新機能を手動探索的に即時検証 |

- shift-left 後は、ストーリーが「テストフェーズ」に到達する前にフィードバックの約半分が得られる。テスターは期待動作の確認作業から解放され、多様な品質側面の探索に集中できる。
- shift-left は機能テストに限らない。セキュリティ（例: pre-commit シークレットスキャン）やパフォーマンスのチェックも左へ寄せられる（詳細は [references/SECURITY-TESTING.md](references/SECURITY-TESTING.md)・[references/PERFORMANCE-TESTING.md](references/PERFORMANCE-TESTING.md)）。
- 標語は「**品質はチームの責任**」。各フェーズの品質チェックをテスター / QA 以外の役割も所有する。全員が相応のテストスキルを身につけることが前提になる。XP（エクストリームプログラミング）系の反復開発フレームワークは、この shift-left を組み込んだ代表例。

## 4. AI 活用の要点

### 4.1 AI サブ分野の整理（テスト文脈での要点）

| サブ分野 | 概要 | テストでの応用例 | 制約 |
|---------|------|-----------------|------|
| 機械学習（ML） | データからパターンを学習する統計モデル | テスト実行ログの分類（欠陥かインフラ問題か） | 訓練データ品質に依存、100% の正解率は出ない |
| 深層学習（DL） | ニューラルネットワークで非構造化データを扱う | 商用モデルによるビジュアルテスト | 巨大データ・計算資源が必要、判断根拠がブラックボックス |
| 生成 AI（GenAI） | 訓練データに基づき新しいコンテンツを生成 | テキスト・コード等の生成全般 | 非決定性・不透明性・著作権リスクを継承 |
| LLM | transformer ベースで次語予測に特化した生成 AI | テストケース生成、テストコード生成、要約 | 「推論しているように見える」出力は近似的検索であるとの研究指摘あり。制御手段はプロンプト / コンテキストエンジニアリングのみ |
| エージェント型 AI | LLM に自律的な計画・実行・修正のループと外部システムアクセスを付与 | チケットのストーリーからテストケース作成〜自動化コード実装までを一気通貫 | 計画の逸脱・過剰な自律性に人間の監督が必要 |

- プロンプトエンジニアリング = 目的・文脈・手順を構造化した指示の設計。コンテキストエンジニアリング = コンテキストウィンドウの制約内で必要情報を最適供給する技術（RAG、MCP、長期メモリ、ナレッジグラフ）。**テスト用途では MCP の利用頻度が最も高い**（AI に実アプリを探索させる MCP ベースの実践は [references/EXPLORATORY-TESTING.md](references/EXPLORATORY-TESTING.md) 参照）。

### 4.2 自律度による 4 分類

人間の介入量を軸にした分類。ツール選定時は「どこまで任せ、どこで人間がレビューするか」をこの軸で決める。

| 分類 | 自律度 | 特徴 | 例 |
|------|--------|------|-----|
| AI-assisted | 低 | 明確な指示が必須。指示の読み飛ばし・誤解あり。出力は必ずレビューが要る | 汎用 LLM にストーリーを渡してテストケース一覧を生成（ドメイン / アーキテクチャ固有のエッジケースは不足しがち） |
| AI-powered | 中 | 特定タスクを少ない介入で遂行 | ML/DL でデータセットの統計特性を学習し、類似分布の合成テストデータを生成 |
| AI-driven | 高 | 自律的な意思決定を伴うワークフロー強化 | セキュリティスキャンを自律実行し脆弱性を優先度付け、修正 PR まで提案 |
| AI agents | 最高 | タスクを分解・計画し、実行・再計画まで自律遂行 | 「テストフレームワークを構築して」でプロジェクト作成〜依存導入〜テストクラス生成まで一括実行 |

### 4.3 提供形態による分類

| 形態 | 役割 | 特記事項 |
|------|------|---------|
| AI チャットボット | タスク支援。MCP サーバーや RAG で文脈を注入可能 | 汎用だが文脈供給の設計が成否を分ける |
| AI コーディングアシスタント | ユニット / 統合 / UI テストの生成。コーディング規約の適用が容易 | 「agent mode」で自律実行に拡張されるものが多い |
| AI 搭載テスティングプラットフォーム | ビジュアル / セキュリティ等の特定タスク用モデル + 実行基盤・根本原因分析・自動修復・レポート・ガバナンス | インフラ込みのマネージドな体験 |
| AI 拡張機能 | 既存ツールに LLM を接続しデバッグ支援・修正提案を追加 | 既存ワークフローへの追加が容易 |

### 4.4 AI 導入の判断基準（落とし穴）

- **AI 導入はツール問題ではなくシステム問題**（業界調査でも一致した知見）。shift-left と健全なエンジニアリング習慣が根付いたチームでは AI がフィードバックループをさらに加速するが、プラクティスが緩いチームでは局所的な効果が下流で失われる。
- AI サブ分野はいずれも「非決定的な出力」「不透明性」「著作権リスク」を継承する。テスト結果の判定に使う場合は特に人間のレビューを外さない。
- **AI ツールに任せきりにすると micro / macro 両レベルでテストケースが抜ける**。生成されたテストは優先度・重要度と突き合わせてレビューする（「カバレッジは高いが価値が低い」状態の回避）。
- 全否定も盲信もしない。かつての「自動テストは ROI に見合うか」論争が手動 / 自動の適切なバランスに収束したように、AI も機能・非機能テストを前進させる道具として位置づける。

### 4.5 AI 特化ソフトスキル

| スキル | 実践内容 |
|--------|---------|
| 抽象化思考 | 問題のメンタルモデルを段階的な指示に落とし込めること。例: 探索的テストセッションの進め方をチャーターとして定義し AI に走らせる |
| タスクの分割 | AI は小さな実行可能タスク単位の指示で最も精度が出る。確立された設計パターン（例: Page Object Model）でも、タスク列に明示分解しないと AI は独自解釈で実装する（page クラス内に assertion を置く、テストクラスにロケーターをハードコードする等） |
| 効果的なライティング | AI との対話は主にテキスト。文法の乱れ・誤字・文脈欠落は成果を直撃する。句読点・デリミタ・適切な空白で構造化した指示を書く |
| 高速な学習と unlearning | AI の進化は既存の仕事のやり方の「棄却」も要求する。古い方法を早く手放すほど変化の波を上手く乗りこなせる |

## 5. テストの第一原理

ツールや技術スタックが変わっても揺るがない 7 原理。新技術（AI ツール含む）を採用する際のガードレールとして機能する。新しいツール・手法が登場して圧倒されそうなときは、一歩引いて「それは第一原理のどれに資するのか」で位置づける。

### 5.1 欠陥予防 > 欠陥検出

欠陥は発見よりも予防が安い。後から直す修繕は周囲と馴染まず、結局大規模な作り直しになる（アーキテクチャ変更級の手戻り）。

予防を実現するプラクティス:

- IPM（スプリント計画）でユーザーストーリーの統合漏れ・エッジケースをブレインストーミング
- Three amigos プロセスで 3 役割の視点を分析フェーズに集約（AI を加えて「four amigos」化も可）
- ストーリーキックオフで開発直前に再検証、テストケースをその場で議論
- ADR（アーキテクチャ決定記録）とテスト戦略の文書化・議論
- TDD により小さなコード単位でもエッジケースを先に考える
- ペアプログラミングで欠陥コード・考慮漏れを未然に防ぐ
- リンター等の静的解析でコードスメルを書いた瞬間に検出
- AI による脅威モデリング等、予防フェーズへの AI 活用

これらはデータプロダクトや AI プロダクトなど、役割構成が異なる新ドメインにも適用できる。

### 5.2 共感テスト（Empathetic Testing）

- テストとはエンドユーザーの靴を履くこと。受入基準の消化で終わらせず、典型的なエンドユーザーとして振る舞ってアプリを探索する。
- 前提として対象ペルソナの理解が必要。デザイン・プロトタイピング段階から UX デザイナーや PM とペアで関与する。
- チームは開発複雑度や納期を理由にユーザーニーズをトレードオフしがち。テスト時にはユーザー視点を最前面に置き、そのトレードオフを再交渉する。

### 5.3 micro / macro レベルテストのバランス

| レベル | 含まれるテスト | 見落としたときの症状 |
|--------|---------------|---------------------|
| micro | ユニットテスト、統合テスト、コントラクトテスト | 境界値・異常値（負の価格、想定外の小数桁）で本番障害 |
| macro | API テスト、UI 機能テスト、ビジュアルテスト | フロー・統合・データ伝播の欠陥 |

- 配分の偏りはフィードバック遅延と品質低下を招く。macro 偏重チームは詳細を見落とし、本番まで問題が発覚しない。
- 常にズームイン / ズームアウトを繰り返し、全体像と詳細の両方を見る。

### 5.4 高速フィードバック

- 欠陥の発見が遅いほど修正コストが増大する（複数の調査研究で相関が示されている）。開発中はコードの文脈が頭にあり root cause の特定が速いが、時間が経つと文脈が失われ、デバッグが長期化する。
- 発見が遅れると欠陥追跡サイクル（バグ票作成 → トリアージ → イテレーション管理 → 担当者確保）自体が日〜週単位の遅延を生む。最悪の場合、その間の開発によりリファクタリングなしでは直せなくなる。
- 実践: shift-left、dev-box テスト、テストピラミッド、スプリントごとの PO サインオフとステークホルダーへのショーケース。AI はこのフィードバックサイクルをさらに加速する。

### 5.5 継続的フィードバック

- 一度テストして放置しない。新機能開発とリファクタリングが続く限り、既存機能と統合が壊れていないかのリグレッションフィードバックを維持する。
- 問題が小さいうちに捕捉でき、継続的デリバリーの前提条件が整う。
- 実現手段は継続的テスト: micro / macro の機能テストと CFR テストをコミットごとに CI で実行（[references/CONTINUOUS-TESTING.md](references/CONTINUOUS-TESTING.md)）。AI ツールは「human in the loop」を保ったまま加速に使う。

### 5.6 品質メトリクスの計測

計測されるものは改善される。ただしメトリクスに固執するとチームは数字を欺く方向に最適化するため、共通の品質ゴールへ向かわせる道具として賢く使う。

| メトリクス | 何が分かるか / 注意点 |
|-----------|----------------------|
| 全レイヤーの自動テストが捕捉した欠陥数 | 自動テストという安全網の強度。多くを早期に捕捉できていれば変更への自信につながる |
| コミットからデプロイまでの時間 | 不安定なテストや環境問題で green build が遅いとフィードバック遅延・生産性損失 |
| テスト環境への自動デプロイ回数 | 少なければ（インフラ障害・テスト失敗等で）フィードバックサイクルに改善余地 |
| ストーリーテスト中に見つかるリグレッション欠陥 | 業務ケースまたは自動テストの欠落のシグナル。自動テストのアンチパターンを疑い、root cause を振り返る |
| 重要度・優先度別の自動化カバレッジ | バックログゼロを目標に記録。AI で自動テストは量産できるため、優先度・重要度と突き合わせないと「カバレッジは高いが価値が低い」状態に陥る |
| 本番欠陥数と重大度 | 見落とした業務ケース・設定・データ不整合の全体像。root cause を特定してテストを自動化し、テスト戦略を生きた文書として進化させる |
| エンドユーザーのユーザビリティスコア | 開発中から UX 関連指標（情報到達までのクリック数等）の改善に接続 |
| インフラ起因の失敗数 | テスト環境の断続ダウン、CI の問題、環境間の設定差異。インフラコードの技術的負債返済の判断材料 |
| 横断的（CFR）メトリクス | 性能 KPI の定点観測、スプリントごとのセキュリティスキャン結果、クロスブラウザ / カオス / ローカライズのカバレッジ |

**DORA メトリクスとの対応**: lead time（コミット → 本番デプロイ可能まで。エリートは 1 日未満）・deployment frequency（エリートは on-demand）・change fail percentage（高パフォーマンスで 0〜15%）に上記メトリクスが接続する。チームで一貫して追跡・議論すれば、高品質は自然と共通目標になる（詳細は [references/CONTINUOUS-TESTING.md](references/CONTINUOUS-TESTING.md)）。

### 5.7 コミュニケーションとコラボレーション

- テストはサイロでは価値を生まない。業務要件・ドメイン知識・技術実装・環境詳細の共有が前提。
- 同期の場: スタンドアップ、ストーリーキックオフ、IPM、dev-box テスト。
- 非同期の担保: ストーリーカード、ADR、テスト戦略、テストカバレッジレポート、動画・メール等（タイムゾーンをまたぐ分散チームの引き継ぎに必須）。

## 6. quality-first マインドセットの要点

品質は一人では「所有」できず、誰も「所有を放棄」できない（リレー競技で一人でも遅ければ勝てないのと同じ）。

| スキル | 実践内容 |
|--------|---------|
| アウトカム駆動 | 各役割が品質アウトカムを所有する（UX = 直感的なユーザージャーニー、PO = 顧客本位のプロダクト、開発者 = 堅牢な設計とコード）。テスターは欠陥予防プラクティスの導入・継続的テストの徹底・自動化カバレッジの追跡と完了を所有する |
| コラボレーション | 「品質はチームの責任」は協働でしか根付かない。テスト戦略は開発者と共同で実装し、欠測テストケースはビジネス代表と共同で発見する |
| 効果的なコミュニケーション | 適切な媒体とタイミングを選ぶ。求める品質水準に必要な事項をチームへ定期的・明確に伝える |
| タスク優先順位付け | 開発視点では小さく見えるタスクが不釣り合いなテスト工数を要することがある。ストーリーごとのテスト活動を事前に優先順位付けし、イテレーションのキャパシティを確保する |
| ステークホルダー期待管理 | 「カバレッジ 100%」や「納期優先」といった非現実的な期待を、協働・伝達・優先順位付けで事前に調整する |
| コーチング / メンタリング | 新メンバーとのペア作業と堅牢なドキュメントでオンボーディングを加速する |
| 影響力の構築 | テスト戦略への支持や新ツール投資の獲得に必須。高品質なアウトカムの継続的な提示 + 他のソフトスキルの掛け合わせで築く |

- テストは継続的な学習の旅である。10 次元は今後さらに増えうる（AI システムの evals、ブロックチェーン、AR/VR など）が、第一原理は変わらない。
- ソフトスキルの習得は技術スキルより難しいことが多く、意識的な練習が要る。テスト技術スキル × ソフトスキルの掛け合わせが高品質ソフトウェアの効率的なデリバリーを実現する。
