# 探索的テスト（Exploratory Testing）

探索的テストは、デプロイ済みアプリケーションを能動的に操作しながら「学習・テスト設計・テスト実行」を同時に行う活動である。要件文書やユーザーストーリーに書かれていない振る舞いまで踏み込み、分析時に想定されなかったユーザーフローや既存フローのバグを発見する。所定の受入基準を順に実行するだけの従来型手動テストと異なり、テスターの分析力・観察力に依存する。本リファレンスでは、テストケース設計フレームワーク8種、機能探索の4観点、戦略（アプリ理解→部分探索→フェーズ反復）、AI併用の実践（テストケース生成・API探索・Web UI探索）、合成テストデータ生成、テスト環境衛生をまとめる。

---

## 位置づけと基本用語

### なぜシフトレフトだけでは不十分か

- ユーザーストーリー分析は主にビジネス視点、開発は現機能の技術スコープに閉じがち
- 「デプロイ済み環境で、エンドユーザー視点・全体像レンズで探索する」工程が欠落すると、統合問題や見落とされたエンドユーザーフローが残る
- 探索的テストはビジネス要件・技術実装・エンドユーザーニーズの3視点を統合し、「正しいとされていること」すべてを疑う
- ベストプラクティス: 探索で発見した新しいユーザーフロー・テストケースが自動化されて初めて機能を「完了」と呼ぶ
- 実施者は専任でも、イテレーションごとの持ち回りでもよい。全ロールが習得する価値のあるスキル

### 用語

| 用語 | 意味 | 例 |
|------|------|-----|
| 機能（feature） | アプリがエンドユーザーに価値を提供する単位 | ログイン（セキュリティを提供） |
| ユーザーフロー | 機能の価値を得るためにユーザーが行う一連の操作 | 認証情報入力→サインイン |
| テストケース | 機能が期待どおり動くことを検証する一連の操作 | 有効な認証情報でログイン成功を確認 |
| 正常系（positive） | ユーザーが価値を得られるケース | 有効入力でログイン成功 |
| 異常系（negative） | エラーが正しく処理されるケース | 無効なユーザー名でエラー表示 |
| エッジケース | 発生頻度が極めて低い異常系 | 境界値ちょうどの入力、競合状態 |

- 探索には正常系・異常系の両方のシミュレーションと観察が必須
- **モンキーテストとの違い**: 探索的テストは対象機能の正確な理解を前提に「未知」を探索する。機能を知らずランダム入力するモンキーテストとは別物

## テストケース設計フレームワーク8種

フレームワークの目的は、直感的に適用できるメンタルモデルを形成し、無限の入力空間を論理的なサンプル集合に区切ってテスト範囲を絞ることにある。入力データに限らず、アプリ内のあらゆる文脈（状態・時刻・権限など）に適用できる。

| # | フレームワーク | 使いどころ |
|---|--------------|-----------|
| 1 | 同値クラス分割 | 連続値・カテゴリ入力を代表値に圧縮したい |
| 2 | 境界値分析 | クラス境界の定義が曖昧・実装ミスが出やすい箇所 |
| 3 | 状態遷移 | 入力の履歴で振る舞いが変わる（ロック、注文状態） |
| 4 | デシジョンテーブル | 入力が AND/OR で論理結合されて結果が決まる |
| 5 | 原因結果グラフ | 論理結合された機能の全体像を分析段階で可視化 |
| 6 | ペアワイズ | 独立変数が複数あり組合せ爆発する |
| 7 | サンプリング | 巨大データセット（移行検証など）でクラス化不能 |
| 8 | エラー推測 | 経験と技術理解から起きそうな欠陥を先回り |

### 1. 同値クラス分割（Equivalence Class Partitioning）

同じ出力・同じ処理経路になる入力をクラスに分割し、**各クラスから1つのサンプルを選べば十分**とする。

例: 所得を入力し税額を返す累進課税計算（〜5,000 は 5%、5,001〜15,000 は 10%、15,000 超は 30%）

- 正常系クラス: [0–5000]・[5001–15000]・[>15000] → 代表値 2,000 / 10,000 / 20,000 の3点でよい
- 異常系クラス: [負数]・[英字]・[記号] など → 各クラスから1値ずつ

単体テストのケース設計にもそのまま使える。時刻ベースの結果（イベント前/後）やシステム内部状態の分割にも応用可能。

### 2. 境界値分析（Boundary Value Analysis）

同値クラス分割を拡張し、**各クラスの境界値を明示的に検証**する。要件が「5,000 未満は5%、5,000〜15,000 は10%」のように書かれると、ちょうど 5,000 や 15,000 がどちらのクラスに属すのかが未定義になりやすく、境界は実装ミスの温床になる。

- 上記の例では「所得 0 は課税なし」という業務ロジックから [0] と [1–5000] に再分割が必要と気づける
- テストすべき境界値: `0, 1, 5000, 5001, 15000, 15001`（＋各クラス内部の代表値）
- 分析フェーズから適用すると効果最大: BA が境界を受入基準に明記 → 開発者が単体テストで正常/異常を自動化 → テスターが探索、と全工程で使う。後から欠陥として直すコストは大きい

### 3. 状態遷移（State Transition）

入力の**履歴**によって振る舞いが変わる機能に使う。各状態をノード、状態を変えるアクション（イベント）を矢印にした**遷移木**を描いてテストケースを導く。

- 例: ログインで誤パスワード1〜2回はエラー表示、3回目でアカウントロック
- 各テストケースは「開始状態 → アクション → 期待される遷移先」として明確化される
- ノード数・遷移数からテスト工数の現実的な見積りも得られる
- 注文管理（支払完了→保留→出荷→キャンセル→完了…）のような複雑な状態機械の全体把握にも有効

### 4. デシジョンテーブル（Decision Table）

入力が `AND`/`OR` で論理結合されて結果が決まる場合、全入力組合せと期待出力を事前に表へ列挙する。

例: メールとパスワードが両方正しいときだけログイン成功（AND 結合）

| 条件/結果 | TC1 | TC2 | TC3 | TC4 |
|-----------|-----|-----|-----|-----|
| メール正しい | ○ | × | × | ○ |
| パスワード正しい | × | ○ | × | ○ |
| → ログイン成功 | − | − | − | ○ |
| → エラー表示 | ○ | ○ | ○ | − |

- 冗長ケースの削除にも使える: AND 結合なら片方が誤りの時点で失敗が確定するため、「両方誤り」（TC3）は削除できる

### 5. 原因結果グラフ（Cause-Effect Graphing）

論理結合された入力と結果を、左に原因・右に結果を置き論理演算子で結んで可視化する。機能の全体像を掴むのに向き、**分析フェーズで特に有用**。作成後はデシジョンテーブルへ変換して詳細なテストケースに落とす。

### 6. ペアワイズ（Pairwise / All-Pairs）

複数の**独立**変数が結果を左右するとき、「任意の2値ペアが少なくとも1回登場すればよい」として組合せを圧縮する。

例: OS 2値 × デバイス 3値 × 解像度 3値 = 全組合せ 18 ケース → ペアワイズ適用で 9 ケースに半減（例: [Windows, タブレット] や [Windows, 中解像度] のようなペアの重複出現が排除される）。

- 前提は変数の独立性。依存関係がある変数にはデシジョンテーブル等を使う
- 変数が多い場合はペアワイズ生成ツールの利用が現実的

### 7. サンプリング（Sampling）

数百万件規模のデータ移行検証のように、同値クラスもペアも定義しきれない巨大・連続データに使う。

| 方式 | やり方 | 例 |
|------|--------|-----|
| ランダムサンプリング | 無作為に一部を抽出して新旧突合 | 1,000 ユーザーから 50〜100 を無作為抽出し移行前後を比較 |
| 基準サンプリング | データ共通特性で層別抽出 | 年齢・契約年数・支払方法・職業、契約側は支払間隔・価格で層別 |

- 発展形: 各基準のサンプル数を実データの分布に比例させ、代表性のあるミニデータセットを構成する

### 8. エラー推測（Error Guessing）

過去の経験・技術理解・論理的推論から起こりそうな欠陥を予測する。固定手順ではなく分析的思考の訓練であり、これを鍛えると探索力全体が上がる。頻出パターン:

- [ ] 無効・空入力に対するバリデーション欠落、修正を促すエラーメッセージの欠如
- [ ] データ検証エラー・技術エラー・業務エラーで HTTP ステータスコードの使い分けが不明瞭
- [ ] ドメイン・データ型・状態に固有の境界条件の未処理
- [ ] サーバーダウンやタイムアウトなど技術エラーが UI 側で未ハンドリング
- [ ] 画面遷移・データ更新・ナビゲーション時の UI のガタつき・描画残り
- [ ] SQL の `LIKE` と `=` の取り違え（結果が全く変わる）
- [ ] キャッシュ未クリア、セッションタイムアウト未定義
- [ ] ブラウザ「戻る」でのリクエスト再送（二重 POST）
- [ ] 異なる OS からのファイルアップロードでフォーマット検証欠落

## 機能探索の4つの観点（Discovery Paths）

どの機能でも、次の4経路を探索すると偏りのないカバレッジになる。探索中に別経路のアイデアが浮かんだらメモして後で回収する。

### 1. 機能的ユーザーフロー

| サブパス | 内容 | 落とし穴 |
|---------|------|---------|
| 単一ユーザー正常フロー（ハッピーパス） | ログイン→検索→カート→住所→配送→支払→確認のような一連の旅程。最初に検証する | 住所・支払方法・配送・商品組合せのバリエーションまで通すこと |
| 反復フロー | 同じフロー（の一部）を繰り返す操作 | 「1回通れば同じ」という仮定は誤り。同一商品を再度カートに入れると数量確認が出る等、初回と挙動が変わりうる |
| 複数ユーザーフロー | 複数ユーザーの同時操作による衝突 | 残り1個の商品を2人が同時にカートへ入れたら？ 単一ユーザーでは完璧でも同時実行で壊れる |

- フロー中で同値クラス分割・境界値分析（税額計算の検証など）や遷移木（住所×配送方法の組合せ）を適宜併用する
- デザイナーが行う UX リサーチ・プロトタイプ検証にペアで同席すると、エンドユーザー行動の理解が深まり探索が豊かになる

### 2. 失敗とエラー処理

探索的テストの核心は「環境をいじる」ことと「実運用シナリオ」の2点。失敗は不可避なので、テスト環境で意図的に再現して観察する。

- コンポーネント間のネットワーク障害でレスポンスが返らない
- ユーザー〜サーバー間の低速ネットワークによる遅延
- ハードウェア障害によるサービスダウン
- 不正なユーザー操作によるエラー

各入力点（ログイン、検索テキスト、住所、支払情報、カート操作…）にバリデーションが組み込まれ、**ユーザーの誤りを指摘し修正方法を示す意味のあるエラーメッセージ**が出ることを確認して初めて機能は完成といえる。

### 3. UI の look & feel

UI はユーザーが直接目にする部分であり、品質問題が最も目立つ。

- 画像が適切な品質で表示されるか
- 表示領域は適切か（長い住所が収まる／短い住所で不自然な余白が出ない）
- ユーザーの主要ブラウザでシームレスに操作できるか
- 遅延時にローディング表示が出るか

### 4. 横断特性（Cross-Functional Aspects）

| 観点 | 探索例 |
|------|--------|
| セキュリティ | UI 入力欄への SQL 注入の試行に対するバリデーション。カード情報が DB・ログに平文保存されていないか |
| プライバシー | 同意なしに個人データを保存していないか。分析利用・第三者送信の事前告知があるか。法規制への準拠 |
| 認証 | SSO・二要素認証・セッション失効・アカウントのロック/解除。未ログインで閲覧可・注文不可の境界 |
| 認可 | ロール（admin/customer 等）と権限の組合せ、複数ロールの重複、権限追加、権限なしでの操作実行時の挙動 |
| その他 | 性能・信頼性・アクセシビリティ・監査可能性など |

## 探索的テスト戦略

「アプリを理解する → 部分に分けて探索する → フェーズを跨いで反復する」の3層構造で回す。

### ステップ1: アプリケーションを理解する（5領域）

| 領域 | 把握すべきこと | 探索への効き方 |
|------|--------------|---------------|
| ユーザーペルソナ | 想定ユーザー群の属性・期待 | 若年層は豊かな体験、シニアは明快な操作を求める等、ペルソナごとの知覚・操作を追体験する |
| ドメイン | 業務ワークフロー・用語 | 例: EC の注文は capture→promise→confirm と流れ、倉庫・配送・仕入先と連携する。ドメイン知識なしに経路は見えない |
| ビジネス優先度 | 何を最重視しているか | 「拡張性あるプラットフォーム化」が優先なら、UI とサービスの疎結合性など「プラットフォーム視点」の探索が要る |
| アーキテクチャ | サービス構成・非同期通信・外部連携 | Web サービスがあれば API 探索、イベントストリームがあれば非同期ケースの探索が必要になる |
| インフラ・設定 | 何がどこにデプロイされ、どんな設定レバーがあるか | レートリミット超過時の挙動、API ゲートウェイ設定、単一/複数マシン構成などが新たなテストケースの源になる |

最初から全部は不要。機能視点から始め、徐々にこれらの問いを増やしていけばよい。

### ステップ2: 部分に分けて探索する

- 人間はスコープが小さいほど深く集中できる。探索単位＝ discovery path、その中のユーザーフロー・機能、あるいはセキュリティ等の横断特性
- 経路とサブ経路の管理には**マインドマップ**が有効（入力バリデーション・UI 挙動・性能・セキュリティ・技術エラー処理・監査性…と枝分かれさせる）。紙でもツール（Coggle, Xmind 等）でも、個人でもチーム共同でもよい
- この段階でフレームワーク8種とツール群を投入する

### ステップ3: フェーズを跨いで反復する

探索的テストは一回きりの活動ではない。コード・機能・統合が増えるたびに探索対象が生まれる。

| フェーズ | スコープ |
|---------|---------|
| dev-box testing | 開発直後のストーリーを開発者マシン上で、ビジネス代表＋テスターが時間を区切って探索。正常フロー・バリデーション・UI look & feel に限定 |
| ストーリーテスト | 開発完了後。クロスブラウザ・横断特性まで拡大 |
| バグバッシュ | チーム全員でこれまでの機能を集中的に探索 |
| リリーステスト | 性能・信頼性・スケーラビリティ等の横断特性を深掘り、正常フローと統合はやや高い視座で確認 |

計画外の経路を発見して時間を使い切るのは正常なこと。次のストーリーテストやバグバッシュへ回すメモを残して先へ進む。

## AI 併用の実践

AI ツールの進化は速いため、個別機能ではなく「探索的テストにおける AI 活用の型」を掴むこと。

### 前提となる注意事項

- **学習利用リスク**: 会話・コンテキスト・コードを学習に使わないと約束されるのは多くの場合エンタープライズ版のみ。一般向け版に機密情報を貼らない（貼る前に機微情報を削る）
- **ハルシネーション**: LLM 生成のテストケースは重複・表層的・架空のドメインロジックを含みうる。生成結果は「スターターキット」であり、素早く検証して浮いた時間でより深いケース（キャッシュ、システム統合、データ問題など）を人間が考える
- **指示は保証でない**: LLM は非決定的で、与えた指示を毎回すべて守るとは限らない。結果は毎回変わる前提で運用する

### テストケース生成ワークフロー

観測されたスケーリング特性:

1. **広いスコープ（ページ全体など）はダメ**: 大量生成されるが重複・表層的・創作ロジック混じり
2. **狭いスコープ（検索機能だけ等）で質が向上**: 入力バリデーション・サジェスト・履歴に加え、性能・クロスブラウザ・国際化など横断ケースも出る
3. **プロジェクト文脈＋探索アプローチ文書を与えると質が桁違いに向上**: 機能→失敗/エラー処理→UI→横断特性の順に、プロジェクトの言葉で生成されるようになる

実装パターン（Claude Desktop ＋ MCP の例）:

- MCP（Model Context Protocol）は、DB・ファイル・プロジェクト文書などの外部コンテキストを LLM に供給する標準プロトコル。ホスト（Claude Desktop）・サーバー（Jira/Confluence 側）・クライアント（両者を1対1で接続）で構成される
- Atlassian 系 MCP サーバー（Docker イメージで配布されているものがある）を `claude_desktop_config.json` の `mcpServers` に登録し、Jira/Confluence の URL・ユーザー名・API トークンを環境変数で渡す
- ワークフロー:
  1. 探索的テストのアプローチ（本リファレンスの「フレームワーク」「4観点」相当）を Confluence ページ化しておく
  2. LLM 側に専用プロジェクトを作り、手順書型のシステム指示を登録する
  3. 「ストーリー ID を探索して」と促すだけで、ストーリー取得→エピック取得→ガイドライン取得→テストケース生成が走る

手順書型指示の骨子（一般化）:

```text
あなたはプロのソフトウェアテスターである。ストーリー探索を依頼されたら:
Step 1: MCP ツールでユーザーストーリーの受入基準を取得する
Step 2: MCP ツールで紐づくエピック（横断要件が書かれている）を取得する
Step 3: MCP ツールで探索的テストガイドライン文書を取得する
Step 4: 1-3 の学習を適用し Given/When/Then 形式で機能テストケースを生成する
        （ただし学習内容のみに限定しない）
Step 5: エピック記載の横断要件カテゴリごとにテストケースを生成する
Step 6: 「機能要件」と横断要件カテゴリ別にグルーピングして提示する
```

- コツ: 良いプロンプトを書くには、まず**期待するアウトカムの姿**を明確にし、そこへ至るメンタルモデルをステップ指示として与える
- エピックに横断要件（セキュリティ・性能・信頼性・クロスブラウザ・アクセシビリティ等）を明記しておくと、探索で忘れられがちな横断特性のケースが必ず生成される
- 生成されたケースは境界値分析やエラー推測といったフレームワークを反映したものになる。レビュー・追記し、ストーリーへの記録まで LLM に指示できる
- プロジェクト設定をチームで共有すれば、探索的テストの最低基準を揃える仕組みになる

### 有償 AI テストツールの評価基準

- [ ] 既存ツールエコシステム（課題管理・テスト管理・CI）との互換性
- [ ] チームの切実な問題を解くか。ドメイン・セキュリティ・アクセシビリティ等の専門知識でファインチューニングされていない限り、LLM サブスクの上に重ねて払う価値は薄い
- [ ] シフトレフトのフィードバックサイクルに組み込めるか（オールインワン志向のツールはここが欠けがち）
- [ ] セキュリティ・プライバシー統制
- [ ] プロジェクト規模での費用見積りの明確さ

### API 探索

API はもはや単なる内部インターフェースではなく**それ自体が製品**である。ビジネスロジックとバリデーションを内包し、モバイルアプリ・サポートポータル・新規事業などから再利用されるため、UI 経由のテストとは別に単体で探索する価値がある。

コアなビジネスロジック以外に見るべき経路:

| 経路 | 確認内容 |
|------|---------|
| リクエスト契約バリデーション | 不正なデータ形式のリクエストをサービスが拒否するか |
| 認証 | トークン（リクエストヘッダーの暗号化文字列）等の認証機構。無効トークンで失敗するか |
| 権限 | ロールごとの操作制限（編集可/閲覧のみ等）が API 層で守られるか |
| 後方互換性 | 契約変更時に旧バージョン API が併存する場合、両バージョンでテストしたか |
| HTTP ステータスコード | 技術的失敗・業務的失敗に適切なコードを返すか |

主要ステータスコード:

| コード | 意味 |
|--------|------|
| 200 OK | GET/PUT/POST 成功 |
| 201 Created | 新規オブジェクト作成成功 |
| 400 Bad Request | リクエスト不正（malformed） |
| 401 Unauthorized | 未認証。認証情報を付けて再要求すべき |
| 403 Forbidden | 認証済みだがアクセス権がない |
| 404 Not Found | リソースが存在しない |
| 500 Internal Server Error | リクエストは正当だがサーバー内部エラー |
| 503 Service Unavailable | サーバーダウン（メンテナンス等） |

ツール別の使い方:

**LLM（Claude Desktop 等）**
- 上記の経路を指示に含め、テストケースごとに適切なリクエストボディまで生成させる
- OpenAPI 仕様をプロジェクトのナレッジとしてアップロードすると API 構造を踏まえたケースが出る（ステータスコード網羅を含め数十件規模の生成が観測されている）
- 生成されたリクエストボディは**送信前に必ずレビュー**する

**Postman**
- HTTP リクエストの手組み・送信・レスポンス（ステータス・ヘッダー・ボディ・Cookie・応答時間）確認の定番。URL 貼り付けでクエリパラメータが自動分解される
- Authorization タブでトークン（無効値を入れて失敗を確認）、Cookies タブで Cookie を操作
- 応答時間表示は入力による性能劣化の探索に使える
- Swagger/OpenAPI からの仕様インポート、GraphQL/SOAP 対応、レスポンス検証の追加によるテスト自動化、コレクション共有と CI/CD 実行も可能
- Agent Mode（AI アシスタント）は API 仕様の理解に基づく共通テストケース・テストスクリプト生成・失敗デバッグを支援する
- **注意**: 既定でワークスペースがクラウド同期される。NDA・社内 IT ポリシーに抵触しないか確認する

**WireMock（スタブによる統合探索）**
- スタブ＝他コンポーネントの挙動を模倣する部品。統合先サービスが未完成でも、合意済み契約からスタブを立てて正常/異常の統合ケースを探索できる
- スタンドアロン JAR を `java -jar wiremock-standalone-x.x.x.jar` で起動（デフォルト 8080 番）
- スタブ登録は管理エンドポイント `/__admin/mappings/new` へ次のような JSON を POST:

```json
{
  "request": { "method": "POST", "url": "/makePayment" },
  "response": { "status": 200, "body": "Payment Successful" }
}
```

- `response` を `{"status": 401, "body": "Payment Unauthorized"}` に差し替えて再登録すれば、UI がエラーを適切に表示するかを観察できる。サービス利用不可（503）や不正リクエストも同様
- **注意**: スタブで探索した後も、実コンポーネントが揃ったら必ず E2E で検証する

### Web UI 探索

**ブラウザカバレッジ**
- 目安: ユーザーベースの 85% 以上をカバーするブラウザ群でテスト（世界シェア統計ではおおむね Chrome ＋ Safari ＋ Edge が最低ライン。自アプリの実ユーザー統計を優先）
- 旧ブラウザや多様な OS×ブラウザ組合せは、BrowserStack や Sauce Labs などのクラウドテストプラットフォームで仮想的に賄う。QA/ステージング環境のプライベートアプリをトンネル経由でテストできるものもある

**データ自動入力系拡張（Auto Data Filler 等）**
- 編集可能要素を右クリックすると、入力長・名前・書式などのエッジケース値の候補一覧が出て、選ぶだけで入力される。実質「探索用チェックリスト」として機能する
- あわせて、公開されている探索的テストのヒューリスティクス・チートシートも、ケース漏れ防止（特に初心者）に有効

**Playwright MCP（AI にブラウザ操作をさせる自律探索）**
- Playwright の E2E ブラウザ操作能力を MCP 経由で LLM に開放するツール。`npm install @playwright/mcp` で導入し、MCP 設定に登録:

```json
{
  "mcpServers": {
    "playwright": { "command": "npx", "args": ["@playwright/mcp@latest"] }
  }
}
```

- テストケース生成の手順書型指示に、次のステップを追加する:
  - 「Web ページ URL が与えられたら Playwright MCP ツールでサイトを開き、**受入基準に限定せず、ただしストーリーで指定された単一機能に絞って**、ツールでステップを1つずつ実行して探索せよ」
  - 「探索の主要な発見（受入基準からの逸脱）を提示せよ」
- 「ストーリー ID をテスト環境 URL で探索して」と促すと、ストーリー・エピック・ガイドライン取得→実ブラウザ探索→テストケース一覧→逸脱レポート（プレースホルダー文言の差異、ページあたり件数の超過など）まで一気通貫で出る
- 発展: 指示を減らして自由探索させ新規ケースを発見させる、特定したケースの自動化スクリプトを生成させる、という使い方もある
- **最重要の注意**: 実行していないのに「合格」と報告するハルシネーションが起こりうる。実行ログ・実画面と突き合わせて監視すること

**Chrome DevTools（右クリック→検証で起動）**

| 機能 | 探索での使い方 |
|------|---------------|
| Console タブ | ページごとにエラーを確認する習慣をつける。画像欠落等のバグレポートに Console のエラーを添付 |
| Network タブ: リクエスト数 | ロジック不良による不要な API 大量呼び出し（ページ低速化の原因）を件数表示で検出 |
| Disable cache | キャッシュを無効化して**初回訪問ユーザーの体験**を再現。開発中の画像差し替え漏れ等を発見 |
| ネットワークスロットリング | 2G/3G/4G/オフラインを模擬し、低帯域での段階的読み込みをスクリーンショット列で観察。オフライン動作（PWA）の検証も可 |
| Network タブ: リクエスト詳細 | UI→API 統合の探索: 入力どおりのクエリパラメータが正しいエンドポイントへ渡るか、404 時に「在庫なし」等の表示が出るか |
| Block Request URL | 特定リクエストをブロックして、**実際にサービスを落とさずに**サービスダウン時の UI 挙動を模擬 |
| Application タブ: Cookies | セッション情報等の Cookie を閲覧・編集・削除して挙動を観察 |

DevTools には AI アシスタント統合もあり、セキュリティ・性能テストにも横断的に使える。

## 合成テストデータ生成（Synthetic Data Vault）

テストデータは探索的テストの成否を分ける。本番データに近い（規模・分布とも）ほど結果は良いが、本番データの匿名化利用にはセキュリティ・プライバシー法の慎重な確認が要るため、合成データが必要になる場面がある。

- **Synthetic Data Vault（SDV）**: ML/DL で本番データの統計的特徴（自覚していない分布も含む）を学習し、同様の分布を持つ表形式合成データを大量生成する研究由来のツール
- オープンソースの Python ライブラリとして利用可能。DB テーブルを CSV で与えると数千件規模の合成レコードを生成する。**入力データそのものは合成データに複製されない**（匿名性・プライバシー維持）
- エンタープライズ版は強化モデルと DB コネクタを持ち、本番 DB から（機微情報を匿名化しつつ）直接学習してテスト DB に生成できる
- 目安: 匿名化済み本番サンプル 2,500 件程度が生成の出発点として妥当とされる
- 相互接続された複数テーブルを持つ大規模レガシー DB で、テーブル間のリレーションを保ったまま生成できる点が特に有用。性能テストのデータ準備にも効く

## テスト環境衛生（Test Environment Hygiene）

テスト環境は探索の「遊び場」であり、その荒廃はテスト結果を直撃する。以下の「臭い」を早期に検知して直す。

| 臭い | 問題 | 対策 |
|------|------|------|
| 共有テスト環境 | サービスを一時停止する等の「環境いじり」に他チームの同意・調整が要る。デプロイ待ちで最新コードを探索できない | サブチームごとの専用テスト環境で自由な探索を担保 |
| デプロイ衛生の欠如 | CI からの自動デプロイが探索中に環境設定を予告なく変える。未修正欠陥入りビルドが探索をブロック | 専用環境へのデプロイは手動トリガーに。自動テスト合格ビルドのみデプロイ可能にする。ファイアウォール・層分離・レートリミット等、本番相当の構成にする（でないと失敗系を探索しきれない） |
| テストデータの荒廃 | 古いデータ・設定が残り、新機能の探索を自ら狂わせる | 新しいストーリー開始時に新ビルドをデプロイして初期状態に戻す。またはストーリーごとに新規テストデータ（既存ユーザーでなく新規ユーザー）を作る |
| チームの非自律性 | 認証情報・設定変更権限・ログ閲覧・スタブ設置のたびに DevOps へ依頼が必要で探索が停滞 | チームが全コンポーネントへアクセスできる自律性を確保 |
| サードパーティ未整備 | 「統合は本番で確認すればいい」という前提が終盤の想定外ブロックを生む | スタブを立てるか、限定アクセス契約を結び、テスト環境で外部統合を探索可能にしておく |

## まとめ（チェックリスト）

- [ ] 探索的テスト＝学習・テスト設計・テスト実行の同時進行。仕様チェックリストの消化（従来型手動テスト）とは別物
- [ ] ビジネス・技術・エンドユーザーの3視点を統合し、「真」とされる前提を疑う
- [ ] フレームワーク8種（同値クラス分割・境界値分析・状態遷移・デシジョンテーブル・原因結果グラフ・ペアワイズ・サンプリング・エラー推測）で思考を構造化する
- [ ] 戦略: 5領域（ペルソナ・ドメイン・ビジネス優先度・アーキテクチャ・インフラ/設定）でアプリを理解 → 4観点（機能フロー・失敗/エラー処理・UI look & feel・横断特性）を部分ごとに探索 → dev-box/ストーリー/バグバッシュ/リリースの各フェーズで反復
- [ ] 探索で見つけた新フロー・ケースの自動化まで終えて「機能完了」とする
- [ ] AI はテストケースのスターターキット生成・API リクエスト生成・自律 UI 探索に有効。ただしスコープを狭く、文脈（受入基準・エピック・探索ガイドライン）を与え、ハルシネーションを常に検証する
- [ ] ツール: API は Postman／WireMock、Web UI はブラウザ＋データ自動入力拡張＋Chrome DevTools＋Playwright MCP、テストデータは SDV 等の合成データ生成
- [ ] テスト環境の衛生（専用環境・デプロイ規律・データ鮮度・チーム自律性・外部統合の手当て）を維持する
