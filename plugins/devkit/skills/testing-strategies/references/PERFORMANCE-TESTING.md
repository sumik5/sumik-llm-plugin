# 性能テスト（バックエンド／フロントエンド）

Web アプリケーションの性能をバックエンド（API 負荷テスト）とフロントエンド（ブラウザ描画性能）の両面から計測・検証するためのリファレンス。性能目標の立て方、KPI、テストタイプと負荷パターン、実施手順、JMeter / Gatling / ApacheBench / Azure Load Testing の使い分け、RAIL モデルと Core Web Vitals、Lighthouse / Chrome DevTools / Playwright / PageSpeed Insights / WebPageTest による計測、そしてシフトレフト型の性能テスト戦略への統合をカバーする。

---

## なぜ性能テストか

- **離脱率（bounce rate）との相関**: 公開されている調査では、ページロード時間が 1→3 秒に伸びると離脱確率が約 32% 増、1→6 秒で約 106% 増、1→10 秒で約 123% 増とされる。遅延 1 秒ごとに顧客は競合へ流れる。
- **SEO への影響**: 検索エンジンのランキングアルゴリズムは遅いサイトを下位に評価する。
- **売上への直結**: 大手 EC・サービス事業者の公開事例では、ロード時間を数百 ms 短縮しただけで年間売上が数百万〜数千万ドル規模で改善している。逆にピークトラフィックの取りこぼしは巨額の機会損失になる。
- **知覚のしきい値**（ユーザビリティ研究の知見）:

| 応答時間 | ユーザーの知覚 |
|---------|--------------|
| 〜0.1 秒 | 即時と感じる |
| 0.2〜1 秒 | 遅延を知覚するが操作の主導権は維持 |
| 1 秒超 | UI が「もっさり」していると感じる |
| 2 秒 | Web サイトとして許容される上限の目安 |

**性能の定義**: 多数の同時ユーザーを、エンドユーザーが許容できる水準を下回らない挙動で捌き続ける能力。テストの出発点は「ピーク時の想定ユーザー数を見積もり、その負荷下で許容水準を満たすか検証する」こと。

## バックエンド性能テスト

### 性能に影響する要因

| 要因 | 典型的な問題 |
|------|------------|
| アーキテクチャ | サービス責務の分割が不適切で UI から多数の呼び出しが必要になる。適切な層でのキャッシュ欠如 |
| 技術スタック | 層ごとのツールの組み合わせが噛み合わない。言語選択がサーバーレスのコールドスタート時間に影響する等 |
| コード品質 | 入力バリデーション欠如により空文字検索が DB まで到達して無駄に遅延する等 |
| DB の選択と設計 | 要件に合わない DB タイプ、1 注文が多テーブルに分散して取得時に集約コストが発生する等 |
| ネットワーク遅延 | コンポーネント間の接続品質。エンドユーザー側の回線（2G〜WiFi）は制御不能だが、重い画像や大容量転送を避ける設計で緩和できる |
| ホスティング地域 | ユーザーが特定地域に集中するならその近くにホストしてホップ数を減らす。グローバルなら複数リージョン複製や CDN |
| インフラ | CPU・メモリ等の能力が負荷耐性を直接決める。性能テスト結果はキャパシティプランニングの入力になる |
| サードパーティ連携 | 外部システムのレイテンシがそのまま自システムのレイテンシに加算される |

**原則**: テスト環境はネットワーク・インフラ・地理条件を本番に近づける。そうでないと計測値は本番性能を代表しない。

### KPI（何を計測するか）

| KPI | 定義 | 補足 |
|-----|------|------|
| 応答時間（response time） | クエリへの応答完了までの時間（API 応答＋ページ完全表示） | Web は 2 秒以内が推奨目安 |
| 同時実行数（concurrency） | ある時点で許容水準内に応答できる同時ユーザー数 | 例: 「500 同時ユーザーに 2 秒以内で応答」 |
| スループット（throughput） | 単位時間あたりに処理できるリクエスト数 | システム視点では同時ユーザーより本質的な指標（リクエストはキューイングされ逐次処理されるため） |
| 可用性（availability） | 継続期間にわたり許容水準内の応答を維持する能力 | メモリリークやバッチ競合で時間経過とともに劣化しうる |

### テストタイプ

| タイプ | 目的 | 設計の要点 |
|-------|------|-----------|
| ロード／ボリュームテスト | 想定ユーザー数で目標応答時間を満たすか検証 | 複数回実行して平均でベンチマーク化する |
| ストレステスト | 劣化開始点とエラー発生点（限界値 X, X+n）を特定 | ロードテストの上限を超えて段階的に負荷を増やす。スケール計画・セール等のイベント準備に使う |
| ソークテスト | 一定負荷を長時間維持して可用性を検証 | メモリリーク等の経時劣化を検出 |
| スパイクテスト | 急激な負荷の立ち上がりに対する耐性を検証 | ピーク–休止パターンの繰り返しで代替可能 |

### 現実的な負荷設計の 3 要素

| 属性 | 定義 | アンチパターン |
|------|------|--------------|
| ramp-up time | テスト開始から全仮想ユーザー接続完了までの時間（例: 100 ユーザーを 1 分で投入） | 全ユーザーが同一瞬間にログインする非現実的な設定 |
| think time | ユーザーがアクション間で考える時間（数秒〜数分） | ユーザーをロボット扱いし、ログイン→検索→購入を数 ms 間隔で実行 |
| pacing | トランザクション間の間隔（例: 1,000 件/時なら 1 時間に分散） | 全トランザクションを冒頭に集中させる |

### 負荷パターン

| パターン | 形状 | 用途 |
|---------|------|------|
| Steady ramp-up | 一定期間で漸増→一定負荷を維持→漸減 | 最も一般的（セール開始時など） |
| Step ramp-up | バッチ単位で段階的に増加（例: 2 分ごとに 100 ユーザー） | 段階ごとの計測で負荷別ベンチマーク・チューニング・キャパシティ計画 |
| Peak–rest | ピークと休止のサイクルを反復 | SNS 型のトラフィック（1 日の中で波がある） |

### 実施手順（6 ステップ）

**Step 1: 目標 KPI の定義**
- まず定性的に考えてから数値化する（例: 「もう 1 か国分スケールできる」「競合 X より速い」「前バージョンより速い」）。
- ビジネス側の数字を鵜呑みにせず**データから導出**する:
  - 既存アプリがある → 本番データを分析して KPI と負荷パターンを導く
  - 新規だが競合がいる → 競合のデータを参考にする
  - 完全新規 → 国別インターネット利用状況・想定ピーク時間帯等から推定する

**KPI 算出の例**（年間 10 万ユーザー・月 2 回アクセス・平均セッション 10 分・活動時間帯 12 時間の場合）:

```
月間アクセス   = 100,000 × 2 = 200,000
日次平均      = 200,000 ÷ 30 ≈ 6,667
時間あたり平均 = 6,667 ÷ 12 ≈ 555 → ピーク余裕を見て 1,000 に切り上げ
同時ユーザー   = 1,000 × (10分/60分) ≈ 166
スループット   = 5 リクエスト/セッション × 1,000/時 = 5,000 リクエスト/時
→ 目標: 「166 同時ユーザーに 3 秒以内で応答」「5,000 req/h を維持」
```

算出結果はステークホルダーと合意し、初年度以降の成長も見込んで再確認する。

**Step 2: テストケースの定義**
- 負荷パターン×テストタイプの語彙で記述する。重要エンドポイント全てについて可用性・スループット・応答時間をカバーする。
- 機能テストと違い、性能テストケースは少数で足りる。
- 例: 全エンドポイントの応答時間ベンチマーク／顧客向けエンドポイントのみ 166〜200 同時ユーザーでボリュームテスト（管理者専用エンドポイントは対象外にしてよい）／100 ユーザー刻みのストレステストで限界点特定／think time 30 秒を挟んだユーザーフローで時間スループット検証／同設計を 12 時間回すソークテスト。

**Step 3: テスト環境の準備** — チェックリスト:
- [ ] 各層・コンポーネントの配置構成が本番同等
- [ ] マシン構成（CPU 数・メモリ・OS バージョン）が同等
- [ ] ホスティングの地理的ロケーションが同一
- [ ] マシン間ネットワーク帯域が同等
- [ ] レート制限等のアプリ設定が完全一致
- [ ] バックグラウンドのバッチジョブ・メール送信系も稼働
- [ ] ロードバランサー配置済み
- [ ] サードパーティ連携（最低限モック）が利用可能
- [ ] テストランナー（負荷生成用の別マシン）を用意。グローバル対象ならリージョン別に複数配置して各国からの遅延を観測

コスト面で本番同等が無理なら妥協点をステークホルダーと交渉し、**その妥協が計測精度を落とすことを明示**する。性能テスト環境は QA 環境と同時期（プロジェクト初期）に用意するのがベストプラクティス。

**Step 4: テストデータの準備**
- データ品質が計測値を大きく左右する。理想は匿名化した本番データ。不可なら本番を模倣する:
  - 本番 DB のサイズを見積もり、投入スクリプトを整備（テストランごとにクリーン＆再投入するため必須）
  - 「Item1」「Item2」ではなく現実的な値を使う
  - 実ユーザー入力に含まれるような誤記・空白などの汚れデータも混ぜる
  - 年齢・国などの分布も本番に近づける
  - 同時ユーザーテスト用にユニークなクレデンシャル等を大量生成する

**Step 5: APM ツールの統合**
- New Relic / Dynatrace / Datadog 等を統合し、テスト中のシステム内部挙動（メモリ不足による失敗等）を観測できるようにする。AI 搭載 APM はテレメトリの根本原因分析（該当コード行の特定まで）を高速化する。

**Step 6: スクリプト化と実行**
- CI 連携可能なツールでスクリプト化する。OSS: JMeter, Gatling, k6, ApacheBench, Locust。商用クラウド: BlazeMeter, NeoLoad, Azure Load Testing 等。
- 本番実行の前に少数ユーザーでドライランして所要時間を見積もる。テストランは数分〜数時間かかる。
- レポート収集・デバッグ・修正・サーバーチューニングの時間とキャパシティを計画に織り込む。

> **警告**: 公開 API への高負荷テストは DDoS 攻撃と見なされうる。演習・検証はスタブ（WireMock 等）か、ツールベンダー提供の練習用サイトに規定の最小負荷で行うこと。

## ツール選定（バックエンド）

| ツール | 特徴 | 向いている場面 |
|-------|------|--------------|
| JMeter | GUI ベース・Java 製・プラグイン豊富・OSS。BlazeMeter 連携でクラウド実行可 | GUI で素早くテスト設計したい／非コーダーも触る／プラグインで多様な負荷パターン |
| Gatling | コードでテストを書く（Java/Kotlin/Scala）。OSS＋商用版。直感的な HTML レポート | テストをコードとしてバージョン管理・レビューしたい開発者チーム |
| k6 | JavaScript でスクリプト記述・CLI 中心 | 開発者フレンドリーな CI 組み込み |
| Locust | Python スクリプトで分散負荷テスト | Python チーム・分散実行 |
| ApacheBench (ab) | コマンド一発の超軽量 CLI（macOS 標準搭載） | 開発中のクイックな数値確認 |
| Azure Load Testing | マネージドサービス。JMeter/Locust スクリプトの AI 支援生成・サーバーサイド監視・AI 診断 | Azure リソースへの大規模負荷＋AI 分析まで一気通貫 |

### JMeter の要点

**セットアップ**: 公式サイトの ZIP を展開し `bin/jmeter.sh` で GUI 起動（互換 Java と JAVA_HOME 必須）。Plugins Manager の JAR を `lib/ext` に置いて再起動。

**基本ワークフロー**:
1. Test Plan 右クリック → Add → Threads (Users) → Thread Group を作成
2. Thread Group 配下に HTTP Request サンプラーを追加（サーバー名・メソッド・パス）
3. リスナーを追加: View Results Tree（個別リクエストの成否・リクエスト/レスポンス内容）と Aggregate Report（平均・中央値・90% Line・スループット等）
4. 保存して実行、リスナーで結果確認

**ベンチマーク設定例**: `Threads=1, Ramp-up=0, Loop Count=10` → 同一リクエストを 10 回実行して平均応答時間を得る。

**ボリュームテスト設定例**: `Threads=166, Ramp-up=0, Loop Count=5`。結果の読み方: Average=801ms / 90% Line=1499ms なら「90% のユーザーは約 1.5 秒以内、平均では 0.8 秒以内に応答を得た」。

**落とし穴**: JMeter はステータスコード 200＝成功と見なす。「200 だが意図した結果ではない」応答（重複登録を 200＋エラーメッセージで返す等）は**明示的なアサーション**（Assertion コンポーネント）を追加しないと成功扱いになる。

**負荷パターン用プラグイン**（Custom Thread Groups）:
- **Concurrency Thread Group**: `Target Concurrency=166, Ramp Up Time=0.5(分), Hold Target Rate Time=2(分)` のように同時実行数を制御。ストレステストでは x ユーザー刻みのステップ増加を設定して限界点を探す
- **Ultimate Thread Group**: 開始遅延・起動時間・保持時間・シャットダウン時間を行単位で設定。行を重ねれば peak–rest パターンも作れる。ソークテスト（長時間一定負荷）にも使う
- **Parallel Controller ＋ Timer**: 複数 HTTP リクエストの並列化と think time 挿入で時間スループット検証。Constant Throughput Timer はスループットを設定値に固定（超過時は自動減速）
- **If / Loop / Random コントローラ**: 条件分岐を含むシナリオ構築

**データ駆動性能テスト**（CSV Data Set Config）:
1. `name,author,...` 等をヘッダーにした CSV を用意（変数名として参照される）
2. Thread Group → Add → Config Element → CSV Data Set Config でファイルパスと変数を指定
3. リクエストボディ内で `${variable_name}` 記法で参照
- ログイン必須アプリのクレデンシャル投入や、テスト開始前のテストデータ生成にも同じ仕組みが使える

**CI 統合**: 性能テストは専用ジョブとして**完全に隔離して**実行する（他ジョブと同居すると計測が汚れる）。

```bash
jmeter -n -t library.jmx -l result.log -e -o ./report/
```

**AI 支援**: GUI に AI チャットを追加するプラグインがあり、Thread Group / HTTP Request / リスナーの追加をプロンプトで指示したり、失敗原因を対話で調べたりできる（LLM ベンダー API キーが必要）。

### Gatling の要点

コードでテストを定義する派向け。`HttpProtocolBuilder`（HTTP 呼び出しの構造定義）と `ScenarioBuilder`（負荷パターン構築）の 2 系統 API を使う。

```java
public class BooksSimulation extends Simulation {
  HttpProtocolBuilder httpProtocol =
      http.baseUrl("https://example.test")
          .acceptHeader("application/json");

  ScenarioBuilder scenario =
      scenario("Scenario").exec(http("Session").get("/books"));

  {
    setUp(scenario.injectOpen(constantUsersPerSec(166).during(5)))
        .protocols(httpProtocol);
  }
}
```

実行は `mvn gatling:test`（Maven プラグイン＋ `gatling-charts-highcharts` 依存）。応答時間・パーセンタイル推移・秒間リクエスト数などを含む HTML レポートが自動生成される。

| 負荷 API | 動作 |
|---------|------|
| `constantUsersPerSec(n).during(t)` | 毎秒 n ユーザーを t 秒間投入 |
| `rampUsersPerSec()` | 指定期間で漸増 |
| `stressPeakUsers()` | ステップパターン |
| `atOnceUsers()` | 全ユーザー一斉投入 |
| `nothingFor()` | 指定時間の休止 |

複数の負荷パターンを別シナリオとして同一シミュレーションに載せ、CI で各性能メトリクスにアサートできる。詳細は公式ドキュメント（https://docs.gatling.io）。

**AI コーディング支援の注意**: 性能テストの OSS コードベースは学習データが少なく、AI アシスタントは依存関係の解決やテスト構文の提案を誤りやすい（存在しないライセンス要求を提案する等）。依存は Maven Central 等の一次ソースで自分で確認すること。

### ApacheBench の要点

```bash
ab -n 200 -c 200 https://example.test/books
```

`-n`=総リクエスト数、`-c`=同時実行数。出力にはスループット（Requests per second）・平均応答時間・接続時間統計・パーセンタイル分布（50%〜100%）が含まれ、開発中のクイックフィードバックに十分。

## フロントエンド性能テスト

### なぜバックエンド計測だけでは不十分か

負荷テストツールは HTML を取得するだけで**ページを描画しない**。実ブラウザは HTML 取得→CSS・画像等のダウンロード→スクリプト実行を行い、スクリプト実行中は並列処理を停止する（スクリプトがページ構造を書き換えうるため）。結果として、**ページロード時間の 80〜90% はフロントエンド側の描画処理**が占めるという推計が広く知られている。API 応答が数百 ms でも、ユーザーが見る体感はそれよりはるかに遅い。サーバー KPI はキャパシティ計画に不可欠だが、ユーザー体験の把握にはフロントエンドメトリクスの計測が別途必要になる。

### フロントエンド性能に影響する要因

| 要因 | 内容 |
|------|------|
| コード品質 | JS の minify、ページあたり HTTP リクエスト数の削減、キャッシュ活用の不徹底が遅延を積み上げる |
| CDN | 地理的に近いサーバーから静的コンテンツを配信して遅延を削減。ただし CDN 自体の性能もロード時間に影響 |
| DNS ルックアップ | 初回 20〜120ms。ブラウザ・OS・ISP がキャッシュするため初回訪問者が最も影響を受ける |
| ネットワーク帯域 | モバイル回線は特に低帯域になりうる。低帯域検出時に軽量版を配信するサイトもある |
| ブラウザキャッシュ | 画像・Cookie 等のキャッシュで初回と 2 回目以降のロード時間が大きく変わる |
| データ転送量 | 転送量が大きいほどネットワーク遅延の影響が増幅される |

### RAIL モデル（目標設定の枠組み）

ユーザー体験を 4 領域に分解し、数値目標を与える思考フレームワーク。「ページがなんか遅い」という曖昧な表現を具体的な言語に置き換える。

| 領域 | 目標 | 意味 |
|------|------|------|
| **R**esponse | 入力遅延 100ms 以内 | クリック等への反応が 100ms を超えるとユーザーはラグを感じる |
| **A**nimation | 1 フレーム 16ms 以内 | 60fps を維持できないとスクロール・D&D・ローディング表示がカクつく |
| **I**dle | アイドル処理は 50ms 単位 | 非重要タスクはアイドル時間にまとめ、ユーザー操作時に 100ms 以内で応答できる余地を残す |
| **L**oad | 1 秒以内に描画開始 | 描画開始が 1 秒を超えるとユーザーは操作の主導権を失ったと感じる |

参考: https://web.dev/rail

### 主要フロントエンドメトリクス

| メトリクス | 測るもの |
|-----------|---------|
| First Contentful Paint (FCP) | DOM 最初の要素が描画されるまでの時間（「何か動いた」までの待ち時間） |
| Largest Contentful Paint (LCP) | 最も大きい要素（メイン画像・テキスト塊）が可視になるまでの時間 |
| Time to Interactive (TTI) | ページが操作可能になるまでの時間（見えるのに反応しない状態の解消） |
| Cumulative Layout Shift (CLS) | 予期しないレイアウトのずれの累積量（視覚的安定性・小さいほど良い） |
| First Input Delay (FID) | FCP〜TTI 間に行われた最初の操作への応答遅延 |
| Max Potential FID | FID の最悪ケース（同区間で最長タスクの所要時間） |

LCP / FID / CLS は **Core Web Vitals**（https://web.dev/vitals）として括られ、非技術者にも性能を伝えやすい共通言語になる。多くの計測ツールがこの 3 つを標準で出力するため、CI で継続計測してシフトレフトする。

### テストケース設計の観点

ターゲットユーザーのデモグラフィックを跨いで設計する:

- デバイスタイプ（デスクトップ／モバイル／タブレット）と、対象地域で主要なデバイスメーカー（CPU・バッテリー・メモリ差）
- ネットワーク帯域（WiFi / 3G / 4G）と対象国の平均回線速度
- 地理的分布（複数ロケーションからの計測）
- 稼働中アプリがあるならアクセス解析ツールの実利用データを設計に使う

テストケース例: 「特定都市の中位モバイル端末ユーザーが 4G 回線でトップページにアクセスする」。

### ツール別ワークフロー

#### Lighthouse

Chrome 同梱の OSS 監査ツール。性能のほかセキュリティ・アクセシビリティも採点する。

**手動ワークフロー**:
1. 対象ページを Chrome で開き DevTools を起動（macOS: Cmd-Option-J / Win・Linux: Shift-Ctrl-J）
2. Network タブでネットワークスロットリング（例: Slow 3G）を設定
3. Performance タブで CPU スロットリング（4x=中位端末 / 6x=低位端末）を設定
4. デバイス解像度を選択
5. Lighthouse タブで Performance カテゴリを選び Generate report

**CI 統合**（Lighthouse CI で性能フィードバックを日次→分単位に短縮した公開事例あり）:

```bash
npm install -g lighthouse
lighthouse https://www.example.com/ --only-categories=performance
```

- オプションでスロットリング値・画面サイズを指定可能
- スコアが閾値（例: 90）未満ならパイプラインを fail させるラッパーを書く
- LightWallet 機能で Web Vitals ごとの performance budget（上限値）を定義し超過を検知
- `playwright-lighthouse` / `cypress-audit` モジュールで E2E スイートに監査を組み込める

#### Chrome DevTools Performance パネル

ネットワークスタック・フレームレート・GPU・メモリ・スクリプト実行時間の詳細分析。ブラウザ組み込みなので開発中のデバッグに最適。

1. DevTools → Performance タブでネットワーク／CPU スロットリングを設定し Record
2. 対象操作（検索など）を実行して記録停止
3. リクエストごとに Core Web Vitals・ネットワーク呼び出し・スクリーンショット付きタイムラインを確認
4. AI 支援機能（設定で有効化）を使うと、LCP 内訳カード等から文脈付きで質問でき、「画像リクエストのキューイングが長すぎる」といった根本原因と最適化提案を得られる

ネットワークトレース付きの分析は「3G で遅い気がする」という曖昧な報告より遥かに具体的な証拠になる。

#### Playwright（機能テストへの性能アサーション組み込み）

既存の UI 機能テストを拡張し、W3C 標準のブラウザネイティブ Performance API で Core Web Vitals を継続監査する。

```typescript
// BasePage 等の共通クラスに置き、各ページクラスから呼ぶ
async getLargestContentfulPaint(): Promise<boolean> {
  const lcp = await this.page.evaluate(() =>
    new Promise<number | null>((resolve) => {
      new PerformanceObserver((list) => {
        const entries = list.getEntries();
        resolve(entries.at(-1)?.startTime ?? null);
      }).observe({ type: 'largest-contentful-paint', buffered: true });
    })
  );
  return lcp !== null && lcp < 2500; // 閾値超過で fail
}

async checkPerformanceVitals() {
  if (!(await this.getLargestContentfulPaint())) {
    throw new Error('Performance Vitals failed');
  }
}
```

- Core Web Vitals チェック・ビジュアルテスト等の反復アクションは `BasePage.ts` に抽象化して全ページクラスに継承させる
- 他のメトリクスも MDN の Performance API ドキュメントに従って同様に取得・アサート可能
- AI アシスタントはこの種の W3C 標準 API コードを概ね正しく生成できるが、**null チェック・タイムアウト・グレースフルな失敗処理を欠落させがち**なのでレビュー必須

#### PageSpeed Insights（フィールドデータ＝RUM）

Lighthouse や DevTools は「ラボデータ」（前提条件を固定した実験室計測）。実ユーザーの回線・端末のばらつきは予測できないため、リリース後は **Real User Monitoring（RUM）** で実測する。PageSpeed Insights は世界中の実ユーザーから収集した Core Web Vitals（フィールドデータ）と Lighthouse ラボデータを併記し、失敗診断と改善示唆を提供する。API 経由の常時監視・アラートも可能。

#### WebPageTest

実ブラウザ（Web／モバイル）でページを描画して評価する無料オンラインツール（https://www.webpagetest.org）。ロケーション・デバイス・解像度を選択でき、DNS 解決・接続確立・HTML/画像ダウンロード等のタスク別所要時間を**ウォーターフォールビュー**で可視化する。異常箇所が最適化の手がかりになる。API・CLI（Node.js モジュール）もある。

**落とし穴**: 認証情報を渡せるが、公開ホストされたテストレポート上にその認証情報が見えてしまう。テスト専用クレデンシャルのみ使用すること。

### ラボデータ vs フィールドデータの使い分け

| データ種別 | ツール | 用途 |
|-----------|-------|------|
| ラボデータ | Lighthouse, Chrome DevTools, WebPageTest, Playwright 計測 | 開発中・CI での再現可能な回帰検知 |
| フィールドデータ（RUM） | PageSpeed Insights, RUM サービス | リリース後の実ユーザー体験の実測・監視 |

## シフトレフト性能テスト戦略（統合）

性能テストを開発ライフサイクル全体に分散配置する。

| フェーズ | やること |
|---------|---------|
| **計画** | プロジェクト開始前にビジネス・マーケ・技術の全ステークホルダーと性能 KPI を合意し、その数値からアーキテクチャと技術スタックを設計する。性能テスト環境をプロジェクト初期に構築（本番同等でなくても先に確保）。全ユーザーストーリーの受け入れ基準に、フロントエンド性能テストケース（回線・地域条件込み）と API の応答時間・同時実行・可用性 KPI を含める |
| **開発** | ユーザーストーリー単位で応答時間・エンドポイント負荷テストによりサーバー KPI を検証。フロントエンド性能テストケースも同時に検証 |
| **CI** | 応答時間検証テストは毎コミット実行。負荷テストは所要時間に応じて毎コミットまたはナイトリー回帰で実行（機能追加による漸進的な性能劣化を可視化できる）。頻出ページのフロントエンド性能テストもパイプラインに含める |
| **ストーリーテスト** | 探索的テスト中に体感できる性能ボトルネックを監視。性能系受け入れ基準が満たされ・自動化され・CI 統合されてからストーリーを完了にする |
| **リリーステスト** | ストレステスト・ソークテスト・APM を使ったデバッグを含む E2E の全体性能テストを実施 |

**quality gate の Tip**: シフトレフト時は、フロントエンド性能を検証するブラウザ・デバイス・解像度の小さなサブセットを「development done」ゲートに、残りを「testing」ゲートに割り当てる。

## チェックリスト（要点）

- [ ] 性能 KPI は勘ではなく本番データ／競合データ／市場データから導出したか
- [ ] ramp-up・think time・pacing を含む現実的な負荷設計になっているか
- [ ] テスト環境・テストデータは本番相当か。妥協点の精度影響を明示したか
- [ ] APM を統合し、テスト中の内部挙動を観測できるか
- [ ] ステータスコード 200 に隠れた業務エラーへのアサーションがあるか
- [ ] 性能テストジョブは CI 上で隔離実行されているか
- [ ] バックエンド KPI とは別にフロントエンドメトリクス（Core Web Vitals）を計測しているか
- [ ] ラボデータ（CI）とフィールドデータ（RUM）の両方を運用しているか
- [ ] 公開 API に対する高負荷テストを避けているか（DDoS 扱いのリスク）
