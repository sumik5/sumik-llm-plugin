# モバイルテスト戦略リファレンス

モバイルアプリのテストは Web テストと共通の基盤（サービス層・DB 層のマイクロ/マクロテスト）に加え、デバイス断片化・アプリ種別・ネットワーク変動というモバイル固有の 3 変数を扱う必要がある。本リファレンスは、デバイス選定からテスト次元別のツール選択、モバイルテストピラミッドの判断基準までを、実務で使えるチェックリストと表で整理する。

## モバイル環境の 3 要素

テスト計画の前提として「デバイス」「アプリ種別」「ネットワーク」の 3 軸を把握する。

### デバイス断片化の 5 次元

| 次元 | テストへの影響 | 確認ポイント |
|------|--------------|-------------|
| 画面サイズ | スマートフォン/タブレット、縦横回転、分割画面で表示領域が変動 | 小画面での過剰スクロール、大画面での余白過多 |
| ピクセル密度 | 同一画面サイズでも密度（LDPI〜XXXHDPI）が異なる | 画像の自動リサイズによるぼやけ・歪み。密度別アセットの用意と検証 |
| OS とバージョン | Android / iOS で約 99% を占めるが、各 OS の旧バージョンが広く現役（断片化） | 機能サポート差・挙動差をバージョン横断で確認 |
| ハードウェア | RAM・CPU・バッテリー・ストレージ・センサー（GPS/カメラ/マイク等）が機種ごとに異なる | 性能だけでなくコア機能に影響しうる（例: 災害時アプリは省電力が必須要件） |
| メーカー | カスタム Android（独自 OS 派生）、ハードキー、指紋センサー位置等の独自仕様 | メーカー固有の挙動差を対象デバイスで確認 |

> 用語: 解像度は水平×垂直のピクセル数（例 1024×768）、ピクセル密度は 1 平方インチあたりのピクセル数。

### アプリ種別と選択のトレードオフ

| 種別 | 実装 | 強み | 弱み | テスト観点 |
|------|------|------|------|-----------|
| Native | OS 別に開発（Android: Java/Kotlin、iOS: Objective-C/Swift） | 最高性能・全 OS API/ジェスチャー・オフライン動作 | OS ごとに別開発＝高コスト。ストア審査が緊急修正にも必須 | ストア配布フロー、OS 機能連携、オフライン |
| Mobile Web | ブラウザで動く Web サイト | OS 非依存・インストール/審査不要 | OS 機能（カメラ・連絡先等）非アクセス、オフライン制限 | ブラウザ互換、レスポンシブ |
| Hybrid | Web 技術＋ネイティブコンテナ | Web 部分はサーバー配信＝審査なしで更新可能。低コストで MVP 向き | 性能・UX はネイティブ未満、オフラインは限定的 | オンライン/オフライン切替、コンテナ経由の OS API |
| Cross-platform | 単一コードベース（Flutter / React Native 等） | 開発効率とネイティブ相当の UX・オフライン対応 | 最新 OS 機能の追従遅延、バイナリ肥大。グラフィック重視ならネイティブ優位 | フレームワーク固有の描画・OS 差分 |
| PWA | URL からインストール可能な高度な Web アプリ | プッシュ通知・オフライン・OS 機能アクセス・省ストレージ・低コスト | ブラウザ/OS のサポート差 | インストール導線、通知、オフライン |

**アプリ種別がテストスコープを決める**: オフライン/オンライン挙動、OS 固有機能、アップデート経路（ストア審査有無）、対応ジェスチャーの範囲が種別ごとに変わる。

### ジェスチャーは横断的要件

スワイプ・ロングプレス・ピンチズーム・ドラッグ・回転などのジェスチャーは、アプリ全体で共通提供されることが多く（例: 左スワイプでメニュー表示）、**すべてのユーザーストーリーに付随する CFR（横断的要件）** として設計・実装・テストする。

### ネットワーク条件

高速回線は普遍ではない。都市部でも接続は揺らぐ。テストすべき条件:

- ネットワーク種別: 2G / 3G / 4G / 5G / WiFi / 完全オフライン
- ネットワークタイムアウト時の挙動とエラー表示
- 回線切替（4G↔3G 等）時の振る舞い
- オフライン時の操作と復帰時の同期
- 各回線種別での起動性能

低帯域市場向けには、軽量版アプリ（2G でも動作する設計）のように**制約をアーキテクチャ段階で織り込む**選択肢もある。

### モバイルアプリのアーキテクチャ

Web アプリとの違いは UI 層がモバイル UI に置き換わる点と、**ローカル DB** が加わる点。ネイティブ/ハイブリッドアプリはユーザー名・プロフィール画像・直近取得コンテンツ等をローカル DB に保持し、オフライン動作と描画高速化を実現する。サービス層・共通 DB 層のテスト手法（ユニット/統合/API テスト、性能・セキュリティ・コンプライアンス）は Web と同一で、**モバイル UI 固有のテストを上乗せ**する。

## デバイス選定戦略（テスト対象の絞り込み）

全デバイス×全 OS の網羅は非現実的。**対象顧客セグメントの約 85% をカバー**する 3〜4 台＋バグバッシュ用の予備数台に絞る。絞り込みの質問:

1. **顧客セグメント**: 高価格帯商材なら低価格帯端末は除外できるか
2. **展開市場**: 対象国・地域で優勢な OS / ベンダーは何か
3. **既存の利用実績**: 既存 Web/アプリのアクセスログから実際に使われているデバイスを特定
4. **回線帯域**: 対象市場の平均回線速度（低価格帯端末を扱う場合は特に重要）

運用上の判断:

- デバイス選定は**プロジェクト開始時**に行う
- 実機購入 vs クラウドデバイスファーム（AWS Device Farm 等・ホスト実機で自動テスト実行可、ただし操作遅延あり）をコスト分析で決定
- シフトレフト: 開発者は最も厳しい解像度（LDPI/MDPI）で開発、dev-box テストで BA/QA/開発者が別デバイスを分担、CI では選定デバイス群で自動回帰

## テスト次元別戦略

### 探索的テスト（手動）

デバイス×ネットワーク×アプリ操作の組合せ爆発があるモバイルでは探索的テストの価値が特に高い。「デバイス」「ネットワーク」「アプリと操作」の 3 レンズで観察する。

| 手段 | 用途 | 限界 |
|------|------|------|
| 実機 | リリース判定の基準。Apple / Google とも実機テストを推奨 | 台数コスト |
| エミュレーター/シミュレーター | テストコード開発中のクイックチェック、サニティ確認 | 一部ハードウェア機能・タッチジェスチャー・センサー統合を再現できない。**これだけでリリース可否を判断しない** |
| ブラウザ DevTools のデバイスモード | モバイル Web の解像度別表示確認 | ネイティブ機能は対象外 |

### 機能自動テスト

ユニットテスト＋UI 駆動 E2E テストを CI に組み込み、スモーク/夜間回帰として対象デバイス群で実行する。

| ツール | 対象 OS | 対応アプリ種別 |
|--------|---------|---------------|
| Appium | Android / iOS（＋デスクトップ/TV） | native / hybrid / cross-platform / mobile web すべて |
| Espresso | Android | native のみ |
| XCUIAutomation | iOS | native のみ |
| Detox | Android / iOS | 主に React Native |

**Appium の位置づけ**: OS 固有フレームワーク（UI Automator / XCUIAutomation）を WebDriver 系の共通 API でラップ。`findElements(By.id)`・`click()` 等の API は OS 間で同一のため、Selenium WebDriver 経験者の学習コストは低い。言語非依存。`DesiredCapabilities` でドライバを初期化する。要素ロケーターの調査には GUI ツール Appium Inspector を使う。

補足: Appium はソースコード不要でデスクトップ/TV アプリも自動化できるため、テスト以外に **RPA ツール**（業務プロセスの E2E 自動化）としても利用可能。

### データテスト

データが「ローカル DB」「共通 DB」「ローカルデバイスストレージ」の 3 箇所に分散する点がモバイル特有。

チェックリスト:

- [ ] 各ストレージへのデータフローを把握したか
- [ ] ローカル DB のキャッシュ表示（例: 直近投稿の即時描画）で**古い情報**を出していないか、保持上限・同期方式は妥当か
- [ ] **双方向同期**: 共通 DB ↔ ローカル DB の同期を、複数デバイス×ネットワーク条件の組合せで検証したか
- [ ] 複数デバイスからのトランザクション（例: 各端末で作成したカレンダー予定）を共通 DB が**競合なく**統合できるか
- [ ] ローカルストレージの境界条件（内部/外部ストレージの満杯・利用不可）と OS のファイル形式制限を試したか

これらはマイクロ/マクロ機能テストとして自動化できる。

### ビジュアルテスト

手動デバイステストで一次カバーしつつ、画面サイズ別の回帰は自動化する。

| 手段 | 特徴 |
|------|------|
| Appium visual testing plugin（`images`） | OpenCV ベースの画像比較。オープンソースで機能テストスイートに同居可能 |
| 商用 AI ビジュアルテストサービス | ページ全体の自動スクロール比較等の高機能。Appium プラグインでは複数スクリーンショットの結合を自前実装する必要がある |

Appium プラグインのセットアップと使い方:

```bash
npm install -g opencv4nodejs
appium plugin install images
appium server -ka 800 --use-plugins=images -pa /wd/hub
```

```java
// ベースライン画像と現画像の類似度を取得
SimilarityMatchingResult result =
    driver.getImagesSimilarity(baselineImg, actualScreen, options);

// スコア(0〜1)が閾値未満なら失敗させる。閾値で感度を調整
assert result.getScore() >= 0.99;

// 失敗時の差分可視化
result.storeVisualization(...);
```

ワークフロー: 各画面のベーススクリーンショット集合を作成 → 現バージョンと比較 → 閾値未満で fail。微小な描画差でスコア 1.0 にはなりにくいため、閾値はアプリごとに調整する。

### セキュリティテスト

シフトレフト戦略（STRIDE による脅威モデリング → SAST/DAST → ペネトレーションテスト）はモバイルでもそのまま適用する。**OWASP Mobile Top 10** への対応を必ず含める。体系的に学ぶなら OWASP MASTG（<https://mas.owasp.org/MASTG>）が起点になる。

| ツール | 種別 | 対象 | 使い方の要点 |
|--------|------|------|-------------|
| MobSF (Mobile Security Framework) | SAST + DAST + マルウェア分析（OSS） | Android / iOS / Windows | Docker で起動し Web UI に APK をアップロード → 脆弱性を深刻度付きでレポート。CI 連携でチェックイン時に自動スキャン可 |
| Qark | SAST（OSS・Python 製） | Android（ソース or APK） | `pip install qark` → `qark --apk <path> --report-type html` で HTML レポート生成 |
| Frida / Snyk Code / Checkmarx 等 | 動的計測・商用 SAST | 各種 | 専門特化する場合の選択肢 |

```bash
# MobSF の起動例
docker pull opensecurity/mobile-security-framework-mobsf:latest
docker run -it -p 8000:8000 opensecurity/mobile-security-framework-mobsf:latest
# → http://0.0.0.0:8000 に APK をアップロード
```

学習用には、意図的に脆弱性を仕込んだ練習用 APK（InsecureBankv2 等）をスキャン対象にできる。自動スキャンで左寄せしつつ、アプリの文脈次第で開発終盤に専門のセキュリティテスターを起用する判断も残す。

### 性能テスト

サービス層の負荷/ストレス/ソークテストは Web と同様に継続する。モバイル UI 層では「性能」の意味が変わる: **共有かつ限られた CPU・メモリ・バッテリー・ネットワーク**の中で動くため、次の 2 基準で判定する。

1. **リソースを独占・枯渇させない**（CPU / メモリ / バッテリー）
2. **ユーザー操作に素早く応答する**

| 観点 | 目安・手段 |
|------|-----------|
| リソース消費 | Android Studio の Android Profiler / iOS の Xcode Instruments でプロファイル。リソース検証を自動ユニットテスト化し CI で継続計測 |
| アプリ起動時間 | アイコンタップ→表示まで **5 秒未満** を目標 |
| 操作応答時間 | アプリ内操作は **3 秒未満**（超えると離脱率が上がる）。サービス呼び出しは回線帯域の影響大 |
| ネットワーク条件下の応答 | エミュレーターの回線シミュレーションで各条件を計測 |
| UI ストレステスト | 連打・ズーム・高速画面遷移などでクラッシュ有無を確認 |
| 本番モニタリング | ベンダー提供の実ユーザー品質ダッシュボード（安定性・バッテリー・クラッシュ等）を継続監視 |

#### Monkey（Android のカオス的ストレスツール）

ランダムなタッチ・キー押下・ジェスチャー列を送り込み、クラッシュや無応答を報告する CLI ツール。

```bash
adb shell monkey -p "com.example.app" -v 2000   # 2000 イベントを送信
```

オプションで送信イベントの種類を絞り込める。

#### エミュレーターのネットワークスロットリング

Android Emulator の Extended Controls → Cellular で、回線種別（GSM/GPRS/EDGE/LTE）と信号強度（Great/Good/Moderate/Poor）、音声/データ状態を切り替えて性能テストに使う。

#### Appium の性能 API（Android 専用）

Android の `dumpsys` 診断を土台にした API で、UI テストと同居する自動性能テストを書ける。

```java
// perf_type: cpuinfo / memoryinfo / batteryinfo / networkinfo
// timeout: データ取得をポーリングする秒数
driver.getPerformanceData("package_name", "memoryinfo", 10);
// 対応 perf_type の一覧は getSupportedPerformanceDataTypes() で取得
```

ユーザーフローの要所で数値を取得し、閾値アサーション（例: 重い処理後のメモリ消費）を追加する。出力値の解釈は `dumpsys` のドキュメントを参照。

### アクセシビリティテスト

W3C WAI は WCAG のモバイル適用ガイダンスを提供しており、原則（知覚可能・操作可能・理解可能・堅牢）は Web と同じ。モバイル固有の確認項目:

- [ ] ズームイン/アウトできるか
- [ ] 小画面でも読めるか
- [ ] 要素間のコントラストは十分か
- [ ] ボタンのタップ領域は妥当な広さか
- [ ] レイアウトはアプリ全体で一貫しているか
- [ ] スクロールなしで必要な要素がビュー内に収まるか

| プラットフォーム | ツール | 用途 |
|-----------------|--------|------|
| iOS | Xcode Accessibility Inspector | シミュレーター上で監査（コントラスト不足・説明欠落等）と属性デバッグ |
| iOS | XCUIAutomation | アクセシビリティツリー経由で要素取得 → a11y アサーションを自動テスト化 |
| iOS | VoiceOver | 実機/シミュレーターでのスクリーンリーダー E2E |
| Android | Android Studio linter | a11y 問題を開発中に警告（シフトレフト） |
| Android | Espresso のアクセシビリティチェック | 各ビューを走査。既存 Espresso スイートに統合し CI 実行 |
| Android | TalkBack | 内蔵スクリーンリーダーでの E2E |
| Android | Accessibility Scanner | ストア配布のスキャナアプリ。対象アプリの操作を記録 → 画面ごとに改善候補をハイライト。手動テストフェーズや開発初期に有効 |
| Android | Switch Access / BrailleBack / Voice Access | 外部支援デバイス・点字ディスプレイ・音声操作の検証 |
| 両対応（商用） | Axe DevTools for Mobile | Appium テスト実行中に自動 a11y スキャン。全アプリ種別対応 |

Google Play はアプリ提出時にプレローンチの a11y 監査レポートも提供する。

### その他の CFR（横断的要件）

監査性・可搬性・信頼性・コンプライアンス・互換性等の CFR はモバイルでも有効。特に重要なモバイル固有 CFR:

| CFR | テスト観点 |
|-----|-----------|
| ユーザビリティ | 言語サポート・ハンズフリー等のパーソナライズが継続利用を左右。対象市場のユーザー行動を事前調査する |
| 信頼性（割り込み） | 着信・通知でアプリがバックグラウンド化→復帰するのが標準的な使われ方。①バックグラウンド化時の進行中リクエスト ②中断→復帰時の認証状態 ③強制終了時の進行中リクエスト ④ワークフロー中のバッテリー切れ、をアプリ全体で検証 |
| インストール性/アップグレード性 | ストレージ不足・権限拒否・OS 非互換などの**失敗系**を含めて検証。アンインストールも対象。アップグレード後に不要なログアウトが発生しないこと。**最新版からだけでなく旧バージョンからの更新**も試す。新権限の要求、各ネットワーク条件下での挙動も含める |
| モニタリング | モバイルのクラッシュは再現困難なものが多い。クラッシュレポートツール（Firebase Crashlytics / Dynatrace / New Relic 等）を**開発中からテスト環境に組み込み**、デバッグを容易にする |

インストール/アップグレード成功や割り込み時挙動の一部は、マイクロ/マクロ機能テストとして自動化し継続フィードバックを得られる。

## AI 支援ワークフロー

### AI 支援探索的テスト（Appium MCP）

Appium MCP サーバーを使うと、LLM が Appium 経由で Android/iOS アプリを操作しながら探索できる。構成の要点:

1. `capabilities.json` に接続情報を定義する

```json
{
  "android": {
    "appium:platformName": "Android",
    "appium:app": "/path/to/app.apk",
    "appium:deviceName": "Android Device",
    "appium:platformVersion": "8.0",
    "appium:automationName": "UiAutomator2",
    "appium:udid": "emulator-5554",
    "appium:noReset": false
  }
}
```

2. MCP クライアント設定に Appium MCP を登録する（`ANDROID_HOME` と `CAPABILITIES_CONFIG` を環境変数で渡す）

```json
"appium-mcp": {
  "type": "stdio",
  "command": "npx",
  "args": ["appium-mcp@latest"],
  "env": {
    "ANDROID_HOME": "/path/to/Android/sdk",
    "CAPABILITIES_CONFIG": "/path/to/capabilities.json"
  }
}
```

3. エージェントへの指示は段階化する: ①チケット管理ツールから受入基準を取得 ②エピック説明を取得 ③**まずアプリを起動**し、対象機能 1 つに限定しつつ受入基準に縛られず探索 ④社内の探索的テストガイドラインを参照 ⑤Given/When/Then 形式で機能テストケースを生成 ⑥エピック記載の非機能カテゴリ別にもテストケースを生成 ⑦機能/CFR 別にグルーピング

期待できる出力: 機能・CFR・デバイス差・メモリリーク・キャッシュ・a11y・UI ストレス（連打）・ジェスチャー・境界条件（長大リスト）・割り込み（バックグラウンド化/強制終了）を含む数十件規模のテストケース群。**プロジェクト固有知識で補強して使う「スターターキット」**と位置づける。

> **警告（trust but verify）**: LLM は指示に完全には従わない。実際に画面遷移できなかった受入基準を「類似フローと同じはず」と推測して Verified と申告する事例がある。実行ログと画面を突き合わせて検証すること。

### エージェントによるテストフレームワーク構築・テスト作成

コーディングエージェント（例: Claude Code）に、Appium の依存関係インストール（Appium server / Java client / UiAutomator2 Driver / TestNG 等の煩雑なセットアップ）と Page Object Model ベースのフレームワーク構築、Gherkin シナリオからのテストコード生成を委任できる。

エージェント定義に含めるべき指示:

1. Appium MCP ツールで対象アプリを探索し UI 構造を把握する
2. 必要な要素・画面・ナビゲーションパスを特定する
3. Page Object Model で構造化した Java テストコードを生成する
4. テストが通るまで依存関係を含めて修正する
5. 既存テストにリグレッションがないことを確認する

> **警告**: エージェントは動くテストを作っても、セットアップ中に作った不要フォルダの掃除・README への構造変更反映を怠り、指示外のテストを勝手に追加することがある。**テストが通る＝品質保証ではない**。生成コードは必ずレビューする。プロジェクト固有の do/don't をエージェント指示に蓄積して育てる。

CI への組み込みは、実機を CI マシンに直結するか、クラウドのデバイステストプラットフォーム（BrowserStack 等）を利用する。

## Android Studio Database Inspector（ローカル DB の検証）

アプリのローカル DB を GUI で確認・編集できるツール。オフライン用データの検証に使う。

手順:

1. Run → Debug でデバッグ有効の APK を選択
2. View → Tools Window → App Inspection でパネルを開く
3. エミュレーターでアプリを実行すると App Inspection 内に Database Inspector が表示される
4. 一般の DB クライアント同様にデータの追加・編集・削除が可能

検証ポイント:

- [ ] オフラインアクセスに必要なデータが期待通り保存されているか
- [ ] **保存される機密情報が暗号化されているか**

## モバイルテストピラミッド

サービス層・DB 層は従来のテストピラミッド（マイクロテスト厚め・マクロテスト薄め）に従う。**モバイル UI 層はピラミッドが逆転しうる**。

逆転を生む 2 要因:

1. **ユニット/UI テストのスコープ制約**: OS の API に依存する機能は「OS 提供者がテスト済み」としてユニットテストを書かないことが多く、その分をデバイス横断の E2E（手動/自動）で担保する必要がある。カメラ・センサー等のハードウェア統合や、スクロールの滑らかさ・高度なジェスチャー等のユーザビリティは**手動でしか十分に検証できない**
2. **アプリの特性**: 機能ロジックが厚く外部依存が少ないアプリ → 従来型ピラミッド。機能ロジックが薄く外部依存が重いアプリ → 逆ピラミッド

判断基準: 自分のアプリがどちらの形になるかを**早期に見極め**、逆ピラミッド型なら自動 UI テストと網羅的な手動回帰のバランスを取るためのテスト工数を計画に織り込む。

## 要点まとめ

- モバイルテストは Web テストと異なる。デバイス×アプリ種別×ネットワークの多様性が開発全工程に影響する
- 戦略の土台は全レイヤー横断のテスト原則（マイクロ/マクロ重視・シフトレフト・セキュリティ/性能/a11y の品質次元）と同じで、その上にモバイル UI 固有のテストを積む
- デバイスは対象顧客の約 85% カバーを目安に 3〜4 台へ絞り、プロジェクト開始時に確定する
- Appium は全アプリ種別・複数 OS を単一 API で自動化でき、ビジュアル（images plugin）・性能（getPerformanceData）・a11y（商用プラグイン）まで拡張できるハブになる
- エミュレーターは開発中の高速チェック用。リリース判定は実機で行う
- AI 支援（MCP 経由の探索・エージェントによるテスト生成）は強力なスターターキットだが、「trust but verify」を徹底する
- テストピラミッドの形はアプリ特性次第で逆転する。早期に形を見極めてテスト工数を計画する
