# 継続的テスト（Continuous Testing）

継続的テスト（CT）は、手動・自動の両方のテスト手法を用いて、あらゆるインクリメンタルな変更のたびにアプリケーション品質を検証し、意図した品質目標からの逸脱を即座にチームへ通知するプロセスである。自動テストが特定担当者のローカルマシンにのみ存在し、リリース時にしか実行されない状態はアンチパターンであり、フィードバックサイクルを著しく遅延させる。CT は継続的インテグレーション（CI）に依存し、CI + CT が継続的デリバリー（CD）を可能にする。本リファレンスでは CI/CT/CD の全体像、CI の原則とエチケット、テスト振り分け戦略、CI パイプラインへの AI エージェント統合、DORA メトリクスとの関係を扱う。

---

## 用語の整理

| 用語 | 定義 |
|------|------|
| 継続的インテグレーション（CI） | チーム全員が少なくとも1日1回、自分の変更を共有コードベースへマージするプラクティス。build-and-test ステージの通過をもって「統合完了」とみなす |
| 継続的テスト（CT） | すべてのコミットに対し、機能・非機能（CFR）両面のアプリケーション品質を自動＋手動（探索的テスト含む）で検証するプロセス。探索的テストで見つけたシナリオの自動化までを「done」の条件に含める |
| 継続的デリバリー（CD） | CI/CT により常時プロダクションレディな状態を維持し、ワンクリックの自動デプロイで任意の環境（QA/UAT/本番）へいつでもリリースできる状態 |
| 継続的デプロイメント | CT 通過後の全コミットを自動で本番へプッシュする方式。CD（セルフサービスで「いつでも出せる」）とは区別する |

### 継続的デプロイメント vs 継続的デリバリーの選択

| 条件 | 選択 |
|------|------|
| デプロイ後に手動テストが必要 / 機能のローンチ日が事前に決まっている / 公開アナウンスと同期させたい | 継続的デリバリー（セルフサービスデプロイ） |
| 全コミットを即時に本番反映してよい | 継続的デプロイメント |

---

## なぜ CI が必要か（統合遅延のコスト）

- 2人の開発者が独立に「ログイン画面」「ホーム画面」を作り、各自ローカルでのみテストして「完了」と報告しても、**統合されていなければ実際には使えない**。誰が統合するのか・統合ストーリーを別に切るのか・統合まで検証を遅らせるのか、といった問題が残る。
- 統合を後回しにするほど、絡み合った不具合の発見と修正が困難になり、大規模な書き直しに至るリスクが増す。開発3週間の機能に統合テスト5週間を要した事例もある。
- CI は統合欠陥を完全には無くさないが、**芽のうちに発見・修正できる**ようにし、場当たり的な書き直しやパッチからチームを守る。

## CI/CT/CD プロセスの全体像

### 4つの構成要素

| 構成要素 | 役割 |
|---------|------|
| バージョン管理システム（VCS） | コードベース全体を保持する中央リポジトリ。全員が最新版を pull し、継続的に統合する場（Git, SVN 等） |
| 自動テスト（機能 + 非機能） | マイクロレベル（unit 等）・マクロレベル（E2E 等）に加え、セキュリティ・パフォーマンス等の品質特性を検証するテスト群 |
| CI サーバー | 変更のたびにアプリケーションをビルドし自動テストを実行（Jenkins, GoCD, GitHub Actions 等） |
| インフラ | CI サーバーとアプリケーションをホストする基盤（実運用ではクラウドまたは共有 VM 上に置き、全員がアクセスできるようにする） |

### 基本ワークフロー

1. 開発者が小さな機能単位を完成させるたびに VCS へ push する（ローカルでは小さなコミットを積み、論理的なまとまりに達したら push。push して初めて CI/CT が始まる）。
2. push が CI サーバーのパイプラインをトリガーし、build-and-test ステージでビルド＋自動テストを実行する。
3. 全テスト通過 → そのコミットは「統合済み」。失敗 → コード所有者が即修正する。修正に時間がかかる場合は revert し、他メンバーが壊れたコードの上に作業を積まないようにする。
4. 後続のコミットは、それ以前の全変更を含むスナップショットとしてテストされるため、**新しい変更が既存機能（他人のテストを含む）を壊していないこと**が毎回保証される。

### CD への拡張

- CD は CI/CT に「セルフサービスデプロイパイプライン」を加えたもの。CI サーバーの後続ステージとして構成する。
- CI サーバーは全コミットとテスト結果を一覧化し、**全テストが通過したバージョンのみ**デプロイ可能として提示する。特定コミット時点のバージョンを選んで UAT 等へデプロイでき、テスト失敗中のコミットはデプロイ候補に載らない。
- 効果: 市場投入タイミングを逃さない／デプロイの属人性を排除（特定個人の「魔法」に依存しない）／ライブラリ不整合・設定漏れ・ドキュメント不足のリスクを削減。

### VCS がもたらす前提能力

- 全変更の履歴保持 → 根本原因分析が容易
- 動作していたバージョンへのロールバック
- 変更とユーザーストーリー／欠陥チケットの紐付け（トレーサビリティ）
- ブランチによる並行作業（ただし**長命なフィーチャーブランチはアンチパターン**）

## CI の原則とエチケット

チームがデリバリーサイクルの開始時に合意し、継続的に強化すべき最低限のルール。

| 原則 | 内容 |
|------|------|
| 頻繁にコミットする | 小さな機能単位を仕上げるたびに VCS へ push し、テスト済みコードを他メンバーが積み上げられる状態にする |
| self-tested code をコミットする | 新しいコードは**同一コミット内に自動テストを伴う**。これにより後続コミットが自分の機能を壊したとき即検知できる |
| 壊れたビルドは10分以内に直す | 自分のコミットが CT を通過するまで次のタスクへ進まない。build-and-test ステージが壊れたら約10分以内に修復、無理なら revert してグリーンな状態を保つ |
| 失敗テストを無効化しない | ビルドを通すために失敗テストをコメントアウト・ignore しない（ありがちだが厳禁） |
| 壊れたビルドに push しない | レッドな状態の上に積むと失敗が連鎖し、どの変更が原因かの切り分けコストが膨らむ |
| すべての失敗を自分事にする | 自分の変更が「自分が触っていない領域」のテストを壊した場合も修復責任は自分にある。必要なら有識者とペアで直し、修正完了まで次タスクへ進まない。責任の押し付け合いで修正が遅れ、テストが CI から外されたまま数日過ぎると、その間の CT フィードバックは不完全・虚偽になる |

### チームで採用されるより厳格なプラクティス（任意）

- push 前にローカルで全マイクロ・マクロレベルテストの通過を必須化
- コードカバレッジ閾値未達で build-and-test を fail させる
- コミットの成否をコミッター名付きでチャットへ通知する
- ビルド破壊時にチームエリアの CI モニターから通知音を鳴らす

> 運用の要諦: ルールの「how」だけでなく「why」をチームに浸透させること。テスター（QA）は CI パイプラインのテスト状況を常時ウォッチし、期限内に修正される状態を維持する。

## 継続的テスト戦略（ステージ分割とテスト振り分け）

単一の build-and-test ステージに全テストを詰め込む構成から始め、プロジェクトの成長に合わせてフィードバックループを分割する。

### フィードバックループの段階的拡張

| 段階 | 構成 | 狙い |
|------|------|------|
| 1ループ | build-and-test（全テスト） | アプリが若いうちは全テストを毎コミット実行できる。まずはこれで十分 |
| 2ループ | ① build-and-test（ビルド＋マイクロレベルテスト） → ② deploy → acceptance（マクロレベルテスト） | 高速なマイクロレベルテストを先行させて素早い一次フィードバックを得る（シフトレフト）。テストピラミッドに従っていればマイクロレベルで広範な機能を検証できる |
| 3ループ | ② の後に ③ CFR ステージ（性能・セキュリティ・アクセシビリティ等の自動テスト） | 品質特性まで含めた全体像のフィードバックを継続的に得る |
| 4ループ | ③ の後に ④ smoke test / nightly regression 分割 | テスト総量の増大に対する実行時間の最適化（後述） |

### 実行時間の目安と高速化

| 対象 | 目安 | 超過時の対策 |
|------|------|-------------|
| build-and-test ステージ | 数分以内（席を立って戻れる程度）。完了を待ってから次タスクへ移れる長さに保つ | コンポーネント単位にステージを並列化する |
| コミット〜セルフサービスデプロイ可能まで（2ループ合計） | 1時間未満 | テスト実行の並列化・テストピラミッドの実装・重複テストの排除・wait の除去や共通処理の抽象化などのテストリファクタリング |

- 実例の規模感: マクロレベル約200本で8時間かかっていたフィードバックが、テストピラミッド準拠に再構成後はマイクロ＋マクロ約470本で約35分に短縮。
- **フィードバックが数時間かかると、失敗を「欠陥チケット化して後回し」にする誘惑が生まれる**。これは欠陥の上に新コードを統合することを意味し、放置された失敗テストの分だけ新コードも未検証になる。ループの短さ自体が品質規律を支える。

### smoke test / nightly regression 分割

チェーンしたパイプライン全体の実行時間・リソースが問題になった段階で導入する。

| ステージ | 内容 | 実行タイミング |
|---------|------|---------------|
| smoke test | 各機能のエンドツーエンドフローをカバーする代表テストだけを抜粋したパック。コミットごとの高レベルなシグナルを素早く得る | acceptance ステージとして毎コミット |
| nightly regression | 全テストスイートをその日の全コミット込みの最新コードベースへ実行 | 毎日1回、チームの業務時間外（例: 19時） |

運用ルール:

- smoke を導入したら nightly regression は**必須**（省略すると回帰の検知が消える）。
- 翌朝一番に nightly の結果を分析し、欠陥・環境起因の失敗の修正を最優先する。テストスクリプト修正が必要ならそれをその日のチーム最優先タスクにする（放置すると以降のコミットへの CT フィードバックが誤りになる）。
- 振り分け対象は**マクロレベルと CFR テストのみ**。マイクロレベルテストは常に build-and-test で毎コミット実行する。
- CFR も同じ考え方で分割できる: 例）クリティカルな1エンドポイントの負荷テストは毎コミット・残りは nightly、静的セキュリティスキャンは build-and-test・機能的セキュリティスキャンは nightly。
- **落とし穴**: この方式はフィードバック（と修正）を最大1日遅らせる。smoke / nightly の振り分けは慎重に選ぶこと。

### 採用順序の指針

1. アプリが若いうちは全テストを毎コミット実行
2. 成長に伴い CI 実行時間の最適化（並列化・ピラミッド・重複排除）
3. それでも足りなくなったら smoke / nightly へ段階的に移行

### CT のメリット

| メリット | 内容 |
|---------|------|
| 共通の品質目標 | 機能・非機能の品質目標に対して全員の作業が継続的に評価される。「品質の作り込み」の具体的手段 |
| 早期欠陥検出 | コンテキストが新鮮なうちに修正できる（数日〜数週間後に戻るコストを回避） |
| 常時デリバリー可能 | コードが継続的にテストされているため、いつでも任意の環境へデプロイできる |
| コラボレーション強化 | どのコミットがどの問題を起こしたか追跡でき、特に分散チームで非難や軋轢を防ぐ |
| デリバリーの共同所有 | テストチームやシニア開発者だけでなく、全員が自分のコミットのデプロイ可能性に責任を持つ |

## CI サーバー構築の要点

CI サーバーのセットアップは DevOps 専任者がいる場合でも、開発者・テスター双方がプロセスを理解し、**CT 戦略に沿ってテストステージを正しくチェーンできる**ことが重要。

### パイプライン設定のチェックリスト（Jenkins 等の汎用形）

- [ ] ビルドツールのパス（`JAVA_HOME`, `MAVEN_HOME` 等）をグローバル設定
- [ ] パイプラインにリポジトリ URL を設定（clone 用）
- [ ] ビルドトリガーを選択（下表）
- [ ] ビルドステップにテスト実行コマンドを設定（例: Maven の `test` ゴール = `mvn test`）
- [ ] post-build で後続パイプライン（CFR テスト等）をチェーンし CD パイプラインを構成
- [ ] ワークスペース（clone 先とテストレポート）とコンソール出力をデバッグ導線として把握

### トリガー方式の比較

| 方式 | 動作 | 用途 |
|------|------|------|
| SCM ポーリング | 一定間隔（例: 2分毎）でリポジトリの新規変更を確認し、あれば実行 | 簡易セットアップ |
| Webhook（push フック） | VCS ホスティング側から push 時に CI へ通知 | 本番運用の標準 |
| 定期実行 | 変更の有無にかかわらず固定スケジュールで実行 | nightly regression |

### 忘れがちなトリガー対象

パイプラインは**アプリケーションコードの変更だけでなく、設定・インフラ・テストコードの変更でも**起動させること。

### Git ワークフローの要点（CI 前提の使い方）

| ステージ | 操作 | CI 観点での意味 |
|---------|------|----------------|
| working directory | 編集 | — |
| staging area | `git add` | 小さな作業単位ごとに積み、レビュー・再利用可能にする |
| local repository | `git commit -m "..."` | 論理単位のスナップショット。失敗時に1かたまりで revert できる |
| remote repository | `git push` | **push して初めて CI/CT が動く**。チーム全員に変更が届く |

- コミットはユーザー名・メールに紐付け（`git config --global user.name / user.email`）、変更のトレーサビリティを確保する。
- チームの既存リポジトリに参加する場合は `git clone`、他メンバーの変更取り込みは `git pull`。

## CI での AI エージェント活用

CI/CD と AI エージェントはどちらも「ワークフロー自動化」を目的とするため親和性が高い。CI/CD がコミット〜本番への伝播を自動化してきた一方、機能開発・品質チェック・テスト作成・CI 失敗分析は人間の作業として残っていた。AI エージェントを CI/CD ワークフローに組み込むと、これらをある程度自律的に肩代わりさせられる。

### ユースケース: OpenAPI 仕様からの API テスト自動生成

API 仕様（OpenAPI spec）がリポジトリへ push された時点で、コーディングエージェントがサービステストを生成し PR を起票する。仕様が先・テストが先というテスト駆動の流れを CI 上で実現できる。

セットアップ（GitHub Actions + コーディングエージェントの例）:

1. エージェントの GitHub App をリポジトリへインストール（リポジトリアクセス権限の管理）
2. API キーをリポジトリ Secrets へ登録（Settings → Secrets & Variables → Actions）
3. Workflow permissions で「Read and Write permissions」と「PR の作成・承認許可」を有効化

ワークフロー定義の骨子（一般化した例）:

```yaml
name: Generate API tests from OpenAPI
on:
  push:
    branches: [main]
    paths: ["api-specs/**"]   # spec の push で自動起動
  workflow_dispatch:           # 手動起動も可能に

jobs:
  generate-tests:
    runs-on: ubuntu-latest
    permissions:
      contents: write
      pull-requests: write
    steps:
      - uses: actions/checkout@v5
      - uses: actions/setup-java@v4
        with: { distribution: temurin, java-version: "17" }
      - name: Generate API tests
        uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          allowed_tools: "Read,Write,Bash,View,GlobTool,GrepTool,BatchTool"
          max_turns: 30
          prompt: |
            api-specs/ 配下の OpenAPI 仕様から REST Assured テストを生成し、
            src/test/java/ に Java テストクラスを作成すること。
      - name: Create Pull Request
        if: success()
        uses: peter-evans/create-pull-request@v6
        with:
          token: ${{ secrets.GITHUB_TOKEN }}
          commit-message: "test: generate API tests from OpenAPI specs"
          branch: automated-tests/run-${{ github.run_number }}
          title: "Generated API tests from OpenAPI specifications"
```

運用ポイント:

- 生成された PR は必ず人間がレビューする。レビュー観点例: 全エンドポイントのカバレッジ／アサーションの妥当性／テストデータ／認証・認可テスト／エラーハンドリング／テストの独立実行可能性。
- レビューで修正が必要なら、PR コメントでエージェントをメンション（例: `@claude`）して修正を依頼できる（対応する workflow ファイルの追加が必要）。
- AI モデル利用料と CI 実行時間（Actions minutes）のコストが発生することを織り込む。

### プロンプトを本番品質にするための強化項目

- 最新コミットで追加された spec ファイルのみを対象にする等のスコープ制御
- プロジェクト固有のテストデータの利用指示
- 生成すべきテストケース種別のガイドライン
- **テスト実行が通った場合のみ PR を起票する**条件付け

### その他の CI × AI 応用パターン

| 応用 | 内容 |
|------|------|
| 自動コードレビュー | PR に対するエージェントレビューをパイプラインに組み込む |
| CI 失敗分析・自動修復 | 失敗ログをエージェントに解析させ、原因特定や修正 PR まで自動化 |
| テスト失敗のリトライ判断 | フレーク判定と再実行の自動化 |
| リリースドキュメント生成 | 変更履歴からリリースノートを自動作成 |
| プラットフォーム統合 | 商用 CI/CD プラットフォームは、インフラコード生成・パイプライン構築・テスト実行最適化・AppSec スキャン・SRE 業務まで AI エージェント化する方向に拡張している |

## DORA メトリクスとテスト

DORA（DevOps Research and Assessment）の研究に基づくソフトウェアデリバリーパフォーマンス指標。チームのパフォーマンスを high/medium/low（最上位は elite）に分類する。中核は次の4指標で、デリバリーの「速度」と「安定性」を対で測る。

| 指標 | 定義 | 測るもの |
|------|------|---------|
| リードタイム | コミットから本番デプロイ可能になるまでの時間 | 速度 |
| デプロイ頻度 | 本番（またはアプリストア）へのデプロイ頻度 | 速度 |
| 障害復旧時間（failed deployment recovery time / MTTR） | サービス停止や失敗からの復旧時間 | 速度・反応性 |
| 変更失敗率（change fail rate） | 本番リリース後にロールバック・ホットフィックス等の是正を要した変更の割合 | 安定性 |

これに加え、**デプロイ手戻り率**（本番インシデント起因で発生した計画外デプロイの割合）を安定性指標として併用する運用もある。速度の指標を追うあまり安定性の指標を犠牲にしない、というバランスが本質。

### 高パフォーマンス（elite）チームの目安

- オンデマンドで1日複数回デプロイ
- コミットから1時間以内に本番へ変更を届けられる
- 本番障害から1時間以内に復旧
- デプロイ後の本番障害率 15% 未満

### CT が各メトリクスをどう改善するか

| メトリクス | CT の寄与 |
|-----------|----------|
| リードタイム / デプロイ頻度 | CI/CT/CD により常時デプロイ可能状態が維持され、オンデマンドでリリースできる |
| リードタイム（1時間以内） | 自動テストを適切なレイヤーに配置（テストピラミッド）すれば、コミットから1時間以内にデプロイ可能になる |
| 変更失敗率（<15%） | 機能・非機能テストを自動化し CT プロセスで毎コミット実行していれば達成は難しくない |
| 生産性の増幅 | 高速なフィードバックループを持つ組織が AI を開発・テストへ導入すると、20〜30% の生産性向上が観測されている（速いループが AI 活用効果の前提条件になる） |

## チェックリスト（要点まとめ）

- [ ] 自動テストは全員のコミットに対し CI 上で継続実行されているか（ローカル止まりになっていないか）
- [ ] CT は機能テストに加え非機能（性能・セキュリティ・アクセシビリティ）も毎変更で検証しているか
- [ ] CI の原則とエチケット（頻繁なコミット・self-tested code・10分修復 or revert・失敗の無効化禁止・壊れたビルドへの push 禁止・全失敗の当事者意識）がチーム合意されているか
- [ ] build-and-test は数分以内、コミット〜デプロイ可能まで1時間未満に収まっているか
- [ ] smoke / nightly 分割はマクロ・CFR テストに限定し、nightly の結果分析を翌朝の最優先にしているか
- [ ] パイプラインはアプリコード以外（設定・インフラ・テストコード）の変更でも起動するか
- [ ] AI エージェント生成物（テスト・修正 PR）に人間のレビューゲートがあるか
- [ ] DORA メトリクス（速度3 + 安定性）を計測し、速度と安定性をトレードオフにしていないか
