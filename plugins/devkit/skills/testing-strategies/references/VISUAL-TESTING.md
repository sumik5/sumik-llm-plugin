# ビジュアルテスト

ビジュアルテストは、アプリケーションの見た目（要素のサイズ・色・相対位置・レイアウト等の視覚属性）が、ブラウザ・デバイス横断で設計どおりに表示されることを検証する活動である。UI 駆動の機能テストはロケーターで要素の存在と動作を確認するだけで「見た目」は検証しないため、ボタンのテキストが欠けて表示されていても合格してしまう。視覚品質は顧客の第一印象と信頼に直結するビジネス価値であり、手動目視・機能テストの隙間を自動ビジュアルテストで補完する戦略が必要になる。本リファレンスでは、ビジュアルテストの位置づけと導入判断、フロントエンドテスト戦略における責務分担、Playwright / Applitools Eyes / Storybook / BrowserStack Percy を用いた実装パターン、そしてフレークとベースライン管理という代表的課題への対策をまとめる。

---

## ビジュアルテストの位置づけ

### 手動目視・機能テストだけでは不十分な理由

| アプローチ | 限界 |
|-----------|------|
| 手動目視 | ピクセル単位の変化（ボタンの角丸消失、ロゴの数 px ずれ等）を検知できない。人間は画像の広い領域の変化でも見落とすことがある（change blindness）。ブラウザ×デバイス×解像度の組み合わせを手動で網羅するのは非現実的 |
| UI 駆動の機能テスト | ロケーター（ID/XPath 等）で要素の存在と機能を確認するだけで、look and feel は検証しない。テキストが切れたボタンでもクリックできれば合格する。全ページ全要素の存在アサーションを追加するのは実行速度・保守コストの面で非現実的 |

- UX デザインそのものの妥当性検証はユーザビリティテストの領域であり、ビジュアルテストとは区別する。

### 自動ビジュアルテストの主な手法

| 手法 | 内容 |
|------|------|
| CSS アサーション | コードで要素の CSS 属性を検証（例: `border-width: 10px` の確認） |
| 静的 CSS 解析 | CSS コードを解析してブラウザと UI 要素の非互換を検出 |
| AI ベースの変化認識 | computer vision と深層学習でページ上の変化を認識 |
| スクリーンショット比較 | ページのスクリーンショットを承認済みベース画像とピクセル比較（screenshot testing）。今日のビジュアルリグレッションで最も一般的 |

- スクリーンショット比較の代表的な OSS: Playwright、BackstopJS。AI 搭載の商用ツール: Applitools Eyes、Functionize 等。
- 運用イメージ: まず手動でデザインカンプと突き合わせて承認 → 以降は自動ビジュアルテストが反復開発の中で継続的フィードバックを返す（機能バグに対する自動機能テストと同じ構図）。

## 導入判断: ビジネスクリティカルな用途

自動ビジュアルテストは全アプリ必須ではない。作成・保守コストがかかるため、費用対効果で判断する（社内管理者数名しか使わない内部アプリなら手動目視で十分なことが多い）。

### 投資を正当化しやすいユースケース

| ユースケース | 理由 |
|-------------|------|
| B2C アプリケーション | 視覚品質が顧客獲得・ブランド価値に直結。開発中の継続的フィードバックが必要（市場検証用の使い捨てプロトタイプは例外） |
| 多デバイス・多ブラウザ対応 | ブラウザ×デバイス×解像度の組み合わせ回帰テストの負荷を自動化で大幅削減。実利用統計ではモバイルがデスクトップを上回り、Chrome/Safari が主要シェアを占める |
| デザインシステムを持つアプリ群 | 中央チームが開発した共有 UI コンポーネントの欠陥はスイート全体へ波及するため、コンポーネントレベルのビジュアルテストが必要 |
| アプリケーションのリビルド | UX を変えずにスケーラビリティ等を改善する再構築や、UI コンポーネントの再編成を伴う大規模リファクタリングでのセーフティネット |
| 多ロケール展開 | 言語・地域別の look and feel 変更がレイアウトへ波及するため、多バージョンの検証を自動化する価値が高い |

### 判断基準チェックリスト

- [ ] 視覚不具合の顧客影響はどの程度か（B2C か内部ツールか）
- [ ] 開発作業の種類（新規/リビルド/リファクタ/ローカライズ）は視覚回帰リスクを伴うか
- [ ] 手動での視覚確認に要する工数はどの程度か
- [ ] チームは変更に対する自信（回帰への安全網）を必要としているか
- 導入する場合も、ビジュアルテストは**最重要フローに限定**し、残りのカバレッジは他のフロントエンドテストに委ねてバランスを取る。

## フロントエンドテスト戦略の全体像

フロントエンドは HTML（構造）・CSS（スタイル）・スクリプト（振る舞い）＋描画するブラウザで構成される。ミクロ（開発時の高速フィードバック）からマクロ（デプロイ後の統合検証）までのテストを組み合わせ、shift-left で早期にフィードバックを得る。一般則として**ミクロレベルのテストを多く、マクロレベル（visual / functional E2E）のテストを少なく**保つ。

### テストタイプ別の責務分担

| テストタイプ | 検証対象 | ビジュアル品質への寄与 | 実施タイミング / 担い手 | 代表ツール |
|-------------|---------|----------------------|------------------------|-----------|
| Unit | 個々のコンポーネントの状態別振る舞い（見出しのテキスト、ボタンの enabled/disabled 等） | 部分的（要素タグや状態のアサーション） | 開発開始時・開発コードベース内 | Jest、React Testing Library |
| Integration / Component | フォーム全体などコンポーネント間連携。サービス呼び出しはモック、状態変化をシミュレート | 部分的（ログイン後の要素消失・disabled 状態等） | コンポーネント開発完了時 | Jest、React Testing Library |
| Snapshot | コンポーネントの DOM 構造をテストレンダラーで描画し、保存済みスナップショット（HTML 断片）と比較 | ミクロレベルで直接寄与 | 開発直後・開発コードベース内 | Jest、Playwright |
| Functional E2E | 実ブラウザでエンドツーエンドのユーザーフローと FE/BE 統合。完全デプロイ＋テストデータが前提 | 部分的（ロケーターベースの存在確認のみ） | デプロイ後・CI | Playwright、Cypress |
| Visual | 全体レンダリング後のページのスクリーンショットを承認済みベース画像と比較 | 主担当（heavy lifting） | デプロイ後・CI（機能スイートへ統合可） | Playwright、BackstopJS、Applitools Eyes、Percy |
| Cross-browser | ブラウザ横断の機能＋視覚品質＋レスポンシブ | 視覚面はビジュアルテストを再利用 | 選定ブラウザでの CI 実行＋バグバッシュ | 上記＋クラウドグリッド |
| Frontend performance | ブラウザのレンダリング遅延。ページロード時間の大半はフロントエンド資材のダウンロードが占める | 間接（視覚品質と性能のバランス） | 開発中〜リリース前 | 性能計測ツール群 |
| Accessibility | WCAG 準拠（一貫レイアウト・可読テキスト・十分なクリック領域等）。多くの国で法的要請 | 視覚品質を底上げ | コンポーネントレベルから追加するのが良い | a11y 検査ツール群 |

### Visual vs Snapshot の使い分け

| 観点 | Snapshot テスト | Visual テスト |
|------|----------------|--------------|
| 比較対象 | HTML/DOM 構造（コード断片） | レンダリング済み画面のピクセル |
| 前提 | テストレンダラーのみ（アプリの完全起動不要） | ブラウザで完全に描画されたアプリ |
| フィードバック | 開発フェーズで高速（shift-left） | デプロイ後・低速 |
| 適用範囲 | 小さな単一コンポーネント、変更頻度の低いコンポーネント | 複数コンポーネントを統合した大きなビュー（ページ全体） |
| 落とし穴 | 対象を大きくするとレイアウト微修正のたびに保守コスト増 | フレーク・ベースライン管理コスト（後述） |

### クロスブラウザテストの要点

- **80/20 ルール**: 利用者の 80% をカバーするブラウザ・解像度に注力し、残り 20% はリリース終盤のバグバッシュで優先度に応じて確認する。
- 機能面: 最重要フローだけを選び、選定ブラウザで UI 駆動機能テストを実行（ブラウザ差異による機能・セキュリティ問題は実例がある）。
- 視覚面: ビジュアルテストを再利用してクロスブラウザ互換とレスポンシブを同時に検証。機能テストとビジュアルテストは同一テストに統合できる。
- 非クリティカルフローは React / Vue.js / Bootstrap / Tailwind 等のフレームワーク組み込みのクロスブラウザ対応に委ねる（ただし古いブラウザは一部機能非対応）。

**shift-left の実践順序:**

1. 標準準拠ブラウザをサポートする開発ライブラリ（React、Vue.js 等）を使う
2. 開発中に CanIUse のサポート表と lint プラグイン（`stylelint-no-unsupported-browser-features`、`eslint-plugin-caniuse`）で対象ブラウザ非対応の CSS/JS 機能を検出する。JS の後方互換には Babel 等のトランスパイラを使う
3. 対象利用の 80% をカバーするブラウザ・デバイスで少数の機能＋ビジュアルテストを CI 実行する
4. 残り 20% は定期的なバグバッシュで補う

## Playwright によるビジュアルリグレッション

Playwright は `toHaveScreenshot()` でネイティブにビジュアルテストをサポートする。内部では pixelmatch ライブラリが参照スクリーンショットと実行時スクリーンショットをピクセル比較し、差分をハイライトする。

### 使いどころ

- ユーザーフローの各ページ到達直後と、状態変化後（カート数の増加等）に `toHaveScreenshot()` を呼ぶのが定石。
- E2E ユーザーフローを覆う標準の機能スモークテストに相乗りさせ、CI でブラウザ・デバイス構成横断の視覚比較を回す。
- エラーフローの見た目はビジュアルテストではなく、ミクロレベルのフロントエンドテストで検証する方がよい。

### 設定

```ts
// playwright.config.ts
expect: {
  timeout: 10000,
  toHaveScreenshot: {
    maxDiffPixelRatio: 0.1,
    pathTemplate: '{testDir}/__screenshots__{/projectName}/{testFilePath}/{arg}{ext}',
  },
},
```

| パラメータ | 役割 | 指針 |
|-----------|------|------|
| `maxDiffPixelRatio` | 全ピクセルに対する差分ピクセルの許容比率（0〜1） | デバイスのバッテリー状態や OS バージョン差で微小なピクセル差が出るため、多様なデバイスで走らせるなら 0.2 未満の小さめの値。単一環境ならアプリの感度要件で決める |
| `pathTemplate` | ベーススクリーンショットの保存パス | プロジェクト（ブラウザ/デバイス）別にサブフォルダが切られる形にする |

### ワークフロー

```ts
// Page Object 内の例
async goto() {
  await this.page.goto('/');
  await this.page.locator('#search-box').waitFor({ state: 'visible' });
  await expect(this.page).toHaveScreenshot('HomePage.png');
}
```

1. **初回実行**: ベースが存在しないため `A snapshot doesn't exist at {location}` で失敗し、構成済みプロジェクトごとに `__screenshots__` フォルダへベース画像が生成される。
2. **2 回目以降**: ベースと比較し、差分があれば差分ハイライト画像を出力して失敗する。
3. Playwright はアサーション前に**連続スクリーンショットを取得して安定するまで待つ**ため、失敗は単純なフレークではなく、カルーセル等の動的コンテンツ起因である可能性が高い。

### 動的コンテンツの扱い（いずれか、または併用）

| 対策 | 方法 | 備考 |
|------|------|------|
| データ固定 | 静的テストデータを使う、または該当 API 呼び出しをスタブする | テスト環境の設計として計画する |
| CSS で非表示 | テスト実行時にページへ上書き適用する CSS ファイルを用意し、`stylePath` で渡す | 非表示化後にベース画像を再取得すること |
| マスク | `mask: [locator]` オプションで該当要素に塗りつぶしを重ねる | ベース側にも同じマスクが乗るため差分にならない |

```ts
// CSS 上書きの例
await expect(this.page).toHaveScreenshot('HomePage.png', {
  stylePath: path.join(__dirname, 'HomePage.css'),  // .carousel { display: none; }
});

// マスクの例
await expect(this.page).toHaveScreenshot('HomePage.png', {
  mask: [this.page.locator('.carousel')],
});
```

その他のケースは公式ドキュメント（https://playwright.dev/docs/test-snapshots）を参照。

### 補足

- `toMatchAriaSnapshot()` はページ/コンポーネントのアクセシビリティツリーを期待値と比較するスナップショット機能で、視覚・機能テストの双方を補完する。
- AI コーディング支援でビジュアルチェックを生成させる場合、素朴な追加は正しく行える一方、**基底ページクラスに既存の視覚チェックメソッドがあるのに個別クラスへ重複実装する**ことがある。生成後は既存メソッドの再利用をレビューし、必要なら追いプロンプトで修正させる。

## Applitools Eyes

AI 搭載の商用ビジュアルテストツール。computer vision ベースの Visual AI がページの構造・レイアウト・色・形状を解析し、人間の眼に近い基準で差分を検出する。オンプレミス・プライベートクラウド・SaaS で提供される。

### AI 搭載ツールに共通する付加価値

| 機能 | 内容 |
|------|------|
| コンテキスト比較 | ページ構造・コンポーネントレイアウトの理解に基づき比較し、微小なピクセル差での偽陽性を減らす |
| 保守支援 | 視覚変化による失敗を認識・グループ化して根本原因分析を容易にし、承認するとベース画像を自動更新する |
| 動的データ処理 | レイアウト比較、パターンマッチ、同一ページ内の要素タイプ別マッチング手法の切り替え |
| 感度制御 | 微小な UI 変化を無視するか厳密に扱うかをチューニング可能 |
| OCR | 画像内テキストも読み取り、UX に影響しない微小なテキストシフトの誤検知を回避 |

### Ultrafast Grid

1 ブラウザでのテスト実行中にページの DOM・CSS 等のリソースをスナップショットとして取得し、それをクラウド側で任意のブラウザ・デバイス・解像度の組み合わせに対して**並列に再レンダリング・比較**する仕組み。実行時間を大幅に短縮できる。

### セットアップ要点

1. サービス登録後、ダッシュボードから API キーを取得し環境変数に設定: `export APPLITOOLS_API_KEY='...'`
2. `@applitools/eyes-playwright` を devDependencies に追加して install。

### ワークフロー（Playwright 統合）

Playwright ネイティブの比較メソッドを `eyes.check()` に置き換えるだけで統合でき、CI パイプラインへの組み込み方も同じ。

```ts
import { BatchInfo, Configuration, VisualGridRunner, BrowserType,
         DeviceName, ScreenOrientation, Eyes, Target } from '@applitools/eyes-playwright';

test.beforeAll(async () => {
  runner = new VisualGridRunner({ testConcurrency: 5 });   // 5 並列
  config = new Configuration();
  config.setBatch(new BatchInfo({ name: 'Home Page' }));
  config.addBrowsers(
    { name: BrowserType.CHROME,  width: 800,  height: 600 },
    { name: BrowserType.FIREFOX, width: 1600, height: 1200 },
    { name: BrowserType.SAFARI,  width: 1024, height: 768 },
    { chromeEmulationInfo: { deviceName: DeviceName.iPhone_11,
                             screenOrientation: ScreenOrientation.PORTRAIT } },
  );
});

test.beforeEach(async ({ page }) => {
  eyes = new Eyes(runner, config);
  await eyes.open(page, 'MyApp', 'Home Page', { width: 1200, height: 600 });
});

test('home page renders as designed', async ({ page }) => {
  await homePage.goto();
  await eyes.check('Home', Target.window().fully());
});

test.afterEach(async ({ page }) => {
  await page.close();
  await eyes.close();
});
```

- 初回実行でベース画像が取得され、ダッシュボードで確認できる。
- 差分検出時は Playwright のレポートにダッシュボードへのリンク付きエラーが出力され、比較結果を UI 上でレビュー・承認できる。

### 動的コンテンツのカスタマイズ

| カスタマイズ | 用途 |
|-------------|------|
| レイアウト比較 | ピクセル比較の代わりにコンポーネントの相対位置のみ検証。動的コンテンツだらけのページに有効 |
| 色の無視 | ロケール別の微小な色差で失敗させない |
| 正規表現マッチ | 日時・ユーザー ID など可変テキストをパターン一致で比較 |

これらはダッシュボード UI からページ全体/コンポーネント単位で設定するか、`eyes.check()` の引数として該当ステップに直接渡せる。設定は以降の実行にも適用される。

### その他の機能

- **self-healing**: 類似の失敗をグループ化し、承認に応じてテストを更新。
- **root-cause analysis**: 失敗の原因となった CSS/HTML の変更を提示。
- Cypress、Storybook、Appium など多数のツールと統合可能。

## Storybook（+ Chromatic）

- UI コンポーネントを**アプリ本体から分離して**開発・レンダリングできる OSS。バックエンドや複雑なスタックのセットアップ、テストデータ準備、画面遷移なしにコンポーネント単体の見た目と振る舞いを確認できる。React / Vue.js / Angular 等と統合。
- コンポーネントの状態ごとの描画結果を **story** として保存する（例: ボタンの large / small をそれぞれ別 story に）。story 群はそのままビジュアルテストの優れたリポジトリになる。
- 中央 UI チームがデザインシステム用の共有コンポーネントを（統合先バックエンドなしで）開発する組織で特に有効。
- **Chromatic**（ホスティングサービス・限定無料枠あり）と組み合わせると、新しい story を以前の story と自動比較する複数ブラウザ横断のビジュアルテストが可能になり、ビジュアルテストを開発段階まで shift-left できる。
- 公式: https://storybook.js.org / https://www.chromatic.com

## BrowserStack Percy

エンドツーエンドテストプラットフォーム上の AI 搭載ビジュアルテスト機能。

| 機能 | 内容 |
|------|------|
| 実デバイス比較 | AI 搭載の視覚比較を実デバイス上で実行 |
| カバレッジ | 数千のブラウザ・数万のデバイスバージョンにアクセス可能 |
| デザイン照合 | Figma の UX デザインとの比較 |
| 動的コンテンツ | 柔軟なハンドリングオプション |
| 保守支援 | 失敗のグループ化・修正支援、根本原因分析 |

- 既存の Cypress / Playwright 等の機能テストスイートに `percySnapshot()` の呼び出しと API キーを追加するだけで統合でき、CI/CD でも通常どおり実行できる。テスト管理用の UI ダッシュボードも提供される。

## 課題と対策: フレークとベースライン管理

### 代表的な課題

| 課題 | 原因 | 対策 |
|------|------|------|
| フレーク（不安定な失敗） | 動的コンテンツ（カルーセル、日時、パーソナライズ） | 静的テストデータ/スタブ、CSS 非表示、マスク、レイアウト比較・regex マッチ（AI ツール） |
| フレーク | デバイスのバッテリー状態・OS バージョン差による微小レンダリング差 | 感度パラメータ（`maxDiffPixelRatio` 等）を適切に緩める |
| フレーク | 導入ステージの誤り | 「骨組み実装のストーリー」にはビジュアルテストを足さず、「見た目を仕上げるストーリー」で追加する。ビジュアルテスト工数をユーザーストーリーに明記して計画する |
| ベースライン管理コスト | 小さな変更のたびに数百枚のベース画像を差し替える運用 | ベース画像の自動クリーンアップ・承認時自動更新を持つツールを選ぶ。ビジュアルテストは最重要フローに限定する |
| 偽陽性の山 | ピクセル完全一致の厳格すぎる比較 | 感度制御、コンテキスト（レイアウト）比較、失敗のグループ化機能を活用 |
| チームの合意形成 | 作成・保守の追加工数への抵抗 | 適切なステージで導入し、保守が容易なツールを選ぶ。導入前にユースケースの費用対効果を示す |

### ツール選定チェックリスト

- [ ] テスト作成 → 保守 → CI 統合までのワークフローが容易か
- [ ] ベース画像の管理が堅牢か（自動クリーンアップ・承認時の自動更新）
- [ ] 感度制御で微小な UI 変化を無視できるか
- [ ] 動的データをハンドリングできるか
- [ ] 多様なブラウザ・デバイスの組み合わせで高速に実行できるか

### アンチパターン

- 全ページ・全要素にビジュアルテストを張る（実行が遅く保守不能になる。ミクロテストとの責務分担で解決する）
- 未完成の UI に対してベースラインを固定する（ストーリー分割の段階を無視した導入）
- スナップショットテストの対象を大きなレイアウトにする（微修正のたびに壊れる）
- エラーフローの見た目をビジュアルテストで検証する（ミクロレベルのテストの守備範囲）
- 内部向けの小規模ツールに自動ビジュアルテストを導入する（手動目視で十分・コスト過剰）

## まとめ

- ビジュアルテストは「デザインどおりに見えること」の検証であり、視覚品質は顧客の信頼とブランド価値に直結する。
- 手動目視と UI 駆動機能テストだけでは不十分（前者は誤り検知に限界、後者は見た目を検証しない）。必要に応じて独立した自動ビジュアルテストを設ける。
- 導入可否は顧客影響・作業種別・手動工数・チームの変更自信で判断し、費用対効果を確認する。B2C・多デバイス・デザインシステム・リビルド・多ロケールが主な適用場面。
- OSS（Playwright、Storybook、BackstopJS 等）でも実施でき、AI 搭載の商用サービス（Applitools Eyes、BrowserStack Percy）は保守・インフラ・ワークフロー管理を上乗せで提供する。
- フレーク回避のため、適切な開発ステージで導入し、動的コンテンツ対策とベースライン管理を計画に織り込む。
- ビジュアルテストはフロントエンドテスト戦略の一部にすぎない。ミクロレベル多め・マクロレベル少なめの構成で、他のテストタイプと責務分担して高速なフィードバックを得る。
