# アクセシビリティテスト

Web アクセシビリティ（a11y）は、永続的・一時的・状況的な障害を持つユーザーを含む全員が Web を利用できるようにする実践であり、ユーザビリティとインクルーシビティの部分集合である。世界人口の約 2 割が何らかの障害を持つとされ、多くの国で法的要件でもある（国連障害者権利条約を基盤とする各国政策・訴訟事例の増加）。本リファレンスは、アクセシビリティのユーザーペルソナと支援技術エコシステム、WCAG の原則とレベル、シフトレフト型のテスト戦略、Lighthouse / axe-core / Accessibility Insights / Pa11y CI などの監査ツールの使い分けをまとめる。

## なぜ投資するか（判断材料）

| 観点 | 要点 |
|------|------|
| ユニバーサル価値 | 「一部の人に必須、すべての人に有用」。明瞭なレイアウト・音声操作・分かりやすいエラーメッセージは全ユーザーの利便性を上げる |
| ビジネス | 障害を持つ人口は購買力ベースで世界有数の市場規模。対応は市場拡大に直結 |
| 法的リスク | 多くの国が Web アクセシビリティを法的に義務化。スクリーンリーダー非対応等を理由とした訴訟が増加傾向 |
| コスト | ページ上のほぼ全要素が対象になるため、後付け（retrofit）は極めて高コスト。設計段階からの組み込み（シフトレフト）が唯一の現実解 |

参照すべき標準は W3C WAI の Web Content Accessibility Guidelines（WCAG）。バージョンは更新され続けるため、開発対象国の法規制と併せて常に最新版を確認する（https://www.w3.org/WAI/ ）。

## アクセシビリティのユーザーペルソナ

ペルソナを定義して要件・ユーザーストーリーに織り込むのが第一歩。代表的なニーズの類型:

| ペルソナ類型 | 状況の例 | 必要な機能 |
|-------------|---------|-----------|
| 一時的な運動制限 | 腕の骨折でマウス操作が困難 | キーボードのみでの完全操作 |
| 色覚の低下・色覚特性 | 高齢による色感度低下、色覚異常 | 背景と前景（リンク・ボタン・画像）の十分なコントラスト、色以外の手がかり |
| 認知・学習の困難 | 新しいことの習得に時間が必要 | 見出し・ナビゲーションバー・一貫した構造を持つ整理されたレイアウト |
| 状況的制約（急ぎ） | 運転中に近くの施設を素早く探す | 即断できる明瞭な情報提示 |
| 全盲 | スクリーンリーダー利用者 | text-to-speech、音声認識、意味のある読み上げ順序 |
| ハンズフリー必須 | 乳児を抱えて手が塞がる | speech-to-text、音声操作、スクリーンリーダー |
| 巧緻性の低下・低視力・ディスレクシア | 細かい操作が困難 | 大きなテキスト・ボタン・操作ターゲット |
| 聴覚障害 | 動画コンテンツの利用 | 同期したキャプション・字幕 |
| 言語・リテラシーギャップ | 非母語話者、学習初期 | 専門用語を避けた平易で理解しやすい文章 |

これらは視覚（全盲/弱視）・聴覚・認知・運動の各困難と一時的/状況的制約を網羅する。ゴールは全員が等しく「知覚・理解・ナビゲート・操作」できること。

## 支援技術エコシステム

アクセシブルな機能は Web 技術単体では完結せず、3 層の連携で成立する:

| 構成要素 | 内容 | 対応する W3C 標準 |
|---------|------|------------------|
| Web 開発ツール・言語 | HTML/CSS/フレームワークが要素情報を支援技術へ渡す構文的手段を持つこと | ATAG（オーサリングツール向け） |
| ユーザーエージェント | ブラウザ・メディアプレーヤー等、コンテンツをレンダリングしスクリーンリーダー等と統合する層 | UAAG |
| 支援技術（AT） | スクリーンリーダー、代替キーボード、スイッチ等、ブラウザと対話しユーザーへ情報を中継する装置・技術 | WCAG（コンテンツ側の要件）＋ UAAG の一部 |

開発チームが直接責任を持つのは WCAG（コンテンツ）だが、エコシステム全体の分業を理解していないと「なぜこの属性が必要か」が判断できない。

### スクリーンリーダーの動作原理

- スクリーンリーダーはページの **accessibility tree**（DOM に似た構造で、要素の role・名前・状態を意味のある順序で持つ）に沿って内容を読み上げ、ユーザーはキーボードショートカットで操作する。
- accessibility tree の順序は「ユーザーフローとして意味が通る順」であるべき。例: 検索フォームなら出発地 → 目的地 → 検索ボタンの順。
- 広告バナーなどフローに寄与しない要素は accessibility tree から隠すコーディングが可能。
- 自分が使わない支援技術は実際に試すのが最善。ブラウザ拡張のスクリーンリーダーや、視覚障害の体験を模したデモサイトで、キーボードのみの操作フローを体験してから設計・テストに臨む。

## WCAG: 4 原則（POUR）と 3 適合レベル

### 適合レベル

| レベル | 内容 | 位置づけ |
|--------|------|---------|
| A | 最低限の適合。キャプション提供・キーボード完全対応・色のみに依存しない情報伝達など、これが無いとサイトが利用不能になる必須要件 | すべてのサイトが満たすべき下限。テストケースへ直訳する対象 |
| AA | A の全要件＋色コントラスト比の制約などより厳しい要件 | 多くの法制度が推奨・要求する実務標準 |
| AAA | A/AA＋動画への手話通訳など、真に全員へ届けるための強化要件 | 到達すれば強いコミットメントの証明 |

### POUR 原則と Level A 要件の要点

**Perceivable（知覚可能）** — 情報と UI はユーザーが知覚できる形で提示する:

- 画像等の非テキストコンテンツに代替テキスト（alt text）を付ける
- 音声・動画に同期したトランスクリプト・キャプションを提供し、一時停止・停止・音量制御を可能にする
- 自動再生される音声には制御手段を必ず付ける
- ページの情報と構造に階層と意味のある流れを持たせる
- 「ボタンが緑になるまで待つ」「ビープ音が鳴ったら」のように形・色・大きさ・位置・音などの感覚的特徴のみに依存する指示を避ける
- 色をアクションの指示・応答の要求・要素の区別の唯一の手段にしない
- 背景と前景のコントラスト比を推奨値に従わせる

**Operable（操作可能）** — UI コンポーネントとナビゲーションは操作可能にする:

- サイト全体でキーボードのみのナビゲーション（前進・後退・モーダルからの脱出）を提供し、フォーカスは明瞭に可視化する
- コンテンツを読み切るのに十分な時間を与える
- 発作等の身体反応を誘発しうる点滅・フラッシュを避ける
- 繰り返しコンテンツのスキップ手段を提供する
- 画面外の（未表示の）コンテンツはスクリーンリーダーからも隠す
- リンクには意味のあるテキストを付ける

**Understandable（理解可能）** — 情報と操作は理解可能にする:

- 「034506451988 is invalid」のような機械的表現を避け、「日付の形式が正しくありません」のような平易で意味のあるメッセージを出す
- 略語は必要に応じて展開する
- 複数ウィンドウを突然開くなどの文脈の急変（キーボードナビゲーションを壊す）を避ける
- ユーザーが大きなフォント等の設定を変えても文脈が変化しないようにする
- 明確で行動可能なラベルを付ける（例: Email ラベル＋サンプル値の提示）

**Robust（堅牢）** — 多様なユーザーエージェント・支援技術が確実に解釈できるようにする:

- マークアップは標準準拠（開閉タグの対応、重複回避、ユニークな ID）でパース可能に保つ
- スクリプト生成要素も含め、全要素の name / role / state を支援技術へ公開する（例: `role="checkbox"`＋`aria-checked="true|false"`。チェック後は更新済み state を通知する）

## WAI-ARIA

カスタム UI 要素（例: CSS で `<li>` をチェックボックス風に見せたもの）は、標準 HTML（`<input type="checkbox">`）と違って支援技術が種類・状態・挙動を自動認識できない。WAI-ARIA の属性（`role`、`aria-checked` 等）を追加して accessibility tree に反映させることで、全支援技術が正しく解釈できるようになる。原則は「まず標準 HTML セマンティクスを使い、足りない場合のみ ARIA で補う」。

## フレームワークの a11y 支援

React（標準 HTML 技法ベースの全面サポート）、Angular Material（アクセシブル設計の再利用 UI コンポーネント群）、Vue.js などの主要フレームワークはアクセシブルなコンポーネント構築を支援する。さらに自動監査ツールが標準的な a11y タグの欠落を警告してくれるため、追加コストは想像より小さい。

## アクセシビリティテスト戦略（シフトレフト）

要件をテストフェーズ後に後付けするのではなく、プロジェクト開始時から継続的に支える。

| 段階 | 活動 |
|------|------|
| プロダクト設計 | アクセシビリティペルソナを定義し、機能・ユーザーフローが全ペルソナを包含するかチームで検証 |
| UX デザイン | デザイン成果物を WCAG 適合の観点で検証（Figma 向けの AI 支援検証ツール等を利用可能） |
| ユーザーストーリー | 全ストーリーに a11y チェックリストを添付 |
| 開発中 | 静的コード解析（linter）＋ローカルでの自動監査で即時フィードバック |
| 機能完成時 | E2E ナビゲーションの一貫性を手動テスト |
| リリース前 | 障害を持つ実ユーザーによるテスト → 専門家による適合性評価 |
| CI/CD | 自動スキャンをパイプラインに常設 |

### ユーザーストーリー用アクセシビリティチェックリスト

WCAG 要件の多くは全ページ共通のため、汎用チェックリストを全ユーザーストーリーへ添付し、アプリ固有項目を追加していく:

- [ ] ブラウザタブのページタイトルがサイト内でのページの文脈を明確に表す
- [ ] CSS を無効化して、要素がスクリーンリーダーに親和的な順序で並ぶことを確認（Chrome DevTools の accessibility tree ビューで順序を確認できる）
- [ ] キーボードのみのナビゲーション（前進・後退・脱出）とフォーカスの明瞭なハイライト
- [ ] エラーメッセージ・ラベル・リンク・その他テキストが正しい意図を伝える
- [ ] OS 設定やブラウザズームでテキストを拡大しても可読性が保たれる
- [ ] グレースケール表示でも判読できる
- [ ] 動画・音声のキャプションが正確・有意味・同期している
- [ ] 画像に意味のある alt text がある（ブラウザの画像ダウンロードを無効化すると alt text 表示で確認できる）
- [ ] スクリーンリーダーでフローを最後まで完了できる

### 自動監査ツールの 2 形態

| 形態 | タイミング | 例 |
|------|-----------|-----|
| 静的コード解析（linter） | コーディング中に即時フィードバック | `eslint-plugin-jsx-a11y`（React/JSX）、Codelyzer（TS/HTML/CSS）、Angular ESLint、Axe DevTools Linter（IDE 拡張・Git pre-commit hook） |
| ランタイムチェッカー | 実際にレンダリングされたページを監査 | axe-core、Lighthouse、Accessibility Insights、Pa11y CI |

補完として、動画下のトランスクリプト表示や明瞭なエラーメッセージなど a11y 要件に由来する機能フローは、通常の機能テスト（ミクロ/マクロ）として自動化する。ローカル実行＋CI 組み込みの両方で回す。

### 手動テストは省略できない

自動ツールが検出するのは HTML 構造の問題が中心（自動検出率はおおむね 5〜8 割との報告）。グレースケール確認・ズーム確認・スクリーンリーダーでのフロー完遂などは人間の検証が必須。スコープを段階で分けると効率的:

| 段階 | スコープ | 手法 |
|------|---------|------|
| ユーザーストーリーテスト | ストーリーが及ぶ全ページ | チェックリスト実施＋WAVE や Lighthouse で問題をブラウザ上に可視化 |
| 機能テスト | 機能完成時の E2E ナビゲーション | 一貫性の欠如を検出。Tab / Shift+Tab で前後移動、Enter で選択、矢印キーでドロップダウン操作を行い、フォーカス位置と可視性を確認。ガイド付きテストツール（Accessibility Insights、Axe IGT）を併用 |
| リリーステスト | リリース全機能 | 障害を持つ実ユーザーを含むエンドユーザーテスト（リモートテスティングサービスで依頼可能）。多様な支援デバイスからのリアルフィードバックを得る |
| 適合性認証 | 公開前 | WCAG 専門家（社内またはコンサルタント）による最終適合性評価 |

## 主要ツール別リファレンス

多くのツールは次の 3 機能を組み合わせて提供する: **自動スキャン**（タグ閉じ・alt text・フォームラベル・ID 一意性などを高速チェック、Playwright 等の機能テストに組み込み可）／**ガイド付きテスト**（ツールの観察結果を人間がレビュー・上書き）／**手動チェックリスト**（前二者で覆えない項目）。

### Lighthouse

- Chrome DevTools の Lighthouse タブで Accessibility カテゴリを選ぶだけで、重み付き計算による総合スコアと改善候補（コントラスト不足、`lang` 属性欠落など）をレポート。
- 問題のあるコード行を特定し、修正方法の解説リンクを併記。レポート末尾に手動確認項目のチェックリストも出す。
- Lighthouse の Node モジュールで CLI 実行・CI 統合が可能。

### axe-core（＋Playwright 統合）

- 多くのツール（Accessibility Insights、Lighthouse 等）の自動スキャンの基盤となっているオープンソースエンジン。WCAG 問題の平均約 57% を自動検出できるとされる。主要ブラウザ対応。
- 機能テストスイートに組み込み、ユーザーフローの各ページで監査を実行するのが定石（Selenium なら Maven 依存として追加可能）。

```bash
npm install @axe-core/playwright
```

```typescript
import AxeBuilder from '@axe-core/playwright';

// Page Object のメソッドとして。goto() 後に呼び、違反 0 件をアサートする
async checkAccessibility() {
  const results = await new AxeBuilder({ page: this.page }).analyze();
  if (results.violations.length > 0) {
    console.error('Accessibility Violations:', results.violations);
    throw new Error(`Accessibility check failed with ${results.violations.length} violations.`);
  }
}
```

- テストスイートが育ったら、この種の共通チェックは `BasePage` クラスへ引き上げて全ページで継承する。
- 特定 WCAG バージョンへの準拠チェックや、ページの一部分だけのスキャンなどの高度な設定も可能（公式ドキュメント参照）。

### Accessibility Insights

- Microsoft 製のオープンソースツール（https://accessibilityinsights.io ）。Chrome / Edge 拡張として無償提供され、WCAG 2.1 / 2.2 向けに自動スキャン（axe-core ベース）＋ガイド付きテスト＋手動チェックリストを備える。
- **FastPass**: ①自動チェック（違反要素を赤枠でハイライト、修正方法を提示）→ ② Tab stops（Tab キーを押しながら要素順を番号付きで可視化レビュー）→ ③手動チェックリスト（コントラスト等）の 3 段階で素早く評価。
- **Assessment**: 完全な WCAG 適合評価用の 20 超のチェック群。例えば画像チェックでは各画像の ARIA role と Decorative/Meaningful 分類を提示し、人間が pass/fail を判定。最終評価レポートをエクスポートして欠陥登録に使える。

### Axe Intelligent Guided Testing（IGT）と Axe 製品群

シフトレフトの各段階を埋める製品群（一部有償）:

| ツール | 段階 | 内容 |
|--------|------|------|
| Axe for Designers | デザイン | Figma 上で UX デザインの WCAG 適合を AI 検証 |
| Axe linters / pre-commit hooks / Axe Assistant | 開発 | 問題の予防・検出・修正コード提案 |
| axe-core エンジン | CI | Lighthouse・Playwright 等から自動スキャン |
| Axe ブラウザ拡張＋IGT（有償） | テスト | テーブル・モーダル・画像・フォーム等のガイド付きチェック。Keyboard IGT は自動で Tab 送りし順序とフォーカス表示をレビューさせる。AI がリンク・ボタン等の name/role/state の正しさを視覚判定（例: alert box と誤マークされたボタンを検出し修正提案）。AI 強化により自動カバレッジが 57%→80% 超に向上したとベンダーは報告 |

### Pa11y CI

- Headless Chrome でページをレンダリングして監査するオープンソース CLI（Node モジュール）。既定エンジン（HTML CodeSniffer）は axe-core と検出領域が異なるため、CI で両者を併用する構成も一般的。
- 設定ファイルの `urls` に複数 URL を列挙、または `--sitemap` で XML サイトマップを渡して一括監査。エラー/警告のしきい値、viewport サイズ、ページロード待機時間を設定でき、CI のビルドステージまたは独立ステージに追加する。

```json
{
  "defaults": {
    "timeout": 1000,
    "viewport": { "width": 320, "height": 480 }
  },
  "urls": [
    "https://example.com/",
    "https://example.com/checkout"
  ]
}
```

## 落とし穴・アンチパターン

| アンチパターン | 問題 | 対策 |
|---------------|------|------|
| テスト終盤での後付け対応 | ほぼ全要素に変更が必要になり修正コストが爆発 | ペルソナ定義・デザイン検証・ストーリーチェックリストで最初からシフトレフト |
| 自動スキャンのみで完了と見なす | 自動検出は WCAG 問題の一部（〜6 割前後）に限られる | ガイド付き・手動テスト・実ユーザーテストを段階的に必ず組み合わせる |
| 色だけで状態を伝える UI | 色覚特性のあるユーザーに情報が届かない | テキスト・アイコン・形状など複数の手がかりを併用 |
| 感覚的特徴に依存した指示文 | 「緑になったら」「音が鳴ったら」は知覚できない人がいる | 感覚に依存しない指示へ書き換え |
| カスタム要素に ARIA 属性なし | 支援技術が種類・状態を認識できない | 標準 HTML 優先、カスタム時は role / aria-* を必ず付与 |
| 機械的なエラーメッセージ | 「invalid」だけでは理解も回復もできない | 平易で行動可能なメッセージに |
| accessibility tree の順序無視 | スクリーンリーダーの読み上げがフローとして破綻 | CSS オフ／DevTools の tree ビューで順序を検証 |
| 支援技術を一度も触らずに実装 | 要件の意図を誤解した形式的対応になる | スクリーンリーダー等を最低 1 つ実際に使ってみる |

## 文化としてのアクセシビリティ

アクセシビリティは Web アプリに限らないマインドセットの転換である。「画像だけのメールに代替テキスト要約を付けたか」「スライドのフォントは全員が読めるサイズか」「難解な語彙より平易で明快な表現を選べているか」——日常のあらゆるコミュニケーションに同じ問いを適用する文化を育てることが、持続的な適合の土台になる。

## キーテイクアウェイ

- アクセシビリティは「一部の人に必須、すべての人に有用」。ビジネス・法務の両面で強い投資根拠がある
- W3C WAI の WCAG（POUR 原則× A/AA/AAA レベル）が実装・テストの基準。常に最新版と対象国の法規制を確認する
- エコシステムは Web 技術・ユーザーエージェント・支援技術の 3 層。最低 1 つの支援技術を自分で体験する
- SDLC の最初から組み込む。後付けは実質的に破綻する
- チェックリスト＋静的解析＋ランタイム自動監査（Lighthouse / axe-core / Accessibility Insights / Pa11y CI / WAVE）で左へシフトし、手動テストと実ユーザーテストで補完する
- 障害を持つ実ユーザーからのフィードバックを得られるテストサービスの活用を検討する
