# 横断的要求（CFR）のテスト戦略

横断的要求（Cross-Functional Requirements: CFR）は、すべての機能フィーチャーに組み込まれるべきアプリケーション特性（性能・セキュリティ・可用性・保守性など）である。機能要求だけを満たしても、応答が遅い・操作が複雑・個人情報が漏れるアプリは使われない。本リファレンスは、FURPS モデルに基づく CFR テスト戦略の全体像と、カオスエンジニアリング・アーキテクチャテスト・インフラテスト・コンプライアンステスト・進化可能性検証の実践パターンをまとめる。

## CFR と NFR — 用語と考え方

| 観点 | 内容 |
|------|------|
| 呼称 | 業界では「非機能要求（NFR）」とも呼ばれるが、「横断的（cross-functional）」の方が実態に合う |
| なぜ「横断的」か | 要求がアプリ全体に分散し、**すべてのユーザーストーリー/フィーチャーの一部として構築・テストされる**必要があるため |
| 「非機能」の弊害 | 「本質的でない」という誤解を招く。CFR は機能要求と同等に必須 |
| 横断的な「機能」 | 認証・認可のように機能として実装されるが全リクエストに適用されるものも CFR として扱う |

### 実行時品質と進化的品質

| 分類 | 定義 | 例 | 欠落時に困る人 |
|------|------|-----|---------------|
| 実行時品質（executional） | ランタイムでの振る舞い | 可用性・認証・監視・回復性 | エンドユーザー（システム停止で不満） |
| 進化的品質（evolutionary） | 静的コードの品質 | 保守性・スケーラビリティ・拡張性・テスト容易性 | まず開発チーム、やがてビジネス全体 |

**原則**: CFR は開発開始時に確立し、機能要求と同様にデリバリーサイクル全体で継続的にテストする。CFR チェックリストを各ユーザーストーリーに組み込むとよい。

### 代表的な CFR（-ilities）

可用性 / 認証・認可 / 監視・可観測性 / 監査可能性 / アーカイブ可能性 / 回復性 / レジリエンス / 一貫性 / 性能 / スケーラビリティ / セキュリティ / プライバシー / アクセシビリティ / ユーザビリティ / ローカライゼーション / 互換性 / 設定可能性 / 拡張性 / インストール容易性 / 相互運用性 / 可搬性 / 保守性 / 再利用性 / テスト容易性 — 網羅列挙ではなく、プロジェクトごとに優先順位を付けて選定する。

## FURPS モデルによる CFR テスト戦略

FURPS は全ソフトウェア要求を分類する広く使われるモデル。CFR を 5 テーマに分解し、テーマごとに手法・ツールを選ぶ。

| テーマ | 対象 | 主なテスト手法 |
|--------|------|---------------|
| **F**unctionality（機能性） | ユーザーフロー（ログイン・予約・決済等）、コンプライアンス機能 | 手動探索的テスト、レイヤー別自動機能テスト、データテスト |
| **U**sability（使用性） | 視覚品質、ブラウザ互換、アクセシビリティ、使いやすさ | ビジュアルテスト、アクセシビリティ監査、L10n/i18n テスト、UX テスト、A/B テスト |
| **R**eliability（信頼性） | 一貫性、耐障害性、回復性 | 機能テスト（エラー処理・リトライ・フォールバック）、カオスエンジニアリング、インフラテスト |
| **P**erformance（性能） | バックエンド KPI、フロントエンド性能指標 | 負荷テスト・フロントエンド性能計測（限界閾値の特定はスケーラビリティ/信頼性の裏付けにもなる） |
| **S**upportability（支援性） | 進化的コード品質全般（保守性・テスト容易性・セキュリティ等） | アーキテクチャテスト、静的解析、セキュリティスキャン |

**運用の要点**: これらを CI に統合して「シフトレフト」する。1 つの CFR が複数テーマにまたがることも多い（例: アクセシビリティは機能性＋使用性、セキュリティは認証機能＋セキュアな静的コード）。

## テーマ別の実践ポイント

### 機能性

- 手動探索的テストと自動機能テストを各レイヤーで組み合わせる。
- コンプライアンス関連機能は、**要件定義とテストの両フェーズで法務チームを巻き込む**（後述「コンプライアンステスト」）。

### 使用性

#### ローカライゼーション / 国際化テストの判断基準

| 状況 | 採用する手法 |
|------|-------------|
| ロケールごとに UI スキンが変わる | ビジュアルテスト |
| スキン不変・言語/日付/通貨フォーマットのみ変化 | ユニットテスト＋手動テスト |
| 翻訳でテキスト長が変わりレイアウトが崩れうる | ビジュアルテスト |

手順の要点:

1. 対象言語のネイティブスピーカーによる翻訳を早期に取得し、ビジネス承認を得る。
2. 承認済み翻訳文字列を各ユーザーストーリーに記載してから開発・手動テストする（省略すると開発者が機械翻訳で埋め、承認前後で二重テストになる）。
3. UI 駆動テストで全文字列を検証しない（低速化する）。ロケール間で**機能フローが異なる箇所だけ** UI テストで検証し、アサーション文字列と要素識別子はパラメータ化して再利用する。

**落とし穴**: ローカライゼーションテストをリリース直前まで先送りすると、翻訳文がレイアウトに収まらず、デリバリー後期に UI 崩れが発覚する。翻訳プロセスを開発初期に開始し継続的にテストする。

#### UX テスト

- 検証観点の例: フローの直感性 / 目的情報までのクリック数 / アイコンの意味伝達 / 配色の受容性。
- 公開されているユーザビリティヒューリスティクス（10 項目など）をチェックリストとして活用する。
- 各ユーザーストーリーの探索的テストに UX 観点を含める。プロダクトオーナーと UX デザイナーのペアによるユーザビリティテストは、テスターがユーザー行動を学ぶ機会にもなる。
- プロトタイプに対して実ユーザーで UX テストを行うツール（UserZoom, Optimal Workshop 等）をデリバリーサイクル中に定期的に、異なるユーザーグループで実施する。

#### A/B テスト

- 同一フィーチャーの異なる UX 案を本番の 2 ユーザーグループに提示し、行動データで最終デザインを決める（例: Sale ボタンは赤と青どちらがクリックされるか）。
- データサイエンスの素養が必要で、PO・データサイエンティスト・開発者・UX デザイナーの協働になる。
- AI 支援の実験プラットフォーム（LaunchDarkly, Statsig 等）は多変量実験の設計とユーザープロファイルに基づくリアルタイムのトラフィック振り分けを自動化し、実験サイクルを短縮する。

### 信頼性

- 信頼性に寄与する CFR: 回復性・レジリエンス・監査可能性・アーカイブ可能性・レポーティング・監視・可観測性・一貫性。
- ユーザーフローとして体験できる側面（エラー処理、リトライ、単一障害点のフォールバック、データ整合性、監視/可観測性ツール統合）は機能テストの手法で検証できる。
- 「未知の未知」を暴くにはカオスエンジニアリング（後述）、土台の検証にはインフラテスト（後述）を使う。インフラ層は autoscaling・アラート・監視・ロードバランシング・アーカイブを支えるよう構成されている必要がある。

### 支援性（Supportability）

- 設定可能性・プロトコル互換性・インストール容易性・相互運用性などの機能的側面は、適切な環境セットアップとスタブがあれば通常の機能テスト手法で検証できる。
- それ以外の代表的手法:

| 手法 | 目的 | ツール例 |
|------|------|---------|
| アーキテクチャテスト | 設計上の必須特性（レイヤー分離・パッケージ独立性等）からの逸脱を自動検知 | ArchUnit, NetArchTest, JDepend, NDepend |
| 静的コード解析 | 保守性の継続的担保 | Checkstyle（コーディングスタイル統一）、PMD（未使用変数・空 catch・重複コード・カスタムルール）、ESLint（JavaScript のスタイル/エラー検査）、SonarQube（カバレッジ・脆弱性スキャン） |

## カオスエンジニアリング

### 位置づけ

- 分散システムに対して意図的に障害を注入し、本番の乱流条件に耐える能力への確信を得るための**実験の規律**。
- ダウンタイム要因（アプリのバグ・単一障害点・ネットワーク/ハードウェア障害・想定外の高負荷・サードパーティ依存の障害）は設計・開発で対策する（例: exponential back-off、blue/green デプロイ）が、大規模分散システムでは多層依存や下流システムのレート制限など**事前に予測困難なエッジケース**が残る。
- 典型的な失敗例: 2 リージョン構成で相互フェイルオーバーをテスト済みでも、片系ダウンとセール時のトラフィックピークが重なった瞬間に、下流サードパーティのレート制限超過で全ユーザーにエラーが波及する — 個々のテストでは捕捉できない。
- 大規模ストリーミングサービスがクラウド移行時に、稼働中クラスタのインスタンスをランダムに毎日停止させるツール（安全策付き）で耐障害性を鍛えた事例が起源として広く知られる。

### テストとの違い

| 観点 | テスト | カオスエンジニアリング |
|------|--------|----------------------|
| 目的 | 期待挙動の検証 | 想定外状況での挙動の**観察**（実験） |
| 前提 | 期待値が定義できる | 未知の未知を探る |
| 実施判断 | 常に実施 | システムの耐乱流性に確信があれば実施しない選択も可 |
| 担い手 | テスター中心 | 全ステークホルダーによるチーム活動（実験設計→実施→デバッグ） |

### 実施手順

1. アプリの信頼性を脅かしうる仮説を立て、クロスファンクショナルなチームを編成する。
2. 定常状態仮説（steady-state hypothesis: 実験中もアプリはこう振る舞うはず）を定義する。
3. ツールで実験をスクリプト化し、**本番で**実行する（実環境の変数はテスト環境で再現困難なため。ただし実験の一時停止と正常状態への復帰手段を必ず用意する）。
4. 定常状態仮説が満たされない（実験失敗）とアラートされたら、チームで原因を調査する。

ツール例: Chaos Toolkit（<https://chaostoolkit.org>・OSS・JSON で実験定義・HTML レポート生成）、Harness、ChaosBlade。AI 支援の商用プラットフォームは、アーキテクチャと運用データから実験の設計・スクリプト化・失敗観察・修正提案までを自動化する。

### 実験定義の例（Chaos Toolkit・簡略化）

```json
{
  "title": "設定ファイル欠損時も別インスタンスでアプリが応答し続ける",
  "contributions": { "reliability": "high", "availability": "high" },
  "steady-state-hypothesis": {
    "title": "アプリは稼働中",
    "probes": [{
      "type": "probe",
      "name": "homepage-must-respond-ok",
      "tolerance": 200,
      "provider": { "type": "http", "timeout": 2, "url": "https://www.example.com/" }
    }]
  },
  "method": [{
    "type": "action",
    "name": "delete-config-file",
    "provider": {
      "type": "python", "module": "os", "func": "remove",
      "arguments": { "path": "/path/config-file" }
    },
    "pauses": { "after": 1 }
  }]
}
```

障害（設定ファイル削除）を注入し、1 秒待って HTTP probe が 2 秒以内に 200 を返すかを検証する。「応答したが 4 秒かかった（フェイルオーバーの再ルーティングに手間取った）」という失敗も貴重な知見になる。

## アーキテクチャテスト

### なぜ必要か — コンウェイの法則

システムの設計は組織のコミュニケーション構造を写し取る（コンウェイの法則）。大規模システムの一部を担当するチームは、全体像を考慮せず自分の担当範囲を局所最適化しがちで（例: 性能優先でレイヤー分離をバイパス）、初期に合意した設計上の必須特性が徐々に崩れる。アーキテクチャテストは、この逸脱を**自動テストとして CI で検知**するガードレールである。

### ツールと使い方

| ツール | 対象 | 用途 |
|--------|------|------|
| ArchUnit | Java | JUnit 形式でアーキテクチャ特性をアサート（パッケージ配置・循環依存・レイヤー規則） |
| NetArchTest | .NET | ArchUnit 相当 |
| JDepend | Java | パッケージ単位の設計メトリクス（抽象クラス/インターフェース数による拡張性評価・外部依存・循環依存検出）。JUnit テストとして CI 統合可 |
| NDepend | .NET | JDepend 相当 |
| AI コードレビューツール | 多言語 | コーディングスタイルだけでなくアーキテクチャ特性の強制に対応するものもある（標準ルール＋利用上限あり。自アプリ固有のテストは別途必要） |

### 例: パッケージ配置の強制（ArchUnit）

```java
@Test
public void order_classes_must_reside_in_oms_package() {
    classes().that().haveNameMatching(".*order.*")
        .should().resideInAPackage("..oms..")
        .as("order classes should reside in the package '..oms..'")
        .check(classes);
}
```

注文管理サービスのクラスを `oms` パッケージに限定する。範囲外にクラスを置く必要が生じたら、チームは全体像を議論してから配置を決めることになる — テストが設計会話のトリガーになる。

### 例: 循環依存の禁止（JDepend）

```java
public class PackageDependencyCycleTest extends TestCase {
    private JDepend jdepend;

    protected void setUp() throws IOException {
        jdepend = new JDepend();
        jdepend.addDirectory("/path/to/project/A/classes");
        jdepend.addDirectory("/path/to/project/B/classes");
    }

    public void testAllPackages() {
        jdepend.analyze();
        assertEquals("Cycles exist", false, jdepend.containsCycles());
    }
}
```

同様に「特定パッケージは期待した依存のみ/依存なし」も検証でき、重要なアーキテクチャ特性への望まぬ変更を即時フィードバックできる。

## インフラテスト

### スコープ

インフラ = 計算資源（マシン・VM・コンテナ）＋ネットワーク構造（VPN・DNS・プロキシ・ゲートウェイ）＋ストレージ資源（オブジェクトストレージ・DB・シークレット管理）。インフラテストはこれらの**セットアップと設定**を検証する新興領域で、急速なスケール（新リージョン展開・ユーザー急増）にワンクリックでスタック全体を複製する必要性から重要度が増している。アプリ側のビルド/デプロイ自動化はあってもインフラ側に同じ能力がない、というギャップが典型的な課題。

### IaC のテストパイプライン（Terraform の例）

| 段階 | 手法 / コマンド | 検証内容 |
|------|----------------|---------|
| 1. 静的検証 | `terraform validate`, TFLint | 構文エラー・非推奨構文・命名規約・クラウドプロバイダに存在するイメージタイプかの確認（開発中の高速フィードバック） |
| 2. 変更プレビュー | `terraform plan` | 既存環境との差分比較。**DB を誤削除する変更もここで発見できる** |
| 3. plan 出力への自動テスト | terraform-compliance（<https://terraform-compliance.com>） | セキュリティポリシー準拠などを apply 前に検証 |
| 4. 実インスタンス検証 | Terratest, awspec, Chef InSpec, Kitchen-Terraform | デプロイ後に「プライベートサブネット内で稼働しているか」「必要なディスク容量があるか」等を自動検証し CI へ。コスト削減には LocalStack や Firebase Local Emulator Suite 等のエミュレータを使う |
| 5. E2E | インフラコンポーネント間の疎通確認 | Web サーバ→アプリサービスの呼び出し等。アプリデプロイ成功＋機能テスト通過で間接的にも確認されるが、**アプリデプロイ前に**問題を捕捉できるのが価値 |

**テスト分布の指針**: 宣言的な低レベルコード（Terraform 等）のユニットテストは最小限に保つのが定石で、レイヤー別のテスト分布はピラミッドではなく**ダイヤモンド型**になりやすい。コードの性質に応じて適切なレイヤーにテストを足す。

### インフラの追加検証観点

| 観点 | 検証内容 / ツール |
|------|------------------|
| スケーラビリティ | 負荷に応じた autoscale の発動と、スケール後も機能が正常動作すること |
| セキュリティ | IaC の脆弱性スキャン（Snyk IaC 等）、開放ポート・public/private 面の妥当性（手動＋自動） |
| コンプライアンス | policy-as-code（Sentinel, Checkov, Open Policy Agent）。例: PCI DSS 準拠のファイアウォール設定 |
| 運用性 | 監査用ログアーカイブ・監視ツール統合・自動メンテナンス機能 |

**制約と落とし穴**: インフラテストツールは発展途上で言語がばらつく（Terratest=Go、terraform-compliance=Python、awspec=Ruby）。AI コーディング支援である程度補えるが、実インフラの起動を要するツールはコストがかかる。制約内で持続可能な戦略を設計する。IaC 入門には、プロンプトからインフラコードとテストを生成する AI インターフェース（Pulumi AI 等）も足がかりになる。

## コンプライアンステスト

手順は 3 段階: ①法規制を理解する → ② CFR テストで準拠を検証する → ③法務チームまたは認定機関が準拠を認証する。**要件収集とテストの両方に法務を巻き込む**こと。以下は代表的な規制の概要であり、法的助言ではない。

### GDPR（EU 一般データ保護規則）

- EU 市民の個人データ保護が目的。EU 市民に物やサービスを提供するサイトは所在地を問わず対象。違反時は年間売上の最大 4% の制裁金。
- 世界的にも同種の法律が主流（約 8 割の国がデータ保護・プライバシー法を保有。カナダの CPPA、英国独自の GDPR 等）。
- 個人データ = 単独または組み合わせで個人を特定できる情報（人種・信条・政治的見解・性的指向・遺伝/生体データ・犯罪歴に加え、IP アドレス・MAC アドレス・デバイス ID・Cookie・アカウント ID などのオンライン識別子も含む）。Privacy by Design の考え方に沿った実装が推奨される。

#### ユーザーの権利とテスト観点

| 権利 | 内容 |
|------|------|
| 知らされる権利 | 個人データの利用方法の告知（プライバシーポリシー等） |
| アクセス権 | 保存された個人記録の開示請求 |
| 忘れられる権利 | 保持理由がない個人データの削除請求 |
| 処理制限権 | 保存は許すが処理（AI 学習等）は禁止できる |
| 訂正権 | 不完全・不正確なデータの訂正 |
| ポータビリティ権 | 個人データの取得と再利用 |
| 異議申立権 | マーケティング・調査・統計・AI 処理への利用拒否 |
| 自動意思決定関連の権利 | プロファイルを自動意思決定に使う前の同意取得 |

検証は機能テスト＋セキュリティテストで可能。例: 「暗黙のオプトインがない」「同意取得後にのみ個人データを保存する」「アプリケーションログに個人情報が残らない」をミクロ/マクロ両レベルの自動テストでアサートする。

**AI 関連の動向**: GDPR は AI 搭載プロダクトも対象。EU の AI 規制（AI Act）はソーシャルスコアリング・職場での感情認識・犯罪予測等への AI 利用を禁止し、透明性と汎用 AI モデルの著作権対応を義務付ける。規制緩和の提案も進行中のため、最新動向の確認が必要。

### 決済関連: PCI DSS と PSD

| 規制 | 概要 | 準拠のポイント |
|------|------|---------------|
| PCI DSS | カード会員データを保存・処理・伝送するすべての事業者（寄付サイト含む）に適用されるグローバル標準。12 のセキュリティ要件。決済事業者との契約で違反時の罰金が規定される | 多くの場合、自己評価質問票で準拠を検証できる |
| PSD / PSD2 | EU の決済サービス規制（法的義務）。PSD2 は強力な顧客認証（SCA・多要素認証に類似）を重視し、**取引の片側が EU 加盟国に関われば**域外にも適用される | 準拠済み決済プロバイダ（Stripe, PayPal 等）を採用するか、SCA を自前実装する |

## 進化可能性（Evolvability）とフィットネス関数

- 要求は市場とともに変わり続け、新要求は既存実装を脅かす（例: 性能改善のために暗号化を安易に外してセキュリティ全体を毀損する）。
- **進化可能性** = 新しい変更を取り込みながら、既存の機能/横断的要求を支えるアーキテクチャ特性（レイヤー構造・データ永続化方式・保存時/転送時の暗号化等）を保持するシステムの能力。
- 達成手段は「必須のアーキテクチャ特性から逸脱した瞬間にフィードバックを返すガードレール」の構築:
  - 自動テスト群: 性能テスト・セキュリティスキャン・アクセシビリティ監査・アーキテクチャテスト・ミクロ/マクロ機能テスト
  - メトリクス群: コードカバレッジ・静的解析メトリクス
- このテストとメトリクスの総体を**フィットネス関数**と呼ぶ。フィットネス関数に導かれてインクリメンタルに変更することで、既存実装を損なわずに進化するアーキテクチャを作れる。
