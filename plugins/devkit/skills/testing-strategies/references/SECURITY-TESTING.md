# セキュリティテスト実践リファレンス

アプリケーションのセキュリティは「多層防御（defense in depth）」で成り立つ。単一の堅牢な外壁ではなく、複数の層それぞれに防御を組み込む。システムの安全性は最も弱い箇所（内部管理者・漏洩したパスワード・未検証の外部連携など）で決まるため、弱点を能動的に探すことが不可欠であり、その主要手段がセキュリティテストである。本リファレンスは、攻撃者の視点で考え（think like a hacker）、脅威モデリング・シフトレフト・CI 組込みを通じて開発チーム自身がセキュリティを日常業務に組み込むための実践知をまとめる。専門のペネトレーションテスターに委ねる前に、自動化ツールと習慣で多くのリスクを潰せる。

セキュリティバグは開発サイクルの後半で見つかるほど修正コストが跳ね上がる。要件定義の段階からセキュリティ要件（機微データの閲覧制限・多要素認証など）を検討し、分析・開発・テストの全フェーズにベストプラクティスを織り込むことが、シフトレフトの出発点になる。

## 基本用語

| 用語 | 意味 |
|------|------|
| Asset（資産） | 防御機構で守るべきアプリケーションの重要要素（データ・資金・インフラ・サービスアクセス・ブランド信頼） |
| Vulnerability（脆弱性） | 攻撃者が悪用できるシステム上の潜在的な穴 |
| Threat（脅威） | 脆弱性を突かれて生じうる負の事象・行為 |
| Attack（攻撃） | セキュリティ侵害を狙った不正な悪意ある行為 |
| Security compromise（侵害） | 防御機構が資産を守れず突破された状態 |
| Encryption（暗号化） | 鍵を持つ受信者だけが復号できるよう情報を撹拌する技術 |
| Hashing（ハッシュ化） | 任意長データを固定長出力へ写像する一方向技術。わずかな入力差でも出力が変わり、実質復号不能。データ真正性の検証に使う |

## 主要サイバー攻撃タイプ

攻撃はアプリケーション層・インフラ層・ネットワーク層のいずれでも発生し、新種が日々生まれる。代表的な型を押さえ、自分のアプリで同種の弱点を探す起点にする。

| 攻撃 | 概要 | 着目点 |
|------|------|--------|
| Web scraping | スクリプトでサイトを自動巡回し公開データ（特に個人情報）を収集 | 公開・露出データを全体で洗い出す。SNS 系は電話番号・位置情報が狙われやすい |
| Brute force | 総当たりで鍵の全組合せを試行 | 弱いハッシュ（旧世代アルゴリズム）は現代の計算力で突破される。強いハッシュ＋ソルトが必須 |
| Social engineering | 心理操作で機密を聞き出す。近年は音声/動画ディープフェイクで上司・役員を偽装する事例が急増 | 人的プロセス（送金承認・本人確認）に多重チェックを設ける |
| Phishing | 偽メール等で偽サイトへ誘導し認証情報・カード情報を窃取。AI アシスタント経由の間接注入（メール要約時の悪性指示実行）も出現 | リンク・添付の検証、AI への外部入力の扱いに注意 |
| Cross-site scripting (XSS) | 入力の未検証を突きスクリプトを注入し挙動を改ざん（決済情報の外部送信・セッション乗っ取り） | 入力のバリデーション/サニタイズ |
| Ransomware | マルウェアでシステムを封鎖し身代金を要求 | バックアップと復旧手順の整備で被害を封じ込める |
| Cookie forging | ユーザー情報を保持する Cookie を偽造しアカウントへ不正アクセス | Cookie 署名・暗号化、セッション管理 |
| Cryptojacking | 他者デバイスで無断に暗号資産採掘。公開リポジトリを巡回しインフラアクセスキーを収集するボットが典型 | シークレットのリポジトリ混入防止、依存/拡張機能の検証 |
| Man-in-the-middle | 正規サイトと攻撃者の間で通信を傍受・中継 | 全通信の暗号化、セキュアな伝送プロトコル |
| (D)DoS | 大量リクエストでサービスを過負荷・停止させる | 後述の DoS 対策を参照 |

> 攻撃防止は法的責任でもある。データ保護・決済関連の各種規制がセキュリティとプライバシーを義務付けている。

## STRIDE 脅威モデル

STRIDE は考えうる全脅威を洗い出すための脅威分類フレームワーク。以下 6 カテゴリの頭字語で、資産（データ・資金・インフラ・サービス・信頼）への脅威を体系的にブレインストームできる。

| 脅威（S.T.R.I.D.E.） | 内容 | 主な防御策 |
|----------------------|------|-----------|
| **S**poofed identity（なりすまし） | 他者の身元を装い資産へアクセス（social engineering・phishing・malware・shoulder surfing） | 多要素認証、強いパスワード、保存/伝送時の強い暗号化 |
| **T**ampering with inputs（入力改ざん） | コード・データ・メモリを不正に改変し整合性を破壊（コード/スクリプト注入） | 入力バリデーション（SQL クエリ混入防止等）、認証/認可、注入を招くコードの排除 |
| **R**epudiation of actions（否認） | 悪意ある操作を本人まで追跡・証明できない | 十分なログと監査機構（タイムスタンプ付き） |
| **I**nformation disclosure（情報漏洩） | 権限のない主体が資産へアクセス（設計不備による意図せぬ露出・MITM 含む） | 強い認可、シークレット暗号化、セキュアな伝送 |
| **D**enial of service（サービス妨害） | サービスを停止させ収益・信頼を毀損（DDoS） | ロードバランサ、IP 単位スロットリング、IP 許可リスト、バックアップ、自動スケール、急増検知アラート |
| **E**scalation of privileges（権限昇格） | 不正に上位権限（super admin 等）を取得。最も危険 | 最小権限の原則、アクセストークン頻繁更新、取引の多重署名、シークレットの vault 保管 |

脅威は「防止」だけでなく「発生時の影響封じ込め（blast radius 縮小）」も併せて設計する。

## 代表的アプリケーション脆弱性

コードに近い視点で脆弱性を把握すると、テスト観点と防御コードが具体化する。

### Code / SQL injection

入力を無検証で組み立てると、悪意ある入力でクエリ全体を書き換えられる。

```sql
-- 脆弱: ユーザー入力 $name をそのまま連結
SELECT * FROM Students WHERE name = '$name'

-- 攻撃入力:  Alice'; DROP TABLE Students; --
-- 実行される羽目になるクエリ:
SELECT * FROM Students WHERE name = 'Alice'; DROP TABLE Students; --'
```

対策: プレースホルダ/パラメータ化クエリ、入力バリデーション。

### その他の脆弱性チェックリスト

| 脆弱性 | 内容 | 対策 |
|--------|------|------|
| Cross-site scripting | 被害者ブラウザでスクリプト実行→セッション乗っ取り・改ざん。入力未検証時に多発 | 投稿内容のスクリプト検証・サニタイズ |
| 既知の未対応脆弱性 | OS・ライブラリ・フレームワーク等の依存に潜む既知脆弱性を放置 | 依存を定期更新。Dependabot による自動更新、Snyk / OWASP Dependency-Check による脆弱性スキャン |
| 認証・セッション管理不備 | セッショントークン窃取、URL へのセッション ID 露出、非暗号化通信での認証情報送信 | セッション ID/機微データの頻繁な更新と旧 Cookie 無効化、暗号化通信 |
| 私的データの平文放置 | ログ・DB・リポジトリ・ドキュメント・公開サービスに未暗号化で露出 | 強い暗号アルゴリズム（AES・HMAC・Argon2・SHA-256）＋動的な salt / pepper |
| 設定ミス | 全ユーザーへ一律 admin 権限付与など。DB・admin エンドポイントへの不正到達を招く | 最小権限、フォルダ/システム権限の適正化 |
| シークレット露出 | 環境認証情報等をコード・設定・CI/CD に平文でハードコード | vault などシークレット管理サービスから都度取得 |

> OWASP（Open Web Application Security Project）は Web の代表的脆弱性トップ 10 を定期公表している。設計・実装・テストの観点として活用する。

## AI 開発エコシステム特有のリスク

上記脆弱性はコーディングアシスタント・MCP サーバー・チャットボット・AI プラットフォームでより顕在化しやすい。理由は 3 点:

1. **導入の拙速**: AI ツール導入と生産性誇示の圧力で、学習途上のまま過剰なタスク・意思決定を抱える。
2. **広範な機微情報アクセス**: AI ツールはドキュメント・メール・インフラ・CI・リポジトリ・開発端末など広く接続され、同時に外部からの内容/指示にも晒される（外部コンテンツ＋機微データアクセス＋外部通信の組合せが危険な「lethal trifecta」を成す）。
3. **ガバナンス不在**: 内部動作が不透明で監査困難、明確な AI ガバナンスがない組織が多い。

原則は「trust, but verify（信頼するが検証する）」。AI 出力は必ずレビューする。

## 脅威モデリング（フレームワークとしての実施法）

潜在的な全脅威を構造的に集約する手法。**スコープを小さく**（例: 1 ユーザーストーリーあたり 15 分）繰り返すのがベストプラクティス。全ロールが参加するチーム作業とし、対面ならホワイトボード＋付箋、リモートなら共同編集ツールを使う。

### 優先順位付けの鉄則

> 潜在脅威に対処するセキュリティ対策の構築コストは、守ろうとする資産の価値を超えてはならない。

影響度（impact）と発生確率（probability）で優先順位を付ける。例: ブログ基盤への ransomware は低確率のため高額監視への投資は過剰になりうる一方、EC サイトのカード情報を狙う code injection は高影響・高確率で優先対応に値する。

### 4 ステップ

| Step | 内容 |
|------|------|
| 1. 機能の定義 | 対象機能のスコープを定める。ユーザー/アクターとユーザーフローを描き、コンポーネント間のデータフローを図示 |
| 2. 資産の定義 | 守るべき資産を特定し、喪失時の影響とリスク発生確率を議論・記録 |
| 3. Black-hat thinking | 「システムを壊す」発想で攻撃手段を列挙。STRIDE で構造化し、可否判断は保留してアイデアを発散。AI をチームの追加メンバーとして活用 |
| 4. 優先順位付けとストーリー化 | 確率と影響を分析して優先度付け。abuser story として起票しチームが対処できる形にする |

### abuser story（悪意ユーザー視点のストーリー）

脅威を通常のユーザーストーリー内で解決するか、"abuser / evil user" ストーリーを新設する。受入基準からセキュリティテストケースを導出できる。

```
As an abusive user, I cannot inject code to redirect the content of the website.
As an abusive user, I should not be able to see customer details even if I gain access to the database.
As an abusive user, I should not be able to take advantage of open browser sessions.
As a store assistant, I should be the only person authorized to make edit requests to the order service.
```

脅威モデリングは開発サイクルを通じて反復する。新機能の脅威を洗ううちに、既存機能の新たな脅威が見つかることも多い。

### AI 支援脅威モデリング

セキュリティ専門性が不足するチームは、GenAI に自チームのユースケースを与えて脅威の叩き台を生成させると、新しい方向で発想を広げられる。ユーザーフロー図・データフロー図をコンテキストとして渡すのが要点。

```
You're a security expert. When asked to design the threats for a given epic,
follow the steps below:

Step 1: 課題管理ツールから epic の詳細を取得する
Step 2: 技術設計と連携仕様をドキュメントリンクから読む
Step 3: 脅威モデリングガイドを読む
Step 4: 学んだ内容を epic に適用し新たな脅威を導く
Step 5: STRIDE カテゴリ別に列挙し重複させない
```

注意: AI はスコープ外・無関係な脅威を挙げたり、重複・誤分類したりする。追加メンバーとして意見を聞き、プロジェクトに合わせて取捨選択する。

### 脅威モデルからのセキュリティテストケース（層別チェックリスト）

脅威モデリング後、探索的テストで各層のセキュリティテストケースを洗い出す。**zero trust**（境界内であっても人・コンポーネント・要求を信頼せず、OAuth 2.0 の bearer token 等で毎回真正性を検証する）を前提に設計する。

**UI 層**
- セッションタイムアウト後は再ログインを促す
- 規定回数の失敗ログインで認証情報をロックする
- 入力欄が不正入力（JavaScript コード・SQL クエリ）を検証する
- アクセストークンは短期で失効し、セッション継続中は refresh token で更新する
- 権限のないロール（管理者・問い合わせ担当）には編集オプションを表示しない

**API 層**
- 失効アクセストークンの再利用は 401（情報を漏らさぬため 400 も可）を返す
- API パラメータ値を UI 同様に検証し、不合格なら適切なエラーを返す
- 権限外ロールのトークンで編集系エンドポイントを叩くと 401 を返す

**DB 層**
- パスワードは動的 salt 付きハッシュで保存する（公的ガイドライン準拠）
- 機微な顧客情報は暗号化して保存する

**アプリケーションログ**
- パスワードを平文でログ出力しない
- ユーザーの機微情報を平文でログ出力しない
- 管理者操作を含む全操作をタイムスタンプ付きで記録する

## セキュリティテスト戦略（シフトレフト・パイプライン組込み）

開発中のコードの脆弱性についてチームへ継続的フィードバックを与え、素早く修正する。開発→CI→テスト→本番の各段階にツールを配置する。

| 技法 | 概要 | 配置・使いどころ |
|------|------|-----------------|
| SAST（静的解析） | ソース/バイト/アセンブリコードの既知脆弱性（未暗号化シークレット等）を静的解析。プラグイン・ライブラリ・SaaS 形態 | 開発中〜CI で毎コミット実行。多くが AI で誤検知低減・自動修正 |
| pre-commit シークレットスキャン | private key・環境認証情報などをコミット前に検出（厳密には SAST ではないがシフトレフトに寄与） | コミット/プッシュ前フック |
| SCA（構成分析） | サードパーティ依存の脆弱性を特定 | 開発中〜CI。SAST と併用で静的コードの穴を早期に発見 |
| 機能セキュリティテスト自動化 | 機能テスト自動化ツールでセキュリティテストケースを網羅（例: 特定ロールのみ編集可を service test 化） | 通常の機能テストと同様に CI |
| イメージスキャン | コンテナイメージの脆弱性を検査。IaC（Terraform/Kubernetes）にはポリシー適用ツール | CI 統合。レジストリ側スキャンや `docker scan` も活用 |
| DAST（動的解析） | ブラックボックスで攻撃を模した特殊リクエストへの応答を分析（注入脆弱性等） | CI 統合。実行が長いため適切なステージ（夜間回帰/手動トリガ）を選ぶ |
| IAST | SAST と DAST を組合せ、実行時にインストルメンテーションでリアルタイム検出。発展途上 | ランタイム |
| 手動探索的テスト | 脅威モデルから導いた各層のケースを実行。ブラウザ開発者ツール・API クライアントを活用 | テストフェーズ |
| ペネトレーションテスト | アプリの重要度とチーム習熟度に応じ、納期終盤に専門家を関与 | 終盤の最終確認 |
| RASP | 本番環境で期待挙動を学習し、実行時に攻撃を検知・自動防御（悪性プロセス停止・不正ペイロード拒否） | 本番。主に有償製品 |

原則: 高コストなペネトレーションテストを待たず、アクセスしやすい自動化ツールで継続的にフィードバックを得る。

## ツール実践

### OWASP Dependency-Check（SCA）

プロジェクトのライブラリ・外部依存の既知脆弱性をスキャンするオープンソース SCA ツール。CLI・CI プラグイン等で実行できる。

```bash
# インストール例（macOS）
brew install dependency-check

# スキャン実行
dependency-check --project project_name -s project_path --prettyPrint
```

- CI パイプラインに組込み、脆弱性検出時にビルドを失敗させられる。
- 同フォルダに HTML レポートを生成し、検出脆弱性（例: 古い jQuery の XSS）と該当 CVE ID・推奨対応を一覧化する。

### ZAP（DAST）

デプロイ済みアプリに対し DAST を行うオープンソースツール。ブラウザとアプリ間の man-in-the-middle として動作し、やり取りを傍受・改変して既知脆弱性を検出する。Selenium 連携で CI 実行も容易。

```bash
# インストール例（macOS）
brew install --cask zap
```

> 警告: 自分が所有しないサイトに ZAP を使わない。無許可のセキュリティテストは違法。学習には OWASP Juice Shop（意図的に脆弱な練習用サイト）を使う。

**ワークフローの要点**
1. **サイト把握**: `Manual Explore`（手動でユーザーフローを一巡、HUD オーバーレイで DL 不要にブラウザ上から操作）または `ZAP Spider`（自動クロール）でアプリの URL/UI を学習させる。JavaScript が絡む場合は AJAX Spider を併用。
2. **Passive scanning**: 通信を改変せず検査（例: 秘匿すべき private IP の露出検出）。重大度で red/orange/yellow に分類。
3. **Active scanning**: 実際にリクエストを改変して攻撃を模し、既知脆弱性（例: SQL injection）を検出。時間がかかる。

**CI 統合の主要 API**

| API | 役割 |
|-----|------|
| `zap.urlopen(target)` | アプリを開く |
| `zap.spider.scan(target)` | Spider 起動＋passive scan |
| `zap.ascan.scan(target)` | active scan 起動 |
| `zap.core.alerts()` | 結果出力 |

active scan は数時間かかりうるため、夜間回帰ステージやユーザーストーリー単位の手動トリガとして組込む。Selenium テストに ZAP API を埋め込む手もある（WebDriver がログインを代行できる）。

```java
@Test
public void testSecurityVulnerabilities() throws Exception {
    zapScanner = new ZAProxyScanner(ZAP_PROXYHOST, ZAP_PROXYPORT, ZAP_APIKEY);
    login.loginAsUser();

    zapSpider.spider(BASE_URL);              // Spider
    zapScanner.setEnablePassiveScan(true);   // passive scan
    zapScanner.scan(BASE_URL);               // active scan（適切な wait を追加）

    List<Alert> alerts = filterAlerts(zapScanner.getAlerts());
    logAlerts(alerts);
    assertThat(alerts.size(), equalTo(0));   // 脆弱性があれば失敗
}
```

HTML レポートを CI の成果物として保存できる。GitHub Actions では ZAP Baseline Scan / Full Scan Action で容易に統合でき、issue も起票される。公式ドキュメント: https://www.zaproxy.org/docs/

### Snyk プラットフォーム（統合）

SCA・SAST・DAST・イメージスキャン・IaC スキャンを SDLC に統合できる AI 駆動の開発者向けセキュリティプラットフォーム。優先度スコア付け、根本原因分析、AI による自動修正、IDE 拡張、CLI、Web UI、MCP サーバーを備える。AI エージェント構築時は skill や MCP サーバーなどエージェント構成要素の既知脆弱性も検査できる。無償版で個人リポジトリ連携を試せる。

### Talisman（pre-commit フック）

コミット時にアプリコードをスキャンし、シークレットや機微情報を検出して警告するオープンソースツール。pre-commit / pre-push フックとして設定でき、シークレットの誤コミットを防ぐ。

```
$ git commit
Talisman Report:
+----------------+-------------------------------------------------+
|     FILE       |                    ERRORS                       |
+----------------+-------------------------------------------------+
| sampleCode.pem | ファイル名が ^.+\.pem$ パターンに一致            |
| sampleCode.pem | hex エンコードされた awsSecretKey= を検出        |
+----------------+-------------------------------------------------+
```

シークレットを狙ってリポジトリを巡回するボットが存在するため重要度が高い。

### ブラウザ開発者ツール・API クライアント（手動探索）

- **API クライアント**: リクエストの Authorization に bearer token を設定し、改ざん済み/失効トークンなどのシナリオを検証できる。
- **ブラウザ開発者ツールの Security タブ**: ページが HTTPS で正しく配信されているか、サードパーティリソースが非セキュアに読み込まれていないか（MITM の温床）を確認できる。サードパーティ Cookie 制限の挙動観察も可能。

## 落とし穴・アンチパターン

| アンチパターン | リスク | 是正 |
|----------------|--------|------|
| 露出データの放置 | scraping・漏洩の起点 | 公開/露出データをアプリ全体で棚卸し |
| 弱いハッシュ・ソルト無し | brute force で突破 | 強アルゴリズム＋動的 salt / pepper |
| 一律 admin 権限 | 権限昇格・不正到達 | 最小権限の原則を徹底 |
| シークレットのハードコード | リポジトリ流出→悪用 | vault から都度取得、pre-commit スキャン |
| セッション ID の URL 露出・非暗号化送信 | セッション窃取 | 暗号化通信・トークン短命化 |
| 依存の更新放置 | 既知脆弱性の悪用 | 自動更新＋SCA を CI に常設 |
| AI 出力の無検証採用 | 幻覚コード・脆弱コードの混入 | 必ずレビュー（trust, but verify） |
| 監査ログ不足 | 否認・追跡不能 | 全操作をタイムスタンプ付きで記録 |
| セキュリティを終盤に一括 | 修正コスト増・積み残し | シフトレフト＋CI で継続フィードバック |
| 無許可サイトへの DAST 実行 | 違法 | 自所有サイト/練習用サイトに限定 |

## セキュリティを習慣にする

どれだけ堅牢に作っても、セキュリティを習慣化しなければ想定外の弱点が残る。チームの日常に潜む侵害要因を毎日問い直す:

- 開発/テストツールがプロジェクトデータをベンダーの私的クラウドに保存していないか
- 環境詳細を含むアーキテクチャ図を外部サイトで作成していないか
- 本番認証情報をチャットに平文共有していないか

## Key Takeaways

- サイバー犯罪は想像以上に蔓延し、AI 導入がリスクを加速させている。
- セキュリティ対策は SDLC 全体を通してアプリに組み込む（多層防御）。
- STRIDE は脅威探索の構造化レンズであり、脅威モデリングに適用できる。
- 小さなスコープ（ユーザーストーリー/機能単位）で全ロール参加の脅威モデリングを頻繁に行い、abuser story とセキュリティテストケースを生む。
- SAST・SCA・DAST の自動化ツールに、手動探索的テストと機能自動テストを組合せてシフトレフトする。
- アクセスしやすい自動化ツールにより、ペネトレーションテストを待たずフィードバックを得られる。
- 何より、セキュリティを習慣にする。
