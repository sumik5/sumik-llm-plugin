# 自動化と開発環境の習熟（AUTOMATION-AND-ENVIRONMENT）

自動化は認知負荷を減らし、作業の信頼性を高める最も即効性の高い投資である。本リファレンスは「自動化を習慣にする」「ワークスペース・端末・エディタ・開発マシンを自分の道具として所有する」実践をカバーする。

---

## 1. 自動化＝信頼性の根幹

手動の手順は「知っている人が手順を覚えている間だけ」機能する。自動化された手順は誰がいつ実行しても同じ結果を生む。認知負荷の削減と、ヒューマンエラーの排除が自動化の二大効果である。

> **基本原則**: 繰り返し実行する作業は、初回に実行コストをかけてでも自動化する。2回目以降はそのコストを回収し、3回目以降は利益になる。

---

## 2. Automate First の5能力チェックリスト

新規プロジェクトを開始するとき、アプリケーションコードを1行も書く前に以下の5能力を整備する。先延ばしにするほど「後でやればいい」という惰性が強まり、デプロイは常に手つかずになる。

| 能力 | 確認事項 |
|------|---------|
| **① 再構成（Reconstitute）** | バージョン管理からプロジェクト全体を一から復元できるか |
| **② 依存インストール** | 依存パッケージを単一コマンドで一括インストールできるか |
| **③ ビルド** | ソースから成果物を単一コマンドでビルドできるか |
| **④ テスト実行** | テストスイート全体を単一コマンド（またはビルドの一部）で実行できるか |
| **⑤ デプロイ** | ターゲット環境へ単一コマンドでデプロイできるか |

チェックリストが5項目すべて✅になってから、機能コードを書き始める。

## 3. Day Zero デプロイ

プロジェクト初日（= Day Zero）に、"Hello, World!" 相当のアプリをターゲット環境へデプロイする。この時点では設定すべきものはほとんどなく、デプロイスクリプトをシンプルに保てる。

**Day Zero デプロイの効果:**

- データベース・キャッシュ・キューなど新しいミドルウェアを追加するたびにプロビジョニングスクリプトを更新する習慣が身につく
- 「全機能を同時に動かす」ような複雑な初回デプロイが発生しない
- ステークホルダーが常に動くものを触れるため、早期フィードバックが得やすい

### 単一コマンドデプロイとは

「アプリを更新するだけ」の単一コマンドでは不十分。新しい素の環境を渡されたとき、OS設定・依存インストール・サービス起動・アプリデプロイ・データ移行・設定投入まで一括で完了できることが目標である。

```
# 概念的な構成例（擬似コード）
deploy-all:
  provision-os          # OS・ミドルウェア設定
  install-dependencies  # 言語ランタイム・パッケージ
  migrate-data          # スキーマ・初期データ
  start-services        # DB/cache/queue 等
  deploy-app            # アプリ本体
  verify-health         # ヘルスチェック
```

定期的に（週1回程度）、ターゲット環境を素の状態にリセットし、単一コマンドで復旧できるかを確認する。それが「早朝障害」への最善の備えになる。

> **組織デプロイパイプライン（CI/CD の全体設計・ブランチ戦略・承認フロー）については、`practicing-software-engineering` を参照。本リファレンスは個人の自動化習慣に焦点を当てる。**

## 4. ワークスペース管理

複数の仮想デスクトップ（ワークスペース）を用途別に分割し、一度に一つの文脈だけを表示する。

**ワークスペース分割の例:**

| ワークスペース | 内容 |
|--------------|------|
| メール・チャット | 通知対応専用。コーディング中は非表示 |
| プロジェクトA | エディタ・ターミナル・ブラウザ（開発用）を配置 |
| プロジェクトB | プロジェクトAとは完全に分離 |
| リサーチ | ドキュメント・参考資料のブラウザ |

文脈スイッチのコスト（再集中に15〜30分かかる）を物理的に防ぐことで、深い集中状態を維持しやすくなる。ワークスペースを切り替えるとき「前の状態がそのまま残っている」安心感も生産性に寄与する。

## 5. 端末とシェルの習熟

「端末＝表示レイヤー・シェル＝処理能力」という役割分担を意識する。

### 5.1 端末評価の観点

端末エミュレータを選ぶ際に確認すべき特性：

| 観点 | なぜ重要か |
|------|-----------|
| **レンダリング速度** | ログやビルド出力を快適にスクロールできるか |
| **スクロールバック行数** | ビルドエラーや長いログを遡れるか（2000行以上推奨） |
| **複数ペイン** | エディタ・サーバー・テスト結果を同時表示できるか |
| **キーマッピング** | エディタと同一のキーシーケンスで操作できるか |
| **テーマ** | 長時間の作業で目が疲れないか |

複数ペインは特に重要。サーバーログを片隅に流しながらコードを書き、別ペインでテストを実行する運用が可能になる。端末マルチプレクサ（例: tmux）が不要なら、端末アプリ側のペイン機能で代替する選択肢もある。

### 5.2 シェル評価の観点

| 機能 | 実践上の価値 |
|------|------------|
| **コンテキスト補完** | ディレクトリごとに補完候補が変わる。ワークスペース切り替え後の認知負荷を減らす |
| **ヒストリ検索** | 過去コマンドをインクリメンタル検索し、長いコマンドを再入力しない |
| **スクリプト言語** | 複雑な処理をワンライナーや短いスクリプトに落とし込める |
| **プロンプトカスタマイズ** | カレントディレクトリ・ git ブランチ・終了コードを一目で把握できる |

> **シェルスクリプトの実装詳細（変数展開・制御フロー・エラーハンドリング・テスト）については、`lang:developing-bash` を参照。**

## 6. エディタの所有

エディタは「脳からコードへの最短経路」である。入力の遅延・不要なコンテキストスイッチ・マウス操作への依存はすべて摩擦になる。

### 6.1 エディタを育てる原則

1. **必要なものだけ追加する** — 使わない拡張は認知の重りになる。インストール後に使わなかった拡張は削除する
2. **段階的に習得する** — 新しい拡張を入れたら、次の拡張を入れる前にそのキーバインドを体に染み込ませる
3. **マウス依存を減らす** — キーボードだけでの操作を試みる。手がホームポジションを離れる頻度が下がるほど思考が中断されにくい

### 6.2 エディタに最低限求める機能

| 機能 | 確認事項 |
|------|---------|
| 言語サポート（LSP 等） | 補完・定義ジャンプ・型情報が表示されるか |
| ファジーファインダー | ファイル名・シンボル名でインクリメンタル検索できるか |
| フォーマット統合 | 保存時または手動で整形できるか |
| テキストオブジェクト | 関数・ブロック・段落などの単位で選択・操作できるか |

## 7. 開発マシンのプロビジョニング

「開発マシンが突然使えなくなったとき、どれくらいで元の生産性に戻れるか」という問いへの答えが、プロビジョニング自動化の完成度を示す。

### 7.1 プロビジョニングの二要素

| 要素 | 管理方法 |
|------|---------|
| **作業ファイル** | バージョン管理（git）でリモートへ push。失っても再 clone で復旧 |
| **ツール＆設定** | パッケージマネージャー＋ dotfiles 管理ツールで自動復旧 |

### 7.2 ツール管理のポイント

- インストールしたツールはパッケージマネージャーのリストファイル（例: `Brewfile`・`requirements.txt`・`.tool-versions` 相当のもの）で記録する
- 「なぜこのバージョンか」を示すためにバージョンロックをする
- パッケージリストはバージョン管理に含め、マシン移行時に再インストールできるようにする

```
# 概念的な初期化スクリプト（擬似コード）
bootstrap-dev-machine:
  install-package-manager   # Homebrew / apt / winget 等
  install-from-list         # パッケージリストからツール一括インストール
  restore-dotfiles          # git から設定ファイルを展開
  verify-versions           # 各ツールのバージョン確認
```

### 7.3 設定（dotfiles）管理のポイント

- シェル設定・エディタ設定・gitconfig・その他ツール設定を一元管理する
- dotfiles 管理ツール（GNU Stow / yadm / Chezmoi 等）を使うか、シンプルに git リポジトリ＋シンボリンクでも構わない
- 「新しいマシンで一つのコマンドを実行するだけで環境が整う」状態を目標にする

> **本番サーバーのプロビジョニング・IaC（Terraform / Ansible 等）については、`cloud:practicing-devops` を参照。**

## 8. 実践チェックリスト

### プロジェクト開始時

- [ ] 5能力チェックリスト（§2）を実装してからアプリコードを書く
- [ ] Day Zero デプロイを完了させる

### 週次

- [ ] ターゲット環境を素の状態にリセットし、単一コマンドでの復旧を確認する

### 環境セットアップ

- [ ] ワークスペースを用途別に分割した
- [ ] 端末のペインを活用して複数の文脈を同時表示できる
- [ ] シェルの補完とヒストリ検索を使いこなしている
- [ ] エディタの拡張は「実際に使うもの」だけに絞っている
- [ ] ツールのインストール履歴をパッケージリストに記録している
- [ ] dotfiles をバージョン管理し、新マシンへの展開を自動化している
