# 複雑さを削る（CUTTING-COMPLEXITY）

> **要旨**: 複雑さには「本質的な難しさ」と「自分たちが招いた複雑化」の2種類がある。後者を継続的に削るための実践ループと評価ルーブリックを示す。シンプリシティは手順ではなく、繰り返しで身につく精神だ。

---

## 1. complexity vs complication — 削れるのは complication だけ

ソフトウェアには避けられない**complexity（本質的複雑さ）** がある。問題領域が複雑であれば、コードもある程度複雑になる。これは設計で管理するものだ。

一方、**complication（自己誘発の複雑化）** は開発チームが自ら持ち込む。ルールがなく、予測がきかない。典型的な発生源:

- 不要な依存の追加
- 誰も求めていない機能
- 早期の抽象化・過度な汎用化
- 「なんとなく便利そう」で増えた設定項目
- 説明できないまま残った特殊ケース

complication は意図的に取り除ける。complexity は取り除けないが、整理して理解を深めることはできる。

---

## 2. シンプリシティとは何か

「シンプル」は「単純すぎる」「ナイーブ」とは異なる。シンプルなコードは:

- 理解しやすい
- 変更しやすい
- 「しっくりくる」

最後の感覚は主観的だが、無視してよい理由はない。「しっくりこない」という直感は、問題の予兆として機能することが多い。

> シンプリシティは **手順ではなく精神** だ。誰かの手順をそのまま模倣しても、別の文脈では機能しない。自分の文脈で Orient-Step-Learn を繰り返すことで身につく。

---

## 3. Orient-Step-Learn ループ

複雑さと向き合う実践的なサイクル。

### Orient（気づく）

**目的**: 「ここが複雑すぎる」という箇所を意識的に発見する。

- 明確なサイン: 「このコードを誰も理解できない」「触るのが怖い」
- 微妙なサイン: なんとなく違和感がある、レビュー中に言葉が出てこない

**問いかける**:
- どうすれば簡潔になるか？
- 「簡潔になった」とわかる基準は何か？

この問いかけを続けることで、複雑化の予兆を早期に察知する感覚（ソフトウェア版の危機察知）が育つ。

**定着のヒント**: 違和感を声に出す。「ここがしっくりこない」と口にするだけで意識が引き起こされる。

---

### Step（小さく試す）

**目的**: 仮説を検証する最小の実験を行う。

| NG | OK |
|----|-----|
| 「まず全体をリファクタリングしよう」 | 「この関数だけ別ファイルに切り出して動作を確認する」 |
| 「依存をすべて入れ替えよう」 | 「この一か所だけ直接実装してみて、コードが減るか見る」 |

**コストが低いほどよい** — 間違っていても損失が少なく、素早くフィードバックが得られる。

**実験の設計原則**:
1. 変更範囲を1つに絞る
2. 判断基準（改善/改悪の定義）を先に決める
3. 既存テストが通ることを確認する

---

### Learn（学ぶ）

**目的**: 実験から何を得たかを整理し、次に活かす。

- うまくいった: 次の候補を同じアプローチで試す
- うまくいかなかった: 仮説を修正して別の Step を設計する

**1回のサイクルで解決しなくていい**。積み重ねることで、判断の精度が上がる。

**記録のすすめ**: 短いメモ（「今日気づいた複雑さ」「試したこと」「結果」）を習慣にすると、パターンが見えてくる。

---

## 4. S vs C 評価ルーブリック

「S（候補 A）は C（候補 B）より簡潔か」を体系的に判断するための3観点。絶対的な正解はなく、文脈によって変わる。

### 観点1: 理解しやすさ

| チェック項目 | S の方が有利であれば ✓ |
|------------|----------------------|
| S のほうが素早く全体像を把握できる | |
| S の説明に使う文が短い | |
| S を読むときに立ち止まる回数が少ない | |
| S を変更することへの自信が高まる | |

### 観点2: 部品の少なさ

| チェック項目 | S の方が有利であれば ✓ |
|------------|----------------------|
| コンポーネント数・依存数・条件分岐数が少ない | |
| 同じ結果をより少ないステップで達成できる | |

### 観点3: 問題の直接反映

| チェック項目 | S の方が有利であれば ✓ |
|------------|----------------------|
| 問題領域の用語・構造に忠実 | |
| 余分な機能・重複・ノイズが C より少ない | |

### 「しっくりくるか」という感覚の扱い

数値では測れないが、「これで大丈夫か」という安心感の有無は判断の一部として認めてよい。ただし感覚だけで判断せず、上記3観点とセットで使う。

**簡潔さは絶対値ではない** — 自分にとっての簡潔さは他者にとってそうでないことがある。また、去年の自分には難しく感じたものが今は自然に読めることもある。判断は文脈依存であることを前提にする。

---

## 5. 精神として身につける

シンプリシティを実現する普遍的な手順はない。文脈が異なれば「正しい選択」も変わる。

できることは:

1. **Orient-Step-Learn を繰り返す** — 続けることで複雑化の察知が速くなる
2. **他者のやり方を参考にするが、コピーしない** — 自分の文脈で考え直す
3. **違和感を無視しない** — 「なんか変」は多くの場合、正しい

時間をかけてシンプルにすることは、後のすべての作業を速くする先行投資だ。「今は複雑化を受け入れるしかない」という状況でも、その複雑さを意識的に把握しておくことが次の削減につながる。
