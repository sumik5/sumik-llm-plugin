# コラボレーションとソフトスキル（COLLABORATION-AND-SOFT-SKILLS）

チームの摩擦はそのままシステムの複雑さに転化する。時間的結合の除去・会議の最小化・スキルの拡散・弁証法的思考・共感・コードへの傾聴・アナロジーによる伝達、これらはシンプリシティを集団で守るための習慣群である。

---

## 1. 時間的結合の除去（チームの非同期化）

コードの結合と同様に、チームメンバー間にも「相手を待たなければ次に進めない」依存関係が発生する。これを**時間的結合（temporal coupling）**と呼ぶ。時間的結合が多いほど変更は遅くなりチームは脆くなる。

### 1.1 待ちの棚卸し

1週間、作業中に「誰かを待った時間」をカテゴリ別に記録する。

| カテゴリ例 | 解消候補の手段 |
|-----------|-------------|
| コードレビュー待ち | レビュー待ち中に並行タスクへ切り替えられるか検討 |
| 技術的な質問の回答待ち | 共有 Q&A ツール・ドキュメント化で非同期化 |
| 会議の招集待ち | Kanban ボード・タスクリストで状態を可視化 |
| 依存するコードの完成待ち | インターフェースをモック化して並行開発 |

1ヶ月後に「実験前」「実験後」の待ち時間を比較し、効果を測定して調整する。

### 1.2 非同期化の基本手段

- **タスクボード（Kanban 等）**: カード状態を更新するだけで他者へ意図を伝達できる
- **非同期コードレビュー**: レビュー後に内部不整合を修正する前提で並行作業
- **共有カレンダー・ドキュメント**: 会議なしで状態を共有できるものは会議にしない

> **チーム構造そのもの（Team Topologies・スプリント設計・アンチパターン）については、`practicing-software-engineering` を参照。**

## 2. 会議の問題と対処

会議は「全参加者の時間×参加人数」を一挙に消費する。10人参加の5分会議は50分の作業時間を失う。それに加え、集中の中断と再集中コストが加わる。

### 2.1 会議の4つの問題

| 問題 | 説明 |
|------|------|
| **非効率** | 開始遅延・脱線・オーバーラン・準備不足の参加者への説明コスト |
| **不公平** | 発言力が立場や積極性に偏る。静かな参加者の意見が失われる |
| **中断コスト** | 集中状態（フロー）を断ち切る。終了後の再集中に15〜30分かかる |
| **コスト大** | 頻繁な全員会議は人件費換算で大きな負荷になる |

### 2.2 DWP（Developer Without Portfolio）ロール

成功しているチームで独立して発見されてきた役割の一つ。DWP の唯一の責務は「チームをスムーズに動かすこと」であり、チームというエンジンにとっての潤滑油として機能する。

**DWP の主な活動:**

- チームを巡回し、各メンバーの状況・詰まり・成功体験を把握する（ペアプログラミング、隣に座ってのリスニング等、方法はその人の個性による）
- 「問題を抱えている人」と「その問題を解決できる人」を結びつける
- デイリースタンドアップの代替として、DWP が常時状況を把握しているため、全員を止めた一斉確認が不要になる
- レビュー会議前に観察内容をまとめて事前配布し、会議を「問題解決」に集中させる
- 設計会議では権限はなく、全体像を把握している立場から代替案の提示などを担う（設計の決定権を持たない）

DWP が1人加わることで節約できる会議時間は、エンジニア1人分以上の稼働に相当するケースがある。実験として3ヶ月前後の開発者センチメントを含む指標を収集し、効果を検証する。

### 2.3 会議を行うなら作法を守る

会議が避けられない場合、主催者は以下の責任を持つ。

**① 明確な目標を設定する**

目標は「測定可能」でなければならない。会議終了時に「達成したか」を検証できる形にする。

| NG目標 | OK目標 |
|--------|--------|
| リリースについて議論する | 次バージョンのリリース日と担当者を確定する |
| 推定精度を改善する | 前回の見積りミスを1件選び、再発防止策を1つ決める |

**② 会議が必要かを確認する**

「自分がもう少し準備すれば会議なしで解決できないか」を正直に自問する。共有カレンダーや非同期ツールで解決できる場合は会議を組まない。

**③ 反転会議（Flip Meeting）**

従来: 会議で内容を提示し、会議後に各自作業する  
反転: 事前に資料を配布して各自で把握させ、会議では課題解決に集中する

事前資料を準備する過程で問題を誤解していたことに気づき、会議が不要になるケースも多い。

**④ 参加者を最小にする**

必要な意思決定に欠かせない人のみを招集する。大きいほど議論は発散する。参加候補者に資料を送り「出席が必要かどうか」を確認してもらうとよい。

**⑤ 時間帯への配慮**

集中の妨げになりにくい時間帯（午前最初・自然な区切りの直後）を選ぶ。タイムゾーンをまたぐ場合は早起きの負担を交互に分担する。

**⑥ 進行する**

定刻に開始し、遅刻者への再説明はしない。測定可能な目標を全員で確認し、議論を目標へ戻し続ける。全員が発言できるよう、独占的な発話を穏やかに制限する。

**⑦ まとめとフォローアップ**

会議末尾に合意事項を読み上げ、書面化する。終了後に全参加者へ議事録と次のアクションを送付する。

**探索型会議は別ルール**

アイデアを収集・発散させる場合（初期ブレスト・未知領域の探索等）は、上記の整理ルールは適さない。事前資料なし・アジェンダなし・外部ファシリテーターの活用なども有効。探索型と意思決定型を混同しない。

> **会議過多のアンチパターン検出については、`practicing-software-engineering` を参照。**

## 3. スキルの拡散

教えることは学ぶことでもある。スキルを共有するとスキルは減らず、教える側・学ぶ側の双方が得る（ポジティブサム）。チームの多様なスキルは変化に対する回復力でもある。

### 3.1 スキル拡散の手法カタログ

| 手法 | 概要 |
|------|------|
| **ペアプログラミング** | ペアを定期的にローテーションし、全員が全員と組む期間を作る |
| **DWP の仲介** | 「詰まっている人」と「解決できる人」を DWP が結びつける |
| **勉強会・ランチセッション** | 自分の発見・調査結果を共有する定期的な場 |
| **チームシャッフル** | 数ヶ月ごとにメンバーを別チームへ一時移動させ、知識と視点を持ち帰る |
| **クロスチームコードレビュー** | チーム外のメンバーにレビューを依頼し、暗黙知を可視化する |
| **モブコーディング** | 複数チームを1日合流させ、共通課題（ビルド改善・ライブラリ更新等）を協働で解決する |
| **スキルの混在チーム編成** | 補完的なスキルを持つメンバーを意図的に混在させる |
| **経験レベルの多様化** | 経験差を活かしてメンタリングと逆メンタリング（新人から学ぶ）を共存させる。ただし差が大きすぎると「直感で動く熟練者」と「手順を求める初学者」の間で対話が困難になる |
| **探索時間** | 週に半日程度の業務外プロジェクト時間を設け、言語・ツール・領域を横断した探索を奨励する |

知識を溜め込むことは腐敗する。共有された知識は相手の既存知識と化学反応を起こし、元の知識より大きな何かになる。

> **ペアプログラミングのパターン・知識サイロの検出については、`practicing-software-engineering` を参照。**

## 4. 弁証法的思考（Dialectical Reasoning）

意見の相違は「どちらかが勝つゼロサムゲーム」ではない。両者の立場に部分的な正しさがある前提で、より良いものを合成することを目的にする。この枠組みを**弁証法的思考**と呼ぶ。

**なぜシンプリシティに関係するか:**

「絶対に正しい」と確信している人が多いチームは、変化への適応コストが高い。根拠のない確信を守るために余分なエネルギーが消費され、より良い折衷案を生む機会も失われる。

**実践の手順:**

1. 相手の意見が「なぜその結論に至ったか」を、論駁のためではなく**理解のために**聞く
2. 「自分が間違っている可能性」を本気で探す（「もしかしたら自分の経験が特殊だったのかもしれない」という姿勢）
3. 双方の立場から正しい部分を抽出し、それを組み合わせた第三の案を作る
4. 「勝ち負け」ではなく「合成」を目標にする

一つの技法として、議論の途中で意見を強制的に入れ替える練習（賛成側が反対を、反対側が賛成を論じる）がある。反対の立場を論じることで、自分の固定観念の限界に気づける。

## 5. 共感（Empathy）

共感とは「相手がどう感じているかを理解し、それを行動に反映させる能力」である。同意する必要はなく、相手の立場を考慮・尊重することが本質。

### 5.1 共感がシンプリシティに貢献するしくみ

共感があると、言葉で詳細を説明しなくても相互理解が進む。これにより：

- 試行錯誤のコストが下がる（小さいステップで進められる）
- 失敗時に非難ではなく学習に向かう
- 感情的な安全網があるため、実験と修正サイクルが速くなる

逆に共感がないと：作業がサイロ化し、決定が絶対化し、失敗の反応が非難になる。

### 5.2 共感を育てる小さな実践

| 実践 | 説明 |
|------|------|
| **相手に時間を与える** | 情報交換を目的としても、人として向き合う時間を先にとる。急いでいる素振りを消す |
| **感情を言語化する** | 自分の状態（疲れている・集中したい等）を自発的に伝える。相手が誤読するのを防ぐ |
| **相手の気持ちを推測して確認する** | 「それは大変でしたね」のような一言で、相手が理解されていると感じる |
| **黄金律** | 「自分がされたいようにする」。単に無礼を避けるのではなく、積極的に気遣う。小さな親切を意識的に実施する |

共感は「個人が実践し、チームが増幅する」ものである。

## 6. 環境への共感・コードに耳を傾ける

「物に共感する」とは、道具や環境を自分の意思に屈服させようとするのではなく、それぞれの性質・制約・動作原理を理解して協調することである。

### 6.1 環境を敵にしない

道具が思い通りに動かないとき、まず「正しい使い方をしているか」「道具の制約に対して適切なアプローチを選んでいるか」を疑う。力で押し切ろうとすると、壊れるか余計な複雑さが生まれる。

### 6.2 コードに耳を傾ける

書きながら感じる違和感を無視しない。

**違和感のシグナル:**
- 「これはもっとシンプルになるはずだ」という感覚
- 同じ理由で変数やクラスを何度も名前変更している
- キーボードを打つ手が重い（書くのが億劫に感じる）
- 席を立つ頻度が増えている

これらは「コードが問題を教えている」サインである。その感覚を認識したら、立ち止まって「なぜこれは難しく感じるのか」を自問する。

### 6.3 デバッグの視点転換

バグは「コードが間違えた」のではなく「コードが指示通りに動いている」。  
「どのコードが悪いか」を探すのではなく「このシステム全体としての動作を引き起こしているのは何か」を問う。データと制御の流れを追い、患者を診る医師のように仮説を立てながら観察する。

直感が「何かおかしい」と告げるとき、その感覚を軽視しない。人間のパターン認識は意識に上らない情報を処理している。散歩・休憩・別作業によって意識を緩めると、潜在的に気づいていたことが浮かび上がりやすくなる。

## 7. アナロジーで語る

技術的な詳細を知らない相手に情報を伝えるとき、正確な説明より**伝わる説明**を優先する。アナロジー（比喩・類推）は精度を一部犠牲にして、相手の既存語彙にマッピングする技法である。

### 7.1 アナロジー作成の4ステップ

| ステップ | 問い |
|---------|------|
| **① 伝えたい情報** | この状況で何が重要で、何は省略してよいか |
| **② 相手の語彙** | 相手がすでに知っているどんな概念を使えば伝わるか |
| **③ マッピング** | 元の概念と相手の語彙を対応づけられる類推はないか |
| **④ 検証** | 類推で失われる情報が、今回の文脈で許容できるか |

**例: データベースのインデックスを非技術者に説明する場合**

| 実際の概念 | アナロジー（書籍の索引） |
|-----------|----------------------|
| DB インデックスで特定レコードを高速に検索できる | 索引ページで知りたいキーワードのページ番号を素早く見つけられる |
| DB インデックスはストレージを消費する | 索引ページは紙面を消費する |
| インデックスが多いほど検索軸が増える | 索引の種類が多いほど探し方が増える |

アナロジーは「完全に正確」である必要はない。重要なのは「受け手が次の意思決定に必要な理解を得られること」。

### 7.2 アナロジーに使うドメインの選び方

**使いやすいドメイン:**
- 両者が日常的に経験しているもの（通勤・料理・図書館・郵便など）
- 話題を持ち出すのに背景説明が要らないもの

**避けるべきドメイン:**
- 片方しか詳しくない専門分野（スポーツ・趣味・地域特有の文化）
- 政治・宗教・社会問題
- 長い前置きが必要な設定

### 7.3 アナロジーの本質

アナロジーが機能するのは、元の問題の**構成要素どうしの相互作用**が、類推先でも**同様に相互作用している**場合である。要素の列挙だけではなく、要素間の関係が類推先でも成立するアナロジーを選ぶ。

アナロジーの練習方法:
- **非同期で練習**: メールの返答でアナロジーを試み、「わかりやすかったか」をフォローアップする
- **日常の場で練習**: 身近な人への技術説明は、プレッシャーの低い練習機会になる
- **自分への類推**: 設計に詰まったとき、問題をアナロジーの世界で語ることで解決の糸口が見えることがある

> **文章・説明の一般技法（構成・論証・文体）については、`writing-effective-prose` を参照。**

## 8. 実践チェックリスト

### 非同期化

- [ ] 直近1週間の「待ち時間」をカテゴリ別に把握した
- [ ] 各カテゴリに非同期化の手段を対応づけた

### 会議

- [ ] 会議の目標は測定可能な形式で設定している
- [ ] 事前資料を配布し、会議では解決に集中している（反転会議）
- [ ] 参加者を「意思決定に必要な人」に絞っている
- [ ] 終了後に議事録と次のアクションを全参加者へ送っている

### スキル拡散・弁証法・共感

- [ ] チームで使っていないスキル拡散の手法を1つ試している
- [ ] 意見の相違に直面したとき「合成」を目標に探索している
- [ ] 相手の状況を推測しながら関わっている

### コード・ツールへの傾聴

- [ ] 書きながら感じた「これは難しすぎる」という感覚を無視していない
- [ ] バグを調査するとき、システム全体として問いかけている
