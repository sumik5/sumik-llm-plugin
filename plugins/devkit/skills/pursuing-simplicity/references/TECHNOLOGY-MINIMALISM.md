# 技術ミニマリズム（TECHNOLOGY-MINIMALISM）

> **要旨**: 依存・フレームワーク・機能・新技術はいずれも「将来の負債」になり得る。追加前の意思決定チェーンと、保守的な技術採用の原則を示す。「追加しない」を最初の選択肢として考えることが簡潔なコードベースを保つ基本だ。

---

## 1. 依存の意思決定チェーン

依存を追加する前に、次の5問を順番に確認する。

### 問1: その機能は本当に必要か？

「あると便利そう」「将来使うかも」は需要ではない。現在のプロジェクトが動作するために**今すぐ必要か**を問う。

- No → 追加しない。サイドプロジェクトや学習セッションで試せば十分。
- Yes → 次の問いへ。

---

### 問2: 自分で書けるか？

必要な機能が小さければ、自前実装のほうが依存を増やすより簡潔なことが多い。

**判断の目安**:

| 状況 | 推奨 |
|------|------|
| 1〜2関数で実装できる規模 | 自作を優先する |
| オープンソースで利用可能な短いコード | ライセンス確認のうえ取り込みを検討 |
| 複雑なプロトコル実装（暗号・ネットワーク等） | 実績ある依存を使う |

---

### 問3: 付随する依存が過大ではないか？

欲しい機能だけを取り出せない場合、不要な依存が大量についてくることがある。使わないコードは潜在的な脆弱性の温床となり、将来の保守コストを押し上げる。

> 「バナナ1本が欲しいのに、ジャングルごと届いた」というエラーに気づいたら、より小さな代替ライブラリや自作を再検討する。

---

### 問4: ローカルコピーや隔離が必要か？

プロジェクトに不可欠な依存であれば:

- **バージョンを固定する** — バージョン管理ツールで依存ツールのバージョンも含めて固定する。
- **ラッパー関数で隔離する** — コードベース全体から直接使うのでなく、薄いラッパーを作り、API変更の影響範囲を1ファイルに抑える。
- **ローカルキャッシュを考慮する** — 外部レジストリからの取得が途絶えてもビルドできる体制を検討する。

---

### 問5: セキュリティ監査をしているか？

依存追加は、外部のコードをプロジェクトの実行環境に招き入れる行為だ。

**実践チェック**:

- [ ] 依存のメンテナが活動的かを確認した
- [ ] ライブラリリポジトリの脆弱性スキャン機能を有効にしている
- [ ] セキュリティ警告を定期的にレビューしている

**四半期ごとの棚卸し**:

```
1. 全依存を一覧表示する
2. 不要になったものを削除する
3. セキュリティアップデートを適用する
4. 直接依存のうち自作できるものがないか再評価する
```

---

## 2. 依存と向き合う姿勢

依存は追加した瞬間から維持コストが発生する。特に依存が依存を持つ場合、1つの追加が連鎖的な更新作業を引き起こすことがある。

**意図的な依存選択のチェックリスト（現在実践）**:

- [ ] 単一の関数のためだけに依存を追加していない
- [ ] 自分で書けるものを外部委託していない
- [ ] 開発中は定期的（週次・月次）に依存を更新している
- [ ] 追加前にメンテナの信頼性を調べた

**改善したい項目（継続目標）**:

- [ ] 依存のセキュリティ状況を自動監視するサービスを導入する
- [ ] ラッパーによる隔離を一貫して実施する
- [ ] 四半期に1回、旧プロジェクトをビルドして依存の健全性を確認する

## 3. フレームワーク選定の原則

フレームワークはアプリケーションを囲い込む。一度採用すると変更コストが高いため、選定時の判断が後々まで影響する。

### フレームワーク肥大化のリスク

フレームワークは時間とともに機能を積み上げる傾向がある。

- 新機能追加のたびに未使用コードが増える
- 破壊的変更で既存コードの書き直しを迫られる
- 機能追加を「改善」として歓迎するチームが不要な移行作業を増やすことがある

**シグナル**: フレームワークの保守に費やす時間が、アプリケーション本体の実装時間を超えたら見直しのサイン。

---

### フレームワーク選定の3原則

1. **安定性** — 長期間メンテナンスされており、破壊的変更の頻度が低い
2. **実績** — 本番環境での採用事例が十分にある
3. **必要十分な最小機能** — 機能数が最多ではなく、自分たちの用途に過不足がない

**FBQ（Framework Bloat Quotient）自問**:

> 現在使っているフレームワークの機能一覧を開いたとき、実際に使っているのは何割か？

多くのチームは10〜20%程度しか利用していないと報告する。使っていない機能は、将来の脆弱性・複雑さの潜在的な源だ。

---

### フレームワーク選定フロー

```
1. 用途を明確化する（API サーバー / SSR / CLI / ...）
2. 用途を満たす最小機能のフレームワークを列挙する
3. 実績・メンテナンス状況・依存数で絞り込む
4. 選んだフレームワークの「使わない機能」の割合を見積もる
5. 過半数が不要なら、より小さい代替を探す
```

## 4. 機能の抑制：Need-Driven Development

機能は追加するたびに将来の負債が増える。コードは書かれた瞬間から保守・理解・拡張の対象になるためだ。

> **機能＝将来の負債** という視点を持つ。「あると便利」は需要ではない。

### 追加すべき機能の判断

追加を正当化する基準:

| 状況 | 判断 |
|------|------|
| 特定のユーザー・利害関係者が今すぐ必要としている | 追加を検討する |
| 「いつか必要になるかも」 | 追加しない |
| 「競合が持っているから」 | 実際のユーザー需要を確認してから判断 |
| 「実装が簡単だから」 | 保守コストは実装コストとは別物——判断基準にしない |

---

### インクリメンタルな価値提供

機能を一括して最後に届けるより、価値の高いものから順番に届けることで複数の利点が生まれる。

**早期提供の効果**:

| 効果 | 説明 |
|------|------|
| 早期の価値実現 | 今の価値は将来の同じ価値より常に大きい |
| 早期フィードバック | 実際に使った人の反応は文書上の要件より正確 |
| 信頼の構築 | 動くものを届けることでステークホルダーの安心感が高まる |
| 手戻りの防止 | 誤解が早期に発覚すれば修正コストが低い |
| 機能の自然な削減 | 早期に使えるようになると、不要だった追加予定機能が自然に見えてくる |

**機能と複雑さの関係**:

```
機能数が増えると複雑さはそれ以上のペースで増加する。

  機能数: 1 → 2 → 3 → ...N
  複雑さ: 1 → 3 → 6 → ...N(N+1)/2 の勢いになることもある

```

「全機能を届けた完成品」より「中心機能を素早く届けたプロダクト」のほうが、チームにもユーザーにも良い結果をもたらすことが多い。

## 5. 保守的な技術採用

新しい技術・ライブラリ・ツールは魅力的に映る。しかし本番環境に投入することと、探索・学習として試すことは別だ。

### 「未来で遊び、過去で働く」原則

| フェーズ | 内容 |
|---------|------|
| **研究・探索（未来）** | 新技術を試す。プロトタイプを作る。失敗してよい空間 |
| **本番実装（過去）** | 実績ある・枯れた道具を使う。安定性を最優先にする |

**プロトタイプ＝偵察隊**: 本番に投入する前に小規模な検証を行い、「使えるか」「デメリットはないか」を確認する。プロトタイプは基本的に捨てる前提で書く。

---

### 新技術採用の判断フロー

```
新技術が気になった
    ↓
研究リストに追加する（今すぐ採用しない）
    ↓
研究時間を使って調査・プロトタイプを作る
    ↓
評価する:
  - 本番での実績はあるか？
  - コミュニティ・メンテナンスは安定しているか？
  - 既存の枯れた代替より明確に優れているか？
    ↓ Yes
採用を検討する
    ↓ No / 不明
研究リストに残し、再評価を待つ
```

**「気になったとき」の対処**: 現在のプロジェクトには組み込まず、研究リストにメモしておく。興味がある限り研究時間で試す機会は必ず来る。

---

### 研究時間の確保

新技術を探索する専用の時間を定期的に設けることで:

- 本番コードの安定性を保ちながら技術変化に追従できる
- 「使ったことがない」への不安が減り、合理的な採用判断ができる
- 学習の楽しさを維持し、日常業務の単調さを和らげる

**目安**: 週あたり30分〜数時間。形式よりも継続性が重要。
