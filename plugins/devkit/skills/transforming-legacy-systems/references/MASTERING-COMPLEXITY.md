# 複雑性の統御

> ソフトウェア開発における複雑性がどこから来て、どの部分は削れて、どの部分は削れないのかを整理する。レガシーシステムの変革は、この区別を誤ると「治療のつもりで別の病気を作る」ことになりやすい。

## 1. 複雑性の起源：問題空間と解決空間

複雑性がどこで生まれるかを考えるとき、まず「問題空間」と「解決空間」を分けて捉える。

| 空間 | 内容 | この空間固有の複雑性 |
|------|------|---------------------|
| 問題空間 | ソフトウェアが対象とする現実の業務ドメインそのもの | **問題由来の複雑性**：ドメインが本来持つ複雑さ |
| 解決空間 | モデリング・アーキテクチャ・設計・採用技術など、実装として作り込む部分 | **解決依存の複雑性**：実装の選択によって生まれる複雑さ |

要件を厳密に見積もり、開発の初期段階で仕様として固定するアプローチは、実務では長続きしない。プロジェクトが計画より長引く、ウォーターフォール型の進め方がうまく機能しない、といった事態の背景には、たいてい「問題空間の複雑性を最初にすべて言語化できる」という誤った前提がある。反復的な開発スタイルが広く支持されるのは、問題空間の複雑性を小さな単位で少しずつ捉え、実装と絡めながら理解を深めていけるからだ。レガシーシステムの変革も同様に、小さなステップで進める。変革の各ステップは新機能開発のステップと絡み合わせることができ、要件が変化し続けるという現実に対応しやすくなる。

## 2. 複雑性の種類：本質的複雑性と偶発的複雑性

複雑性には、もう一つの軸として「本質的（essential）」と「偶発的（accidental）」の区別がある。

### 本質的複雑性

**本質的複雑性**は、対象の性質そのものに内在する複雑さであり、どれほど設計が優れていても消し去ることはできない。複雑なドメインを扱うなら、解決空間の複雑さもそれに見合って高くなるのが自然であり、それ自体は問題ではない。指針はシンプルで、「できる限り単純に、ただしそれ以上単純にはしない」ことである。

たとえば、映画館の座席予約システムを考えると、上映回・座席の空き状況・会員ランクごとの割引・複数チケットの同時予約・払い戻しポリシーなどを同時に扱う必要があり、この複雑さはドメインの性質そのものに由来する本質的複雑性である。単に会員の名前と連絡先を記録するだけの名簿システムとは、要求される複雑さの水準が最初から異なる。

### 偶発的複雑性

**偶発的複雑性**は必然ではなく、次のような要因から生まれる。

- **ドメインの誤解**：分析が不十分なまま実装し、実際より複雑な（あるいは不適切な）解決策を作ってしまう
- **ドメインの変化への追随不足**：昨日は適切だった設計が、業務の変化によって今日はそぐわなくなる
- **使われない機能の実装**：開発者が「利用者はこう働くはずだ」と思い込み、実際には必要とされない支援機能を作り込む
- **設計・アーキテクチャの劣化**：不適切な意思決定、あるいは保守・変更・無理解の積み重ねによる経年劣化
- **技術選定のミスマッチ**：古くなった技術、目的に合わない技術、あるいは想定外の使い方をしている技術

学習・経験・慎重な検討が不足した状態で開発が進むと、シンプルな解決策が見つからないままシステムは不必要に複雑化する。重複実装、不要な機能追加、設計原則の軽視などがその典型であり、なかには「新しい技術を試したいから」という理由だけで導入される、意図的な偶発的複雑性も存在する。

## 3. アーキテクチャの意思決定領域

複雑性を扱いやすくするため、意思決定の領域を次の4つに分けて考える。

```
              問題空間                        解決空間
        ┌────────────────┐           ┌────────────────┐
  上段   │  要件分析         │           │                  │
(本質+偶発)│  ドメインモデリング  │           │                  │
        └────────────────┘           └────────────────┘
  下段   │                  │           │ ドメインアーキテクチャ │
(本質+偶発)│                  │           │ 技術アーキテクチャ    │
        └────────────────┘           └────────────────┘
```

要件分析とドメインモデリングは問題空間側で、本質的複雑性と偶発的複雑性の両方に働きかける。ドメインアーキテクチャと技術アーキテクチャは解決空間側で、同じく両方の複雑性に影響する。4つの領域それぞれに固有の手法・技法があり、複雑性の発生源に応じて使い分ける。

## 4. 本質的・偶発的複雑性への対処

4つの意思決定領域の手法を正しく適用すれば、偶発的複雑性を大きく削減できる。ただし完全に取り除くことはできない。ソフトウェア開発・保守は継続的な学習の過程であり、開発者はドメインや技術の一部を誤解し続けるし、部分的には不十分な解決策を実装し続けるからだ。ドメイン自体が時間とともに変化することも、偶発的複雑性が増える要因になる。業務側の変化に加えて、ソフトウェアが実際に使われることで利用者が新しい業務のやり方に気づき、それがさらなる変化を促すこともある。かつて最適だった設計は次第に精度を失い、ドメインの概念や業務プロセスが洗練されるにつれて偶発的複雑性が忍び込む。

> **陥りやすい罠**：「このドメインはよく知っているから簡単だろう」と決めつけ、見慣れた要求だけを基準に単純な（しばしばCRUDだけの）解決策から着手してしまうこと。設計が不適切だと気づいた頃には手戻りのコストが大きくなっており、偶発的複雑性が染み付いた状態になっている。

複雑性を減らすための3つの手法（協働モデリング／DDD／アーキテクチャ概念）は、この意思決定領域にそれぞれ対応する。

```
問題空間 ─────────────────────────────────────▶ 解決空間

 協働モデリング          ドメイン駆動設計（DDD）        アーキテクチャ概念
 （要件分析を担う）        （ドメインモデリング〜         （技術アーキテクチャの
                          ドメインアーキテクチャを担う）    健全性を担う）
```

DDDは技術的な実装の詳細を指定せず、問題空間のドメインについても「代わりに考える」ことをしない。あくまでドメイン知識を深く理解するための道具と考え方の集合であり、それが幅広い適用可能性と息の長さにつながっている。

## 5. レガシーシステムにおける複雑性

長年にわたり複数世代の開発者・アーキテクトの手を経てきたレガシーシステムには、偶発的複雑性が蓄積しやすい。主な理由は次のとおりである。

- 利用者の業務とドメインは年月とともに変化してきたが、ソフトウェアの**ドメインモデリング**とアーキテクチャ上の**ドメイン駆動のモジュール分割**が、その変化に十分追従できていない
- 組織のIT戦略や、ITシステムの位置づけ自体が変化してきた。かつては再利用性によるスケールメリットが重視されコストセンターと見なされていたが、現在はスピードによる競争優位や戦略的資産としての側面が重要になっている
- 時間的制約・理解不足・関心の欠如により、当初の設計・アーキテクチャが徐々に薄まり、各部分が絡み合って相互依存するようになった
- 多くのレガシーシステムは、境界づけられたコンテキストに基づくドメイン固有の概念に整理されるのではなく、大きな一枚岩のドメイン概念を中心に構築されてきた
- あらゆる技術は10〜15年ほどで陳腐化し置き換えが必要になるが、レガシーシステムではその適切なタイミングを逃しがちである

ドメイン駆動設計は本来、新規開発における「これから作るシステムをどうドメイン駆動にするか」という問いに答えるものだったが、レガシーシステムの再設計・再構築にも同じ考え方が有効であることが実務を通じて確かめられてきた。鍵になるのは、DDDと他の道具を組み合わせた、地に足のついた段階的なモダナイゼーションである。

## 変革プロセスとのつながり

この複雑性の区別は、変革の進め方そのものを左右する。ドメイン層が偶発的複雑性で埋め尽くされた状態（大きな一枚岩のクラスに業務ロジックが詰め込まれている等）のまま境界づけられたコンテキストへ分割しようとすると、分割後の各コンテキストにも同じ混乱がそのまま複製されるだけになりやすい。そのため、ドメイン層が構造を持たない状態であれば、先に技術変革・戦術変革で偶発的複雑性を削ってから戦略変革（4ステップ）に進む方が安全であり、逆にドメイン層にある程度の構造がすでに見えているなら、先に境界を引いてから内部を磨く順序でも成立する。この判断がMMI経路セレクター（[SKILL.md](../SKILL.md) §3）の考え方の土台になっている。
