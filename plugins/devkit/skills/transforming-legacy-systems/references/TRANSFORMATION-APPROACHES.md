# 変革アプローチの選択（Transformation Approaches）

レガシーシステムをどう変革するかは、大きく「置き換える」か「今のシステムを活かしたまま作り直す」かの二択に分かれる。置き換えにはさらに一括切替と段階的切替の2種類がある。本ファイルでは3つのアプローチ（ビッグバン置換・段階的置換・リシェイピング）を共通の6評価軸で比較し、選定指針を示す。

---

## 1. 3つのアプローチ

| アプローチ | 概要 |
|-----------|------|
| **ビッグバン置換** | 旧システムを稼働させたまま、新システムをゼロから並行して構築する。新システムが完成した時点で一斉に切り替える |
| **段階的置換** | 旧システムの一部分を少しずつ新システムに置き換え、完成した部分から順次本番投入する。旧機能は置き換えが済んだ分だけ順次停止する |
| **リシェイピング（改造）** | システムを置き換えず、稼働させながらリファクタリングで内部構造を作り直す |

段階的置換は、旧システムを新システムが徐々に包み込み最終的に置き換える様子から、比喩的に「絞め殺しの木（Strangler Fig）型」と呼ばれることがある。この呼称・技術的な実装パターン（ルーティング切替・並行稼働・データ同期戦略等）の詳細は `cloud:architecting-infrastructure` のマイグレーションパターンへ委譲し、本ファイルでは「3アプローチのどれを選ぶか」の意思決定に絞る。

---

## 2. 6つの評価軸

3アプローチはいずれも一長一短があり、次の6つの軸で比較すると差が明確になる。

| 評価軸 | 何を見るか |
|--------|-----------|
| 未知の未知（unknown unknowns） | 見落としている機能・シャドーIT・想定外の使われ方をどの程度早く発見できるか |
| 停滞と動く標的 | 旧システムの改善が止まる停滞感と、新システム側の要件が動き続ける負荷 |
| 二重作業 | 法改正対応やバグ修正を新旧両方に実装する手間 |
| ノウハウ喪失 | 旧システムの開発者が意欲を失い離脱するリスク |
| 遅いデプロイ | 完成まで本番投入されず、フィードバックが遅れるリスク |
| ユーザー不満 | 利用者が変化にどれだけ早く・小さく適応できるか |

### 比較表

| 評価軸 | ビッグバン置換 | 段階的置換 | リシェイピング |
|--------|--------------|-----------|--------------|
| 未知の未知 | 全機能を一度に分析する必要があり発見コストが高い | 置換対象部分ずつ発見が進み、後続の置換ほど精度が上がる | リファクタリング対象を触るたびに発覚するが範囲は限定的 |
| 停滞と動く標的 | 旧システムの改修を止める間に新要件が積み重なり、完成が遠のく | 旧システムに触れ続けるため改修が完全には止まらない | 常に同じシステムに手を入れるため進捗が体感しやすい |
| 二重作業 | 法改正・重大バグは新旧両方に反映が必要で負担が大きい | 置換中の部分のみ二重対応が必要で負担は限定的 | 十分小さい単位でリファクタリングできれば二重作業はほぼ発生しない |
| ノウハウ喪失 | 新規開発から外れた開発者の離脱リスクが高い | 担当のローテーションで緩和しやすい | 新しい技術を都度取り込むため知識更新が進みやすい |
| 遅いデプロイ | 完成まで本番投入されずフィードバックが遅い | 完成した部分から順次本番投入されフィードバックが早い | 改修した部分から順次本番投入されフィードバックが早い |
| ユーザー不満 | 全機能が一度に変わり、使用感の落差が大きい | 少しずつ変化するため慣れる時間がある | 見た目の変化は小さく、新技術導入時のみ影響が出る |

**総評**: ビッグバン置換は6軸すべてで最も不利になりやすい。段階的置換とリシェイピングはいずれも「小さく進めてフィードバックを早める」という共通の強みを持つ。リシェイピングの弱点は、既存の設計判断そのものを問い直す契機が生まれにくいことにある（後述）。

## 3. 各アプローチの実務上の注意点

### 3.1 ビッグバン置換

一見「新システムをきれいに作り直せる」という理想論に見えるが、実務では次の理由でしばしば失敗する。

- 旧システムには長年にわたり多数のチームの手が入り、標準的な業務フロー以外に**特殊業務・稀にしか実行されないバッチ処理・想定外の使い方**が積み重なっている。これらを事前に洗い出すコストは非常に高い。
- 旧システムのエクスポート機能を使って利用者が独自に構築した**シャドーIT**（表計算・簡易データベース等）は、IT部門の管理外にあり見落とされやすい。
- 開発が長期化するほど、旧システムに積もる新規要件との差が拡大し、当初の計画より作り直す範囲が膨らむ。

これらの理由から、既定ではビッグバン置換は推奨しない。

### 3.2 段階的置換

旧システムの一部を新システムへ置き換え、置き換えた部分から本番投入する。置換のたびに未知の未知が段階的に発見されるため、後続の置換ほど見積もりの精度が上がる。旧システムに触れ続ける必要があるため、旧システムの改修が完全に止まる停滞感を避けやすい。担当チームを境界づけられたコンテキスト単位で再編し、旧・新それぞれの保守をローテーションすれば、ノウハウ喪失も抑えられる。

### 3.3 リシェイピング

旧システムに蓄積された業務知識を捨てずに活かしたい場合に選ばれる。稼働させたままリファクタリングで内部構造を改善するため、安全網として**リファクタリング対象部分の特性テスト（characterization test）を先に用意する**ことが前提になる（詳細は `TECHNICAL-STABILIZATION.md`）。

最大の弱点は、既存の設計をそのまま前提にしてしまい、「そもそも別の作り方の方が適切ではないか」という抜本的な問い直しの機会が生まれにくいことにある。この弱点を補うのが、変革プロセス全体（MMI評価→4ステップ戦略変革）である。リシェイピングを選んでも、目標アーキテクチャを都度モデリングし現状と突き合わせることで、抜本的な設計見直しの契機を作り出せる。

### 3.4 サブシステム単位での組み合わせ

システム全体を1つのアプローチで統一する必要はない。実務では、サブシステムごとに評価軸への当てはまりが異なるため、複数アプローチを組み合わせるのが一般的である。

| サブシステムの性質 | 適したアプローチの傾向 |
|-------------------|----------------------|
| コアドメイン（差別化の源泉・業務ロジックが複雑） | リシェイピングで既存の業務知識を守りながら段階的に改善する |
| 汎用サブドメイン（認証・決済等、既製品やSaaSで代替可能） | 段階的置換で外部サービスへ切り替える |
| 使用頻度が極端に低い機能・利用者がごく少数の機能 | 当面は現状維持し、優先順位付けの対象から外す |

この組み合わせ判断は、変革ステップの後半（優先順位付けと実装）で扱う内容と直結する。

## 4. 選定指針

- **既定は段階的置換またはリシェイピング**。ビッグバン置換は上記の理由から原則非推奨とし、安易に選ばない。
- 旧システムを技術的に刷新したい（言語・プラットフォームの移行を伴う）場合は段階的置換が選びやすい。既存の実装・業務知識をそのまま活かしたい場合はリシェイピングが選びやすい。
- 実際には、サブシステムごとに異なるアプローチを組み合わせることも多い（例: コアドメインはリシェイピング、周辺の汎用機能は段階的置換で外部サービスに委譲）。
- 判断に迷う場合は推測で進めず、関係者に確認する。

### AskUserQuestion 実装例

```
AskUserQuestion(questions=[{
  "question": "このレガシーシステムの変革アプローチはどれが最適ですか？",
  "header": "変革アプローチ",
  "options": [
    {"label": "段階的置換", "description": "旧システムを少しずつ新システムで置換。既定・低リスク・早期フィードバック"},
    {"label": "リシェイピング", "description": "旧システムを稼働させたままリファクタリングで改造。ノウハウ温存。抜本的な作り直しはしない"},
    {"label": "ビッグバン置換", "description": "新システムを完成させ一斉切替。未知の未知・遅いフィードバックのリスク大。原則非推奨"}
  ],
  "multiSelect": false
}])
```

AskUserQuestion非対応環境（Codex CLI等）では、同じ3択を通常のテキスト質問として提示する。
