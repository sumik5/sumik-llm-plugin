# 戦略ステップ1-2: ドメイン再発見とターゲットアーキテクチャのモデリング

戦略的変革の4ステップ手法のうち、前半2ステップを扱う。ステップ1では既存システムを一旦忘れ、業務ドメインを純粋な形で再発見してサブドメインを見つける。ステップ2では、そのサブドメインから理想の境界づけられたコンテキスト群（ターゲットコンテキストマップ）をモデリングする。両ステップとも、まだ実際のソースコードには踏み込まない「仮想的なグリーンフィールド」上の作業である。

---

## 0. 4ステップ手法における位置づけ

```
評価（MMI） → [Step1 ドメイン再発見] → [Step2 目標アーキのモデリング]
                                              → Step3 現状とのすり合わせ → Step4 優先順位付けと実装 → （反復）
```

- Step1・Step2 = 本ファイル
- Step3・Step4 = `STRATEGIC-STEPS-ALIGNMENT-EXECUTION.md`
- 各ステップは共通の構造（Actors / Substeps / Tools / Results / Goals / Challenges）で記述する。この構造自体が「変革プロセスをチームに説明するときのテンプレート」として再利用できる。

## 1. Step 1: ドメイン再発見（Domain Rediscovery）

### 1.1 概要

| 要素 | 内容 |
|------|------|
| Actors | ドメインエキスパート／開発チーム／ドメインモデラー |
| Substeps | 1.1 現状（As-Is）の把握 → 1.2 ドメイン知識の抽出（純化） → 1.3 業務プロセスの最適化（To-Be） → 1.4 サブドメインの発見 |
| Tools | ファシリテーション資材（EventStormingキット等）／オンラインホワイトボード等の協働ツール／プロセスモデリングツール |
| Results | As-Is / To-Be の業務プロセス、サブドメイン一覧 |
| Goals | ドメイン知識を前面に引き戻す。既存システムに引きずられた「解決策の埃」を払い落とす |
| Challenges | 適切なドメインエキスパートの招集／ドメインエキスパートを実際のドメインへ導く／作業項目で分解しない／適正サイズを見つける |

ドメインモデラー・ドメインエキスパート・開発チームがワークショップに集まり、協働モデリングの手法（ドメインストーリーテリングやEventStorming等。詳細は `COLLABORATIVE-MODELING.md`）を使って以下を明らかにする。

- **How**: ドメインエキスパートがどう働いているか（業務プロセス）
- **What**: 何を対象に働いているか（作業対象・ドメイン概念）
- **What**: それを何と呼んでいるか（ドメイン言語）
- **Why**: なぜその作業が必要か

最初から細部に踏み込まず、中粒度で全体像を掴むことを優先する。議論が細部に寄りすぎたら「一度俯瞰しよう」と引き戻す進行が要る。

### 1.2 Challenge: 適切なドメインエキスパートの招集

「ソフトウェアを改善したいだけなのに、なぜドメインエキスパートの時間を割くのか」という反応はよくある。しかし業務側の再分析を省くと、既存ソフトウェアが前提にしてしまった暗黙の制約ごと引き継ぐことになる。既存ソフトウェアを意識せずに問題領域（何を解決すべきか）へ立ち返るには、ドメインエキスパートの説明が不可欠である。

### 1.3 Challenge: ドメインエキスパートを実際のドメインへ導く

問題領域にも解決領域にも、本質的な複雑性と偶有的な複雑性がある（詳細は `MASTERING-COMPLEXITY.md`）。長年使われてきたシステムほど、「なぜこの作業をするのか」が「業務上必要だから」なのか「システムがそう要求するから」なのか区別しづらくなる。「システムがそう要求するから」は取り払うべき解決策側の埃であり、これを剥がして業務の本質に到達するのがドメイン再発見の目的である。ドメインモデラーを介在させることで、長年の間に生じた「システム都合の思い込み」を洗い出しやすくなる。

### 1.4 Substep 1.1: 現状の把握

分析初期のドメインエキスパートは、既存のIT系システムに強く紐づいた形でしか業務を語れないことが多い。この段階では無理に理想を語らせず、まず「今、システムを使ってどう仕事をしているか」を粗い粒度でモデル化する（デジタル化されたAs-Is）。開発チームとドメインエキスパートの間で密なやり取りを重ね、共通理解を作る。開発者は問題を見ると即座に解決策を考えたがる傾向があるため、ドメインモデラーは「まだAs-Isの段階だ」とチームを踏みとどまらせる役割を担う。

### 1.5 Substep 1.2: ドメイン知識の抽出（純化）

レガシーシステムを分解する際は、ソフトウェアを登場人物に含めない「純粋な」業務プロセスに立ち返るのが最も効果的である。As-Isプロセスから次の「デジタル的な要素」を取り除く。

- ソフトウェアシステムそのもの
- システムを満足させるためだけに存在し、業務上の根拠がない手順

#### 技法: 思考実験「紙のプロセス」

プロセスをIT支援なしで完全に「紙だけ」で行うとしたらどうなるかを想像する。各手順（ドメインストーリーの各文、EventStormingの各イベント）について次を問う。

- ITがなかった時代、この作業はどう行われていたか
- 今も手作業でこの業務を行っている組織があるとしたら、どうしているか
- 紙ベースなら、この手順は本当に必要か

例えば「システムがメールを送る」という手順が出てきたら、「紙ではできない」と反応しがちだが、そこで踏みとどまって「なぜメールを送るのか、その中身は何か」を掘り下げる。「注文確認を送る」ことが目的だとわかれば、「システムがメールを送る」ではなく「注文確認を伝える」という業務本位の表現に置き換えられる。

#### 技法: 思考実験「仮想アクター」

プロセスを完全に明示化するために、現実には存在しないアクターを導入する手法。「注文確認を伝える」という手順から「システム」という語を消すには、「昔、システムがない時代は誰が注文確認を伝えていたか」を問う。「それは時間がかかりすぎて誰もやっていなかった」という答えが出れば、「もし時間もコストも度外視するなら、誰がその役目を担うか、その役割を何と呼ぶか」を重ねて問う。こうして生まれた「確認担当者」のような仮想アクターを使って手順を書き換える。仮想アクターは、後にサブドメインの輪郭を示す最初の手がかりになることが多い。

> 現場事例: ある安全性検証用コンパイラの開発チームは、当初「人でなく機械の振る舞いを対象にした手法は合わないのでは」と懐疑的だったが、実際にモデリングを進めると、人のアクター（設備技師・自動化技術者）に加え「パーサー」「レキサー」「配信基盤」のような仮想アクターが浮かび上がった。これらの仮想アクターが後の境界づけられたコンテキストの輪郭を示す指標になった。

#### 代替: 最初から純粋にモデリングする

ワークショップ参加者が問題領域と解決領域を自然に区別できる場合（協働モデリングの経験がある場合など）、デジタル化されたAs-Isのモデリングを省略し、最初から純粋なプロセスをモデリングしてよい。その際も上記の質問技法は補助的に使える。

#### ユビキタス言語の蒸留

ドメイン知識の抽出と並行して、ユビキタス言語の整理を進める。次の観点に注意する。

- 同じ用語が違う意味で使われている箇所はどこか
- 同義語だが実は同じ意味を指している用語はどこか
- 業務言語が技術用語に汚染されている箇所はどこか

ここで明確にした言葉の違いは、後述のサブドメイン発見（Substep 1.4）で効いてくる。

### 1.6 Substep 1.3: 業務プロセスの最適化

As-Isプロセスから解決策由来の要素を取り除いたら、To-Beプロセスへ最適化する。これまでの手順で見えてきた兆候をもとに、次の観点で議論する。

- As-Isプロセスに冗長な手順はないか
- その役割は本当に必要か
- 手順自体を減らせるより良い方法はないか
- モデリング中に見つかった「シャドーIT」（非公式な運用の補完手段）を、システムに正式に組み込むべき artifact に置き換えられないか

### 1.7 Substep 1.4: サブドメインの発見

As-Isから始め、業務の核心へ純化し、To-Beへ最適化したプロセスをもとに、サブドメインを見出す。中心的な問いは次の2つである。

- 業務プロセスの中で、何がひとまとまりとして結束しているか
- 業務プロセスのどこで区切りが生まれるか

手法ごとに、この問いは次のように言い換えられる。

| 手法 | 言い換え |
|------|---------|
| ドメインストーリーテリング | アクターの視点で、どの活動がひとまとまりか。どの活動が新しいサブプロセスを開始するか |
| EventStorming | どのイベント列がひとつながりか。どのイベントが新しいイベントの流れを起点にするか（ピボットイベント） |
| シナリオキャスティング | オリエンテーションシナリオのどの地点でミッションが完了し、サブプロセスが確定するか |

#### サブドメイン境界の指標

次の指標が、業務プロセスからサブドメインの境界を見出す手がかりになる。

| 指標 | 説明 |
|------|------|
| 完了した作業成果物 | ある部分プロセスで作業対象が仕上げられ、完成した成果物として次のサブプロセスに引き渡される |
| 成果物の一方向の受け渡し | 完成した成果物が後戻りなく次のサブプロセスへ渡される（共同作業ではなく順次処理） |
| ドメインエキスパートのグループ | あるサブプロセスに精通した専門家グループが存在し、他のサブプロセスは外側からしか知らない |
| ドメイン用語の定義の違い | 同じ用語が異なるサブプロセスで異なる意味を持つ |
| 作業対象の扱い方の違い | 同じ名前の対象が、サブプロセスごとに異なる操作をされる |
| リズムとトリガーの違い | サブプロセスが異なる周期・異なる契機で実行される |

例えば、契約成立後に与信担当へ引き渡す業務（成約の完了・一方向の引き渡し）、営業担当と与信担当という異なる専門家グループ、「契約」という言葉に対する解釈の違い（営業側は交渉可能なパラメータの集合、与信側は承認/却下の対象という不変の情報）、月次の支払いサイクルと都度発生する営業サイクルの違い、といった具合に、これらの指標は同時に複数現れることが多い。

### 1.8 Challenge: 作業項目で分解しない

サブドメインの境界を引く際、最も避けるべき反パターンが「作業項目（エンティティや集約）を軸に分解すること」である。特にレガシーシステムでは、既存の実装に引きずられて「このエンティティを扱うサブドメイン」という単位で区切りたくなる誘惑が強い。

例えば、ある業務プロセスを「週間スケジュール」を扱うサブドメインと「座席表」を扱うサブドメインに分けてしまうと、ある担当者が週間スケジュールから座席表を開こうとするたびにサブドメイン境界をまたぐ矢印が発生する。これをそのままソフトウェアに落とすと、次のような設計になりやすい。

```
[番組編成コンテキスト]        [チケット販売コンテキスト]
   WeeklyScheduleService        SeatingPlanService
        ▲                              │
        └──────────呼び出し────────────┘
   （UI層は両方のサービスを横断して呼び出す必要がある）
```

この設計では、UI層が複数コンテキストのサービスを横断して呼び出す必要が生じ、`SeatingPlanService` が `WeeklyScheduleService` を直接呼び出す密結合も発生する。ビッグボールオブマッドを解消するどころか、サービス指向の見た目を被せて温存してしまう。

業務プロセスが示す指標（成果物の受け渡し・専門家グループ・用語の違い等）に沿って境界を引き直すと、「番組編成」と「チケット販売」のように独立して動けるコンテキストになる。番組編成が完了した時点で、確定した週間スケジュールと座席表をドメインイベントとして非同期にチケット販売へ渡せば、両者は疎結合を保てる。

### 1.9 Challenge: 適正サイズを見つける

サブドメインへの分割を進める前に、細かく分けすぎる傾向に注意する。各サブドメインは、1つのチームが担当できる程度に独立した業務プロセスの単位であるべきである（戦略的設計における「サブドメイン」、および Team Topologies の認知負荷の考え方と符合する）。

#### 適正サイズの目標

- **独立したストリームアラインドチーム**: 互いに疎結合な境界づけられたコンテキストに対して、チームが独立して働けること。1コンテキストにつき1チームが理想だが、重要度に差があるため、現実には1チームが2〜3コンテキストを担当することも多い。それ以上に増やすと開発・運用のオーバーヘッドが目立ち始める
- **独立したリリース**: 境界づけられたコンテキストを個別にリリースできること。ある技術的・法的・業務的な理由でまとまって変更・公開せざるを得ないコンテキスト群が見つかったら、暗黙の結合がある証拠であり、粒度が細かすぎる可能性が高い。統合するか、同じチームに割り当てるべきである

#### 適正サイズの指標

サブドメインへの分割時には、その単位が次を満たすか自問する。

- 職能横断・ストリームアラインドなチームが高品質なソフトウェアを届けられるか
- そのチームがソフトウェア・アーキテクチャ・ビジネスを進化させられるか
- そのチームが業務の複雑さを十分に理解できるか

この助言に従うと、最初に見積もったサブドメインは1チームには大きすぎることが多い。しかし、変革プロセスの初期段階ではあえて大きめのサブドメインから始め、後の反復で必要に応じて分割する方が、最初から細かく割りすぎるより安全である。ドメイン駆動の変革は一度きりのイベントではなく、繰り返し行われる継続的なプロセスだからである。

### 1.10 AIとの議論

業務プロセスがEventStormingの記録やドメインストーリーのファイルのように機械可読な形で保存されていれば、ソースコードと合わせてAIアシスタントに渡し、境界の議論相手にすることもできる。ただし結果は一様ではなく、AIも「作業項目で分解する」という同じ罠に陥ることがある点には注意が必要である。

## 2. Step 2: ターゲットアーキテクチャのモデリング（Modeling the Domain-Driven Target Architecture）

### 2.1 概要

| 要素 | 内容 |
|------|------|
| Actors | ドメインモデラー／アーキテクチャアナリスト／開発チーム／ドメインエキスパート |
| Substeps | 2.1 コンテキストマップの作成 → 2.2 サブドメインの分類 → 2.3 境界づけられたコンテキストのチーム割当 → 2.4 関係性の決定 |
| Tools | ファシリテーション資材／オンラインホワイトボード等の協働ツール |
| Results | ターゲットコンテキストマップ |
| Goals | 業務視点で妥当な境界づけられたコンテキストの導出／チームの決定／コンテキスト間インターフェースの明示化 |
| Challenges | ソリューション空間由来の境界づけられたコンテキスト／チーム割当は往々にしてチーム再編を意味する |

Step1でサブドメインを発見したら、Step2ではそれをもとに「将来、システムがどうあるべきか」を定義する新しい成果物、コンテキストマップを作る。これも依然としてグリーンフィールド上の作業であり、既存のソースコードとの照合はStep3で行う。コンテキストマップは一度作って終わりではなく、ドメイン理解の深化に応じて更新され続けるスナップショットである。

### 2.2 Substep 2.1: コンテキストマップの作成

Step1で見つけた各サブドメインを、そのままコンテキストマップ上の境界づけられたコンテキストへ写像するところから始める。複数のドメインストーリーやEventStormingセッションから見つかったサブドメイン群を1枚のコンテキストマップに集約すると、ドメイン全体を俯瞰する粗い見取り図が得られる。

### 2.3 Challenge: ソリューション空間由来のコンテキスト

業務ドメインから一対一で導出される、正真正銘のドメイン駆動な境界づけられたコンテキストから着手するのが基本である。それによってドメインの本質的な複雑性をシステムに正しく映せる。そのうえで、ソフトウェアが使われることそのものに起因するコンテキスト（=偶有的な複雑性を持つコンテキスト）にも目を向ける必要がある。

典型例はユーザー管理である。「ユーザー」という概念は業務ドメイン自体には存在しないが、IT基盤にログインを設けて利用者を制限する以上、事実上必須のコンテキストになる。ロールや権限も同様に、業務側の役割分担に由来しつつ、実装上は独立したコンテキストとして扱われることが多い。「ユーザー」「ログイン」「パスワード」「ロック」といった語彙は特定の業務ドメインに固有ではなく汎用的であり、これはそのコンテキストが解決策由来の汎用サブドメインであることを示すシグナルでもある（詳細は Substep 2.2）。

> 現場事例: 産業機器のフリート管理システムをモノリスからマイクロサービスへ移行する際、老朽化した車載端末が古い独自メッセージ形式でデータを送ってくる問題があった。当初は「各マイクロサービスが端末と直接つながり、独立してデプロイできるようにすべき」という案が出たが、老朽端末（と新端末）のメッセージ形式に関する専門知識を全チームに複製すると知識の分散が大きくなりすぎる懸念があった。最終的に、この接続知識を専属チームが担当する独立した境界づけられたコンテキストとして切り出し、アンチコラプションレイヤーの役割を持たせた。これにより他のコンテキストは端末形式の詳細から解放され、この接続コンテキストが一時的に落ちても短時間なら他のコンテキストは自律的に稼働し続けられるようになった。独立デプロイ可能性と知識の分散はトレードオフの関係にあり、レガシー変革では頻繁に直面する判断である。

### 2.4 Substep 2.2: サブドメインの分類

コンテキストマップが揃ったら、それぞれの重要度を議論する。ここで使うのが戦略的設計のサブドメイン分類（コア／サポート／汎用。概念定義は `applying-clean-architecture` の DDD 戦略設計参照へ委譲）である。この判断はドメインエキスパートだけでは決めきれず、組織の戦略層を巻き込む必要がある。議論では次のような問いを使う。

- 何で（最も）収益を得ているか
- 市場で何を評価され、知られているか
- 競合より優れているべき点はどこで、平均でよい点はどこか
- 自社ならではの強み・差別化要因は何か
- 特に多様なプレイヤーが存在する業務領域はどこか（差別化が進み重要性が高い兆候）
- 特に高度な作業が重要な成果につながっている業務領域はどこか
- 多くの他領域から情報を集約する中心的な業務領域はどこか

すべての境界づけられたコンテキストに分類を付ける。まだ判断がつかないコンテキストがあれば、それは「判断が保留中」であって「判断不要」ではないことを明示する。コア・サポート・汎用の中間形（サポート寄り汎用、など）が出てくることも珍しくない。

この分類は固定ではなく、組織の重点が変わるにつれて見直される。コアドメインチャート（詳細は `applying-clean-architecture` へ委譲）は、この分類の推移を可視化する補助になる。長年運用されてきたレガシーシステムでは、構築当時は自作するしかなかった領域が、現在では既製の COTS / SaaS / OSS で代替可能になっているケースが多く、コアでなくなった領域ほどこの置き換えを検討する価値がある。

### 2.5 Substep 2.3: 境界づけられたコンテキストのチーム割当

コンテキストマッピングとチーム割当は密接に関係する。分類（コア／サポート／汎用）もチーム割当の判断材料になる。実務では次のような割当方針が組織ごとに採用される。

- 「コアドメインには必ず専属チームを置く」（Team Topologies の認知負荷の考え方に近い）
- 「すべてのチームに1つはコアドメインを持たせる」（担当作業の重要感を保つため）
- 「コアドメインは経験豊富なチームに、新規チームはサポートドメインから」
- 「コアドメインは優秀なチームに」
- 「コアドメインは社内チームに、外部委託チームは知識流出の影響が小さいドメインに」

これらの方針は互いに矛盾することもある。特に「チームを優劣で選別する」考え方には注意が必要で、割当は開発チームとそのマネージャーを交えた議論を通じて、組織が納得できる形に落とし込むべきである。

### 2.6 Challenge: チーム割当は往々にしてチーム再編を意味する

多くの組織では、既存のチーム構成はビッグボールオブマッドの構造に最適化されており、これから目指す境界づけられたコンテキスト単位の構成とは一致しない。そのため、ソフトウェアの分割にあわせてチームの再編も必要になるケースがほとんどである。ここで Team Topologies の知見（詳細は `cloud:architecting-infrastructure` および本スキルの `TEAM-ORGANIZATION.md` へ委譲）と、社会技術的リファクタリング（レイヤー編成のチームから機能横断チームを形成する手順。詳細は `REFACTORING-CATALOG.md`）が役立つ。まず1チームを機能横断型に再編し、そこで得た知見を次のチーム再編に活かす、という段階的な進め方が現実的である。

### 2.7 Substep 2.4: 関係性の決定

目標アーキテクチャのモデリングの最後のサブステップとして、コンテキスト（とそれを担うチーム）間の関係を決める。次のような問いを開発チームと議論する。

1. 境界づけられたコンテキスト間のインターフェースはどこにあるか
2. どのコンテキスト間にはインターフェースを一切設けるべきでないか
3. これらのインターフェースは、コンテキストマッピングのどのパターンに従うか
4. インターフェースを介して連携するチーム同士の関係はどうあるべきか

1・2の問いには、Step1で作った分割済みのTo-Beプロセス（ドメインストーリーテリングやEventStormingの成果物）が手がかりになる。3・4の問いには、コンテキストマッピングパターン（Open Host Service / Anticorruption Layer / Conformist / Customer-Supplier 等。詳細は `applying-clean-architecture` の DDD 統合参照へ委譲）と Team Topologies の考え方を組み合わせる。

例えば、あるコンテキストが標準製品（COTS/SaaS）で賄われており、その公開インターフェースが自社の理想と完全には一致しない場合、汎用サブドメイン側は素直にそのインターフェースへ合わせる（Conformist）選択が現実的な一方、コアドメイン側は自身のモデルを守るためにアンチコラプションレイヤーで包んで防御する、といった使い分けが生まれる。同じ提供元インターフェースに対して複数のコンテキストが接続する場合は、それをOpen Host Serviceとして扱う、という判断も典型的である。こうした関係を1本ずつ洗い出し、コンテキストマップに書き加えていく。

### 2.8 まとめて可視化する

分類・チーム・関係性のすべてを1枚のコンテキストマップに重ねて描くこともできるが、情報量が増えすぎて一度に消化しづらくなる。多くのチームは、目的に応じて簡略化した複数のマップ（分類だけを示す版、チーム割当だけを示す版、関係性だけを示す版）を使い分ける方を好む。完成したコンテキストマップは、チーム間の全体会議などで頻繁に参照できる場所に掲示し、継続的な議論の土台とするとよい。
