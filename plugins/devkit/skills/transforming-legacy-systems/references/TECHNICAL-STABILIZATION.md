# 技術的安定化 / Technical Stabilization

ドメイン駆動の変革に着手する前に、まず技術基盤を安定させる。IDE・ビルド・依存関係の近代化から、依存の切断（Seam）、メトリクスに基づく構造改善、エラー堅牢性の強化まで、費用対効果の高い施策を着手しやすい順に並べる。

---

## 1. まず費用対効果の高い施策から着手する

技術的安定化は、あれもこれも同時に手を付けるのではなく、**最小のコストで最大の改善が得られる施策（Low-Hanging Fruits）から順に着手する**のが原則である。以降の節はおおむねこの順序（着手しやすさ・効果の出やすさ）で並べてある。焦って全面刷新を狙わず、小さな改善を積み重ねる姿勢がレガシーシステムの変革では特に重要になる。

---

## 2. 開発基盤を近代化する

### 2.1 IDEを更新する

古いバージョンの統合開発環境を使い続けているチームは、自動リファクタリングやAI支援など、現代のIDEが提供する生産性向上の恩恵を受けられていないことが多い。IDEの更新（あるいは未導入なら導入）は最も着手コストの低い施策の一つであり、優先的に実施する価値がある。IDE統合型のAIコーディングアシスタントも、まだ導入していなければ早期に試す価値がある。

### 2.2 ビルドとデプロイを自動化する

開発チームの成熟度に応じて、以下のような段階的な施策が有効になる。

| 施策 | 内容 |
|------|------|
| バージョン管理の近代化 | 個人PC上でのファイル管理やUSBメモリでの受け渡しから、標準的なバージョン管理システムへ移行する |
| ビルドの自動化 | 手作業のビルド手順を、ビルドツールによる自動化へ置き換える |
| ビルドの最適化 | ビルドサーバーの増強、部分デプロイの導入、ホットデプロイの活用などで、ビルドの所要時間を短縮する |
| CI/CDサーバーの導入 | プッシュのたびに自動でビルド・テスト・デプロイを行う仕組みを整える |
| Infrastructure as Code | 設定やインフラの定義もコードとしてバージョン管理下に置き、再現性と巻き戻し容易性を確保する |

ビルドが大規模なレガシーシステムほど、ビルド所要時間の長さが開発者のフィードバックサイクルを遅らせ、結果として大きな単位でバグが埋め込まれやすくなる。ビルドパイプラインの整備は、その後のあらゆる変更作業の土台になるため、早い段階で着手する価値が大きい。

### 2.3 依存関係を更新する

放置された依存ライブラリはセキュリティ上の弱点になりやすい。依存関係の追跡・更新の仕組みを持たないレガシーシステムでは、気づかないうちにライブラリが古くなり、セキュリティ更新の恩恵を受けられなくなっていることが多い。

**ライブラリ・フレームワークの更新**は、多くの場合は後方互換性が保たれているため、既存コードへの影響は限定的である。一方で、フレームワークそのものを刷新する必要がある場合（UIライブラリの世代交代など）は大規模な作り直しになりやすく、場合によってはプログラミング言語自体の変更を伴う。このようなケースでは、無理に今着手せず、他の水準での安定化を先に進め、システム全体の状態が良くなってから取り組む方が現実的である。同じバージョンのフレームワークを複数チームが共有しているために更新しづらい場合も同様で、戦略的変革によって境界づけられたコンテキストごとに独立したバージョンを持てるようになってから更新する、という順序が有効なことがある。

**プログラミング言語の更新**も見逃せない。多くの現代言語は、モジュールシステムによる境界づけられたコンテキストの実装支援、null安全性の設定による堅牢性向上、軽量な値オブジェクト構文（読み取り専用プロパティ、record/data class的な機能）など、変革を助ける機能を追加し続けている。言語のバージョンを設定ファイルで切り替え可能にし、バージョン管理ツール（asdf・SDKMAN・pyenv系のツール等）を併用して、問題が起きたときにすぐ切り戻せる体制を整えておく。

**コンパイラ・静的解析の警告解消**も効果が高い。静的型付け言語のコンパイラ警告、動的型付け言語向けの静的解析ツール、IDEの警告、専用のバグ検出ツールなど、現代の開発者には多くの手段がある。レガシーシステムにはこの種の警告が大量に蓄積していることが多く、これは技術的負債の一形態である。警告を解消したら、可能であれば警告レベルをより厳しく設定し、この作業を繰り返す。

> **現場事例（一般化）**: あるプロジェクトでは、十数年分の警告が積み重なり、変更のたびに新しい警告が増えても誰も気づかない状態になっていた。あるとき「警告が一つも残っていないコードしかコミットしてはならない」というルールを導入したところ、最初の数か月は非常に手間がかかったものの、変更頻度の高い部分から順に警告が解消され、半年後にはチーム全体が「片付いたシステムで開発できる」ことを実感できるようになった。

依存関係やコンパイラ警告の解消は件数が膨大になりがちなので、AIツールに次のような役割を担わせるのも有効である。

- 現在使われている技術要素の棚卸しと最新度の判定
- 影響範囲が小さく機械的に適用できる更新の洗い出し
- コーディングアシスタントへの指示による、更新に必要な広範囲の修正の実施

## 3. バグ修正のたびにテストを増やす

多くのレガシーシステムには、充実したバグ管理の仕組みがある一方で、テストカバレッジは乏しいという傾向が見られる。自動テストなしで運用されているシステムが今なお多く存在するのは残念な現実である。

テストカバレッジの低さとバグの多さは、しばしば相互に悪循環を作る。開発チームは自動テストという安全網がないままバグ修正や機能追加を行うため、新しいバグを埋め込みやすい。ある観察によれば「変更によってどれだけの既存の振る舞いが危険にさらされるか、私たちはしばしば把握できていない」という指摘があり、これは既存の機能を変更から守るテストの重要性を端的に示している。

### 3.1 スカウトルール

既存コードベース全体に対して後追いでテストを書き切ることは現実的ではない。そこで有効なのが**スカウトルール**（来たときよりも美しく、の精神）である。バグ修正・計画的なリファクタリング・新機能実装のためにソースコードへ手を加える際は、その手直しの一環として必ずテストと必要なリファクタリングを実施し、触れた場所を来たときより良い状態にして去る。これを続けることで、バグ修正のたびにテストカバレッジが少しずつ積み上がっていく。

> **現場事例（一般化）**: ある複雑なアルゴリズムを、単体テストで少しずつ安全網に包んでいった現場がある。別のバグが見つかるたびに、まずそのバグを再現するテスト（赤）を書き、それから修正する（緑）。修正が既存の振る舞いに影響を与えていれば、他のテストが失敗して知らせてくれる。テストで守られた領域では、リファクタリングをむしろ楽しめるようになったという実感が、チーム全体でテストを増やす強い動機付けになった。

AIツールに既存機能に対するテストを書かせることも有効である。ただし、AIが最初に書くテストは既存の振る舞いをなぞるだけで、バグそのものは見つけてくれない点に注意する。それでも、これから行う変革のためのリファクタリングを検証する土台としては十分な価値がある。

### 3.2 テストの種類とテストピラミッド

単体テスト・統合テスト・UIテストはそれぞれ検証範囲と実行コストが異なる。以下のような比率で組み立てるのが一般的な指針である。

```
        /\
       /UI\        少数・低速・広い範囲（画面全体）
      /----\
     / 統合  \      中間量・中速・複数クラスにまたがる
    /--------\
   /  単体    \    多数・高速・狭い範囲（クラス/値オブジェクト単位）
  /------------\
```

単体テストは狭い範囲を高速に検証できるため、テストカバレッジ向上の主軸に据えるべきである。統合テストやUIテストは複数のクラスやシステムの広い範囲にまたがるため、デバッグの手間が単体テストより大きくなりやすい。

ドメイン駆動設計の文脈では、単体テストの対象は「クラス一つ」に限らない。以下のいずれも意味のある単体テスト対象になりうる。

- 値オブジェクト単体
- 関連する値オブジェクトを含むエンティティ
- 内部にエンティティ・値オブジェクトを含む集約
- 集約・エンティティ・値オブジェクトを扱うサービス（リポジトリはモック化する）

テストを追加する際は、オブジェクトの依存関係グラフの末端（他に依存されていない葉）から始めて、徐々に上位へ広げていくとよい。

密結合したレガシーシステムでは単体テストの導入自体が難しいことが多い。テスト可能に（そして後の分解も可能に）するには、まず依存関係を減らす必要がある。これは次節で扱う。

## 4. 依存を切断してテスト容易性を取り戻す（Seam）

「変更したい箇所があるなら、まずその変更を容易にし（これは難しいかもしれない）、それから容易になった変更を行う」という考え方は、レガシーシステムの改善において繰り返し引用される指針である。小さなステップに分解してリファクタリングを進めようとしても、レガシーシステムでは絡み合った依存関係のせいで、百件のつもりが千件に膨れ上がり、収拾がつかなくなることが少なくない。

この問題への実践的な解決策として広く知られているのが**Seam（継ぎ目）モデル**である。

> **Seamの定義（再構成）**: ある箇所を編集することなく、そこでの振る舞いを差し替えられる場所のこと。

一見矛盾しているように聞こえるが（編集せずにどうやって振る舞いを変えるのか）、実際にはいくつかの手段がある。代表的なのはオブジェクト指向のポリモーフィズムを利用する方法で、これを**オブジェクトSeam**と呼ぶ。型`T`の変数`v`に対してメソッド`m()`を呼んでいる箇所があれば、`v`を`T`のサブタイプ`S`のインスタンスに差し替え、`S`側で`m()`をオーバーライドして望みの振る舞いを実現できる。このほかにも、プリプロセッサを利用する手段や、リンク（クラスパス等）を利用してテスト用の実装を「差し込む」手段が存在する。

それぞれのSeamには、「どちらの振る舞いを使うか」を決める箇所が対応する。これを**Enabling Point（有効化点）**と呼ぶ。オブジェクトSeamでは依存性注入の設定がEnabling Pointになることが多く、リンクSeamでは読み込むライブラリの指定（クラスパス等）がEnabling Pointになる。

Seamを見つけてテストの支配下に置ければ、それまで手を付けられなかった箇所にも安全にテストを追加できるようになる。テストで守られたコードは、壊れる不安なしに変更できるようになり、次の一歩である内部構造の改善が可能になる。

### 4.1 小さな一歩でリファクタリングする

リファクタリングは、外部から見た振る舞いを変えずに内部構造だけを改善する作業であり、常にコンパイル・実行・テスト可能な状態を保ちながら進めるという哲学に支えられている。メソッドの改名を例に、避けるべき進め方と推奨される進め方を対比する。

**避けるべき進め方**:
1. メソッド名をいきなり変更する → 呼び出し側全てがコンパイルエラー（最悪の場合は実行時エラー）になる
2. コンパイルエラーを一つずつ潰していく

この進め方では、全ての修正が終わるまでソフトウェアは一度もコンパイル・実行できる状態に戻らない。途中のどの一歩が問題を起こしたのか、フィードバックを得られない。

**推奨される進め方**:
1. 新しい名前のメソッドを追加する（シグネチャは同じにし、実装は旧メソッドへの委譲のみにする）
2. 新旧メソッドの呼び出し箇所を少しずつ新メソッドへ切り替える
3. 旧メソッドを非推奨（deprecated）としてマークする
4. 非推奨警告を手がかりに、呼び出し箇所を一つずつ新メソッドへ置き換える
5. 呼ばれなくなった旧メソッドを削除する

この進め方であれば、各ステップの後で必ずソフトウェアはコンパイル・実行・テスト可能な状態を保つ。途中で作業を中断し、後で再開することもできる。現代のIDEであれば、単純な改名はこの手順を自動リファクタリング機能でサポートしてくれる。

依存を切断するリファクタリングと、内部構造を改善するリファクタリングは特に重要な2つのカテゴリである。

### 4.2 メトリクスで内部構造の改善箇所を見つける

静的解析ダッシュボードなどのツールを使うと、コードの「におい」を集計し、メトリクスとして可視化できる。ただし、これらのツールが報告する問題は非常に大量になりがちで、技術的負債の解消に必要な作業量まで見積もってくれる場合もある。数字に圧倒されず、優先順位付けから始めることが重要である。

メトリクスを扱う際の指針は次の2点である。

| メトリクスの種類 | 着眼点 |
|-----------------|--------|
| サイズ系メトリクス | コード行数・循環的複雑度が突出して大きい単位（メソッド・クラス・パッケージ・モジュール）に着目する。サイズや複雑度が突出する単位は、実装や設計が良くない兆候であることが多い。テストで守った上でリファクタリングし、より小さな単位に分割する |
| 結合系メトリクス | 循環依存の数、ファンイン（依存されている数）、ファンアウト（依存している数）に着目する。多くの単位から使われている、あるいは多くの単位を使っている単位は、密結合の兆候であることが多い。テストで守った上でリファクタリングし、より独立した単位にする |

メトリクスの値が時間とともにどう変化するかを追うことも有効である。あるパッケージ内のクラス間の関係数が増え続けているといった兆候は、問題がゆっくり進行しているサインであり、早期に対処するほどコストは小さくて済む。

> **現場事例（一般化）**: 外部委託されていたレガシーシステムを開発部門へ内製化した組織の例がある。委託時代はシステムが頻繁に不安定になり、データの不整合も発生していたが、内製化後にまず既知のバグに対するテストを整備しながら地道に修正を重ね、1年ほどで安定稼働にこぎつけた。その後、次のステップとして推奨されたのが、静的解析ダッシュボードの導入による問題の可視化と優先順位付けだった。

## 5. エラーへの堅牢性を高める

レガシーシステムのユーザーや開発チームが、エラーによるクラッシュに日常的に悩まされているケースは珍しくない。運用担当者が深夜や早朝にサーバーを定期的に再起動して、翌日のクラッシュを予防するような極端な例もある。ここではエラーへの堅牢性を高めるための代表的な技法を扱う。

### 5.1 nullに対する堅牢性を確保する

null参照は広く知られた設計上の誤りとして指摘されており、静的型付け言語の多くで、未割り当ての参照に起因するエラーが繰り返し発生する原因になってきた。現代の言語の中にはデフォルトでnullを許可しない設計を採るものもあり、既存の言語でも設定によってnullを制限できるものが増えている。

レガシーコードのnull起因エラーを減らすには、次のような段階的な対策が有効である。

- 言語やツールチェーンがnullの事後的な禁止をサポートしているなら、その機能を有効にする
- nullを許容する/しないを表明するアノテーションを導入し、静的チェックにかける
- 上記が使えない場合は、パラメータの受け渡しや変数への代入箇所を手作業で確認し、アサーションでチェックする

手間はかかるが、システムの堅牢性を大きく引き上げる効果がある。

### 5.2 例外の使い方を見直す

例外の濫用は、正常な処理の流れの中に予期せぬジャンプを持ち込み、コードの見通しを悪くする。例外が引き起こす最大の問題は、メソッドからの脱出経路を不必要に増やしてしまうことである。開発者は、あるメソッドを呼び出すすべての経路について、脱出のたびに整合性のある状態が保たれているかを考え続けなければならなくなる。

> **指針**: 例外は原則として「予期しない技術的な事象」にのみ使い、業務上のエラーには使わない。たとえば入力値の検証で誤りが見つかることは、ユーザーが入力を間違えたり外部インターフェースが不正なデータを送ってきたりする以上、ごく普通に起こることであり、例外で扱うべき事象ではない。

予期しない振る舞いは大きく2種類に分けられる。

| 種類 | 特徴 | 対応方針 |
|------|------|---------|
| 外部環境エラー | ファイルオープン失敗や配列範囲外アクセスなど、開発チームの制御が及ばない事象。原理的には予見可能 | 例外を自作するのではなく、既存の例外機構で捕捉し、整合性のある状態で終了するか、適切なエラーメッセージとともに処理を継続する |
| 内部プログラミングエラー | 開発中に混入するバグ。あらゆる可能性を事前に網羅することはできない | ペアプログラミング・コードレビュー・自動テスト・契約による設計といった予防策を組み合わせ、できるだけ早期に発見する |

### 5.3 契約による設計を導入する

**契約による設計（Design by Contract）**は、クラスやメソッドが「呼び出し側が一定の条件を満たせば、一定の振る舞いを保証する」という契約を提示するという考え方であり、内部プログラミングエラーへの堅牢性を高める土台になる。

契約は**事前条件**（メソッド実行前に成り立つべき状態）・**事後条件**（メソッド実行後に成り立つべき状態）・**不変条件**（オブジェクトが常に満たすべき状態）という3種類のアサーションとして表現される。多くの言語では、専用の契約構文の代わりにアサーション機構（`assert`文やそれに類する仕組み）を使って実現する。

以下は、座席予約を例にした事前条件導入の再構成例である（TypeScript風の擬似コードで表現）。

```typescript
class SeatingPlan {
  markAsSold(soldSeat: Seat): void {
    // 事前条件: 引数がnullでないこと
    assert(soldSeat != null, "sold seat must not be null");
    // ... 本来の処理
  }
}
```

まず技術的な事前条件（null対策など）から導入するのが着手しやすい。呼び出し側が誤った引数でメソッドを呼び出せば、その場でアサーションが失敗し、早期に問題を検知できる。契約による設計の業務的な使い方（ドメイン指向の事前条件）については、次のファイル（`DOMAIN-KNOWLEDGE-IN-CODE.md`）で扱う。

契約による設計をスカウトルールに組み込み、日常的な変更のたびに事前条件を追加していく運用を推奨する。少なくとも事前条件は、全ての開発者が当たり前に使う道具にする価値がある。事後条件や不変条件は、メソッドの効果を説明するために旧い状態を参照する必要が生じやすく、実装の難度は事前条件より高い。

さらに踏み込みたい場合は、契約による設計で表現していた制約を型システムに押し込め、「不正な状態をそもそも表現できなくする」という設計方針も存在する。これはドメインモデルの型設計における発展的なテーマである。

## 6. 自己評価チェックリスト

以下の項目について、1（全くそう思わない）〜5（強くそう思う）の5段階で自己評価してみる。低い評価が付いた項目が、次に着手すべき技術的安定化の候補になる。

| # | 項目 |
|---|------|
| 1 | システムは容易に拡張・保守できる |
| 2 | システムの一部を理解していれば、他の部分も理解しやすい |
| 3 | コード品質を静的解析ダッシュボード等で継続的に監視している |
| 4 | 問題のあるコードは定期的にリファクタリングされている |
| 5 | 継続的インテグレーション・継続的デリバリーが実践されている |
| 6 | 変更を予測可能にするだけの自動単体テストがある |
| 7 | 変更を予測可能にするだけの自動統合テストがある |
| 8 | 変更を予測可能にするだけの自動UIテストがある |
| 9 | システムは大きな泥だんご（Big Ball of Mud）状態ではない |
| 10 | システムに「触れずにおくべき」場所は少ない |
| 11 | システムに「もう誰も理解していない」領域は少ない |
| 12 | システムの一部を取り除いたり、単独でテストしたりするのは容易である |
