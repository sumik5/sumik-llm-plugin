# ドメインパターン — 変革時の実装選択指針

変革を重ねるうちに気づくのは、「ドメインには分解しやすいものと分解しにくいものがある」「同じ分解方針でも実装のされ方によって難易度が変わる」という経験則である。本参照は、この経験則を3つの次元（ワークフロー・ソフトウェア内ドメインモデル・ドメインの形）に整理し、境界づけられたコンテキストへの実装方針を選ぶための判断材料としてまとめる。ここで示す分類は確立した理論ではなく、現場での観察から抽出した傾向であり、絶対的な分類ではなく「傾向を掴むための道具」として使うとよい。

---

## この参照が扱う3つの次元

| 次元 | 問い |
|------|------|
| 1. ワークフロー | ドメイン内の人々はどう協働しているか（順番か、並行か、交渉か） |
| 2. ソフトウェア内ドメインモデル | ドメインモデルはどう実装されているか（中心的な項目に依存しているか、挙動が豊かか貧血か） |
| 3. ドメインの形 | ドメインはどれくらい成熟し、どれくらいデジタルか、変革の余地はどこにあるか |

---

## 次元1: ドメイン内のワークフロー

### Pipeline（逐次処理）パターン

作業が段階的に進み、しばしば異なる役割の人が順番に担当するドメインは **Pipeline** パターンに従う。ある工程は前工程の結果なしには成立せず、各役割は自分の作業結果を次の工程に引き渡す。

```
[役割A: 工程1] → [役割B: 工程2] → [役割C: 工程3]
     順序が明確で、後工程は前工程の結果を前提とする
```

**代表的なドメイン例**:
- 物流プロセス
- 購買・配送プロセス、取引プラットフォーム
- 保険・与信契約の締結プロセス
- 規制対応プロセス
- 興行チケット販売（週間スケジュール確定 → チケット販売の順）
- リース業務（「引き合い」→「与信・リスク審査」の順）

ほとんどのドメインには部分的にPipeline的な工程が含まれる。ドメインの最上位レベルがPipelineパターンに従う場合、業務プロセス上の境界がそのまま見えやすいため、協働モデリング（`COLLABORATIVE-MODELING.md`）による分析・分解は比較的容易に進む。

### Blackboard（並行協調）パターン

複数の利用者が1つまたは複数の業務対象に対して、決まった順序なく並行して作業するドメインは分析・分解が格段に難しい。この形を **Blackboard** パターンと呼ぶ。役割間の作業関係を分析しても、繰り返し現れる決まった手順は見つからない。

```
役割A ─┐
役割B ─┼─→ 共有された業務対象（順不同・並行してアクセス）
役割C ─┘
```

全体としての粗粒度プロセスは、順番に実行される活動ではなく並行して実行される活動の集まりであり、Pipelineパターンよりはるかに多くの状態遷移が起こり得る。

**代表的なドメイン例**:
- 車両・プラント設備の計算ソフトウェア
- 大規模イベントの企画・運営管理
- 監視センターでの観測・監視（プラント、物流、交通流、サプライチェーンの乱れ等）
- ソフトウェア開発そのもの
- コミュニケーションツール（会議・チャット等の協働ツール）
- 多人数が同じ世界に同時参加するゲーム

Blackboardドメインを分析・分解するときは、関与する各アクターと、その協働のステップに注目して業務プロセス上の境界を探す必要がある。

### Dialog（交渉・合意形成）パターン

粗粒度プロセスの内側、より細かい粒度でBlackboardパターンの変種が見られることがある。単に1つの業務対象を共同作業するのではなく、**複数の当事者が情報を出し合い、互いに応答しながら共通の計画を明示的に交渉する**プロセスである。これを **Dialog** パターンと呼ぶ。

```
当事者A ⇄ 当事者B
   情報を出し合い、互いに応答する
        ↓
   合意された共通の計画（明示的な合意）
```

この交渉プロセスの終わりには、すべての当事者が明示的に合意した結果が生まれる。互いに指揮命令権を持たない企業間・行政部門間の交渉や、行政上の明示的な処分行為が伴わない場合には、Blackboardではなくこちらの変種を選ぶのが基本になる。

**代表的なドメイン例**:
- 船舶の運航スケジュールを海運会社と港湾間で調整する
- 大型設備の種類・納入・保守サイクルを発注者と顧客の間で調整する
- 建設スケジュールの長期調整を行政と建設申請者の間で行う
- 交渉プロセスに基づくゲームの一部

Dialogドメインでは、交渉プロセスの中で共通する部分を見つけ出し、それをサブドメインや境界づけられたコンテキストとして切り出すことが鍵になる。アクターや役割単位で分解するアプローチは、Dialogドメインには適さない。

### パターンは固定ではなく進化する

ドメインが従うパターンは、システムの発展段階に応じて変化しうる。ある変革プロジェクトでの現場事例では、複数組織にまたがる調整業務を支援するソフトウェアにおいて、当初は各当事者が持ち場から意見をまとめて突き合わせるだけの、比較的無秩序なBlackboard的な運用だった。数年の運用経験を経て、業務側が「一定の状態で承認や見解表明を必須とする」品質ゲート付きのプロセスを段階的に導入し始め、システムはBlackboardからDialogへと緩やかに移行しつつあった。

この事例で興味深いのは、ソフトウェアがドメインの発展を後押しする側面と、逆にソフトウェアの機能拡張がドメインの発展に追いつけていない側面の両方が同時に観察された点である。ソフトウェアが多数の業務対象の全体像を利用者に提供したことで初めて調整が可能になった一方、次のプロセス支援機能はまだソフトウェア側に実装されておらず、利用者の不満につながっていた。このドメインは複数の組織・行政間の合意形成を伴うため、今後もPipelineパターンへ発展する可能性は低いと見られている。

## 次元2: ソフトウェア内ドメインモデルの実装

この次元は、ドメインモデルがソフトウェアの中でどう実装されているかを扱う。利用者は全員同じ作業対象を扱うのか、それぞれ異なる対象を扱うのか。作業対象はモデルとして貧血なのか、挙動が豊かなのか（貧血ドメインモデルの解消は `DOMAIN-KNOWLEDGE-IN-CODE.md` を参照）。

### 中心的ドメイン項目 vs 多様なドメイン項目

分解の難易度は、システムに**中心的なドメイン項目**（あらゆる部門・工程が同じ1つの概念に依存する）が存在するかどうかに大きく左右される。

**中心的ドメイン項目の例**: 港湾のコンテナターミナルでは、荷受け（鉄道・トラック）、ヤード管理、荷役機械、積付計画、管制まで、あらゆる部門・工程が「コンテナ」という同一の概念とそのデータを必要とする。この中心的な項目の存在自体が分解を難しくする。開発初期に「コンテナは業務の場面ごとに異なる役割を持つ」という発想があれば中心的項目化を避けられた可能性はあるが、既存システムが出来上がった後にそれを見直すのは容易ではない。

**多様なドメイン項目の例**: 銀行業務では、融資・証券・顧客サービス・決済など各部門がいずれも「顧客」に関わるが、扱う商品は大きく異なる。融資と証券を無理に統一抽象化しようとする者はまずいない。同様に、保険会社の各事業ライン（家財保険・生命保険・医療保険・賠償責任保険等）も、商品としての違いが大きい。ただし、いずれも「保険」という呼び名を共有しているという表面的な共通性に引きずられ、事業ラインを横断する統一モデルを作りたくなる誘惑には注意が必要である。

中心的ドメイン項目がどこまで肥大化するかは、開発者・アーキテクトが抽象化やクラス階層による統一化にどれだけ傾きやすいか、そしてシステム内でどこまで再利用を狙うかにも左右される。

> 💡 **対処の指針**: 中心的な作業対象を持つドメインでは、データではなく**挙動**に着目することが特に重要である。一見「中心的」に見える対象であっても、境界づけられたコンテキストごとに、その文脈に特化した専用モデルを構築することを目標にする。

### フォームベースドメインモデル

一部のソフトウェアでは、ドメインモデルの大半が「フォーム（帳票）」で構成されている。行政系の申請処理や、税務申告の受付・審査・承認のようなシステムに多く見られる形で、紙の帳票を前提にした業務を、そのままソフトウェアに置き換えた歴史的経緯を持つことが多い。

こうしたフォームベースのモデルは、多くの場合CRUD（Create/Read/Update/Delete）アプリケーションとして実装される。利用者がフォームにデータを入力し保存する、という形である。「これは単純なCRUDアプリケーションだから、モジュールや境界づけられたコンテキストへの分割に意味はない」という判断がしばしば下されるが、実際に真にCRUDだけで説明が付くドメインは、経験上、想定より少ない。

> ⚠️ **見落とされがちな事実**: フォームそのものが、すでにドメインの中核概念とプロセスの一部をモデル化したものである。フォームの各項目・各セクションには、業務上の意味が織り込まれている。

CRUDアプリケーションでは、業務ロジックが本来あるべきドメイン層より1つ上のレイヤー（サービス層、ひどい場合はUI層）に漏れ出し、全レイヤーに散在しがちである。この散らばりを整理し、CRUDデータオブジェクトのどの部分がどの境界づけられたコンテキストで使われているかを分析した上で、UI・サービス層に漏れ出た業務ロジックをドメインオブジェクトへ引き戻せば、フォームベースのドメインモデルであっても分割は十分に意味を持つ。ただし、この整理は事実上システム全体の再構築に近く、新規実装との費用対効果を慎重に比較検討する必要がある。

## 次元3: ドメインの形

最後に、ドメインそのものの性質に関する3つの観点を扱う。

### 成熟度（ドメインの年齢）

分析対象のドメインが、業務プロセスや中核概念として長い歴史を持つのか、それとも最近生まれたばかりなのかを意識する必要がある。年齢によって、中核概念の明確さや、既存ソフトウェアによる業務プロセス支援の成熟度は大きく異なる。

銀行・保険・トレーディング・コンテナ輸送のような古参のドメインでは、中核用語やプロセスについて業界横断で公開された整理済みの資料が存在することが多く、分析時にはこうした資料を積極的に参照すべきである。一方、カーシェアリングのような新しいモビリティ、デジタルツイン、ドローン輸送、ソーシャルメディアのような若いドメインでは、そうした整理はまだ乏しい。

ソフトウェアの成熟段階を捉える視点として、次の4段階に分類するモデルが知られている。

| 段階 | 内容 |
|------|------|
| Genesis（黎明期） | 未知の新しいドメインを探索している段階 |
| Custom Built（独自構築） | 各社が独自に実装している段階 |
| Product（製品化） | 市場に製品として提供され、提供者ごとに差がある段階 |
| Commodity（コモディティ化） | 高度に標準化された実装が広く普及している段階 |

自分たちのドメイン向けソフトウェアをこの段階のどこに位置づけられるか整理すると、そのドメインの発展度合いについて多くの示唆が得られる。

### 有形物ドメイン vs 完全デジタル化ドメイン

保険業、多くの行政手続き（税務申告等）、銀行業務の大部分など、近年完全にデジタル化されたドメインでは、現実世界にはもはや対応する「モノ」が存在せず、データとプロセスだけがデジタル世界に存在する。

一方、コンテナ・自動車・貨物・リース資産・輸送手段のように、現実世界の有形物を前提とするドメインも多く残っている。こうした「表現型」のドメインでは、ソフトウェア内のドメインモデルは現実の一種の写し・シミュレーションであり、現実との整合を保ち続ける必要がある。有形物を表現する部分のモデルには、その物の状態遷移を表す**ステートマシン**を持たせるのが望ましい。

現実世界に対応物が一切存在せず、完全にデジタルの中だけで生まれたドメイン（ソーシャルメディア、開発ツールそのもの等）もある。このようなドメインではユビキタス言語や実体のあるドメインモデルの構築が特に難しく、メタファーを積極的に活用することが助けになる。

### 変革の余地

最後に、境界づけられたコンテキストごとに変革の伸びしろは異なる。

- ドメインモデル自体の業務知識の反映が乏しく、モデルそのものに改善余地が大きいコンテキスト（フォームベースのドメインモデル等）
- ドメインモデルとしての完成度は高いが、プロセスレベルでの変革余地が大きいコンテキスト
- 数十年の経験でプロセスもモデルもすでに高度に整理されており、新しいビジネスモデルの導入以外に伸びしろを見出しにくいコンテキスト（成熟した小売業等）

ある変革プロジェクトの現場事例では、イベント管理システムにおいて、要件がエンティティ単位で細かく規定され複雑な検証ルールも整備されている一方、業務プロセスの支援がソフトウェアに組み込まれておらず、プロセスの全体像は利用者の頭の中にしか存在しない、という状況が観察された。メニュー構成もエンティティ単位で作られており、経験の浅い利用者ほど迷いやすい。ドメインモデル自体は挙動が豊かでも、業務プロセスレベルでの変革余地は大きく残っていたことになる。

## パターン→実装選択の指針

| ドメインの性質 | 分解のしやすさ | 実装での注目点 |
|----------------|---------------|----------------|
| Pipeline | 比較的易しい（境界が業務プロセス上に見える） | 工程の切れ目をそのままコンテキスト境界の候補にする |
| Blackboard | 難しい（繰り返し手順が見えない） | アクターと協働のステップから境界を探す |
| Dialog | 難しい（アクター単位の分解は不適） | 交渉プロセスの共通部分を境界にする |
| 中心的ドメイン項目あり | 難しい | データでなく挙動に着目し、コンテキストごとに専用モデルを作る |
| 多様なドメイン項目 | 比較的易しい | 表面的な類似性（同じ呼び名）に引きずられない |
| フォームベース | 見かけ上は簡単に「分割不要」と誤判断されやすい | フォームは既にモデルであると捉え直し、業務ロジックの散在を洗い出す |
| 有形物ドメイン | モデルはステートマシンで表現しやすい | 現実世界との整合維持コストを考慮する |
| 完全デジタルドメイン | ユビキタス言語の確立が難しい | メタファーを積極的に使う |

これら3次元の判断は独立ではなく組み合わさって現れる。たとえば「中心的ドメイン項目を持つ、成熟した有形物ドメインのPipeline」と「多様なドメイン項目を持つ、若い完全デジタルドメインのDialog」では、分析の出発点も変革の優先順位もまったく異なる。実際の境界づけられたコンテキストの発見・目標アーキテクチャの検討（`STRATEGIC-STEPS-DISCOVERY-MODELING.md`）では、対象ドメインをこの3次元に照らして仮説を立て、協働モデリング（`COLLABORATIVE-MODELING.md`）で検証するとよい。
