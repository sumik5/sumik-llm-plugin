# モジュール性成熟度指標（MMI: Modularity Maturity Index）

> 要旨: MMI は、モジュール性・階層性・パターン一貫性という3つの評価軸を重み付けして合成し、レガシーシステムの構造的な成熟度を 0〜10 のスケールで数値化する評価指標である。算出結果は「技術変革を先にすべきか、戦略変革から着手できるか」という変革の出発点を選ぶための判断材料になる。

---

## 1. MMIとは何か

MMI はモノリスかマイクロサービスかを問わず、ソフトウェアシステムがどれだけドメイン指向の分解に適した状態にあるかを、モジュール性・階層性・パターン一貫性という3つのカテゴリで評価し、0〜10 の1つの数値にまとめたものである。数値は「良い/悪い」の二値判定ではなく、現状を客観視し、変革の優先順位を決めるための出発点として使う。

MMI の主な用途:
- レガシーシステムの現状把握（このスキル全体の入口となる評価）
- 新規開発中のシステムの健全性モニタリング
- 変革の進捗を継続的に追跡する定点観測指標

## 2. 3つの評価軸と重み付け

### 2.1 モジュール性（重み 45%）

もっとも重みが大きい軸。プログラム単位（レイヤー・モジュール・パッケージ・クラス・メソッド）の内部が高凝集で、他の単位とは疎結合になっているかを問う。定量計測だけでは判断しきれず、レビュアーによる定性判断を要する。

評価の切り口:
- **均整の取れた粒度**: 同じ階層にあるプログラム単位同士が、コード量や下位要素数の点で同程度の大きさになっているか。極端に肥大した単位は分割候補として要注意
- **単一責務**: 各プログラム単位について「この単位の仕事は何か」に一言で答えられるか。名前が曖昧な単位は責務も曖昧であることが多い
- **結合による凝集**: あるブロックの下位要素同士が、兄弟要素よりも他のブロックの要素と強く結びついていないか。強く結びついている場合は所属先の見直しが必要
- **目標アーキテクチャと実アーキテクチャの一致度**: 設計時に意図した構造（目標アーキテクチャ）と、実装後にできあがった構造（実アーキテクチャ）がどれだけ一致しているか

モジュール性はさらに、次の3つのサブカテゴリへ大まかに分解できる（カッコ内はモジュール性45%のうちの目安の内訳）。

| サブカテゴリ | 内容 |
|---|---|
| 技術・ドメイン指向のモジュール化（最も重い） | ソースコードがドメインモジュール／技術レイヤーへどれだけ明確に帰属しているか、両者のサイズ比率、レイヤーとモジュールのマッピングの明確さ |
| 内部インターフェース | ドメイン指向モジュール・技術レイヤーの少なくとも一方が、パッケージやプロジェクト分割を通じて内部インターフェースを明確に表現できているか |
| サイズ比率 | 極端に大きいクラス・メソッド・パッケージが占める割合、循環的複雑度が高いメソッドの割合 |

### 2.2 階層性（重み 30%）

システムがどれだけ「あらゆる要素が相互依存する塊（Big Ball of Mud）」から遠いかを表す軸。こちらは定量的なメトリクスで計測できる。

評価の切り口:
- **循環に含まれるクラス/パッケージの比率**: 目安として、循環に含まれるクラスの割合が概ね1割未満であれば階層は健全といえる。1〜2割程度なら解きほぐしの作業が必要になり、3割を超えるようであれば構造がかなり崩れているか、そもそも設計されていない可能性が高い。パッケージレベルの循環はクラスレベルよりもやや大きい割合まで許容されるが、割合が上がるほど深刻な問題になる点は同じ
- **循環1つあたりのクラス/パッケージ数**: 数個規模の小さな循環であれば実害は限定的。数百規模の巨大な循環が見つかる場合は、解消に大掛かりな作業を要する
- **技術レイヤリングの違反率**: レイヤー間の呼び出し規約（上位から下位への一方向）がどれだけ守られているか
- **境界づけられたコンテキスト間の関係違反率**: 意図して設計した結合方針が、実際のコードでどれだけ守られているか

階層性も同様に2つのサブカテゴリへ分解できる（カッコ内は階層性30%のうちの目安の内訳）。

| サブカテゴリ | 内容 |
|---|---|
| 技術レイヤリング・ドメイン構造の違反（やや重い） | レイヤー間の呼び出し規約、境界づけられたコンテキスト間の結合方針に対する違反の割合 |
| クラス・パッケージの循環（やや重い） | 循環に含まれるクラス／パッケージの割合、循環1つあたりの規模 |

### 2.3 パターン一貫性（重み 25%）

システムで意図されたアーキテクチャ・設計パターンがどれだけ一貫して適用されているかを表す軸。パターンはクラスやパッケージの責務にルールを課すことで凝集を高め、許される関係を制限することで結合を減らす役割を持つ。

評価の切り口:
- **パターン適用率**: アーキテクチャ・設計パターンに沿って書かれているソースコードの割合。パターンに沿わない部分ほど保守が難しくなる傾向がある
- **パターンの明示性**: 命名規則・継承関係・アノテーション等によって、そのクラスがどのパターンの一部であるかが視覚的に判別できるか。明示性が高いほど、パターンの一貫した運用が定着しやすい

### 2.4 重み付けの根拠

モジュール性が最も重く扱われる（45%）のは、モジュール性が階層性・パターン一貫性の土台になるという位置づけによる。階層性（30%）はそれに次ぐ基盤要素として扱われる。パターン一貫性（25%）がやや軽いのは、パターンによる構造化が開発現場全般にまだ十分浸透しているとは言い切れないという実務的な判断による。

## 3. 採点の考え方（0〜10スケール）

MMI の集計は次の2段階構造をとる。

1. 個々の評価観点（例: 循環に含まれるクラス比率、責務の明確さ など）ごとに 0〜10 の点数を付ける
2. カテゴリ内の観点の点数を平均し、そのカテゴリの重み（45% / 30% / 25%）を掛けて合算する

採点方式は観点によって異なる:

| 採点方式 | 特徴 | 該当する観点の例 |
|---------|------|-----------------|
| 線形 | 最低ライン（0点）と理想ライン（10点）の間を均等な刻みで補間する | ドメインモジュールへのソースコード割当率 など |
| 対数的 | 値が悪化するほど採点の刻み幅が急激に厳しくなる。値の大きい側ではわずかな改善が点数に反映されにくい | 循環に含まれるクラス比率、循環1つあたりのクラス数 など |
| 3段階の定性判定 | 「該当なし（0点）」「部分的に該当（5点）」「完全に該当（10点）」の3段階で採点する簡易運用 | 「明確な責務を持つか」等の定性観点 |

> 🔴 上記は採点の考え方の骨格を示すものであり、原典が定める詳細な11段階のしきい値表（各観点ごとの厳密なパーセンテージ・件数の閾値）はここでは転記しない。実務でMMIを採用する場合は、自組織のコードベースの分布から独自の基準値を再定義することを推奨する。重要なのは「モジュール性45%・階層性30%・パターン一貫性25%」という重み構造と、観点ごとに0〜10へ正規化して合成するという算出方法の考え方であり、絶対値の閾値そのものではない。

## 4. 算出例（簡易シミュレーション）

3軸をどう合成するかを、独自の簡易な数値例で示す。あるシステムについて、モジュール性 7.0点・階層性 5.0点・パターン一貫性 6.0点という評価が出たとする。

```
MMI = モジュール性 × 0.45 + 階層性 × 0.30 + パターン一貫性 × 0.25
    = 7.0 × 0.45  + 5.0 × 0.30 + 6.0 × 0.25
    = 3.15        + 1.50       + 1.50
    = 6.15
```

この場合の MMI は 6.15 となり、「MMI 4〜8: 技術レイヤード」の帯に該当する。実際にはさらに、各軸を構成する複数の観点（2章参照）ごとに 0〜10 点を付け、その平均をカテゴリ得点として使う。上記はあくまで3軸を合成する最終段階のみを簡略化した例である。

## 5. 評価の進め方（レビュー運用）

- **定量指標**は静的解析やアーキテクチャレビューツールで自動計測できる（循環比率、レイヤリング違反率など）
- **定性指標**（責務の明確さ、パターンの一貫性など）は、レビュアー一人の主観に委ねず、必ずペアでレビューし、結果をより大きなレビュアーグループで議論して比較可能性を担保する
- ソースコードを読むだけでは判断がつかない項目は、開発者・アーキテクトを交えたワークショップ（対面またはリモート）で補う。アーキテクチャレビューツールを使い、複数の視点から構造を議論する進め方が実務的
- 自己流でMMIを算出するのが難しい場合は、技術変革・戦術変革・チーム編成それぞれの章末に用意された自己評価チェックリストを代替手段として使うとよい（`TECHNICAL-STABILIZATION.md` / `DOMAIN-KNOWLEDGE-IN-CODE.md` / `TEAM-ORGANIZATION.md` 参照）

## 6. MMIによる変革経路の選択

MMI は 0〜10 のスケールを3つの帯に分け、それぞれで典型的なアーキテクチャの状態と推奨される経路が変わる。

### 6.1 MMI 0〜4（赤信号）: Big Ball of Mud

構造と呼べるものがほとんど存在せず、あらゆる要素が相互に依存している状態。技術コードと業務ロジックが渾然一体となっており、ドメインモデルへ変換できそうなクラスを見分けることすら難しい。バグの多さやセキュリティ・インフラ面の課題を抱えていることも多い。

この状態でいきなり戦略変革（境界づけられたコンテキストへの分解）に踏み出しても土台が無く長続きしない。推奨する順序は次のとおり:

```
技術変革（ビルド/デプロイの自動化・依存更新・テスト増強・エラー堅牢性の改善）
      ↓
戦術変革（ドメイン知識によるモデルの強化）
      ↓
戦略変革（4ステップによるドメイン駆動の分解）
```

### 6.2 MMI 4〜8（黄信号）: 技術レイヤード

技術的なレイヤー構造（UI → アプリケーション → ドメイン → データベース）は概ね一方向に保たれているが、ドメインに沿った垂直方向の分割はほとんど存在しない状態。ここでは2つの経路に分岐する。

| 経路 | 適用条件 | 進め方 |
|------|---------|--------|
| **Path A（戦術先行）** | ドメイン層そのものが Big Ball of Mud 化している（データベースのテーブル構造がそのままコードへ転写されているようなケースに多い） | まず戦術変革でドメインモデルを強化し、その後に戦略変革で境界づけられたコンテキストへ分解する |
| **Path B（戦略先行）** | ドメイン層に、多少の依存循環はあっても認識可能な構造が残っている | まず戦略変革で境界づけられたコンテキストへ分解し、その後に戦術変革でモデルを強化する |

判断が難しい場合は **Path B を既定**とする。ドメイン層の状態が明確に劣悪と分かっているときのみ Path A を選ぶ。両者は排他的な二択ではなく、「一部を戦略的に切り出す → その部分だけ戦術的に強化する → 次を切り出す」という反復的な組み合わせも実務上有効である。コアドメイン（最も重要な境界づけられたコンテキスト）については、支援・汎用サブドメインより優先的にドメイン知識での強化を行うことが推奨される。

**AskUserQuestion 実装例**:
```
AskUserQuestion(questions=[{
  "question": "MMIが4〜8の技術レイヤード構成です。ドメイン層はBig Ball of Mud化していますか？",
  "header": "変革経路の選択",
  "options": [
    {"label": "Path B（戦略先行）", "description": "ドメイン層に一定の構造が残っている、または判断がつかない場合の既定選択。先に境界づけられたコンテキストへ分解する"},
    {"label": "Path A（戦術先行）", "description": "ドメイン層がBig Ball of Mud化している場合。先にドメインモデルを強化してから分解する"}
  ],
  "multiSelect": false
}])
```

### 6.3 MMI 8〜10（緑信号）: ドメイン駆動モジュラー

すでにドメインに沿ったモジュール分割が実現している状態。ただし、モジュール間の一部の関係はなお整理の余地があり、実装されているモジュールが「本物の」境界づけられたコンテキストと呼べるかどうかも検証が必要なことが多い。4ステップの戦略変革を一巡させ、微調整を積み重ねる運用が適している。

## 7. スコア帯と変革アプローチの関係

MMI のスコア帯は、そのまま「置換するか改造するか」を機械的に決めるものではないが、目安として次の傾向がある。

- **0〜4帯**: 技術的負債が深く構造の再現性も低いため、境界を少しずつ切り出しながら置き換える段階的な移行が現実的な選択肢になりやすい。全面的な一斉置換は「未知の未知」のリスクが大きく、既定では避ける
- **4〜8帯**: 技術的な土台は保たれているため、リファクタリングを中心とした改造（リシェイピング）が選択肢になりやすい
- **8〜10帯**: 大規模な置換や作り直しは不要で、既存構造を維持したままの微調整で十分なことが多い

変革アプローチ自体の比較軸・選定指針は `TRANSFORMATION-APPROACHES.md` に委譲する。本節はあくまで MMI のスコア帯との大まかな対応関係を示すものである。

## 8. アーキテクチャレビューツールとの関係

MMI の定量指標（循環比率、レイヤリング違反率など）は、静的解析を行うアーキテクチャレビューツールで計測するのが実務的である。ツールで現状の依存関係を可視化し、その上に目標アーキテクチャ（意図した境界づけられたコンテキストやレイヤー構造）を重ね合わせることで、両者のズレを直接確認できる。この可視化は MMI 算出の材料であると同時に、現状と目標を突き合わせるステップ（`STRATEGIC-STEPS-ALIGNMENT-EXECUTION.md` のStep3）でも再利用される。
