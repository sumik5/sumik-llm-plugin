# アーキテクチャ概念（変革文脈での評価軸）

レガシー変革を成功させるには、対象システムの構造を「見える化」し、モジュール性・凝集・結合という基準で現状を評価できる必要がある。本参照は、変革の意思決定に直接使う概念（C4モデル・モジュール性・凝集/結合・Big Ball of Mud・アーキテクチャスタイル）を、判断軸として整理する。Clean Architecture の詳細な同心円モデルやDDD戦術パターンの定義は `applying-clean-architecture` に譲り、ここでは「変革の現状把握・目標設定にどう使うか」に絞る。

---

## 1. C4モデルによるアーキテクチャの可視化

変革の第一歩は、対象システムの現在のアーキテクチャを関係者全員が同じ粒度で理解できる形にすることである。C4モデルは、粗い粒度から細かい粒度へ4段階でドリルダウンする軽量な可視化手法で、変革プロジェクトの共通言語として使いやすい。

```
Level 1  Context    : システム全体 ⇄ 利用者・外部システムとの関係
             ↓ drill down
Level 2  Container  : デプロイ可能な単位（プロセス）とその関係
             ↓ drill down
Level 3  Component  : 1つのコンテナ内に存在するまとまり
             ↓ drill down
Level 4  Code       : クラス・メソッドレベルの詳細設計
```

各レベルで「人・要素・関係」を描く点は共通で、より細かいレベルは1つ上のレベルの要素を分解したものになる。

**変革での使い方**:
- Context/Container図は、ステークホルダーとの合意形成・現状把握（後述の戦略的変革Step1〜Step3）で使う共通資料になる。
- Component/Code図は、境界づけられたコンテキストの候補を技術的に裏づける材料として使う。
- 図がまだ存在しない場合は、変革の初期タスクとして関係者を集めて描き起こすこと自体に価値がある（現状の暗黙知を可視化する効果）。

## 2. モジュール性 — 情報隠蔽と狭いインターフェース

プログラムを構造化せず全データに自由にアクセスできる状態は、変更の影響範囲が読めなくなり破綻する。この反省から生まれたのが「モジュール」という単位であり、モジュールの実装は狭いインターフェースの内側に隠すべきだという原則（情報隠蔽）が確立した。

```
┌───────────────────────────┐
│           モジュール          │
│   ┌─────────────────┐   │
│   │   内部実装（隠蔽）   │   │
│   └─────────────────┘   │
│          ▲                │
│     狭いインターフェース      │
└──────────┼───────────────┘
           │
     呼び出し側は実装の詳細を知らずに使える
```

情報隠蔽が効いていると、呼び出し側の認知負荷が下がり（実装を理解する必要がない）、インターフェースさえ変わらなければ実装は自由に変更できる。

この考え方はメソッド・クラス・パッケージ・アセンブリなど、あらゆる粒度に適用できる普遍的な概念である。DDDの文脈では、**境界づけられたコンテキスト**とその内部の戦術的な構成要素（詳細は `applying-clean-architecture` の DDD 参照群）が、この意味でのモジュールに相当する。変革プロセスにおいて「モジュールの境界をどこに引くか」を決める基準が、次節の凝集と結合である。

---

## 3. 凝集と結合 — モジュール分割の判断基準

モジュールを変更するとき、開発者はそのモジュールと隣接モジュールの両方をある程度理解する必要がある。モジュール内部が濃密にまとまり、他モジュールとの関わりが少ないほど、この理解コストは下がる。

| 用語 | 定義 |
|------|------|
| **凝集（cohesion）** | モジュール内部の要素が概念的・機能的にどれだけ強く結びついているか |
| **結合（coupling）** | モジュール間の依存の度合い。依存が多いほど結合は強い |

**高凝集・疎結合**は変更容易性の土台であり、逆の状態（低凝集・密結合）が行き着く先が Big Ball of Mud である。

### 3.1 DDDにおける凝集と結合

戦略レベルでは「境界づけられたコンテキストの中のモデルは、単一の統一された目的の周りで高凝集であるべき」という原則が重視される。戦術レベルでは「クラス間の相互依存が強いほど、モデルと設計は理解・テスト・保守が難しくなる」という原則が繰り返し語られ、他クラスへの参照をほとんど持たない「スタンドアロンクラス」が疎結合の極致として位置づけられる。値オブジェクトはオブジェクトツリーの葉であるため、この理想にもっとも自然に近づける構成要素である。

### 3.2 ドメイン的な疎結合 vs 技術的な疎結合

> ⚠️ **注意**: 「疎結合」という言葉には誤解が伴いやすい。
>
> 一部の開発者は「同期的な直接呼び出しをやめてメッセージ／イベントをバス経由で送るようにすれば自動的に疎結合になる」と考える。しかし変革の文脈で重要な疎結合は**ドメインの観点での依存の少なさ**であり、技術的な通信方式とは別軸である。
>
> メッセージを大量にやり取りするシステムは技術的には疎結合に見えても、変更のたびに「メッセージの送り先・理由・返信内容」まで理解しないといけないなら、実質的な結合はモジュール単体を理解するより大きくなる。**ドメインに基づく狭いインターフェース**を持つことが本質であり、通信方式の変更だけでは疎結合は得られない。

凝集・結合はDDDが目指すゴールそのものであり、その対極にあるのが次に扱う Big Ball of Mud である。

## 4. Big Ball of Mud（BBoM）

高結合・低凝集で絡み合った構造は「Big Ball of Mud（泥だんご）」と呼ばれる。BBoM内部の依存は非巡回ではなく、循環依存を形成している点が特徴。

### 4.1 クラスレベルの兆候

アーキテクチャレビューツールでソースコードから自動生成した依存グラフでは、数十万行規模のJavaシステムで数百クラス・数十パッケージにまたがる循環依存の塊が観測されることがある。同じパッケージに属するクラス（色分けで識別可能）が、依存グラフ全体にばらばらに散らばっている場合、上位レベルの秩序（パッケージによる構造化）がほとんど機能していないことを示す。同じ業務機能を扱うクラスを同じパッケージにまとめる、という単純な原則がまず出発点になる。

### 4.2 アーキテクチャレベルの兆候

プロジェクト（デプロイ単位）同士の依存関係を俯瞰したとき、「ほとんどのプロジェクトが他の多数のプロジェクトから使われ、かつ他の多数のプロジェクトに依存する」状態もBBoMの一形態である。要するに、**依存関係の総数をコントロールしてこなかったシステムはすべてBBoMに至る**。依存を制御できていなければ、アーキテクチャを制御できているとは言えない。

### 4.3 BBoMはなぜ生まれるか

システムは時間とともに理解しやすさと保守性が下がり、変更コストが上がっていく傾向がある。この劣化を「偶有的複雑性」の蓄積として図式化すると、次のようになる。

```
偶有的複雑性
  高 │                                     ╱─→ 不良なレガシー系の領域
     │                              ╱───╯
     │                        ╱───╯   （放置すると複雑性が積み上がる）
     │                  ╱───╯
     │            ╱───╯
  低 │  良いレガシー系の領域（拡張と改善を往復し続けられる限り、ここに留まる）
     │  ∿∿∿∿∿∿∿∿∿∿∿∿∿∿∿∿∿∿∿∿∿∿∿∿∿∿∿∿∿∿
     └──────────────────────────────────→ 時間
        拡張のたびに複雑性は増える → 定期的な改善で押し戻す、を繰り返す
```

開発当初は高凝集・疎結合な設計でも、機能追加のたびに偶有的複雑性は必ず増える。これは学習過程の自然な結果であり問題ではない。問題は、**その複雑性を定期的なアーキテクチャ改善で押し戻す**という往復運動を維持できるかどうかである。この往復を怠ると、機能追加のたびに開発速度が落ち、バグの追跡が難しくなり、最終的に「不良なレガシー系」に至る。この往復運動こそ、変革プロセス全体を反復的に設計する根拠になる。

### 4.4 結合と再利用という誤解

古いシステムを分析すると、開発チーム自身が「なぜここまで結合度が高くなったのか」と驚くことが少なくない。原因の多くは「既存のコードは再利用すべきで、重複コードは絶対悪」という信念にある。既存コードが今のニーズに完全には合わなくても、それを抽象化・パラメータ化・条件分岐で無理に使い回そうとする判断が、結合を静かに増やし続ける。アーキテクチャを見直す機会があれば、モジュール・パッケージ・クラス間の結合を減らすことを最優先にすべきである。

## 5. アーキテクチャスタイル

変革でよく検討されるスタイルは、モジュラーモノリス・マイクロサービス・ヘキサゴナル/オニオン/クリーンアーキテクチャ・Self-contained Systemsである。以下、それぞれの位置づけと変革プロセスでの意味を整理する。

### 5.1 モノリス vs マイクロサービス

「モノリス」という言葉は否定的な意味を帯びがちだが、本来は**デプロイ単位が1つ**であることを指すに過ぎない。単一デプロイ単位が問題になるのは、次のような場合である。

- あるチームの変更頻度が、システム全体を担当する他チームより著しく高い
- システムが大きくなりすぎてテストやリリースが遅くなっている
- モジュールごとに異なるスケーリング要件がある

マイクロサービスは独立してデプロイできる複数のプログラムでシステムを構成する。技術的な利点（スケーラビリティ等）よりも、**チームが互いに待たずに独立して開発を進められる**という組織的な利点の方が本質的である。1つのマイクロサービスは1つ以上の境界づけられたコンテキストを実装し、1つの境界づけられたコンテキストは（他のコンテキストと共有せず）単一のマイクロサービスにデプロイされるのが望ましい対応関係になる。

### 5.2 モジュール性 × 分散度の4象限

密結合なモノリスを「とりあえずデプロイ単位だけ分割する」と、最悪の組み合わせである**分散版Big Ball of Mud**に陥る。内部結合の問題はそのままに、分散システム特有の複雑性まで抱え込むためである。

```
モジュール性
  高 │  モジュラーモノリス      │  マイクロサービス
     │ （まずここを目指す）     │  （最終形態）
     ├────────────────────┼────────────────────
  低 │  Big Ball of Mud       │  分散版 Big Ball of Mud
     │ （出発点）             │ （最悪の組み合わせ：安易な分散化の罠）
     └────────────────────┴────────────────────→
              低                        高
                     分散度（デプロイ単位の数）
```

安全な経路は、まず**左下→左上**（境界づけられたコンテキストを形成し、モジュラーモノリスへ）へ進み、その後**左上→右上**（コンテキストを個別のデプロイ単位へ切り出し、マイクロサービスへ）へ進む段階的な移行である。分散度を先に上げてはならない。

### 5.3 レイヤードアーキテクチャ

上位層が下位層に依存し、その逆はない、という一方向の階層構造。「厳格なレイヤード」（直下の層しか使えない）と「緩やかなレイヤード」（下位のどの層も使える）の2種類がある。ネットワークプロトコルの階層化と同様、抽象を積み上げる汎用パターンとして古くから使われてきた。

### 5.4 インサイドアウト系アーキテクチャ（変革での位置づけ）

「上下」の関係だけでは現代的な設計を説明しきれなくなり、「内外（inside-out）」という捉え方に発展したのが、ヘキサゴナル（Ports & Adapters）・オニオン・クリーンアーキテクチャの系譜である。詳細な同心円モデル・依存性ルール・コンポーネント原則は `applying-clean-architecture` に委譲するが、変革の文脈で押さえるべき要点は次の2つ。

1. **業務コードと技術コードの分離**という目的は共通しており、これがそのまま「戦術変革（第8章の内容）」の到達目標になる。
2. アプリケーションコアはUI・DB・外部インターフェースを知らずにテストできる状態を目指す。レガシーの安定化・貧血モデルの解消（`TECHNICAL-STABILIZATION.md` / `DOMAIN-KNOWLEDGE-IN-CODE.md`）は、この状態に近づくための具体的な手段である。

### 5.5 アーキテクチャハンバーガー — 上下と内外の統合

「内外」だけでは UI 層とインフラ層が同じ外周リングに同居してしまい、両者に依存関係がないという事実を表現しにくい。そこで、DDDの4層構造を上下関係として捉え直した図式が次のものである。

```
┌─────────────────────────┐
│   UIレイヤー（MVC / MVP / MVVM）  │
├─────────────────────────┤
│   アプリケーションレイヤー          │  ← ユースケース／アプリケーションサービス
├─────────────────────────┤
│   ドメインレイヤー（中核＝パティ）   │  ← 戦術的DDDパターン
├─────────────────────────┤
│   インフラレイヤー                │
└─────────────────────────┘
        すべての依存はドメインレイヤーへ向かう
```

ドメインコアを中心に、UI・インフラという「具材」を自由に載せ替えられる、というイメージ。アプリケーションレイヤーは全システムに存在するとは限らない（あってもなくてもよい）点も特徴。この様式はコストが高いため、**中核業務領域（core subdomain）でのみ**採用するのが実践的な落とし所である。

### 5.6 サービス指向アーキテクチャ（SOA）とアンチパターン

SOAは、サービスを中心概念とする全社的なITアーキテクチャで、各サービスが1つの業務機能を実装・支援する。エンタープライズサービスバス（ESB）経由でアプリケーションから利用される構成が典型的である。

> ⚠️ **entity-serviceアンチパターン**: 実際に見かけるSOAの多くは、中心的なエンティティ1つにつきサービスを1つ作る設計に陥っている。これは肥大化した巨大サービスを生み、あるサービスへの変更が必要になったチーム全員が、そのサービスの担当チームの実装完了を待たなければならなくなる。結果としてチームの自律性が失われ、DDDが目指す「独立して作業できる自己完結モジュール」からは対極の状態になる。

同様の罠は、マイクロサービスでも「**1アグリゲート = 1マイクロサービス**」という単純化された規則として現れることがある。アグリゲートはエンティティ群に対するトランザクション境界であって、そのままデプロイ単位にすべきという意味ではない。DDDが目指すのは「1つの境界づけられたコンテキストが、業務プロセスの一部（1つのビジネスケイパビリティ）を実装し、必要な集約・エンティティをすべて内包した上で、1つのデプロイ単位に収まる」という対応関係である。

既存の大規模基幹システムをすべてマイクロサービス化する必要はない。SOAとマイクロサービスが混在し、各マイクロサービスが自分のデータ管理を持てる形であれば、十分に機能する構成になり得る。

### 5.7 Self-contained Systems（SCS）

マイクロサービスの「マイクロ」という語感が誤解を招きやすいことから生まれた、独立した複数のシステムの協調で全体を構成するスタイル。SCS間の通信はできる限り非同期にすることで、システム同士を疎結合にし、障害の影響範囲を抑え、自律性を高める。

### 5.8 ITランドスケープの振り子

システムをSCS/マイクロサービス的に分割すると、各システムが自前のデータベースを持ち、インターフェース経由で連携する構成になる。これは実は、2000年代初頭の「独立したシステム群がインターフェースで連携するITランドスケープ」と驚くほど似た構造である。当時はデータの不整合を避けるため、顧客データなどを一箇所に集約する「唯一の万能サービス（canonical model）」を目指してSOAへ向かったが、万能サービスは実践では「誰のニーズにも中途半端にしか応えない汎化の塊」になりがちだった。今日、業界は再び「自前のデータベースを持つ独立したサブシステム」へと回帰しつつあり、それを今はマイクロサービスやSCSと呼んでいる、という見方ができる。歴史は繰り返しているが、今回は「独立チームが共通の全体像を持って開発する」という組織的な工夫が加わっている点が異なる。

## 6. 変革文脈での評価軸まとめ

現状のアーキテクチャを評価するときは、次の軸で見ると変革の出発点が明確になる。

| 評価軸 | 見るべきポイント | 悪化のサイン |
|--------|-----------------|--------------|
| 凝集 | 業務機能ごとにまとまっているか | 同じ業務機能のコードが複数モジュールに散在 |
| 結合 | ドメイン的に狭いインターフェースになっているか | 技術的には疎結合でも意味的に密結合（チャットが多い） |
| モジュール性 | パッケージ／サービス境界が業務境界と一致しているか | 循環依存、色（業務単位）がグラフ上でばらける |
| デプロイ形態 | 分散度とモジュール性が揃っているか | モジュール性が低いまま分散度だけ上げている |
| 依存の意味 | 1エンティティ=1サービスのような技術的分割になっていないか | entity-serviceアンチパターン、1アグリゲート=1サービス |

この評価は、後述する定量的な成熟度評価（`MODULARITY-MATURITY-INDEX.md`）の土台になる。
