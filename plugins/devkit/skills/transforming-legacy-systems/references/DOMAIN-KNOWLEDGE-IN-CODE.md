# ドメイン知識のコードへの注入 / Domain Knowledge in Code

技術的安定化を終えたら、次はドメイン知識でソースコードを強化する段階に進む。凝集度を高め結合度を下げる具体的なリファクタリングと、コードでアーキテクチャを可視化する方法を扱う。

---

## 0. 全体像: 凝集度と結合度という2つの軸

モジュール性の基本原則である**凝集度（cohesion）**と**結合度（coupling）**を手がかりにすると、ドメイン知識でソースコードを強化する施策を整理しやすい。大きく次の4カテゴリに分類できる。

| カテゴリ | 施策 |
|---------|------|
| 前提 | 業務コードと技術コードを分離する |
| 凝集度を高める | モデリングをドメイン知識で充実させる／値オブジェクトを活用する／ドメイン指向の事前条件を導入する／エンティティの同一性を明示する |
| 結合度を下げる | 集約境界でID参照を使う／業務コードの継承を減らす／ドメインイベントで疎結合にする |
| 仕上げ | コードでアーキテクチャを文書化する |

これらの施策はいずれもリファクタリングによって実現する。前段の技術的安定化で整えたテストと、AIコーディングアシスタントの支援は、ここでも変わらず土台になる。

---

## 1. 業務コードと技術コードを分離する

ドメイン駆動設計は、その多くの部分が「ドメインに基づくコード」、すなわち**アプリケーションコア**（業務コード）を対象にしている。アプリケーションコアは、技術的な都合で存在するコード（永続化・UI・外部連携などの技術コード）に周囲を取り囲まれている、というのが基本のイメージである。ヘキサゴナル/オニオン/クリーンアーキテクチャなど、この数十年で提唱されてきた多くのアーキテクチャスタイルは、この構図を実現する設計パターン群として理解できる（詳細は`devkit:applying-clean-architecture`に委譲する）。

レガシーシステムの戦術的変革では、技術コードとアプリケーションコアがどの程度絡み合っているかを点検し、絡み合いを解くリファクタリングを検討する。着眼点は次の通りである。

| 着眼点 | 兆候 | 対応 |
|--------|------|------|
| 業務ロジックへのデータベースクエリの混入 | アプリケーションコアの中に直接クエリが書かれている | リポジトリパターンでデータベースアクセスを分離する。極端に古いシステムではUI層にまでクエリが漏れていることもあり、この場合はより大掛かりな分離作業が必要になる |
| データベース側への業務ロジックの漏出 | ストアドプロシージャやトリガーが技術的な用途を超えて業務ルールを含んでいる | 可能であれば業務ロジックをドメイン層へ引き上げる。難しい場合は、少なくとも業務ロジックと技術的な処理を別のプロシージャに分ける |
| UIへの業務ロジックの漏出、あるいは業務コードによるUI表示の直接制御 | 業務クラスが自分自身の表示形式を組み立てている、逆にUIコードの中に本来ドメインモデルにあるべきロジックがある | UIから業務クラスを抽出する、あるいはUIからサービスを抽出するリファクタリングで境界を引き直す |
| 技術基底クラスへの実装継承 | アプリケーションコアのクラスが技術的な基底クラスを継承している | 委譲による分離を検討する。ただしフレームワークによっては継承が要求されることもあり、その場合は該当フレームワークの更新・置き換えまで解消を待つ判断もありうる |

## 2. 凝集度を高める

### 2.1 貧血ドメインモデルを解消する

ドメイン駆動設計の基本は、ユビキタス言語をもとに、メソッドそのものがドメインの語彙で語られる「振る舞いの豊かなドメインモデル」を育てることにある。しかし実務では、その正反対の状態、すなわち**貧血ドメインモデル**にしばしば出会う。クラスがほぼgetterとsetterしか持たず、実質的にデータの入れ物でしかない状態を指す、広く知られたアンチパターンである。

> **注意**: 厄介なのは、クラス名や属性名だけはドメインの言葉で名付けられているため、一見するとドメインをきちんとモデル化しているように見える点にある。しかしよく見ると、これらのクラスはドメインの「データ」だけをモデル化しており、ドメイン指向の「振る舞い」を何も持っていない。インターフェースにはgetterとsetterしかなく、オブジェクト指向の基本原則であるカプセル化が損なわれている。

以下は、ポイント残高を管理する会員クラスを例にした、貧血ドメインモデルの再構成例である。

```typescript
// 悪い例: 貧血ドメインモデル
class MemberPoints {
  private balance: number = 0;

  setBalance(amount: number): void {
    this.balance = amount;
  }

  getBalance(): number {
    return this.balance;
  }
}
```

このクラスに何が問題なのだろうか。

| 問題 | 説明 |
|------|------|
| ドメインの語彙が失われる | 現場の担当者は「残高を設定する」とは言わず、「会員がポイントを獲得する／利用する」と言う。貧血な実装からは、このクラスが何のために存在するのかがメソッド名からは読み取れない |
| 誤用されやすい | ポイントの獲得も利用も同じ`setBalance()`で表現されてしまい、意味的に大きく異なる操作を区別できない |
| あり得ない状態が表現できてしまう | 残高を任意の値へ直接書き換えられるため、業務上あり得ない値も簡単に設定できてしまう |
| 一貫性違反が容易に起こる | 新しい残高の計算がクラスの外側（呼び出し側）に漏れ出し、クラス自身が自分の一貫性を守れなくなる |

同じドメインを、振る舞い中心に書き直すと次のようになる。

```typescript
// 良い例: 振る舞いが豊かなドメインモデル
class MemberPoints {
  private balance: number = 0;

  earn(amount: number): void {
    this.balance += amount;
  }

  redeem(amount: number): void {
    this.balance -= amount;
  }

  getBalance(): number {
    return this.balance;
  }
}
```

`earn()`・`redeem()`という2つのメソッドが`setBalance()`を置き換えている。呼び出し側は「残高をどう計算するか」を知る必要がなくなり、クラス自身が自分の一貫性を守れるようになる。これがDDDでいう**意図が伝わるインターフェース（intention-revealing interface）**である。

貧血ドメインモデルを見つけたら、初期段階で有効な3つのリファクタリング指針がある。

1. **ドメイン言語の徹底**: 変数・メソッド・クラス・データベーステーブルなどの識別子を、ドメインの言葉に揃える。現場の用語をそのまま使った方が業務専門家にとって理解しやすい場合は、無理に英語化しない判断も含む
2. **setterの見直し**: setterを、意味のあるドメイン指向のメソッドに置き換えられないか検討する。UI表示のために必要なgetterは残ってよいことが多い
3. **値オブジェクトの導入**: プリミティブ型を値オブジェクトに置き換える（次節で詳述）

戦術的変革は、戦略的変革（境界づけられたコンテキストへの分解）の前に行っても後に行ってもよい。どちらの順序を取るかは、システムとチームの状況次第である。

### 2.2 値オブジェクトを活用する

DDDの戦術的パターンの中でも、**値オブジェクト**は比較的小さな単位で段階的に導入しやすいビルディングブロックである。狙いは、業務ロジックからプリミティブ型を追い出し、ドメイン指向の型に置き換えることにある。これにより、プログラミング言語の型システムを使って表現力と型安全性の両方を高められる。

先ほどのポイント残高を`number`のままにしておくと、次のような問題が起きる。

- ポイント以外の任意の数値（重量・距離など）を誤って加算できてしまう
- 単位（ポイントなのか、金額なのか）が型からは分からない

そこで、次のような値オブジェクトを導入する。

```typescript
type PointUnit = "LOYALTY" | "PROMOTIONAL";

class PointAmount {
  constructor(
    private readonly value: number,
    private readonly unit: PointUnit
  ) {}

  add(other: PointAmount): PointAmount {
    return new PointAmount(this.value + other.value, this.unit);
  }

  subtract(other: PointAmount): PointAmount {
    return new PointAmount(this.value - other.value, this.unit);
  }

  getValue(): number {
    return this.value;
  }
}
```

現代の言語の多くは、値オブジェクトを不変なレコード型・データクラスとして簡潔に書ける構文を提供しており、コンストラクタ・等価比較・文字列表現といった定型コードを大幅に削減できる。既存のコードベースが対応するバージョンであれば、こうした言語機能への移行も合わせて検討する価値がある。

このままではまだ実用に耐えない部分もある（等価比較の実装、単位が異なる値同士の加減算の扱いなど）が、単位や通貨のような「量」を安全に表現するモデリングは発展的なテーマとしてここでは深入りしない。

会員クラスに組み込むと、`earn()`・`redeem()`のシグネチャに`PointAmount`という型が現れるようになる。誤って生の数値を渡すことはコンパイル時に検出されるようになり、意図が伝わるインターフェースがさらに強化される。

```typescript
class Member {
  private points: PointAmount;

  earn(amount: PointAmount): void {
    this.points = this.points.add(amount);
  }

  redeem(amount: PointAmount): void {
    this.points = this.points.subtract(amount);
  }
}
```

### 2.3 ドメイン指向の事前条件を導入する（契約による設計の応用）

技術的安定化の段階では、契約による設計を主にnull対策など技術的な堅牢性のために導入した。ここでは同じ考え方を、業務ルールを表現するために一段深く使う。

異なる単位のポイントを加算すると誤った結果になる、という問題を防ぐには、値オブジェクトのメソッドにドメイン指向の事前条件を追加すればよい。会員クラス側にも、「利用できるポイントを超えて利用しようとしていないか」という事前条件を追加できる。

```typescript
class PointAmount {
  add(other: PointAmount): PointAmount {
    assert(this.unit === other.unit, "unit mismatch");
    return new PointAmount(this.value + other.value, this.unit);
  }
}

class Member {
  private points: PointAmount;

  isRedeemable(amount: PointAmount): boolean {
    return amount.lessThanOrEqualTo(this.points);
  }

  redeem(amount: PointAmount): void {
    assert(!amount.isNegative(), "amount must not be negative");
    assert(this.isRedeemable(amount), "insufficient points");
    this.points = this.points.subtract(amount);
  }
}
```

ここで重要なのは、事前条件のチェックに使う判定（`isRedeemable()`）を、呼び出し側からも使えるブールメソッドとしてインターフェースに公開している点である。これにより呼び出し側は事前にチェックしてから安全に呼び出せるようになり、契約による設計が本来目指す「契約をインターフェース上で確認できる」状態に近づく。単に名前をドメインの言葉に揃えるだけの表面的な対応より一歩進み、クラスの意味がコードそのものに表現される。

### 2.4 エンティティの同一性を明示する

値オブジェクトと異なり、エンティティは不変の同一性（アイデンティティ）を必要とする。ここでいう同一性は、データベースの技術的な主キーとは別物であり、あくまで業務上意味のある識別子を指す。会員番号・契約番号・注文番号のようなものが典型例である。

この同一性は、他の値と同様に値オブジェクトとして表現し、フィールドはイミュータブルにする。同一性を比較する`equals()`/`hashCode()`（あるいは各言語の等価比較機構）は、この同一性の値だけに基づいて実装する。

```typescript
class MemberId {
  constructor(private readonly value: string) {}

  equals(other: MemberId): boolean {
    return this.value === other.value;
  }
}

class Member {
  constructor(
    private readonly id: MemberId,
    private address: string
  ) {}

  equals(other: Member): boolean {
    return this.id.equals(other.id);
  }
}
```

レガシーシステムのモデリングを見直す際は、こうしたドメイン上の同一性を探し、プリミティブな主キーの代わりに専用の値オブジェクトへ置き換えていく。この改善は、次節で扱う集約境界でのID参照の土台にもなる。

## 3. 結合度を下げる

戦略的変革の核心は「境界づけられたコンテキストを分離し、それぞれが高い凝集度を持てるようにすること」にある。戦術的な水準でも、結合を緩めるリファクタリングには十分な価値がある。

### 3.1 循環依存・依存関係を断ち切る

モジュール成熟度の評価が低いシステムほど、クラス間・パッケージ間の循環依存に注意を払う必要がある。データベースのテーブル関係をそのまま反映した結果、ドメインモデルの全クラスが直接循環的に関係し合っている、という構図によく出会う。このような大きな循環は後のコンテキストへの分解を妨げるため、断ち切っておく必要がある。

プログラミング言語の仕様上は循環参照が許されていても、凝集度と結合度の観点からは2つのドメインクラス間の関係がどちらの向きを向くべきかを意図的に決め、直接循環を避けるべきである。パッケージレベルでも同様に、実際に協調して動くクラス同士が同じパッケージに収まるよう再配置すると凝集度が上がり結合度が下がる。絡み合ったクラス循環・パッケージ循環への具体的な対処技法は`devkit:writing-clean-code`にも詳しい。

### 3.2 集約境界でID参照を使う

DDDの**集約**というビルディングブロックは、まさにこの戦術的な境界を引き、結合を緩めるために存在する。多くのレガシーシステムでは、ドメインモデルのクラスが実オブジェクト参照でつながっており、結果として大きな階層ツリー、あるいは最悪の場合は巨大な循環構造を形成している。

データベース構造をそのまま反映した「大きな木」または「単純な循環」程度であれば、実オブジェクト参照を**ID参照**に置き換えるのが有効な戦術的変革の手段になる。「集約は他の集約を同一性で参照する」という設計指針は、集約設計のベストプラクティスとして広く知られている。たとえば、ポイント残高を管理する会員システムで注文が会員を直接参照している構図は、次のようにID参照へ書き換える。

```typescript
// 悪い例: 集約が他の集約を実オブジェクトで直接参照している
class Order {
  constructor(private readonly member: Member) {}
}

// 良い例: 同一性(ID)で他の集約を参照する
class Order {
  constructor(private readonly memberId: MemberId) {}
}
```

これにより、注文だけを単独で扱えるようになり、常に会員オブジェクト全体を一緒に持ち歩く必要がなくなる。集約境界を明確に引くことは、小さな単位での分割を助け、大きな単位（境界づけられたコンテキスト）での分割の準備にもなる。

### 3.3 業務コードの継承を減らす

オブジェクト指向言語が普及して以来、共通機能をスーパークラスに一度だけ実装したいという動機から、業務コードに3段・4段の継承階層が作られてしまう例は少なくない。しかしクラス階層を作ることは詳細の単純化・抽象化を意味し、ドメインを深く掘り下げ新しいサブドメインを取り込むにつれて単純化がそぐわなくなりやすい。継承階層は結合の最も強い形態の一つであるため、こうなってからの手直しは難しくなる。

業務専門家とユビキタス言語を作り直す作業では、専門家の口から用語同士の親子関係が語られることは意外に少なく、むしろ「類似」「同義」という形で語られることが多い。この類似性をクラスモデルで表現するには、**委譲**という単純な仕組みが向いている。あるクラスが自分の役割の一部を別のクラスに委ねる形にすれば、クラス階層ほど強い結合を持たずに構造化できる。実装継承が使われている業務コードの箇所を洗い出し、可能な限り委譲へ置き換えることが、後のシステム分解を大きく容易にする。

### 3.4 リスコフ置換原則（LSP）

実装継承だけを目的にした継承階層は、ソフトウェア工学的にはそもそも誤用である。継承は、サブクラスがスーパークラスに対して「である（is-a）」関係を持つ場合にのみ使うべきである。たとえば「プレミアム会員は会員である」は成り立つが、「ゲスト会員はプレミアム会員である」は成り立たない、といった具合である。

この関係は**リスコフ置換原則（LSP）**として広く知られており、「サブクラスはスーパークラスの代わりに使えなければならない」と要約される。サブクラスのオブジェクトをスーパークラス型の変数に代入しても、その変数を使うすべての場面で正しく動作しなければならない。

これを具体的に確認するには契約による設計の考え方が役立つ。**サブクラスは、スーパークラスのすべての契約を満たさなければならない**——事前条件をスーパークラスより強くしてはならず（同じか弱くてよい）、事後条件をスーパークラスより弱くしてはならない（同じか強くてよい）。つまり、サブクラスは求めるものをスーパークラス以上に増やしてはならず、提供するものをスーパークラス以下に減らしてはならない。

以下は、会員ポイントの例で、is-a関係を満たす継承の再構成例である。標準会員のスーパークラスに対して、プレミアム会員はマイナス残高（一時的な立替枠）を許容するサブクラスとして定義する。

```typescript
class Member {
  protected points: PointAmount;

  isRedeemable(amount: PointAmount): boolean {
    return amount.lessThanOrEqualTo(this.points);
  }

  redeem(amount: PointAmount): void {
    assert(this.isRedeemable(amount));
    this.points = this.points.subtract(amount);
  }
}

class PremiumMember extends Member {
  private advanceLimit: PointAmount;

  // 事前条件を弱める: マイナス残高分だけ多く利用できる
  isRedeemable(amount: PointAmount): boolean {
    return amount.lessThanOrEqualTo(this.points.add(this.advanceLimit));
  }
}
```

`PremiumMember`は`isRedeemable()`をオーバーライドして事前条件を緩めており、`Member`が保証していたすべての契約を満たしたまま、より多くの状況で利用を許可する。これはまさにLSPが要求する「求めるものを増やさず、提供するものを減らさない」という形になっている。一方で、`redeem()`が要求する条件をスーパークラスより厳しくするようなオーバーライドは、LSP違反であり避けるべきである。

is-a関係が成り立つ継承関係は、レガシーシステムにそのまま残してよい。この基準を満たさない継承関係は、可能な限り委譲へ解消していく。

## 4. コードでアーキテクチャを文書化する

DDDのビルディングブロック（エンティティ・値オブジェクト・集約ルートなど）を、クラス図の注釈にとどめず、コードそのものに明示するという考え方がある。これは「生きたドキュメント」の実践の一つであり、ドメイン注釈ライブラリ（クラスに`@Entity`・`@ValueObject`・`@AggregateRoot`のような注釈を付与する仕組み）を使って実現できる。

```typescript
// @ValueObject
class MemberId { /* ... */ }

// @ValueObject
class PointAmount { /* ... */ }

// @AggregateRoot
class Member {
  // @Identity
  private readonly id: MemberId;
  // ...
}
```

こうした注釈には主に3つの利点がある。

1. **可読性の向上**: どのDDDビルディングブロックを意図して実装したのかが、コードを見るだけで直接伝わる
2. **アーキテクチャ検証の自動化**: 「サービスはエンティティと値オブジェクトを使ってよいが、値オブジェクトとエンティティはサービスを使ってはならない」といったDDDの規則を、アーキテクチャ検証ツールで自動チェックできるようになる
3. **定型コードの削減**: 注釈をもとに、永続化フレームワークなどが定型的な処理（マッピング・シリアライズ等）を肩代わりしてくれる場合がある

これら3点はいずれも戦術的変革にとって重要である。特に1点目は、戦略的変革の準備としても効いてくる。値オブジェクトは複製されたり共有カーネルへ移されたりしやすく、エンティティは分割対象になりやすいというように、ビルディングブロックの種類によって扱いが変わるためである。

## 5. AIはドメイン知識を持っているか

IDE統合型のAIコーディングアシスタントは、戦術的リファクタリングに大いに役立つ。これらのモデルはDDDの戦術的パターン、契約による設計、ドメイン注釈ライブラリといった一般的な概念を知っているため、次のような簡単な指示でも良い提案を返してくれることが多い。

- 「このクラスにどんな戦術的リファクタリングを勧めますか」
- 「このクラスを値オブジェクト活用型に書き換えてください」
- 「このフィールドはenumにできますか」
- 「契約による設計を導入してください」
- 「ドメイン注釈ライブラリを使えますか」

ただし、結果は割り引いて受け止める必要がある。特に本章のテーマであるドメイン知識については注意したい。AIは一般的な業界知識（たとえば「ポイントプログラムとはこういうものだ」という一般論）は持っているが、変革対象の組織固有の業務ルールの機微までは知らない。AIの提案はあくまで出発点として扱い、最終判断は業務専門家やチームが行う。

## 6. 自己評価チェックリスト

以下の項目について、1（全くそう思わない）〜5（強くそう思う）の5段階で自己評価してみる。

| # | 項目 |
|---|------|
| 1 | ドメインクラスは技術的な実装の詳細から自由になっている |
| 2 | DDDのビルディングブロックがシステムの広い範囲で一貫して使われている |
| 3 | DDDのビルディングブロックや他の設計パターンによって、各クラスの目的が明確になっている |
| 4 | ドメインクラスのインターフェースは事前条件で保護されている |
| 5 | ドメインクラスの継承構造はis-a関係に基づいている |
| 6 | DDDのビルディングブロックの使い方が、クラス構造の階層化を過度に招いていない |
| 7 | ドメインクラスのインターフェースには振る舞いの豊かなメソッドがある |
| 8 | 値オブジェクトのように振る舞うクラスが存在する |
| 9 | ソースコードはパッケージ／モジュールの水準でドメイン指向の構造を持たず、技術的な分類のみになっている（この項目は評価が高いほど改善余地が大きいことを示す） |
| 10 | 貧血ドメインモデルが存在する（この項目は評価が高いほど改善余地が大きいことを示す） |
| 11 | ドメインクラスのインターフェースにsetter/getterが多く残っている（この項目は評価が高いほど改善余地が大きいことを示す） |
| 12 | ドメインロジックがドメインクラスの外側（サービス層など）に漏れ出している（この項目は評価が高いほど改善余地が大きいことを示す） |
