# 協働モデリング（Collaborative Modeling）

レガシー変革の出発点は「今のドメインで実際に何が起きているか」を業務側と開発側が同じ絵で理解することである。協働モデリング（CoMo）は、業務専門家と開発者が同じ場で図を描きながら合意形成する一群のワークショップ手法を指す。本ファイルではドメインストーリーテリング・EventStorming・シナリオキャスティングの3手法と、使い分けの指針をまとめる。

---

## 1. なぜ協働モデリングが必要か

UMLやBPMNのような技術的記法を使ってドメインを分析すると、その記法に慣れた開発者・ビジネスアナリストと、慣れていない業務専門家との間に理解力の差が生まれる。この差は社会学で「モデル独占（model monopoly）」と呼ばれる現象に近い。理解力が劣る側は疑問を口にしにくくなり、誤った分析にも異論を挟まないまま合意してしまう。結果として、開発が進んだ段階で「実際の業務はこう動いていない」という食い違いが発覚し、手戻りコストが跳ね上がる。

協働モデリングはこの問題を、**5分で説明できる軽量な記法**（付箋・矢印・簡単なアイコン）を採用することで回避する。技術者向けの厳密さより、参加者全員が対等に発言できることを優先する設計である。

共通する原則は次の4点に整理できる。

1. **全員参加の共同作業**: 個別インタビューではなく、関係者を一室に集めて同時に議論する。
2. **具体的なシナリオで語る**: 抽象的な仕様ではなく、実際に起こった（起こりうる）具体的な事例を語り、そこから一般化する。
3. **可視化による理解の共有**: 口頭説明だけでは伝わらない構造を、図として全員の目の前に置く。
4. **探索的な学習**: 最初から全体を知っている人はいない。ワークショップの中で発見しながら理解を組み立てる。

## 2. ドメインストーリーテリング

### 2.1 記法

ドメインストーリーテリングは「誰が・何を・何に対して・いつ・誰と・なぜ行うか」を問う記法で、以下の3要素だけで構成される。

| 要素 | 意味 |
|------|------|
| アクター（人型アイコン） | 業務を行う人・組織単位（場合によりITシステム） |
| 作業対象（物のアイコン） | アクターが操作するモノや情報 |
| アクティビティ（矢印） | アクターが作業対象に対して行う行為。連番が振られる |

図に表せない補足情報は注釈として書き添える。読む際は連番の順に矢印をたどればよい。

**簡易例（配送サービスの注文受付）**:

```
1. 顧客 --[注文する]--> 注文書
2. 受付担当 --[確認する]--> 注文書
3. 受付担当 --[渡す]--> 配送担当
```

**粒度の対比例**: 同じ配送サービスでも、粒度を変えると描かれる内容がまったく異なる。

粗い粒度（業務全体の流れ）:
```
1. 顧客 --[申し込む]--> 配送サービス
2. 配送サービス --[手配する]--> 配送担当
3. 配送担当 --[届ける]--> 荷物
```

細かい粒度（受付処理の内訳のみ）:
```
1. 受付担当 --[検索する]--> 空き時間表
2. 受付担当 --[提示する]--> 候補時間
3. 顧客 --[選ぶ]--> 配送時間
4. 受付担当 --[確定する]--> 予約
```

1枚の図の中で粗い部分と細かい部分が混在すると読み手が混乱する。混在に気づいたら「この部分だけ詳しく話せる人が別にいるのでは」と疑うとよい手がかりになる。

### 2.2 スコープの3因子

同じ手法でも描かれる粒度は大きく変わる。作成前に次の3因子を意識して合わせておくと、混在による混乱を防げる。

| 因子 | 選択肢 | 説明 |
|------|-------|------|
| 粒度 | 粗い ⇔ 細かい | 1枚の図の中で粒度を揃える。混在は「誰かが知識不足」の兆候 |
| ドメイン純度 | 純粋（pure） ⇔ システム混在（digitalized） | 純粋な図はアクターに人間のみを置き、既存システムの都合を排除する |
| 時点 | 現状（as-is） ⇔ 将来像（to-be） | 今の姿を理解する図か、次の一手を描く図かを明確にする |

レガシー変革では、まず**現状の業務を「純粋」な図として描き直す**ことが重要になる。既存システムの操作手順に慣れた業務専門家は、しばしば元の業務の姿を忘れている。あえて「もし今のシステムがなかったら」と問い直すことで、本来の業務語彙が浮かび上がる。この語彙がユビキタス言語の種になる。

### 2.3 境界の発見

ドメインストーリーの図には「グループ」という要素も使える。一連のアクティビティを囲むことで、サブドメインの候補を可視化できる。作業結果を完成させて一方向に渡す境目（例: 「準備担当が完成物を配送担当へ渡す」）は、サブドメイン境界の手がかりになりやすい。境界発見の詳細な判断基準は、戦略ステップ（ドメイン再発見）側の参照ファイルに委ねる。

## 3. EventStorming

### 3.1 3つの形式

EventStormingは付箋を使ってドメインイベントを洗い出す一群のワークショップ形式で、目的に応じて3段階のズームインとして使い分ける。

| 形式 | スコープ | 目的 |
|------|---------|------|
| Big Picture | 業務全体（end-to-end） | 大規模な発見。戦略変革の起点 |
| Process Modeling | 個別の業務プロセス | ソフトウェア抜きでのプロセス精緻化 |
| Software Design | 特定の境界づけられたコンテキスト | ソフトウェア設計に直結する詳細化 |

戦略変革（ドメイン再発見）ではBig Pictureを主に使い、戦術変革（貧血モデル解消などコードレベルの改善）ではSoftware Designを使う。

### 3.2 ワークショップの進行

1. **無秩序な探索**: 参加者が思いつく「ドメインイベント」（過去形で書く、業務上意味のある出来事）を各自オレンジの付箋に書き、壁に貼る。この段階は正誤を問わず、まず量を出す。
2. **時系列の強制**: 付箋を左から右へおおよその時系列に並べ替える。並べ替える過程で欠落イベントの発見や、表現の重複整理が進む。多数の付箋（数百枚規模になることもある）を整理するには次の工夫が有効:
   - **要となるイベント（pivotal event）**: 多くの流れに共通して現れ、フェーズや責任範囲の自然な境目になりやすいイベントに着目する。
   - **レーン分け（swim lane）**: サブストーリーを帯状のレーンに分けて視覚的に整理する。
3. **人とシステムの追加**: 役割（アクター）や外部システムを付箋に加え、ナラティブに厚みを持たせる。
4. **通し読み（explicit walkthrough）**: 進行役が語り手となり、壁全体を最初から最後まで物語として読み上げる。聞き手全員が納得できるまで並べ替えを繰り返す。
5. **価値づけ**: 通し読みで得られた合意済みの流れを土台に、価値の可視化・課題の洗い出し・境界発見・チーム責任範囲の明確化など、目的に応じた発展作業を行う。
6. **境界の描画**: 要となるイベントやレーンの切れ目を手がかりに、サブドメイン（将来の境界づけられたコンテキストの候補）を線で区切る。

### 3.3 カラーコーディング

| 色/形 | 意味 | 使用フェーズ |
|-------|------|------------|
| オレンジ（横長） | ドメインイベント | 基本（全フェーズ） |
| 赤 | ホットスポット（未解決の論点・対立） | 議論が停滞したとき随時 |
| 黄（縦長・人型） | 人・役割 | 人とシステムの追加以降 |
| ピンク（横長） | 外部システム | 人とシステムの追加以降 |
| 淡黄（横長） | 集約（イベントを発生させる業務上の単位） | Software Design |
| 青 | コマンド（人の行為がトリガーとなる操作） | Software Design |

色の割り当ては固定ルールではなく、その場の必要に応じて凡例をその都度更新してよい。カラーコーディングが使えない状況（印刷物・色弱の参加者がいる等）では、色に加えて記号やアイコンを併用する。

Software Designレベルでは、コマンド（青）→イベント（オレンジ）の流れに集約（淡黄）を紐づけることで、オブジェクト指向設計ではクラス図の草案に、イベント駆動設計ではイベントとコマンドがそのままアーキテクチャ要素になる。

### 3.4 使いどころの目安

- 一見構造が見えないドメインを大人数で一気に俯瞰したいとき → Big Picture
- 個々の境界づけられたコンテキストの業務ロジックを深掘りしたいとき → Software Design
- タイムラインで全体を一望したいとき（構造化されていない業務の整理） → EventStormingが有利

## 4. シナリオキャスティング

シナリオキャスティングは、ドメインストーリーテリングやEventStormingに**先立って**「どこから手を付けるべきか」を決めるための手法である。対象領域が広大・未知・変化が速い場合に、闇雲に細部から入るのではなく戦略的に焦点を当てる場所を選ぶ。

### 4.1 コアとなる考え方

具体的な事例（シナリオ）は複雑な問題を理解可能にする強力な手段である。「新規注文は15分後に物流へ送る。これは、顧客が重複注文をすぐに取り消した場合に、出荷処理の取り消しを避けるためだ」というように、我々は日常的に具体例で意思決定を説明している。この力を意図的に使い、複数の具体例シナリオを**シナリオバックログ**に集め、その中から戦略的に意味のある部分集合を選んで組み合わせ、1つの**方向づけシナリオ（orientation scenario）**を作る。方向づけシナリオが、その開発インクリメントの共通の焦点になる。

小さな例示シナリオを1つだけ見ても得られる洞察は少ない。複数の関連する例示シナリオを束ねることで、根底にあるパターンやサブドメイン間の関係、ドメイン全体の構造が見えてくる。

### 4.2 ワークフロー

シナリオキャスティングは次の3ステップのワークショップと、それに続く3ステップの実行フェーズから成る。

1. **ブレインストーミング**: 関係者が懸念・アイデア・疑問を、抽象的な要望ではなく具体的な事例シナリオとしてドメイン語彙で書き出し、バックログに集める。
2. **フォーカシング**: ビジネス目標や新たな知見に基づき、最も影響の大きいシナリオを選ぶ。対象外（out of scope）に振り分けるシナリオも同時に決める。
3. **組み立て（assembling）**: 選ばれたシナリオをタイムライン上に並べ、筋の通った1つ（または少数）の方向づけシナリオへ合成する。

方向づけシナリオが決まった後は:

4. **分析**: EventStormingやドメインストーリーテリングを使い、方向づけシナリオをイベント・サブドメイン・作業対象などに分解する。
5. **設計・実装**: 方向づけシナリオを支えられるようにソフトウェアを拡張・改修する。
6. **検証**: 方向づけシナリオを主要な受け入れテストとして、意図した通りに実行できるかを確認する。

各インクリメント終了後はバックログを見直す。新たに発見されたシナリオを追加し、対応済み・不要になったシナリオを取り除く。この繰り返しにより、バックログは自然に先細り（バーンダウン）していく。方向づけシナリオを小さく絞ることで、複数チームが厳密な調整なしに自律的な意思決定を続けられる。

### 4.3 運用上のコツ

- **具体的すぎることを恐れない**: 「注文処理を改善する」ではなく「常連客が先週と同じ内容を再注文し、1点だけ追加する」のように、詳細まで書く。詳細自体が業務理解の手がかりになる。
- **解決策ではなく状況を書く**: 「モバイル注文に対応する」ではなく「冷蔵庫を見て急に食材が要ると気づいた瞬間」のように、解決策より状況を記述する。状況記述の方が本質的な制約を明らかにする。
- **少なめに絞る**: 方向づけシナリオが豊富すぎると、1インクリメントで実装しきれない。焦点は小さく持つ方がうまくいく。
- **前提を明示する**: 議論の中で除外した条件（例:「冷蔵が必要な商品は対象外」）は前提として書き残し、定期的に妥当性を見直す。
- **未解決の論点は赤い付箋で残す**: EventStormingのホットスポットと同様に、その場で決めきれない疑問は目立たせて後で扱う。

## 5. どの手法をいつ使うか

| 状況 | 推奨手法 |
|------|---------|
| ドメインが広大・未知で、どこから手を付けるか分からない | シナリオキャスティングで焦点を絞る |
| 構造が一見見えない業務を、大人数でタイムラインとして俯瞰したい | EventStorming（Big Picture） |
| 少人数の関係者間の協力関係に着目してプロセスを深掘りしたい | ドメインストーリーテリング |
| 特定の境界づけられたコンテキストのソフトウェア設計に踏み込みたい | EventStorming（Software Design） |

3手法は排他的ではなく、シナリオキャスティングで絞り込んだ範囲をEventStormingやドメインストーリーテリングで分析する、という組み合わせが典型的な流れになる。

## 6. リモート実施とAI活用

- **リモート開催**: 専用のオンラインホワイトボードツールを使えば、ドメインストーリーテリングもEventStormingもオンラインで実施できる。ただし身振りなど非言語的な情報は失われやすい。全員が対面か、全員がオンラインかのいずれかに統一し、対面とオンラインが混在するハイブリッド開催は避けることが望ましい。
- **AI活用**: ワークショップの録音から議事録・要約を自動生成させる用途、ドメイン用語の意味を参加者に補足説明する用途で活用できる。生成された図（デジタルドメインストーリー等)をAIに読ませてサブドメイン候補を提案させる試みも進んでいるが、最終的な境界判断は人間が行う。ワークショップそのものは人と人の学習プロセスであり、AIはあくまで補助に留める。
