# 戦略的ステップ：現状とターゲットの整合・優先順位付けと実行（Step 3・Step 4）

> 要旨: レガシー変革の4ステップ手法における後半2段階を扱う。Step 3ではソースコードの現状アーキテクチャをターゲットコンテキストマップと突き合わせ、必要なリファクタリングと未解決の発見事項を洗い出す。Step 4では洗い出したリファクタリングに優先順位を付け、実際にコードへ手を入れて境界コンテキストを切り出す。

Step 1（ドメイン再発見）・Step 2（ターゲットアーキテクチャのモデリング）は [STRATEGIC-STEPS-DISCOVERY-MODELING.md](STRATEGIC-STEPS-DISCOVERY-MODELING.md) を参照。

---

## Step 3: 現状とターゲットアーキテクチャの整合

ターゲットコンテキストマップができたら、次に明らかにすべきは「では今、自分たちはどこにいるのか」である。現状（as-is）とターゲット（to-be）のギャップを可視化し、そのギャップを埋めるための作業リストを作るのがStep 3の役割である。

### スキーマ

| 要素 | 内容 |
|------|------|
| Actors | アーキテクチャアナリスト／開発チーム |
| Substeps | 3.1 現状アーキテクチャの特定 → 3.2 ターゲットコンテキストマップとソースコードの比較 → 3.3 リファクタリングと発見事項の編纂 |
| Tools | ファシリテーション資材／協働ツール（例: オンラインホワイトボード）／アーキテクチャレビューツール（依存関係の解析・可視化）／C4モデル |
| Results | 現状アーキテクチャの記述／現状とターゲットの差分／戦略的・戦術的リファクタリングの一覧／発見事項リスト |
| Goals | レガシーソフトウェアの現状把握、再構築の規模の見積もり |
| Challenges | 非境界モデル／非境界ドメインモデル／非境界貧血ドメインモデル／非境界データモデル／ユーザーインターフェースの分解 |

> ⚠️ ターゲットコンテキストマップを100%再現する必要はない。発見事項の中にはコストが見合わないものもある。完璧な分割よりも「その後の開発が速くなる」ことを優先する。ターゲット自体も、実際にコードを分解しながら得た知見によって書き換わっていくものと捉える。

### Substep 3.1: 現状アーキテクチャの特定

現状アーキテクチャを最速で把握する方法は、アーキテクチャレビューツールでソースコードを解析することである。多くのツールはモノリスの解析を主眼とするが、複数リポジトリに分かれたマイクロサービス群の相互関係を解析できるものも増えている。

解析結果は開発チームと一緒に読み解く。2000年代前半に主流となったレイヤードアーキテクチャ（UI／アプリケーション・ドメイン／データアクセスの分離）が、今も大半のレガシーシステムの基盤になっている。5〜9人程度の単一チームで完結する規模ならレイヤードアーキテクチャで十分だが、それを超える規模になると技術レイヤーの上にドメイン指向の構造（境界コンテキスト）が必要になる。開発初期から境界コンテキストを意識していなかった場合、この構造化は後からの穴埋め作業になり、相応の労力を要する。

アーキテクチャ文書（arc42やC4モデルによる記述）が残っていれば手がかりになる。文書がなければ、開発チームへのヒアリングで現状の構造を明らかにする。

**AIツールによる現状アーキテクチャの発掘**: システムの当初の設計意図が誰にも分からなくなっている場合、AIツールを使ったソフトウェアアーケオロジー（現存するコードから設計意図を逆算する作業）が効果を発揮する。典型的な問いかけは次のとおり。

- 技術的なレイヤリングは見つかるか。破られている箇所はどこか
- 境界コンテキストは見つかるか。違反はあるか
- インターフェースの種類は識別できるか
- アーキテクチャパターンやデザインパターンは見つかるか。それらはどう連携しているか、規約に従っているか
- このシステムだけに現れる独自パターン（文献やフレームワークに現れない局所パターン）はあるか

> **現場事例**: 自動車リース業務を担うレガシーシステムのレビューでは、開発チームが「レイヤードアーキテクチャです」と即答した。実際にC4モデルのコンテナ図を見ると、フロントエンド・バックエンド・データベースの3層に分かれてはいたが、境界コンテキストは存在しなかった。ドメイン用語はある程度コードに残っており（契約は`Contract`、顧客は汎化された`Partner`として表現）、テーブル名も大文字でドメイン用語を反映していた。しかし中心的なクラス（`Contract`・`Partner`）はどのサービスからも参照される巨大な共有クラスで、フィールドへのアクセスはgetter/setterのみだった。レイヤリングはあるが境界コンテキストがない、典型的な「レイヤードだが非境界」な状態である。

### Substep 3.2: ターゲットコンテキストマップとソースコードの比較

ターゲットコンテキストマップをソースコードに重ねる作業は、幸運にもモジュール／パッケージ単位で境界コンテキストと一致することもあるが、多くの場合は境界線がパッケージやクラスの内部を横切る。既存の構造がドメイン指向にどれだけ近いかによって、この対応付け作業の重さが変わる。

以下は独自の対応付け例（自動車リースの支店システムを想定した簡略化イメージ）である。

| 現状の構造 | 対応するターゲット境界コンテキスト | 判定 |
|-----------|----------------------------------|------|
| `LeaseCalculator` クラス | 見積（Offering） | 明確に1コンテキストへ割当可能 |
| `PartnerRiskService` クラス | リスク評価（Risk Assessment） | 明確に1コンテキストへ割当可能 |
| `ContractService` クラス | 見積／リスク評価の両方から利用 | 非境界モデル（要分割） |
| `Contract` エンティティ | 見積／リスク評価の両方から利用 | 非境界モデル（要分割） |

複数コンテキストにまたがるクラスが見つかった時点で、次節「非境界モデル」の分解作業に進む。

### Challenge: 非境界モデル

**非境界モデル（unbounded model）**とは、1つの境界コンテキストにきれいに収まらず、境界を横断してしまうモデルを指す。非境界性はドメインモデルだけでなく、データモデル・インターフェース・サービス層のそれぞれに現れる。

```
UI層        : 複数コンテキストの概念をまとめて表示する画面（非境界な総覧ビュー）
サービス層   : 複数コンテキストのロジックを抱え込む巨大サービス（非境界サービス）
ドメイン層   : 1つの業務概念に対して1つの巨大な表現しかない（非境界ドメインモデル）
データ層     : ドメインモデルと同型の巨大な共有テーブル（非境界データモデル）
```

これらは互いに依存し合うため、まとめて現れることが多い。ドメインモデルが境界を横断していれば、その上に乗るサービス・データモデル・UIも連動して境界を横断する。

**分解の順序**: 変革では内側（ドメインモデルとサービス）から外側（データモデル・UI）へ向かって分解するのが基本パターンである。小さく分割したドメインモデルの知識を使うことで、データモデルとUIの分解が後から容易になる。

### Challenge: 非境界ドメインモデル

非境界ドメインモデルは、既存クラスへの機能追加を繰り返した結果として自然に生まれる。新機能が必要になったとき、既存の中心クラス（`Product`・`Contract`・`Customer` 等、業務の中心概念を表すクラス）にメソッドやフィールドを追加していくと、「1箇所にまとめておけば管理しやすい」という発想のもとでクラスが際限なく肥大化する。

**問題の本質**: 追加されたメソッドの多くは、追加した機能以外では使われない。再利用を意図した設計判断が、実際には強い結合を生む。複数チームが同じ巨大クラスを同時に変更するため、チーム間の独立性が失われ、開発速度が落ちる。

分解の考え方は、境界コンテキストごとに専用の小さなドメインモデルを持たせることである。同じ業務概念でも境界コンテキストが違えば別のクラスとして存在してよい。例えば「顧客」はコンテキストごとに異なる形を取り、「口座」という概念は「決済口座」「融資口座」「証券口座」のように、コンテキストの関心に応じてより具体的な名前へ研ぎ澄まされる。

### Challenge: 非境界貧血ドメインモデル

貧血ドメインモデル（すべてのフィールドがgetter/setterで公開され、業務ロジックを持たないデータの入れ物）が非境界である場合、状況はさらに悪化する。業務ロジックはドメインモデルの上に乗るサービスクラス、あるいはUI層に押し出されている。この場合、貧血なドメインモデルとそれに付随するサービスクラスを合わせて境界コンテキストへ分解する必要がある。

分解の手順は次の3ステップに整理できる。

1. **クラスの複製**: 新しい境界コンテキスト用のクラスを作成し、旧クラスに一時的な橋渡しフィールドを追加する
2. **メソッドの移動**: 移動対象のメソッドを新クラスへ移し、旧メソッドは新メソッドへ委譲するだけの薄いラッパーにして非推奨（deprecated）とマークする
3. **呼び出し元の切替**: 呼び出し元を1つずつ新クラスへ向け直し、旧メソッドの呼び出しがゼロになったら旧メソッドを削除する

```typescript
// Step1: 新コンテキスト用クラスを作り、旧クラスに橋渡しを仕込む
// offering/leaseQuoteService.ts
export class LeaseQuoteService {
  offer(contract: LeaseContract): void { /* ... */ }
}

// legacy/contractService.ts
class ContractService {
  private readonly bridgeToOffering = new LeaseQuoteService();

  /** @deprecated LeaseQuoteService.offer() を使用 */
  offer(contract: Contract): void {
    this.bridgeToOffering.offer(toLeaseContract(contract));
  }
}
```

呼び出し元の切替が完了するまでの間、システムは常にリリース可能な状態を保つ。この橋渡し構造こそが、大規模リファクタリングを小さな安全な単位に分解する鍵になる。同様の手順はフィールド・getter/setterの移動にも適用できる。1つのコンテキストでしか使われないフィールドは、そのコンテキスト専用のクラスへ完全に移してよい。複数のコンテキストで必要になるフィールドは、Step 4の「複数コンテキストへの複製」で扱う。

**エンティティの同一性**: 複数の境界コンテキストに同じ実体を表すエンティティが分かれて存在する場合、「これは同じものである」という同一性を実装上で明示する必要がある。契約番号や注文IDのような識別子は、分割後も全コンテキストで一致させ、生成元のコンテキストから他コンテキストへドメインイベントで伝播させる。エンティティ同一性の明示化そのものは [DOMAIN-KNOWLEDGE-IN-CODE.md](DOMAIN-KNOWLEDGE-IN-CODE.md) に譲る。

### Challenge: 非境界データモデル

非境界データモデルは通常、非境界ドメインモデルの下に存在する。中心的なテーブルはシステム全体から参照されるため変更が難しく、カラムは追加され続ける一方でほとんど削除されない。

データベースのリファクタリングは、常に3つの側面を同時に扱う必要があるためコードのリファクタリングより難易度が高い。

1. **スキーマの変更**
2. **スキーマにアクセスするコードの変更**
3. **既存データの移行**

以下はベビーステップで進める場合の典型的な手順である（自動車リース業務の契約テーブルを分割する例）。

| 手順 | 内容 |
|------|------|
| 1 | 新しい空テーブル（例: `offering_contracts`）を作成する |
| 2 | 旧テーブルと同じ主キーで1対0..1の関係を張る |
| 3 | 新テーブルに必要なカラムを追加し、両テーブル間に同期トリガーを設置する |
| 4 | データ移行スクリプトで既存データをコピーする |
| 5 | アクセスコードを段階的に新テーブルへ切り替える |
| 6 | 移行完了後、旧テーブルの不要カラム・同期トリガー・テーブル間の関係を削除する |
| 7 | 新テーブルを専用のデータベース（またはスキーマ）へ移し、ドメイン用語に沿った名前へリネームする |

各手順の後、システムは常に本番投入可能な状態を保つ。

### Challenge: ユーザーインターフェースの分解

境界コンテキストへ分解する際、UIの扱いは大きく2方針に分かれる。

| 方針 | 構造 | 利点 | 欠点 |
|------|------|------|------|
| モノリシックUI | 境界コンテキスト群の上に単一のUIコンポーネントを重ねる | 見た目・体験を1つの型で統一しやすい | UI担当チームが全境界コンテキストに依存し、コンテキストの変更ごとに再リリースが必要。機能を1チームで垂直に実装しづらい |
| マイクロフロントエンド | 各境界コンテキストが自分のUIを持つ（ハイパーリンクで並置、または動的読み込みで統合） | チームが全レイヤーを横断して機能を独立実装できる | 境界コンテキスト間で一貫した見た目・体験を保つのが難しい |

UIの構成方針は戦略設計そのものに影響する。UI層で他コンテキストの機能を直接統合できるなら、ドメインイベントによる連携を減らせる場合がある。逆にUIが分かれていれば、コンテキスト間のドメインイベント・実行時依存・機能の重複がどう発生するかを先に洗い出しておく必要がある。

ドメインエキスパートはUIを通してしかシステムを知らないことが多く、境界コンテキストとUIの関係を理解していないと、戦略的設計の意思決定を正しく評価できない。そのため、この方針はStep 2（ターゲットアーキテクチャのモデリング）に着手する前に固めておくことを推奨する。

> 🔴 **確認ポイント**: モノリシックUIとマイクロフロントエンドのどちらを選ぶかは推測で進めず、`../SKILL.md` のAskUserQuestion原則に従ってユーザーに確認する（既存UIの技術的負債・チーム構成・リリース頻度を選択肢の判断材料として提示する）。

### Substep 3.3: リファクタリングと発見事項の編纂

アーキテクチャレビューのワークショップで得られた気づきは、次の2種類に分けて記録する。

| 種別 | 定義 |
|------|------|
| 発見事項（Findings） | 問題は認識できたが、その場では実装方針まで詰められない欠陥 |
| リファクタリング（Refactorings） | 実装方針を段階的に詰められた改善項目 |

リファクタリングはさらに、上位の**戦略的リファクタリング**（例:「見積コンテキストの抽出」）と、それを実現する**戦術的リファクタリング**に分かれる。管理を容易にするため、戦略的リファクタリングを`X.0`、付随する戦術的リファクタリングを`X.1`、`X.2`…と番号付けし、個々の作業が最終的にどの戦略目標につながるかを常に見えるようにする。

発見事項とリファクタリングの境界は流動的であり、発見事項を戦略的リファクタリングとして解決するかどうかは事業判断による。

AIツールは、現状アーキテクチャとターゲットアーキテクチャの記述を読み込ませることで、両者のギャップを埋めるリファクタリング案の提示や、戦略的リファクタリングを1スプリントで実装できる粒度の戦術的リファクタリングへ分解する提案に活用できる。ただしAIが提示した案のレビュー・修正にかかる労力が本当に開発を速めるのかは、継続して検証すべき論点として残る。

以下は独自の記録例（自動車リース業務、見積コンテキストの切り出しを想定）。

**発見事項の例**

| No. | 発見事項 |
|-----|---------|
| A | 旧世代のUI技術を最新のWebフロントエンドへ置き換える必要がある（規模が大きく即断できない） |
| B | 支払遅延管理機能を汎用サブドメインとして外部サービスに置き換えられないか検討する余地がある |

**リファクタリングの例**

| No. | リファクタリング | カタログ上の対応パターン |
|-----|-----------------|------------------------|
| 1.0 | 境界コンテキスト「見積」の抽出（戦略的） | Extract Bounded Context |
| 1.1 | 貧血エンティティ`Contract`をモノリスから分離・分解 | Extract Specialized Anemic Entity |
| 1.2 | テーブル`CONTRACTS`をモノリシックなデータベースから分離 | Extract Specialized Table |
| 1.3 | サービス`ContractService`をモノリスから分離・分解 | Extract Specialized Service |

各パターンの詳細な手順は [REFACTORING-CATALOG.md](REFACTORING-CATALOG.md) を参照。

## Step 4: 変革施策の優先順位付けと実装

Step 3で編纂したリファクタリングと発見事項を、実際に着手する順序に並べ、コードへ反映するのがStep 4である。4ステップ手法の中でも最も工数を要する段階であり、実際にコードへ手を入れる作業がここから始まる。

### スキーマ

| 要素 | 内容 |
|------|------|
| Actors | アーキテクチャアナリスト／ドメインモデラー／開発チーム／プロダクトオーナー |
| Substeps | 4.1 最初の戦略的リファクタリングの選択 → 4.2 戦術的リファクタリングをプロダクト（改善）バックログへ追加 → 4.3 実現可能なリファクタリングをスプリントバックログへ計画 → 4.4 コンテキストの抽出 |
| Tools | リファクタリング支援機能を持つIDE／課題管理ツール |
| Results | 着手対象の決定／変革のための設計指針／優先順位付けされたリファクタリングリスト |
| Goals | 支援・汎用サブドメインから着手しリスクを最小化する、再構築の経験を積む |
| Challenges | サービス内ビジネスロジックの解きほぐし／ローカルデータ・setter・getterのコンテキストへの移動／複数コンテキストへの複製／サービス間へのドメインイベント導入／サービスから貧血ドメインモデルへのロジック移動／発見事項の着手タイミング／根気の維持 |

### Substep 4.1: 最初の戦略的リファクタリングの選択

戦略的リファクタリングの一覧ができたら、どの境界コンテキストから着手するかを決める。経験のないチームほど「一番重要なコア（core）から始めるべきだ」と直感的に判断しがちだが、これは推奨しない。

**パターン: 支援サブドメインから始める** — 最初の切り出し作業は失敗から最も多くを学べる機会である。品質の面でも、1回目の抽出が完成度の高い成果になることは稀で、2回目・3回目でようやく安定した結果を出せるようになる。そのため、業務上の重要度が低い支援・汎用サブドメインから始めることを推奨する。最初の抽出で発生する粗さが、事業の中核（core）を汚さないようにするためである。加えて、他部分との結合が少なく独立性の高い境界コンテキストを選ぶと、チームは短期間で成果を出しやすい。

**パターン: コアドメインへ進む** — 最初のサブドメインを切り出したら状況を再評価する。コアサブドメインに挑む準備ができたか、もう一度支援サブドメインで経験を積むべきか、あるいは少し結合度の高いコンテキストを次に選ぶべきかを判断する。技術的負債の観点では「利率の高い」コードを狙うのが最大の効果を生む。中核部分を段階的に切り出せれば、保守性・拡張性の改善が組織にはっきりと見える形で現れる。

> 🔴 **確認ポイント**: 最初に着手する境界コンテキストの選択（支援サブドメイン優先 / コアドメイン優先）は、`../SKILL.md` のAskUserQuestion原則に従い、チームの変革経験の有無を踏まえてユーザーに確認する。

### Substep 4.2: 戦術的リファクタリングをバックログへ追加

モノリスの分解は大規模リファクタリングであり、完了までに時間がかかる。ほとんどの開発組織はその間も機能開発を止められないため、戦術的リファクタリングはさらに小さな単位へ分割する必要がある。各分割の後にシステムが稼働可能な状態を保つことが前提であり、リファクタリングは構造だけを変え、振る舞いを変えないという原則を守る（振る舞いが変わるなら、それは新機能かバグ修正である）。

規模の大きなリファクタリングほど慎重な分割が必要になる。極端に小さい単位で進める考え方は、俗に「ベビーステップ」と呼ばれる。

粒度をアジャイル計画要素に置き換えると、戦略的リファクタリングはエピック、戦術的リファクタリングはユーザーストーリーに相当する。ユーザーストーリーと同様に、戦術的リファクタリングもタスクレベルまで分解できる。

戦術的リファクタリングはプロダクトバックログに投入され、次のスプリント計画に載せられる。リファクタリングと発見事項を独立した「改善バックログ」として分けて管理するチームもある。

### Substep 4.3: 実現可能なリファクタリングをスプリントバックログへ計画

スプリント計画では、ユーザーストーリーと戦術的リファクタリングのどちらをどれだけスプリントに入れるかを決める。基本方針は2つある。

| 方針 | 内容 |
|------|------|
| 片面スプリント | スプリント全体を機能開発、または変革のどちらかに専念させる |
| 混合スプリント | 毎スプリントで一定の工数を変革用に確保し、残りを機能開発に使う（例: ベロシティ10ポイントのうち3ポイントを変革に割く） |

変革の初期段階では混合スプリントを推奨する。ユーザーへの機能提供を丸1スプリント止めずに済み、変更が目に見えない形（構造のみ）であっても継続的にテスト・承認のサイクルを回せるためである。チームとプロダクトオーナーの間で信頼関係が築けたら、変革専念スプリントを検討してもよい。

**リファクタリングの一時停止は緊急時のみ**: 変革用に確保した工数を他の目的に転用してよいのは、本当の緊急事態に限る。目安として、真の緊急事態は年1回以下・1スプリントを超えて続かないものである。それより頻繁に発生する、あるいは長期化するなら、それは緊急対応で解決すべき問題ではなく、技術的安定化（[TECHNICAL-STABILIZATION.md](TECHNICAL-STABILIZATION.md)）やチーム編成（[TEAM-ORGANIZATION.md](TEAM-ORGANIZATION.md)）から見直すべき構造的な問題である。

**見積りの現実**: 戦術的リファクタリングの見積りはユーザーストーリーと同様、経験を積むほど精度が上がる。ただし完全に正確な見積りは、実装を終えてからでなければ本質的に不可能である。見積りに時間をかけすぎず、実用的な精度で十分とする姿勢が要る。

**カンバンでの運用**: Scrumではなくカンバンを使うチームは、WIP制限の一部を変革用に確保し、専用のスイムレーンでリファクタリングを機能開発と分離して管理するとよい。カンバンにはスプリントという単位がないため、見積り自体を省略できるという利点もある。

### Substep 4.4: コンテキストの抽出

実装に入る際に守るべき基本原則は3つ。

1. リファクタリングの前に、既存の振る舞いを保証する**特性テスト**を用意する（詳細: [TECHNICAL-STABILIZATION.md](TECHNICAL-STABILIZATION.md)）
2. 変更のたびにテストを実行し、どれだけ小さな変更でも都度統合して常に本番投入可能な状態を保つ
3. 品質保証として、ペア・モブ／アンサンブルプログラミングで実装する。それが叶わない場合は最低限コードレビューで四眼原則を確保する

以降のChallengeでは、この原則に基づいて実際に発生する典型的な課題を扱う。

### Challenge: サービス内のビジネスロジックの解きほぐし

貧血ドメインモデルでは業務ロジックがサービスクラスのメソッドに存在する。レガシーの振る舞いを保ったまま、メソッドを境界コンテキストへ正しく振り分けるには、メソッド単位で「これはどのコンテキストに属するか、複数に分解すべきか」を問い直す必要がある。

Step 3で示した「クラスの複製 → メソッドの移動 → 呼び出し元の切替」という手順（[非境界貧血ドメインモデル](#challenge-非境界貧血ドメインモデル)参照）をメソッド単位で繰り返すことで、少しずつサービスをコンテキストへ振り分けていく。

### Challenge: ローカルデータ・setter・getterのコンテキストへの移動

ビジネスロジックを解きほぐすと、クラスの設計意図が明確になり、どのフィールド・setter・getterがどのコンテキストに属するかも見えやすくなる。目標は高い凝集度であり、フィールドはそれを使うsetter/getterと同じ境界コンテキストへ移す。

まずは**1つのコンテキストでしか使われない**データを切り離す。手順は次のとおり。

1. 新コンテキスト用クラスへ対象フィールド・setter/getterを作成する
2. 旧クラスに橋渡しフィールドを追加し、旧クラスのsetter/getterを新クラスへの委譲に変え、非推奨とマークする
3. 呼び出し元を新クラスへ切り替え、呼び出しがなくなった時点で旧フィールド・メソッドを削除する

複数コンテキストで必要になるデータについては、次の「複数コンテキストへの複製」で扱う。

### Challenge: 複数コンテキストへの複製

分解の過程では、同じデータとそのsetter/getterが複数の境界コンテキストで必要になることが多い。この場合はデータを複製し、複製後に値がずれないよう、コンテキスト間でドメインイベントを使った更新の仕組みを導入する。

```typescript
// offering コンテキストが契約成立を通知する
interface ContractFinalized {
  contractNumber: string;
  vehiclePrice: Money;
}

class LeaseQuoteService {
  constructor(private readonly publisher: DomainEventPublisher) {}

  finalizeOffer(quote: LeaseQuote): void {
    // ...見積確定のロジック...
    this.publisher.publish<ContractFinalized>({
      contractNumber: quote.contractNumber,
      vehiclePrice: quote.vehiclePrice,
    });
  }
}

// riskAssessment コンテキストが受け取って自分の Contract を更新する
class ContractFinalizedListener {
  constructor(private readonly contracts: ContractRepository) {}

  handle(event: ContractFinalized): void {
    const contract = this.contracts.findByNumber(event.contractNumber);
    contract.setVehiclePrice(event.vehiclePrice);
    this.contracts.save(contract);
  }
}
```

モジュラーモノリスの内部であればオブザーバーパターンで十分だが、分散したマイクロサービス間ではメッセージバス等を介した非同期のイベント配信を選ぶことになる。

この仕組みが機能したら、複製元のコンテキストから複製先への逆方向の呼び出し（例えば複製先が複製元のsetterを直接呼ぶような経路）は不要になるため削除する。最終的に両コンテキスト間の依存関係は、共有インターフェース（ドメインイベント）だけに絞られる。

### Challenge: サービス間へのドメインイベント導入

レガシーの多くはサービスが互いを様々な形で直接呼び合っている。

> **現場事例**: あるレガシーシステムのレビューでは、すべてのサービス間呼び出しを共通のインターフェース層に集約するルールが敷かれていた。ルール自体は単純で、クライアント・サーバー間のインターフェースとしても妥当な設計だった。しかし同じ設計をサーバー内部のサービス間にも適用した結果、業務ロジック層とフレームワーク層の間の強い結合がインターフェース層の背後に隠れてしまい、可視化されていなかった。共通インターフェースをドメインごとのプロジェクトへ統合し直して初めて、レイヤリングに逆行する依存関係が多数存在する「分散したビッグボールオブマッド」であることが判明した。

サービス間・マイクロサービス間で、結果を期待せずデータやオブジェクトを渡すだけの呼び出しは、ドメインイベントに置き換えられる典型的な箇所である。すでにイベントで疎結合になっている場合も、そのイベントが本当に業務プロセス上の出来事に対応しているかを、Step 1で洗い出した業務プロセスの区切りと照らして検証する。対応する業務上の出来事がないイベントは、境界コンテキストの切り方自体を見直す合図になる。

### Challenge: サービスから貧血ドメインモデルへのロジック移動

貧血ドメインモデルでは、本来エンティティや値オブジェクトが持つべき業務ロジックがサービスクラスに置かれている。サービス内のロジックを次の3つの観点で仕分け、ドメインクラスへ押し下げる。

- **1つの集約だけに関係する処理** → その集約のエンティティへ移す
- **複数の集約をまたぐ処理** → サービスに残す
- **集約の整合性・不変条件を守る処理** → 集約のルートエンティティへ移す

処理全体が丸ごとエンティティへ属するなら、標準的な「メソッドの移動」で移す。エンティティ側とサービス側の責務が1つのメソッドに混在している場合は、先に「メソッドの抽出」で分割してから該当部分だけを移す。どちらに属するかの判断に迷う場合は、対象領域を協働モデリング（[COLLABORATIVE-MODELING.md](COLLABORATIVE-MODELING.md)）で細かくモデル化すると切り分けやすい。

```typescript
// Before: ロジックがすべてサービスに集中している
class LeaseQuoteService {
  offer(contract: LeaseContract, price: Money, lessee: Lessee): void {
    contract.setVehiclePrice(price);
    contract.setLessee(lessee);
    const installment = this.calculator.calculate(price, contract.termMonths);
    contract.setMonthlyInstallment(installment);
    this.notifyCustomer(contract, lessee);
  }
}

// After: 集約の整合性に関わる部分をエンティティへ押し下げる
class LeaseContract {
  fillOut(price: Money, lessee: Lessee): void {
    this.vehiclePrice = price;
    this.lessee = lessee;
  }

  calculateInstallment(rateTable: RateTable): void {
    this.monthlyInstallment = rateTable.monthlyRateFor(this.vehiclePrice, this.termMonths);
  }
}

class LeaseQuoteService {
  offer(contract: LeaseContract, price: Money, lessee: Lessee, rateTable: RateTable): void {
    contract.fillOut(price, lessee);
    contract.calculateInstallment(rateTable);
    this.notifyCustomer(contract, lessee); // 顧客への通知は複数集約に関わるためサービスに残す
  }
}
```

この分割により`setVehiclePrice()`や`setMonthlyInstallment()`のような無防備なsetterが不要になり、フィールドは`fillOut()`・`calculateInstallment()`という業務意味を持つ操作を通してのみ変更されるようになる。契約による設計（事前条件で`fillOut()`が先に呼ばれていることを要求する等）や値オブジェクトの導入といった、さらなる貧血対策は [DOMAIN-KNOWLEDGE-IN-CODE.md](DOMAIN-KNOWLEDGE-IN-CODE.md) に譲る。

### Challenge: データモデルの解きほぐし

ドメイン層の分解が済んだら、データベースにも同じ分解を適用する。手順は[非境界データモデル](#challenge-非境界データモデル)で示したベビーステップと同じであり、スキーマ・コード・データの3面を並行させず、段階的に進める。

### Challenge: 発見事項の着手タイミング

発見事項に対する戦略的・戦術的リファクタリングを事前にすべて洗い出すのは、実質的にStep 3をもう一度繰り返すことになり非効率である。事前の完全な設計は多くの無駄な作業を生むことが分かっているため、発見事項は「経営判断としてシステムの拡張・変更が決まったタイミング」で初めてリファクタリング計画に落とし込む。

対象の境界コンテキストに絞って戦略的・戦術的リファクタリングを組み立て、通常のリファクタリングと同じ手順で改善バックログへ載せる。1つの境界コンテキストで経験を積めば、他の境界コンテキストに同種の発見事項が出てきたときの見積り・対応も速くなる。

### Challenge: 根気の維持

大規模なリファクタリングはスプリントという単位で区切られていても、実態は長距離走に近い。次の姿勢を持ち続けることが変革の成否を分ける。

- タスクの規模の大きさに圧倒されない
- システム全体を一度に直すことはできないと理解し、継続的な改善を日々の習慣にする（[TECHNICAL-STABILIZATION.md](TECHNICAL-STABILIZATION.md)のスカウトルール参照）
- モチベーションを長期間保つのは体力を使う作業だと認め、小さな成功を都度称える
- システム全体が分解し切れていなくても、部分的に解きほぐせたこと自体に価値があると捉える
- リファクタリングが1スプリントに収まらないと分かったら、迷わず分割する。戦略的リファクタリング→戦術的リファクタリング→メソッド/フィールド単位のリファクタリングという順で細分化していく
- 分解にどう取り組むべきかが最初から100%分かっていることはない。うまくいかなければ別のやり方を試し、実験して学ぶ

図やホワイトボード上の設計だけでは変革は完了しない。最終的に問われるのは、本番で動くコードである。
