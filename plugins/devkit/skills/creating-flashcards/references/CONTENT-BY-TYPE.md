> 🔴 ここで示すパース戦略は `scripts/parser_scaffold.py` をコピーした使い捨てスクリプトの `parse()` に**毎回手書き**する（同シリーズ・同一級でも構造変異するためジェネリック化禁止）。投入・HTML整形・冪等性は `scripts/anki_toolkit.py` を再利用する。

## 判定1: 前半後半分離型

**特徴:** ファイルの中盤〜後半に「解答」「答え」等の大きなセクション見出しが1つだけ存在し、前半に問題、後半に解答が並ぶ。

**パース方法:** 解答セクション見出しで2分割 → 問題番号でマッチング。

```python
# 解答編開始位置を検出
answer_section_start = None
for i, line in enumerate(lines):
    if re.match(r"^#{1,2}\s+解答[・･]?\s*解説", line.strip()):
        answer_section_start = i
        break
problem_section = lines[:answer_section_start]
answer_section = lines[answer_section_start:]
```

問題セクション内で問題マーカー（`# 問N`）の位置をリスト化し、解答セクション内の `問N 解答` で対応する解答・解説を抽出。両者を問題番号で join してカード化。

### 「解答・解説マーカー先行型」サブパターン

> ⚠️ 過去問題集の解答編では `**問N 解答・解説**` または `問N 解答・解説` がカード境界マーカーとして使われ、問題文本体はマーカー後の解説テキスト内に「ア × 適切ではありません。〜」のように選択肢ごとの判定とともに散在することがある。問題編（問題文だけが並ぶページ）と解答編（解答・解説が並ぶページ）が別場所に配置される「前半後半分離型」と組み合わさるケースが多く、**過去問では解答編のマーカー数で問題数を数える**のが信頼性が高い。問題編側はOCR汚染を受けやすい一方、解答編側は規則的な「正解：X」表記を含むため抽出の起点として安定する。実装: 解答編の `**問N 解答・解説**` を起点に分割 → 各ブロックから正解・解説を抽出 → 対応する問題文は問題編側の `^問\s*N` で起点同期させる、の二段抽出を推奨する。

> ⚠️ **解答編が自己完結する分離型は解答編だけパースする + 正解は三段フォールバック**: ドリル系の前半後半分離型では、解答編に問題文・選択肢・正解・解説がすべて揃う（自己完結）ことがある。この場合**問題編は捨て、解答編だけをパース**するのが最短かつ確実（問題編との起点同期が不要）。各章解答編の冒頭には正解一覧（`問1：ア　問2：イ　…` を1スパンに収めた表）が置かれることが多い。個別問の正解は (1) インライン正解スパン `[正解：ア]{.class}` → (2) 問題文先頭に**プレーンテキストで漏洩した `正解：X`** の救出（救出後 front から除去）→ (3) **章冒頭の正解一覧 dict** の順で三段フォールバックすると取りこぼしゼロにできる。正解一覧は全件クロスチェック（インライン正解との一致検証）にも使える。実観測: ドリルで20問のインライン正解欠落（page_N マーカー分断・class変異由来）を、漏洩救出＋正解一覧で全補完。

## 判定2: 交互配置型

**特徴:** 問題マーカーと解答マーカーが交互に出現する。

**パース方法:** マーカーの出現位置をリスト化 → 交互にペアリング。

> ⚠️ **番号ラベルが破損していてもマーカー順序は信頼できる → 位置ベース・グルーピング**: 交互配置型（問題→正解→解説→問題…）で、`【問N 正解】` `【問N 解説】` の番号ラベルが壊れている（実観測: `【問1 正解】` が8回出現し、本来の番号 241/262… が欠落）ケースがある。番号をキーに問題↔正解↔解説を辞書照合すると、同番号が上書きされて問題文と解説がズレる（番号網羅検証は dups では気づけても、問題文と解説の**内容の混線**は見逃しやすい）。**対策**: 番号でなく**マーカーの出現順**でグルーピングする。問題マーカーで新レコードを開始し、後続の正解・解説マーカーを（番号ラベルを無視して）直前の問題に属させる。検算: 総マーカー数 = 問題数 + 正解数 + 解説数 が「問題1件につき正解1・解説1」で整合するか確認する。これは質問側/解答側の番号集合がズレる誤植（[CONTENT-COMMON.md](CONTENT-COMMON.md) の番号アンカー正規化）とは別の罠で、**番号ラベル全体が信頼できない**場合の対処。

> ⚠️ **自動生成系コンテンツの極端な書式揺れ → 解答ブロックを真実源 + 多段フォールバック**: 大量問題を自動生成した教材では、同一書籍内で選択肢書式（`(A)` / `A)` / `A.`）・空欄番号書式・正解表記（`正解: (A)` / `正解: A.` / `正解: 単語`＝レター無し）がページごとに不規則に混在し、内部矛盾（正解レターが選択肢に存在しない等）すら混じる。**対策**: (1) 区切り文字を `[).．。]?` で吸収する多書式 regex にする、(2) **解答ブロックを真実源**にして件数を確定する（候補番号は `blanks ∪ answers` の和集合で網羅）、(3) 正解は「先頭レター抽出 → 失敗時は選択肢本文との単語照合（完全一致→部分一致）」の二段で解決する、(4) 選択肢が項目ごとに無く語群を共有する（word bank）ケースは単一選択肢グループ／マーカー無し選択肢列をフォールバックに使う（ただし**6個超の語群は複数項目の混線**とみなしフォールバック無効化＝選択肢なしのリコール型に退避）、(5) 解決不能なものは `needs_fix` タグで識別可能にし全件保持する。完璧を狙わず 95%+ クリーン + 残りを flag する現実解を採る。

## 判定3: 同一セクション型

**特徴:** 各セクション内に問題マーカーと解答マーカーの両方が含まれる。

**パース方法:** セクション内で解答マーカーを境界に分割。

### <a id="テーマ見出し1問型"></a>「テーマ見出し = 1問」型（テーマ別一問一答書籍）

国家資格・IT資格の「一問一答問題集」EPUB では、**各問題が独立した `## テーマ名 {.heading_sH}` 見出しのセクション**になり、その中に次の規則的構造が並ぶサブパターンが頻出する:

```
# 第N章 章名 {.heading_sGH}      ← 章見出し
## テーマ名 {.heading_sH}         ← 問題1件 = 1テーマ（Knowledge Area になる）
### 問題 {.heading_sGZ}           ← 問題見出し（番号なし固定文字列「問題」）
　問題文…
::: class_sHH-0                   ← 選択肢コンテナ div
::: class_sH3
ア
:::
選択肢アの本文
（イ・ウ・エ …）
:::
令和N年度 春期 午前II 問21        ← 出典行（試験回・問番号）
### 正解：エ {.heading_sHU}        ← 正解レター（インライン見出し）
正解選択肢の本文
### 解説 {.heading_sHX}           ← 解説本文
```

**判別シグナル**: (1) 問題見出しが `### 問題`（番号を持たない固定文字列）で全件同一、(2) 選択肢が `::: class_sH3` div で1レターずつ囲まれる（ア/イ/ウ/エ が独立 div）、(3) 正解が `### 正解：X` のインライン見出し、(4) 各問題の前に `## テーマ名` 見出しがある。

**パース戦略**:
1. **問題見出し `^### 問題 \{[^}]*\}` を全件 finditer** し、各問題ブロックを「この問題見出し終端 〜 次の問題見出し開始」で切り出す
2. 章は直近の `^# 第\d+章 .+? \{\.heading_sGH\}`、テーマ（Knowledge Area）は直近の `^## (.+?) \{\.heading_sH\}` を位置ベースで lookup
3. 正解レターは `^### 正解[：:]\s*([アイウエオ])`、解説開始は `^### 解説` で検出
4. 選択肢は `^::+ class_sH3\s*\n\s*([アイウエオ])\s*\n::+\s*\n(本文)` で letter/text を抽出
5. 出典行 `^((?:令和|平成)[0-9０-９元]+年度\s+\S+\s+午前[IⅡ]+\s+問[0-9０-９]+)` を抽出し Back 末尾の出典 div + `出典::` タグに使う

> ⚠️ **「テーマ見出し = 1問」型の解説ブロック終端は「次の問題見出し」ではなく「次のテーマ見出し」**: このサブパターンでは問題ブロックを「次の `### 問題` 見出し」で区切ると、解説の後ろに付く**次の問題の `## テーマ名 {.heading_sH}` 見出しまで解説に取り込まれてしまう**（テーマ見出しは次問題の `### 問題` より前に出現するため）。**対策**: 解説テキスト抽出時に `^#{1,2} .+? \{\.heading_s[A-Za-z0-9_-]+\}\s*$`（テーマ/章見出し）の最初のマッチ位置で truncate する。検出の兆候: 品質検証で Answer 末尾に `## 別のテーマ名 {.heading_sH}` や `# 第N章 … {.heading_sGH}` が混入する。

> ⚠️ **「総合演習章」による章末の完全重複問題（exact-duplicate 再収録）**: 章別一問一答書籍では、最終章が「○○問題だけに挑戦」「総仕上げ」等の名前で、**前章までの問題を問題文そのままに再収録**するケースがある（実観測: 全460問中、最終章の121問が第1〜9章と Front 完全一致）。この再収録分は学習上は同一問題なので、`addNotes` の `duplicateScope: "deck"` + 事前 Front 集合 dedup で**自動的に1枚に畳まれる**のが正しい挙動（投入数 = ユニーク問題数）。**判別**: 「期待件数より投入件数が大幅に少ない」を OCR 抽出漏れと誤認しないこと。最終章名に「挑戦」「総仕上げ」「総合」「演習」「まとめ」等が含まれ、かつ skip 件数 ≒ 最終章の問題数なら再収録と判断してよい。dedup で残る1枚は最初に出現した章（第1〜9章）のタグを保持する。

### <a id="5パーツ型"></a>「予想問題 + プラスの枝 + テーマ解説 + 解答」5パーツ・モジュラー型サブパターン

資格試験対策スピード問題集で頻出する、各問題が次の5パーツで構成される同一セクション型のサブパターン。

| パーツ | マーカー | 内容 |
|--------|---------|------|
| 予想問題 | `### 予想問題` / プレーン `予想問題` | メイン問題（4肢・3肢択一） |
| プラスの枝 | `**プラスの枝**` / `## プラスの枝` | 5〜6サブ問題の○×型番号並び |
| テーマ解説 | `### テーマ解説 [テーマ名]` | 解説本文 |
| 問題N 解答 | `# 問題N 解答` / `### 問題N 解答` / `**問題N 解答**` | メイン問題の正解＋各選択肢解説 |
| プラスの枝 解答 | `# プラスの枝 解答` / `**＋ プラスの枝 解答**` | 各サブ問題の○×＋解説 |

**パース戦略**:
1. 問題マーカー `^(?:#{0,4}\s*)?\**問\s*題\s*\d+\**\s*$` で問題ブロック開始位置を検出（「問題N 解答」は除外）
2. 各ブロック内で5パーツの開始位置を行 index ベースでマップ化
3. メイン問題は予想問題セクション → 問題N 解答セクションの対応で抽出
4. プラスの枝は連番性チェック付きインライン番号分解（[CONTENT-DETECTION.md](CONTENT-DETECTION.md) の「インライン番号並びの『連番性チェック』」参照）で5〜6サブ問題に分解
5. メインカード + サブカード を個別生成

1問題ブロックあたり平均5〜7枚のカードが生成される（メイン1 + プラスの枝5〜6）。

**カード化方針の選択肢**: ユーザーに「サブ問題を個別カード化」「メイン+プラスの枝を1枚に統合」「プラスの枝はカード化しない」の3択を Step 4 のサンプル確認時に提示することを推奨。

> ⚠️ **5パーツ・モジュラー型における末尾セクション境界欠落の罠**: 上記サブパターンでは各問題ブロックの最後が「プラスの枝 解答」セクションになるが、**このセクションには終端マーカーが無い**ため、単純な「次の問題ブロック開始位置を上限とする」戦略を取らないと、最終問題のプラスの枝 解答セクションが**ファイル末尾まで全テキストを取り込んで**カード Answer が異常な長さ（数千〜1万字超）になる罠が発生する。**対策**: 各セクションの終了位置を `secs.get("next_section_start") or next_question_block_start or len(lines)` のように、(1) 同ブロック内の次セクション、(2) 次の問題ブロック開始位置、(3) ファイル末尾、の優先順位で決定する。検出の兆候: 品質検証で Answer 文字数が異常値（中央値の10倍以上）のカードが1〜数件発生する。

### <a id="過去問選択式2スタイル"></a>過去問題集（本試験形式）選択式問題の2スタイル

> ⚠️ **過去問題集（本試験形式）選択式問題の2スタイル**: 同一年度・同一資格でも、**選択式問題のみ書籍内で配置が異なる**。Style A (科目末尾集中型): `### 【選択式問題】` セクション内に全問題が並び、セクション末に **唯一の `【解答・解説】`** ブロックが置かれ、すべての問題のマーカー解答 (`【 A 】(7) 打切補償`等) が列挙される。Style B (問題ごと独立型): 各問題の直後にその問題専用の `[ 解答 ]` `[ 解説 ]` ブロックが配置される (角括弧 + 全角空白)。**対策**: section 内の `(?:【解答[・、]?(?:解説)?】|【正解】|\[\s*解答\s*\])` 出現回数で判別 (`>= 2` で Style B 採用)。Style B では問題番号 `\n\s*([0-9０-９一二三四五六七八九十]+)[\.．]` で section をスライスしてから各 sub-block を parse する。実観測: 平成28年度社労士過去問3冊で、ある書籍は Style A、別書籍は Style B を採用しており、Style B 専用パーサーが必要だった (一般常識/社会保険が Style B、労働関係が Style A)。

> ⚠️ **選択式問題の選択肢が画像形式のみで本文未抽出のケース**: 過去問題集の一部書籍では、選択肢パネル（`(1)アメリカ (2)イギリス (3)ドイツ...` 等）が **画像 (JPEG/PNG) として埋め込まれ Markdown テキスト化できない**。問題文（`【A】【B】【C】` マーカー入り）と解答（`【A】(3) ドイツ`）はテキスト抽出可能だが、Front 表示時に「選択肢が見えない」状態となる。**対策**: 選択肢取得失敗時には Front 末尾に `<div style="background:#fff8e1;color:#996600;font-size:0.85em;">⚠️ 選択肢は EPUB 内の画像に記載（画像未抽出）。各 【X】 の正解語句は解答側で確認可能。</div>` の警告 div を埋め込み、Back 側のマーカー→語句テーブルで学習を担保する。`画像のみ選択肢` タグも併用すると後日 OCR 補完時の絞り込みが容易。

### <a id="過去問実技型"></a>過去問題集（実技）の「大問（事例リード文）+ 複数小問」構造

資格試験対策の過去問題集 EPUB（実技試験）では、1大問単位で次の構造が連続する:

1. **大問見出し** `# [N]{.class_sYA} {.heading_s48}` または `## [N]{.class_s101} {.heading_s4K-0}`（H1/H2両対応）
2. **事例リード文**（発言1〜3を含む3〜4段落の事例設定）
3. **複数小問** `# 問M {.heading_s4K-0}` が大問内に5〜8問連続
4. **【理由群Ⅰ】〜【理由群Ⅲ】** が小問の合間に配置（複数小問に共有される）
5. **解答編** はファイル後半に集約

**パース戦略**:

1. **解答編開始マーカー** `^#{1,2}\s+解答[・･]?\s*解説` で問題セクションと解答セクションを2分割
2. **大問範囲特定**: 問題セクション内で大問見出し位置をリスト化し、各大問の範囲 `[start, next_start)` を確定
3. **小問の所属大問判定**: 各小問の line index がどの大問範囲に含まれるかで親大問を決定
4. **大問リード文取得**: 大問見出し直後〜最初の小問開始までのテキスト
5. **小問ごとのカード生成**: Front = 大問リード文 + 小問本文（理由群があれば含む）、Back = 解答編の `問N 解答 X` から正解＋解説を抽出

**学科 vs 実技の構造差**: 学科試験は大問なしのフラット構造（独立小問のみ）、実技試験は大問あり構造。汎用パーサーは「大問が検出されない場合は全範囲を1大問として扱う」フォールバックを実装することで両構造に対応できる:

```python
if not big_questions:
    big_questions = [{"num": 1, "start": 0, "end": len(problem_section_lines), "small_qs": []}]
    big_questions[0]["small_qs"] = find_smallq_in_range(0, len(problem_section_lines))
```

> ⚠️ **大問範囲内【理由群】の broadcast パターン**: 過去問題集（実技）では【理由群Ⅰ】【理由群Ⅱ】が**大問範囲内の複数小問**（例: 問1の○×判定 → 問2の理由選択、問3の○×判定 → 問4の理由選択）に共通参照される。**パース戦略**: 大問範囲内の `【理由群[ⅠⅡⅢⅣ\d]+】` 開始位置を全件検出し、各小問のFrontに「直前または直後の最も近い【理由群X】」を broadcast 付与する。○×判定問（「適切なら○、不適切なら×」型）には理由群を付与しないことに注意。検出漏れの兆候: 「【理由群Ⅰ】の中から1つだけ選び」と問題文に明記されているのに、Front内に【理由群Ⅰ】本文（ア〜エ）が含まれない。

> ⚠️ **出典行と問題番号の前後順序が同一書籍内で両方向に揺れる**: 過去問題集では各問の出典行（`(47回 実技 問17)` 等）と問題番号マーカー（`**N**`）の**前後順序が同一書籍内で双方向に揺れる**ことがある（`N`→`(出典)` 順と `(出典)`→`N` 順が混在）。一方向だけを仮定して隣接行から出典を拾うと、逆順配置の問で出典が取得できない。**対策**: 出典の関連付けは問題番号マーカーの**直前・直後の両方向を探索**し、最も近い未使用の出典行を採用する双方向リンク方式にする。

> ⚠️ **組合せ大問の小問は出典(src)を持たず範囲親から継承する**: 過去問題集の組合せ問題では、大問リード文側に範囲出典（`(49回 実技 問5～問8)` 等）が1つだけ付き、**配下の各小問（問5・問6…）は個別の出典行を持たない**。小問単位で出典を探すと取得ゼロになり `出典::` タグ・Back の出典 div が欠落する。**対策**: 範囲親出典（`問M～問N`）を検出したら、その範囲に含まれる全小問へ**親出典を継承（broadcast）**する。【理由群】・【語群】の伝播と同じく、伝播範囲は「同一大問内」に限定する。

> ⚠️ **共有リード文/設例の画像は全小問カードへ broadcast、小問マーカーは「行末付帯・単問グループでの欠落」に頑健化**: 実技型の大問+小問構造で設例（リード文）に画像（家族構成図・資料表・年金加入歴図等）が含まれる場合、テキストの broadcast（理由群・出典）と同様に**設例画像を当該大問の全小問カードの `media` へ複製（broadcast）**する（設例は各小問で共有される問題の一部のため）。画像取り込みの手順・接頭辞・プレースホルダ退避は [CONTENT-COMMON.md](CONTENT-COMMON.md) の「EPUB 埋め込み画像」節を参照。さらに小問マーカーには変種があり、(a) `《問5》（改題）` のように**行末に付帯テキスト**が付く（`^《問N》$` の行末限定では漏れる→ `^《問(\d+)》` で行末非限定にする）、(b) **単問グループでは小問マーカー自体が無く**設例直後に問題本文が直接続く（マーカー検出0個なら「設例後〜解答ブロック前の全体を1問」とするフォールバックを置く）。正解は解答ブロック（`《問N》 正解：N` の N件）を**権威**とし、各小問本文を問番号で照合する。実観測: 実技問題集で設例画像のカード封入＋問5の行末付帯・5つの単問グループのマーカー欠落に対応し全24問成立。

## 判定4: 章末解答型

**特徴:** 複数の章に分かれ、各章の末尾に解答セクションがある。

**パース方法:** 章単位で分割 → 章内の問題・解答をマッチング。

### <a id="インライン解答型"></a>章末集中問題集型サブパターン（要点解説テキスト主体）

> ⚠️ テキスト本文の各テーマ節は要点解説テキストのみで構成され、問題マーカーを含まない。問題と解答は章の末尾に一括集約される構造（`# 第N章 → Themeセクション群 → 章末問題集（**問N**） → 答え合わせ（**問N 正解：①**）`）のサブパターン。パース戦略: **テーマ単位**ではなく**章単位**を抽出アンカーとし、章末から `\*\*問\d+\*\*` 起点で問題ブロック、`\*\*問\d+\s+正解\s*[：:]` 起点で解答ブロックを切り出す。Theme節のテキストは問題と紐付けず、章見出し情報としてタグ化のみに利用する。

### 共有語群末尾配置型サブパターン（前方伝播必須）

> ⚠️ 同一大問内の複数択一問題（例: 問35-37 / 問38-40）が共通の【語群X】を共有し、語群ブロック（ア〜カ等の選択肢群）が**大問末尾の最終問の後ろにのみ配置**されているサブパターン。判別キー: 問題文が `【語群X】の中から、空欄 N に入る語句として最も適切なもの〜` 等で語群参照しているにも関わらず、当該問題の直下に選択肢ブロックが存在せず、同一大問の最後の問題の後ろにのみア〜カ等の選択肢リストが出現する。パース戦略: 大問単位で問題を集約後、語群ブロックを**前方の兄弟問題群へ複製伝播**する後処理を入れる。伝播範囲は「同一大問内で語群参照キー（`【語群\w+】`）が一致する問題群」に限定する。これを怠ると前方兄弟問が選択肢ゼロのまま Anki に投入され、回答不能カードが発生する。

### ○✕＋理由群ペア型（後方伝播抑制必須）

> ⚠️ 過去問の「会話文・事実列挙＋複数選択問題」形式で頻出する、奇数問（○✕判定）と直後の偶数問（理由群択一）が **2問1組のペア構造**になっているサブパターン。pandoc 変換後、【理由群Ⅰ〜Ⅸ】等の理由ブロックは**偶数問（理由を問う側）の体内にトレイル**して配置される。注意点: 知見「共有語群末尾配置型」の前方伝播ロジックを無批判に適用すると、理由群ブロックが**前置の○✕問にも伝播**し、本来選択肢を持たない○✕問にア〜エが付与される誤りが発生する。判別キー: 直前問が `次の発言は適切か。○か✕か答えよ` 系の○✕判定文の場合、当該問題には**選択肢を伝播しない**。実装: 伝播ロジックは「同一大問内かつ問題文が `【\w+群】` を参照しているもの」のみを対象にし、○✕問は明示的に除外する。

### インライン埋め込み型解答マーカー

> ⚠️ **インライン埋め込み型解答マーカー（`**問N 正解：①**` 形式）の致命的失敗パターン**: 解答情報が `**問1 正解：①**` のように bold 装飾の中にインラインで埋め込まれているケースでは、既存の `^解答|^正解|^Answer` 等の行頭マッチ正規表現では**検出件数ゼロ**になる。検出に成功する正規表現例: `^\*\*問\s*\d+\s+正解\s*[：:]\s*[①-⑳\d]+\s*\*\*`。解答ブロックの境界は「次の `\*\*問\d+\s+正解` 行」または「次の `\*\*問\d+\*\*` 行（新問題開始）」または「章末」とする。

### 番号正解（丸数字）方式

> ⚠️ **番号正解（丸数字 ①〜⑩）方式の追加対応**: 正解表示がレター方式（A./B./C./D.）・数字方式（1./2./3./4.）だけでなく、丸数字（① U+2460〜⑩ U+2469）を使用するパターンがある。これは「組み合わせを①〜⑥の中から1つだけ選びなさい」型のメタ問題形式と組み合わせて使用される。Front の選択肢は `<ol>` の連番付与が可能。Back の正解表示は `<b>正解:</b> ① [組み合わせ内容]` のように丸数字を保持して表示する。

### ア・イ・ウ・エ小問 + 組み合わせ番号選択型

> ⚠️ 複数小問（ア/イ/ウ/エ）の正誤判定を組み合わせた番号（①〜⑥等）を1つ選ぶ問題形式。Front: 主問題文 + 「次のア〜エの記述のうち〜組み合わせを①〜⑥の中から1つだけ選びなさい」+ 小問ア〜エの平叙文を `<ol>` で表示 + 組合せ選択肢 ①〜⑥ を別 `<ol>` で表示する。Back: 正解番号 + 各小問ア〜エの○×判定 + 解説を記載する。

### 「正しいものはいくつあるか」型本試験形式 — 最終正解レター自動導出

> ⚠️ 国家資格本試験(社労士・宅建等)で頻出する形式で、Front に5記述(ア〜オ)+ A〜E選択肢(`Ａ. 一つ` `Ｂ. 二つ`...`Ｅ. 五つ`)が並ぶが、**ファイル内に最終正解(A〜E)が明示されない**サブパターンがある。各記述ごとに `① 解答: 【正しい】/【誤り】`が並列記載されているため、最終正解はパーサで導出する必要がある。導出ロジック: `correct_count = sum("正しい" in answer.verdict for answer in answers); final_letter = ["A","B","C","D","E"][correct_count - 1]`。Backには「正解: D (正しい記述: 4つ)」のように両方を表示すると学習効率が高い。前述の「ア・イ・ウ・エ小問 + 組み合わせ番号選択型」は番号を1つ選ぶ別形式で、本パターンとは導出ロジックが異なる。

### 法令穴埋め問題集の解答セクション・プレフィックス変異（同シリーズでも書籍別）

> ⚠️ 法令(条文)穴埋め形式の問題集シリーズで、同一出版社・同年版にもかかわらず**書籍ごとに解答行のプレフィックスが変異**する。観測パターン: (a) `・１条：B` `・１条②【★】：C　【■】：A`(中点プレフィックス + 漢数字)、(b) `第２条：B` `第３条①【★】：C　【■】：B`(中点なし + 「第」付き + 漢数字 or 算用数字)、(c) `第２条１号：C` `第２条４号【★】：A　【■】：D`(「号」単位の解答)。同シリーズ内の科目本でも一部の書籍のみ (a)、他の書籍は (b)+(c) 混在、というケースを実観測。**対策**: 解答パース regex は `^[・]?\s*(?:第)?([一二三四五六七八九十百０-９0-9]+)\s*条\s*(?:([①-⑩])|([０-９0-9]+)\s*号)?` のように「中点」「第」「項(①〜⑩)」「号」すべてオプショナル許容にする。書籍別の差を吸収せずに既存 regex を再利用すると、書籍間で「解答取得 0 件」の致命的失敗が発生する。

> ⚠️ **法本体と施行令・施行規則が同一章内に混在する穴埋め書籍**: 健康保険法のような書籍では 法本体・施行令・施行規則 の3レイヤーが同一章内に交互配置され、解答ブロックは `施行令第41条【★】：C` のようにサブ法令プレフィックスで識別される。本文側は単に `第四十一条　...` と書かれているため、article_num だけでは法本体/施行令を区別できない。**対策**: (1) answer key を `(law='', article='41', para='1', marker='★') → 'C'` のように **4タプル**で保持し、`law` には `''`/`施行令`/`施行規則` を入れる。(2) parse_answers でトークン先頭の `施行令|施行規則` プレフィックスを current_law として trackingする。(3) card の sub-law tag は問題ブロック側のヒューリスティック (`注：施行令第N条` 注釈) ではなく **答えがマッチした law キーから passive 決定** すると false positive が激減する。`※施行令第N条` は参照注釈（無視）、`注：施行令第N条` 行頭出現は section marker（採用）の区別が必要。実観測: 健康保険法穴埋めパース初期、`※施行令第34条` の inline 注釈を section marker と誤判定して 法第34条 のカードを sublaw='施行令' でタグ付け、結果 32 件で letter 欠落となった。match-based 判定に切り替え後 91.6%→94.8% に改善。

### 法令穴埋め問題集の解答セクション・高密度フォーマット

> ⚠️ 上記の解答セクション・プレフィックス変異のサブ問題として、年金法・社会保険一般常識等の解答セクションは更に複雑な高密度フォーマットを取る。観測パターン: (a) **1行に複数条連続**: `第３条【★】：A　【■】：C　第４条【★】：B　【■】：B`、(b) **「第N条のM」枝番**: `第４条の２：A　第４条の３①：C`、(c) **省略形による直前条継承**: `第４条の３①：C　②：D`(「②：D」は直前 `第4条の3` の第2項を意味)。これらを単純な行単位パーサ(line.startswith("・") 必須・1行1解答前提)で処理すると **30〜45% の解答取得失敗** が発生する(実観測: 65問中28件、118問中39件等の高欠落率)。**対策**: (1) `re.finditer` で行内マルチマッチを取る、(2) 条/項/号に加えて `条のM` 枝番をキーに含める `(jou, branch, key_tag)` の3要素タプル設計、(3) 「条」マッチがない match は直前の `(jou, branch)` を継承する文脈依存パーサ。さらに同条内で `(jou, ('kou', N))` で見つからない場合は、その条のすべての解答エントリを集約して blank marker(★/■) でマッチングする jou_index フォールバックを実装する(下記知見も参照)。

> ⚠️ **複数項で選択肢パネルを共有する条文**: 法令穴埋め問題集では `第N条第2項` の本文末に【★】、`第N条第3項` の本文末にも同じ【★】、そして第3項の直後に **両項共通の選択肢パネル** `A：理事長 B：理事 C：監事 D：厚生労働大臣` という配置が頻出する。項単位の panel detection だけでは第2項の選択肢が空となる。**対策**: 項単位検出が失敗した場合、**条文全体 (article_body) から選択肢クラスタを抽出** して各マーカーに割当てるフォールバックを実装する。フォールバック検出 regex は `\n\s*\n([A-E][：:].*?)(?=\n\s*\n\s*(?:[^A-E\s]|\Z)|\Z)` のように blank line を経て [A-E]: から始まる cluster を、blank line + 非[A-E] (条文本文) で終端する。実観測: 健康保険法第16条第2項・第3項が共通パネル `A：理事長 B：理事 C：監事 D：厚生労働大臣` を共有しており、article-wide fallback 導入で 5 カード回復。さらに `A：1か月 B：2か月\n\nC：3か月 D：4か月` のように **選択肢クラスタが空行で分断**されている場合、`\n\s*\n` で cluster を打ち切ると C/D が欠落する。終端 lookahead を `(?=\n\s*\n\s*(?:[^A-E\s]|\Z)|\Z)` のように「blank line + 非[A-E]」に変更すると複数空行を跨いだ cluster を統合できる。

> ⚠️ **答え記述の `①・②` 多項共通指定と `③：B` 単独項単発穴**: 法令穴埋めの解答ブロックには (a) `第7条の18①・②【★】：B　【■】：B` のように複数項が同一マーカーで同じ答えを共有する指定、(b) `③：B　④：A　⑥：C` のように **項マーカー単独 + コロン** で項単位の単発穴解答を示す指定がある。両方とも単純な `第N条NUM【★】：X` パターンとマッチしない。**対策**: (a) parse_answers でトークンに丸数字が連続出現する場合 `multi_paras` リストに集約し、続く 【マーカー】：X を各項に複製投入する。実装スケッチ: `while rest and rest[0] in PARA_MAP: multi_paras.append(rest[0]); rest = rest[1:]; if rest.startswith('・'): rest = rest[1:] else: break`。(b) `if rest startswith ':'` の時に `(law, article, current_paragraph, None) → letter` を保存する。さらに card 側の lookup フォールバックとして `(law, article, ANY_para, marker)` で同条文内に唯一のマッチがあれば採用する Level 4 検索を実装する。実観測: 健保第7条の18パースで両パターン未対応で letter 欠落、対応後 32→17 件に削減。

### 法令穴埋め問題集 — 「条全体で1セット」発想による解答エントリ統合フォールバック

> ⚠️ 上記知見の追加対策として、解答セクションは「条」単位で集約されており、項境界を解答側が無視する設計が多い(例: 第19条第1項に【★】、第19条第2項に【■】が分散配置されても、解答は `・19条【★】：B 【■】：B` と統合)。**対策**: パーサで `(jou, kou)` 完全一致が見つからない場合、同 jou のすべての解答エントリを集めて、現在のセグメントに含まれる blank marker(`【★】` or `【■】`)とマッチングするキーを抜き出すフォールバックを実装する。実装スケッチ:
>
> ```python
> if not ans_entry:
>     seg_markers = {b["marker"] for b in parsed_seg["blanks"]}  # {'star', 'square'}
>     merged = {}
>     for e in jou_index.get(jou, []):
>         for k_marker, letter in e.items():
>             if k_marker in seg_markers and k_marker not in merged:
>                 merged[k_marker] = letter
>     ans_entry = merged
> ```
>
> このフォールバックで解答取得率が約60% → 約90% に改善した実観測例あり(94件中25件→8件 missing)。

### 難易度マーカー混在形式

> ⚠️ **難易度マーカー ★☆ 混在形式への対応**: 難易度マーカーが `★★☆`（中）/ `★☆☆`（易）/ `★★★`（難）のように ★ と ☆ が混在するパターンがある。既存の `★+` パターンでは ☆ も含むマーカー文字列に対して「★の数だけ難易度」を正しく算出できない。抽出ロジック: `re.findall(r"★", marker_line)` で★の個数を集計するか、`★(?=[★☆\s」$])` 等の lookahead で連続★数をカウントする。値域: ★☆☆→易（`難易度::★`）/ ★★☆→中（`難易度::★★`）/ ★★★→難（`難易度::★★★`）。マーカー不明・未記載は「難易度不明」タグを付与する。

## 判定6: テーマ単位ページ分離型

**判定条件:**

- ファイルが `---` の水平線で頻繁に分割されている（書籍のページ区切り由来）
- `**問題N**` 形式のbold装飾問題番号が連続して並ぶ「問題ページ」と、`**解答N**` 形式が連続する「解答ページ」が**ペアで繰り返される**
- 問題ブロックと解答ブロックの間に章・節・テーマ見出しが挟まることがある
- ○×判定マーカー（`○` / `✕` / `×`）が解答ブロックの先頭に存在する

**パース方法:**

1. **テーマセクション抽出**: `**テーマ\d+`、`## 第\d+節`、`# 第\d+章` の見出しでファイルを階層分割
2. **問題ブロック検出**: 各テーマ内で `^\*\*問題\s*\d+` で始まる行を起点として、次の `^\*\*問題\s*\d+` または `---` までを1ブロックとして切り出す
3. **解答ブロック検出**: 同様に `^\*\*解答\s*\d+` で始まる行をブロック単位で切り出す
4. **番号マッチング**: 問題N と 解答N を**問題番号で対応付け**（出現順ではなくテーマ内での番号一致）
5. **メタ情報抽出**: 各問題ブロックから難易度（`★+`）、重要マーカー（`\*\*重要[!！]\*\*`）を抽出してタグ化
6. **○×正規化**: 解答先頭の判定マーカーを `○` または `✕` に統一（`×`→`✕`、`〇`→`○`）

> ⚠️ **目次・要点整理セクションの除外**: ファイル冒頭の「# 目次」、各章末の「要点整理」、書籍紹介ページ（はしがき、傾向と対策等）は問題ブロックを含まない。テーマ見出しが存在しないブロックはスキップする。

> ⚠️ **テーマ見出しの欠落補完**: 一部のテーマブロックではpandoc変換でテーマ見出しが欠落することがある。直前のテーマを継承するか、解答ブロックに含まれるテーマ見出しから逆引きする。

> ⚠️ **コンテキスト参照渡しの罠**: 章/節/テーマのコンテキストを `Context` インスタンスやdictで保持しながらパースする場合、問題関数と解答関数で**同一インスタンスを共有**すること。コピー（`dict.copy()` / `dataclasses.replace()` 等）を解答関数に渡すと、解答ページ内の見出しから取得した章情報が次の問題ランに伝播せず、解答ページが章境界をまたぐケースで章情報が古いまま固定される。

> ⚠️ **冒頭プレビュー問題の重複**: 資格試験書籍の冒頭には「サンプル問題」「お試し問題」として本編章節の最初の数問を再掲するパターンがある。問題文の**完全一致で重複検出し、後出現を残してデダプ**すること（先出現を残すとサンプルセクションには章コンテキストが付与されておらず章タグが欠落する）。後出現の本編側を採用することで、章・節・テーマ情報が正しく付与されたカードが生成される。

> ⚠️ **Theme番号の表記揺れ（`# Theme N` / 単独数字見出し / 改行分離型）**: 同一書籍内でもテーマ見出しの形式が揺れるケースがある。(a) `# Theme 1 タイトル` 形式、(b) `# 1`（単独数字見出し）形式、(c) 前行が `Theme` のみでその次行に番号タイトルが来る改行分離型。パース時に「Theme番号＋タイトル」を正規化して連結する。改行分離パターンの検出: 前行が `^Theme\s*$` または `^テーマ\s*$` の場合、次の見出し行と論理的に連結する。

### 箇条書き形式の解答ブロック

> ⚠️ 一部のページでは解答が箇条書きで列挙される（`*   **解答1 ○**\n    元本が確定すると…`）。`clean_line` で先頭の箇条書きマーカー（`*  ` / `-  ` / `+  `）を除去するとともに、続く**インデント行を継続行として連結**すること。これを処理しないと、解答1の本文がパーサーから消える。

### 省略された判定マーカーの推論

> ⚠️ 同じ書籍シリーズでも、章・節・テーマ毎にレイアウト方針が異なり、一部の解答ブロックでは ○/× の判定マーカーが**完全に省略され、解説文のみ**が表示されるケースがある（読者は解説文から判定を推論することが暗黙的に要求される）。この場合の処方は2段階:
> 1. **明示マーカー検出を尽くす**: 解答ヘッダの直前・直後・同一行・次行のすべてを ○/× 候補としてスキャンする
> 2. **残った欠落ケースの意味推論**: 解答テキストの末尾に「誤り」「誤って」「設問前段が誤り」等の否定キーワードがあれば ✕。質問文の主張を再述・補強する解説（「〜できる」「〜認められる」「本問のとおり」）であれば ○。両義的な場合は Claude が個別に質問と解答を読み比べて判定する
> 3. 判定不能のものは `判定マーカー欠落` タグを付与して保留する（Step 8 完了報告で件数を提示）

### 複数ページにまたがる解答ブロックの分断

> ⚠️ **テーマ単位ページ分離型サブパターン — 複数ページにまたがる解答ブロックの分断**: 同じ「テーマ単位ページ分離型」でも、1つの解答ブロックが複数ページにまたがるバリアントが存在する（解答ヘッダ直後の解説段落が次ページに送られる、または1ページあたり数件の解答のみで占有されるなど）。この場合、`---` 区切りをページ境界としてページ単位で即時に問題/解答をペア化する素朴な実装では、後続ページに続く解答テキストが捨てられ、解答フィールドが欠損・切り詰めになる。
>
> **対策**: `pages` バッファにページを**累積**しながら、(a) 次のテーマ見出し（`**テーマ\d+`・`## 第\d+節` 等）を検出した時、または (b) 次の問題ページ群が始まると判定できた時にまとめて `flush` して Q&A ペアを構築する。パース単位のアンカーは「解答ヘッダが出現した開始ページ」ではなく**「テーマセクション全体」**とすること。

> ⚠️ **解答ページの「解答解説」見出し欠落と `正解` マーカーによるページ分類**: テーマ単位ページ分離型で1テーマの解答が複数ページに渡るとき、2ページ目以降の**継続解答ページには「解答解説」見出しが付かない**ことがある（見出しは解答群の先頭ページにのみ出現）。`---` 区切りページを「解答解説」文字列の有無で問題/解答に二分類すると、継続ページが問題ページへ誤分類され解答テキストが欠落する。**対策**: ページのブロック分類は「解答解説」見出し文字列ではなく **`正解` マーカーの有無**で判定し、`正解` を含むページ＝解答ページ群として直前の見出し付き解答ページへ累積（flush）連結する。実観測: 問21-22 で継続解答ページが見出しを欠き、`正解` 判定へ切替えて復旧した。

## <a id="OCR汚染対策"></a>OCR汚染対策（画像主体EPUB特有）

> 🔵 **二層防衛の前提**: ソースが `recognize-image-to-markdown` 経由のOCR出力なら、思考ログ除去・反復崩壊検出・自動再OCRは**コマンド側で一次対処済み**（ログタグ `[ocr-cleanup]` で実施。`--no-ocr-cleanup` で無効化、`--max-repeat` / `--contamination-threshold` で調整可能）。本セクションは **①コマンドをすり抜けた残存ケース ②コマンド未経由の他ツールOCR出力** のフォールバック（最終防衛線）として読むこと。
>
> 消費側（本スキル）の機械的前処理は、`scripts/parser_scaffold.py` の素材ヘルパを `parse()` から呼んで行う:
>
> | ヘルパ | 役割 | 閾値（消費側） |
> |--------|------|--------------|
> | `collapse_repeated_lines(text, max_repeat=3)` | 同一行の連続反復（無限ループ崩壊）を1回に畳む | **3**（コマンド側=4 と意図的に非対称。最終防衛線なので厳しめ） |
> | `strip_thinking_logs(text)` | OCRメタ思考ログ専用行のみ除去（○×単独行は保護） | — |
>
> これらは「素材」であり一括統合関数にはしない。ソースを見て呼ぶ/呼ばないを判断する。以下に列挙する検出パターンの大半はコマンドが一次対処済みのため、消費側では上記ヘルパ呼び出しで足りる。残るのは**ヘルパで取りきれない高度サルベージ**（構造化記述復元・regex本文救出・段階的サルベージ・選択的再OCR）であり、これらが第2層の存在意義である。

画像主体EPUBを生成AIでOCR化したMarkdownでは、OCR LLMの思考プロセスが本文に混入することがある。判定6（テーマ単位ページ分離型）の画像主体書籍で特に頻発するが、全判定で発生する可能性があるため独立セクションとして扱う。

### OCR変換ログの混入（残存フォールバック）

> ⚠️ OCR LLMの「思考プロセス」混入（英語メタ文 `1. **Analyze the Request:**` 系、システムプロンプト復唱 `The user wants the text` / `ユーザーは…画像…` 系、同一文字判定の無限ループ等）は、**コマンドが一次対処済み**。コマンド未経由ソース／すり抜けた残存行は、消費側で `parser_scaffold.py` の `strip_thinking_logs()`（メタ専用行のみ除去）と `collapse_repeated_lines(text, 3)`（反復崩壊圧縮）を呼んで対処する。
>
> **残存対処時の注意（ヘルパが内蔵済みの原則）**:
> - **単独の `O` / `X` / `○` / `×` 行は除外しない**（判定マーカーである可能性が高い）。`strip_thinking_logs()` は `_OXMARK_ONLY_RE` を先にチェックして○×単独行を保護する。手書きフィルタを足す場合も同順序を守る
> - 汚染ページは**ページ全体を破棄しない**（本来のQ&Aが同ページ内に部分的に再構成されているため）。OCR-thinking内にネストされたQ&Aコンテンツ（例: `* Label: 解答 1 / * Symbol: ○ / * Text: 使用者責任の成立要件として...`）は手動抽出で復元する
> - パーサーは汚染ページを許容し、対象問題の解答が空欄になったケースを後段で個別補正する戦略を取ること

### OCRメタ思考トリガー行とリスト継続行のブロック除去

> ⚠️ `is_likely_english_meta` / OCRメタログ行フィルタを適用する際、メタ思考トリガー行（例: `* Analyze the Request:` / `* Scan the Image:` / `* 見出し：`）に続く **インデント付きリスト継続行**（`  * サブ項目` / `    - 補足`）は同一メタブロックの一部であるため、トリガー行と一緒にブロック単位で除去すること。行単位でトリガー行だけを除去すると、インデント継続行が本文パーサーに取り込まれ、意図しない解説テキストや小問候補として誤抽出される。**実装**: トリガー行検出後、インデントが続く限り後続行を消費する（`while next_line.startswith("  ")` 等）。

### OCR debug 構造化記述からの完全復元（高度サルベージ・第2層の主機能）

> 🔵 これ以降の4手順（構造化記述復元・regex本文救出・段階的サルベージ・選択的再OCR）は、コマンドの一次対処や `strip_thinking_logs()` では取りきれない**残存崩壊ページからの高度サルベージ**であり、第2層（消費側）の主たる存在意義。コマンド経由・未経由を問わず、ヘルパで除去しきれなかったブロックに適用する。

> ⚠️ OCR LLM の思考ログには大きく分けて 2 種類のパターンが観測される: (A) 同一文字を延々と反復するだけの**完全破壊型**（リカバリ困難）、(B) 画像の構造を構造化された Markdown リスト（`1. **ヘッダー部分**: ...` / `2. **問題文**: "..."` / `3. **選択肢**: ① "..." ② "..." ③ "..." ④ "..."` / `4. **解答**: ...` 等）で**整然と記述したまま無限ループに突入する記述型**。後者は思考ループのノイズ行を除けば**問題文・選択肢・解答が原文ほぼそのまま埋め込まれている**ため、ページ全体破棄ではなく**手動復元による完全カード化が可能**。
>
> **検出パターン**: ブロック先頭が以下のような構造化マーカーで始まり、後続行に引用符（`"..."` / `「...」`）で囲まれた本文・選択肢が並ぶ:
>
> - `ユーザーは提供された画像のテキストを Markdown 形式で出力することを求めています。`
> - `The user wants the text from the image transcribed into markdown format.`
> - `**画像の(分析|構造)**:` / `**Identify the main title:**`
> - 続いて `①: "..."` `②: "..."` `③: "..."` `④: "..."` または `a. "..."` `b. "..."` `c. "..."` `d. "..."` の選択肢列挙
>
> **復元手順**:
>
> 1. パーサーで取れなかった問題番号を warning として列挙し、ソース md の該当ブロックを目視確認
> 2. 構造化記述ブロックが見つかった場合、その記述に含まれる引用符内テキストを問題文・選択肢として手動抽出
> 3. 直後の `---` 区切り後にある解答ブロック（こちらは比較的クリーン）から正解番号と解説を取得
> 4. 別 Python スクリプト（例: `recover_missing.py`）で `addNotes` API 経由で個別カード追加
> 5. 復元カードには `OCR汚染復元` タグを付与し、Step 8 完了報告で件数を提示

### OCRメタ思考からの小問本文救出 — regex 完全仕様

> ⚠️ OCR系VLMが出力するメタ思考ログには、問題本文が構造化リスト形式で埋め込まれているケースがある。以下4形式から小問・難易度を救出可能:
>
> | パターン | 対象 | regex例 |
> |---------|------|--------|
> | 小問本文 | `* ア：[本文]` | `^\s*\*\s+([アイウエオ])[:：]\s*(.+?)$` |
> | 問題タイトル | `* 見出し：第N問 [タイトル]` | `^\s*\*\s+見出し[:：]\s*第\d+問\s+(.+?)(?:\s+難易度.*)?$` |
> | 問題指示文 | `* 指示：[指示文]` | `^\s*\*\s+指示[:：]\s*(.+?)$` |
> | 難易度マーカー | `難易度 ★★` | `難易度\s*([★☆]+)` |
> | 解答（装飾プレフィックス付き） | `*Line N:* ○` / `Item N: 使用者責任…` / `` `○` `` | `^\s*\*?Line\s*\d+[:：]\*?\s*(.+)$` / `^\s*Item\s*\d+[:：]\s*(.+)$`（前後のバッククォートは別途 strip） |
>
> 救出成功条件: メタ思考ブロック内に上記パターンが出現し、直後の `---` 区切り後の解答ブロックがクリーンであること。救出カードには `OCR汚染復元` タグを付与する。

> ⚠️ **解答が思考ログ装飾行内にしか存在しないケース（`strip_thinking_logs` の前提を覆す）**: 上記救出と前掲「OCR変換ログの混入」の「ネストされたQ&A救出」は、いずれも**「思考ログ＝冗長な汚染（クリーンな本文が同ページ内に別途再構成済み）」を暗黙の前提**にしている。だがページによっては**本物の解答が `*Line N:*` / `Item N:` / バッククォート装飾を付けた思考ログ行の中にしか存在しない**ことがある（実観測: 約20問の解答がこの形でのみ残存）。この前提崩れの下では `strip_thinking_logs()` が解答の唯一のコピーごと該当行を削除し、解答が**永久に失われる**（単純な思考ログ除去で約20問が消失する事故が起きる）。**対策**: 該当行を行ごと除去せず、**装飾プレフィックス（`*Line N:*` / `Item N:` / 前後のバッククォート）だけを剥がして本文を温存**する。実装順序は「装飾プレフィックス付き行を先に救出 → その後 `strip_thinking_logs()` を適用」とし、唯一コピーの破壊を防ぐ。判定が曖昧な行は strip 対象から除外し欠落報告に回す。

### 段階的サルベージ戦略

> ⚠️ **OCR汚染ブロックからの段階的サルベージ**: ハルシネーション・思考ループでページ全体が汚染されていても、汚染ブロックの内部に**断片的に正しい構造**が残っていることが多い。汚染を理由にページ全体を破棄する前に以下の優先順位でサルベージを試みること:
>
> 1. **解答ヘッダ式救出**: `第\s*\d+\s*問\s*解答\s*[①-⑳]` パターンで番号と正解だけ救出する（解説は失われるが正解は得られる）
> 2. **○×サマリー行救出**: `^\d{3}\s+[○✕×〇◯]\s+\S+` のような「3桁問題番号 + 判定 + 短文」パターンが解答リストとして残存している場合がある
> 3. **答え合わせ集計表フォールバック**: 章末・解答セクション冒頭に `\| 第\s*\d+\s*問 \| [①-⑳\d] \|` 形式のサマリー表が出力されていることがある。これを最終フォールバックとして利用し、最低限「正解番号のみ」のカード（解説欠落）を生成する
>
> 救出したカードには `OCR汚染復元` `解説欠落` タグを付与し、Step 8 完了報告で件数を提示する。

### 画像主体 EPUB の選択的ページ再 OCR フォールバック戦略

> ⚠️ pandoc変換では本文テキストが少ない「画像主体ページ」が空白または数行のみになる。このとき、パーサーが解答件数と問題件数の差異を検出した場合に、不足分のページを特定して再OCR（VLMによるMarkdown化）するフォールバックを実装すること。**実装手順**: (1) パース後に問題番号の連番ギャップを検出（例: 第3問が欠落、第4問の直前ページが空白）、(2) 該当ページ番号をEPUBから画像として抽出（`epub.read(item)` で画像バイナリ取得）、(3) VLM OCR API に再送してMarkdown化、(4) 再OCR結果をパーサーに再投入。再OCR対象は最大10ページ程度に限定し、10ページを超える場合はユーザーに手動確認を求める。

### OCRメタ思考ログの系統別パターン（コマンド一次対処済み・残存はヘルパ呼出）

> ⚠️ OCRメタ思考ログには複数系統がある（英語メタ「Analyze the Request」系 / 日本語自問自答「画像の分析を行います」「いや、待て」「よし、「[文字]」と判断する」系 / 三人称セルフ説明「ユーザーは…画像…」「これは…の問題だと」「実際には右から…」系 / 英語仮説検証「Hypothesis:」「Let's look at the character」「Testing the hypothesis」「Confirming the reading」系 / Mental Draft `*Mental Draft:*` 系 / 視覚特徴説明文「白抜き文字」「縦書きで」「太字で表示」の反復）。**これらの行頭定型句マッチによる除去はコマンドが一次対処済み**であり、消費側ですり抜けた残存行は `parser_scaffold.py` の `strip_thinking_logs()`（系統別アンカーを内蔵。○×単独行は保護）で除去する。連続反復による崩壊は `collapse_repeated_lines(text, 3)`（消費側閾値3）で1回に畳む。
>
> **ヘルパで取りきれない残存に対する運用知見（温存）**:
> - **反復除去のスキャン範囲**: 視覚特徴説明文・三人称セルフ説明は同一範囲で**数十行（10〜50行規模）にわたって同一・類似フレーズを反復**する実観測ケースがある。手書きでブロック単位除去する場合はスキャン範囲を最大50行先まで広げること
> - **語彙密度フィルタ**: 視覚特徴説明文（`白抜き` 等）は単独行では本物のQ&A解説と区別できない。**同一語彙が3行以上連続出現する場合のみ汚染ブロックとして除去**する（`strip_thinking_logs()` の行頭アンカーで取りきれない場合の手書き補完）
> - **復元不能ページの扱い**: 英語仮説検証ループや簡体字ループで実コンテンツが破壊されたページは OCR 復元不可。該当問題番号を欠落報告に含め、ユーザーが原典から手動補完する経路を残す（系統別カテゴリ例: `OCR_JP_THINKING` / `OCR_HYPOTHESIS_LOOP` / `OCR_MENTAL_DRAFT` / `OCR_VISUAL_DESCRIPTION`）
> - **簡体字を崩壊の補助シグナルに**: OCR LLM が崩壊・幻覚ループに入ると日本語原文にない**簡体字・中国語トークン**（`一组`・`组` 等）を反復出力することがある。反復検出（[CONTENT-DETECTION.md](CONTENT-DETECTION.md) のOCR無限ループ崩壊警告）に加え、簡体字の出現を崩壊の補助シグナルとして扱い該当範囲を破棄候補＋欠落報告対象にする。🔴 ただし**簡体字の機械削除は `strip_thinking_logs()` には入れない**（本文巻き込み＝誤爆回避）。あくまで人手判断のシグナルに留める。書籍末尾の定型フッター（`最新の情報は下記ウェブサイト…` 等）が自己ループするケースは1行に凝縮して無害化できるが、本文ページで簡体字ループが起きた問は復元不能なので欠落報告に含めること
> - 系統別で取りきれずブロックに埋もれた本物のQ&Aは Step 8 完了報告で件数を提示し、手動補完を促す

### 章フッターのページ内位置不定混入

> ⚠️ 紙面の章フッター（章番号 + 章タイトルの2行構造）は、通常はページ区切り `---` の直前に出現する想定だが、OCR時のレイアウト崩れにより**問題文の中盤・選択肢の合間・解説段落の途中**にも混入する。位置に依存した単純な「`---` の直前のみ削除」戦略では検出漏れが発生し、抽出された問題文に章フッター断片が残留する。
>
> **対策**: 章フッターの語彙を**明示リスト化**してファイル全体をスイープする:
>
> ```python
> CHAPTER_FOOTERS = [
>     # コンテンツ構造分析（Step 3）で目次見出しから動的に抽出した章タイトル群
>     "章タイトル1",
>     "章タイトル2",
> ]
> for footer_title in CHAPTER_FOOTERS:
>     text = re.sub(rf'^\d{{1,2}}\s*\n{re.escape(footer_title)}\s*$', '',
>                   text, flags=re.MULTILINE)
> ```
>
> 章フッター語彙は書籍ごとに異なるため、コンテンツ構造分析（Step 3）の段階で目次見出しから抽出して動的に構築する。**位置依存ではなく語彙依存**にすることで、ページ内位置に関わらず確実にスイープできる。

## 判定7: 混在型

**判定条件:** 1つのEPUB内に異なる構造パターンのセクションが共存する。

**例:** Assessment Test（同一セクション型: `Correct Answer:` + `Explanation:` がQuestion直後）とPractice Exams（前半後半分離型: Questions セクションとSolutions セクションが分離）が共存。

**パース方法:** セクション単位で構造パターンを判定し、各セクションに適したパーサーを適用。最終的に全セクションのQ&Aペアを統合する。

実装手順:

1. ファイル全体をセクション境界（章見出し・大セクションタイトル）で分割
2. 各セクションを CONTENT-DETECTION.md の判別ロジックに通して個別に判定
3. 判定結果に応じて判定1〜6のいずれかのパース戦略を適用
4. 各セクションのQ&Aペアを統合し、共通の出力フォーマットで整形

判定漏れを防ぐため、(a) 全セクションのQ&Aペアが抽出されているか、(b) セクション間のQ&Aペアの重複・欠落がないか、を Step 7 品質検証で確認する。

## 判定8: 組み合わせ問題（サブアイテム + 選択肢の二層構造）

**判定条件:** 「次のうち、~の組み合わせはどれか」型の問題が混在する。問題文の下に **半角数字 `1`, `2`, `3`, `4` のサブアイテム**（命題候補）が並び、その下に **`ア 1と2` `イ 1と3` `ウ 2と4` `エ 3と4`** のような選択肢（組み合わせ）が続く二層構造。

**例:** 著作権検定上級、ビジネス法務検定2級、行政書士、公務員試験等で頻出する出題形式。

**典型構造:**

```
問題4 二次的著作物に関する次の記述のうち、正しいものの組み合わせはどれか。

1 二次的著作物となるためには、原著作物に依拠して創作される必要がある。
2 二次的著作物の利用には、二次的著作物の著作権者の許諾を得る必要はない。
3 二次的著作物の同一性保持権は、原著作物の著作者の同意を得なくても行使することができる。
4 二次的著作物の公衆への提供に際し、二次的著作物の著作者は氏名表示権を有しない。

ア 1と3
イ 1と4
ウ 2と3
エ 2と4
```

**パース戦略:**

サブアイテムと選択肢を同一の `choices` 辞書に押し込むと判定ロジックが崩れる（「ア」が「1と3」だけになり、命題1や命題3のテキストが失われる）。**サブアイテム用と選択肢用の二層 dict を別フィールドで保持する**:

```python
problem = {
    "q_text": "次のうち、〜の組み合わせはどれか。",
    "subitems": {"1": "命題1テキスト", "2": "命題2テキスト", "3": "命題3テキスト", "4": "命題4テキスト"},
    "choices": {"ア": "1と3", "イ": "1と4", "ウ": "2と3", "エ": "2と4"},
}
answer = {
    "correct": "ア",
    "subitems_expl": {"1": "命題1の○×解説", "2": "命題2の○×解説", ...},
    "explanations": {"ア": "選択肢ア全体の解説", "イ": "選択肢イ全体の解説", ...},
}
```

**パース時の正規表現分離**:

```python
# 選択肢（ア/イ/ウ/エ）優先で照合
choice_m = re.match(r"^([アイウエ])\s+(.*)$", stripped)
if choice_m:
    # 選択肢として扱う
    ...

# サブアイテム（1/2/3/4）は選択肢未開始時のみ受け付ける
sub_m = re.match(r"^([1-4１-４])\s+(.*)$", stripped)
if sub_m and cur_choice_letter is None:
    # サブアイテムとして扱う（全角数字も半角に正規化）
    ...
```

**HTML出力時は二層 `<ol>` でレンダリング**:

```html
[問題文]<br><br>
<ol style="list-style-type: none; padding-left: 0;">  <!-- 上の層: サブアイテム -->
  <li>1 命題1テキスト</li>
  <li>2 命題2テキスト</li>
  <li>3 命題3テキスト</li>
  <li>4 命題4テキスト</li>
</ol>
<br>
<ol style="list-style-type: none; padding-left: 0;">  <!-- 下の層: 選択肢 -->
  <li>ア 1と3</li>
  <li>イ 1と4</li>
  <li>ウ 2と3</li>
  <li>エ 2と4</li>
</ol>
```

**解答 Back の構成**:

```html
<div>正解: ア</div>
<b>各記述（命題）の解説:</b><br>
<b>1</b> 命題1の○×解説<br>
<b>2</b> 命題2の○×解説<br>
...
<b>各選択肢の解説:</b><br>
<b>ア</b> 選択肢ア全体の解説<br>
...
```

**実観測**: ビジネス著作権検定上級40問のうち約半数（17問）がこの構造だった。サブアイテム未対応のパーサーでは命題テキストが消失し、選択肢欄に「1と3」のような短文だけが残って学習に使えなくなる。判定3（同一セクション型）や判定4（章末解答型）の問題に**サブパターンとして混在**することが多い。

## <a id="判定8-パッセージ共有型"></a>判定8: パッセージ共有型（長文穴埋め・長文読解）

**特徴:** 1つの長文（Eメール・記事・通知）に複数の空欄または読解設問がぶら下がる。各空欄/設問を1カードにし、**パッセージ全文を各カードの Front に文脈として共有**する。語学長文読解対策書籍などで採用される。

**セグメント構造:** `No.N`（パッセージ番号）→ 本文 → 設問群 → `No.N 解答と解説`（`(n) 正解: X` + 解説）。

**パース戦略:**

1. **逐次セグメント化**: パッセージマーカー `No.N` で新パッセージを開始し、`No.N 解答と解説` で解答モードに切り替えて本文/解答を蓄積する。🔴 パッセージ番号は**再利用・欠番がありうる**（同じ `No.3` が複数の別パッセージに付く）ので、番号でなく**位置**でセグメントを区切る。Front が一意なので番号衝突は dedup に影響しない。
2. **本文と設問の分離**: `Questions:` ヘッダーがあればそこで分割する（信頼度高）。無ければ「**選択肢グループの直前の設問マーカー以降を設問**、それより前を本文」とする。本文中の番号リスト（`1. Plan Upgrade:` のような箇条書き）は実設問マーカーより前にあるため本文側に残り、設問と誤認されない。
3. **設問↔解答の対応**: パッセージ内の設問番号は `1./2./3.`（または `No.1/2/3`）とリセットし、解答側は `(1)(2)(3)` で対応する。番号がリセットするため**番号でなく出現順で zip** する。
4. **設問テキスト抽出**: 最初の選択肢グループ直前の run は `[本文 + Q1テキスト]` なので設問マーカーで分割し、2問目以降の run は選択肢グループ間にあるため**run 全体が設問テキスト**（本文混入なし）として扱う。
5. **検算と flag**: 設問数 ≠ 解答数のパッセージ、設問文が極端に短い（フラグメント化）カードは `needs_fix` で flag する。選択肢書式・正解書式の揺れは判定2の「自動生成系コンテンツ」の多段フォールバックを併用する。

## 資格試験テキストの「レッスン末〇✕テスト」型（解答編先行・番号脱落カスケード対策）

教科書型資格対策書（速習レッスン等）に頻出。各レッスン末に「理解度把握〇✕テスト」等の見出し＋番号付き設問（1〜N）が並び、同一ブロック末尾の「解答・解説」に `N.<○/✕>[解説]` 形式で正解＋解説が集約される**自己完結型**（前半後半分離でなく1ブロック完結）。実観測の抽出戦略:

1. **解答編先行で問題数を確定**: 「解答・解説」ブロックを `(\d+)\s*[.．]\s*([0○◯〇✕×xX])` で走査し、番号→判定を辞書化。これが設問数 N の信頼できる基準（設問本文側はOCR汚染を受けやすいが解答キーは規則的）。🔴 **○ が数字 `0` にOCRされる**（`1.0` = 1.○）。`anki_toolkit._VERDICT_NORMALIZE` が `0`→○ を吸収するが、パーサー側でも明示マップを持つと安全。
2. **設問を番号マーカーで逐次切り出し**: 設問番号は「`N本文`（glued）」か「`N`単独行」の両形がある。`(?m)^N(?=\D|$)` で境界を取り、`pos` を進めながら 1..N を順に抽出。
3. **🔴 番号脱落カスケードの回避**: OCRが設問番号を散発的に脱落させると（例: 3,4が無番号化）、固定で「次番号 N+1」を境界探索すると見つからず前設問が末尾まで吸収→以降全滅する。**END は「次に*実在する*解答番号」を探す**（`for nxt in range(N+1, nmax+1): if nxt in ans …`）ことで脱落を飛ばし、カスケードを断つ。
4. **吸収しすぎ設問の除外**: 上記でも無番号設問は前設問に併合される。○×設問は通常120字未満なので、**160字超の設問は複数併合の疑いとして除外**し品質を守る（量より質）。
5. **サイドバートピック名の混入除去**: 段組レイアウトのOCR（特に Apple Vision）では、欄外のトピック名（「物質の三態」等の短い名詞句）が設問本文の行間に注入される。設問クリーニングで「7字以下・句読点/括弧/。なし・かな漢字のみの独立行」を除去する。先頭の箇条記号 `•/■/●/・` も剥がす。
6. **タグ・テーマ**: ブロック末尾の `Lesson N■<タイトル>` からテーマ名を取り、`テーマ:<title>` タグへ。ノートタイプは ○×型＝`Basic`{Front,Back}。実績: 529ページ教科書から59ブロック・抽出率93%（846問中788問）。

## 資格試験テキストの「章末まとめ問題・予想模擬試験」型（5択MCQ・前半後半分離）

同じ教科書の章末「まとめ問題」や巻末「予想模擬試験」は **5択MCQ**。`問題N` ＋設問文 ＋ `（1）`〜`（5）` の選択肢で、解答は別位置（章末解答型 or 巻末解答編）に分離する:

- **まとめ問題（章末解答型）**: `まとめ問題`→`問題N`＋選択肢 →（別ページ）`解説`→`問題N`解説 →（別ページ）`正解（X）` 列（`DLessonN P.xx`参照付き・問題順に並ぶ）。設問・解説・正解が3箇所に分離するため**出現順で zip** する。
- **予想模擬試験（巻末解答編）**: 問題編（`問題N`＋選択肢・回ごと PDF 連続ページ）と解答編（`問題N 解答（X）`＋`〔解説〕`・`■<科目>` 見出し付き）が分離。第1回/第2回で `問題N` がリセットするので**回でスコープ**して番号照合。`■<科目>` は `Knowledge Area`／分野タグへ。
- **🔴 表・2D図のOCR崩壊**: MCQ の設問に埋め込まれた表（臨界温度表等）は Apple Vision で列が崩壊する。該当設問は (a) `needs_fix` タグ付与で投入、または (b) 当該ページのみ VLM(recognize-image-to-markdown) で再OCRして表を復元（[OCR-CONVERSION.md](OCR-CONVERSION.md) の「Apple Vision の弱点と VLM 併用」参照）。解答編はテキストのみで Vision でもクリーンなことが多く、問題編だけ VLM 再OCRすれば足りる。
- ノートタイプは選択肢型＝`資格試験(ランダム)`{Question,Answer,Knowledge Area}・`RenderOptions(choice_list_style="ol")`。
