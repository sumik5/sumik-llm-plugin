# AUGMENTED-PRACTICES.md

チームと個人がLLMとの対話を日常に組み込むための実践ガイド。
個人プラクティス・チームプラクティス・ワークショップ・アジャイルリチュアル拡張・マニフェストを収録。

---

## 1. 個人プラクティス

### LLMをパーソナルパートナーとして育てる

LLMは単なる補助ツールではなく、自分の思考を映す**対話パートナー**として機能する：

| 役割 | 説明 |
|------|------|
| 思考の伴走者 | アイデアを声に出しながら整理する相手 |
| 選択肢シミュレーター | 「この設計にはどんな代替案があるか」を素早く探索 |
| 技術アドバイザー | 特定のドメイン知識を即座に参照 |
| 思考の鏡 | 自分の意図の曖昧さや矛盾を可視化してくれる存在 |

> LLMはあなたの専門性を置き換えない。正しい問いを立てられる人間の能力を増幅する。

### 自分のパターンを育てる

試行錯誤を通じて、自分に最も効果的な対話シーケンスを発見できる：

- プロジェクトコンテキストの導入方法
- 回答精度を上げるプロンプト構造
- 曖昧な回答を絞り込む再フレーミング手順

**発見したパターンには名前をつけて記録する**。良いパターンは、乗り越えてきたフラストレーションから生まれることが多い。

### パーソナルプロンプトライブラリの構築

有効なインタラクションを記録・管理することで長期的な学習リポジトリを構築できる：

```markdown
## プロンプトライブラリ構成例

### カテゴリ別分類
- 設計探索（アーキテクチャ代替案、トレードオフ比較）
- コード分析（レガシー理解、リファクタリング方針）
- テスト設計（受け入れ基準生成、エッジケース発見）
- 要件整理（ユーザーストーリー、曖昧さの解消）

### 各プロンプトに記録する項目
- 使用したプロンプト
- 得られた結果の質（有効 / 失敗 / 改善が必要）
- 改善後の再フレーミング
- 再利用可能な条件・コンテキスト
```

**ツール例**: Obsidian、Notion、Markdownファイル、チームWiki

## 2. チームプラクティス

### パターンを可視化する会話習慣

スタンドアップ、レビュー、デブリーフ中に以下の問いを添える：

- 「そのプロンプト、どう構成した？」
- 「どのパターンを使った？」
- 「インテントテストはどう設定した？」

これらの問いによって、チームのAI対話の実践が**可視化され、共有知**になる。

### パターン強化コードレビュー

コードレビューに「LLMとの対話」という次元を加える：

1. 使用したプロンプトを提示する
2. AIとのやり取りの中で行ったトレードオフを説明する
3. AIの提案の信頼性や論理を問い直す

> レビューの対象はコードだけではない。そのコードに至った**推論プロセス**もレビューする。

### ペア対話によるプロンプト共同設計

2人のチームメンバーがLLMと協働する：

| 役割 | 担当 |
|------|------|
| ドライバー | プロンプトを入力し、LLMに質問する |
| オブザーバー | 回答を観察し、再フレーミング・バリエーションを提案する |

プロンプトが「2人で構築した共有オブジェクト」になることで、チームパターンが生まれる：
- 「私たちの設計探索方法」
- 「私たちのリファクタリングテンプレート」

### チームプロンプトブックの運用

アジャイルチームがLLM対話の「方言」を作り上げる実践：

- 各スプリント：有効なプロンプトを文書化し、新興パターンを形式化する
- 蓄積されたプロンプトブックは、新メンバーの研修資料にも、レトロスペクティブの議論ツールにもなる

**環境整備のポイント**：
- **実験文化**: 失敗を恐れず試せる心理的安全性
- **自律信頼**: チームが自分たちのペースでLLMを活用する裁量
- **内省時間の評価**: 即時生産だけでなく、対話・再フレーミングに費やす時間を価値として認める

## 3. 5つのワークショップ

| # | ワークショップ名 | 目的 | 所要時間 | 参加者 | 形式 |
|---|----------------|------|----------|--------|------|
| 1 | Team Prompt Design | プロンプトの集団的な構成・改善・テスト | 90〜120分 | 3〜6名（開発者・PO・UX・QA） | ペア作業 + 全体ファシリテーション |
| 2 | Dialogue Daily | LLM対話の経験を毎日共有する軽量リチュアル | 5〜10分 | チーム全員 | デイリー or 週2回の定例 |
| 3 | Dialogue Motif Mapping | チームで有用なLLM対話パターンを特定・マッピング | 90〜120分 | 4〜8名 | 二重エントリーマトリクス（頻度×有用性） |
| 4 | Absurd Prompts Game | LLMの限界・ハルシネーション・バイアスをユーモアで探索 | 60〜90分 | 4〜10名 | 全体 + グループ討議 |
| 5 | LLM Ready Team Reference | チームのLLM活用ガイドを実体験から共同構築 | 120分（2×60分に分割可） | 4〜8名 | 壁面 or Miro + 共同編集 |

### Workshop 1: Team Prompt Design（詳細フロー）

```
1. イントロ & フレーミング（10分）
   - 実際のユースケースを選択（モジュール分割、テスト設計、技術選定...）

2. 最初のナイーブなプロンプト（15分）
   - 「このコンテキストでLLMに何を聞く？」をチームで初稿作成
   - 問題の特定：曖昧さ、技術用語の未定義、暗黙の意図

3. イテレーション & リフレーミング（30〜40分）
   - 異なる角度でプロンプトを再構成：
     - 明示的なロール（「あなたはソフトウェアアーキテクトです...」）
     - ステップバイステップ形式
     - 箇条書き構造化版
     - 批判的 / 探索的 / 生成的バージョン

4. プロンプトパターンの抽出（15〜20分）
   - 再利用可能なパターンを形式化：基本構造、オプションモジュール、使用条件、落とし穴

5. 振り返りと学習（10〜15分）
   - 「今日学んだこと」「明日使えること」「まだ試したいこと」
```

**落とし穴**:
- 「正しい答え」を求めて、バリエーションのテストを怠る
- プロンプトの背後にある隠れた意図を言語化しない
- 一人がタイプしながら他全員が見ているだけ

### Workshop 2: Dialogue Daily（詳細フロー）

```
1. オープニング（1分）
   「今日、AIは何を教えてくれた？」

2. 3つの問い（掲示または口頭）
   - LLMで何を試した？
   - 何が驚き、役立ち、または失望させたか？
   - 何を持ち帰るか、または次に試したいか？

3. 自発的または順番での共有（5〜8分）
   - 成功・失敗・興味深いプロンプト・バイアスの観察など

4. クロージング & 記録（1〜2分）
   - プロンプトグリモワール（呪文書）への追加
   - Slack #daily-llm、Miro/Notionへの記録
```

### Workshop 5: LLM Ready Team Reference（詳細フロー）

Co-createしたチーム固有のリファレンスドキュメントを構築する：

```
ドキュメント構成：
- プロンプトタイプ集（用途別：コード分析、テスト、変換、探索）
- 回答品質評価グリッド（関連性・堅牢性・透明性・セキュリティ・チーム標準への整合）
- 使用ルール（いつLLMを使うか、いつ人間がバリデーションするか）
- ブラックリスト（よくある落とし穴：曖昧なプロンプト、ハルシネーション傾向）
- ガバナンス：1名のリファレンス管理者 + 1〜2名の定期更新担当
```

> リファレンスは完成しない。チームとともに進化する。

## 4. アジャイルリチュアル拡張

LLMはアジャイルを置き換えない。**各リチュアルを拡張する**。

### Planning ― サイレントペアとしてのLLM

| 活用例 | 具体的な問い |
|--------|-------------|
| ユーザーストーリーの明確化 | 「このストーリーで考慮すべきエッジケースは？」 |
| 代替実装案の生成 | 「この機能を実装する3つの方法とその長所・短所を提案して」 |
| 受け入れ基準の自動生成 | ストーリーからAcceptance Criteriaのドラフトを生成し、チームでレビュー |

LLMは意思決定を行わない。**探索空間を広げる**。

### Daily ― 同期の質を高める

```
クイックアクション例:
- ブロッカーをアクション可能なプロンプトとして再フレーム
- 前日のLLM活用を「即席レッスン」として共有
  例：「昨日、実装ミラーパターンを使ってPRのブロックを解除した」
- Kanbanボードの動きの自動サマリー（Jira/GitHub連携）
```

**リスク**: Dailyがスタティックなレポートに化骨化しないよう、LLMサマリーを**定性的な対話の時間を増やすため**に使う。

### Review ― 成果物とプロセスの両方を評価

- PRに使用したプロンプトと意思決定の記録を添付する
- 「LLMはどのトレードオフを明示してくれたか？」を問う
- AI生成コードと手書きコードのレビュー基準を統一する

### Retrospective ― 反省の深化

```
LLMを使ったRetrospective活用例:
- チケットログ・コードコメント・会議記録から繰り返しのパターンを分析
  「過去3スプリントで2回以上のテスト反復が必要だったストーリーは？」
  「本番に報告されたバグの共通点は？」
- AIの使用状況を振り返るセクションを追加（AI Watch Time）
```

**チームへの5つの実践原則**:

| 原則 | 内容 |
|------|------|
| Co-build | 「いつ・どう使うか」をチームで決める |
| 透明性維持 | モデルとのやり取りを文書化、重要なプロンプトをアーカイブ |
| 人間コントロール | AIは提案する、チームが決定する |
| 相互学習 | AIの利用状況を定期的に振り返り、継続的改善の文化を作る |
| バイアス検出 | 常にLLMの提案を検証。特にアーキテクチャ・セキュリティ・パフォーマンスに関して |

## 5. 拡張開発マニフェスト v0.1

```
私たちはAIと共に設計する、その代わりではなく。
意図を精確に、好奇心と厳密さを持って表明する。
批判的思考と協働によって回答を評価する。
明示的で共有可能な実践を中心にチーム文化を構築する。
学ぶために実験し、改善するために共有し、続けるために適応する。
```

これは開かれたマニフェストである。あなたの拡張・再フレーミング・実世界のケースを待っている。

## 6. 文化的ポスチャー：沈黙か、対話か

AIの使用をめぐる羞恥心は、多くの場合**個人の問題ではなく、集団がAI活用を受け入れる方法を知らないことの反射**である。

### 2つのポスチャーの対比

| | ポスチャーA（沈黙・回避） | ポスチャーB（対話・共有） |
|---|---|---|
| AIの使用 | 秘匿、プライベートタブで実施 | 公開・ペアでの探索として位置づけ |
| コミット | AI使用を記録しない | PRにプロンプト使用を明記 |
| チームへの影響 | 孤立した生産性、知識の断絶 | 集団的学習、知識の継承 |
| 内面状態 | 「ズルをした」という後ろめたさ | 「より良く考えるためのツール」という誇り |

### 羞恥心ではなく、明晰さを

IDEを使うことを恥じていたか？ライブラリを使うことを恥じていたか？Stack Overflowを恥じていたか？LLMも、この長い開発者拡張ツールの連鎖の一部である。

**本当に注意すべきこと**：
- テスト・検証なしに回答を統合すること
- 理解していない解決策をコピーすること
- 生成されたコードの出所を隠すこと
- 再フレーミングも思考もしていないアイデアを自分のものとして提示すること

### No-Shame Cultureの構築

「AIを使わない開発者」ではなく、「**適切なツールを使ってより良く考える開発者**」を目指す文化は共同責任である。それはリチュアル・対話の場・失敗が学習に変わるフレームワークによって実現される。

AIをどう使うかではなく、**それについて人間として、共に語り合う方法**こそが問われている。
