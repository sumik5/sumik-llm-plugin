# エンタープライズアカウント設計（組織・フォルダ・プロジェクト・IAM）

エンタープライズ企業が Google Cloud を利用する際のアカウント設計の全体像を扱う。リソースコンテナ階層設計、Cloud Billing 管理、IAM によるユーザー権限管理、組織ポリシーの適用まで、運用・管理・統制の観点から実践的な設計指針を提供する。

## リソースコンテナ階層設計

### 3 層リソースコンテナの概念

Google Cloud では「組織」「フォルダ」「プロジェクト」の 3 層でリソースを管理する。

| コンテナ | 役割 | 課金単位 | スコープ |
|---------|------|---------|---------|
| **組織** | 企業全体のルートコンテナ。企業ドメイン（例: example.co.jp）に紐付く | - | 組織全体のポリシー管理 |
| **フォルダ** | プロジェクトをグルーピングする中間層。部署・事業部単位で分割 | - | フォルダ配下の一括ポリシー |
| **プロジェクト** | リソースをグルーピングする最小単位。VM・DB・Storage 等を含む | ✅ 課金最小単位 | プロジェクト内リソース |

**ポイント:** 上位階層のポリシーは下位に継承される。組織ポリシーを最上位に設定すれば、全配下リソースに自動適用できる。

### 階層構造パターン

| パターン | 設計方針 | メリット | デメリット | 適用場面 |
|---------|---------|---------|-----------|---------|
| **環境指向** | 開発/テスト/本番の環境ごとにフォルダ分割 | 管理負荷小 | 部署間の共有制御が難しい | 小〜中規模、部署が少ない |
| **機能指向** | 部署や事業部ごとにフォルダ分割 | 組織構造と連動しやすい | 環境分離が横断的になる | 部署が多い中規模組織 |
| **詳細指向** | 部署×環境の組み合わせで細かく分割 | 最大の柔軟性・細粒度制御 | 管理負荷が最大 | 大規模エンタープライズ |

**推奨設計手順:**
1. 企業の組織構造をフォルダ階層にマッピング
2. 環境（開発/テスト/本番）はプロジェクト単位で分離
3. IAMポリシーを階層に合わせて設計し、継承を活用

### Resource Manager の活用

Resource Manager は、リソースコンテナの作成・移動・ポリシー設定を一元管理するサービス。

| 機能 | 説明 | ユースケース |
|------|------|------------|
| **階層構成** | 組織/フォルダ/プロジェクトの CRUD | 新部署設立時のフォルダ追加 |
| **ポリシー設定** | IAMポリシー・組織ポリシーの適用 | 組織全体セキュリティポリシー |
| **REST API対応** | プログラム的な操作・自動化が可能 | Terraform/Deployment Managerによる IaC |

## Cloud Billing（請求先管理）

### Billing の 2 層構造

```
Googleお支払いプロファイル（Google レベル）
    └─ Cloud請求先アカウント（Google Cloud レベル）
            └─ プロジェクト（課金対象）
```

| 要素 | 説明 | 企業利用における注意点 |
|-----|------|-------------------|
| **Googleお支払いプロファイル** | すべての Google サービスの支払い管理 | 企業は「ビジネス」タイプを選択。複数ユーザーが管理可 |
| **Cloud請求先アカウント** | Google Cloud 専用の課金管理単位 | IAM ロールで管理ユーザーを厳密に制限する |
| **プロジェクト** | 実際の課金発生場所 | 作成時に必ず Cloud 請求先アカウントを指定 |

### Cloud請求先アカウントの種類

| 種別 | 支払い方法 | 特徴 | 選択基準 |
|-----|----------|------|---------|
| **セルフサービスアカウント** | クレジットカード等 | 即時利用可能 | 小規模・スタートアップ向け |
| **請求書発行アカウント** | 請求書払い | 所定の審査条件を満たす必要あり | 大企業・安定的な取引実績がある場合 |

### 費用管理ベストプラクティス

| 施策 | 内容 | 効果 |
|-----|------|------|
| **予算アラート** | 費用が閾値（50%/80%/100%等）に達したらアラート送信 | 過剰課金の早期検知 |
| **Pub/Sub 連携** | アラートをトリガーにプロジェクト自動停止等を実装 | 予算超過の自動制御 |
| **Pricing Calculator** | 事前に概算費用を試算 | 計画段階でのコスト把握 |
| **ラベル付け** | リソースにラベルを付与してコスト按分 | 部署別・プロジェクト別コスト分析 |

## アイデンティティ（IAM）管理

### 3 種類のアイデンティティ

| 種別 | 説明 | 主な用途 | 管理主体 |
|-----|------|---------|---------|
| **ユーザーアカウント** | 個人に紐付く Googleアカウント | コンソール操作、人間による作業 | 企業管理推奨（Cloud Identity） |
| **Googleグループ** | 複数ユーザーをグループ化する仮想グループ | ロールの一括付与・管理委任 | 企業管理 |
| **サービスアカウント** | アプリや自動化ツールが使用するアカウント | API コール、機械間連携 | Googleまたはユーザー管理 |

### ユーザーアカウント管理方式

| 方式 | 特徴 | 推奨ケース |
|-----|------|----------|
| **個人管理** | 個人の Gmail アカウントで操作 | 個人開発・検証のみ |
| **企業管理（Cloud Identity）** | 企業ドメインで一元管理。パスワードポリシーや MFA を強制適用可 | **エンタープライズでは必須** |
| **外部 IdP 連携（SSO）** | Active Directory / Azure AD / Okta 等と連携してシングルサインオン | 既存 ID 基盤がある場合 |

**外部 IdP 連携方式:**
- Active Directory → **Google Cloud Directory Sync（GCDS）** でユーザー同期
- Azure AD / Okta → 専用アダプタで Cloud Identity にユーザー情報を複製

### サービスアカウントの種類

| 種別 | 作成主体 | キー管理 | 使用場面 |
|-----|---------|---------|---------|
| **デフォルト SA** | Google が自動作成 | Googleが管理 | GCE・GKE のデフォルト SA（編集者ロールは**無効化推奨**） |
| **ユーザー作成 SA** | ユーザー | ユーザーが管理 | アプリから特定 API を呼び出す場合 |
| **Google API SA** | Google が自動作成 | Googleが管理 | Google サービス内部利用 |

⚠️ **サービスアカウントキー管理の注意点:**
- Google 管理キー: 自動ローテーション、漏洩リスク低
- ユーザー管理キー: オンプレミスからの API 呼び出し時に使用。定期ローテーション・アクセス制限を自分で実施

### IAM ロールの 3 種類

| 種別 | 説明 | 推奨度 |
|-----|------|--------|
| **基本ロール** | オーナー/編集者/閲覧者。広範囲の権限 | ❌ 本番での使用は避ける |
| **事前定義ロール** | Google が提供するサービスごとの細粒度ロール | ✅ 基本はこれを使用 |
| **カスタムロール** | ユーザーが独自に権限を組み合わせて定義 | 事前定義ロールで不足する場合のみ |

### IAM ポリシーの適用スコープと継承

IAM ポリシーは **組織 → フォルダ → プロジェクト → リソース** の階層で継承される。

| 適用レベル | 効果 |
|----------|------|
| 組織レベル | 全プロジェクト・リソースに適用 |
| フォルダレベル | フォルダ配下の全プロジェクトに適用 |
| プロジェクトレベル | プロジェクト内の全リソースに適用 |
| リソースレベル | 特定リソースのみに適用（最も細粒度） |

**IAM Conditions の活用:** 時間帯・リソース属性などの条件付きでロールを付与可能。例:「業務時間帯は管理者権限、それ以外は閲覧者権限」

## ベストプラクティス

### 最小権限の原則を徹底する

| 施策 | 内容 |
|-----|------|
| 事前定義ロールを活用 | 必要最小限の権限を含むロールを選択 |
| IAM Recommender 利用 | 実際の利用状況から過剰な権限を検出・削減提案 |
| IAM Conditions 活用 | 時間や条件に応じた細粒度アクセス制御 |
| デフォルト SA の編集者ロール無効化 | 組織ポリシーで SA 作成時のデフォルト編集者ロール付与を禁止 |

### 組織ポリシーの活用

**組織ポリシー** は IAM ポリシーとは別に、組織・フォルダレベルで適用できる制約。

| ポリシータイプ | 説明 | 使用例 |
|-------------|------|--------|
| **リスト型（許可リスト）** | 許可する値を列挙 | 特定リージョンのみリソース作成を許可 |
| **リスト型（拒否リスト）** | 拒否する値を列挙 | 外部 IP アドレスの付与を禁止 |
| **ブール型** | 制約の有効/無効を設定 | サービスアカウントキーの作成を禁止 |

⚠️ **継承ルール:** 許可リストと拒否リストが競合した場合、**拒否リストが優先**される。混在設定は避けること。

### ID の一元管理

| ケース | 推奨対応 |
|-------|---------|
| Google Workspace 利用中 | そのまま Cloud Identity として活用 |
| Active Directory 保有 | GCDS でユーザー同期 → SSO を実現 |
| 企業メールで個人 Google アカウントを作成済みの社員がいる | アカウント統合（Cloud Identity への移行）が必要 |

### プロジェクト作成の自動化

| ツール | 用途 |
|-------|------|
| **Terraform** | IaC でプロジェクト・IAM・組織ポリシーを一括管理 |
| **Cloud Deployment Manager** | Google Cloud ネイティブの IaC ツール |
| **Ansible/Puppet** | サードパーティの構成管理ツール |

自動化によって得られるメリット:
- 命名規則・ラベル付けの標準化
- セキュリティ設定の確実な適用
- プロジェクト再構成（リファクタリング）の効率化

### グループを使った業務委任

| グループ例 | 付与するロール例 | 効果 |
|----------|---------------|------|
| data-scientists@example.com | BigQuery データ閲覧者 | 新メンバー追加時はグループへの追加だけでOK |
| network-admins@example.com | ネットワーク管理者 | 役割変更時はグループのロール変更だけでOK |
| app-developers@example.com | Cloud Run 開発者 | 個人へのロール付与が不要 |

## 設計チェックリスト

| カテゴリ | 確認項目 |
|---------|---------|
| **組織設計** | Cloud Identity または Google Workspace で組織を設定しているか |
| **階層設計** | 企業の組織構造がフォルダ階層に反映されているか |
| **環境分離** | 開発/テスト/本番がプロジェクトレベルで分離されているか |
| **IAM設計** | 最小権限の原則に基づいて事前定義ロールを使用しているか |
| **グループ管理** | 個人ではなく Googleグループに対してロールを付与しているか |
| **SA設計** | デフォルト SA の編集者ロールを無効化しているか |
| **組織ポリシー** | 外部 IP 禁止・SA キー作成禁止等の組織ポリシーを適用しているか |
| **費用管理** | 予算アラートを設定しているか |
| **ID管理** | 既存 ID 基盤と Cloud Identity が連携されているか（必要な場合） |
| **自動化** | Terraform 等でプロジェクト作成を自動化しているか |
