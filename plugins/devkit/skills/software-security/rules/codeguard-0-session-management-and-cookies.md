## セッション管理と Cookie

使いやすさを維持しながら、固定化・ハイジャック・盗難を防ぐ、堅牢で攻撃耐性のあるセッション処理を実装してください。

### セッション ID の生成と特性
- CSPRNG を使用してセッション ID を生成してください。エントロピーは 64 ビット以上（128 ビット以上を推奨）。不透明で推測不可能、かつ意味を持たないものにしてください。
- フレームワークのデフォルトではなく、汎用的な Cookie 名（例：`id`）を使用してください。サーバーが発行していない受信 ID はすべて拒否してください。
- すべてのセッションデータはサーバーサイドに保存し、トークンに PII や権限情報を埋め込んではいけません。機密性が高い場合は、サーバーサイドのセッションストアを保存時に暗号化してください。

### Cookie のセキュリティ設定
- セッション Cookie には `Secure`、`HttpOnly`、`SameSite=Strict`（フローに必要な場合は `Lax`）を設定してください。
- `Path` と `Domain` で Cookie のスコープを狭く限定してください。クロスサブドメインへの露出を避けてください。
- 非永続的なセッション Cookie（Expires/Max-Age なし）を優先してください。完全な HTTPS を要求し、サイト全体で HSTS を有効化してください。

ヘッダーの例：
```
Set-Cookie: id=<opaque>; Secure; HttpOnly; SameSite=Strict; Path=/
```

### セッションのライフサイクルとローテーション
- セッションはサーバーサイドでのみ作成し、提供された ID は信頼されていない入力として扱ってください。
- 認証・パスワード変更・権限昇格のたびにセッション ID を再生成してください。以前の ID は無効化してください。
- フレームワークのパターンで必要な場合は、認証前と認証後の Cookie 名を区別してください。

### 有効期限とログアウト
- アイドルタイムアウト：高価値セッションは 2〜5 分、低リスクセッションは 15〜30 分。絶対タイムアウト：4〜8 時間。
- タイムアウトはサーバーサイドで適用してください。サーバーセッションを完全に無効化し Cookie をクライアントサイドで消去する、見えるところにログアウトボタンを設置してください。

### 転送とキャッシュ
- セッション全体を通じて HTTPS を強制してください。1 つのセッション内で HTTP/HTTPS を混在させては絶対にいけません。
- セッション識別子や機密データを含むレスポンスには `Cache-Control: no-store` を送信してください。

### Cookie 盗難の検出と対応
- セッション確立時にサーバーサイドでコンテキストをフィンガープリントしてください（IP、User-Agent、Accept-Language、利用可能な場合は関連する `sec-ch-ua`）。
- 受信リクエストを保存済みフィンガープリントと比較し、無害なドリフト（サブネット変更・UA 更新など）は許容してください。
- リスクベースの対応：
  - 高リスク：再認証を要求し、セッション ID をローテーションする。
  - 中リスク：ステップアップ検証（チャレンジ）を行い、セッション ID をローテーションする。
  - 低リスク：不審なアクティビティをログに記録する。
- ハイジャックの可能性が検出された場合は、常にセッション ID を再生成してください。

### クライアントサイドストレージ
- XSS リスクがあるため、`localStorage`/`sessionStorage` にセッショントークンを保存しないでください。転送には HttpOnly Cookie を優先してください。
- セッション以外の機密情報にクライアントサイドストレージがどうしても避けられない場合は、Web Workers を通じて分離し、ページのコンテキストには絶対に露出しないでください。

### フレームワークと複数 Cookie のシナリオ
- 組み込みのセッションフレームワークを優先し、常に最新の状態にハードニングしてください。
- 複数の Cookie がセッション状態に関与する場合は、それらの関係を検証してください。パス/ドメインをまたいで同じ Cookie 名を使用しないでください。

### 監視とテレメトリ
- セッションのライフサイクルイベント（作成・ローテーション・終了）は、セッション ID の生の値ではなくソルト付きハッシュを使用してログに記録してください。
- セッション ID のブルートフォース攻撃と異常な同時使用を監視してください。

### 実装チェックリスト
1) CSPRNG によるセッション ID（エントロピー 64 ビット以上）、不透明でサーバー発行のみ。
2) Cookie フラグ：`Secure`、`HttpOnly`、`SameSite` を設定；ドメイン/パスを厳格に限定。
3) HSTS を伴う HTTPS のみ；混在コンテンツなし。
4) 認証と権限変更時に ID を再生成；古い ID を無効化。
5) アイドルおよび絶対タイムアウトをサーバーサイドで適用；完全なログアウトを実装。
6) 機密性の高いレスポンスに `Cache-Control: no-store` を設定。
7) サーバーサイドのフィンガープリンティングと異常に対するリスクベースの対応。
8) クライアントストレージにセッショントークンを保存しない；フレームワークのデフォルトをハードニング。
