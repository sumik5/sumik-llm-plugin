# Infrastructure as Code (IaC) セキュリティ

Terraform や CloudFormation などを用いてクラウドインフラを設計・IaC を記述する際は、常にセキュアなプラクティスとデフォルト（公開露出の防止や最小権限の原則の遵守など）を採用してください。セキュリティ上の設定ミスを積極的に特定し、安全な代替案を提示してください。

## Infrastructure as Code における重要なセキュリティパターン

### ネットワークセキュリティ
- リモート管理サービス・データベース・LDAP・TACACS+ などの機密性の高いサービスへのアクセスは、**常に**制限してください。必要でないサービスをインターネット全体から到達可能にしてはなりません。代わりに、アクセスが必要な特定の IP アドレスまたは CIDR ブロックに限定してください。
    - Security Group および ACL のインバウンドルールで、リモート管理ポート（SSH 22、RDP 3389 など）に対して `0.0.0.0/0` を許可しては**絶対にいけません**。
    - Security Group および ACL のインバウンドルールで、データベースポート（3306、5432、1433、1521、27017 など）に対して `0.0.0.0/0` を許可しては**絶対にいけません**。
    - Kubernetes API エンドポイントの許可リストで `0.0.0.0/0` を許可しては**絶対にいけません**。EKS・AKS・GKE などすべての Kubernetes API エンドポイントは、管理アクセスが必要な CIDR アドレスの許可リストに限定してください。
    - クラウドプラットフォームのデータベースサービス（RDS・Azure SQL・Cloud SQL）をすべての IP アドレス `0.0.0.0/0` に公開しては**絶対にいけません**。
- パブリックネットワークアクセスが必要な場合を除き、内部 VPC・VNET・VPN などのプライベートネットワークを優先してください。
- ネットワーク監視とセキュリティ分析のために、VPC/VNET フローログを**常に**有効化してください。
- デフォルト拒否ルールと、必要なトラフィックのみを対象とした明示的な許可ルールを**常に**実装してください。
- デフォルトでインターネットへのエグレストラフィックをブロックすることを優先してください。エグレスが必要な場合、適切なトラフィック制御ソリューションとして以下が考えられます：
    - 特定の必要なサービスへのアクセスを許可するルールを持つエグレスファイアウォールまたはプロキシ。
    - 特定の必要な IP または CIDR ブロックへのアクセスを許可するルールを持つエグレスセキュリティグループ（SG）またはアクセス制御リスト（ACL）。
    - 悪意あるドメインへのアクセスを防ぐ DNS フィルタリング。

### データ保護
- データベース・ファイルシステム・オブジェクトストレージ・ブロックストレージを含むすべてのストレージサービスで、保存時のデータ暗号化を**常に**設定してください。
    - クラウドストレージサービス（S3・Azure Blob・GCS バケット）の暗号化を有効化してください。
    - すべてのデータベースエンジン（RDS・Azure SQL・Cloud SQL・DocumentDB など）で保存時のデータベース暗号化を設定してください。
    - 仮想マシンのストレージボリュームで EBS/ディスク暗号化を有効化してください。
- すべてのデータ通信で転送時の暗号化を**常に**設定してください。
    - すべての HTTPS/API 通信で TLS 1.2 以上を使用してください。
    - 証明書検証を伴う SSL/TLS をデータベース接続に設定してください。
    - VPC/VNET 内のサービス間通信の暗号化を有効化してください。
    - リモートアクセスには暗号化されたプロトコル（SSH・HTTPS・セキュアな RDP）を使用してください。
- 機密レベルに基づいたデータ分類と保護コントロールを**常に**実装してください。
    - PII・PHI・財務データ・知的財産には、より厳格な暗号化とアクセス制御を適用してください。
    - データ分類レベルごとに個別の暗号化キーを使用してください。
- セキュアなデータ保持・廃棄ポリシーを**常に**設定してください。
    - 規制要件およびビジネス要件に基づいてデータ保持期間を定義してください。
    - セキュアな削除を伴う自動データライフサイクル管理を実装してください。
- 包括的なデータアクセス監視と監査を**常に**有効化してください。
    - すべてのデータアクセス・変更・削除操作をログに記録してください。
    - 異常なデータアクセスパターンや潜在的なデータ流出を監視してください。
    - 機密データのアクセス違反に対するリアルタイムアラートを実装してください。
- データバックアップを**常に**暗号化してください。
    - 本番データとは別の暗号化キーを使用してすべてのバックアップデータを暗号化してください。
    - 適切なアクセス制御を伴い、地理的に分散した場所にバックアップを保存してください。
    - バックアップ復元手順を定期的にテストし、バックアップの整合性を検証してください。

### アクセス制御
- 重要な管理サービスやデータサービス（バックアップ・ストレージ・コンテナレジストリ・ファイル共有など）は、パブリック分類として明示的にラベル付けされているか公開を意図している場合を除き、匿名アクセスを許可したままにしては**絶対にいけません**。
- IAM ポリシーやクラウド RBAC でワイルドカード権限（`"Action": "*"`、`"Resource": "*"`）を使用しては**絶対にいけません**。
- 必要でない場合にサービスアカウントに Owner/Admin ロールを過剰付与しては**絶対にいけません**。
- サービスの API キーやクライアントシークレットを使用せず、代わりにロールベースのアクセス制御を伴うワークロードアイデンティティを使用して、長期有効な認証情報の必要性をなくしてください。**絶対に**サービス API キーとクライアントシークレットを使用しないでください。
- AWS でレガシーのインスタンスメタデータサービスバージョン 1（IMDSv1）を有効化・使用しては**絶対にいけません**。
- OAuth のようなより安全な代替手段がある場合、レガシーまたは時代遅れの認証方式（ローカルユーザーなど）を使用しては**絶対にいけません**。

### コンテナおよび VM イメージ
- ハードニングされていない VM やコンテナイメージを使用しては**絶対にいけません**。
- ディストロレスまたは最小限のコンテナイメージを**常に**選択してください。
- 信頼できるソースからのセキュアなベースライン仮想マシンイメージの使用を**推奨**します。
- 信頼できるソースからの最小限のディストロレスコンテナイメージの使用を**推奨**します。

### ロギングと管理アクセス
- 機密サービスの管理アクティビティログを無効化しては**絶対にいけません**。
- 特権操作の監査ログを**常に**有効化してください。

### 機密情報管理
- 機密情報・パスワード・API キー・証明書を IaC ソースコードに直接ハードコードしては**絶対にいけません**。
- Terraform では機密情報に `"sensitive = true"` を**常に**付与し、その他の IaC コードでは適切なアノテーションやメタデータを使用して機密性の高い値を示してください。

### バックアップとデータ復旧
- 保存時および転送時の暗号化なしにバックアップを作成しては**絶対にいけません**。
- バックアップにはクロスリージョンレプリケーションを伴うマルチリージョンデータストレージを**常に**設定してください。
- 保持ポリシーとライフサイクル管理なしにバックアップを設定しては**絶対にいけません**。
