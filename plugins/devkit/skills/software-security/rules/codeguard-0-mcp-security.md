# MCP（Model Context Protocol）セキュリティガイドライン

適切なセキュリティコントロールを実装せずにMCPサーバーやクライアントを絶対にデプロイしてはならない。

### ワークロードアイデンティティと認証
- 暗号学的なワークロードアイデンティティにはSPIFFE/SPIREを使用する
  - SPIFFE（Secure Production Identity Framework For Everyone）はサービスアイデンティティの標準を提供する
  - SPIRE（SPIFFE Runtime Environment）は短命の暗号アイデンティティ（SVID）を発行・ローテーションする

### 入力とデータのサニタイズ
- すべてのトラスト境界においてアローリストを使用してすべての入力を検証する
- 正規化（canonicalization）によってファイルパスをサニタイズする
- データベース操作にはパラメータ化クエリを使用する
- コンテキストアウェアな出力エンコーディングを適用する（SQL、シェル、HTML）
- ツール出力をサニタイズする: 最小限のフィールドのみ返し、すべてのPIIとセンシティブデータを秘匿する
- すべての入力、ツールスキーマ、メタデータ、プロンプト、リソースコンテンツを信頼できない入力として扱う
- プロンプトインジェクション検出システムをデプロイする
- 命令とデータの境界を維持するために厳格なJSONスキーマを使用する

### サンドボックスと分離
- MCPサーバーは最小権限で実行されるよう設計する
- ホスト環境（ファイル、コマンド、ネットワーク）と対話するMCPサーバーは必ずサンドボックスコントロールを実装する
- LLMが生成したコードは絶対にフルユーザー権限で実行してはならない
- 追加のサンドボックス層を実装する: gVisor、Kata Containers、SELinuxサンドボックス

### リソースの暗号検証
- すべてのサーバーコードに暗号署名とSBOMを提供する
- サーバーをロードする前にMCPクライアントで署名検証を実装する
- すべての転送中データにTLSを使用する
- サーバーが期待通りのコードを実行していることを確認するためのリモートアテステーション機能を実装する

### トランスポート層セキュリティ

#### stdioトランスポート（ローカルサーバー）
- DNSリバインディングリスクを排除するためにローカルMCPに強く推奨する
- ダイレクトパイプベースのストリーム通信
- 権限昇格を防ぐためのサンドボックスを実装する

#### HTTPストリーミングトランスポート（リモートサーバー）
実装が必要なセキュリティコントロール:
- ペイロード制限（大容量ペイロードおよび再帰ペイロードDoSの防止）
- ツール呼び出しとトランスポートリクエストのレート制限
- クライアント・サーバー認証/認可
- 相互TLS認証
- TLS暗号化
- CORS保護
- CSRF保護
- 整合性チェック（リプレイ、スプーフィング、汚染されたレスポンスの防止）

### 安全なツールとUXの設計
- 明確な境界を持つ単一目的ツールを作成する。「何でもできる」ツールは避ける
- 検証や認可の決定をLLMに依存しない
- 影響の大きいアクションには二段階コミットを使用する: まずドラフト/プレビュー、次に確認を伴う明示的なコミット
- ロールバック/アンドゥパスを提供する（ドラフトID、スナップショット、可逆的なアクション）。可能であれば時間制限付きコミットを使用する

### ヒューマン・イン・ザ・ループ
- MCPサーバーでリスクのある操作に確認プロンプトを実装する
- MCPサーバー側でelicitationを使用して、リスクのあるアクションに対するユーザー確認を要求する
- セキュリティに関連するメッセージは必ず影響を明確に示す
- 人間の承認のみに依存してはならない（ユーザーは疲労する可能性がある）

### ロギングとオブザーバビリティ
- MCPサーバーとクライアントでロギングを実装する
- ログに記録するもの: 使用されたツール、パラメータ、発生元のプロンプト
- アクションのエンドツーエンドのリンク可能性のためにOpenTelemetryを使用する
- アクションと認可の不変のレコードを維持する

## デプロイメントパターンのセキュリティ

### オールローカル（stdioまたはhttp）
- セキュリティはホストシステムのポスチャーに完全に依存する
- DNSリバインディングリスクを避けるために`stdio`トランスポートを使用する
- 権限昇格攻撃を制限するためにサンドボックスを使用する
- 開発および個人使用に適している

### シングルテナントリモート（http）
- クライアントとサーバー間の認証は必須である
- セキュアな認証情報ストレージを使用する（OSキーチェーン、シークレットマネージャー）
- 通信は必ず認証および暗号化されていなければならない
- エンタープライズクライアントは明示的なアローリストを持つ認証済みサーバーディスカバリーを強制すべきである

### マルチテナントリモート（http）
- 堅牢なテナント分離、アイデンティティ、アクセス制御を要求する
- 強力なマルチテナンシーコントロールを実装する（テナントごとの暗号化、ロールベースアクセス制御）
- サービスプロバイダーが直接ホストするMCPサーバーを優先する
- 可能であればリモートアテステーションを提供する
