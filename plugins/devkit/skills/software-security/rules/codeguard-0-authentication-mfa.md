## 認証 & MFA

認証情報への攻撃に耐性を持ち、機密情報を保護し、フィッシング耐性のある強力なMFAと安全なリカバリーをサポートする、弾力性と使いやすさを兼ね備えた認証システムを構築する。

### アカウント識別子とUX
- 非公開かつランダムで一意の内部ユーザー識別子を使用する。ログインには確認済みメールアドレスまたはユーザー名を許可する。
- 常に汎用的なエラーメッセージ（例: 「ユーザー名またはパスワードが無効です」）を返す。アカウント列挙を防ぐためタイミングを一定に保つ。
- パスワードマネージャーをサポートする: `<input type="password">`、ペーストを許可し、JSによるブロックは設けない。

### パスワードポリシー
- パスフレーズとフル Unicode を受け付ける。最低8文字。文字種の組み合わせルールは避ける。合理的な最大長（64文字以上）のみを設定する。
- 新しいパスワードは漏洩コーパス（例: k‑anonymity API）に照合し、漏洩済み・一般的なパスワードは拒否する。

### パスワード保管（ハッシュ化）
- 暗号化ではなくハッシュ化する。ユーザーごとの一意のソルトと定数時間比較を伴う、低速でメモリ負荷の高いアルゴリズムを使用する。
- 推奨順序とパラメータ（ハードウェアに合わせてチューニングし、サーバー上で1秒未満を目標とする）:
  - Argon2id: m=19–46 MiB, t=2–1, p=1（または同等のセキュリティトレードオフ）
  - scrypt: N=2^17, r=8, p=1（または同等）
  - bcrypt（レガシーのみ）: cost ≥10。72バイト入力制限に注意
  - PBKDF2（FIPS）: PBKDF2‑HMAC‑SHA‑256 ≥600k、または SHA‑1 ≥1.3M
- ペッパー（任意）: DB外部（KMS/HSM）に保管する。使用する場合はHMAC前適用またはプレハッシュで適用する。ペッパーをローテーションする場合はユーザーリセットの計画を立てる。
- Unicodeとnullバイトはライブラリがエンドツーエンドでサポートしていなければならない。

### 認証フローの堅牢化
- すべての認証エンドポイントとトークン転送においてTLSを強制し、HSTSを有効化する。
- IP・アカウント・グローバルごとのレート制限を実装する。プルーフオブワークやCAPTCHAは最終手段としてのみ追加する。
- ロックアウト/スロットリング: 段階的バックオフ。リセット/アラートによる恒久的ロックアウトは避ける。
- オラクル/タイミングシグナルを低減するため、均一なレスポンスとコードパスを使用する。

### 多要素認証（MFA）
- センシティブなアカウントには、デフォルトでフィッシング耐性のある要素を採用する: パスキー/WebAuthn（FIDO2）またはハードウェアU2F。
- 許容できる要素: TOTP（アプリベース）、PINを伴うスマートカード。センシティブな用途では避けるべき要素: SMS/音声、メールコード。セキュリティ質問には絶対に依存してはならない。
- 次の場面でMFAを要求する: ログイン、パスワード/メール変更、MFA無効化、権限昇格、高価値トランザクション、新規デバイス/ロケーション。
- リスクベースMFAシグナル: 新規デバイス、地理的速度、IPレピュテーション、異常な時刻、漏洩した認証情報。
- MFAリカバリー: シングルユース バックアップコードを提供し、複数の要素を推奨し、リセット時には強力な本人確認を要求する。
- MFA失敗時の対応: 登録済みの代替方法を提供し、失敗をユーザーに通知し、コンテキストをログに記録する（機密情報は含めない）。

### フェデレーションとプロトコル（OAuth 2.0 / OIDC / SAML）
- 標準プロトコルのみを使用する。独自実装は絶対に行わない。
- OAuth 2.0/OIDC:
  - パブリック/ネイティブアプリにはPKCEを伴うAuthorization Codeを優先する。ImplicitとROPCは避ける。
  - stateとnonceを検証する。リダイレクトURIは完全一致を使用する。オープンリダイレクトを防止する。
  - トークンはaudience/scopeに制約する。可能であればDPoP またはmTLSで送信者制約を行う。
  - リフレッシュトークンをローテーションする。ログアウト時またはリスクシグナル検出時に失効させる。
- SAML:
  - TLS 1.2以上。レスポンス/アサーションに署名する。センシティブなアサーションは暗号化する。
  - 発行者、InResponseTo、タイムスタンプ（NotBefore/NotOnOrAfter）、Recipientを検証する。信頼済みキーに対して検証する。
  - 厳格なスキーマ検証と堅牢なXPath選択によりXML署名ラッピングを防止する。
  - レスポンスの有効期間を短く保つ。SP起点フローを優先する。RelayStateを検証する。リプレイ検出を実装する。

### トークン（JWTと不透明トークン）
- シンプルさと失効のために、不透明なサーバー管理トークンを優先する。JWTを使用する場合:
  - アルゴリズムを明示的にピン留めする。「none」を拒否する。iss/aud/exp/iat/nbfを検証する。短い有効期間とローテーションを使用する。
  - 機密情報/キーは安全に保管する（KMS/HSM）。強力なHMAC機密情報または非対称キーを使用する。絶対にハードコードしてはならない。
  - リプレイを低減するために、トークンをクライアントコンテキスト（例: Cookieにフィンガープリントハッシュ）にバインドすることを検討する。
  - ログアウト時および重大なイベント時の失効のために、拒否リスト/許可リストを実装する。

### リカバリーとリセット
- 既存アカウントと存在しないアカウントに同じレスポンスを返す（列挙不可）。タイミングを均一化する。
- 32バイト以上のCSPRNGトークンを生成する。シングルユース。ハッシュとして保管する。有効期間を短くする。
- ピン留めされた信頼済みドメインへのHTTPSリセットリンクを使用する。UIにreferrerポリシー（no-referrer）を追加する。
- リセット後: 再認証を要求し、セッションをローテーションし、自動ログインは行わない。
- リセット試行によるアカウントロックアウトは絶対に行わない。代わりにレート制限と監視を行う。

### 管理者アカウントと内部アカウント
- 管理者ログインはパブリックフォームから分離する。より強力なMFA、デバイスポスチャチェック、IPアローリスト、ステップアップ認証を強制する。
- 管理者操作には、別個のセッションコンテキストとより厳格なタイムアウトを使用する。

### モニタリングとシグナル
- 安定したフィールドと相関IDを用いて認証イベント（失敗/成功、MFA登録/検証、リセット、ロックアウト）をログに記録する。機密情報やrawトークンは絶対にログに残さない。
- クレデンシャルスタッフィングを検出する: 高い失敗率、多数のIP/エージェント、不可能な移動。新規デバイスログインをユーザーに通知する。

### 実装チェックリスト
- パスワード: Argon2id（推奨）とユーザーごとのソルト、定数時間検証。変更/設定時に漏洩パスワードチェックを実施する。
- MFA: 高リスク用にWebAuthn/パスキーまたはハードウェアトークン。フォールバックとしてTOTP。バックアップコードによる安全なリカバリー。
- フェデレーション: Authorization Code + PKCE。厳格なリダイレクトURI検証。audience/scopeを強制する。トークンローテーションを実施する。
- トークン: 短命、可能であれば送信者制約。失効を実装する。機密情報はKMS/HSMに保管する。
- リカバリー: シングルユース・ハッシュ化・時間制限付きトークン。一貫したレスポンス。リセット後に再認証を要求する。セッションをローテーションする。
- 不正使用対策: 認証エンドポイントにレート制限、スロットリング、異常検知を実施する。均一なエラーハンドリングを行う。
- 管理者: より厳格なポリシーとデバイスチェックを持つ独立したフローを使用する。

### テスト計画
- ログイン、MFA登録/検証、リセット、ロックアウトに対する均一エラーのユニット/インテグレーションテストを実施する。
- プロトコルテスト: PKCE、state/nonce、リダイレクトURI検証、トークンaudience/scope。
- クレデンシャルスタッフィング耐性とトークンリプレイの動的テスト。ログアウトおよびロール変更後の失効を検証する。
