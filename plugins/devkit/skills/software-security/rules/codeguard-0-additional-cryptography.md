## 追加暗号化と TLS

保存データと転送データに対して、最新の実証済み暗号化を適用する。鍵を安全に管理し、TLS を正しく設定し、HSTS を展開し、適切な場合にのみピン留めを検討する。

### アルゴリズムとモード
- 対称：AES‑GCM または ChaCha20‑Poly1305 を優先する。ECB を避ける。CBC/CTR は encrypt‑then‑MAC と組み合わせた場合のみ使用する。
- 非対称：RSA ≥2048 または最新の ECC（Curve25519/Ed25519）。RSA 暗号化には OAEP を使用する。
- ハッシュ：整合性には SHA‑256 以上を使用。MD5/SHA‑1 を避ける。
- 乱数生成：プラットフォームに適した CSPRNG を使用する（例：SecureRandom、crypto.randomBytes、secrets モジュール）。非暗号 RNG を絶対に使用しない。

### 鍵管理
- 検証済みモジュール（HSM/KMS）内で鍵を生成し、パスワードや予測可能な入力から生成することは絶対にしない。
- 目的ごとに鍵を分離する（暗号化、署名、ラッピング）。侵害、暗号期間、またはポリシーに基づいてローテーションする。
- KMS/HSM またはボールトに鍵を保管する。ハードコードせず、平文の環境変数を避ける。KEK を使用して DEK をラップし、別に保管する。
- トラストストアへのアクセスを制御し、更新を検証し、すべての鍵アクセスと操作を監査する。

### 保存データ
- 機密データを暗号化し、保管する機密情報を最小化し、可能な場合はトークン化する。
- 認証付き暗号化を使用し、nonce/IV を適切に管理し、アイテムごとに一意のソルトを保持する。
- バックアップを保護する：暗号化し、アクセスを制限し、リストアをテストし、保存期間を管理する。

### TLS 設定
- プロトコル：TLS 1.3 を優先。レガシー互換性のためのみ TLS 1.2 を許可。TLS 1.0/1.1 と SSL を無効化する。TLS_FALLBACK_SCSV を有効化する。
- 暗号スイート：AEAD スイートを優先。NULL/EXPORT/anon を無効化する。ライブラリを常に更新し、圧縮を無効化する。
- 鍵交換グループ：x25519/secp256r1 を優先。必要に応じて安全な FFDHE グループを設定する。
- 証明書：2048 ビット以上の鍵、SHA‑256、正しい CN/SAN。ライフサイクルと失効を管理する（OCSP ステープリング）。
- アプリケーション：サイト全体で HTTPS を使用。HTTP を HTTPS にリダイレクトする。混在コンテンツを防止する。Cookie に `Secure` を設定する。

### HSTS
- HTTPS 経由のみで Strict‑Transport‑Security を送信する。段階的なロールアウト：
  - テスト：includeSubDomains 付きの短い max‑age（例：86400）
  - 本番：max‑age ≥ 1 年。安全な場合は includeSubDomains
  - 成熟後はオプションの preload を検討。永続性とサブドメインへの影響を理解すること

### ピン留め
- ブラウザの HPKP を避ける。制御されたクライアント（例：モバイル）でかつ両エンドを所有している場合のみピン留めを検討する。
- バックアップピン付きの SPKI ピン留めを優先し、安全な更新チャンネルを計画する。ユーザーによるバイパスを絶対に許可しない。
- ローテーションと障害処理を十分にテストし、運用リスクを理解する。

### 実装チェックリスト
- あらゆる場所で AEAD を使用。実証済みライブラリのみ使用。カスタム暗号実装はしない。
- KMS/HSM で鍵を生成・保管。目的ごとにスコープを限定。ローテーション手順を文書化。
- 強力な暗号スイートで TLS 1.3/1.2 を使用。圧縮オフ。OCSP ステープリングオン。
- 段階的計画に従って HSTS を展開。混在コンテンツを排除済み。
- ピン留めは正当化される場合のみ使用し、バックアップと更新パスを確保。

### テスト計画
- プロトコル・暗号スイート・HSTS の自動設定スキャン（例：SSL Labs、testssl.sh）。
- 暗号 API の誤用のコードレビュー。鍵ローテーション・バックアップ・リストアのテスト。
- 展開している場合はローテーション・障害のピン留めシミュレーション。
