## API・Webサービス セキュリティ

REST、GraphQL、SOAP/WSサービスをエンドツーエンドで保護する: 通信、認証/認可、スキーマ検証、SSRF制御、DoS制限、マイクロサービス安全パターン。

### 通信とTLS
- HTTPSのみを使用する。高価値サービスや内部サービスにはmTLSを検討する。証明書（CN/SAN、失効確認）を検証し、混在コンテンツを防止する。

### 認証とトークン
- クライアントには標準フロー（OAuth2/OIDC）を使用し、独自スキームは避ける。サービス間にはmTLSまたは署名付きサービストークンを使用する。
- JWT: アルゴリズムを固定し、iss/aud/exp/nbfを検証し、短い有効期限・ローテーション・ログアウト/失効時のdenylistを設定する。失効が必要で中央ストアが利用可能な場合は不透明トークンを優先する。
- APIキー: スコープを最小化し、レート制限を設け、使用状況を監視し、機密性の高い操作には単独で使用しない。

### 認可
- エンドポイントごと・リソースごとにサーバーサイドで確認を強制し、デフォルト拒否とする。
- マイクロサービスでは、ゲートウェイ（粗粒度）とサービス（細粒度）の両レイヤーで認可を行い、外部トークンではなく署名付き内部IDを伝搬する。

### 入力とコンテンツの処理
- OpenAPI/JSON Schema、GraphQL SDL、XSDを使用してコントラクトで入力を検証する。不明なフィールドや過大なペイロードは拒否し、サイズ上限を設定する。
- コンテンツタイプ: 明示的なContent‑Type/Acceptを強制し、未対応の組み合わせは拒否する。XXE/エンティティ展開に対してXMLパーサーを堅牢化する。

### リゾルバーとハンドラーにおけるSQL/インジェクション安全性
- パラメーター化クエリ/ORMバインドパラメーターを使用し、ユーザー入力をクエリやコマンドに絶対に連結してはならない。

### GraphQL 固有の制御
- クエリの深さと全体的な複雑度を制限し、ページネーションを強制し、実行タイムアウトを設定し、本番環境ではイントロスペクションとIDEを無効化する。
- IDOR/BOLAを防ぐためにフィールド/オブジェクトレベルの認可を実装し、バッチ処理を検証してオブジェクトタイプごとにレート制限する。

### アウトバウンド呼び出しのSSRF防止
- 生のURLを受け入れない。ライブラリを使用してドメイン/IPを検証し、HTTP/HTTPSのみに制限する（`file://`、`gopher://`、`ftp://`等はブロック）。
- ケース1（固定パートナー）: 厳格な許可リスト、リダイレクト無効化、ネットワーク出口許可リストを使用する。
- ケース2（任意）: プライベート/リンクローカル/localhostレンジをブロックし、すべてのIPがパブリックであることを解決・検証し、可能であればターゲットからの署名付きトークンを要求する。

### SOAP/WSとXML安全性
- XSDでSOAPペイロードを検証し、メッセージサイズを制限し、必要に応じてXML署名/暗号化を有効化する。
- XXE、エンティティ展開、再帰的ペイロードに対してパーサーを設定し、添付ファイルをスキャンする。

### レート制限とDoS対策
- IP/ユーザー/クライアントごとの制限、サーキットブレーカー、タイムアウトを適用する。サーバーサイドのバッチ処理とキャッシュを使用して負荷を軽減する。

### 管理エンドポイント
- インターネットに公開しない。強固な認証（MFA）、ネットワーク制限、専用ポート/ホストを要求する。

### テストとアセスメント
- 正式なAPI定義を維持し、仕様からコントラクトテストとファジングを実施する。
- 認証/認可バイパス、SSRF、インジェクション、情報漏洩についてエンドポイントを評価し、トークン検証の失敗をログに記録する。

### マイクロサービスのプラクティス
- 埋め込み判定ポイントを持つポリシーアズコードを使用し、サイドカーまたはライブラリPDPを採用する。
- mTLSまたは署名付きトークンによるサービスIDを使用し、外部トークンを内部で絶対に再利用してはならない。
- 相関IDを用いた集中型の構造化ロギングを行い、機密データをサニタイズする。

### 実装チェックリスト
- HTTPS/mTLSを設定し、証明書を管理し、混在コンテンツがないことを確認する。
- エッジとサービスでコントラクト検証を行い、不明なフィールドを拒否し、サイズ/時間制限を強制する。
- エンドポイントごとに強固な認証/認可を行い、GraphQLの制限を適用し、本番環境でイントロスペクションを無効化する。
- アプリとネットワークレイヤーにSSRF保護を施し、リダイレクトを無効化し、可能な限り許可リストを使用する。
- レート制限、サーキットブレーカー、回復力パターンを実装する。
- 管理エンドポイントを分離し、強固な認証を要求する。
- ログを構造化してプライバシーに配慮し、相関IDを付与する。

### テスト計画
- スキーマ準拠のコントラクトテストと、スキーマ対応ツールによるファジングを実施する。
- SSRF、IDOR/BOLA、認可バイパスのペンテスト、DoS制限のパフォーマンステストを実施する。
- エンドポイントごとにすべてのHTTPメソッドをテストし、クエリ文字列以外のURLパス、ヘッダー、構造化データのパラメーターも探索する。
- トークン検証と失効動作の自動確認を行う。
