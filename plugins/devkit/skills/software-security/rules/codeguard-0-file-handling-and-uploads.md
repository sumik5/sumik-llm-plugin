## ファイルアップロードセキュリティガイドライン

このルールは、悪意あるファイル攻撃を防ぎシステムの整合性を保護するためのセキュアなファイルアップロードのベストプラクティスを提供します。

- 拡張子の検証
  - ビジネス上重要な機能のみを対象に、許可する拡張子をリストアップしてください。
  - 拡張子を検証する前に入力検証が適用されることを確認してください。
  - 二重拡張子（例：`.jpg.php`）やヌルバイトインジェクション（例：`.php%00.jpg`）を避けてください。
  - ファイル拡張子には拒否リストではなく許可リストアプローチを使用してください。
  - バイパスの試みを防ぐため、ファイル名をデコードした後に拡張子を検証してください。

- コンテンツタイプとファイルシグネチャの検証
  - クライアントが送信した Content-Type ヘッダーはスプーフィングされる可能性があるため、絶対に信頼しないでください。
  - Content-Type チェックと組み合わせて、ファイルシグネチャ（マジックナンバー）を検証してください。
  - 素早い保護レイヤーとして MIME タイプの許可リストアプローチを実装してください。
  - ファイルシグネチャ検証を使用しますが、単独のセキュリティ手段としては使わないでください。

- ファイル名のセキュリティ
  - ユーザーが提供した名前の代わりに、ランダムなファイル名（UUID/GUID）を生成してください。
  - ユーザーのファイル名が必要な場合は、最大長の制限を実装してください。
  - 文字は英数字・ハイフン・スペース・ピリオドのみに制限してください。
  - 先頭のピリオド（隠しファイル）や連続したピリオド（ディレクトリトラバーサル）を防いでください。
  - シェルスクリプト処理を安全にするため、先頭のハイフンやスペースを避けてください。

- ファイルコンテンツの検証
  - 画像の場合、悪意あるコンテンツを破壊するために画像書き換えテクニックを適用してください。
  - Microsoft ドキュメントの場合、検証に Apache POI を使用してください。
  - 多数の攻撃ベクターがあるため、ZIP ファイルは避けてください。
  - リソースが許す場合、サンドボックス環境での手動ファイルレビューを実装してください。
  - 対象ファイルタイプにはアンチウイルススキャンと Content Disarm & Reconstruct（CDR）を統合してください。

- ストレージセキュリティ
  - 可能な場合、完全な分離のためにファイルを別のサーバーに保存してください。
  - ファイルは管理アクセスのみとし、web ルートの外部に保存してください。
  - web ルート内に保存する場合は、適切なアクセス制御を伴う書き込み専用権限を設定してください。
  - パブリックアクセスには、ID をファイル名にマッピングするアプリケーションハンドラーを使用してください。
  - DBA の専門知識を伴う特定のユースケースでは、データベースストレージを検討してください。

- アクセス制御と認証
  - ファイルアップロードを許可する前にユーザー認証を要求してください。
  - ファイルアクセスと変更に対して適切な認可レベルを実装してください。
  - 最小権限の原則でファイルシステム権限を設定してください。
  - 実行権限が必要な場合は、実行前にファイルをスキャンしてください。

- アップロードとダウンロードの制限
  - アップロード保護のために適切なファイルサイズ制限を設定してください。
  - 圧縮ファイルには展開後のサイズ制限を検討してください。
  - DoS 攻撃を防ぐため、ダウンロードサービスへのリクエスト制限を実装してください。
  - ZIP ファイルのサイズを安全に計算するためのセキュアな方法を使用してください。

- 追加のセキュリティ対策
  - ファイルアップロードエンドポイントを CSRF 攻撃から保護してください。
  - すべてのファイル処理ライブラリをセキュアな設定に保ち、常に更新してください。
  - アップロードアクティビティのロギングと監視を実装してください。
  - 違法なコンテンツに対するユーザーの報告メカニズムを提供してください。
  - 圧縮ファイルにはセキュアな展開方法を使用してください。

まとめ：  
多層的な検証・セキュアなストレージのベストプラクティス・適切なアクセス制御・包括的な監視によって、ファイルアップロードに対する多層防御を実装してください。単一の検証方法に頼らず、攻撃を防ぐために常に安全なファイル名を生成してください。
