# 認証情報のハードコード禁止

機密情報、パスワード、APIキー、トークン、その他いかなる認証情報もソースコードに直接保存してはならない。

コードベースは公開されており信頼できないものとして扱うこと。ソースコードに現れた認証情報は漏洩したものとみなし、セキュアな代替手段で管理しなければならない。

#### 絶対にハードコードしてはならない値の種類:

パスワードと認証情報:
- データベースパスワード、ユーザーパスワード、管理者パスワード
- APIキー、シークレットキー、アクセストークン、リフレッシュトークン
- 秘密鍵、証明書、署名キー
- 認証情報を含む接続文字列
- OAuthクライアントシークレット、Webhookシークレット
- 外部サービスへのアクセスに使用できるその他の認証情報


#### 認識パターン - これらのフォーマットを見抜く

絶対にハードコードしてはならない一般的な機密情報フォーマット:

- AWS Keys: `AKIA`、`AGPA`、`AIDA`、`AROA`、`AIPA`、`ANPA`、`ANVA`、`ASIA`で始まる
- Stripe Keys: `sk_live_`、`pk_live_`、`sk_test_`、`pk_test_`で始まる
- Google API: `AIza`の後に35文字が続く
- GitHub Tokens: `ghp_`、`gho_`、`ghu_`、`ghs_`、`ghr_`で始まる
- JWT Tokens: ドットで区切られた3つのbase64セクション、`eyJ`で始まる
- 秘密鍵ブロック: `-----BEGIN`から`-----END PRIVATE KEY-----`の間のテキスト
- 接続文字列: `mongodb://user:pass@host`のような認証情報を含むURL

コード中の警告サイン:
- `password`、`secret`、`key`、`token`、`auth`を含む変数名
- 何であるか不明な長くランダムに見える文字列
- 認証コードの近くにあるBase64エンコードされた文字列
- 外部サービスへのアクセスを許可するあらゆる文字列

このルールがどのように適用されたか、またなぜ適用されたかを常に説明しなければならない。
