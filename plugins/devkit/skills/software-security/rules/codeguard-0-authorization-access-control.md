## 認可 & アクセス制御

すべてのリクエストとリソースに対して最小権限と正確なアクセス決定を強制し、IDORとマスアサインメントを防ぎ、必要に応じて強力なトランザクション認可を提供する。

### コア原則
1.  デフォルト拒否: アクセスリクエストのデフォルトは「拒否」であるべきである。明示的な拒否ではなく、ロールやユーザーに明示的に権限を付与する。一致するアローリストルールがない場合は、HTTP 403 Forbiddenを返す。
2.  最小権限の原則: 職務を遂行するために必要な最小限のアクセスレベルをユーザーに付与する。過剰でないことを確認するために権限を定期的に監査する。
3.  すべてのリクエストで権限を検証する: ソース（AJAX、API、直接）に関わらず、すべての単一リクエストに対して認可を確認する。一貫した適用を確保するためにミドルウェア/フィルターを使用する。
4.  RBAC よりもABAC/ReBACを優先する: シンプルなロールベースアクセス制御の代わりに、きめ細かな権限のためにABAC（属性ベースアクセス制御）またはReBAC（関係ベースアクセス制御）を使用する。

### システム的コントロール
- ミドルウェア/ポリシー/フィルター経由でサービス境界における認可を一元化する。
- リソースレベル（所有権/テナンシー）で権限をモデル化し、データクエリでスコーピングを強制する。
- リソースの存在を漏洩しないよう、汎用的な403/404レスポンスを返す。
- ユーザー、アクション、リソース識別子（非PII）、理由コードを含むすべての拒否をログに記録する。

### IDORの防止
- ユーザー提供の識別子だけを信頼してはならない。常に各オブジェクトインスタンスへのアクセスを検証する。
- ユーザースコープのクエリまたはサーバーサイドルックアップでリソースを解決する。例: `Project.find(id)` の代わりに `currentUser.projects.find(id)` を使用する。
- 多層防御として非列挙可能な識別子（UUID/ランダム）を使用する。難読化のみに依存しない。

### マスアサインメントの防止
- センシティブなフィールドを含むドメインオブジェクトにリクエストボディを直接バインドしない。
- DTOを通じて安全で編集可能なフィールドのみを公開する。パッチ/更新のための明示的なアローリストを維持する。
- アローリストが実現不可能な場合は、フレームワーク機能を使用してセンシティブなフィールドをブロックリストに追加する。

### トランザクション認可（ステップアップ）
- センシティブなアクション（送金、権限昇格、データエクスポート）には第2要素を要求する。What‑You‑See‑Is‑What‑You‑Signを適用: ユーザー確認のために重要なフィールドを表示する。
- トランザクションごとに一意の時間制限付き認可認証情報を使用する。フロー途中でデータが変更された場合は拒否する。
- 選択した認可方法をサーバーサイドで強制する。クライアントサイドのダウングレードを防止する。
- スロットリングと失敗後の完全なフロー再起動でブルートフォースを防護する。

### テストと自動化
- エンドポイント/リソース、ロール/属性、期待される結果を列挙した認可マトリクス（YAML/JSON）を維持する。
- マトリクスを反復し、ロールトークンを発行し、トークン有効期限/失効ケースを含む許可/拒否結果をアサートするインテグレーションテストを自動化する。
- ネガティブテストを実施する: IDの入れ替え、ロールのダウングレード、スコープの欠如、バイパス試行。

### 実装チェックリスト
- ミドルウェア/ポリシーがすべてのエンドポイントにデフォルト拒否とリソースチェックを強制する。
- クエリスコーピングにより、ユーザーが許可された行/オブジェクトにのみアクセスできることを確認する。
- DTOとアローリストがマスアサインメントを防止する。センシティブなフィールドは絶対にバインド不可にする。
- センシティブな操作に一意の短命の認証情報を伴うステップアップ認可を配置する。
- 認可マトリクスがCIテストを駆動し、失敗時はマージをブロックする。
