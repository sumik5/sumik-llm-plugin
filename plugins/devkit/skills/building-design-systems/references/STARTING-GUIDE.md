# デザインシステム立ち上げ・浸透ガイド

DSを「小さく始めて」組織全体に浸透させるための実践ガイド。
立ち上げから運用・ツール選定まで、プロダクトチームが直面する実践的な課題への処方箋。

> **理論・フレームワーク・Figma実装は INSTRUCTIONS.md 参照。本ファイルは実践プロセスに特化。**

---

## 1. DSの3原則

デザインシステムが「システム」として機能するための共通原則。

### 一貫性（Consistency）

目的に対して力を発揮するために、情報に一貫性を持たせる。

- 一貫性があると、プロダクトが作りやすくなるだけでなく、レビューや評価もしやすくなる
- 組織やプロダクトを取り巻く環境の変化があったときにも適応しやすい
- 各自がバラバラなベクトルで動くと全体の力は弱まる → チームが同じ方向を向くことで最大化

### 一元化（Single Source of Truth）

ブレないために、情報を一元化する。

- 各チームが善意で独自の改善を始めると、知見が分散し車輪の再発明が繰り返される
- 一元化されているからこそ、個々のチームが他への影響を気にせず自分の仕事に集中できる
- 一元化された場所に還元することが、組織全体の推進力につながる

### 循環（Circulation）

システムは「利用→還元」のサイクルで生き続ける。

| 役割 | 説明 |
|------|------|
| **仲介者（頒布）** | DSの内容をチームに届ける。窓口・スポークスパーソン。専任でなくても必須 |
| **貢献者（還元）** | 現場の知見をDSに反映する。「使うこと」も立派な貢献 |

- DSは一度公開して完成ではない → 常に生きるドキュメントとして変化し続ける
- 循環サイクルを意識的に作ることが、システムとしての機能維持に不可欠

## 2. 立ち上げ3ステップ

「どこから着手するか」という壁を乗り越えるプロセス。

### STEP1: 便利なコンテンツから作る

「完璧なDSを作る」より「まず使われている状態を実現する」が先。

**「便利なコンテンツ」の3条件:**

| 条件 | 説明 |
|------|------|
| 最初の利用者が実際に使いたいと思える | ターゲットユーザーの具体的なニーズを満たす |
| 導入・学習コストが低い | ドキュメント1枚で理解できる水準 |
| 使うと成果が見えやすい | 開発速度向上など、効果を数値・体感で確認できる |

**やってはいけないこと:**
- ❌ 先にツールを導入してDSの全体像（目次）を描く
- ❌ 多様なコンテンツを先に揃える
- ❌ 最初から立派なシステムを目指してコストをかける

**コンポーネントライブラリを最初のコンテンツとした場合の効果例:**
- ウェブ標準を基盤にするため制作コストを抑えられる
- 開発速度向上という成果が可視化しやすい
- エンジニア・デザイナー両者にとって利用価値が明確

> **AskUserQuestion ─ 出発点の判断:**
> ```
> 1. 最初のターゲット利用者は誰ですか？（エンジニア・デザイナー・非技術職など）
> 2. 今最も繰り返し発生している「車輪の再発明」は何ですか？
> 3. 「使ったら成果がすぐ見える」コンテンツとして何が思い浮かびますか？
> ```

---

### STEP2: 課題発見と課題解決のサイクルを繰り返す

最初のコンテンツが使われ始めたら、課題ドリブンで拡充していく。

**サイクルの実践:**

```
現場のリスク・課題を調べる
    ↓
課題を1箇所に書き留める（解決策の目星もメモ）
    ↓
優先度の高い課題から解決策（コンテンツ）を作る
    ↓
公開・頒布 → また現場の課題を調べる（繰り返し）
```

**WIPの精神（Work In Progress）:**

| 状況 | 判断 |
|------|------|
| 優先度の高い課題が解決できた | 次の優先度の高い課題に移る |
| 一般的なDSと比べて足りない要素がある | 今は追加しない（必要になったときに対処） |
| 他社DSと自分たちのDSを比較したくなる | 他社はあくまで参考。自組織の課題解決が最優先 |

**重要な考え方:** 課題を集め、解決策を言語化し、1箇所に残し続けることが「システム」の本質。課題解決のプロセスそのものがデザインシステムである。

---

### STEP3: 原則を言語化する

STEP2のサイクルが安定してから、背景にある「なぜ」を言語化する。

**原則をSTEP3にする理由:**

| 理由 | 説明 |
|------|------|
| 十分な時間の確保 | 原則の言語化には集中した時間が必要。STEP1・2と並行すると中途半端になる |
| 骨格・表層レベルの負債を防ぐ | 具体的なガイドラインを先に作ることで、抽象的な方針と実装のズレを防ぐ |
| 読まれる原則にするため | DSが使われている状態になってから公開する方が、読者が理解しやすい |

**原則言語化プロジェクトの進め方:**

1. **ゴール共有**: 関係者が「これは嫌だ」「これが良い」を対比でブレスト → ゴールの解像度を上げる
2. **少人数で進める**: 合議ではなく志ある少人数が意思決定 → 折衷案で目的が曖昧になることを防ぐ
3. **決め方を合意**: プロセスとメンバーを関係者全員で承認してから進める

**原則ドキュメントが目指す状態:**
- DSで定義されていないことを判断しなければいけない場面でも適切な判断ができる
- 誰が判断しても結論に大きなズレが生まれない

## 3. 浸透3ステップ

「どうやって組織に浸透させるか」という壁を乗り越えるプロセス。

### STEP1: 草の根活動

どんなに便利なものでも、知られていなければ使ってもらえない。

**草の根活動の具体例:**

| 活動 | 効果 |
|------|------|
| 全社ミーティング・定例での発表枠確保 | 注目度の高い場での認知獲得 |
| チャットツールのキーワード通知設定 | 話題になった瞬間にリアクション・案内が可能 |
| DS読書会・解説会の開催 | 理解度把握 + フィードバック収集の機会 |
| 社内報・社内ブログへの掲載 | ストック型コンテンツとして継続的な認知 |
| コンテンツ更新のたびに告知 | DSが生きているドキュメントであることの体感 |

**ポイント:** 最初のコンテンツ作成時だけでなく、更新・改善のたびに積極的に発信し続ける。

---

### STEP2: 課題発見に小さく協力してもらう

立ち上げのSTEP2（課題発見サイクル）は、関与者を増やす最大のチャンス。

**現場ヒアリングで聞くこと:**

```
・日々の業務で困っていることはあるか？
・DSを使ったことはあるか？使ってみてどうか？
・（未使用者に）そもそもDSを知っているか？
・既存コンテンツで改善してほしい点はあるか？
```

**ヒアリング後のフォローアップ:**

課題解決できたら → その人にお知らせに行く

```
困っていたことを伝えた → コンテンツが作られた → 他の人も便利になった
この体験がDSの意義理解を促し、積極的なフィードバックにつながる
```

---

### STEP3: コンテンツの作者になってもらう

課題発見に協力してくれる人を、次は課題解決のパートナーに誘う。

**段階的な巻き込み方:**

```
まず使う（最小の貢献）
    ↓
課題をフィードバックする
    ↓
小さな解決策を一緒に作る（WIPの精神で小さく）
    ↓
コンテンツの「作者」になる
    ↓
自然とDSへのアンテナが高まり、課題を見つけ続ける
```

**作者になることの効果:**
- 自分が作ったコンテンツが気になり始める（使われているか、改善点はないか）
- 無意識にDSへのアンテナが立ち、課題を見つけやすくなる
- その課題への解決サイクルが個人の中で自然に回り出す

**多職種の関与事例（汎用パターン）:**

| 職種 | 貢献しやすいコンテンツ領域 |
|------|------------------------|
| UXライター | ライティングガイドライン・用字用語・マイクロコピー |
| エンジニア | コンポーネント実装・技術ドキュメント・コードスニペット |
| プロダクトマネージャー | 機能要件・ユーザーストーリー・用語定義 |
| マーケター | トーン&マナー・ブランドガイドライン |

> **AskUserQuestion ─ 浸透段階の把握:**
> ```
> 1. 現在のDS認知度はどの程度ですか？（知っている人の割合）
> 2. 最もアクティブにフィードバックをくれる職種・チームはどこですか？
> 3. 「作者」になってもらえそうな候補者はいますか？
> ```

### 浸透の壁と対処法

| 壁 | 症状 | 対処法 |
|----|------|--------|
| 存在が知られていない | 「そんなものがあったの？」という反応 | 定期的な告知活動・ミーティング枠確保 |
| 使い方がわからない | 見つけてもどう使うか不明 | オンボーディング資料・ウォークスルー動画 |
| 自分には関係ないと思っている | 「デザイナーのものでしょ？」 | 非デザイナー向けのコンテンツ整備・多職種の成功事例共有 |
| 更新が止まって信頼を失う | 「古い情報ばかり」という印象 | WIPの精神で小さく更新し続ける仕組み |
| 貢献のハードルが高い | GitHubが使えない・Markdownが書けない | 非エンジニア向けのウェブUI編集、クラウドDB連携 |

## 4. チーム構成

### 3つのチームモデル

| モデル | 説明 | 向いている組織 |
|--------|------|--------------|
| **孤独型（Solitary）** | 特定プロダクト・チームの問題解決から自然発生 | 小規模・スタートアップ初期 |
| **中央集権型（Centralized）** | DSに特化した専任グループが要望を聞き実装・展開 | 大規模・ブランド統一が重要 |
| **連合型（Federated）** | プロダクトチーム代表者による合議で意思決定 | 高度な技術文化・スケーラビリティ重視 |

> **補足:** 3つのモデルを複合した「ハイブリッド型」も有効。最適解は組織の中で考え抜くしかない。

> **AskUserQuestion ─ チーム構成の選択:**
> ```
> 1. DS専任メンバーを確保できますか？それとも兼任での運用になりますか？
> 2. 組織の規模（デザイナー・エンジニア数）と今後の成長見込みは？
> 3. DSに対して強い関心・オーナーシップを持つ人はいますか？
> ```

### 兼任 vs 専任の判断基準

| 状況 | 推奨 |
|------|------|
| 小規模・立ち上げ期 | 兼任推奨 — 各チームでの認知向上と利用促進のサイクルが自然に回る |
| 組織規模が拡大・複数プロダクト展開 | 専任または準専任チームの設置を検討 |
| DSの内容が複雑化・運用コストが増大 | 専任エンジニアの配置が不可欠 |

**兼任のメリット:**
- 各チームでのDS認知が高まる
- 利用促進→結果反映のサイクルが自然に回る
- 個別のベストプラクティスが組織全体のプラクティスになる

### 「仲介者」と「貢献者」の役割定義

| 役割 | 主な活動 | 必要なスキル |
|------|---------|------------|
| **仲介者（スポークスパーソン）** | DSを届ける・展開する・コンテキストを共有する | コミュニケーション、ドメイン横断的な視点 |
| **貢献者（コントリビューター）** | 現場知見をDSに還元する・コンテンツを作る | 実務知識、ドキュメント作成能力 |

- 組織規模が大きくなるほど、仲介者の役割を**明示的に定める**必要がある
- 窓口になる人は専任でなくとも必須

## 5. コンテンツ戦略

### 4種のスタイルガイドの概要

| ガイドライン | 対象 | 主なコンテンツ例 |
|-------------|------|---------------|
| **ブランドガイドライン** | 企業・サービス全体 | ロゴ・カラー・タイポグラフィ・ビジョン・行動規範 |
| **UIガイドライン** | エンジニア・デザイナー | コンポーネント集・パターンライブラリ・コードスニペット |
| **コンテンツガイドライン** | 文章を書く全員 | ボイス&トーン・表記ルール・用字用語・ライティングパターン |
| **運用ガイドライン** | DS関係者全員 | 更新プロセス・教育コンテンツ・ガバナンス |

> **重要:** すべてのガイドラインを同時に作る必要はない。「便利なコンテンツから作る」原則に従い、最も利用者に近いものから着手する。

### 優先順位判断フロー

```
最初のターゲット利用者は誰か？
    ↓
エンジニア・デザイナーが主 → UIガイドラインから（コンポーネントライブラリ）
    ↓
ブランドを扱う全員が主 → ブランドガイドラインから（カラー・ロゴ）
    ↓
文章を書く全員が主 → コンテンツガイドラインから（表記ルール）
    ↓
運用の仕組みが整ってきたら → 運用ガイドラインを整備
```

> **詳細な策定プロセスは CONTENT-GUIDELINES.md 参照**（作成予定）

## 6. 運用・ツール選定

### 外部サービス vs フルスクラッチの判断基準

| 判断軸 | 外部サービス推奨 | フルスクラッチ推奨 |
|--------|--------------|-----------------|
| **立ち上げスピード** | 速く公開したい・リソースが少ない | 時間と人材がある |
| **コード連携** | デザインデータ・画像の共有が主 | 実装コードをDSに直接組み込みたい |
| **共同編集** | 小規模なチームで編集者が少ない | 複数職種が同時編集・PRレビューが必要 |
| **エンジニアリング文化** | デザイナー主導 | エンジニアリングへの理解がチームにある |
| **移行性** | — | コンテンツをエクスポート・移行しやすい形式を選ぶ |

> **AskUserQuestion ─ ツール選定:**
> ```
> 1. DSの主な運用者はデザイナーですか、エンジニアですか？
> 2. プロダクトのコードをDSに直接組み込んで表示したいですか？
> 3. 複数人が同時にドキュメントを編集する運用を想定していますか？
> ```

### 外部サービスの種類と特徴

| カテゴリ | 代表ツール | 特徴 |
|---------|---------|------|
| **DS専用プラットフォーム** | zeroheight, Frontify | デザインツール連携が容易。技術知識不要。コード組み込みは限定的 |
| **UIデザインツール** | Figma, Sketch | デザインデータとの連携が強力。デザイントークン管理も可能 |
| **コンポーネント管理** | Storybook | コードベース。エンジニア主導。メンテナンスコスト低 |
| **コラボレーションツール** | Notion, Miro | 手軽に始めやすい。DSテンプレートあり。技術的な制約が多い |

### フルスクラッチ構成の例（汎用Jamstackアーキテクチャ）

コードとドキュメントを一元管理したい場合の技術スタック例:

```
コンテンツ管理: Markdown / MDX + Gitリポジトリ
    ↓ PRレビュー・変更履歴管理
フレームワーク: 静的サイトジェネレーター（Gatsby / Next.js / Astro等）
    ↓ コンポーネントをドキュメントに直接組み込み可能
デプロイ: Netlify / Vercel（GitリポジトリとCI/CD連携）
    ↓ プルリクエスト単位でプレビュー・レビュー
```

**メリット:**
- プロダクトと同じレビューフロー（プルリクエスト）でドキュメント品質を保証
- Markdownで書ける文化があれば、エンジニア以外も参加しやすい
- ライティングガイドラインをデータベース（Airtable等）と連携して非エンジニアが更新可能

### CI/CD自動化

持続可能なDSには自動化が不可欠。段階的に整備する。

| 自動化項目 | ツール例 | 効果 |
|----------|---------|------|
| **コード規約チェック（linting）** | ESLint, Stylelint | 一貫したコード品質を維持 |
| **文章校正（textlint）** | textlint | 用字用語・表記ルールの自動チェック |
| **アクセシビリティテスト** | axe, Storybook a11y addon | WCAG準拠の継続的な確認 |
| **視覚的回帰テスト（VRT）** | Chromatic, Percy | コンポーネントの意図しないビジュアル変更を検知 |
| **クロスブラウザテスト** | Playwright, BrowserStack | 複数環境での表示確認 |

**導入の優先順位:**

```
1. linting（コード規約） → 最初から導入コスト低
2. 視覚的回帰テスト（VRT） → コンポーネントが増えてきたら
3. textlint → ライティングガイドラインのルールが増えてきたら
4. アクセシビリティテスト → a11y対応を本格化するタイミングで
```

**分割して考える:** 全自動化をひとつのリポジトリで行う必要はない。コンポーネントライブラリ・ドキュメントサイト・デザイントークンそれぞれで必要な自動化を小さく行う。

## 7. AskUserQuestion Pattern

### 立ち上げ初期状況の把握（4問）

```
1. DSは新規立ち上げですか、既存のものを改善・拡張しますか？
2. 最初にDSを使ってほしい職種・チームはどこですか？
3. 現在最も繰り返し発生している「車輪の再発明」や「不整合」は何ですか？
4. DSに取り組むことに賛同してくれるメンバーはいますか？
```

### チーム・ツール選定（3問）

```
5. DS専任メンバーを確保できますか？それとも兼任での運用になりますか？
6. DS運用の主導者はデザイナー寄りですか、エンジニア寄りですか？
7. プロダクトのコードをDSに直接組み込んで表示したいですか？
```

### 浸透フェーズの確認（2問）

```
8. 現在のDS認知度・利用率はどの程度ですか？
9. 貢献者になってもらえそうな候補者（非デザイナー職種も含めて）はいますか？
```

## 8. DS失敗パターンと対策（立ち上げ・浸透固有）

> **一般的なDS失敗パターンは INSTRUCTIONS.md の「DS失敗パターン」参照。**
> 本テーブルは立ち上げ・浸透フェーズ固有のパターンを補足する。

| 失敗パターン | 症状 | 対策 |
|------------|------|------|
| **完成してから公開症候群** | 「まだ足りない」と思い公開が遅れ続ける | STEP1の「便利なコンテンツから作る」を徹底。WIPの精神で小さく公開 |
| **ツール先行でコンテンツ後回し** | 素晴らしいシステムができたが中身がない | ツール選定前に「最初のコンテンツ」を決めてから環境を整える |
| **担当者の孤立** | 一人が頑張り続けて疲弊・退職でDS消滅 | STEP2・3で課題発見・作者化のサイクルを早期に回す |
| **課題なき浸透活動** | 告知はしているが誰も使わない | 現場に出向いて課題を聞き、それを解決するコンテンツを届ける |
| **貢献ハードルの設定ミス** | 技術知識がないと更新できない | 非エンジニアが更新できる仕組み（ウェブUI・クラウドDB連携）を整備 |
| **外部サービスへの過度な依存** | サービス廃止・仕様変更で移行コスト爆発 | 移行性（コンテンツのエクスポート可否）を選定基準に入れる |
| **原則の先行策定** | 抽象的な原則が先にあり、具体的な実装がバラバラになる | 原則の言語化はSTEP3（具体的なコンテンツが安定してから） |
