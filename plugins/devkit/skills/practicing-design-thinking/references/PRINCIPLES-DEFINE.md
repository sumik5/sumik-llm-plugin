# UX実践原則：Define（原則 38–51）

フェーズ「Define」はプロジェクトの問題と目標を正確に定義する段階。
正しいクライアント・要件定義・MVP・ユーザーフローの設計で方向性を確立する。

---

### 38. 正しいクライアントを選ぶ

**核心**: すべてのクライアントと仕事をするべきではない。UX品質を諦めさせる関係は双方を不幸にする。

- UXへの理解・尊重・予算が不十分なクライアントは、最終的に低品質な成果物を生む
- 「安くて速くて良いもの」は存在しない。3つのうち2つしか選べないことをキックオフで合意する
- 初期の違和感（コミュニケーション・意思決定スタイル）は後に増幅される

**If/Then**: If 新しいプロジェクトを受注するなら then UXプロセスへの理解と承認プロセスについて事前に合意を得る

---

### 39. 探偵になる（キックオフミーティング）

**核心**: キックオフは情報収集の最重要機会。表面的な要件の背後にある「なぜ」を掘り下げる。

- 「何を作りたいか」ではなく「誰のどんな問題を解決したいか」を問う
- ステークホルダーは異なる期待・優先度を持っている。全員を一堂に集めて齟齬を可視化する
- 「制約」（予算・技術・期限・コンプライアンス）をすべて洗い出す

**If/Then**: If キックオフミーティングを準備するなら then 「ビジネス目標」「ユーザー課題」「成功指標」「制約」の4テーマに絞った質問リストを用意する

---

### 40. 要件をまとめる（定性インタビュー）

**核心**: ユーザーインタビューから得た定性データを整理し、設計の根拠とする。

- アフィニティダイアグラムでインタビューデータをグルーピングしパターンを発見する
- 「ユーザーが言ったこと」と「ユーザーが意味したこと」は異なる場合がある
- 要件は「〜できる」（機能）だけでなく「〜と感じる」（体験品質）も含める

**If/Then**: If ユーザーインタビューを実施したなら then 24時間以内にメモを整理し、パターンと矛盾点を抽出する

---

### 41. プロブレムステートメントを書く

**核心**: 解決すべき問題を1文で明確に定義することで、チーム全体の方向性が揃う。

- フォーマット例：「[ユーザーセグメント]は[目標]を達成したいが、[障壁]があるため[問題が発生している]」
- 問題定義が曖昧なまま解決策を設計すると、方向性がずれ続ける
- プロブレムステートメントは解決策を含まない。「どうやって解決するか」は次のフェーズ

**If/Then**: If 設計を始める前に then チーム全員がサインオフできるプロブレムステートメントを1文で書き、合意を得る

---

### 42. 近道を探す（How Might We）

**核心**: 問題を「どうすれば〜できるか（HMW）」の形式に変換することで、創造的解決策のアイデアが生まれやすくなる。

- HMW質問は問題を解決可能な機会として再定義する
- 広すぎず（「どうすれば世界を変えられるか」）、狭すぎない（「どうすれば色を変えられるか」）範囲に設定する
- チームで複数のHMW質問を生成し、最も有望なものに投票する

**If/Then**: If プロブレムステートメントが完成したなら then それをHMW形式に変換し、5〜10個のバリエーションを作る

---

### 43. MVPから始める（最初から完璧を目指さない）

**核心**: 完璧な初版は存在しない。最小限の機能で価値を検証し、学びながら進化させる。

- MVPは「最低限度の機能セット」ではなく「最小限の投資で最大限の学びを得る製品」
- ユーザーが実際に使い始めると、想定外の使い方・課題が必ず発見される
- リリース後のデータが設計改善の最も信頼できる根拠

**If/Then**: If 機能リストを作るなら then 「これがなければ製品が成立しない」機能のみをV1に含め、残りをV2以降に分類する

---

### 44. 約束は控えめ、提供は期待以上（カノモデル）

**核心**: カノモデルでは製品品質を「当たり前品質」「一元的品質」「魅力的品質」に分類する。

- **当たり前品質**: 存在して当然。なければ不満（ページが表示されない等）
- **一元的品質**: 多いほど満足度が上がる（読み込み速度・機能数等）
- **魅力的品質**: 予期しない喜び。あると感動、なくても不満なし
- 魅力品質は時間が経つと当たり前品質に変化する（例：スマートフォンのカメラ）

**If/Then**: If 機能優先度を決めるなら then カノモデルで分類し、当たり前品質→一元的品質→魅力品質の順に投資する

---

### 45. 複雑さは必要なときだけ（オッカムの剃刀）

**核心**: 同等の結果を得られる解決策が複数あるなら、最もシンプルなものを選ぶ。

- 複雑な解決策はバグを生み、メンテナンスコストを増やし、ユーザーを混乱させる
- 「もし〜だったら便利かも」という仮定の要件を追加しない
- エンジニア・デザイナーともに「複雑な解決策を作る能力」より「シンプルな解決策を見つける能力」が価値を生む

**If/Then**: If 解決策を複数検討しているなら then 同等の価値を持つ最もシンプルな案を採用する

---

### 46. テスラーの法則（複雑さの保存）

**核心**: システムの複雑さの総量は一定。デザイナーが複雑さを引き受けることでユーザーが楽になる。

- フォームの住所自動入力・スマートデフォルト・自動補完はデザイン側が複雑さを吸収した例
- ユーザーに複雑さを押しつけるのは「設計の放棄」
- ただし、簡略化しすぎると制御が奪われ上級ユーザーが不満を持つ

**If/Then**: If ユーザーに手動入力を求めているなら then その入力をシステムが推測・自動化できないか検討する

---

### 47. ユーザージャーニーマップを作る

**核心**: ユーザーが製品を発見してから目標を達成するまでの全体フローを可視化し、感情の浮き沈みを特定する。

- 「認知→検討→購入→利用→推薦」などのフェーズに分けてタッチポイントを洗い出す
- 各タッチポイントでのユーザーの感情・疑問・不満を記録する
- 複数のチャネル（Web・アプリ・メール・カスタマーサポート等）を横断した体験を設計する

**If/Then**: If 新しいサービスを設計するなら then ユーザージャーニーマップを作成し、感情が最も低い箇所を優先改善ポイントとする

---

### 48. ユーザーフローを設計する

**核心**: 特定のタスクをユーザーが完了するまでの画面遷移を図示し、迷子になる分岐を排除する。

- 「最短ハッピーパス」（正常系）を先に設計し、次にエラー・例外を処理する
- 分岐が多すぎるフローは認知負荷が高い。統合・削除できる分岐を探す
- フローの各ステップで「ユーザーは次に何をすべきかわかるか」を問う

**If/Then**: If ユーザーフロー図が10ステップ以上になるなら then サブフローに分割するか、ステップを統合できないか検討する

---

### 49. 障壁を除去する

**核心**: ユーザーが目標達成を諦める原因となる摩擦・障壁を体系的に特定し除去する。

- コンバージョンファネルの離脱ポイントを数値で把握し、最大離脱箇所を優先改善する
- 「もし〜がなければどうするか」の思考実験でUIの必要性を問い直す
- 強制的な会員登録・ゲスト決済拒否・長いフォームは代表的な障壁

**If/Then**: If ユーザーがある画面で多く離脱するなら then その画面のアクションに至るまでの障壁を列挙し、削除または簡素化する

---

### 50. ないものが意味を持つ（ネガティブスペース）

**核心**: 余白・沈黙・空白は「何もない」のではなく、その周囲の要素を引き立てる積極的なデザイン要素。

- 要素間の余白は視覚的な呼吸を与え、スキャンしやすくする
- 空の状態（empty state）はonboardingの機会であり、情報設計の一部
- テキストの行間・文字間隔は可読性に直接影響する

**If/Then**: If UIが窮屈に感じるなら then 要素を削除するより余白を増やすことを先に試みる

---

### 51. フィッツの法則（ターゲットまでの距離とサイズ）

**核心**: インタラクティブ要素に到達する時間は、要素までの距離に比例し、要素のサイズに反比例する。

- 重要なアクションボタンは大きく・近い場所に置く
- 画面の端・角は常にターゲットとして有効（マウスが端で止まるため）
- モバイルでは親指が届きやすい画面下部に主要操作を配置する

**If/Then**: If 主要CTAの押下率が低いなら then サイズを大きくし、ユーザーの操作起点（親指・カーソル）から近い位置に移動する
