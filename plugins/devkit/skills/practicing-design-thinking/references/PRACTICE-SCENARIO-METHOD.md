# 構造化シナリオ法 実践ガイド

ユーザーの本質的な欲求から出発し、「価値→行動→操作」の3層シナリオを作成することで、企業目線に陥らない画面設計を実現する手法。

---

## 仕事としての始めやすさ

| こっそり練習 | 一部業務でトライアル | クライアント巻き込み |
|---|---|---|
| ★★ | ★★★ | ★★★★ |

**こっそり練習**: 過去に担当したプロジェクトや個人的な関心テーマで、シナリオを書く習慣をつける段階。Webサイトの画面設計経験があれば始めやすい。

**一部業務でトライアル**: 要件定義・設計タイミングのプロジェクトを選び、ワイヤーフレームに入る前のシナリオ作成として組み込む。実装機能が細かく決まっているプロジェクトは避ける。

**クライアント巻き込み**: 何度も練習し、親和図法（→ PRACTICE-USER-RESEARCH.md）も扱えるようになってから、規模の小さいプロジェクトで試みる。クライアントと共にシナリオを考える場合は、画面設計の論拠を論理的に説明できる準備が必要。

---

## 概要

構造化シナリオ法は、「現状の問題解決」ではなく「理想のあるべき姿」を描くための手法。ユーザーの「本質的な欲求＝価値」を中心に据え、3段階のシナリオとして具体化する。

```
ユーザー調査・親和図法
        ↓
価値のシナリオ（バリューシナリオ）
  ユーザーの「本質的な欲求」を満たすシナリオ
        ↓
行動のシナリオ（アクティビティシナリオ）
  ユーザーが「価値」を満たすために行動するシナリオ
        ↓
操作のシナリオ（インタラクションシナリオ）
  ユーザーが「行動」に伴い機器・画面等を操作するシナリオ
```

---

## なぜ構造化シナリオ法が必要か

**企業側の落とし穴**: Web制作側は「操作シナリオ」から考えてしまいがち。画面構成やUIから発想すると、ユーザーの本音（価値・行動）を見落とす。

**具体例**:
- 企業側：「問い合わせボタンを大きくして入力フォームを改善した」
- ユーザーの本音：「困ったら友達に聞くか電話する。Webフォームは面倒」

ユーザーにとってWebサイトは目的達成のひとつの手段に過ぎない。「価値→行動→操作」の順で考えることで、本質的な解決策を導ける。

## 4-1 ペルソナ作成

### ペルソナとは

シナリオの主人公となる典型的ユーザーの人物像。ペルソナなしに構造化シナリオ法は成立しない。

### ペルソナの根拠となる資料

| 資料・データ | 活かせる項目 |
|---|---|
| Webサイトの企画書・提案書 | 年齢・性別・職業・趣味・嗜好・よく見るWebサイト等 |
| アクセス解析レポート | 利用曜日・時間帯・デバイス・閲覧ページ・会員/非会員等 |
| アンケート・インタビュー調査結果 | デモグラフィック + 認知経路・利用理由・利用頻度・感想等 |

### 簡易ペルソナの作り方

A4用紙に手書き。早ければ10分で完成。数時間かける必要はない。

必須項目:
- **似顔絵**: 手書きでラフに描くか写真を貼る
- **基本属性**: 名前（仮名）・年齢・性別・居住地・職業・家族構成・趣味
- **行動の特徴**: 調査テーマに関連する日常的な行動・状況
- **目標**: Webサイト・製品・サービスを利用することで得たいこと

### ペルソナ作成の注意点

❌ **やってはいけないこと**:
- 「F1層（20〜34歳の女性）」など幅を持たせた記述（→ 最多利用年齢層を特定して書く）
- 「会社員」「独身」など一括りにした記述（→ 趣味・行動の特徴を重視する）

✅ **良いペルソナの基準**: 読んだ人がその人物の行動を予想できる具体性があること

---

## 4-2 当事者の思い込みから逃れる

ユーザーの本音は「なぜWebで解決しようとするのか？」「そもそも何を達成したいのか？」から始まる。現状のWebサイトや製品・サービスを前提にシナリオを考えると、発想が狭まる。

価値のシナリオから始めることで、今までにない新しい製品・サービスを企画できる。

---

## 4-3 価値のシナリオ（バリューシナリオ）

ペルソナの本質的な欲求を満たす、最小限のシナリオ。

### 書き方

1. ペルソナの「本質的な欲求」を1文で書く（親和図法で導き出したもの）
2. その欲求が満たされる状態を「価値のシナリオ」として記述する
3. そのシナリオに関係する「シーン」を複数列挙する

### 価値のシナリオ テンプレート

```
年  月  日

【テーマ】
「〇〇の製品・サービスやWebサイトの提案」

【ユーザーの本質的な欲求】
（親和図法などから導き出した本質的な欲求）

【ユーザーの特徴】
（ペルソナの特徴を箇条書き）

【価値のシナリオ】
（ペルソナにとっての価値のシナリオ）

【シーン】
- シーンA
- シーンB
- シーンC
```

### NG例

❌ 「どんな天候の日でも、**ヘアケア商品△□とドライヤーを使って**、常に自分の納得する髪型をキープできる」

→ 具体的な商品や道具＝「行動のシナリオ」の要素が入っている

✅ 「どんな天候の日でも、常に自分の納得する髪型をキープできる」

価値のシナリオは短くてシンプルな文章になることが多い。

---

## 4-4 行動のシナリオ（アクティビティシナリオ）

価値のシナリオのシーンを選び、ユーザーが目標を達成するための行動を具体的に描く。

### 書き方

1. 「ユーザーの目標」を1文で書く
2. 「シーン」から1つを選ぶ
3. そのシーンで、ユーザーがどのように行動するかを物語形式で記述する
4. 行動に伴う「タスク」を箇条書きにする

### 注意点

❌ **書いてはいけないもの**:
- 特定の製品名・機能名（「Facebookアプリ」「〇〇ボタン」等）
- Webサイトやアプリの具体的な操作

→ これらは操作のシナリオで書く。どうしても必要なら「〇〇できる仕組みを利用し……」のように特定製品に限定されない表現にする

### 行動のシナリオ テンプレート

```
年  月  日

【テーマ】
「〇〇の製品・サービスやWebサイトの提案」

【ユーザーの目標】
（ペルソナの目標）

【シーン】
●シーンC

【行動のシナリオ】
（そのシーンにおいてペルソナが目標を達成するための行動）

【タスク】
- タスクA
- タスクB
- タスクC
```

**MEMO**: 最初に「価値のシナリオ」が書けない場合は、先に「行動のシナリオ」を書くことも可。ただし、「価値のシナリオ」の前に「行動のシナリオ」を固めることは避ける（先入観に縛られるため）。

---

## 4-5 操作のシナリオ（インタラクションシナリオ）

行動のシナリオのタスクを選び、具体的な製品・機能・画面操作を詳細化する。

### 書き方

1. 「ユーザーの目標」を記述
2. 「タスク」から1つを選ぶ
3. 画面名・ボタン名・デバイスなど具体的な操作を番号付きで記述する

### 操作のシナリオ テンプレート

```
年  月  日

【テーマ】
「〇〇の製品・サービスやWebサイトの提案」

【ユーザーの目標】
（ペルソナの目標）

【タスク】
●タスクB

【操作のシナリオ】
① （操作1）
↓
② （操作2）
↓
③ （操作3）
```

### 代替手段

操作のシナリオの代わりに、画面構成をポンチ絵（概略図）で書くことも可。設計経験者はワイヤーフレームで書き進めた方が早い場合もある。

---

## 4-6 シナリオ作成の練習手順

構造化シナリオ法の練習は、実際の仕事と同様に「新しい製品・サービスの企画提案」という想定で行う。

### ステップ

1. **リクルーティング**: 同僚2人にペアインタビュー（各20分）を依頼。ヒアリングテーマは調査したい分野で設定
2. **インタビュー実施**: 感情曲線インタビューや弟子入りインタビュー（→ PRACTICE-USER-RESEARCH.md 参照）
3. **親和図法**: インタビューデータを付箋に書き、本質的な欲求を発見（→ PRACTICE-USER-RESEARCH.md 参照）
4. **簡易ペルソナ作成**: インタビュー相手のプロフィールと親和図の結果をベースに作成
5. **価値のシナリオを書く**: 本質的な欲求を軸に。製品名・具体的な操作は書かない
6. **行動のシナリオを書く**: 価値のシナリオのシーンから1つを選び肉付け
7. **操作のシナリオを書く**: 行動のシナリオのタスクを選び詳細化

## 4-7 ツールキット

### 簡易版テンプレート（A3印刷推奨）

**価値のシナリオシート**

| 項目 | 内容 |
|---|---|
| テーマ | |
| ユーザーの本質的な欲求 | |
| ユーザーの特徴 | |
| 価値のシナリオ | |
| シーン | シーンA / B / C / D |

**行動のシナリオシート**

| 項目 | 内容 |
|---|---|
| テーマ | |
| ユーザーの目標 | |
| シーン | |
| 行動のシナリオ | |
| タスク | タスクA / B / C / D |

**操作のシナリオシート**

| 項目 | 内容 |
|---|---|
| テーマ | |
| ユーザーの目標 | |
| タスク | |
| 操作のシナリオ（番号付き） | |

### 3シートの関係性

```
価値のシナリオ
  └── シーン（複数）
        └── 行動のシナリオ（シーンごとに1枚以上）
              └── タスク（複数）
                    └── 操作のシナリオ（タスクごとに1枚以上）
```

ひとつの価値のシナリオから複数のシーン → 複数の行動のシナリオ → 複数の操作のシナリオが生まれる。

---

## 4-8 客観性の保ち方

自分で書いたシナリオは「いいシナリオ」に見えがち。以下の方法で客観性を保つ。

### 他者レビュー

- 同僚や友人にシナリオを読んでもらい、おかしい展開を指摘してもらう
- 複数案を書いて発表し、メンバーで投票する
- クライアント側にもレビューしてもらう

### ロールプレイ（上級者向け）

- 「行動のシナリオ」を声に出して演じる
- ペルソナになりきって、行動・発話・感情を表現する
- 実際に存在しない機器・製品は「使っているふうに演じる」
- これはシナリオのプロトタイピング検証でもある

## Light Case / Heavy Case

### Light Case（ライト級）

| 項目 | 内容 |
|---|---|
| 期間 | 1週間程度 |
| 体制 | 3〜4名（ディレクター・プランナー・専門家） |
| 調査 | 社員同士のペアインタビュー（定性データ収集） |
| シナリオ | 価値のシナリオ + 行動のシナリオ（操作のシナリオは割愛） |
| 目的 | 企画提案段階での候補案出し |
| ポイント | 予算なしでも社員同士のインタビューで実施可能。企画段階のため操作のシナリオは不要 |

### Heavy Case（ヘビー級）

| 項目 | 内容 |
|---|---|
| 期間 | 4週間程度 |
| 体制 | ディレクター・デザイナー・HCD専門家 |
| 調査 | ユーザーインタビュー + 行動観察 |
| シナリオ | 3層すべて（操作のシナリオの代わりにラフプロトで代用も可） |
| 目的 | 業務ツール改善など、実態に即した設計が必要なケース |
| ポイント | 現場の行動観察でリアリティのあるペルソナを作成し、業務の実態に即したシナリオを構築 |

## チェックリスト

### 価値のシナリオ
- [ ] 本質的な欲求（親和図法の分析結果）が軸になっているか
- [ ] 具体的な製品名・機能名が含まれていないか
- [ ] シーンが複数列挙されているか

### 行動のシナリオ
- [ ] 特定の製品名・アプリ名が含まれていないか
- [ ] 画面操作の詳細が書かれていないか（→ 操作のシナリオへ）
- [ ] タスクが箇条書きで明確化されているか

### 操作のシナリオ
- [ ] ボタン名・画面名など具体的な要素が含まれているか
- [ ] 操作が番号付きで順序立てて書かれているか

### 全体
- [ ] 3シートが「価値→行動→操作」の順で繋がっているか
- [ ] 他者にレビューしてもらったか
- [ ] ペルソナがシナリオの主人公として機能しているか
