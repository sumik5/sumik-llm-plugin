# プロセスワークフロー・反復開発リファレンス

スプリント計画からKanban設計、Sprint Zero、リリース管理、
リアルタイムアラートまで——継続的デリバリーを支える実践的なワークフロー手法。

---

## 1. スプリントケイデンスと反復ワークフロー

### 2週間スプリントの設計原則

2週間サイクルは「継続的な進捗」と「変化への応答性」のバランスを確保する実践的な選択肢。

**スプリント構成（月曜開始）:**

```
月曜午前  スプリントプランニング
  - PMが最上位バックログアイテムをKPIとの紐付きで提示
  - エンジニア・デザイナー・サポートが一緒にキャパシティと複雑度を評価
  - スコープをロック → 未計画作業は専用バッファキューへ

水〜金   実装・進捗確認
  - デイリースタンドアップで完了タスク・計画作業・ブロッカーを共有
  - 予期せぬブロッカー発生時は低優先ストーリーと入れ替えて進捗維持

翌週木曜午後  バックログリファインメント（60分）
  - 上位10件のストーリーを多機能メンバーでレビュー
  - エンジニアが工数見積もり、デザイナーがワイヤーフレーム提示
  - ストーリーを5分タイムボックスで議論 → 深い技術議論は非同期スレッドへ
  - セッション終了時には次スプリント用の「定義済み・サイズ済み・優先順位付き」ストーリーが揃う

翌週金曜  スプリントレビュー + レトロスペクティブ
```

### リリース候補の品質基準

リリースに進むための3つの条件：

| 基準 | 閾値 |
|------|------|
| 新規コードのカバレッジ | 80%以上 |
| 高優先度バグ | 0件（対象機能） |
| キーエンドポイントの応答速度 | 標準負荷下で200ms以内 |

### リグレッションテスト戦略

```
ナイトリーリグレッション:
- ステージング環境の「リリーストレイン」に対して毎夜実行
- カバー範囲: アカウント作成、検索、予約、決済等の重要フロー
- 失敗時: 自動アラート + ロールバック → 翌営業日開始前に解決

デプロイ後スモークテスト:
- 本番デプロイ直後に実施
- 失敗パターンの場合はロールバック判断を即時行う
```

**「リリーストレイン」モデル:** 複数フィーチャーをステージングでまとめて検証してから次の本番デプロイに統合する方式。個別デプロイの調整コストを削減しながら品質を維持する。

## 2. 分散チームのアジャイルケイデンス

### インフラ制約下での適応

分散チームでは帯域幅・タイムゾーン・接続環境の違いを前提としたケイデンス設計が必要。

#### 非同期デイリースタンドアップ

リジッドな9時MTGは低帯域・高通勤地域のメンバーを排除する。代替手法：

```
毎朝11時までにチャットチャンネルに3点投稿:
1. 完了タスク
2. 本日の計画作業
3. ブロッカー（複雑な場合は短音声メモを添付）

PMが確認・クリティカル課題をフラグ → フォローアップアクションをトリアージ
（長時間ビデオ通話なしで透明性を維持）
```

#### 軽量バックログリファインメント（45分・隔週）

- 低帯域最適化ビデオルームで実施
- 低接続ゾーンのメンバーは音声のみ参加
- ストーリーの核心情報（受入条件・依存関係・概算見積もり）はオフライン参照可能なテキストベースボードに記録
- ファシリテーターが1項目5分でタイムボックス → 深い技術議論は非同期でIssueトラッカーへ

#### 隔週ショーアンドテル（30分）

- 完成フィーチャーのデモをステークホルダーに公開
- 事前収録のウォークスルー動画を共有 → ライブでQ&Aのみ
- デザイナーはクラウドリポジトリのインタラクティブプロトタイプを使用（ダウンロード不要）

#### 四半期プランニングワークショップ（3時間）

- テキストアウトライン＋静的ワイヤーフレーム＋軽量ポール（重い資料は不使用）
- モバイル最適化ポーリングツールでフィーチャーシーケンスを投票
- スプリントコミットメントはRICEスコアと直近ベロシティデータに基づいて有機的に決定

#### 低コスト同期ポイント

| リズム | 形式 | 目的 |
|--------|------|------|
| スプリント中間 | 10分 音声通話 | 顕在化しつつあるリスクの特定・スコープ調整 |
| 隔週 | チャンネルに短動画ピン | 新フィーチャーのデモ → 非同期フィードバック |
| 週次 | 一問アンケート | チームセンチメント・接続問題・ツールフィードバック |

### 接続コスト管理

- チームメンバーが個人データプランを使用している場合、データ軽量セッション（テキスト主体・事前収録）でコストを抑制
- ツールアセット（プロトタイプ・ドキュメント）はクラウドに保存し、必要時のみダウンロード

## 3. Kanbanプロセス設定

### 2レーン Kanbanボードの設計

スプリントリズムとアドホックな作業が混在する環境では、専用の2レーン構成が機能する。

```
レーン1: Support-to-Dev キュー（カスタマーサポート所有）
  [New] → [Triage: In Progress] → [Ready for Dev]

レーン2: Engineering Implementation レーン（開発チーム所有）
  [To Do] → [In Progress] → [Code Review] → [QA Testing] → [Done]
```

**ボード全体を「Single Source of Truth」として機能させる:**
- サポートチームは開発進捗をリアルタイムで参照可能
- エンジニアリングは顧客コンテキスト（再現手順・ユーザーインパクト）を保持したまま実装

#### Support-to-Dev レーンの詳細

1. **Triage:** インパクト・緊急度で評価 → カラーコードカードで種別・深刻度を分類
2. **Ready for Dev:** 高優先チケットに再現手順・ユーザーコンテキストを付記して渡す

#### Engineering Implementation レーンの詳細

各ステージに明示的な入退場基準をボードに掲示（推測排除）。

### WIP（Work In Progress）制限

| カラム | WIP上限 | 効果 |
|--------|---------|------|
| Triage | 設定値（例: 5） | サポートの負荷を可視化 |
| In Progress | 設定値（例: 3/人） | コンテキストスイッチを抑制 |
| Code Review | 設定値（例: 4） | レビュー滞留を防止 |
| QA Testing | 設定値（例: 3） | テストボトルネックを可視化 |

上限到達時はデジタルボードで視覚的警告 → 新作業を開始せずダウンストリームへ進める作業を優先。

### スイムレーン分割

```
スイムレーン構成:
- Critical Production Issues（本番障害・最優先）
- Planned Enhancements（計画済みフィーチャー）
- Technical Debt（技術的負債）
```

緊急案件が通常タスクに埋もれることを防ぐ構造的な分離。

### 「Done」定義の策定

「完了」の曖昧さをなくすため、ステージごとに明確な完了基準を設定する：

**例:**
- Code Review完了: 指摘事項0件 or Approved
- QA Testing完了: 受入条件100%通過、高優先バグ0件
- Done: 本番デプロイ済み、スモークテスト通過

## 4. Sprint Zeroロードマップ計画

### Sprint Zeroの役割

本格実装の前段階として、主要ソリューションを検証するための期間。3つの主要活動：

1. **Spikeタスク**（クイックバリデーション）
2. **APIインテグレーション PoC**（技術検証）
3. **モニタリングセットアップ**（可観測性の確立）

### Spikeタスクの設計

各スパイクは2〜3日のタイムボックス。解決されないままにすると大規模プロジェクトを止める可能性のある技術的・設計的クエスチョンに答えることが目的。

```
スパイク定義テンプレート:
- 検証する問い（1文で明確に）
- 期間（2〜3日）
- 成功基準（定量的・測定可能）
- 担当チーム
```

**例:**

| スパイク | 成功基準 |
|---------|---------|
| コピー明確性A/B | 48時間・5,000ユーザーでキャンセルリンククリック -10% |
| SDKエラー処理パッチ | ステージングで5日間連続クラッシュ0件 |
| フォールバックAPIプロトタイプ | 99%リクエストで500ms以下、UIへのエラー0件 |

**デイリーチェック:** 「このスパイクは仮説を確認したか、それとも反証したか？」

Jiraの「Spike」イシュータイプで追跡、結果はConfluenceページに記録。

### APIインテグレーション PoC

外部サービス（例: 航空会社スケジュールAPI）が不安定な場合のフォールバック戦略を検証する：

- シンプルなプロキシサービスをビルドし、API障害時にキャッシュデータを返す
- 成功基準: 応答500ms以内・エラー率0%
- 本格実装前にPoC成果でアーキテクチャ判断を下す

### モニタリングセットアップ

既存ツール（Prometheus / Grafana / Slack等）を活用して、Sprint Zero期間中から実験の観測可能性を確保：

```
アラートルール例:
- フォールバックプロキシが300ms超 → Slack通知
- エラー率が5分ウィンドウで1%超 → Slack通知
```

「スプリント中にすべてのステークホルダーが実験結果をリアルタイムで確認できる」ことが目標。

## 5. リアルタイムアラートダッシュボード

### ダッシュボードの設計目的

- **定常状態の確認:** SREチームへの「異常なし」シグナル
- **スパイク検出:** 閾値超過時の即時エスカレーション
- データ → 判断のサイクルを数分以内に完結させる

### 2シナリオのダッシュボード設計

**シナリオ1: 正常運用**

```
キャンセル率: 5〜7%（10分ローリング平均）
状態: グリーン
アノテーション: "Steady state: No action required."
アラート: サイレント
```

**シナリオ2: 急激なスパイク検出**

```
外部API障害 → キャンセル急増
14:30: 10分平均が15%を2分超過
→ パネルが赤に変化
→ Slack通知:
   "🚨 キャンセル率が15%を2分間超過。現在: 16.8% [ダッシュボードリンク]"
→ イベントログテーブルに詳細記録
→ エンジニアがフォールバックサービスを確認・デプロイ
→ 15:00までに閾値以下に回復
```

### アラートルール設計の原則

| 要素 | 推奨 |
|------|------|
| ウィンドウ | 10分ローリング平均（ノイズ軽減と即時性のバランス） |
| 発火条件 | 閾値を「N分間継続」超過（一時的なスパイクの誤報を防ぐ） |
| 通知先 | Slackチャンネル＋ダッシュボードリンク |
| 視覚インジケーター | 正常:緑 / アラート:赤 |
| ログ | タイムスタンプ付きイベントログテーブルで証跡を保持 |

### 既存ツールスタックの活用

新規ツール導入前に既存スタック（Prometheus + Grafana + Slack）で実現可能な範囲を最大化する。Sprint Zeroでの監視は特に「軽量・即座に起動可能」であることが重要。

## クイックリファレンス: ワークフロー選択ガイド

| 状況 | 適切なアプローチ |
|------|----------------|
| 計画的な機能開発（2週間サイクル） | スプリントケイデンス |
| 分散・低帯域チーム | 非同期スタンドアップ + 軽量セレモニー |
| 緊急修正と計画作業が混在 | 2レーンKanban + WIP制限 |
| 不確実性が高い新問題への対処 | Sprint Zero + Spikeタスク |
| 問題の早期検知・即時対応 | リアルタイムアラートダッシュボード |
| リリース前品質チェック | リリース候補基準 + ナイトリーリグレッション |
