# AI PM 3つの専門化ロール

AI時代のプロダクトマネジメントにおける3つの専門化領域を定義したリファレンス。各ロールの担当領域・核心スキル・課題と、自分に合った専門化の選び方を解説する。

> **前提**：3ロールは排他的な職種ではなく、特定組織では1人が複数の役割を担うこともある。ここではそれぞれの「フォーカスエリア」として捉える。

---

## ロール1：AI-Experiences PM

**ミッション**：AI搭載機能をユーザーが理解・活用できる体験として具現化する。AIの技術的能力とユーザーのニーズを橋渡しする「UXの守護者」。

### 担当領域と核心スキル

| カテゴリ | 内容 |
|---------|------|
| **主な担当** | ユーザー向けAI機能のUX設計・実験・改善 |
| **「顧客」** | エンドユーザー |
| **成功指標** | クリック率、タスク完了率、ユーザー満足度 |
| **協働相手** | データサイエンティスト、UXデザイナー、エンジニア |
| **コアスキル** | ユーザーリサーチ、A/Bテスト設計、プロンプトエンジニアリング、GenAI評価理解 |
| **技術深度** | AI/ML の概念理解（実装不要）、API連携の基礎、デプロイメントのUX影響理解 |

### 主な責務

- ユーザーニーズを起点にAI機能の機会を特定・優先順位付け
- AI出力の提示方法・不確実性の表現・フィードバック設計を主導
- 倫理・公平性・透明性をUX要件として組み込む
- A/Bテストで機能インパクトを定量検証し、反復改善を推進
- データサイエンスチームにユーザー視点でのモデル評価フィードバックを提供

### 課題とジレンマ

| 課題 | 具体的な緊張関係 |
|------|----------------|
| 革新性 vs 使いやすさ | 最先端の技術を使いたいが、UIが複雑になりすぎる |
| 期待管理 | ハイプに溢れたAIへの過剰期待を現実的に管理する |
| 不確実性の設計 | モデルが外れた時のフォールバックUXを設計する必要がある |
| 倫理的責任 | ユーザー向けAIのバイアス・有害コンテンツへの第一線防御者 |
| データへの依存 | 機能改善がデータ収集・品質チームの進捗に依存することが多い |

## ロール2：AI-Builder PM

**ミッション**：組織全体のAI開発を可能にする基盤（プラットフォーム・モデル・フレームワーク）を構築・管理する「AIインフラの設計者」。

### 担当領域と核心スキル

| カテゴリ | 内容 |
|---------|------|
| **主な担当** | AI基盤プラットフォーム・再利用可能モデル・MLOpsフレームワーク |
| **「顧客」** | 社内開発チーム（他のPM・エンジニア・データサイエンティスト） |
| **成功指標** | プラットフォーム採用率、モデルデプロイ時間、コスト効率 |
| **協働相手** | データサイエンティスト、MLエンジニア、DevOps、法務・コンプライアンス |
| **コアスキル** | 技術アーキテクチャ理解、MLOps設計、データガバナンス、GenAI評価管理 |
| **技術深度** | クラウドインフラ（AWS/GCP/Azure）、API設計、MLパイプライン、モデル評価フレームワーク |

### 主な責務

- AIプラットフォームの長期ビジョンとロードマップを定義・所有
- 複数プロダクトチームが共用できる基盤モデル・APIを開発・デプロイ
- A/Bテストフレームワーク・モデル監視・再学習パイプラインを整備
- データガバナンスポリシーを確立し、法律・コンプライアンスチームと連携
- GenAI評価（GenAI Evals）プロセスを設計・監督し、モデル品質・安全性を保証

### 課題とジレンマ

| 課題 | 具体的な緊張関係 |
|------|----------------|
| 優先順位の競合 | 複数プロダクトチームからのAI基盤要求が常に競合する |
| 技術的複雑性 | スケーラブルで信頼性の高いプラットフォーム設計には深い技術知識が必要 |
| プラットフォーム採用 | 優秀な基盤を作っても使われなければ価値ゼロ。内部マーケティングが必要 |
| 成功指標の定義 | ユーザー向け機能と異なり、間接的なインパクト指標を定義しなければならない |
| 技術革新への対応 | AI分野の急速な変化に合わせてプラットフォームを継続進化させる必要がある |

## ロール3：AI-Enhanced PM

**ミッション**：AIツールを活用して自身のPM業務を高度化・効率化する「AIパワーユーザー」。AIプロダクトを作るのではなく、AIでPM業務そのものを変革する。

### 担当領域と核心スキル

| カテゴリ | 内容 |
|---------|------|
| **主な担当** | PM業務へのAIツール統合・チームへのAI活用促進 |
| **「顧客」** | 自分自身とプロダクトチーム |
| **成功指標** | PM生産性向上、意思決定の質改善、ツール採用率 |
| **協働相手** | 他のPM、データアナリスト、IT/エンジニアチーム |
| **コアスキル** | データ分析・解釈、AIツール評価・選定、プロンプトエンジニアリング、変革マネジメント |
| **技術深度** | APIの基本概念、クラウドサービスの基礎知識、各種AIツールの機能理解 |

### 主な責務（PM業務の変革領域）

| PM業務領域 | AIによる変革方法 |
|-----------|----------------|
| **市場・競合分析** | AI競合インテリジェンスツールで自動集約・分析 |
| **ユーザーリサーチ分析** | NLPツールでレビュー・サポートチケット・インタビューを自動分類・要約 |
| **ロードマップ優先順位** | AIツールが多元的データを統合し優先順位案を生成（人間が最終判断） |
| **PRD・ドキュメント作成** | LLMでドラフト生成、PMが編集・改善 |
| **会議効率化** | AI文字起こし・要約ツールでアクションアイテムを自動抽出 |
| **プロトタイピング** | AI UIジェネレーターで迅速なMVPUI作成 |

### 課題とジレンマ

| 課題 | 具体的な緊張関係 |
|------|----------------|
| ツール選択オーバーロード | 急増するAIツールの中から真に有効なものを見極める難しさ |
| AI過信リスク | AIの出力を批判的評価なしに受け入れる危険性 |
| ブラックボックス懸念 | なぜそのインサイトが生成されたか説明できないツールへの信頼 |
| チーム採用の難しさ | 既存ワークフローを変えることへの組織的抵抗 |
| ROI証明 | ツール投資対効果を定量的に示す必要性 |

## 3ロール比較テーブル

### スキルセット比較

| スキル | AI-Experiences PM | AI-Builder PM | AI-Enhanced PM |
|--------|-------------------|---------------|----------------|
| ユーザーリサーチ | ⭐⭐⭐ 必須 | ⭐ 基礎 | ⭐⭐ AIツール活用 |
| 技術アーキテクチャ理解 | ⭐⭐ 中程度 | ⭐⭐⭐ 深い | ⭐ 基礎 |
| データ分析 | ⭐⭐ 中程度 | ⭐⭐ 中程度 | ⭐⭐⭐ 主要スキル |
| MLOps/インフラ | ⭐ 基礎 | ⭐⭐⭐ 深い | ⭐ 基礎 |
| プロンプトエンジニアリング | ⭐⭐ 中程度 | ⭐ 基礎〜中程度 | ⭐⭐ 中程度 |
| GenAI評価 | ⭐⭐ 中程度 | ⭐⭐⭐ 深い | ⭐⭐ 中程度 |
| 変革マネジメント | ⭐ 基礎 | ⭐⭐ 中程度 | ⭐⭐⭐ 必須 |
| 適応力・学習敏捷性 | ⭐⭐ 中程度 | ⭐⭐ 中程度 | ⭐⭐⭐ 必須 |

### フォーカス領域と典型業務

| 観点 | AI-Experiences PM | AI-Builder PM | AI-Enhanced PM |
|------|-------------------|---------------|----------------|
| **フォーカス** | ユーザー向けAI体験 | AI基盤インフラ | PM業務の高度化 |
| **「製品」** | エンドユーザー機能 | 社内AIプラットフォーム | PM自身の生産性 |
| **典型的な1日** | ユーザーフィードバック分析、A/Bテスト設計、UXレビュー | プラットフォームKPI確認、モデルリクエスト評価、GenAI Evals設計 | 競合インテリジェンス確認、AI要約でUXリサーチ合成、LLMでPRD下書き |
| **成果の可視性** | ユーザー指標に直接反映 | 間接的（他チームの能力向上） | 個人・チーム生産性 |
| **組織規模適性** | 全サイズ | 中〜大規模組織 | 全サイズ |

## ロール間の連携パターン

```
AI-Builder PM
    │
    │ 基盤API・モデル・フレームワークを提供
    ▼
AI-Experiences PM ◄── ユーザーニーズ・機能要件をフィードバック
    │
    │ ユーザー行動データ・実験インサイトを共有
    ▼
AI-Enhanced PM ──► AI-Builder PMへ内部ツール改善要望
                └──► AI-Experiences PMへのユーザーリサーチインサイト共有
```

### 連携が重要な場面

| シナリオ | 誰がリード | 誰がサポート |
|---------|-----------|------------|
| 新AI機能のゼロからの開発 | AI-Experiences PM | AI-Builder PM（基盤提供） |
| 組織横断的なAI評価プロセス整備 | AI-Builder PM | AI-Experiences PM（評価基準へのUX要件提供） |
| PM組織全体のAIツール導入 | AI-Enhanced PM | AI-Builder PM（内部ツール整備） |
| ユーザー向け機能の倫理審査 | AI-Experiences PM + Legal | AI-Builder PM（技術的実装の倫理対応） |

## 自分に合った専門化の選び方

### 傾向チェック

| 問い | Yes → 向いている専門化 |
|------|----------------------|
| ユーザーインタビューやユーザビリティテストが好き？ | **AI-Experiences PM** |
| なぜ機能が動くかより「どう感じるか」を重視する？ | **AI-Experiences PM** |
| システムアーキテクチャや技術インフラの議論に引き込まれる？ | **AI-Builder PM** |
| 一つの機能より組織全体のプラットフォーム整備に興奮する？ | **AI-Builder PM** |
| 新ツールを試して自分のワークフローを最適化するのが好き？ | **AI-Enhanced PM** |
| データ分析でインサイトを抽出することが得意・好き？ | **AI-Enhanced PM** |

### キャリアパスの観点

| 背景・志向 | おすすめの入口 |
|-----------|--------------|
| UX・デザイン出身 | AI-Experiences PM |
| エンジニア・データサイエンス出身 | AI-Builder PM |
| 現役PM・BizDev出身でAIツール活用を加速したい | AI-Enhanced PM |
| スタートアップで一人PM | AI-Enhanced PM + AI-Experiences PM の複合 |

> **重要な前提**：どの専門化を選んでも、Ch 1-4の基礎AI/MLリテラシー（`AI-PLAYBOOK-FOUNDATIONS.md`参照）は全ロール共通の必須知識。専門化はその上に積み上げるものである。
