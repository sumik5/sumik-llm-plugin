# AIプロダクトライフサイクル詳細

---

## AI Lifecycle 8段階 — 詳細ガイド

### Stage 1: Problem Definition & Business Understanding（問題定義）
> *「なぜ作るか？」— ビジネス問題の本質を明確化する*

**PMの核心タスク**:
- ビジネスゴールをAI/MLで解決可能な形に変換する
- 「AI機能を作りたい」ではなく「ユーザーのXという問題を解決したい」から始める
- 成功の定義（Success Criteria）を定量的に設定する

**重要な問い**:
```
□ どのビジネス問題を解決するか？
□ AI/MLなしでは解決できないか？
□ 成功したとき何が変わるか（ビジネス指標で）？
□ ステークホルダーは誰で、何を期待しているか？
```

**アウトプット**: Problem Statement・Success Metrics・制約条件一覧

**具体例（チャーン予測）**: サブスクリプションサービスでユーザーが解約している。ビジネスゴールは既存加入者の維持。定量目標: 「次の30日以内に解約する可能性が高いユーザーはどれか？」をモデルに答えさせる。PMの役割: ステークホルダーと協力してこのフェーズをリードし、プロダクト全体戦略との整合を確保する。

---

### Stage 2: Data Collection & Exploration（データ収集・探索）
> *「どんなデータが必要か？」— データの食材を見極める*

**PMの核心タスク**:
- 必要なデータ種類・量・品質の要件定義
- 既存データソースの棚卸しとギャップ分析
- データプライバシー・規制要件の確認

**データ品質チェック**:
- 完全性（Completeness）: 欠損値の割合
- 正確性（Accuracy）: ラベルの信頼性
- 一貫性（Consistency）: 異なるソース間の整合性
- 適時性（Timeliness）: データの鮮度

**PMへの警告**: 「完璧なデータが揃うまで待つ」は誤り。
小規模・高品質なデータでAI MVPを作り、イテレーションで改善する。

**具体例（チャーン予測）**: 収集すべきデータ — ユーザーデモグラフィクス（年齢・地域・プランタイプ）、視聴履歴・ログイン頻度・コンテンツ完了率、サポートインタラクション・支払い履歴・フィードバックデータ。探索で「月5時間未満のコンテンツ視聴ユーザーが解約しやすい」等の洞察を得る。

---

### Stage 3: Data Preprocessing（データ前処理）
> *「食材を調理する」— データをモデルが扱える形にする*

**主要タスク**:
- データクリーニング（欠損値・外れ値・重複の処理）
- 正規化・標準化（数値スケールの統一）
- エンコーディング（カテゴリ変数→数値変換）
- 訓練・検証・テストセットの分割

**PMの役割**:
- データエンジニアリング工数をロードマップに適切に反映する
- データ前処理パイプラインの品質保証プロセスを要求する
- データプライバシー・倫理的配慮を前処理判断に影響させる

**具体例（チャーン予測）**: 欠損した年齢データを類似ユーザー特性から補完、日付フォーマットを統一規格に変換、視聴時間等の数値特徴量を共通スケールに正規化。

---

### Stage 4: Feature Engineering（特徴量エンジニアリング）
> *「モデルへの入力を設計する」— 問題を解くための情報を抽出する*

**概念**:
- Feature（特徴量）: モデルが学習に使う入力変数
- Feature Engineering: 生データから意味のある特徴量を作り出すプロセス

**PMの役割**:
- ドメイン知識（ビジネス理解）をエンジニアに提供する
- 「このユーザー行動が重要」という仮説をデータチームと共有する
- Feature Importanceを確認し、強い予測因子をプロダクト判断に活用する

**具体例（チャーン予測）で作成する特徴量**:

| 特徴量名 | 例 | 説明 |
|---------|-----|------|
| Days Since Last Login | 15 | 最終ログインから15日経過 |
| Avg Courses Completed/Month | 1.5 | 月平均1.5コース完了 |
| Forum Posts per Week | 0.2 | 約5週に1投稿（低関与） |
| Subscription Length (months) | 24 | 2年継続中 |
| Discount Usage Rate | 0.10 | 支払いの10%で割引使用 |

---

### Stage 5: Model Selection & Training（モデル選択・訓練）
> *「どのアルゴリズムを選ぶか？」— 問題に適したツールを選択する*

**モデルタイプとPMへの含意**:

| タイプ | 特徴 | 代表的なユースケース |
|--------|------|------------------|
| 線形回帰 | シンプル・説明しやすい | 価格予測・需要予測 |
| 決定木/ランダムフォレスト | 解釈可能・ロバスト | チャーン予測・分類 |
| ニューラルネットワーク | 高精度・複雑 | 画像認識・自然言語処理 |
| 大規模言語モデル（LLM） | 汎用・生成 | チャットボット・コンテンツ生成 |

**PMの役割**:
- 精度 vs 解釈可能性のトレードオフを事業要件に基づき判断
- ハイリスク領域（医療・融資）では解釈可能モデルを優先
- 成功指標（KPIs）を明確に定義・伝達する

**具体例（チャーン予測）**: 分類タスクのため以下を検討 — Logistic Regression（シンプル・解釈可能）、Decision Trees（可視化しやすい）、Random Forest（精度が高いアンサンブル手法）、Neural Networks（非線形関係を捕捉できるが解釈困難）。

---

### Stage 6: Model Evaluation & Tuning（モデル評価・チューニング）
> *「品質を確認し磨く」— ビジネス目標に対して十分に機能するか検証する*

**主要評価指標（PMが知るべき）**:

| 指標 | 意味 | PMとの関連 |
|------|------|-----------|
| **Accuracy（正解率）** | 全体的な正確さ | 平均的な性能把握 |
| **Precision（適合率）** | 陽性予測の正確さ | 誤検知コストが高い場合（スパム・詐欺検知） |
| **Recall（再現率）** | 実際の陽性の捕捉率 | 見逃しコストが高い場合（病気検知） |
| **F1 Score** | PrecisionとRecallの調和平均 | バランスが必要な場合 |
| **AUC-ROC** | 全体的な判別性能 | モデル比較・閾値設計 |

**GenAI評価手法（3種）**:
- **Human Evaluation**: 評価者が品質を直接判断（Gold Standard・コスト高）
- **Automated Metrics**: BLEU/ROUGE/Perplexity等（スケーラブル・客観的）
- **Hybrid Approaches**: 自動フィルタリング → 人手評価（バランス型・推奨）

**PMの役割（Stage 6）**:
- 評価指標の意味を理解し、ビジネスゴールとの関連を把握する
- 改善優先度を評価結果に基づきデータサイエンスチームと決定する
- モデル開発がイテレーティブなプロセスと認識し、複数回の評価・チューニングを想定する

**具体例（チャーン予測）**: 高Accuracyながら低Recallの場合（実際の解約ユーザーを多く見逃す）、閾値を調整してRecallを向上させる。解約を見逃すコスト（収益損失）の方が誤警告コストより高いため、Recallをプライマリ指標にする。Precision・F1-Scoreをセカンダリとして監視。

---

### Stage 7: Deployment & Monitoring（デプロイ・監視）
> *「本番稼働と健全性維持」— AIを生きたシステムとして管理する*

**デプロイ戦略**:

| 戦略 | 内容 | 使いどころ |
|------|------|-----------|
| **Shadow Deployment** | 本番影響なしにモデル評価 | 新モデルの安全な検証 |
| **A/B Testing** | モデルバージョン比較 | インパクト測定 |
| **Canary Deployment** | 一部ユーザーに段階的展開 | リスク最小化 |

**監視すべき指標**:
```
技術的指標:
  - モデルの予測精度（継続的）
  - レイテンシ・スループット
  - エラー率・可用性

ビジネス指標:
  - North Star Metricへの貢献
  - ユーザー行動の変化
  - 収益・コストへのインパクト

Model Health:
  - Data Drift（入力データ分布の変化）
  - Concept Drift（問題の性質変化）
  - Feature Importance変化
```

---

### Stage 8: Retraining & Maintenance（再訓練・保守）
> *「モデルを最新状態に保つ」— living systemとして継続管理する*

**Model Driftへの対応**:
- **データドリフト**: 入力データの分布が変化 → 再訓練トリガー
- **コンセプトドリフト**: 問題の性質が変化 → 特徴量・モデル再設計
- **パフォーマンス劣化**: 精度指標の低下 → 調査・再訓練

**再訓練トリガーの種類**:
- **スケジュールトリガー**: 月次/四半期定期再訓練
- **パフォーマンストリガー**: 精度が閾値を下回ったとき
- **イベントトリガー**: 重大なビジネス変化時

**PMのロードマップへの組み込み方**:
```
✅ 初回リリース後の保守工数を最低30%見積もる
✅ モデル再訓練サイクルをスプリント計画に含める
✅ モデル劣化のアラートシステムを要件に含める
```

**PMの役割（Stage 8）**:
- 再訓練スケジュールをデータサイエンスチームと確立する（月次、または性能閾値トリガー）
- 継続的な保守・更新のためにリソースを確保する
- 再訓練に必要なデータ収集を継続的に優先する
- ユーザーフィードバックがモデルの問題を示している場合は原因を追究するフィードバックループを設ける

**具体例（チャーン予測）**: 最新ユーザーデータで毎月再訓練。新機能の大規模リリース時はユーザー行動が大きく変化するため追加再訓練を検討。継続的なA/Bテストで介入施策（割引・メール文面等）の有効性も最適化する。

## NPD（新製品開発）の7段階とAI製品

一般的なNPDプロセスとAI製品特有の考慮点：

| ステップ | 内容 | AI製品特有の考慮点 |
|---------|------|-----------------|
| 1. Discovery | ユーザーリサーチ・市場分析 | データ可用性の調査を追加 |
| 2. Define | 要件定義・成功指標設定 | AI評価指標（MLメトリクス）の定義 |
| 3. Design | UX/UIデザイン | AI特有のUX（不確実性表示・説明可能性）設計 |
| 4. Implementation | 開発・訓練 | AI Lifecycle 8段階の実行 |
| 5. Marketing | GTM準備 | AI機能のポジショニング・リスク開示 |
| 6. Beta Testing | 限定公開テスト | 人間評価・多様なユーザーでのバイアステスト |
| 7. Launch | 本番リリース | 段階的デプロイ（Shadow→Canary→Full） |

### AI/MLドリームチーム — 批判/非批判ロール詳細

AI MVP立ち上げに必要なロールを**CRITICAL（必須）**と**NON-CRITICAL（状況次第）**に分類する。

#### CRITICAL ロール（6つ）

| ロール | 部門 | 主な責任 |
|--------|------|---------|
| **AI PM** | プロダクト管理 | プロダクトビジョン・要件・成功指標の定義。AIとビジネスゴールの整合 |
| **Data Scientist** | データサイエンス | AI/MLモデルの設計・実験・データ分析とインサイト抽出 |
| **ML Engineer** | エンジニアリング | データサイエンスと本番環境をつなぐ橋渡し。スケーラブルなモデルの実装 |
| **Data Engineer** | エンジニアリング | データパイプライン構築・データフロー管理・訓練用クリーンデータの確保 |
| **Full Stack Developer** | ソフトウェア開発 | AI出力をフロントエンド・バックエンドに統合し、シームレスなUI/UXを実現 |
| **DevOps Engineer** | オペレーション | デプロイ・インフラ・CI/CDの管理。AIモデルの安定的な本番稼働を担保 |

#### NON-CRITICAL ロール（状況依存）

| ロール | 部門 | 活用タイミング |
|--------|------|--------------|
| **AI/ML Data Strategist/Architect** | データサイエンス | AI旅の初期段階（最初の6〜8ヶ月）。データ戦略・採用・ワークフロー設計のコンサルタント |
| **Data Analyst** | データサイエンス | データ基盤の確立フェーズ。クエリ作成・探索的分析・ダッシュボード提供 |
| **UX/UI Designer** | デザイン | AI出力のプレゼンテーション最適化。ユーザーとAI機能のインタラクション設計 |
| **UX/UI Researcher** | デザイン | ユーザーリサーチ・行動・嗜好・ペインポイント調査。ユーザー中心設計への反映 |
| **QA Engineer** | エンジニアリング | AI機能のテスト・信頼性・パフォーマンス検証。モデルドリフト・バイアス監視 |
| **Cloud Architect** | エンジニアリング | AIインフラのスケール。クラウドリソース最適化・大規模デプロイ戦略 |
| **Business Analyst** | アナリティクス | ビジネスニーズを技術要件に変換。ステークホルダーアラインメントの追加サポート |
| **Ethics and Compliance Officer** | 法務/コンプライアンス | モデルの倫理的考慮・バイアス評価・規制コンプライアンス確認 |

> **PMの判断基準**: 必ずしも全ロールが必要ではない。MVP段階ではCRITICAL 6ロールに集中し、スケールに応じてNON-CRITICALを追加する。

### テックスタック投資判断

AI/MLプロダクトのテックスタック選定において、PMが知るべき判断基準とツール体系。

#### 投資判断の3原則

```
原則1: スケールに応じて追加投資する
└ 「まず問題が発生してから」ツールに投資する。早期の過剰投資は避ける。

原則2: チームのコラボレーション環境を確保する
└ アナリスト・データサイエンティスト・MLエンジニア・開発チームが
  同一プラットフォームで連携できる環境を優先する。

原則3: 自動化できるものは自動化する
└ 人材は高レベルな意思決定に集中させる。
  データ更新・テスト・監視は自動化の対象とする。
```

#### ツール体系（目的別）

| 目的 | ツール候補 |
|------|----------|
| **モデルバージョン管理** | MLflow, DVC (Data Version Control), Weights & Biases (W&B) |
| **モデルパッケージング** | Docker, TensorFlow Serving, Seldon |
| **モデルレジストリ** | MLflow Model Registry, Amazon SageMaker Model Registry |
| **自動再訓練** | Kubeflow, Apache Airflow, Tecton |
| **モデル監視** | Prometheus, Grafana, Evidently AI, Neptune.ai |
| **ログ・デバッグ** | Prometheus, Grafana, ELK (Elasticsearch, Logstash, Kibana) |
| **MLワークステーション** | SageMaker, Kubeflow, DataRobot, Dataiku, Domino |
| **実験管理** | MLflow, Weights & Biases, Neptune.ai |
| **コンテナ・オーケストレーション** | Docker, Kubernetes |

#### プログラミング言語

- **Python**: 圧倒的に最も広く使われる（pandas/NumPy/TensorFlow/PyTorch/scikit-learn/Keras）
- **R**: 学術・統計分野での使用が主
- **Java/C++**: 特定の高パフォーマンス要件時

#### クラウドプラットフォームと選択基準

| プラットフォーム | 主な特徴 |
|---------------|---------|
| **AWS (Amazon SageMaker 等)** | 最も成熟した MLOps エコシステム |
| **Google Cloud Platform (Vertex AI 等)** | TensorFlow との深い統合 |
| **Microsoft Azure (Azure ML 等)** | エンタープライズ・Microsoft製品との統合 |

> **判断のポイント**: 既存組織のクラウド契約・チームの習熟度・予算制約が選択を左右する。クラウドプロバイダー選定後は、そのプラットフォーム固有のMLツールを優先的に評価する。

#### Generative AIのテックスタックへの統合

生成AIはMLテックスタック自体の効率化にも活用できる:

```
活用領域:
  □ 合成データ生成（トレーニングデータ不足の補完・エッジケーステスト）
  □ 探索的データ分析（生データから可視化・洞察・傾向を自動抽出）
  □ 特徴量エンジニアリング（高寄与特徴量の自動選定）
  □ コーディング支援（コード生成・デバッグ・リファクタリング）

注意: 生成AIはツールであり、人材を置き換えるものではない。
      AIを使いこなす人材が依然として必要。
```
