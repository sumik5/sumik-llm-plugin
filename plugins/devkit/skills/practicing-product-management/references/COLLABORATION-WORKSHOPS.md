# コラボレーション・ワークショップ実践ガイド

クロスファンクショナルな協働を具体化する5つのパターン。
カスタマージャーニーマッピング、フィッシュボーン分析、協働リチュアル、SLA交渉、エグゼクティブブリーフィングを体系化する。

---

## 1. カスタマージャーニーマッピング・ワークショップ

### 目的と参加者

複数部門（プロダクト・エンジニアリング・デザイン・オペレーション・カスタマーサポート）が一堂に会し、ユーザー体験の全体像を視覚化する。定性的エピソードと定量データ（サーベイ結果・サポートコール記録・ドロップオフ分析）を事前に準備して参加する。

### ジャーニーの7段階フレーム

水平タイムラインを以下の段階に分割する（例: 航空券予約後体験）:

| # | ステージ | 代表的タッチポイント |
|---|---------|-----------------|
| 1 | 予約確認 | メール通知・SMS・ロイヤルティ表示 |
| 2 | フライト前コミュニケーション | ゲートアサイン・天候アラート・ビザリマインダー |
| 3 | オンラインチェックイン | モバイルアプリ・手荷物計算機 |
| 4 | 座席選択・手荷物管理 | シートマップ・追加料金表示 |
| 5 | 空港到着・搭乗 | ボーディングパス・ラウンジアクセス |
| 6 | 機内体験 | Wi-Fi・食事サービス |
| 7 | 到着・フライト後サポート | 荷物追跡・払い戻し |

### スティッキーノートの色コーディング

| 色 | 意味 | 記入内容 |
|----|-----|---------|
| 黄 | 感情（ポジティブ）| 喜び・期待・安心感 |
| 赤 | ペインポイント | 混乱・フラストレーション・エラー |
| 緑 | 解決案 | 提案中の改善策・プロトタイプアイデア |

### ファシリテーション手順

1. **データ共有**（15分）: PMがサーベイ結果・サポートコール要約・ドロップオフ分析を提示
2. **ステージ別マッピング**（各段階20分）: 参加者が全員スティッキーを記入し感情・利用システムを書き出す
3. **ペインポイント特定**: 赤ノートを集約し、影響度の高いものにドット投票
4. **解決案ブレスト**: 上位ペインポイントに対して緑ノートで解決案を生成
5. **プロトタイプ優先順位付け**: RICE採点でネクストアクション確定

### ワークショップ成果物

- 全ステージを網羅したビジュアルマップ（Miro等に転記）
- ペインポイントランキング（ドット投票集計）
- 解決案バックログ（次スプリント候補リスト）

## 2. フィッシュボーン・ダイアグラム・ワークショップ

### 構造と参加者

**問題定義**: ホワイトボードの右端に問題を記載（例: "高いキャンセル率"）し、水平スパインを引く。4本のボーンに軸ラベルを付ける。

参加者: データエンジニア・PM・UXリサーチャー・オペレーションズ・サポートリード（各専門領域からの声が4軸を満たす）。

### 4軸構造

| 軸 | 代表的な原因例 |
|----|-------------|
| **People** | エージェント対応速度・スキルギャップ・研修不足 |
| **Process** | 払い戻しフロー・ポリシー文言の曖昧さ・エスカレーション手順 |
| **Technology** | モバイルSDKバグ・API障害・Retry Logic失敗 |
| **Environment** | 外部スケジュールAPI・天候アラート・規制変更 |

### 2時間アジェンダ（タイムボックス）

| フェーズ | 時間 | 内容 |
|--------|-----|------|
| 1. イントロダクション | 10分 | 問題のデータ提示（キャンセル率・早期スパイク等）。ファシリは「ブレームレス・好奇心重視」マインドセットを強調 |
| 2. カテゴリブレスト | 20分 | 各テーブルが1軸を担当。参加者はサイレントライティングで付箋を記入 |
| 3. ドット投票 | 10分 | 全員が高インパクトと判断する原因にドットを貼る。各軸トップ5が深掘り対象 |
| 4. ドリルダウン根本分析 | 40分 | 入選した原因にサブブランチを追加（例: Technology→Mobile SDK Bugs→Retry Logic Failure/Schema Mismatch） |
| 5. クロスブランチ依存関係 | 10分 | 軸間のつながりを矢印でマッピング（例: Environment:API障害 → People:サポートスパイク → Process:ポリシー文言ギャップ） |
| 6. アクションマッピング | 20分 | 各根本原因に対してミティゲーション案を1対1で紐づけ。担当チームと期限を決定 |

### ワークショップ後の成果物

- **デジタル化フィッシュボーン**: Miro等にアーカイブし全員に共有
- **高インパクト原因リスト（6件目安）**: 各原因に証拠ソース（ログ・ダッシュボード・ポリシー文書）を紐付け
- **担当アサイン**: Engineering=SDK修正、Product=ポリシー文言改善、Operations=APIフォールバック設計、Support=SLAコミットメント見直し
- **実行タイムライン**: クイックウィン（ポリシーコピー更新・UX例外パッチ）vs. 中期案件（APIリダンダンシー・SDKバージョン監査）の区分

## 3. クロスファンクショナル協働リチュアル

### デイリー・クロスチームスタンドアップ

サイロ型のチーム別ミーティングを廃止し、全機能から1名ずつ参加するシングルスタンドアップを実施。

**3点フォーマット**:
1. 昨日の進捗
2. 今日の計画作業
3. 浮上している課題・ブロッカー

5分タイマーを厳守し、行動アイテムはリアルタイムで共有ボードに記録する。
UIコンポーネントがAPIに依存している場合、サポートが見ているユーザビリティギャップ、こうした依存関係を即座に可視化できる。

### ウィークリー・バックログワークショップ

毎週2時間。目的はバックログ精査と優先順位付け。

**アジェンダ構成**:
1. キーメトリクスレビュー（予約完了率・セッション頻度等）でデータ起点に立つ
2. クロスファンクショナルで新規リクエスト・バグレポートを評価 → RICE採点・受け入れ基準を明確化
3. デザイナーがUXフロースケッチを仮想ホワイトボードに描く
4. エンジニアが技術工数を見積もる
5. オペレーションズが物流上の影響をフラグ立て

**バックログアイテムごとの議論は10分でタイムボックス**し、充実と効率のバランスを保つ。

### バイウィークリー・シンクセッション

**奇数回**: フィーチャーロードマップ回 — エンジニアリングがUXプロトタイプをデモし、サポート・オペレーションズからエッジケースや制約のフィードバックを受ける
**偶数回**: インフラ改善回 — 予定デプロイ（DBレプリカ・コンテナクラスター等）を共有し、デザイン・サポートが文書・カスタマーガイドを事前準備

### 共有アーティファクトの管理

| アーティファクト | 役割 |
|----------------|-----|
| 統合プロダクトロードマップ | 全機能の優先順位の単一ソース |
| コラボレーティブデザインライブラリ | UIコンポーネント・パターンの共有 |
| 優先順位付きバックログボード | スプリント計画の根拠 |
| デプロイカレンダー | リリース調整・影響予測 |

変更通知（イシュートラッカー更新・コードマージ）は関連ステークホルダーに自動アラート → メールノイズを削減。

### タイムボックスチェックポイントの原則

- アジェンダを24時間前に配布（目標・事前資料を明記）
- 全セレモニーは開始・終了時間を厳守
- 各リチュアルの最後に2分レトロ（「始めること」「止めること」を各1つ収集）

## 4. SLA交渉スクリプト

### 交渉前の準備

以下の3種類のデータを事前収集する:

1. **現状データ**: コアサービス（PNRステータス更新・手荷物追跡イベント・通知配信）のインシデント応答時間とアップタイム率のチャート
2. **ユーザーインパクト証拠**: カスタマージャーニーWSで特定した、システム遅延によるユーザーフラストレーション例
3. **業界SLA標準**: 現実的なターゲット設定のためのベースライン

### 3フェーズ構造

#### フェーズ1: オープニング

**目的**: 友好的な雰囲気の確立と目標合意

モデレーター発言例:
*「本日は乗客の方々が予約変更から手荷物追跡までシームレスな体験を受けられるよう、双方にとって現実的・測定可能なコミットメントを合意することを目指します」*

フォローアップ質問:
*「現在の標準応答時間と、ピーク負荷シナリオへの対応方法を教えていただけますか？」*

#### フェーズ2: エクスプロレーション

**目的**: 技術的実現可能性の深掘りとトレードオフの特定

発言例（数値ターゲット提示）:
*「PNRステータスクエリのP90レイテンシが現状350msのところ、200ms超で問い合わせが急増することがわかっています。P95で200ms以内を提案しますが、アーキテクチャ上の実現可能性はいかがでしょうか？」*

段階的合意のアプローチ:
*「段階的な展開でリスクを最小化するとしたら、タイムラインはどう見積もりますか？」*

#### フェーズ3: アグリーメント

**目的**: ターゲット・エスカレーションパス・レビュー方法の確定

合意例（数値コミットメント）:
- PNRステータスクエリ: P95レイテンシ ≤200ms（Q2目標）
- 手荷物追跡イベント: 平均伝播 ≤750ms（Q1末）→ ≤500ms（Q3末）
- 通知配信: 250ms以内で99.9%成功

**エスカレーション階層**:

| ティア | トリガー | アクション |
|-------|---------|----------|
| Tier-1 | 違反検知 | 5分以内にオンコールエンジニアへ自動アラート |
| Tier-2 | エラーバジェット10%超過 | オペレーションズマネージャーと日次レビュー |
| Tier-3 | エラーバジェット25%超過（月次） | エグゼクティブレビュー |

### 言語パターンの原則

ウルティマトムを避け、win-winフレーミングを一貫して使う:
- "we propose" / "together we can" / "our shared objective"
- 技術指標をビジネスインパクトに変換（例: 「レイテンシ150ms改善 → サポートコール20%削減 → 双方チームの工数余力が生まれる」）

## 5. エグゼクティブブリーフィング・アウトライン

### 使用場面

C-suiteが迅速な意思決定を求める場面（急激な性能劣化・トラフィックスパイク・重大インシデント等）。10分以内で根本原因・統計的信頼度・次のアクションを伝える6スライド構成。

### 6スライド構造

| スライド | タイトル | 記載内容 |
|--------|---------|---------|
| 1 | **Situation & Objective** | 何が起きているか（事実）と本ブリーフィングの目的 |
| 2 | **Problem Framing** | 問題の定義・スコープ・ビジネスへの影響 |
| 3 | **Methodology Snapshot** | 分析手法・信頼区間・サンプリング方法の簡潔な説明 |
| 4 | **Key Insights & Data** | 主要発見事項・チャート・統計的有意性 |
| 5 | **Risks, Sensitivity & Mitigations** | 感度分析・リスクシナリオ・ミティゲーション案 |
| 6 | **Recommendations & Action Plan** | 推奨アクション・オーナー・期限 |

### 作成原則

- **ナラティブドリブン**: データを先に見せるのではなく、状況→問題→証拠の順で語る
- **不確実性の明示化**: 信頼区間・上下限を意図的に示し、判断の根拠を透明化する
- **アクション可能性**: 各スライドの結論が次のアクションにつながるよう設計する
- **インフラ意思決定への接続**: 予測の上限値をサーバーオートスケーリング閾値に対応させる等、数値と運用判断を直結させる

### 準備チェックリスト

- [ ] ブリーフィング24時間前にアジェンダと事前読み資料を配布
- [ ] スライド1の「Situation」は1〜2文で完結させる
- [ ] チャートは1枚1メッセージ（視覚的ノイズ排除）
- [ ] スライド6のアクションプランはオーナー名と期限を明記
- [ ] Q&A時間を確保（ブリーフィングの25%目安）

## クイックリファレンス: ワークショップ選択ガイド

| 状況 | 適用ワークショップ |
|-----|-----------------|
| ユーザー体験の全体像を可視化したい | カスタマージャーニーマッピングWS |
| 問題の根本原因を多角的に特定したい | フィッシュボーンダイアグラムWS |
| チーム間の連携・情報共有を仕組み化したい | 協働リチュアル設計 |
| ベンダー・他部門とのSLA合意が必要 | SLA交渉スクリプト |
| 経営陣への迅速な状況報告が必要 | エグゼクティブブリーフィング |
