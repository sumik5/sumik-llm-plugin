## AI-native UXデザイン原則

### AI製品特有のUX課題

一般的なソフトウェアと異なり、AI製品のUXには独自の課題がある：

```
1. 不確実性（Uncertainty）の扱い
   - AIの出力は確率的——100%の確信はない
   - ユーザーに適切な信頼度を伝える方法が必要

2. 説明可能性（Explainability）
   - 「なぜこの推薦をしたのか」をユーザーが理解できるか
   - 特に高リスク意思決定（ローン・医療）での説明義務

3. エラーハンドリング
   - AIのミスはルールベースと異なり「奇妙」に見える
   - 失敗モードを予測し、ユーザーへの対話設計が必要

4. 継続的変化（Model Updates）
   - モデル更新でUXが突然変化する可能性
   - 変化についてユーザーをどう案内するか
```

### AI UXの7原則

1. **透明性（Transparency）**: AIを使っていることを開示する
2. **制御性（Control）**: ユーザーがAI推薦を上書きできる
3. **信頼性（Reliability）**: AIは一貫して期待通りに動く
4. **解釈可能性（Interpretability）**: なぜその結果になったか理解できる
5. **誠実性（Honesty）**: AIの限界・不確実性を適切に表示
6. **包括性（Inclusivity）**: 多様なユーザーグループに公平に機能する
7. **人間中心性（Human-Centered）**: AIはユーザーを補助し、代替しない

## AI製品のプロダクト設計要素101

出典: AI PM Handbook 2nd Ed. Ch.9

### 基本設計要素

```
エンドユーザー理解:
  □ AIに慣れているか（技術的リテラシー）
  □ AIへの信頼レベルはどの程度か
  □ 利用文脈（業務中・モバイル・緊急時）
  □ エラーが起きたときの許容度

問題定義（Problem Definition）:
  □ ユーザーが「解決されたと感じる」状態の定義
  □ AIなしではどう解決していたか（比較基準）
  □ AIが誤った時の影響度・リカバリー方法

実験（Experimentation）:
  □ AI機能のA/Bテスト計画
  □ AIと人間の比較評価（Human-in-the-Loop）
  □ ユーザーフィードバックループの設計

検証（Validation）:
  □ ユーザーテストでAI体験を検証
  □ エッジケースのシミュレーション
  □ 多様なユーザーグループでの検証

イテレーション（Iteration）:
  □ フィードバックをモデル改善に還元するループ
  □ スプリント単位での改善サイクル

ドキュメンテーション:
  □ AIの動作・限界・利用規約の明示
  □ AI利用に関する変更ログ
```

## AI-native プロダクトが特別な理由

出典: AI PM Handbook 2nd Ed. Ch.9「What makes the AI-native product design process special?」

### ユーザー執着（User Obsession）
AI製品は継続的なユーザーフィードバックがモデルを改善する——これは普通のソフトウェアにない「共進化」の関係だ。ユーザーの使用データが製品を良くし、良くなった製品がユーザーを引き込む。

### Machine Learning による差別化
```
AI-native製品が作れる体験:
  - パーソナライゼーション: 個人の行動パターンを学習
  - 予測: ユーザーニーズを先取り
  - 自動化: 繰り返し作業の排除
  - 発見: ユーザーが気づかなかった価値の提示
```

### 説明可能性のUX実装

```
低リスク AI機能（推薦・提案）:
  → 簡単な理由表示で十分
  例: 「あなたの過去の閲覧履歴に基づいた推薦」

中リスク AI機能（コンテンツモデレーション・スコアリング）:
  → 詳細な説明 + 不服申し立て機能
  例: 「このコンテンツは[理由]のためフラグを立てました。異議がある場合は...」

高リスク AI機能（融資・採用・医療）:
  → 完全な説明 + 人間レビューへのエスカレーション
  例: 「あなたの申請が承認されなかった主な理由は[A][B][C]です。担当者に相談するには...」
```

## HEARTフレームワーク詳細

出典: *The Resilient Product Manager* (Kaelin Rhun)
もともとGoogleが提唱したUX品質測定フレームワーク。

### GSM（Goals-Signals-Metrics）アプローチとの組み合わせ

```
各HEARTカテゴリに対してGSMを適用:

H - Happiness（幸福・満足）:
  Goal: ユーザーが製品に対して好意的な感情を持つ
  Signal: ポジティブな評価・低いクレーム率・高いNPS
  Metrics: NPS（Net Promoter Score）・CSAT・アプリストア評価

E - Engagement（関与）:
  Goal: ユーザーが製品を深く・頻繁に利用する
  Signal: セッション数の増加・機能利用の多様性
  Metrics: 週次アクティブ率・セッション時間・機能利用率

A - Adoption（採用）:
  Goal: 新ユーザー・新機能が定着する
  Signal: 初回利用から継続利用への移行
  Metrics: 新規機能の7日利用率・オンボーディング完了率

R - Retention（維持）:
  Goal: ユーザーが離脱せず継続利用する
  Signal: 解約率の低下・長期利用者の増加
  Metrics: 1ヶ月/3ヶ月/12ヶ月リテンション率・Churn Rate

T - Task Success（タスク成功）:
  Goal: ユーザーが目標タスクを達成できる
  Signal: エラーなしでのタスク完了・ヘルプ参照の減少
  Metrics: タスク完了率・エラー率・解決までの時間
```

## ユーザーリサーチ手法（AI製品向け）

### AI体験の検証に特に有効な手法

```
Wizard of Oz Testing:
  内容: AIの動作を人間が裏でシミュレート
  目的: 実際のAI構築前にユーザーニーズを検証
  適用場面: AI機能のコンセプト検証・早期フェーズ

Think-Aloud Protocol:
  内容: ユーザーがAI使用中に思考を口に出す
  目的: AI出力への反応・理解度・信頼感を把握
  適用場面: AIが生成するコンテンツのUX評価

Longitudinal Study:
  内容: 数週間〜数ヶ月かけてユーザーを追跡
  目的: AIへの慣れ・信頼の変化・使い方の進化を観察
  適用場面: AI学習機能（推薦精度の向上）の評価

Bias Detection Session:
  内容: 多様なユーザーグループによる体験テスト
  目的: 特定グループへの偏りや不公平な体験を発見
  適用場面: リリース前の公平性確認
```

## スプリント設計・クロスファンクショナル協働

出典: *The Resilient Product Manager*

### AIプロダクトのスプリント構造

```
2週間スプリントの典型構成:

Day 1-2: スプリントプランニング
  - バックログリファインメント（AI実験タスクを含む）
  - データチーム・MLエンジニア・PM・UXの全員参加
  - 前スプリントのモデル性能レビュー

Day 3-12: 実行フェーズ
  - デイリースタンドアップ（15分・進捗・ブロッカー）
  - 中間チェックポイント（Day 7）：実験の暫定結果確認
  - データドリフトの日次モニタリング

Day 13: スプリントレビュー
  - デモ（AI機能の実際のアウトプット含む）
  - ステークホルダーフィードバック収集
  - 指標レビュー（成功/失敗の評価）

Day 14: レトロスペクティブ
  - 実験から学んだこと
  - プロセス改善
  - 次スプリントへの知見転用
```

### AI製品のクロスファンクショナル協働

```
主要ステークホルダーとPMとの関係:

ML/Data Science チーム:
  接点: モデル性能・実験設計・評価メトリクス
  PMの役割: ビジネス目標をML指標に翻訳する「通訳者」

Engineering チーム:
  接点: システム設計・APIインターフェース・スケーラビリティ
  PMの役割: 技術負債とビジネス優先度のバランス調整

UX/Design チーム:
  接点: AI体験設計・ユーザーリサーチ
  PMの役割: ユーザーニーズとモデル能力のギャップを埋める

Legal/Compliance:
  接点: データ利用規約・AI規制対応
  PMの役割: 倫理要件を開発要件に組み込む

C-Suite/Stakeholders:
  接点: AI投資戦略・ROI説明責任
  PMの役割: 技術的成果をビジネス価値に変換して報告
```

## 優先度付けフレームワーク

### RICE スコアリング（The Resilient PM）

```
RICE = (Reach × Impact × Confidence) / Effort

Reach（リーチ）: 影響を受けるユーザー数（月次）
Impact（インパクト）: ゴールへの貢献度
  3 = 大きなインパクト
  2 = 中程度のインパクト
  1 = 最小限のインパクト
  0.5 = 微小なインパクト
Confidence（確信度）: 見積もりの確信度
  100% = 高い確信
  80% = 中程度の確信
  50% = 低い確信
Effort（工数）: 人月

AI機能へのRICE適用例:
  推薦エンジン改善: (50,000 × 2 × 0.8) / 3 = 26,667
  AI検索機能追加: (80,000 × 3 × 0.5) / 8 = 15,000
  → 推薦エンジン改善を優先
```

### 優先度付けの追加考慮事項（AI製品特有）

```
□ データ依存性: この機能に必要なデータは既に揃っているか
□ モデル成熟度: 十分な訓練データがあるか
□ 倫理リスク: バイアス・プライバシーリスクの水準
□ 保守工数: リリース後のMLOps・再訓練コスト
□ 規制リスク: 将来的な規制変更による影響度
```
