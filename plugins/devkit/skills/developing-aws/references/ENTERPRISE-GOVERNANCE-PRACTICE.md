## 1. アカウント管理の考え方

### 1.1 AWSアカウント vs IAMユーザー

AWSのアカウント管理は、**AWSアカウント**と**IAMユーザー**の大きく2つに分けられる。

| 概念 | 説明 | 備考 |
|------|------|------|
| **AWSアカウント** | AWSリソースの論理的な境界・請求単位 | メールアドレスで識別、ルートユーザーを持つ |
| **IAMユーザー** | アカウント内の操作権限を持つ個人識別子 | アカウントごとに作成が必要 |
| **IAMグループ** | IAMユーザーをまとめる論理グループ | グループにポリシーをアタッチして一括管理 |
| **IAMロール** | 一時的な権限付与・クロスアカウント連携に使用 | EC2インスタンスやLambdaに付与、STS経由で取得 |
| **IAMポリシー** | JSON形式で定義した権限のセット | Effect/Action/Resource で構成 |

### 1.2 クロスアカウントロール

一時的な権限付与やクロスアカウントアクセスは **IAMロール + STS (Security Token Service)** で実現する。

```
利用パターン:
1. アカウントA のIAMユーザーが、アカウントBのIAMロールをAssumeRole
2. STSから一時的な認証情報（アクセスキー・シークレットキー・セッショントークン）を取得
3. 一時的な認証情報でアカウントBのリソースを操作
```

| IAMロールの種類 | 用途 |
|--------------|------|
| **EC2インスタンスロール** | EC2上のアプリがAWSサービスをSDK経由で操作 |
| **Lambda実行ロール** | Lambda関数がAWSリソースにアクセス |
| **クロスアカウントロール** | 別アカウントのリソースに一時的にアクセス |
| **Identity Federation Role** | Cognito/SAML経由の外部ID連携 |

## 2. AWS Organizations

### 2.1 Organizations の構造

```
Organization（会社全体や部門単位）
  └── Root（最上位ノード）
        ├── Organization Unit (OU)  ← ディレクトリ/フォルダ相当
        │     ├── アカウント
        │     └── Organization Unit (入れ子可能)
        │           └── アカウント
        └── アカウント（OUに属さない直接配置も可）
```

| 要素 | 説明 |
|------|------|
| **Organization** | 請求をまとめる単位。マスターアカウント（=作成元）が子アカウントを管理 |
| **Root** | 木構造の最上位。全アカウントに制限をかけたい場合はここにSCPを設定 |
| **Organization Unit (OU)** | アカウントをグルーピングする論理単位。SCPはOUに設定すると配下全体に継承 |
| **アカウント** | 個別のAWSアカウント。ファイルシステムのファイルに相当 |

### 2.2 Organizations の機能

Organizations 作成時に機能を選択する（後から`ENABLE ALL FEATURES`へ変更可能だが逆は不可）:

| 機能 | `ENABLE ONLY CONSOLIDATED BILLING` | `ENABLE ALL FEATURES` |
|------|------------------------------------|-----------------------|
| 一括請求 | ○ | ○ |
| ボリュームディスカウント共有 | ○ | ○ |
| リザーブドインスタンス共有 | ○ | ○ |
| サービスコントロールポリシー (SCP) | × | ○ |
| アカウント作成（API経由） | × | ○ |

### 2.3 サービスコントロールポリシー (SCP)

SCP はアカウントが利用できる AWS サービスを制限するポリシー。**IAMポリシーより優先**される（SCPで拒否されたサービスはIAMで許可しても使えない）。

#### ホワイトリスト型 vs ブラックリスト型

| 形式 | Effect | 動作 | 使い分け |
|------|--------|------|---------|
| **ホワイトリスト型** | `Allow` | 明示したサービス・操作のみ許可 | 厳格な制限が必要なアカウント |
| **ブラックリスト型** | `Deny` | 明示した操作のみ拒否、それ以外は許可 | 特定操作だけ禁止したい場合 |

**⚠️ 重要**: Organization内でホワイトリストとブラックリストを混在させない（権限の把握が困難になる）

#### SCP の JSON 例（ホワイトリスト型）

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowOnlyEC2AndRDS",
      "Effect": "Allow",
      "Action": [
        "ec2:*",
        "rds:Describe*"
      ],
      "Resource": ["*"]
    }
  ]
}
```

> **注意**: SCPの `Resource` には `"*"` しか指定できない（IAMポリシーと異なる制約）。また `Condition` も利用不可。

#### SCP とIAMポリシーの違い

| 項目 | SCP | IAMポリシー |
|------|-----|------------|
| 対象 | アカウント・OU・Root | IAMユーザー・グループ・ロール |
| Resource指定 | `*` のみ | 個別リソースARN指定可能 |
| Condition | 使用不可 | 使用可能（IP制限等） |
| マスターアカウントへの適用 | 適用外 | 適用される |

## 3. AWS Organizations 導入手順

### 手順1: マスターアカウントでOrganization作成

1. マネジメントコンソール → AWS Organizations → `Create organization`
2. 機能を選択: **`ENABLE ALL FEATURES`** を推奨
3. Organization作成完了（マスターアカウントはこの時点で確定）

### 手順2: アカウントの設定

**既存アカウントを招待する場合:**
1. Organizations画面 → `Add account` → `Invite account`
2. 招待したいアカウントIDまたはメールアドレスを入力 → `Invite`
3. 招待されたアカウント側で承認（承認が必要）

**新規アカウントを作成する場合:**
1. Organizations画面 → `Add account` → `Create account`
2. アカウント名・メールアドレスを入力 → `Create`
3. 初回ログイン時にパスワードリマインダーが必要（パスワード未設定状態で作成される）

### 手順3: Organization Unit (OU) の作成

1. Organizations → `Organize accounts` タブ
2. `New organizational unit` をクリック
3. OUの名前を入力 → `Create organizational unit`
4. アカウントをOUに移動（ドラッグ&ドロップまたは操作メニュー）

### 手順4: SCP の作成とアタッチ

1. `Policies` タブ → `Service control policies` → `Create policy`
2. JSON形式でポリシーを定義（またはポリシジェネレータを使用）
3. `Organize accounts` タブ → アタッチしたいOU/Rootを選択
4. `Service control policies` → `Enable`（Rootの場合）→ 作成したポリシーを `Attach`

> **推奨**: 個別アカウントではなく**OUにアタッチ**する。新規アカウント追加時に自動適用されるため。

## 4. AWS Organizations ベストプラクティス

| # | ベストプラクティス | 理由 |
|---|----------------|------|
| 1 | **マスターアカウントでCloudTrailを有効化** | マスターアカウントへの不正操作（SCP変更、新規アカウント発行等）を記録する |
| 2 | **マスターアカウントはアカウント管理専用** | 強力な権限を持つため、ログインユーザーを最小化。他の用途は子アカウントに委譲 |
| 3 | **OUにSCPをアタッチ（個別アカウントへの直接アタッチを避ける）** | 新規アカウント追加時の自動適用、一括変更の容易性 |
| 4 | **ホワイトリストとブラックリストを混在させない** | 権限の把握が困難になるため、組織全体で形式を統一 |
| 5 | **RootへのSCPは最小限** | 将来アカウント数が増えた際、Rootポリシー変更の影響範囲把握が困難 |

## 5. 環境分離: アカウント分割のメリット・デメリット

### 5.1 アカウント分割のメリット

**運用面:**

| メリット | 説明 |
|---------|------|
| **誤操作リスクの排除** | 担当外システムのリソースが見えないため、誤った操作の可能性がなくなる |
| **シンプルな権限管理** | 「ECサイト担当者にはECサイトアカウントの権限のみ付与」という単純な設計で済む |
| **環境間の独立性** | 開発作業が本番環境に影響しない |

**コスト管理面:**

| メリット | 説明 |
|---------|------|
| **1アカウント1請求先** | アカウント単位でコスト把握が明確 |
| **ボリュームディスカウント** | Organizations内で使用量を合算してディスカウントを受けられる |
| **リザーブドインスタンス共有** | 利用されていないRIを同一Organization内の他アカウントに自動適用 |

### 5.2 アカウント分割のデメリット

**運用面:**

| デメリット | 対策 |
|----------|------|
| アカウントごとにIAMユーザー作成が必要 | CloudFormationによるIAMユーザー作成の自動化 |
| 複数アカウントのログイン管理が煩雑 | IAM Identity Center (旧AWS SSO) でシングルサインオン |

**コスト面:**

| デメリット | 説明 |
|----------|------|
| NATゲートウェイ等の重複コスト | 環境数×リソース数だけコストが増加 |
| サポート料金の増加 | アカウントごとにサポート最低料金（例: Business $100/月）が発生 |

### 5.3 推奨分割パターン

```
【推奨構成】
Organization
  └── 請求用アカウント（マスター）
  ├── ECサイト OU
  │     ├── ECサイト 本番アカウント
  │     ├── ECサイト 検証アカウント
  │     └── ECサイト 開発アカウント
  └── 分析基盤 OU
        ├── 分析基盤 本番アカウント
        ├── 分析基盤 検証アカウント
        └── 分析基盤 開発アカウント
```

**判断基準:**

- 特別なポリシーがない場合 → **システム・環境ごとにアカウントを分割**
- 権限管理の複雑さ > サポート料金増加コスト（人件費換算で検証せよ）

## 6. アカウント管理のロードマップ

企業内でAWSを初めて導入する際の推奨ステップ:

### Step 1: 管理主体と管理範囲を決定

| パターン | 適した状況 |
|---------|---------|
| クラウド担当部署が全社アカウントを管理 | 大企業・集中管理 |
| 事業部のインフラ担当者が部内を管理 | 中規模・事業部制 |
| プロジェクトのインフラ担当者が管理 | 小規模・スタートアップ |

→ **最低限**: アカウントがバラバラに無秩序に作成されないよう、管理者を定める

### Step 2: Organizationを管理する担当を決定

- マスターアカウントにログインできる担当者を限定
- その人がアカウント払い出し・権限付与の責任を持つ
- 誰でもマスターアカウントにログインできる状態はNG

### Step 3: 権限付与のルールを作成

例:
- 本番環境: 申請のあった機能のみ許可
- 開発環境: 基本的に全サービスを解放（要件確認のため）
- 法律準拠でNG なサービス: 全アカウントで禁止

### Step 4: OU構成設計を行う

1. **粒度の決定**: システムごとに親OU、その配下に環境ごとの子OUを作成
2. **SCP設計**: 親OUにシステム固有の制限、子OUに環境固有の制限を付与
3. **SCPの形式統一**: ホワイトリスト or ブラックリストを組織全体で統一

## 7. CloudWatch による監視

### 7.1 CloudWatch の概要

AWSリソースのメトリクスを収集・可視化・通知するための中心サービス。

| 種別 | 説明 | 設定方法 |
|------|------|---------|
| **標準メトリクス** | AWSが自動収集するメトリクス | 設定不要（EC2のCPU使用率等） |
| **カスタムメトリクス** | 利用者が独自に追加するメトリクス | スクリプト or CloudWatch Agent で送信 |

### 7.2 標準メトリクスでアラームを設定する

1. マネジメントコンソール → CloudWatch → `アラーム` → `アラームの作成`
2. メトリクスを選択: `EC2メトリクス` → `インスタンス別メトリクス` → `CPUUtilization` にチェック → 次へ
3. アラーム条件を設定:
   - しきい値: `CPUUtilization >= 60`
   - 期間: `5分間` / データポイント: `1`
   - 欠落データの処理: `見つかりません`
4. アクションを設定: SNSトピックを選択（通知先）
5. `アラームの作成` をクリック

### 7.3 カスタムメトリクスで監視する

**標準メトリクスで監視できない主要項目:**

| 監視項目 | CloudWatch標準 | カスタムメトリクス |
|---------|--------------|----------------|
| CPU使用率 | ○ | × |
| CPUロードアベレージ | × | ○ |
| **メモリ使用率** | × | ○ |
| **swap使用率** | × | ○ |
| **ディスク利用率** | × | ○ |

**カスタムメトリクス設定手順（Linuxインスタンス）:**

```bash
# 1. 必要パッケージをインストール
sudo yum install perl-Switch perl-DateTime perl-Sys-Syslog perl-LWP-Protocol-https

# 2. AWS公式スクリプトをダウンロード
curl http://aws-cloudwatch.s3.amazonaws.com/downloads/CloudWatchMonitoringScripts-1.2.1.zip -O
unzip CloudWatchMonitoringScripts-1.2.1.zip
rm CloudWatchMonitoringScripts-1.2.1.zip
cd aws-scripts-mon

# 3. メトリクスをCloudWatchに送信（テスト実行）
./mon-put-instance-data.pl --mem-used --mem-avail

# 4. cronに登録して定期実行
# crontab に追記: */5 * * * * /path/to/aws-scripts-mon/mon-put-instance-data.pl --mem-used --mem-avail
```

**必要なIAMロールポリシー:**

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "cloudwatch:PutMetricData",
        "cloudwatch:GetMetricStatistics",
        "cloudwatch:ListMetrics",
        "ec2:DescribeTags"
      ],
      "Resource": "*"
    }
  ]
}
```

### 7.4 AWS特有の監視項目

| メトリクス | サービス | 活用方法 |
|-----------|---------|---------|
| `NetworkIn` / `NetworkOut` | EC2 | ネットワーク帯域の使用状況。インスタンスタイプ変更の判断材料 |
| `ReadIOPS` / `WriteIOPS` | RDS, EBS | I/O負荷の確認。gp2は1GBあたり3IOPSの制限があるため、io1への移行判断に活用 |

## 8. CloudWatch Logs

### 8.1 CloudWatch Logs の機能

EC2インスタンス内のログを収集・監視するサービス。

| 機能 | 説明 |
|------|------|
| ログの蓄積・エクスポート | ログをAWS上に蓄積し、S3へエクスポート |
| フィルタリング・グラフ化 | 特定文字列でフィルタリング、メトリクスとして可視化 |
| SNSアラート | フィルタ条件に一致した場合にSNS通知 |
| KinesisやLambdaとのリアルタイム連携 | ストリーム処理・自動対応 |

### 8.2 エージェントのインストール

```bash
# Amazon Linux の場合（yum でインストール）
sudo yum install -y awslogs

# リージョン設定（東京リージョンの場合）
sudo vi /etc/awslogs/awscli.conf
# [plugins] → [default] の region を ap-northeast-1 に変更
```

> ⚠️ EC2インスタンスにIAMロールを割り当てておくこと。サーバ内ファイルにアクセスキーを記載するのはセキュリティリスクが高いため禁止。

**EC2に付与するIAMロールポリシー:**

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "logs:CreateLogGroup",
        "logs:CreateLogStream",
        "logs:PutLogEvents",
        "logs:DescribeLogStreams"
      ],
      "Resource": ["arn:aws:logs:*:*:*"]
    }
  ]
}
```

### 8.3 監視対象ログの設定

設定ファイル: `/etc/awslogs/awslogs.conf`

```ini
[/var/log/messages]
datetime_format = %b %d %H:%M:%S
file = /var/log/messages
buffer_duration = 5000
log_stream_name = {instance_id}
initial_position = start_of_file
log_group_name = /var/log/messages
```

| 設定項目 | 説明 |
|---------|------|
| `file` | 監視するログファイルのパス |
| `log_group_name` | CloudWatch Logsのロググループ名（ログの種類単位） |
| `log_stream_name` | ロGグストリーム名（インスタンス単位）。`{instance_id}` で動的設定可能 |
| `buffer_duration` | ログイベントのバッファ時間（ミリ秒） |

### 8.4 エージェントの起動

```bash
# 自動起動設定
sudo chkconfig awslogs on

# 起動確認
sudo chkconfig --list | grep awslogs
# awslogs 0:off 1:off 2:on 3:on 4:on 5:on 6:off

# エージェント起動
sudo service awslogs start
```

### 8.5 特定文字列の検知（メトリクスフィルタ）

1. CloudWatch → `ロググループ` 画面でラジオボタンを選択
2. `メトリクスフィルタの作成` をクリック
3. **フィルタパターン**を入力（例: `Error`）
4. テスト: サンプルログを入力して `パターンのテスト` → 一致を確認
5. `メトリクスの割当て` → フィルタ名・メトリクス名前空間・メトリクス名を入力 → `フィルタの作成`
6. CloudWatchアラームを設定（しきい値・SNS通知先）

**フィルタパターン例:**

| パターン | 検知内容 |
|---------|---------|
| `Error` | 「Error」という文字列を含むログ |
| `"INFO"` | 「INFO」という文字列 |
| `[host, logName, user, timestamp, request, statusCode=4, size]` | 400番台HTTPレスポンス |
| `{ $.errorCode = "AccessDenied" }` | JSONログのAccessDeniedエラー |

### 8.6 他サービスとの連携

```
EC2インスタンス
  → CloudWatch Logs エージェント
    → CloudWatch Logs
      → Lambda（リアルタイム処理）
      → Elasticsearch Service（ログ可視化）
      → Kinesis（ストリーム処理）
      → SNS（通知）
```

## 9. CloudWatch Events

### 9.1 CloudWatch Events の概要

AWSで発生するイベントをほぼリアルタイムに検知し、SNS/Lambda等と連携するサービス。

| 機能 | 説明 |
|------|------|
| **イベントパターン** | AWSリソースの状態変化を検知（EC2停止、RDS障害等） |
| **スケジュール** | cron式またはrate式で定期実行 |
| **イベントバス** | 別アカウントのターゲットにイベントを転送（マルチアカウントでの集約管理） |

### 9.2 EC2スケジュールイベント通知の設定手順

EC2インスタンスのメンテナンス（ホストサーバ障害・メンテナンス）をSNS経由で通知する設定:

1. CloudWatch → `イベント` → `ルールの作成`

2. **イベントソースの設定:**
   - 実行タイミング: `イベントパターン`
   - サービス名: `Health`
   - イベントタイプ: `特定のヘルスイベント`
   - Specific service: `EC2`
   - Specific event type category: `scheduledChange`

3. **ターゲットの設定:**
   - `ターゲットの追加` → `SNSトピック` を選択
   - 既存のSNSトピックを選択

4. **ルールの定義:**
   - 名前: 任意（例: `Ec2ScheduleChange`）
   - 説明: 用途を記載
   - 状態: `有効`
   - `ルールの作成`

### 9.3 イベントバスの活用

```
各AWSアカウント
  ├── アカウントA → イベントバス → 管理用アカウントのターゲット（Lambda/SNS）
  ├── アカウントB → イベントバス → 管理用アカウントのターゲット
  └── アカウントC → イベントバス → 管理用アカウントのターゲット
```

複数アカウントのイベントを管理用アカウントに集約することでITガバナンスを強化できる。

## 10. AWSサポートとサービス運用

### 10.1 AWSサポートプラン

| サポートプラン | 問い合わせ手段 | IAM対応 | 対応時間 | 最低費用（月間） |
|------------|------------|---------|---------|--------------|
| **ベーシック** | フォーラム・Web | × | ベストエフォート | $0 |
| **開発者** | Web | × | 平日9時〜18時 | $29〜 |
| **ビジネス** | Web・チャット・電話 | ○ | 24h/365d | $100〜 |
| **エンタープライズ** | Web・チャット・電話 + TAM | ○ | 24h/365d | $15,000〜 |

> **推奨**: プロダクションサービスを提供するアカウントはビジネスプラン以上

**緊急性と応答時間:**

| 緊急性 | 応答時間 | 対応プラン | 具体例 |
|-------|---------|----------|-------|
| 非常事態 | 15分 | エンタープライズ | 最優先対応 |

| 高（発生中の障害） | 4時間 | ビジネス・エンタープライズ | 重要機能の低下 |
| 普通（急ぎの問い合わせ） | 12時間 | 開発者・ビジネス・エンタープライズ | 致命的でない機能の異常 |
| 低（通常の問い合わせ） | 24時間 | 開発者・ビジネス・エンタープライズ | 一般的な質問・機能要望 |

## 11. Amazon Inspector（セキュリティ評価）

### 11.1 Inspector の評価範囲

| 評価項目 | 概要 | 対象OS |
|---------|------|-------|
| **共通脆弱性識別子 (CVE)** | 導入製品のバージョンからCVEの脆弱性を確認 | Linux / Windows |
| **セキュリティのベストプラクティス** | SSH設定・パスワードポリシー・メモリ保護設定・アクセス権限 | Linux |
| **実行時の動作の分析** | 暗号化されていないプロトコル使用・不要なポートのリッスン・不正実行対策 | Linux / Windows |
| **CISベンチマーク** | CIS（Center for Internet Security）が定めるセキュリティ推奨設定 | Linux（Amazon Linux 2015.03のみ） / Windows |

### 11.2 Inspector 実行手順

**事前準備:**

**1. Inspector実行用IAMロールの作成:**
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["ec2:DescribeInstances"],
      "Resource": ["*"]
    }
  ]
}
```
マネジメントコンソール → Inspector → `ロールの選択または作成` をクリック

**2. EC2インスタンスにタグを付加:**
- EC2ダッシュボード → `タグの管理`
- キー: `inspector-check`、値: `on` を追加

**3. EC2インスタンスへのエージェントインストール:**
```bash
curl -O https://d1wk0tztpsntt1.cloudfront.net/linux/latest/install
sudo bash install
sudo /etc/init.d/awsagent start
```

**Inspector設定（評価ターゲット・テンプレート定義）:**

1. Inspector → `今すぐ始める`
2. `評価ターゲットの定義`:
   - 名前: 任意
   - タグ: `inspector-check: on`
   - 続行（タグに一致するインスタンスの一覧が表示される）
3. `評価テンプレートの定義`:
   - 名前: 任意（例: `test-template`）
   - ルールパッケージ: `Common Vulnerabilities and Exposures-1.1`（他も選択可）
   - 所要時間: `1時間`（推奨）
4. `作成`

**実行と結果確認:**

1. 評価テンプレートを選択 → `実行` をクリック
2. 直前の実行列が「データを収集中」になることを確認
3. 完了後、`評価の実行` メニュー → Reports列のアイコンをクリック
4. レポート形式を選択（Findings report / Full report）、形式（HTML / PDF）→ `Generate report`
5. レポートの内容を確認して必要な対策を実行

> **ベストプラクティス**: システムリリース前・改修後に加え、定期的にInspectorを実行して継続的に安全な状態を維持

## 13. AWS CloudTrail

### 13.1 CloudTrailの概要

AWSサービスに対して実行された**全てのAPI操作**をログとして記録するサービス。

| 機能 | 説明 |
|------|------|
| 監査ログ取得 | 「誰が」「いつ」「どのAPIを」実行したかを記録 |
| 整合性検証 | ログファイルの改ざん防止・検証 |
| 暗号化 | KMS によるログファイル暗号化 |
| マルチアカウント集約 | 別アカウントのS3バケットにログを集約 |
| CloudWatch Logs連携 | 特定操作の通知・自動対応 |

### 13.2 CloudWatch Logs との連携ユースケース

| ユースケース | 説明 |
|-----------|------|
| **誤操作への気づき** | 東京リージョン以外でのEC2/RDS作成を検知・通知 |
| **MFA未導入IAMへの警告** | MFA認証されていないAPIコールを検知して設定を促す |
| **想定外サービス利用の検知** | 利用予定外のAWSサービスへのAPIコールを管理者に通知 |

### 13.3 CloudTrailの設定手順

1. マネジメントコンソール → CloudTrail → `今すぐ始める`

2. **証跡情報の作成:**
   - 証跡名: 任意（例: `CloudTrail-all-region`）
   - 証跡情報を全てのリージョンに適用: **`はい`** を選択
     （利用予定のないリージョンの操作も記録。将来の新リージョンにも自動適用）

3. **管理イベントの設定:**
   - 読み込み書き込みイベント: **`すべて`** を推奨
   - 変更操作と参照操作を分けたい場合: 複数の証跡を作成して「読み取り専用」「書き込み専用」を分割

4. **データイベントの設定:**
   - S3オブジェクトレベルのAPI操作ログ（GetObject/PutObject等）が必要な場合に設定
   - 対象バケットを指定、読み込み/書き込みを選択（追加料金が発生）

5. **ストレージの場所:**
   - 新しいS3バケットを作成（または既存を指定）
   - ログファイルを暗号化しますか: **`はい`**
   - ログファイルの検証を有効化しますか: **`はい`**
   - SNS通知: 任意（ログファイル配信のたびに通知が必要な場合は有効化）
   - `作成` をクリック

6. **ログの参照:**
   - `イベント履歴` から過去7日間のイベントを参照可能
   - フィルター: ユーザー名・イベント名・リソースタイプで絞り込み

## 14. AWS Config / Config Rules

### 14.1 Config の概要

AWSリソースの設定情報を記録し、変更を追跡・通知するサービス。

| 機能 | 説明 |
|------|------|
| **設定履歴の記録** | リソースの追加・変更・削除の履歴を保持 |
| **Config Rules** | リソースがルールに適合しているか自動チェック |
| **SNS通知** | 設定変更やルール違反時に通知 |
| **別アカウントへの集約** | ログ管理専用アカウントのS3バケットに集約可能 |

> ⚠️ Config Rulesはルール違反を検出・通知するが、**事前に不正操作を抑止しない**。抑止にはSCPを使用。

### 14.2 Config Rules の種類

| 種別 | 説明 |
|------|------|
| **マネージドルール** | AWSが事前定義したテンプレートルール（35種以上） |
| **カスタムルール** | Lambda関数として独自ルールを定義 |

**主なマネージドルール:**

| ルール名 | 内容 | 関連サービス |
|---------|------|------------|
| `ec2-volume-inuse-check` | EBSボリュームがEC2にアタッチされているか確認 | EC2 |
| `encrypted-volumes` | アタッチ済みEBSボリュームが暗号化されているか確認 | EC2 |
| `restricted-ssh` | セキュリティグループが制限なしSSHトラフィックを不許可にしているか | EC2 |
| `cloudwatch-alarm-action-check` | CloudWatchアラームにアクションが設定されているか | CloudWatch |
| `db-instance-backup-enabled` | RDSインスタンスのバックアップが有効か | RDS |
| `rds-multi-az-support` | RDSインスタンスの高可用性が有効か | RDS |
| `rds-storage-encrypted` | RDSインスタンスのストレージ暗号化が有効か | RDS |
| `iam-password-policy` | IAMユーザーのパスワードポリシーが要件を満たすか | IAM |
| `root-account-mfa-enabled` | ルートユーザーがMFAを必要とするか | IAM |
| `s3-bucket-logging-enabled` | S3バケットのログ記録が有効か | S3 |
| `cloudtrail-enabled` | CloudTrailが有効か | CloudTrail |
| `required-tags` | リソースに指定タグがあるか（例: CostCenter） | EC2/RDS/S3等 |

### 14.3 Config の設定手順

1. マネジメントコンソール → Config → `今すぐ始める`
   （Configはリージョン単位で設定が必要）

2. **記録するリソースタイプ:**
   - `すべてのリソース` → `このリージョンでサポートされているすべてのリソースを記録します` にチェック
   - グローバルリソース（IAM等）の記録も有効化

3. **S3バケットの設定:**
   - `バケットの作成`（新規）または既存バケット・別アカウントのバケットを選択
   - バケット名: 任意（例: `config-bucket-{accountid}`）
   - 別アカウントのバケット指定により、ログ管理専用アカウントに集約可能

4. **SNSの設定:**
   - 設定変更の通知が必要な場合は `Amazon SNSトピックへのストリーム設定の変更と通知` にチェック

5. **Config ロールの設定:**
   - `ロールを作成する` を選択（推奨）→ 必要最低限の権限が自動付与
   - ロール名: 任意（例: `config-role-ap-northeast-1`）

6. **ルールの設定（手順2）:**
   - 初回設定時はスキップ可能（後からルールを追加できる）

7. **確認 → `確認` ボタンで決定**

### 14.4 Config Rules の作成

**マネージドルールの追加:**
1. Config → `ルール` → `ルールの追加`
2. ルール名・ラベル・説明でフィルタリング
3. 適用したいルールを選択

**カスタムルールの作成:**
1. Lambda関数を作成（変更内容がルールに適合しているかをYes/Noで返す）
2. Config → `ルール` → `ルールの追加` → `カスタムルール` を選択
3. Lambdaファイルの ARN を指定

### 14.5 Config と CloudTrail の連携

| 役割 | サービス |
|------|---------|
| 変更内容・変更履歴の記録 | **Config** |
| 「誰が」「いつ」「どのAPI操作を」行ったかの追跡 | **CloudTrail** |

Configのタイムライン画面から関連するCloudTrailイベントを直接参照でき、操作した IAMユーザーや操作内容を特定できる。

### 14.6 Config 履歴の確認

1. Config → `リソース`
2. リソースで `EC2:Instance` を選択 → `検索`
3. 一覧から確認したいインスタンスのConfig タイムラインアイコンをクリック
4. 変更前後の設定差分（例: `Configuration.State.Name: stopped → running`）を確認
5. 同画面の `CloudTrailイベント` タブで操作者・イベント名を確認
