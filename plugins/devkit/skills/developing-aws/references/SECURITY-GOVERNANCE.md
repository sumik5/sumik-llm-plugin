# Security Governance on AWS

> **関連**: AWS技術セキュリティ（IAM/VPC/KMS実装）は [SECURITY.md](./SECURITY.md)、エンタープライズセキュリティは [SECURITY-ADVANCED.md](./SECURITY-ADVANCED.md)、検知・対応は [SECURITY-DETECTION-RESPONSE.md](./SECURITY-DETECTION-RESPONSE.md) を参照

セキュリティガバナンスの基礎概念・フレームワーク・方法論ガイド。「WHY（なぜセキュリティが必要か）」と「WHAT（何をするべきか）」を中心に、リスクアセスメントとセキュリティポリシー策定まで体系的にカバーする。

## 1. セキュリティの基本思考法

### 1.1 WHY / WHAT / HOW 思考法

セキュリティを学ぶ上で最も重要なのが、この3段階の思考法。技術（HOW）だけを追いかけると目的を見失う。

| 思考レベル | 問い | セキュリティにおける意味 |
|-----------|------|----------------------|
| **WHY** | なぜそのセキュリティが必要か | ビジネス目的・リスク・規制要件の理解 |
| **WHAT** | 何を守るべきか | 情報資産・サービス・データフローの識別 |
| **HOW** | どのように実現するか | 技術実装・管理策・AWSサービス選定 |

> **ポイント**: AWSのサービスはお客様のフィードバックから生まれる。新サービスは「どんな課題をどう解決するか」という観点で理解すると定着が早い。

### 1.2 セキュリティ業務の特徴

セキュリティはサービスの中心ではなく、**サービスの価値を損なうリスクを低くするもの**。

- **サービスの中心ではない**: 主目的（売上・ユーザー体験等）の外縁を守る役割
- **使い勝手とのトレードオフ**: 多要素認証等はセキュリティを高めるが操作手間も増やす
- **見えにくい投資効果**: 守られていて当然、事故があって初めて注目される構造
- **終わりのない活動**: 攻撃の高度化に合わせて守る側も進化し続けなければならない

### 1.3 IT セキュリティの3つの課題

| 課題 | 内容 |
|------|------|
| **理解** | セキュリティ人材不足。クラウド×セキュリティを横断理解できる人材は稀 |
| **負荷** | パッチ適用・監視・バックアップ等の運用負荷が大きく、継続が難しい |
| **リソース配分** | セキュリティへの投資対効果が見えにくく、経営資源が配分されにくい |

## 2. 責任共有モデルの深掘り

### 2.1 説明責任 vs 実行責任

責任には2種類あり、この区別がクラウド活用の本質を理解する鍵。

| 責任の種類 | 英語 | 対応する思考 | 特徴 |
|-----------|------|------------|------|
| **説明責任** | Accountability | WHY / WHAT | アウトソーシング不可。「なぜ任せたか」「どこまで任せたか」「どう監督したか」を常に問われる |
| **実行責任** | Responsibility | HOW | アウトソーシング可能。専門知識を持つ外部組織に委任できる |

> **重要**: クラウドに移行しても説明責任は利用者に残る。「AWSに任せているから知りません」は通用しない。

### 2.2 責任共有モデル（Shared Responsibility Model）

```
┌─────────────────────────────────────────────────┐
│ お客様の責任（Security IN the Cloud）           │
│  ・お客様のデータ                               │
│  ・プラットフォーム・アプリケーション          │
│  ・ID・アクセス管理                             │
│  ・OS・ネットワーク・ファイアウォール構成       │
│  ・クライアント側データ暗号化                   │
│  ・ネットワークトラフィック保護                 │
├─────────────────────────────────────────────────┤
│ AWSの責任（Security OF the Cloud）             │
│  ・コンピュート・ストレージ・DB・ネットワーク  │
│  ・ハードウェア・AWSグローバルインフラ          │
│  ・リージョン・AZ・エッジロケーション          │
└─────────────────────────────────────────────────┘
```

### 2.3 サービスレイヤー別の責任パターン

| サービスモデル | AWS例 | AWSの責任 | 利用者の責任 |
|--------------|-------|----------|------------|
| **IaaSモデル** | EC2 | 物理・仮想化基盤 | OS・MW・アプリ・ネットワーク設定 |
| **コンテナサービス** | RDS・ECS | 物理・OS・プラットフォーム | データ・アクセス制御・暗号化設定 |
| **アブストラクトサービス** | S3・DynamoDB | 物理〜プラットフォーム全体 | データ・アクセス制御・暗号化の可否 |

> **設計原則**: 抽象度が高いサービスほど利用者の管理負荷は減る。運用負荷を削減した分を高度な監視や自動化に充てることでセキュリティレベルが上がる。

### 2.4 Security の3分類

| 分類 | 説明 | 例 |
|------|------|-----|
| **Security 'of' the Cloud** | AWSインフラ自体のセキュリティ | データセンター物理セキュリティ、仮想化基盤管理 |
| **Security 'in' the Cloud** | 利用者がクラウド上で構築するシステムのセキュリティ | IAM設定、VPC設計、アプリセキュリティ |
| **Security 'by' the Cloud** | クラウドサービスを活用したセキュリティ強化 | マネージドWAF、GuardDuty、Shield |

## 3. CIA三要素（セキュリティの基本要件）

ISO/IEC27000で定義される情報セキュリティの3要素。

### 3.1 各要素の設計指針

| 要素 | 英語 | 定義 | 設計指針 |
|------|------|------|---------|
| **機密性** | Confidentiality | 認可されていない個人・エンティティ・プロセスに情報を使用・開示しない特性 | 認証・認可・暗号化・アクセスコントロール・物理的入退室管理 |
| **完全性** | Integrity | 正確さおよび完全さの特性 | 改竄防止・電子署名・バージョン管理・変更監査ログ |
| **可用性** | Availability | 認可されたエンティティが要求したときにアクセス・使用可能である特性 | 冗長化・バックアップ・BCP・DDoS対策・Auto Scaling |

### 3.2 ビジネス観点での優先順位

セキュリティ検討はCではなくAから始めることが実践的。

```
A（可用性）→ I（完全性）→ C（機密性）
```

- **可用性**: サービスが使えることがビジネスの前提条件。止まれば売上・信頼が直結して失われる
- **完全性**: 間違いのない情報が届けられることで、可用性に意味が生まれる
- **機密性**: 正しい人に必要な情報だけが届く状態が最終目標

> **落とし穴**: 「情報を全て金庫に入れる」と機密性は最大化されるが可用性がゼロになる。3要素のバランスが重要。

## 4. セキュリティフレームワーク

### 4.1 NIST CSF（Cyber Security Framework）

米国NIST（国立標準技術研究所）が策定した、組織のセキュリティ対応能力を継続的に改善するためのフレームワーク。予防だけでなく**事後対応**にも重点を置いている点が特徴。

```
識別（Identify）→ 防御（Protect）→ 検知（Detect）→ 対応（Respond）→ 復旧（Recover）
      ↑                                                                          │
      └──────────────────── 継続的な改善サイクル ─────────────────────────────┘
```

| フェーズ | 内容 | 本書での扱い |
|---------|------|------------|
| **識別** | 守るべき資産の把握・リスク評価 | Chapter 6（本リファレンスの主テーマ） |
| **防御** | リスクを低減する管理策の実装 | Chapter 7（SECURITY.md参照） |
| **検知** | 異常なアクティビティの発見 | Chapter 8（SECURITY-DETECTION-RESPONSE.md参照） |
| **対応** | インシデント発生時の対応 | Chapter 9（SECURITY-DETECTION-RESPONSE.md参照） |
| **復旧** | 元の状態への速やかな復帰 | Chapter 9（SECURITY-DETECTION-RESPONSE.md参照） |

> **ISMS（ISO/IEC27001）との違い**: ISMSは予防的管理策中心。CSFは高度化するサイバー攻撃を踏まえ「防げなかった場合の検知・対応・復旧」にも注力している。

### 4.2 ISO/IEC 27001（ISMS）

情報セキュリティマネジメントシステムの国際規格。日本では特に広く採用されている。

| 特徴 | 内容 |
|------|------|
| **目的** | セキュリティポリシー策定・管理体制構築のための網羅的フレームワーク |
| **構造** | トップダウン（方針→戦略→戦術）とボトムアップ（実運用要件→設計）のバランス |
| **認証制度** | B2B取引等での信頼証明として活用。認証未取得でもテンプレートとして参照可能 |
| **AWS対応** | AWS Artifactから第三者監査レポートが取得可能（SOC2・ISO認証等） |

## 5. セキュリティ原則

### 5.1 主要なセキュリティ原則一覧

| 原則 | 内容 | AWS実装例 |
|------|------|---------|
| **Need to Know** | 必要な人・組織にのみ情報を公開する | IAM最小権限ポリシー、S3バケットポリシー |
| **Authentication & Authorization** | 認証（誰か）と認可（何ができるか）を分離して管理 | IAM、Cognito、Federation |
| **Least Privilege（最小権限）** | 必要最低限の権限のみ付与する | IAMポリシー設計、SCPs |
| **Defense in Depth（多層防御）** | 複数のレイヤーで独立したセキュリティ管理策を施す | VPC+SG+NACL+WAF+アプリ認証 |
| **Fail Safe（フェイルセーフ）** | 障害時にはデフォルトで安全側に倒す | SG deny-all-by-default |
| **Separation of Duties** | 一人に全権限を持たせず役割を分担する | IAMグループ・ロール分離 |
| **Three Lines Defence** | 現場・管理部門・内部監査の3層でリスク管理する | 開発者・セキュリティチーム・監査の役割分担 |
| **Security by Design** | 設計段階からセキュリティを組み込む | DevSecOps、CDKセキュリティコンストラクト |

### 5.2 ゲートキーパー型 vs ガードレール型

AWSが推奨するのはガードレール型のアプローチ。

| タイプ | 説明 | 問題点/メリット |
|--------|------|--------------|
| **ゲートキーパー型** | 定期的に業務を止めてセキュリティレビューを行う | 開発速度を損ない、ビジネスのアジリティを妨げる |
| **ガードレール型** | 逸脱が起きない仕組みを組み込み、逸脱を自動検知する | 開発の柔軟性を許容しつつ継続的なセキュリティを実現 |

## 6. セキュリティポリシー策定

### 6.1 文書体系の構造

```
ポリシー（Policy）     → WHY  → 志・ミッション・組織がセキュリティに取り組む目的
スタンダード（Standard）→ WHAT → 戦略・守るべき基準・管理項目
プロシージャ（Procedure）→ HOW → 戦術・具体的な実施手順・チェックリスト
```

### 6.2 策定プロセスのポイント

**トップダウン × ボトムアップのバランス**

| アプローチ | 内容 | 注意点 |
|-----------|------|--------|
| **トップダウン** | 組織のあるべき論・目標から管理策を導く | 現場の実態と乖離したポリシーになりやすい |
| **ボトムアップ** | 実装・実運用面の要件から定義する | 全体整合性が取れないことがある |

### 6.3 セキュリティポリシーの落とし穴

| 落とし穴 | 内容 | 対策 |
|---------|------|------|
| **形骸化** | ルールが増えすぎて目的を忘れ「ルールだからやる」状態になる | 「何のためか」を常に問い続ける原則中心アプローチ |
| **陳腐化** | 環境変化に合わせてポリシーが更新されない | 定期的なレビューサイクルの設定 |
| **先例主義** | 事例がないと採用しない保守的姿勢が新技術の活用を阻む | 小さく試す・アジャイル的アプローチ |
| **過剰規制** | 詳細すぎるルールが変化への障壁になる | 原則ベースで記述し、実装の選択余地を残す |

### 6.4 効果的なポリシー運用の3原則

1. **原則中心のアプローチ**: 細かいルールより「何を実現したいか」という原則を軸にする
2. **小さく何度も試す**: アジャイル的に経験から学び、サービスに反映させる
3. **一緒に考える（DevSecOps）**: 開発・運用・セキュリティ・監査が協力して進める

## 7. リスクアセスメント方法論

### 7.1 リスクの定義

ISO/IEC27005による定義:

> 資産が抱える脆弱性を狙う攻撃などの**脅威**により、組織に対して**悪影響が生じる可能性**

| 要素 | 説明 |
|------|------|
| **脅威** | DDoS・不正アクセス等、セキュリティを脅かす事象 |
| **脆弱性** | リスクが顕在化する原因。セキュリティ上の欠陥・設定不備 |
| **資産** | 保護すべき情報・システム・ビジネス価値 |
| **リスク** | 脅威 × 脆弱性 × 資産重要度の組み合わせで評価 |

### 7.2 リスクアセスメントのアプローチ比較

| | ベースラインアプローチ | リスクベースアプローチ |
|---|---|---|
| **概要** | 定評あるフレームワーク等のベースラインに基づき管理策を選定 | 資産ごとに脅威・脆弱性を評価しリスクを分析 |
| **難易度** | 易しい | 難しい |
| **期間** | 短い | 長い |
| **適切さ** | ミスマッチの可能性あり | マッチする |
| **用途** | 複数システム共通のセキュリティガイドライン・監査項目 | システム固有のセキュリティ要件・設計 |

> **現在のトレンド**: ベースラインアプローチは共通基準の点検に限定し、システム固有の管理策はリスクベースアプローチで検討する組み合わせが主流。

### 7.3 情報資産の識別と分類

**情報資産の重要度分類基準（例）**

| 重要度 | 分類基準 | データ例 | システム例 |
|-------|---------|---------|----------|
| **高** | 情報漏洩・サービス停止が組織・個人に致命的/壊滅的な悪影響 | 個人の病歴情報・機密設計情報 | 医療情報システム・決済基盤 |
| **中** | 情報漏洩・サービス停止が重大な悪影響 | 従業員の役職情報・顧客データ | 営業支援システム・ECサイト |
| **低** | 情報漏洩・サービス停止が限定的な悪影響 | 従業員数・公開情報 | 社内情報共有システム |

分類の観点: 機密性・完全性・可用性それぞれの喪失時の影響を評価する。

### 7.4 データフローによる保護範囲の把握

データフロー図（DFD）はセキュリティ設計の基本ツール。

**DFDの活用目的**
- 保護すべきITリソース・通信経路の明確化
- 想定される攻撃拠点の特定
- 信頼境界線（Trust Boundary）の可視化

**信頼境界線のポイント**
- 境界線をまたぐ通信はセキュリティ管理策が必要
- 境界線が引かれていない箇所はセキュリティ検討が不十分なサイン
- 境界線の内側がすべて安全とは限らない（ゼロトラストの考え方）
- 多層防御: レイヤーごとに独立した管理策を施す

### 7.5 リスク分析の2アプローチ

#### 資産ベースのリスク分析

保護対象の情報資産ごとにリスクを評価する手法。

```
資産重要度確認 → 脅威の確認 → 現状確認（脆弱性レベル）→ リスク値算出

リスク値 = （脅威レベル × 脆弱性レベル） × 資産重要度
```

**評価基準例（各3段階）**

| 指標 | レベル3 | レベル2 | レベル1 |
|------|---------|---------|---------|
| **脅威レベル** | 悪意ある第三者が主体 | 悪意ある内部関係者 | 過失による内部関係者 |
| **脆弱性レベル** | 防御対策を未実施 | 部分的に実施 | 網羅的に実施 |
| **資産重要度** | 長期停止・巨額損失リスク | 一定時間停止・ある程度の損失 | 短時間停止・少額の損失 |

**メリット**: 資産ごとに効率的な管理策検討が可能
**デメリット**: システム全体観点での優先度判断がしにくい

#### 攻撃シナリオベースのリスク分析（STRIDEモデル活用）

攻撃者視点で攻撃シナリオ・攻撃ルート・攻撃ツリーを整理する手法。

**STRIDEによる脅威分類**

| 脅威 | 説明 | 対策方向 |
|------|------|---------|
| **S**poofing（なりすまし） | 他者になりすましてアクセス | 認証強化・MFA |
| **T**ampering（改竄） | データやシステムの不正変更 | 完全性チェック・署名 |
| **R**epudiation（否認） | 「やっていない」という証拠隠滅 | 監査ログ・CloudTrail |
| **I**nformation disclosure（情報漏洩） | 意図しない情報公開 | 暗号化・アクセス制御 |
| **D**enial of Service（DoS） | サービス停止 | Shield・スロットリング |
| **E**levation of privilege（権限昇格） | 不正に高い権限の取得 | 最小権限・SCPs |

```
攻撃シナリオ選定 → 被害レベル確認 → 現状確認 → リスク値算出

リスク値 = （脅威レベル × 脆弱性レベル） × 被害レベル
```

**メリット**: 多層防御の観点で管理策を策定でき、攻撃シナリオへの対応優先度が明確
**デメリット**: 攻撃シナリオの網羅性に依存、検討量が多く時間がかかる

### 7.6 クラウド固有のリスク（ENISA準拠）

ENISA（欧州ネットワークセキュリティ庁）が指摘するクラウド特有の考慮リスク：

| リスク | 内容 | AWS対策 |
|--------|------|---------|
| **責任分界の不十分な理解** | 責任共有モデルを理解せず対応漏れが発生 | 本ガイドの責任共有モデル理解 |
| **管理インターフェイスへの不正アクセス** | Console・CLI・APIへの不正ログイン | MFA・IAM設計・CloudTrail |
| **マルチテナント隔離の失敗** | 仮想化の隔離不備による情報漏洩 | AWSが責任を負う（ホワイトペーパー参照） |
| **EDoS攻撃** | 従量課金を悪用したDDoSによる経済的損失 | Shield・コスト監視・バジェットアラート |
| **クラウド事業者の不適切な管理** | 特権悪用・データ消去不備・暗号鍵消失 | AWS Artifactで第三者監査レポート確認 |

### 7.7 リスクへの対応方法

| 対応方法 | 内容 | 例 |
|---------|------|-----|
| **低減** | 管理策でリスクの発生可能性・影響度を下げる | 暗号化・MFA・パッチ適用 |
| **移転** | 保険や契約でリスクを第三者に移す | 責任共有モデル・サイバー保険 |
| **回避** | リスク原因となる活動・資産をやめる | 不要なサービスの廃止・データ削除 |
| **受容** | リスクを認識したうえで許容する | コストと見合わない低リスクへの対応見送り |

## 8. AWS利用開始時のセキュリティ初期設定

### 8.1 AWSユーザーの種類と役割

| ユーザー種別 | 特徴 | 推奨利用場面 |
|------------|------|------------|
| **ルートユーザー** | AWSアカウント全リソースへの全権限。メールアドレス+パスワードでサインイン | 初期IAMユーザー作成・ルートユーザー専用タスクのみ |
| **IAMユーザー** | 必要な権限のみ付与可能。1人に1つが原則 | 日常的なAWS操作全般 |
| **IAMロール** | 一時的な権限委任。人間だけでなくサービス（Lambda等）も使用 | クロスアカウント・EC2/Lambda実行権限 |

**ルートユーザーにしかできない主な操作**:
- AWSアカウント設定変更（メールアドレス・連絡先情報）
- IAM管理者ユーザーが自身のアクセス許可を誤って削除した場合の復元
- AWSサポートプラン変更
- アカウントのクローズ

### 8.2 ルートユーザー保護チェックリスト

```
[ ] 多要素認証（MFA）の有効化
    └─ 認証アプリケーション（推奨）/ セキュリティキー / ハードウェアTOTPトークン
[ ] アクセスキーの削除（デフォルトでは存在しないが要確認）
[ ] 強力なパスワードの設定
[ ] ルートユーザーの日常業務での使用禁止
```

**MFAの種類**

| 種類 | 方法 | 推奨度 |
|------|------|--------|
| **認証アプリケーション（仮想MFA）** | スマホアプリで6桁OTPを生成 | ◎ 最も利用しやすい |
| **セキュリティキー** | YubiKey等のFIDOデバイスにタッチ | ◎ フィッシング耐性が高い |
| **ハードウェアTOTPトークン** | 専用ハードウェアデバイス | △ 管理コストが高い |

### 8.3 IAM初期設定のベストプラクティス

**IAMグループとユーザーの設計原則**

```
1. IAMグループを作成（役割別: Administrators, Developers, Auditors等）
2. グループにIAMポリシーをアタッチ
3. IAMユーザーをグループに所属させる（ポリシーの直接アタッチは避ける）
4. 1人につき1つのIAMユーザーを作成
5. 各IAMユーザーにもMFAを設定
```

**IAMポリシーの種類**

| 種類 | 説明 | 用途 |
|------|------|------|
| **AWS管理ポリシー** | AWSが提供するプリセットポリシー | `AdministratorAccess`・`ReadOnlyAccess`等 |
| **カスタムポリシー** | 組織が独自に定義するポリシー | サービス固有の最小権限設計 |
| **インラインポリシー** | ユーザー・グループ・ロールに直接埋め込む | 例外的な個別設定（原則避ける） |

### 8.4 AWS Trusted Advisor の活用

AWSアカウントのベストプラクティス準拠状況を自動チェックするサービス。

**セキュリティカテゴリの主要チェック項目**

| チェック項目 | 内容 |
|------------|------|
| ルートアカウントMFA | MFAが未設定の場合に警告 |
| IAMの使用 | IAMユーザーが1つも存在しない場合に警告 |
| アクセスキーのローテーション | 90日以上ローテーションされていないキーを警告 |
| S3バケットのパブリックアクセス | 意図しないパブリック設定を検出 |
| セキュリティグループの開放 | 0.0.0.0/0での危険なポート開放を警告 |

## 9. ガバナンスとコンプライアンス

### 9.1 セキュリティガバナンスの役割

| 概念 | 意味 | セキュリティとの関係 |
|------|------|------------------|
| **ガバナンス** | 組織が健全な経営を行うための機能・プロセス | 説明責任を果たし、不正行為を未然に防ぐ管理・監督体制 |
| **コンプライアンス** | 法令・社会規範・組織自身のミッションへの準拠 | セキュリティポリシーの遵守・第三者への説明 |
| **ITガバナンス** | IT活用を適切に管理・監督すること | セキュリティはITガバナンスを健全に保つための仕組み |

### 9.2 IT×ガバナンスの関係

**ITがガバナンスを強化する理由**:
- 業務を自動化することで人為的ミスや不正の機会を減らす
- ログ記録により業務の透明性・監査可能性が向上する
- 統制（Internal Control）を効率的に実現できる

**IT自体のリスク管理（ITガバナンス）の必要性**:
- プログラムのバグが大規模な誤出力を引き起こす可能性
- ログが適切に管理されていないと事実の改竄・追跡不能が発生
- セキュリティへの過剰投資は経営上不健全（コスト対効果の意識）
