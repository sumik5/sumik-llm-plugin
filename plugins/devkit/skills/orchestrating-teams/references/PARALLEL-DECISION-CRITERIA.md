# 並列実行判断基準（共通リファレンス）

> **正本ファイル。** `orchestrating-codex` スキルはこのファイルを参照する。
> このファイルを編集した場合は両スキルの INSTRUCTIONS.md との整合性を確認すること。

---

## 使用タイミング（ユーザー要求のテキストから即座に判断）

**🔴 ファイルを読んで判断しない。ユーザーの要求文が以下に該当しそうなら即座にplanner（製品マネージャーagent）を起動:**

1. **複数の機能・コンポーネント** に言及している（例: 「UIとAPIを作って」）
2. **異なる関心事** が含まれる（例: 「フロントエンドとバックエンドを変更」）
3. **複数のサブタスク** が明示的または暗示的に含まれる
4. **「〜を追加して」＋「テストも書いて」** のような複合要求

**以下の場合のみ単体起動:**
- 明らかに1ファイルのみの変更（「このファイルのバグを直して」）
- 単一の小さなタスク（「typoを修正して」）

---

## ファイル所有権パターン（必須ルール）

**並列agent / タチコマが同一ファイルに書き込むと競合する。plannerが排他的所有権を厳密に定義すること。**

plannerが事前にパスベースの所有権を定義（同一並列グループで重複禁止）:

```
frontend: src/components/**, src/pages/**
backend:  src/api/**, src/services/**, src/models/**
test:     tests/e2e/**, tests/integration/**
architect: docs/design/**
```

- **各agent / タチコマは自身の所有範囲外のファイルを編集しない。**
- 共有ファイル（型定義・設定ファイル等）は先行グループ（Wave / Phase）に配置するか、1つのagentに所有権を集約する。
- 競合が予想される場合はタスクを順次実行に変更する。

## タスク分解の共通原則

| 指標 | orchestrating-teams | orchestrating-codex |
|------|---------------------|---------------------|
| 1担当あたりの目標タスク数 | **5–6タスク**（実証済みの生産性最適値） | **3–5タスク**（並列度を上げるためスコープを絞る） |
| 多すぎる場合 | メンバーのオーバーヘッド増加（8+） | agentのコンテキスト過負荷で後続Waveをブロック（6+） |
| 少なすぎる場合 | 待機時間の増加（1–2） | agent起動コストが相対的に大きい（1） |

> スキル固有の差分（teams: `blockedBy` フィールド / codex: Wave分割テクニック）は各 INSTRUCTIONS.md を参照。

## 依存関係に基づくグループ化

```
独立タスク群  →  同一並列グループ（同時起動）
      ↓ 全完了待ち
次の依存タスク群  →  次の並列グループ
```

### 判断基準

| 条件 | グループ配置 |
|------|------------|
| 他タスクへの依存なし | 最初のグループに並列起動 |
| 先行グループの成果物を読むだけ（書かない） | 次のグループで並列起動可能 |
| 先行グループが作成したファイルを**編集**する | 別グループに分離（先行完了後に起動） |
| 同一ファイルに複数agentが書き込む | **同一グループに入れてはならない** → 所有権を分離するか1agentに統合 |

### 並列度を上げるテクニック

1. **共通型定義・インターフェースを先行グループに分離** → API/UIが同時に着手可能
2. **テスト作成は実装と同一グループに配置可能**（テストagentは実装agentと異なるファイルを所有）
3. **同一agentを複数起動してスコープ分割**（例: tachikoma-nextjs × 2 で異なるページ群を並列実装）
4. **ドキュメントagentは実装と並列起動可能**（docsファイルは実装agentの所有外）
