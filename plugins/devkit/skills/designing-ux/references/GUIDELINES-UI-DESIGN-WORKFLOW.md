# UIデザインワークフロー

UIデザイナーが担う業務フロー・プロセス・組織内の役割を整理したリファレンス。
UIコンポーネントの視覚ルール（Do/Don't）は `GUIDELINES-ui-design.md` を参照。

---

## 1. デザインの本質的な役割

### デザイン = 解像度を高める行為

| 定義 | 内容 |
|------|------|
| **コア定義** | 混沌とした情報やニーズの中で、適切な課題を設定し、その解決手段をビジュアルで思考・表現する行為 |
| **情報処理** | 混沌とした状況において、複雑・不確実な情報に輪郭を与え、サービスの世界観の解像度を高める |
| **ファシリテーション** | アイデアをプロトタイプ等で具現化し、チームをビジュアルでファシリテーションする |

**インプット → デザイナー → アウトプット の流れ:**

```
[アイデア・調査・技術] → デザイナー → [ドキュメント / グラフィック / プロダクト / 体験]
```

UIデザインは、その情報を適切な形に表現する方法の一つ。ビジネスモデル・ドメイン知識・システム要件を考慮し、その状況に合ったスタイルで表現することが求められる。

## 2. UIデザイナーの担当業務とスキルマップ

### 開発フロー別の担当範囲

| フェーズ | 主な作業 | UIデザイナーの関与 |
|---------|---------|-----------------|
| リサーチ | 市場調査・インタビュー・データ分析 | 補助的 |
| 企画 | アイデア・利用体験・指標設定 | 補助的 |
| **設計** | **要件定義・情報設計・画面仕様・プロトタイプ・ビジュアル** | **中心的** |
| **レビュー** | **エキスパートレビュー・ユーザーテスト** | **中心的** |
| 実装 | マークアップ・コーディング | 連携 |
| テスト | QA・ユーザーテスト | 補助的 |
| 分析 | データ分析・定性調査 | 補助的 |

### UIデザイナーのスキル軸

```
What & Why（何をなぜ作るか）
  ユーザーニーズ / ビジネスゴール / コンテンツ
         ↓
  UIデザイナー が橋渡し
         ↓
How（どう作るか・見せるか）
  情報設計 / 画面仕様 / ビジュアルデザイン
```

**スキルマップ（主要スキル一覧）:**
- ファシリテーション・論理的思考
- 定性/定量分析・ユーザビリティ評価
- 情報設計・プロトタイピング
- アートディレクション・タイポグラフィ・カラー
- エンジニアリング連携

> スキルの境界は曖昧。「自分がやらなければならない」と追い詰めるより、チームで得意分野を共有して協力する姿勢が重要。

## 3. UIデザインの制作プロセス

### 4ステップ制作フロー

```
1. 要件定義
   ヒアリング・データ分析で情報収集
   → コンテンツ要素の優先度・機能要件を整理

2. 画面設計
   ワイヤーフレーム・プロトタイプで設計
   → 実装面を考慮した画面仕様を策定

3. ビジュアルデザイン
   カラー・サイズ等のグラフィック表現を作り込み
   → スタイルガイドに基づいて仕上げる

4. レビュー
   PMがゴール達成度を確認
   デザイナーがガイドライン遵守を確認
   エンジニアが実装再現性を確認
   → 必要に応じてユーザーテストで仮説検証
```

## 4. 情報設計とUI要素への翻訳

### 情報整理の手法

サービスを運用し続けると情報が膨れ上がり混沌とした状況に陥る。その中で情報を整理するのがUIデザインの重要な土台。

| 目的 | 手法 | 用途 |
|------|------|------|
| 時系列でユーザー体験を把握 | カスタマージャーニーマップ | 利用シーンの漫画化・仕様漏れの発見 |
| 複数画面のアイデア発散 | 6up Sketches | 多様な案の並置・比較 |
| 画面構造の全体把握 | サイトマップ | 画面単位で情報を構造化 |
| 複雑なフロー・状態の整理 | UIフロー | ユーザーが見るもの・することを粒度別に整理 |

**UIフローが特に有効な場面:**
- 機能数が多いプロダクト
- 複数画面でデータを共有している場合
- 会員登録・決済など分岐が多い場合

### UI要素への翻訳チェックリスト

仕様要件・データ・ユーザー行動から、以下のUI要素を精査する:

- [ ] ナビゲーション
- [ ] レイアウト
- [ ] コンポーネント（ボタン・フォーム等）
- [ ] アイコン
- [ ] 文言
- [ ] スタイル（カラー・サイズ感）
- [ ] 状態遷移（空状態・エラー・ローディング等）

## 5. 画面設計の忠実度（プロトタイピング戦略）

求めるフィードバックに応じて**抽象度・忠実度**を使い分ける。

### 忠実度マトリクス

| 項目 | スケッチ | ワイヤーフレーム | モックアップ | プロトタイプ |
|------|---------|--------------|-----------|-----------|
| **忠実度** | 低 | 低〜中 | 中〜高 | 高 |
| アイデア | ○ | ○ | ○ | ○ |
| 情報設計 | ○ | ○ | ○ | ○ |
| ビジュアル | × | × | ○ | ○ |
| インタラクション | × | × | × | ○ |
| **主な用途** | アイデア出し・要件定義 | 情報設計・UI要素の認識合わせ | ビジュアルの作り込み | 実機での遷移・操作性チェック |

### 忠実度選択の判断基準

| 状況 | 推奨する忠実度 | 理由 |
|------|-------------|------|
| 仮説がぼんやりしている初期フェーズ | スケッチ（紙・ホワイトボード） | ビジュアルでなくアイデアに議論を集中させる |
| 情報設計の認識合わせが必要 | ワイヤーフレーム | 構造・要素の過不足を確認する |
| ビジュアルの方向性確認 | モックアップ | 複数案を比較しやすい |
| 実装前の操作性確認 | プロトタイプ | 画面遷移・インタラクションの問題を発見する |

### UIデザインを説明するときの5点

1. **ゴール** — そのデザインで達成したい目的
2. **背景** — 課題が発生した経緯・前提条件
3. **仮説** — 分析結果から推測される原因と対策
4. **Use Case** — プロダクトの操作手順
5. **影響範囲** — 関連画面・データ

## 6. スタイルガイドによる品質管理

### スタイルガイドの構成要素

| カテゴリ | 内容 |
|---------|------|
| Colors | プライマリ・セカンダリ・セマンティックカラー |
| Typography | フォントファミリー・サイズ・ウェイト |
| Buttons | サイズ・状態（通常/ホバー/無効）・バリエーション |
| Forms | インプット・セレクト・バリデーション |
| Icons | サイズ規則・用途別分類 |

### スタイルガイドがもたらす効果

```
スタイルガイドなし                スタイルガイドあり
─────────────                ─────────────────
色・形がバラバラ          →   共通言語が確立される
何が正しいか不明          →   「プライマリの送信ボタン」と一言で伝わる
新機能の影響範囲が不明    →   影響範囲を体系的に把握できる
チームの認識コストが高い   →   オンボーディングが効率化される
```

**スタイルガイド整備のトリガー:**
- 機能・画面数が増え始めたとき
- デザイナー・エンジニアが複数人になったとき
- 同じコンポーネントについて毎回説明が必要になったとき

## 7. 組織内でのUIデザイナーの役割

### ステークホルダー別の期待役割

| 視点 | UIデザイナーの役割認識 | 求められること |
|------|-------------------|-------------|
| **ビジネスサイド**（経営者・マーケ・営業） | 商品の使いやすさ・見た目を整える専門職 | 顧客満足度・セールスポイントの担保 |
| **開発サイド**（PM・エンジニア） | 画面仕様を設計し、要件をプロダクトに具体化する役割 | ステークホルダーとユーザー行動を考慮した実装しやすい設計 |

> どちらか一方の役割ではなく、**両方の役割を同時に担う**ことがUIデザイナーとして機能する条件。

### 組織体制のパターン

| パターン | 特徴 | 注意点 |
|---------|------|-------|
| 独立・横断組織 | デザイン部門が独立 | 開発との距離ができやすい |
| 開発部署に組み込み | 開発チームの一員 | ビジネス視点が薄くなりやすい |
| 外部制作会社に委託 | 社内デザイナーが不足している場合 | ドメイン知識の共有コストが高い |

### 継続的改善のための姿勢

立派なガイドラインや技術があっても活用されなければ形骸化する。**小さな改善サイクルを回し続ける仕組み**と、プロダクト全体像を俯瞰してビジュアルでファシリテーションするスキルが長期的に重宝される。

## 8. If/Then ルール集

| If（状況） | Then（アクション） |
|-----------|----------------|
| 初期フェーズでビジュアルを作り込もうとしている | スケッチ/ワイヤーフレームに立ち返り、情報設計の認識合わせを先に行う |
| 同じコンポーネントについて毎回説明している | スタイルガイドを整備して共通言語を確立する |
| 機能が増えて画面の全体像が把握しにくい | サイトマップ・UIフローで構造を可視化する |
| ステークホルダーから的外れなフィードバックを受ける | フィードバックの焦点（アイデア/情報設計/ビジュアル）を明示した上でプレゼンする |
| ビジネスサイドとデザイン価値観のズレを感じる | 「課題解決のアイデアをビジュアルで具現化してチームをファシリテーションする」という貢献を言語化する |
| 分岐が多くUIフロー設計が複雑 | UIフロー手法でユーザーが見るもの・することを粒度別に整理する |
