## AIと協調するICの日常ワークフロー

AIは「代替」ではなく「増幅器」である。ICとしての判断力・専門知識を保ちながら、AI を日常ワークフローに統合することで、スピードと品質の両方を底上げできる。

---

## AIが得意なこと・苦手なこと

### AIが効果を発揮するタスク

| タスク | AIの貢献 | 注意点 |
|--------|---------|--------|
| ボイラープレート・スキャフォールド生成 | REST APIエンドポイント、Reactコンポーネントの雛形を数秒で生成 | 生成物のレビューは必須 |
| ユニットテスト初稿 | エッジケース・エラー条件を網羅したテスト一式を生成 | 網羅性・意図の正確性を人間が確認 |
| コードの説明・要約 | PRのdiff、不慣れなコードの解説、概念説明 | オンボーディングや大きなPRのレビューに有効 |
| マイグレーション・反復編集 | ライブラリ移行、大規模リファクタリングの機械的作業 | スコープを明確に指定すること |
| 構造化ドキュメント生成 | APIドキュメント、リリースノート、ランブック更新 | セマンティック検索でコードとの連動も可能 |

**現実の数字**: エンタープライズ環境でのAI提案受け入れ率は平均約30%。「全部AIに書かせる」は幻想。

### 人間が主導し続けるべきタスク

| タスク | 理由 |
|--------|------|
| アーキテクチャ・設計の意思決定 | AIはユーザー・ビジネス・長期的影響を理解しない |
| セキュリティクリティカルな実装（認証・暗号・決済） | レースコンディション・微妙なメモリ問題を見落とす |
| テスト・監視のない機能の実装 | 安全網なしのAI生成コードは潜在的に危険 |
| 障害モード・複雑な並行処理 | AIは非決定的な振る舞いを確実に予見できない |
| 本番ホットフィックス | 深いシステム知識が必要な局面でのAI頼りはリスク |

## AIを活用した1日のワークフロー

### フェーズ1: 計画・設計（午前）

#### ADR（Architecture Decision Record）のブレインストーミング

**プロンプトパターン（決定概要の生成）:**
```
[問題の概要] を解決するための1ページの決定概要を作成してください。
以下の観点でトレードオフを含む3〜5つの実装候補を提案してください:
スケーラビリティ、レイテンシ、セキュリティ、データモデルの複雑さ、運用性。
類似する先行事例（他社・他プロジェクトの実例）も2件以上挙げてください。
```

**プロンプトパターン（NFRの圧力テスト）:**
```
以下のSLAと制約条件を前提に、潜在的な障害モードについて敵対的な質問をしてください:
- 突発的なトラフィックスパイクやリトライストームが発生した場合は？
- バックプレッシャーはどこで蓄積するか？
- リソース制限のボトルネックはどこか？
各シナリオに対して障害注入テストを提案してください。
```

**ポイント**: AIは「最初の草稿の労力」を代替するが、「思考」は代替しない。外部ファクトは必ず自分で検証する。

#### PRDからバックログへの変換

```
以下の要件ドキュメントを構造化されたバックログに変換してください:
- エピック一覧（高レベル）
- 各エピック配下のユーザーストーリーと受け入れ基準
- 各ストーリーに必要なテレメトリ/監視
- テストデータ・マイグレーション手順
- 段階的ロールアウト計画
Jiraにそのまま投入できる形式で番号付きリストとして出力してください。
```

**効果**: PRDの「コンプライアンス要件」「パフォーマンス目標」などの重要事項が、受け入れ基準に自動的に含まれるようになる。

---

### フェーズ2: 実装（午後）

#### テストファーストパターン（推奨アプローチ）

```
ステップ1: AIにユニットテストを生成させる
  → "processImageUpload(file) のユニットテストを書いてください。
     5MB超はリジェクト、非画像形式はリジェクト、有効な画像はプロファイルに保存。"

ステップ2: テストを実行して失敗を確認
  → 意図通りに失敗しているか検証（テスト自体のバグを排除）

ステップ3: 実装コードをAIに生成させる
  → "これらのテストをパスするように processImageUpload を実装してください"
  → テストが安全網として機能し、AIのミスを自動的に検出

ステップ4: 小さな単位でコミット
  → 1関数/1論理ブロック単位 → テスト実行 → コミット → 次へ
```

**重要**: AIに一度に大きなコード塊を生成させない。小さく分割することで問題の切り分けが容易になる。

---

### フェーズ3: コードレビュー

#### AIを「増幅器」として使う（ゲートキーパーではなく）

**レビュアー向けプロンプトパターン:**
```
このプルリクエストを分析して以下を行ってください:
(a) 主要な変更点を数文で要約
(b) セキュリティ・パフォーマンス・保守性に影響する可能性のある
    リスクのある部分または複雑な部分を列挙
(c) 人間のレビュアーが指摘するであろう2〜5件の具体的なコードレビューコメント
    （行番号付きで、コーディング基準を参照しながら、礼儀正しく建設的に）
```

**大原則**: AIは「疲れないジュニアレビュアー」。スタイル上の問題・既知のバグパターンを指摘させ、人間は高レベルの判断（設計の妥当性・ビジネス要件への適合）に集中する。

---

### フェーズ4: マージ後の後処理

AIが最も「忘れられがち」な作業を効率化する場面:

| タスク | AIへの依頼例 |
|--------|------------|
| ランブック・ドキュメント更新 | "新機能の動作と障害時の対処方法をオンコールランブックに追記してください" |
| Grafanaダッシュボード生成 | "upload_sizeメトリクスとupload_counterの時系列パネルJSONを生成してください" |
| リリースノート・ステークホルダー向けメール | "ユーザー価値に焦点を当てた2〜3文のリリースノートと、サポートチーム向け案内メールを作成してください" |

## コンテキストの供給: AIを「チームメンバーレベル」に引き上げる

**コンテキストサンドイッチ構造**（プロンプト品質を大幅に向上させる4層モデル）:

```
[層1] ゴール/アウトカム
  → "Xの機能を実装したい。期待する結果はYである。"

[層2] 関連コンテキスト
  → 最小限の関連スニペット（関数シグネチャ、エラーメッセージ、設定など）
  → 各スニペットに1行の補足（"// これは実装すべきインターフェース定義"）

[層3] 制約・要件
  → パフォーマンス制約（"O(n)で動作すること"）
  → スタイル指針（"内部スタイルガイドに従うこと"）
  → 技術制約（"ライブラリXのバージョン3を使用"）

[層4] 依頼
  → "実装を提供し、セキュリティ上の考慮事項を説明してください。
     外部のコードや公式ドキュメントを参照した場合はソースを明示してください。"
```

**その他のコンテキスト供給戦略:**

| 手法 | 説明 |
|------|------|
| コードベース連携 | IDEのリポジトリインデックス機能を有効化し、AIがコードを「見える」状態にする |
| セマンティック検索 | "このリポジトリでOAuth2クライアント認証フローはどこで実装されているか？" |
| プロンプトテンプレート共有 | `prompts/` ディレクトリにバージョン管理。チームの共有資産として育てる |

## セキュリティ: AIを使った安全な開発ライフサイクル

### AI生成コードのセキュリティリスク

**現実の統計**: AI生成コードサンプルの約45%にOWASPクラスのセキュリティ欠陥（SQLインジェクション、XSSなど）が含まれる（Veracode 2025年分析）。

### セキュリティ統合チェックリスト

- [ ] **プロンプトにセキュリティ要件を明示する**
  - "非推奨の暗号関数を使用しないこと"
  - "すべての入力を検証し、出力エンコーディングを実施すること"
  - "OWASP Top 10の関連する問題をどのように軽減したかをチェックリストで説明すること"
- [ ] **CI/CDにSAST・SCAを統合する**: AI生成コードに既知の脆弱性パターンがないか自動スキャン
- [ ] **AI生成コードには必ずテストを付ける**: 「AIが書いたなら、テストも必須」のルールを設ける
- [ ] **Policy-as-Codeで禁止パターンを強制する**: `eval`、直接SQL文字列結合などをlinterルールで禁止
- [ ] **インシキュア出力のフィードバックループを作る**: 問題が発見されたらプロンプトテンプレートを更新し、再発を防ぐ
- [ ] **センシティブデータをAIに渡さない**: パスワード、APIキー、PII、機密アルゴリズムをプロンプトに含めない

### アンチパターン: Vibe Coding

> 仕様もテストもなく、AIが生成した「なんとなくそれっぽい」コードをそのままマージすること。

**結果**: 表面上は動くが、セキュリティ欠陥・隠れたバグ・技術的負債の温床。一度の予防可能なセキュリティ事故でAIツールへの組織的信頼が崩壊する。

## プライバシー・ライセンス・コンプライアンス

| リスク | 対処方針 |
|--------|---------|
| センシティブデータの漏洩 | 本番ログのPIIは匿名化。外部AIサービスへの送信は「潜在的に人間が見る可能性あり」と仮定 |
| ライセンス汚染 | 大きなコード塊の類似チェックを実施。AIに "このコードはどのソースから来たか？" と確認 |
| AIコードの帰属 | PR・コメントに "AI支援で生成" を記録。監査証跡として機能 |
| 規制対応 | EU AI Act等の動向を把握。Legal/Complianceチームと連携して自社への適用範囲を確認 |

### NIST AIリスク管理フレームワーク（実用的解釈）

| 機能 | ICとして問うべき質問 |
|------|-------------------|
| Govern（統治） | チームにAI利用ガイドラインはあるか？誰がポリシーをレビューするか？ |
| Map（マッピング） | AIをどこで使っており、失敗した場合の影響は何か？ |
| Measure（測定） | AIの成果・エラー率・セキュリティインシデントを追跡しているか？ |
| Manage（管理） | 問題が起きたときの対応プロセスはあるか？継続的改善はしているか？ |

## AIエージェントの活用と管理

### エージェントが有効なタスク（現時点）

| タスク | 説明 |
|--------|------|
| 大規模リファクタリング | 関数リネーム、インポート更新を数百ファイルに適用 |
| 依存関係アップグレード | バージョンバンプ + ビルドエラーの自動修正 |
| レガシーコードのテスト生成 | テストのないモジュールへのユニットテスト追加 |
| バグの再現・切り分け | フレーキーなテストの調査・最小再現リポジトリの生成 |

### エージェント管理の安全レール

- **スコープ制限**: サンドボックスブランチで実行、修正可能なディレクトリを指定
- **ステップ・時間制限**: 暴走を防ぐためのタイムアウト・最大ステップ数
- **テストをガードレールとして組み込む**: テストが通らない限りエージェントのタスクは未完了
- **すべてのエージェント出力にPRを経由させる**: 人間のレビューを経てからマージ
- **全操作のラベル・ログ保持**: "AI agent Alpha 1.2 - 2026-01-15" のようなコミットメッセージ
- **非クリティカルなタスクから開始**: 内部ツール・ドキュメントサイトで信頼性を積み上げてから本番コアに適用

> **2026年時点のエージェント**: 高速道路での自動運転に例えられる。好条件下では有効だが、ドライバーは常に介入可能な状態を維持すること。

## インパクト測定: AIの効果をエンジニアとして定量化する

### 採用・行動メトリクス

| メトリクス | 意味 |
|-----------|------|
| 週間アクティブユーザー数 | ライセンスを付与した開発者の何割が実際に使用しているか |
| 提案受け入れ率 | 業界平均は約30%。5%以下なら使い方の改善が必要。60%超は過信のリスク |
| AI生成コードの保持率 | 受け入れ後に大幅修正・削除が多い場合、時間節約効果が薄い可能性 |
| コンテキスト活用率 | 提案にリポジトリ固有のAPIが正しく含まれているか |

### 品質・スループットメトリクス

| メトリクス | 計測方法 |
|-----------|---------|
| タスク完了時間 | AI導入前後での同種タスクの所要時間比較 |
| PRサイクルタイム | オープンからマージまでの時間（AIレビュー支援で短縮可能） |
| スループット（機能数/スプリント） | AI支援前後での完了ストーリー数比較 |
| コード品質 | 欠陥密度（バグ/KLOC）、本番逸脱バグ率 |
| テストカバレッジ変化 | AI支援コミットのカバレッジが向上しているか |
| セキュリティインシデント率 | AI導入前後での脆弱性発見数の変化 |

### 構造化パイロット実験の設計

```
フェーズ1（ベースライン）: AI未使用での2週間。全メトリクスを計測。
フェーズ2（コントロールグループ）: チームの半数にAI導入。残り半数は対照群。
フェーズ3（比較・分析）: 両グループのメトリクスを比較。
フェーズ4（定性フィードバック）: チームへのサーベイ。生産性の体感・信頼度。
フェーズ5（反復改善）: データに基づいてプロンプト・プロセスを調整。
```

## 即使えるプロンプトパターン集

### パターン1: テストファースト・ジェネレーター

```
あなたはシニアエンジニアです。以下の機能に対して、ユニットテスト
（必要に応じてプロパティベーステストも）を生成してください:

[機能の説明またはユーザーストーリー]

要件:
- エッジケースを網羅すること
- 機能が実装されていない状態で明確に失敗すること
- 実装の詳細を仮定せず、入出力の振る舞いに集中すること
- プロジェクトの標準テストフレームワークを使用すること
```

### パターン2: セキュア・スキャフォールダー

```
[フィーチャー/サービス] のスキャフォールド実装を [フレームワーク] で作成してください。

必須セキュリティ要件:
- すべての入力を検証（長さ・型・SQLインジェクション・スクリプトインジェクション対策）
- ユーザー向けテキストに出力エンコーディングを適用
- 安全でない関数（eval、生のSQLクエリ等）を使用しないこと
- センシティブデータをマスクした構造化ログを含めること

コード作成後に、どのOWASP Top 10リスクをどのように軽減したかを
チェックリストで説明してください。
```

### パターン3: レビュアー・アシスタント

```
あなたはコードレビューアシスタントです。以下のdiffを分析して:
(a) 変更内容を2〜3文で要約
(b) セキュリティ・パフォーマンス・保守性のリスクがある箇所を列挙
(c) 人間のレビュアーが指摘するであろう2〜5件の具体的なコメントを提案
    （行番号付き、コーディング標準参照、礼儀正しく建設的なトーンで）
```

### パターン4: マイグレーション・プランナー

```
[ライブラリ/技術A] から [ライブラリ/技術B] へのマイグレーション計画を作成してください。

含めるべき内容:
- フェーズ分けされたステップ（Phase 1: X, Phase 2: Y...）
- 移行中の並行稼働方法（可能な場合）
- データマイグレーション要件
- 各ステップでの検証方法
- ロールバック計画
- 既知の落とし穴や注意点

ダウンタイムとリスクの最小化を優先してください。
```

### パターン5: インシデント・エクスプレイナー

```
以下のインシデントタイムラインとログを分析してください:
[インシデントデータ]

(a) 平易な日本語でインシデントの経緯を簡潔に記述（ポストモーテムレポート形式）
(b) 主要な根本原因・寄与要因を2〜3件特定
(c) 再発防止のための現実的でブレームレスな対策を2件提案
    （個人ではなく、システム・プロセスの改善に焦点を当てること）
```

## 90日チーム導入計画

### Week 1-4: 基盤の確立とガードレール整備

| アクション | 目的 |
|-----------|------|
| パイロットツールを1つ選定・ライセンス取得 | 変数を最小化して比較しやすく |
| データ・セキュリティポリシーを定義 | 「センシティブデータを渡さない」など基本ルールの明文化 |
| ベースラインメトリクスを計測 | 導入前のタスク完了時間・品質指標を記録 |
| ボランティア数名で低リスクタスクから開始 | 内部ツール・小機能で学習 |
| ランチ&ラーン（キックオフ勉強会）の開催 | 使い方デモ・ポリシー周知・フィードバック収集 |

**Day 30のゴール**: ツール稼働、全員認知、初期メトリクス取得、ガードレール設置。

### Week 5-8: スケールアップとチームトレーニング

| アクション | 目的 |
|-----------|------|
| 初期ユーザーの知見をプロンプトライブラリに集約 | チーム全体の再利用可能なベストプラクティス化 |
| AIを使ったコードレビューの奨励 | レビュアー負荷軽減、品質管理への活用 |
| エージェントの限定試験 | 大規模リファクタリング等でエージェント実用性を評価 |
| セキュリティファーストの習慣を徹底 | PRテンプレートに "AIを使用・セキュリティスキャン実施" チェックボックス追加 |
| 60日時点で中間メトリクス分析・レトロ実施 | 課題の早期特定と軌道修正 |

**Day 60のゴール**: チームの大半がAIを活用中、初期の問題点が解消、改善の兆候が数値に現れている。

### Week 9-12: 最適化と次フェーズの決定

| アクション | 目的 |
|-----------|------|
| ベースラインとの詳細比較 | タスク完了時間・PRサイクル・品質の変化を定量化 |
| 有効・無効なユースケースを特定 | テスト・ボイラープレートは有効、複雑アルゴリズムは不向き、等 |
| AI生成コードのミニ監査 | マージされたコードの品質・セキュリティを遡及的に確認 |
| 幅広い展開or軌道修正の判断 | データに基づいてダブルダウン/調整/一時停止を決定 |
| ナレッジベース・プレイブックの最終化 | 次のメンバーが再学習不要な「AIを使ったエンジニアリングガイド」作成 |
| 成功事例・教訓の共有 | 具体的な数字で懐疑的なメンバーを巻き込む |

**Day 90のゴール**: 成功事例（証拠付き）とAI利用ガイドが完成。全員が「なぜこのツールをこう使うのか」を説明できる状態。

## 次世代ICの姿: AIとの協調によるハイブリッドIC

### AIが解放するもの

AIがルーティン・ボイラープレート・反復作業を担うことで、ICは以下に集中できる:

- **大局的な設計と創造性**: アーキテクチャ判断、ユーザー体験設計
- **クロスドメインの知識統合**: DevOps・セキュリティ・プロダクト感覚を持つハイブリッドIC
- **高品質なコードレビューと技術メンタリング**: AIが低レベルを担い、人間が本質を議論
- **コミュニケーションとステークホルダー管理**: AIが補助できないソフトスキルがより差別化要因に

### T字型スキルの進化

```
             [深い専門性]
                  |
技術基礎 ─────────┼──────── 隣接領域への理解
                  |
  DevOps・セキュリティ・プロダクト感覚・基本的なML知識
```

- **深さ**: 特定技術領域（パフォーマンス、DB内部構造、ML等）の専門的理解
- **広さ**: クラウドDevOps、セキュリティ、プロダクト思考、AI活用能力
- **禁物**: 何でも浅くなる「ジャック・オブ・オール・トレーズ」化

### 生涯ICとして活躍し続けるために

| 変わらないもの（ファンダメンタルズ） | 変化に対応するもの |
|-----------------------------------|------------------|
| データ構造・アルゴリズム | AIツールの使い方と限界の理解 |
| システム設計原則 | 新しいフレームワーク・パラダイムへの適応 |
| コミュニケーション・批判的思考 | ドメイン知識のアップデート |
| 問題分解・抽象化 | コンテキストエンジニアリングのスキル |

> **重要な視点**: AIが「シンタックスとボイラープレート」を担うほど、アーキテクチャ・設計の正しさを判断できる「概念的理解」の価値が相対的に高まる。ファンダメンタルズへの投資はAI時代においてより重要になる。
