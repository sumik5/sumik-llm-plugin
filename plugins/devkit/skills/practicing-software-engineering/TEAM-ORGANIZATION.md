# チーム組織とペアプログラミング（TEAM-ORGANIZATION）

チームの組織設計からペアプログラミングパターンまで、
開発チームの効果的な運営方法を体系化したガイド。

---

## チームサイズと構造

### 小さなチームが優れる理由

チームサイズが大きくなるにつれ、以下が指数関数的に増加する:

- **認知負荷（Cognitive Load）**
- **通信の組み合わせ数**
- **システムの複雑性**

> Conway の法則: 「システムを設計する組織は、その組織のコミュニケーション構造を
> コピーした設計を生み出す。」— Melvin E. Conway

### 推奨チームサイズ

**5〜9人**が最適レンジ。大規模プロジェクトでは、この規模の複数チームに分割する。

### 大規模組織でよくある失敗パターン

機能別（コーディング・テスト・アーキテクチャ・UIなど）にチームを分けると:

- すべてのチームが同時に進捗しなければ何も完成しない
- チーム間の依存が増大し、スピードが低下する
- 誰もシステム全体の責任を持たない

---

## チームスキルと責任

### チームが一次的な作業単位

「個人」ではなく「チーム」が作業の基本単位。
チームメンバー全員がチームの目標に対して責任を共有する。

### 避けるべきハンドオフ

| ❌ 悪い例 | ✅ 良い例 |
|---------|---------|
| バックエンドを作ってからフロントエンドチームに渡す | 同一チームが両方を担当 |
| アーキテクチャは別チームの責任 | チームが自分たちのアーキテクチャを担当 |
| テスト・QAは別チームに任せる | チームが品質に責任を持つ |

### チームが担うべきスキルの全領域

中心から外側へ:

1. **開発・コーディング / 設計・アーキテクチャ**（コア）
2. **インフラストラクチャ / オペラビリティ**
3. **メトリクス・監視 / テスト・QA**
4. **アプリセキュリティ / 法的コンプライアンス / 商業・運用上の実行可能性 / UX**

高パフォーマンスチームは小規模でありながら、これらすべての領域を担える多機能チームである。

---

## チームトポロジー

### 4種類のチームタイプ

#### 1. Stream-Aligned Team（ストリーム整合チーム）

**全チームの大多数がこれであるべき**

- ビジネスドメインの bounded context に整合したチーム
- ユーザーへの価値を素早く確実に届けることにフォーカス
- 他チームへの依存なしに独立して進捗できる

> 成功の予測指標: 「チームが他チームの承認なしに、意思決定し進捗できる能力」
> — State of DevOps

#### 2. Enabling Team（イネーブリングチーム）

- 必要に応じて専門知識を貸し出す
- Stream-Aligned Team が「難しい20%」に直面したときに支援
- 役割は2つ: **機能実装の支援** + **Stream-Aligned Team へのナレッジ移転**
- 目的: Stream-Aligned Team の認知負荷を減らすこと

#### 3. Complex Subsystem Team（複雑サブシステムチーム）

- 技術的に特殊・複雑なコンポーネントを担当（例: 複雑なハードウェアとのインターフェース）
- 狭い分野の深い専門知識を持つ
- Stream-Aligned Team が技術的詳細を気にせず進めるようにする

#### 4. Platform Team（プラットフォームチーム）

- Stream-Aligned Team の仕事を楽にするツール・インフラを構築
- **長期的ニーズ**に対応（戦術的なフィーチャーリクエストではない）
- 詳細は後述の「Platform Teams」セクションを参照

---

## Platform Teams の原則

### 目的

Stream-Aligned Team が他チームから独立して作業できるように支援すること。

### よくある失敗パターン

| パターン | 問題 |
|---------|------|
| 「残りものの集合体」 | 分類できなかったものを全部 Platform に押し込む → 相互に無関係なコンポーネント群 |
| 「グランドデザイン病」 | 完璧なプラットフォームが完成するまで誰も使えない → 全チームが待ち続ける |
| 「強制変更」 | Platform の変更で Stream-Aligned Team のコード変更を強いる |

### 良い Platform Team の設計原則

1. **ユーザー視点での使いやすさ優先**
   - Platform コード側で多少の追加実装が必要でも、消費者側が使いやすい設計を選ぶ

2. **疎結合 API 設計**
   - API 変更が Stream-Aligned Team のコード変更を強制しない設計

3. **Platform 側が変更コストを負担**
   - API 変更が必要な場合、Platform Team がすべての消費者コードを変更する責任を持つ
   - CI を通じて行うのが最も効率的

4. **インクリメンタルな開発**
   - 小さなステップで実際のユースケースを通じて何が本当に機能するかを発見する
   - シンプルなドキュメント（新サービスの追加方法・ディレクトリ構造など）から始めることも有効

5. **Stream-Aligned Team のニーズにレーザーフォーカス**
   - チームが自分たちの作業について意思決定できているか定期的に確認する

## チーム構造の変革

既存組織でチーム構造を改善する際の指針:

1. **Enabling Team を作る**: CD プラクティスの普及・Deployment Pipeline 等のインフラ構築
2. **チームの独立性を高める**: チーム間の結合・依存を削減
3. **多機能チームへの移行**: チームを大きくせず、専門知識を一時的に貸し借りしながら
4. **Platform Team の見直し**: 本当に Stream-Aligned Team を助けているか確認
5. **インクリメンタルな変革**: 大規模な一括再編を避け、徐々に自律性を高める
6. **5〜9人の規模目標**: 大きいチームは複数の小チームに分割する方法を検討
7. **設計上のコストを受け入れる**: チームが独立して働けるなら、多少の設計上の冗長性は許容する

## ペアプログラミング

### ペアプログラミングとは

2人のプログラマーが協力してコードを書く手法。
コード記述だけでなく、計画・調査・設計・テスト・ドキュメントもペアで行う。

CI/CD・TDD・ATDD と組み合わせることで相乗効果を発揮する。

---

## ペアプログラミングパターン

### ✅ Driver & Navigator（推奨：通常の開発）

最も基本的なペアリングパターン。

| 役割 | フォーカス | 思考例 |
|------|-----------|--------|
| **Driver** | コードの中 | 「この構文は？どう表現する？次のテストは？」 |
| **Navigator** | 戦略的視点 | 「どこへ向かっているか？次は何？ブロッカーは？」 |

- 定期的にロールを交代する

### ✅ Ping Pong（推奨：TDD実践時）

TDD のサイクルに最適化されたパターン。

```
Ping: 失敗するテストを書く（RED）
↓
Pong: テストを通すコードを書く（GREEN）
↓
2人で REFACTOR
↓
ロール交代
↓
Ping: 次の失敗するテストを書く
↓ ... 繰り返し
```

### ✅ Strong-Style（推奨：ナレッジ移転・オンボーディング）

経験者から初心者へのナレッジ移転に特化したパターン。

| 役割 | 担当者 | 特徴 |
|------|--------|------|
| **Navigator** | 経験者 | 方向を示しながらガイド |
| **Driver** | 初心者 | 手を動かして学ぶ |

- 各ステップ後に、コード・進捗・学習内容を2人でレビュー
- 十分な学習が行われるまでロールを維持（毎回交代しない）

### ❌ Parent-Child（アンチパターン：避けるべき）

経験差があるペアで陥りがちな失敗パターン。

| 役割 | 問題 |
|------|------|
| 経験者 | 過度に細かい指示を出す「親」的役割になる |
| 初心者 | 受動的な「子」的役割になり、学習が限定される |

**学習効果が著しく低下するため、意識的に避けること。**

## Pair Rotation（ペアローテーション）

### 目的と効果

- ナレッジの共有
- チームビルディング
- 強い開発文化の構築

### 仕組み

1人が「Anchor」として留まり、もう1人が別のペアに移動。
これにより継続性を保ちながらナレッジが循環する。

### 推奨ローテーション頻度

- **通常**: 2〜3日ごと
- **初心者が多い場合**: より長い間隔
- **毎日**: 慣れていない人には頻繁すぎる可能性あり

---

## ペアプログラミングの効果

### 継続的レビュー

コードレビューを「後から」ではなく「書きながら」行う。
フィードバックが最も価値ある瞬間（コードを書いている時）に提供される。

### 学習の増幅

- パートナーの異なる知識・技術から双方が学ぶ
- シニア→ジュニアの一方向ではなく、双方向の学習
- アイデアを説明することが理解を深める

### コード品質

- 複数の選択肢を検討する
- 前提を確認・言語化する
- 結果: シンプルさ・可読性・正確性・機能性・速度が向上

### 生産性

- ペアは平均的にソロの60%の時間で作業を完了
- バグが少ない → バグ修正時間が減る
- WIP（仕掛中作業）を制限 → フローを維持

### イノベーション

- パートナーとのアイデア交換で創造性が高まる
- リスクを取りやすい（パートナーのサポートがある）
- バグ修正に時間を取られないため、新しい作業に充てる時間が増える

### ナレッジ分散とビジネス継続性

- コードがチーム全員の「共有所有物」になる
- 特定の人が離脱してもシステムへの影響が最小化される
- ハンドオーバーが不要になる

## ペアのパートナーシップ組み合わせ

| | 初心者 | 経験者 | スペシャリスト |
|--|--------|--------|--------------|
| **初心者** | 一緒に実験・学習。外部のコーチが必要な場合も。 | ナレッジ移転。マイクロマネジメントに注意。 | ナレッジ移転。依存を避けるため定期的にペア変更。 |
| **経験者** | 初心者がフレッシュな視点をもたらす。 | 創造的・生産的・楽しい。分析麻痺に注意。 | スペシャリストが広い視点・知識から学ぶ。 |
| **スペシャリスト** | 初心者が特定ツール・スキルを学ぶ。他プロジェクトに定期的に移動を。 | 経験者が新しいドメインでスキルを習得。 | ナレッジ移転。変革的。エゴとテリトリー争いに注意。 |

## ペアプログラミング成功のコツ

| コツ | 内容 |
|------|------|
| **デフォルトとして採用** | ペアリングを「通常」として、単独作業を例外とする |
| **問題の理解から始める** | コードを書く前に、ユーザーストーリー・解決策・アプローチをペアで検討 |
| **自然なペア形成** | 特に最初は、メンバーが自分でやりたい作業を選べるようにする |
| **基本的なセットアップ** | カレンダー調整・ペアリング時間・作業環境の設定を事前に合意 |
| **リモートペアリング** | 画面共有・コントロール共有・ビデオ通話・ヘッドフォン使用 |
| **定期的なローテーション** | 通常2〜3日ごと |
| **頻繁な休憩** | 集中度が高く疲れやすい。定期的な休憩を促す |
| **抵抗の克服** | 数スプリント試してから本格導入を検討。経験してメリットを体感してから |
| **柔軟性と忍耐** | 異なるパターンが異なる人・プロジェクトに合う。常にペアリングする必要はない |
