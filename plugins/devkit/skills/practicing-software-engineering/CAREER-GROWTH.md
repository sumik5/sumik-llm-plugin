## キャリア成長とレベルアップ

ソフトウェアエンジニアとしての成長は「時間の経過」ではなく「インパクトの範囲と質」によって決まる。本リファレンスでは、各キャリアステージで何が求められるか、Technical Depth vs Breadth をどう判断するか、そして昇進・キャリアパスをどう戦略的に歩むかを整理する。

---

## 基礎スキルの概要（Junior → Mid-Level）

初期キャリア（経験0〜4年、L3〜L4相当）では、**技術的卓越性と優れたエンジニアリング習慣**が成果の源泉となる。

### 主要な基礎スキル（概要）

| スキル | 習得ポイント |
|--------|------------|
| クリーンコード | 明確性・シンプルさ・一貫性・リファクタリング容易性 |
| テストと品質 | ユニットテスト・統合テスト・回帰防止・テストをドキュメントとして使う |
| バージョン管理 | コミット = ストーリーテリング、git bisectによるデバッグ |
| デバッグ | 仮説→実験→証明の科学的アプローチ |
| ドキュメント・コミュニケーション | 決定事項の記録、詰まったら早期に発信 |

> **詳細リファレンス**:
> - クリーンコード・SOLID原則 → `writing-clean-code` スキル参照
> - テスト戦略・TDD → `testing-code` スキル参照

## キャリアステージ別スキルマトリクス

### 全レベル比較表

| 次元 | Junior (L3) | Mid-Level (L4) | Senior (L5) | Staff (L6+) |
|------|------------|----------------|-------------|------------|
| **スコープ** | タスク単位 | 機能〜中規模 | プロジェクト全体 | 複数チーム・組織 |
| **自律性** | 頻繁なガイダンス必要 | 明確なタスクは独立実行 | 曖昧な要件も自己分解 | 問題を自ら発見し解決 |
| **技術的意思決定** | 実装詳細 | 実装詳細 + 設計への参加 | アーキテクチャ設計 | 組織横断技術方針 |
| **コミュニケーション** | スタンドアップ・質問 | PR説明・推定提供 | ステークホルダー対話 | 経営層・全社技術ビジョン |
| **リーダーシップ** | なし | コードレビューで貢献 | チームのTech Lead | リードなき権限行使 |
| **メンタリング** | 受け手 | 受け手 + 後輩への簡単なサポート | Junior/Mid指導 | 複数レベル・複数チーム指導 |
| **技術的負債** | 認識できる | 指摘できる | 計画的に解消 | 組織優先度として主導 |

### 各レベルで「次へ行く」ために必要な転換

```
Junior → Mid:  「タスクをこなす」→「品質と一貫性で信頼を積む」
Mid → Senior:  「コードを書く」→「プロジェクトを定義・推進し、チームを引き上げる」
Senior → Staff: 「チームを改善する」→「組織の技術戦略に影響を与え、正しい問題を選ぶ」
```

## Technical Depth vs Breadth の判断フレームワーク

Senior以上では「T字型スキル」—— 1つ以上の縦（深さ）と横（広さ）の両立が求められる。

### Depth（専門性）とは何か

- 特定技術ドメイン（DB、分散システム、ML、セキュリティ等）の細部と落とし穴を熟知
- そのドメインで他の人より速く・エレガントに解く
- 他者のメンタリングとプロジェクトリードができる

**Depthによるインパクト例**: DBエキスパートは「インデックス戦略」「シャーディング」「レプリカ構成」を即座に判断できる。

### Breadth（汎用性）とは何か

- 複数ドメイン・スタック全体の動作原理を理解
- 各専門家ほど深くはないが、コンポーネント間の相互作用を予見できる
- フロントエンド変更がバックエンドに波及するリスクを前もって察知できる

**Breadthによるインパクト例**: DB以外も理解するエンジニアは「Redisキャッシュで90%のDB呼び出しを削除」「50回の個別クエリをバッチ化」など、より単純な解法を発見する。

### 投資判断テーブル

| 状況 | 推奨投資先 | 理由 |
|------|-----------|------|
| Junior / Mid初期 | Depth優先 | 信頼性・専門性の構築 |
| Mid後半 | Depth維持 + Breadth拡大開始 | 隣接領域の理解が設計改善に直結 |
| Senior | T字型バランス | プロジェクトリード + 技術設計に両方必要 |
| Staff以上 | Breadth重視（深さは維持） | 複数チームの仲裁・組織横断設計 |
| 急激な技術変化がある時 | Breadth | 適応性でキャリアリスク低減 |
| 会社がスペシャリスト不足な時 | Depth | indispensableになれる |

### 専門化のプロセス（3ステップ）

1. **Specialtyの選択**: 興味・会社のニーズ・市場需要が交わる領域を特定する。プロジェクト配属が決めることもある。どちらも有効。
2. **Depthへのコミット**: 書籍・上級コース・カンファレンス・挑戦的な問題への取り組み。エキスパートとは「多くのパターンの失敗を見てきた人」である。
3. **Depthの活用**: デザインレビューのリード、そのドメインのプロジェクト主導、社内Tech Talk、内部ドキュメント執筆。

### 技術的負債への Depth vs Breadth の適用

| 役割 | Depth の使い方 | Breadth の使い方 |
|------|--------------|----------------|
| 負債の特定 | コードの臭い・アンチパターンを認識 | どの負債が「高金利」かビジネス文脈で判断 |
| 解消計画 | 適切な設計パターンで正確に修正 | ビジネスロードマップと同期し、適切なタイミングで実施 |
| ステークホルダーへの説明 | 技術的根拠を示す | 「新機能の開発速度が2倍遅い」と非技術者の言語に翻訳 |

**負債の優先度判断**: 「この変数は2人しか使わない小規模ビルドツールの問題」vs「認証とユーザーモジュールが密結合でバグを量産している」—— インパクトと利息率で優先度を決める。

## 昇進の力学

### プロモーションの核心原則

> **「テニュア（在籍年数）ではなく、インパクトとスコープで昇進が決まる」**

昇進を得るためには、**すでに次のレベルで動いている状態**を示す必要がある。

### プロモーション準備チェックリスト

| アクション | 具体的な行動 |
|-----------|------------|
| ラダーを把握する | 自社のレベル定義・ルーブリックを入手し、自分へのチェックリストとして使う |
| Show, don't tell | Seniorを目指すなら今からSeniorの仕事（技術提案・後輩ガイド・自発的改善）を始める |
| マネージャーと会話する | 「Senior Engineerになりたい。何を示す必要があるか？」と明示的に聞く |
| 成果を記録する | 成果物・メトリクス・インパクトを年間通じて記録する（"Brag Document"） |

### Brag Document の書き方

```
# Brag Document - [Quarter/Year]

## プロジェクト名
- 状況: [背景]
- 担当: [自分が行ったこと]
- アクション: [具体的な手順]
- 結果: [定量的な成果] (例: 検索応答時間50%改善、コンバージョン5%向上)
```

> AI活用ヒント: JiraチケットやPRリンクをAIに渡し、「STARメソッドでビジネスインパクトを強調したバレットポイントを書いて」とプロンプトすることで、自己PR文書の草案を素早く作成できる。

## Mid → Senior → Staff への具体的移行戦略

### Mid → Senior への移行

**変化の本質**: 「実装する人」→「定義・推進・人を引き上げる人」

| 証明すべきこと | 具体的な行動 |
|-------------|-----------|
| プロジェクトリード | 高レベル要件を受け取り、自分でタスク分解・設計・調整を行う |
| 技術的メンタリング | 後輩に「考え方を教える」（答えを教えるだけでなく） |
| 曖昧さへの対処 | 要件が不明確でも判断を下し、エスカレーションを最小化 |
| 信頼性 | 推定精度が高く、プレッシャー下での技術的判断を信頼される |
| 技術文化への貢献 | 新ツール提案・技術的負債の改善・チームプロセス向上を主導 |

**準備行動の例**:
- アーキテクチャレビューで積極的に発言する
- 後輩エンジニアのPRレビューを引き受ける
- 機能の技術提案書（design doc）を自分で書く
- インシデント対応でコーディネーター役を担う

### Senior → Staff への移行

**変化の本質**: 「チームを改善する」→「組織に正しい問題を解かせる」

| 証明すべきこと | 具体的な行動 |
|-------------|-----------|
| 組織横断のインパクト | 複数チームに影響する課題を発見し主導する |
| 権限なきリード | 管理権限なしに複数チームを動かす |
| 技術方針の設定 | コーディング規約・アーキテクチャ標準・ポストモーテム主導 |
| 戦略的技術判断 | ビジネス戦略に沿った技術的優先順位の設定 |

**準備行動の例**:
- 複数チームに影響するデプロイメント改善を提案・実行する
- 組織全体のセキュリティ基準確立を主導する
- モノリスからマイクロサービスへの移行のアーキテクチャ設計
- 複数のSeniorエンジニアを束ねるイニシアティブを起案する

## 昇進の「不文律」

表向きの評価基準に加え、以下の暗黙のルールが昇進に影響する。

| 不文律 | 解説 |
|-------|------|
| **コード以上のもの** | Mid → Senior にはリーダーシップ（タイトルなし）の証明が必要。Senior → Staff には組織的影響力。「黙々とコードを書くだけ」ではMid-Levelが天井になりうる |
| **Personal Brand** | 「あの人に聞けば一番難しいバグが直る」「彼女のデザインは堅牢だ」という評判が昇進議論を加速する |
| **可視性** | 大組織では直属の上司だけでなく、より広い範囲での認知が重要。社内テックトーク・イントラネット記事・横断プロジェクトへの参加が有効 |
| **メンターとスポンサー** | Mentorは助言する人。Sponsorは昇進議論の場で「この人は次のレベルで動いている」と発言してくれる人。両方を育てる |
| **一貫性** | 1つの大きな成功より、持続的な高いパフォーマンスが重要。波があっても平均値を高く保つ |
| **プロフェッショナリズム** | 協力的で信頼できる姿勢は、組織が「この人に影響力を与えていいか」を判断する基準になる |

## IC vs マネジメントパスの選択

SeniorレベルになるとIC（Individual Contributor）のままStaff Engineerを目指すか、Engineering Manager（EM）に転向するかの岐路に立つ。

### 2つのパスの比較

| 次元 | IC（Staff Engineer）パス | EM（Engineering Manager）パス |
|------|------------------------|------------------------------|
| **主な仕事** | 技術的解決策・設計・直接貢献・技術メンタリング | 採用・1on1・チーム調整・上層部への報告 |
| **影響力の形** | 技術的アイデアを通じて間接的に影響 | 人事決定・プロジェクトアサインの権限 |
| **コーディング** | 継続（Staff以上でも深く関わる） | 減少（マネジメント業務に移行） |
| **満足の源泉** | 難しい技術問題を解くこと自体 | チームメンバーの成長・チームの成功 |
| **会議量** | Junior〜Mid期は少ない。Staff以上は多い | 一貫して多い |
| **給与・評価** | Senior以降は同等 | Senior以降は同等（マネージャー階層も存在） |

### 選択基準フロー

```
コードを書くことと技術問題を解くことに喜びを感じるか？
├── Yes → IC継続を検討
│         → Staffを目指すなら「権限なきリード」「組織横断インパクト」にフォーカス
└── No（または不確か）
      ↓
チームメンバーの成長・組織の成功に喜びを感じるか？
├── Yes → EM検討。まず「Tech Lead Manager」的な経験を試してみる
└── 不確か → 1〜2年のEM経験後にIC復帰も一般的。パスは可逆である
```

### 重要な考慮事項

- **両パスはどちらも「昇進」ではなく「並行トラック」**である。Staff EngineerとEMは同等に評価されるべき。
- ICでもStaff以上は大量のクロスコラボレーション業務（=会議）が伴う。
- ICからEMへの転向・復帰は一般的。EM経験はIC戻り後のビッグピクチャー理解を深める。
- EMを「昇進の唯一の道」として選ぶのは危険。優秀なエンジニアが不幸なマネージャーになるパターンを生む。

## セルフアドボカシーの実践

### 基本原則

自己推薦は「自慢」ではなく「事実の報告」である。マネージャーは多くのメンバーを抱えており、あなたの貢献を覚えていない可能性がある。

### アクションプレイブック

| タイミング | 具体的なアクション |
|-----------|----------------|
| **日常的に** | 成果・メトリクス・フィードバックをBrag Documentに記録 |
| **1on1で** | キャリア目標を明示する：「Senior Engineerを目指しています。次のサイクルで何を達成すべきか教えてください」 |
| **360フィードバックで** | リーダーシップや横断的インパクトを語れる人を選ぶ。自分も他者に誠実なフィードバックを返す |
| **昇進サイクルの前** | HR/昇進プロセスのタイミングを把握し、マネージャーと「準備ができているか」を事前に確認 |
| **昇進見送り後** | フィードバックを具体的に求め、6ヶ月プランを立てて再挑戦 |

### セルフアドボカシーのNGパターン

| NGパターン | 代替アプローチ |
|----------|-------------|
| 「一人でやった」と誇張する | 「チーム3名をリードし、自分が〇〇を主導した」と事実ベースで述べる |
| 成果をメトリクスなしで語る | 「応答時間を50%改善し、コンバージョン5%向上」のように定量化 |
| 昇進を「黙って待つ」 | 明示的に目標を伝え、プランを一緒に作る |
| 1つの成功で完結する | 一貫したハイパフォーマンスを維持し続ける |

### キャリア成長を「エンジニアリングプロジェクト」として扱う

```
1. 要件収集   → レベル期待値（ラダー）を理解する
2. 実装       → スキルを開発し、スコープを広げる
3. テスト     → フィードバックを得て調整する
4. リリース   → 貢献を可視化する
5. 受け入れ検査 → マネージャー・ピアが「次のレベル」と認める
```

## ラテラル移動（横断的成長）

昇進だけが成長ではない。チームや役割を変えることで、必要なスキル・経験を戦略的に積める。

| 状況 | ラテラル移動の判断基準 |
|------|-------------------|
| 現チームに「リードの機会」がない | 新しいプロダクトラインでの主導経験を積む |
| 同じスタックしか触れていない | 隣接技術（バックエンド↔フロントエンド等）の経験を積む |
| 会社の昇進が構造的に詰まっている | 外部で適切な評価・機会を探すことも選択肢 |

## まとめ：レベルアップの複利モデル

キャリア成長は短期的な成果の積み上げではなく、**技術力・影響範囲・信頼の複利**によって加速する。

```
基礎習慣（Junior）
  ↓
信頼性と技術的深さ（Mid）
  ↓
T字型スキルとチームへのレバレッジ（Senior）
  ↓
組織への戦略的インパクト（Staff）
```

各段階で「今より少し広いスコープ」で行動し続けることが、次のレベルへの最も確実な道である。
