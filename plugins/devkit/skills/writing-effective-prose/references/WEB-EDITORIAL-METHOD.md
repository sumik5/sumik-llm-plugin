# Web編集メソッド

## 概要

月3,000本以上の記事を配信する大規模ニュースサイトでの新人研修から体系化された、未経験者を現場ライターに育てた実績に基づく実践的なWeb文章メソッド。

> **核心**: 「いきなり書き始めてはいけない」— 書く前の準備で文章の7割が決まる

---

## 1. 「完読」の哲学

### 良い文章とは何か

現場編集ゼミの回答：**完読される文章が良い文章**。

「わかりやすい」「テンポがいい」「間違いがない」…無数の良回答を万能解に集約したマジックワードが「完読」。この一点を目指せば他の問題はほぼカバーできる。

### 完食されないラーメンのたとえ

文章が完読されない理由 ＝ ラーメンが完食されない理由と対応する：

| 完食されない理由 | 完読されない理由 |
|----------------|----------------|
| 多すぎる | 文章が長すぎる |
| 伸びている | タイムリーでない |
| 麺のどごしが悪い | リズムが悪い |
| 虫が入っている | 事実誤認がある |
| 具が単調 | 繰り返しが多い |
| 食べたかった味じゃない | 求めていない内容 |

### 完読を阻む要因の優先順位

1. **事実の誤り**（発売日が間違い → 即0点）
2. **ロジックの破綻**（言葉づかいでは挽回不能）
3. **言葉づかいの荒さ**（ここは磨けば磨くほど難しくなる）

## 2. 3層構造（事実→ロジック→言葉づかい）

```
┌─────────────────┐
│   言葉づかい      │  ← 表層。美文でも事実誤認があれば0点
├─────────────────┤
│   ロジック        │  ← 中間層。事実を組み合わせて論理を形成
├─────────────────┤
│   事実            │  ← 基盤。出来事・日時・人名・場所など
└─────────────────┘
        下から順に積み上げる
```

### 実践上の意味

- どれだけ美文を連ねても、**事実に誤認があれば実用文として0点**
- ロジックがおかしな文章は、言葉づかいでは挽回できない
- 多少言葉が素朴でも、事実に誤りがなく筋が通っていれば60点前後は取れる
- **まず事実を固める、次にロジック、最後に言葉づかい**を磨く順序が正しい

> 流暢な言葉づかいで読ませたとしても、発売日が間違っていたなら0点。

## 3. 主眼と骨子

### 主眼（テーマ・切り口）

書き始める前に必ず設定する「この文章で何を言うのか」。

- 別名：コンセプト、キモ、切り口、意義付け、テーゼ
- **主眼 ≠ 単なる要約**。記者が頭を使った形跡が見える切り口が必要

**例：同じ素材でも切り口が変わる**

- マンガ編集部視点：「人気漫画家Aが音楽バンドBと、たった50人のプレミアムイベント」
- 音楽編集部視点：「音楽バンドBがトークイベント、相手は楽曲タイアップつながりの漫画家A」

### 骨子（要素・順番・軽重）

主眼を達成するための骨組み。必ず3つセットで考える：

| 要素 | 意味 | 問い |
|------|------|------|
| 要素 | 何を話題にするか | 何を？ |
| 順番 | どの順で並べるか | どれから？ |
| 軽重 | どこを重点にするか | どれくらい？ |

**軽重の格付け**：A（重点的に）/ B（基本情報として）/ C（付帯情報として）

### 取捨選択の判断基準

主眼との関係性が薄い話題は、たとえ正確な情報でも捨てる。**捨てる勇気**が完読を促す。

> 常にジャッジの基準を「主眼を伝えることに奉仕できているか」に置く。

## 4. 構造シート

書き始める前の下ごしらえ。**プラモデル化**ともいう（箱絵＝主眼、パーツ＝要素、取説＝順番・軽重）。

### 作成手順

1. **紙の上方に線を引いてテーマ欄を作る**（この段階は空欄）
2. **書こうとする話題を箇条書きで列挙する**（順番・整合性は気にしない）
3. **並んだ話題を眺めながら主眼を見定め、テーマ欄に書き込む**
4. **話題の順番を考えて番号を振る**
5. **紙を替え、主眼を書き込み、順番通りに並べ直す**（必ず手書き・書き直す）
6. **アピール優先度をABCで格付けする**

### なぜ手書きで書き直すのか

- パソコンでは「かったるい」となりがちで、いきなり書き始めるスタイルに戻ってしまう
- 順番通り書き直すことで全体の流れがクリアになり、仕上がりのイメージが明確になる
- 最初は30分かかるが、すぐ数分で埋められるようになる

### 構造シートの卒業目安

- **30本目まで**：手書きで作成（じっくり）
- **~100本目**：エディタ上にメモ書き
- **2年後**：瞬間的に主眼と骨子を脳内で構造シート化（10〜15秒）

### 書き詰まったときの対処法

- スタックした箇所を**放置して先を書く**（修正はあとからいくらでもできる）
- いま書いている内容を**人に話してみる**（デパート販売員がお客に説明するイメージ）
- 2,000字超は**章ごとに構造シートを作る**（構造シートの構造シートも必要に）

## 5. サビ頭構成

### 定義

J-POPの「サビ頭」からの転用。**冒頭に最も重要・魅力的な内容を持ってくる構成**。

```
サビ頭の基本パターン：
結論 ─ 問題提起 ─ 状況説明 ─ 付帯情報
```

ビジネス界でいう PREP法（Point→Reason→Example→Point）や SDS法（Summary→Detail→Summary）と同義。

### サビ頭が求められる理由

クリックひとつで別ページに移れるウェブ環境では、冒頭で読者の興味をグイっと引きつけないと離脱される。**冒頭でどういう話かわからない文章は完読率が格段に下がる**。

### サビ頭の例外（2割の記事）

徐々に盛り上がり最終段落のクライマックスまで引っ張る構成。使える条件：
- 取り扱う題材・各話題のレベルがすべて高い
- スロースタートでも読者がついてきてくれる自信があるとき

## 6. 推敲: 多階層重複チェック

### 推敲の3つの視点

| 視点 | 器官 | チェック内容 |
|------|------|------------|
| 意味（ミーニング） | 脳 | 事実誤認・論理破綻・文法の適切さ |
| 字面（ビジュアル） | 目 | 同字の連続・見間違えやすい箇所・段落の見た目 |
| 語呂（オーディオ） | 耳 | 音韻の重複・発音しづらい語句（音読してチェック） |

### 重複チェックはマルチスケールで

重複はあらゆるスケールで起こる。ミクロからマクロまで網羅的に発見する：

```
単語レベル  →  同じ単語の繰り返し
文節レベル  →  文末表現のダブリ（最頻出）
文型レベル  →  同じ構造の文が続く
段落レベル  →  段落構造の類似
記事レベル  →  テンプレート化
```

**鉄則**: 2連は黄色信号、**3連はアウト**

### レベル別チェックポイント

**単語レベル**
- 「の」「では」など助詞の連続に注意
- 意図的な韻踏み（「たり〜たり」）は除く

**文節レベル（最重要）**
- 文末表現が「〜した」「〜した」「〜した」と単調になっていないか
- 過去形と現在形を混在させて推進力を出す（過去の出来事でも現在形で書ける）
- 基本の文末パターン：動詞（現在/過去）・断定（だ/です）・体言止め

**文型レベル**
- 「私は〇〇しました。兄は〇〇しました。」→ 同じ構造が続く
- 意図がなければバラす

**段落レベル**
- 各段落の1文目・2文目の構造が似通っていないか
- 接続詞で始まる段落が連続していないか

**記事レベル**
- 固有名詞だけ変わったテンプレート記事になっていないか

## 7. 速度制御

### 「スピード感」の定義

```
文章のスピード感 = 情報量 ÷ 文字数
```

同じ情報を1,000文字で書いた文章より500文字で書いた文章のほうがスピード感が高い。

### 速度は「出す」より「コントロール」

初心者 → まずスピードを出せるようになる（冗長さの排除）
上級者 → 読者を完読まで導くため**適切に制御する**

> 冗長すぎれば離脱する。ソリッドすぎても無愛想で読む気が失せる。

### 速度を上げるテクニック

| テクニック | 効果 | 注意点 |
|-----------|------|--------|
| 体言止め | 文字数削減・文末変化 | 読者が省略語句を推測する負担あり。**最小限に** |
| 名詞化（〜化・〜的） | コンパクト化 | 冗長化・ぶっきらぼうの諸刃の剣。湯加減が必要 |
| 余計な言葉を削る | 直接的になる | エクスキューズ・メタ言及・定型文を削る |
| 翻訳文体の回避 | 簡潔さ | 「することができる」→「できる」に置換 |

### 速度を落とすテクニック（意図的な間）

- 具体的なエピソードを書く（臨場感を持たせる）
- セリフ・会話の引用
- 数字で具体性を追加

## 8. Web編集式文章改善テクニック集

### 係り受けの整理

**原則**: 修飾語句は**大きく長い順**に並べる。係る言葉と受ける言葉の位置を近づける。

複雑な複文は分解：
1. まず述語から読む
2. 入れ子の従属節を把握する
3. いったん全部単文にバラす
4. 必要な関係のみ複文でつなぐ

**読点の使い方**：
- 係り受けの関係を明示する（「美しい日本の、私」）
- 視覚的な区切りとしても役立てる
- 文頭一語目の直後の読点は避ける（間の抜けた印象）

### 言葉づかいの精度向上

**濁し言葉を取る勇気**
「〜など」「〜といった」「〜ら」は誠実さを示すが、キャッチーさを損なう。媒体・読者属性を見極めた上で言い切る覚悟を持つ。

**ボンヤリワードを避ける**
「作品」「企画」「楽曲」「コンテンツ」など抽象度の高い言葉は具体的な名詞に置き換える。
例：「作品」→「学園ラブコメディ」

**伝聞表現は腰を弱くする**
裏が取れた事実は断定的に語る。「〜とのこと」「〜らしい」は媒体の信頼性を下げる。

**可能表現に頼らない**
「〜できる」「〜可能だ」を連発すると安直な印象。具体的な動詞に置き換える。

**呼応する動詞を選ぶ**
- 映画は「掛かる」（「上映される」より強い）
- コーヒーは「淹れる」、抹茶は「点てる」、将棋は「指す」
- 「〜となる」の多用を避ける（最も汎用性が高い動詞＝力がない）

**「こと」「もの」を減らす**
- 「自分のことを理解すること」→「自分を理解すれば」
- 「〜ことで」→「〜すると」「〜すれば」

**数字を入れると具体性が増す**
「久しぶり」→「20年ぶり」。数字があれば「かつて」「昔」などの相対語が不要になる。

### 主観と客観のバランス

**主観の押し付けは読者を白ける**
感動を伝えたいときほど「感動」「感激」という言葉を使わない。
→ 客観的な状況を丁寧に描写するだけで伝わる。

**具体的なエピソードが独自性をつくる**
同じプレスリリースから書いても、書き手が自分で考えた切り口と具体的な取材事実があれば、そこにオリジナリティが宿る。

**「〜らしさ」「〜ならでは」には根拠を**
既知を前提とした表現は独善的。必ず客観的根拠を添える。
例：「豪華キャスト」→「映画賞受賞俳優〇〇主演」

## 9. タイトル設計

### 基本原則

タイトル ＝ 主眼（切り口）をそのまま素案にする。
文章の要約ではなく、**テーマや切り口を端的に提示するもの**。

### 集中 vs 拡散

| アプローチ | 内容 | 効果 |
|-----------|------|------|
| **集中** | 一点に絞り込む（他の要素を切り捨て） | 特定層に強く刺さる。派手な釣り針 |
| **拡散** | 複数の要素を盛り込む（全部乗せ） | 広い層にアピール。小さい釣り針を多数 |

どちらが正解かは**ターゲットを認識した上で決める**。

### タイトルで注目を引く要素

- **人物名始まり**：ヒト > コト > モノの順で注目される。「アーティストAが〇〇フェスに参戦」の方が「〇〇フェスにアーティストAが参戦」より反応がよい
- **閉じた言葉**（上級技）：ファンにだけわかる引用でピンポイントに距離を縮める
- **具体的な数字**：字面が明快になり訴求力が上がる
- **助詞終わり**：「開催へ」「決定！」など見出しで多用

### タイトルの禁止事項（Web編集の鉄則）

- 文末の「！」多用 → 安易な強調。品性を下げる
- 「!?」「!!!!」連発 → 記号インフレ
- 約物の乱用（かぎかっこ・ダブルコーテーションの過剰使用）
