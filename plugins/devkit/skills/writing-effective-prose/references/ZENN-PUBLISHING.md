# Zenn記事の作成と公開

Zenn CLIを使う技術記事リポジトリで、記事を作成し、Lintを通し、公開する。

## 最初に確認すること

- リポジトリ固有のファイル名規則
- `articles/`と画像ディレクトリの配置
- package scriptsと導入済みLintツール
- 既存記事の文体、見出し、topicsの付け方

既存規約がこの文書と異なる場合は、リポジトリ側を優先する。

## フロントマター

```yaml
---
title: "記事タイトル"
emoji: "🎯"
type: "tech"
topics: ["tag1", "tag2"]
published: false
---
```

- `type`は記事内容に合わせて`tech`または`idea`を選ぶ。
- topicsは実際の主題を表す英数字小文字にし、プラットフォーム上限を越えない。
- 初稿では`published: false`にする。
- H1はフロントマターのタイトルが担うため、本文はH2から始める。

## 企画とタイトル

1. 読者が解決したい問題を1文にする。
2. 公式資料と現行バージョンを確認する。
3. Zenn内の同じtopicsと最近の記事を調べ、既存記事との差を決める。
4. 体験、数値、問題解決、比較、入門など、内容に合うタイトル型を選ぶ。
5. タイトルと本文の約束が一致することを確認する。

トレンドへ合わせるために、未検証の効果や過剰な断定をタイトルへ入れない。

## 本文

- 冒頭で対象読者、解決する問題、前提条件を短く示す。
- 手順には期待結果を添え、読者が成功・失敗を判断できるようにする。
- コード、エラーメッセージ、設定値は実際に確認したものを使う。
- 公式仕様と自分の観察を区別する。
- 長い背景説明は必要な箇所へ移し、最初の実行例までを遠ざけない。
- Zenn固有のmessage記法は、補足や警告に必要な場合だけ使う。

```markdown
:::message
前提条件や補足情報
:::

:::message alert
実行前に確認すべき警告
:::
```

## 標準ワークフロー

```bash
# 1. 既存記事と命名規則を確認
ls articles/

# 2. published: false で記事を作成

# 3. 本文と画像を追加

# 4. リポジトリで定義されたLintを実行
pnpm run lint

# 5. 差分とプレビューを確認

# 6. 公開時だけ published: true に変更
```

個別ツールが設定されている場合は、リポジトリのscriptを優先する。代表例:

```bash
pnpm exec textlint articles/example.md
pnpm exec markdownlint-cli2 articles/example.md
pnpm exec textlint --fix articles/example.md
```

Lintルール同士が循環する場合、記事の意味を不自然に変えて回避せず、設定とルールの意図を確認する。

## 画像

- リポジトリの配置規則に従う。
- 代替テキストで、画像を見なくても要点が分かるようにする。
- UI、ログ、個人情報、通知、タブ、ブックマークの映り込みを確認する。
- 画像がなくても伝わる内容を本文に残す。

## 文体

- 既存記事の敬体・常体を維持する。
- 技術用語はプロジェクトの表記へそろえる。
- 体験談は実際の観察、判断、失敗、変更点に限る。体験を補作しない。
- 根拠と確度を保ち、自然さのために断定へ変えない。
- チャット前置き、入力指示、内部引用IDを完成稿へ残さない。

## 公開前チェック

- [ ] フロントマターの必須項目が正しい
- [ ] title、type、topicsが本文と一致する
- [ ] 見出しがH2から始まり、階層を飛ばしていない
- [ ] コードとコマンドを実行確認した
- [ ] バージョン、URL、引用を確認した
- [ ] コードブロックに言語指定がある
- [ ] 画像の配置と代替テキストが正しい
- [ ] リポジトリのLintとプレビューが通る
- [ ] `published`の値が公開意図と一致する
- [ ] 根拠のない効果、体験、推薦を加えていない
