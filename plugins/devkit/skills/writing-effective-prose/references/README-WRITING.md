# README作成

プロジェクトの利用者が、目的を理解し、最短で動作確認できるREADMEを作る。

## 読者を決める

| 読者 | 主なニーズ | 優先する内容 |
|------|-----------|--------------|
| 利用者 | インストール、使い方、トラブル解決 | 概要、Quick Start、リファレンス |
| 開発者 | ローカル環境、テスト、設計 | セットアップ、テスト、コントリビューション |
| 両方 | 位置付け、状態、ライセンス | 概要、バッジ、関連資料 |

主要な読者が使う言語を選ぶ。複数言語版を用意する場合は、正本と翻訳版の関係を明記する。

## 最小テンプレート

```markdown
# プロジェクト名

プロジェクトが何をし、誰のどの問題を解決するかを1〜3文で説明する。

## インストール

\`\`\`bash
npm install your-package
\`\`\`

## Quick Start

\`\`\`bash
your-package --help
\`\`\`

## ドキュメント

詳細は `docs/` を参照。

## ライセンス

[MIT](LICENSE)
```

## 基本構成

1. **名前と概要** — H1は1つだけにし、差分と用途を短く示す。
2. **状態表示** — バッジは読者の判断に必要なものだけを3〜5個程度に絞る。
3. **前提条件とインストール** — 対応バージョン、依存ツール、コマンドを明記する。
4. **Quick Start** — 最短で成功を確認できる例を1つ示す。
5. **詳細資料** — API、チュートリアル、設計、運用資料へリンクする。
6. **開発者向け情報** — ローカル実行、テスト、Lint、ビルドの手順を示す。
7. **参加方法** — Issue、機能要望、貢献規約、行動規範へ案内する。
8. **ライセンス** — 実ファイルと一致する名称とリンクを示す。

アーキテクチャや複数のチュートリアルをREADMEへ詰め込まず、専用文書へ分離してリンクする。

## プロジェクト種別による追加項目

| 種別 | 追加するとよい情報 |
|------|-------------------|
| ライブラリ | API例、互換性、peer dependency、代替手段との違い |
| Webアプリ | スクリーンショット、デモ、環境変数、デプロイ |
| CLI | コマンド、引数、終了コード、使用例 |
| OSS | 貢献方法、セキュリティ報告窓口、引用方法 |
| 社内ツール | 所有チーム、問い合わせ先、運用手順、権限申請 |

## Markdown規範

- H1はプロジェクト名の1つだけにする。
- 見出し階層を飛ばさない。
- コードブロックへ言語を指定する。
- 画像に内容を説明する代替テキストを付ける。
- リンクラベルだけで行き先が分かるようにする。
- 表は比較や対応に必要な場合だけ使う。
- バージョンやコマンドを実環境で確認する。

## 公開前チェック

- [ ] 概要だけで用途と対象読者が分かる
- [ ] インストールとQuick Startを新しい環境で再現できる
- [ ] コマンド、パス、設定名、対応バージョンが現行である
- [ ] リンク切れがない
- [ ] コードブロックに言語指定がある
- [ ] 画像に代替テキストがある
- [ ] READMEとライセンスファイルの表記が一致する
- [ ] 詳細すぎる手順を専用文書へ分離している

## 避けること

| 問題 | 修正 |
|------|------|
| バッジを大量に並べる | 読者の判断に使うものだけ残す |
| 複数の長いチュートリアルを入れる | 最小例を1つ残し、詳細へリンクする |
| 未確認のコマンドを載せる | 新しい環境で実行して結果を確認する |
| 更新予定や品質を自己宣言する | 実際の状態、CI、変更履歴で示す |
| リリース履歴を本文へ蓄積する | CHANGELOGやReleasesへ分離する |
