# レプリケーション・HA・モニタリング リファレンス

PostgreSQLにおける物理・論理レプリケーションの設定手順、レプリケーションスロットの管理、高可用性（HA）設計の考え方、およびCumulative Statistics Systemと pg_stat_statements を活用したモニタリング手法をまとめる。

---

## 1. レプリケーション概要

PostgreSQLは2種類のレプリケーションをサポートする。

| 種別 | 仕組み | 粒度 |
|------|--------|------|
| **物理レプリケーション** | WALのバイトレベルの変更をレプリカに転送 | クラスター全体（すべてのDB） |
| **論理レプリケーション** | WALをデコードしてDMLとして転送 | テーブル単位（フィルタリング可） |

---

## 2. 物理レプリケーション（ストリーミング）

WALをリアルタイムでストリーミングする方式。データ量に関わらず低レイテンシーの同期が可能。

### 特徴と制約

| 特徴 | 説明 |
|------|------|
| **全体複製** | クラスター全体が複製対象。個別DB・テーブルの選択不可 |
| **バージョン制約** | 同一メジャーバージョン間のみ動作（ページフォーマットが異なるため） |
| **読み取り専用レプリカ** | レプリカはデフォルトで読み取り専用 |

### 同期モード

| モード | 動作 | リスク |
|--------|------|--------|
| **非同期（デフォルト）** | プライマリはレプリカのACK待ちなしでコミット | レプリカが若干遅延する可能性 |
| **同期** | プライマリは指定数のレプリカのACKを待ってコミット | レプリカが応答不能でブロック発生 |

> **同期レプリケーション使用時の推奨**: 最低2台以上のレプリカを用意し、全体の50〜75% のACKを要求する設定にすること。全台必須にすると1台の障害でプライマリがブロックされる。

---

## 3. 論理レプリケーション（パブリケーション/サブスクリプション）

WALをデコードしてDML操作として転送するパブリッシュ/サブスクライブモデル。

### 特徴と利点

| 特徴 | 説明 |
|------|------|
| **テーブル単位** | 特定テーブルのみを選択して配信可能 |
| **クロスバージョン** | 異なるメジャーバージョン間でも動作 |
| **行フィルタリング** | WHERE句で特定行のみを特定サブスクライバーに送信 |
| **カスケード** | 物理レプリカからさらに論理レプリケーションを設定可能 |

### 制約

| 制約 | 説明 |
|------|------|
| **DDL非対応** | テーブル作成・変更などのスキーマ変更は転送されない |
| **スキーマ事前準備** | サブスクライバー側のテーブルは事前に作成が必要 |

---

## 4. レプリケーションスロット

レプリカ（物理）またはサブスクライバー（論理）がデータを受信するまで、WALファイルを保持し続ける仕組み。

```sql
-- アクティブなレプリケーションスロットを確認
SELECT * FROM pg_replication_slots;
```

### スロットの注意点

| リスク | 説明 | 対策 |
|--------|------|------|
| **ディスク圧迫** | レプリカが長期切断されるとWALが蓄積してプライマリのディスクが枯渇 | `pg_replication_slots` を定期監視 |
| **同期モードでのブロック** | 同期レプリケーション中にスロットレプリカが切断されると、プライマリがブロック | 十分なレプリカ数と安定ネットワークを確保 |

---

## 5. 物理 vs 論理 — 選択ガイド

| ユースケース | 推奨 |
|------------|------|
| 温スタンバイ・読み取りオフロード | 物理（ストリーミング） |
| 特定テーブルのみを別クラスターへ配信 | 論理 |
| 部署ごとにフィルタリングして配信 | 論理（行フィルタリング） |
| メジャーバージョンアップグレード | 論理（バージョン間対応） |
| ディザスタリカバリ（全体フェイルオーバー） | 物理 |
| 両方の組み合わせ | 物理 + 論理（カスケード） |

---

## 6. ストリーミングレプリケーション設定手順

### Step 1: レプリケーションロール作成（プライマリ）

```sql
-- REPLICATIONロール属性を持つユーザーを作成
CREATE ROLE repusr LOGIN REPLICATION PASSWORD 'rep-password';
```

### Step 2: pg_hba.conf 修正（プライマリ）

```
# TYPE   DATABASE      USER     ADDRESS          METHOD
host     replication   repusr   192.168.1.2/32   scram-sha-256
```

```bash
# pg_hba.confを反映（再起動不要）
sudo systemctl reload postgresql-16.service
```

### Step 3: レプリカクラスターのデータディレクトリを準備（レプリカ）

```bash
# PostgreSQLを停止し、データディレクトリを空にする
sudo systemctl stop postgresql-16
sudo rm -rf /var/lib/pgsql/16/data/*
```

### Step 4: ベースバックアップ作成（レプリカから実行）

```bash
# postgres ユーザーで実行
sudo su - postgres

# pg_basebackup で完全バックアップを取得しレプリケーション設定も自動化
pg_basebackup \
    -h 192.168.1.1 \
    -U repusr \
    -D /var/lib/postgresql/16/data \
    -c fast \   # チェックポイント即時実行
    -R \         # postgresql.auto.conf と standby.signal を自動生成
    -S 'replica1' \  # スロット名
    -C \         # スロットをプライマリ側に作成
    -P           # 進捗表示
```

| フラグ | 説明 |
|--------|------|
| `-c fast` | プライマリでチェックポイントを即時実行してバックアップ開始点を確定 |
| `-R` | `postgresql.auto.conf` に接続情報を追記し、`standby.signal` を作成 |
| `-S <name>` | 接続するスロット名 |
| `-C` | スロットが存在しない場合にプライマリ側で作成 |
| `-P` | バックアップ進捗を表示 |

### Step 5: レプリカ起動と検証

```bash
# レプリカを起動
sudo systemctl start postgresql-16
```

```sql
-- プライマリ側: レプリカの接続を確認
SELECT * FROM pg_stat_replication;

-- プライマリ側: スロットの状態を確認
SELECT * FROM pg_replication_slots;

-- レプリカ側: WAL受信状態を確認
SELECT * FROM pg_stat_wal_receiver;
```

### フェイルオーバー（レプリカの昇格）

```bash
# pg_ctl でレプリカを昇格（読み書き可能に）
pg_ctl promote -D /var/lib/pgsql/16/data
```

```sql
-- または SQL 関数で昇格
SELECT pg_promote();
```

> **自動フェイルオーバーはPostgreSQLに組み込まれていない。** Patroni または repmgr などの外部ツールが必要。

## 7. 論理レプリケーション設定手順

### Step 1: WALレベルの設定変更（ソースクラスター）

```
# postgresql.conf
wal_level = logical    # minimal / replica / logical のいずれか
```

```bash
# wal_level変更は再起動が必要
sudo systemctl restart postgresql-16
```

### Step 2: パブリケーション作成（ソースDB）

```sql
-- 特定テーブルのパブリケーション
CREATE PUBLICATION rental_pub FOR TABLE bluebox.rental;

-- 行フィルタリング付きパブリケーション
CREATE PUBLICATION rental_pub_filtered
    FOR TABLE bluebox.rental
    WHERE (store_id > 100);

-- スキーマ全体のパブリケーション
CREATE PUBLICATION my_schema_pub FOR TABLES IN SCHEMA bluebox;

-- データベース全体のパブリケーション
CREATE PUBLICATION all_tables_pub FOR ALL TABLES;

-- パブリケーション一覧
SELECT * FROM pg_publication;
```

### Step 3: サブスクリプション作成・検証（ターゲットDB）

```sql
-- ターゲット側にテーブルが事前に存在している必要がある
CREATE TABLE bluebox.rental (
    rental_id int8 GENERATED BY DEFAULT AS IDENTITY NOT NULL,
    rental_period tstzrange NULL,
    inventory_id int4 NOT NULL,
    customer_id int4 NOT NULL,
    last_update timestamptz DEFAULT now() NOT NULL,
    store_id int4 NULL,
    CONSTRAINT rental_pkey PRIMARY KEY (rental_id)
);

-- サブスクリプション作成（即時ストリーミング開始）
CREATE SUBSCRIPTION sub_rental
    CONNECTION 'host=192.168.1.1 dbname=bluebox user=repusr password=rep-password'
    PUBLICATION rental_pub
    WITH (enabled = true, create_slot = true, slot_name = rental_pub1);

-- フィルタリング済みパブリケーションへのサブスクリプション
CREATE SUBSCRIPTION sub_rental_filtered
    CONNECTION 'host=192.168.1.1 dbname=bluebox user=repusr password=replicapw1'
    PUBLICATION rental_pub_filtered;

-- サブスクリプションの状態確認
SELECT * FROM pg_stat_subscription WHERE subname = 'sub_rental_filtered';

-- スロットの統計確認（ソース側）
SELECT * FROM pg_stat_replication_slots;

-- 行数の一致を検証
-- ソース側
SELECT count(*) FROM bluebox.rental;
-- ターゲット側（同数になるはず）
SELECT count(*) FROM bluebox.rental;
```

> **初回サブスクリプション**: 大規模テーブルの場合、初期データ同期が完了するまでターゲット側に行が現れない。`pg_stat_activity` で `backend_type = 'logical replication worker'` を確認して進捗を把握する。

## 8. PostgreSQL モニタリング

### 8.1 エラーログの設定

```
# postgresql.conf
logging_collector = on         # ファイルへのログ出力を有効化（デフォルト off）
log_destination = 'stderr'     # stderr / csvlog / jsonlog / syslog / eventlog
```

```sql
-- ログコレクターの状態確認
SHOW logging_collector;

-- ログファイルの保存先確認
SHOW log_directory;

-- 現在のログファイル名確認
SELECT pg_current_logfile();
```

### ログレベル（重要度の昇順）

| レベル | 説明 |
|--------|------|
| `WARNING` | 潜在的な問題（デフォルトで記録） |
| `ERROR` | コマンドが中断 |
| `LOG` | 管理情報（起動・リカバリ等） |
| `FATAL` | セッションが中断 |
| `PANIC` | インスタンス全体が中断 |

### 8.2 Cumulative Statistics System（累積統計システム）

PostgreSQL 16 では22個のビューで内部メトリクスを提供。集計済みデータで負荷が少ない。

**注意点:**
- 書き込みは1秒間隔で反映
- クラッシュ・フェイルオーバー・PITR後はリセットされる
- autovacuum はこの統計を使用するため、`pg_stat_reset()` は慎重に実行

### 主要なpg_stat_*ビュー

| ビュー | 用途 |
|--------|------|
| `pg_stat_activity` | 現在のセッションとクエリ（動的ビュー） |
| `pg_stat_database` | DB全体の行数・コミット数・ロールバック数 |
| `pg_stat_user_tables` | テーブルごとのスキャン数・行操作・VACUUM/ANALYZE状況 |
| `pg_stat_all_tables` | ユーザーテーブル + システムテーブルの統計 |
| `pg_stat_user_indexes` | インデックスごとのスキャン数・読み取り行数 |
| `pg_statio_user_tables` | テーブルのブロック読み取り（ディスク vs メモリ） |
| `pg_stat_replication` | ストリーミングレプリケーションの状態 |
| `pg_stat_wal_receiver` | WAL受信の状態（レプリカ側） |
| `pg_replication_slots` | レプリケーションスロットの状態 |
| `pg_stat_bgwriter` | バックグラウンドライターの統計 |

### 8.3 主要な監視クエリ例

#### アクティブな接続の監視

```sql
-- 現在アクティブなクエリを表示
SELECT
    psa.datname,
    psa.usename,
    psa.query,
    psa.xact_start,
    psa.query_start,
    now() - psa.query_start AS duration
FROM pg_stat_activity AS psa
WHERE psa.state = 'active'
ORDER BY duration DESC;
```

#### ロック待ちの検出

```sql
-- ロック待ちが発生しているセッションを特定
SELECT
    blocked.pid AS blocked_pid,
    blocked.query AS blocked_query,
    blocking.pid AS blocking_pid,
    blocking.query AS blocking_query,
    now() - blocked.query_start AS wait_duration
FROM pg_stat_activity AS blocked
JOIN pg_stat_activity AS blocking
    ON blocking.pid = ANY(pg_blocking_pids(blocked.pid))
WHERE cardinality(pg_blocking_pids(blocked.pid)) > 0;
```

#### レプリケーションラグの監視

```sql
-- レプリケーションの遅延を確認（プライマリ側）
SELECT
    client_addr,
    state,
    sent_lsn,
    write_lsn,
    flush_lsn,
    replay_lsn,
    write_lag,
    flush_lag,
    replay_lag
FROM pg_stat_replication;
```

#### Dead tupleが多いテーブルの特定

```sql
-- Dead tupleの割合が高いテーブルを検出
SELECT
    schemaname,
    relname,
    n_live_tup,
    n_dead_tup,
    round((n_dead_tup::numeric / NULLIF(n_live_tup, 0)) * 100, 2) AS pct_dead,
    last_autovacuum,
    last_autoanalyze
FROM pg_stat_user_tables
WHERE n_dead_tup > 1000
ORDER BY pct_dead DESC
LIMIT 20;
```

#### シーケンシャルスキャンの多いテーブル（インデックス追加候補）

```sql
SELECT
    psut.schemaname,
    psut.relname,
    psut.seq_scan,
    psut.last_seq_scan,
    psut.idx_scan
FROM pg_stat_user_tables AS psut
WHERE psut.schemaname = 'public'
    AND psut.seq_scan > 0
ORDER BY psut.seq_scan DESC;
```

#### ディスク vs メモリの読み取り比率（キャッシュヒット率）

```sql
SELECT
    relname,
    heap_blks_read AS disk_reads,
    heap_blks_hit AS cache_hits,
    round(
        heap_blks_hit::numeric / NULLIF(heap_blks_read + heap_blks_hit, 0) * 100,
        2
    ) AS cache_hit_ratio
FROM pg_statio_user_tables
WHERE heap_blks_read + heap_blks_hit > 0
ORDER BY disk_reads DESC
LIMIT 20;
```

## 9. pg_stat_statements によるクエリパフォーマンス監視

クエリレベルのパフォーマンスデータを収集する拡張機能。ほぼすべての主要クラウドプロバイダーでデフォルト有効。

### セットアップ

```
# postgresql.conf
shared_preload_libraries = 'pg_stat_statements'  # 再起動が必要

# オプション設定
pg_stat_statements.max = 5000        # 追跡するクエリ数（デフォルト 5000）
pg_stat_statements.track = 'top'     # top / all / none
```

```sql
-- 拡張機能を有効化（再起動後）
CREATE EXTENSION pg_stat_statements;

-- 有効化確認
\dx
```

> **詳細ガイド**: → [POSTGRESQL-EXTENSIONS.md](POSTGRESQL-EXTENSIONS.md) の pg_stat_statements セクションを参照

### 低速クエリの特定

```sql
-- 平均実行時間が長いクエリ上位10件
SELECT
    pss.query,
    pss.calls,
    round(pss.total_exec_time::numeric, 2) AS total_ms,
    round(pss.mean_exec_time::numeric, 2) AS mean_ms,
    round(pss.min_exec_time::numeric, 2) AS min_ms,
    round(pss.max_exec_time::numeric, 2) AS max_ms,
    round(pss.stddev_exec_time::numeric, 2) AS stddev_ms
FROM pg_stat_statements AS pss
ORDER BY pss.mean_exec_time DESC
LIMIT 10;

-- 総実行時間が最も長いクエリ（システム全体のボトルネック）
SELECT
    pss.query,
    pss.calls,
    round(pss.total_exec_time::numeric, 2) AS total_ms,
    round(pss.mean_exec_time::numeric, 2) AS mean_ms
FROM pg_stat_statements AS pss
ORDER BY pss.total_exec_time DESC
LIMIT 10;
```

### 特定クエリのパフォーマンス追跡

```sql
-- queryid を特定
SELECT queryid
FROM pg_stat_statements
WHERE query LIKE '%bluebox.rental%';

-- 特定クエリの詳細メトリクス
SELECT
    pss.calls,
    pss.total_exec_time,
    pss.min_exec_time,
    pss.max_exec_time,
    pss.mean_exec_time,
    pss.stddev_exec_time,
    pss.rows
FROM pg_stat_statements AS pss
WHERE pss.queryid = 427240950862911343;
```

| メトリクス | 説明 |
|-----------|------|
| `calls` | 累積実行回数 |
| `total_exec_time` | 累積実行時間（ms） |
| `min_exec_time` | 最速実行時間 |
| `max_exec_time` | 最遅実行時間 |
| `mean_exec_time` | 平均実行時間 |
| `stddev_exec_time` | 実行時間の標準偏差 |
| `rows` | 累積返却行数 |

### 統計のリセット

```sql
-- 全クエリ統計をリセット（パラメーター調整の前後比較に使用）
SELECT pg_stat_statements_reset();
```

## 10. 監視のベストプラクティス

### 時系列比較の重要性

単点の値ではなく、**変化のトレンド**を把握することが重要。以下を定期的に収集して比較できるようにする。

| 収集対象 | 方法 |
|---------|------|
| クエリ実行時間の推移 | pg_stat_statements を定期スナップショット |
| Dead tuple の増加率 | pg_stat_user_tables を Cron で定期保存 |
| レプリケーションラグ | pg_stat_replication を監視ツールに連携 |
| キャッシュヒット率 | pg_statio_user_tables を定期収集 |

### モニタリングの注意点

| 原則 | 説明 |
|------|------|
| **Observer Effect** | 監視自体がシステムに負荷をかける。適切な間隔設定が重要 |
| **統計リセット注意** | `pg_stat_reset()` は autovacuum の判断にも影響するため慎重に |
| **pg_stat_statementsオーバーヘッド** | 負荷の高いシステムでは有効化による追加負荷を考慮 |

## まとめ

| 概念 | 要点 |
|------|------|
| 物理レプリケーション | WALバイトレベル転送。クラスター全体。同一メジャーバージョン必須 |
| 論理レプリケーション | DMLとして転送。テーブル単位・行フィルタ・クロスバージョン対応。DDL非対応 |
| レプリケーションスロット | WAL保持を保証。長期切断でディスク圧迫リスクあり |
| 同期レプリケーション | データロス防止だがブロックリスク。十分なレプリカ数と部分ACKが必要 |
| 自動フェイルオーバー | PostgreSQL組み込みなし。PatroniまたはrepmgrなどHA管理ツールが必要 |
| エラーログ | `logging_collector = on` で有効化。csvlog/jsonlogで外部ツール連携容易 |
| Cumulative Statistics | 22個のpg_stat_*ビュー。クラッシュでリセット。1秒遅延あり |
| pg_stat_statements | クエリ統計の最重要拡張。デフォルト無効。クラウドはほぼデフォルト有効 |
| 監視の鉄則 | 単点ではなく時系列比較。比較基準がないメトリクスは無意味 |
