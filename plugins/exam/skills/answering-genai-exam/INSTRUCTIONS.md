# answering-genai-exam: 解答生成ワークフロー

生成AI活用試験の問題画像（見開きスクリーンショット）から、各小問の「対話の要点」と提出物を生成し、入力画像と同じ場所の `answers/<問番号>/` に保存する。

> このスキルは特定の試験ブランド・主催団体に依存しない汎用設計。同様な問題形式（左ページ=問題文＋小問・右ページ=回答欄）であれば何にでも対応する。

---

## 問題形式の前提（読み取り対象）

各画像は **1画像 = 1大問（問一・問二…）の見開き**。同一テンプレートを想定する。

- **左ページ = 問題**: ヘッダー（試験名・SECTION・テーマ）／状況設定文／回答指示／【条件】【制約】ボックス（任意）／小問リスト `(1)(2)(3)…`（各小問にタイトル・指示文・提出物名）。
- **右ページ = 回答欄**: 小問ごとに「対話の要点」入力欄（プレースホルダ `自分: ○○ → AI: ○○ → 自分: ○○ → AI: ○○`・`0/1000` の文字数上限表示）と「ファイル提出」欄（提出物名 例 `q1_1.md`）。

各小問に対し **2 種類の成果物** を生成する:

1. **対話の要点** — 生成AIと対話して答えに至った流れ（`自分: ○○ → AI: ○○ → …` 形式・**最大1000文字**）。全小問必須。
2. **提出物** — 小問ごとに指定された拡張子に応じた成果物（後述の拡張子分岐）。

---

## ワークフロー

### Step 0: 入力受領

- 引数または対話で **ローカル画像パス（複数可）** を受け取る。
- 各パスの存在を確認する。画像でないもの・存在しないものは除外し、報告に明記する。
- 出力先は「**入力画像と同じディレクトリ**」と確定する（例: `…/exam/toi1.png` → `…/exam/answers/問一/`）。

### Step 1: 問題構造の抽出（呼び出し側＝本体が実行）

各画像を `Read` で読み、以下を抽出してテキスト化する。これが後で solver へ渡す一次情報になる。

- 試験名（あれば。**特定ブランド名は記録せず汎用化**して保持）／ SECTION ／ テーマ ／ 問番号
- 状況設定文 ／ 回答指示 ／【条件】【制約】
- 小問リスト `[(番号, タイトル, 指示文, 提出物名)]`

抽出結果を **構造化メモ** にまとめる。例:

```
[問一] SECTION xx / テーマ: <テーマ名>
状況設定: <状況設定文>
回答指示: 各小問について対話の要点を「自分:→AI:→…」形式で記入。最大1000字。
小問:
  (1) <タイトル> / 指示: <指示文> / 提出物: q1_1.md
  (2) <タイトル> / 指示: <指示文> / 提出物: q1_2.png
  (3) <タイトル> / 指示: <指示文> / 提出物: q1_3.jpg
```

### Step 2: 曖昧点を AskUserQuestion で一括解消（🔴 subagent 起動前に必ず）

**ここがユーザー要求「subagent 起動前に全解消」の実装ポイント。** background で起動する exam-solver は AskUserQuestion を使えないため、確認は必ず呼び出し側（本体）がこの時点で済ませる。明確な場合は飛ばし、**推測で進めない**。

確認候補と、実際に呼べる AskUserQuestion の例:

```python
AskUserQuestion(
    questions=[
        {
            "question": "この試験の分野・到達レベルを教えてください（解答の専門性・粒度に影響します）。",
            "header": "試験分野",
            "options": [
                {"label": "基礎/入門レベル", "description": "概念理解中心。平易な用語で簡潔に解答"},
                {"label": "実務/応用レベル", "description": "業務適用を想定。具体例・トレードオフを含めて解答"},
                {"label": "専門/上級レベル", "description": "高度な技術判断・設計根拠まで踏み込んで解答"},
            ],
            "multiSelect": False,
        },
        {
            "question": "小問(2)の提出物 q1_2.png は『比較図』指定ですが、出力形式の方針を選んでください。",
            "header": "提出物形式",
            "options": [
                {"label": "画像生成指示として出力", "description": "描画要素・配置・矢印・ラベルを自然言語で詳述（既定）"},
                {"label": "Mermaid等のコードも併記", "description": "画像生成指示に加え、図のソース記法も補助的に添える"},
            ],
            "multiSelect": False,
        },
        {
            "question": "状況設定が抽象的で解答方針に複数解釈があります。どの方向で求解しますか？",
            "header": "解答方針",
            "options": [
                {"label": "解釈A", "description": "（抽出した状況設定に基づく具体的な解釈Aを記述）"},
                {"label": "解釈B", "description": "（代替解釈Bを記述）"},
                {"label": "両論併記", "description": "両解釈を提示し、より妥当な方を主張として採用"},
            ],
            "multiSelect": False,
        },
        {
            "question": "出力先 answers/問一/ に既存ファイルがあります。どう扱いますか？",
            "header": "既存answers",
            "options": [
                {"label": "上書き", "description": "既存の同名ファイルを新しい解答で置き換える"},
                {"label": "別名で保存", "description": "answers/問一_v2/ など別ディレクトリに保存し既存を温存"},
            ],
            "multiSelect": False,
        },
    ]
)
```

- 上記4種は **必要なものだけ** 1回の AskUserQuestion にまとめて提示する（複数 question を1コールに同梱）。
- 提出物拡張子が読み取り不可で出力形式を判断できない小問があれば、ここで必ず確認する（並列 solver 起動後は解消できない）。
- 確定した回答は Step 3 で各 solver のプロンプトへ「確定方針」として埋め込む。

### Step 3: 求解の振り分け

抽出メモ（Step 1）と確定方針（Step 2）が揃ったら、画像枚数と実行環境で経路を選ぶ。

- **画像 1枚**: 呼び出し側が直接 Step 4 を実行する（並列化不要）。
- **画像 2枚以上 / Claude Code**: **Workflow で問ごとに並列求解（第一推奨）**。Workflow 不可なら background-Agent 手動起動（代替）。
- **画像 2枚以上 / 単一エージェント環境（Codex 等）**: 逐次フォールバック。

いずれの並列経路でも、各 worker へは以下を **プロンプトに全文埋め込んで** 渡す（worker は Desktop など作業ディレクトリ外の画像を読めない／Workflow 標準 agent はスキル未 preload のため、テキストで完結させる）:

1. **Step 1 の抽出メモ**（その画像の大問1つ分・小問リスト含む）
2. **Step 2 の確定方針**（試験分野・レベル・解答方針・上書き可否など）
3. **出力先パス**（`<入力画像dir>/answers/<問番号>/`・絶対パス）
4. **問番号**（`問一` など）
5. **出力規約 / 求解契約**（`references/OUTPUT-FORMAT.md` 準拠＝ファイル名規則・拡張子分岐・対話要点≤1000字。Workflow ではこの契約を凝縮してプロンプトに埋め込む）

> 🔴 **共通の掟**: どの並列 worker も AskUserQuestion を使えない。本質的な曖昧点は Step 2 で本体が解消済みであること。不明点は worker が自己判断（best-effort）し報告/構造化結果に明記する。

#### 経路A: Workflow 並列求解（Claude Code・第一推奨）

`parallel()` で全画像（大問）を一斉ファンアウトし、結果を `schema` で構造化受領する。手動の完了待ちが消え、`/workflows` で進捗が可視化され、失敗時は journal resume で未完の問だけ再実行できる。

- スクリプト雛形・`args` の形・埋め込む求解契約・本体側の集約手順は **`references/WORKFLOW-SCRIPT.md` を参照**。
- 手順の要点:
  1. Step 1・Step 2 の成果を `args = { approach, questions: [{ q, outputDir, memo }, …] }` に詰める（実 JSON 値で渡す）。
  2. `Workflow({ script: <雛形>, args })` を呼ぶ（求解契約は雛形内の定数に埋め込み済み・`agentType` は指定しない）。
  3. 返却 `{ solved: [...] }` を Step 5 の報告に使う。`null` の問だけ経路B/逐次で再求解する。

> **Workflow opt-in の根拠**: Workflow ツールは無断起動禁止だが、本スキルの指示が呼び出しを命じることが正当な opt-in 経路（スキル/コマンドの指示による起動は許可される）。

#### 経路B: background-Agent 手動起動（Workflow 不可時の代替）

Workflow が使えない／無効化されている場合は、画像ごとに `exam-solver` agent を `run_in_background: true` で並列起動する（チーム編成は orchestrating-teams の手順に従う）。例（概念）:

```
Agent(subagent_type: "exam-solver", run_in_background: true, prompt: <問一の抽出メモ + 確定方針 + 出力先 + 出力規約>)
Agent(subagent_type: "exam-solver", run_in_background: true, prompt: <問二の抽出メモ + 確定方針 + 出力先 + 出力規約>)
…
```

各 solver（preload 済みの本スキルに従う）は担当1大問の全小問を Step 4 の手順で解き、ファイル出力後に報告する。本体は全 solver の完了を待って Step 5 へ進む。

#### 経路C: 単一エージェント環境のフォールバック（Codex 等・agent も Workflow も使えない場合）

agent も Workflow も起動できない実行環境では、**呼び出し側が画像を1枚ずつ逐次処理** する:

1. 画像1の Step 1（抽出）→ Step 2（必要なら確認）→ Step 4（全小問求解・出力）。
2. 画像2へ進み同様に処理。以降の画像も順に繰り返す。

並列化しないだけで成果物の中身・出力規約は同一。Codex では exam-solver agent は配布されず、このスキル単体（逐次）で完結する。

### Step 4: 各小問の求解（直接 or solver 内）

抽出メモの各小問について、以下を順に行う:

a. **模範解答の導出**: 状況設定・指示文・確定方針に沿って答えを論理的に組み立てる。必要に応じて GenAI 設計や実装のドメイン知識（`ai:designing-genai-patterns` 等）を活用する。

b. **「対話の要点」の再構成**（≤1000字・後述フォーマット厳守）: 答えに至る自然で説得力のある思考プロセスを `自分: ○○ → AI: ○○ → …` 形式で表現する。実際の往復ログではなく **模範的な対話を再構成** する。内容は提出物本体（c）と整合させる。

c. **提出物本体の生成**（拡張子分岐・後述）。

d. **出力**: `<入力画像dir>/answers/<問番号>/` に `references/OUTPUT-FORMAT.md` のファイル名規則で保存する。`Write` は親ディレクトリを自動生成する。

### Step 5: 集約・報告

- 全問完了後、生成ファイル一覧と各問の要旨を報告する。
- 並列時は各 worker が報告した「自己判断した不明点」を集約し、ユーザーへ提示する（Workflow 経路では返却 `solved[].selfJudged`、background-Agent 経路では各 solver の完了報告から拾う）。
- 対話要点の文字数が規定（≤1000字）を満たしているか最終確認する（Workflow 経路では `solved[].charCounts` で照合）。

## 提出物の拡張子による出力分岐（🔴 核心ロジック）

各小問の提出物名（例 `q1_2.png`）の **拡張子** で生成する成果物の種類を決める。詳細表は `references/OUTPUT-FORMAT.md` を参照。

| 拡張子 | 生成するもの |
|--------|-----------|
| `.md` | markdown 回答本文（見出し・表・箇条書きで論理的に記述） |
| `.png` `.jpg` `.jpeg` `.gif` `.svg` 等の画像 | **別の画像生成AIに渡すテキスト指示**（描くべき要素・配置・矢印・ラベル・スタイルを自然言語で詳述。「画像生成AI使用不可（手描き指定）」の図でも、人間/別AIが描けるよう指示文として出す） |
| `.py` `.ts` `.js` `.sql` `.ipynb` 等のコード | 実際に動くソースコード（コメント付き） |
| `.csv` `.json` `.yaml` 等のデータ | 該当形式のデータ |
| 不明・読み取り不可 | Step 2 の AskUserQuestion で確認（並列 solver 時は本体が事前解消） |

## 対話の要点フォーマット（全小問共通・必須）

- 形式: `自分: ○○ → AI: ○○ → 自分: ○○ → AI: ○○ → …`
- **最大1000文字（厳守）**。超過したら要約して収める。
- 「自分」＝受験者: 問題の言い換え・仮説提示・追及・反例検証・要約要求を担う。
- 「AI」＝生成AI: 概念提示・定義・例示・反証・整理を担う。
- **模範的な対話を再構成** する（実ログの転記ではなく、答えに至る思考プロセスの再現）。
- 内容は提出物本体（回答）と整合させる（対話で導いた結論 ＝ 提出物の主張）。
- 出力ファイル末尾に文字数を `（XX/1000字）` の形で付記する。
