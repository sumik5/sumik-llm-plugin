# OUTPUT-FORMAT: 出力ファイル名規則・拡張子分岐・テンプレート

このファイルは `answering-genai-exam` スキルおよび `exam-solver` agent が共有する **出力規約の単一基準**。ファイル名・拡張子分岐・対話要点フォーマット・画像生成指示文の書き方をここに集約する。

---

## 1. 出力ディレクトリとファイル名規則

入力画像 `…/exam/toi1.png`（問一）に対する出力例:

```
…/exam/answers/
└── 問一/
    ├── _summary.md              # 任意: 大問全体の概要＋小問へのリンク
    ├── q1_1.md                  # 小問(1) 提出物（.md 指定 → 回答本文）
    ├── q1_1_対話の要点.md        # 小問(1) 対話の要点（≤1000字）
    ├── q1_2_画像生成指示.md      # 小問(2) 提出物が q1_2.png → 画像生成AIへの指示
    ├── q1_2_対話の要点.md        # 小問(2) 対話の要点
    ├── q1_3_画像生成指示.md      # 小問(3) 提出物が q1_3.jpg → 画像生成AIへの指示
    └── q1_3_対話の要点.md        # 小問(3) 対話の要点
```

### 規則

- **問番号フォルダ**: 画像から読んだ問番号（`問一`/`問二`…）。読めなければ `問<連番>` または画像 basename を使う。
- **出力先の親**: 必ず「入力画像と同じディレクトリ」の `answers/<問番号>/`。`Write` は親ディレクトリを自動生成する。
- **提出物本体**:
  - `.md` → 提出物名そのまま（`q1_1.md`）に回答本文を書く。
  - 画像系（`.png`/`.jpg`/`.jpeg`/`.gif`/`.svg` 等）→ `<basename>_画像生成指示.md`（`q1_2.png` → `q1_2_画像生成指示.md`）。冒頭に「対象提出物: q1_2.png（この指示を画像生成AIに渡して画像化する）」を明記する。
  - コード系（`.py`/`.ts`/`.js`/`.sql`/`.ipynb` 等）→ 提出物名そのまま（`q1_3.py`）にソースコードを書く。
  - データ系（`.csv`/`.json`/`.yaml` 等）→ 提出物名そのままに該当形式のデータを書く。
- **対話の要点**: `<basename>_対話の要点.md`（全小問必須）。本文1行目に小問タイトル、本体に `自分:→AI:→…` を記述し、末尾に文字数（例 `（XX/1000字）`）を付記する。
- **`_summary.md`**: 任意。大問の概要と各小問への相対リンクを並べると、後から見返しやすい。

### 実バイナリレンダリング時の追加出力（opt-in）

Step 2 で「実画像もレンダリング」を選んだ場合、上記に **加えて** 実バイナリと原稿を併置する（既定では生成しない）:

- 画像系: `q1_2_画像生成指示.md`（既定）＋ `q1_2.png`（実画像）＋ `q1_2_src.svg`（再レンダリング用 SVG 原稿）。
- `.md` 回答: `q1_1.md`（既定）＋ `q1_1.pdf`（実PDF）＋ `q1_1_src.md`（PDF 原稿。`q1_1.md` と同一なら省略可）。

手順・ツールチェーン・グリフ脱落の罠・検証は `references/RENDERING.md` を参照。

## 2. 拡張子分岐の詳細表（🔴 核心ロジック）

提出物名の拡張子から、生成する成果物の種類・出力ファイル名・記述要件を一意に決める。

| 拡張子グループ | 例 | 提出物の意味 | 生成するもの | 出力ファイル名 | 記述要件 |
|--------------|----|------------|------------|--------------|---------|
| markdown | `.md` | 回答本文 | markdown 文書 | 提出物名そのまま `q1_1.md` | 見出し・表・箇条書きで論理構成。結論先行（PREP）。根拠を伴う |
| 画像 | `.png` `.jpg` `.jpeg` `.gif` `.svg` `.webp` | 図（比較図・構造図・概念図） | **画像生成AIへのテキスト指示** | `<basename>_画像生成指示.md` | §3 のガイドに従い、描画要素・配置・矢印・ラベル・スタイルを自然言語で詳述 |
| コード | `.py` `.ts` `.js` `.tsx` `.sql` `.ipynb` `.go` `.java` 等 | ソースコード | 実際に動くコード | 提出物名そのまま `q1_3.py` | 動作する完全なコード。コメント・docstring 付き。`writing-clean-code` 準拠 |
| データ | `.csv` `.json` `.yaml` `.yml` `.tsv` | データファイル | 該当形式のデータ | 提出物名そのまま `q1_4.csv` | 仕様に沿った妥当なデータ。スキーマ整合・エンコーディング配慮 |
| 不明 / 読み取り不可 | — | — | — | — | スキル: Step 2 で AskUserQuestion 確認。solver: 自己判断（best-effort）＋報告に明記 |

> **画像系の重要原則**: 「画像生成AIの使用は不可（丸・矢印・単語のみで手描き）」と指定された図であっても、出力はあくまで **指示文（`_画像生成指示.md`）** とする。これは「人間または別の描画手段が、その指示だけで図を再現できる」ことを目的とした設計図であり、実画像そのものは生成しない。なお実バイナリ生成は opt-in（既定は指示文のみ）で、手順は `references/RENDERING.md` を参照する。

## 3. 画像系提出物「画像生成AIへの指示文」の書き方ガイド

画像系提出物は、別の画像生成AI（または人間）がそれだけで図を再現できる **自然言語の設計図** として書く。以下の型を満たすこと。

### 3.1 冒頭メタ情報（必須）

```
対象提出物: q1_2.png（この指示を画像生成AIに渡して画像化する）
図の種類: 比較図 / 構造図 / 概念図 / フロー図 …
目的: この図が示したい主張を1〜2文で
```

### 3.2 本体に含める6要素

| 要素 | 書き方 |
|------|--------|
| **描画要素** | 図に登場する具体物（ノード・箱・アイコン・人物・矢印）を漏れなく列挙。各要素にラベル文言を明記 |
| **配置** | 画面のどこに何を置くか（上下/左右/中央、グリッド、対比なら左右2カラム など）を相対位置で指定 |
| **矢印・線** | 始点→終点、向き（単方向/双方向）、意味（因果・流れ・依存）、線種（実線/破線） |
| **ラベル** | 各要素・各矢印に付すテキストを「」で正確に。図タイトル・凡例も指定 |
| **スタイル** | 配色（役割で色分けする方針）、図形（角丸/直角）、トーン（手描き風/フラット）、フォント雰囲気 |
| **制約反映** | 問題が課す制約（例「丸・矢印・単語のみ」「画像生成AI不可＝手描き想定」）を指示文にも明記し、逸脱しない |

### 3.3 比較図の追加要点

- 比較軸（行）と比較対象（列）を表形式に落とせる構造にする。
- 「同型（同じ構造原理で動く事例）」と「反例（似て非なる事例）」を対比させる図では、**何が同じで何が違うか** が一目で分かる視覚的差異（色・位置・記号）を指示する。

### 3.4 指示文の良し悪し

- ✅ 良い: 「中央に半径の異なる同心円を3つ描き、外側から『入力増加』『臨界点』『飽和』とラベル。臨界点の円に向けて左から赤い実線矢印（『負荷』）、飽和の円から右へ灰色破線矢印（『性能低下』）。」
- ❌ 悪い: 「渋滞とDDoSの関係をいい感じの図にして」（要素・配置・ラベルが不明で再現不能）。

## 4. 対話の要点テンプレート

全小問必須。`<basename>_対話の要点.md` に保存する。

```markdown
# 小問(N) <小問タイトル> — 対話の要点

自分: <問題を自分の言葉で言い換え、何を明らかにしたいか提示> → AI: <概念・定義・枠組みを提示> → 自分: <仮説や具体例を当てて追及/反例で検証> → AI: <反証・補足・整理して論点を絞る> → 自分: <結論候補を要約し確認を要求> → AI: <結論を確定し提出物の主張に対応する形でまとめる>

（XX/1000字）
```

### 厳守事項

- 形式は `自分: ○○ → AI: ○○ → …` の連鎖。1行（または読みやすい範囲の改行）で往復を表現する。
- **最大1000文字**。超過したら要約して収める。末尾に `（XX/1000字）` を付記する。
- 「自分」＝提問・追及・検証・要約要求。「AI」＝回答・定義・例示・反証・整理。
- 実際の往復ログではなく **模範的な対話を再構成** する。
- 対話で導いた結論を、その小問の **提出物本体の主張と一致** させる。

---

## 5. markdown 回答本文（`.md` 提出物）の書き方

- 結論先行（PREP: Point → Reason → Example → Point）。
- 見出し（`##`）で論点を区切り、対比は表（`| … |`）、列挙は箇条書きで可読性を上げる。
- 専門用語は初出で1行定義を添える。状況設定で与えられた具体物に必ず接続する（抽象論で終わらせない）。
- 特定の試験ブランド名・書籍名・著者名・出版社名は **一切書かない**（汎用記述に置換）。
- opt-in で実PDF化する場合は `references/RENDERING.md` のグリフ脱落検証（`pdftotext`）を必ず通す。
