# AIシステム性能最適化チェックリスト（175+項目）

プロファイリング・デバッグ・チューニング時の体系的な確認リスト。
ボトルネック特定 → 適切なカテゴリへ → 項目を順次確認する。

## 1. パフォーマンス最適化マインドセット

- [ ] **最も高コストな処理を先に最適化する**: 80/20ルールでランタイムの上位ボトルネックを特定し集中する
- [ ] **最適化の前後でプロファイリングする**: 理論に頼らず実測値でスループット・レイテンシ・利用率を比較する
- [ ] **適応型オートチューニングを活用する**: リアルタイムフィードバック（強化学習・ベイズ最適化）でパラメータを動的調整する
- [ ] **最適化コストを予算化する**: 低ハンギングフルーツ（AMP有効化・データプリフェッチ）から着手し、ROIで優先順位を決める
- [ ] **フレームワークの最新版を維持する**: PyTorch/TensorFlow のアップグレードで融合カーネルや改良ヒューリスティックを無料で得る
- [ ] **ベンダー・コミュニティと協調設計する**: ドライバ更新・ベータ版テストに参加し新たな最適化機会を把握する
- [ ] **クラウドコストレバーを活用する**: スポットインスタンス・リザーブドインスタンスを使い分け、スループット/ドルを評価基準にする
- [ ] **利用率メトリクスを継続監視する**: GPU利用率・SM効率・メモリ帯域幅・ネットワーク利用率を常時監視する
- [ ] **スループット向上のためにハイパーパラメータを調整する**: バッチサイズ・シーケンス長をスループット重視でチューニングする
- [ ] **最適化内容を文書化して再利用する**: 最適化の適用内容と効果を内部 Wiki に記録し次プロジェクトへ継承する
- [ ] **複雑さとのバランスを保つ**: `torch.compile` で要件が満たせるなら、カスタム CUDA カーネルは不要

- [ ] **フォールトトレランスを設計する**: 分散チェックポイント・冗長ハードウェア・動的ジョブ再スケジュールで障害に備える

## 3. システムアーキテクチャとハードウェア計画

- [ ] **Goodput（有用スループット）を設計目標にする**: 生の FLOPS ではなくドル/ワット当たりの有用な作業量を最大化する
- [ ] **適切なアクセラレータを選択する**: FP8/FP4 対応・高速インターコネクト搭載の最新 GPU アーキテクチャを優先する
- [ ] **高帯域幅インターコネクトを活用する**: PCIe のみより NVLink/NVSwitch（例：GB200 NVL72）で GPU 間帯域幅を最大化する
- [ ] **CPU/GPU/メモリ比率を最適化する**: GPU あたり高速 CPU コア・DRAM・ストレージを適切に配分し GPU を飢餓から守る
- [ ] **データローカリティを確保する**: タイトに結合したワークロードは同じ NVLink/NVSwitch ドメイン内に配置する
- [ ] **チェーン内のボトルネックを排除する**: 最も遅いリンク（CPU/メモリ/ディスク/ネットワーク）を特定しスケールアップする
- [ ] **クラスタサイズを慎重に選択する**: GPU 追加時の逓減収益に注意し、N GPU の利用率 95% 達成後に 2N GPU へ拡張する
- [ ] **冷却と電力供給を設計する**: GB200/GB300 の高 TDP に対応した液冷と電源設備を確保し熱スロットリングを防ぐ

## 4. 統合 CPU-GPU スーパーチップアーキテクチャ

- [ ] **統合 CPU-GPU メモリを活用する**: Grace Blackwell の NVLink-C2C（900 GB/s）でメモリプールを共有し超大規模モデルを扱う
- [ ] **データローカリティを最適化する**: ホットな重み・アクティベーションは GPU HBM、コールドデータは CPU LPDDR に配置する
- [ ] **CPU-GPU 直接メモリアクセスを活用する**: NVLink-C2C 経由で PCIe 経由のコピーを省略し、プリフェッチで遅延を隠蔽する
- [ ] **Grace CPU を効率的に活用する**: 72 コア CPU でデータ前処理・拡張を高速実行し GPU への供給を途切れさせない
- [ ] **超大規模モデルに対応する**: `cudaMallocAsync` とマネージドメモリで GPU メモリオーバーフロー時に CPU メモリを拡張に使う
- [ ] **スーパーチップ最適化アルゴリズムを検討する**: SuperOffload 等の STV・異種最適化でオフロード効率を向上させる

## 5. マルチ GPU スケーリングとインターコネクト最適化

- [ ] **高速全対全トポロジーを設計する**: NVL72 の完全接続 GPU 群（NVLink 5: 1.8 TB/s 双方向）でスケールを実現する
- [ ] **トポロジーアウェアスケジューリングを使用する**: 同じ NVLink Switch ドメインにジョブを集約しクロスドメイン通信を避ける
- [ ] **前例のない帯域幅を活用する**: NVL72 の総帯域幅 ~130 TB/s を活かし、勾配データを数ミリ秒で全リデュースする
- [ ] **最新集合通信アルゴリズムを採用する**: PAT（Parallel Aggregated Tree）等 NVSwitch 最適化 NCCL アルゴリズムを使う
- [ ] **細粒度モデル並列化を検討する**: 1.8 TB/s の全対全接続でテンソル並列/レイヤー並列を積極的に活用する
- [ ] **リンク飽和を監視する**: DCGM の NVLink カウンターで通信パターンを検証しボトルネックがあれば調整する
- [ ] **将来の拡張を計画する**: 576 GPU（2段スイッチ）規模では、階層的集合通信でイントララック NVLink を優先使用する

## 6. OS とドライバの最適化

- [ ] **HPC チューニング済み Linux を使用する**: 不要なサービスを無効化し、CPU ガバナーを "performance" に設定する
- [ ] **スワップを無効化する**: 訓練サーバーでスワップを無効化し、クリティカルバッファは `mlock` / `cudaHostAlloc` でロックする
- [ ] **積極的なプリアロケーションでメモリ断片化を防ぐ**: 頻繁に使用するテンソル用に大きな連続メモリブロックを事前確保する
- [ ] **CPU ライブラリの環境変数を最適化する**: `OMP_NUM_THREADS`・`MKL_NUM_THREADS` でスレッド競合を削減する
- [ ] **NUMA アウェアネスを設計する**: `numactl` / `taskset` で GPU プロセスを最近傍 CPU に束縛しクロス NUMA トラフィックを最小化する
- [ ] **NIC と GPU タスクの IRQ アフィニティを設定する**: NIC 割り込みを同一 NUMA ノードの CPU コアに固定し性能の安定性を確保する
- [ ] **透明ヒュージページを有効化する**: THP を `always` か `madvise` モードで有効化し大規模メモリ割り当てを最適化する
- [ ] **最大ロックメモリを拡大する**: `ulimit -l unlimited` を設定しデータローダーがピンドバッファを確保できるようにする
- [ ] **最新 NVIDIA ドライバと CUDA スタックを維持する**: 全ノードで統一バージョンを使用し、永続化モード（`nvidia-smi -pm 1`）を有効にする
- [ ] **MIG 構成で永続化モードを有効にする**: MIG パーティションをジョブ間で維持しリセットオーバーヘッドを回避する
- [ ] **システムタスクを隔離する**: OS ハウスキーピング用に専用コアを確保し GPU フィードスレッドへの割り込みを排除する
- [ ] **システム I/O 設定を最適化する**: `noatime` マウント・適切なディスクスケジューラ（NVMe には `mq-deadline` か `noop`）を設定する
- [ ] **BIOS/ファームウェアを定期更新する**: PCIe 帯域幅改善・IOMMU 問題修正のファームウェアアップデートを適用する
- [ ] **Docker/Kubernetes を GPU 向けに調整する**: `--ipc=host`・`--ulimit memlock=-1` でコンテナの共有メモリとメモリロックを設定する

## 7. GPU リソース管理とスケジューリング

- [ ] **トポロジーアウェアジョブスケジューリングを実施する**: NUMA・NVLink 境界を考慮し、Kubernetes/SLURM でジョブを適切に配置する
- [ ] **MPS（Multi-Process Service）を活用する**: 複数プロセスが単一 GPU を共有する場合、時分割の代わりに並列実行で利用率を向上させる
- [ ] **MIG（Multi-Instance GPU）を適用する**: 軽量ワークロード多数の場合に GPU を最大 7 スライスに分割し ROI を最大化する
- [ ] **MIG の永続化モードを維持する**: クラスタ起動時に設定し、ジョブ間でパーティションを維持してスケジュール予測性を確保する
- [ ] **GPU クロックと電力設定を管理する**: `nvidia-smi -lgc/-lmc` でクロックを固定し実行間の一貫性を確保する
- [ ] **ECC メモリを有効にする**: データセンター GPU では ECC を有効にし長時間訓練でのメモリエラーによる破損を防ぐ
- [ ] **ジョブスケジューラに GPU トポロジーを統合する**: デバイスプラグイン・Gres を使い MIG スライスをスケジュール可能にする
- [ ] **CPU オーバーサブスクリプションを避ける**: GPU ジョブの CPU 要求（データローダースレッド等）を考慮してジョブをパックする
- [ ] **NVSwitch 環境で NVIDIA Fabric Manager を使用する**: NVSwitch トポロジー管理に Fabric Manager を常時起動する

## 8. I/O 最適化

- [ ] **データを並列ロードする**: `DataLoader(num_workers=N)` 等でワーカー数を増やし、GPU へのデータ供給がボトルネックにならないよう調整する
- [ ] **ホストメモリをピン留めする**: `pin_memory=True` で GPU が DMA 直接転送できるページロック済みメモリを確保する
- [ ] **計算とデータ転送をオーバーラップする**: CUDA ストリームと `cudaMemcpyAsync` / `non_blocking=True` でダブルバッファリングを実現する
- [ ] **高速ストレージ（NVMe/SSD）を使用する**: GPUDirect Storage（GDS）で GPU が NVMe から直接データをストリーム取得できるようにする
- [ ] **I/O 並列度とストライピングを調整する**: 単一ファイルアクセスを避け複数ストレージターゲットへ分散しアグリゲート帯域幅を最大化する
- [ ] **小ファイルアクセスを最適化する**: tar/RecordIO 等の大型コンテナに小ファイルをパックしメタデータオーバーヘッドを削減する
- [ ] **クライアントサイドキャッシングを活用する**: NFS キャッシュサイズ拡大や分散 FSキャッシュで冗長な I/O を削減する
- [ ] **データを賢く圧縮する**: I/O ボトルネック時は LZ4/Zstd 軽量圧縮を使い、CPU が新たなボトルネックにならないよう多スレッド展開する
- [ ] **スループットを測定しボトルネックを排除する**: GPU が ~100% 近くで動かない場合は MB/s 単位でデータ読み取り速度を測定する
- [ ] **マルチノード向けに I/O をスケールする**: 各ノードのローカルデータコピーやストレージシャーディングで集合 I/O スループットを確保する
- [ ] **チェックポイントとロギングのオーバーヘッドを最小化する**: 非同期書き込みで訓練を止めず、N イテレーションごとのログ集約で I/O を削減する

## 9. データ処理パイプライン

- [ ] **バイナリデータフォーマットを使用する**: TFRecords・LMDB・メモリマップド配列に変換し小ファイル群のオーバーヘッドを削減する
- [ ] **ファイルシステムを調整する**: `noatime` マウント・リードアヘッド拡大・複数ストレージノードへのシャーディングを実施する
- [ ] **CPU 集約ワークロードでハイパースレッディングを無効化する**: スレッド競合を減らし単一スレッド性能を重視するパイプラインに対応する
- [ ] **スレッド優先度を上げる**: `chrt` / `pthread_setschedparam` でデータローダースレッドの優先度を上げ供給遅延を防ぐ
- [ ] **頻繁にアクセスされるデータをキャッシュする**: OS ページキャッシュ・RAM ディスクで繰り返し読み取られるデータの冗長 I/O を排除する
- [ ] **データをプリフェッチしバッファリングする**: バックグラウンドスレッドと `prefetch_factor` で次バッチを事前ロードする
- [ ] **データ変換を並列化する**: 画像拡張・トークン化等の CPU 前処理が重い場合はワーカーを増やすか GPU 変換（DALI）を使う
- [ ] **モデル状態と出力をキャッシュする**: LLM 推論時は頻出トークンの KV キャッシュを再利用しリコンピュートを避ける
- [ ] **マルチノードでデータをシャーディングする**: 各ノードが固有のデータサブセットを読み取り、全ノードが同一ソースから読まない設計にする
- [ ] **パイプラインを監視しバッチサイズを調整する**: GPU 待機を減らすためバッチサイズと I/O スループットのバランスを調整する
- [ ] **GPU でデータ拡張を行う**: 単純な拡張操作（ノイズ付加・正規化）は CUDA ストリームで次バッチに対し非同期実行する

## 10. 性能プロファイリング・デバッグ・監視

- [ ] **ボトルネックを見つけルート原因を分析する**: Nsight Systems で CPU/GPU タイムラインを確認し compute-bound/memory-bound/IO-bound を特定する
- [ ] **Python オーバーヘッドを排除する**: 過剰ループやログの Python コードを特定しベクトル化操作・最適化ライブラリで置き換える
- [ ] **GPU 利用率とアイドルギャップを測定する**: 定期的な利用率低下を発見しチェックポイント保存等のイベントと相関させる
- [ ] **NVTX マーカーを使用する**: フェーズ（データロード・フォワード・バックワード）にラベルを付け Nsight Systems でアトリビュートする
- [ ] **Nsight Compute でカーネル分析する**: Occupancy・メモリスループット・命令スループットを確認しメモリバウンド/コンピュートバウンドを判定する
- [ ] **Warp ダイバージェンスを確認する**: 分岐効率メトリクスを確認し条件分岐を再構成してイントラワープ分岐を最小化する
- [ ] **ロードバランシングを検証する**: マルチ GPU ジョブで全ランクのタイムラインをプロファイルし最遅ランクがボトルネックになっていないか確認する
- [ ] **メモリ使用量を監視する**: `torch.cuda.memory_summary()` で OOM 直前の状態やリークを検出する
- [ ] **ネットワークとディスクを監視する**: OS ツールで実際のスループットを期待値と比較しボトルネックを特定する
- [ ] **異常のアラートを設定する**: ECC エラー・過熱・異常に遅いイテレーションを DCGM で自動検出する
- [ ] **回帰テストを実施する**: ドライバ・フレームワーク・コードの変更後に標準ベンチマークで性能比較し回帰を早期検出する

## 11. GPU プログラミングと CUDA チューニング

- [ ] **GPU メモリ階層を理解する**: レジスタ→共有メモリ/L1→L2→グローバル HBM の順で高速なメモリにデータを最大限再利用する
- [ ] **グローバルメモリアクセスをコアレス化する**: ワープ内スレッドが連続アドレスを参照するよう再構成しメモリトランザクションを最小化する
- [ ] **データ再利用に共有メモリを使用する**: 行列タイル等の頻繁にアクセスされるデータを共有メモリにロードしグローバルメモリトラフィックを削減する
- [ ] **メモリアラインメントを最適化する**: データ構造を 128 バイト境界に揃え `float2`/`float4` 型ベクトルロードを活用する
- [ ] **メモリ転送を最小化する**: 小さな転送を一つの大きな `cudaMemcpy` に統合し、Unified Memory では `cudaMemPrefetchAsync` を使う
- [ ] **過剰な一時アロケーションを避ける**: `cudaMalloc/cudaFree` の頻繁呼び出しを避け PyTorch のキャッシュアロケータや `cudaMallocAsync` を使う
- [ ] **スレッドとリソース使用のバランスを取る**: Occupancy 計算ツールでスレッド数を調整し、レジスタスピルを避けながら適切な Occupancy を維持する
- [ ] **レジスタと共有メモリ使用量を監視する**: Nsight Compute で Occupancy が 25% 以下の場合はスレッド数増加を検討する
- [ ] **メモリ転送と計算をオーバーラップする**: `cudaMemcpyAsync` + 複数 CUDA ストリームでデータ転送を計算と並行実行する
- [ ] **バルクプリフェッチを活用する**: `cp.async.bulk.prefetch` や TMA で L2/共有メモリへのプリロードを行い遅延を隠蔽する
- [ ] **Cooperative Groups を使用する**: 全ブロック同期の代わりにスレッドサブセットの局所同期でオーバーヘッドを削減する
- [ ] **Warp ダイバージェンスを最適化する**: データをソートし全 32 スレッドが同一分岐を取るようにし `__ballot`/`__shfl` で分岐レス解を実装する
- [ ] **Warp レベル操作を活用する**: `__shfl_sync` でワープ内リダクションを共有メモリなしに実行する
- [ ] **CUDA ストリームで並列処理する**: 独立したカーネルを別ストリームで起動しリソース使用率を高める
- [ ] **ライブラリ関数を優先する**: cuBLAS・cuDNN・Thrust・NCCL 等の高度最適化ライブラリを独自実装より優先する
- [ ] **繰り返し起動には CUDA Graphs を使用する**: 静的な訓練ループを CUDA Graphs でキャプチャし CPU 起動オーバーヘッドを大幅削減する
- [ ] **スケーラビリティの限界を確認する**: 入力サイズ拡大時の性能変化とロフトライン分析で Occupancy と帯域幅の限界を把握する
- [ ] **`torch.compile` を活用する**: TorchInductor で操作を融合した最適化カーネルに変換し 10-40% の高速化を得る
- [ ] **動的シェイプを適切に管理する**: `mark_dynamic()` で動的次元を指定し不要なリコンパイルを防ぐ
- [ ] **Triton カーネルを `torch.compile` 経由で統合する**: `torch.library.triton_op` でカスタム GPU カーネルを PyTorch に登録する
- [ ] **オートチューニングを活用する**: `cudnn.benchmark=True` で cuDNN が最適な畳み込みアルゴリズムを自動選択する
- [ ] **読み取り専用パスを活用する**: `const __restrict__` ポインタで GPU の L1 読み取り専用キャッシュを活用する

## 12. カーネルスケジューリングと実行最適化

- [ ] **GPU 同期呼び出しを最小化する**: `cudaDeviceSynchronize()` を必要最小限にし GPU が継続的に作業を処理できるようにする
- [ ] **小カーネルを融合して起動オーバーヘッドを償却する**: 連続する小カーネルを一つに統合し `torch.compile(mode="reduce-overhead")` を試す
- [ ] **GPU 動的並列化をイントラ GPU スケジューリングに使用する**: 子カーネルを GPU から直接起動し CPU 介入なしに不規則なタスクを処理する
- [ ] **繰り返しワークロードに永続カーネルを使用する**: 一度起動したカーネルがループ内で多数の作業単位を処理し起動コストを削減する
- [ ] **スレッドブロッククラスターを評価する**: Blackwell では最大 16 ブロックのクラスターを形成し局所性とリローンチオーバーヘッドを削減する

---

## 13. 算術最適化と低精度/混合精度

- [ ] **混合精度訓練を使用する**: AMP で FP16/BF16 を使い Tensor Core を活用し約 1.5-3.5× の高速化を得る
- [ ] **新しいハードウェアでは BF16 を優先する**: FP16 よりも指数範囲が広く損失スケーリング不要で安定した訓練を実現する
- [ ] **FP8/FP4 と新精度の活用を評価する**: Transformer Engine 経由で FP8 を使い訓練時に FP16/FP32 アキュムレータを維持する
- [ ] **Tensor Core と TMA を活用する**: FP16/BF16/TF32 の行列演算で CUTLASS テンプレートを使い峰値 FLOPS に近づける
- [ ] **TF32 で簡単な高速化を得る**: `torch.set_float32_matmul_precision("high")` で FP32 行列積に TF32 Tensor Core を使う
- [ ] **構造化スパース性を活用する**: 2:4 スパース性で GEMM スループットを ~2× にするためにモデルを剪定する
- [ ] **勾配とアクティベーションの精度を下げる**: FP16/BF16/FP8 で勾配通信し、アクティベーションチェックポイントも 16 ビットで保存する
- [ ] **推論にカスタム量子化を使用する**: TensorRT で INT8 量子化しキャリブレーション後に FP16 比 2-4× の高速化を得る
- [ ] **スケーリングと計算操作を融合する**: FP4 のマイクロスケーリング等、精度維持のために融合操作を分離パスではなく一括で実施する

---

## 14. 高度なチューニング戦略とアルゴリズムの工夫

- [ ] **カーネルパラメータをオートチューニングする**: ブロックサイズ・タイルサイズ・アンロール係数を GPU 世代ごとに探索し 20-30% の改善を狙う
- [ ] **ML ワークロードでカーネル融合を使用する**: 融合オプティマイザ・融合マルチヘッドアテンション・融合 LayerNorm + dropout を採用する
- [ ] **FlashAttention 等のメモリ効率的アテンションを使用する**: タイル型ストリーミングアテンションで中間行列を具現化せずメモリを削減する
- [ ] **通信と計算をオーバーラップする**: 各レイヤーの勾配が準備でき次第、非同期 NCCL で AllReduce を開始しバックワードと並行実行する
- [ ] **深いモデルにパイプライン並列化を使用する**: マイクロバッチで全ステージをビジーに保ち NVLink 経由でアクティベーションを高速転送する
- [ ] **分散オプティマイザシャーディングを活用する**: ZeRO で最適化状態・勾配を GPU 間でシャーディングし超大規模モデルをメモリ内に収める
- [ ] **可能な場合は非同期訓練を行う**: Stale SGD 等の非同期更新で全ワーカーの同期待ちなしにスループットを向上させる
- [ ] **スパース性と剪定を組み込む**: 訓練時剪定・蒸留・構造化剪定でモデルサイズを縮小し推論コストを削減する

## 15. 分散訓練とネットワーク最適化

- [ ] **可能な場合は RDMA ネットワークを使用する**: InfiniBand/RoCE で GPU からカーネルネットワークスタックをバイパスし遅延を削減する
- [ ] **Ethernet 使用時は TCP/IP スタックを調整する**: ネットワークバッファを拡大しジャンボフレーム（MTU 9000）を有効化する
- [ ] **NIC に CPU アフィニティを割り当てる**: NIC 割り込みを同一 NUMA ノードの CPU コアに固定し NUMA ペナルティを回避する
- [ ] **NCCL 環境変数を最適化する**: `NCCL_NTHREADS` を 8-16 に増やし、SHARP 対応環境では `NCCL_COLLNET_ENABLE=1` を設定する
- [ ] **低速ネットワークには勾配累積を使用する**: N ミニバッチに一度だけ同期する大規模 AllReduce でネットワークオーバーヘッドを削減する
- [ ] **AllReduce トポロジーを最適化する**: ノード内 NVLink → ノード間 InfiniBand の階層的 AllReduce で帯域幅を最大活用する
- [ ] **ネットワークオーバーサブスクリプションを避ける**: GPU 数に見合った NIC 帯域（200/400 Gbps InfiniBand 等）を確保する
- [ ] **通信を圧縮する**: 16 ビット/8 ビット勾配圧縮でネットワーク転送量を削減し帯域幅制約環境での訓練を可能にする
- [ ] **ネットワーク健全性を監視する**: パケットロスを検出し iPerf/NCCL テストで期待帯域幅とレイテンシを定期的に検証する

## 16. 効率的な推論とサービング

- [ ] **動的リソースを効率的にオーケストレーションする**: カスタムメトリクスで拡張する Kubernetes でライブ需要に応じてスケールする
- [ ] **推論にサーバーレスアーキテクチャを採用する**: バースト性トラフィック対応のマイクロサービス設計でアイドルリソースオーバーヘッドを削減する
- [ ] **バッチと並列処理を最適化する**: 動的/継続的バッチングで GPU を常にビジーに保ちながらレイテンシ SLO を維持する
- [ ] **分散推論に NIXL を活用する**: GPUDirect RDMA 経由でプリフィル GPU からデコード GPU へ KV キャッシュを CPU 介入なしに転送する
- [ ] **必要時は KV キャッシュをオフロードする**: Paged KV キャッシュ・量子化済み KV キャッシュ・階層的オフロードで GPU メモリ超過を回避する
- [ ] **最適化済み推論システムを使用する**: vLLM・SGLang・TensorRT-LLM・NVIDIA Dynamo で低レイテンシ・高スループット推論を実現する
- [ ] **テールレイテンシを監視しチューニングする**: P99 レイテンシを監視し CPU 介入・GC 停止・コンテキストスイッチによる外れ値を特定する
- [ ] **コールドスタートレイテンシを避けるためウォームアップする**: モデルをロードし dummy 推論を数回実行してから本番リクエストを受け付ける
- [ ] **QoS のためにリソースを適切に分割する**: 訓練と推論を別ノードに分離するか MIG で推論に保証済み GPU スライスを割り当てる
- [ ] **エッジ AI と低レイテンシデプロイに慎重にチューニングする**: エッジアクセラレータとデータ転送プロトコルを最適化し超低レイテンシを実現する

## 17. マルチノード推論とサービング

- [ ] **推論パイプラインを分解する**: プリフィル（プロンプト処理）とデコード（逐次生成）フェーズを別リソースに分離して独立スケールを実現する
- [ ] **LLM に継続的バッチ処理を使用する**: vLLM 等のトークンプーリングで常に高い GPU 占有率を維持しスループットを最大化する
- [ ] **GPU とノード間でモデルをシャーディングする**: テンソル並列/パイプライン並列で超大規模モデルを複数デバイスに分散し、高速インターコネクトで通信を補う
- [ ] **拡張コンテキストのためにメモリをオフロードする**: コールド KV キャッシュエントリを CPU RAM/NVMe に退避し GPU メモリ不足による OOM を防ぐ

## 18. 電力と熱管理

- [ ] **効率的で環境負荷の小さいエネルギーを使用する**: エネルギー消費量を性能と並行して追跡し動的電力キャッピングや再生可能エネルギー時間帯への移行を検討する
- [ ] **熱とクロックを監視する**: GPU 温度とクロック周波数を監視し熱スロットリングを検出したら冷却を改善する
- [ ] **エネルギーアウェア動的電力管理を使用する**: 適応的な電力キャッピングと動的クロックスケーリングでスループット/ワットを最適化する
- [ ] **性能/ワットを最適化する**: メモリバウンドなワークロードは若干低クロックでも速度低下が少なく電力を 20% 削減できる場合がある
- [ ] **適応的冷却戦略を使用する**: 冷却効率や電力価格に応じてジョブをスケジュールし非緊急ジョブを電力安価な時間帯に移す
- [ ] **ワークロードを統合する**: 多数の GPU を低利用率で動かすより少数の GPU を高利用率で動かす方がエネルギー効率が良い
- [ ] **冷却を効率的に設定する**: 重い実行時はファン速度を固定高速に設定し水冷の場合は流量と水温を管理する
- [ ] **電力を慎重に監視する**: `nvidia-smi` でフェーズ別電力プロファイルを把握し電力制約環境でスパイクの集中を避ける
- [ ] **長時間ジョブのレジリエンスを向上させる**: 90% 電力ターゲットで部品へのストレスを軽減し複数プロジェクトにわたるハードウェア寿命を延ばす
