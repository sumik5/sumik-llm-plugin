# Continuous Delivery 実践ガイド

## 1. Continuous Deliveryとは何か

Continuous Delivery（CD）とは、**ソフトウェアを常にリリース可能な状態に保ちながら開発し続ける**アプローチだ。

CDの核心的な目標：「アイデアから価値あるソフトウェアをユーザーの手に届けるまでの時間を、繰り返し確実に、短縮すること」

CDは単なるツールや自動化ではない。以下の要素が統合されて初めて機能する：

| 要素 | 説明 |
|------|------|
| **マインドセット** | 常にリリース可能な状態を維持する意識 |
| **自動化** | 繰り返し作業を機械に任せ、人間の判断に集中する |
| **計測** | データに基づいて改善を続ける |
| **チームワーク** | 個人ではなくチームとして品質に責任を持つ |

## 2. デプロイメントパイプライン

### 定義

デプロイメントパイプラインとは、**コミットからリリース可能な成果物へ**至るプロセスを、繰り返し確実に、高速かつ効率的に処理する「機械」だ。

新しいコードがパイプラインを通過し終えた時点で、追加作業は不要でそのまま本番リリースできる状態になっていなければならない。

### よくある誤解

| 誤解 | 正しい理解 |
|------|-----------|
| ビルド・テスト・デプロイの自動化ワークフロー | コミットから本番リリースまでの全プロセスを統合した「一つの機械」 |
| ビルド用・テスト用・デプロイ用の別々パイプライン | 単一の連続したパイプラインとして設計する |
| ツールとプロセスの寄せ集め | チームの行動規範と技術的な仕組みの統合 |
| コードが良いかどうかを証明するもの | 問題を発見し素早く修正するための「反証機構」 |

パイプラインは失敗から学ぶためのプラットフォームであり、証明のためのものではない。

### 最小構成の4ステージ

```
コミット → [Commit Stage] → [Artifact Repository] → [Acceptance Stage] → [Production]
```

| ステージ | 役割 | 目標時間 |
|---------|------|---------|
| **Commit Stage** | 高速な技術的テスト（Unit Tests中心）、開発者への即時フィードバック | < 5分 |
| **Artifact Repository** | Release Candidateの保存・バージョン管理、一意IDの付与 | - |
| **Acceptance Stage** | ユーザー視点の受け入れテスト、本番同等環境での検証 | - |
| **Production** | 手動または自動デプロイ（Continuous Deployment） | - |

## 3. パイプライン構築の原則と手順

### 原則: Walking Skeletonから始める

大規模なパイプラインを最初から構築しようとしてはいけない。まず**最小限の機能するシステム（Walking Skeleton）**に対してパイプラインを作り、そこから育てていく。

Walking Skeletonとは、システムのアーキテクチャ全体を通過する最もシンプルな実装のことだ。最初の数週間は大きな先行投資に感じられるが、一度稼働し始めるとチームの生産性が加速する。

### 構築ステップ

**Step 1: 準備**
- チームで最小の開始ケース（ユーザーストーリー）を合意する
- VCSを設定し、すべてを1つのリポジトリに置く
- CI/CDシステム・バージョン管理・パッケージング方法をチームで決定する

**Step 2: Commit Stageの構築**
- ビルド管理システム（BMS）を選び、VCSと連携させる
- すべてのコミットでUIテスト・ロジックテストが実行されるよう設定する
- テストが自動的に検出される命名規則・フォルダ構造を確立する

**Step 3: Artifact Repositoryの構築**
- デプロイ可能な成果物（Release Candidate）のパッケージング方式を決定する
- 各Release Candidateに一意のIDを付与する
- テスト通過後、Artifact Repoに自動保存する

**Step 4: Acceptance Stageの構築**
- Acceptance Testを作成し、パイプラインで自動実行する
- DBのクリーンセットアップスクリプトを作成してコミットする
- Acceptance Test環境のデプロイを自動化する
- Release Candidate IDをキーとして結果を保存する
- **テストが一つでも失敗したら、そのRelease Candidateは廃棄する**

**Step 5: Production Deploymentの構築**
- 全テストを通過したRelease Candidateをデプロイする仕組みを作る
- 手動リリース（Web UI経由）または自動デプロイを選択する

### 成長のサイクル

最初のパイプライン構築後、次のストーリーからは「テストとコード」だけを追加すればよい。パイプラインは要件に応じて徐々に拡充していく：

- セキュリティテスト
- パフォーマンステスト
- レジリエンステスト
- データマイグレーションテスト

## 4. 主要設計原則

### Automation（自動化）

自動化できることはすべて自動化する。目的：

- 繰り返し作業を確実かつ効率的に実行する
- 人間のエラーによるリスクを削減する
- 監査可能な記録を残す

手動テストはリグレッションには適さない。ただし、外部ユーザー視点で使いやすさを評価する探索的テストには価値がある。

### Efficiency（効率性）

1日に複数回「コミットからリリース可能な成果物へ」サイクルを回すことが目標：

- 各ステップは1回だけ実行する（重複排除）
- Unit Testsへの高速フィードバックにより、開発者は並行作業を継続できる
- **1時間以内のサイクル**を目指す → 同日中に問題を発見・修正できる

### Continuous Integration（継続的インテグレーション）

CIはCDパイプラインの最初のステージ。CIの本質は**「公開」プロセス**：

- チーム全員が常に最新の状態を共有する
- **「興味深いバージョンは現在のバージョンだけ」**という状態を維持する
- 1日最低1回のコミットを目指す

| ブランチ戦略 | CI観点での評価 |
|-------------|--------------|
| 長寿命ブランチ（数日〜数週間） | ❌ Integration Hellのリスク |
| 短命ブランチ（最大1日） | △ 許容範囲 |
| Trunk-Based Development | ✅ 推奨。高いスループットと安定性の予測因子 |

### Test Driven Development

TDDはパイプラインの「Commit Stage」に高速なテストを提供するための最善の方法：

- コードより先にテストを書くことで全体的な開発効率が向上する
- テストとコードの疎結合により、テストがコード変更に対して堅牢になる
- テスタビリティを考慮した設計は高品質なコードと同じ特性を持つ

### Version Control（バージョン管理）

本番にデプロイするすべてのビット・バイトが意図したものであることを保証する。
**バージョン管理の対象: コード、依存関係、設定、インフラストラクチャ、すべて。**

## 5. CI実践: チーム行動規範

### 個人の行動原則（10ヶ条）

| # | 行動 | 理由 |
|---|------|------|
| 1 | 小さなステップで作業する（機能全体ではなく小さな変更をコミット） | 複雑な問題を隠しにくく、テストと修正が簡単 |
| 2 | コミット前にローカルでCommit Testsを実行する | チームへの影響を事前に防ぐ |
| 3 | 10〜15分ごとにコミットし、5分以内のフィードバックを待つ | 自分のミスを最初に自分で見つけることが目標 |
| 4 | 全評価（Unit Tests + Acceptance Tests）の進捗を監視する | 変更の成功確認は自分の責任 |
| 5 | 失敗は10分以内に修正する。できなければ変更をrevertする | チームの作業を止めないため |
| 6 | チームメイトが結果を確認せず離席したら変更をrevertする | 本番への経路を維持するため |
| 7 | 失敗の原因が不明な場合、関係するコミッター全員で担当を決める | チームとしてリリース可能状態を維持する |
| 8 | Build SinsをゲーミフィケーションでCI行動を促進する | 文化的な定着のため |
| 9 | TDDを採用する | 高速で信頼性の高いCommit Stageテストのため |
| 10 | Deployment Pipelineを常に改善する | ソフトウェア開発の最も効率的な組織化方法 |

### パイプラインのチームへの価値

デプロイメントパイプラインはチームに**可視性と制御**を与え、コードの設計・品質・テスト・デプロイに対する責任を担えるようにする。これはツールではなく、チームのエンパワーメントの仕組みだ。

## 6. CD成熟度自己評価

### 13のCD指標

各プラクティスについて **0〜5点** で自己評価する（5: 完全実践、0: 未実施）：

| プラクティス | 評価基準（YesなのにNoならスコアが低い） |
|-------------|--------------------------------------|
| **Test Automation** | 手動リグレッションテストなしで安全に本番リリースできるか？ |
| **Deployment Automation** | テスト環境・本番環境へのデプロイをボタン一つで実行できるか？ |
| **Trunk-Based Development** | 少なくとも1日1回、trunk/master/headにマージしているか？ |
| **Shift Left on Security** | セキュリティテスト・検証がデプロイメントパイプラインに組み込まれているか？ |
| **Loosely Coupled Architecture** | システムの一部を変更した際、全体をテストせずに本番リリースできるか？ |
| **Empowered Teams** | チーム外の許可なしに技術・設計を変更できるか？ |
| **Continuous Integration** | 小さな変更を安全に1日複数回コミットできるか？ |
| **Version Control** | 履歴上の任意の時点から動作するバージョンをテスト環境にデプロイできるか？ |
| **Continuous Testing** | コミット時に全テストが自動実行され、リリース可能な成果物が生成されるか？ |
| **Test Data Management** | 自動テストが独立・自己完結しているか？ |
| **Test Consistency** | テストを並列・繰り返し実行しても常に一貫した結果が得られるか？ |
| **Monitoring and Observability** | ユーザーから苦情が来る前にシステムの問題を検知できるか？ |
| **Proactive Notifications** | 問題に気付く前にシステムが能動的に通知してくるか？ |
| **Database Change Management** | データ・データ構造の変更をCDプラクティスに含めているか？ |

### スコア解釈

| スコア帯 | 状態 | 推奨アクション |
|---------|------|--------------|
| 5 | 優秀 | 維持・他チームへの展開 |
| 3〜4 | 良好 | 継続的な改善 |
| 1〜2 | 要改善 | 重点的な投資 |
| 0 | 未実施 | 優先度高で取り組む |

## 7. CD改善フレームワーク

5点未満のプラクティスへの取り組み方（小さなステップで改善する）：

**1. Where Do You Think You Are Now?**
→ 現状の客観的な評価（データがあれば使う、なければ主観評価でも可）

**2. What Is Your Objective?**
→ 達成したいゴールの明確化

**3. What Is Your Next Step?**
→ 最初の小さな一歩の特定

**4. How Will You Know If You Have Succeeded?**
→ 成功の測定方法の定義

このサイクルを繰り返すことで、CDの成熟度を段階的に向上させていく。
