## 防御設計の基盤

### VPCネットワーク制御

#### サブネットゾーニング

ITリソースの重要度に応じてセグメントを設けるゾーニングがAWSでも有効。

| サブネット種別 | 用途 | 配置リソース例 |
|--------------|------|----------------|
| パブリックサブネット | インターネット直接通信 | ELB（ロードバランサー） |
| プライベートサブネット（アプリ層） | ELBからの通信のみ受付 | EC2（Webサーバー） |
| プライベートサブネット（データ層） | アプリ層からの通信のみ受付 | RDS、ElastiCache |

**推奨構成**: EC2はプライベートサブネットに配置し、ELBをパブリックサブネットに置くリバースプロキシ構成。EC2にグローバルIPを付与しないことでインターネットからの直接アクセスを隠蔽。

#### セキュリティグループ連携

**NACL vs セキュリティグループ**

| 機能 | NACL | セキュリティグループ |
|------|------|-------------------|
| デフォルト | すべて許可 | すべて拒否 |
| ステートフル性 | ステートレス（戻り通信を明示的に許可要） | ステートフル（戻り通信は自動許可） |
| 適用単位 | サブネット | インスタンス |
| 設定方式 | 拒否ルールを記述 | 許可ルールを記述 |

**SG連携パターン**（ソースにSGを指定）:
```
インターネット → [ELB用SG: 0.0.0.0/0:443] → [Web用SG: ELB-SG:8080] → [DB用SG: Web-SG:3306]
```

複数インスタンスで共用できるため、Auto Scalingによるスケールアウト時も自動的にルールが適用される。

### IAMロールと最小権限

#### アクセスレベルによる権限管理

IAMポリシー作成時にアクセスレベルでアクションを分類・制御する。

| アクセスレベル | 内容 | S3の例 |
|--------------|------|--------|
| List | リソース一覧の閲覧 | ListBucket, ListAllMyBuckets |
| Read | コンテンツの読み取り（編集不可） | GetObject, GetBucketLocation |
| Write | リソースの作成・削除・変更 | PutObject, DeleteBucket, CreateBucket |
| Tagging | タグの追加・変更 | PutBucketTagging |
| Permissions management | アクセス権の付与・変更 | PutBucketPolicy, DeleteBucketPolicy |

**最小権限の実装**: IAMポリシー作成画面のビジュアルエディタでアクセスレベルごとにアクションを選択し、必要最低限の権限のみ付与する。

#### IAMロールによる一時認証情報

**アクセスキー漏洩リスクを排除する推奨パターン**:

```
EC2インスタンス
  ↓ (IAMロールをアタッチ)
AWS STS (Security Token Service)
  ↓ (一時的な認証情報を発行)
AccessKeyId + SecretAccessKey + SessionToken
  ↓ (短期間で自動ローテート)
AWS APIへのアクセス
```

コードへのハードコード・環境変数・認証情報ファイルによるアクセスキー管理は非推奨。EC2にIAMロールをアタッチし、一時的な認証情報を使用することを推奨。

### KMS暗号化とACM証明書管理

#### KMSキーの種類と選択

| キー種別 | 管理主体 | ローテーション | キーポリシー変更 | 推奨ユースケース |
|---------|---------|--------------|----------------|----------------|
| カスタマーマネージドキー（CMK） | カスタマー | 年1回自動更新可 | 可能 | クロスアカウントアクセス制御、詳細な権限管理 |
| AWSマネージドキー | AWS | AWS管理 | 不可 | 運用負荷を削減したい場合、アカウント内のみ |
| AWS所有のキー | AWS（不可視） | AWS管理 | 不可 | - |

**エンベロープ暗号化の仕組み**:
1. データキー（DEK）でデータを暗号化
2. KMSキーでデータキー自体を暗号化
3. 暗号化されたデータキーをデータと同じ場所に保管
4. 復号時はKMSに暗号化DEKの復号をリクエスト

**メリット**: 鍵管理の簡易化（KMSキーのみ管理）、CloudTrailによる鍵アクセス監査の一元化。

#### ACM証明書管理

| 認証レベル | 検証内容 | ACM対応 | 用途 |
|----------|---------|---------|------|
| DV（Domain Validation） | ドメイン所有者確認のみ | ✅ | 一般的なHTTPS通信 |
| OV（Organization Validation） | 組織の法的実在性も確認 | ❌（外部CA） | 企業向けサイト |
| EV（Extended Validation） | 厳格な実在性・申請者確認 | ❌（外部CA） | 金融・EC等の高信頼性要求 |

ACM発行証明書は**有効期間395日で自動更新**。ELB・API Gateway・CloudFrontなどのAWS統合サービスに証明書を適用可能。

## ガードレール型セキュリティ設計

### ゲートキーパー型との比較

| 設計思想 | 概念 | 特徴 | 課題 |
|---------|------|------|------|
| **ゲートキーパー型** | 道路標識だらけの交差点 | 強固な予防的対策で通過を制限 | クラウド利用の自由度・利便性を損なう |
| **ガードレール型** | 道路のガードレール | 「検知」を併用し、クラウドの道を塞がない | 検知の仕組みが機能する前提が必要 |

**AWS Control Towerのガードレール**: 「予防」と「検知」の両面を含む。一部は強制的にAWSアカウントに適用される。

### 検知の位置付け

検知（発見的統制）は予防的対策（予防的統制）を補完する管理策。

**NIST CSFにおける検知の取り組み**:
- **異常とイベントの確認**: 異常なアクティビティを的確なタイミングで検知し、潜在的影響を把握
- **セキュリティの継続的モニタリング**: 情報システムとデータを別の間隔で監視してサイバーセキュリティイベントを洗い出し

### 発見的統制の業務分類

| 脅威種別 | 業務分類 | 観点 | 活用AWSサービス |
|---------|---------|------|----------------|
| 外部脅威 | 監視業務 | 異常アクティビティの有無（ニアリアルタイム） | GuardDuty, WAF |
| 外部脅威 | 分析業務 | 影響範囲分析（ログ遡及） | CloudWatch Logs Insights, Detective, Athena |
| 内部脅威 | 監視業務 | 異常アクティビティの有無（ニアリアルタイム） | GuardDuty, CloudWatch Logs フィルター, CloudTrail Insights |
| 内部脅威 | セキュリティ設定の不備確認 | ベストプラクティス準拠状況 | Trusted Advisor, Config, Security Hub |
| 内部脅威 | 分析業務 | 影響範囲分析（ログ遡及） | CloudWatch Logs Insights, Detective |
| 内部脅威 | 点検・監査業務 | 操作内容の正当性確認 | CloudWatch Logs Insights |

## 検知の仕組み

### 異常アクティビティ監視

#### CloudWatchによるパターンマッチング

CloudTrailログをCloudWatch Logsに転送し、**メトリクスフィルター**でパターンマッチングを実施。

**CIS AWS Foundations Benchmarkの主要モニタリング項目**:

| 検知対象 | フィルターパターン例 |
|---------|-------------------|
| 認証されていないAPIコール | - |
| MFAなしのコンソールアクセス | - |
| rootアカウントの使用 | - |
| IAMポリシーの変更 | - |
| CloudTrail設定の変更 | - |
| コンソールへの認証失敗 | - |
| CMKの無効化・削除スケジュール | - |
| S3バケットポリシーの変更 | - |
| Config設定の変更 | - |
| セキュリティグループの変更 | - |
| ネットワークACLの変更 | - |
| VPCの変更 | - |

**CloudWatch Logs Insights**: インタラクティブなクエリでアドホック調査。フィルターパターン開発にも活用。

SSHログイン試行確認クエリ例:
```
# ログイン失敗（Invalid user）
[Mon, day, timestamp, ip, id, msg1= Invalid, msg2 = user, ...]

# ログイン成功（要注意）
[Mon, day, timestamp, ip, id, msg1= Accepted, msg2 = password, ...]
```

#### AWS Config によるセキュリティ対策有効性検証

| 機能 | 内容 |
|------|------|
| Config Rules（マネージドルール） | AWSが提供するベストプラクティスルール（例: `s3-bucket-level-public-access-prohibited`） |
| Config Rules（カスタムルール） | 独自の検証ルールをLambdaで実装 |
| 適合パック（Conformance Pack） | 複数ルールと自動修復アクションをまとめたパッケージ |

### GuardDuty脅威インテリジェンス

Amazon GuardDutyは機械学習・脅威インテリジェンス（CrowdStrike・Proofpoint）を活用し継続的監視。

**データソース**:
- CloudTrailログ（API操作の異常検出）
- VPC フローログ（ネットワークアクティビティ）
- DNSログ（疑わしいドメインへの通信）
- S3データプレーンイベント
- EKSコントロールプレーンログ
- EBSボリュームのマルウェアスキャン

#### Findings重大度分類

| 重大度 | 表示色 | 意味 |
|--------|--------|------|
| 低（Low） | 青 | 疑わしいアクティビティだが、リソース侵害前にブロックされた |
| 中（Medium） | オレンジ | 通常の動作から逸脱したアクティビティを検出 |
| 高（High） | 赤 | 対象リソースが侵害され、不正な目的で活発に使用されている状態 |

#### 検知対象リソースとFindingsタイプ例

| 対象 | Finding例 | 内容 |
|------|----------|------|
| EC2 | `UnauthorizedAccess:EC2/SSHBruteForce` | SSHブルートフォース攻撃 |
| EC2 | `CryptoCurrency:EC2/BitcoinTool.B!DNS` | BitcoinツールのDNSクエリ |
| EC2 | `Backdoor:EC2/DenialOfService.Tcp` | EC2がTCP DoS実行 |
| S3 | `UnauthorizedAccess:S3/MaliciousIPCaller.Custom` | 悪意のあるIPからのS3操作 |
| S3 | `Discovery:S3/TorIPCaller` | TorIPからのS3探索 |
| IAM | `UnauthorizedAccess:IAMUser/MaliciousIPCaller.Custom` | 悪意のあるIPからのIAM操作 |

**FindingsはSecurity Hub・S3・EventBridgeへエクスポート可能**。重大度別のアラート通知も設定可。

### Detective詳細調査

GuardDuty FIndingsのイベント頻度・前後のアクセス状況を詳細調査するためのサービス。

**特徴**:
- VPC Flow Logs・CloudTrail・GuardDutyの複数データソースを統合分析
- 機械学習・統計的分析・グラフ理論でリンクされたデータセットを構築
- 時系列グラフでイベント傾向を可視化
- GuardDuty/Security Hubの操作画面からシームレスに遷移可能

### Security Hub統合管理

GuardDuty・Inspector・IAM Access Analyzer・Macieなどの検出結果を一元集約。

**主要機能**:

| 機能 | 内容 |
|------|------|
| 検出結果の正規化 | ASFF（AWS Security Finding Format）形式に変換・統一 |
| インサイト | 特定観点の事象を絞り込み表示（AWS提供＋カスタム） |
| Detectiveとの連携 | 検出結果から根本原因特定のためDetective画面へ遷移 |
| マルチアカウント管理 | 管理者アカウントでメンバーアカウントの検出結果を一元確認 |

## ログ保管・分析

### ログの転送と集約

**取得対象ログのドメイン分類**:

| ドメイン | 構成要素 | ログ例 |
|---------|---------|--------|
| アプリケーション | Webアプリケーション | アクセスログ、アプリケーションログ |
| インフラストラクチャ | OS | `/var/log/secure`、`/var/log/messages`、Windowsイベントログ |
| インフラストラクチャ | ネットワーク | VPCフローログ、VPCトラフィックミラーリング |
| インフラストラクチャ | ミドルウェア | DBアクセスログ、DBエラーログ |
| サービス | AWSサービス | CloudTrail（全API操作の4W1H） |

**CloudTrailの4W1H**:
- **Who**: API呼び出し元のID
- **When**: API呼び出し時刻
- **Where**: ソースIPアドレス
- **What**: 操作対象AWSリソース
- **How**: 呼び出されたAPI名

**EC2のCloudWatchエージェント設定例**（OS→CloudWatch転送）:
```bash
# CloudWatchエージェントインストール後、/var/log/secureなどを自動転送
```

#### マルチアカウントにおけるログ集約

**ベストプラクティス**: ログ集約専用アカウントのS3バケットに各アカウントのCloudTrailログを集約。

メリット:
- 個別アカウント内での改竄を防止
- 分析・監視の一元化
- マルチアカウント環境でのコンプライアンス対応

### ログの保護（WORM保護）

**S3ログ保護の3要素**:

| 保護手段 | 機能 | 実装方法 |
|---------|------|---------|
| ログの転送と集約 | 単一箇所への集中管理で改竄リスク分散 | CloudTrailログを専用アカウントS3へ転送 |
| アクセス制御 | 最小限の人員のみアクセス可能 | S3バケットポリシー＋IAMポリシー |
| 整合性検証 | 改竄の有無を確認 | CloudTrailの「ログファイルの整合性の検証」機能 |
| WORM保護 | 削除・上書き不可 | S3オブジェクトロック（コンプライアンスモード） |

**S3オブジェクトロック コンプライアンスモード**: rootユーザーを含むいかなるユーザーも保護オブジェクトを削除・上書き不可。

### ログ分析アプローチ

| アプローチ | 手段 | 用途 |
|----------|------|------|
| シグネチャ型 | WAF・IDS/IPS | 既知の攻撃パターンをシグネチャとマッチング。ゼロデイ攻撃には限界 |
| アノマリ型 | GuardDuty（機械学習） | 通常パターンとの乖離を検知。誤検知チューニングが必要 |
| パターンマッチング | CloudWatch Logsメトリクスフィルター | ルールベースのアラート。攻撃ツリーから手動でルール作成 |
| アドホック分析 | CloudWatch Logs Insights, Athena | 過去ログを対象に遡及調査 |

## インシデントレスポンス準備

### AWS CAFが定める5つの準備要件

1. **インシデントレスポンスのプロセスを整備しておく**
2. **調査や分析のためにデータを保存しておく**
3. **フォレンジックのための基盤を用意しておく**
4. **インシデントレスポンスの対応内容を自動化しておく**
5. **シミュレーションを行ってインシデントレスポンスの訓練をしておく**

### インシデントシナリオの定義

#### インシデントドメイン分類（AWS推奨）

| ドメイン | 説明 | シナリオ例 |
|---------|------|-----------|
| アプリケーション | アプリコード・ソフトウェアデプロイに起因 | プログラム不具合による情報露出、ソースコードの誤公開 |
| インフラストラクチャ | EC2・OS層への通信・プロセス・データの侵害 | 不要ポート公開による侵入、パッチ未適用によるOS乗っ取り、S3誤公開 |
| サービス | AWSアカウント・IAM権限・メタデータの侵害 | IAM権限不備による不正アクセス、認証情報漏洩、不正な請求 |

#### セキュリティイベント指標（シナリオ洗い出しの観点）

| 指標 | 具体例 |
|------|--------|
| ログ監視によるイベント | CloudTrail監視アラート、GuardDuty通知 |
| 脅威インテリジェンス | IoC（侵害指標）に基づくGuardDutyアラート |
| セキュリティベンダー製品 | ネットワークIPS・DLPからのアラート |
| AWSの請求アクティビティ | 利用料金の急激な変化 |
| AWSからの不正使用連絡 | Abuse Report（AWSサポートからのメール） |
| 社内・社外からの連絡 | メール・口頭による事故報告 |

### 対応方法の定義（4フェーズ）

| フェーズ | 内容 |
|---------|------|
| 原因調査 | アラート発生理由の特定と正当性評価（誤検知確認を含む） |
| 影響調査 | システム・ビジネスへの影響範囲確認 |
| 対応判断 | 影響度合いに基づく対応策の決定（復旧タイミングも含む） |
| 復旧作業 | 復旧手順の実施（復旧完了基準を事前定義） |

#### GuardDuty検知タイプ別対応方針

| 検知カテゴリ | 対応方針 |
|------------|---------|
| **悪意のあるアクティビティ**（Backdoor, Cryptocurrency） | 侵害を前提に調査開始。ファイル・プロセス確認→隔離→フォレンジック→再構築検討 |
| **ベースライン逸脱アクティビティ**（Behavior, UnauthorizedAccess） | 正規操作かを確認。操作者・システム特性を確認し侵害有無を判定 |

### Runbook整備

**Runbook記載項目**:

| 項目 | 内容 |
|------|------|
| インシデントの概要 | 攻撃シナリオの端的な表現 |
| 攻撃の詳細 | 攻撃の具体的情報・対応トリガーとなる情報 |
| 重大度 | 基本重大度と状況に応じた加算条件 |
| 連絡手順 | 連絡先・エスカレーションタイミング |
| 対応手順 | 原因調査・影響調査・対応判断・復旧作業の手順概要 |

**Runbook例（DDoS攻撃対応）**:
```
インシデントの概要: Webシステムへの DDoS 攻撃
攻撃の詳細: AWS Shield Adv. からの DDoS 通知メール
重大度: 中（繁忙期+1、4時間以上継続+1）
連絡手順:
  即時: 運用/企画/開発チームML、セキュリティ管理チームML
  中+1以上: CTO、CISO（復旧前）
対応手順:
  調査: CloudFront/ALBのメトリクス確認（同時接続数・リターンコード）
  対応判断: Shield Adv. DRTチームの提案確認と誤検知リスク評価
```

**担当者・アクセス権の定義**: IAM管理ポリシー `SecurityAudit` をインシデント調査担当者に事前付与。ログ・イベントへのアクセス権を整備しておく。

## フォレンジック基盤

### フォレンジック基盤の3要件

| 要件 | 内容 | クラウドでの対応 |
|------|------|----------------|
| 可用性 | 24時間いつでも調査可能 | マネージドサービスで常時稼働 |
| 処理能力 | 大量ログの高速検索・関連分析 | オートスケールで処理能力を動的確保 |
| 費用 | 長期ログ保存とスケール対応 | 利用量課金で予測不能な拡張にも対応 |

### AWSサービスを活用したフォレンジック

#### Amazon Detective

| 機能 | 詳細 |
|------|------|
| データソース | VPC Flow Logs、CloudTrail、GuardDuty |
| 分析手法 | 機械学習・統計的分析・グラフ理論 |

## 自動修復パターン

### EventBridge→SNS→Lambda連携

```
GuardDuty / Config Rules
    ↓ (Findings発生)
Amazon EventBridge
    ↓ (イベントパターンでフィルタリング)
Amazon SNS
    ├─→ メール通知（管理者・SOC）
    ├─→ AWS Chatbot → Slack通知
    └─→ AWS Lambda
           ↓ (自動修復処理)
        ・セキュリティグループ変更
        ・S3バケットポリシー修正
        ・IAMロールのセッション無効化
```

**EventBridgeルール設定例（Config変更検知）**:
```json
{
  "source": ["aws.config"],
  "detail-type": ["Config Rules Compliance Change"]
}
```

### マルウェア感染時の自動対応フロー（EC2）

GuardDutyがマルウェア感染を検知したときのLambdaによる自動化手順:

```
1. EC2インスタンスからメタデータをキャプチャ
2. インスタンスの終了保護を有効化（偶発的な終了防止）
3. VPCセキュリティグループを切り替えてネットワーク隔離
4. Auto Scaling グループ・ELBからインスタンスを除外
5. EBSデータボリュームのスナップショットを作成
6. 調査用インスタンスを起動しスナップショットからストレージ復元
```

**自動化のメリット**:
- 工数削減（24時間対応が不要に）
- 迅速な対応による被害低減
- 人的ミス（アラートへの鈍感化・見逃し）の防止
- セキュリティ担当者を判断が必要なインシデントに集中させる

## 実践シナリオ：疑似攻撃環境構築

### 環境構築（モジュール1：検知対応環境）

1. **GuardDuty有効化**: CloudTrailログ・VPCフローログ・DNSクエリログの監視開始
2. **Security Hub有効化**: GuardDuty等の検出結果を集約
3. **CloudFormation実行**（モジュール1テンプレート）: EventBridge・SNS・Lambda等の自動対応基盤を構築
4. **メール通知設定**: GuardDuty Findingsのメール通知
5. **Config Rulesの設定**: `s3-bucket-level-public-access-prohibited`を有効化
6. **EventBridgeルール作成**: Config Rules変更を検知しSNS→メール通知

### 疑似攻撃の実行（モジュール2）

CloudFormationテンプレート2により以下を自動構築:
- 攻撃対象のWebサーバーEC2インスタンス（パスワード認証SSHが有効、IAMロールあり）
- 攻撃者役のEC2インスタンス（同一VPC内の別サブネット）
- GuardDutyカスタム脅威リストに攻撃者のIPを登録

**攻撃の流れ**:
1. 攻撃者インスタンスがWebサーバーにSSHブルートフォース攻撃
2. SSHログイン成功（パスワード認証 + 弱いパスワードにより）
3. 攻撃者がIAMロールの一時認証情報を取得
4. S3バケットのパブリックアクセス設定を変更
5. 取得した認証情報でその他APIコールを実行

### S3バケット侵害の調査手順

**トリガー**: Configのコンプライアンス変化通知メール（`s3-bucket-level-public-access-prohibited`）

```
1. Config コンソール → ダッシュボード
2. 非準拠ルール「s3-bucket-level-public-access-prohibited」を確認
3. 非準拠リソースのS3バケット一覧を確認
4. 影響バケットを特定（データバケットへのパブリックアクセスが許可されている）
```

**対応（封じ込め）**:
```
1. 対象S3バケットを選択 → 「アクセス許可」タブ
2. 「ブロックパブリックアクセス」→「編集」
3. 「パブリックアクセスをすべてブロック」を有効化
4. 変更を保存し「非公開のバケットとオブジェクト」表示を確認
```

### EC2インスタンス侵害の調査手順

**トリガー**: GuardDuty Finding `UnauthorizedAccess:EC2/SSHBruteForce`

```
調査ステップ1: SSH認証方式の確認
  Security Hub → 検出結果 → フィルタ「製品名: Inspector」
  「Instance is configured to support password authentication over SSH」を確認
  → パスワード認証が有効でブルートフォース攻撃に脆弱な状態を確認

調査ステップ2: 攻撃成功の有無確認
  CloudWatch → ロググループ → /threat-detection-wksp/var/log/secure
  フィルタパターン: [Mon, day, timestamp, ip, id, msg1= Accepted, msg2 = password, ...]
  → aliceユーザーで攻撃元IPからのログイン成功を確認
```

**対応（SSH接続遮断）**:
```
1. EC2コンソール → 侵害インスタンスを選択
2. 「セキュリティ」タブ → セキュリティグループを選択
3. 「インバウンドルール」→「インバウンドのルールを編集」
4. SSHルール（ポート22）を削除 → 「ルールを保存」
```

### IAM認証情報侵害の調査手順

**トリガー**: GuardDuty Finding `UnauthorizedAccess:IAMUser/MaliciousIPCaller.Custom`

```
調査ステップ1: アクセスキーの特定
  メールの本文からAccess Key IDをコピー（一時的な認証情報/Assumed Role）

調査ステップ2: GuardDutyで詳細確認
  GuardDuty → 「フィルタの追加」→「アクセスキーID」で検索
  検出結果「UnauthorizedAccess:IAMUser/MaliciousIPCaller.Custom」を確認
  「影響を受けるリソース」→ instanceID・IAM instance profile（ARN）を記録

調査ステップ3: 侵害IAMロールの特定
  ARNからIAMロール名を特定（例: threat-detection-wksp-compromised-ec2）
```

**対応（セッション無効化と認証情報ローテーション）**:
```
1. IAMコンソール → 「ロール」→ 侵害ロールを選択
2. 「セッションを取り消す」タブ → 「アクティブなセッションの無効化」
   ※ 確認チェックボックスにチェックを入れてから実行
3. EC2インスタンスを停止→再起動
   ※ 一時認証情報をリフレッシュし、攻撃者の認証情報を無効化
4. 認証情報のローテーション確認（Systems Manager → セッションマネージャー）
   curl http://169.254.169.254/latest/meta-data/iam/security-credentials/<role-name>
   → AccessKeyIdが変更されていることを確認
```

### 調査全体フロー（統合シナリオまとめ）

```
攻撃発生
  ↓
GuardDuty: SSHBruteForce検知
Config Rules: S3パブリックアクセス変更検知
  ↓ (EventBridge→SNS→メール)
セキュリティ担当者へアラート通知
  ↓
S3調査・対応（Config→S3コンソール）
  → パブリックアクセスブロックを修復
  ↓
EC2調査（Security Hub Inspector結果→CloudWatch Logs）
  → SSHログイン成功を確認
  → セキュリティグループからSSH削除
  ↓
IAM調査（GuardDuty→MaliciousIPCaller）
  → 侵害IAMロールを特定
  → セッション無効化・インスタンス再起動で認証情報ローテーション
  ↓
Detective で関連イベントを詳細調査
  ↓
Runbookに従って根絶・復旧・再発防止
```
