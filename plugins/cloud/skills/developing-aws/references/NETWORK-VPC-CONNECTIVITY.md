# VPC ネットワーク接続 実践ガイド

VPC の基礎概念、IPアドレス管理、インターネット接続、セキュリティ制御、DNS/負荷分散、および外部ネットワークとの接続パターンを実践的手順で解説するガイド。

> **関連**: VPC設計ガイドは [VPC-ARCHITECTURE.md](./VPC-ARCHITECTURE.md)、ALB/NLB/CloudFrontは [NETWORKING.md](./NETWORKING.md) を参照

---

## 1. VPC基礎とオンプレミス対比

### 1.1 オンプレミス vs AWS ネットワーク対応表

| オンプレミスコンポーネント | AWS相当 | 役割 |
|--------------------------|---------|------|
| データセンター（建物） | リージョン | 地理的に分離された運用拠点 |
| データセンター内フロア | アベイラビリティゾーン（AZ） | 電気・物理・ネットワーク的に隔離 |
| プライベートネットワーク | VPC サブネット | RFC 1918 プライベートIPで閉じたネットワーク |
| スイッチ/ルーター | VPC内蔵ルーター | サブネット間を暗黙的に接続 |
| パケットフィルタリングFW（インスタンス単位） | セキュリティグループ | ENI/EC2インスタンス単位のFW（ステートフル） |
| パケットフィルタリングFW（セグメント単位） | ネットワークACL | サブネット単位のFW（ステートレス） |
| インターネット引込線 | Internet Gateway (IGW) | VPC⇔インターネット接続点 |
| NATルーター | NAT Gateway | プライベートIPからインターネットへのOutbound |
| NIC（ネットワークインターフェースカード） | ENI（Elastic Network Interface） | EC2インスタンスの仮想NIC |
| 固定IPアドレス | Elastic IP | 静的パブリックIPアドレス |

### 1.2 リージョン・AZ概念

```
AWSグローバルインフラストラクチャ
└── リージョン（例: ap-northeast-1 東京）
    ├── AZ-a（ap-northeast-1a）: データセンター群A
    ├── AZ-c（ap-northeast-1c）: データセンター群C
    └── AZ-d（ap-northeast-1d）: データセンター群D

VPC（リージョンをまたぐ）
└── サブネット（各AZに属する・AZをまたがない）
    └── EC2インスタンス（特定AZに固定）
```

**設計上の重要ポイント**:
- AZ間通信は同一AZ内より若干遅延・費用発生（通常は無視できる程度）
- 冗長性確保には複数AZへの配置が必須
- 大阪リージョン（ap-northeast-3）は東京より利用可能サービスが少ない

### 1.3 非VPCサービス vs VPCサービス

| 区分 | 代表サービス | IPアドレス | インターネットアクセス |
|------|------------|-----------|----------------------|
| **非VPCサービス** | S3, CloudFront, Route 53, IAM | パブリックIP | デフォルトで可能 |
| **VPCサービス** | EC2, RDS, ELB, Lambda（VPC内） | プライベートIP | ゲートウェイが必要 |

**重要**: VPCサービスは契約者ごとに閉じたネットワーク。インターネットや他サービスと接続するには IGW・VPC Endpoint・VGW などのゲートウェイを設置する。

## 2. VPC作成とCIDR設計

### 2.1 デフォルトVPC vs カスタムVPC

| 特性 | デフォルトVPC | カスタムVPC |
|------|------------|------------|
| CIDR | `172.31.0.0/16`（固定） | 任意（推奨: `/16`） |
| サブネット | AZごとに`/20`が自動作成 | 手動で設計 |
| IGW | 自動でアタッチ済み | 手動でアタッチ |
| パブリックIP自動割当 | 有効 | デフォルト無効 |
| 用途 | 開発・PoC・学習 | 本番・セキュリティ要件あり |

### 2.2 VPC・サブネット作成手順

**VPC作成の制約**:
- リージョンあたりデフォルト最大5つ（申請で増加可）
- CIDRは作成後に変更不可（追加は最大5ブロックまで可）
- 最大サイズ: `/16`（65,536 IP）

**サブネット予約IPアドレス（各サブネットで5つ）**:

| IPアドレス | 用途 |
|-----------|------|
| 先頭（x.x.x.0） | ネットワークアドレス |
| 先頭+1（x.x.x.1） | VPCルーター |
| 先頭+2（x.x.x.2） | AmazonDNS |
| 先頭+3（x.x.x.3） | 将来の予約 |
| 末尾（x.x.x.255） | ブロードキャスト（VPCでは機能しない） |

例: `/24`サブネット（256 IP）は実際251台まで利用可能。

**CIDR設計のポイント**:
- VPCは`/16`が実用上の最大（将来拡張を考慮）
- オンプレミス接続予定がある場合は **IPアドレス重複禁止**（VPCピアリング・VPN接続が不可になる）
- RFC 1918 プライベートアドレス（`10.x.x.x`、`172.16.x.x`〜`172.31.x.x`、`192.168.x.x`）を使用

## 3. EC2とIPアドレス管理

### 3.1 ENI（Elastic Network Interface）の仕組み

```
EC2インスタンス
└── ENI（Elastic Network Interface）= 仮想NIC
    ├── プライマリプライベートIP（変更不可・DHCPで永続割当）
    ├── セカンダリプライベートIP（任意追加・変更可能）
    ├── パブリックIP（オプション、再起動で変更）
    ├── Elastic IP（オプション、固定・手動関連付け）
    └── セキュリティグループ（複数適用可、最大5つ/ENI）
```

**ENIの特徴**:
- 1つのENIに複数のセキュリティグループを適用可能（最大5つ）
- 1つのセキュリティグループを複数のENIで共有可能（変更すると全ENIに反映）
- EC2インスタンスには複数のENIをアタッチ可能（複数サブネット接続）
- EC2インスタンス作成後に別のENIに付け替えることで、プライマリIPを「移動」できる

### 3.2 プライベートIPアドレスの割当ルール

| IPタイプ | 割当方法 | 変更可否 | 特徴 |
|---------|---------|---------|------|
| **プライマリプライベートIP** | DHCPで自動 or 作成時に手動指定 | **変更不可**（ENI作成時に決定） | 再起動後も同一IPを維持 |
| **セカンダリプライベートIP** | 手動で追加 | 変更・削除可能 | 複数のIP割当に使用 |
| **パブリックIP** | サブネット設定で自動割当 | 停止/起動で変更される | 動的・無料 |
| **Elastic IP** | 明示的に割当・関連付け | 手動で付け替え可能 | 静的・未割当時に課金 |

**重要**: AWS では OS の IP設定を DHCP にすること（固定IPをOS側に直書きは禁止）。固定させたい場合はAWS側でENIに特定IPを予約する。

### 3.3 パブリックIPとNAT変換

EC2インスタンス内では `ip addr` コマンドではパブリックIPが見えない。IGWレベルでNAT変換されているため、インスタンス自体はプライベートIPのみ保持する。

```
インターネット → IGW(NAT変換) → EC2インスタンス
                  パブリックIP ↔ プライベートIP
```

パブリックIPの確認方法（メタデータサービス経由）:
```bash
curl http://169.254.169.254/latest/meta-data/public-ipv4
```

### 3.4 Elastic IP管理

| 操作 | 説明 |
|------|------|
| **割当（確保）** | AWSアドレスプールから静的IPを確保。リージョンあたり最大5つ |
| **関連付け** | 確保したEIPをENIに紐付け。即座に有効化 |
| **付け替え** | 障害発生時に新しいEC2インスタンスに関連付けを変更 |
| **解放** | 確保を解除。別アカウントが取得する可能性あり |

**コスト注意**: 稼働中EC2インスタンス1台につき1つのEIPは無料。関連付けていないEIPや停止中インスタンスのEIPは時間課金される。

## 4. インターネット接続

### 4.1 IGW接続とルートテーブル構成

**Public Subnetの設定手順**:

```
1. VPCにIGW（Internet Gateway）を作成・アタッチ
2. 専用のルートテーブルを作成
3. ルートテーブルにデフォルトルート追加:
   宛先: 0.0.0.0/0  →  ターゲット: igw-xxxxxxxx
4. サブネットに上記ルートテーブルを関連付け
5. EC2インスタンス起動時にパブリックIP自動割当を有効化
```

**Public Subnetルートテーブル例**:

| 宛先 | ターゲット | 備考 |
|------|----------|------|
| `10.0.0.0/16` | local | VPC内通信（自動設定） |
| `0.0.0.0/0` | igw-xxxxxxxx | インターネット向け |

### 4.2 NAT Gatewayによるプライベートサブネット接続

**配置パターン**:

```
インターネット
      ↓
  IGW（Internet Gateway）
      ↓
┌─────────────────────────────┐
│  Public Subnet              │
│  ・NAT Gateway（EIPあり）   │
│  ・Webサーバー等            │
└─────────────────────────────┘
      ↑（プライベートサブネットのデフォルトGW）
┌─────────────────────────────┐
│  Private Subnet             │
│  ・DBサーバー等             │
│  Route: 0.0.0.0/0 → nat-gw │
└─────────────────────────────┘
```

**Private Subnetルートテーブル例**:

| 宛先 | ターゲット | 備考 |
|------|----------|------|
| `10.0.0.0/16` | local | VPC内通信 |
| `0.0.0.0/0` | nat-xxxxxxxx | NAT Gateway経由でインターネットへ |

**NAT Gatewayの特徴**:
- パブリックサブネットに配置する（EIPが割当される）
- Outbound（プライベート→インターネット）のみ。Inboundは不可
- NAT Gatewayを通る通信の送信元IPは割当EIPになる（IP制限用途に利用可）
- 費用: 時間課金 + データ処理量課金（不要時は削除推奨）
- S3/DynamoDBへのアクセスにはVPCエンドポイントを使うとNAT費用を回避可能

## 5. セキュリティグループ vs ネットワークACL

### 5.1 機能比較

| 特徴 | セキュリティグループ（SG） | ネットワークACL（NACL） |
|------|--------------------------|------------------------|
| **適用対象** | ENI（EC2インスタンス）単位 | サブネット単位 |
| **ステートフル/ステートレス** | **ステートフル**（応答パケット自動許可） | **ステートレス**（戻りパケットも明示的許可必要） |
| **ルール種別** | 許可ルールのみ | 許可 + 拒否ルール |
| **評価順序** | 全ルールを評価 | 番号順に評価（最初マッチで確定） |
| **デフォルト動作** | インバウンド全拒否 / アウトバウンド全許可 | インバウンド全許可 / アウトバウンド全許可 |
| **最大ルール数** | インバウンド60 + アウトバウンド60 | インバウンド20 + アウトバウンド20 |
| **1対多の関係** | 1つのSGを複数ENIに適用可、1つのENIに複数SG適用可 | 1つのサブネットに1つのNACL |

### 5.2 ステートフル vs ステートレスの実践的影響

**セキュリティグループ（ステートフル）**:

SSH接続許可の場合:
```
インバウンドルール: TCP 22（SSH）← 設定だけでOK
アウトバウンドルール: 設定不要（戻りパケットは自動許可）
```

**ネットワークACL（ステートレス）**:

SSH接続許可の場合:
```
インバウンドルール:  TCP 22 許可（SSH要求）
アウトバウンドルール: TCP 1024-65535 許可（エフェメラルポートの応答）← 必須！
```

### 5.3 セキュリティグループ設計パターン

#### レイヤー別チェーン（推奨パターン）

```
sg-alb:     Inbound: 0.0.0.0/0:443    (インターネットからHTTPS)
sg-app:     Inbound: sg-alb:8080      (ALBからのみ許可 ← セキュリティグループ参照)
sg-db:      Inbound: sg-app:3306      (Appからのみ許可)
sg-bastion: Inbound: <社内IP>/32:22   (特定IPからのみSSH)
```

**セキュリティグループIDでの参照のメリット**: IPアドレスが変わっても（EC2再作成時など）ルールの変更不要。同じSGを持つインスタンスが増えれば自動的に対象に含まれる。

#### NACLの適切な使い方

- **NG**: ポート単位の細かいフィルタリング（ステートレスのため複雑になる）
- **OK**: 特定IPアドレス範囲の拒否（例: 社内LANからのみ許可、DDoS源IPのブロック）
- ルール番号は100刻みで設定（将来の挿入に対応）

### 5.4 ウェブ/コンテナ/DB層の設定シナリオ

| 層 | セキュリティグループ設定例 |
|----|--------------------------|
| **Web（ALB）** | Inbound: 0.0.0.0/0:80,443 |
| **App（ECS/EC2）** | Inbound: sg-alb:80 ← ALBからのみ |
| **DB（RDS）** | Inbound: sg-app:5432 ← Appからのみ |
| **管理（Bastion）** | Inbound: <会社IP>/32:22 |

## 6. DNS・負荷分散

### 6.1 Route 53 基本概念

| 概念 | 説明 |
|------|------|
| **ホストゾーン** | ドメインの設定範囲。`example.co.jp`に対する全サブドメインを管理 |
| **NSレコード** | そのゾーンを担当するDNSサーバーを指し示す |
| **SOAレコード** | 更新情報・連絡先・TTLなどゾーンのメタ情報 |
| **Aレコード** | ホスト名 → IPv4アドレスの対応 |
| **CNAMEレコード** | ホスト名 → 別ホスト名（Alias）の対応 |
| **エイリアスレコード** | AWS特有。ALBなどのDNS名を直接指定（IPが変動するリソース向け） |
| **TTL** | DNSキャッシュ保持時間（秒）。小さいほど変更が素早く反映 |

### 6.2 ルーティングポリシー

| ポリシー | 動作 | 適用場面 |
|---------|------|---------|
| **シンプル** | 登録した値を返す（複数あればランダム） | 単一エンドポイント |
| **加重（Weighted）** | 重みに比例して返す | カナリアデプロイ、A/Bテスト |
| **位置情報（Geolocation）** | ユーザーの地理的位置で返す値を変える | リージョン別コンテンツ配信 |
| **フェイルオーバー** | ヘルスチェックに基づきプライマリ/セカンダリを切り替え | DR構成 |
| **レイテンシー** | 最も低レイテンシーのリージョンに誘導 | グローバルサービス |

### 6.3 ALB構築手順

**構成概要**:
```
インターネット
      ↓
   ALB（複数AZにまたがって配置）
   ├── リスナー: HTTP:80
   │       ↓
   │  ターゲットグループ（mytarget-group）
   │  ヘルスチェック: HTTP /healthcheck
   │  ├── EC2インスタンス-A（AZ-a）
   │  └── EC2インスタンス-B（AZ-c）
   └── リスナー: HTTPS:443（SSL証明書: ACM）
```

**ALB構築の主要設定項目**:

| 設定項目 | 説明 | ポイント |
|---------|------|---------|
| **スキーム** | `internet-facing`（外部公開）or `internal`（VPC内） | Webは通常 internet-facing |
| **ネットワークマッピング** | AZとサブネットを選択 | 最低2AZ推奨 |
| **ターゲットグループ** | 負荷分散先のEC2グループ | インスタンス/IP/Lambda |
| **ヘルスチェック** | 正常判定のパス（例: `GET /`） | 失敗したインスタンスを除外 |
| **リスナー** | 受信ポート（80/443）とターゲットグループ | HTTPSはACM証明書が必要 |

**Route 53とALBの連携**:
```
Aレコード（エイリアス有効）
  ホスト名: www
  値: ALBのDNS名（エイリアスで選択）
  ← ALBのIPが変動してもRoute 53側の設定変更不要
```

**ALBの特徴**:
- 配下のEC2インスタンスにパブリックIPは不要（ELB→EC2はプライベートIP通信）
- SSL/TLS終端をALBで実施（AWS Certificate Manager: ACMで無料証明書取得可）
- URL パスベースのルーティング、認証機能あり
- ELBのエンドポイントDNS名: `<名前>-<アカウントID>.<リージョン>.elb.amazonaws.com`

## 7. VPC外部接続パターン

### 7.1 接続パターン比較

| パターン | 接続先 | 費用 | 用途 |
|---------|--------|------|------|
| **ゲートウェイエンドポイント** | S3, DynamoDB のみ | **無料** | プライベートIPからS3/DynamoDBへ |
| **インターフェースエンドポイント（PrivateLink）** | 多数のAWSサービス | 時間+データ課金 | プライベートIPから各種AWSサービスへ |
| **VPCピアリング** | 別VPC | データ転送量 | VPC間の直接接続（1対1） |
| **Transit Gateway** | 複数VPC/オンプレ | 時間+データ課金 | 多数VPCの集約管理 |
| **Site-to-Site VPN** | オンプレミス/他クラウド | 時間+データ課金 | IPsecによる拠点間接続 |
| **Direct Connect** | オンプレミス | 専用線費用+ポート課金 | 安定・大帯域のオンプレ接続 |

### 7.2 ゲートウェイエンドポイント（S3・DynamoDB）

**仕組み**: ルートテーブルにエンドポイントへのルートを追加するだけ。ENIは作成されない。

```
Private Subnetのルートテーブル（エンドポイント設定後）:
  宛先: pl-xxxxxxxx (S3 prefix list)  →  ターゲット: vpce-xxxxxxxx
  宛先: 0.0.0.0/0                     →  ターゲット: nat-xxxxxxxx
```

**設定手順**:
1. VPCメニュー → [エンドポイント] → [エンドポイントを作成]
2. サービスカテゴリ: `AWSのサービス`、サービス: `com.amazonaws.<リージョン>.s3`（ゲートウェイ）
3. VPCを選択
4. ルートテーブルを選択（エンドポイントへのルートが自動追加される）

**メリット/制約**:
- NAT Gateway経由のS3通信費用（データ処理量課金）を節約できる
- エンドポイントポリシーで特定バケットのみに制限可能
- 同一リージョン内のS3/DynamoDBのみ。クロスリージョンは不可
- VPCピアリングや Transit Gateway を経由した先のVPCからは利用不可

### 7.3 インターフェースエンドポイント（PrivateLink実践）

**仕組み**: VPC内にENIが作成され、AWSサービスのDNSがそのENIのプライベートIPに解決される。

```
Private Subnet
└── ENI（インターフェースエンドポイント）
    プライベートIP: 10.0.1.xxx
    ↓ (PrivateLink経由)
    AWS サービス（Systems Manager / CloudWatch / ECR / STS 等）
```

**設定手順**:
1. VPCメニュー → [エンドポイント] → [エンドポイントを作成]
2. サービスカテゴリ: `AWSのサービス`、サービスタイプ: `インターフェース`
3. VPC・サブネット・セキュリティグループを選択
4. プライベートDNS有効化（オプション）: AWSサービスのDNSがENIのIPに解決される

**ゲートウェイ vs インターフェース 使い分け**:

| 判断軸 | ゲートウェイ | インターフェース |
|--------|------------|----------------|
| 対象サービス | S3, DynamoDB のみ | その他AWSサービス全般 |
| 費用 | **無料** | 時間+データ転送量課金 |
| プライベートDNS | 非対応（`--endpoint`オプション必要） | 対応（透過的にアクセス可） |
| AZ跨ぎ | ルートテーブルのみ | ENIをAZごとに配置 |

**主なユースケース**:
```
パブリックIPなし環境 → Systems Manager SessionManager でEC2接続
パブリックIPなし環境 → CloudWatch Logs への書き込み
パブリックIPなし環境 → ECR からのコンテナイメージ取得
コンプライアンス要件 → インターネット経由なしでAWSサービスを利用
```

### 7.4 VPCピアリング（設定手順・ルーティング制約）

**概要と制約**:

| 項目 | 詳細 |
|------|------|
| **接続形態** | 2つのVPC間の1対1接続 |
| **トランジットルーティング不可** | VPC①→VPC②→VPC③ の中継は禁止。①と③を繋ぐには別途ピアリングが必要 |
| **CIDR重複禁止** | 接続するVPCのCIDRが重複すると設定不可 |
| **クロスアカウント可** | 他のAWSアカウントのVPCとも接続可能 |
| **クロスリージョン可** | 異なるリージョン間のVPCも接続可能 |
| **費用** | 同一AZ内は無料。AZ間・リージョン間はデータ転送量課金 |

**設定手順**:
```
1. ピアリング接続の作成
   [VPC] → [ピアリング接続] → [ピアリング接続を作成]
   リクエスタ（接続元VPC）とアクセプタ（接続先VPC）を選択

2. ピアリング接続の承認
   アクセプタ側で「承認」の操作が必要（同一アカウントなら自動承認設定可）

3. ルートテーブルの編集（両側で必要）
   myvpc01のルートテーブル:
     宛先: 10.1.0.0/16 → ターゲット: pcx-xxxxxxxx（ピアリング接続）
   myvpc02のルートテーブル:
     宛先: 10.0.0.0/16 → ターゲット: pcx-xxxxxxxx（ピアリング接続）

4. セキュリティグループの調整
   相手VPCのCIDRからの通信を許可するルールを追加
```

**3つ以上のVPC接続時の注意**:
```
VPC①←→VPC②←→VPC③ の構成では、①と③は通信できない
→ ①←→③ の直接ピアリングが別途必要
→ 多数VPCの場合は Transit Gateway の使用を検討
```

### 7.5 Site-to-Site VPN（VGW・CGW・IPsec）

**コンポーネント**:

| コンポーネント | 役割 | 配置場所 |
|-------------|------|---------|
| **VGW（仮想プライベートゲートウェイ）** | AWS側のVPN接続点。VPCにアタッチ | AWS VPC |
| **CGW（カスタマーゲートウェイ）** | オンプレミス側ルーターの論理表現（固定IP保持） | AWS上の設定（実体はオンプレ側） |
| **VPN接続** | VGWとCGWを繋ぐIPSec VPNトンネル（冗長で2本） | AWS管理 |

**接続構成**:
```
オンプレミス環境 (192.168.0.0/24)
└── IPSec対応ルーター（固定グローバルIP必須）
        ↕  IPSec (2本のVPNトンネルで冗長化)
    VGW（仮想プライベートゲートウェイ）
        ↕
    VPC (10.0.0.0/16)
```

**設定手順**:
```
1. 仮想プライベートゲートウェイ（VGW）の作成・VPCへのアタッチ
   [VPC] → [仮想プライベートゲートウェイ] → [作成] → [VPCへのアタッチ]

2. VPN接続の作成（カスタマーゲートウェイも同時設定）
   [VPC] → [Site-to-Site VPN接続] → [VPN接続を作成]
   - カスタマーゲートウェイIP: オンプレ側ルーターのグローバルIP
   - ルーティング: 動的（BGP推奨）or 静的

3. ルーター設定ファイルのダウンロード・適用
   [アクション] → [設定をダウンロード]
   ベンダー（Cisco/Yamaha/Fortinet等）を選択してファイルを取得・適用

4. ルート伝達の有効化（BGP使用時）
   サブネットのルートテーブル → [アクション] → [ルート伝達の編集] → 有効化
   ← BGPで交換されたルート情報が自動的にルートテーブルに反映される
```

**ルーティング方式の比較**:

| 方式 | 特徴 | 推奨 |
|------|------|------|
| **動的ルーティング（BGP）** | ルート情報を自動交換。ルート伝達を有効化すると自動設定 | 推奨 |
| **静的ルーティング** | 宛先ネットワークを手動でルートテーブルに追加 | 小規模・シンプルな構成 |

**注意点**:
- VPN接続を作り直すとAWS側の接続受入IPが変わる → オンプレ側ルーターの再設定が必要
- VGWは稼働中ずっと課金される（NAT Gatewayと同様）
- VPNトンネルは2本あり冗長化（Tunnel1とTunnel2がそれぞれUpであることを確認）
- セキュリティグループでICMPが禁止されているとpingテストは失敗する
