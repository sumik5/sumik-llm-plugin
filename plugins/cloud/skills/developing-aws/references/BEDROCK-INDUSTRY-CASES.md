# Bedrock 業界別ケーススタディ

E-commerce / Finance / Media & Entertainment の3業界における本番グレードBedrockアーキテクチャパターン。

---

## 業界別特性概観

| 業界 | リスク許容度 | 主要課題 | 採用スピード |
|------|-------------|---------|------------|
| E-commerce | 高 | 個人化ROI・コスト削減 | 速い |
| Finance | 低 | コンプライアンス・監査可能性 | 慎重 |
| Media & Entertainment | 中〜高 | クリエイティブ速度・多言語展開 | 積極的 |

**共通原則**: GenAI成功は「モデル性能」よりも「エンジニアリングの厳密さ」に依存。信頼性・スケーラビリティ・ビジネス制約への適合がアーキテクチャ品質を決定する。

## E-commerce

### フライホイールモデル

GenAI投資は線形ではなく複利的な価値を生む。

```
Engage（AIによる豊かなUX）
  → Capture（リッチなファーストパーティデータ蓄積 → S3データレーク）
  → Refine（RAG KB拡充 + Personalizeモデル再学習）
  → Improve（精度向上 → さらなるEngagement）→ (サイクル継続)
```

**ポイント**: データインフラ（S3・ベクトルDB・パイプライン）への投資は複利で価値を生むプラットフォーム投資。

---

### ユースケース1: RAGベース生成的レコメンデーション

従来の協調フィルタリングを超え、コンテキストを考慮したナラティブ提案を生成。

```
従来: 「おすすめ商品」+ 商品画像カルーセル
GenAI: 「先日ハイキングブーツをご購入いただいたので、
        この防風ジャケットはいかがでしょうか。本日40%オフです。」
```

#### Conversational Commerce Engine アーキテクチャ

```
[ユーザー] → [Bedrock Agents (オーケストレーター)]
  ├─→ KB Retrieve API → [Vector Store]          (RAGによる商品・ポリシー検索)
  ├─→ Lambda → Amazon Personalize               (統計的レコメンデーション)
  ├─→ Lambda → DynamoDB                         (ユーザープロフィール・注文履歴)
  └─→ Lambda → ERP/外部API                      (リアルタイム在庫確認)
       ↓ 拡張プロンプト (全コンテキスト統合)
  [Bedrock FM] → 個人化ナラティブ応答
```

#### ベクトルストア選択: 速度 vs 耐久性

| 特性 | Redis (ElastiCache/MemoryDB) | DynamoDB | OpenSearch Service |
|------|------------------------------|----------|--------------------|
| ストレージ | インメモリ (RAM) | SSD | SSD |
| レイテンシ | マイクロ秒 | 一桁ミリ秒 | ミリ秒 |
| ベクトル検索 | ネイティブ (RediSearch) | 非ネイティブ | ネイティブ k-NN |
| ベストユースケース | リアルタイム会話型RAG | ユーザー/注文データ | 検索+分析複合 |
| スケーラビリティ | 水平 (Cluster Mode) | 実質無限 | 水平 |

**推奨**: ポリグロット永続化 — RAG検索にRedis/OpenSearch + 耐久性データにDynamoDB。

### ユースケース2: AIパワードカスタマーサポートチャットボット

レコメンデーションと同一RAG/エージェントフレームワークを使用するが、**KBの内容**が異なる。

**KBコンテンツ**: FAQ・返品ポリシー・配送情報・保証詳細・トラブルシューティングガイド

**能力の進化**:

| タイプ | 対応例 |
|--------|--------|
| 汎用チャットボット | 「返品ポリシーは？」 (静的) |
| RAGベース | 「破損品の返品方法は？」 (文脈的) |
| RAG + 注文DBアクセス (DynamoDB read-only) | 「注文#12345のテントが破損。どうすれば？」 → 注文詳細取得・保証確認・解決策提示 |

**コスト最適化**: 高スループット定型クエリには高コストモデル不要。コスト効率の高いモデル選択で品質維持しながら最大66%コスト削減可能。

## Finance

### コンプライアンスサンドイッチ: 基本アーキテクチャ原則

確率論的LLMを決定論的制御レイヤーで挟む。

```
[入力レイヤー (決定論的)]
  RAGグラウンディング + バージョン管理プロンプト + PIIフィルタリング
          ↓
[LLMコア (確率論的)]
  Amazon Bedrock Foundation Model
          ↓
[出力レイヤー (決定論的)]
  Automated Reasoningチェック + 機密情報リダクション + 包括的ロギング
```

**3つの非交渉的柱**:

| 柱 | 内容 | 実装 |
|----|------|------|
| Compliance | AML/GFC等規制への準拠を証明可能に | Guardrails + 正式AIポリシー |
| Auditability | 6ヶ月前の決定を完全再現可能 | CloudWatch + S3不変ログ |
| Latency | 遅延 = 直接的リスク (不正エクスポージャー拡大) | CloudWatch P99 + SLAアラーム |

---

### ユースケース1: 説明可能AIによる不正検知強化

既存の検知エンジンを置き換えるのではなく、説明可能性で拡張する。

```
[トランザクションデータ] → [S3 / RDS]
[既存ML検知システム] → 異常検知時にブロックせずイベント発行
                              ↓
[Amazon EventBridge] ← トランザクションID + リスクスコア
                              ↓ Lambda起動
[Lambda (説明オーケストレーター)]
  1. S3/RDSから取引全コンテキスト取得
  2. Guardrails (PIIリダクション) を通してBedrockに送信
  3. Bedrock FM → シニア不正アナリストとして箇条書き説明生成
  4. Guardrails出力チェック
     ├─→ DynamoDB: 説明テキスト保存 (アナリストダッシュボード)
     └─→ S3 (不変監査ログ): プロンプト + 生応答 + Guardrails評価
[CloudWatch] InvocationLatency P99 > 2秒 → リスクアラート
```

### ユースケース2: コンプライアンスレポート自動化 (SAR生成)

```
[ナレッジコーパス: S3]
  コンプライアンスマニュアル・規制ガイドライン・AMLポリシー・SAR記述例
    ↓ [Bedrock Knowledge Bases]
  Titan Embeddingsベクトル化 → OpenSearch Serviceインデックス

[プロンプトレジストリ: DynamoDB]
  PromptTemplateID / semver (v1.2.0) / プレースホルダー付きテキスト
  ※ コンプライアンス担当者がコードデプロイ不要で更新可能

[ワークフロー]
1. アナリストがSAR対象取引をフラグ
2. RDSから構造化取引データ取得
3. DynamoDBからプロンプトテンプレート取得・充填
4. Bedrock RetrieveAndGenerate → SAR草案生成
5. [最重要] Guardrails Automated Reasoning検証
   ポリシー例: "取引額>$10,000 AND 高リスク国関与 → SAR必要"
   → LLMナラティブが論理的に一致するか数学的検証
   → Valid/Invalid + 説明を返却 (確率論的出力への決定論的ゲート)
6. CloudWatchに完全監査証跡 (テンプレートバージョン・KBチャンク・生応答・検証結果)
```

### ユースケース3: 責任ある金融アドバイザリーアシスタント

```
[データ源]
RDS ← ユーザーポートフォリオ・保有銘柄・取引履歴 (個人化基盤)
KB  ← 承認済みマーケットコメンタリー・目論見書・教育記事 (公開インターネット不使用)

[免責事項動的注入: DynamoDB]
法務部作成・バージョン管理済みテンプレート → クエリタイプに応じてプロンプトへ動的挿入
例: "過去のパフォーマンスは将来の結果を保証しません..."

[Guardrails: 禁止トピック]
- 株式の将来価格予測
- 特定投資のBuy/Sell/Hold推奨
- 投資リターンの保証
→ ユーザー入力 + モデル応答の両方をチェック → トリガー時: コンプライアント定型応答
```

### プロンプト再現性・監査可能性のベストプラクティス

**Semantic Versioning**:
- `MAJOR` (v1.5.2→v2.0.0): コンプライアンス影響のある非互換変更
- `MINOR` (v2.0.0→v2.1.0): 後方互換性のある機能追加
- `PATCH` (v2.1.0→v2.1.1): タイポ修正・指示の明確化

**監査ログ必須スキーマ**:

| フィールド | 型 | 説明 |
|-----------|-----|------|
| `InvocationID` | UUID | Bedrock APIコール一意識別子 |
| `TraceID` | String | 上位ビジネスプロセスへの紐付け |
| `PromptTemplateID` | String | 使用テンプレートID (DynamoDB PK) |
| `PromptTemplateVersion` | String | semverまたはGitコミットハッシュ |
| `InputVariables` | JSON | テンプレートに注入した動的データ |
| `FinalPrompt` | String | 実際に送信した完全プロンプト (完全再現に必須) |
| `ModelID` | String | 呼び出したモデルのフルARN |
| `InferenceParameters` | JSON | temperature・max_tokens・top_p等 |
| `RawModelResponse` | String | Bedrock APIの未加工JSONレスポンス |
| `GuardrailEvaluation` | JSON | 評価結果 (PASS/BLOCK + アセスメント詳細) |

## Media & Entertainment

### ユースケース1: 自動ローカライゼーションパイプライン

機械翻訳の速度 + FMの文化的精緻化による「トランスクリエーション」エンジン。

```
[S3: 新規動画/音声アップロード]
        ↓ S3 Event Notification
[AWS Step Functions]
  Step1: Amazon Transcribe → タイムスタンプ付き書き起こし (話者識別対応)
  Step2: Amazon Translate → ベースライン機械翻訳 (多言語並列)
  Step3: Bedrock FM (プロのローカライゼーション編集者として動作)
         → トーン・文化的適切性・簡潔性の精緻化
         → 字幕フォーマット制約の遵守 (例: 1行42文字以内)
  Step4: Amazon Polly → 高品質音声ファイル生成 (地域アクセント対応)
        ↓
[S3: 完成したローカライズ済みコンテンツ]
```

**効果**: 数日→数分、数市場→数十言語市場への同時展開。

#### トランスクリエーションプロンプトの4原則

1. **ペルソナ割り当て**: "アクション映画の日本語字幕専門家として..."
2. **明確な制約**: "各行42文字以内、フォーマルなトーン"
3. **XMLタグによる内容分離**: `<text_to_translate>...</text_to_translate>`
4. **Few-Shotサンプル提供**: 理想的な入出力ペアを1〜2例プロンプトに含める

### ユースケース3: インタラクティブ・ナラティブエージェント

**NPC (非プレイヤーキャラクター) の動的化**: 事前スクリプトの会話ツリーを超えた没入型体験。

#### ナラティブエージェントの構成要素

| コンポーネント | 役割 | 実装 |
|-------------|------|------|
| Instruction | ペルソナ定義 (性格・目標・話し方) | エージェントのシステムプロンプト |
| Knowledge Base | ゲームローア・世界観・クエスト詳細 | Bedrock KB (ハルシネーション防止) |
| Action Groups | ゲームワールドとのインタラクション | Lambda + OpenAPI Schema |
| Memory | 会話コンテキスト保持 | Bedrock AgentCore (短期+長期) |

**マルチエージェント・ナラティブシステム**:

```
[Story Director (スーパーバイザー)]
  ├─ Quest Giver
  ├─ Suspicious Guard  ← プレイヤーの行動に応じて動的に態度変化
  └─ Shady Merchant    ← 情報共有でGuardに通知

※ ストーリーは事前スクリプトではなくエージェント間インタラクションから「創発」する
```

### ユースケース4: エンドツーエンドメディアパイプライン

**ポッドキャスト → 多言語パーソナライズ動画クリップへの完全自動変換**:

```
S3アップロード → EventBridge → Lambda
  → Amazon Transcribe: 書き起こし (話者・タイムスタンプ付き)
  → Bedrock (Claude): 要約・テーマ抽出・メタデータタグ付与
  → Amazon Translate: 多言語並列翻訳
  → Bedrock: プラットフォーム別トーン再生成

  → Amazon Polly: 地域アクセント対応の音声生成
  → S3にメタデータ付き保存
  → Amazon Personalize: 適切なスニペットを適切なユーザーへ配信
```

## 業界横断: アーキテクチャパターン比較

| パターン | E-commerce | Finance | M&E |
|---------|-----------|---------|-----|
| RAG (Bedrock KB) | ✅ 商品・購買履歴 | ✅ コンプライアンス文書 | ✅ ゲームローア・コンテンツ |
| Bedrock Agents | ✅ ショッピングアシスタント | ✅ レポーティング | ✅ NPCエージェント |
| Guardrails | ✅ コンテンツ品質 | ✅ PII・コンプライアンス | △ キャラクター制約 |
| DynamoDB | ✅ ユーザー・注文 | ✅ プロンプトテンプレートレジストリ | ✅ ゲームステート |
| Step Functions | △ バッチ | ✅ 多段階検証 | ✅ メディア変換 |
| EventBridge | △ | ✅ 異常検知トリガー | ✅ S3アップロードトリガー |
| Amazon Personalize | ✅ 商品推薦 | ✗ | ✅ コンテンツ推薦 |
| 監査ログ (S3不変) | △ | ✅ 必須 | △ |

**5つの汎用原則**:
1. **RAG first**: 全業界で組織固有データへのグラウンディングが必須
2. **ポリグロット永続化**: 高速検索 (Redis/OpenSearch) + 耐久性 (DynamoDB/S3)
3. **Guardrailsは設計要件**: 後付けではなく最初から組み込む
4. **プロンプトをコードとして扱う**: バージョン管理・中央管理 (DynamoDB)・ランタイム更新
5. **モデル選択はユースケース依存**: 高スループット定型クエリにはコスト効率優先
