# クラウドネイティブ・アーキテクチャパターン

AWSを前提としたクラウドネイティブなシステム設計の基礎概念、責任共有モデル、アーキテクチャパターン、認証・認可の仕組みを解説します。

---

## クラウドネイティブとは

### 定義

クラウドネイティブとは、**クラウドのサービスを前提に構築されたシステムおよびアプリケーション**のことです。仮想サーバ上に自前でミドルウェアをインストール・設定する従来方式ではなく、SaaSを最大限活用して拡張性・可用性のあるシステムを組み立てていくスタイルです。

> 「Cloud is the New Normal」— Amazon副社長 Andy Jassy氏

### クラウドファーストとの違い

| 概念 | 意味 | 実践例 |
|------|------|--------|
| **クラウドファースト** | システム導入時はまずクラウドで実現できないか検討する | EC2/RDSへのリフト&シフト |
| **クラウドネイティブ** | クラウドサービスの特性に沿って設計・構築する | Lambda/DynamoDB/API Gatewayを組み合わせてサーバレス構築 |

### クラウドネイティブの第一歩

- EC2上に自前でアプリケーションサーバ・DBをインストールするのではなく、ElasticBeanstalk（PaaS）やRDS（DBaaS）を活用
- LambdaやDynamoDB等、ユーザーがサーバを管理しないマネージドサービスを利用
- **AWSのアプリケーションサービスをいかに上手く取り入れるかがAWS活用の鍵**

## AWSサービスの責任共有モデル

### モデルの概念

AWSのセキュリティの考え方のベースとなるのが**責任共有モデル**です。AWSと利用者が、それぞれ責任を持つ範囲を明確に分担します。

### EC2（IaaS）の責任分担

```
利用者の責任範囲
├── アプリケーション
├── アカウント管理
├── データ暗号化
├── ウイルスチェック
├── セキュリティグループ
├── ネットワークACL
├── OSファイアウォール機能
└── OS（脆弱性対応含む）
─────────────── 責任分解点 ───────────────
AWSの責任範囲
├── 仮想インフラ（ハイパーバイザ）
├── ネットワークインフラ
├── 物理インフラ
└── ファシリティ（データセンター入館等）
```

### 役割分担の具体例

| AWSが対応すべき事項 | 利用者が対応すべき事項 |
|---------------------|------------------------|
| データセンターの堅牢性確保 | OSの脆弱性対応 |
| 電源・ネットワークの冗長化 | データのウイルスチェック |
| データセンターへの入館ポリシー | HTTPS通信によるデータ暗号化 |
| サーバやネットワーク機器の安全性 | セキュリティグループでの通信制御 |
| 利用者アカウントの管理 | アプリケーションの脆弱性（XSS等） |
| ハイパーバイザの脆弱性対応 | — |

### サービス形態別の責任分担

| サービス形態 | 代表サービス | 利用者の責任範囲 | AWSの責任範囲 |
|-------------|------------|----------------|--------------|
| **IaaS** | EC2 | OS以上のすべて | ハイパーバイザ以下 |
| **PaaS** | RDS, ElasticBeanstalk | アプリケーション・データのみ | OS・ミドルウェア以下 |
| **フルマネージド（SaaS的）** | Lambda, DynamoDB | ビジネスロジック・権限設定のみ | インフラ・スケーリングすべて |

**フルマネージドサービスの利点**: AWSが可用性・キャパシティ調整を担うため、利用者はビジネスロジックに集中できます。

## サーバレスアーキテクチャ

### 「サーバレス」の正確な意味

AWSの文脈での「サーバレス」とは**「サーバを使用しない」ではなく「サーバ（メンテナンス）レス」**です。AWSが管理するサービスを利用し、ユーザーはサーバを直接管理しないという意味です。

### 従来アーキテクチャとの比較

**従来のEC2ベースアーキテクチャ:**
```
モバイル/PC → ロードバランサー(ELB) → Web/APサーバ(EC2) → データベース(RDS)
```
課題:
- EC2上のOS・ミドルウェアのセキュリティパッチ適用が必要
- 可用性確保のため最低2台を常時稼働（コスト効率が悪い）
- 急激な負荷上昇への対応が困難

**サーバレスアーキテクチャ:**
```
モバイル/PC → API Gateway → Lambda → DynamoDB
                                    ↓
                            S3 / SQS / Cognito
```
利点:
- サーバ管理不要（AWSが担当）
- 自動スケーリング
- LambdaはEC2（1時間課金）と異なりミリ秒課金 → アイドル時のコストゼロ
- イベントドリブンなリソース利用でリソース効率が高い

## 2-Tier vs 3-Tierアーキテクチャ

### アーキテクチャの選択基準

| 観点 | 2-Tier | 3-Tier |
|------|--------|--------|
| **クライアントの役割** | AWSリソースを直接操作 | APIを介して間接的に操作 |
| **シンプルさ** | シンプル（直結） | 複雑（サービス経由） |
| **疎結合性** | 密結合になりやすい | 疎結合（フロント/バックエンド分離） |
| **オーバーヘッド** | 少ない | サービス経由のオーバーヘッドあり |
| **AWS SDK** | クライアント側に必要 | 不要（REST APIのみ） |
| **マルチプラットフォーム対応** | OS毎にロジック構築が必要 | バックエンドを共通化しやすい |
| **主な認証方式** | Cognito（Identity Federation） | アクセストークン（JWT等） |

## 2-Tierアーキテクチャ

### 構成概要

クライアント（モバイル/JavaScript）がAWSリソースを直接操作するパターン。

```
モバイルクライアント(iOS/Android SDK)
クライアントPC(JavaScript SDK)
       ↓ 直接操作
AWS:
  - 認証管理 (Cognito)
  - ストレージ (S3)
  - キューサービス (SQS)
  - NoSQL (DynamoDB)
  - 計算処理 (Lambda)
```

### セキュリティの仕組み（STSとCognito）

**問題**: クライアント側にAccess Key/Secret Access Keyを埋め込む方法はセキュリティ上不可（漏洩リスク）。

**解決策**: **AWS STS（Security Token Service）** による一時的セキュリティ認証情報の発行。

- 有効期限付き（数分〜数時間）の認証情報を発行
- 期限切れで自動失効
- アクセス可能なリソースを権限として設定可能

**CognitoによるSTSの実装**:

```
① クライアントがIdentity Provider（Facebook/Google/Twitter等）でID/パスワード認証
② 認証成功 → Identity ProviderがトークンをCognitoに送付
③ CognitoがトークンIDを受け取りIAMロールと紐付け
④ Cognitoが一時的Credentialトークンを発行
⑤ クライアントはトークンを使ってAWSリソースを利用
```

Cognitoの補足:
- 未認証ゲストアクセス（IAMロール設定次第）も可能
- Cognito Sync機能により、同一ユーザーの複数端末間でデータ同期も実現

### 2-Tierのメリット・デメリット

| メリット | デメリット |
|---------|-----------|
| バックエンド開発・運用コストを最小化 | クライアントとAWSが密結合になりやすい |
| フルマネージドサービスで自動スケーリング | クライアント側にロジックが集中・肥大化 |
| Lambdaのミリ秒課金でコスト効率が高い | 複数OS（iOS/Android/JS）で同ロジック実装が必要 |
| 構成がシンプルで通信オーバーヘッドが少ない | — |

**適した用途**: ファイル・画像を大量に扱うアプリ、シンプルなCRUDアプリ

---

## 3-Tierアーキテクチャ

### 構成概要

クライアントがAPI Gateway（HTTPエンドポイント）を介してバックエンドリソースを操作するパターン。2015年6月のAPI Gateway発表により普及。

```
モバイルクライアント(iOS/Android)
クライアントPC(ブラウザ)
       ↓ リクエスト/レスポンス
AWS:
  API Gateway
       ↓ 処理要求
  Lambda（計算処理）
       ↓ リソース呼び出し
  S3 / SQS / DynamoDB / Cognito
```

> 実質的には4層（Client → API Gateway → Lambda → DB/Storage）になりますが、「2-Tier」の対比として「3-Tier」と呼びます。

### API Gateway + Lambdaの役割分担

- **API Gateway**: HTTPリクエストの受付、APIバージョニング、スロットリング、認証・アクセス権管理
- **Lambda**: ビジネスロジックの実装。HTTP直接受付はできないためAPI Gatewayと組み合わせる

### 3-Tierの認証フロー

```
①クライアント → 認証要求（ID/パスワード等）
②認証サーバ/Cognito → アクセストークン発行
③クライアント → APIリクエスト（Authorizationヘッダーにトークン付与）
④API Gateway/Lambda → トークン検証・処理
```

クライアント側ではAWS SDKは不要。REST APIとJSONがわかれば開発可能。

### 3-Tierのメリット・デメリット

| メリット | デメリット |
|---------|-----------|
| フロントエンド/バックエンドを疎結合化 | 2-Tierより経由サービスが多くオーバーヘッド増 |
| バックエンドをAWSと意識させない（クライアント非依存） | 構成が複雑に見える |
| バックエンドを従来EC2構成→サーバレスへ段階移行しやすい | — |
| iOS/Android/WebでバックエンドAPIを共通化できる | — |
| WebHook対応でイベントドリブン連携が可能（GitHub/CircleCI等） | — |

**適した用途**: マルチクライアント対応が必要なサービス、段階的なサーバレス移行、チーム分業が必要な中〜大規模サービス

## 認証と認可の仕組み

### 認証(Authentication) vs 認可(Authorization)

| 概念 | 定義 | 例 |
|------|------|-----|
| **認証（Authentication）** | 本人かどうかを確認する | ID/パスワードによる本人確認 |
| **認可（Authorization）** | リソースに対する権限を付与する | 一般ユーザーは読取のみ、管理者は書込も可能 |

**重要な認識**: 認証と認可は別の役割。GoogleやTwitterで認証し、その事実をもって別サービスが権限付与を行うことがある（OAuth）。

### AWSの認証方式

| 認証方式 | 利用者 | 用途 |
|---------|-------|------|
| ID/パスワード | AWSコンソール利用者 | Webブラウザでのマネジメントコンソール |
| アクセスキー/シークレットアクセスキー | プログラム・CLI | SDK/CLI経由のAPI呼び出し |
| IAMロール | EC2インスタンス | サーバ自体への認証情報付与 |
| X.509証明書 | 一部サービス | 証明書ベースの認証 |
| MFA（多要素認証） | 全ユーザー | 認証強化（上記に加えて） |
| Cognito（外部IdP連携） | モバイル/フロントエンド | Facebook/Google/Twitter等でのFederation認証 |

### AWSの認可（IAMポリシー）

```json
// IAMポリシーの構造（JSON形式）
{
  "Resource": "対象AWSサービス（ARN）",
  "Action": "許可する操作（s3:GetObject等）",
  "Condition": "利用条件（IPアドレス制限等）"
}
```

| ポリシー種別 | 適用範囲 | 特徴 |
|------------|--------|------|
| **インラインポリシー** | ユーザー/グループ/ロール単位 | 個別に直接適用 |
| **管理ポリシー** | 複数のユーザー/グループ/ロール | 再利用可能・一元管理 |

### AWSの認可 vs アプリケーションの認可

**重要**: AWSの認証・認可とアプリケーションの認証・認可は基本的に別物。

| 観点 | AWSの認可 | アプリケーションの認可 |
|------|----------|---------------------|
| 制御の粒度 | 「利用できる・できない」の2値が基本 | ユーザーごとに参照可能なデータを制御 |
| 実装方法 | IAMポリシー（JSON定義） | バックエンドのビジネスロジック（Lambda/EC2） |
| 例 | S3バケットへのアクセス可否 | Facebookのタイムライン表示範囲 |

**アプリケーション認可の実装指針**:
- Lambda（サーバレス）: 小規模・シンプルな権限制御に適する
- EC2（既存APサーバ活用）: 複雑な権限設定・既存ミドルウェア活用が必要な場合

## アーキテクチャのパターン

### フロントエンドアプリケーション構成パターン

HTML5 + JavaScript + CSSで構築されたWebアプリケーション（PC/ブラウザ）向けのパターン。

#### フロントエンド 2-Tierパターン

デスクトップアプリケーション（Electron等）で採用されることが多い。

```
Electronアプリケーション
  HTML5 + JavaScript/Node.js
  AWS JavaScript SDK（直接接続）
  セキュリティキー（設定画面で入力）
        ↓ SDKを介してAWSリソース直接操作
AWS:
  S3（ストレージ）
  SQS（キューサービス）
  DynamoDB（NoSQL）
  Lambda（計算処理）
```

認証方法: アクセスキーを設定画面で登録 or Cognito（Facebook/Twitter連携）

#### フロントエンド 3-Tierパターン

ブラウザアプリケーションで最も一般的なパターン。

```
ブラウザアプリケーション
  HTML5 + JavaScript
        ↓ REST API（JSON）
AWS:
  API Gateway（エントリポイント）
  Lambda（処理）
  Cognito（認証管理）
  S3/DynamoDB（データ保管）
```

認証フロー: ① 認証要求 → ② アクセストークン発行 → ③ 以降のAPIリクエストにトークン付与

特徴: クライアントはバックエンドがAWSかどうかを意識不要。JSON I/Fのみ理解すれば開発可能。

---

### モバイルアプリケーション構成パターン

iOS/Androidアプリ向けのパターン。フロントエンドとほぼ同様の2-Tier / 3-Tier構成。

#### モバイル 2-Tierパターン

ファイル・画像を大量に扱うアプリに適する。

```
モバイルアプリ（iOS/Android SDK）
        ↓
認証プロバイダ（Facebook/Google/Twitter等）
        ↓ トークン
Cognito（IAMロール紐付け・一時的Credential発行）
        ↓
AWSリソース直接操作（S3/DynamoDB/Lambda等）
```

注意: アクセスキーのモバイル埋め込みはセキュリティリスク → **Cognito経由が推奨**

**AWS Mobile Hub** を利用すると2-Tier構成のひな形を自動生成可能:

| 機能 | 対応サービス |
|------|------------|
| ユーザーサインイン | Cognito |
| ユーザーストレージ | Cognito + S3 |
| クラウドロジック | Lambda |
| プッシュ通知 | SNS |
| アプリ分析 | Amazon Mobile Analytics |
| コンテンツ配信 | CloudFront + S3 |
| アプリテスト | AWS Device Farm |

#### モバイル 3-Tierパターン

バックエンドシステムとAPI経由で通信するタイプ。

```
iOS / Android / フロントエンドWebアプリ
（同一APIエンドポイントを共有可能）
        ↓ REST API（JSON）
AWS API Gateway
        ↓
Lambda / Cognito / S3 / DynamoDB
```

**最大のメリット**: iOS/Android/Webの3クライアントが同一バックエンドAPIを利用 → 開発・運用コスト削減

### IoT構成パターン

IoT（Internet of Things）では対象デバイスにより構成が多様。デバイスとの接続形態とデータ収集方式で3パターンに分類。

#### パターン1: 単一デバイスとの接続

1台〜少数のデバイスとの接続。データ量が少ないため、通常のモバイルアーキテクチャと同様。

```
iBeacon/BLEセンサー
        ↓ BLE信号
iOSアプリ（モバイルハブ）
        ↓ REST API
AWS: API Gateway → Lambda → DynamoDB
```

**活用例**: iBeaconと連動した勤怠管理、Apple Watchによる健康データ収集

#### パターン2: 大量デバイスからのデータ収集（一方向）

数千〜数万のセンサーから大量データをリアルタイム収集するパターン。

```
BLEセンサー群（多数）
        ↓ PUT（SDK or MQTT）
AWS Kinesis（リアルタイムストリーミング処理）
        ↓ ETL処理
DynamoDB / S3（データ保管）
        ↓
Lambda（データ処理・分析）
```

**Kinesisの特徴**: リアルタイムストリーミングデータ処理、低コスト・伸縮自在・高可用性

**MQTTプロトコル利用のメリット**: クライアント側にAWS SDKが不要 → センサー等のリソース制限デバイスに適する

#### パターン3: デバイスとの双方向接続

デバイスからのデータをトリガーに、何らかの動作を指示するパターン。

```
デバイス群
        ↕ 双方向接続（AWS IoT SDK）
AWS IoT（デバイス管理・シャドウ機能）
        ↓
Lambda（データ処理）
Kinesis（ストリーミング）
DynamoDB / S3（データ保管）
```

**AWS IoTの特徴**:
- デバイス管理のための証明書・認証の仕組みを提供
- **デバイスシャドウ**: クラウド側でデバイスと1対1対応する仮想状態を保持
- 1台1台のデバイスと双方向でやり取り可能

| IoTパターン | 特徴 | 主要サービス |
|------------|------|------------|
| 単一デバイス | データ量少・シンプル | API Gateway + Lambda |
| 大量収集（一方向） | 大容量・リアルタイム | Kinesis |
| 双方向制御 | デバイス管理・シャドウ | AWS IoT |

## REST API設計の基本

### RESTとは

REST（Representational State Transfer）は、Roy Fielding氏が2000年に提唱したソフトウェア連携のための設計スタイル。API Gatewayを設計するうえで重要な基礎概念。

### 狭義のREST（RESTful）の3原則

| 原則 | 説明 | メリット |
|------|------|---------|
| **ステートレス** | クライアント/サーバともにセッション状態を持たない | スケーラビリティの確保が容易 |
| **URIによるリソース識別** | リソースはURIで一意に表現。動詞ではなく名詞で表現 | URL体系がシンプルで直感的 |
| **HTTPメソッドを利用した操作** | GET/POST/PUT/DELETE/PATCHで操作を表現 | URIに動詞を含める必要がなくなる |

### HTTPメソッドとCRUD対応

| HTTPメソッド | 動作 | CRUDとの対応 | 補足 |
|-------------|------|------------|------|
| **GET** | リソースの取得 | READ | 状態変化を伴ってはならない（冪等） |
| **POST** | 子リソースの作成 | CREATE | 既存URIへのデータ追加 |
| **PUT** | リソースの更新・作成 | UPDATE / CREATE | URIの決定権がある場合に作成で使用 |
| **DELETE** | リソースの削除 | DELETE | — |
| **PATCH** | リソースの部分更新 | UPDATE（部分） | 一部のフィールドだけ変更する際に使用 |

### URI設計パターン（ユーザーリソースの例）

| 動作 | URI | メソッド |
|------|-----|---------|
| 一覧取得 | `https://api.example.org/users` | GET |
| 新規登録 | `https://api.example.org/users` | POST |
| 特定ユーザー取得 | `https://api.example.org/users/{id}` | GET |
| 更新 | `https://api.example.org/users/{id}` | PUT / PATCH |
| 削除 | `https://api.example.org/users/{id}` | DELETE |

**URI設計のポイント**: 名詞で表現、複数形を使用、動詞を含めない（動作はHTTPメソッドで表現）

### RPC vs REST の比較

| 観点 | RPC（Remote Procedure Call） | REST |
|------|------------------------------|------|
| URIの設計 | `getUser`, `addUser`, `listUsers` | `/users`, `/users/{id}` |
| 重視するもの | 動作の多様性 | リソース（名詞） |
| URI複雑性 | 動詞が多くなり複雑化しやすい | シンプルなURI体系 |

### 広義のREST（Web API）

厳密なRESTの定義を満たさないが、「HTTPでアクセスし、データをJSONやXMLで返すAPI」を広義のREST（Web API）と呼ぶ。本書で扱うAPI Gatewayを利用したAPIは、主にこのWeb API形式。

**HTTPステータスコードの使い分け**:
- `2xx`: 成功
- `4xx`: クライアントエラー（400: Bad Request, 401: Unauthorized, 404: Not Found）
- `5xx`: サーバエラー（500: Internal Server Error）

## アーキテクチャ選択の判断フロー

```
要件確認
    │
    ├── サーバ管理を最小化したい？
    │         └─ Yes → クラウドネイティブ（マネージドサービス中心）
    │
    ├── クライアントは1種類（デスクトップのみ等）？
    │         └─ Yes → 2-Tier（シンプル・直結）
    │         └─ No（iOS+Android+Web等）→ 3-Tier（バックエンド共通化）
    │
    ├── データ量・デバイス数が大量？（IoT）
    │         └─ 一方向収集 → Kinesis活用パターン
    │         └─ 双方向制御 → AWS IoTパターン
    │
    └── 認証が必要？
              └─ 2-Tier → Cognito（Identity Federation）
              └─ 3-Tier → アクセストークン（JWT等）
```
