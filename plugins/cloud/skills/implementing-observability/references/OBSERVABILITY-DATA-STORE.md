# オブザーバビリティ特化データストア設計

オブザーバビリティワークロード固有のデータストア設計原則。高カーディナリティ・高ディメンションなイベントデータを任意のフィールドで数秒以内にクエリー可能にするための列指向ハイブリッドアーキテクチャ。

> 出典: 「オブザーバビリティ・エンジニアリング」16章（Honeycomb Retriever実装事例を参考）

---

## 1. オブザーバビリティデータの機能要件

### 1.1 5つのコア要件

| 要件 | 説明 | 違反した場合の影響 |
|------|------|-----------------|
| **リアルタイムクエリー** | 結果は数秒以内に返す（コーヒーを飲む時間があればNG） | インシデント対応が遅延 |
| **高カーディナリティ対応** | ユーザーIDなど無数のユニーク値を持つフィールドでクエリー可能 | 調査が行き詰まる |
| **高ディメンション対応** | 数十〜数百フィールドのイベントを任意の組み合わせで分解 | 問題の原因特定不可 |
| **フレッシュネス** | 取り込みからクエリー可能になるまで数秒以下 | 古いデータで誤判断 |
| **耐障害性** | データストア障害 ≠ オブザーバビリティ停止 | 本番障害中に手がかり消失 |

### 1.2 クエリー特性

```
オブザーバビリティのクエリーパターン:
  特定の時間範囲  →  インデックス優遇される唯一のディメンション
  任意のフィールド →  インデックスなし・事前集計なしでクエリー可能が必須
  反復調査        →  1クエリーで終わらず仮説を変えながら何度もクエリー
```

**重要な原則**: 「どのフィールドが関連するかを調査開始前に知ることはできない」ため、特定フィールドへの事前インデックスは機能要件を満たせない。

## 2. 従来のデータストアの限界

### 2.1 時系列DB（TSDB）の限界

```
通常のTSDB（メトリクス向け）:
  タグ: { method: GET, status: 200 }  → 1時系列
  タグ: { method: POST, status: 500 } → 1時系列
  合計: (メソッド数) × (ステータス数) = 少数の時系列

高カーディナリティフィールドを追加すると:
  タグ: { method: GET, status: 200, user_id: "u-0001" } → 1時系列
  タグ: { method: GET, status: 200, user_id: "u-0002" } → 1時系列
  ...
  ユーザー100万人 → 100万の新規時系列 = カーディナリティ爆発💥
```

| TSDBの問題点 | 詳細 |
|------------|------|
| **カーディナリティ爆発** | タグの一意な組み合わせごとに時系列が作成される |
| **構造化イベント非対応** | 各イベントがユニークな場合、行作成コストが件数に比例 |
| **事前集計の前提** | 取り込み時に集計済み → 後から任意分割不可 |
| **インデックスの限界** | 全カラムにインデックスを付けると元データより大きくなる |

### 2.2 行ベースストア（Bigtable系）の限界

| 問題 | 説明 |
|------|------|
| **コンパクション必要** | ミュータブルデータストアはオーバーレイ層の定期マージが必要 |
| **コンパクションと取り込みの競合** | 新規データを数秒以内にクエリー可能にしたい要件と矛盾 |
| **行全体の読み込み** | 任意フィールドの分析にはタブレット全列の読み取りが必要 |
| **インデックスコスト** | Google Dapper事例: 3フィールドのインデックス = データの76%のサイズ |

### 2.3 データストア適合度まとめ

| データストア | 高カーディナリティ | 任意ディメンション | リアルタイム | オブザーバビリティ適合度 |
|------------|----------------|-----------------|------------|----------------------|
| TSDB (Prometheus/InfluxDB) | ✗ 爆発する | ✗ 事前集計前提 | ✓ | ✗ 不適 |
| 行ベース (Bigtable) | △ | ✗ 全行スキャン | △ コンパクション問題 | △ 困難 |
| NoSQL (MongoDB) | ✓ | △ インデックス依存 | ✓ | △ 任意クエリー困難 |
| インメモリ (Scuba) | ✓ | ✓ | ✓ | ✗ コスト不可能 |
| **列指向ハイブリッド** | ✓ | ✓ | ✓ | ✓ **推奨** |

## 3. 列指向ハイブリッドアーキテクチャ（Retriever参考）

### 3.1 全体アーキテクチャ

```
[イベント受信]
     ↓
[Kafka パーティション]  ← 耐久性・スケーラビリティのバッファ
     ↓
[インジェストワーカー]
     ↓
[セグメント生成]  ← 時間ウィンドウで分割された列指向ファイル
  ├── timestamp.col
  ├── user_id.col
  ├── status_code.col
  └── [任意のフィールド].col
     ↓
[ローカルSSD]  ──→  [S3/Cloud Storage]  ← 時間経過で階層移動
  (ホット)              (ウォーム/コールド)
     ↓
[クエリーエンジン]
  ├── セグメント選択（時間フィルタ）
  ├── 列スキャン（関連列のみ）
  └── MapReduce並列集計
```

### 3.2 行ベース vs 列ベース vs ハイブリッドのトレードオフ

| 戦略 | 得意 | 苦手 |
|------|------|------|
| **行ベース** | 単一行の高速取得・個別トレース取得 | 全列スキャン・任意ディメンション分析 |
| **列ベース（純粋）** | 特定列の高速スキャン | 行全体の再構築・管理コスト |
| **ハイブリッド（推奨）** | 時間で絞り込み + 関連列のみスキャン | 実装複雑度が高い |

## 4. 時間によるセグメント分割

### 4.1 セグメントの概念

```
時間軸 ──────────────────────────────────────────>

セグメントA  [============================]   (最古〜最新タイムスタンプ)
セグメントB          [======================]
セグメントC                  [======================]

クエリー範囲        [──────────────────]
                         ↑ここに重なるセグメントのみスキャン対象
```

### 4.2 セグメント確定のトリガー条件

| 条件 | 例 |
|------|-----|
| 経過時間 | 1時間経過 |
| 書き込み件数 | 25万件超 |
| ファイルサイズ | 1GB超 |

### 4.3 セグメント内のデータ構造

各セグメントは列ごとのファイルに分割して保存：

```
セグメントディレクトリ/
  timestamp.col     ← すべてのイベントが持つ特権的インデックス
  trace_id.col
  user_id.col
  status_code.col
  duration_ms.col
  [その他フィールド].col
```

**列ファイルの構造（シーケンス番号 + 値）**:
```
Raw format:
  idx  value
  1    "foo"
  2    "foo"
  3    "bar"

Compacted format（辞書圧縮 + スパースエンコーディング）:
  dictionary: { 1: "foo", 2: "bar" }
  presence:   [bitmask of non-null rows]
  data:        [1, 1, 2, ...]
```

### 4.4 遅延到着データの扱い

| 到着パターン | 対処方法 |
|------------|---------|
| 数秒〜数分の遅延 | セグメントのタイムスタンプウィンドウが重複を許容 |
| バッチジョブ（数時間〜数日前） | セグメントの時間ウィンドウが広がり、その期間のクエリー対象になる |
| 書き込み中のセグメント | 追記専用ファイル → コンパクション不要 |

## 5. クエリー実行ワークフロー（6ステップ）

```
Step 1: セグメント選択
  クエリー時間範囲 ∩ セグメントタイムスタンプウィンドウ → 対象セグメントリスト

Step 2: 各セグメントを独立処理
  WHERE/SELECT/GROUP BY で使用する列ファイルのみ読み込み

Step 3: 行単位のストリーミング処理
  タイムスタンプ確認 → 時間範囲内なら値評価 → 条件一致なら出力
  ※1行ずつ処理して廃棄 → RAMに1値のみ保持

Step 4: セグメント内集計
  GROUP BY値ごとに COUNT/SUM/MAX等を計算

Step 5: セグメント間マージ
  各セグメントの集計結果をReduceワーカーが統合

Step 6: 上位K件の選択とソート
  デフォルトのK値で返すグループ数を制限
```

### 5.1 クエリー擬似コード（Go形式）

```go
// GROUP BY a, b で x を集計し、y > 0 の行をフィルタ
groups := make(map[Key]Aggregation)
for _, s := range segments {
  for _, row := range fieldSubset(s, []string{"a", "b", "x", "y"}) {
    if row["y"] > 0 { continue }
    key := Key{A: row["a"], B: row["b"]}
    aggr := groups[key]
    aggr.Count++
    aggr.Sum += row["x"]
    if aggr.Max < row["x"] { aggr.Max = row["x"] }
    groups[key] = aggr
  }
}
```

### 5.2 複雑な集計のアルゴリズム

| 集計タイプ | アルゴリズム |
|-----------|------------|
| P99などパーセンタイル | t-digest による分位推定 |
| COUNT DISTINCT | HyperLogLog |
| 単純な COUNT/SUM/MAX/AVG | セグメント間でマージ可能 |

## 6. リアルタイムクエリーの実現

### 6.1 3つのアプローチ比較

| アプローチ | 方式 | トレードオフ |
|-----------|------|------------|
| **ファイル部分フラッシュ（推奨）** | 書き込み中の列ファイルを強制フラッシュしてクエリー | 取り込みとクエリーの関心事を分離できる |
| RAMデータ構造クエリー | 未確定セグメントをメモリ上でクエリー | 共有状態の管理が複雑 |
| 定期強制フラッシュ | 数秒ごとにデータを強制書き出し | 規模拡大でオーバーヘッド増大 |

**Retrieverが採用**: ファイルベースのアプローチ（取り込みとクエリーを完全分離）

## 7. 階層化ストレージ（コスト最適化）

### 7.1 ホット / ウォーム / コールド 階層

```
[ホット層]  ローカルSSD
  対象: 直近数分〜数時間のデータ
  特性: 最速アクセス、コスト高
  操作: シリアライズ中・書き込みアクティブなセグメント

[ウォーム層]  ネットワークストレージ (S3/GCS)
  対象: 数時間〜数日前のデータ
  特性: 中速、コスト中
  操作: 確定・圧縮済みセグメント。辞書圧縮 + LZ4等

[コールド層]  アーカイブストレージ
  対象: 長期保存（数週間〜数ヶ月）
  特性: 最低コスト、アクセス遅延あり
```

### 7.2 コスト比較（参考値）

| ストレージ種別 | 相対コスト | 特性 |
|-------------|----------|------|
| RAM（インメモリ全体） | 40x | Facebook Scuba方式。大規模運用は経済的に不可 |
| ローカルSSD | 1x | ホット層として最適 |
| S3/GCS | 0.025x | ウォーム/コールド層として最適 |

## 8. 並列処理によるクエリー高速化

### 8.1 MapReduce形式の分散クエリー

```
[クエリーリクエスト]
        ↓
[リーダーノード] → セグメントリストをLambda(サーバーレス)に配布
        |
        ├── Lambda #1: セグメントA, B処理 → 中間集計結果
        ├── Lambda #2: セグメントC, D処理 → 中間集計結果
        ├── Lambda #3: セグメントE, F処理 → 中間集計結果
        └── ...
        |
[Reduceワーカー] → 中間結果をマージ → 最終結果
```

### 8.2 テイルレイテンシー対策（ヘッジドリクエスト）

```
通常のアプローチ:
  100%完了を待つ → テイルの遅延で全体が遅れる

Retrieverのアプローチ:
  90%完了 → 残り10%を再リクエスト（元リクエストはキャンセルしない）
  先に返ってきた方の結果を採用

コスト: +10%のリクエスト増
メリット: 中央値50ms、P99 5秒以内を実現（7億件/日、150万span/秒のクラスタで）
```

## 9. スケーリング・耐久性戦略（Kafka統合）

### 9.1 Apache Kafkaを中心としたアーキテクチャ

```
[受信プロセス（ステートレス）]  → Kafkaトピック/パーティション
         ↓
[インジェストワーカー（ステートフル）]  ← 各パーティションを順次消費
  ・Kafkaオフセットをチェックポイント
  ・再起動後はそのポイントから再開
         ↓
[セグメントファイル生成] → S3アップロード
```

### 9.2 耐障害性の設計原則

| 障害シナリオ | 対策 |
|------------|------|
| 受信プロセス再起動 | Kafkaがバッファリング → データ欠損なし |
| インジェストワーカー再起動 | Kafkaオフセットから再開 → 冪等性で重複なし |
| クエリーノード障害 | 取り込みは継続 → 古いデータのクエリーのみ影響 |
| S3の一時的障害 | ローカルSSDにホットデータが存在 → 直近データはクエリー可能 |

### 9.3 スケーリング設計

| スケーリング軸 | 方法 |
|-------------|------|
| 書き込みスループット | Kafkaパーティション数を増加 → 各パーティションが独自セグメントを生成 |
| クエリー並列度 | Lambda同時実行数を増加 → 追加コストはあるがボトルネックなし |
| データ保持量 | S3に無制限 → ローカルSSD容量に依存しない |

**Retriever実績（2021年11月）**:
- 受信ワーカー: ~1,000 vCPU
- Kafkaブローカー: 100 vCPU
- Retrieverワーカー: ~1,000 vCPU
- Lambda同時実行: 最大30,000
- データ総量: 700TB（5億セグメント・2ヶ月分）
- 取り込みレート: 150万span/秒（クエリー可能まで数ms）

## 10. 高カーディナリティへの対応

### 10.1 グループ数が爆発した場合の対処

```
問題: user_idでGROUP BYすると数十万グループが発生
       → メモリ不足 / レスポンス不可能なデータ量

対処オプション:
  1. Reduceワーカーをファンアウト（Chordハッシングでキー空間を分割）
  2. ORDER BY / LIMIT 条件を満たす可能性が高いグループを推定
  3. 利用可能メモリ合計でグループ数を上限設定
  4. 過剰グループを返すクエリーを中断（ユーザーに通知）
```

### 10.2 カーディナリティ限界の目安

| グループ数 | クエリー可否 | 対応 |
|-----------|-----------|------|
| ~1,000 | ✓ 問題なし | - |
| ~10,000 | △ 注意 | Reduceファンアウト推奨 |
| ~100,000以上 | ✗ 困難 | グラフ上2,000px → 10万本の線は無意味 |

## 11. 自前構築 vs 既存ソリューション選択

### 11.1 既存オープンソースの選択肢

| ソリューション | 特性 | オブザーバビリティ適合度 |
|-------------|------|----------------------|
| **ClickHouse** | 列指向、高速クエリー | ✓✓ 推奨（自動シャーディング要カスタム） |
| **Apache Druid** | 列指向、リアルタイム | ✓✓ 推奨（同上） |
| **BigQuery** | フルマネージド列指向 | ✓✓ 運用テストは少ない |
| **SigNoz** | ClickHouseベース・OTel対応 | ✓✓ オブザーバビリティ特化 |
| **Elasticsearch** | 全文検索・行指向 | △ カーディナリティ制限あり |
| **Cassandra** | 行指向・高スループット | △ 任意ディメンション分析は困難 |

### 11.2 自前構築時の注意点

```
正しいデータ抽象化を選ぶことが最重要。
問題が難しすぎる → 間違ったデータ抽象化を使っているサイン。

推奨アーキテクチャ:
  時間分割 + 分散 + ハイブリッド列指向ストア
  = スピード × コスト × 信頼性 の要件を同時に満たす唯一の現実的アプローチ
```

### 11.3 選択判断フロー

```
オブザーバビリティデータストアを選ぶ
          |
          ├─ 自社規模が小さい（シングルノードで足りる）
          |    → SQLiteやPostgreSQLでも短期は可能
          |
          ├─ オープンソースで構築したい
          |    → ClickHouse or Druid + SigNoz参考アーキテクチャ
          |
          ├─ マネージドサービスを使いたい
          |    → Honeycomb, Grafana Cloud, Datadog
          |
          └─ 自前構築する
               → Kafka（耐久性） + 列指向ファイル（セグメント）
                 + S3（階層化） + Lambda（並列クエリー）
```
