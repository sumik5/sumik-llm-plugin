# オブザーバビリティによるデバッグ手法

オブザーバビリティツールを使ったデバッグの実践ガイド。コア分析ループ・BubbleUp・AIOpsの限界・人間と機械の役割分担を体系化する。

---

## デバッグ手法の2分類

### 既知の状態からのデバッグ（従来手法）の限界

| 問題 | 説明 |
|------|------|
| **ダッシュボード軍拡競争** | 新たな問題のたびにダッシュボードを追加 → 数十枚をめくるツールホッピング状態に陥る |
| **事前定義の制約** | 調査前に条件を宣言しなければならない → 調査中の新発見を反映できない |
| **カーディナリティ制限** | 高カーディナリティデータをダッシュボードに読み込めない（UUID, user_id等） |
| **直感依存** | シニアエンジニアの「魔法」に頼る → スケールしない、引き継ぎ不可 |
| **手順書の陳腐化** | 同じ方法で2度故障することはほとんどない → 手順書はすぐに無効になる |
| **確証バイアス** | 仮説を立てた後にその証拠だけを探し始めるリスク |

> **罠**: `tail -f | grep` や無限ダッシュボードのビジュアルパターンマッチは、オブザーバビリティデータに対して行っても「既知の状態からのデバッグ」にすぎない。データを持つだけでなく、**アプローチ**を変えることが本質。

---

### 第一原理からのデバッグ（オブザーバビリティ手法）

**第一原理** = 他の仮定から演繹されない基本的な仮定。何も決めつけず、観察から検証する科学的手法。

| 特性 | 詳細 |
|------|------|
| **事前知識不要** | システムアーキテクチャへの深い精通がなくても調査できる |
| **体系的・再現可能** | 直感に依存せず、誰でも同じプロセスをたどれる |
| **未知の問題に対応** | 過去に見たことがない障害モードを発見できる |
| **民主化効果** | 好奇心旺盛で本番コードを熱心に見る人が最も活躍できる |

## コア分析ループ

### 4ステージの実行フロー

```
[トリガー] アラート受信 / 顧客クレーム / 異常検知
      ↓
1. 全体像の把握
   「何が起きているか」の概況を掴む
   ─────────────────────────────────
2. 真実の検証
   パフォーマンスの顕著な変化を確認
   データ可視化でカーブの変化を特定
   ─────────────────────────────────
3. 要素の探索（メインループ）
   a) サンプル行を調べて異常値の手がかりを探す
   b) さまざまなディメンション（status_code, region等）で group by
   c) フィルタリングで異常値をより明確に絞り込む
   ─────────────────────────────────
4. 仮説の確認
   「発生していることが理解できたか？」
   → Yes: 完了
   → No: ビューをフィルタリングして 3 に戻る
```

### 分析における切り口の優先順位

| 優先度 | ディメンション例 | 理由 |
|--------|----------------|------|
| 高 | `availability_zone`, `region`, `instance_type` | インフラ起因の問題を迅速に特定 |
| 高 | `user_id`, `tenant_id`, `team_id` | 特定顧客への影響範囲を特定 |
| 高 | `request.endpoint`, `handler_route` | コード起因の問題の絞り込み |
| 中 | `build_id`, `deploy_version` | リリース起因の問題の特定 |
| 中 | `status_code`, `error_type` | エラーの種類による分類 |
| 低 | `user_agent`, `accept_language` | クライアント起因の確認 |

## BubbleUp: コア分析ループの自動化

### 動作原理

```
1. ヒートマップで「気になる異常領域」をポイント&クリックで選択
         ↓
2. BubbleUp が全ディメンションの値を2つの領域で計算
   ┌──────────────────┬────────────────────────┐
   │ 選択領域（異常）  │ ベースライン（通常）     │
   └──────────────────┴────────────────────────┘
         ↓
3. 差分を計算してパーセンテージ差でソート
         ↓
4. 最も差が大きいディメンションを上位表示
```

### 実例: AZ障害の発見

```
調査対象: レイテンシースパイクのデバッグ

BubbleUp 結果（差分降順）:
┌─────────────────────────────┬──────────┬──────────┐
│ ディメンション                │ 異常領域  │ ベース    │
├─────────────────────────────┼──────────┼──────────┤
│ global.availability_zone    │          │          │
│   = us-east-1a              │  98%     │  17%     │ ← 最大差分
├─────────────────────────────┼──────────┼──────────┤
│ instance_type = m5.xlarge   │  87%     │  32%     │
├─────────────────────────────┼──────────┼──────────┤
│ request.endpoint = /batch   │  100%    │  20%     │
└─────────────────────────────┴──────────┴──────────┘

→ クラウドプロバイダーの AZ 全体のネットワーク問題と即座に特定
```

### BubbleUp が機能する前提条件

- **任意の幅の構造化イベント** が必要（メトリクスでは不可能）
- 高いカーディナリティデータが計装に含まれていること
- 手動コア分析ループの代替として機能するが、データ品質に依存する

## AIOps の限界と批判的評価

### 異常検知が失敗するパターン

```
問題: BubbleUP の「ボックスをどこに描くか」を AI が自動決定する困難さ

[安定したシステム]
 ベースライン ─────────────────────────── 異常が明確に検出可能
              ┌───────────────┐
              │               │ ← ボックスサイズが適切
              └───────────────┘

[革新的な環境（頻繁な変更がある現代のシステム）]
 ベースライン ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
                  ↑        ↑       ↑
              新機能 バグ修正  最適化
         これらはすべて「異常」だが「正常な変化」

結果1: ボックスが小さすぎる → 多くの正常な変化を異常と誤検知
結果2: ボックスが大きすぎる → 本当の異常を正常と見逃す
```

### AIOps が失敗する環境の特徴

| 条件 | 影響 |
|------|------|
| 頻繁なデプロイ（CD実践チーム） | ベースラインが常に変化し、モデル訓練が追いつかない |
| 新機能リリース | パフォーマンスカーブの変化が「異常」だが意図的 |
| バグ修正 | 修正後の改善が「異常」として検出される |
| 最適化導入 | パフォーマンス向上が「異常」と誤認される |
| マイクロサービス環境 | サービス間の相互依存でベースラインが不定 |

> **結論**: AIOps は明確に識別できるパターンが安定して存在する場合にのみ有効。変化の激しい現代のソフトウェア開発環境では根本的な限界がある。

## 人間と機械の役割分担

### 最適な協働モデル

| 担当 | 得意なこと | コア分析ループでの役割 |
|------|----------|---------------------|
| **コンピューター** | 膨大なデータ処理・全ディメンションの差分計算・スパイク検出 | 総当り分析の自動化（BubbleUp） |
| **人間** | 文脈的判断・価値評価・問題の意味づけ | パターンへの認知的文脈付加、重要シグナルの選別 |

**鉄則**: スパイクを検出するのは機械。そのスパイクが「良いか悪いか、意図的か否か」を判断するのは人間。今日の技術でAIOpsにその価値判断はできない。

### 実践的な分業フロー

```
異常発生
  ↓
[機械] BubbleUp が全ディメンションを計算・ソートして候補を提示
  ↓
[人間] 上位候補を確認し文脈を付加
       「今日の朝 us-east-1a に何かあったか？」
       「さっきのデプロイで何が変わった？」
  ↓
[人間+機械] 仮説を立ててフィルタリングで絞り込む
  ↓
[人間] 根本原因の特定・対応・関係者への連絡
```

## 仮説駆動型デバッグの民主化効果

### 従来手法 vs オブザーバビリティ手法の人材要件

| 要因 | 既知の状態からのデバッグ | 第一原理からのデバッグ |
|------|----------------------|---------------------|
| **有利なスキル** | システムへの深い精通・長年の経験 | 好奇心・体系的な調査能力 |
| **スケーラビリティ** | シニアエンジニアの属人化 | ジュニアエンジニアも独立して調査可能 |
| **引き継ぎコスト** | 高い（暗黙知の移転が必要） | 低い（プロセスが再現可能） |
| **オンボーディング** | 長期間のシステム学習が必要 | ツールの使い方を習得すれば開始可能 |

### 民主化を最大化する計装設計

- **高カーディナリティフィールドを積極的に含める**: user_id, request_id, trace_id
- **コンテキストを豊富に**: build_id, feature_flag, region, instance_type
- **計装がドキュメントになる**: 何をディメンションとして選択したかが意図を示す
- **300〜400フィールドが成熟した計装の目安**: 広いイベントが「針を見つける」力を与える

## デバッグ判断チェックリスト

### インシデント発生時の初動判断

```
□ アラートの全体像を把握した（何が遅い/壊れているか）
□ パフォーマンスの変化をデータで確認した（直感だけでない）
□ 最初に疑う仮説を1つ決めた（先入観を自覚している）
□ ディメンション切り取りで仮説を検証した
□ BubbleUpまたは同等の手法でパターンを自動検出した
□ 人間の文脈判断（デプロイ・インフラ変更等）を付加した
□ 特定した根本原因に証拠があるか確認した
```

### ダッシュボード依存度の自己診断

| 質問 | Yes → | No → |
|------|-------|-------|
| 問題発生時、最初にダッシュボードを見るか？ | 注意: 既知の状態依存の可能性 | 良い傾向 |
| 調査中に「あのディメンションを事前に追加すべきだった」と思うか？ | ダッシュボード制約の典型的症状 | 計装が十分 |
| ジュニアエンジニアが独立してデバッグできるか？ | 民主化できている | 属人化の改善余地あり |
| 見たことがない種類の障害を自力で調査できるか？ | オブザーバビリティ実践できている | 未知の未知への対応要検討 |
