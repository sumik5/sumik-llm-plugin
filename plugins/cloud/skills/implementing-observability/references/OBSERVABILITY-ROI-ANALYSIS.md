# オブザーバビリティ ROI分析・組織展開リファレンス

「オブザーバビリティ・エンジニアリング」（Charity Majors・Liz Fong-Jones・George Miranda著）
Ch.3, 15, 19, 20 の知見をまとめた実践リファレンス。
構築か購入か・投資対効果・組織展開の意思決定を支援する。

---

## 1. ROI分析の基本フレームワーク

### 1.1 総所有コスト（TCO）を正確に見積もる原則

オブザーバビリティのROIを分析する際に陥りがちな罠は、**目に見えるコストだけを比較する**ことだ。
正確なTCO計算には、以下の2段階が必要:

```
Step 1: 目に見えるコストを計算する
  - 構築: 初期開発工数・インフラコスト
  - 購入: ベンダー請求書の金額

Step 2: 隠れたコストを明確化する
  - 構築: 採用コスト・人件費・メンテナンス・機会費用
  - 購入: 価格モデルの将来的拡大・ベンダーロックイン
```

> **重要な教訓**: オープンソースは「子犬のようにフリーだが、ビールのようにフリーではない」。
> 無料に見えるソリューションの本当のコストは、ほとんどの場合、見積もりの2倍以上になる。

## 2. Build（自社構築）の分析

### 2.1 自社構築の隠れたコスト

| コスト項目 | 内容 | 見落とされやすい理由 |
|-----------|------|------------------|
| **エンジニア人件費** | 年間25〜30万ドル/人（米国水準） | 既存コストに埋め込まれている |
| **採用コスト** | 年収の15〜25%（1人あたり約7万5千ドル） | 一時的費用として計上されない |
| **機会費用** | コアビジネス価値を提供できない時間 | 定量化が難しく見過ごされやすい |
| **継続的メンテナンス** | サードパーティ更新・機能追加対応 | 初期構築後に著しく過小評価される |
| **組織トレーニング** | 社内ツール習熟・ナレッジ移転 | 「無料の時間」扱いされがち |

**実際の事例（ELKスタック）:**

あるエンジニアリングマネージャーは商用ソリューション（月8万ドル）を「高い」と拒否した。
しかし彼の「無料のELKスタック」の実際のコスト:

```
ハードウェア費用:       月 8万ドル
エンジニア人件費(3名): 年 75〜90万ドル
採用コスト:            年 22.5万ドル
               合計: 年200万ドル以上
```

商用オプション（年96万ドル）の**2倍以上のコスト**が、見えにくい形で発生していた。

### 2.2 自社構築の利点

| 利点 | 詳細 |
|------|------|
| **予算的柔軟性** | 新規外部予算承認より社内工数確保の方が組織的に簡単な場合がある |
| **深い専門知識** | 組織固有のニーズを完全に理解し、カスタムソリューションを開発できる |
| **カスタム統合** | 独自のビジネスルール・ワークフローに深く統合できる |
| **競争優位性** | 組織固有のプラクティスと文化に根ざしたシステムを構築できる |

### 2.3 自社構築のリスク

- **プロダクトマネジメントの専門性不足**: 社内ユーザーのニーズに応える製品を作れるかどうか
- **採用遅延**: 機能する商用ソリューションをすぐに使う代わりに、数ヶ月待つことになる
- **開発の停滞**: 最初のバージョンを構築後に次のプロジェクトへ移行し、内部ツールが陳腐化するパターン
- **組織的採用の失敗**: UIや体験が不十分で、開発チームにしか使われないリスク

## 3. Buy（商用ソフトウェア購入）の分析

### 3.1 商用ソフトウェアの隠れた金銭的コスト

価格モデル別のリスク評価:

| 価格モデル | 予測可能性 | 罠 |
|-----------|-----------|-----|
| **ユーザー数課金** | 中 | チーム拡大と連動して増加 |
| **ホスト数課金** | 中 | インスタンス変更で予測が難しい |
| **データ量課金** | 低 | オブザーバビリティが普及するほど指数関数的に増加 |
| **クエリー数課金** | 低 | 探索的利用・組織全体展開を阻害する |
| **固定シート制** | 高 | スケールには制約が出る |

> **教訓**: 好奇心にペナルティを与える価格モデルのツールは避けよ。
> 組織内の多くの人がデータを探索できることがオブザーバビリティの本質的価値だ。

### 3.2 商用ソフトウェアの隠れた非金銭的コスト

**ベンダーロックインのリスク:**

- 独自エージェントや計装ライブラリで構築すると、移行時に初期投資をやり直す必要がある
- **対策**: OpenTelemetry（OTel）でネイティブに計装し、ベンダー設定はディストリビューションに分離する
  - 別ベンダーへの移行時は設定のみ変更すればよい

### 3.3 商用ソフトウェアのメリット

| メリット | 内容 |
|---------|------|
| **Time to Value** | 数分〜数時間で使い始められる |
| **UX品質** | 特定のワークフローに最適化された製品体験 |
| **メンテナンス委託** | ベンダーがアップデート・スケール対応を担う |
| **専門知識の獲得** | 問題解決を専門とするパートナーの知見を活用できる |

### 3.4 商用ソフトウェアのリスク

- **社内専門性の欠如**: ベンダー依存で組織内のオブザーバビリティ深度が浅くなる
- **ロードマップ依存**: 自社固有の機能要求がベンダーに優先されないリスク
- **汎用性のトレードオフ**: 多様なユーザー向けに設計されており、特定のニーズへの深い最適化は難しい

## 4. ハイブリッドアプローチ（推奨）

### 4.1 「Build かつ Buy」戦略

Build vs. Buyは**誤った二者択一**だ。最善の選択は「購入しつつ構築する」ことが多い。

```
推奨モデル:
  ┌─────────────────────────────────────┐
  │  商用ベンダーソリューション（購入）        │
  │  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━  │
  │  オブザーバビリティチーム（構築）          │
  │  ・計装ライブラリ・抽象化機能            │
  │  ・命名規約の標準化                    │
  │  ・各チームへの導入支援               │
  │  ・ベンダーAPIによるカスタム統合         │
  └─────────────────────────────────────┘
```

### 4.2 ハイブリッドが機能する条件

オブザーバビリティチームの役割:

- **車輪の再発明をしない**: 商用ツールをベースに使う
- **「最後のマイル」を構築する**: 組織固有のワークフロー・統合ポイントのみ自社開発
- **APIフレンドリーなツールを選ぶ**: データを自由に取得・カスタマイズできる製品が前提

### 4.3 ハイパフォーマーとローパフォーマーの違い

| | ハイパフォーマー | ローパフォーマー |
|--|-----------------|-----------------|
| ツール姿勢 | 最高のツールを使ってエンジニア効率を最大化 | 平凡なツールを惰性で使い続ける |
| カスタム開発 | コアビジネスの妨げになった場合のみ | 問題をすべて自分たちで解決しようとする |
| 意思決定基準 | ビジネス価値への集中度 | コスト削減の可視性 |

## 5. 導入タイミングの戦略

### 5.1 リアクティブ導入（問題後の対応）

**オブザーバビリティ不足の典型的な症状:**

- 重大バグを社内より顧客が先に発見・報告する
- マイナーなインシデントが長期サービス停止に発展することが頻繁にある
- 壊れたものを直す運用作業が新機能開発を上回る
- サポートがパフォーマンス問題を再現・解決できず、顧客満足度が低下
- 予測できない作業が積み上がり、新機能リリースが数週間〜数ヶ月遅延

これらの症状があれば、リアクティブでもビジネスケースを構築する根拠がある。

### 5.2 プロアクティブ導入（能動的な改善）

**ROIの4本柱:**

| 柱 | 内容 | 測定指標 |
|----|------|---------|
| **収益増加** | 稼働時間・パフォーマンス向上、コード品質向上による直接収益増 | 稼働率、エラー率、コンバージョン率 |
| **インシデント対応コスト削減** | MTTD・MTTR短縮、クエリー応答時間改善 | 平均検出時間、平均解決時間 |
| **インシデント回避** | 重大化する前に問題を発見・防止 | インシデント件数、回避率 |
| **従業員離職率低下** | 燃え尽き・オンコール疲労の軽減 | エンジニア満足度、離職率 |

> Forrester Research TEIフレームワークによる調査でも、これら4つの利点が定量的に確認されている。

### 5.3 プロアクティブ導入の進め方

```
1. 現状ベースラインの計測
   └── TTD（検出時間）・TTR（解決時間）を記録

2. パイロットチームで導入開始
   └── 明確なスコープ（1チームまたは1ビジネスライン）で

3. 早期の成果を証明
   └── TTDとTTRの改善を数値で示す

4. 二次的な利点を測定・訴求
   └── オンコールストレス低下・新機能開発時間の増加

5. 組織全体への展開を推進
   └── 隣接チームを巻き込み、支持者を作る
```

## 6. 非エンジニアへの展開

### 6.1 各チームへの価値訴求

オブザーバビリティデータを「民主化」することで、組織全体の意思決定品質が向上する。

| チーム | 主なユースケース | 提供できる価値 |
|--------|---------------|--------------|
| **カスタマーサポート** | 顧客から問題報告を受ける前に内部で検知。テンプレート化されたクエリーでセルフ解決 | サポートコストの削減・顧客満足度向上 |
| **プロダクトマネジメント** | 新機能採用状況の把握・勘に頼らないデータ駆動の意思決定 | 機能開発の優先順位の精度向上 |
| **営業チーム** | デモで使用する機能の可用性確認・顧客ごとの利用パターン把握 | 売上予測精度・商談成功率の向上 |
| **カスタマーサクセス** | 解約リスクのある顧客の早期検知・成功パターンの発見 | 解約率の低下・LTV向上 |
| **経営層・役員** | 信頼性トレンドの把握・障害発生件数の推移 | ビジネスリスクの可視化 |

### 6.2 非エンジニア展開のビジネスケース例

```
典型的なユースケース:
  ✓ 新機能の採用状況把握（誰が使っているか？放棄パターンは？）
  ✓ 最新のサービスステータス（サポートが自己解決できる情報）
  ✓ 積極的な問題解決（顧客報告前にサポートチケット化を防ぐ）
  ✓ 顧客体験の可視化（どのユーザーがどの問題に遭遇しているか）
  ✓ デプロイ品質の監視（新機能リリースが既存体験を壊していないか）
```

## 7. オブザーバビリティツール vs BIツールの使い分け

### 7.1 比較マトリクス

| 特性 | オブザーバビリティツール | BIツール |
|------|----------------------|---------|
| **クエリー実行時間** | 秒単位〜数秒（探索性が命） | 分単位も許容（定期レポート向け） |
| **精度** | 99.5%でも即時の方が良い | 100%正確さが必須（課金等） |
| **最新性** | ほぼリアルタイム必須（インシデント対応） | 時間遅延は許容（戦略的意思決定） |
| **データ構造** | スキーマレス・事後推論、任意の幅 | 事前定義スキーマ、一貫した構造 |
| **時間枠** | 数秒〜数分のセッション | 数日〜数週間のジャーニー |
| **データ保持** | 数週間〜数ヶ月（刹那的） | 永続的・無制限拡大 |
| **主な用途** | コード・インフラ・ユーザー・時間の交差点の理解 | 財務・CRM・ERP・マーケティング指標 |

### 7.2 使い分けの判断フロー

```
質問: 「この問題に答えたい」
  │
  ├─ 今この瞬間に何が起きているか？
  │   └── → オブザーバビリティツール
  │
  ├─ 先月どの機能が最も使われたか？
  │   └── → BIツール（または両方の組み合わせ）
  │
  ├─ 過去2年のP95レイテンシートレンドは？
  │   └── → モニタリング（長期集計メトリクス）
  │
  └─ 個々のユーザーリクエストの内部動作は？
      └── → オブザーバビリティツール
```

### 7.3 オブザーバビリティとBIの補完的な使い方

**マクロ + ミクロの組み合わせ:**

- **BIツール**: 月次・週次レベルの大局的な傾向（例：2月に最も使われた機能）
- **オブザーバビリティ**: 個々のリクエストレベルまでドリルダウン（例：その機能が遅い特定ユーザーの調査）

> BIが「何が起きたか（集計値）」を示し、オブザーバビリティが「なぜ起きたか（個別事例）」を示す。
> 両者を組み合わせることで、仮説検証の精度が劇的に向上する。

## 8. ケーススタディ: Parse（Ch.3 スケーリングの教訓）

### 8.1 背景

- **Parse**: モバイルバックエンド・アズ・ア・サービス（MBaaS）プラットフォーム（2013年Facebook買収）
- 当初6万→最終100万以上のモバイルアプリをホスティング
- Ruby on Rails / MongoDB / マルチテナント共有プールアーキテクチャ

### 8.2 スケーリングに直面した問題

**従来のモニタリングツール（Nagios・Ganglia等）が機能しなかった場面:**

| 問題の種類 | 従来ツールの限界 |
|-----------|----------------|
| 毎日新しいトップアプリが急増する | 何が問題になるかを事前に予測できない |
| マルチテナント共有環境で1テナントが全体に影響 | 集計メトリクスでは個別テナントの影響を切り分けられない |
| 全体の信頼性99.9%でも1シャードが100%ダウン | 平均値は個別テナントの実体験を隠す |
| ユーザーが実行する任意クエリーによる障害 | ホスティングサービス特有の予測不可能性 |
| 遅いクエリーリストが症状であり原因でない | 根本原因特定のために任意の軸での分析が必要 |

### 8.3 問題の本質

**旧来のアプローチが失敗した根本理由:**

```
従来の前提（モノリス時代）:
  ✓ 問題は予測可能なパターンを持つ
  ✓ 熟練者が過去の問題をパターンマッチングできる
  ✓ ダッシュボードとしきい値で問題を発見できる

現実（マイクロサービス / マルチテナント時代）:
  ✗ 毎日が「初めて見る問題」
  ✗ シニアエンジニアの直感・知識が追いつかない
  ✗ ヒーロー文化が組織のスケーラビリティを破壊する
```

### 8.4 転換点：Scubaとの出会い

Facebook買収後に出会ったScuba（Facebookのリアルタイムデータベースシステム）が変革をもたらした:

| 変化 | Before（従来のモニタリング） | After（高カーディナリティ分析） |
|------|--------------------------|------------------------------|
| 問題解決時間 | 何日も、場合によっては永遠 | 数分〜数秒 |
| 必要な事前知識 | 問題を事前に知っていることが前提 | 症状から「パンくず」を辿れる |
| 最適なデバッガー | 最も長く在籍する人 | 最も好奇心が強く粘り強い人 |
| 知識の所在 | 個人の頭の中（属人的） | 共有ツール・データ（民主的） |
| ヒーロー依存 | 特定エンジニアに依存 | 方法論とデータに依存 |

### 8.5 得られた教訓

**Parse事例から導けるオブザーバビリティ導入の判断基準:**

- [ ] 本番環境で「初めて見る問題」が増加していないか
- [ ] デバッグが「熟練者の直感」に頼っていないか
- [ ] マイクロサービスやマルチテナントアーキテクチャへの移行が進んでいるか
- [ ] 従来のダッシュボード・アラートが「症状の表示」にとどまり「原因特定」ができていないか
- [ ] オンコール対応が特定のシニアエンジニアへの依存になっていないか

**いずれか1つでも当てはまれば、オブザーバビリティへの投資を優先検討すべきタイミングだ。**

## 9. 導入成熟度チェックリスト

### 9.1 ツール選定前の確認事項

- [ ] Build vs Buy のTCOを隠れたコスト込みで比較したか
- [ ] 価格モデルが「好奇心にペナルティを与えない」か（クエリー数課金は要注意）
- [ ] OpenTelemetry対応でベンダーロックインを回避できるか
- [ ] APIが充実しており、ハイブリッドアプローチが可能か
- [ ] データ保持・分析が同一ソリューションにバンドルされているか（断片化回避）

### 9.2 組織展開の確認事項

- [ ] TTD・TTRのベースラインを計測しているか
- [ ] パイロットチームで成果を測定・証明したか
- [ ] 非エンジニアチームへの展開ユースケースを特定したか
- [ ] オブザーバビリティチームの役割を定義したか（Build and Buy の統合担当）
- [ ] 心理的安全性・非難のない文化の基盤があるか

### 9.3 「十分なオブザーバビリティ」の判断基準

以下が達成できていれば、オブザーバビリティが実践として定着している:

- **文化的指標**: コードレビューで計装が自然に議論される
- **プロセス指標**: デプロイ後に本番挙動を確認することが当然になっている
- **成果指標**: 未解決の「ミステリー」インシデントの割合が減少している
- **組織指標**: 非エンジニアがセルフサービスで本番データをクエリーしている
- **品質指標**: 「このデータで答えられない」場合にテレメトリーのギャップを埋めることが優先される
