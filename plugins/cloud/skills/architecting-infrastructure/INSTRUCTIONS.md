# パターンベース設計手順

## 1. パターンベース設計の手順

非機能要件をインフラ設計に落とし込む際、以下の4ステップで体系的に進める。

```
❶要件を確認する
    ↓
❷設計方式を抽出する
    ↓
❸インフラ構成を検討する
    ↓
❹インフラ構成を確認する
```

### ❶ 要件を確認する

非機能要件を確認し、同時に制約条件も整理する。

**確認すべき内容:**

| 観点 | 内容 |
|------|------|
| 非機能要件 | 可用性・セキュリティ・性能・拡張性・運用性の各目標値 |
| コスト制約 | ハードウェア投資予算・運用コスト上限 |
| 技術的制約 | 既存システムとの接続要件・技術標準 |
| データセンター | ロケーション・建物設備の制約 |
| コンプライアンス | 法規制・業界標準への準拠要件 |
| 納期 | 設計・構築・テストの工程スケジュール |

> **AskUserQuestion**: 非機能要件の優先度が不明な場合は、可用性・セキュリティ・性能のどれを最優先とするかユーザーに確認する（[セクション4参照](#4-ユーザー確認の原則askuserquestion)）。

### ❷ 設計方式を抽出する

確認した要件に適合する設計パターンを選択する。

**選択のポイント:**
- パターンの選択基準テーブルで、要件を満たすパターンを絞り込む
- 選択したパターンが制約条件（コスト・技術・納期）に対応しているか確認する
- パターンの特徴・留意点を理解してから選択する
- 複数のパターンから選びきれない場合は、選択基準の項目について要件をさらに深掘りする

**追跡記録の重要性:**

選択した設計方式と、その根拠となった要件・制約条件の関係を記録しておく。
後工程での要件変更時に変更箇所を容易に特定できる。

> **AskUserQuestion**: クラウド環境かオンプレミスかによりパターン選択が大きく変わる。対象環境が不明な場合は確認する（[セクション4参照](#4-ユーザー確認の原則askuserquestion)）。

**詳細パターン一覧:**

| カテゴリ | 詳細 |
|---------|------|
| 可用性パターン（35種） | [references/AVAILABILITY.md](references/AVAILABILITY.md) |
| セキュリティパターン（18種） | [references/SECURITY.md](references/SECURITY.md) |
| 性能・拡張性パターン（14種） | [references/PERFORMANCE-SCALABILITY.md](references/PERFORMANCE-SCALABILITY.md) |
| 運用・保守性パターン（16種） | [references/OPERATIONS.md](references/OPERATIONS.md) |
| インフラ構成パターン（25種） | [references/INFRASTRUCTURE-CONFIG.md](references/INFRASTRUCTURE-CONFIG.md) |
| クラウドパターン（19種） | [references/CLOUD.md](references/CLOUD.md) |

### ❸ インフラ構成を検討する

選択した設計方式を組み合わせてインフラ構成図を作成する。

**検討のポイント:**
- パターンの組み合わせによっては干渉が起きる場合がある → 必ず各パターンの「留意点」を確認する
- 複数パターンの組み合わせ時は、障害時の動作も含めてシミュレーションする
- コスト制約に対して構成が過剰・不足になっていないか確認する

### ❹ インフラ構成を確認する

検討したインフラ構成について関係者とレビューする。

**確認の視点:**

| 確認項目 | 内容 |
|---------|------|
| 制約条件の妥当性 | 考慮した制約条件は正しく反映されているか |
| 要件の完全性 | 要件と制約条件がすべて反映されているか |
| 戦略整合性 | 経営戦略・情報戦略と整合が取れているか |
| 追跡可能性 | パターン選択の根拠となる要件を容易に見つけ出せるか |
| 論理的妥当性 | その方式が最善である理由を論理的に説明できるか |
| 対策の網羅性 | 障害対策・バックアップ機能を確認したか |
| リスク対策 | リスク要因の抽出と対策の検討は十分か |

確認の結果、変更や追加検討が必要であれば、手順をさかのぼって再検討する。

## 2. リスク要因の確認手順

パターンの組み合わせや採用する設計方式にはリスクが伴う場合がある。
以下の3ステップで検討漏れを防ぐ。

```
❶リスク要因を抽出する
    ↓
❷リスクを整理する
    ↓
❸リスク対策を検討する
```

### ❶ リスク要因を抽出する

採用する設計方式のデメリット・留意事項を洗い出す。

**抽出方法:**
- パターンの比較表で △/× 評価の項目を確認する
- パターンの選定基準で基準を満たしていない項目を確認する
- 経験者や有識者に意見を求めて抽出漏れを防ぐ
- 少しでも関係する可能性がある事項はすべて抽出する（過度な絞り込みは避ける）

### ❷ リスクを整理する

抽出したリスクを「設計時に検討すべきリスク」と「後工程で注意すべき事項」に分類する。

**設計時に検討すべきリスクの例:**

| リスク種別 | 内容 |
|---------|------|
| 要件不適合リスク | パターン組み合わせにより要件を満たさなくなる可能性 |
| 実現性リスク | 特定条件下で実施が困難になる可能性 |
| コストリスク | 製品が高額・構築/運用コストが増大する可能性 |
| 効率性リスク | 構築・運用作業の効率が低下する可能性 |

**後工程で注意すべき事項の例:**
- 製品選定作業での注意点
- 構築作業での注意点
- 稼働後の運用作業での注意点

### ❸ リスク対策を検討する

特定したリスクに対して回避・軽減策を検討する。

**対策の手法:**
- 設計方式の組み合わせを変更してリスクを回避する
- 代替パターンを採用して影響を軽減する
- リスクを受容したうえで運用手順で対応する（残留リスクとして明示）
- 経験者・有識者の意見を求めてより有効な対策を導く

## 3. 要件変更への対応手順

環境変化に伴う要件変更が発生した場合、以下の3ステップで対応する。

```
❶要件とインフラ構成要素との関係を確認する
    ↓
❷構成を部品化する
    ↓
❸変更内容を反映する
```

### ❶ 要件とインフラ構成要素との関係を確認する

変更対象のインフラ構成要素と要件の対応関係を明確にする。

**関係が不明確な場合の整理手順:**
1. インフラ構成図と設計方式の図を見比べ、変更対象に対応するパターンを特定する
2. 特定したパターンの選定基準・比較表から要件を逆引きする
3. 要件定義書との照合を行い、対応するシステム要件を特定する
4. 要件定義書に記載がない場合は、当時の事業環境から要件を推測して追加する（元の要件定義書との区別を明記すること）

### ❷ 構成を部品化する

インフラ構成要素を関係性の強いものでまとめ「部品」として扱う。

**部品化の例:**

| 部品名 | 含む構成要素の例 |
|-------|----------------|
| Webサーバ | フロントエンドサーバ・ロードバランサ |
| APサーバ | アプリケーションサーバ・ミドルウェア |
| DBサーバ | データベースサーバ・ストレージ |
| バックアップ | バックアップ装置・バックアップソフト |
| システム監視 | 監視サーバ・監視エージェント |
| ジョブ管理 | ジョブ管理サーバ・スケジューラ |
| ネットワーク | ルータ・スイッチ・ファイアウォール |

部品化することで、要件変更時の影響範囲を特定しやすくなる。

### ❸ 変更内容を反映する

変更要件に関係する部品を特定し、新しい要件に応じたパターンに変更する。

**変更後の確認:**
- 変更した部品が新要件を満たしているか確認する
- インフラ構成全体への影響を確認する（他の部品・パターンとの整合性）
- 必要に応じてリスク要因の確認手順（セクション2）を再実施する

## 4. ユーザー確認の原則（AskUserQuestion）

**判断分岐がある場合、推測で進めず必ずAskUserQuestionツールでユーザーに確認する。**

### 確認すべき場面（4箇所）

| # | 場面 | 確認内容 |
|---|------|---------|
| 1 | **非機能要件の優先度** | 可用性・セキュリティ・性能・運用性のうち最優先とすべき要件はどれか |
| 2 | **コスト制約** | 高可用性パターン（コスト高）vs. コスト効率パターン（可用性を一部妥協）のどちらを優先するか |
| 3 | **クラウド/オンプレミス** | 対象インフラはクラウド・オンプレミス・ハイブリッドのいずれか（パターン選択に直結） |
| 4 | **DR要件レベル** | 広域クラスタ（完全冗長化）vs. バックアップ転送（一定のRPO/RTO許容）どちらのDRレベルを求めるか |

### AskUserQuestion コード例

```python
# 例: 非機能要件の優先度確認
AskUserQuestion(
    questions=[
        {
            "question": "最優先とする非機能要件を選択してください。",
            "header": "非機能要件の優先度",
            "options": [
                {
                    "label": "可用性優先",
                    "description": "SPOF排除・HA構成・DR対策を最重視。コストは上昇する可能性あり"
                },
                {
                    "label": "セキュリティ優先",
                    "description": "認証強化・暗号化・ネットワーク分離を最重視"
                },
                {
                    "label": "性能・拡張性優先",
                    "description": "スケールアウト・サイジング精度・過負荷対策を最重視"
                },
                {
                    "label": "運用性優先",
                    "description": "監視・ジョブ管理・バックアップの自動化・効率化を最重視"
                }
            ],
            "multiSelect": False
        }
    ]
)
```

### 確認不要な場面

- セキュリティ必須対策（通信の暗号化・ファイアウォール設置など）
- SPOF排除の基本（単一障害点の冗長化）
- バックアップの基本設計（物理障害と論理障害の両方への対応）

## 5. 非機能要求の基本

各カテゴリの基本概念を理解したうえで、対応するパターンリファレンスを参照する。

---

### 5.1 可用性の基本

**詳細パターン:** [references/AVAILABILITY.md](references/AVAILABILITY.md)

システムの稼働率を高めるため、「故障の防止」「迅速な復旧」「継続的改善」を組み合わせる。
費用対効果を考慮し、低コストで効果的な対策から優先的に実施することが基本方針。

#### 故障の防止

| アプローチ | 概要 |
|-----------|------|
| SPOF排除 | サーバ・ディスク・ネットワークの冗長化で単一障害点を除去 |
| 品質の作り込み | 設計工程で品質目標を設定し、テスト項目に反映 |
| フェイルセーフ | 想定外の故障時も致命的状態に陥らない安全弁を設計（例: DBレコード最大取得件数制限） |
| 実績ある製品の利用 | 多くのユーザーに使用され致命的問題が修正された安定バージョンを採用 |
| 単純化 | システムの複雑さを減らし、オペレーションミスを誘引する要素を排除 |
| 運用・設定の自動化 | 人手の介在を減らして設定ミス・オペレーションミスを防ぐ |

#### 迅速な復旧

| アプローチ | 概要 |
|-----------|------|
| 運用体制の整備 | 問題発生時の連絡体制・処理体制・エスカレーション先を事前整備 |
| 運用手順の整備 | 障害対応の優先度・目標復旧時間を事前定義 |
| BCP/BCM | 事業継続計画と継続的管理プロセスの策定 |
| バックアップ設計 | MTTR短縮のためのバックアップ取得・リストア設計 |
| 監視項目の適正化 | 必要十分な監視項目と監視対象を設定 |
| エラーログ設計 | 障害発生箇所・影響範囲の特定に役立つログの設計 |

> 可用性レベルの選定（高可用性 vs コスト効率）は **AskUserQuestion** で確認（[セクション4](#4-ユーザー確認の原則askuserquestion)）。

### 5.2 セキュリティの基本

**詳細パターン:** [references/SECURITY.md](references/SECURITY.md)

「予防」「検知」「回復」の3つの観点でセキュリティ対策を設計する。
セキュリティ技術は日々進歩するため、定期的な見直しが必要。

#### 攻撃手法の理解

対策を検討する前に、主要な攻撃手法カテゴリを把握する:

| 標的 | 代表的な攻撃手法 |
|------|----------------|
| クライアント端末 | マルウェア感染・フィッシング |
| サーバ | 脆弱性悪用・不正アクセス |
| ネットワーク | 盗聴・DoS攻撃・中間者攻撃 |
| ユーザー（人） | ソーシャルエンジニアリング |

#### 暗号化

データの盗聴・改竄からの保護のため、公開されたアルゴリズムによる暗号化を採用する。
暗号強度はアルゴリズムの種類・鍵長・鍵管理の組み合わせで決まる。

**独自アルゴリズムは安全性が検証されていないため採用しない。**

#### 認証

| 認証対象 | 概要 |
|---------|------|
| 本人認証 | 情報認証（パスワード）・所有物認証（トークン）・バイオメトリクス認証の3要素 |
| サーバ・端末認証 | デジタル証明書によるなりすまし防止 |
| データ認証 | データの完全性確認（デジタル署名等） |

**2つ以上の認証要素を組み合わせた多要素認証で高いセキュリティレベルを実現する。**

#### アクセス制御

| 方式 | 特徴 | 採用場面 |
|------|------|---------|
| 任意アクセス制御（DAC） | リソース所有者がアクセス権を設定。柔軟だが機密性が低下しやすい | 一般的なOS・ファイルシステム |
| ロールベースアクセス制御（RBAC） | ユーザーの役割単位で権限を設定。人事異動にも対応しやすい | OS・ミドルウェアのユーザー管理 |
| 強制アクセス制御（MAC） | システムがアクセス権を決定。高い機密性を実現 | 高セキュリティ要件のシステム |

#### ネットワークセキュリティの基本

| 対策 | 目的 |
|------|------|
| ファイアウォール設置 | 「原則拒否」思想による不正アクセス防止 |
| DMZ構築 | 公開サーバ侵害時の内部ネットワークへの二次被害防止 |
| IDS/IPS導入 | ファイアウォールで検知できない不正アクセスのリアルタイム検知 |

### 5.3 性能・拡張性の基本

**詳細パターン:** [references/PERFORMANCE-SCALABILITY.md](references/PERFORMANCE-SCALABILITY.md)

性能・拡張性対策の基本はサイジング。システム全体のボトルネックを把握したうえで、適切なハードウェア構成を決定する。

#### サイジングの目的

| 問題 | 結果 |
|------|------|
| 性能不足 | 処理速度の非機能要件を満たせず、下流工程で問題発生 → 手戻りコスト |
| 過剰性能（オーバースペック） | 不要な機材投資による費用対効果の悪化 |

#### サイジングの手順（4ステップ）

| ステップ | 内容 |
|---------|------|
| ❶ 性能要件の整理 | ユーザー数・ピーク負荷傾向・レスポンスタイム目標値・バッチ完了時間など業務特性を整理 |
| ❷ 机上サイジング | 類似システム事例・ベンチマーク値をもとに必要ハードウェア構成を机上で推定。安全率を設定 |
| ❸ 実機サイジング | プロトタイプアプリで実機測定。既存システム更改の場合は本番データを活用 |
| ❹ ハードウェアの比較・選定 | 複数のベンチマーク値を比較。前提条件（業務特性・ボトルネック）を考慮して選定 |

**拡張性設計はサイジングと同時に行う。** 将来の業務量増加に備えた最小限の追加投資で対応できる設計を検討する。

> コスト制約と高可用性・高性能パターンのトレードオフは **AskUserQuestion** で確認（[セクション4](#4-ユーザー確認の原則askuserquestion)）。

### 5.4 運用・保守性の基本

**詳細パターン:** [references/OPERATIONS.md](references/OPERATIONS.md)

システム監視・ジョブ管理・バックアップ管理の3機能を中心に、開発フェーズで運用機能を作り込む。
運用機能（インフラ実装）と運用管理（運用プロセス）は相互にフィードバックしながら設計する。

#### システム監視

| 実装ポイント | 内容 |
|-------------|------|
| 監視対象と監視項目の最適化 | 闇雲に増やさず、重要度と特性に基づいてマトリクスで整理 |
| 監視メッセージの重要度定義 | 「致命的」「警告」「情報」等の重要度をあらかじめ定義 |
| 通知先と通知方法 | 重要度別の通知先（監視センター・システム管理者・顧客）と通知方法を決定 |
| フィルタリング設定 | 大量メッセージ発生時の視認性低下を防ぐ絞り込み設定 |
| 重要度の変換 | OSやミドルウェアのログ重要度を定義した重要度体系に変換 |
| テストとチューニング | 他のインフラテスト中に出力されるログを活用して監視定義をチューニング |

#### ジョブ管理

| 実装ポイント | 内容 |
|-------------|------|
| 自動化範囲の決定 | 手動作業の許容範囲・非同期処理の自動化範囲を決定 |
| ジョブ管理ソフトウェアの選定 | サーバ間処理・実行順序保証が必要な場合は専用ソフトウェアを導入 |
| ジョブの一元管理 | ジョブ管理サーバは原則1台。複数のジョブ管理機構の混在を避ける |
| ジョブ管理サーバの可用性 | システム全体制御を担う場合はHA/FT構成で高い可用性を確保 |
| ジョブ設計規約 | 命名規則・リトライ設定・ジョブネット構成など設計規約を策定 |

#### バックアップ管理

| 実装ポイント | 内容 |
|-------------|------|
| 物理・論理障害の両対応 | RAID（物理障害対応）とバックアップ（論理障害対応）を組み合わせる |
| 上流での機器構成検討 | バックアップ装置・ネットワーク・ライセンスはコスト影響が大きく上流から検討 |
| バックアップ対象別設計 | バックアップ頻度・保存期間はデータ種別ごとに設計 |
| バックアップ機能の統合 | バックアップ専用ネットワーク・共有バックアップ装置で運用性を向上 |
| バックアップ装置の選定 | テープストレージ vs. 磁気ディスクストレージの特性を考慮 |
| バックアップソフトの選定 | OS・RDBMSサポート確認。メジャーバージョンアップ直後は動作検証を実施 |

#### 運用管理の3P

| P | 内容 |
|---|------|
| People（体制・役割） | システムに関わる役割（オペレーター・管理者・エスカレーション先等）と責任範囲を明確化 |
| Process（手順） | 管理手順（組織的な動作フロー）と作業手順（個別作業の手順書）を整備 |
| Product（プロダクト） | 運用管理ツール（商用製品・OSS）の導入を費用対効果で判断 |

> DR（Disaster Recovery）要件のレベル（広域クラスタ vs. バックアップ転送）は **AskUserQuestion** で確認（[セクション4](#4-ユーザー確認の原則askuserquestion)）。
