# Zero Trust Architecture

Zero Trust（ゼロトラスト）の原則・コアコンポーネント・実践的手法を解説する。NIST SP 800-207 をベースとし、ハイブリッドクラウド環境においてアーキテクチャ設計の全フェーズへ Zero Trust 思考をどう統合するかを整理する。

← [SKILL.md へ戻る](../SKILL.md)

---

## 1. Zero Trust とは

従来の境界型セキュリティは「ネットワーク境界を越えた通信だけを精査し、内部は信頼できる」という前提に立つ。クラウドコンピューティングの普及とモバイルデバイスの多様化により、この前提は成立しなくなった。境界そのものが曖昧になった現代では、**ネットワーク位置に依存した暗黙の信頼を排除**し、あらゆる通信を潜在的に悪意あるものとして扱う考え方が必要である。

Zero Trust の核心は次の２つのスタンスに集約される。

| スタンス | 意味 |
|---------|------|
| **Never trust, always verify**（決して信頼せず、常に検証する） | リクエストの発信元が内部か外部かを問わず、アクセス要求のたびに認証・認可を行う |
| **Assume breach**（侵害を想定する） | 同一ネットワーク上の隣接システムを既に侵害されている可能性があるものとして設計する |

Zero Trust はテクノロジーの導入だけで実現できるものではなく、プロセス・実践・組織文化の変革を伴う包括的なアプローチである。また、エンドツーエンドのメソッドではなく、他のアーキテクチャ思考技法と組み合わせて初めて効果を発揮する。

## 2. 境界型セキュリティからの移行

```
従来モデル（境界型）          Zero Trust モデル
────────────────────         ──────────────────────────────
外部 ─[Firewall]─ 内部        あらゆるアクセスを検証
内部 = 暗黙の信頼              アイデンティティ = 新しい境界
ネットワーク位置で判断          コンテキスト（誰・何・どこから・いつ）で判断
静的ポリシー                   動的ポリシー（リアルタイム評価）
```

移行において最も重要なパラダイムシフトは、**境界（perimeter）ではなくアイデンティティを信頼の基点**とすることである。識別・認証・認可が行われる地点が新しい境界となり、そのすべてが継続的に評価される。

## 3. ガイディング原則と標準規格の対応

NIST SP 800-207 および UK NCSC Zero Trust Architecture Design Principles を実務的な観点で整理したものが以下の表である。

| ガイディング原則 | 内容 | NIST SP 800-207 テネット（代表） | UK NCSC 原則（代表） |
|----------------|------|-------------------------------|---------------------|
| **データ中心セキュリティ** | すべてのデータソースとコンピューティングサービスをリソースとして管理する | テネット①: すべてのデータソースとコンピューティングサービスはリソース | 原則①②: アーキテクチャ・アイデンティティの把握 |
| **Never trust, always verify** | アイデンティティ検証 + アクセス制御 + 最小権限 + マイクロセグメンテーションを組み合わせる | テネット③④⑤: セッション単位アクセス・動的ポリシー・動的認証認可 | 原則④⑤: ポリシーによる認可・あらゆる場所での認証 |
| **あらゆる場所でのデータ保護** | ネットワーク位置に関係なく通信を暗号化する | テネット②: すべての通信はネットワーク位置に依存せず保護 | 原則⑦: 自組織ネットワークも信頼しない |
| **Assume breach / 継続的監視** | 資産の完全性とセキュリティ態勢を常時監視・計測する | テネット⑤⑥: 資産の監視・計測・状態情報の収集・活用 | 原則③④: ユーザー行動・サービス・デバイスの健全性評価 |
| **Zero Trust コンポーネント選定** | Zero Trust 設計を前提とするサービスを選ぶ | （サービス選択基準は NIST では非明示） | 原則⑧: Zero Trust 設計のサービスを選択 |

> **注意**: 「Zero Trust ネットワークアーキテクチャ原則」はソリューションベンダーが製品説明に使う用語で、上記の包括的な Zero Trust 原則の一部にすぎない。混同しないこと。

## 4. コアアーキテクチャコンポーネント（NIST SP 800-207）

NIST SP 800-207 は Zero Trust の論理的なコンポーネントとその相互関係を定義している。

```
                    [コントロールプレーン]
                           │
Subject ─── PEP ──── Enterprise Resource
(アクター)   │
            PDP
          ┌──┴──┐
          PE    PA
         (ポリシーエンジン) (ポリシー管理者)
              ↑
         [データソース群]
         CDM / SIEM / IAM / PKI / ...
```

| コンポーネント | 役割 |
|--------------|------|
| **Subject（サブジェクト）** | アクセスを要求する人間・システム・機能コンポーネント |
| **Enterprise resource（エンタープライズリソース）** | 組織が管理するデータ・デバイス・機能コンポーネント |
| **PEP（Policy Enforcement Point）** | サブジェクトとリソースの間に位置するインラインアクセス制御プロキシ。アクセス要求をすべて PDP へ転送し、PDP の判断を強制する |
| **PDP（Policy Decision Point）** | PE と Policy Administrator で構成。PE は「誰が・どのリソースに・どの条件でアクセスできるか」を定義したポリシーと、コンテキストデータを用いてアクセスを決定する |
| **Policy Engine（PE）** | コンテキストデータ（アイデンティティ・デバイス状態・ネットワーク経路等）と静的ポリシーを組み合わせてアクセス可否を決定するエンジン |
| **Policy Administrator（PA）** | PEP との通信管理を担う。PEP が複数存在する場合の調整を行う |
| **Control plane / Data plane** | PEP–PDP 間の通信はコントロールプレーン（制限されたパス）を使い、データプレーン（アプリケーションデータの経路）と分離する |
| **データソース群** | CDM（継続的診断・緩和システム）、SIEM、IAM、PKI 等、PE にコンテキスト情報を提供するコンポーネント群 |

> **現実の注意点**: NIST モデルでは PEP が一元化された判断拠点として描かれているが、実際のシステムでは**複数の PEP が存在する**。クラウドプラットフォームが独自の PEP を持ち、ネットワーク PEP・アプリケーション PEP とは独立して稼働する構成が一般的である。

## 5. 原則から実践へのマッピング

ガイディング原則を設計作業で扱える「実践（Practice）」に変換する。

| ガイディング原則 | 設計実践 |
|----------------|---------|
| データ中心セキュリティ | アイデンティティ・データ・トランザクション識別 |
| Never trust, always verify | 継続的認証 |
| Never trust, always verify | 適応型アクセス制御 |
| Never trust, always verify | 最小権限（Least Privilege） |
| Never trust, always verify | マイクロセグメンテーション |
| あらゆる場所でのデータ保護 | 転送中・保存時・使用中の暗号化 |
| Assume breach | 脅威検知と対応 |

## 6. 各実践の詳細

### 6-1. アイデンティティ・データ・トランザクション識別

Zero Trust で保護すべきリソースを特定するには、システムを流れるすべてのアクター・データ・トランザクションフローを体系的に把握することが出発点となる。

**作業手順**:
1. **システムコンテキスト図**を作成し、システム外部に存在するユーザー・デバイス・サービスを特定する（アーキテクチャ設計 Phase）
2. **トランザクションフロー図**（シーケンス図・データフロー図）で機能コンポーネント間の処理と通信を可視化し、脅威モデリングに接続する
3. **インフラ設計**において、デプロイ済みコンポーネントへのフローを再分析し、PEP の配置箇所を決定する

この識別プロセスはライフサイクル全体（Plan→Design→Build→Run）を通じて繰り返し実施する。

---

### 6-2. 継続的認証（Continuous Authentication）

従来の IAM ではセッション開始時に一度認証すれば、そのセッションは数日〜数週間有効であることが多い。しかし、セッション開始から時間が経過した後も同一人物・同一システムであることを保証できるのか、という問いに答えられない。

**継続的認証の考え方**:
- 認証はセッション開始時だけでなく、**セッション全体を通じて継続的に実施**する
- セッション中にコンテキスト情報（位置・デバイス状態・行動パターン等）が変化した場合、アクセスポリシーを再評価し、場合によってはセッションを終了させる

**コンテキスト情報の例**:

| カテゴリ | 具体例 |
|---------|--------|
| 行動パターン | タイピング速度・操作リズムの異常検知 |
| 生体情報 | 指紋・顔認証等の再確認 |
| デバイス状態 | セキュリティコンプライアンス準拠度 |
| 環境情報 | アクセス時刻・地理的位置の逸脱 |

継続的認証の失敗は、脅威検知のシグナルとしても活用できる。現時点では広範な実装には至っておらず、適応型アクセス制御と組み合わせて実現するケースが多い。

---

### 6-3. 適応型アクセス制御（Adaptive Access Control）

アクセス権限をコンテキストに応じて動的に増減させる仕組みである。固定的な権限付与ではなく、**アクセス時のコンテキストをリアルタイムに評価してアクセスレベルを決定**する。

**コンテキストによる権限調整の例**:
- オフィス内（信頼済みネットワーク）からのアクセスは広い権限、リモートからは絞った権限
- 通常の勤務時間外のアクセスは追加認証を要求
- 組織管理デバイスからのアクセスは高い信頼度

**技術的構成要素**:
- リアルタイムの PEP または組み込みアクセス制御ロジック
- 利用可能なコンテキストデータの最大活用
- RBAC（ロールベース）・ABAC（属性ベース）・リスクベースアクセス制御の組み合わせ

現在のクラウド環境では、システム間・サービス間通信においてきめ細かいアクセス制御と強固な認証が組み合わせられており、Zero Trust Network Access（ZTNA）がネットワークエッジでの代表的な実装形態である。

---

### 6-4. 最小権限（Least Privilege）

ユーザーまたはシステムは、業務遂行に必要な最小限のアクセス権限のみを持つべきというセキュリティ原則である。

**設計上の要点**:

| 観点 | 内容 |
|------|------|
| デフォルトゼロアクセス | 明示的に許可されない限り、アクセス権は一切付与しない |
| 職務分掌の強制 | 機微なプロセスには複数人の関与を求め、単独での不正操作を防止する |
| 特権クリープへの対処 | 過去の役職から引き継いだ不要な権限（Privilege Creep）を定期的に棚卸しし、剥奪する |
| 最小権限での認可 | セッション単位でリアルタイムにコンテキストを評価し認可する |

**設計への組み込み箇所**:
- **要件定義**: スイムレーン図と職務分掌マトリクスで必要な権限を文書化する
- **インフラ設計**: コンポーネント間・サービス間・デバイス間のセッション認可で最小権限を適用する
- **セキュリティ運用**: RACI・プロセス・手順書で運用レベルの最小権限要件を定義する

> **複数 PEP の考慮**: クラウドプラットフォームはネットワーク PEP・アプリケーション PEP とは独立した独自の PEP を持つことが多い。最小権限の実施ポイントを単一箇所に限定せず、各レイヤーで個別に適用することを前提に設計する。

---

### 6-5. マイクロセグメンテーション（Microsegmentation）

従来のネットワーク分離では、数百台のデバイスを含む大きなセグメントに分割していた。マイクロセグメンテーションでは、コンピュートリソースを**小さく隔離されたセグメント**に細分化し、侵害が発生した場合のラテラルムーブメント（横移動）を最小化する。

**主な効果**:
- 単一のホスト侵害が他のセグメントに波及するリスクを低減
- セグメント境界ごとにアクセス制御を適用することで最小権限の実施箇所を増加
- 境界内部の「暗黙の信頼」を排除

インフラアーキテクチャ設計（デプロイメントアーキテクチャ・クラウドアーキテクチャ）フェーズで適用する。

---

### 6-6. 転送中・保存時・使用中の暗号化

クラウド環境ではインフラ運用を CSP（クラウドサービスプロバイダー）に委ねるため、ネットワーク経路・ストレージ・処理中のいずれにおいても機密データを保護する暗号化が不可欠である。

| フェーズ | 主な技術 |
|---------|---------|
| 転送中（In transit） | mTLS・TLS・IPSec |
| 保存時（At rest） | ストレージ暗号化・鍵管理サービス（KMS）・HSM |
| 使用中（In use） | 機密コンピューティング（Confidential Computing） |

特に保存時の暗号化は、クラウドの特権ユーザー（CSP の管理者）からも最も機微なデータにアクセスできないよう設計することが目標となる。証明書管理・鍵管理サービスの適切な運用を組み合わせる。

---

### 6-7. 脅威検知と対応（Threat Detection and Response）

「侵害を想定する」スタンスでは、**同一ネットワーク上のすべてのシステムを潜在的な攻撃者として扱う**。隣接するシステムを信頼するのではなく、接続するすべてのシステムが危険にさらされている可能性を前提とする。

**アーキテクチャへの組み込み**:
1. 脅威モデリング（機能コンポーネントのトランザクションフロー分析）において、潜在的な脅威と異常挙動の識別を行う
2. インフラアーキテクチャの脅威分析で、デプロイ済みコンポーネントの脅威を識別する
3. 識別した脅威を脅威検知ユースケースとインシデント対応ランブックに変換する（セキュリティ運用フェーズ）

---

## 7. ハイブリッドクラウドでの Zero Trust 適用パターン

クラウドネイティブ環境では Zero Trust を実現するための基盤が整いつつある。NIST はマルチクラウド・ハイブリッド環境向けに、アプリケーションレベルのきめ細かいポリシー適用を実現するためのガイダンスを公開している。

| 適用パターン | 設計上のポイント |
|------------|----------------|
| **マルチクラウド PEP** | 各クラウドプラットフォームが独自の PEP を持つ。ネットワーク・アプリケーション・クラウドの各レイヤーで独立した PEP を設計し、それぞれで最小権限を適用する |
| **ZTNA（Zero Trust Network Access）** | VPN 代替として、ネットワークエッジでのアクセス制御を実現。ユーザー・デバイス・アプリケーション単位で接続を許可し、ネットワーク全体への暗黙アクセスを排除する |
| **サービスメッシュ** | マイクロサービス間の通信に mTLS を適用し、コンポーネント間の暗黙の信頼を排除。サービス間認可もポリシーで管理する |
| **クラウドネイティブ IAM** | クラウドの細粒度 IAM（RBAC・ABAC）を活用し、サービスアカウント・ワークロードアイデンティティに最小権限を適用する |

## 8. Zero Trust ソリューション技術マップ

原則ごとに代表的な技術ソリューションを整理する（特定製品への言及は含まない）。

| 原則 | IAM | データ | アプリケーション | エンドポイント | ネットワーク |
|------|-----|-------|---------------|-------------|------------|
| **Never trust, always verify** | JIT アクセス・特権アクセス管理・継続的認証・適応型アクセス制御 | デジタル権利管理・適応型アクセス制御 | 適応型アクセス制御・許可プロセスリスト | 信頼できるコンピューティングベース・特権アクセス管理 | 適応型アクセス制御・マイクロセグメンテーション・動的ネットワークアクセス制御・ZTNA |
| **データ保護** | — | mTLS・証明書管理・KMS/HSM | mTLS・証明書管理・KMS/HSM | mTLS・証明書管理・KMS/HSM | IPSec |
| **Assume breach** | ユーザー・エンティティ行動分析（UEBA） | データ損失防止（DLP） | エンドポイント検知・応答（EDR） | EDR・ファイル整合性監視 | ネットワーク検知・応答（NDR） |

## 9. アーキテクチャへの統合方法

Zero Trust はアーキテクチャ設計の特定フェーズだけで適用するものではなく、ライフサイクル全体に分散して組み込む。

```
Plan         → 適用コンプライアンス要件・リスク許容度の確認（AskUserQuestion）
Design       → システムコンテキスト図でアクター識別 → データフロー図でトランザクション識別
              → 脅威モデリングで脅威・コントロール特定
              → PEP 配置・マイクロセグメント境界の設計
Build        → IaC でマイクロセグメント・mTLS・最小権限ポリシーをコード化
Run          → 継続的認証・UEBA・DLP・NDR による継続的監視
              → インシデント対応ランブックに基づく対応
Close        → 権限・セグメント境界の棚卸し、アーカイブデータの暗号化確認
```

各フェーズでのアーティファクトとの接続:
- **システムコンテキスト図** → アクター・信頼境界・PEP の初期配置
- **脅威モデル** → 脅威検知ユースケースの入力
- **インフラアーキテクチャ図** → マイクロセグメント境界・暗号化ポイントの可視化
- **RACI マトリクス** → 最小権限・職務分掌の組織的定義
- **ランブック** → Assume Breach 発動時のインシデント対応手順

## 10. アンチパターン

| アンチパターン | 問題点 | 対処 |
|-------------|--------|------|
| 境界型モデルの踏襲 | 内部ネットワーク = 信頼という前提を残したまま Zero Trust を名乗る | すべてのアクセス経路（内部・外部）に PEP を配置し、暗黙の信頼をゼロにする |
| セッション開始時のみの認証 | セッション乗っ取りや長時間不正アクセスに対応できない | 継続的認証とセッション中のコンテキスト再評価を実装する |
| デフォルトアクセス許可 | 最小権限の逆。不要な権限が蓄積し特権クリープが発生する | デフォルトゼロアクセスを徹底し、明示的な許可のみアクセスを付与する |
| 単一 PEP への集中 | クラウドプラットフォーム・アプリ・ネットワーク層に独立した PEP が必要なのに一元化しすぎる | 各レイヤーで独立した PEP を設計し、それぞれでポリシーを適用する |
| コンプライアンス充足で満足 | Zero Trust を「チェックリストの消化」として扱い、実質的なリスク低減を行わない | コントロールの有効性を継続的に計測・監視し、検知されない脅威への対応能力を維持する |

## 11. QA チェックリスト

| 観点 | 確認内容 |
