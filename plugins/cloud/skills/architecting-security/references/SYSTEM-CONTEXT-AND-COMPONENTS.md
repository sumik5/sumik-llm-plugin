# システムコンテキスト図とコンポーネントアーキテクチャ

システムの境界を定義し、外部アクターとのインターフェースを可視化する2つの基礎的な作図技法を解説する。これらはアーキテクチャ分解の最上位2層を形成し、後続の脅威モデリングや展開設計の基盤となる。

## 1. システムコンテキスト図

### 1.1 目的と位置づけ

システムコンテキスト図は、設計対象システムを黒箱として中央に置き、外部の人間アクター・システムアクターとのやり取りを俯瞰する図である。システムの境界（スコープ）を確定し、外部インターフェースを可視化することで、後続のコンポーネント設計・脅威モデリングの出発点となる。

アーキテクチャ分解の第1層として位置づけられ、「何が外部で何が内部か」という根本的な問いに答える。

### 1.2 アーキテクト役割の分担

| 役割 | システムコンテキスト図への関与 |
|------|-------------------------------|
| アプリケーションアーキテクト | 外部アクターとユースケースを整理し、システム境界を定義する |
| インフラアーキテクト | 外部システムアクターとの物理的・論理的接続を検討する |
| セキュリティアーキテクト | 各インターフェースで流通するデータを特定し、Zero Trust実装ポイントを識別する |

セキュリティアーキテクトは他の担当者が作成したコンテキスト図にセキュリティ情報を追記することが多いが、セキュリティサービス自体が設計対象の場合は単独でシステムコンテキスト図を作成する場合もある。

### 1.3 技術設計図の表記法

組織横断・クラウドサービスプロバイダー（CSP）非依存の共通表記法を用いることで、ステークホルダー間の意思疎通を効率化できる。主要な図要素は以下のとおり。

| 要素 | 説明 | 抽象度 |
|------|------|--------|
| **アクター** | 人間アクターと外部システムを円形アイコンで表現 | — |
| **ターゲットシステム** | 設計対象システム（コンテキスト図の中央） | — |
| **デプロイメントユニット** | ノード上で動作する機能の配備インスタンス（矩形） | 論理/規定 |
| **ロケーション** | 地理的・組織的な配置場所（左上マーカー付き矩形） | 論理/規定 |
| **コンポーネント** | 機能を担う部品（コネクタ付き矩形） | 論理/規定 |
| **論理グループ** | 非機能要件を共有するコンポーネントの集合 | 論理のみ |
| **ノード** | コンポーネントが展開される基盤（左上シンボル付き矩形） | 論理/規定 |
| **ネットワークゾーン** | 共通セキュリティポリシーを適用するゾーン（破線枠） | 規定のみ |

> **表記の選択**: 丸角は論理（論理ビュー）、角張りは規定（prescribedビュー、クラウド上への展開で使用）を示す。仮想化・コンテナ化環境では物理ビューの代わりに規定ビューを用いる。

### 1.4 アクターの特定と分類

#### 人間アクター

人間アクターは**ロール単位**で識別する（個人ではなくロールで記述）。組織内外を問わず、システムとインタラクションをもつすべてのロールを列挙する。

- 業務プロセスを担うロール（ビジネスアクター）
- システムを運用するロール（オペレーターアクター）
- 外部委託先のロール：委託先がシステム経由でアクセスする場合はシステムアクターとなる場合がある

#### システムアクター

**外部システム**（IT/OT/IoT）のうち、設計対象システムの外側に位置するものをシステムアクターとして扱う。判断基準は「自組織がセキュリティポリシーを直接制御できるか」である。制御できない場合は外部システムアクターとして扱い、契約・ドキュメントオブアンダースタンディング（DoU）によるセキュリティ要件の合意が必要となる。

**チェックポイント**: あるコンポーネントがPaaSサービスとして外部に委託されているなら、そのサービスのカスタマイズや制御がどこまで可能かを精査する。完全なカスタマイズが不可能であれば、内部コンポーネントではなくシステムアクターとして扱う方が適切な場合がある。

### 1.5 ビジネス/ITコンテキストの活用

システムコンテキスト図を正確に描くには、以下のインプットが必要である。

- **ビジネスコンテキスト**: ビジネス目標・業務プロセス→人間アクターの識別源
- **現行ITランドスケープ**: 既存システム・技術制約→システムアクターの識別源

情報が不足している段階でも、ステークホルダーと一緒にシステムコンテキスト図を作成することがビジネスコンテキストの明確化に役立つ。

### 1.6 システムコンテキストの文書化

図だけでは不十分なプロジェクトでは、アクター一覧表を作成して補完する。

| アクター | 説明 | インターフェース |
|---------|------|----------------|
| （例）Webユーザー | 外部向けWebアプリにアクセスするエンドユーザー | TLS 1.3セッション経由のHTTPS |
| （例）外部決済システム | 支払い処理を担う外部PaaS | REST API over TLS 1.2+ |
| （例）監視オペレーター | セキュリティオペレーション担当者 | ジャンプホスト経由のSSH |

初期段階ではインターフェースの詳細が未確定でよい。ドラフトとして記録し、設計の進行とともに更新する。

### 1.7 システムコンテキストのQAチェックリスト

- [ ] 組織内外の人間アクターをすべてロール単位で列挙したか
- [ ] 各ロールのユースケース（データフローを発生させる操作）を紐づけたか
- [ ] 遠隔勤務者・オフィス勤務者など、ロールのサブ分類が必要か検討したか
- [ ] 外部システムアクターの所有・制御状況を確認したか
- [ ] バッチ処理・タイマーイベントなど、アクター以外がトリガーするフローも洗い出したか
- [ ] データのライフサイクル（バックアップ・サイバーリカバリーなど）を考慮したか

## 2. コンポーネントアーキテクチャ

### 2.1 位置づけと目的

コンポーネントアーキテクチャはアーキテクチャ分解の第2層であり、システム境界の内部を機能的なビルディングブロック（コンポーネント）に分解する。セキュリティアーキテクトはこの図から以下を識別する。

- 転送中のデータ（data in transit）
- 保存中のデータ（data at rest）
- 処理中に生成されるデータ（data in use）
- 機密データ・センシティブデータ・高度機密データ

### 2.2 2つの抽象レベル

| レベル | 内容 | 用途 |
|--------|------|------|
| **論理レベル** | 実装を特定しない機能コンポーネントと責任範囲 | 高レベル設計・ベンダー非依存の议論 |
| **デプロイレベル** | 実際の実装技術・製品を選択した具体的な構成 | 詳細設計・実装の起点 |

サブシステムは、関連する機能コンポーネントをまとめるための論理的なグルーピングであり、それ自体は固有の能力やインターフェースを持たない。

### 2.3 可視化技法

#### コンポーネント図（静的モデル）

依存関係と静的な構造を表す。アクターはシステムコンテキスト図と対応させる。

```
主要な図要素
- コンポーネント（丸角矩形）
- サブシステム（コンポーネントを囲む矩形）
- 関係（矢印の方向がインタラクションの向きを示す）
```

#### シーケンス図（動的モデル・時系列）

重要ユースケースにおける、コンポーネント間のやり取りの順序と時系列を示す。

```
主要な図要素
- ライフライン（上部のボックス）
- アクティベーションバー（処理中の期間を示す縦棒）
- メッセージ（実線＝呼び出し、点線＝戻り値）
- セルフメッセージ（処理前に必要な内部処理を示す）
```

シーケンス図は認証フロー（OAuth 2.0 / OpenID Connect など）の詳細記述に特に有用である。

#### コラボレーション図（動的モデル・関係）

シーケンス図と同じ情報を「コンポーネント間の関係」として表現する。少数のコンポーネント間の単純なやり取りを業務関係者に説明する際に適している。コンポーネント数が多くなるとシーケンス図の方が読みやすくなる。

```
主要な図要素
- コンポーネント（矩形）
- リンク（関係を示す実線）
- 矢印（メッセージ、番号でフローの順序を示す）
```

#### データフロー図（DFD）

セキュリティ観点から最重要な図。データがどのコンポーネント間をどの方向に流れ、どこに保存されるかを示す。

```
主要な図要素
- プロセス（楕円：データを変換する処理）
- データフロー（矢印：単方向。脅威モデリングで方向が重要）
- データストア（平行線付き矩形：DB・ファイルなど永続化領域）
- 外部エンティティ（矩形：システム外部のアクター）
```

DFDはデータのボトルネック・依存関係・潜在的リスクの発見を助け、脅威モデリングの直接的なインプットとなる。

### 2.4 コンポーネントアーキテクチャの構築プロセス

| フェーズ | 内容 |
|---------|------|
| **Plan（何を作るか）** | 主要コンポーネントを識別し、サブシステムに分割して責任を割り当てる。参照アーキテクチャ・パターンを積極的に活用する |
| **Design（どう機能させるか）** | インターフェース・転送データ・事前・事後条件・インタラクションシーケンスを定義する |
| **Deploy（どう実装するか）** | 実装戦略・使用製品・第三者パッケージを特定する。重要なアーキテクチャ決定が行われるフェーズ |

### 2.5 コンポーネントアーキテクチャのQAチェックリスト

- [ ] すべての機能アクターがコンポーネントアーキテクチャに表現されているか
- [ ] すべての機能要件を満たすコンポーネントが定義されているか
- [ ] コンポーネント間インターフェースで流通するデータを特定し、方向を確認したか
- [ ] 前段のシステムコンテキストで識別したデータが正しく機能コンポーネントを流れているか
- [ ] 不明な実装詳細を前提として置き、それをRAIDログに記録したか
