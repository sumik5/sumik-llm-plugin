# COMPLIANCE-AND-GOVERNANCE.md

コンプライアンス管理・エンタープライズセキュリティアーキテクチャ（ESA）・ガバナンス体制のリファレンス。外部コンテキスト（法規制・標準・脅威）と内部コンテキスト（戦略・ポリシー・原則・プロセス）を整理し、RAIDログのガバナンス上の位置づけを定義する。

## 1. コンプライアンス ≠ セキュリティ

コンプライアンスとセキュリティは **別の概念** であり、混同するとアーキテクチャの品質が低下する。

| 観点 | コンプライアンス | セキュリティ |
|------|----------------|------------|
| 目的 | 規制・標準・契約上の義務を証明 | リスクに基づく実質的な保護 |
| アプローチ | 最低限の要件を満たすコントロールを選択 | 脅威・リスク許容度に応じた追加コントロールを選択 |
| 合格基準 | 監査・認証での適合 | インシデントへの耐性とリカバリ能力 |
| 限界 | チェックリスト合格 ≠ 攻撃者への耐性 | 規制対応が不完全になりうる |

**設計指針**: コンプライアンス要件を最低ベースラインとして扱い、脅威ランドスケープとリスク許容度に基づく追加コントロールを積み上げる。法規制が高レベルのガイダンスに止まる場合は、業界標準・専門組織のフレームワークを参照して具体的なコントロールを導出する。

## 2. エンタープライズセキュリティアーキテクチャ（ESA）

### 2.1 セキュリティ「プロセス」から「サービス」への転換

組織がセキュリティ能力を「プロセス」と呼ぶ慣習があるが、「サービス」として捉えることを推奨する。

**「サービス」として扱う理由**:
1. セキュリティ能力は技術・プロセス・人の三要素から構成される。技術のみ、またはプロセスのみでは機能しない。
2. 一つのセキュリティ能力が複数のコントロールフレームワーク領域（アクセス制御・ネットワーク・暗号化など）の要件を横断的に受け取ることが多い。プロセスオーナーによる縦割り管理では整合が取れない。
3. サービスとして定義することで、可用性・レスポンスタイム・キャパシティ計画・回復時間目標（RTO）などの非機能要件が設計対象になる。

ハイブリッドクラウド環境では、シークレット管理やIDライフサイクル管理といったセキュリティサービスが失敗するとアプリケーション全体が即座に停止しうる。24×7 の高可用性と迅速な自動回復を前提にサービスを設計する必要がある。

### 2.2 ESAの3階層分解: ドメイン → カテゴリ → サービス

標準的な分類体系（taxonomy）を持つことで、組織横断でセキュリティ能力を一貫して議論・管理できる。各層は認知負荷を考慮して7個以下のグループを目安に設計する。

```
Security Domain（上位）
    └── Security Category（中位・各ドメイン5カテゴリ程度）
            └── Security Service（下位・単一技術コンポーネントで実装できる粒度）
```

**6つのセキュリティドメイン（例）**:

| ドメイン | 保護対象・役割 |
|---------|-------------|
| Identity and Access | エンドユーザー認証・認可・ID ライフサイクル管理 |
| Network | ネットワークアクセス制御・セグメンテーション・暗号化転送 |
| Application | アプリケーション層の保護（WAF・API セキュリティ・脅威モデリング対象） |
| Data | データ保護・暗号化・DLP（→ `DATA-CENTRIC-SECURITY.md` 参照） |
| Endpoint | クライアントデバイス・サーバー・コンテナの保護 |
| Detect and Respond | 全ドメインの脅威検知・インシデント対応（SIEM・SOAR） |

> **補足**: ドメインを水平方向に並べると、エンドユーザーのアクセスフロー（Identity → Network → Application → Data → Endpoint）を表現できる。Detect and Respond はその全体に横断する。上位に GRC（ガバナンス・リスク・コンプライアンス）、下位にサービス管理プレーンが位置する。

### 2.3 NIST CSF とのマッピング

NIST Cybersecurity Framework (CSF) は6つの機能（Govern / Identify / Protect / Detect / Respond / Recover）でサイバーセキュリティの成果を整理する。ESAのドメインとCSFの機能は対応関係を持つ。

| NIST CSF 機能 | ESA ドメインとの主な対応 |
|-------------|----------------------|
| Govern | GRC・ポリシー・リスク管理（ESA 上位層） |
| Identify | 資産管理・リスクアセスメント（全ドメインに横断） |
| Protect | Identity / Network / Application / Data / Endpoint |
| Detect | Detect and Respond |
| Respond | Detect and Respond |
| Recover | サービス継続・DR設計 |

両フレームワークを同時に使うと管理が複雑になる。プロジェクトでは**どちらかを主体に選択**し、もう一方との対応表を補足として保持する方が実用的である。

### 2.4 セキュリティサービス管理

マイクロセグメンテーション環境では、数百万件のアクセス制御ルールと設定を横断的に管理する必要が生じる。サービス管理を2つの中核カテゴリで設計する。

**ポリシー・標準管理（Policy and Standards Management）**:
- セキュリティリソースを個別設定でなくポリシーに対応させるポリシーベース管理
- リソース増加に比例してスケールできる自動化が前提
- ポリシーと標準の一元管理で組織全体の整合性を保証

**レポート・分析・測定（Reporting, Analytics, and Measurements）**:
- ポリシー・標準の遵守状況を独立して報告（コントロールの自己評価ではなく客観的な計測）
- セキュリティサービスのサービスレベル達成を可視化
- 行動変容のドライバーとして機能（報告が存在しないコントロールは形骸化しやすい）

### 2.5 コントロールフレームワークのマッピングとギャップ分析

複数の規制・標準に対応する場合、個別のフレームワークに1対1で対応すると重複コントロールが増加し運用コストが膨らむ。**ESAを共通基盤としてコントロールをマッピング**する。

```
[法規制A] ─┐
[法規制B] ─┤
[標準X]   ─┼──→ ESA ドメイン/カテゴリ/サービス ──→ 統合コントロールカタログ
[標準Y]   ─┤
[業界規格] ─┘
```

**ギャップ分析の手順**:
1. 適用対象のフレームワーク（PCI DSS・NIST SP 800-53・ISO/IEC 27001・CSA CCM 等）を特定
2. 各フレームワークのコントロール要件を ESA の ドメイン/カテゴリ/サービスに割り当て
3. 未割り当て（ESAに存在しないコントロール）と要件重複（複数フレームワークから要求される共通コントロール）を特定
4. 統合コントロールカタログを構築し、プロジェクトの要件ベースラインとして活用

> **注意**: 業界フレームワークは最低ベースラインである。ギャップ分析の結果は「追加コントロールの不要証明」ではなく「ベースライン + リスク基づく追加コントロール」の出発点として使う。

## 3. 外部コンテキスト

ソリューションアーキテクチャ設計を開始する前に、外部から課されるコンプライアンス・セキュリティ要件を網羅的に把握する。

### 3.1 法規制の5領域

| 領域 | 主要内容 | 設計への影響 |
|------|---------|------------|
| データ保護・プライバシー | 個人データの収集・利用・保管・開示ルール。同意取得・個人の権利保障 | 暗号化要件・データレジデンシー・アクセス制御設計 |
| 侵害通知 | データ漏洩時の通知義務（通知先・タイミング・内容の法定要件） | インシデント検知・対応プロセスの設計 |
| サイバー犯罪・法執行 | 不正アクセス・詐欺・破壊行為の刑事的枠組み。国内CSIRT機関との連携義務 | インシデント管理プロセスの公的機関連携ポイント |
| 重要インフラ保護 | エネルギー・医療・金融・交通等の追加セキュリティ要件・インシデント報告義務 | 追加コントロール・リスクアセスメント・情報共有要件 |
| 運用レジリエンス | 金融機関等へのレジリエンス要件（インシデント対応・サードパーティリスク・定期テスト） | BCPおよびDR設計・クラウドワークロードへの適用解釈 |

**注意点**: データ処理・転送・アクセスの **地理的な場所** に応じた法規制が適用される。リモートからのITサポートや開発者がデータを表示する場合もデータ移転に該当しうるため、法的助言を受けることを推奨する。

### 3.2 業界・専門組織のベストプラクティス

法規制は高レベルのガイダンスに留まることが多く、具体的なコントロール実装には専門組織のフレームワークが有用である。

**コントロールフレームワーク（高レベル・汎用）**:
- NIST Cybersecurity Framework (CSF): 6機能でサイバーセキュリティ成果を整理。広く国際的に参照される
- CSA Cloud Controls Matrix (CCM): クラウドコンピューティング向けのコントロール集

**詳細コントロールカタログ**:
- NIST SP 800-53: 情報システムに対するセキュリティ・プライバシーコントロールの詳細カタログ。OSCAL形式で自動コンプライアンスチェックに対応
- CIS Critical Security Controls: 優先順位付きの実践的なサイバーセキュリティベストプラクティス

**設定ベースライン（特定技術向け）**:
- CIS ベンチマーク: クラウドプロバイダ・OS・サーバーソフトウェア等の設定標準
- NIST STIG/National Checklist Program: 設定チェックリスト。SCAP形式でツール自動化可

**産業別標準**:
- PCI DSS: 決済カード処理に関わる組織が遵守すべきコントロール標準

### 3.3 ISO/IEC 27001 の位置づけ

ISO/IEC 27001 は**情報セキュリティ管理システム（ISMS）**の構築・運営に関する国際標準であり、技術的なセキュリティコントロールのカタログではない。

- 役割: CISOチームが組織全体のセキュリティポリシー・実践・手順を整備するための枠組みを提供
- 特徴: ハイブリッドクラウドアーキテクチャの個別技術コントロールを規定するものではない
- 適用: 組織・クラウドサービスプロバイダの認証・監査対象として位置づける。ソリューションアーキテクトは認証状況を前提に設計する

### 3.4 法人・消費者の期待

取引先法人（B2B）は、自社データおよびその顧客データを適切に保護することを委託先に求める。要求水準は次の3段階に分かれる。

1. **最低ライン**: 業界・国・州に対応した標準コントロールフレームワークへの準拠
2. **外部監査・認証**: 独立機関によるコンプライアンス証明（主要クラウドプロバイダはこれを公開する）
3. **リスクベースの追加要件**: 規制要件を上回るコントロールをサプライヤーに要求するケース

消費者（B2C）の期待は可視性に依存する。暗号化の明示・多要素認証（MFA）・プライバシー保証の提示によってサービスへの信頼が高まる。

**対処方針**: 認証取得は保証の一部に過ぎない。システムの統合設計次第でセキュリティ品質が変わるため、認証の存在に頼らずアーキテクチャレベルで保護を設計する。

### 3.5 脅威ランドスケープの把握

脅威ランドスケープは業種・ホスティング地域・使用技術・インターネット露出度によって異なる。主要な参照先：

- 政府機関・業界団体のサイバーセキュリティ脅威報告（年次・四半期）
- MITRE ATT&CK: 脅威アクターのTTP（戦術・技法・手順）カタログ
- CVEデータベース: 特定ソフトウェアの既知脆弱性

**設計への活用**: 脅威レポートから自プロジェクトに関連する上位脅威グループ（ランサムウェア・サプライチェーン攻撃・ソーシャルエンジニアリング等）を特定し、対応するコントロールメカニズムが設計に含まれているか確認する。

### 3.6 サイバーセキュリティ脆弱性

使用する技術コンポーネントには脆弱性が含まれる可能性が高い。アーキテクトはパッチ適用を設計に組み込む必要がある。

- **クラウドプロバイダの責任範囲**: クラウドプラットフォーム自体のパッチはクラウドサービスプロバイダ（CSP）の責任。ただし CSP のパッチ適用がアーキテクチャの可用性に影響を与える場合がある
- **利用者の責任範囲**: クラウドプラットフォーム上のワークロード・ミドルウェア・アプリケーション層のパッチは利用者の責任
- **ハイブリッドクラウドの複雑性**: 複数技術プラットフォームの組み合わせは攻撃表面を拡大する。脆弱性に対するアーキテクチャ的な防御（多層防御）が必要

## 4. 内部コンテキスト

組織内のガバナンス文書・戦略・ポリシーもアーキテクチャを制約・誘導する。これらを事前に把握することで設計の手戻りを防ぐ。

### 4.1 事業・情報システム戦略

事業戦略のなかにセキュリティ設計に影響する要素が存在する。

| 要素 | セキュリティ設計への影響例 |
|------|------------------------|
| ビジョン・ミッション | B2B vs B2C の差で適用すべきコントロールフレームワークが変わる |
| 市場分析 | 競合他社の侵害事例から自社が強化すべき領域を特定 |
| 価値提案 | 「信頼性」を差別化要因とする場合、外部認証取得が戦略的に意味を持つ |
| 戦略目標・施策 | 進行中の他プロジェクト（ポリシー更新・サプライヤーコンプライアンス枠組み策定）との整合 |
| IS戦略・エンタープライズアーキテクチャ | 技術ビジョン（モノリス→コンテナ移行等）は必要なセキュリティサービスを変化させる |
| IT運用・サービス管理 | セキュリティサービスは組織標準の IPC プロセス（インシデント・問題・変更）に統合する必要がある |

### 4.2 現行IT環境とセキュリティコントロールプレーン

セキュリティコントロールプレーン（セキュリティサービスとその管理ツールのホスティング場所）の配置はワークロードのパフォーマンスと可用性に直接影響する。

**主な配置オプションと考慮事項**:

| 配置先 | 特徴 | リスク |
|-------|------|--------|
| オンプレミスデータセンター | 既存インフラ活用。物理コントロール | クラウドへの延伸時にレイテンシ・可用性問題が発生しうる |
| クラウドプラットフォーム | クラウドネイティブサービスとの低レイテンシ | クラウド障害時のフェイルオーバー設計が必要 |
| PoP（ポイントオブプレゼンス）/ コロケーション | オンプレミス・クラウド双方から独立 | 追加コスト・管理複雑性 |
| 分散配置 | レイテンシ敏感なサービスはクラウドネイティブ、低頻度サービスはオンプレミス | サービス間整合性の設計が必要 |

**設計原則**: コントロールプレーンの配置がワークロードの可用性要件（RTO・RPO）を満たせるか検証する。セキュリティサービスの障害がアプリケーション全体の停止を引き起こすリスクを評価する。

### 4.3 ポリシー・実践・標準

CISOチームが外部法規制・組織文化・リスク許容度を統合して策定したポリシー・実践・標準は、コントロール実装の基準となる。

- **既存サービスとの統合**: 組織が定義済みのセキュリティサービスを持つ場合、その統合ポイントに従って設計する
- **コントロールチェックリスト**: プロジェクトにはコントロールチェックリストが課されることが多い。マッピング文書で対応状況を示す
- **リスクベース追加コントロール**: ベースラインのポリシー適用に留まらず、プロジェクト固有のリスクに基づく追加コントロールを特定する

### 4.4 リスク管理

組織のリスク管理プロセスはソリューションアーキテクチャに追加のセキュリティ要件をもたらすことがある。

- **組織レベルのリスクレジスター**: 情報システムアーキテクチャに関連するリスクを確認する
- **リスクの一時的受け入れ**: 他プロジェクトへの依存等でコントロール実装が遅延する場合、リスク管理プロセスが一時的受け入れを承認することがある
- **ソリューション固有リスク**: 脅威モデリングで特定する（`THREAT-MODELING.md` 参照）

### 4.5 エンタープライズアーキテクチャ（EA）

組織のEAは事業目標とITインフラを整合させる戦略的枠組みを提供する。ソリューション設計時に確認すべき点：

- 組織全体のEA・エンタープライズセキュリティアーキテクチャが存在するか
- 採用が求められるアーキテクチャパターン・標準が定義されているか
- ESAとの整合（§2のESA参照）

### 4.6 ガイディング原則

組織レベルで定義されたセキュリティガイディング原則は設計上の判断の拠り所になる。以下は主要原則の概要である（詳細はソリューション設計に応じて展開する）。

| 原則 | 概要 |
|------|------|
| 防衛縦深（Defense in Depth） | 複数の防御層を重ねる。一つのコントロールが突破されても次の層で保護する |
| 最小権限（Least Privilege） | ユーザー・コンポーネントに必要最低限の権限のみを付与。定期レビューで権限クリープを防ぐ |
| 攻撃表面最小化 | 未使用サービスの無効化・定期パッチ適用・ネットワークセグメンテーションで侵入ポイントを削減 |
| 職務分掌（Separation of Duties） | 重要プロセスを複数の個人・役割に分割。不正・誤操作・品質問題を抑止する |
| Zero Trust | ネットワーク内部でも「信頼しない、常に検証する」を前提にアクセスを制御（`ZERO-TRUST-ARCHITECTURE.md` 参照） |
| マイクロセグメンテーション | アプリケーション層・コンポーネントごとに独立したネットワークセグメントを設ける（`CLOUD-INFRA-AND-PATTERNS.md` 参照） |
| Secure by Default | デフォルト設定が高セキュリティ状態になるよう設計。追加設定なしで保護される |
| Secure by Design | 設計開始時からセキュリティを組み込む（本スキル全体が体現するアプローチ） |
| KISS | コントロールの追加が複雑性を増し、かえってリスクを高めないか常に問い直す |
| オープンデザイン | 設計を広範なレビューにさらしセキュリティ上の弱点を早期発見する |

> 組織がこれらの原則を未定義の場合、アーキテクトが上記を出発点として提案・適用する。プロジェクト固有の原則を追加してもよい。

### 4.7 アーキテクチャパターンと自動化

ガイディング原則とベストプラクティスに従った繰り返し可能な設計をアーキテクチャパターンとして定義し、組織横断で再利用する。パターンを自動化した成果物を**Deployable Architecture**と呼ぶ（詳細は `CLOUD-INFRA-AND-PATTERNS.md` 参照）。

### 4.8 エンタープライズプロセス

静的なコントロールの設定だけでなく、**活動の順序・役割・制御ポイント・監査証跡**を定義するプロセスが必要である。

- プロセスは技術非依存で定義し、技術が変わっても手順の骨格は維持する
- セキュリティ固有のプロセスを独自に作るのではなく、組織標準のインシデント・問題・変更（IPC）管理プロセスに統合する
- 手順（procedure）・作業指示（work instruction）との3段階（プロセス→手順→作業指示）で詳細化する

## 5. RAIDログ: ガバナンスでの位置づけ

RAIDログは、プロジェクトチームが設計・実装を通じて特定した「不確実性」を系統立てて管理するガバナンスツールである。Build 段階での実運用は `BUILD-RUN-OPERATIONS.md` を参照。本セクションでは **概念定義とガバナンス上の役割** を扱う。

### 5.1 4カテゴリの定義

| 記号 | カテゴリ | 定義 | 基本アクション |
|------|---------|------|--------------|
| **R** | リスク（Risk） | 発生していないが、プロジェクトに負の影響を与えうる不確実な事象 | 回避・軽減・移転・受け入れ・共有のいずれかで対処 |
| **A** | 前提条件（Assumption） | 確認されていないが、プロジェクト成功に関連する事項 | 検証・削除（できるだけ早期に確認する） |
| **I** | 課題（Issue） | 既に発生し、対処が必要な問題 | 解決策・計画変更 |
| **D** | 依存関係（Dependency） | プロジェクト成功に必要な外部要因（第三者・他プロジェクト）への依存 | 合意取得・追跡 |

**記録フォーマット（推奨文例）**:

```
R: 「<リスク> が発生するリスクがある。原因は <原因>、結果として <影響> を招く可能性がある」
A: 「<前提条件> と仮定している。この前提が無効だった場合、<リスク> のリスクがあり <影響> を招く」
I: 「<課題> という問題があり、<影響> を引き起こした。原因は <原因>」
D: 「<対象> が <アクション> を実施することへの依存がある。期限通りに実施されない場合、<リスク> が発生し <影響> を招く」
```

### 5.2 ガバナンスでの役割

RAIDログが果たすガバナンス機能：

1. **透明性の担保**: プロジェクト全体のリスク・前提・課題・依存関係を一元可視化。チーム全員（PMだけでなく）が継続的に更新する
2. **プロアクティブなリスク管理**: 潜在的な障害を早期に特定し、先手の対処を可能にする
3. **引き継ぎ文書としての機能**: プロジェクト終了時・運用チームへの引き継ぎ時に、設計上の判断根拠と未解決事項の文脈を提供する
4. **設計レビューとの連携**: アーキテクチャ決定記録（ADR）の決定時に発生したRAIDアイテムを記録し、設計とリスク管理を接続する

### 5.3 セキュリティアーキテクトとRAIDログ

セキュリティアーキテクトが特にRAIDアイテムを生成しやすい設計局面：

- **脅威モデリング（Plan/Design フェーズ）**: 前提条件（A）が多数発生する（例：「個人情報を処理しないと仮定している」）
- **コンプライアンス要件の解釈**: 法的要件の適用範囲が不明確な場合にリスク（R）として記録
- **コントロール実装の遅延**: 他プロジェクトへの依存（D）や既存インストールとのギャップ（I）が発生する
- **アーキテクチャ決定（ADR 策定）**: 決定の影響として生じたリスク・依存関係を必ず記録する

### 5.4 QAチェックリスト（RAIDエントリ品質向上）

**共通**:
- [ ] 第三者との合意が必要な事項は「依存関係（D）」として記録しているか
- [ ] 全エントリに所有者・完了目標日・具体的なアクションが設定されているか
- [ ] 記録したアクションがエントリをクローズする内容になっているか

**リスク（R）**:
- [ ] そのリスクはすでに発生しているか → 発生済みなら「課題（I）」に変換
- [ ] 軽減策が適切なフェーズまでに実施されるよう計画されているか

**前提条件（A）**:
- [ ] 設計フェーズ完了前に検証されるスケジュールになっているか
- [ ] 検証不可能な場合、そのリスクを「リスク（R）」として記録し承認取得済みか

**課題（I）**:
- [ ] まだ発生していない事象を「課題」として記録していないか → 未来の事象は「リスク（R）」
- [ ] フェーズ内で解決できない場合のリスクを「リスク（R）」として分離しているか

**依存関係（D）**:
- [ ] 全依存関係が契約・内部合意書でカバーされているか
- [ ] 本番稼働前に合意取得状況を確認するチェックポイントが設定されているか

> **実運用（Build フェーズでの日常的な RAID ログ更新・インシデントへの連携）については `BUILD-RUN-OPERATIONS.md` を参照する。**

## 6. 使用時の確認ポイント（AskUserQuestion）

本リファレンスを参照する際に、以下が不明な場合は推測せず確認する。

| 確認事項 | 影響する設計 |
|---------|------------|
| 適用されるコンプライアンス要件は何か（PCI DSS / GDPR / 金融規制 / 医療規制 / 特になし） | コントロールマッピングの対象フレームワーク選定（§2.5・§3） |
| 組織はどのレベルの規制対象か（高規制・監査対応必須 / 中規模 / 軽量） | 成果物の詳細度・RAIDログの管理水準（§5） |
| 組織のEA・ESAは整備されているか | §2のESA活用 vs 新規定義の判断 |
| セキュリティガイディング原則は既に定義されているか | §4.6原則の採用 vs 組織固有原則への適合 |
| RAIDログを管理するプロジェクト管理プロセスは存在するか | §5のRAIDログの新規導入 vs 既存プロセスへの統合 |
