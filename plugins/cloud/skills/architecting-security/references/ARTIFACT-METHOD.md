# ARTIFACT-METHOD — アーティファクト駆動設計手法・ライフサイクル・役割

セキュリティアーキテクチャを「段階的に成果物を積み重ねる」アーティファクト駆動手法として実践するための全体論。設計思考からエンジニアリング・運用までの流れ、エンタープライズとソリューションアーキテクチャの位置づけ、セキュリティアーキテクトの役割と RACI、そして継続的アーキテクチャの考え方を扱う。

## 1. アーキテクチャ思考の位置づけ：設計思考から運用まで

### 1.1 設計思考（Design Thinking）の限界

設計思考は「ユーザーの課題を人間中心に解決する」優れた手法だが、**セキュリティ・スケーラビリティ・レジリエンス・コンプライアンス**といった非機能要件・品質特性を体系的に扱わない。設計思考だけでコーディングを開始すると、後から大幅な再設計が必要になるリスクがある。

### 1.2 アーキテクチャ思考（Architectural Thinking）

アーキテクチャ思考は情報システムの設計において、問題解決と意思決定を体系的に行うプロセスである。

- **対象**: システム境界の定義・機能コンポーネントとその相互作用・インフラへの展開・脅威の識別とコントロールの実装・ビジネス戦略とのアラインメント
- **特徴**: 非定量的な手法が中心（判断・原則・過去の教訓）であり、演繹的なエンジニアリングとは性質が異なる
- **アーキテクトの役割**: アーキテクチャ思考を体系的に適用して情報システムアーキテクチャを設計する人。セキュリティアーキテクトはセキュリティ観点でこれを実践する

### 1.3 エンジニアリング（Engineering）との違い

| 観点 | アーキテクチャ思考 | エンジニアリング |
|-----|-----------------|----------------|
| 目標 | クライアント満足・ビジネス要求への適合 | 技術的最適化・実装の精緻化 |
| 手法 | 非定量・原則・判断（帰納的） | 数学・科学的分析（演繹的） |
| 成果物 | 設計図・アーキテクチャ文書・決定記録 | 設定ファイル・コード・スクリプト・ハードウェア |

> 実際には境界は曖昧で重複する。例えばセキュリティアーキテクトがファイアウォールルールを個別に定義する作業はエンジニアリングに近い。重要なのは役割の棲み分けではなく、双方の活動が適切に行われることである。

### 1.4 運用思考（Operational Thinking）

セキュリティは設計・実装だけでなく、稼働後の継続的な運用で担保される。設計段階から運用アーキテクチャ（監視・検知・対応）を組み込まないと、後から運用インフラを追加するための大規模改修が必要になる（実際に多くのプロジェクトで発生している問題パターン）。

例: 暗号化セッション内の通信を検査するインライン検査装置を後から追加しようとすると、ネットワーク構成とアプリケーションアーキテクチャの両方を変更しなければならない。

## 2. PoC（概念実証）vs MVP（最小実用製品）

この区別はセキュリティアーキテクチャの適用スコープを決定する重要な判断点である。

| 観点 | PoC（概念実証） | MVP（最小実用製品） |
|-----|---------------|-------------------|
| 目的 | 機能原理の実証・技術検証 | 最小機能での本番サービス開始 |
| アーキテクチャ思考の深さ | 浅い（一部の主要決定のみ） | 深い（本番品質の特性を満たす全決定） |
| セキュリティ設計 | 最低限（テストデータ・非本番環境） | 完全（コンプライアンス・脅威対策・監視込み） |
| テスト要件 | 基本機能の動作確認 | 本番readiness を確認する正式テスト |

> **警告**: PoC を「MVP」と呼ぶと、ステークホルダーが本番投入可能と誤解するリスクがある。アーキテクチャ思考なしの「MVP」は実質 PoC であり、本番化には再設計が必要になる可能性が高い。

## 3. エンタープライズアーキテクチャとソリューションアーキテクチャ

### 3.1 エンタープライズアーキテクチャ（EA）

組織全体のビジョン・戦略と情報システムを継続的に整合させる方法論的枠組み。

- **目的**: 組織横断での一貫性・ベストプラクティスの適用・情報システム開発の効率化
- **構成要素**: アーキテクチャビルディングブロック（ABBs）・ガイディング原則・パターン・プロセスとサービスの定義
- **セキュリティへの応用**: エンタープライズセキュリティアーキテクチャはセキュリティサービスを分類・記述し、組織全体での一貫した実装を導く

### 3.2 ソリューションアーキテクチャ（SA）

特定の情報システム（インフラ・アプリケーション・ワークロード）の設計と実装を記述するアーキテクチャ。

- **目的**: 特定のビジネス課題を、人・プロセス・技術の観点から解決する方法を記述
- **構成**: エンタープライズコンテキスト + ビジネス/IT 要件 + 機能コンポーネント + 展開アーキテクチャ
- **セキュリティへの応用**: 本方法論の主対象。ソリューションのセキュリティ・コンプライアンス側面を記述する

> EA は SA を導く「地図」であり、SA は特定プロジェクトの「詳細設計」である。EA が定めるガイディング原則やパターンを SA が継承し、実装レベルに落とし込む。

## 4. 継続的アーキテクチャとアジャイル適用

### 4.1 ウォーターフォールとアジャイル

| 特性 | ウォーターフォール | アジャイル |
|-----|-----------------|-----------|
| 要件定義 | 開始前に固定 | 反復的・継続的 |
| 開発サイクル | 順次フェーズ | 短いスプリント |
| 変更対応 | 困難 | 柔軟 |
| アーキテクチャ | 大規模な先行設計 | 「ちょうど十分・ジャストインタイム」 |

実際のプロジェクトでは、インフラ設計（複数アプリが共有するためウォーターフォール的な深い事前設計が有効）とアプリケーション開発（アジャイル）を使い分けることも多い。

### 4.2 継続的アーキテクチャ（Continuous Architecture）の原則

アジャイル開発環境でアーキテクチャ思考を継続的に実践するための指針：

1. **プロジェクトからプロダクトへ**: アーキテクチャをプロジェクト一時的成果物ではなく、継続進化するプロダクトとして設計する
2. **品質特性への集中**: 機能要件だけでなく、セキュリティ・スケーラビリティ・レジリエンス等の品質属性を継続的に追求する
3. **設計判断の先送り**: 絶対必要になる直前まで取り消し困難な決定を遅らせ、情報が揃ってから判断する
4. **小さな変更の力**: 大規模なアーキテクチャ変更よりも、小さく取り消し可能な変更を積み重ねる
5. **ビルド・テスト・デプロイ・運用を含む設計**: 開発後の全フェーズを見越した設計を行う
6. **チーム構造と設計の整合**: システムアーキテクチャがチーム分割・コミュニケーションパスを反映するよう設計する

## 5. アーティファクトフレームワーク詳細

### 5.1 6グループの関係性

```
【エンタープライズコンテキスト】
    法規制・業界標準・組織ポリシー・EA・ガイディング原則
            │
            ▼ 入力
【要件】
    ユースケース・ユーザーストーリー・非機能要件・要件トレーサビリティマトリクス
            │
            ▼ 入力
【アーキテクチャ】 ─────────────────────────────────────┐
    システムコンテキスト図 → コンポーネント図 → 脅威モデル     │
    → デプロイメント図 → クラウドアーキテクチャ図 → ADR      │
            │                                            │
            ▼ 入力                                       │
【オペレーション】                                         │
    RACI・プロセス図・手順書・脅威検知ユースケース・ランブック  │
            │                                            │
【ガバナンス】 ─── 全段階に影響 ────────────────────────────┤
    RAIDログ・アーキテクチャ決定記録                         │
            │                                            │
【アシュアランス】 ─── 継続的活動 ──────────────────────────┘
    テスト計画・侵入テスト・コンプライアンス監査
```

### 5.2 成果物の詳細度調整原則

| プロジェクト特性 | 成果物の方針 |
|---------------|------------|
| 高規制環境（金融・医療・公共インフラ等） | 全成果物を詳細に作成。外部監査官への説明可能な文書化を必須とする |
| 中規模・中リスク | 「ちょうど十分」の成果物。アーキテクチャ上重要な項目に集中 |
| 低規制・スタートアップ | 代表的な取引フロー・最重要アーティファクトに絞る |

> 重要: 全取引フローを列挙する必要はない。システムの主要パスを少なくとも1回ずつカバーする**アーキテクチャ上重要なフロー**のサンプルで十分である。

### 5.3 成果物の命名と他手法との対応

本方法論で使用する成果物名は、他の多くのソフトウェアアーキテクチャ手法や産業標準と一致しない場合がある。以下の原則で命名している：

- できる限り業界で認知される一般的な名称を採用する
- 同等の成果物がない場合は、インフラ・アプリケーションアーキテクトが認識できる名称を新設する
- 既存の手法（C4モデル・TOGAF等）に重ねて使用できるよう汎用性を持たせる

## 6. ソリューションアーキテクチャ分解

ソリューションアーキテクチャは3つの層に分解して設計する。

| 層 | 内容 | 代表成果物 |
|---|------|-----------|
| **システムコンテキスト層** | システム境界・外部アクター（人・外部システム）との接続関係 | システムコンテキスト図 |
| **コンポーネント層** | システム内部の機能コンポーネントとその相互作用 | コンポーネントアーキテクチャ図・DFD・脅威モデル |
| **デプロイメント層** | 機能コンポーネントのインフラへの展開と Zero Trust 適用 | デプロイメントアーキテクチャ図・クラウドアーキテクチャ図 |

各層は論理的な視点（何を・どのように接続するか）から物理的/プリスクリプティブな視点（どこに・どの製品で実装するか）へ掘り下げられる。

## 7. QAチェックリスト運用哲学

各アーティファクトに QA チェックリストを設けることで、成果物の完全性・整合性・品質を担保する。

### 設計原則

- **「十分に良い（good enough）」の基準を定義する**: 全脅威・全フローを検証することはスコープ外。代表的なものをカバーしたら完了とする
- **コンテキストに応じた適用**: チェックリストの全項目が必須というわけではない。プロジェクトの規制対象度・リスクレベルに応じて取捨選択する
- **段階的改善**: 初期版は簡潔に、後続フェーズで詳細化する。アジャイルの「ちょうど十分・ジャストインタイム」に準じる
- **汎用化の優先**: 演習問題の設問・解答をそのまま流用するのではなく、プロジェクトで再利用可能な汎用チェックリスト形式に変換する

### チェックリストの種類と対応 references

| 種類 | 対応ファイル |
|-----|------------|
| アーティファクト全般の品質確認 | [ARTIFACT-METHOD.md](ARTIFACT-METHOD.md)（本ファイル） |
| 機能/非機能要件の品質 | [REQUIREMENTS-ENGINEERING.md](REQUIREMENTS-ENGINEERING.md) |
| 情報資産登録簿・システムコンテキスト図 | [DATA-CENTRIC-SECURITY.md](DATA-CENTRIC-SECURITY.md)・[SYSTEM-CONTEXT-AND-COMPONENTS.md](SYSTEM-CONTEXT-AND-COMPONENTS.md) |
| 脅威モデルの完全性 | [THREAT-MODELING.md](THREAT-MODELING.md) |

## 8. セキュリティアーキテクトの役割と RACI

### 8.1 役割定義

| 役割 | 典型的スキルプロファイル | 主な責務 |
|-----|-------------------|---------|
| **エンタープライズセキュリティアーキテクト** | セキュリティ全般 + EA 専門 | 組織全体のセキュリティ戦略・EA・ガイディング原則の策定。業界動向・現在の脅威ランドスケープへの深い理解と高いコミュニケーション能力が必要 |
| **ソリューションセキュリティアーキテクト** | セキュリティ全般 + 特定技術領域 | 特定プロジェクト・イニシアティブのセキュリティアーキテクチャ全般を担う。EA・ポリシー・インフラ・アプリ両方の理解が必要。エンジニア・テスター・PM との協働スキル必須 |
| **プロダクトセキュリティアーキテクト** | 特定セキュリティドメイン（IAM・脅威検知等） + 製品アーキテクチャ | セキュリティ製品・サービスのアーキテクチャ設計。製品がエンタープライズ環境にどう統合されるかの理解と、プロダクトマネジメントへの説明スキルが必要 |
| **アドバイザリ / コンサルティングセキュリティアーキテクト** | T字型（深いセキュリティ + 広いインフラ・アプリ知識） | 他アーキテクトへのセキュリティ設計支援。様々なステークホルダー（非技術者含む）と対話できる幅広いコミュニケーション能力が必要 |
| **インフラ・アプリケーションアーキテクト** | インフラ / アプリ専門 | セキュリティアーキテクト不在のプロジェクトで自らセキュリティを統合設計する |
| **セキュリティチャンピオン** | アーキテクチャ思考 + エンジニアリングスキル | Agile / DevSecOps 環境でアドバイザリセキュリティアーキテクトの役割を担い、開発者にセキュアコーディングを助言する |

### 8.2 文脈的役割（Contextual Roles）

全役割が方法論のエンドツーエンドプロセスとアーティファクトを理解している必要があるが、各成果物への関与（リード / 支援 / 確認）は以下の文脈によって変動する：

- **組織規模**: 大企業では役割分担が明確、スタートアップでは一人が複数役割を兼務
- **プロジェクト性質**: セキュリティサービス構築 vs 一般アプリへのセキュリティ統合
- **規制環境**: 高規制業種では EA とソリューション SA の連携が不可欠
- **Agile vs ウォーターフォール**: Agile 環境ではセキュリティチャンピオンの役割が重要

### 8.3 RACI の考え方

RACI（Responsible / Accountable / Consulted / Informed）は、誰がどの成果物に対してどう関与するかを明確化する。

| RACI 要素 | 意味 | セキュリティアーキテクチャでの適用 |
|---------|-----|--------------------------------|
| **R（担当）** | 実際に作業を実施する | 成果物を作成・更新する担当者 |
| **A（説明責任）** | 成果物の品質・完成に最終責任を持つ | 通常はアーキテクト（作業者と同一の場合も多い） |
| **C（相談）** | 専門知識を提供・意見を求められる | セキュリティ、コンプライアンス、インフラの専門家 |
| **I（情報共有）** | 決定結果の通知を受ける | PM・エグゼクティブ・運用チーム |

> **重要**: セキュリティアーキテクトの責任は「成果物の所有」だけでなく「成果物のデリバリーの担保」にある。自分で作成しない成果物（例: 運用手順書の作成はセキュリティ運用チーム）でも、その品質と納期に責任を負う場合がある。

## 9. アーティファクト依存関係ダイアグラムの使い方

アーティファクト依存関係ダイアグラムは方法論全体の「道路地図」として機能する。

**主な使い方**:
1. **プロジェクト開始時**: どの成果物が必要か、どの順序で作成すべきかを把握するロードマップとして活用
2. **進捗確認**: どの成果物が完成し、次は何を作るべきかを追跡
3. **スコープ調整**: プロジェクトの規制環境・リスクレベルに応じて、必要な成果物サブセットを特定
4. **担当割り当て**: 各成果物の所有権と RACI をダイアグラムに重ねて可視化

**アーティファクト間の依存関係の例**:
- システムコンテキスト図 → 情報資産登録簿（システム境界が決まって初めて資産が定まる）
- 情報資産登録簿 → 脅威モデル（保護すべき資産が明確になって脅威の優先順位付けが可能）
- コンポーネントアーキテクチャ図 → 脅威モデル（コンポーネント構成が決まって詳細な脅威分析が可能）
- 脅威モデル → テスト計画（特定された脅威に対するテスト戦略を設計）
- 脅威モデル → 脅威検知ユースケース（脅威が特定されて初めて検知シナリオを設計できる）
