# セキュリティ要件工学

セキュリティアーキテクトが情報システムに必要なセキュリティ要件を抽出・記述・精緻化し、トレーサビリティを確保する手法を体系的にまとめる。

## 1. 要件の基本概念

### 1.1 機能要件（Functional Requirements）

機能要件は「システムが**何を**実現するか」を規定する。具体的には一連の処理フロー・入出力・データ処理の振る舞いを記述し、そのシステムが存在する第一義的な目的を満たす機能を定義する。

セキュリティの観点では、機能要件は以下の問いを含む:

- どの操作がセキュリティ上の判断を伴うか？
- 操作の証跡（監査ログ）を残す必要があるか？
- 不審な活動を検知するための処理を追加すべきか？

> **セキュリティアーキテクトの役割**: 機能要件の収集自体はビジネスアナリストやコンサルタントが担うことも多い。ただし、セキュリティアーキテクトは完全で高品質な要件を確保する責任を持つ。セキュリティ上有害な機能要件を発見した場合は積極的に修正を提案する（例: 多要素認証の必須化など）。

### 1.2 非機能要件（Non-Functional Requirements）

非機能要件は「システムが**どのように**機能を提供するか」を規定する。別名「アーキテクチャ特性」や「品質特性（ISO/IEC 25010）」とも呼ばれる。セキュリティ設計に重要な非機能要件のカテゴリ:

| カテゴリ | セキュリティへの影響 |
|---------|-------------------|
| セキュリティ | CIA（機密性・完全性・可用性）の保護要件。ポリシー・標準・規制から多数派生する |
| プライバシー | 個人情報保護規制（GDPR等）への対応。セキュリティと密接に連動 |
| スケーラビリティ | セキュリティサービスが処理量増大に追随できない場合、可用性に影響（例: コンテナ環境での特権パスワード取得頻度の急増） |
| 可用性 | クラウドネイティブ環境ではセキュリティサービスの可用性が依存アプリの可用性を上回る必要がある |
| 回復性（Recoverability） | RPO・RTO の明確化が必須。セキュリティサービスはアプリより先に復旧させる必要がある |
| 使いやすさ（Usability） | 使いにくいセキュリティ制御はユーザーが迂回策を採るリスクを生む |

**RTO と RPO の定義**

- **RPO（Recovery Point Objective）**: 障害発生時に許容されるデータ損失の最大範囲。非同期レプリケーションは RPO がゼロにならない点を考慮する
- **RTO（Recovery Time Objective）**: 障害からサービス復旧までの許容時間。依存アプリの RTO を達成するため、セキュリティサービスはより短い RTO を持つ必要がある

> **「鍵が車内に閉じ込められる」パターン**: 回復シーケンスの循環依存に注意。例えばファイアウォール再起動に HSM（ハードウェアセキュリティモジュール）への通信が必要だが、その通信がファイアウォールに依存している場合、復旧が詰まる。シークレット管理・証明書管理でも同様の循環依存が起きる。

### 1.3 制約（Constraints）

制約は法規制・現行 IT 環境・選定ソフトウェアから生じる「回避できない条件」となる。主な発生源:

| 発生源 | 具体例 |
|-------|-------|
| ソフトウェアバージョン依存 | 旧バージョンのライブラリにしか対応していないアプリが他のアップグレードを阻む（脆弱性が蓄積） |
| セキュリティプロトコル | TLS 1.3 要求に対して TLS 1.2 までしか対応していない場合、IPsec VPN 等の補完策が必要 |
| API バージョン | クラウドサービスが旧 API を廃止する際の強制的な移行コスト |
| エージェント非互換 | EDR（Enterprise Detection and Response）等のセキュリティエージェントが旧 OS や専用アプライアンスに対応しない |

制約の発見は設計初期に行うほど手戻りコストが下がる。新たな制約は RAID ログ（リスク・課題・前提・依存）への記録対象とする。

## 2. 要件の品質基準 — SMART フレームワーク

要件の品質を担保するために SMART 基準を用いる。全要件が **Specific・Measurable・Relevant** を満たすことを必須とし、**Attainable・Time-bound** は プロジェクトの全体像が見えてから判断する。

| 基準 | 意味 | セキュリティ文脈での適用 |
|-----|------|----------------------|
| **S**pecific（具体性） | 明確・一貫・曖昧さなし。5W（誰が・何を・いつ・どこで・なぜ）で詳細化 | 「アラート対象」「通知先」「検知基準」を明記する |
| **M**easurable（測定可能性） | 実装の検証が可能な形式。テストを書けるか？ | SLA（例: 15分以内にアラート発報）など数値で表現する |
| **A**ttainable（達成可能性） | 技術的・制約的に実現できること。不確かな場合は RAID ログに記録 | SME（技術専門家）レビューを通じて実現性を確認する |
| **R**elevant（関連性） | ビジネス目標・リスク許容度に整合。余分なセキュリティ要件を追加しない | 組織のリスク許容度に基づき必要なコントロールに絞る |
| **T**ime-bound（時間軸） | いつその能力が必要か明示。ADR で根拠を文書化 | 「クライアントデータ移行前」など具体的マイルストーンと紐づける |

**良い要件の例:**

```
NFR_SEC_TD_001: 脅威検知システムは、クライアントデータが存在する前に稼働し、
ランサムウェアを示す可能性のあるイベントパターンを検知してから
15分以内に SOC（セキュリティオペレーションセンター）へアラートを発報しなければならない。
```

**ID 命名規約の例:**  
`NFR_SEC_TD_001` = 非機能要件（NFR） / セキュリティ（SEC） / 脅威検知（TD） / 連番（001）

## 3. 要件の優先順位付け — MoSCoW 手法

長大な要件リストをスプリントや MVP（Minimum Viable Project）に割り当てるため、MoSCoW 手法を用いて優先度を分類する:

| ラベル | 意味 |
|-------|-----|
| **M**ust have | このリリースで必ず実装する。未達ではリリース不可 |
| **S**hould have | 期間内に実装を試みるが、必達ではない |
| **C**ould have | 実装できれば良いが、通常は未実装になる |
| **W**ould / **W**on't have | 将来への言及または明示的な除外（コスト・複雑性を理由に実装しないと明示することも価値がある） |

スプリントが進むにつれてラベルを見直し、優先度の変化を記録する。

## 4. 機能要件の記述技法

### 4.1 ユースケース（Use Cases）

ユースケースは人間・システムアクターとシステムのやり取りを記述する。UML ユースケース図でシステム境界・アクター・ユースケースを可視化し、以下のテンプレートで詳細を文書化する:

| 項目 | 説明 |
|-----|-----|
| ユースケース名 | 処理の名称 |
| 説明 | 何を達成するか |
| アクター | 関与する人間・システムアクター |
| 事前条件 | 処理開始の前提 |
| フロー | ステップ番号付きの処理手順 |
| 事後条件 | 処理完了後の状態 |
| 例外 | 失敗シナリオと対処 |
| 要件 | 関連する機能要件 |

セキュリティ関連ユースケースはシステムへのログインから始まりセキュリティインシデント報告まで多岐にわたる。すべてのセキュリティ関連アクターを特定し、ユースケースを文書化することが重要。

### 4.2 ジャーニーマップ（Journey Maps）

ジャーニーマップは人間中心設計（デザイン思考）の手法で、ペルソナ（アクター）の行動・感情・ペインポイントを段階的に可視化する。フェーズ・ステップ・感情・ペインポイントをワークショップで付箋を用いて構築するのが典型的な進め方。

目的: システム設計の前に「ユーザーの立場」を理解すること。詳細な要件定義の前段として適しているが、それ単体では実装に必要な詳細が不足するため、後続の技法と組み合わせる。

### 4.3 ユーザーストーリー（User Stories）

アジャイル開発における機能要件の表現形式:

```
As a <役割>, I want <機能・操作>, so that I <ビジネス価値・目的>.
```

構成要素:
- **User story 本文**: As a ～ / I want ～ / so that ～
- **コンテキスト**: 背景情報・要件の発生経緯
- **受け入れテスト**: 実装完了を確認するテストシナリオ

ポイント:
- 「As a user」のような抽象的なロールは避け、具体的なロール（例: 「コンプライアンス担当者」）を指定する
- セキュリティ要件は「I want ～」ではなく「I'm required ～」と書くことが適切な場合がある
- 複数スプリントにまたがる大きなユーザーストーリーは「エピック（Epic）」として管理し、サブストーリーへ分解する
- 関連するストーリーとエピックは「テーマ（Theme）」としてグループ化する

開発チームはプライマリビジネスユーザー以外（コンプライアンス担当者・セキュリティ運用者等）を見落としがち。システムコンテキスト図でこれらのアクターを明示することで、コンプライアンス要件が主要機能のひとつであることを認識させる。

### 4.4 スイムレーン図（Swimlane Diagrams）

複数アクター間の処理フロー・引き継ぎ・判断分岐・職務分掌を視覚化するダイアグラム。各アクターの「レーン（行または列）」に処理ステップを配置する。

**主要な構成要素:**

| 要素 | 役割 |
|-----|-----|
| アクター | プロセスに関与する人間・システムアクター |
| スイムレーン | アクターが担当する処理ステップの領域 |
| 終端（Terminator） | プロセスの開始または終了 |
| 処理ステップ | アクターが行う活動 |
| 判断（Decision） | Yes/No などの分岐点。ポリシー判断では監査ログへの記録が必要 |
| サブプロセス | 複雑な処理ステップを別図へ分解 |
| ステップ番号 | 左→右 または 上→下 の時系列順に番号付け |

スイムレーン図単体では情報が不足するため、アクター・番号・タイトル・説明を記載した補足テーブルをセットで作成する。

セキュリティ判断が発生するすべてのプロセスについてスイムレーン図の作成を推奨する。

### 4.5 職務分掌マトリクス（Separation of Duties Matrix）

スイムレーン図で定義したプロセスステップに対して、どのロールが「どの組み合わせのステップを実施してはいけないか」を可視化するマトリクス。

構成:
- 行・列に処理ステップ ID を配置
- 交点に以下の記号を記入:
  - `X`: このロールがこの2つのステップを組み合わせて実施するのは**高リスク・禁止**
  - `*`: **低リスクだが避けることが望ましい**（やむを得ない場合は許容）
  - チェック（✓）: **実施可能**

典型例: 権限アクセスを申請したユーザーが自分の申請を承認したり、権限を自己付与してはならない。

## 5. 非機能要件の記述

### 5.1 要件の出所

非機能要件は以下の3つの文脈から導出する:

```
外部文脈（法規制・ISO/IEC 27001・業界ベストプラクティス・脅威ランドスケープ）
    ↓
内部文脈（組織ポリシー・標準・ガイドライン・ガイディング原則・リスク管理）
    ↓
プロジェクト文脈（目標・スコープ・予算・工数・スキル・タイムライン）
    ↓
ソリューションアーキテクチャ内のセキュリティコントロール
```

プロジェクト文脈はコスト・時間とスコープ・品質・リスクのバランスを要求する（プロジェクトマネジメントの三角形）。全要件を満たすことがコスト・期間の制約上困難な場合、より低コスト・短期実装が可能な代替コントロールを検討する。

また、**暗示的（Implicit）な非機能要件**にも注意する。例えば、依存アプリの RTO が 8 時間なら、セキュリティサービスの RTO はそれより短くなければならない。

### 5.2 非機能要件の依存関係

既存のセキュリティサービスが新規ワークロードの要件を満たすか事前に確認する:

- 対応 OS・ミドルウェアバージョンは揃っているか？
- EDR・アンチウイルス等のエージェントがサポートされているか？
- データベースのアクティビティ監視ソリューションは対応しているか？
- クラウドサービス固有のアクセス制御要件との整合性はあるか？

ハイブリッドクラウド環境ではソフトウェアバージョン確認を設計初期に実施することが特に重要。後段で判明すると、クラウドサービスから VM 上のソフトウェアへの変更など、アーキテクチャの大幅な見直しを強いられる。

### 5.3 非機能要件の文書化

要件を一覧化するスプレッドシート/テーブルを以下の列構成で作成する:

| 列 | 内容 |
|----|-----|
| NFR ID | ドメイン・カテゴリ・連番を組み合わせた一意識別子 |
| ドメイン | セキュリティドメイン（例: IAM / PAM） |
| カテゴリ | より細かい分類 |
| 優先度 | MoSCoW ラベル |
| 要件本文 | SMART 基準を満たした記述 |
| 外部参照 | 出所となる規制・フレームワーク文書への参照 |
| 解決策 | 採用するセキュリティコントロール・サービス |
| サービスオーナー | 実装・運用の責任者 |

## 6. 要件カタログの精緻化

複数のソースから集めた要件は品質・形式がバラバラになりがち。繰り返し利用できる高品質な要件カタログを作成するために以下の手法を順次適用する:

| 手法 | 説明 |
|-----|-----|
| **並べ替え（Reorder）** | 後続の要件が前の要件を前提とする場合、論理的な順序に並べ替える |
| **分割（Split）** | 「and」を含む複合要件を個別要件に分解して担当者を明確化する |
| **統合（Merge）** | 実質的に同一の要件を1つにまとめる（統合前にニュアンスの差異を確認） |
| **再分類（Recategorize）** | 誤ったドメイン/カテゴリに割り当てられた要件を適切な責任者に再マッピング |
| **サブ分類（Subcategorize）** | 大量の要件を共通テーマでグループ化して管理しやすくする |
| **次元化（Dimensions）** | 要件が適用される技術スコープを明示する（例: ネットワーク・サーバー・コンテナ・クラウドプラットフォーム等） |
| **パラメータ化（Parameterize）** | 組織ごとに設定が異なる値（スキャン頻度・通知時間等）を要件本文から分離して管理する |
| **ブランド非依存化（Non-align）** | 特定製品名・ブランド名を含む要件を製品中立的な表現に書き換える |

> 注意: 改善後の要件は元の要件との**マッピングを常に維持**する。分割・統合・書き換えで元要件との対応が崩れないよう定期的にバックマッピングを確認する。

### 実践例: NIST SP 800-53 の要件を精緻化するフロー

1. **NIST CSF でカテゴリを特定** → ハイレベルの機能/カテゴリ/サブカテゴリを確認
2. **NIST SP 800-53r5 コントロールを参照** → より詳細な制御要件を入手
3. **分析**: 1つのコントロールに複数の要件・次元・パラメータが混在していることが多い
4. **分解**:
   - 制御要件（Control Requirements）: 実施すべき行動
   - 次元（Dimensions）: スキャン対象などのスコープ
   - パラメータ（Parameters）: 頻度・時間などの設定値

**例（脆弱性管理の精緻化）:**

| 分類 | ID | 記述 | コントロールマッピング |
|-----|-----|-----|---------------------|
| 制御要件 | VM-01 | 脆弱性管理プロセス・手順を定義・実装する | NIST SP 800-53 RA-5(a) |
| 制御要件 | VM-02 | 脆弱性スキャンを {VM-P-01,02,03,04} の頻度でコンポーネント種別 {VM-D-01} に対して実施する | NIST SP 800-53 RA-5(a) |
| 制御要件 | TIM-01 | システムコンポーネントの脆弱性通知を脆弱性管理サービスへ {TIM-P-01} 以内に通知する | NIST SP 800-53 RA-5(a) |
| 次元 | VM-D-01 | ネットワーク機器・物理/仮想サーバー・コンテナ・クラウドプラットフォーム・コンテナプラットフォーム・アプリケーション | NIST SP 800-53 RA-5(a) |
| パラメータ | VM-P-01 | 週次以上の頻度 | NIST SP 800-53 RA-5(a) |
| パラメータ | VM-P-02 | 重要なシステム変更後 | NIST SP 800-53 RA-5(a) |
| パラメータ | VM-P-03 | 脆弱性通知から 24 時間以内 | NIST SP 800-53 RA-5(a) |
| パラメータ | VM-P-04 | ペネトレーションテスト実施時 | NIST SP 800-53 CA-8 |
| パラメータ | TIM-P-01 | 24 時間以内 | NIST SP 800-53 RA-5(a) |

> 注意: NIST CSF のファンクション/カテゴリとセキュリティサービスのドメインは必ずしも一対一対応しない。脆弱性管理サービスに関する要件が DETECT（検知）と PROTECT（保護）の両ファンクションにまたがるように、フレームワークのコントロールファミリーをそのままオーナー割り当てとして使うことはできない。

## 7. 要件トレーサビリティマトリクス

要件が設計・実装・テスト・運用ドキュメントにどう反映されたかを追跡するマトリクス:

| 要件 ID | 要件本文 | 高レベル設計（HLD） | 低レベル設計（LLD） | テスト計画 | 運用ドキュメント |
|--------|---------|-----------------|-----------------|----------|----------------|
| SEC_IAM_PAM_003 | 特権クレデンシャルのページ応答時間は極高負荷時でも 5 秒以内とする | 4.2 特権アクセス管理 | 11.3 特権アクセス管理 | 6.5 ユニット / 7.8 統合 / 9.7 回復性テスト | 運用マニュアル 5.6 緊急ブレークグラス VPN 手順 |

マトリクスの最小列構成: **要件 ID / 要件本文 / 高レベル設計 / 低レベル設計 / テスト計画 / 運用ドキュメント**。プロジェクト規模に応じて列を追加する。

### 要件インデックス化のヒント

設計文書内に要件 ID を `[SEC_IAM_PAM_003]` のようにインラインで記載するとドキュメントの索引化が可能になり、全要件のカバレッジ検証が容易になる。

## 8. QA チェックリスト

### 機能要件チェック

- [ ] システムコンテキスト図から特定されたユースケースに対してユースケース記述が存在するか
- [ ] ジャーニーマップ・ユーザーストーリーにセキュリティ制御を含む活動/ストーリーが識別されているか
- [ ] セキュリティを執行するプロセスにスイムレーン図と職務分掌マトリクスがあるか
- [ ] ビジネスプロセスの所有権がビジネスプロセスオーナーに帰属しているか（セキュリティアーキテクトが定義を独占しない）

### 非機能要件チェック

- [ ] 外部文脈・内部文脈・プロジェクト文脈の3ソースが全て考慮されているか
- [ ] 依存アプリ・コンポーネントのソフトウェアバージョン依存・ワークロード固有要件・容量要件が洗い出されているか
- [ ] 各要件は SMART 基準を満たしているか
- [ ] 各要件に MoSCoW 優先度ラベルが付与されているか
- [ ] 要件 ID に一意識別子・出所ラベルが含まれているか
- [ ] 次元（適用スコープ）が明示されているか
- [ ] パラメータ（組織固有の設定値）が外部化・管理されているか
- [ ] コスト・スコープ/品質/リスク・時間のバランスが考慮されているか
