# アプリケーションセキュリティ（クラウド環境）

クラウドにホストされるアプリケーション（自社開発・SaaS 利用の双方）に対する脅威の分類と、CASB / ZTNA / SSE / SASE といったクラウド前提のセキュリティコントロール、アウトソーシング時のガバナンス、Web アプリケーションの多層防御、API 利用時の認証・認可・レート制限・キー管理をまとめた実践リファレンス。責任共有モデルの下では、データ・ユーザー・ポリシーの責任は常に自組織に残る——この前提がすべての判断基準になる。

---

## 前提: 責任共有モデルと責任の残余

アプリケーションをどこにホストしても、ビジネスとして顧客・従業員・データに対する最終責任は移転できない。

| 委譲できるもの | 委譲できないもの |
|---|---|
| 物理インフラ・データセンター運用 | 自組織のデータの分類・保護方針 |
| マネージドサービスの基盤パッチ | ユーザー ID とアクセス権の管理 |
| アプリケーションのホスティング | セキュリティポリシーの策定・執行 |
| （SaaS の場合）アプリ自体の脆弱性修正 | 自社開発アプリのセキュア SDLC 全体 |

**判断基準**: 「プロバイダーがやってくれているはず」と推定した瞬間にギャップが生まれる。コンポーネントごとに責任分界を文書化し、パッチ・ログ・暗号化の担当を明示する。

## アプリケーションセキュリティ脅威

ホスト場所が変わっても脅威の本質とビジネスインパクトは変わらない。クラウドでは顕在化の仕方が変わるだけである。

### 1. ID・アクセス管理（IAM）の脅威

認証情報への攻撃は依然として最上位の侵入経路。

| 攻撃・弱点 | 内容 | 対策 |
|---|---|---|
| 弱いパスワード | 短い・頻出パスワードは辞書攻撃で即時突破される | 長いパスフレーズ + パスワードマネージャー |
| ハッシュのオフライン解析 | 漏洩ハッシュは GPU で数十億件規模の候補を秒単位で照合可能 | ソルト付き低速ハッシュ（bcrypt/argon2 等）・漏洩監視 |
| レインボーテーブル | 事前計算済みハッシュ DB による即時ルックアップ（計算をストレージと交換） | ソルト必須化 |
| パスワード使い回し | 1 サービスの漏洩が他アカウントの侵害に連鎖 | MFA・漏洩パスワードチェック |
| ソーシャルエンジニアリング | フィッシング等で認証情報自体を詐取 | MFA（特にフィッシング耐性のある方式）・教育 |
| 特権クレデンシャルの悪用 | 一般アカウントと同じ弱点で、影響は桁違いに大きい | パスワードボールト・Just-in-Time アクセス |

**落とし穴**:
- 「複雑なパスワードの定期強制ローテーション」は、多数の複雑な認証情報を回し続けられないユーザーに悪い運用（メモ・使い回し）を強いる逆効果になり得る。管理しやすい強い認証 + MFA を優先する。
- 認証情報の侵害がなくても、過剰な権限付与だけでデータ喪失は起こる。権限の適正化はそれ自体が脅威対策。
- IAM の具体的機能はプラットフォームごとに異なるため、特定技術に依存しない「ID ライフサイクル全体を守るポリシーとプロセス」を先に固める。

### 2. 設定ミス（Misconfiguration）

クラウドの設定項目の多さ・複雑さが設定ミスの温床になる。

| 典型パターン | リスク | チェックポイント |
|---|---|---|
| ストレージバケットの公開設定 | パブリック IP + アクセス制御なしで内容が全公開 | 外部から直接到達する必要が本当にあるか。デフォルト非公開か |
| 暗号化の未設定・弱い方式 | 保存データ・通信の露出 | 必須箇所で有効か。廃止されたプロトコルを許可していないか |
| 鍵・パスフレーズの平文保存 | 暗号化を無効化するのと同義 | 鍵を解錠するパスワードの保管場所と保護 |
| デフォルトクレデンシャル | オンラインで検索可能な初期認証情報の放置 | 導入時の変更を手順化。棚卸しで残存を検出 |

### 3. 脆弱性管理の不備

- クラウド移行しても脆弱性管理はなくならない。一部はプロバイダー側の責任になるが、自分が担当する層は自分で監視・修正する。
- 自社開発では、アプリ本体だけでなく**依存ライブラリ・コンテナベースイメージ**の更新プロセスが必須。
- 脆弱性管理は技術ではなく**プロセス**の問題。修正パッチが他ソフトウェアとの非互換を生むリスクとのトレードオフを扱う体制（テスト・依存関係の把握）が要る。

### 4. ロギング・監視の不足

| 問題 | 内容 |
|---|---|
| ログ収集の欠落 | SIEM が容量課金のため収集対象を絞りがち。攻撃者の潜伏期間を遡れる保持期間を確保する |
| 責任分界の誤解 | 「クラウド側がログを取っているはず」という推定でギャップが生じる |
| 収集だけで監視なし | 人力でログを目視するのは不可能。アラート生成の仕組みが必須 |
| アラート疲れ | 価値の低いイベント（例: ping スイープ検知）を大量アラート化すると、本物のインシデントを見逃す |

**MITRE ATT&CK の活用と注意**（https://attack.mitre.org/）:
- ATT&CK は既知の攻撃者の tactics（ステップの目的）/ techniques（手法）/ procedures（具体的実装・ツール）を体系化したナレッジベース。tactic は 14、technique は 200 超、subtechnique はさらに多い。
- 全 technique を検知ルール化しようとするとルール数もアラート数も爆発し、SOC を疲弊させる。**環境と脅威モデルに関係する項目を選別**して段階導入する。
- 可視性は**スタック全体**で設計する: ネットワーク（ファイアウォール・セキュリティグループ・IDS）からアプリケーションログまで。特に**認証・アクセス制御に関するイベント**を優先。

### 5. セグメンテーション不足

- フラットネットワークはラテラルムーブメント（east-west 移動）をほぼ制限できない。
- 従来の n-tier（Web / アプリ / DB を別セグメント）に相当する分離は、マイクロサービス・サーバーレスでも依然必要。
- VLAN を分けるだけでは不十分。**セグメント間トラフィックを制御・制限できること**（最低 ACL、望ましくはファイアウォール）が本質。
- 分割基準は**信頼レベル / データ機微度**: 公開データのシステムと PII を持つシステムを同一セグメントに同居させない。
- ホストベースファイアウォールは中央管理がないと各ホストのルールが把握不能になり穴を生む。**マイクロセグメンテーション**（ホスト単位の分離 + 中央ポリシー管理 + アプリ単位の通信許可）で補う。

### 6. インフラ設定ミス

- **パッチ管理**は脆弱性対応に限らない。サポート切れバージョンの放置自体がリスク。依存関係更新のコスト（テスト・改修）を開発コストに織り込む。
- ファイアウォール / セキュリティグループの過剰許可: 大規模環境ではどの通信がどの境界を跨ぐべきか追跡困難になり、不要なポート開放が累積する。
- **DNS の設定ミス**: ネットワーク構成の過剰な露出や、**サブドメインテイクオーバー**（放置されたレコードを攻撃者が乗っ取り自分の IP を配信）を許す。

## クラウドのセキュリティコントロールメカニズム

境界（ペリメーター）が企業ネットワークからすべてのエンドポイントへ拡張した前提で、クラウドから配信するコントロール群。

| コントロール | 一言で | 主目的 |
|---|---|---|
| CASB | エンドポイントとクラウドの間でポリシー執行 | シャドー IT 対策・クラウド利用の統制と可視化 |
| ZTNA / ゼロトラスト | 「決して信頼せず常に検証」 | 内部 = 信頼という前提の廃止。ID ベースのアプリアクセス |
| SSE | セキュリティ機能のクラウド配信バンドル | 場所を問わず同一の保護（SWG + CASB + ZTNA + FWaaS） |
| SASE | SD-WAN + SSE の統合パッケージ | ネットワークとセキュリティの一体的なクラウド配信 |

### CASB（Cloud Access Security Broker）

- **課題**: クラウドは導入が容易すぎるため、部門が IT を介さず利用を始め（シャドー IT）、データがセキュリティ・コンプライアンスの統制外で処理される。ファイアウォールでクラウドを一律遮断はできない（正当な利用がある）。
- **実装方式**: ①プロキシ型（オンプレ or クラウドに配置しトラフィックを強制通過）②エージェント型（各エンドポイントに導入。プロキシの制約を回避）。
- **できること**:
  - どのユーザーがどのクラウドプロバイダー・**どのテナント**にアクセスできるかの粒度制御（時間帯ベース等の細粒度制御を含む）
  - 全社ポリシーの一元執行（リモートワーカーも同一統制下）
  - 複数クラウドを横断したアクティビティログの集約による可視化
  - 規制・法令遵守の支援（CASB の統制範囲内で）

### ゼロトラスト / ZTNA

- 「内部にいれば信頼できる」は常に誤謬だった。外殻だけ固く中は無防備な構造（境界型・城と堀モデル）は defense in depth ではない。VPN 経由も同罪。
- 原則は **never trust, always verify**（2010 年頃に業界で提唱された概念）。
- 実装の核は **ID**:
  - 強い認証（可能な限り MFA）
  - デバイスへのログインで終わらせず、**ID とアプリケーションを直接紐付ける**（アプリアクセスは常にユーザー ID とアクセス権に基づく）
  - 最小権限の原則が前提
  - マイクロセグメンテーションを併用し、各エンドポイントを独立セグメントのように扱う実装もある
- ベンダーにより「ゼロトラスト」の実装内容は大きく異なるため、製品名でなく上記原則で評価する。

### SSE（Security Service Edge）

セキュリティサービスをクラウドから配信し、ユーザーの所在に依存しない保護を提供する。コア構成要素:

| 要素 | 役割 |
|---|---|
| Secure Web Gateway（SWG） | Web トラフィックのカテゴリフィルタリング・マルウェアスキャン・暗号化トラフィックの検査。バイパス防止（全 Web トラフィックの強制通過） |
| CASB | クラウドアクセスの組織ポリシー準拠を保証 |
| ZTNA | ユーザー⇔アプリ間のすべての通信に強い認証を要求 |
| Firewall as a Service（FWaaS） | 次世代ファイアウォール機能を全拠点・全リモートユーザーに配信 |

**価値**: 全トラフィックを本社にバックホールする必要がなくなる。全エンドポイントが**同一のクラウド配信サービス**から保護を受けるため、拠点ごとの機器差異が消え、中央から一貫したポリシー統制ができる。

### SASE（Secure Access Service Edge）

- **SD-WAN**（インターネット回線上にトンネルをオーバーレイして安価に構築する仮想 WAN。edge / gateway / orchestrator / controller で構成）を核に、SSE 系のセキュリティ機能（NGFW・CASB・SWG・IDS・アンチマルウェア等）をバンドルした概念。2019 年に業界アナリストが提唱した用語で、本質的にはマーケティング用語だが、ベンダー製品カテゴリとして定着している。
- MPLS 等の専用接続より安価にサイト間接続と回線の冗長性を得つつ、全拠点に同一のセキュリティを配信できる。
- クラウド専業構成である必要はなく、オンプレ要素を含む**ハイブリッド構成**も可能。目的は「全拠点を等しく守る中央管理型の統合」。

**選定の指針**: 個別製品の寄せ集めか統合プラットフォームかより、「全ユーザー・全拠点で同一ポリシーを中央から執行できるか」「既存の ID 基盤・ログ基盤と統合できるか」で評価する。

## アプリケーションのアウトソーシング / SaaS 利用

アプリケーションの管理を委託しても、データとユーザーの統制責任は残る。契約前に検討すべき観点:

### データガバナンス

| 観点 | 問い |
|---|---|
| 所在とデータ主権 | データはどこに**保存**され、どこで**処理**されるか（両者は一致しない場合がある）。どの法域の法律が適用されるか |
| 分類と取り扱い | 自組織のデータ分類ポリシーと取り扱い要件（例: 保存時暗号化必須）をプロバイダーが遵守できるか |
| 保持・破棄 | 保持期間・破棄ポリシーへの準拠 |

### ID 管理の設計

- プロバイダー側にユーザーを作らせる方式は、**ID 保護の負担をプロバイダーに委ねる**ことになる。ID リポジトリは常に攻撃対象であり、すべてのプロバイダーが強固な保護を実装しているわけではない。
- 可能なら自組織の **IdP（Identity Provider）**を使うフェデレーションを選ぶ:

**SAML フローの要点**（OAuth / OIDC でも考え方は同様）:
1. ユーザーが SaaS アプリにアクセス → 認証済みトークンがないためアプリが認証要求を返し IdP へリダイレクト
2. ユーザーは **IdP と直接**認証する（MFA を含む強い認証。SaaS 側は認証情報に触れない）
3. IdP が署名付きアサーション（ID + 属性）を発行し、アプリはそれに基づき適切なアクセスレベルを付与

利点: 認証フローが標準プロトコルで保護され、クレデンシャルが中間者に晒されない。認証ポリシー（MFA 等）を自組織で一元統制できる。

### 規制・契約・ベンダー管理

- GDPR 等の privacy 規制を、データガバナンスと合わせて適用法令の棚卸しに含める。
- **要求事項は必ず契約書面にする**: データ取り扱い要件・監査権（right to audit）・インシデント通知義務・要件未達時の是正/解約条項。
- 大手プロバイダーには交渉余地が小さくても、**ベンダー審査ポリシー**自体は持つ: 各ベンダーにデータ管理・脆弱性管理等の重要プラクティスへの回答を求めるセキュリティスコアカードを作り、**少なくとも年 1 回**再評価する。
- ベンダー管理は継続的活動であり、セキュリティ部門と法務の共同管掌にする。

## Web アプリケーションセキュリティ

クラウドでは多層防御スタックを低コストで揃えられる。リクエストの流入経路に沿った標準構成:

```
ユーザー → CDN → WAF → ロードバランサー → Web サーバー → アプリサーバー → DB
          (DDoS 吸収)  (攻撃検知)   (負荷分散/TLS 終端)     (静的)      (動的処理)   (データ)
```

| 層 | セキュリティ上の役割 | 注意点 |
|---|---|---|
| CDN | 地理分散による可用性 + 大規模 DoS/DDoS の吸収 | オリジン直撃を防ぐ設定（オリジンの直接公開を避ける） |
| WAF | HTTP を検査し既知の攻撃パターンを遮断。OWASP Top 10（https://owasp.org/www-project-top-ten/）系の脆弱性悪用を検知 | WAF は緩和層であり、アプリ側の脆弱性修正の代替ではない |
| ロードバランサー | 負荷分散。TLS 終端を担うことが多い | **終端後は内部が平文になる**。内部区間の保護（再暗号化 or 厳格なセグメント分離）を設計判断として明示する |
| Web / アプリ / DB | n-tier の役割分離 | 各 tier を**別ネットワークセグメント + セキュリティグループ**（別ゾーン）に置き、機微度の異なるシステムを分離 |

### 横断的に必須の要素

ユーザーから見えない部分こそ攻撃者の狙い目になる:

- **ロギングと監視**: すべての Web アプリに必須。認証・アクセス制御イベントを最優先で収集。
- **ID 管理**: エンドユーザーだけでなく**運用管理者のアクセス**も対象。
- **シークレット管理**: システム間認証の保護。アプリサーバーに DB 接続情報を平文で置くのは、セグメント分離を実質無効化するアンチパターン（攻撃者はそのクレデンシャルで目的のデータに直行できる）。シークレットストア / ボールトを使う。
- **鍵管理**: 暗号鍵の保管・ローテーションを担う専用の仕組み（KMS 等）。

この構成は n-tier を例にしているが、**サーバーレス・マイクロサービス・API 構成でも同じ原則**（層ごとの検査・ゾーン分離・シークレット/鍵管理・可視性）が適用される。

## API 利用のセキュリティ

### 背景: なぜ API か

- HTTP はステートレス（各リクエストは独立）であり、これが負荷分散を容易にしている。REST はこの性質の上で、JSON/XML 等によりクライアント⇔サーバー間で複雑な情報をやり取りするアーキテクチャスタイル。
- API はアプリの機能をエンドポイントとして外部（他アプリ・モバイルクライアント・自アプリの別コンポーネント）に公開する。状態管理はクライアント側が持ち、サーバーはデータの保存・取得・加工に徹する構図になりやすい。
- **含意**: 機能がエンドポイントとして直接露出するため、Web ページ経由では隠れていたロジックが攻撃面になる。API は「もう一つの UI」としてフル装備のセキュリティ設計を要する。

### 認証（Authentication）

| 方式 | 適する場面 | 注意 |
|---|---|---|
| OAuth 2.0 / OIDC | ユーザー代理アクセス・SaaS 間連携 | 適切なフロー選択（公開クライアントは PKCE 必須）。トークン有効期限を短く |
| API キー | サーバー間のシンプルな識別 | 識別であって強い認証ではない。単独での機微 API 保護に使わない |
| JWT（署名付きトークン） | ステートレスなセッション伝搬 | 署名検証・`alg` 固定・有効期限検証を省略しない |
| mTLS | 内部サービス間・高機微 API | 証明書ライフサイクル管理の運用コストを見込む |

### 認可（Authorization）

- 認証の成功 ≠ 全エンドポイントへのアクセス許可。**エンドポイント単位 + オブジェクト単位**で権限を検証する（他人のリソース ID を指定するだけで読める、が最頻出の API 欠陥）。
- 最小権限: クライアントごとに必要なスコープ / エンドポイントのみ許可。
- サービス間呼び出しにも principal（サービス ID）を割り当て、人間の ID と同じ統制（監査・権限レビュー）下に置く。

### レート制限

- 目的は 3 つ: ①ブルートフォース・クレデンシャルスタッフィングの抑止 ②スクレイピング/列挙攻撃の抑止 ③過負荷（意図的・偶発的）からの可用性保護。
- クライアント識別子（API キー・トークン・IP）単位で制限し、認証エンドポイントには特に厳しい閾値 + ロックアウト/バックオフを設ける。
- 制限超過は 429 を返し、リトライ情報の出しすぎ（攻撃者への情報提供）に注意。

### キー / シークレット管理

| ルール | 理由 |
|---|---|
| コード・リポジトリ・設定ファイルにキーを直書きしない | 漏洩の最頻出経路。シークレットストア / 環境注入を使う |
| キーはローテーション可能に設計する（複数キー併存期間を許容） | 漏洩時に無停止で無効化できる |
| キーごとに権限スコープと所有者を紐付ける | 漏洩時の影響範囲特定と失効判断が速くなる |
| 発行・使用・失効をログに残す | API キーも ID の一種。IAM と同じ監査対象 |

### API セキュリティチェックリスト

- [ ] 全エンドポイントで認証を必須化（「内部用だから不要」を認めない——ゼロトラスト原則）
- [ ] オブジェクトレベル認可を全 CRUD 操作で検証
- [ ] レート制限とアカウントロックアウト（認証系は特に厳格に）
- [ ] TLS 必須。内部区間の平文化を設計として明示・最小化
- [ ] キー/トークンのローテーションと失効手順が運用に存在する
- [ ] API アクセスログを収集し、認証失敗・権限エラーの急増をアラート化
- [ ] 非推奨・未使用エンドポイントの棚卸しと廃止（シャドー API 対策）

## アンチパターン早見表

| アンチパターン | なぜ危険か | 代替 |
|---|---|---|
| 「内部ネットワークだから信頼」 | 侵入後のラテラルムーブメントを許す境界型思考 | ゼロトラスト（ID ベース検証 + マイクロセグメンテーション） |
| ストレージのとりあえず公開設定 | データ全露出の最短経路 | デフォルト非公開 + 明示的な最小許可 |
| 複雑パスワードの強制定期ローテーション | ユーザーが悪い運用に逃げ、逆効果 | 管理可能な強い認証 + MFA |
| ログは取っているが誰も見ていない | 事後の調査にしか使えない | アラート設計 + 保持期間の要件化 |
| ATT&CK 全項目の検知ルール化 | アラート疲れで本物を見逃す | 脅威モデルに基づく選別と段階導入 |
| アプリサーバーに DB クレデンシャル平文保存 | セグメント分離を無効化 | シークレットストア + 短命クレデンシャル |
| 「プロバイダーがやってくれているはず」 | 責任分界の推定はギャップを生む | 責任共有モデルの文書化と定期レビュー |
| SaaS ベンダーへの要求を口頭・実装依存で済ませる | 未達時に打つ手がない | 契約への明記（監査権・通知義務・是正条項）+ 年次再評価 |
