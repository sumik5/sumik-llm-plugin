# クラウドセキュリティの新潮流 — クラウドネイティブ・モダンIAM・ゼロトラスト・AI/ML

クラウド移行はアプリケーションの開発・提供形態を変え、セキュリティ制御のあり方も変えた。基礎（NIST Cybersecurity Framework に沿った能力整備・十分な可視性の確保）は不変だが、エフェメラルなワークロード・ソフトウェア定義の環境・認証情報を狙う攻撃の常態化により、「どう守り、どう可視化するか」の実装は大きく進化している。本リファレンスは 4 つの潮流 — クラウドネイティブセキュリティ、モダン IAM、ゼロトラストアーキテクチャ、AI/ML とセキュリティ — の実装パターン・判断基準・落とし穴を整理する。

---

## 1. クラウドネイティブセキュリティ

### 1.1 前提: 守る対象の変化

| 従来 | クラウドネイティブ |
|------|-------------------|
| モノリシック（物理/VM 上） | コンテナ・マイクロサービス・サーバーレス・マネージド DB・API ゲートウェイ・メッセージキューの分散構成 |
| 長寿命ホストの防御 | エフェメラルな（常に生成・消滅する）コンポーネントの防御 |
| 境界防御中心 | コンポーネントごとに異なる保護方式の組み合わせ |

- 抽象化の進展はコンポーネント単体の攻撃対象領域（attack surface）を縮小する一方、**守るべき構成要素の数と種類は増える**。
- エフェメラル性により、ログ・フォレンジック情報が消滅前に収集される設計（外部への転送・集約）を事前に計画する必要がある。

### 1.2 デプロイパイプラインの防御（シフトレフト）

| 対象 | 実施事項 |
|------|---------|
| IaC（Infrastructure as Code） | デプロイ前に脆弱性・設定不備をスキャン（静的解析をパイプラインに組み込む） |
| コンテナイメージ | ベースイメージ（ランタイム・最小 OS）を含めてスキャン。自作コードが健全でも同梱バイナリに脆弱性が潜む |
| 依存関係 | 言語ランタイム（.NET / Python / Java / Ruby 等）を問わず、依存パッケージの脆弱性スキャンを必須化 |

**落とし穴**: 「自分のコードはクリーン」でも、イメージに含まれるランタイムバイナリ・OS パッケージ・推移的依存が脆弱性を持ち込む。スキャン対象をアプリコードに限定しない。

### 1.3 ランタイム保護

- **カーネルレベル制御**: Linux では seccomp / SELinux / AppArmor を適用する。
- **プロセス挙動監視**: プロセスとその振る舞いを監視し異常を検出する。
- **ファイル整合性監視（FIM）**: 運用負荷は高いが改ざん検出に有効。
- **ネットワークトラフィック監視**: 通信の異常パターンから攻撃兆候を検出。
- 実施レイヤー: 可能ならコンテナ内部、最低でも**コンテナホスト側では必ず**実施する。

### 1.4 Kubernetes の多層防御

Kubernetes の防御は単一の対策では成立せず、レイヤーごとの制御を積み上げる（defense in depth）。

| レイヤー | 主な制御 |
|---------|---------|
| Infrastructure & Host | ホスト OS のハードニング・パッチ管理 |
| Cluster（API Server / RBAC） | 最小権限の RBAC。管理者権限の奪取＝イメージ差し替え・ネットワーク制御改変を許すため最重要 |
| Namespace & Network | namespace による論理分離＋NetworkPolicy |
| Workload & Pods | セキュリティコンテキスト・Pod セキュリティ標準 |
| Container & Image | イメージスキャン・署名検証・最小イメージ |

**namespace の使いどころ**:
- 関連要素をまとめ、無関係な要素を分離する論理境界（dev / staging / production の分離）。
- 同一物理基盤上のマルチテナンシー分離にも利用可能。
- アクセス制御・ネットワーク制御は namespace 単位で内包され、配下の全要素に適用される。

**Pod / Workload の信頼境界**: Pod は 1 つ以上のコンテナの単位で namespace・ストレージ等を共有し得る。Workload は関連 Pod の上位抽象で、**同一 Workload 内の全要素は同一のセキュリティコンテキスト（信頼境界・要件）を共有する**。信頼レベルの異なるコンポーネントを同一 Pod / Workload に同居させない。

### 1.5 API ゲートウェイによる保護の集約

- リクエストをエンドポイント別に適切な Pod / Workload へルーティングする抽象層。
- 認証・レート制限などの入力制御をゲートウェイに集約でき、下流のアプリコンポーネントが個別実装せずに済む。
- アプリ本体を直接露出させない防波堤として機能する。

### 1.6 基盤の基本衛生は消えない

コンテナ・サーバーレスの下には常に物理ハードウェアと汎用 OS（Linux / Windows）がある。パッチ管理・IAM・物理保護・バックアップ・監視・インシデント対応といった従来型の衛生管理は、クラウドネイティブでも（責任分界に応じて）依然として必要である。

### 1.7 CNAPP（Cloud Native Application Protection Platform）

上記の個別制御（IaC スキャン・イメージスキャン・ランタイム保護・設定監査・権限監査）を単一プラットフォームに統合する潮流。

| 構成要素 | 役割 |
|---------|------|
| CSPM | クラウド設定不備の継続監査 |
| CWPP | ワークロード（VM/コンテナ/サーバーレス）のランタイム保護 |
| KSPM | Kubernetes 固有の設定・RBAC 監査 |
| CIEM | クラウド権限（過剰な IAM 権限）の可視化・削減 |
| IaC / イメージスキャン | パイプライン段階の検査 |

**判断基準**: 個別ツールの寄せ集めはアラート分散と文脈欠如を招く。コードからランタイムまでを相関できる統合型を選ぶと、「この脆弱なイメージが、この過剰権限で、外部露出している」というリスクの連鎖を優先度付けできる。

## 2. モダン IAM（パスワードレス・ワークロードアイデンティティ）

### 2.1 パスワード運用ガイダンスの転換

| 旧来の推奨 | 現在の推奨（NIST 等） | 理由 |
|-----------|---------------------|------|
| 定期的なパスワードローテーション | ローテーション強制の廃止 | 強制変更は最小限の変更・使い回し・書き留めなど最悪の行動を誘発し、攻撃防止効果もない |
| 複雑性ルールの強制 | 長く強いパスワード＋サービス間の使い回し禁止 | 長さが強度に効く |

ただし強いパスワードでも、精巧なフィッシング 1 通で資格情報は窃取される。**根本策はパスワード自体の廃止方向**にある。

### 2.2 パスワードレス認証

原則: **資格情報（パスワード）をユーザーと ID プロバイダの間で一切送らない**。

| 方式 | 要素 | 特性 |
|------|------|------|
| 従来型 | ユーザー名＋パスワード（知識） | フィッシング・リプレイに弱い |
| パスワードレス | 生体認証 or ハードウェアキー＋PIN | 「パスワードレス」でも実態は多要素（所持＋知識/生体） |
| サービスプリンシパル | 証明書 or クライアントシークレット | 非人間主体用。対話ログイン等の人間向け権限を持たせない |

**ハードウェアトークン（FIDO2 系セキュリティキー）の要点**:
- タッチ操作で暗号材料をワンタイム生成 → 傍受されてもリプレイ不能。
- タッチ＋PIN の組み合わせ＝所持要素＋知識要素の多要素認証。
- 1 本のキーに複数アカウントを収容でき、複数クラウドプラットフォームで共用可能。
- ID プロバイダ側がトークンと同期しており、期待値を検証できる。

### 2.3 非人間アイデンティティ（サービスプリンシパル）

**背景**: アプリ・Pod・Workload・システム自体が認証主体になった。旧来の「証明書解錠用パスワードをディスクに保存」「アプリの DB パスワードを設定ファイルに記載」は、侵入者による回収リスクが大きすぎる。

| 設計要素 | 内容 |
|---------|------|
| サービスプリンシパル | アプリに紐づく専用アカウント。権限を最小限に割当て、対話ログイン等の人間向け権利を持たせない |
| 資格情報 | シークレット or X.509 証明書。**キーボールト**に保存し、実行時に権限付きで取得 |
| 認証フロー | JWT を生成（発行者 ID・対象・有効期限を含む）→ 専有証明書で署名 → ID プロバイダが署名検証 → アクセストークン発行 |

### 2.4 ワークロードアイデンティティ（SPIFFE/SPIRE）

コンテナが高頻度で生成・消滅する環境では、長期トークンの発行が原理的に不適合。サービスメッシュ型のワークロード ID がこれを解決する。

**SPIFFE（Secure Production Identity Framework for Everyone）の仕組み**:

1. SPIRE server が認証局（CA）兼 ID レジストリとして稼働。
2. ワークロード起動時、SPIRE agent が **アテステーション**（Kubernetes ServiceAccount やクラウドの IAM ロール等でワークロードの素性を検証）を実施。
3. SPIRE server が短命の X.509 証明書 **SVID**（SPIFFE Verifiable Identity Document）を発行。ID は URI 形式の SPIFFE ID で一意化。
4. SVID を用いた **mTLS** で、サービス間通信の相互認証＋暗号化を実現。

**採用効果**:

| 効果 | 説明 |
|------|------|
| シークレットレス | 管理すべき長期シークレットが消滅。キーボールト自体が不要になる |
| 自動ローテーション | 証明書は短命で自動更新。サービスインスタンスの寿命を超えて生き残る資格情報がない |
| ゼロトラスト適合 | 「同じ環境内にいるから信頼」を排し、暗号学的に検証された ID のみを信頼 |

### 2.5 分散型アイデンティティ（発展）

パスワードレスの延長として、ID 情報を中央プロバイダに集約せずユーザー自身が保持・提示する分散型 ID（DID / Verifiable Credentials）の標準化が進む。クラウド間・組織間のフェデレーションで、単一 IdP への依存と大規模漏えいの単一障害点を減らす方向性として押さえておく。

## 3. ゼロトラストアーキテクチャ

### 3.1 原則

- モットー: **「決して信頼せず、常に検証する（never trust, always verify）」**。
- 旧来の「境界内＝信頼」モデルは、リモートワークの拡大とリソースのクラウド移転で完全に破綻した。
- **侵害前提（assume breach）**: どのコンポーネントも既に侵害されている可能性を前提に、あらゆる通信で相手を確認する。
- デバイス認証＋ユーザー認証の重畳: デバイスが自動認証できても、ユーザー認証を併せて要求すれば、侵害済み端末の横断通信を抑止できる。

### 3.2 構成要素

| 要素 | 役割 |
|------|------|
| マイクロセグメンテーション | 全要素をネットワーク的に分離し、明示的に許可された通信（プロトコル・ポート単位）以外を全拒否。境界ファイアウォールとの違いは「境界の内側でも全システムが互いに閉じている」こと |
| ポリシーエンジン | 通信可否のルールを一元管理。ID ベースアクセス（ユーザー・サービスアカウント・ワークロード ID）＋ RBAC ＋ デバイスポスチャ |
| PDP（Policy Decision Point） | 全ポリシー要素をリアルタイム評価しアクセス可否を判定 |
| PIP（Policy Information Point） | ID プロバイダ・デバイス管理・脅威インテリジェンス等から判定材料を収集 |
| デバイスポスチャ | パッチレベル・OS バージョン・ヘルスチェック項目が基準を満たすかを接続条件に含める |
| mTLS | X.509 による相互認証＋ネットワーク経路の暗号化（エンドポイント到達後は復号されるため、経路上のみの保護である点に注意） |
| ID プロバイダ | ユーザー・デバイス・アプリの全認証を支える基盤。強い認証方式が前提 |
| 監視 | 全要素の認証イベント（特に失敗）を継続監視 |

### 3.3 実装の要点と落とし穴

| 要点 | 説明 |
|------|------|
| 製品ではなく設計哲学 | 「ゼロトラスト製品」を買って完了ではない。設計原則＋運用プラクティスの総体 |
| 目的は侵入防止＋横展開の抑止 | 1 システムへの侵入が他システムへの信頼済みアクセスに直結しない構造を作ることが本質 |
| ソフトウェア定義環境が追い風 | クラウドでは全てが設定で完結し、ルーター・スイッチ等の追加ハードウェア構成が不要なため導入しやすい |
| 端末証明書 | オンプレ端末にも証明書を配布し、未許可デバイスの接続を防ぐ |

**アンチパターン**:
- 境界ファイアウォールの内側で East-West 通信を無制限に許す（マイクロセグメンテーションの不在）。
- mTLS を導入して「全て暗号化済み」と考える（エンドポイント上の平文処理・アプリ層の防御は別問題）。
- ネットワーク層のみに注力し、ID・デバイスポスチャ・監視を欠く。

## 4. AI/ML とクラウドセキュリティ

### 4.1 用語の整理

| 用語 | 定義 |
|------|------|
| AI | 会話・推論・問題解決・言語理解・曖昧な入力に基づく判断など、人間的な柔軟さを計算機で模倣する概念 |
| ML | AI の一形態。明示的なルールではなく訓練データからパターンを学習し、類似データへ外挿するアルゴリズム群 |
| LLM | ML の一種。言語要素間の関係の統計的表現。Transformer により大規模テキストから「文脈上次に来る語」を予測するモデルを構築し、チャットボット等の基盤となる |

マーケティング文脈では用語が曖昧に使われがち。**設計判断では「何をどう学習し、何を出力するのか」で区別**する。

### 4.2 防御側での AI/ML 活用

ML の強みは**人間には不可能なスケールのパターン認識**。テラバイト級のログ・トラフィックから、人間がルール化できていない（見たことのない）異常を発見できる。

**セキュリティオペレーションへの適用フロー**:

```
データソース（クラウドログ・ネットワークトラフィック・アプリ通信）
  → ML 分析エンジン
      ├ 異常検知（未知パターンの検出）
      ├ 脅威分類（悪性か否かの判定）
      ├ リスクスコアリング（ビジネス影響許容度に基づく優先度付け）
      └ 予測分析
  → 出力
      ├ 自動応答（トラフィックのブロック/許可の自律調整）
      ├ セキュリティアラート
      └ 調査支援
```

| 活用領域 | 内容 |
|---------|------|
| 異常検知 | 人手ルール（既知パターン前提）では拾えない未知の悪性挙動を、通信パターン学習済みモデルが検出 |
| トリアージ | 重要資産と脅威の理解に基づき、エスカレーション要否とスコアを付与して優先順位付け |
| 自動対応 | 環境・ビジネス・アプリ利用者のニーズ理解に基づき、ブロック/許可を自動調整（人間の承認ゲートを挟む設計が現実的） |
| LLM による IaC 支援 | 実装一貫性の欠如はセキュリティポスチャを毀損する。LLM で IaC 生成・自動化タスク開発を支援し一貫性を高める |
| LLM によるドキュメント生成 | プロセス・ポリシー・標準文書の生成を省力化（文書化が敬遠されがちな組織で特に有効） |

**落とし穴**: ML は人間のアナリストを置き換えるものではなく、スケール面の補完。自動ブロックは誤検知時の業務影響を評価してから有効化する。

### 4.3 AI システム自体の保護

AI を防御に使う一方、クラウド上の AI/ML システム自体が新たな攻撃対象になる。

| 脅威 | 対策の方向性 |
|------|-------------|
| プロンプトインジェクション | 入力検証・出力フィルタリング・権限を絞ったツール実行 |
| 訓練データ汚染（poisoning） | データ来歴管理・パイプラインのアクセス制御 |
| モデル・訓練データの窃取 | モデルアーティファクトをシークレット同様に扱う（保管暗号化・最小権限） |
| 過剰な自律性 | エージェント型 AI に与える権限を最小化し、破壊的操作に人間の承認を要求 |
| 推論エンドポイントの悪用 | 認証・レート制限・利用監視を API と同水準で適用 |

AI ワークロードも本質はクラウドワークロードであり、本リファレンス前半の制御（IAM・ワークロード ID・セグメンテーション・監視）がそのまま土台になる。

## 5. まとめ: 潮流別チェックリスト

### クラウドネイティブ
- [ ] IaC・コンテナイメージ・依存関係をパイプラインでスキャンしている
- [ ] ランタイム保護（seccomp/SELinux/AppArmor・プロセス監視・FIM）をホスト側で必ず実施している
- [ ] Kubernetes は RBAC 最小権限＋namespace 分離＋レイヤー別制御で守っている
- [ ] エフェメラルなコンポーネントのログを消滅前に外部集約している
- [ ] API ゲートウェイに認証・レート制限を集約している

### モダン IAM
- [ ] パスワードローテーション強制を廃止し、長いパスワード＋使い回し禁止に移行した
- [ ] 人間の認証はパスワードレス（セキュリティキー＋PIN / 生体）へ移行中
- [ ] アプリにはサービスプリンシパル（証明書/シークレット＋キーボールト）を割り当てている
- [ ] 短命ワークロードには SPIFFE/SPIRE 等で短命証明書＋mTLS を自動発行している

### ゼロトラスト
- [ ] 「境界内＝信頼」を廃し、全通信で認証（ユーザー＋デバイス＋ワークロード）している
- [ ] マイクロセグメンテーションで East-West 通信をデフォルト拒否している
- [ ] PDP/PIP を備えたポリシーエンジンでリアルタイムにアクセス判定している
- [ ] デバイスポスチャ（パッチ・OS バージョン）を接続条件に含めている
- [ ] 目的を「横展開の抑止」に置き、製品導入で完了と見なしていない

### AI/ML
- [ ] 大規模ログの異常検知・トリアージに ML を活用し、人手ルールの盲点を補完している
- [ ] 自動対応の誤検知リスクを評価し、必要な箇所に人間の承認ゲートを置いている
- [ ] LLM を IaC・自動化・文書生成の一貫性向上に活用している
- [ ] AI システム自体（プロンプト・訓練データ・モデル・推論 API）を攻撃対象として保護している
