# プラットフォーム・インフラストラクチャセキュリティ

クラウド基盤（パブリック/プライベート問わず）のセキュリティを、セキュリティアーキテクチャ設計 → データセンター物理セキュリティ → IAM → クラウド内ネットワークセキュリティ → 仮想化・コンテナセキュリティの順に整理したリファレンス。プライベートクラウドを自前構築する場合は、パブリッククラウドなら事業者が担う物理層・可用性の責任もすべて自組織に降りてくる。どの提供形態でも IAM・データ保護・ロギング/モニタリング・構成管理・ネットワークセキュリティ・コンプライアンスという基本機能は共通であり、Shared Responsibility Model に基づいて「誰がどの層を守るか」を最初に確定させる。

---

## 1. インフラセキュリティアーキテクチャ

### 1.1 セキュリティアーキテクチャの構成要素

セキュリティアーキテクチャ＝「システム/アプリケーションでセキュリティをどう実装すべきかを記述した全体設計」。技術だけでなく人とプロセスを含める。

| 要素 | 内容 |
|------|------|
| セキュリティ原則・目標 | 最も守りたいものを定義。目標がなければ達成可否を判定できない |
| 統制カテゴリ | 予防的（preventive）/ 発見的（detective）/ 是正的（corrective）、技術的 / 管理的のどれを扱うか明示 |
| 統制の詳細 | 期待するサービス・能力のレベル（実装技術は固定しない） |
| 役割と責任 | 誰が運用・維持するか。技術は勝手に動かない |
| ポリシー・プロセス | 脆弱性管理・ロギング/モニタリング等の横断プロセス |

### 1.2 リファレンスアーキテクチャの選定

ゼロから作らず、公開されているリファレンスアーキテクチャを出発点にする。

| フレームワーク | 特徴 | 向くケース |
|---------------|------|-----------|
| NIST Cloud Computing Security Reference Architecture | 脅威・脆弱性の特定を支援。政府機関は準拠が前提 | 公共系・規制対応 |
| CSA (Cloud Security Alliance) Enterprise Architecture | ガバナンス・データ保護等のドメイン分類、脅威モデリング、マルチクラウド統制 | ベンダー非依存で始めたい |
| Microsoft Cloud Security Architecture | ゼロトラスト基盤・アイデンティティ中心・多層防御 | Azure 中心 |
| AWS Well-Architected Framework | 6 本柱（運用性・セキュリティ・信頼性・パフォーマンス効率・コスト最適化・持続可能性）。セキュリティ以外も網羅 | AWS 中心 |
| Google Cloud Security Foundations Guide | security-by-default・リソース階層・ゼロトラスト・自動化重視 | GCP 中心 |
| IBM Cloud Security Architecture | 脅威ベース・概ねベンダー非依存・データ保護/ID管理/継続的コンプライアンス監視 | マルチクラウド |

**判断基準**: 単一ベンダー展開ならそのベンダーのフレームワークを第一候補に。マルチクラウドなら NIST / CSA / IBM 系のベンダー非依存フレームワークを軸にし、ベンダー固有のものからも横断的な知見を拾う。

### 1.3 信頼境界とセグメンテーション

システムの本質は「情報と、そこへのユーザーアクセス」。コンポーネントごとに扱う情報の機微度とユーザー信頼度が異なるため、信頼境界（trust boundary）でゾーン分割する。

**ゾーン設計の型**:

| ゾーン | 例 | 扱い |
|--------|-----|------|
| 非信頼（untrusted） | API ゲートウェイ等の公開受け口 | 接続も入力データもすべて非信頼。専用セグメントに隔離 |
| 低機微 | マーケティングコンテンツ配信 | 公開されても被害は小さいが、他システムへの接続情報を持ちうる → ラテラルムーブメント防止のため分離 |
| 高機微 | データベース、ID リポジトリ | 機密情報を保持。独立セグメント＋最小限の許可経路のみ |

**セグメンテーション実装の選択肢**:

| 手法 | 要点 | 落とし穴 |
|------|------|---------|
| VLAN | 同一ブロードキャストドメイン（L2）に共通性のあるシステムを収容。MAC アドレスで通信 | **VLAN は整理術であってセキュリティ統制ではない**。同一 VLAN 内・外からの通信を制御しない → ファイアウォール併用必須 |
| ファイアウォール | ネットワーク境界の出入りを制御。ホストファイアウォールは端末保護にはなるがセグメンテーションにはならない | 境界型のみだと内側は素通し |
| マイクロセグメンテーション | ゼロトラストの要素。各デバイスを実質単独ネットワーク化し、ホスト側ファイアウォールで外部アクセスを全面制限 | 管理負荷。SDN との組み合わせが現実的 |
| SDN | 制御を物理層からソフトウェアへ移す。VM・クラウドと親和性が高い。物理網はトンネルのキャリアに徹する | — |
| VPC | クラウドで顧客がネットワークハードに触れない代わりに提供される仮想ネットワーク環境。VLAN 相当の分離統制を実現 | — |
| Security Group | 開放ポートの定義。一度作れば VPC 内の複数箇所へ再利用適用できる | ステートフル（多くの実装）。ACL との違いを混同しない |
| Network ACL | サブネット入口でのフィルタ。送信元/宛先アドレス・ポート・プロトコルを制御 | **ステートレス**：往路が許可されていても復路は独立に判定される |

### 1.4 多層防御（DiD）と広域防御（DiB）

- **Defense in Depth（多層防御）**: 同じ壁を何枚も重ねるのではなく、**層ごとに異なる種類の統制**を置く。1 枚破る手口が全層に通用する構成は多層防御ではない。例: ステートフルファイアウォール → ロードバランサ → Network ACL → Security Group → WAF / API ゲートウェイ。
  - API ゲートウェイ: アクセス制御・レート制限等で API サーバーを保護
  - WAF: アプリケーション層攻撃の悪性メッセージを検知・遮断/警告
- **Defense in Breadth（広域防御）**: 技術統制に加えプロセス統制（脆弱性管理プログラム、ロギング/モニタリング運用等）まで幅を広げてカバレッジを確保する考え方。
- **統制の継承（inheritance）**: 上位に置いた統制（FW・LB・WAF・API GW・横断プロセス）は配下の全要素を保護する。**重複配置を避けつつ最大カバレッジになる配置点を選ぶ**のがコスト効率の要諦。
- **アンチパターン**: ログ収集基盤だけ作って各システムのログ出力設定・転送設定をしない（「持っているが使えていない」統制）。

## 2. データセンター物理セキュリティ

自前データセンターでは物理統制のすべてが自組織の責任。「ローカルだから可用性要件を下げてよい」は成り立たない。

### 2.1 立地選定

| 観点 | チェック |
|------|---------|
| 自然災害 | 竜巻・ハリケーン・洪水の常襲地帯を避ける。回避しきれない残余リスクは環境監視でカバー |
| 電力の信頼性 | 停電が多い地域・発電容量が不足する地域を避ける。可能なら複数電力源・複数引き込み口 |
| 人為的脅威 | 車両が突入しうる交差点至近・隣接駐車場等を避ける |
| ネットワーク | 上流プロバイダを複数確保できる立地 |

### 2.2 境界設計とアクセス制御

| 統制 | 要点 |
|------|------|
| ボラード・障壁 | 車両突入からの建物保護 |
| フェンス・ゲート | 高機微施設で人の接近を制御 |
| 窓 | 空調負荷・内部レイアウト/資産の可視化・侵入口になる。最小化または排除 |
| 入館認証 | **物理鍵は非推奨**（複製容易・追跡困難）。バッジ/カードキー等のデジタル手段で失効管理可能に |
| 機器室の二要素 | バッジ＋生体（虹彩・指紋）。生体には**ライブネス検知**を併用（高解像度スキャン複製対策） |
| 共連れ（tailgating）対策 | 1 人ずつしか通れない回転扉等。バリアフリー・機材搬入との両立を設計段階で考慮 |
| 賃借ビルの場合 | 建物オーナーとの責任分界を明確化 |

### 2.3 監視

- CCTV: 建物外周（接近者・不審行動）と入口ドアを撮影。常時監視要員が置けない場合も**最低限録画**して事後レビュー可能に。
- 警備員: 拠点によってはカメラの代替/補完として常駐。
- センサー: 開放ドア検知（長時間開放でアラート）、モーションセンサー。
- **認証イベントの相関監視**: 深夜 3 時の「正常な」バッジ成功、バッジ成功＋生体失敗の組み合わせ等、成功イベントも文脈で疑う。

### 2.4 運用セキュリティ

| 対象 | 実践 |
|------|------|
| アクセス権管理 | 役割ベースで付与基準を文書化。バッジの配布/失効、生体情報の収集/保護手順を定義 |
| 要員 | データセンター入室者には通常より厳格な身元調査＋データ漏えい防止の追加教育 |
| 訪問者 | 視認可能な訪問者バッジ・入退記録・常時同伴。期限切れが視覚化される自己失効バッジ（一定時間でインクが浮き出て VOID 表示）も有効 |
| ベンダー | 訪問者と同等に記録・監視・同伴。定常的に出入りするベンダー従業員には身元調査（正規の審査を迂回する侵入経路になりうる） |
| 資産管理 | 全ハードウェアを取得〜廃棄までID管理・追跡。特にディスク等の記録媒体 |
| 統制の維持 | 錠・バッジリーダー・カメラの定期保守と、訪問者記録・媒体破壊等の人的プロセスの定期点検 |

**媒体廃棄の判断基準**:

| 媒体 | 手法 | 注意 |
|------|------|------|
| 磁気ディスク | 消磁（degauss）または複数回上書き | 消磁は保証がない。1 回の上書きでは不十分 |
| SSD | **上書き消去を信頼しない** | wear leveling により退役セルへ書かれたデータが残存しうる |
| 共通の最終解 | **物理破砕（シュレッド）** | 認定業者による証明付き破壊の利用が現実的 |

### 2.5 冗長性・信頼性

| 領域 | 設計要点 |
|------|---------|
| 冷却 | ラック集積の発熱は換気では賄えない（開口部は侵入経路にもなる）。空調＋排水（除湿水）管理 |
| 湿度・環境 | 機器の動作パラメータ内に環境条件を常時監視・制御。逸脱は故障→可用性低下に直結 |
| 電力 | 可能なら複数電力源/引き込み口。パワーコンディショナ（スパイク対策）→ UPS（瞬断/瞬低のブリッジ）→ 発電機（長時間停電）。**発電機は停電後にしか起動できないため、起動までの空白を UPS が必ず埋める**。切替は自動転送スイッチ（ATS）で電源系統を選択 |
| ネットワーク | 上流プロバイダ複数化＋物理引き込み口複数化。内部もルーター/スイッチを HA 構成で冗長化 |
| 火災 | 煙感知＋熱感知の併用（煙が先に出ない火災・煙感知が効かない区画がある）。検知はアラートだけでなく消火をトリガー |

**消火の原理**: 火は熱・燃料・酸素の 3 要素で成立し、1 つ除去すれば止まる。**データセンターで水系消火は禁止**（電子機器を破壊する）。不活性ガス（Argonite・Inergen 等）による酸素遮断、粉末系、燃焼連鎖を断ちつつ冷却する薬剤系を選ぶ。

### 2.6 災害復旧（DR）と事業継続（BCP）

- 予防のみに賭けない。**障害は起きる前提**で復旧を計画する。
- DR 計画: 施設・事業への脅威を評価し、脅威別の復旧手順を文書化。最終目標は完全復旧。
- BCP: 危機の最中に事業を回し続ける計画。手作業への切替（紙での受注等）と、収束後にデジタルへ戻す手順まで含める。発災時はまず BCP を発動し、DR 計画が完全復旧へ導く。
- バックアップは攻撃者の標的になるため**オフライン保管またはイミュータブル**に。

**フェイルオーバーサイトの選択**:

| サイト | 状態 | 復旧時間 / コスト |
|--------|------|------------------|
| ホットサイト | 本番と並走（HA 的運用） | 最短 / 最高 |
| ウォームサイト | データ・システムをレプリケーション済み、即切替可能 | 中 / 中 |
| コールドサイト | 設備はあるがリストアが必要 | 最長 / 最低 |

**計画テストの成熟段階**（文書化しただけの計画は思い込みを含む）:

1. 構造化ウォークスルー: 関係者で手順を読み合わせ、欠落・変更点を洗い出す
2. 机上演習（tabletop）: シナリオを与え、計画に沿った判断・招集を模擬
3. シミュレーション: 実際に（または擬似的に）系を落としてバックアップへ切り替え、手順を実地で踏む

テストの目的は訓練ではなく**ギャップ発見と改善**。文書更新に加え、必要ならシステム側の増強まで行う。

## 3. Identity and Access Management（IAM）

IAM はあらゆるシステム実装の基盤。ユーザー・サービス等すべてのプリンシパルにライフサイクルがあり、**プロビジョニング → 認証 → 認可 → 監視 → 維持 → デプロビジョニング**を一貫して管理する。各フェーズは可能な限り自動化する（ヒューマンエラーの排除）。

### 3.1 ライフサイクル各フェーズの要点

| フェーズ | 実践 | 落とし穴 |
|---------|------|---------|
| プロビジョニング | ID 付与前に本人性を検証（身元調査、ビデオ通話での身分証照合等）。上長が権限・グループを明示。HR システム起点で自動化 | リモート採用でのなりすまし。検証なしの ID 発行 |
| 認証 | 知識（パスワード）・所持（デバイス/トークン）・生体の複数要素で **MFA**。IdP 発行のトークンで各システムが認証処理自体を持たない構成も有効 | 単一要素依存 |
| 認可 | 最小権限の原則: 職務に必要なものだけ。ロール/グループを職位に整合させる | ログインできる＝全操作可能、という設計 |
| 監視 | 想定外のユーザー作成・ログインを検知。失敗だけでなく成功も記録（調査・コンプライアンス立証に使う） | 「成功ログは不要」という思い込み |
| 維持 | ロール所有者への定期アクセスレビュー（誰がそのロールにいるか＋ロールの権限が適正か） | 棚卸しの形骸化 |
| デプロビジョニング | HR イベント起点でほぼ即時に全アクセスを自動除去し、除去の検証・証跡まで自動化 | 退職後アカウントの残存 |

### 3.2 アクセス制御モデルと条件付きアクセス

| モデル | 概要 | 使いどころ |
|--------|------|-----------|
| RBAC | ロールに権限を束ね、ユーザーをロールへ割当 | 個人単位管理を避けたい大半のケース |
| 属性ベース | 所在地・所属等の属性で判定 | 文脈依存の細粒度制御 |
| DAC（任意アクセス制御） | 個々のユーザーがアクセスレベルを決める | 汎用 OS 等の既定 |
| MAC（強制アクセス制御） | 経営/管理側が策定したポリシーが決める | 高機密環境 |

**条件付きアクセスポリシー**: 「何に」だけでなく「どんな条件で」アクセスできるかを制御。会社支給デバイス限定・地理的制約・**impossible travel 検知**（2 時間前に別大陸から認証していたら片方は不正）等。

### 3.3 特権アクセス管理

- 特権作業用アカウントは**通常業務アカウントと分離**する。
- MFA があっても特権の**常設アカウントを持たない**運用が望ましい: 時間制限付きのチェックアウト（Just-in-Time 昇格）で必要時間だけ付与。
- 分離の理由: 特権のままメール閲覧・Web ブラウズをすると侵害の起点になる。
- ログ/監視により最小権限・職務分離の遵守をコンプライアンス上も立証できる。

### 3.4 フェデレーション

大規模組織や B2B では ID が単一リポジトリに収まらない。**フェデレーション**＝複数の ID ソース間に信頼関係を張り、認証は ID の記録元（system of record）が担い、認可はリソース所有側システムが担う分担構成。

## 4. クラウド内ネットワークセキュリティ

対象は IaaS / PaaS / プライベートクラウド。SaaS やストレージ利用のみの消費者は通常ネットワークセキュリティの統制権も責任も持たない（責任共有の確認が先）。

### 4.1 レイヤ構成

VPC（仮想ネットワークコンテナ）を基本単位に、外側から順に統制を重ねる。

| レイヤ | 統制 |
|--------|------|
| 境界（VPC エッジ） | ファイアウォール、WAF、DDoS 防御、API ゲートウェイ。監視のための TLS 終端 |
| セグメンテーション | サブネット分割＋ Network ACL（ステートレス、サブネット入口で送信元/宛先/ポート/プロトコル制御） |
| インスタンス単位 | Security Group（ホストベースFW定義を複数インスタンスへ再利用適用）。例: Web 層は default-deny ＋ TCP 80/443 のみ許可 |
| セグメント内 | 同一分類・同一要件の要素が集まるセグメントには標準構成のFWを追加。ホスト個別設定の乱立を避ける |
| 検知（detective） | 各統制のログ＋フローログ（パケット数・サイズ・方向・ポート・プロトコル）を SIEM へ集約 |

### 4.2 検知側の要点

- **ログを集めるだけでは検知にならない**。SIEM でルール＋異常検知ベースのアラートを構成する。
- フローログとネットワーク統計はインシデント調査の一次資料。
- 予防統制と検知統制をセットで層状に配置する（どちらか片方に寄せない）。

## 5. 仮想化セキュリティ

クラウドは本質的に仮想化されており、システム・ネットワーク・ストレージ・アプリケーションすべてがソフトウェアで抽象化される。ゲスト OS・アプリに加えて**仮想化基盤そのもの**が保護対象になる。

### 5.1 ハイパーバイザの脅威モデル

- ハイパーバイザは 1 台で多数の VM を集約する。**ハイパーバイザが侵害されると配下の全 VM が侵害される**（単一障害点かつ高価値標的）。
- 分離の仕組み: ハイパーバイザが各 VM のシステムコールを捕捉し、VM ごとの仮想メモリページテーブルで他 VM のメモリアクセスを遮断。仮想ディスクは物理側ではファイルとして保持され、VM 内のディスクアクセスをファイル内オフセットへ変換して越境参照を防ぐ。

**VM エスケープの 2 経路**:

| 経路 | 内容 | 対策 |
|------|------|------|
| 脆弱性経由 | ハイパーバイザソフトウェアの脆弱性を突いてゲストから脱出 | パッチ適用・脆弱性管理 |
| 管理インターフェース経由 | ネットワーク防御が不十分だと、ゲストからハイパーバイザの管理画面（Web UI / SSH）へ到達できる。**実質エスケープと同じ結果** | 管理プレーンをゲストネットワークから完全分離 |

### 5.2 仮想ネットワークとリソース管理

- ハイパーバイザ内のネットワークは定義上すべて SDN。VM ごとに独立した IP ネットワーク（セグメント）を割り当てる。
- 物理網へ出る際は **802.1q VLAN タグ**で分離を維持し、スイッチが明示設定なしにタグをまたがないようにする。
- オーバーサブスクリプションは仮想化の目的そのものだが、行き過ぎると実際のボトルネックになる。ハイパーバイザ/管理ソフトウェアで割当と使用率を監視する。

### 5.3 コンテナセキュリティ

コンテナは OS 全体ではなく**アプリケーションだけを仮想化**する形態。操作はコンテナ管理ソフトウェア（Docker 等）経由に限られ、アプリが提供する機能以外の接点がない。

**分離の仕組みと固有の性質**:

- namespace: OS が管理するメタデータでメモリアドレス等をタグ付けし、同一タグのプロセスからしかアクセスできなくする。ディスク等他リソースへのアクセスも制御。
- エフェメラル性: コンテナは短命で消える → 侵害されても永続化しにくい。**ただしそれだけでは不十分**。
- コンテナ侵害時に得られるのは、アプリ実行に必要な最小限だけの「偽のシステム」。

**コンテナエスケープの経路**: ①異なる namespace のメモリへアクセスできるカーネル脆弱性、②ホスト OS の管理インターフェース露出（ホストへ脱出すると他コンテナと転送データまで掌握される）。

**ハードニングのチェックリスト**:

| 統制 | 内容 |
|------|------|
| cgroups | リソースクォータでリソース枯渇型 DoS を防止、優先度制御。**単独では不十分** |
| MAC（SELinux 等） | ラベルとポリシーで Linux 上の全リソースアクセスを強制制御 |
| capabilities の削減 | `CAP_SYS_ADMIN`・`CAP_NET_ADMIN` 等、侵害時に特権昇格へ使われる capability を drop |
| seccomp プロファイル | デバイスの mount/unmount・プロセストレース・システム時刻変更・スワップ管理等の syscall を遮断 |

これらを組み合わせ、アプリが侵害されても攻撃者が得られるものを最小化し、攻撃面自体を削る。

## 6. 横断チェックリスト

- [ ] 責任分界（Shared Responsibility）を提供形態（IaaS/PaaS/SaaS/プライベート）ごとに文書化したか
- [ ] リファレンスアーキテクチャを選定し、自環境の統制へマッピングしたか
- [ ] 信頼境界で非信頼/低機微/高機微ゾーンを分離し、ラテラルムーブメント経路を潰したか
- [ ] VLAN/VPC 分割に**加えて** FW・ACL・Security Group による通信制御を入れたか（VLAN 単独をセキュリティ統制と誤認していないか）
- [ ] 各層に**異なる種類の**統制を配置したか（同種の壁の重ね掛けになっていないか）
- [ ] ログは出力設定・転送・SIEM 相関まで通貫しているか
- [ ] 物理: 入館の二要素＋ライブネス、共連れ対策、訪問者/ベンダー管理、媒体の物理破砕
- [ ] 電源: パワーコンディショナ → UPS → 発電機のチェーンと ATS、UPS が発電機起動までの空白を埋めるか
- [ ] 消火: 煙＋熱の二重検知、水系以外の消火剤
- [ ] DR/BCP: ウォークスルー → 机上演習 → シミュレーションまで実施し、結果を計画とシステムに反映したか
- [ ] IAM: HR 起点の自動プロビジョニング/デプロビジョニング、MFA、条件付きアクセス、JIT 特権、定期アクセスレビュー
- [ ] 仮想化: ハイパーバイザ/ホストの管理プレーンをワークロードネットワークから分離したか
- [ ] コンテナ: capabilities drop・seccomp・MAC・cgroups を既定で適用しているか
