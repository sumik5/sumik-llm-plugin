# 法規制・プライバシー・契約・クラウドフォレンジック

クラウド利用を前提としたセキュリティプログラムに必要な法務・規制対応のリファレンス。データプライバシー規制（GDPR / CCPA / PIPL 等）の要点と越境移転・データ主権、契約と SLA に盛り込むべきセキュリティ条項、ベンダー管理ライフサイクル、法令遵守ベストプラクティス、クラウド環境でのデジタル調査（証拠保全・eDiscovery）を、チェックリストと判断基準の形で整理する。コンプライアンス達成は法規制対応の一部にすぎず、プライバシー・契約・調査対応を含む広い法務連携が必要という前提に立つ。

---

## 1. データプライバシー規制

### 1.1 なぜ規制対応が難しくなったか

- 顧客接点のデジタル化により、保存・処理する個人データ量が増大
- 処理の多くが自社所有システム外（クラウド・SaaS・外部ベンダー）で行われる
- 地域・国ごとに独自のプライバシー法が乱立 → **管轄（jurisdiction）問題**が常態化
- クラウドのリージョン選択・マルチナショナル展開が規制適用範囲に直結する

### 1.2 GDPR（EU 一般データ保護規則）

#### 基本原則（Article 5 の骨子）

| 原則 | 要点 |
|------|------|
| 適法・公正・透明な処理 | 収集・利用の根拠と範囲を本人に明示 |
| 目的の限定 | 収集時に特定した目的以外に使わない |
| データ最小化 | 目的に必要な最小限のデータのみ収集 |
| 本人の権利保障 | アクセス権・ポータビリティ・削除（忘れられる権利） |
| 責任の明確化 | 誰がデータを所有し、誰が処理に責任を負うかを定義 |

#### 役割モデル

| 役割 | 定義 | クラウド文脈での典型 |
|------|------|--------------------|
| Data Subject（データ主体） | 情報を収集・処理・保存される本人 | エンドユーザー・消費者 |
| Data Controller（管理者） | 収集目的と処理方法を決定する組織 | クラウド利用企業（顧客側） |
| Data Processor（処理者） | 管理者の指示でデータを処理する組織 | クラウドプロバイダー |
| Supervisory Authority（監督機関） | 国・地域単位の執行機関。管理者・処理者の両方を監督 | 各 EU 加盟国の当局 |

- Controller ↔ Processor 間の責任分担は **DPA（Data Processing Agreement）** で規定する
- 侵害（breach）の疑いが生じたら、被害組織は**可及的速やかに監督機関へ通知**する義務がある

#### DPO（Data Protection Officer）の要否判断

| 条件 | DPO 必須か |
|------|-----------|
| 公的機関（裁判所を除く） | 必須 |
| コア業務が大規模かつ定常的・体系的なデータ主体の監視 | 必須（「大規模」の定義は曖昧＝グレーゾーン） |
| 特別カテゴリ（健康・生体情報等）の大規模処理 | 必須 |
| クラウドプロバイダー | 通常必須（大量の主体データを処理するため） |
| 小規模なクラウド利用企業 | 多くは不要（ただし兼務でも置く価値はある） |

DPO の職務: GDPR 遵守の統括・教育訓練・プライバシー影響評価の実施・侵害時の窓口。

#### 適用範囲（域内・域外）

| スコープ | 適用条件 |
|---------|---------|
| 域内適用（territorial） | **処理が EU 域内で行われる**なら、データ主体の国籍・所在を問わず適用 |
| 域外適用（extraterritorial） | 処理が EU 域外でも、**EU 居住者に商品・サービスを提供**または**EU 居住者を監視**していれば適用 |

罠: クラウドのリージョンを米国内に選んでも、サービス提供先が EU 居住者なら GDPR 対応が必要。**リージョン選択は GDPR 適用回避の手段にならない。**

### 1.3 CCPA（カリフォルニア州消費者プライバシー法）

- 焦点: データが**商品として売買される**ことへの規制（GDPR との最大の違い）
- 執行主体: 州司法長官（AG）単独（GDPR のような複数監督機関の調整は不要）
- 登場主体: Consumer / Business（収集者）/ Service Provider（クラウド事業者等）/ Third Party（データ購入者）

#### 消費者の権利

| 権利 | 内容 |
|------|------|
| 知る権利 | 何がなぜ収集されているかを知る |
| 削除権 | 事業者にデータ削除を求める |
| オプトアウト権 | **情報の売却を拒否する**（CCPA 特有） |
| 非差別権 | 権利行使を理由に不利益を受けない |
| ポータビリティ権 | 個人情報を可搬形式で受け取る |

#### 適用対象となる事業者（いずれか 1 つ該当で適用）

- 年間総収入 2,500 万ドル以上
- 年間 5 万人以上の消費者の個人情報を売買
- 収入の 50% 以上が個人情報の販売由来

制裁金: 違反 1 件あたり 2,500 ドル、故意の違反は 7,500 ドル。Service Provider は特定の業務目的以外に個人情報を使えない。

#### 適用法の判断基準

- **どの法が適用されるかは原則データ主体の居住地で決まる**（カリフォルニア居住者 → 処理者が欧州にいても CCPA）
- ただし管理者・処理者の所在地は、制裁・訴訟手続の場所として意味を持つ

### 1.4 その他の主要規制

| 法規制 | 対象 | 特徴 |
|--------|------|------|
| NY SHIELD Act（米ニューヨーク州） | NY 居住者 | 事業規模を問わず「合理的なセキュリティ対策」と侵害通知を義務化 |
| LGPD（ブラジル） | ブラジル居住者 | GDPR に類似。南米にクラウド展開があれば適用可能性を確認 |
| PIPEDA（カナダ・連邦法） | カナダ市民 | カナダ企業に加え、カナダ顧客を持つ米国企業にも関係 |
| PIPL（中国） | 中国内の個人 | 下記参照。データローカライゼーションが最も厳格 |
| HIPAA（米・医療） | PHI（保護対象保健情報） | 医療記録・請求情報等、個人を特定しうる医療データの保護。医療関連事業者は遵守必須 |

### 1.5 PIPL（中国）とデータ主権・越境移転

PIPL はデータ主権（data sovereignty）要求が最も強い例であり、越境移転設計の代表的な制約条件になる。

| 項目 | 要求内容 |
|------|---------|
| 適用範囲 | 中国内の個人へ製品・サービスを提供する事業者、中国内の個人の行動を分析・評価する事業者 |
| 個人情報（PI）の定義 | 識別された/識別可能な個人に関するあらゆる情報（広義） |
| センシティブ PI | 生体情報・宗教・健康・金融口座・位置追跡など。**追加の同意**が必要 |
| 同意 | 明示・具体的・明確・十分な情報に基づくこと。**撤回は付与と同程度に容易**でなければならない |
| 本人の権利 | 知る/アクセス・訂正/削除・処理制限・ポータビリティ・自動化された意思決定の説明を受ける権利 |
| データローカライゼーション | 重要情報インフラ運営者・大規模処理者は**中国国内での保存が必須** |
| 越境移転 | CAC（国家インターネット情報弁公室）へ詳細文書を添えて申請 → 技術的/組織的措置・必要性・影響のセキュリティ審査（数か月単位）→ **2〜3 年ごとに承認更新** |
| 侵害通知 | 当局および本人への通知が義務 |
| 制裁 | 年間売上の最大 5% の制裁金・事業停止・責任者個人への責任追及 |

クラウド実務への影響: 中国向けサービスは中国国内のクラウドインフラが事実上必要になりやすく、ベンダー評価に PIPL 遵守検証を含める。ローカライゼーション要求によりマルチナショナル構成は複雑化する。

### 1.6 規制対応の設計チェックリスト

- [ ] サービス提供先の居住者ベースで適用法を棚卸しした（リージョン選択で回避できると誤解していない）
- [ ] Controller / Processor の役割と DPA の締結状況を整理した
- [ ] 本人の権利（アクセス・削除・ポータビリティ・オプトアウト）を実装で満たせる
- [ ] 侵害通知の宛先（監督機関/AG/本人）と期限を規制ごとに把握した
- [ ] データローカライゼーション要求のある市場（中国等）を特定し、保存場所を制御できる
- [ ] 越境移転の承認手続き（申請・審査・更新サイクル）をスケジュールに織り込んだ

## 2. 契約と SLA

### 2.1 前提

- 顧客との契約とベンダー（クラウドプロバイダー含む）との契約の**両面**を管理する
- セキュリティ条項は現代の契約に不可欠。**弁護士とセキュリティ実務者の協働**が必要（セキュリティ側は契約を書けず、法務側はセキュリティ統制を理解しない前提で分業する）
- 最終的な起草・レビューは必ず法務（弁護士）が行う

### 2.2 契約に含めるべきセキュリティ条項カタログ

| 条項カテゴリ | 確認・記載する内容 |
|-------------|------------------|
| データ保護・プライバシー | データ分類と取扱要件／保存・処理場所の制限（規制産業では必須）／アクセス・利用の禁止事項／準拠すべきプライバシー法（GDPR・CCPA・HIPAA 等）の明記／DPA の特定 |
| セキュリティ統制・標準 | ISO 27001・SOC 2 Type II 等のフレームワーク準拠／PCI DSS 等の追加要件／保存時・転送時の暗号化／MFA 等のアクセス制御／**自社が重視する統制（パッチ管理等）はベンダーにも同等統制を明記** |
| インシデント対応・侵害通知 | 侵害通知法のない法域もあるため**契約でインシデント/侵害の定義と通知期限を規定**／自社側インシデント時にプロバイダーが管理範囲の情報・アクセスを提供する義務 |
| 監査・モニタリング | **監査権（right to audit）**の明記（自社の規制要件による場合は特に）／文書レビュー・場合によりオンサイト／SOC レポート等の適時提供／ログ・監視情報やダッシュボードの提供 |
| ベンダー管理 | パフォーマンスレビュー／要員の身元確認・セキュリティ教育／秘密保持（NDA） |
| 責任・リスク配分 | 賠償上限（liability cap）／保険付保要件と免責範囲／セキュリティ障害に対する補償（indemnification）／規制違反時の取扱い／知的財産の保護 |
| データライフサイクル | ベンダー側障害時のバックアップ責任（SaaS・ストレージ型で特に）／削除データの完全破壊（自社のデータ保持要件との整合）／データを含んだ**媒体**の廃棄方法 |
| 終了・移行 | データ・アカウント/クレデンシャル情報を含む全情報の持ち出し可能性／**ベンダーロックイン禁止**／移行完了までサービスを維持する移行支援義務 |

### 2.3 SLA の設計・確認観点

| 観点 | 内容 |
|------|------|
| 対象指標 | サービス可用性／サポート応答時間（チケットのピックアップまでの時間）等、性能への期待値を数値で定義 |
| 実測との突合 | 期待性能を文書化し、継続的に実績と比較する（ベンダー管理の定期レビューで使用） |
| 未達時の救済 | ペナルティ・返金等の条項を契約に明記（明記がなければ実質執行できない） |

### 2.4 アンチパターン

- 契約に監査権がなく、規制当局から求められた証跡をベンダーから取得できない
- 侵害の「定義」がなく、ベンダーが「これは侵害でない」と主張して通知が遅延する
- 解約時のデータ返却・破壊手順が未規定で、移行時にデータが人質化する
- SLA 未達のペナルティ条項がなく、性能劣化を交渉材料にできない

## 3. ベンダー管理

### 3.1 ライフサイクル

ベンダー管理はデューデリジェンスに始まりオフボーディングで終わる**継続的ライフサイクル**である。段階と要点:

| 段階 | 主な活動 |
|------|---------|
| 1. デューデリジェンス | 必須基準の定義／セキュリティ統制・コンプライアンス要件の充足確認／**財務健全性の検証**（契約条項を履行できる相手かの見極め） |
| 2. 契約 | §2 の条項カタログをすべて契約に反映 |
| 3. オンボーディング | ベンダー台帳への登録／初期セキュリティベースライン取得／**セキュリティ質問票**（契約要件はすべて質問票に反映）／必要に応じセキュリティテストで統制を実地検証 |
| 4. 継続管理 | 年次等の定期アセスメント・質問票再送付／回答を契約・SLA と突合し、逸脱があれば是正を協働／文書の裏取りとしてのセキュリティテスト／SLA 実績監視とペナルティ執行／**双方向の課題トラッキングと関係管理** |
| 5. 定期パフォーマンスレビュー | SLA 実績評価／質問票回答と期待値の比較／結果はアウトカムに関わらずベンダーと共有（関係管理の一部） |
| 6. 更新 or 終了 | 更新: 既存契約の実績を交渉材料に条件改善・新たなコンプライアンス要件の反映／終了: **データ回収 → ベンダー側リソースからの破壊**を契約に基づき管理 |

### 3.2 運用上の要点

- **質問票は「紙集め」ではない**: 回答は契約要件・SLA に対して評価し、劣化があれば是正計画に落とす
- セキュリティインシデント時、ベンダー管理チームは影響把握とベンダー↔社内（セキュリティ・法務ほか）の**リエゾン**を務める
- マルチクラウド利用時は、ベンダー管理プログラムで**プロバイダー間の統制パリティ**（同等の統制を全プロバイダーに要求）を確保する
- 小規模組織はベンダー数に比例して管理負荷が増える点を計画に織り込む（全ベンダーに同じプロセスを回す必要がある）
- ベンダーロックイン（移行を著しく困難にする設計・条件）への対抗もベンダー管理の守備範囲

## 4. 法令遵守ベストプラクティス

| プラクティス | 要点 |
|-------------|------|
| ガバナンス基盤 | 適用法・業界慣行に基づいたポリシー・手順・標準を整備する |
| 組織的アラインメント | 法務・コンプライアンスチームとセキュリティチームの整合。**目標もアクティビティも別物**である前提で、経営層のスポンサーシップによりトップダウンで揃える（草の根連携より効果的） |
| 文書管理と discovery | 訴訟時は**ディスカバリー（eDiscovery）**により作成・保存した文書の開示を求められうる。何を残し何を残さないかは法務の助言に従う |
| 弁護士依頼者間秘匿特権 | 重大インシデント時等、法務経由のコミュニケーションで attorney-client privilege による保護を受けられる場合がある（万能ではない） |
| リスク評価への法務参画 | リスクは金銭的損失で測る事業概念。**制裁金・訴訟費用等の規制コストを損失額に織り込む**。クラウドは管轄をまたぐため、侵害時の事業影響評価に法務の関与が必須 |
| 教育・訓練 | コンプライアンス上の要件であることが多く、顧客からも実施状況を問われる。**ロールベース研修**が見落とされがち: 標的にされやすい役割（高権限者等）には攻撃態様に即した個別研修を行う |
| 経営報告 | コンプライアンス状況をレポート・ダッシュボードで経営層へ可視化（是正活動の進捗を含む） |
| 規制当局との関係 | 法務主導（または法務同席）で当局と関係を維持し、報告要件・統制変更等の規制変更の影響を早期把握 |
| 共有責任モデルの理解 | クラウドではデータ保存・アプリ利用の実態を法務も把握する。**ログ・監視基盤にはセンシティブ情報が流れ込むため、ロギングプラットフォーム自体を機密として扱う** |

## 5. クラウド環境でのデジタル調査（フォレンジック）

### 5.1 目的と前提

- 目的: 攻撃者が触れた全リソースの特定 → 侵入経路・横移動の解明 → **完全な排除**と再発防止
- 「フォレンジック」は法廷適合性を含意するが、実務の大半は法廷と無関係な**デジタル調査**。ただし顧客データや知財が絡むため法務の関与は常に必要
- クラウドは調査を容易にしない。物理層の統制がなく、取得できる情報が制限される

### 5.2 調査プロセスの要点

| フェーズ | 要点 |
|---------|------|
| 検知 | 適切なロギング・監視・検知ロジックが前提（外部通報起点の場合もある） |
| トリアージ | イベント → インシデントへの昇格判定 |
| 通知 | インシデント確定後は**まず経営層と法務へ通知**。サイバー保険・breach coach・外部 IR ベンダー等の外部関係の発動を調整 |
| 封じ込め | システム・コンテナ・アプリケーションをまたぐアクセス・横移動を遮断 |
| 証拠収集 | 法務を関与させて情報・コミュニケーションを保護。重要証拠には**chain of custody（証拠管理の連鎖）文書**を維持（法廷用途は稀でも、後日の再分析に耐える証拠完全性の担保として有効） |
| プロバイダー連携 | マルチプロバイダー・マルチリージョン構成では調整が必須。顧客が持たない情報をプロバイダーが保持している場合がある（稀だが起こりうる） |
| 分析 | 収集情報の相関分析 → 新たな情報源の発見 → 再収集、の**反復サイクル**（調査は線形に進まない） |
| 根本原因と修復 | 全侵入経路の特定後、脆弱性修復・アカウントリセット計画を策定。**IaC ベースの環境は再構築が高速**（再イメージング不要・コード修正込みで環境ごと再構築できるのがクラウドの利点） |

### 5.3 オンプレ調査との違い（制約と対策）

| オンプレでの手法 | クラウドでの現実 | 対策 |
|-----------------|----------------|------|
| メモリダンプ取得 | 物理制御がなく通常不可。フルオーケストレーションのコンテナ環境では特に不可 | ログ・イベント・アラートへの依存を前提に検知を設計 |
| ディスクイメージ取得 | 同上 | 同上 |
| ローカルログ収集 | VM/コンテナ内ログに加え、IdP ログ・ネットワークログ/トレース等ソースが分散 | すべてを集約・相関できる収集基盤を平時から用意 |
| 事後の証拠取得 | **エフェメラルな環境では証拠が消滅済み**のことがある | 「後で取りに行く」前提を捨て、**発生時に収集・保存**する |
| 全レイヤーへのアクセス | 顧客がアクセスできる情報に制限される。サービス固有ログが通常の監視基盤に流れていないことがある | 必要なログの棚卸しと収集経路を**プレイブックとして事前整備** |

### 5.4 事前準備チェックリスト

- [ ] 調査に必要なログ種別（環境内・IdP・ネットワーク・サービス固有）を棚卸しした
- [ ] ログはリアルタイムに収集し、改ざん・削除から保護された場所へ保存している
- [ ] インシデント時の法務・経営への通知経路と外部関係（保険・IR ベンダー）を定義した
- [ ] chain of custody の様式と証拠保管手順を用意した
- [ ] クラウドプロバイダーへの調査協力依頼窓口・手順（契約上の根拠含む→§2.2）を確認した
- [ ] マルチリージョン/マルチプロバイダー横断の相関分析手順を演習した
- [ ] IaC による環境再構築（修正込み再デプロイ）の手順を検証した

## 6. 全体の要点（サマリー）

- コンプライアンス ≠ セキュリティだが、大半の組織にとって不可避。近年の中心はプライバシー規制
- 適用法は**データ主体の居住地**基準で棚卸しする。GDPR は域外適用があり、中国 PIPL は保存場所自体を制約する
- ベンダーには自社と**同等の統制**を契約で要求し、監査権・侵害通知・データ破壊・移行支援を明文化する
- ベンダー管理はデューデリジェンス→契約→オンボーディング→継続管理→レビュー→更新/終了のライフサイクルで回し、質問票は契約・SLA と突合して評価する
- 法務とセキュリティの整合は経営層スポンサーシップで担保し、discovery・秘匿特権・制裁金評価で法務を活用する
- クラウド調査はメモリ/ディスク取得に頼れない。**平時からのログ収集・保護保存とプレイブック整備**が成否を分け、IaC は修復・再構築を高速化する
