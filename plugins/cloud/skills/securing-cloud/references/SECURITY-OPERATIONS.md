# セキュリティ運用（Security Operations）

セキュリティ運用とは、組織への攻撃の試みを「特定・検知・対応」するために必要な人・プロセス・技術の総体である。可用性を保つ通常運用とは異なり、「悪いことが起きないようにする／起きたときに対処する」ことが目的となる。本リファレンスでは、敵対者のタイプと動機、攻撃者の行動をモデル化するフレームワーク（Cyber Kill Chain・攻撃ライフサイクル・MITRE ATT&CK）、防御側の組織化フレームワーク（NIST CSF）、SOC のインシデント対応プロセス、ログ・監視・SIEM/SOAR、およびセキュリティエンジニアリング（堅牢化・構成管理・パッチ）を、クラウド環境固有の差分を交えて整理する。

---

## 敵対者のタイプと動機

現実の脅威主体は「個人のいたずら」ではなく、組織化された 3 タイプが中心となる。自組織が保有する資産（情報・アクセス・資金・象徴性）によって、どのタイプに狙われやすいかが変わる。

| タイプ | 動機 | 特徴 | 典型的な行動 |
|--------|------|------|--------------|
| ハクティビスト | 政治的・社会的主張、抗議 | 主張への「報復」として行動。スキルは集団により幅がある | 大規模 DoS/DDoS、情報窃取と公開による「見せしめ」、Web 改ざん |
| 組織犯罪 | 金銭 | 企業のように運営される（雇用・R&D・シフト勤務・外注）。収益が出る限り執拗に狙い続ける | ランサムウェア、情報窃取と転売、侵害インフラの貸し出し（サービス化） |
| 国家支援グループ | 諜報、政治的影響力、（一部は）外貨獲得 | 潤沢な資金と独自の R&D を持ち、既製のマルウェアに頼らず新規攻撃を開発する＝検知が困難 | 標的型攻撃、ゼロデイ悪用、長期潜伏（APT） |

### 判断基準

| 観点 | 問い |
|------|------|
| 標的価値 | 自組織の情報・アクセス・資金・象徴性は、どの敵対者にとって価値があるか |
| 執拗さ | 金銭目的の敵対者は「一度追い出しても戻ってくる」前提で防御を設計しているか |
| 検知難度 | 国家支援グループを想定するなら、シグネチャ依存の検知だけで足りるか |

### ランサムウェアの動向と対策上の含意

- 「暗号化して復号鍵を売る」型から「データを窃取して公開すると脅す」型へ移行している。
- 暗号化型は健全なバックアップからの復旧で対抗できるが、**窃取型はバックアップでは防げない**。データ分類・暗号化・流出検知・最小権限が対策の中心になる。

## セキュリティ運用フレームワーク

攻撃者は自然な順序（プロセス）に従って行動する。攻撃者のプロセスを理解すれば、①検知した振る舞いから攻撃の進行度（緊急度）を推定でき、②自組織の統制をプロセスの各段階にマッピングしてギャップを特定できる。

### Cyber Kill Chain

軍事概念「キルチェーン」をサイバー空間に適用した理論モデル。攻撃を段階に分解し、どこで「チェーンを断ち切るか」を考える。

| フェーズ | 内容 |
|----------|------|
| Reconnaissance（偵察） | 標的選定、情報収集 |
| Weaponization（武器化） | マルウェア開発と配送手段との組み合わせ |
| Delivery（配送） | ネットワーク・USB 等でマルウェアを標的へ届ける |
| Exploitation（悪用） | 標的システム上でマルウェアを実行させる |
| Installation（インストール） | バックドアを設置し遠隔制御の足場を作る |
| Command & Control（C2） | 侵害システムへ遠隔からコマンドを送る |
| Actions on Objectives（目的の実行） | データ窃取、認証情報の追加奪取など本来の目的を実行 |

各フェーズに対して取り得る防御アクション:

| 防御アクション | 内容 |
|----------------|------|
| Detect | 攻撃者の存在の特定 |
| Deny | 不正アクセス・情報開示の防止 |
| Disrupt | 攻撃トラフィックの停止・変更 |
| Degrade | 攻撃と制御の弱体化 |
| Deceive | C2 への妨害・欺瞞 |
| Contain | ネットワークセグメンテーションによる封じ込め |

**限界（採用時の注意）**:

- 実際の攻撃者観察ではなく、物理戦の理論から導かれたモデルである。
- 最初の 2 フェーズ（偵察・武器化）は防御側の統制がほぼ及ばない。
- マルウェア中心の想定であり、マルウェアを使わない現代型攻撃（正規認証情報の悪用等）を扱いにくい。

### 攻撃ライフサイクル（Attack Lifecycle）

実際に観測された攻撃者の行動に基づくモデル。**全フェーズが防御側の統制範囲内**にあり、攻撃者・被害者・標的システム・TTP を問わず適用できる。長期潜伏と環境内での勢力拡大を明示的に扱う点、マルウェアに限定しない点が Kill Chain との違い。

| フェーズ | 内容 | 検知の着眼点 |
|----------|------|--------------|
| 初期偵察 | ホスト名・従業員・メールアドレス等の標的情報収集 | 外部からのスキャン、OSINT 収集の痕跡 |
| 足場の確立（Initial Compromise / Establish Foothold） | 侵害済み認証情報・フィッシング・アプリ侵害・マルウェアで環境へ侵入 | 不審なログイン、フィッシング着弾、Web アプリ異常 |
| 権限昇格 | 以降の行動に必要な管理者権限の獲得を試みる | 権限変更イベント、資格情報ダンプの兆候 |
| 内部偵察 | 共有・他ユーザー・他システム等、次の行動に役立つ情報を収集 | 大量の列挙クエリ、通常と異なる探索行動 |
| 横展開（Lateral Movement） | 他システムへのアクセスを拡大 | システム間の異常な認証・リモート実行 |
| 潜伏維持（Maintain Presence） | バックドア等の永続化機構を追加 | 新規アカウント・スケジュールタスク・スタートアップ変更 |
| 目的の完遂（Complete Mission） | データ窃取等。ただし攻撃者は可能な限り**年単位で環境内に留まる**（実際に 4 年以上潜伏が発覚した大規模侵害事例がある） | 大量データ転送、異常な外向き通信 |

- 検知した振る舞いがどのフェーズに属するかで、対応の緊急度を判断できる。
- 権限昇格〜横展開はループする（環境内で繰り返される）ことを前提に調査する。

### MITRE ATT&CK

観測ベースの TTP（Tactics, Techniques, and Procedures）カタログ。攻撃ライフサイクル上の位置特定と検知ルール設計の共通言語として使う。

| 概念 | 定義 | 例 |
|------|------|-----|
| Tactic（戦術） | 攻撃者の直近の目標 | 初期アクセスの獲得 |
| Technique（技法） | 目標達成の手法 | フィッシング |
| Procedure（手順） | 具体的な実装詳細 | 認証情報収集に使う特定形式のメール |

- 各 TTP にはシグネチャ（IoC）が対応付けられることが多く、検知エンジニアリングの入力になる。
- 公式: <https://attack.mitre.org/>

### NIST Cybersecurity Framework（CSF）

防御側の組織化フレームワーク。ガバナンス（Govern）を土台とし、その上に 5 つの機能を置く。各機能について人・プロセス・技術のカバレッジを点検することで、情報セキュリティプログラム全体を組織化できる。

| 機能 | 内容 | 要点 |
|------|------|------|
| Govern | 文書化・管理・メトリクス等、全機能に共通する経営基盤 | 各機能に個別のガバナンスを持たせず、基盤として全体に適用する |
| Identify | 資産・脅威・能力の特定 | 資産と脅威は常に変化するため、一度きりでなく継続的に文書化・改訂する |
| Protect | 攻撃者を侵入させないための予防的統制 | 必須だが「境界防御で防ぎ切れる」という前提には立たない |
| Detect | ロギングと監視、逸脱時のアラート | 可視性＋アラートの両方が揃って初めて機能する |
| Respond | アラート発生後の調査と攻撃者の排除 | アラートを出すだけで誰も動かないのは「知らないより悪い」 |
| Recover | 復旧・通知・侵入経路の閉塞・失敗分析と再発防止 | 対応だけでは不十分。根本原因を潰すまでが Recover |

- 公式: <https://www.nist.gov/cyberframework>

### コントロールマッピング（ギャップ分析）

CSF の各機能 × 攻撃ライフサイクルの各フェーズでマトリクスを作り、保有する統制を配置する。

| 手順 | 内容 |
|------|------|
| 1 | 攻撃ライフサイクルの各フェーズに対し、Identify〜Recover の統制を列挙する |
| 2 | 空欄（統制のないセル）＝攻撃者が通り抜ける経路として優先的に手当てする |
| 3 | 「侵害される前提」で後段（Detect/Respond/Recover）にも統制を配分する。予防に全振りしない |
| 4 | 攻撃ライフサイクル上の活動を検知するたびにマトリクスへ立ち返り、統制を追加する |

## セキュリティ運用（SOC）

大規模組織は自前の SOC を持ち、小規模組織は外部委託することが多い。いずれの場合もプロセス自体は共通で、検知から解決までを一貫して扱う。

### イベント → 関心イベント → インシデントの絞り込み

すべてのイベントを調査する時間もリソースもないため、段階的に絞り込む。

| 概念 | 定義 | 例 |
|------|------|-----|
| イベント（Event） | 観測可能な出来事。異常を意味しない | 電源投入、ログイン、Web アクセス |
| 関心イベント（Event of Interest） | 深掘りする価値のある逸脱 | 連続ログイン失敗、想定外の場所からのログイン |
| インシデント（Incident） | 「何が起きたか解明が必要」なレベルに達した関心イベント | 侵害の証拠を伴うアラート |

- 関心イベントごとに検知手段（アラート）を定義しておく。検知→トリアージ→インシデント判定、が分析の入り口となる。

### インシデント対応プロセス

検知 → 分析 → 封じ込め → 根絶 → 復旧。直線的ではなく、調査結果に応じて分析と封じ込めの間をループする。

| 段階 | 要点 |
|------|------|
| 検知（Detection） | 関心イベントのアラート化。通知手段（画面・メール・メッセージ）は「人が気づく」ことが唯一の要件 |
| 分析（Analysis） | トリアージでインシデント該当性を判定。該当しなければクローズ。該当すれば発生箇所から情報収集し調査へ |
| 封じ込め（Containment） | 感染拡大の抑止。EDR やネットワーク制限でアクセスを絞る。**全容解明まで攻撃者を排除しない**（下記アンチパターン参照） |
| 根絶（Eradication） | 影響範囲の全リストが揃ってから一斉クリーンアップ（資格情報リセット・ワイプ・再構築）。清掃中に攻撃者が後追いで再侵害するリスクがあるため、**ネットワークを切り離してオフラインで実施**する選択肢を検討 |
| 復旧（Recovery） | 事業の通常運転への復帰＋根本原因分析（侵入経路・横展開経路）＋人・プロセス・技術のギャップ修復 |

### IoC（Indicator of Compromise）の運用

| ルール | 内容 |
|--------|------|
| 定義 | 攻撃者が環境内にいることを示す情報片（想定外の場所からのログイン、在席中の VPN ログイン等） |
| 全域スキャン | 新しい IoC を発見するたびに**環境全体**をその IoC でスキャンし、未特定の侵害システムを洗い出す |
| スコープ確定の目安 | 「新しい IoC が見つからなくなったとき」が全容把握完了のサイン。それまで根絶に進まない |

### クラウド環境での差分

| 観点 | オンプレミス | クラウド |
|------|--------------|----------|
| 可視性 | 自組織が全レイヤーを保有 | 基盤（物理・ハイパーバイザ・下位ネットワーク）はプロバイダ責任で参照不可 |
| 横展開の形 | 同一セグメント上のデスクトップ群を辿る | VM・コンテナ・サーバーレスは別々の仮想セグメントに分散。**資格情報とパーミッションを軸に調査**する |
| ログの主対象 | OS 中心 | OS から抽象化されたアプリケーション層、フローログ、セキュリティグループ、LB/Web/アプリサーバー |

### 根絶・復旧の見落としがちな対象

- DR サイト（ウォーム/ホットスタンバイ）が侵害環境を複製している可能性を確認する。
- バックアップは「要注意（suspect）」とマークし、攻撃者の再侵入経路となる成果物を復元しないようにする。
- 手動・アナログ運用に切り替えていた場合は、通常運転への切り戻しまでを復旧に含める。

### 発見事項の優先順位付け（脆弱性管理への接続）

インシデントで判明した露出・恒常的な脆弱性管理の両方で、明確な枠組みに基づいて優先順位を付け、完全に修復してからクローズする。

| 枠組み | 用途 |
|--------|------|
| DREAD（Damage Potential / Reproducibility / Exploitability / Affected Users / Discoverability） | 発見事項の相対優先度付け。発見も悪用も容易な脆弱性は最優先で即時修復 |
| CVSS ＋ 悪用可能性情報（既知悪用リスト・エクスプロイト公開状況） | 定常的な脆弱性トリアージ。基礎スコアだけでなく自環境での到達可能性・資産重要度で補正 |

脆弱性管理の定常サイクル: 資産インベントリ → スキャン/検出 → トリアージ（優先度付け） → 修正（パッチ・設定変更・緩和策） → 修正の検証 → 記録とメトリクス化。インシデント対応の Recover で見つかったギャップも同じサイクルに合流させる。

## ログ・監視・SIEM/SOAR

### ロギングの原則

可視性はセキュリティ運用の前提であり、可視性はロギングから始まる。「見えないものは検知できない」。

| 原則 | 内容 |
|------|------|
| リモート集約必須 | ローカルログは侵害端末上で攻撃者に改竄され得る。ログサーバー等へ必ず転送し、証跡の信頼性を保つ |
| 一元化 | クラウドプロバイダのログ収集サービス（例: Azure Monitor、Google Cloud Logging）または自前基盤へ集約し、保管・相関・検索を一箇所で行う |
| クラウドのログ源 | アプリケーションログ、フローログ、セキュリティグループ、LB・Web・アプリサーバー |
| フォーマット | syslog（Unix 系）と Windows Event ログが代表。収集サーバーは転送受信とエージェント収集の両方に対応させる |

### 監視パイプライン

ログ生成 → 収集（正規化） → 保管（検索可能なインデックス付きストア） → 分析・相関 → アラート → 人によるトリアージ。

| 落とし穴 | 対策 |
|----------|------|
| ログの大半は正常業務の記録でノイズ | ルールベースの相関でパターンを抽出し、人には**アラートだけ**を見せる（生ログの目視監視は成立しない） |
| 異常＝即問題ではない（例: いつものユーザーがログインしない日） | コンテキスト（休暇・退職等）と突き合わせるエンリッチメントを組み込む |
| マイクロサービスでは 1 リクエストが多数のイベントに分散 | 関連イベントを相関させて 1 つのまとまりとして提示する仕組みを監視チェーンに入れる |

### SIEM（Security Information and Event Management）

ログ・脅威インテリジェンス・脆弱性データ等を取り込み、正規化・保管・索引化し、ルールと照合してアラートを生成するシステム。

| 機能 | 内容 |
|------|------|
| 取り込み | ログ、脅威フィード、脆弱性データ等の外部情報 |
| 相関・ルール | 自組織定義のルール＋脅威フィード由来の「IoC ↔ 既知の脅威アクター」マッピング |
| エンリッチメント | IP → 逆引き FQDN、whois、地理情報。過去の類似アラートの処理結果（false positive 履歴等）も付与 |
| フィードバック | アナリストの false positive 判定を学習材料として蓄積し、以後のアラートを文脈化 |
| レポート | アラート件数・応答速度・最終処理結果等のダッシュボード。コンプライアンス証跡としても利用 |

### SOAR（Security Orchestration, Automation, and Response）

| 項目 | 内容 |
|------|------|
| 役割 | アラートを起点に、エンドポイントからの追加データ取得（予備調査）や定型対応を自動実行 |
| プレイブック | アラート種別ごとの自動アクション手順書。単純なケースでは初動アラートからクローズまで自動処理できる |
| 位置づけ | SIEM の補完。多くのクラウドプロバイダの SIEM ソリューションは SOAR 機能を内包 |

### マルチクラウドの注意

プロバイダ間でログを相互取り込みする場合、ログデータをオープンなインターネットに素通しで流さないよう、プロバイダ間の相互接続（専用線・プライベート接続）を含めてネットワークを設計する。

## セキュリティエンジニアリング

セキュリティ運用チームが使う**ツールチェーン全体**（ロギング基盤・監視・アラートルール）を構築・維持する機能。脅威インテリジェンスを取り込み、関心イベントに対する検知ルールへ変換する役割を担う。

| 責務 | 内容 |
|------|------|
| ログ基盤 | 収集・正規化・保管・検索基盤の構築と維持 |
| 検知エンジニアリング | 脅威インテリジェンス → 相関ルール → アラートへの変換。ルールのチューニング（false positive 削減） |
| ツールチェーン管理 | SIEM/SOAR/EDR 等、SOC が依存する全ツールの選定・統合・運用 |

### 堅牢化（ハードニング）

攻撃対象領域を最小化する予防的統制。Protect 機能の中核。

| 施策 | 要点 |
|------|------|
| ベースライン適用 | 公開されている堅牢化ベンチマーク（CIS Benchmarks 等）を OS・コンテナ・クラウドサービスの基準線とする |
| 攻撃面の縮小 | 不要なサービス・ポート・アカウント・パッケージを削除。デフォルト認証情報を排除 |
| 最小権限 | 人・サービスアカウントとも必要最小限のパーミッションに限定。クラウドでは IAM が横展開の主経路になるため特に重要 |
| ネットワーク分離 | セグメンテーション（封じ込めの事前準備でもある）。クラウドではセキュリティグループ・仮想ネットワーク境界で実装 |

### 構成管理

| 施策 | 要点 |
|------|------|
| ベースラインの文書化 | 「あるべき構成」をコード（IaC）またはポリシーとして定義し、レビュー可能にする |
| ドリフト検出 | 実構成とベースラインの乖離を継続監視。乖離は関心イベントとして扱う（攻撃者による変更の可能性） |
| 変更管理 | 構成変更を承認フローと監査ログに乗せる。「誰がいつ何を変えたか」が調査時の証跡になる |

### パッチ管理

| 手順 | 要点 |
|------|------|
| 把握 | 資産インベントリと脆弱性情報（ベンダー通知・既知悪用リスト）の突き合わせ |
| 優先度付け | 悪用容易性 × 露出（インターネット到達可能か） × 資産重要度で決定。緊急パッチの SLA を事前定義 |
| 適用 | 検証環境 → 段階的ロールアウト。クラウドでは可変インフラの場合、パッチ適用済みイメージへの再デプロイ（イミュータブル方式）を優先 |
| 検証 | 適用後の再スキャンで修正を確認。適用できない資産は緩和策（分離・仮想パッチ）と例外管理台帳に記録 |

## アンチパターン集

| アンチパターン | 問題 | 正しいアプローチ |
|----------------|------|------------------|
| 発見即排除 | 全容解明前に攻撃者を追い出すと、攻撃者は把握されていない足場を温存して潜伏化し、数か月後に再検知することになる | IoC が収束する（新規発見が止まる）まで封じ込めに留め、全容確定後に一斉根絶 |
| 予防偏重の統制配分 | 境界防御に全リソースを投じ、Detect/Respond/Recover が空洞化 | 「侵害される前提」で攻撃ライフサイクル全域に統制をマッピング |
| ローカルログ依存 | 侵害端末のログは攻撃者に改竄され、証跡が失われる | リモート集約と改竄困難な保管を必須化 |
| 生ログの目視監視 | 人間はスクロールするログから異常を拾えない | ルール相関＋アラート駆動のトリアージに限定 |
| アラートを出して終わり | 対応者がいなければ「知らないより悪い」状態 | アラートごとに対応責任・トリアージ手順を定義 |
| Kill Chain の無批判な採用 | 偵察・武器化は統制不能、かつマルウェア前提で現代型攻撃を見落とす | 攻撃ライフサイクル＋ATT&CK を主軸にし、Kill Chain は補助的に使う |
| 単一システムの隔離で完了と判断 | 攻撃者（多くは複数人の組織）は他システムにも足場を持つ | 横展開を前提に環境全体を IoC でスキャン |
| バックアップ・DR サイトの無検査復元 | 侵害済み成果物の復元により攻撃者が再侵入 | バックアップを suspect 扱いにし、DR サイトも調査範囲に含める |
