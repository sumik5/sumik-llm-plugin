# セキュア開発ライフサイクルとクラウド開発パターン

クラウド環境でのソフトウェア開発にセキュリティを統合するためのリファレンス。SDLC の各フェーズへのセキュリティ組み込み、DevOps / DevSecOps の実践、脅威モデリング（STRIDE / PASTA / DREAD）、CI/CD パイプラインへの自動化ゲート、Infrastructure as Code、SRE（SLI/SLO/SLA）、そして設計スタイル（多層アーキテクチャ・API・クラウドネイティブ・マイクロサービス・イベント駆動）ごとのセキュリティ含意をカバーする。

---

## 1. SDLC とセキュリティ統合の全体像

### 1.1 なぜ SDLC にセキュリティを組み込むか

| 原則 | 内容 |
|------|------|
| セキュリティはプロセス | 大規模開発と同様、セキュリティも単発の作業ではなく継続的プロセス。両プロセスの接合点を計画的に設計する |
| 早期修正ほど低コスト | セキュリティ問題は要件・設計フェーズで対処するほど修正コストが劇的に下がる（シフトレフトの根拠） |
| 協働が前提 | セキュリティチームが開発プラクティスを理解し、開発チームがセキュリティ目標を理解する双方向の歩み寄りが必要 |
| ビジネス目標への整合 | セキュリティも開発も最終的にはビジネス目標に貢献する活動として位置づける |

### 1.2 SDLC 方法論の比較

| 方法論 | 特徴 | セキュリティ上の含意 |
|--------|------|---------------------|
| ウォーターフォール | 要件→設計→開発→テスト→デプロイ→保守が逐次進行。前段完了が後段の前提 | チーム間が分断され問題の協調的解決が遅れる。改善のフィードバックループが構造的に欠如 |
| 反復型（RAD / XP） | 小さな増分変更・コミュニケーション重視 | 変更単位が小さく、セキュリティレビューを回しやすい |
| アジャイル | 顧客との協調・動くソフトウェア・変化への対応を重視する価値観の集合 | 2〜3週間のスプリント内で SDLC 全フェーズを完結させる前提。セキュリティ活動もスプリント内に収める設計が必要 |
| DevOps / DevSecOps | 開発・運用（＋セキュリティ）の統合と自動化 | セキュリティゲートをパイプラインに自動化して組み込める |

**アジャイルのアンチパターン**: 「アジャイル採用」を口実にプロセスと文書を放棄すること。プロセスの欠如は品質管理の劣化、文書の欠如は最終成果物の理解不足を招き、いずれもセキュリティ観点で有害。スプリント分割とチャットツール導入だけではアジャイルの意図を満たさない。

代表的フレームワーク: Scrum（自己組織化チーム・意思決定の現場移譲）、Kanban（可視化されたワークフロー管理）、Lean（リーン製造原則の翻訳）、Scrumban（ハイブリッド）。

## 2. DevOps と DevSecOps

### 2.1 DevOps の中核原則

| 原則 | 内容 |
|------|------|
| 運用をステークホルダー化 | Web 配信への移行により、開発組織自身が運用も担うようになった。運用要件（監視性・管理性）を開発要件に含める |
| 自動化最優先 | インフラ構築・テスト・デプロイのすべてを自動化し、一貫性と速度を得る |
| 品質と速度の両立 | アジャイルの開発速度を運用面から支える |
| コミュニケーション | 開発と運用の分断をなくす。単に同室に押し込むことではなく、原則とプラクティスの共有 |

**構成ドリフトの罠**: システムイメージの自動デプロイ（PXE / Kickstart / VM クローン等の系譜）は初期状態の一貫性しか保証しない。デプロイ直後からドリフトが始まり、性能劣化・複雑化・脆弱性混入の原因になる。デプロイ後の継続的な構成管理・更新の自動化が別途必要。

### 2.2 DevSecOps: セキュリティをチームに加える

DevOps は品質と速度を目指すが、セキュリティは自動的には含まれない。DevSecOps はセキュリティをフルメンバーとしてチームに統合する。

セキュリティチームの責務:

- 強固な identity and access management（IAM）プラクティスの全面適用
- すべてのシステムイメージのハードニング
- セキュリティテストのパイプライン実装
- 要件定義段階からの参画
- 脅威モデリングのファシリテーション

## 3. 脅威モデリング

### 3.1 脅威と脆弱性の区別

| 用語 | 定義 |
|------|------|
| 脅威（threat） | システム・ユーザー・データに危害を及ぼしうる状況。理論上の問題 |
| 脆弱性（vulnerability） | アプリケーション/システムの弱点。脅威が実際に悪用される現実的な入口 |

脅威は脆弱性がなければ悪用できない。両者を分けて分析する。

### 3.2 STRIDE（脅威の分類タクソノミー）

| カテゴリ | 内容 |
|----------|------|
| Spoofing | 別のエンティティになりすます |
| Tampering | 情報の不正な改ざん |
| Repudiation | 行為者の特定・同一性の保証ができない（否認） |
| Information Disclosure | アクセス権のない者にデータが漏れる |
| Denial of Service | システム/データへのアクセス妨害 |
| Elevation of Privilege | 追加の権限・特権を不正に取得する |

非構造的なブレインストーミングより、タクソノミーを使った方が開発チームでも脅威を洗い出しやすい。

### 3.3 PASTA（プロセス）

Process for Attack Simulation and Threat Analysis。STRIDE と排他ではなく組み合わせて使う。

1. **目標定義** — ビジネス・セキュリティ・コンプライアンス要件を明確化
2. **技術スコープ定義** — 対象アプリの全コンポーネントを特定
3. **アプリケーション分解** — 信頼境界とコンポーネントを含む図式化
4. **脅威分析** — 脅威インテリジェンスと分解結果から脅威を特定・分類
5. **脆弱性分析** — 特定した脅威に対して悪用されうる脆弱性を特定
6. **攻撃モデリング/シミュレーション** — 攻撃ベクターをマッピングし悪用経路を特定
7. **リスク/影響分析** — 影響と発生可能性で優先順位づけし、修復計画を立てる

### 3.4 DREAD（優先順位づけ）

| 軸 | 問い |
|----|------|
| Damage potential | アプリ・組織・データ・顧客への潜在的影響は？ |
| Reproducibility | 脆弱性はどれだけ容易に再現できるか？ |
| Exploitability | 悪用の容易さは？（必要スキルレベル・攻撃ツールの入手性を含む） |
| Affected users | 影響を受けるユーザーの範囲・権限レベルは？ |
| Discoverability | 攻撃者はどれだけ容易に発見できるか？ |

**推奨の組み合わせ**: PASTA（プロセス）＋ STRIDE（特定のタクソノミー）＋ DREAD（優先順位づけ）。過度な形式化は不要で、緩い概念として適用してもよい。重要なのは「潜在的問題の特定に時間を割く」こと。セキュリティ実務者が脅威特定を支援し、開発チームがアプリ内部の知見を提供する分業が機能する。

## 4. CI/CD パイプラインとセキュリティゲート

### 4.1 パイプラインの標準構成と各段のセキュリティ

| 段階 | 内容 | セキュリティゲート |
|------|------|-------------------|
| コミット / PR | バージョン管理（Git が事実上の標準）へ push、pull request 作成 | MFA 必須のリポジトリアクセス。コードレビュー（脆弱性検出の第一線） |
| マージ | 主コードベースへ統合 | SAST（静的アプリケーションセキュリティテスト）を自動トリガー |
| ビルド | コンパイル、またはインタープリタ言語の健全性チェック | 依存関係スキャン・SBOM 自動生成 |
| テスト | ユニットテスト実行 | ミスユースケース（misuse case）ベースのセキュリティユニットテスト |
| 動的テスト | 統合的な自動テスト | DAST ツール（OWASP ZAP・Burp Suite 等）のパイプライン統合 |
| デプロイ | ステージング→本番 | スモークテスト・段階的リリース（フェーズドロールアウト） |

**フェイルクローズ原則**: パイプラインのどの段階でも否定的な結果が出たら停止する。失敗や潜在的脆弱性が本番に到達するのを構造的に防ぐ。

**デプロイの判断基準**: 完全自動で本番まで押し出すことも可能だが、ステージング環境への段階的リリース＋スモークテストを挟むのが安全。変更の重要度に応じて、一部ユーザーへの先行リリース（段階的デプロイ）で評価する選択肢もある。

### 4.2 コードリポジトリのセキュリティ要件

- **強固なアクセス制御**: 多要素認証を理想とする。ホスティング型（GitHub / GitLab 等）は自組織 IAM とのフェデレーションを提供することが多い
- **監査証跡**: 認証済みユーザー名を全変更に紐づけ、問題発生時に変更の連鎖を再構成できるようにする（悪意の有無を問わず、破壊的変更の起点特定に必要）
- **コードレビューの強制**: 変更受け入れ前に別の（できればシニアな）開発者によるレビューを必須化する自動化フック
- **バックアップ**: 集中リポジトリは障害に備えてバックアップし、端末ローカルのみのコード保持を避ける

### 4.3 サプライチェーンリスク

| リスク | 内容 | 対策 |
|--------|------|------|
| パッケージリポジトリの侵害 | npm / PyPI 等で大量の侵害パッケージが発見された実績がある。攻撃者は可能な限り上流を狙う | 信頼できるソースのみ使用。導入前評価プロセスを持つ |
| typosquatting | 打ち間違えやすい名前で悪性パッケージを登録し、誤入力したユーザーに配布する | 依存追加時の名前の厳密確認・依存レビュー |
| 依存経由の脆弱性 | OSS・商用を問わずどのソフトウェアソースも侵害されうる | SBOM（software bill of materials）を維持 |

**SBOM 運用の要点**: ソフトウェアを構成する全コンポーネントの包括的インベントリ。ビルドプロセスで自動生成すれば、常にビルド実体と一致した最新状態を保てる（手動更新は陳腐化する）。

## 5. Infrastructure as Code（IaC）

### 5.1 概念と利点

IaC は環境の全要素（コンピュート・ストレージ・ネットワーク・ソフトウェア構成）を記述された構成として定義・デプロイする手法。すべてが仮想＝ソフトウェアであるクラウドでは、構成をコードとして扱える。

| 利点 | 説明 |
|------|------|
| 再現性・一貫性 | 何度デプロイしても同一の結果 |
| テスト可能 | ソース管理に入れ、期待結果に対して検証できる |
| 変更容易性 | ハードウェア交換ではなくコード変更で済む |
| 自己文書化 | 構成コード自体がプロセスの明確なドキュメントになる |
| 差分適用 | 多くの IaC 基盤は目標状態を追跡し、変更分のみ適用する（plan → apply の 2 段階で変更内容を事前確認できる） |

### 5.2 ツールの分類

| ツール | 記述言語 | 特徴 |
|--------|---------|------|
| Terraform | HCL | 宣言的。provider ブロックでターゲット（AWS 等）を指定し、plan で差分確認後に apply |
| Ansible / Salt | YAML | 宣言的構成 |
| Puppet / Chef | Ruby ベースの DSL | 独自言語 |
| PowerShell | スクリプト言語 | 完全なプログラミング言語。クロスプラットフォーム・マルチクラウド対応モジュールあり |
| 汎用言語＋クラウド SDK | Python 等 | 柔軟だが可読性が下がりやすい |

宣言的な構成記述の例（Terraform / HCL）:

```hcl
terraform {
  required_providers {
    aws = { source = "hashicorp/aws", version = "~> 5.0" }
  }
}

provider "aws" {
  region = var.aws_region
}

variable "cluster_name" {
  type    = string
  default = "small-container-cluster"
}
```

### 5.3 IaC のパイプライン統合とセキュリティ

- IaC 自体をコードパイプラインに通す: push 時に構文評価・ミスコンフィグ検出ツールを実行してから apply
- **シークレットスキャン必須**: 保護されたリポジトリであっても、認証情報をディスク上（コード内）に置くのは悪手。シークレット検出ツールをパイプラインに常設する
- サードパーティコンポーネントを含む全構成要素のミスコンフィグ・脆弱性スキャンをデプロイフェーズで実施

## 6. SRE と信頼性メトリクス

Site Reliability Engineering は運用にソフトウェアエンジニアリングの厳密さを持ち込む実践。継続的モニタリングでプラットフォーム/アプリの信頼性を担保する。

### 6.1 SLI / SLO / SLA の階層

| 用語 | 定義 |
|------|------|
| SLI（Service Level Indicator） | 環境の振る舞いを示すメトリクス。典型: レイテンシ・スループット・可用性・エラー率 |
| SLO（Service Level Objective） | SLI が収まるべき許容範囲。逸脱すると成果や性能が保証できない |
| SLA（Service Level Agreement） | SLO の集合として構成される合意。いずれかの SLI が SLO 範囲外になると SLA 違反となりうる |

SLA は外部プロバイダーとの契約に限らない。社内の組織単位間でも非公式な SLA を設定すれば業務が予測可能になる。

### 6.2 可用性（nines）と許容ダウンタイム

| 稼働率 | 年間ダウンタイム |
|--------|------------------|
| 99.9%（スリーナイン） | 8.76 時間 |
| 99.99%（フォーナイン） | 52.56 分 |
| 99.999%（ファイブナイン） | 5.26 分 |

9 が 1 つ減るごとにダウンタイムは約 10 倍。メンテナンスウィンドウを取る運用では高可用性目標の達成が難しくなる点に注意。

### 6.3 計測の実装要件

- メトリクスは自然には集まらない。ロギング・外部監視（合成監視: Web リクエストを定期送信して稼働確認等）を設計段階から組み込む
- 収集したメトリクスの保存・レポート基盤と、SLI が許容範囲を逸脱した際のアラートを用意する（SIEM 等のログ分析・アラート基盤を活用）

## 7. シフトレフト: フェーズ別セキュリティ活動

開発プロセスを左（企画）から右（デプロイ）へ並べたとき、セキュリティ活動をできるだけ左（早期）に押し込むのがシフトレフト。早いほど修正コストが低い。

| フェーズ | セキュリティ活動 |
|----------|------------------|
| 計画（要件） | 既存要件のセキュリティ観点レビュー（IAM・暗号化・分離を担保するアーキテクチャ）。セキュリティ独自要件の追加。開発チームへのセキュリティ視点の啓発 |
| 設計 | 脅威モデリングの実施。信頼境界と分離の設計レビュー。アクセス制御とユーザー/管理者ロールの定義 |
| 開発 | セキュア開発プラクティス・スタイルガイドの共有。セキュリティ指向ユニットテスト（ミスユースケース）。SAST のパイプライン組み込み。シークレットスキャンの導入 |
| テスト | セキュリティテストケースの追加。DAST 等の動的テストツールの自動化・パイプライン統合 |
| デプロイ | IaC・サードパーティコンポーネントを含む全要素のミスコンフィグ/脆弱性スキャン |
| 運用（保守） | 継続的モニタリング（SLI/SLO）・ログ分析・アラート・脆弱性対応とアップデート |

## 8. 設計スタイルとセキュリティ含意

### 8.1 多層（n-tier）アーキテクチャ

プレゼンテーション層 / アプリケーション層 / ビジネスロジック層 / データアクセス層に分離する古典的パターン。クラウドでもそのまま実装可能。

**セキュリティ含意**:

- 層ごとに機密度が異なるため、need-to-know に基づく分離が可能
- 各層を独立ネットワークに分離し、ファイアウォール / ACL で層間を制御する。公開面（Web サーバー）を突破されても DB サーバーへ直接跳べない構造にする
- プレゼンテーション層はユーザーの支配下（アプリ所有者は制御できない）。アプリ層は公開面なので静的コンテンツ中心・データ最小に。機密データは内側の層に置く

### 8.2 API 中心設計

モバイルアプリと Web の両プレゼンテーションを持つ場合、バックエンドを API に一本化してインフラの重複を避ける。API ゲートウェイがルーターとして HTTP リクエストを適切なハンドラーへ振り分ける。

**API セキュリティの要点**:

| 項目 | 内容 |
|------|------|
| HTTP verb | GET（取得）/ POST・PUT（送信）/ DELETE（削除）。verb ごとにエンドポイントの認可を設計 |
| トークン認証 | `Authorization: Bearer ...` ヘッダーで認証済みを示すトークンを検証。JWT（Base64 エンコード・署名アルゴリズムとクレーム＝ユーザー・権限・有効期限を内包）や OAuth トークン（認証自体は SAML 等で実施し、認可情報を共有） |
| API キー | アプリ/サービス単位に事前発行するランダム値をヘッダー（例: `X-API-Key:`）で送信。いつでも失効でき、該当アプリのアクセスだけ止められる |

### 8.3 クラウドネイティブ設計

#### コンテナ化

OS 全体でなく単一アプリを仮想化する。カーネルの namespace で分離され、アプリ実行に必要なバイナリ・ライブラリのみ同梱（Docker が代表的な管理ソフトウェア）。

| 特性 | セキュリティ含意 |
|------|------------------|
| 軽量・高速起動 | 常時起動が不要 → 攻撃者が足がかりを得る待ち受け時間が減る |
| 短命（ephemeral） | 侵害されてもコンテナ終了で攻撃者が排除される |
| OS 非依存の分離 | コンテナ内の足がかりから OS 本体へ到達しにくい |
| イミュータブル更新 | コード差し替え後の次回起動で最新版が動く → パッチ適用が容易 |

#### オーケストレーション

Kubernetes 等が複数コンテナを pod にまとめ、その上の service が受信リクエストを pod 内の実行中インスタンスへ分配する。オートスケール（需要に応じた spin up / tear down）はオーケストレーションがあって初めて成立する。

#### マイクロサービスとイベント駆動（バス）アーキテクチャ

- モノリシックな単一バイナリではなく、単機能のサービス群に分割してそれぞれコンテナ化する
- サービス間連携は (a) 待ち受けサービスへの直接呼び出し、または (b) **メッセージバス**経由。バスでは送信側がメッセージ（リクエスト＋データ）をバスに置き、宛先サービスが取得して処理し、応答をバスに戻す。関数呼び出しの分散版として機能し、サービス同士を疎結合にする
- **サーバーレス関数**: コンテナすら持たず、コードのみが存在。トリガー受信時にランタイム（Python / Java / .NET 等）が起動してコードを実行する。マイクロサービスの実装手段の一つ

#### クラウドネイティブの必須要素

| 要素 | 内容 |
|------|------|
| スケーラビリティ | 需要に応じた拡張・縮退でコストを実使用量に近づける |
| マイクロサービス | 構成要素への分割がスケーリングを支える |
| コンテナ化 | 実行に必要な最小構成のみ同梱 → **攻撃面の縮小** |
| 動的オーケストレーション | オートスケールの実現機構 |
| 可観測性 | モニタリング・ロギング・トレーシングで機能性と稼働を常時確認 |

### 8.4 データストアの選択（NoSQL）

| タイプ | 構造 | 代表例 | 適する用途 |
|--------|------|--------|-----------|
| ドキュメント | JSON/XML 等の半構造化文書を格納 | MongoDB | CMS・カタログ・ユーザープロファイル |
| キーバリュー | key → value の辞書 | Redis | ショッピングカート・セッション・キャッシュ |
| カラムファミリー | 行が構造を定義し、行ごとに列構成が異なってよい | — | 時系列データ・IoT |
| グラフ | ノードとエッジの関係で格納・探索 | — | ソーシャルネットワーク・不正検知 |
