# クラウドデータセキュリティ

データは組織の最重要資産であり、クラウドではデータが従来の境界（ペリメータ）の外に出るため、ライフサイクル全体（作成→保存→利用→共有→アーカイブ→破棄）を通じた保護設計が不可欠になる。本リファレンスは、クラウドデータの概念（ライフサイクル・分散・タイプ・ロケーション）、ストレージアーキテクチャと脅威、設計考慮点、データ保護技術（暗号化・鍵管理・トークナイゼーション・マスキング・匿名化・DLP）、Information Rights Management（IRM）、可視性・監査の実践パターンをまとめる。

---

## データライフサイクル

データは永続するものではなく、明確な終わりを持つライフサイクルとして管理する。古いデータの無期限保持はコストであると同時に攻撃対象・法的リスクになる。

| 段階 | 内容 | 主なセキュリティ考慮点 |
|------|------|----------------------|
| 1. 作成 / 取込（Create/Ingest） | 人手入力・アプリ生成・ログ・センサー等の外部ソースからの取込 | 生成時点での分類付与、取込元の信頼性検証 |
| 2. 保存（Store） | メモリ（一次記憶）→ディスク等の二次記憶。クラウドでは常にネットワーク越し | 保存時暗号化、バックアップ（オンライン侵害への備え） |
| 3. 処理 / 整理（Process/Organize） | 生データを利用可能な形式へ変換・格納（ファイル / RDB / NoSQL） | 変換過程での平文露出の把握、一時領域の管理 |
| 4. 利用 / 分析（Use/Analyze） | アプリがデータをメモリへ展開して読取・分析（人手を介さないバッチ含む） | アクセスログ（メタデータ）の収集、利用時暗号化の要否 |
| 5. アーカイブ / 保持（Archive/Retain） | 低頻度アクセスデータの退避。保持ポリシーに基づく | アーカイブ先にも同等のアクセス制御・暗号化を適用 |
| 6. 削除 / 破棄（Delete/Destroy） | 論理削除ではなく復元不能な破棄まで完遂する | 「削除≠破棄」。バックアップ・複製・版も対象に含める |

> 共有（Share）を独立段階に置くフレームワーク（Create→Store→Use→Share→Archive→Destroy）もある。いずれの枠組みでも、段階ごとに必要な統制が異なる点が本質。

### 保存の階層（アクセス即時性）

| 区分 | 特性 | 例 |
|------|------|-----|
| オンライン | 即時アクセス可能 | 稼働中ストレージ |
| ニアライン | 追加操作（リモートシステムへの接続等）で到達可能 | ネットワークファイル共有、低頻度アクセス層 |
| オフライン | 物理的操作が必要 | 外付けディスク、テープ |

オフライン媒体は「接続すれば読める」状態になりがちなので、施錠管理か媒体暗号化を必須とする。

### 保持（Retention）ポリシーの駆動要因

- 規制・業界ガイダンスへの準拠（一定期間後の削除を求めるものもある）
- 古いデータの保持自体が攻撃者の標的価値を高める → 不要データの削除は露出削減
- ストレージコスト
- データ分類に応じた保持期間の差別化（機微度が高いほど厳格に）

### 削除と破棄（Deletion vs Destruction）

| 論点 | 要点 |
|------|------|
| OSの「削除」 | ごみ箱→空にする、でも実体はディスクに残存しうる（メタデータ上の削除マークのみ）。フォレンジックツールで容易に復元可能 |
| ブロック再利用の罠 | 大きいファイルは複数ブロックを占有。新ファイルが一部ブロックしか使わなければ、残りから断片が復元できる |
| 磁気ディスクの論理破棄 | 上書き消去。NIST SP 800-88 では 1 回の上書きで十分とされる |
| SSD | ウェアレベリングでセルのアドレスが固定されないため上書き戦略が通用しにくい。TRIM による未使用ブロック消去にドライブ側が対応 |
| 暗号消去（Crypto Shredding） | 強い暗号＋大きな鍵で暗号化し、鍵を破棄する。上書きと同等の効果。鍵がバックアップ・自動化経由で複製されていないことの確認が前提 |
| 物理破棄 | ディスクの物理シュレッディング。大容量ディスクの上書きは時間がかかるため、確実性重視なら物理破壊 |
| バックアップの破棄 | 破棄要件はバックアップにも及ぶ。長期保管バックアップは破棄対象データの負債になる |
| パブリッククラウドでの破棄 | 媒体はプロバイダの所有。全痕跡の破棄をプロバイダ責務として**契約に明記**し、**監査権（right to audit）**を含める |
| 責任共有モデルの活用 | 顧客側で暗号化し鍵を顧客のみが保持すれば、プロバイダ側に残存データがあっても鍵なしでは読めない |

## データ分散（Data Dispersion）とデータマッピング

データは複製・レプリケーション・バックアップ・利便性目的の移動などで自然に拡散する。分散は「複数コピーのうちどれが正か分からない」バージョン問題や、把握外の保存先（＝破棄漏れ・保護漏れ）を生む。

### データマップに含めるべき要素

- [ ] データソース（発生源・取込元）
- [ ] データタイプ
- [ ] 保存ロケーション（すべて）
- [ ] 処理ポイント
- [ ] アクセス制御
- [ ] 保持期間
- [ ] データ変換（transformation）

作成責任は「データの動きを最もよく知る」アプリケーション / IT チームに置く。

### データフロー図（DFD）

| 項目 | 指針 |
|------|------|
| 含める内容 | システム間を流れる**データの種類**（認証情報・注文情報・決済情報等）、暗号化等の保護策、ポート・プロトコル |
| 抽象度 | ネットワーク図・アーキテクチャ図より抽象的でよい。VM やコンテナの詳細は不要。逆にネットワーク図にはデータ種別が載らないので相互補完 |
| 信頼境界（Trust Boundary） | 信頼・保護レベルの異なる要素間の論理的境界線を明示する。暗号化には鍵が要るため「どのシステムが平文を見られるか」が境界を定める |
| 変換の記録 | 復号も変換。平文化された機微データがどこに現れるかを記録する。文書形式変換（例: 文書→PDF 生成）はコピーが増える操作として明記 |
| 付随文書 | プロトコル・保護策の選定理由（justification）を残す |
| 保守 | アプリ変更・データ要件変更に追随して更新するプロセスを持つ。**陳腐化した DFD は追跡外の場所への情報漏出（＝処分漏れ）を生む** |

## データタイプ（構造の観点）

| タイプ | 定義 | 例 | 含意 |
|--------|------|-----|------|
| 構造化 | 事前定義モデル・一貫した編成・要素間の明確な関係 | SQL データベース（スキーマあり）、スプレッドシート | スキーマベースの検出・分類・統制が容易 |
| 半構造化 | 共通の骨格はあるが厳密・不変ではない。自己記述的 | JSON / XML、NoSQL、メールヘッダ、ログ（syslog 等） | フィールドの欠落・自由記述が混在。分類・DLP 検出が難しくなる |
| 非構造化 | 定義済み構造なし | 自由文書、画像、音声 | 分類は内容解析かラベル付けに依存 |

クラウドサービスは保存方式を抽象化するため、利用者はデータそのものの管理・保護に集中できる一方、タイプに応じたストレージ / アプリ選定は依然必要。

## データロケーションと規制

| 観点 | 要点 |
|------|------|
| 規制の基準の違い | 事業者所在地やデータ主体所在地を基準にする法制もあれば、GDPR のように**データの保存場所**を基準に適用される法制もある。中国等も強い規制を持つ |
| リージョン選択 | 大手プロバイダはマルチリージョン / デュアルリージョン / 単一リージョンを選択可能。顧客への近接性（性能）と規制適用の両面で決める |
| 設計への組込み | 顧客所在地・データ保存場所・適用規制の組合せをアーキテクチャ意思決定の入力にする。規制は変化し続けるため継続ウォッチ |
| データレジデンシーの罠 | データ所在地とデータ主体の不一致により予期しない法律が適用される（コンプライアンス脅威の代表例） |

## クラウドストレージアーキテクチャ

### ストレージタイプの選定

| タイプ | 特性 | 用途 / 注意点 |
|--------|------|--------------|
| インスタンス（エフェメラル）ストレージ | VM に付随する一時ディスク。**インスタンス終了（terminate）で消滅**（停止・一時停止は別） | OS ボリューム等。重要データを置かない |
| オブジェクト / ブロブストレージ | URL（Web プロトコル）でアクセス。メタデータ・バージョニング・一意識別子（GUID）付き。下層ファイルシステムは非公開 | 汎用の長期保存。ローカルディスクのようにマウントする使い方は性能面で不利 |
| 低頻度アクセス層（warm） | 可用性は通常層と同等、保存コスト低・アクセスコスト高・最低保持期間の設定あり | バックアップ・DR 用。アプリ実行用ではない |
| アーカイブ / コールド層 | 取り出しに数時間かかることがある | コンプライアンス保管・長期バックアップ |
| ブロックストレージ | VM に通常ディスクとして見える。パーティション・フォーマット等を利用者が完全制御。VM を破棄しても存続し、別 VM へ付け替え可能 | DB 格納や OS と分離したデータ領域。ファイルシステム互換性のみ注意 |
| ファイルストレージ（マネージド共有） | SMB / NFS でマウントする階層型ファイル共有 | 従来の社内ファイルサーバ相当。遠隔アクセス時のレイテンシ・帯域に注意 |

**判断基準**: コンピュートと運命を共にしてよいか（エフェメラル可否）→ アクセス頻度・取り出し許容時間（hot / warm / cold）→ アクセス様式（URL / ブロックデバイス / ファイル共有）。

## ストレージへの脅威

### アクセス脅威

| 脅威 | 対策 |
|------|------|
| IAM の設定ミス・過剰権限 | 最小権限。要件チーム⇔運用チーム間の齟齬・伝達ミスも原因になる前提でレビューする |
| ストレージバケットの公開設定 | 公開はデフォルトではなく**意図的な選択**。アカウントレベルで公開化自体をブロックする設定を使う。「公開バケットだから機微データはない」は成立しない（誰かが機微文書を置くのは防げない） |
| 認証情報の漏洩・推測可能パスワード・鍵の流出 | MFA 必須化。ただしプッシュ承認のみ・SMS コードは迂回されうるため、フィッシング耐性のある方式を選ぶ |
| 退職・異動後のアクセス残存 | アクセス失効の即時性を担保（オフボーディングプロセス） |

### データ保護脅威

| 脅威 | 対策 |
|------|------|
| 暗号化していても認証情報が破られる | 正規の資格情報を持つ者は復号もできる。暗号化と IAM は不可分。鍵の保護＝アクセス制御 |
| バージョン削除（ランサムウェアは支払いを確実にするため版を消しに来る） | バージョニング＋**イミュータブル（書換・削除不能）バックアップ** |
| データ消失・誤削除 | バージョニングと独立したバックアップの併用 |

### ネットワーク脅威

| 脅威 | 対策 |
|------|------|
| ソフトウェア定義ネットワークの設定ミス | ACL でアクセス元を限定。コンポーネント間通信も必要最小限に制約 |
| 管理プロトコル（SSH / RDP）のインターネット露出 | デフォルトの許可ルールが全世界に開くクラウドがある。送信元 IP 制限、VPN / 専用接続、NAT・踏み台経由に限定 |
| VPC 境界の甘さ | 外部公開が必要なサービスのみ公開し、残りは VPC 内部に閉じる |
| DoS（大規模フラッド / スロー攻撃によるコネクション枯渇） | 帯域面とアプリ面の両方で対策を検討 |

### 運用・コンプライアンス脅威

- **ロギング欠如**: 可視性ゼロでは悪用検知も性能問題の把握もできない。SaaS はプロバイダ任せでログ設計が漏れやすい
- **暗号化欠如**（保存時・転送時）: ただし暗号化は盗難を確実に防ぐものではない（アカウント侵害者はアプリ経由で平文を読める）
- **誤削除・ハードウェア障害・プロバイダ側の事故**: バックアップは利用者責任。「データの責任はプロバイダではなく自組織にある」
- **データレジデンシー不整合・保持期間超過・越境移転**: 規制違反の典型経路
- **監査・文書化要件の不備**: 変更管理の欠如は「誰が何をいつ承認したか」を追えなくする
- **データ分類の不在・不統一**: 所有者不明・責任未定義。DLP ツールは助けになるが導入・維持は容易ではない
- **SLA の不備**: 稼働率・障害時復旧・インシデント調査支援を契約で担保する

## 設計考慮点

クラウドデータ保護戦略は 3 つの軸を同時に満たすように設計する:
**① データ保護 ② 規制コンプライアンス ③ ビジネス効率・継続性**

### 設計チェックリスト

- [ ] **ビジネス目的から扱うデータ種別を特定**し、既存のデータ分類ポリシーを適用する（分類が取扱い＝保護レベルを決める）
- [ ] **暗号化の適用面を決める**: 転送時（クライアント⇔サーバだけでなくマイクロサービス間 API 呼び出しも）／保存時（ボリューム暗号化か DB 暗号化か、マネージド DB の場合の方式）／利用時
- [ ] **特に機微なデータはトークナイゼーション**（SSN・カード番号・患者 ID・口座情報等）
- [ ] **非本番環境には本番データを置かない**: マスキングで「本物に見えるが本物でない」データを用意する（非本番はソフトターゲット）
- [ ] **鍵管理**: 鍵の露出＝暗号化しなかったのと同じ。管理方式・暗号種別・鍵保護を設計する
- [ ] **IAM**: 強い資格情報プラクティス＋MFA、認可は最小権限。管理者には just-in-time 資格情報・専用アカウント・パスワードボールト・追加監視を検討
- [ ] **ロギング**: セキュリティ・性能・トラブルシュート・コンプライアンス報告のそれぞれの要件を文書化して適切なログレベルを決める
- [ ] **ライフサイクル全段階をカバー**: プロバイダの削除・破棄における役割を確認。**バージョニングにより「削除したはずの旧版」がプロバイダ側に残る**ケースを想定する
- [ ] **コーナーケース**: 非機微文書に機微情報が誤って書かれ、削除後も旧バージョンに残存するケースを扱えるか
- [ ] **BC/DR**: クラウド利用＝自動的に守られる、ではない。プロバイダ側の大規模障害や攻撃を想定し、重要データをバックアップ
- [ ] **RTO / RPO を定義**: RTO（復旧までの許容時間）は IaC による再構築で短縮できる。RPO（許容データ損失量）はレプリケーション / バックアップ頻度を決める

## データ保護技術

### 暗号化

「暗号化せよ」だけでは設計にならない。暗号種別・アルゴリズム・鍵強度・鍵保護までを一体で決める。

#### ハイブリッド暗号システムと PKI

| 概念 | 要点 |
|------|------|
| ハイブリッド暗号 | 公開鍵暗号でセッション確立（パラメータ・鍵交換）し、以降は対称鍵で通信を暗号化（TLS の基本構造）。非対称は計算コストが高いため、時間に敏感な本文には対称鍵を使う |
| 信頼モデル | 公開鍵暗号は推移的信頼に依存。分散型（web of trust、PGP 型）と、集中型の認証局（CA）がある。CA が本人性を検証して証明書を発行 |
| X.509 証明書 | Version / Serial Number / Signature Algorithm / Issuer / Validity（Not Before・Not After）/ Subject / Subject Public Key Info / Extensions（Key Usage・SAN 等）/ Signature で構成 |
| 失効管理 | 証明書は有効期限があり更新が必要。漏洩時は失効させる。CRL（失効リスト）と OCSP（オンライン照会）の 2 方式があり、方式の主流は変遷している |

#### 主なアルゴリズム

| アルゴリズム | 種別 | 特徴 |
|-------------|------|------|
| DSA | 非対称（署名専用） | デジタル署名の検証のみ。否認防止（non-repudiation）に使う |
| RSA | 非対称（汎用） | 素因数分解の困難性に基づく。長い鍵が必要。現役だが計算効率は劣る |
| ECC | 非対称 | 楕円曲線離散対数問題に基づく。**同等強度で大幅に短い鍵**（256-bit ECC ≒ 3072-bit RSA）。ストレージ・計算量で有利 |
| AES | 対称（ブロック） | 現在の標準。高速で複数の鍵長をサポート。ストリームモードでも利用可 |
| ChaCha20 | 対称（ストリーム） | ストリーム暗号の代表例 |

- **ブロック暗号**: 固定長ブロックに分割（必要ならパディング）して暗号化。前ブロックの情報を連鎖させるモードがある
- **ストリーム暗号**: 到着したデータを逐次（バイト単位等）暗号化

#### 鍵・シークレット管理

| 課題 | パターン |
|------|---------|
| 証明書パスワードをディスクに置くとアプリ起動用に発見可能になる | **キーボールト / シークレットボールト**を使い、アプリはワークロードアイデンティティでボールトに認証して取得（パスワードを露出させない）。クラウド設計の定石 |
| 事前共有鍵（PSK）の配送問題 | 256-bit のバイナリ鍵は人間が記憶・口頭共有できない。鍵交換プロトコルかボールト経由で管理 |
| 鍵のローテーション | 対称鍵は定期的に交換する前提で運用を設計する |

### ハッシュ（完全性検証）

| 性質 | 要点 |
|------|------|
| 固定長出力 | 任意長の入力から固定長の値を返す。ファイル内容のみを対象とし、ファイル名・タイムスタンプ等のメタデータは影響しない。1 bit の違いで出力が大きく変わる |
| 衝突（collision） | 出力空間が有限である以上、理論上は異なる入力が同じハッシュになりうる（誕生日パラドックス）。攻撃者は同一ハッシュを生む別データの生成を狙う |
| MD5（128-bit） | 衝突攻撃が既知。単純なファイル同一性確認程度なら使えるが、**機微データ・暗号用途では使わない** |
| SHA-1（160-bit） | 旧世代。新規採用は避ける |
| SHA-2 / SHA-3 | 224〜512 bit の出力長を選択可能。現行の推奨 |
| 用途 | 転送・ダウンロードの改ざん検証、バックアップ／リストアの一致確認（バイト比較より高速）、ログファイルの完全性保全、証拠保全（chain of custody） |

### 本番データの取扱い（非本番環境・機微フィールド保護）

非本番（テスト・ステージング・開発）システムは本番と同等に守られないことが多く、同じデータを置けば攻撃者は弱い方を突く。「本物に見えて本物でないデータ」を用意するのが原則。

| 技術 | 仕組み | 用途 / 注意点 |
|------|--------|--------------|
| データマスキング | 機微な値を構造を保ったまま偽値に置換（例: SSN → 999-99-9999）。計算対象の数値はランダム値の加減算等で加工 | 非本番環境用データ生成。ゼロから合成するより容易な場合が多い |
| データ匿名化 | 個人を特定できる情報を除去・置換する。**可逆な加工（文字順反転等）は匿名化ではない** | データセット全体の有用性を保ったまま個人の保護を実現する。分析用データ共有 |
| トークナイゼーション | 実データの代わりにランダムなトークンを保存し、実データは強く保護された**トークンボールト**にのみ保持。トークンはレコードロケータとして機能 | カード番号・SSN 等の最機微データ。機微情報がアプリ内を流通しなくなる |

### DLP（Data Loss Prevention）

DLP の本質は「データの移動（送信・ダウンロード・メール添付等）を検知し、理想的には阻止、最低でも把握する」こと。前提として**何が重要データかを定義する分類**が必要。

#### データ分類カテゴリ（代表例）

| カテゴリ | 内容 |
|---------|------|
| Public | 制限なく共有可能（マーケティング資料・公開文書） |
| Internal | 組織内利用を意図するが機微ではない（内部手順・非機微な運用データ）。流出は望ましくないが法的影響はない |
| Confidential | 特定の個人・グループに限定。開示は重大な損害・法的影響・競争劣位につながる（財務報告・従業員記録・内部戦略） |
| Restricted | 法規制・契約上の保護対象。不正開示は重い罰則・侵害に直結（PII・PHI・営業秘密・知的財産） |
| Highly Restricted | 最高度の保護。クリアランス・need-to-know を持つごく少数のみ（機密指定情報等） |

#### 分類運用の要点

- 分類定義の次に「該当データがどこに存在するか」の**データディスカバリ**が必要。SSN・カード番号のような定型データは検出しやすいが、半構造化・非構造化データは難しい。文書に proprietary / confidential 等のマーキングがあれば容易になる
- 分類でグルーピングすることで、個々のデータ単位ではなく**保存場所単位**でアクセス制御を適用できる
- ラベル付けと合わせて、所有者・保持要件・規制該当の有無をメタデータとして追跡。文書にはヘッダ / フッタで分類レベルを表示する

#### クラウドストレージ標準機能で DLP を支える統制

| 領域 | 機能 |
|------|------|
| アクセス制御・認証 | ロールベースのきめ細かなアクセス制御、位置ベース制限、ストレージアクセスへの MFA 要求 |
| ファイル / フォルダレベル | バージョニング（破損時のロールバック）、ロック（同時編集防止）、保持ポリシー（重要データの削除防止） |
| 監査・監視 | アクセス・変更・共有のアクティビティログ、大量ダウンロード / 削除等の異常アラート、ファイル完全性監視 |
| 暗号化 | 保存時・転送時の暗号化と、顧客側での鍵管理・ローテーション |

#### DLP と DFD の接続

アプリごと・業務プロセスごと（人手による転送を含む）に DFD を整備し、①発生源 ②移動方法（ポート / プロトコル）③転送時暗号化の有無 ④受信者（システム / 共有 / アプリ / 個人）を押さえる。形式変換（文書→PDF 等）で**コピーが増える箇所**と、その両方の行き先を文書化する。

## Information Rights Management（IRM）

IRM は「保護がデータ自体に付いて回る」仕組み。システム側のアクセス制御だけでは、PDF 化・印刷・スクリーンキャプチャといった迂回で情報が持ち出せる。IRM は内容を暗号化し、操作レベルの権限で持ち出しを制御する。

### IRM の能力チェックリスト

- [ ] 閲覧 / 編集に加え、**印刷・コピー&ペースト・別名保存 / エクスポート・スクリーンキャプチャ**を個別に許可 / 禁止できる
- [ ] ユーザー / グループ単位のアクセス制御（既存の ID 管理システムと統合できる）
- [ ] アクセスの**有効期限**設定（例: 24 時間限定、特定時間帯・曜日のみ）
- [ ] **配布後の権利失効**（ばら撒かれた後でも読めなくできる）
- [ ] **オフライン時も強制**される（ネットワーク切断でアクセス不能になる fail-closed 設計。切断で保護が外れる実装は無意味）
- [ ] 場所バインド（ネットワーク上 / 地理的位置 / 特定端末への限定）
- [ ] ロールベースの権限セット（author/owner・editor・reviewer〈コメントのみ〉・reader＋カスタムロール）

主要クラウドプロバイダのオフィススイート / ストレージにはある程度の IRM が組み込まれており、まず既存機能の活用を検討する。

## 可視性・監査

クラウドではコンテナ・サーバーレスにより「システムへのアクセス」という概念自体が薄れるため、**データの保存と移動**を軸にした可視性が要る。

| 目的 | 内容 |
|------|------|
| Auditing（監査） | アクセスと活動の詳細記録。不正アクセス検知・規制対応・運用健全性の確認。コンプライアンス / リスク / 法務 / セキュリティの各チームが利用者 |
| Traceability（追跡可能性） | データの誕生から現在までの全移動・全変更のタイムスタンプ付き証跡。インシデント調査と法的対応の基盤 |
| Accountability（説明責任） | 各操作を**誰が**行ったかの帰属。犯人断定ではなく「最初に事情を聞くべき人」を特定する出発点 |

### 収集すべき最低限のフィールド

1. **Who**: 操作主体のアイデンティティ
2. **Where**: 送信元 IP（geo-IP で物理位置の推定も可能）
3. **What**: 対象リソース
4. **When**: 正確なタイムスタンプ

Who / What / When を正確に取れて初めて Why / How の調査に進める。

### ログとフォレンジックの保護

- ログ自体へのアクセス・消去・削除もログに残す（メタロギング）
- クローズ済みログファイルへの**暗号学的ハッシュ付与**で改ざん検知
- 鍵の適切な保護（パスワード保護＋物理 / システムアクセス制御）により、鍵の所有者しか操作できない状態を作れば**否認防止**が成立する（「自分ではない」を言わせない）
- 法廷に持ち込む可能性があるなら **chain of custody**: ログの保管・ハッシュによる無改ざん証明・ログへのアクセス履歴の 3 点を揃える

## 落とし穴・アンチパターン早見表

| アンチパターン | 何が起きるか | 是正 |
|---------------|-------------|------|
| 「削除した」で終わる | 実体・旧バージョン・バックアップにデータ残存 | 破棄はバックアップ・版・プロバイダ側残存まで含めて設計。契約に破棄責務と監査権を明記 |
| データを無期限保持 | 攻撃対象価値・コスト・法的リスクの増大 | 分類ベースの保持ポリシーと期限後削除 |
| DFD を作って放置 | 追跡外の場所へ情報が漏出し、処分・保護から漏れる | アプリ変更に連動した更新プロセス |
| 「暗号化したから安全」 | 資格情報が破られればアプリ経由で平文が読める。鍵が漏れれば暗号化は無意味 | 暗号化＋IAM＋鍵管理を一体で設計 |
| 公開バケット＋「機微データはないはず」 | 誰かが機微文書を置くのは設定では防げない | アカウントレベルで公開化をブロック |
| 本番データを非本番へコピー | ソフトターゲット経由で本番相当の情報が漏れる | マスキング / 匿名化 / トークナイゼーション |
| MFA をプッシュ承認 / SMS だけにする | 承認疲労攻撃・SIM スワップで迂回される | フィッシング耐性のある方式を選定 |
| バージョニングだけでランサム対策済みとする | 攻撃者は版そのものを削除しに来る | イミュータブルバックアップの併用 |
| 「クラウドだからバックアップ不要」 | プロバイダ事故・攻撃で顧客データが失われた事例は実在する | データの責任は常に利用者側。独立したバックアップと RTO/RPO 定義 |
| ログを取らない / ログを守らない | 検知不能・証跡改ざん・監査不合格 | ログ設計を要件化し、ログ自体をハッシュとアクセス制御で保護 |
