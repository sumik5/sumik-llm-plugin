# ガバナンス・リスクマネジメント・コンプライアンス（GRC）

セキュリティプログラムはビジネス目標に従属する監督機能を必要とする。セキュリティ機能がビジネスと整合しないと、事業の阻害要因に転落する。本リファレンスは、リスクマネジメント（定量/定性分析・リスク対応の類型・NIST/ISO/FAIR フレームワーク）、コンプライアンス（規制・業界標準・認証）、ポリシー/標準/手順の3層ガバナンス文書、および BC/DR（BIA・RTO/RPO・計画とテスト）を、クラウド環境を含む実務判断基準として整理する。

---

## ガバナンスの全体像

| 構成要素 | 役割 | 問い |
|---------|------|------|
| リスクマネジメント | 脅威を「可能性×影響」で金額評価し優先順位付け | 何をどれだけ守るべきか |
| コンプライアンス | 規制・業界標準への適合を証明 | 求められる統制を実装・証明できているか |
| ポリシー/標準/手順 | 組織の期待値を階層的に文書化 | 従業員は何をすべきか |
| BC/DR | 障害時の事業継続と復旧 | 止まったらどう続け、どう戻すか |

原則: **セキュリティはビジネスに従う**。ビジネス側がビジョンを示し、セキュリティ側と同じ「リスクの言語」を使う。標準化されたリスクアプローチの採用は、事業部門と技術部門が別言語を話す事態を防ぐ。

## リスクマネジメント

### リスクの定義 — よくある誤用

**リスク = 発生可能性（Likelihood）× 影響（Impact）**。可能性は確率、影響は金額で表す。

| 誤用 | 正しい理解 |
|------|-----------|
| 「雷雨のリスクが30%」（可能性だけをリスクと呼ぶ） | 30%は可能性。リスクは 30% × 被害額（例: $50,000）= 期待損失 $15,000 |
| 「未パッチの Windows はリスク」 | それは脆弱性（影響への経路）であってリスクではない。リスク登録簿に載せる粒度を誤る典型例 |
| 「可能性が1割でもあればリスク」 | 影響が無視できるなら優先度は低い。逆に影響が甚大でも可能性が極小なら議論対象にならない |

### 定量的リスク分析の基本式

| 指標 | 式 | 意味 |
|------|-----|------|
| SLE (Single Loss Expectancy) | AV × EF | 1回のインシデントの期待損失 |
| AV (Asset Value) | — | 資産価値（金額） |
| EF (Exposure Factor) | — | 事象発生時の損失割合（%）。同じ資産でも事象が異なれば別計算（部分損傷と全焼は別事象） |
| ARO (Annualized Rate of Occurrence) | — | 年間発生回数の期待値 |
| ALE (Annualized Loss Expectancy) | SLE × ARO | 年間期待損失。**優先順位付けと対応選定の基準値** |

注意: 確率は準備のための道具であり、予言ではない。全事象を列挙できた保証も、想定どおり起きる保証もない。

### 定量 vs 定性

| 方式 | 特徴 | 使いどころ |
|------|------|-----------|
| 定量 | 金額・確率の実数値。比較・費用対効果判断が可能 | データがある領域。経営判断の根拠 |
| 定性 | High/Medium/Low、Tシャツサイズ（S/M/L/XL） | データ不足時の初期評価。**各ラベルに対応する金額レンジを必ず定義**し、平易な言語で経営層・取締役会と会話する |
| 半定量 | ラベル＋金額レンジの併用 | 実務の主流。「影響 $100,000」単体では意味が伝わらないため比較の文脈を添える |

### リスク登録簿（Risk Register）

特定したリスクは登録簿に文書化する。必須列: **脅威・可能性・影響・ALE・対応（Treatment）・根拠（Rationale）**。

| 脅威例 | 可能性 | 影響 | ALE | 対応 | 根拠 |
|--------|--------|------|-----|------|------|
| ストレージバケット設定ミスによるデータ漏えい | 中 (30%) | 高 ($250K) | $75K | 低減 | バケットポリシー・アクセス制御・自動構成スキャン |
| ランサムウェア | 中 (25%) | 高 ($300K) | $75K | 低減＋移転 | エンドポイント保護・バックアップ・IR計画＋サイバー保険 |
| フィッシングによる資格情報侵害 | 高 (60%) | 中 ($50K) | $30K | 低減 | MFA・意識向上訓練・メールフィルタ |
| レガシーシステムの脆弱性悪用 | 中 (35%) | 中 ($150K) | $52.5K | 受容 | 6か月後に廃止予定。暫定監視のみ |
| クラウドプロバイダ障害 | 低 (8%) | 高 ($200K) | $16K | 回避 | マルチクラウド＋自動フェイルオーバー |

ポイント: 1つのリスクに複数の対応を組み合わせてよい（ランサムウェア＝統制で低減しつつ保険で移転）。

### リスク対応の類型

中核の4類型:

| 類型 | 内容 | 例 |
|------|------|-----|
| 回避 (Avoid) | リスクを生む活動自体を行わない | 単一クラウド依存をやめる |
| 低減 (Mitigate) | 統制導入で可能性または影響を下げる | MFA・DLP・監視強化 |
| 移転 (Transfer) | 財務的影響を第三者に負わせる | サイバー保険・DDoS対策サービス |
| 受容 (Accept) | 情報に基づく判断として残余の影響を引き受ける | 廃止予定システムの暫定容認 |

拡張分類（ISO 系で使われる追加区分）:

| 類型 | 内容 |
|------|------|
| 除去 (Removal) | リスクの発生源そのものを取り除く |
| 機会費用 (Opportunity cost) | 活動の便益が潜在コストを上回るため受容する |
| 共有 (Share) | 影響の一部を他組織と分担する |

判断基準の要点:

- **回避 vs 除去**: パスワード認証を使う限り資格情報侵害は回避できない（MFA で低減は可能）。パスワードレス認証（証明書・生体認証）へ移行すれば発生源ごと**除去**できる。
- **共有 vs 移転**: クラウド移行は**共有**（プロバイダ過失分はプロバイダが負うが、データの責任は常に自組織に残る）。保険は**移転**（保険料と引き換えに補償を受ける）。
- **統制の費用対効果**: $1,000 の統制で $10 しかリスクが減らないなら導入しない。受容が合理的な場合がある。
- **残存リスク (Residual Risk)**: 統制導入後に残るリスク。経営層が納得できる水準まで下げるのがゴールであり、ゼロ化ではない。

### NIST RMF（SP 800-37）— 7ステップ

参照: https://csrc.nist.gov/projects/risk-management

| ステップ | 内容 | 要点 |
|---------|------|------|
| 1. Prepare | 役割・責任の特定と文書化、人員配置・訓練 | 多くの組織は創業時でなく成熟途上で導入するため準備が必須 |
| 2. Categorize | 情報資産とデータフローの棚卸し、脅威領域の特定 | **機微性**（保護対象情報か）と**重要度**（事業運営への必須度）を記録 |
| 3. Select | 資産の分類結果に基づき統制を選定 | 防御的統制だけでなく重要資産への**検知的統制**も選ぶ。統制コストとリスク削減額を比較 |
| 4. Implement | 選定した統制の展開 | — |
| 5. Assess | 統制が実際に機能しているか検証 | 統制は「バイパス・無効化されうる」「経年劣化する」前提で評価。機能しない統制は可能性を押し上げる |
| 6. Authorize | 経営層による正式承認 | 可能性と影響を完全に理解した**インフォームドな受容**。無理解の承認は組織を財務的破綻リスクに晒す |
| 7. Monitor | 資産・統制・評価の継続監視 | 一度きりではない。資産も脅威も変化し続ける |

### ISO リスクマネジメント（ISO 31000 / ISO 27005）

- ISO 31000 = 汎用リスクマネジメントフレームワーク。ISO 27005 = ISMS（ISO 27000 シリーズ）における情報セキュリティリスク管理ガイダンス。ISO 27001 は ISMS 実装に Plan-Do-Check-Act を推奨。
- 前提条件: **リーダーシップのコミットメント**（同意だけでなく実行権限を伴う）、組織全体への統合、継続的改善。

プロセスの流れ:

1. **コンテキスト確立** — 組織にとってのリスクの意味を定義。可能性・影響のスケール（ラベルと金額レンジ）を定義し、経営層と平易に会話できるようにする。
2. **リスク選好（Risk Appetite）の決定** — 存続を脅かさず、利害関係者の信頼を損なわない範囲で受容できる影響水準を定義。対応選定の基準になる。
3. **リスクアセスメント** — 特定・分析・評価の3サブプロセス。
4. **リスク対応** — 各リスクに最低1つの対応を割当て。
5. **モニタリング・レビュー** — リスクは増減する。継続的に見直す。

#### リスク特定の構造化質問（資産カタログを起点に）

| 質問 | 目的 |
|------|------|
| 誰が我々に害を与えたいか | 脅威の特定 |
| 何を達成しようとしているか | 動機 |
| どの資産を狙うか | 資産の特定 |
| どう攻撃するか | 攻撃ベクトル/脆弱性 |
| 事業への影響は何か | 帰結の評価 |

良いリスクステートメントの形: 「**脅威アクターXが、侵害した資格情報で製造システムに侵入して暗号化し、1日あたり$1Mの生産性損失を生じさせる**」— 脅威・経路・資産・金額影響が揃い、実行可能（actionable）。可能性の推定だけは自組織データが乏しいため外部データが必要になることが多い。

### FAIR（Factor Analysis of Information Risk）

参照: https://www.fairinstitute.org

- **タクソノミ（用語体系）**であり、組織内の全員が同じリスク用語を使うことを保証する。部門ごとに競合フレームワークが乱立して経営層への伝達が壊れる事態を防ぐ。
- 定量アプローチを推奨しつつ、確率推定の困難さも認める（データなき確率は当てずっぽうと同じ）。

分解構造:

```
リスク = Loss Event Frequency (LEF) × Loss Magnitude (LM)
LEF = Threat Event Frequency（資産が狙われる頻度）× Vulnerability（脅威行動が損失に至る確率）
LM  = Primary Loss（生産性低下・交換費用など直接損失）
    + Secondary Loss（罰金・顧客離反など間接損失）
```

プロセス: ①シナリオ定義（脅威×資産の組合せ）→ ②情報収集 → ③LEF算出 → ④LM算出（一次＋二次）→ ⑤リスク計算 → ⑥ステークホルダーへの提示と意思決定。

- 可能性の点推定を避けるため **Monte Carlo シミュレーション**（数千回の反復ランダムサンプリング）で値のレンジを得ることを推奨。単一の思いつき数値より良い結果を生む。
- 分解により巨大に感じられるリスク管理をサブタスク化でき、直接損失と間接損失の両方を明示的に扱える。

## コンプライアンス

### 基本原則と落とし穴

| 原則 | 内容 |
|------|------|
| 準拠 ≠ 防御 | PCI DSS 準拠企業でも大規模カード情報漏えいは多数発生している。認証はセキュリティの保証ではない |
| それでも無視不可 | 証明書類の未提出は重い罰金につながる。金融規制などは実効的な制裁を伴う |
| アーティファクト（証跡）が必要 | 統制の実装・有効性・「文書と実態の一致」を証明する成果物を規制主体の要求に応じて生成する |
| 法務と協働 | 法律・契約への影響が絡むため、コンプライアンス対応は法務の助言のもとで進める |
| 監査運営のコストを見積もる | 成熟したプログラムなら監視フェーズで証跡は日常的に生成されるが、監査対応自体の時間・人員は別途大きい |

### NIST vs ISO — 「認証できるか」の違い

| 項目 | NIST | ISO |
|------|------|-----|
| 性質 | ガイダンス（CSF はプログラム構造化の枠組み、SP 800-53 は20ファミリー・1,000超の統制カタログ） | 監査・認定につながる標準を発行 |
| 認証 | **不可**。「NIST CSF 認証」は存在しない | **可能**（ISO 27001 等） |
| 実務での使い方 | 自組織プログラムの設計指針。クラウドプロバイダへ「800-53 準拠か」を尋ねて統制水準を推し量る | 顧客・市場への適合証明 |

### ISO 27001 認証プロセス

1. **スコープ定義** — ISMS 全体でなく部分（特定拠点の特定システム等）に絞ってよい。文書化され明確なら部分スコープは正当。グローバル企業は全社スコープの文書量が過大になるため意図的に絞ることが多い。
2. **文書収集＋SOA** — ポリシー・標準・手順に加え **Statement of Applicability** を提出。ISO 27001:2022 Annex A の **93統制・4テーマ（組織・人・物理・技術）**（2013年版の114統制から再編）のうちどれが適用対象かを宣言。適用外の統制があって当然。
3. **文書審査** — 監査人がスコープ理解・ギャップ特定・リスク管理実務のレビューを行い、重大ギャップはここで洗い出す。
4. **現地審査** — スタッフインタビュー、証跡（ログ・議事録・記録）レビュー、統制テスト。例: 情報分類統制なら「分類ポリシーの存在」と「文書が実際にポリシーどおり分類されているか」の両方を検証。
5. **判定** — 重大不適合（major nonconformity）がなければ概ね認証。軽微な不適合は指摘として記録。**有効期間3年**、以後再認証。

限界の理解: 認証は「スコープ内で統制が概ね存在し、宣言どおりで、概ね機能する」ことしか意味しない。組織全体への適用も、統制の100%の信頼性も、防御の保証もしない。

### CSA Cloud Controls Matrix（CCM）と STAR

参照: https://cloudsecurityalliance.org

- Cloud Security Alliance が策定する**クラウド特化**の統制フレームワーク。汎用フレームワークが扱わないクラウド固有論点を補完。
- 統制ドメイン例: アプリ/インターフェイスセキュリティ、監査保証、事業継続・運用レジリエンス、変更/構成管理、データセキュリティとプライバシー、暗号鍵管理、ガバナンスとリスク、IAM、インフラ/仮想化、ロギング監視、インシデント管理、サプライチェーン、脅威/脆弱性管理 等。

クラウド固有の2大論点:

| 論点 | 要点 |
|------|------|
| マルチテナンシー | プロバイダはテナント間アクセスを絶対に許さない統制が必要。自社が SaaS 提供者なら、**同一企業所有の複数テナント間でも**明示許可なきデータ混在を防ぐ |
| 共有責任モデル | オンプレは物理〜データまで全部自己責任だが、クラウドは層ごとに責任が分割される |

**STAR プログラム**（段階的認証）: レベル1 = CAIQ（Consensus Assessment Initiative Questionnaire）による自己評価 → レベル2 = 第三者監査による認証または保証（attestation）。

### 金融系の規制

| 規制 | 対象 | 要点 |
|------|------|------|
| SOX | 全上場企業の IT 統制 | 株主・財務利害関係者の保護。重大インシデントは報告義務対象＝インシデント管理が直結 |
| FFIEC | 銀行 | 検査協議会の規制要件 |
| GLBA | 銀行を超えた金融機関全般 | 顧客金融情報の保護 |
| PCI DSS | カード決済を扱う全事業者 | 業界標準（法律ではない）だが契約上実質必須 |

### SOC 2（Type I / Type II）

AICPA が策定した監査基準。クラウドプロバイダ・SaaS・マネージドサービスプロバイダなど**サービス組織**が対象。ISMS に限定される ISO より広い統制範囲をカバーし、「認証を名乗れる」点で NIST 系と異なる。

| 項目 | Type I | Type II |
|------|--------|---------|
| 評価対象 | 統制の**設計**の適切さ（ある時点） | 統制の**運用有効性**（数か月の期間） |
| 証明内容 | 統制が文書・ポリシー上存在し、正しく実装されれば有効に機能しうる | 統制が実在し、**期間を通じて実際に機能した** |
| 証跡 | 設計文書 | ログ・監視レポート・インシデント記録・訓練記録 等 |
| 期間目安 | 1–3か月 | 運用実績6–12か月＋監査1–3か月 |
| 顧客への意味 | 「統制があると言っている」 | 「統制が時間経過に耐えて機能している」 |

大手クラウド（Azure/AWS/GCP）や SaaS プロバイダには **Type II** の取得が事実上の標準要求。

### クラウドプロバイダのコンプライアンス支援機能

| プロバイダ | 主なツール | 用途 |
|-----------|-----------|------|
| AWS | Config / Security Hub / Systems Manager Compliance / Audit Manager | 準拠状況の可視化。例: OS パッチポリシー準拠を判定するルール定義で脆弱性管理を運用化 |
| Azure | Azure Policy / Security Center / 規制テンプレート（NIST・SOC・PCI DSS 等） | 統制セットを定義し実装済みリソースへ適用。監査アーティファクト生成を大幅短縮 |
| GCP | セキュリティ/コンプライアンス機能群 | 同様の機能。上位ティア契約が前提の機能あり |

落とし穴: 高度なコンプライアンス機能は**上位サービスティア限定**のことがある。必要機能とライセンス階層を事前に突き合わせる。

## ポリシー・標準・手順の3層構造

ガバナンスはルールなしに成立しない。ミッションステートメント（セキュリティ機能の達成目標）を頂点に、下位に行くほど具体化する:

```
ポリシー（最も抽象・組織全体の期待値・技術非依存）
  └─ 標準（実装要件・特定技術への言及可・1ポリシー文に複数標準）
       └─ 手順（実装/監視/管理の具体的ステップ・1標準に複数手順）
```

### ポリシーの必須特性

| 特性 | 内容 |
|------|------|
| 権威 | 経営層の支持を得て指示としての重みを持つ |
| 具体性 | 曖昧でなく**測定可能**な指針。「ポリシーのためのポリシー」は組織を守らず、規制適合にも寄与しない |
| 網羅性 | 人・プロセス・技術をカバー |
| 生きた文書 | ポリシー自身が定めるスケジュールで定期レビュー。事業とともに進化しないポリシーは事業に即していない疑いあり |

### ポリシー文書の標準構成

1. ヘッダ/識別情報（名称・発効日・最終レビュー日）
2. 目的とスコープ（存在理由・達成目標・適用対象・対応するセキュリティゴール）
3. 定義（読者はセキュリティ用語に不慣れという前提で多めに）
4. ポリシーステートメント
5. コンプライアンスと執行（測定・監視方法、違反時の制裁＝懲戒〜解雇、報告経路とエスカレーション）
6. 文書管理（レビュー・更新履歴、承認者）

### 助動詞の使い分け（ポリシー/標準共通）

| 語 | 意味 |
|----|------|
| shall | 必須の要求（行動） |
| will | 組織が行う意図の表明 |
| must not / shall not | 禁止 |
| should | 任意の推奨（行動） |
| may | 任意の許容（意図） |

### 規制・フレームワークが標準的に要求するポリシー群

| ポリシー | 内容 |
|---------|------|
| 情報セキュリティポリシー | プログラムの確立・スコープ・権限 |
| 利用規程（AUP） | IT リソースの適正利用 |
| データ分類ポリシー | データカテゴリと取扱要件 |
| リスクマネジメントポリシー | リスクへのアプローチ定義 |
| IAM ポリシー | パスワード管理・アクセス制御・特権管理・MFA（複数に分割可） |
| ネットワークセキュリティポリシー | ファイアウォール・セグメンテーション・監視要件 |
| モバイルデバイス管理ポリシー | モバイル利用要件 |
| クラウドセキュリティポリシー | クラウド利用の統制・管理 |
| 訓練ポリシー | 意識向上訓練とロール別訓練 |
| リモートアクセスポリシー | 遠隔地からのアクセスに要求する統制 |

### 標準・手順の整合性ルール

- 標準はポリシーと同種の言語（shall/should 等）を使いつつ、**実装詳細・特定技術**に踏み込む。
- 手順は標準が要求する人・プロセス・技術の実装/監視/管理の具体策。
- **マッピング検査**: ポリシーステートメントに対応しない標準を見つけたら、標準を削除するかポリシー側に文を追加して整合させる。標準はポリシー準拠を担保できるだけの詳細を持つこと。

## 事業継続（BC）と災害復旧（DR）

### BC と DR の区別

「BCDR」と一括りにされがちだが、別機能である:

| 項目 | 事業継続（BC） | 災害復旧（DR） |
|------|---------------|---------------|
| ゴール | 障害の最中も**事業機能を動かし続ける** | 全機能を**障害前の能力に復元**（喪失データの復旧、障害中に発生したオフラインデータの取込を含む） |
| 位置づけ | 戦略的フレームワーク | **BC のサブセット** |
| オーナー | 各事業機能・アプリケーションの責任部門（プロセスの問題であり技術の問題ではないため、原則 IT 部門ではない） | IT 部門が主導 |
| 手段 | **非技術的な代替手段を含む**（手作業運用への切替等） | システム・データの技術的復旧 |

### 策定手順

1. **事業機能・プロセスの棚卸し** — 顧客等の重要ステークホルダーへの提供を支える機能をすべて特定。
2. **BIA（Business Impact Analysis）** — 各機能について以下を特定:
   - 目的 / 依存関係 / 影響 / 必要リソース / 重要度
3. **優先順位付け** — リソースは有限。重大障害時は「顧客支援と収益創出（回収を含む）に不可欠な機能」へ集中する。
4. **回復目標の設定**（機能ごと）:

| 指標 | 定義 | 関係 |
|------|------|------|
| MTO (Maximum Tolerable Outage) | 存続の危機に至るまでに許容できる最大停止時間 | 上限 |
| RTO (Recovery Time Objective) | 障害発生から機能復旧までの目標時間 | **RTO ≤ MTO** を厳守 |
| RPO (Recovery Point Objective) | 許容できるデータ損失量（時間で測る）。超えると存続を脅かす | バックアップ頻度・レプリケーション方式を規定 |

5. **計画文書化** — BIA の結果を基に BC 計画・DR 手順を作成。非技術的代替手段を計画に含める。

### テストと維持

計画は検証しなければ機能しない。負荷の低い順に段階的に実施する（業界標準の演習類型）:

| 演習 | 内容 | 頻度目安 |
|------|------|---------|
| 机上演習（tabletop） | 関係者がシナリオを議論し役割・手順を確認 | 年1回以上 |
| ウォークスルー | 計画書を実際の手順に沿って読み合わせ・検証 | 計画更新時 |
| シミュレーション | 模擬インシデントで対応を演習 | 年1回 |
| 並行テスト | 本番を止めずに復旧環境を実際に立ち上げ検証 | RTO/RPO 実測に有効 |
| 完全切替テスト | 本番を停止し復旧サイトへ切替 | リスクが高く要経営承認 |

- テスト結果と実インシデントの教訓を計画へフィードバックし、資産・組織変更のたびに BIA と回復目標を見直す。
- **クラウド活用**: クラウドは障害対策（地理的冗長・自動フェイルオーバー）と、必須アプリの新インスタンスを迅速に立ち上げる復旧手段の両方に使える。RTO/RPO 要件からレプリケーション/バックアップ構成を逆算する。

## チェックリスト（要点の再確認）

- [ ] リスクを「可能性×影響（金額）」で表現しているか。可能性だけ・問題リストをリスクと呼んでいないか
- [ ] リスク登録簿に脅威・可能性・影響・ALE・対応・根拠が揃っているか
- [ ] 各リスクに最低1つの対応（回避/低減/移転/受容＋除去/共有）が割当てられ、統制コスト < リスク削減額か
- [ ] 経営層は残存リスクをインフォームドに承認したか
- [ ] モニタリングが継続プロセスとして回っているか（一度きりの評価で止まっていないか）
- [ ] 「準拠＝安全」と誤解していないか。認証のスコープと限界を顧客・経営層に正しく説明できるか
- [ ] SOC 2 は Type I か Type II か、要求と提出物を区別しているか
- [ ] ポリシー→標準→手順のマッピングに孤児（対応先のない文書）がないか
- [ ] BIA・MTO/RTO/RPO が機能ごとに定義され、RTO ≤ MTO が成立しているか
- [ ] BC/DR 計画を演習でテストし、結果を計画に反映しているか
