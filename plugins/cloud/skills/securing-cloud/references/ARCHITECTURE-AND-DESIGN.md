# クラウドセキュリティのアーキテクチャと設計原則

クラウド環境におけるセキュリティアーキテクチャの基礎リファレンス。脅威の分類、暗号技術（対称/非対称・TLS）、IAM、ネットワークセキュリティ（ファイアウォール/NSG・VPN・多層防御）、データ保護、セキュリティ衛生、ゼロトラスト、BIA・BC/DR、設計フレームワーク（Well-Architected・Critical Security Controls・エンタープライズセキュリティアーキテクチャ）、DevOps セキュリティ、クラウドプロバイダのセキュリティ評価（PCI DSS・ISO 27017・FIPS 140・Common Criteria）を扱う。基本原則: **概念はオンプレミスと共通だが、実装するコントロールはクラウドプロバイダごとに異なる。データ保護の責任は最終的にプロバイダではなく利用者にある**（Shared Responsibility Model）。

---

## セキュリティの基本概念

### CIA トライアド

| 特性 | 目的 | クラウドでの観点 |
|------|------|----------------|
| 機密性 (Confidentiality) | 認可されていない者に情報を読ませない | 暗号化・アクセス制御・ストレージ公開設定 |
| 完全性 (Integrity) | 予期しない変更・破損を防ぐ | 署名・変更管理・改ざん検知 |
| 可用性 (Availability) | 必要なときに使える状態を維持 | マルチリージョン・自動復旧・DoS 対策 |

- セキュリティ事故の多くは悪意によるものではない（設定ミス・人的エラー）。「攻撃を防ぐ」だけが仕事ではない。
- 高可用性はクラウドの地理的ゾーンにより実現しやすいが、責任は利用者に残る。

### 脅威・脅威ベクター・脅威アクターの区別

| 用語 | 定義 | 例（ウイルス感染のアナロジー） |
|------|------|------------------------------|
| 脅威 (threat) | 組織や資源に悪影響を及ぼしうる事象・状況（NIST 定義ベース） | ウイルスそのもの |
| 脅威ベクター (threat vector) | 脅威が顕在化する経路 | 飛沫という伝播経路 |
| 脅威アクター (threat actor) | 脅威を顕在化させる主体（悪意の有無を問わない） | 感染を広める人 |

## クラウドの代表的な脅威分類

| 脅威 | 内容 | 主な対策 |
|------|------|---------|
| 設定ミス・人的エラー | 最も見落とされやすい脅威。typo・要件誤解・ベストプラクティス不知から生じ、CIA すべてを損ないうる。大規模障害の実例も更新配布ミス起因 | テスト・自動化・レビュー体制 |
| IAM の不備 | 弱いパスワード運用・使い回し（credential stuffing / password spraying を誘発）・MFA 未導入 | MFA 必須化・最新のパスワードガイダンス準拠 |
| 機微データの露出 | パブリック公開されたオブジェクトストレージ（例: S3 バケット）、保存時/転送時の暗号化欠如 | 公開設定の監査・デフォルト非公開・暗号化 |
| マルウェア | クラウドでもオンプレミス同様に不正アクセス・データ流出・破壊をもたらす | anti-malware・EDR |
| DoS | 帯域消費型だけでなく、リクエスト完了待ちでサーバリソースを枯渇させる低速攻撃も存在 | プロバイダの緩和サービス＋アプリ側対策 |
| 貧弱なアプリケーション設計 | API に十分な防御がなく、入力を信頼した結果インジェクション等の Web 攻撃を許す | 入力を信頼しない・API 保護 |
| 脆弱なコンポーネント | COTS/内製を問わずソフトウェアには脆弱性がある | 脆弱性スキャン・アプリケーションテスト |
| サプライチェーンの弱点 | OSS リポジトリに悪意あるコードが混入し、依存するソフトウェアが即座に危殆化 | 依存関係の検証・内部リポジトリ運用 |
| インフラ障害・過剰構成 | サブネット未保護・必要以上のリソース使用（コンテナで足りるのに VM 等）・需要を支えない設計 | 最小構成・セグメント化・キャパシティ設計 |

## 暗号技術

### 基本用語

| 用語 | 意味 |
|------|------|
| 平文 (plaintext) | そのまま読める情報 |
| 暗号文 (ciphertext) | 鍵なしには読めない変換後の情報 |
| 鍵 (key) | 平文⇔暗号文の変換に適用する情報 |
| 対称鍵暗号 | 暗号化と復号に同一の鍵を使う |
| 非対称鍵暗号 | 数学的に関連する公開鍵/秘密鍵のペアを使う |

### 対称 vs 非対称の比較

| 観点 | 対称鍵暗号 | 非対称鍵暗号 |
|------|-----------|-------------|
| 代表アルゴリズム | AES、ChaCha20、Blowfish、Twofish、RC4 | RSA、ECC、DSA |
| 典型的鍵長 | 128 / 256 ビット | 1024 / 2048 / 4096 ビット（RSA） |
| 速度・処理コスト | 高速・低コスト | 低速・高コスト |
| 鍵配送問題 | あり（事前に安全に共有が必要・定期的な再鍵化も必要） | なし（公開鍵は配布してよい） |
| 鍵の寿命 | 短命（使うほど破られるリスク増） | 長命（公開/秘密鍵ペアは長期利用前提） |
| 用途 | セッション暗号化・大量データ | 鍵交換・デジタル署名・証明書 |

**落とし穴**: 鍵長は同一方式内でしか比較できない。AES-128 が RSA-2048 より弱いわけではない（数学的アルゴリズムが異なるため鍵長の直接比較は無意味）。128 ビット鍵の鍵空間は 2^128（約 3.4×10^38）であり、総当たりは非現実的——攻撃は鍵推測以外の経路を狙う。

### 非対称暗号の運用ポイント

- **公開鍵は共有してよい・秘密鍵は厳重保護**。秘密鍵は必ずパスフレーズで保護する（漏えいしても即座に悪用されない）。
- **信頼の確立**: 鍵を自己生成し手渡しする方式はスケールしない。信頼できる第三者＝認証局 (CA) を用いる PKI が標準。CA が身元を検証し証明書（暗号鍵＋エンティティ識別情報を含むデジタル文書）を発行する。互いを直接信頼しなくても、双方が CA を信頼することで信頼が推移する。
- **デジタル署名と否認防止 (nonrepudiation)**: 秘密鍵で署名し公開鍵で検証。本人しかアクセスできない鍵に行為が紐づくため「やっていない」と主張できない状態を作れる。

### TLS（Transport Layer Security）

- 転送中データの標準的な暗号化手段。Web だけでなくサイト間 VPN にも使われる（IPsec より設定・運用が容易）。
- 前身の SSL は 3 バージョンすべてに深刻な脆弱性があり公式に非推奨化済み。**新規設計で SSL を使わない**。

**ハイブリッド方式**: TLS は非対称暗号でセッションを確立し、その後は高速な対称鍵（セッション鍵）で通信する。

ハンドシェイクの要点:

1. `ClientHello` — クライアントが対応可能な暗号スイートを提示
2. `ServerHello` — サーバが双方対応のスイートを選択
3. サーバ証明書送付 — クライアントが有効性と FQDN 一致を検証
4. 鍵交換 — セッション鍵を確立し、以降は対称暗号で通信

**Diffie-Hellman 鍵交換**: 鍵そのものは一切ネットワークを流れない。双方が公開値（素数と基数）と各自の秘密値から計算を行い、計算結果のみ交換して同一の共有秘密を導出する。長いセッションでは同じ手順で再鍵化（re-keying）できる。

## Identity and Access Management (IAM)

### アイデンティティライフサイクル

アイデンティティ＝エンティティ（人・アプリケーション・デバイス・サービス）の一意なデジタル表現。作成→管理→削除の完全なライフサイクルと、それに紐づく権限を一元管理する。

| フェーズ | 要点 |
|---------|------|
| 作成・検証 | 人は公的身分証明で本人確認＋組織との所属関係確認。サービス/デバイスアカウントは既知のオーナーが必要性と用途を承認 |
| 保管 | 全利用箇所で一貫する単一レコードに保存（ディレクトリサービス）。クラウドではプロバイダの ID 基盤か、オンプレミス ID プロバイダとのフェデレーション |
| 認証 | 「知識（パスワード）・所持（トークン/鍵）・生体（指紋等）」の要素を組み合わせる |
| 認可 | どのリソースへのアクセスを許すかの決定 |
| 削除 | 退職・用途終了時の確実な無効化 |

### アクセス制御モデルの選択

| モデル | 仕組み | 向いている状況 | 注意点 |
|--------|--------|--------------|--------|
| RBAC | ロール（グループ）に権限を割当て、個人をグループに所属させる | 職務が明確な組織の標準解 | 個別割当（ad hoc）より管理性が高い |
| ABAC | 属性（クリアランス・部署・役職等）とポリシーで判定 | 機密レベル管理が重要な組織。RBAC より粒度が細かい | ポリシー設計の複雑さ |
| DAC | リソース所有者が任意に権限付与 | 柔軟だが組織統制が効かない | 機微情報が不適切に共有されるリスク |
| MAC | 組織/システムレベルのポリシーでのみアクセス決定（例: SELinux） | 高セキュリティ環境 | 運用の硬さ |

### 特権アクセス管理（PAM / PIM）

| 実践 | 内容 |
|------|------|
| 最小権限 | すべての ID は業務に必要なリソースのみアクセス可能にする（IAM 全体の大原則） |
| アカウント分離 | 管理作業専用 ID と日常業務（メール・ブラウジング）用 ID を分ける |
| パスワードボールト | 特権アカウントのパスワードを保管し、必要時にチェックアウト |
| セッション管理 | 特権アクセスをすべて記録・ログ化 |
| Just-in-Time アクセス | 必要なときに限定時間だけ特権を付与 |
| PIM | 特権 ID のライフサイクル管理（プロビジョニング〜廃棄）。自動化で一貫性を担保し、定期的に権限の要否をレビュー |

### パスワード運用の現行ガイダンス

「長く複雑なパスワードの定期強制ローテーション」は、書き留め・使い回し等の回避行動を誘発し、かえってリスクを高める——現行の標準ガイダンス（NIST 系）はこの旧慣行を否定する方向に更新済み。MFA を基本とし、self-service のパスワードリセット基盤を整備する。

### 行動分析（Behavior Analytics）

- ユーザーが「何を・どこから・いつ」行うかを把握し、異常をフラグする。
- 例: 普段と異なる地理的位置からのアクセス→追加認証要求。
- **impossible travel**（同時に離れた 2 地点からログイン）はアカウント侵害のシグナルだが、自宅に残した端末の自動再接続という偽陽性もある。

## ネットワークセキュリティ

### ファイアウォールの進化と NSG

| 世代 | 判定基準 | 限界 |
|------|---------|------|
| パケットフィルタ（ACL） | IP アドレス・ポート番号 | アドレス偽装（spoofing）に弱い |
| ステートフルファイアウォール | 通信の方向と状態（TCP 3-way handshake の追跡。UDP はフローの期待値で判定） | 上位プロトコルの中身は見ない |
| モダンファイアウォール | 状態＋上位プロトコルの妥当性＋IDS/IPS/anti-malware 統合 | 復号が必要な暗号化トラフィック |

- **IDS**: 既知パターンに基づき異常トラフィックを検知してアラート。**IPS**: 検知に加えブロックまで行う。
- **ネットワークセキュリティグループ (NSG)**: クラウドにおけるファイアウォール相当。NIC またはサブネットレベルに適用するルール集合。IP・ポート・プロトコルで ingress/egress を制御し、通常ステートフル。**ルールに優先度を付けられる**ため、複数ルールが同一パケットに該当しうる曖昧さを解消できる。

### VPN

| 方式 | 特徴 |
|------|------|
| IPsec | デバイス間暗号化。IPv6 由来で IPv4 にも適用可。設定・管理が比較的難しい |
| TLS ベース | 汎用プロトコルで暗号化。セットアップが容易 |
| PPTP / L2TP / GRE | **カプセル化のみで暗号化なし**。「プライバシーのない仮想ネットワーク」であり、単体では機密性を提供しない |

設計上の要点:

- VPN はトンネルによりリモートユーザーがローカルネットワーク上にいるように見せる（暗号化＋カプセル化）。
- 戻りの通信は必ず VPN 装置経由にする。直接ユーザーへ返すと暗号化・カプセル化なしで公衆網を流れる。
- クラウドでは VPC 内の特定サブネットに VPN コンセントレータを置き、管理系など非公開リソースへの入口とする構成が典型。

### 多層防御（Defense in Depth）と関連戦略

| 戦略 | 考え方 |
|------|--------|
| Defense in Depth (DiD) | 攻撃者が標的に到達するまでに複数の層を突破させる「垂直」の防御。ネットワークセグメント化・ファイアウォール・IDS/IPS・ホストベースファイアウォールを重ねる |
| Defense in Breadth (DiB) | 企業全体をカバーする「水平」の防御。脅威インテリジェンス（誰がどのベクターで狙うか）・全資産の脆弱性管理・EDR 等 |
| Defensible Network Architecture | **侵入されることを前提に**、隔離・切断できるセグメント設計と、影響範囲の監視を強化できる十分なロギングを備える |

- 攻撃者は外部サーバだけでなくユーザーエンドポイントを狙う。境界防御に加えエンドポイント保護（ホストファイアウォール・anti-malware・EDR）が必須。
- メール・Web ブラウジングは通す必要がある経路＝ソーシャルエンジニアリングの主要ベクター。

### トラフィック検査

- フルパケットキャプチャはフォレンジック情報として有用。ネットワーク入口のファイアウォール・クラウド内のソフトウェアファイアウォール・エンドポイント・API ゲートウェイなど複数の収集ポイントがある。
- **メタデータだけで十分な場合も多い**（フルキャプチャが常に必要なわけではない）。
- 暗号化通信（TLS）が主流のため、検査には復号を担うファイアウォール/TLS ゲートウェイが必要になる点を設計時に織り込む。

## データ保護

守るべき本質はシステムではなく**データ**（知的財産・PII・カード情報・PHI・財務/法務データ・認証情報）。システム保護はデータへの到達経路を守るための手段。

### データライフサイクル（6 フェーズ）

1. 生成/取り込み
2. 保存
3. 処理/利用
4. 共有
5. アーカイブ
6. 削除/破棄

**全フェーズで保護が必要**。特に削除/破棄は「消したつもり」の検証まで行う。削除が検証されないまま放置されたストレージバケットが後年発見され、管理外のまま PII が悪用される——ライフサイクル終端の管理漏れが典型的な事故パターン。

## セキュリティ衛生プラクティス

| プラクティス | 要点 |
|-------------|------|
| IAM 保護 | アカウント保護と MFA 徹底。アカウント乗っ取りは常套手段 |
| 脆弱性管理 | 特定（スキャナ＝シグネチャ照合・バージョン確認であり exploit はしない）→優先度付け→修正→検証のサイクル。ベンダーの脆弱性通知の追跡には正確な資産管理（バージョン含む）が前提 |
| パッチ適用 | 脆弱性管理で最重要の実行段階。自動更新は再起動タイミング・内製アプリとの互換性の観点で常に最善ではない。**更新は配布前にテストする**（未テスト更新の一斉配布が世界規模の障害を起こした実例あり）。配布基盤（Intune 等のパッチサーバ、Puppet/Chef 等の自動化プラットフォーム）を整備 |
| ベースライン/ゴールドイメージ | 標準イメージまたは自組織テンプレートで OS 展開の一貫性を確保し、定期更新する |
| anti-malware / EDR | クラウドだからマルウェア免疫があるとは考えない。コンピューティング基盤を運用するなら導入する |

## ゼロトラスト

### 前提の変化

- クラウド環境には伝統的な「境界」がない。サービス（メール・チャット・ファイル共有等）は分散し、どこからでもアクセスされる。
- 旧来の誤り: 「ネットワークに物理接続している＝信頼できる」という暗黙の前提（接続後の認証が緩い）。ゼロトラストはこの前提を捨てる。

### 主要な構成要素

| 要素 | 内容 |
|------|------|
| マイクロセグメンテーション | ネットワーク/ホストファイアウォールでデバイス同士を相互に隔離し、接続先ネットワークへの全面露出を防ぐ |
| 強い認証と単一 ID ソース | ほぼすべてが公衆網経由の接続になるため、強い認証が正当な利用者を保証する唯一の手段。認証・認可を中央管理する |
| デバイス認証 | ユーザーだけでなくデバイスも認証（組織が配布する暗号証明書で正当性を検証） |
| 条件付きアクセスポリシー | 中央の認証ソースで「どこから・いつ認証したか」等に基づき、キャッシュ認証の許可/再認証/追加要素要求を細かく制御 |
| 侵害前提 (assume breach) | 認証済みマシンからのリクエスト＝本人とは限らない（端末が乗っ取られユーザーコンテキストで操作されうる）。アプリケーション側でも passkey・トークン・OTP 等の追加要素で認証する |

## 設計原則

良い設計は技術要素の組み立てではなく、**顧客要件とビジネス要件から始まる**。ビジネスが費用を負担し、ビジネスの継続が最優先目標である。

## ビジネス影響分析（BIA）

**目的**: 重要なビジネスプロセス・機能・サービスの中断が組織にもたらす影響を特定する。

### 手順

1. **カタログ化**: 重要/クリティカルなプロセス・機能・サービス（アプリ・システム含む）を洗い出す
2. **影響評価**: 各項目について「財務損失・レピュテーション毀損・法的/規制上の帰結・業務非効率」を評価し、金額換算して優先度付けの材料にする
3. **RTO の特定**: Recovery Time Objective＝許容できる最大ダウンタイム。短いほどクリティカル
4. **RPO の特定**: Recovery Point Objective＝時間換算で許容できるデータ損失量
5. **優先順位付け**: RTO・RPO・影響から機能ごとの優先度をランク付けし、保護・復旧に充てるリソースを決める
6. **リソース特定**: 各領域を支える人・プロセス・技術を、現状と復旧時に必要なものの両面で文書化
7. **文書化と定期レビュー**: BIA は一度きりの作業ではない。組織は変化し続けるため定期的に更新する

## 事業継続（BC）と災害復旧（DR）

| | 事業継続 (BC) | 災害復旧 (DR) |
|---|--------------|---------------|
| 目的 | 災害・事故・緊急事態の**最中**にクリティカル機能の運用を維持する | 暫定運用から**通常運用へ復帰**する |
| 起点 | BIA で影響と対象を把握 | BC 計画の暫定手段を前提に復帰手順を設計 |
| 中身 | シナリオ策定（自然災害・ランサムウェア等のサイバーインシデント・大量の人員喪失）、代替プロセス（オフライン処理・代替処理施設）、人・プロセス・技術 | バックアップと代替処理能力の確保、オフラインデータの復元・マージ、データ損失箇所の把握、代替施設からの切り戻し手順 |

### BC/DR チェックリスト

- [ ] BC 計画の対象は BIA に基づき「絶対に必要なプロセス」へ集中する（全機能の維持は大組織では非現実的）
- [ ] 顧客対応・売上直結の機能だけでなく、**給与支払・福利厚生・雇用/退職処理など人に関わるプロセス**も対象にする
- [ ] 短期だけでなく長期の中断を想定する（自然災害・パンデミック・サイバーインシデントは数か月以上続きうる）
- [ ] 計画は定期的に**訓練**する。訓練で判明した不備は計画に反映する
- [ ] 訓練は現実の制約を模す——システム停止シナリオなら**計画書はハードコピーでも参照可能に**しておく
- [ ] バックアップとリストアは必ずテストし、**復元所要時間を計測**する（意思決定と復旧期待値の設定に使う）
- [ ] 代替施設・別プロバイダへのフェイルオーバーを含む場合、**切り戻し手順**を明文化する
- [ ] 完全な原状回復に固執しない。災害後は環境自体が変わり、新常態への適応が正解になることがある

## 機能要件の扱い

| 原則 | 内容 |
|------|------|
| What に集中 | 解決策（How）を仕様に書かない。解決策を先に固定すると目標達成の検証が困難になる |
| 検証可能性 | すべての要件はテストで達成可否を確認できる形で書く |
| 全ステークホルダー | エンドユーザーだけでなく、運用・管理者も「ユーザー」。管理要件も収集する |
| 早期フィードバック | インタビュー・ワークショップで反復的に収集。長期間後の一括納品で認識齟齬が発覚する事態を避ける |
| 競合の解決 | 要件は重複・競合する。要件とステークホルダー双方の優先順位付けで解決する |

**優先順位付けの標準手法**:

| 手法 | 内容 |
|------|------|
| MoSCoW | Must / Should / Could / Won't Have の 4 区分 |
| Kano モデル | ユーザー満足度と必要性で分類 |
| 加重スコアリング | 基準を定義し重み付けし、要件ごとに採点 |

## 設計パターン・フレームワーク

複数フレームワークは排他ではない。主軸を 1 つ選びつつ、他から知見を取り込んでよい。

### Well-Architected 型フレームワーク（6 本の柱）

クラウドベンダーが公開する構造化された設計指針。代表的な 6 本の柱:

| 柱 | 要点 |
|----|------|
| セキュリティ | 強い ID 基盤から始める。最小権限。**全レイヤーに**セキュリティコントロールを適用（ネットワークだけではない）。インシデント対応・復旧の備え |
| 信頼性 | 障害から**自動で**回復する（手動介入はダウンタイムを生む）。水平スケールで需要に耐える。復旧手順をテスト。変更は自動化し、デプロイ前にテスト |
| パフォーマンス | リソースの効率利用。実測メトリクスに基づく事実ベースの改善。メトリクスは「その値で何を変えるか」を意識して設計 |
| 運用の卓越性 | 変更の自動化・IaC によるデプロイ自動化。**小さく・頻繁で・巻き戻せる変更**を優先。障害は想定し、学習イベントとして手順改善につなげる |
| コスト最適化 | ビジネス目標を満たす最低価格帯の構成。serverless・マネージドサービスの活用。利用状況の定期分析 |
| サステナビリティ | 環境影響を計測・予測し、リソース使用率の最大化でインフラ量を最小化。マネージドサービス活用 |

### Critical Security Controls（CIS 系・18 コントロール）

セキュリティ組織が維持してきたコントロール群は現在 CIS（Center for Internet Security）が管理し、v8 系で 18 項目に統合されている（20→18 は重要性の低下ではなく統合。例: 管理特権の制御＋need-to-know ベースのアクセス制御→アクセス制御管理）。

| # | コントロール | クラウドでの要点 |
|---|-------------|----------------|
| 1 | 企業資産のインベントリと管理 | 他の多くのコントロールの土台。リソースが需要に応じ増減するクラウドでは難度が上がるが、所有者・管理方法まで把握する |
| 2 | ソフトウェア資産のインベントリと管理 | ライセンスコストの把握に加え、脆弱なソフトウェアへの露出範囲を知る手段 |
| 3 | データ保護 | アクセス制御＋転送時/保存時の暗号化＋データ損失防止 (DLP) |
| 4 | 資産・ソフトウェアのセキュア設定 | 設定ミスは悪用・情報窃取に直結。ハードニングと継続的な設定管理 |
| 5 | アカウント管理 | プロバイダの IAM 実装を適切に利用 |
| 6 | アクセス制御管理 | 最小権限・RBAC・特権アカウント管理 |
| 7 | 継続的脆弱性管理 | プロバイダ責任範囲でも定期スキャンで自組織の露出を把握。修正・検証まで管理 |
| 8 | 監査ログ管理 | クラウドリソースのログを確実に収集・分析（クラウドのログ基盤利用可） |
| 9 | メール・ブラウザ保護 | 脅威防止・添付ファイルスキャン・サンドボックス検証（隔離環境で添付ファイルを開封・実行して検査する detonation）。VDI 専用でない仮想環境をメール/ブラウジングに使わない |
| 10 | マルウェア対策 | サーバ含めクラウドにも anti-malware を配備 |
| 11 | データ復旧 | バックアップ＋完全な DR 計画 |
| 12 | ネットワークインフラ管理 | 仮想ネットワーク・セキュリティグループの適切な管理 |
| 13 | ネットワーク監視と防御 | クラウドネイティブのログ/監視＋侵入検知 |
| 14 | セキュリティ意識向上とスキル訓練 | 一般的な訓練に加え、クラウド固有の脅威への認識を明示的に含める |
| 15 | サービスプロバイダ管理 | 契約・エスカレーション・サポートを含むベンダー管理 |
| 16 | アプリケーションソフトウェアセキュリティ | SDLC 全フェーズにセキュリティを組み込む。CI/CD への統合を含む |
| 17 | インシデント対応計画 | クラウド上のインシデントを包含。クラウド運用チーム外からのアクセス経路と、対応チームへのログ提供を計画 |
| 18 | ペネトレーションテスト | 脆弱性スキャンで見つけにくい問題を補完するため定期実施 |

### エンタープライズセキュリティアーキテクチャ（CSA 型）

コントロールの「リスト」ではなく、**レイヤー構造のアーキテクチャ**としてクラウドセキュリティを整理するアプローチ。各レイヤー（Business / User / Application / Data / Infrastructure / Security / Management & Governance）が異なるステークホルダーとニーズを表す。

セキュリティ観点のドメイン例: ガバナンスとリスク管理、法務・契約・電子証拠開示、コンプライアンスと監査、情報ガバナンス、管理プレーンと事業継続、インフラセキュリティ、仮想化とコンテナ、インシデント対応、アプリケーションセキュリティ、データセキュリティと暗号化、ID/エンタイトルメント/アクセス管理、Security as a Service、関連技術。

**導入手順**: ①現状のエンタープライズアーキテクチャを評価 → ②セキュリティ/コンプライアンスのギャップを特定 → ③クラウド移行ロードマップを策定 → ④全レイヤーにセキュリティコントロールを実装 → ⑤継続的な監視と改善（プロセスと統制の有効性を測定し非効率を発見する——最も難度が高い段階）。Cloud Control Matrix 等の参照資料で全レイヤーのコントロール網羅を確認でき、NIST・ISO 27001 等の他フレームワークと統合しやすい設計になっている。

## DevOps セキュリティ

### DevOps の基本要素とセキュリティへの寄与

| 要素 | 内容 | セキュリティへの寄与 |
|------|------|--------------------|
| コラボレーション | チーム間の壁を壊し、コミュニケーションを増やす | セキュリティ担当が要件を初期から表明できる |
| 自動化 | 開発・デリバリーの高速化 | 人的エラーの削減 |
| 継続的インテグレーション | 中央リポジトリへ定期的にマージ | 修正が滞留しない |
| 継続的デプロイ | ステージング/本番への自動投入 | 脆弱なコードが本番に長期滞留しない |
| Infrastructure as Code | コードで一貫・再現可能なデプロイ | 構成のテスト・管理が可能に |
| 監視とロギング | 本番のテレメトリ収集 | 検知・対応の基盤 |
| マイクロサービス | 小さな分離コンポーネント | **攻撃面の縮小**——外部公開を必要最小の部品に限定し、他はネットワーク層の背後に保護 |

### DevSecOps: パイプラインへのセキュリティ組み込み

| 組み込み箇所 | 手段 |
|-------------|------|
| コードレビュー | 通常レビュー（バグ・競合・スタイル）に加え、セキュリティ担当が脆弱性観点でレビュー |
| ビルドパイプライン | 静的解析 (SAST) を組み込み、深刻な問題はビルド前に開発者へ差し戻し。軽微な問題は後続変更でフラグ管理 |
| 自動テスト | 動的セキュリティテスト (DAST) を自動実行。手動テストほど深くないが、**毎回のパイプライン通過時に基本的な脆弱性を検出できる**（デプロイ後にしかセキュリティテストしない慣行への対策） |
| コード・環境の保護 | アプリコードだけでなく環境構築コード (IaC) も強いアクセス制御で保護 |
| コンテナイメージ | 公開リポジトリからの汚染を継承するリスクに備え、**自組織のイメージレジストリを運用**し改ざんのないベースを保証 |

原則: セキュリティチームを開発チームの正式メンバーとして迎え入れる（「妨害者」ではなく貢献者として機能させる）。

## クラウドプロバイダのセキュリティ評価

法規制対象の資産を預ける場合は特に、プロバイダの実務をフレームワークで評価する。

### PCI DSS（ペイメントカード業界）

カード情報を扱う場合の必須基準。6 グループ → 12 要件 → 220 のサブ要件に展開される。

| グループ | 12 要件の例 |
|---------|------------|
| セキュアなネットワークとシステムの構築・維持 | ファイアウォールの設置・維持／ベンダー既定パスワードの排除 |
| カード会員データの保護 | 保存データの保護／公衆網での伝送暗号化 |
| 脆弱性管理プログラムの維持 | マルウェア対策と更新／セキュアなシステム・アプリの開発維持 |
| 強力なアクセス制御 | need-to-know によるアクセス制限／アクセスの識別と認証／物理アクセス制限 |
| 定期的な監視とテスト | ネットワークリソースとカード会員データへのアクセス追跡・監視／セキュリティシステムの定期テスト |
| 情報セキュリティポリシーの維持 | 全人員を対象とするポリシー |

- 毎年のコンプライアンス認証が必要。取引量に応じ自己評価〜外部評価者 (QSA) による評価まで変わる。
- **クラウド上でカードを扱うなら、その基盤はオンプレミス同様に評価対象**になる。プロバイダの能力が PCI DSS の期待に応えられるかを事前に確認する。

### ISO 27017 / 27002

- ISO 27002＝情報セキュリティマネジメントシステムの実装ガイドライン（組織・人・技術・物理コントロール）。ISO 27017 はその上に**クラウドプロバイダ固有のコントロール**を追加する。

クラウドプロバイダを評価すべき領域:

| 領域 | 確認内容 |
|------|---------|
| 役割と責任の共有 | Shared Responsibility の分界線が明確か |
| 顧客資産の返却/削除 | 解約時に資産が返却または破棄されるか。**バックアップ含め**データが残らないか |
| 仮想環境の分離 | 他顧客からの完全な分離と、その実現方法の開示 |
| 仮想マシンのハードニング | プロバイダ管理インフラへのハードニング基準適用と（少なくとも方法論レベルの）文書化 |
| 管理者の運用セキュリティ | 管理者アクセスの文書化と統制 |
| クラウドサービスの監視 | サービス監視と、顧客に関わる情報の必要に応じた伝達 |
| データ取扱の整合 | 削除・保持ポリシーが顧客側と同期しているか（プロバイダ側の長期保持が顧客の法令違反を招かないか） |

### その他の基準

| 基準 | 対象 | 判断ポイント |
|------|------|-------------|
| FIPS 140 | 非機密情報の暗号化要件 | 特定種別のデータを扱う場合、自組織とプロバイダ双方に準拠義務が及ぶ |
| Common Criteria | 情報セキュリティ製品・技術の厳格な評価の国際標準（Protection Profile・Security Target・EAL 等の階層） | 標準化と厳密さを提供するが、時間とコストが大きい。プロバイダ評価目的にはオーバースペックになりうる |

## まとめ: 設計時の判断基準

| 状況 | 指針 |
|------|------|
| クラウド設計を始める | 脅威の把握 → BIA → BC/DR → 要件収集（全ステークホルダー）→ フレームワーク適用の順で、ビジネス起点で進める |
| コントロール選定 | オンプレミスの概念は通用するが実装は別物。プロバイダ固有の実装（NSG・IAM・ログ基盤）に読み替える |
| IaaS/コンテナを使う | ネットワークセキュリティ・パッチ・anti-malware は自分の責任 |
| SaaS 中心 | ネットワーク層の責任は軽くなるが、**データ保護と IAM は常に自分の責任** |
| 保存時暗号化 | 推奨だが万能ではない——データへアクセスできるアカウントが侵害されれば暗号化は守ってくれない。だからこそ IAM が最重要 |
| プロバイダ選定 | PCI DSS（該当時）・ISO 27017/27002・FIPS 140・Common Criteria 等で評価。規制・法域差で「どのプロバイダも同じ」ではない |
| 障害・侵害への備え | プロバイダは無謬ではない。BC/DR は利用者の責任として維持・訓練し続ける |
