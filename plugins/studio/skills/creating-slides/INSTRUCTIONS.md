# HTMLスライド作成ガイド

HTMLスライドの作成・カスタマイズ・ソース素材変換を体系的にガイドするスキル。

## 実行委譲

**スライド作成の実行は `tachikoma-doc-slide` エージェントに委譲する。**

本スキルがロードされた場合、Claude Code本体（オーケストレーター）は以下のフローで動作する:

1. **本体の役割**: ユーザーとの対話・要件整理・承認確認
2. **タチコマの役割**: 実際のHTMLスライド作成・テーマカスタマイズ・プロジェクト初期化

### 委譲フロー

```
ユーザー指示 → 本体（要件整理）→ Agent(subagent_type: "sumik:tachikoma-doc-slide") → スライド完成
```

### 本体が行うこと
- ソース素材の受け取りと確認
- スライドの方向性・トピック・聴衆のヒアリング
- タチコマからの質問の中継（AskUserQuestion）
- 完成物の最終確認とユーザーへの報告

### 本体が行わないこと
- HTMLファイルの直接作成・編集
- テーマCSSの変更
- プロジェクト初期化のファイルコピー

> **注意**: tachikoma-doc-slideは `creating-slides` スキルと `writing-effective-prose` スキルをプリロード済みのため、本体がスキル内容を転送する必要はない。

## Stage 0: 制作要件・スタイル決定（最優先・AskUserQuestion）

**運用原則**: スライド生成前に、以下の選択肢を AskUserQuestion で確認する。**ただしユーザーの指示や提供素材から答えが明らかな項目は質問せず確定値として進める**（冗長な質問をしない）。各質問は推奨オプションを先頭に置き末尾に「(推奨)」、相互排他でない項目は multiSelect、回答は具体的なデッキ変更にマッピングする。

このStage 0は**コンテンツ面より先**に行う「ビジュアル/技術面の決定」フェーズである。下記の決定ポイントを確認してから、後続の「プレゼン品質ヒアリングフロー（Stage 1〜5）」でコンテンツ面を詰める。

### 決定ポイント表

| # | 項目 | 質問文案 | 選択肢 | 回答 → 反映先 |
|---|------|---------|--------|--------------|
| 1 | **テーマ** | スライドの配色テーマはどうしますか？ | ・ライト白背景・青アクセント (推奨)<br>・ダーク背景<br>・ブランド配色に合わせる | ライト → デフォルト（`theme/sample.css` のまま）。ダーク → `theme/dark.css` を作成し各デッキの `<link>` を差し替え（詳細: `references/CSS-OVERFLOW-AND-DESIGN.md` 「ダークテーマを作る場合」）。ブランド → `:root` 変数を上書き |
| 2 | **アクセントカラー** | アクセント色はデフォルトの `#0033FF` のままでよいですか？ | ・デフォルト `#0033FF` のまま (推奨)<br>・ブランドHEXを指定する | デフォルト → 変更なし。指定 → `theme/sample.css` の `--color-accent` を指定HEXに設定 |
| 3 | **フォント** | 本文フォントはどうしますか？ | ・システム標準（埋め込みなし・推奨）<br>・ブランドフォントを使う | システム標準 → 変更なし。ブランド → `.woff2` を `shared/fonts/` へ同梱し `@font-face` 定義 + `--font-sans` を差し替え（CDN直リンク禁止・詳細: `references/CSS-OVERFLOW-AND-DESIGN.md` 「カスタムフォントの使い方」） |
| 4 | **用途・トーン** | このプレゼンの用途は？（1つ選択） | ・学術発表<br>・社内研修<br>・提案・営業<br>・ピッチ<br>・講義・動画配信 | 構成・情報密度・参照リファレンスの選択に影響（学術なら ACADEMIC-* 系、ピッチなら QUALITY-LOGICAL-STRUCTURE.md のピッチ10ステップ等） |
| 5 | **情報密度** | 1枚あたりの情報量はどうしますか？ | ・標準 (推奨)<br>・高密度・ハンズオン<br>・余白多め・講演向け | 標準 → デフォルト。高密度 → 第一選択は「スライド分割」、収まらない場合のみ `references/CSS-OVERFLOW-AND-DESIGN.md` の縮小オーバーライドを適用。講演向け → 余白を広めに、1枚の要素数を絞る |
| 6 | **シンタックスハイライト** | コードに色付け（シンタックスハイライト）しますか？ | ・コードあり → 有効 (推奨)<br>・コードなし → 無効 | 有効 → デフォルト（engine が自動適用）。主要言語を併せて確認し `<code class="language-xx">` を付与。無効 → デッキに `<div class="deck" data-no-highlight>` を付ける（詳細: `references/CSS-OVERFLOW-AND-DESIGN.md` 「シンタックスハイライト」） |
| 7 | **言語** | スライドの言語は？ | ・日本語 (推奨)<br>・英語<br>・日英併記 | 日本語 → デフォルト。英語 → ENGLISH-* 系リファレンス（表現・発音）を参照。日英併記 → 主要見出しに英語併記 |
| 8 | **ロゴ** | ロゴは何を使いますか？（**必ず確認する。勝手にロゴを決めない**） | ・指定ロゴを使う（パス or 画像を提供してもらう）<br>・プレースホルダー `shared/logo/logo.svg` のまま (推奨/未定時)<br>・ロゴなし | 指定 → `shared/logo/` に配置し各スライドに `<img class="slide-logo" src="../../shared/logo/{file}" alt="Logo" onerror="this.style.display='none'">`。プレースホルダー → 既定の `logo.svg`（差し替え前提）。なし → `<img class="slide-logo">` を置かない。配置面はタイトル・コンテンツ・エンディングのみ（詳細: `references/BUILD-SYSTEM.md` 「ロゴの使い方」） |
| 9 | **最終用途・エクスポート** | 最終的にどう使いますか？（複数選択可） | ・ブラウザ発表 (推奨)<br>・PDF配布<br>・録画・配信<br>・オフライン会場PC | 録画・配信 / オフライン → 全アセットをローカル化（CDNフォント排除・hljs はローカル vendoring が既定）し、PDF出力を必ず検証。PDF配布 → `@media print` のスケール調整を検証 |
| 10 | **素材忠実度・QCD** | 素材の扱いと優先軸は？（2軸） | ・原文を極力保持 (推奨) / 発表用に要約・再構成可<br>・品質優先 / 納期優先 | 原文保持 → `references/SOURCE-CONVERSION-RULES.md` を厳守。要約可 → 1メッセージ化を優先。品質優先 → リハーサル・はみ出し検証を厚く。納期優先 → 最小構成で素早く |

### 運用ルール（Stage 0 と Stage 1〜5 の関係）

- **全ケースで Stage 0 を最初に実施する**（ただし既知項目はスキップし、確定値として進める）。
- Stage 0 完了後、下記「プレゼン品質ヒアリングフロー（Stage 1〜5）」でコンテンツ面を詰める。
- **ユーザーが「急ぎ」と指示した場合**: Stage 0 はテーマ・言語・ロゴの3点のみ最低確認し、残りはデフォルト（ライト白/日本語/ロゴなし）で進める。

## プレゼン品質ヒアリングフロー

tachikoma-doc-slideがスライド作成前にAskUserQuestionで収集すべき情報。回答に応じてスライド構成・スタイル・メッセージ設計を最適化する。

### Stage 1: 聴衆分析（必須）

以下を確認する：
- **誰に向けて？**: 聴衆の属性（意思決定者/技術者/一般参加者/学生等）
- **知識レベル**: テーマに対する専門度（初心者/中級/専門家）
- **期待**: 聴衆がこの場に何を期待しているか
- **人数・形式**: 対面/オンライン、少人数/大人数
- **DMU分析**（組織向けプレゼンの場合）: 門番・ユーザー・決定者・承認者・影響力者・購買者の役割マップ。キーパーソンの特定
- **視座・視点・視野**: 聴衆の「2つ上の視座」で設計すべきかを判断
- **「読む人」vs「聞く人」**: 聴衆スタイル判定でスライド情報密度を調整
- **プレゼン種別**: 学術プレゼンか商業プレゼンか（学術なら ACADEMIC-* 系リファレンスを参照）
- **発表形式**: 国際学会か国内発表か（国際学会なら ENGLISH-* 系・文化差への配慮を必須化）
- **ノンネイティブ参加者比率**: 比率が高い場合は語彙選択・スライド設計・配慮を追加適用（ENGLISH-PRONUNCIATION-DELIVERY.md 参照）

→ 回答に基づき、情報密度・専門用語の量・ストーリー vs データの比率を調整

### Stage 2: プレゼンタイプ判定（必須）

以下を確認する：
- **プレゼンの目的**: 提案/報告/啓発/ピッチ のいずれか
- **求める行動**: プレゼン後に聴衆にとってほしい具体的行動
- **心理状態**: 説明後に聴衆にどんな心理状態になってもらいたいか（「これならできそう」「危機感を持ってほしい」等）

- **提案書の場合**: 提案書構成10ステップの前提確認（ゴール設定・相手分析・現状/あるべき姿/ギャップ）をヒアリングに組み込む
- **段階的ゴール設定**: 初回提案では高すぎるゴールを避け、次のアクション（次回商談に応じる等）を目標とする
- **学術研究発表の場合**: 材料/方法・結果/考察・結論の構成を優先 → ACADEMIC-RESEARCH-STRUCTURE.md 必読
- **ポスター発表の場合**: 通常デッキと異なる目的・構成・声かけ戦略を適用 → ACADEMIC-POSTER-PRESENTATION.md 必読

→ 回答に基づき、4基本タイプから構成パターンを選択、展開型（分解型/積上型/選択肢型/意外性展開型）を決定

### Stage 3: ゴール・メッセージ設計（推奨）

以下を確認する：
- **1メッセージ**: このプレゼンで聴衆に持ち帰ってほしい、たった1つのことは何か
- **お土産**: 聴衆が翌日同僚に話すとしたら、どう伝えてほしいか
- **発表時間**: 持ち時間（分）

→ 回答に基づき、メッセージの核を設計、スライド枚数を算出（持ち時間÷3分/枚を目安）

### Stage 4: ストーリー構成（オプション）

素材が提供されない場合、以下を確認する：
- **共感ポイント**: 聴衆が「あるある」と感じるポイント
- **HSM（Holy Shit Moment）**: 聴衆を驚かせたいデータ・事実はあるか
- **ストーリータイプ**: 個人体験/第三者事例/ブランドストーリー のいずれを使うか

→ 回答に基づき、感情のアーク・エモーションカーブを設計

### Stage 5: 学術プレゼン専用設定（推奨）

学術プレゼン・国際学会発表の場合に確認する：
- **Q&A 想定時間**: 質疑応答の持ち時間（分）と形式（挙手制/指名制）
- **批判的質問への備え**: 想定される批判的質問の有無と重点分野
- **ネットワーキング目標**: 共同研究獲得 / 認知向上 / フィードバック収集 のいずれか
- **懇親会・交流セッション参加**: 発表後のネットワーキング機会の活用有無

→ Q&A 対策は ACADEMIC-QA-MASTERY.md、ネットワーキングは ACADEMIC-NETWORKING.md を参照

### ヒアリングフローの実行ルール

| 状況 | ヒアリング範囲 |
|------|-------------|
| **全ケース共通** | **まず Stage 0（制作要件・スタイル決定）を最初に実施**（既知項目はスキップ） |
| ソース素材なし + トピック指定のみ | Stage 0 → Stage 1〜4 すべて実行 |
| ソース素材あり（Markdown等） | Stage 0 → Stage 1〜2 必須、Stage 3 推奨 |
| 既存デッキの修正 | Stage 0 のうち変更対象項目のみ → 必要なStageのみ |
| ユーザーが「急ぎ」と指示 | Stage 0 はテーマ・言語・ロゴの3点のみ → Stage 1のみ（最低限） |
| 学術プレゼン・国際学会発表 | Stage 0 → Stage 1〜3 必須、Stage 5 推奨 |
