## Googleアルゴリズムの全体像

### コアアルゴリズムとアップデートの方向性

Googleの検索アルゴリズムは単一のルールセットではなく、複数のシステムが協調して動作する。主要な評価軸は「コンテンツの品質・関連性」「ユーザー体験」「権威性」「技術的なアクセス可能性」の4つに集約される。

アルゴリズム更新の方向性は一貫している。**「人間にとって有用なコンテンツ（people-first content）を上位表示し、検索順位操作目的のコンテンツを排除する」**。2024年3月以降、Helpful Content System（HCS）はコアアルゴリズムの一部として統合され、有用性の評価はコアアップデートを通じて継続的に行われる。

凡例: 🟢=Google公式発表 / 🔵=業界コンセンサス・第三者分析

### アップデート年表（2024–2026/6）

| 時期 | 種別 | 焦点・内容 |
|------|------|-----------|
| 🟢 2024年3月（3/5〜） | コア＋スパムポリシー | HCSをコアへ統合。複数システム刷新。非オリジナルコンテンツ約45%削減を発表。**新スパムポリシー3種**を同時導入 |
| 🟢 2024年3月 | スパムポリシー新設3種 | ①Expired Domain Abuse（期限切れドメイン悪用）②Scaled content abuse（大量生成コンテンツ悪用=AI/人間問わず無価値量産）③Site Reputation Abuse（寄生サイト/パラサイトSEO） |
| 🔵 2025年3月（3/13–3/27, 14日間） | コア | 全種別サイトから関連性・満足度の高いコンテンツを表示。クリエイター由来コンテンツの可視化を強調 |
| 🔵 2025年6月（6/30–7/17, 16日間） | コア | 同方針継続。一部サイトで過去ペナルティからの部分回復。トピカルオーソリティ重視を再強化との業界分析 |
| 🔵 2025年8月（8/26–9/22, 27日間） | スパム | 全言語・全地域での広範なスパム検出。24時間以内に影響が出始め9/9前後に強化 |
| 🔵 2025年12月（12/11–12/29, 18日間） | コア | 同方針継続。週末を中心に激しい変動 |

> **スパムポリシー3種は「ポリシー」**（🟢公式）であり、コアアップデート（ランキングシステム調整）とは別軸。両者がしばしば同時告知される点に留意。

### 運用方針（示唆）

- [ ] AI/人手を問わず「検索順位狙いの大量生成」は Scaled content abuse の対象になり得る → 量より独自価値を追求
- [ ] 権威ドメインを借りた寄生コンテンツ（Site Reputation Abuse）は明確なポリシー違反
- [ ] コア更新で下落した場合は個別ペナルティではなく「相対的に有用性が低い」評価。次回更新時の回復には品質の底上げが必要（数週間以上かかる）
- [ ] 2026年以降もアップデートは継続されるため、公式 Search Central ブログを定期確認する

## ランキング要因の全体像

### 相対的重要度（業界コンセンサス・公式は重み非公開）

🔵 **Googleは個々のランキング要因の重み付けを公式に開示していない**。以下は複数の業界調査・専門家分析に基づく現時点のコンセンサスであり、相関関係に基づく推計である。

| 要因 | 相対的重要度の傾向 | 補足 |
|------|------------------|------|
| コンテンツ品質・関連性 | **最重要** | 「満足できるコンテンツの継続的な公開」が筆頭。people-first content |
| ユーザー体験（UX） | **高（2位級）** | ナビゲーション・モバイル最適化・表示速度。低UXは高品質コンテンツの効果を打ち消す |
| 被リンク（バックリンク） | **中〜高（約13%との推計）** | 歴史的な支配的地位から相対的に低下。ただし権威性・トピック関連性の高いリンクは依然強力 |
| 技術SEO | **基盤（必須条件）** | クロール可能性・サイト構造・Core Web Vitals・モバイルファースト索引。欠けると上位は困難 |

### 重要な原則

- 🔵 要因は独立して働かず、**コンテンツ・権威性・UX・技術の総合力**で評価される。複数要因が相互補強したとき最良の結果が得られる
- 🟢 GoogleはCore Web Vitals等のページエクスペリエンスがランキングに影響すると明言。ただし優れたコンテンツを覆すほどではなく、同等品質間のタイブレーカー的位置づけ

### 技術SEO基本チェックリスト（詳細は TECHNICAL-SEO.md）

- [ ] Core Web Vitals（LCP / INP / CLS）を良好域に維持（詳細は TECHNICAL-SEO.md）
- [ ] モバイルファースト索引対応（レスポンシブ・モバイル可読性）
- [ ] クロール/インデックス最適化（XMLサイトマップ・robots.txt・内部リンク・canonical）
- [ ] HTTPS・構造化データ（Schema.org / JSON-LD）

## ユーザー行動シグナル（公開一次資料が示す代理測定モデル）

### 検索エンジンは「意味を理解する」のではなく「ユーザー反応を蓄積する」

🔵 公開された検索アルゴリズム関連の一次資料（流出した大規模APIドキュメント・反トラスト訴訟で開示された内部資料など）から読み取れる核心は、**検索エンジンは文書の意味そのものを理解しているわけではなく、ユーザーがその文書にどう反応したかを観測・蓄積し、将来の順位に反映する代理測定（proxy measurement）構造**を持つという業界コンセンサスである。つまり「良いコンテンツか」を直接判定するのではなく、「ユーザーが満足したように振る舞ったか」を手がかりに品質を推定している。

観測・追跡されるとされるユーザー操作シグナルの例:

- [ ] **クリック**（どの結果を選んだか・クリック関連だけで30超の変数があるとされる）
- [ ] **結果への注視（dwell / 滞在の度合い）**
- [ ] **カルーセル等のスワイプ操作**
- [ ] **再クエリ入力**（同じ意図で言い換えて検索し直す＝直前の結果に満足しなかったシグナル）
- [ ] **スクロール深度**（どこまで読み進めたか）
- [ ] **マウスホバー / ポインタ挙動**
- [ ] **読了**（最後まで読まれたか）

これらの行動データは新しい概念ではなく、長年（少なくとも2000年代前半から）順位付けに利用されてきたとされる。ブラウザを提供する主目的の一つも、こうした行動データの収集にあると分析される。

### ランキングの3本柱（文書・Web・ユーザー）

🔵 ランキング評価は、大きく次の3本柱で構成されると整理できる。

| 柱 | 何が語るか | 具体シグナル |
|----|-----------|-------------|
| (1) **文書が自身について語る** | ページ自体の記述 | 本文の語句・title・heading・太字・フォントサイズ・構造 |
| (2) **Webが文書について語る** | 外部からの言及 | 被リンク（バックリンク）・第三者の引用 |
| (3) **ユーザーが文書について語る** | 実際の利用反応 | クリック・注視・再クエリ・スクロール・読了などのエンゲージメント |

このうち **(3) ユーザーが文書について語るエンゲージメントは、relevance（関連性）に次ぐ規模のシグナルである可能性が高い**とされる。すなわちエンゲージメントは、被リンクやオンページ要素と並ぶ「事実上の主要ランキング要因」として扱う価値がある。なお「誰が書いたか」という著者シグナルも追跡対象に含まれるとされ、E-E-A-T の権威性評価とも接続する。

### 実務含意: 低エンゲージメント＝低品質判定と AI 低品質量産リスク

- 🔵 ユーザーが滑り読みで離脱し、すぐ再クエリし、読了されないページは、**「低エンゲージメント＝低品質」と推定され順位低下圧を受けうる**。UX・エンゲージメント施策を「SEOの周辺」ではなく中核に据える論拠がここにある。
- 🔵 これは、検索意図を満たさない AI 生成コンテンツの大量公開が機能しない理由の一次資料的根拠でもある。表面的に体裁が整っていてもユーザー反応が伴わなければ低品質と判定され、量産すればサイト全体の評価を毀損しうる（Scaled content abuse の文脈とも連動。詳細は `RISK-GOVERNANCE.md`）。
- 🔵 逆に言えば、**期待されるクリック率（期待CTR）を実CTRが上回ると順位の昇格圧、下回ると降格圧がかかる**というCTRフィードバックループも、この代理測定モデルの一断面である。meta description をCTRレバーとして最適化する具体手順は `CONTENT-KEYWORDS.md` の「CTRフィードバックループとしての扱い」を参照。

> **注記**: 上記は流出資料・訴訟開示資料・専門家分析に基づく業界コンセンサスであり、Googleが個々のシグナルの重み付けを公式に追認したものではない（🟢公式は重み非公開）。相関に基づく推計として扱う。

出典: 🔵 https://searchengineland.com/google-ranking-systems-leaked-documents / https://www.searchenginejournal.com/google-search-document-leak/

## E-E-A-T

### 4要素とTrust中心の構造

🟢 2022年12月、Googleの品質評価者ガイドラインが更新され、E-A-T（専門性・権威性・信頼性）に**Experience（経験）**が追加されE-E-A-Tとなった。一次的・実体験に基づくコンテンツへの重視が明確化された。

**E-E-A-Tは単一のランキングシグナルではない**（🔵業界コンセンサス）。品質評価者がコンテンツ品質を判定するためのガイドライン概念であり、評価者の判定はアルゴリズム改良フィードバックに使われる。

4要素と**Trust（信頼）が中心**である構造:

```
           ┌────────────┐
           │   Trust    │ ← 中心。E/E/Aはすべて信頼を裏づける証拠として機能
           └─────┬──────┘
     ┌───────────┼───────────┐
 Experience  Expertise  Authoritativeness
（経験）      （専門性）    （権威性）
```

信頼性はユーザー保護に直結する最終判断軸。サイトの安全性・正確性・透明性・目的の正当性が問われる。経験・専門性・権威は「なぜそのコンテンツが信頼に値するか」を裏づける証拠として機能する。

### YMYL領域

🔵 **YMYL（Your Money or Your Life）**とは、健康・金融・安全・法律・人生の重大判断に影響するトピックの総称。E-E-A-T基準がとりわけ厳格に適用される。

- 誤情報が実害を生むため、著者の資格・出典の明示・組織の信頼性が一般トピックより重く評価される
- 2025年はAI生成コンテンツの氾濫を受け、Googleは「人間の実体験・専門性が表れたコンテンツ」の識別強化との業界分析がある（相関であり因果の公式確認ではない点に留意）

YMYL例: 医療診断・薬の服用・投資アドバイス・法的手続き・安全に関する情報

### 各要素を高める施策チェックリスト

**Experience（経験）**
- [ ] 一次情報（実際に使用・訪問・体験した記録）を本文に含める
- [ ] オリジナル写真・スクリーンショット・動画・計測データを掲載
- [ ] 「実際に試した」「現地で確認した」等の体験ベース記述を具体化
- [ ] レビューは実使用に基づく長所短所の両論併記

**Expertise（専門性）**
- [ ] 著者バイオに資格・経歴・専門領域を明記（author byline）
- [ ] 著者プロフィールページを用意し外部の権威ソースとリンク
- [ ] 専門用語の正確な使用、最新の事実・統計で裏づけ
- [ ] YMYLでは有資格者・専門家による執筆/監修を明示

**Authoritativeness（権威性）**
- [ ] 業界の権威サイト・一次研究からの被リンク獲得
- [ ] 専門メディア・公的機関での言及（メンション）を蓄積
- [ ] トピックを網羅し「その分野の情報源」と認識される構造化（トピカルオーソリティ）
- [ ] 著者・ブランドの第三者評価（受賞・引用・ナレッジパネル）

**Trust（信頼）**
- [ ] HTTPS・明確な連絡先・運営者情報・所在地を掲載
- [ ] 著者情報・公開日/更新日・出典リンクで透明性確保
- [ ] レビュー/評判の管理（外部の評価が信頼性に影響）
- [ ] 正確性の担保（事実確認・引用・誤情報の排除）
- [ ] EC/金融はプライバシーポリシー・返金規定・セキュリティ表示

出典: 🟢 https://developers.google.com/search/blog/2022/12/google-raters-guidelines-e-e-a-t / 🔵 https://www.semrush.com/blog/eeat/ / https://backlinko.com/google-e-e-a-t

## 検索意図（Search Intent）

### 4分類

| 分類 | ユーザー状態 | クエリ例 | 最適なコンテンツ |
|------|------------|---------|----------------|
| **Informational（情報収集型）** | 学びたい・調べたい（🔵全クエリの約70%との推計） | how/what/why/「とは」 | ガイド・How-to・解説記事・FAQ |
| **Navigational（案内型）** | 特定サイト/ページへ行きたい | ブランド名・ログイン | 公式ページ・ブランドハブ（第三者は上位獲得困難） |
| **Commercial Investigation（商業調査型）** | 購入前の比較検討 | best/比較/「○○ vs ○○」/レビュー | 比較記事・ランキング・レビュー・購入ガイド |
| **Transactional（取引型）** | 行動準備完了（購入/登録） | buy/価格/割引/near me | 商品ページ・LP・料金ページ |

### 意図合致コンテンツ設計（チェックリスト）

- [ ] 対象キーワードの意図を4分類で判断し、コンテンツ種別（ガイド/比較/商品ページ）を意図に一致させる
- [ ] 1ページ1主要意図の原則。意図がずれると上位表示困難・直帰率上昇を招く
- [ ] Informational コンテンツに取引導線を詰め込みすぎない（意図逸脱=ユーザー体験低下）
- [ ] 意図が混在するクエリではSERPで複数コンテンツ型が混在する → 自社が狙える型を選定

### SERP分析の手順

検索意図はGoogleがSERP上位で示す答えを観察することで判断できる。

- [ ] ターゲットクエリを実際に検索し、**上位10件のコンテンツ型を観察**（記事中心か、商品ページ中心か、動画か）
- [ ] 表示されるSERP機能（強調スニペット/People Also Ask/ローカルパック/ショッピング）から意図を逆算
- [ ] 上位ページの見出し構成・網羅トピック・コンテンツ長を分析し自社コンテンツのギャップを特定
- [ ] 意図に合わせてコンテンツ種別を決定してから執筆に入る

## トピッククラスター / トピカルオーソリティ

### ピラーページとクラスターページの構造

🔵 **トピッククラスター**とは、1つの「ピラーページ」と複数の「クラスターページ」を内部リンクで結ぶ構造的なコンテンツ戦略。

```
         [ピラーページ]
        「トピック全体を広く・浅く網羅」
         ↕            ↕            ↕
 [クラスターA]  [クラスターB]  [クラスターC]
「サブトピックを  「サブトピックを  「サブトピックを
 狭く・深く」    狭く・深く」    狭く・深く」
```

ピラーが各クラスターへ、各クラスターがピラーへ相互リンクし、「サイトがそのトピックの権威（トピカルオーソリティ）」であることをGoogleに示す。

2025年6月コア更新では「主題を網羅的・一貫的・信頼性高くカバーするサイト」が評価され、トピカルオーソリティ重視の傾向が強まっているとの業界分析がある。

### 構築手順（チェックリスト）

- [ ] **コアトピック選定**: 事業・専門領域と重なる主要テーマを1〜3つ定める
- [ ] **サブトピック洗い出し**: People Also Ask・検索サジェスト・競合サイト分析で網羅性を担保
- [ ] **ピラーページ作成**: トピック全体を俯瞰し、各サブトピックへ内部リンクを設置
- [ ] **クラスターページ作成**: 各サブトピックを深掘りし、必ずピラーへ戻る内部リンクを設置
- [ ] **継続的な更新**: 一貫した公開・更新で網羅性と鮮度を維持

### 内部リンク設計（チェックリスト）

内部リンクはもっとも過小評価されつつ低コストで改善できるSEOレバーとされる（🔵業界コンセンサス）。多くのサイトでは大半のページが内部リンク1本以下 → 監査して補強することで改善余地が大きい。

- [ ] **双方向リンク**（ピラー↔各クラスター）でPageRankを分配しトピックシグナルを強化
- [ ] クラスター間も関連があれば相互リンク（孤立ページを作らない）
- [ ] アンカーテキストは説明的にする（対象ページの主題を表す語を使い、「こちら」「詳細」等の無意味なテキストを避ける）
- [ ] 重要な内部リンクはモバイル版にも含める（モバイルファーストインデックス対応）
- [ ] リンクはHTMLの `<a href>` で実装（JSのみで生成されるリンクはクロールされにくい）
- [ ] 内部リンク監査ツール（Screaming Frog / Ahrefs / Search Console）で孤立ページを定期検出

### ベクトル空間／エンティティ検索の実務翻訳

🔵 トピカルオーソリティが効く背景には、**検索エンジンがキーワードの完全一致だけでなく「意味的近さ」で関連性を測っている**という仕組みがある。これを実務施策へ翻訳すると、関連語の網羅とエンティティ結合の強化という具体策に落ちる。

- **トークンのベクトル化と類似度計算**: 検索エンジンはテキスト・概念・画像といったトークンを多次元空間上のベクトルに変換し、ベクトル間の距離で概念どうしの類似度を計算する。これにより、完全一致だけでなく**部分一致・類似概念**も関連ありと判定される。
- **含意（語を使わなくても該当しうる）**: たとえば、ある詩人の作品を解説する記事は「ソネット（特定の詩形）」という語を本文で一度も使っていなくても、「その詩人のソネット」を問うクエリの結果に出得る。意味空間上で近接しているためである。
- **関連語追加でエンティティ結合を強化**: 逆にこの性質を利用し、**主題に関連する語（例: 特定の韻律名など、そのトピックの専門語彙）を意図的に本文へ含めると、対象エンティティとの結び付きが強まり関連性評価が上がる**。トピッククラスターのクラスターページで「サブトピックを狭く深く」掘るとき、関連語彙を取りこぼさないことがエンティティ結合の観点でも効く。
- **源流は「Things not strings」**: この発想は、文字列の一致ではなく実体（エンティティ）どうしの関係で世界を捉える知識グラフ（2012年に提唱された「Things not strings」の考え方）に連なる。
- **失敗様式: 情報不足だと失敗する＝人間介入が必要**: ベクトル空間／エンティティ検索もLLMによる生成も、**利用可能な情報が不足していると正しく機能しない**（必要な関連情報を持たないと誤った近接判定・生成をしうる）。ゆえに「どの関連語・どの一次情報が欠けているか」を補う**人間の介入が前提**となる。網羅性の判断を完全自動化しきれない理由でもある。

> 関連語の網羅は「キーワード詰め込み」とは異なる。読者の意図を満たす文脈の中で自然に専門語彙を扱うことが条件で、無関係な語の羅列はかえって品質シグナルを下げる。

## ゼロクリック時代の基盤対応

### 現状（🔵第三者調査・数値は調査機関・時期により変動）

- ゼロクリック検索（SERP内で完結）: Semrush 2025調査で米国約58.5%・EU約59.7%。SparkToro 2026調査では米国Google検索の約68%がクリックなしで終了
- AI Overviews（AIによる概要回答）: 情報系クエリの約30%で表示。全キーワードの約20.5%に出現（一部カテゴリは60%）との分析。表示クエリでオーガニックCTRが約61%低下との分析
- People Also Ask（PAA）: 検索結果の約78%に出現

> AI Overviews の詳細・GEO・AEO・LLMO・llms.txt の最適化手法は `AI-SEARCH-GEO.md` を参照。

### 対応策（チェックリスト）

強調スニペット・PAA・AI Overviews への対応は「answer-first」構造が共通の鍵。

- [ ] **強調スニペット狙い**: 質問を見出し化し、直下に簡潔な定義/手順（40–60語前後）を置く。表・リスト・段落の各形式に対応
- [ ] **PAA活用**: PAAの質問をコンテンツ構成に取り込み、各質問を見出し＋構造化回答にする
- [ ] **構造化データ（Schema.org / JSON-LD）実装**: Article・Product・Review等の主要タイプを実装（詳細は TECHNICAL-SEO.md）
- [ ] **JSで注入される構造化データを避ける**: AIクローラはJS注入のマークアップを取りこぼすことがある → サーバーサイドレンダリング/静的HTMLで実装
- [ ] **answer-first 構造**: キーワードでなく「問いへの答え」を冒頭に置き、AI応答に直接引用されやすい構造にする
- [ ] **KPI再設計**: 従来の順位・クリック数だけでなく、SERP機能内での露出・ブランド想起を計測対象に追加（詳細は MEASUREMENT.md）
