## 執筆前リサーチ（結果の質を決める工程）

記事の質は、書き始める前にどれだけ調べたかでほぼ決まる。着手前に済ませておくべき3種のリサーチを扱う。

### 読者リサーチ（最重要）

読者を具体的に解像できていなければ、どれだけ優れた文章技術を使っても響かない。以下の手法を組み合わせて読者像を立体化する。

**調査手法チェックリスト**:
- [ ] **検索行動分析**: ターゲットKWのSERP上位を読者の目線で読み、どんな悩み・言葉遣いで検索しているかを推測する
- [ ] **既存顧客への直接ヒアリング/アンケート**: 購入・利用の決め手、迷った点、比較検討した競合を直接聞く
- [ ] **レビュー・Q&Aサイト・SNSの実際の発言収集**: 読者自身の生の言葉（語彙・言い回し）をそのまま拾い、文章に反映する
- [ ] **カスタマーサポート・営業担当への聞き取り**: 現場が日常的に受ける質問・誤解・クレームは読者の本音の宝庫になる

**読者解像度チェックリスト**（埋まっていない項目は執筆前に埋める）:
- [ ] 属性（年齢層・性別・職業・居住地など）
- [ ] 悩み・欲求の具体的な言語化（表面的な悩みだけでなく背景の欲求まで）
- [ ] 情報収集の段階（今なぜ・何を知りたくて検索しているか）
- [ ] 購買検討段階（比較検討中か、今すぐ客か）
- [ ] 抱えている誤解・思い込み
- [ ] 過去に試して失敗した方法
- [ ] 意思決定を妨げている不安・反対意見
- [ ] 普段使っている語彙・言い回し（専門用語か口語か）
- [ ] 情報を得ている媒体（SNS・検索・雑誌・知人など）
- [ ] 記事を読んだ後にどう行動してほしいか（記事のゴール）

### 商品・サービスリサーチ

読者理解と対になるのが、紹介する商品・サービス側の理解。執筆の説得力を大きく左右する。

- [ ] 機能・スペックの一覧化（後述の「特徴→ベネフィット変換」の材料になる）
- [ ] 競合と比較した際の優位点・弱点の整理
- [ ] 実際に使ってみた使用感・仕上がり・所要時間の記録
- [ ] 想定される利用シーン・利用者像の洗い出し
- [ ] 価格帯と、価格に納得できる根拠の整理
- [ ] よくある質問・誤解の収集（FAQ設計の材料になる）
- [ ] 提供者側があまり語りたがらない弱点の把握（後述の「先回り自己開示」の材料になる）

### 切り口リサーチ（差別化・穴場発見）

同じテーマでも「どの切り口で書くか」が競合との差別化を決める。競合分析の定量チェック手順そのものは `OPERATIONS-DIAGNOSTICS.md`（本スキル内）を参照し、ここでは執筆前の切り口発想に絞る。

- [ ] 競合上位記事が扱っていない疑問・角度を探す
- [ ] 「自社だからこそ言える」一次情報・立場（後述のMSPの材料）を洗い出す
- [ ] ターゲット読者のセグメントを絞り込むことで生まれる独自の切り口を検討する
- [ ] 季節性・トレンド・時事性と絡めた切り口の可能性を確認する

### 検索ニーズ深掘り「なぜ?」2回ルール

表面的なキーワードの奥にある本当の検索意図まで踏み込むための簡易フレーム。

- [ ] 表面クエリに対して「なぜそれを知りたいのか?」を自問する（1回目の「なぜ」）
- [ ] 出てきた答えに対してさらに「なぜそれが必要なのか?」を自問する（2回目の「なぜ」）
- [ ] 2回掘り下げた先に見える本質的な欲求・不安を記事の核に据える
- [ ] 表面クエリへの回答だけで終える記事は、本質的な欲求まで答える競合記事に負けやすい

## 情報の階層と一次情報の調達

### 一次/二次/三次情報と配合比率

記事の説得力は情報の出どころで決まる。3階層に分類し、配合を意識する。

| 階層 | 定義 | 具体例 | 配合の目安 |
|------|------|--------|-----------|
| **一次情報** | 自分自身が直接体験・調査・観察して得た情報 | 実使用レビュー・独自調査・現場取材・インタビュー | 最大化することを目安にする |
| **二次情報** | 一次情報を編集・解説した情報 | 専門家の解説記事・公的機関の報告書の要約 | 一次情報を補強する程度に絞る |
| **三次情報** | 二次情報をさらにまとめた情報 | まとめサイト・他記事の要約の孫引き | 原則使わない |

- [ ] 三次情報（孫引き・まとめの再まとめ）は独自性がなく、検索エンジン・読者双方から低評価になりやすいため使わない
- [ ] 二次情報を使う場合は出典を明示し、可能な範囲で一次情報の裏付けを添える
- [ ] 配合比率は執筆時の自己点検の目安であり、「まず一次情報を最大化してから二次情報で補う」という優先順位で運用する

### 一次情報（＝E-E-A-T「経験」）の調達手法

一次情報は `FOUNDATIONS-RANKING-EEAT.md` が扱うE-E-A-Tの **Experience（経験）** を、コンテンツ本文レベルで実装する部分に当たる。

**調達手法チェックリスト**:
- [ ] 自分で実際に使用・購入・体験し、その過程を記録する
- [ ] 現場・店舗・拠点に足を運び取材する
- [ ] 顧客・利用者に直接インタビューする
- [ ] 独自にアンケート・調査を実施し一次データ化する
- [ ] 社内の専門家・経験者から監修・談話をもらう

**禁止事項**:
- [ ] 🔴 **エアレビュー（実際に使っていないのに使ったかのように書くレビュー）は行わない**。E-E-A-Tの信頼性を毀損し、発覚時のブランド毀損リスクが大きい

**経験の資産化パターン**:
- [ ] 体験談・失敗談を隠さず書く（成功事例だけでは説得力が薄い）
- [ ] 現場写真・独自データ・スクリーンショットなど一次情報の証拠を本文に添付する
- [ ] 一次情報は記事本文だけでなく、事例ページ・お客様の声ページ（後述）にも展開する

### 独自コンテンツ比率の目安

- [ ] 記事全体に占める「他では読めない独自要素（一次情報・独自見解・独自データ）」の比率を**5割以上を目安**に確保する
- [ ] 残りを二次情報の整理・一般的な解説に充てる
- [ ] この比率は執筆時の自己点検の目安であり、厳密な数値検証の対象ではない

## 記事構成の型（コンバージョン文章フレームワーク）

### 全体骨組み（問題提起→解決方法→根拠）

詳細な型に入る前段階として、記事の骨組みは次の3要素で捉えると設計しやすい。

- [ ] **問題提起**: 読者が抱える悩み・疑問を言語化して見せる（「わかってくれている」という信頼の起点）
- [ ] **解決方法**: その悩みに対する解決の方向性・手段を提示する
- [ ] **根拠**: なぜその解決方法が有効なのか、一次情報・データ・専門性で裏付ける

### コンバージョン文章フレームワークと使い分け

コンバージョンを狙う文章にはいくつかの定番フレームワークがある。記事の目的・読者の検討段階に応じて使い分ける。

| フレームワーク | 構造 | 向いている場面 |
|---|---|---|
| **AIDA** | Attention（注意）→Interest（興味）→Desire（欲求）→Action（行動） | 広告寄りの短めのセールスコンテンツ |
| **QUEST** | Qualify（対象者を絞る）→Understand（理解を示す）→Educate（教育する）→Stimulate（刺激する）→Transition（行動へ導く） | 情報収集型の読者を行動に導く長文記事 |
| **PASONA** | Problem（問題）→Affinity（親近感）→Solution（解決策）→Offer（提案）→Narrowing down（絞込）→Action（行動） | 悩み解決型の商品・サービス紹介記事 |
| **PASBECONA** | PASONAにBridge（橋渡し）・Explanation（説明）・Confirmation（確認）を加えた拡張型 | より丁寧な合意形成が必要な高単価商材 |
| **BEAF** | Benefit（便益）→Evidence（証拠）→Advantage（優位性）→Feature（特徴） | 商品スペック比較・購入直前の検討記事 |

- [ ] 読者の検討段階（情報収集・比較検討・購入直前）に応じてフレームワークを選ぶ
- [ ] 1記事1フレームワークに固執せず、セクションごとに部分的に組み合わせてもよい

### PREP法（answer-first・AIが引用しやすい回答の型）

- [ ] **Point（結論）→Reason（理由）→Example（具体例）→Point（結論の再提示）** の順で書く
- [ ] 結論を先に述べるanswer-first構造は、AI Overviews等の生成AI回答に引用されやすい型でもある（詳細は `AI-SEARCH-GEO.md` の引用されやすいコンテンツ構造の節を参照）
- [ ] FAQ・見出し直下の要約・強調スニペット狙いの導入文と特に相性がよい

### パラグラフ論理順と逆ピラミッド

- [ ] 段落単位でも「結論→根拠→具体例」の順を徹底し、結論を後回しにしない
- [ ] 記事全体も**逆ピラミッド構造**（最も重要な情報・多数派のニーズへの回答を先頭に、詳細・少数派向けの情報を後方に）で組み立てる
- [ ] スマホでの流し読み・途中離脱を前提に、前半だけ読んでも要点が伝わる設計にする

## 離脱を防ぐ設計

### 4つのNOT

読者は記事に対して4段階の心理的ハードルを持つ。各段階の離脱要因を潰す設計が離脱防止の骨格になる。

| NOT | 意味 | 対策の方向性 |
|---|---|---|
| **Not Open（開かれない）** | タイトル・meta descriptionの時点で読まれない | タイトル設計（後述）でクリックを獲得する |
| **Not Read（読まれない）** | 開いても最初の数秒で離脱される | 惹きつけ技法・リード文設計（後述）で引き込む |
| **Not Believe（信じられない）** | 読んでも内容を信用してもらえない | 一次情報・E-E-A-T・反論処理（後述）で信頼を積む |
| **Not Act（行動されない）** | 信じても行動に移さない | 損失回避・行動障壁の除去・明確なCTA設計で後押しする |

- [ ] 4段階のどこで離脱が起きているかをGA4等の行動データで仮説立てし、該当箇所を優先的に改善する

### 書き出し（リード文）の心理設計

- [ ] **共感**: 読者の悩みへの共感を最初の数行で示す（「わかってくれている」という安心感）
- [ ] **ゴール提示**: この記事を読み終えるとどうなれるかを明示する
- [ ] **本文への興味喚起**: 続きを読みたくなる一言を添えて本文へ橋渡しする

### 惹きつけ技法

- [ ] **すべり台効果**: 一文ごとに「次を読みたくなる」フックを仕込み、読者を滑らせるように最後まで読ませる
- [ ] **5W2Hの早期解消**: 「これは誰向けの何の記事で、読めば何が得られるか」を冒頭で解消し、読者の不安を消す
- [ ] **冒頭に「得られること」を明示**: 記事の要点・結論を先出しする（PREP法・逆ピラミッドとも整合する）
- [ ] **目次を必ず設置する**: 読者が自分に必要な箇所へ直接ジャンプできるようにする（離脱防止と回遊性を両立させる）

### 読みやすさの数値基準

- [ ] 一文60字程度を目安にする（長文は分割する）
- [ ] **一文一義**（1つの文には1つの主張のみ）を徹底する
- [ ] 冗長な修飾語・重複表現を削る
- [ ] **スマホビューで実際に表示確認する**（PC編集画面と改行位置・余白感が大きく異なる）
- [ ] **音読して推敲する**（黙読では気づかない読みにくさ・リズムの悪さを検出できる）

## 信頼と説得の設計

### 特徴→ベネフィット変換

商品・サービスの特徴（スペック）をそのまま並べても読者には響かない。読者にとっての意味（ベネフィット）に変換する。

| 変換 | 内容 |
|---|---|
| **機能的ベネフィット** | その特徴によって読者が得られる実利（時間短縮・費用削減・作業効率化など） |
| **感情的ベネフィット** | その特徴によって読者が得られる感情の変化（安心・自信・ステータス・不安の解消など） |

- [ ] 特徴を並べたら「だから読者にとって何が嬉しいのか」を必ず一段深掘りする
- [ ] 機能的・感情的の両方のベネフィットに触れると説得力が増す

### 説得でなく納得

- [ ] 人は**感情で意思決定し、後から論理で正当化する**傾向がある（感情先行・論理後付け）ことを前提に構成する
- [ ] 感情に訴える具体的なエピソード・情景描写を根拠の前段に置く
- [ ] **損失回避**（得ることより失うことを避けたい心理）を意識した訴求を適所に使う（例:「これをしないと機会損失になる」）。ただし過度な不安煽りは信頼を損なうため節度を持つ
- [ ] 「説得しよう」とする文章は身構えられる。読者が自分で「納得した」と思える情報提示を心がける

### 嫌われない文章のアンチパターン＋反論処理＋先回り自己開示

**アンチパターン（避けるべき書き方）**:
- [ ] 上から目線の断定口調（読者を見下す言い回し）
- [ ] 根拠のない誇張・煽り文句
- [ ] 自社に都合の良い情報だけを並べる一方的な主張
- [ ] 読者の疑問を無視して自社が言いたいことだけを書く
- [ ] 冗長で結論が見えない文章展開

**反論処理・先回り自己開示**:
- [ ] 読者が抱きそうな疑問・反論をあらかじめ記事内で取り上げ、答えを用意しておく
- [ ] 商品・サービスの弱点や不向きなケースも正直に開示する（不都合な情報を隠さない姿勢が信頼を生む）
- [ ] 先回りして自己開示することで、読者の「都合の良いことしか書いていないのでは」という警戒心を解く

### 執筆者・監修者の記名

- [ ] 執筆者名・肩書きを**記事冒頭**に明記する（末尾のみの記名より離脱率が下がる傾向があるとされる）
- [ ] 専門家による監修がある場合は監修者名も明記する
- [ ] 著者プロフィールページへのリンクを設置する（`FOUNDATIONS-RANKING-EEAT.md` のExpertise/Authoritativeness施策と対応するE-E-A-T実装粒度）

### MSP（個の価値観・信念での差別化）

- [ ] 情報の正確さだけでなく、**書き手個人の価値観・信念・スタンス**を文章に反映させる
- [ ] 「なぜ自分/自社がこれを語るのか」という立場を明示すると、同じ情報でも他記事との差別化になる
- [ ] 個の権威性は、感情的な共感や継続的なファン化にもつながる

## 事例・お客様の声のコンテンツ化

一次情報（経験）を最も具体的な形で資産化するのが事例・お客様の声コンテンツ。

- [ ] 体験談・失敗談・現場事例は、記事内で信頼を補強したいタイミングに配置する
- [ ] 導入事例・お客様の声ページは以下の要素で構成する:
  - [ ] 利用前後の変化（Before/After）
  - [ ] 定量的な数値（削減時間・向上率など）
  - [ ] 利用者の属性（業種・規模・役職等、読者が自分事化できる情報）
  - [ ] 実際の写真・利用シーン
- [ ] 事例は記事本文への引用元としてだけでなく、独立した資産ページとしても構築し、記事から相互リンクする

## タイトル設計（クリックされる入口）

### CTR向上フレーズの型

タイトルは「Not Open」を突破する最初の関門。以下の型を組み合わせて設計する。

- [ ] **数字**を入れる（具体性・信頼性が増す。例:「5つの方法」「3ステップで」）
- [ ] **簡易性**を示す（「誰でも」「すぐできる」等、ハードルの低さを伝える）
- [ ] **権威**を示す（「専門家が解説」「〇〇監修」等）
- [ ] **感情**に訴える語を入れる（読者の悩み・欲求に直接刺さる言葉）
- [ ] **隅付き括弧【】**等で視覚的に目立たせる（多用しすぎるとスパム的に映るため節度を持つ）

### タイトル運用の勘所

- [ ] 重要なキーワードはできるだけ**前方に配置**する
- [ ] タイトルタグとH1の整合を取る（完全一致は必須ではない。詳細は `CONTENT-KEYWORDS.md` のタイトルタグ節）
- [ ] 競合タイトルの表現をそのまま模倣（パクリ）しない。自社独自の切り口を反映する
- [ ] タイトルは一度で決め切れるものではなく、**公開後の実CTRを見ながら試行錯誤・改善する前提**で運用する
- [ ] 文字数は絶対的な字数より**表示ピクセル幅が基準**（半角・全角・記号で表示幅が変わる）。字数はあくまで目安として捉え、`CONTENT-KEYWORDS.md` のタイトルタグ最適化チェックリスト（50〜60文字目安）と整合させる

## FAQの中身の書き方

FAQをページに実装する技術的な手順（構造化データ・リッチリザルト対応）は `TECHNICAL-SEO.md` と `AI-SEARCH-GEO.md` を参照。ここではFAQの**中身**の書き方を扱う。

- [ ] 質問文は検索クエリの体裁ではなく、**読者が実際に話しかけるような自然言語**で書く
- [ ] 回答は**150〜300字程度**を目安に簡潔にまとめる（answer-firstで結論から書く）
- [ ] 初出の専門用語には簡単な説明を添える（読者の前提知識に依存しない）
- [ ] 検索トレンド・製品情報の変化に合わせて**月次程度で内容を見直す**（陳腐化した回答を放置しない）

## 文章以外の制作要素

SEO記事は文章だけでなく、マルチモーダルな要素の組み合わせで完成度が決まる。

- [ ] **図解**: 概念・手順・比較を視覚化し、文章だけでは伝わりにくい情報を補う。記事導入部への1枚設置で全体の理解を助ける
- [ ] **表・グラフ**: 数値比較・スペック比較は表形式にすると可読性が上がる
- [ ] **配色**: ブランド・トーンに合わせた一貫した配色ルールを持つ
- [ ] **余白**: 詰め込みすぎない余白設計で圧迫感を減らす
- [ ] **語り手設定**: 誰の視点・声で語るコンテンツかを一貫させる（記名・MSPとも接続する）

**図解制作の注意点**:
- [ ] 他サイトの図解をそのまま流用しない（盗用対策として自社ロゴ・自社デザインを入れる）
- [ ] 「SEO目的だけ」の中身の薄い図解（内容が伴わない挿絵的な図）は避ける。理解を助ける実質的な情報を持たせる

## メディア別の文章最適解

SEO記事・広告・SNS投稿は、同じ情報を伝える場合でも構成・語彙が異なる。制作の担当スキルもそれぞれ異なる。

| メディア | 文章の特徴 | 担当 |
|---|---|---|
| **SEO記事** | 長文・網羅的・検索意図駆動・一次情報重視・answer-first | 本ファイル（`CONTENT-WRITING.md`） |
| **広告コピー・見出し** | 短文・瞬間的なインパクト・心理トリガー重視 | `studio:creating-content` |
| **SNS投稿** | プラットフォームごとの文化・カジュアルな語彙・即時的な反応設計 | `studio:creating-content` |

- [ ] SEO記事で使う説得技法（PASONA等）をそのままSNS投稿に転用しない。媒体ごとに文章の型を作り直す
