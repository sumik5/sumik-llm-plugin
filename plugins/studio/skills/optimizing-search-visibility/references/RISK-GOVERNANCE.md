# リスクとガバナンス リファレンス（R-7）

AI生成コンテンツを活用したSEO施策における主要リスクと、Googleポリシー・法規制への準拠を体系的に整理する。品質基盤（E-E-A-T・ランキング要因）は `FOUNDATIONS-RANKING-EEAT.md`、AI活用ワークフロー全般は `GENAI-WORKFLOWS.md` を参照。

---

## 1. Googleポリシーの核心：制作方法でなく品質

### 1-1. 基本原則

GoogleはAI生成コンテンツの利用を**一律に禁止していない**。公式の立場は一貫して「どのように作られたかではなく、コンテンツの品質と有用性で評価する」というものだ。

> 「AIや自動化の適切な利用は、コンテンツを人々に届けることを主目的とするかぎり、Googleのガイドラインに反しない。問題となるのは、主にランキング操作を目的とした利用である。」
> （出典: https://developers.google.com/search/blog/2023/02/google-search-and-ai-content 、 https://developers.google.com/search/docs/fundamentals/using-gen-ai-content）

この「**people-first**」の原則はAI生成コンテンツの文脈でも変わらない。重要なのは制作プロセスではなく、ユーザーへの実質的価値だ。

### 1-2. スパムポリシーとコアアップデート

2024年3月、Googleはコアアップデートと同時に3つの新スパムポリシーを導入した（出典: https://developers.google.com/search/blog/2024/03/core-update-spam-policies）。

| ポリシー | 内容 |
|---------|------|
| **Scaled content abuse** | ランキング操作目的の大規模ページ量産（AI/人間問わず） |
| **Expired domain abuse** | 期限切れドメインを悪用した低品質コンテンツの流通 |
| **Site reputation abuse** | 第三者による寄生コンテンツ（パラサイトSEO） |

このアップデートにより、低品質・非オリジナルコンテンツの検索表示が**約45%削減**されたとGoogleが報告している。

また2025年1月23日改訂の品質評価者ガイドラインには以下の記述がある（出典: https://searchengineland.com/google-quality-raters-content-ai-generated-454161）:

> *"The use of Generative AI tools alone does not determine the level of effort or Page Quality rating."*

つまり、**生成AIを使ったこと自体**はページ品質に影響しない。人間レビューなし・独自価値なしの純AI出力は "Lowest" 評価になるが、それは「AIを使ったから」ではなく「品質が低いから」だ。

### 1-3. Scaled content abuse の定義（2024年3月導入）

公式定義（原文・出典: https://developers.google.com/search/docs/essentials/spam-policies）:

> *"Scaled content abuse is when many pages are generated for the primary purpose of manipulating search rankings and not helping users."*

重要な特徴が2つある。

**手法非依存の執行**: 定義には「Using generative AI tools or other similar tools」と「automation, human efforts, or some combination」が明示されている。AIツールを使わない手書きの薄コンテンツ量産でも適用される。

**意図と結果による判断**: 「primary purpose of manipulating search rankings」＝ランキング操作が**主目的**かどうかが判断基準。ユーザーへの価値追加が主目的であれば、AI補助で大量のページを作成すること自体は問題にならない。

### 1-4. Human curated（人間による編集監督）

Google担当者は「human created（人間が作った）」ではなく**「human curated（人間が監督した）」**が正確な表現だと述べている（出典: https://www.searchenginejournal.com/google-says-ai-generated-content-should-be-human-reviewed/553486/）。

「human curated」の実質要件:
- 編集者が内容を確認し、正確性を検証している
- 監督は**本物**（形式的なラベル貼りではない）かつ**社内で実施**されている
- ページに「human reviewed」ラベルを表示することは信頼できるランキングシグナルにはならない

#### AIコンテンツ監査を実施すべき4トリガー

人間レビューとオフブランド検知をワークフローに組み込んでいても、それとは別に**AIコンテンツ監査**を定期・随時で実施する。監査ではサンプルコンテンツをベンチマークとし、QAチェックリスト（事実正確性・ブランドボイス・独自価値・開示の有無）を参照する。発火タイミングは次の4点。

| トリガー | タイミング |
|---------|----------|
| **初回トレーニング時** | 生成AIをチームに初導入した直後（運用が定着する前のベースライン確認） |
| **新ローンチ時** | 新製品・新サービス・新ブランドラインの立ち上げ時 |
| **定期** | 通常は四半期ごと（出力品質のドリフトを継続検知） |
| **アドホック** | 重大エラーやオフブランド出力が観測された時の臨時監査 |

### ✅ チェックリスト: Googleポリシー準拠

- [ ] AIをランキング操作目的でなくユーザー価値追加に使っている（people-first）
- [ ] 各ページに独自価値（一次体験・独自研究・専門家の知見）がある
- [ ] 全AI出力に**本物の社内編集監督**（human curated）を通している
- [ ] 事実・統計・引用を人間がファクトチェックしている（特にYMYL）
- [ ] E-E-A-Tの"Experience"（一次体験）を示す要素を盛り込んでいる
- [ ] site reputation abuse（第三者コンテンツの寄生）を行っていない

## 2. 主要リスクと対策

### 2-1. 低品質・大規模薄コンテンツ

Scaled content abuseは意図と結果で判断され、**手書きの薄コンテンツも同等に処罰対象となる**（出典: https://developers.google.com/search/docs/fundamentals/using-gen-ai-content）。

リスクの兆候:
- 日50本超の公開ペース（人間チームは通常週10〜15本程度とされる）
- 複数の薄いページが同一トピックを部分的にカバーしている
- 一次体験・独自研究・専門的洞察が欠如している

対策チェックリスト:
- [ ] 公開ペースを人間チームの2〜4倍以内に抑制する（40〜100倍にしない）
- [ ] 薄いページを包括ガイドに統合し301リダイレクトで集約する
- [ ] ゼロ価値ページは「改善」でなく削除を選ぶ
- [ ] SME（Subject Matter Expert）がアウトラインを作成 → AIがドラフト → 専門家がレビューの順序を守る
- [ ] 一次研究・実験・体験に基づく要素を含める

### 2-2. 公開LLMへの禁止入力（Prohibited Inputs）

公開された生成AIツール（チャットボット・無料プラン等）に絶対入力してはならないデータ分類がある。入力データは学習や他ユーザー応答に流用され得るため、漏えいすると取り返しがつかない。ある業界で報じられた事例では、大手メーカーの従業員が機密ソースコードを公開チャットボットにアップロードし、内部情報が露出した。これら7分類は**カスタムGPT/エンタープライズ契約（データ非学習・隔離保証あり）の設定下でのみ**、承認済みユースケースとして極めて慎重に扱う。

| 分類 | 具体例 |
|------|--------|
| **第三者の著作権/IP** | ライセンスコンテンツ・他社ロゴ・有償素材 |
| **機密ビジネス情報** | 営業秘密・未公開財務データ・原価 |
| **内部戦略文書** | SWOT分析・買収メモ・経営会議資料 |
| **認証データ** | ユーザー名・パスワード・APIキー・トークン |
| **パートナー/顧客データ** | 契約書・アカウント詳細・顧客との会話ログ |
| **PII（個人を特定できる情報）** | 氏名・メールアドレス・政府発行ID番号 |
| **機微な個人データ** | 健康記録・クレジットカード番号・生体情報 |

対策チェックリスト:
- [ ] 公開LLMへの禁止入力7分類を社内ポリシーに明文化し、全社員に周知する
- [ ] 機密データを扱う用途はエンタープライズ契約（データ非学習・隔離保証）下のツールに限定する
- [ ] プロンプトに貼り付ける前に「これは公開されても問題ないか」を一律に自問するルールを徹底する

### 2-3. 公開スピードの罠（量産の品質比率管理）

AIによりコンテンツ量産が容易になると、チームが自然と量を追い始める。これが品質監督の劣化を引き起こす。さらに**公開速度そのものが信頼シグナルとして評価される**側面がある。週数本のペースから急に週数百本へ跳ね上げると、検索側に「低品質を大量投下している」シグナルを送り、相対評価で順位を落としうる。

典型的な失敗パターン:
1. AIで大量生成 → レビューが形骸化 → 品質基準が下がる
2. 速度指標（本/月）で評価 → 品質指標（コンバージョン・滞在時間）が後回し
3. 公開数が増えるほどレビュアーの集中力が低下する（automation bias の一形態）

#### 量産を「品質比率（quality ratio）」で管理する

量産判断は「何本出すか」でなく**低品質コンテンツの混入比率**で捉える。ある観測例では、低品質が50本に1本（50:1）に抑えられていた運用が、レビュー無しの量産に切り替えたことで60本中30本がほぼ低品質（≒半数）という比率へ悪化し、サイト全体の順位を毀損した。AI生成ノイズの多い領域では、検索側が長期に高品質シグナルを蓄積した老舗ブランドへ信頼を回帰させる調整を行ったとされる観測例もある。

運用則:
- **一貫してゆっくりランプアップ**する（量を主目的にせず、品質比率を一定以下に保ったまま漸増）
- 専任のレビュー体制が無いなら、数百本/週級の量産は行わない
- 量（本/月）でなく品質指標（オーガニック流入・エンゲージメント率）でチームを評価する
- 公開承認ゲートを必須化し、スキップできない仕組みにする

### 2-4. ハルシネーション・事実誤り

生成AIは存在しない情報を自信満々に出力する。ハルシネートされたURL・統計・機能説明はブランド信頼を損ない、誤情報として拡散する可能性がある（出典: https://infomineo.com/artificial-intelligence/stop-ai-hallucinations-detection-prevention-verification-guide-2025/）。

SEO上の追加リスク: AI検索エンジン（ChatGPT、Perplexity等）はWebコンテンツをRAGで参照する。誤情報を含むページが引用されると、AI回答を通じて誤情報がさらに拡散する。

対策チェックリスト:
- [ ] 数値・統計・引用・固有名は必ず人間が一次ソースで検証する
- [ ] 新規ローンチ前に「ゼロ知識テスト」（存在しない機能をAIが捏造しないか確認）を実施する
- [ ] 四半期ごとに複数のAIモデルへ「自社について」質問し、誤情報が流通していないか監査する
- [ ] RAGによるリアルタイム検索でグラウンディングし、AI推測値を排除する
- [ ] Organization/Person/Productスキーマ + `sameAs` リンクで自社エンティティを明確化する

### 2-5. 著作権

米国著作権局の公式見解（出典: https://www.copyright.gov/ai/）:

- **AIのみで作成された作品は保護対象外**。プロンプト作成や軽微な編集では保護適格性を満たさない
- 保護が及ぶのは**分離可能な人間の創作的寄与**（選択・配列・編集など）のみ
- 2025年5月の見解: 著作物で学習し原作と「競合する表現的コンテンツ」を生成するのはフェアユースの範囲を超え得る

実務上の含意:
- 純AI出力への著作権保護を事業戦略の前提にしてはならない
- 競合が自社のAI生成コンテンツをコピーしても法的保護が限定的になる可能性がある

対策チェックリスト:
- [ ] 実質的な人間の創作的関与（編集・選択・構成変更）を文書化する
- [ ] 純AI出力に著作権保護を頼らない事業設計にする
- [ ] AIツールのベンダーが提供する学習データ来歴・補償条項を精査する
- [ ] 自社コンテンツの無断スクレイピング対策（robots.txt・利用規約）を整備する

> **注記**: 上記は法的助言ではなく、公開されている一般情報に基づく整理である。個別の状況については法律専門家に相談すること。

#### 人間制作を死守すべきハイステークス・コンテンツ一覧

米国では「AI単独生成物は著作権保護不可（人間が十分な表現要素を決定した場合のみ保護）」であり、**保護したいアセットは人間が制作・refineする必要がある**。著作権の問題だけでなく、ブランド評判・真正性・法的リスクの観点からも、以下のカテゴリは100% AI生成を避け人間が主導して制作する。これは `GENAI-WORKFLOWS.md` の「AI-first vs AI-assisted」二分法における **AI-assisted（人間をループ内に保つ）を選ぶべき領域そのもの**であり、両輪の関係にある。

**人間制作必須のマーケティング資産**:
- ソートリーダーシップ・署名記事
- 主要ページ（FAQ・About Us・幹部bio）
- 高露出の商品/キャンペーンLP
- 重要な購入ガイド・設置/how-toコンテンツ
- ケーススタディ・顧客の声
- スポンサー/パートナー/共同ブランドコンテンツ
- ライフサイクルメール（welcome/nurture/retention）
- 戦略的・機微なSNSキャンペーン
- ブランド広告のクリエイティブ・コピー
- プレスキット・メディアリリース

**人間制作必須の戦略的ブランドコミュニケーション**:
- ミッション/ビジョン/バリュー
- ブランドメッセージ・ポジショニング
- クライシス対応計画
- 経営コミュニケーション
- ESG/CSRレポート・メッセージ
- 製品/ローンチメッセージ

運用手順:
- [ ] 各部門リーダーが「人間制作必須」リストを自部門ぶん作成し提出する
- [ ] 提出物をマスター文書に統合し、全社で共有・更新する
- [ ] マスター文書を公開承認ゲートのチェック項目に組み込む（該当カテゴリは純AI出力での公開を技術的に止める）

### 2-6. 盗用・重複コンテンツ

AIの言い換えは**意味的重複**を生み、単純な文字列一致ツールでは検出しにくい。多くのチームで問題になるのは、外部サイトへの盗用よりも**自社内のトピック重複**だ（出典: https://searchengineland.com/guide/detect-ai-content-prevent-plagiarism）。

対策チェックリスト:
- [ ] 意味的（非文字列）検出を使うオリジナリティAPIをCMSに統合し、公開前に全ドラフトをスキャンする
- [ ] コンテンツ在庫でトピック・キーワードをタグ付けし、新規発注前に網羅状況をチェックする
- [ ] AIドラフト → 人間編集 → オリジナリティ検証のハイブリッドフローを標準化する

#### 外注ライターのオリジナリティ担保（プロセス計測）

外注ライターが未編集のAI生成物を自作と偽って納品するのを防ぐには、成果物のチェックだけでなく**執筆プロセスそのものを計測**する。前提として、AI検出ツールは誤検出が多い（公的文書ですらAI判定されることがある）ため**方向性の参考に留め、人手で裏取りする**。

| 手段 | 内容 |
|------|------|
| **公開前の二重スキャン** | AI検出＋盗用チェックを公開前に必須化。盗用の一次確認は**本文の部分文をそのまま検索エンジンに貼り**、大ブロックの一致が出ないかを見る |
| **ノーペーストCMS** | 執筆をCMS等のツール内で行わせ、外部からのコピー&ペーストを技術的に禁止する |
| **キーストローク生体認証** | タイピング速度・打鍵ダイナミクスを指紋化し、実際に人間がその場で書いているかを検証する |

著作権・盗用回避の実務則:
- [ ] 特定の著者・人物の文体やトーンを真似させない
- [ ] 単一ソースのみでの生成を禁止し、**複数ソースの利用＋ソース一覧の出力**を必須化して脚注/リンクで帰属する

### 2-7. 自動化の罠・自動化バイアス

稀なエラーが続くとレビュアーの警戒心が時間とともに低下する（**automation bias**）。「AIが出したから大丈夫だろう」という思い込みが監督の劣化につながる（出典: https://www.3dissue.com/human-in-the-loop-how-editorial-teams-safely-scale-with-ai-in-2025/）。

症状と対策:

| 症状 | 対策 |
|-----|------|
| レビュアーが内容を流し読みするようになった | レビュアーを受動的承認者でなく「批判的挑戦者」として位置づけるワークフロー設計 |
| AI出力をそのまま公開する割合が増えた | 完全自動公開を技術的に不可能にする（承認ゲートを必須化） |
| ファクトチェックが形骸化している | ファクトチェック項目を編集ゲートの明示ステップに組み込み、スキップ不可にする |
| 品質よりも量の指標で評価している | エンゲージメント・コンバージョン等の品質指標に評価基準を移行する |

対策の原則: **事実とブランドボイスは人間が所有する。AIが担うのは形式・構造・量産の部分のみ。**

#### 自動化の「set & forget」ドリフト

SEO業務を自動化（パイプライン・エージェント・定期スクリプト）した場合、「設定して放置」はできない。AIは更新が速く、環境変化で出力が黙って劣化（ドリフト）する。次のドリフト源を継続監視する。

| ドリフト源 | 起きること | 対策 |
|-----------|----------|------|
| **LLMバージョン更新** | 出力内容・トーンが変わる | 更新後は出力を必ず再レビューし、プロンプト/スクリプトを追従更新する |
| **学習データソースの変更** | 検索側が新たな掲示板系コンテンツ等を学習に取り込み、生成トーンが自社ポリシー/プロダクトボイスから乖離する | 定期的に出力サンプルをブランドボイス基準で点検する |
| **競合分析の自動化** | 以前見ていなかった新ドメインを誤って拾う／競合サイトの変更を反映しない | 解析ロジックの出力を定期的に人手レビューする |
| **単一ギャップ分析での大量生成** | AI回答が画面を占有しユーザーがスクロールしない種別のクエリを狙ってしまい、流入もCVも出ずROIマイナス（APIコストだけが残る） | 計測（流入・CTAクリック）で当たり外れを検知し、撤退判断する |
| **ロジックエラーでのAPI暴走** | 想定外のリクエスト増（例: 5→50リクエスト）で第三者API課金が膨張する | API利用量・課金アラートを監視する |

### 2-8. バイアス

限られたデータで学習したモデルは特定の地域・集団・視点を系統的に不利にすることがある。編集者が明示的に監視しなければ、アルゴリズム的バイアスが公開コンテンツに永続化する（出典: https://www.3dissue.com/human-in-the-loop-how-editorial-teams-safely-scale-with-ai-in-2025/）。

対策チェックリスト:
- [ ] 編集者がAI出力をバイアス・不公平なフレーミングで明示的にレビューする
- [ ] プロンプトと人間補強でソース・視点・例示を多様化する
- [ ] バイアスチェックをファクトチェックと並ぶ編集ゲートの明示ステップに組み込む
- [ ] 特にYMYL（健康・金融・法律）領域では多様な専門家によるレビューを義務化する

## 3. コンプライアンス・規制

### 3-1. AI生成コンテンツの開示義務（FTC等）

米国連邦取引委員会（FTC）は「実質的 vs 表面的」テストを適用する（出典: https://www.ftc.gov/ai 、 https://thestacc.com/blog/ftc-ai-disclosure-rules-2026/）。

**開示が必要なケース（実質的影響あり）**:
- AI生成の画像・動画・音声
- AIが生成した広告コピーや推薦文
- AI合成パフォーマー・アバター

**開示不要のケース（実質的影響なし）**:
- 文法・スペルチェック
- コンテンツフィルタリング
- スケジューリング・分析ツール

**「二重開示」の要件**: 商業的関係とAI関与は**別々に開示**しなければならない。一つのラベルで両方をカバーすることはできない。

**ペナルティ**: 2026年時点の最大民事罰は1違反あたり$53,088、各非準拠投稿が個別にカウントされる。州法による追加義務もある（NY GBL §396-B、California AB 3211 等）。

### 3-2. 国際的な開示規制と著作権の最新動向

FTC以外にも、AIコンテンツの開示・著作権に関する規制が各国で整いつつある。施策設計時は以下の最新事実を踏まえる。

**EU AI Act（2024年導入・2025年施行）**:
- 生成AIプロバイダにはAI出力を「人工的に生成/操作されたもの」と検出可能にすることを、利用者（マーケター含む）には開示を要求する。
- **重要な例外**: 人間がレビュー・編集し**編集責任を持つ**AI生成コンテンツは**開示不要**。人間が編集責任を引き受けるワークフロー（human-in-the-loop）が、ここでも開示負担の軽減につながる。
- 適用範囲はEU内、およびEU内の人に影響を与える全組織（域外適用あり）。

**米国 NO FAKES Act（提案中）**:
- 個人の声・視覚的な肖像（likeness）の無断デジタル複製を規制する法案。AI合成パフォーマー・ディープフェイク広告のリスク管理に直結する。

**米国著作権局「Copyright and AI Report」Part 2（2025年1月）**:
- 生成AI出力は、**人間著者が十分な表現的要素を決定した場合のみ**著作権保護され得ると結論。**単なるプロンプト提供では不可**。
- AI支援での創作や、AI生成物を人間制作の作品へ内包することは保護を妨げない。
- Part 3（訓練データへの著作物使用）は本記述時点で未公開。

含意: **AI単独生成物は米国で著作権保護されない**ため、保護したいアセットは人間が制作・十分にrefineする（2-5「人間制作を死守すべきハイステークス・コンテンツ一覧」と連動）。

### 3-3. 開示の実装

対策チェックリスト:
- [ ] 使用AIツールを棚卸しし、各用途を「表面的/実質的」に分類する
- [ ] 実質的影響のある用途に二重開示を適用する（コンテンツ近接・動画は冒頭3〜5秒以内）
- [ ] プラットフォーム純正AIラベルを積極的に使用する
- [ ] クリエイター・インフルエンサー契約にAI開示義務を明記する
- [ ] 開示関連の文書記録を3年間保持する
- [ ] 日本市場では景品表示法・電気通信事業法上の適正広告基準も確認する

## 4. 社内AIガバナンス

開示や著作権といった対外規制（セクション3）に加え、AI活用を組織として統制する**社内ガバナンス**の枠組みが必要だ。SEO/コンテンツ部門は全社AI戦略の一部として運用されるべきで、部門単独の判断で公開LLMを使い始める状態はリスクを集積する。

### 4-1. AI使用ポリシーの3層構造

責任ある利用を担保するため、AI使用ポリシーを3層で整備する。各層は粒度が異なり、上位の枠組みを下位が具体化する関係にある。

| 層 | 対象 | カバー範囲 |
|----|------|----------|
| **Company（全社）** | 組織全体 | 責任ある利用の枠組み。バイアス・倫理・法的トピックを含み、AI委員会が所管する |
| **Department-specific（部門）** | チーム（例: マーケティング） | 全社枠組みを部門のニーズ・リスク・規制に合わせて具体化。承認ユースケース・リスク管理・承認プロセス・トレーニング・レポーティングを規定する |
| **Third-party（第三者）** | 代理店・パートナー・アフィリエイト・フリーランス | 契約に開示・準拠の文言を入れるだけでなく、**検証プロセス**で第三者の作業者全員がレビュー・理解したことを示す。第三者が重大エラーを犯した場合の法的サポートにもなる |

### 4-2. AI委員会（AI committee）

3層ポリシーの前提として、全社横断の**AI委員会**を組成する。

- **構成**: 法務・財務・人事・技術・マーケティング等の各部門の幹部で構成する（特定部門に閉じない）。
- **役割**: AI戦略・ポリシー・リスク管理を統括し、Companyポリシーを所管する。新ツール導入・新ユースケースの承認、バイアス/倫理/法的リスクの判断を担う。

運用チェックリスト:
- [ ] 全社のAI委員会を組成し、Companyポリシーの所管を明確にする
- [ ] SEO/コンテンツ部門のDepartment-specificポリシーで承認ユースケースと承認ゲートを定義する
- [ ] 外注契約にThird-partyポリシー文言＋検証プロセス（理解確認）を組み込む

### 4-3. 規制業界でのAIコンプライアンス（医療）

PHI（保護対象保健情報）など、規制対象データを扱う業界ではAI利用要件が一段と厳格になる。医療を例に要点を示す。

**HIPAA下でのAI利用要件**:
- PHIを患者の許諾なく露出しない（意図せぬ露出も含む）
- 使用・共有の前に de-identification（非識別化）を保証する
- 暗号化・セキュアなアクセス制御・監査ログを備える
- PHIを安全でない形で保存・送信しない

**最重要原則: 「準拠」と「準拠の主張」を峻別する**。ツールやモデルが「HIPAA準拠」と自称しても鵜呑みにせず、最終的な準拠責任は医療提供者側にある（後述のアクセシビリティ・オーバーレイと同じ構図）。

**医療画像生成は特に危険**: ある研究では生成された医療画像736枚のすべてに解剖学的ディテールの重大な欠陥があったと報告されている。医療画像はGenAIに委ねず、資格ある医療イラストレーターに依頼する。

## 5. AI生成コンテンツ検出の限界

### 5-1. 検出ツールの信頼性

**AIテキスト検出ツールは証拠としては信頼できない**。査読論文が複数これを文書化している。

- 誤検出率: 15〜45%（出典: https://link.springer.com/article/10.1007/s40979-026-00213-1）
- 主要ツールの総合精度: 約61〜69%（出典: https://pmc.ncbi.nlm.nih.gov/articles/PMC12331776/）
- 人間とAIの混成テキストでの精度: ほぼ0%に低下（出典: https://www.mdpi.com/2078-2489/16/10/905）

特に問題となる誤分類:
- 非ネイティブ英語話者の文章がAI生成と誤判定される傾向
- 洗練された人間の文章がAI生成と判定されるケース

### 5-2. 実務上の含意

検出スコアは「一つのシグナル」であり「証拠」ではない。Googleも検出スコアをランキングシグナルとして使用しているとは公式に述べていない。

**検出ツールに頼らない品質管理を設計すること**が実務上の正解だ。コンテンツの評価基準は「AIが書いたか否か」ではなく「ユーザーに価値を提供しているか」に置く。

対策チェックリスト:
- [ ] AI検出スコアを「証拠」として懲戒・削除の根拠にしない
- [ ] 品質評価を検出スコアでなく実質的な内容（正確性・独自性・有用性）で行う
- [ ] 学術・採用・法的文脈では検出ツールへの過信が特に危険であることを組織内で周知する

### 5-3. AIアクセシビリティ・オーバーレイ神話

AI検出ツールと同様、「AIツールの準拠主張を鵜呑みにすると逆にリスクになる」典型例が**アクセシビリティ・オーバーレイ**だ。サイトに**コードスニペットを差し込むだけでADA/WCAG準拠になる**と謳うAIオーバーレイは、実際には機能せず、かえって法的・評判上のリスクを増やす。

リスクの実態（出典種別を付して示す。個別企業は特定しない）:
- **ある業界調査による**と、ある年の州/連邦アクセシビリティ訴訟の**約30%**がオーバーレイを使用しているサイトに対するものだった。
- 実務者を対象とした調査では、**約67%**がオーバーレイを「全く/あまり効果的でない」と評価した。
- **ある事例では**、オーバーレイ提供事業者が「自社AIがサイトを完全に準拠させる」と謳ったことが欺瞞的取引慣行と判断され、規制当局から罰金命令を受け、集団訴訟にも発展したと報じられている。

**準拠を保証できるソフトウェアは存在しない**。AIは課題（issue）の特定・修正のショートカット・保守には使ってよいが、最終的には**人間が監視・頻繁にテスト**する必要がある（「準拠」と「準拠の主張」を峻別する、4-3医療と同じ構図）。

法規制の整理（これらは事実として記載できる）:

| 枠組み | 適用対象 |
|--------|---------|
| **ADA** | 私企業・公的機関の双方（裁判所はWebを「公共accommodationの場」と解釈する判断を示してきた） |
| **Section 508** | 連邦機関およびそのベンダー |
| **WCAG 2.0 AA** | 法そのものではないが、Section 508がベンチマークとして採用し、ADA執行でもガイドラインとして参照される事実上の基準 |

対策チェックリスト:
- [ ] アクセシビリティ・オーバーレイの「ワンタグで準拠」主張を信用しない
- [ ] WCAG 2.0 AAをベンチマークに、人間による監視・継続テストで準拠を担保する
- [ ] AIは課題特定・修正補助・保守に限定して用いる

## 6. ペナルティ・順位喪失への対応

### 6-1. コアアップデートによる順位下落

コアアップデートによる下落は**個別ペナルティではない**。「他のコンテンツと比較して相対的に有用性が低い」と評価されたことを意味する。したがって、急な対策（大量削除・大幅改変）よりも継続的な品質底上げが有効だ。

対応フロー:
1. **現状分析**: Google Search Consoleでクリック数・表示回数の変化を確認。アップデートのロールアウト日（約2週間）完了後まで待つ
2. **影響範囲の特定**: 下落したページのカテゴリ・共通点を把握する
3. **原因仮説の立案**: E-E-A-T不足・検索意図不一致・薄コンテンツ・技術的問題のどれか
4. **改善実施**: コンテンツの深化（専門家知見・一次データの追加）、構造改善、技術的修正
5. **次回更新での回復を狙う**: 通常、次のコアアップデートまで大幅回復は期待しにくい

### 6-2. スパムポリシー違反への対応

Scaled content abuse など明示的なスパムポリシー違反は**手動対応（Manual Action）**の対象になりうる。

手動対応への対処:
- [ ] Google Search Consoleの「手動による対策」レポートを確認する
- [ ] 問題ページを特定し、スパムポリシーに反するコンテンツを削除または大幅改善する
- [ ] 改善完了後、Search Consoleから「再審査リクエスト」を送信する
- [ ] 再審査には数週間〜数ヶ月かかることがある

### 6-3. 回復の基本原則

- 個別ページの急ぎ対応より、**サイト全体の品質水準**を引き上げることを優先する
- 「改善ではなく削除」の判断基準: 独自価値を追加できないページは改善に時間をかけるより削除して内部リンクジュースを集中させる
- 回復の確認指標: オーガニック流入だけでなく、ページの平均滞在時間・直帰率・CTRを複合的に追う
