# OOUI設計方法論 — 3ステップ設計プロセス詳細

Object-Oriented UI（OOUI）を実践するための設計ステップ詳細リファレンス。
SKILL.md の「OOUI設計3ステップ」概要に対応する実践ガイドとして使用する。

---

## 全体像

OOUI設計は次の3ステップで構成される。各ステップは順序通りに進めても、
行き来しながら進めても構わない。慣れれば途中の作業を頭の中だけで処理したり、
手戻りしながら反復的に洗練させることが自然になる。

```
ステップ1: オブジェクトの抽出
  → タスク文言から名詞を拾い、メインオブジェクト・プロパティ・アクションを確定する

ステップ2: ビューとナビゲーション
  → 各オブジェクトのコレクション/シングルビューを整理し、呼び出し関係を定義する

ステップ3: レイアウトパターンの適用
  → 実際の画面として、適切なパターンを選択して適用する
```

各ステップの成果物は互いに対応している。
モデルレイヤーで定義されたオブジェクトは、インタラクションレイヤーで複数のビューに表象され、
それらが組み合わさってプレゼンテーションレイヤーの画面になる。

---

## ステップ1: オブジェクトの抽出

### 1-1. 名詞を抽出する

要件・ユーザーストーリー・ユースケース等のタスク記述から**名詞**を列挙する。
機械的に抽出するのではなく「ユーザーの関心の対象となる概念は何か」を考えながら行う。

- **「もの」的オブジェクト**: 食べ物・書物・製品・ファイルなど
- **「こと」的オブジェクト**: 食事・予約・イベント・取引など

どちらもオブジェクトになりうる。日常生活でも人はあらゆる名詞に囲まれて行動している。

### 1-2. 名詞同士の関係を整理する

タスクごとに名詞同士を線でつなぎ、関連を可視化する。
UMLクラス図のエンティティ関連整理に近い作業。

### 1-3. 名詞を汎化して粒度を揃える

類似する名詞を共通概念にまとめ、粒度を統一する。

- 例：「部」「野球部」「サッカー部」→ 汎化して「部」に統合
- 抽象化しすぎると「もの」「こと」レベルになり設計の手掛かりを失う
- 上位概念に行き過ぎたと感じたら戻る

タスクに明示されていない名詞（例：「カテゴリー」）も、
必要に応じて概念として立ててよい。

### 1-4. 名詞同士をつなぎ、オブジェクトを整理する

タスクをまたいで同じ名詞を統合し、ネットワーク状に接続する。
この段階でタスク単位の視点からオブジェクト中心の構成に移行する。

### 1-5. メインオブジェクトを特定する

オブジェクトを「主要なもの（メインオブジェクト）」と「付随するもの（サブオブジェクト）」に分類する。

**メインオブジェクト選定の判断基準:**

| 判断基準 | メインオブジェクトらしい | サブ（プロパティ）らしい |
|---------|----------------------|----------------------|
| 関心の度合い | ユーザーが直接操作・参照したい | 他のオブジェクトの属性として存在 |
| 関連の数 | 多くのオブジェクトと接続されている | 限られたオブジェクトにのみ関連 |
| 独立した情報 | それ自体に関する情報が複数ある | 値・数値・短いテキスト程度 |
| 関連メインオブジェクト数 | 複数のメインオブジェクトと関連 | 単一オブジェクトの絞り込み条件程度 |

> **例外**: 「名前」「電話番号」は通常プロパティだが、
> 姓名判断アプリや電話番号検索アプリでは主役なのでメインオブジェクトになる。
> **アプリケーションの関心領域によって判定が変わる**点に注意する。

### 1-6. メインオブジェクト同士の多重性を確定する

関連する2つのメインオブジェクト間で、「多重になりうる側」に `*` を付ける。

- 例：「1つの組に複数の生徒」→ 生徒側に `*`
- この多重性は後のナビゲーション設計（ステップ2）で参照関係の方向性を決める根拠になる

### 1-7. プロパティとアクションを定義する

**プロパティ（サブオブジェクト）:**

- メインオブジェクトに付随するサブオブジェクトをプロパティとして紐付ける
- 関連し合うメインオブジェクト同士も互いのプロパティになる
- タスクに明示されていなくても、オブジェクトを識別するための「名称／氏名」は
  ほぼ全てのオブジェクトに必要

**アクション:**

- タスク文言の**動詞**を手掛かりに抽出する
- 「〜を確認する」はオブジェクトを表示するだけなのでアクションに含めない
- 編集可能なアプリでは「新規作成」「削除」が暗黙的に存在することが多い。明示的に補足する
- アクションはメインオブジェクトに紐付けるのが基本（プロパティへの紐付けも有り）

## ステップ2: ビューとナビゲーション

### 2-1. 基本ビュー形式

オブジェクトを表象するビューの基本形は2種類:

| ビュー種別 | 役割 | 対応する行動 |
|-----------|------|------------|
| **コレクションビュー** | 複数オブジェクトを集合として表示 | 眺める・一覧する |
| **シングルビュー** | 単一オブジェクトの詳細を表示 | よく見る・詳細確認 |

人間は「眺めて、気になるものに近寄る」という行動を日常的に繰り返している。
この普遍的なパターンがコレクション→シングルの遷移に対応する。

### 2-2. 各メインオブジェクトにビューを割り当てる

まず機械的に全メインオブジェクトに対してコレクションビューとシングルビューを用意する。
コレクション→シングルへの矢印でつなぎ、呼び出し関係を示す。

**省略判断（後工程で行う）:**

| 省略パターン | 条件 |
|------------|------|
| **シングルビューの省略** | 一覧表示だけで事足りるオブジェクト（例：チェックリスト項目） |
| **コレクションビューの省略** | アプリ内に1件しか存在しないオブジェクト（例：自分のアカウント設定） |

> 省略の最終判断はレイアウト面積との兼ね合いで決まることが多い。
> この段階では省略確定より「省略候補」として扱うのが安全。

### 2-3. ナビゲーション（呼び出し関係）を定義する

メインオブジェクト同士の多重性（ステップ1-6）を元に、ビュー間の参照関係を定義する。

**参照可能性のルール:**

| 参照元 | 多重性 | 参照先 |
|--------|--------|--------|
| オブジェクトAのシングルビュー | 関連先が複数（`*`） | オブジェクトBの**コレクション**ビュー |
| オブジェクトAのシングルビュー | 関連先が単数 | オブジェクトBの**シングル**ビュー |

- 全ての参照可能性をUIに実装する必要はない。実際の画面を想像しながら必要なものだけ選ぶ
- 参照可能性のない方向（例：生徒→組のコレクション）はつなげない

### 2-4. ルートナビゲーション項目を選定する

ルートナビゲーションはアプリケーションの最上位ナビゲーション。
**メインオブジェクトのうち特に重要なもの**を選ぶ。

**選定基準:**

- ユーザーがアプリを使う際に「思考の起点」となるオブジェクト
- 特定のオブジェクトからしか呼び出されないオブジェクトは優先度を下げる
- ルートナビゲーションを選択するとそのコレクションビューが表示される

**ラベリング原則（やってはいけない例）:**

```
❌ 「生徒管理」「生徒一覧」「生徒確認」「生徒情報」
✅ 「生徒」

❌ 「注文照会」「注文参照」「注文登録」
✅ 「注文」
```

末尾に「管理・一覧・確認・照会・参照・登録・情報・編集」をつけない。
オブジェクトの種類名のみを使う。動詞ではなく名詞。

## ステップ3: レイアウトパターンの適用

### 3-1. ルートナビゲーションの配置パターン

| プラットフォーム | 推奨配置 |
|---------------|---------|
| スマートフォン | 下部タブバー または 開閉式メニュー（ハンバーガー） |
| デスクトップ | 上部ナビゲーションバー または 左サイドバー |

- 横並びの場合は項目数を限定する（通常5項目以内）
- 項目数が多い場合は縦方向にスクロール可能なサイドバーが適する

### 3-2. ビューの配置パターン（ワンカラム vs マスターディテール）

**ドリルダウンの方向性**: コレクション→シングルは「上から下」または「左から右」

| パターン | 構成 | 適するケース |
|---------|------|------------|
| **① 画面分割展開（1オブジェクト）** | コレクション画面→シングル画面と個別遷移 | スマートフォン、ループするナビゲーション |
| **② ペイン分割（1オブジェクト）** | 1画面を左右2ペインに区切り同時表示 | デスクトップ、オブジェクト横断参照なし |
| **③ 画面分割展開（複数オブジェクト）** | ルートナビ付き、コレクション→シングルを個別遷移 | スマートフォン、複数オブジェクト間のループ |
| **④ ペイン分割（複数オブジェクト）** | ルートナビ付き、左右2ペイン同時表示 | デスクトップ、横断参照なし |

> **使い分けポイント:**
> オブジェクトA→B→Aのようなループが発生する場合は①③（画面分割展開）が有効。
> ループがなく単方向参照のみなら②④（ペイン分割）が使いやすい。

### 3-3. コレクションビューの表示形式

| パターン | 特徴 | 適するオブジェクト |
|---------|------|----------------|
| **1行1項目リスト** | 多数項目を省スペースで一覧 | 頻繁に増減する大量データ |
| **複数行リスト（固定高）** | 各項目に複数プロパティを表示 | 名前＋サブ情報をセットで確認したい |
| **複数行リスト（可変高）** | 本文量に応じて高さが変わる | SNS投稿・コメント・メモ等 |
| **サムネイルグリッド** | 画像・ビジュアルを強調 | 商品・写真・アルバム・部活 |
| **マッピング（カレンダー/地図）** | 日付や位置で2次元配置 | イベント・予定・店舗・スポット |

**選定の視点**: ユーザーがそのオブジェクトのどの属性に着目するか
- 文字情報 → リスト
- 外観 → サムネイル
- 位置 → 地図
- 時間 → カレンダー

複数の視点が必要な場合は表示形式の**切り替え機能**を用意する。

### 3-4. コレクションのフィルタリングパターン

| パターン | 概要 | 適するケース |
|---------|------|------------|
| **キーワード検索** | 自由入力テキストで絞り込み | 件数が多く任意条件で探したい |
| **グループ切り替え** | プリセット条件でグルーピング切り替え | ステータス・カテゴリ等で分類がある |
| **AND条件追加** | 選択式で条件を追加して絞り込む | 複数プロパティの組み合わせ検索 |
| **クエリビルダー** | AND/OR等の論理演算で複雑な条件を構成 | 上級ユーザー向け、デスクトップ専用 |
| **ソート** | プロパティ値基準の並び替え | 優先度・日付・名前順での確認 |

### 3-5. シングルビューの表示形式

シングルビューの「詳細情報」の解釈はアプリによって異なる:
- プロパティ群（通常の詳細情報）
- 関連オブジェクトの中身（例：フォルダ→ファイル一覧）

**他オブジェクトのコレクションを組み込む3パターン:**

| パターン | 概要 | 適するケース |
|---------|------|------------|
| **一部を先出し表示** | 関連コレクションの代表アイテムのみ表示、全件は別画面 | 関連数が多い可能性あり、省スペース優先 |
| **コレクション強調表示** | 関連コレクション全件を同一画面に表示 | 関連オブジェクトが主役に近い |
| **コレクションのみ表示** | シングルビュー自体の情報を省略し関連コレクションのみ | フォルダ=中身のようなケース |

### 3-6. アクションの表示形式（CRUDパターン）

**Create（作成）パターン:**

| パターン | 概要 |
|---------|------|
| ブランク | 即時に空のアイテムを作成 |
| パラメーター | モーダルで必要な初期値を入力してから作成 |
| プレースホルダー | リスト末尾の入力欄に直接入力して作成 |
| セーブアズ | 既存アイテムを複製して作成 |
| テンプレート | あらかじめ用意したテンプレートを選んで作成 |
| マスター | マスター（型）を選んでインスタンスを生成（マスター変更が全インスタンスに反映） |
| ガッツ | モード遷移なしにデフォルトスタイルで即時作成 |

**Delete（削除）パターン:**

| パターン | 概要 |
|---------|------|
| モーダルコンファーム | 削除前にモーダルで確認 |
| アンドゥアブル | 即時削除＋Undoまたはゴミ箱への移動 |
| モードレスコンファーム | モードレスで確認UIを出し、他操作でキャンセル |

**Update（更新）パターン:**

| パターン | 概要 |
|---------|------|
| モーダルエディット | 参照モード＋編集モードを分離。保存ボタンで確定 |
| モードレスエディット | 常時編集可能＋自動保存 |

## タスクの扱い方

### タスクとOOUIの関係

タスクとはユーザーが「やること（ユースケース・ユーザーゴール）」の総称。
OOUIにおけるタスクの重要な特性を理解する:

| 特性 | 説明 |
|------|------|
| **粒度が不定** | 「水を飲む」〜「生きる」まで様々な粒度で捉えられる |
| **増えていく** | オブジェクトの定め方次第で想定外のタスクが生まれる |
| **変化する** | ユーザーはアプリの特性に合わせてタスクを変化させる |
| **事後でしか確定しない** | 行動しながら考えるため、完了して初めて確定する |

**結論**: タスクのみを根拠にUI構造を設計すると、タスクの増減・変化に伴い設計が複雑化・破綻しやすい。
タスクをオブジェクト+アクションに分解してOOUI構造に統合するのが正しいアプローチ。

### タスクが反映される7つの場所

OOUIでは、タスクはUIの構造（ナビゲーション・ビュー分類）には反映されない。
代わりに以下の要素に反映される:

| 反映先 | 説明 | 例 |
|-------|------|-----|
| **コンテンツ** | ヘルプ・プロモーション・ガイドの情報構成 | 「〜するには」形式のヘルプ |
| **表示形式** | 特定プロパティを強調した別表示形式 | 「施設名」を大きく表示するビュー |
| **アクション** | オブジェクトへの操作バリエーション | 「共有する」「エクスポートする」 |
| **初期値** | プロパティのデフォルト値 | 今日の日付・現在地・前回選択値 |
| **テンプレート** | Create時のプリセット初期値セット | 「週次レポート用テンプレート」 |
| **フィルタリンググループ** | タスク文脈に合ったプリセット絞り込み | 「未完了」「発送済み」「完了」 |
| **表示順（ソート）** | タスク優先度に合わせたデフォルト順序 | 期限日昇順・重要度降順 |

### タスク的要素（ウィザード等）の位置づけ

タスクフローが複雑で「段階的に案内したい」場合、
ウィザード形式（ステップ制UI）をCreate/Updateアクションの一形態として実装することは許容される。
ただしウィザードはアクション内の「入力補助」であり、
ルートナビゲーションやビュー分類の基準にはしない。
