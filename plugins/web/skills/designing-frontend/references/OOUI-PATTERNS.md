# OOUI レイアウトパターンカタログ

オブジェクト指向UI設計における標準的なレイアウトパターンの参照集。
ルートナビゲーション・ビュー配置・コレクション表示・フィルタリング・シングルビュー・CRUDアクションのパターンを網羅する。

---

## 1. ルートナビゲーション配置パターン

ルートナビゲーションはアプリケーション最上位のナビゲーション。メインオブジェクトをそのまま並べる。

| 配置位置 | 向いているプラットフォーム | 特徴 |
|---------|------------------------|------|
| 下部 (Bottom Navigation) | スマートフォン | 親指で届きやすい。4〜5項目まで |
| 上部 (Tab Bar) | Web・デスクトップ | 視線の起点。横並びのため項目数を制限 |
| 左サイドバー | デスクトップ・タブレット | 縦スクロール可能。多項目に対応 |
| 開閉式ドロワー | スマートフォン・タブレット | 画面面積を節約。頻度の低い項目に向く |

### ルートナビゲーション設計の鉄則

| 鉄則 | 説明 |
|------|------|
| オブジェクト名を使う | 「注文管理」→「注文」、「ユーザー一覧」→「ユーザー」 |
| 動詞は使わない | 「検索する」「登録する」は項目名にしない |
| 接尾語を付けない | 〜管理・〜一覧・〜確認・〜照会・〜情報は冗長 |
| 選択→コレクション表示 | 項目選択でそのオブジェクトのコレクションビューを表示する |
| 思考の起点を選ぶ | ユーザーが最初に意識するオブジェクトを最上位に置く |

## 2. ビュー配置パターン

コレクションビューとシングルビューをどのように画面上に配置するかのパターン。
「ドリルダウンの方向性」として、上→下・左→右の順で詳細化する。

### 2-A. 単一メインオブジェクト（ルートナビゲーション不要）

| パターン | 説明 | 向いている状況 |
|---------|------|--------------|
| **① ワンカラム（画面遷移型）** | コレクション→シングルを別画面に展開 | スマートフォン。オブジェクト間でナビゲーションがループする場合 |
| **② マスターディテール（ペイン分割型）** | 1画面を左右ペインに分割し、左にコレクション・右にシングル | デスクトップ。オブジェクト間のループが発生しない場合 |

### 2-B. 複数メインオブジェクト（ルートナビゲーション必要）

| パターン | 説明 | 向いている状況 |
|---------|------|--------------|
| **③ ワンカラム＋ルートNav（画面遷移型）** | ルートNavから各オブジェクトのコレクションに遷移 | スマートフォン。オブジェクト間でナビゲーションがループする場合 |
| **④ マスターディテール＋ルートNav（ペイン分割型）** | ルートNavとコレクションとシングルを同時表示 | デスクトップ。オブジェクト間のループが発生しない場合 |

### ビュー配置パターン選択ガイド

| 条件 | 推奨パターン |
|------|------------|
| 画面面積が小さい（スマートフォン） | ①または③（ワンカラム遷移型） |
| オブジェクト間でナビゲーションがループする | ①または③ |
| 画面面積が大きく、ループなし | ②または④（ペイン分割型） |
| 3ペイン以上 | メインオブジェクト数×2ビュー必要。シングルビュー省略も考慮 |

## 3. コレクションビュー表示形式パターン

オブジェクトの一覧をどのように表示するかのパターン。オブジェクトのどの側面に着目するかで選ぶ。

| パターン | 特徴 | 向いているオブジェクト |
|---------|------|---------------------|
| **1項目1行リスト** | シンプルな一行表示。多項目の一覧に最適 | テキスト情報が主体のオブジェクト（名前・タイトル等） |
| **1項目複数行リスト** | 1項目内に複数行の情報を表示 | 説明・補足情報も一覧で確認したいオブジェクト |
| **可変高リスト** | 項目ごとに高さが可変（SNS投稿形式） | 内容量が不定のオブジェクト（投稿・コメント等） |
| **サムネールグリッド** | 画像・外観を強調したタイル表示 | 視覚的外観が重要なオブジェクト（写真・商品・部屋等） |
| **マッピング** | カレンダー・地図など2次元上に配置 | 日付・位置情報を持つオブジェクト（イベント・店舗等） |

### 表示形式の選択基準

| ユーザーの着目点 | 推奨表示形式 |
|---------------|------------|
| テキスト情報 | リスト形式 |
| 視覚的外観・画像 | サムネールグリッド |
| 日付・時系列 | マッピング（カレンダー） |
| 位置情報 | マッピング（地図） |
| 複数の視点が必要 | 表示形式の切り替えUI |

## 4. コレクションフィルタリングパターン

コレクションビューの絞り込み機能のパターン。オブジェクトの性質とユーザーのタスクに応じて選択する。

| パターン | 説明 | 向いている状況 |
|---------|------|--------------|
| **キーワード検索** | 自由入力テキストで絞り込み | 項目数が多く、任意の文字列で検索したい場合 |
| **グループ（タブ/セグメント）** | プロパティ値でグループ化し選択式に切り替え | カテゴリ・タグ・ステータス等で事前に分類できる場合 |
| **AND条件フィルター** | 複数条件を選択式に追加して絞り込み | 複数軸（期間・種別・状態など）を組み合わせる場合 |
| **クエリビルダー** | AND/OR条件を組み合わせてクエリ構築 | 高度な検索が必要なデスクトップ向け業務システム |
| **ソート** | プロパティ値を基準に並び替え | 優先度・日付・名前順など特定の並び順で一覧確認する場合 |

> **グループパターンの重要ルール**: 全件を表示する「すべて」グループを必ず用意する。

### タスク連動フィルタリンググループの設計

タスクを意識したフィルタリングプリセットを用意することで特定業務を行いやすくする。

| 例：「注文」コレクション | グループ |
|----------------------|--------|
| デフォルト表示 | すべて |
| 業務上の重要ステータス | 未完了 / 発送済み / 完了 |

## 5. シングルビュー表示形式パターン

シングルビューから他のオブジェクトのコレクションを呼び出す際のパターン。

| パターン | 説明 | 向いている状況 |
|---------|------|--------------|
| **コレクションの一部を表示** | 関連オブジェクトの代表的なアイテムを一部だけシングルビューに埋め込む | 関連オブジェクトが多い・他の情報と共存が必要な場合 |
| **コレクションを強調表示** | 関連オブジェクトの全アイテムをシングルビューに統合して表示 | 関連オブジェクトが主役・アイテム数が少ない場合 |
| **コレクションのみ表示（シングルビュー省略）** | シングルビューを省略して関連コレクションを直接表示 | フォルダ→ファイルのように、中身そのものが「詳細」の場合 |

### シングルビュー設計の原則

シングルビューには「オブジェクトについての詳細情報」を表示するが、「詳細」の定義はコンテキストによる。

| 解釈 | 例 |
|-----|-----|
| オブジェクト自体の属性情報 | ユーザーの氏名・メールアドレス・登録日 |
| オブジェクトの中身（コンテナ） | フォルダの中のファイル一覧 |
| オブジェクトに紐づく別オブジェクト | アルバムの収録曲リスト |

## 6. CRUDアクションパターン

### 6-A. Create（作成）パターン

| パターン | 説明 | 向いている状況 |
|---------|------|--------------|
| **ブランクパターン** | 作成アクション実行で即座に空のアイテムを作成 | シンプルなオブジェクト。入力内容が少ない |
| **パラメーターパターン** | モーダルダイアログで必要パラメーターの入力を促す | 作成時に必須パラメーターが必要な場合 |
| **プレースホルダーパターン** | リスト最下行が常に新規入力欄として存在 | 素早く連続して作成したい場合（Todoリスト等） |
| **セーブアズパターン** | 既存アイテムを複製して新規作成（別名保存） | テンプレート的な利用が多い場合 |
| **テンプレートパターン** | 作成時にテンプレートを選択してアイテムを初期化 | パターンが決まっている複雑なオブジェクト |
| **マスターパターン** | マスター（型）を選択してインスタンスを作成。マスター変更が全インスタンスに反映 | 繰り返し構造を持つオブジェクト（イベント・ドキュメント等） |
| **ワンタイムモードパターン** | 作成アクション後にキャンバス上のドラッグ/クリックでアイテムを配置 | 図形・要素の座標配置が必要なキャンバス系UI |
| **ガッツパターン** | モードなしで即座にデフォルトスタイルのアイテムを作成（ワンタイムモード不使用） | モードレスを優先するキャンバス系UI |

### 6-B. Delete（削除）パターン

| パターン | 説明 | 向いている状況 |
|---------|------|--------------|
| **モーダルコンファームパターン** | 削除前にモーダルダイアログで確認を求める | 取り消し不能な削除。重要なデータ |
| **アンドゥアブルパターン** | 即時削除 + アンドゥで取り消し可能（またはゴミ箱移動） | 操作を中断せずスムーズに削除したい場合 |
| **モードレスコンファームパターン** | 削除ボタン押下で確認UIがモードレス表示。他操作でキャンセル | モーダルを避けつつ誤削除を防ぎたい場合 |

### 6-C. Update（更新）パターン

| パターン | 説明 | 向いている状況 |
|---------|------|--------------|
| **モーダルエディットパターン** | 参照モードと編集モードを分離。編集アクションでモーダル編集画面を開く | 参照と編集を明確に分けたい場合。編集後の保存確認が必要な場合 |
| **モードレスエディットパターン** | 常に編集可能。変更は自動保存 | リアルタイム編集が必要な場合（ドキュメント・メモ等） |

## 7. アクション配置の原則

| 原則 | 説明 | 例 |
|------|------|-----|
| オブジェクトの近くに配置 | アクションはその対象オブジェクトのそばに置く | 新規作成ボタン→コレクションビュー内、印刷ボタン→シングルビュー内 |
| 一貫した表現 | 同じ種類のアクションは統一したUIパターンで表現 | 全削除ボタンは同じスタイル・位置 |
| プロパティ固有アクション | 特定プロパティに対するアクションはそのプロパティのそばに配置 | ファイル添付ボタン→添付ファイル欄のそば |
