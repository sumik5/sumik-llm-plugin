# REST（Representational State Transfer）

REST はインターネット規模の分散システムのために設計された、最も普及した同期型 API スタイルである。
HTTP・リソース・URI の3要素を中心に構成され、**リソース指向（Resource style）** アーキテクチャを採る。
2000 年にネットワークアーキテクチャの研究成果として体系化され、現在あらゆる規模・業界で採用されている。

> **スコープ限定**: エンドポイント設計の命名規則・バージョニング戦略・ページネーション実装・OAS ドキュメント化
> などの実装実務は `developing-web-apis` → `DESIGN-ENDPOINTS.md`・`DESIGN-VERSIONING.md` を参照。
> 本ファイルは**APIスタイル選定判断に効く特性**に絞って解説する。

---

## 1. 基本構成要素

### HTTP・リソース・URI の役割

| 要素 | 役割 |
|------|------|
| **HTTP** | トランスポート。メソッド（動詞）・ヘッダー・ステータスコード・ボディで通信を構造化 |
| **Resource（リソース）** | 名前付け可能な情報。文書・画像・ユーザー・注文など。静的リソース（不変）と動的リソース（クライアント個別）に分類 |
| **URI（URL/URN）** | リソースの一意識別子。URL はアクセス方法＋場所を含み、URN は場所非依存の永続識別子 |

**メディアタイプ（MIME type）**: リソースのフォーマットを示す識別子。`application/json`・`text/html`・
`image/png` など1,600以上の公式メディアタイプが存在し、クライアントがデータの処理方法を判断する基準となる。

### REST 制約（6原則）

REST が定義する6つの設計制約。これらを充足するほど REST 本来の性質を発揮する。

| 制約 | 内容 | 選定への影響 |
|------|------|------------|
| クライアント・サーバー | 役割分離でポータビリティと独立進化を実現 | フロント・バックの個別スケールが可能 |
| **ステートレス** | 各リクエストは自己完結（セッション状態をサーバーが保持しない） | 水平スケーリングが容易。大きめのペイロードになりやすい |
| **キャッシュ可能** | レスポンスが明示的にキャッシュ可否を示す | `GET` は CDN・ブラウザでキャッシュ可能。レイテンシ削減の鍵 |
| 統一インターフェース | URI によるリソース識別・表現による操作・自己記述メッセージ・HATEOAS | API の普遍性・相互運用性の源泉 |
| 階層化システム | ゲートウェイ・プロキシ・CDN が透過的に挿入可能 | インフラ柔軟性。レガシー移行にも対応 |
| コード・オン・デマンド（任意） | サーバーからクライアントへコードを転送・実行 | 拡張性を高めるが XSS など攻撃面が増える |

## 2. API 成熟度モデル（Richardson Maturity Model）

HTTP ベース API の成熟度を4段階で評価するフレームワーク。**レベル2が実用上の標準**。

| レベル | 特徴 | 代表例 |
|--------|------|--------|
| **0** | 単一 URI + 単一 HTTP メソッド | SOAP（`POST` 1点集中） |
| **1** | リソース別 URI、単一 HTTP メソッド | 初期の REST-like API |
| **2** | リソース別 URI + 複数 HTTP メソッド（CRUD） | 現在の標準的 REST API |
| **3** | レベル2 ＋ HATEOAS（レスポンス内ハイパーメディアリンク） | 純粋な RESTful API |

**RESTful vs RESTless**:  
- **RESTful** = HATEOAS を実装したレベル3の API（REST アーキテクチャスタイルの原義による「本物の REST」）  
- **RESTless（REST API）** = レベル2。HATEOAS なしだが実用上十分。現在の主流  
- Google・Meta などの大手企業も含め、大半の実装はレベル2（RESTless）を採用している

### HATEOAS（レベル3）を採用しない主な理由

HATEOAS はサーバー・クライアント双方の複雑度を高め、実用コストが高い。

| 課題 | 詳細 |
|------|------|
| サーバー複雑性 | 動的リンク生成・HAL/Hydra/JSON-LD 等の複数ハイパーメディア形式の乱立 |
| クライアント複雑性 | 動的なリンク解釈とランタイム遷移が必要。デバッグとテストが困難 |
| バージョニング難度 | リンク数増加で破壊的変更が起きやすい |
| 開発コスト | バックエンド・フロントエンド双方に高い規律と専門知識が要求される |

```json
// HATEOAS レスポンス例（レベル3 の参考）
{
  "data": { "id": "a-001", "status": "pending" },
  "_links": {
    "self":    { "href": "/orders/a-001", "method": "GET" },
    "cancel":  { "href": "/orders/a-001/cancel", "method": "POST" },
    "confirm": { "href": "/orders/a-001/confirm", "method": "POST" }
  }
}
```

`_links` フィールドにより、クライアントが次に可能なアクションをサーバーから発見できる。
レベル3 の採用が有効なのは、アプリケーションロジックが頻繁に変更され、その変更をクライアントに自動発見させたい場合に限られる。

## 3. HTTP メソッドの冪等性・安全性・キャッシュ性

REST の選定特性として重要な HTTP メソッドの分類。

| HTTP メソッド | 冪等 | 安全 | キャッシュ | 主な用途 |
|--------------|:----:|:----:|:--------:|---------|
| `GET` | ✅ | ✅ | ✅ | Read（一覧・詳細取得） |
| `HEAD` | ✅ | ✅ | ✅ | メタ情報のみ取得 |
| `POST` | ❌ | ❌ | 条件付き | Create / 任意操作 |
| `PUT` | ✅ | ❌ | ❌ | Replace（完全置換） |
| `PATCH` | 条件付き | ❌ | ❌ | Update（部分更新） |
| `DELETE` | ✅ | ❌ | ❌ | Delete |
| `OPTIONS` | ✅ | ✅ | ❌ | CORS プリフライト |

- **冪等（Idempotent）**: 同じリクエストを複数回送っても、サーバーの最終状態が同じ
- **安全（Safe）**: サーバー状態を変更しない（読み取り専用に近い）
- **キャッシュ**: `GET`/`HEAD` は CDN・プロキシ・ブラウザでキャッシュ可能。これが REST の大きな強みの1つ

### リソース設計の概念（選定判断用）

```
# 良い例: リソース指向（REST らしい）
GET    /orders          # 一覧取得
POST   /orders          # 新規作成
GET    /orders/{id}     # 詳細取得
PATCH  /orders/{id}     # 部分更新
DELETE /orders/{id}     # 削除

# 避けるべき例: 操作指向（REST のメリットを損なう）
POST /createOrder
POST /getOrders
POST /deleteOrder/{id}
```

URI はリソース（名詞・複数形）で設計し、操作（動詞）は HTTP メソッドで表現するのが REST の慣習。
操作指向の設計は gRPC・GraphQL の思想に近い（本スタイルが REST を選ぶ理由がなくなる）。

## 4. キャッシュ戦略との関係（選定上重要）

REST の最大の差別化特性の1つが **HTTP キャッシュとの親和性**。

```
[クライアント] → CDN（キャッシュ） → [オリジンサーバー]
                     ↑
           GET /items/popular?region=jp
           Cache-Control: public, max-age=300
           → 300秒間 CDN でキャッシュ。同じリクエストはオリジンに届かない
```

- `GET` + `Cache-Control` ヘッダーで CDN・逆プロキシ・ブラウザが連携
- 高トラフィック・読み取り中心の API では REST + CDN がコスト効率を大幅に改善
- GraphQL の単一 `POST` エンドポイントではこのネイティブキャッシュが機能しない

## 5. Trade-Offs

### 利点

| 特性 | 詳細 |
|------|------|
| **成熟度・普及度** | 最も広く採用された API スタイル。ブラウザ・`curl`・全言語ライブラリで標準対応 |
| **HTTP キャッシュ** | `GET` レスポンスが CDN・プロキシ・ブラウザでキャッシュ可能。大規模なレイテンシと帯域の削減が可能 |
| **水平スケーリング** | ステートレス設計によりリクエストをサーバー間で均等分散。ロードバランサとの相性が良い |
| **相互運用性** | HTTP を話せるシステムはすべてクライアントになれる。言語・フレームワーク・OS 非依存 |
| **デバッグ容易性** | JSON/XML などテキスト形式。`curl`・ブラウザ DevTools・Postman で即座に確認可能 |
| **ツール生態系** | OpenAPI Specification・Swagger・Postman・Kong・WAF・レートリミッターなど成熟したエコシステム |
| **学習コストの低さ** | HTTP の知識がそのまま適用可能。バックエンド・フロントエンド・インフラ担当が共通言語で話せる |

### 欠点

| 特性 | 詳細 |
|------|------|
| **HTTP 動詞の貧しさ** | `GET`/`POST`/`PUT`/`PATCH`/`DELETE` 程度の語彙では複雑なドメイン操作を表現しにくい |
| **Over-fetching** | エンドポイントが固定のレスポンス形状を返すため、クライアントに不要なフィールドが含まれやすい |
| **Under-fetching** | 関連データを複数エンドポイントに分けて叩く必要が生じることがある |
| **ステートレスのコスト** | セッション情報を毎回ペイロードに含めるため、リクエストサイズが増大しやすい |
| **標準バージョニング不在** | URI・クエリパラメータ・ヘッダー等の戦略が非標準で実装依存。クライアントが API 変更に敏感 |
| **深いリソース関係の扱い** | 関連データ取得のために複数エンドポイントへの複数リクエストが必要になる場合がある |
| **コード・オン・デマンドのリスク** | オプション機能だが有効化すると XSS・インジェクション攻撃の経路になりうる |

## 6. When to Use REST

以下の条件に合致する場合、REST は最適または有力な選択肢となる。

### 適している状況

| 状況 | 理由 |
|------|------|
| **パブリック API・サードパーティ公開** | HTTP を知っていれば誰でも使える普遍的なインターフェース。ドキュメントも OAS で標準化可能 |
| **ブラウザ・モバイルが主要クライアント** | ネイティブ HTTP サポート。CDN キャッシュで静的・準静的データを高速化 |
| **CRUD 中心のリソース操作** | HTTP メソッドと URI が CRUD に自然にマッピングされる |
| **北向き（North-South）トラフィック** | データセンター出入り口の外部向け API として業界標準 |
| **異種システム・レガシーとの統合** | HTTP を話せれば言語・プラットフォーム非依存。レガシーとの橋渡しに適する |
| **読み取り中心の高トラフィック API** | `GET` + CDN キャッシュでインフラコストとレイテンシを大幅削減 |
| **既存インフラ（WAF・ゲートウェイ）を活用したい** | HTTP ステータスコード・ヘッダー準拠のため既存セキュリティツールがそのまま機能する |

### 慎重に検討すべき状況

| 状況 | 代替スタイル |
|------|-------------|
| 複雑なリレーションを少ないリクエストで取得したい | GraphQL → `STYLE-GRAPHQL.md` |
| 内部サービス間の低レイテンシ・高スループット RPC | gRPC → `STYLE-GRPC.md` |
| サーバーからのリアルタイムイベント通知が必要 | WebSocket / Webhook → `STYLE-WEBSOCKET-MESSAGING.md` / `STYLE-WEBFEEDS-WEBHOOKS.md` |
| 非同期の大量メッセージ処理 | ブローカー型メッセージング → `STYLE-WEBSOCKET-MESSAGING.md` |

> **意思決定マトリクス**: `SELECTION-MATRIX.md` で REST を含む7スタイルの横断比較と決定木を参照。
