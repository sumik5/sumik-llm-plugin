# GraphQL

GraphQL は Facebook（現 Meta）が 2012 年ごろモバイルアプリの課題解決を目的として開発し、2015 年に
オープンソース化した**クエリ言語 ＋ サーバーサイドランタイム**の API スタイルである。
クライアントがレスポンスの形状を指定できる柔軟性を持ち、REST の over-fetching / under-fetching 問題を
設計レベルで軽減する。HTTP をトランスポートとして使用するが、GraphQL の仕様自体はトランスポート非依存。

> **スコープ限定**: スキーマ定義の実務・リゾルバー実装・認証フロー・OAS 生成などの実装詳細は
> `developing-web-apis` を参照。本ファイルは**APIスタイル選定判断に効く特性**に絞って解説する。

---

## 1. REST との根本的な違い

### 概念対比

| 観点 | REST | GraphQL |
|------|------|---------|
| エンドポイント数 | リソースごとに複数 | 原則1つ（`/graphql`） |
| データ形状の決定者 | サーバー（固定） | クライアント（指定） |
| 思想 | リソース指向（名詞） | クエリ指向・グラフ思考 |
| 型システム | なし（OAS で補完） | スキーマ・静的型システムを内包 |
| 主な HTTP メソッド | `GET`/`POST`/`PUT`/`PATCH`/`DELETE` | 主に `POST`（`GET` でクエリも可） |
| バージョニング | URI/ヘッダー/パラメータで管理 | スキーマ進化でバージョンレスを目指す |
| HTTP キャッシュ | `GET` の CDN キャッシュが容易 | 単一 `POST` エンドポイントのためキャッシュが困難 |

### Over-fetching / Under-fetching の解消（比較）

```
# REST: エンドポイントが固定の形状を返す
GET /items/a-001
→ { "id": "a-001", "name": "Laptop", "price": 1500,
    "stock": 20, "description": "...", "category": {...} }
  ↑ 名前だけ必要なのに全フィールドが返る（over-fetching）

# GraphQL: クライアントが必要なフィールドのみ指定
query { item(id: "a-001") { name } }
→ { "data": { "item": { "name": "Laptop" } } }
  ↑ 指定したフィールドのみ返却（ネットワーク効率化）
```

## 2. グラフ思考（Thinking in Graphs）

GraphQL の核心はデータを**グラフ構造**として表現する思想にある。

- **ノード** = データエンティティ（REST のリソースに対応）
- **エッジ** = ノード間のリレーションシップ（フィールドのネスト）

```graphql
# ユーザー・注文・商品を1リクエストで取得
query GetUserOrders {
  user(id: "u-001") {
    name
    orders {
      id
      status
      items {
        name
        price
      }
    }
  }
}
```

REST では3エンドポイント（`/users`・`/orders/{id}`・`/items/{id}`）を叩く場面を
1リクエストで完結させられる。これは特にモバイル・IoT 環境での通信効率に直接影響する。

## 3. GraphQL の構成要素

### 操作種別（Operations）

| 操作 | CRUD 対応 | HTTP | 説明 |
|------|-----------|------|------|
| **Query** | Read | `POST` または `GET` | 宣言的なデータ取得。複数フィールドを1度に取得可能 |
| **Mutation** | Create / Update / Delete | `POST` | データ変更操作。CUD を `POST` 1メソッドで表現 |
| **Subscription** | リアルタイム受信 | WebSocket（主）または SSE | イベント発生時にサーバーからデータをプッシュ配信 |

> Subscription は WebSocket プロトコルを利用することが多いが、HTTP Server-Sent Events (SSE) や
> long-polling による実装も可能。双方向通信の要件がある場合は WebSocket との比較が必要（→ `STYLE-WEBSOCKET-MESSAGING.md`）。

### 型システムとスキーマ（SDL 骨子）

```graphql
# GraphQL Schema Definition Language（SDL）の骨子
# フレームワーク非依存の汎用表現

type Order {
  id:     ID!
  status: String!
  items:  [Item!]!     # 他の型への参照（エッジ）
}

type Item {
  id:    ID!
  name:  String!
  price: Float!
}

type Query {
  order(id: ID!): Order
  orders: [Order!]!
}

type Mutation {
  createOrder(items: [ID!]!): Order!
  cancelOrder(id: ID!): Boolean!
}

type Subscription {
  orderStatusChanged(id: ID!): Order!  # WebSocket でプッシュ
}
```

- **スカラー型**: `Int`・`Float`・`String`・`Boolean`・`ID`（プリミティブ値）
- **`!`（Non-null）**: `null` を許容しないことを示す
- **リゾルバー**: 各フィールドに対応する関数。DB・外部 API・マイクロサービスと接続する橋渡し役

## 4. Code-First vs Schema-First

GraphQL API の構築アプローチは2通りある。選択は開発プロセスの好みや体制による。

| アプローチ | 方向 | 特徴 | 向いている状況 |
|----------|------|------|--------------|
| **Code-First** | コード → スキーマ自動生成 | 実装言語のクラス・デコレータでスキーマを定義。型安全性が保ちやすい | 1チームが実装とスキーマを管理する場合 |
| **Schema-First** | スキーマ（`.graphql`）→ コード実装 | SDL をコントラクトとして先行定義。フロント・バックが並行開発可能 | チームを分けてコントラクト先行設計をする場合 |

## 5. セキュリティ上の考慮点（選定に影響する特性）

GraphQL は単一エンドポイントに高い表現力が集中するため、固有の攻撃ベクトルがある。
これらの対策コストは採用時の考慮事項として事前に把握しておく必要がある。

### 主要攻撃ベクトルと対策

| 攻撃 | 内容 | 対策 |
|------|------|------|
| **イントロスペクション悪用** | スキーマ全体を取得して非公開エンドポイントを把握する | 本番環境でイントロスペクションを無効化 |
| **再帰クエリ** | 深いネストにより指数的なデータ取得でサーバーを圧迫 | クエリのネスト深度に上限を設定 |
| **バッチクエリ攻撃** | 1リクエストに多数のオペレーションをまとめてサーバーを過負荷にする | バッチ操作数の上限・タイムアウト設定 |
| **エイリアス悪用** | エイリアスで同一フィールドを大量に並列実行する | クエリ複雑度スコアリング・レートリミット |

```graphql
# 再帰クエリ攻撃の例（深いネストで指数的にデータが増加）
query RecursiveAttack {
  user {
    friends { user { friends { user { friends {
      id name
    } } } } }
  }
}
# → ネスト深度制限（例: max_depth=5）で防御
```

```graphql
# エイリアス悪用の例（同一フィールドを複数名で並列実行）
query AliasAttack {
  a1: items(filter: {name: "Laptop"}) { id }
  a2: items(filter: {name: "Phone"}) { id }
  # ... a100 まで続く
}
# → バッチ数上限・クエリ複雑度スコアリングで防御
```

### N+1 問題（パフォーマンス上の重要課題）

グラフのリレーション取得で、1 のクエリに対して N 件の追加クエリが発生する問題。

```
# ナイーブな実装の場合
query { orders { id item { name } } }

# DB 実行パターン
1. SELECT * FROM orders         → 10件取得
2. SELECT * FROM items WHERE id = 1  ┐
3. SELECT * FROM items WHERE id = 2  │ N 回繰り返す
   ...                               ┘
   合計: 1 + N クエリ（= N+1 問題）

# DataLoader パターンで解決
2. SELECT * FROM items WHERE id IN (1, 2, ..., 10)
   → 1回のバッチクエリに集約
```

REST でも N+1 問題は発生するが、GraphQL はネストが深くなりやすいため顕在化しやすい。
採用時は DataLoader 等の対策を標準的に組み込む前提で設計する必要がある。

## 6. ドキュメント・イントロスペクション

GraphQL はスキーマに基づいてドキュメントを自動生成する能力を標準で持つ。

```graphql
# イントロスペクションクエリ（スキーマの型一覧を取得）
{
  __schema {
    types {
      name
      kind
      fields { name type { name } }
    }
  }
}
```

| ツール | 用途 |
|--------|------|
| **GraphiQL** | ブラウザベースのインタラクティブ IDE。スキーマ閲覧・クエリ実行・ドキュメント探索 |
| **SpectaQL** | スキーマから静的ドキュメントを生成（本番向け） |
| **Voyager** | スキーマをビジュアルグラフで可視化 |
| **GraphQL Codegen** | スキーマからクライアントコード・型定義を自動生成 |

> ⚠️ イントロスペクションは攻撃面になるため、本番環境では無効化を検討。
> 開発環境のみ有効にし、本番向けドキュメントは事前生成・外部ホストする構成が一般的。

## 7. マイクロサービス統合パターン

GraphQL はマイクロサービスアーキテクチャの統合層としても機能する。

### BFF（Backends for Frontends）パターン

```
[ブラウザ用 GraphQL BFF] ──┐
[モバイル用 GraphQL BFF]   ├── 複数バックエンドサービス
[TV/IoT 用 GraphQL BFF] ──┘   （REST / gRPC / DB）
```

各クライアントプラットフォームに最適化した GraphQL スキーマを個別に持つ。
クライアントの多様性が高い（Web・モバイル・IoT 等）場合に有効。

### フェデレーション（Federated GraphQL）

複数の独立した GraphQL サービスを単一グラフに統合。
各マイクロサービスが自身の GraphQL スキーマを持ち、ゲートウェイが統合クエリを解決する。

## 8. Trade-Offs

### 利点

| 特性 | 詳細 |
|------|------|
| **カスタマイズ可能なレスポンス** | クライアントが必要なフィールドのみ取得可能。ネットワーク転送量を最適化 |
| **少ないリクエスト数** | 関連データを1リクエストで取得可能。複数エンドポイントへの往復を削減 |
| **スキーマ型安全性** | スキーマと型システムにより、実行前にリクエストの妥当性が検証される |
| **自己ドキュメント化** | スキーマからドキュメントが自動生成。GraphiQL で対話的に API を探索可能 |
| **バージョンレス進化** | 新フィールドの追加は破壊的変更なし。既存クライアントへの影響を最小化 |
| **マイクロサービス統合** | 複数バックエンドを単一グラフに統合可能（BFF・フェデレーション） |
| **クライアント主導開発** | フロントエンドが API 形状を自律的に決定。バックエンドへの変更依頼が減る |

### 欠点

| 特性 | 詳細 |
|------|------|
| **HTTP キャッシュの難しさ** | 全リクエストが `POST` でシングルエンドポイントに集中。CDN・HTTP キャッシュが効きにくい |
| **セキュリティ複雑性** | イントロスペクション悪用・深いクエリ・バッチ攻撃への対策が必要。攻撃面が1点に集中 |
| **N+1 問題** | リレーションのネスト取得で N+1 クエリが発生しやすい。DataLoader 等の対策が必須 |
| **重いクエリのリスク** | 複雑なクエリ・深いネストがパフォーマンス劣化を起こしやすい。複雑度制御が必要 |
| **ファイルアップロード非標準** | `multipart/form-data` は仕様非対応。追加ライブラリ・Base64 エンコード等の回避策が必要 |
| **HTTP ステータスコードの限定活用** | エラーも基本的に HTTP 200 で返り、エラー詳細はボディに含まれる。監視・ログ管理に工夫が必要 |
| **学習コスト** | SDL・リゾルバー・型システム・DataLoader など REST にはない概念の習得が必要 |

## 9. When to Use GraphQL

以下の条件に合致する場合、GraphQL は最適または有力な選択肢となる。

### 適している状況

| 状況 | 理由 |
|------|------|
| **モバイル・IoT・省電力デバイスが主要クライアント** | 必要最小限のデータのみ転送でネットワーク効率と電力消費を最適化 |
| **クライアントごとに異なるデータ形状が必要** | Web・モバイル・TV など多様なクライアントが同一 API を最適形状で利用可能 |
| **深くネストした関係データを少ないリクエストで取得したい** | ユーザー→注文→商品→カテゴリなどのグラフ構造を1クエリで表現 |
| **複数データソースの統合（BFF・フェデレーション）** | 単一スキーマで複数バックエンドを抽象化 |
| **フロント主導の高速プロトタイピング** | バックエンドへの変更依頼なしにフロントが必要なデータ形状を決定できる |
| **スキーマ先行のコントラクト設計** | Schema-First でフロント・バックの並行開発が可能 |

### AskUserQuestion: スキーマ定義アプローチの確認

GraphQL 採用が確定した場合、以下を確認する。
> ツールが使えない環境では、同じ選択肢をテキスト質問として提示して確認すること。

**スキーマ定義アプローチをどちらにしますか？**
1. **Schema-First（契約先行）** — `.graphql` ファイル（SDL）を先に定義し、コードを後から実装する
2. **Code-First（実装から生成）** — クラス・デコレータでスキーマを定義し、SDL ファイルを自動生成する

### 慎重に検討すべき状況

| 状況 | 代替スタイル |
|------|-------------|
| シンプルな CRUD API で形状の柔軟性が不要 | REST → `STYLE-REST.md` |
| 低レイテンシ・高スループットの内部 RPC | gRPC → `STYLE-GRPC.md` |
| ファイルアップロードが中心 | REST（`multipart/form-data` を標準サポート） |
| CDN キャッシュで高頻度読み取りを最適化したい | REST（`GET` キャッシュが容易） |
| 既存の HTTP 監視・WAF をそのまま活用したい | REST（HTTP ステータスコードが標準的に活用可能） |
| 双方向リアルタイム通信 | WebSocket → `STYLE-WEBSOCKET-MESSAGING.md` |

> **意思決定マトリクス**: `SELECTION-MATRIX.md` で GraphQL を含む7スタイルの横断比較と決定木を参照。
