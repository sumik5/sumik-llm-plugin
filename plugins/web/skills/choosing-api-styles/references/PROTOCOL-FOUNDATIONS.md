# プロトコル基盤：HTTP進化とTCP特性

APIスタイル間の性能差（レイテンシ・スループット・HOL blocking・接続確立コスト）を説明するために、
TCP/IP の基本と HTTP/0.9→1.0→1.1→2→3（QUIC）の進化を整理する。

---

## 1. ネットワークプロトコルとは

**ネットワークプロトコル** とはネットワークデバイス間の通信ルールの集合。

ネットワークの目的はバイナリデータの交換を可能にすること。送信側と受信側が同じルールで通信するためにプロトコルが必要となる。

多くのネットワークプロトコル（TCP/IPなど）はOSが実装するため、アプリケーション開発者はプロトコルの詳細を実装せずOSのAPIを通じて利用できる。

---

## 2. Socket API と TCP/IP

**TCP/IP**（Transmission Control Protocol / Internet Protocol）はインターネットトラフィックの大部分を処理するプロトコルスイート。

**Socket API** はアプリケーションがOSのネットワークスタックと対話するための主要なOS呼び出しセット。
以下の操作を提供する：

- ネットワーク接続のオープン
- メッセージの送受信（`send` / `recv`）
- 接続のクローズ

クライアント・サーバー方向での典型的なフロー：

```
Client                       Server
  │── socket() ──────────────►│
  │── connect() ─────────────►│  (TCP 3-wayハンドシェイク)
  │── send() ────────────────►│
  │◄─ recv() ─────────────────│
  │── close() ────────────────►│
```

> **クライアント** = 接続を開始する側。**サーバー** = 接続を待ち受ける側。

## 3. TCP/IPとOSIモデル

**OSIモデル**（7層）は1984年制定のネットワーク概念モデルで、プロトコルのレイヤーを整理する枠組みとして今も参照される。

| OSI層 | TCP/IP層 | 代表プロトコル | データ単位 |
|-------|----------|----------------|-----------|
| 7 Application | Application | HTTP / DNS / XMPP | message |
| 6 Presentation | ↑ | TLS | record |
| 5 Session | ↑ | — | — |
| 4 Transport | Transport | TCP / UDP / QUIC | segment / datagram |
| 3 Network | Internet | IP | packet |
| 2 Data link | Link | Ethernet | frame |
| 1 Physical | ↑ | — | bit / signal |

**プロトコルのカプセル化**：上位層のデータはヘッダーを付けられて下位層のデータの「内側」に包まれる（マトリョーシカ人形の比喩）。受信側は逆順にヘッダーを剥がして上位層へ渡す。

> **注意**: QUICは暗号化（TLS由来）をトランスポート層に統合し、UDPの上で動作する。OSI厳格分離の例外として、クロスレイヤー最適化を実現している。

## 4. TCPの特性と制限

### 4-1. TCPが保証するもの

| 特性 | 説明 |
|------|------|
| 信頼性（Reliability） | 失われたパケットを自動的に再送する |
| 順序保証（In-order delivery） | 送信順にデータが届く |
| フロー制御（Flow control） | 受信バッファを超えないよう送信量を調整 |
| 輻輳制御（Congestion control） | ネットワーク混雑時に送信量を下げる |

HTTPはTCPの信頼性と順序保証の恩恵を受けるが、同時にその制限にも縛られる。

---

### 4-2. TCP Head-of-Line (HOL) Blocking

**HOL blocking** とは、TCPストリーム中のあるセグメントが失われたとき、後続のセグメントが届いていても受信側がそれを上位層へ渡せなくなる問題。

「Head（先頭）」のセグメントが詰まることで行列全体が止まる、というキューイング理論の概念に由来する。

```
送信順:  [Seg1][Seg2 ← 消失][Seg3][Seg4]
                  ↑
受信側: Seg1 ✓, Seg2 未着, Seg3 バッファ待機, Seg4 バッファ待機
         → Seg2 が再送されるまで Seg3・Seg4 は上位層へ渡せない
```

**影響**：
- パケットロスが起きるとレイテンシが急激に増大する
- HTTP/1.1 のパイプラインングが普及しなかった主因のひとつ
- HTTP/2 はアプリケーション層で多重化するが、**単一 TCP 接続上のため TCP HOL blocking は解決しない**

---

### 4-3. TCP スロースタートと輻輳回避

**スロースタート** とは、新規TCP接続でネットワーク帯域を探索するフェーズ：

```
初期送信量: 小（例: 2セグメント）
   ↓ ACKを受け取るごとに倍増
ssthresh（閾値）到達
   ↓ 輻輳回避フェーズ（緩やかな増加）に移行
```

**影響**：短命な TCP 接続（ブラウザが小さいリソースを1リクエストで取得する場合）では、
帯域を最大限活用できないまま接続が終わる。これが HTTP/1.1 で「1ドメイン6接続」が推奨された背景のひとつ。

---

## 5. HTTP の進化

HTTP は TCP の上に構築され、バージョンごとに TCP の制限を段階的に緩和してきた。

### 5-1. HTTP/0.9（1991年）

**最小仕様**: `GET` メソッドのみ。ヘッダーなし。レスポンス後に TCP 接続をクローズ。

```
Client → Server: GET /index.html\r\n
Server → Client: <html>...</html>
                 [TCP FIN]
```

- テキスト文書の検索・取得に特化
- リクエストごとに TCPハンドシェイク（1 RTT）が必要

---

### 5-2. HTTP/1.0（1996年）

主な追加機能：

| 機能 | 説明 |
|------|------|
| ヘッダーとボディの分離 | メタデータを `Content-Type`・`Content-Length` 等で伝達 |
| `GET` 以外のメソッド | `HEAD`・`POST`・`PUT`・`DELETE` など |
| ステータスコード | `1xx` 情報 / `2xx` 成功 / `3xx` リダイレクト / `4xx` クライアントエラー / `5xx` サーバーエラー |
| コンテンツ圧縮 | `Content-Encoding: gzip` などでボディを圧縮 |
| 基本認証 | `Authorization` ヘッダーでの認証機構 |

**制限**: デフォルトでリクエストごとに TCP 接続を切断・再確立する。

---

### 5-3. HTTP/1.1（1997年/1999年）

最重要の改善：**TCP 持続接続（Persistent Connection）** がデフォルト化。

```
HTTP/1.0（接続のたびにハンドシェイク）:
  GET /a → [TCP SYN/SYN-ACK/ACK] → response → [TCP FIN]
  GET /b → [TCP SYN/SYN-ACK/ACK] → response → [TCP FIN]

HTTP/1.1（単一 TCP 接続で複数リクエスト）:
  [TCP SYN/SYN-ACK/ACK]
  GET /a → response
  GET /b → response
  [TCP FIN]
```

主な追加機能：

| 機能 | 説明 |
|------|------|
| 持続接続 | `Connection: keep-alive` でTCP接続を使い回す |
| チャンク転送エンコーディング | ボディを分割してストリーミング送信 |
| パイプラインニング | 応答を待たずに複数リクエストを送信（HOL blockingで普及せず） |
| `HOST` ヘッダー必須 | バーチャルホスティング（単一IPで複数ドメイン）を実現 |
| コンテンツネゴシエーション | `Accept` ヘッダーでクライアントの希望形式を伝達 |
| 追加ステータスコード | `101 Switching Protocols`（WebSocket昇格に使用）など |
| HTTP キャッシュ | `Cache-Control`・`ETag`・`If-None-Match` 等 |

**残課題**: TCP HOL blocking・スロースタートによるパフォーマンス限界。
ブラウザは1ドメインあたり約6接続を並列に張る回避策を採用した。

---

### 5-4. HTTP/2（2015年）

バイナリフレームと多重化（Multiplexing）による大幅な性能改善。

主要な特徴：

| 特徴 | 説明 |
|------|------|
| **バイナリフレーム** | テキストから機械可読バイナリへ。フレーム種別（`HEADERS`・`DATA`・`SETTINGS`・`GOAWAY` 等）で構造化 |
| **ヘッダー圧縮（HPACK）** | 繰り返し送るHTTPヘッダーを差分圧縮 |
| **ストリームと多重化** | 1つのTCP接続上で複数HTTPリクエストを並列処理。応答順序制約なし |
| **サーバープッシュ** | クライアントが要求する前にリソースを送信 |

HTTP/2 の多重化イメージ：

```
単一 TCP 接続（HTTP/2）:
  ┌──────────────────────────────┐
  │ Stream 1: GET /api/users     │
  │ Stream 2: GET /api/orders    │  ← 同時送受信
  │ Stream 3: GET /static/app.js │
  └──────────────────────────────┘
```

**重要な制約**: HTTP/2 は TCP HOL blocking を解決しない。
TCP 接続は1本なので、あるセグメントが失われると全ストリームがブロックされる。
損失のない環境では多重化が効果を発揮するが、パケットロスが多い環境では HTTP/1.1 の複数接続に劣ることもある。

**暗号化**: 仕様上は任意だが、主要ブラウザは TLS 必須（`h2`）。
平文バージョン（`h2c`）はサービス間通信等で利用される。

---

### 5-5. HTTP/3 と QUIC（2022年 RFC 9000/9114）

HTTP/2 の TCP 依存を断ち切り、UDP 上の **QUIC** プロトコルで動作する新世代 HTTP。

**QUIC の主要特性**：

| 特性 | 説明 | 選定への影響 |
|------|------|-------------|
| **必須暗号化（TLS 1.3統合）** | 全パケットを個別暗号化。ミドルボックスによるプロトコル破壊を防ぐ | セキュリティ前提が強化 |
| **1-RTT 接続確立** | クライアントがサーバーの TLS パラメータを予測して初回から HTTP を送れる | モバイル・高レイテンシ環境でのコスト削減 |
| **0-RTT 再接続** | 過去の TLS セッション情報を再利用。安全なメソッド（GET等）は即送信可能 | 再接続コストをほぼゼロに |
| **接続マイグレーション** | 接続 ID ベースの識別でIPアドレス変更（WiFi↔5G切替）を透過的に扱う | モバイルアプリに特に有効 |
| **HOL blocking 軽減** | 独立したストリームごとに喪失影響を局所化。ただしバースト損失時は複数ストリームが影響を受ける | 損失が多いネットワーク環境での改善 |

HTTP バージョン別の接続確立コスト比較：

```
HTTP/1.1 と HTTP/2:  [TCP 3-way] + [TLS ハンドシェイク] = 2 RTT
HTTP/3（初回）:       [QUIC ハンドシェイク（TLS統合）]   = 1 RTT
HTTP/3（再接続・0-RTT）:                                  = 0 RTT
```

**多重化と HOL blocking の比較**：

```
HTTP/1.1（単一TCP接続、HOL有り）:
  [S1] [S2 ← 消失] [S3] [S4]
   → S2 再送まで S3, S4 ブロック

HTTP/2（単一TCP接続、HOL有り）:
  Stream A + Stream B が同一 TCP 上
  → TCP セグメント消失で両ストリームがブロック

HTTP/3（QUIC、HOL軽減）:
  Stream A, Stream B が独立したストリーム
  → Stream A のパケット消失が Stream B には影響しない（理想ケース）
```

QUIC は **QPACK** ヘッダー圧縮を採用（HTTP/2 の HPACK はストリーム間の順序を前提とするため QUIC に適合しない）。

## 6. レイテンシ vs 帯域幅

API設計において混同しやすいが、異なる概念：

| 概念 | 定義 | 改善手段 |
|------|------|---------|
| **レイテンシ（Latency）** | 1つのリクエスト-レスポンスが完了するまでの時間 | 接続再利用・QUIC 0-RTT・CDN |
| **帯域幅（Bandwidth）** | 単位時間に転送できるデータ量 | 圧縮・バイナリプロトコル・多重化 |

- **gRPC（HTTP/2）** は Protobuf + バイナリフレームで帯域幅効率が高い
- **WebSocket** は持続的接続で接続コスト（レイテンシ）を排除する
- **REST（HTTP/1.1）** は新しい接続ごとに TCP+TLS のコストが発生しうる（持続接続で緩和）

---

## 7. DNS とブラウザ並列接続

HTTP のレイテンシ分析に DNS を含めることが重要：

- ドメイン名→IPアドレスの解決（DNS解決）は通常50〜150ms
- 初回接続では DNS → TCP → TLS → HTTP の順でオーバーヘッドが積み重なる

ブラウザの最適化（HTTP/1.1時代）：

- 1ドメインあたり約6TCP接続を並列に張る
- サブドメイン分散（domain sharding）でさらに接続数を増やす

HTTP/2・HTTP/3 では単一接続で多重化されるため、domain sharding は不要（むしろ逆効果になる）。

## 8. プロトコル選択とAPIスタイルの関係

プロトコルの特性はAPIスタイルの選定に直結する：

| シナリオ | 推奨スタイル | 理由 |
|----------|-------------|------|
| サービス間の高スループット通信 | **gRPC（HTTP/2）** | バイナリ・多重化・ストリーミング |
| ブラウザ↔サーバーのリアルタイム通信 | **WebSocket** | full-duplex・HTTP/1.1 Upgrade で確立 |
| モバイル・高レイテンシ環境 | **HTTP/3 対応のREST** | QUIC 0-RTT・接続マイグレーション |
| シンプルなリソース公開 | **REST（HTTP/1.1/2）** | 広いクライアント互換性 |
| イベント通知（サーバー→クライアント） | **Webhook** | HTTP POST でコールバック |
| 非同期・高信頼配信 | **ブローカー型** | AMQP 等・ACK・Dead Letter Queue |
