# Web フィード・Webhook スタイルリファレンス

Web フィードと Webhook はいずれも**非同期**の情報配信スタイルだが、
通信の方向性が逆である。

| スタイル | 方向 | クライアントの役割 |
|---------|------|-----------------|
| **Web フィード** | pull（クライアント主導） | 定期的にフィードをポーリングして差分取得 |
| **Webhook** | push（ソース主導） | コールバック URL を用意して待ち受ける |

---

## Web フィード（RSS / Atom）

### 概念

Web フィードは定期的に更新されるコンテンツをクライアントが取得できるよう
HTTP `GET` エンドポイントとして公開するしくみ。
クライアント（フィードリーダー）が定期的にフィード URL をポーリングし、新しいエントリを取得する。

```
[フィードリーダー] --GET /feed--> [サーバー]
                  <-- フィード文書 --
        ↑定期的に繰り返す
```

> **pull 型・非同期**: 新コンテンツの到着通知はサーバーから行われない。
> クライアントが問い合わせることで最新情報を知る。

### フォーマット

| フォーマット | 仕様 | 特徴 |
|-------------|------|------|
| **RSS 2.0** | Netscape 発祥の XML ベース。IETF 標準化なし。最も普及 | シンプルだがバージョン断片化（0.9x・1.x・2.x） |
| **Atom** | IETF RFC 4287 標準。XML ベース | Unicode 対応・要素再利用性・AtomPub 公開プロトコル付属 |
| JSON feed | JSON ベース（新興） | 開発者フレンドリー・軽量 |

**Atom フィードの最小構造例**:

```xml
<?xml version="1.0" encoding="utf-8"?>
<feed xmlns="http://www.w3.org/2005/Atom">
  <id>https://example.com/blog</id>
  <title>製品アップデートフィード</title>
  <updated>2024-06-01T00:00:00Z</updated>
  <link rel="self" href="/blog/feed"/>
  <entry>
    <id>https://example.com/blog/2024/v2-released</id>
    <title>バージョン 2.0 リリース</title>
    <updated>2024-06-01T00:00:00Z</updated>
    <link href="/blog/2024/v2-released"/>
    <summary>新機能と改善点の概要</summary>
  </entry>
</feed>
```

**RSS 2.0 の最小構造例**:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<rss version="2.0">
  <channel>
    <title>製品ニュース</title>
    <description>最新の製品情報</description>
    <link>https://example.com</link>
    <item>
      <title>v2.0 リリース</title>
      <description>新機能を追加しました</description>
      <link>https://example.com/blog/v2</link>
      <guid>https://example.com/blog/v2</guid>
    </item>
  </channel>
</rss>
```

### 主なユースケース

- ニュースサイト・技術ブログの更新配信
- クラウドベンダーのサービス障害・変更通知（AWS・Azure・Google Cloud 各社が公開）
- セキュリティ勧告・脆弱性情報の配信
- ポッドキャスト・動画チャンネルの新着通知

### セキュリティ上の注意点

| リスク | 対策 |
|--------|------|
| DoS 攻撃 | レート制限・WAF の適用 |
| フィードインジェクション | フィードコンテンツのサニタイズ・バリデーション |
| XSS | フィードリーダーによる HTML レンダリング時のエスケープ |

## Webhook（コールバック型 push API）

### 概念

Webhook は**ソースサービスがイベント発生時に宛先サービスのコールバック URL へ
HTTP `POST` を送信する**しくみ。宛先はコールバック URL を事前に登録して待ち受ける。

```
[ソース] --event.created--> POST /callback --> [宛先]
         <-------------- 200 OK（確認応答） <--
```

Webhook の別名：
- **HTTP push API**: プッシュ型の HTTP API
- **HTTP callback**: HTTP コールバック
- **リバース API**: クライアント-サーバーが逆転した構造

### 受信・送信 Webhook の分類

観点はどのサービスを基準にするかで変わる。

| 分類 | 定義 |
|------|------|
| **受信 Webhook (incoming)** | 対象サービスがメッセージを受け取る方向 |
| **送信 Webhook (outgoing)** | 対象サービスがメッセージを送り出す方向 |

例：在庫サービスが注文サービスへ在庫切れを通知する場合、
在庫サービスからは「送信 Webhook」、注文サービスからは「受信 Webhook」。

### 統合フロー

```
1. 宛先サービス  → コールバック URL（HTTP エンドポイント）を実装
2. 宛先サービス  → ソースサービスへ登録（POST または設定画面で URL を入力）
3. ソースサービス → イベント発生時に HTTP POST をコールバック URL へ送信
4. 宛先サービス  → ペイロードを検証・処理 → HTTP 200 で確認応答
5. ソースサービス → 未応答の場合は再試行（宛先側の冪等性処理が必要）
```

### ペイロード構造例

```json
{
  "id": "evt_01HQ3XYZ",
  "type": "order.created",
  "timestamp": "2024-06-01T12:00:00Z",
  "data": {
    "order_id": "ord_001",
    "user_id": "usr_999",
    "total_amount": 5000
  }
}
```

### セキュリティ（署名検証・HMAC）

Webhook は公開エンドポイントへの POST のため、
**ペイロードの完全性と送信元の正当性を HMAC で検証**する。

署名生成の仕組み：

```
# ソース側
sig_scheme = "{webhook-id}.{timestamp}.{payload}"
signature  = HMAC-SHA256(shared_secret, sig_scheme)
header: Webhook-Signature: v1,{base64(signature)}

# 宛先側
expected = HMAC-SHA256(shared_secret, sig_scheme)
if received_sig != expected: reject(401)
```

主要なセキュリティ対策：

| 対策 | 理由 |
|------|------|
| HMAC 署名検証 | ペイロード改ざん・なりすまし防止 |
| タイムスタンプ検証 | リプレイアタック防止（受信後 5 秒等のウィンドウで拒否） |
| TLS（HTTPS） | 通信経路の暗号化 |
| IP アドレス許可リスト | 信頼できるソース IP のみ受け入れ |
| ペイロードサイズ制限 | 過大リクエストによるリソース消費を防ぐ（例: 20 KB 以内） |
| 冪等性処理 | 同一 `webhook-id` の重複受信を排除 |

> [standard-webhooks](https://www.standardwebhooks.com) は
> HMAC・再試行・ヘッダー形式を統一する業界仕様。Go・Python・Node.js 等に
> リファレンス実装がある。

### 再試行と冪等性

配信が失敗した場合、ソースは指数バックオフで再試行する。
宛先側は **`webhook-id` ベースの冪等性チェック**が必須。

```
再試行スケジュール例（指数バックオフ）:
1 回目失敗 → 30 秒後
2 回目失敗 → 5 分後
3 回目失敗 → 30 分後
4 回目失敗 → 2 時間後（最大試行回数まで繰り返す）
```

## Trade-Offs

### Web フィード — 利点

| 利点 | 説明 |
|------|------|
| **シンプルな実装** | 静的ファイルまたは HTTP `GET` エンドポイントを提供するだけ。相互登録不要 |
| **プライバシー保護** | ログインなしで購読できる。ソーシャルプラットフォームへの個人情報提供が不要 |
| **コンテンツ集約** | フィードリーダーが複数フィードを一元管理できる。訪問先ごとに回遊しなくてよい |
| **コンテンツシンジケーション** | ブログ・ポッドキャスト・動画を他プラットフォームへ容易に配信できる |
| **標準化** | Atom は IETF RFC 4287 で標準化。RSS 2.0 も広く普及 |

### Web フィード — 欠点

| 欠点 | 説明 |
|------|------|
| **ポーリング遅延** | サーバーから能動的にプッシュされない。ポーリング間隔だけ通知が遅れる |
| **フォーマット断片化** | RSS の複数バージョン（0.9x・1.x・2.x）と Atom が混在。リーダーが複数形式に対応する必要がある |
| **ユーザー追跡の困難** | 誰がフィードを読んでいるかを直接把握しにくい（匿名性の裏返し） |
| **コントラクト欠如** | フィード提供者が更新を止めても購読者に通知されない |
| **情報過多リスク** | 多くのフィードを購読すると読み切れない情報量になりがち |

---

### Webhook — 利点

| 利点 | 説明 |
|------|------|
| **ポーリング不要** | イベント発生時にのみ通信が発生するため、無駄なリクエストが発生しない |
| **低遅延** | イベント発生直後にコールバックが届く（ポーリング間隔の待ち時間なし） |
| **確認応答（ACK）** | 各メッセージに対して宛先が HTTP レスポンスで確認。未確認は再試行される |
| **イベント駆動** | サービス間の疎結合なリアルタイム連携が可能 |
| **シンプルな技術基盤** | HTTP `POST` のみ。特別なプロトコルやブローカー不要 |

### Webhook — 欠点

| 欠点 | 説明 |
|------|------|
| **配信保証の複雑さ** | 宛先がダウン中の未配信メッセージには、再試行キュー・ストレージが必要。設計コストが増す |
| **冪等性が必要** | ネットワーク障害による重複配信が発生しうる。宛先側で `webhook-id` 等による重複排除が必須 |
| **バージョン管理の難しさ** | ペイロード変更が下位互換を壊す可能性。バージョニング戦略（ヘッダー or ボディ）を事前設計する必要がある |
| **標準化の不足** | 業界共通の標準がない。standard-webhooks が統一を試みているが普及途上 |
| **公開エンドポイントのリスク** | コールバック URL はインターネットに公開される。HMAC 検証・TLS・IP 制限の実装が実質必須 |
| **多対多は非効率** | 1:1 または 1:N の構成に向く。多数のサービスが生成するイベントを集約する多:多構成にはブローカー型が適する |

## When to Use

### Web フィードを選ぶ場合

| シナリオ | 理由 |
|---------|------|
| **コンテンツ配信・ブログ更新通知** | 最も典型的なユースケース。低コストで広くコンテンツを届けられる |
| **クラウドサービス・OSS のリリース情報** | サービス状態・新機能・非推奨のお知らせを標準フォーマットで購読できる |
| **セキュリティ勧告の購読** | CVE・脆弱性情報を標準フォーマットで受け取れる |
| **ポッドキャスト・動画チャンネル** | RSS はポッドキャスト配信の de facto フォーマット |
| **リアルタイム性が不要な更新通知** | 数分〜数時間の遅延が許容できる場合 |

### Web フィードが不向きな場合

| シナリオ | 推奨代替 |
|---------|---------|
| ユーザー固有のプッシュ通知 | Webhook（1:1 通知）・WebSocket（リアルタイム） |
| 即時性が必要なアラート | Webhook・WebSocket |
| 大量イベントの非同期処理 | ブローカー型メッセージング（RabbitMQ・Kafka） |

---

### Webhook を選ぶ場合

| シナリオ | 理由 |
|---------|------|
| **サービス間の非同期イベント連携** | 注文作成 → 在庫・物流サービスへの通知など |
| **プロセス自動化** | コードプッシュ → CI/CD 起動・PR マージ → デプロイなど |
| **外部 SaaS との統合** | 決済完了・フォーム送信などのイベントを受信する |
| **長時間非同期処理の完了通知** | 動画変換・ファイルアーカイブ・データ同期完了の通知 |
| **リアルタイム性が必要だが双方向不要** | サーバーが一方向に通知を送れれば十分なユースケース |

### Webhook が不向きな場合

| シナリオ | 推奨代替 |
|---------|---------|
| ブラウザや SPA への通知 | WebSocket・SSE（Server-Sent Events） |
| 多:多の大量メッセージング | ブローカー型（RabbitMQ・Kafka 等。詳細: `STYLE-WEBSOCKET-MESSAGING.md`） |
| 双方向リアルタイム通信 | WebSocket |

## 比較表：Web フィード vs Webhook

| 軸 | Web フィード | Webhook |
|---|-------------|---------|
| 方向 | pull（クライアント主導） | push（ソース主導） |
| 通信種別 | 非同期（ポーリング） | 非同期（コールバック） |
| HTTP メソッド | GET | POST |
| フォーマット | XML（RSS・Atom）・JSON feed | JSON・XML |
| 確認応答 | なし | HTTP レスポンスで確認 |
| 再試行 | クライアント任意 | ソースが責任を持つ |
| 署名検証 | 一般に不要 | HMAC 推奨（実質必須） |
| 実装コスト | 低 | 中 |
| 標準 | Atom = IETF RFC 4287 | 非標準（standard-webhooks 勧告） |

| 特性 | Webフィード | Webhook |
|------|-------------|---------|

| 通信種別 | 非同期 | 非同期 |
| バイナリ対応 | Partial | Yes |
