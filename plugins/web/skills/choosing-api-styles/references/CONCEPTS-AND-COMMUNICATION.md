# API概念と通信モデル

APIとは何か、ネットワーク通信の基本概念（メッセージ・伝送モード・同期/非同期）、
APIライフサイクル、API-first、API-as-a-Product を整理し、スタイル選定の前提知識を提供する。

---

## 1. APIとは

API（Application Programming Interface）の「Interface」とは、エンティティ同士が通信する相互作用点のことを指す。
エンティティはユーザー・システム・組織など多様であり得る。

代表的なインターフェースの例：

- **HID（Human Interface Device）**：リモコンやマウスなど人とデバイスの接点
- **CLI（Command-Line Interface）**：テキスト行でシステムに命令を渡す
- **GUI（Graphical User Interface）**：アイコン操作でシステムと対話する

APIはさまざまなコンピュータシステムに存在する：

- ソフトウェアライブラリ・フレームワーク（OS・ネットワーク機能を抽象化）
- オペレーティングシステム（POSIX のような標準 API でクロスOS互換性を確保）
- **ネットワークベースアプリケーション**（本スキルの主題）

APIは内部の複雑さを隠蔽し、利用者はドキュメントに従って入力を与え出力を受け取る「不透明なボックス」として扱える。

---

## 2. ネットワークベースAPI

**ネットワークベースAPI** とは、ネットワーク（通常はインターネット）越しにソフトウェアが通信するための相互作用点。
Web API はその代表で、HTTP 等の Web テクノロジーを利用してブラウザ・サービス・CLIツール・モバイルアプリ等が通信する。

本スキルでは「API」と言及した場合、原則としてネットワークベースAPIを指す。

クライアント-サーバー構成における各用語：

| 用語 | 説明 |
|------|------|
| Application | クライアント（フロントエンド）とサーバー（バックエンド）の両側に存在 |
| Programming | クライアントがサーバーの機能を「プログラム」する＝呼び出す |
| Interface | サーバーがクライアントへ機能を公開する相互作用点 |

## 3. API通信の概念

### 3-1. メッセージ

**メッセージ** とは、送信者から1人以上の受信者へ送られる、ペイロードとメタデータを含む自己完結したデータ単位。
OSIモデルではアプリケーション層プロトコルのデータ単位に相当する。

代表的なアプリケーション層メッセージの形式：

**HTTP（テキスト形式）**
```
POST /orders HTTP/1.1
Host: api.example.com
Content-Type: application/json
Content-Length: 27

{"item":"book","qty":1}
```

**MQTT PUBLISH パケット（バイナリ構造）**
```
Fixed Header:    PUBLISH ...
Variable Header: sensor/temperature
Payload:         24.8 C
```

XML / JSON / バイナリ（Protobuf / Avro）など多様な形式でシリアライズできる。

---

### 3-2. 伝送モード（Transmission Modes）

伝送モードは通信当事者間でデータをどのように共有するかを記述する。
API選定において「一方向か」「同時双方向か」を判断する基礎となる。

各通信当事者は「送信」「受信」「送受信同時」の3状態のいずれかをとる。

| モード | 方向 | 同時性 | 代表例 |
|--------|------|--------|--------|
| **Simplex（単方向）** | 一方向のみ | — | ラジオ放送・プッシュ型通知 |
| **Half-duplex（半二重）** | 双方向だが非同時 | 片方向ずつ | 無線機（トランシーバー）、HTTP/1.1パイプライン |
| **Full-duplex（全二重）** | 双方向かつ同時 | 同時 | 電話、WebSocket、gRPC双方向ストリーミング |

> **選定への影響**: リアルタイム双方向通信が必要なら Full-duplex（WebSocket / gRPC）を選ぶ。
> イベント通知のみなら Simplex（Webhook / Webフィード）で十分なことが多い。

---

### 3-3. 同期通信と非同期通信

APIスタイルを選ぶ際に最初に問うべき軸のひとつ。

**同期通信（Synchronous）**

- 送信者はレスポンスが返るまでブロックする
- 送受信者が同時に稼働している必要がある
- 例：電話、HTTP の curl クライアントなど

**非同期通信（Asynchronous）**

- 送信者はメッセージを送った後、ブロックせず他のタスクを実行できる
- 送受信者が同時に存在しなくてよい
- 例：メール、JavaScript `fetch()` による HTTP リクエスト、メッセージキュー

> **注意**: 同一のプロトコル（HTTP）でも、クライアントの実装次第で同期・非同期のどちらにもなりうる。

```
同期（ブロッキング）
  Client  ──── Request ────►  Server
          ◄─── Response ────
  （待機中）

非同期（ノンブロッキング）
  Client  ──── Request ────►  Server
          （他の処理を継続）
          ◄─── Response ────（後で受け取る）
```

## 4. APIスタイルの定義

**APIスタイル** とは、APIを設計・実装・公開するためにパターン・慣行・プロトコルを組み合わせたパラダイム。

各スタイルは複数の特性（プロトコル・設計制約・通信種別等）から構成されるが、**支配的な特性の集合** によってスタイルが定義される。

本スキルが対象とする7スタイルの分類：

| スタイル分類 | 代表技術 | 支配的特性 |
|---|---|---|
| Resource（リソース指向） | REST | HTTP + URI でリソースを操作 |
| Query（クエリ指向） | GraphQL | クライアントがデータ形状を指定 |
| RPC | gRPC | リモートプロシージャ呼び出し |
| Web feed | Atom / RSS | 継続更新コンテンツの配信（pull型） |
| Callback | Webhook | イベント発生時のHTTPコールバック（push型） |
| Bidirectional（双方向） | WebSocket | full-duplex でリアルタイム通信 |
| Broker-based（ブローカー型） | RabbitMQ / Kafka | ブローカー経由のメッセージパッシング |

> 詳細な比較表と選定マトリクス → `SELECTION-MATRIX.md`

## 5. APIライフサイクル

APIはソフトウェアであり、SDLC（Software Development Lifecycle）の観点で管理すべき。
以下の7フェーズで構成される（順序は固定でなく相互連携する）：

### 5-1. 計画（Planning）

- APIのスコープ（プライベート / パブリック / パートナー）を確定
- 対象ユーザー（IoT機器 / ブラウザ / モバイル / サービス間）を特定
- 機能要件（FR）・非機能要件（NFR）を定義

**機能要件 vs 非機能要件**

| 種別 | 定義 | 例 |
|------|------|-----|
| 機能要件（FR） | システムが果たすべき具体的な機能・動作 | 「ユーザーが検索クエリを送れる」 |
| 非機能要件（NFR） | システムの品質・制約（"ilities"） | 「応答は1秒以内」「可用性99.9%」 |

計画時の主要な問いかけ：

- APIのスコープは？（プライベート / パブリック / パートナー）
- ユーザーデバイスは？（IoT / ブラウザ / モバイル / AI エージェント）
- 言語・フレームワーク制約は？
- スケール要件は？（想定ユーザー数）
- バージョニング戦略は？
- GDPRやHIPAA等の規制制約は？

### 5-2. 設計（Design）

- APIの機能だけでなく「どのように提供するか」を決定
- サードパーティとの統合・技術選定
- **API-firstアプローチ**の採用可否を決定

### 5-3. 実装（Implementation）

- APIの仕様に基づいて実装
- インターフェースと実装は分離して管理する（Interface vs Implementation）

### 5-4. テスト（Testing）

- 統合テスト・コントラクトテスト・ファジングテスト・E2Eテスト
- セキュリティはテストフェーズだけでなく全フェーズを横断する

### 5-5. デプロイ（Deployment）

- API ゲートウェイ・CI/CD の統合
- Blue/Green・カナリアリリース等の展開戦略

### 5-6. メンテナンス（Maintenance）

- バージョニング・後方互換性・前方互換性の維持
- ドキュメントはメンテナンス期だけでなく全フェーズで更新するのが現実的

### 5-7. 退役（Retirement）

- 廃止通知・サンセット期間の通知
- HTTP `Deprecation` ヘッダー・`Sunset` ヘッダーで機械的に通知できる

```http
Deprecation: Sat, 01 Jan 2026 00:00:00 GMT
Sunset: Sat, 01 Jan 2027 00:00:00 GMT
Link: <https://developer.example.com/api/v2>; rel="successor-version"
```

## 6. API-firstアプローチ

**API-first** とは、実装に先立ってAPIの仕様を作成するアプローチ。
意図したユーザーと合意した後に実装を行うことで、設計の質が高まる。

代表的なAPI仕様フォーマット：

| フォーマット | 用途 |
|---|---|
| OpenAPI (Swagger) | REST API の記述・ドキュメント生成・コード生成 |
| AsyncAPI | 非同期API（Webhook・メッセージング）の記述 |
| Protobuf IDL | gRPC のサービス定義 |

> **利点**: 仕様が先に固まるため、クライアント・サーバーを並行開発できる。
> ドキュメント・テスト・モックの自動生成も可能になる。

## 7. API as a Product

**「APIをプロダクトとして扱う」** とは、単なる統合技術ではなく、他のプロダクトと同じ注意を払ってAPIを設計・管理すること。

プロダクトとして意識すべき観点：

- **開発者体験（DX）**: 第三者開発者のニーズを満たすことでAPI採用率を高める
- **ポータル整備**: 内部APIの発見可能性とドキュメントを Developer Portal で管理
- **使用状況分析**: パワーユーザー・地域別トラフィック・エラー率・レスポンスタイムを把握
- **セキュリティ**: 認証・認可・レートリミット・異常トラフィックのアラート

### API収益化モデル

| モデル | 概要 | 適用例 |
|---|---|---|
| 従量課金（Pay-per-use） | 呼び出し数・データ量で課金 | 地図API・OCR API |
| サブスクリプション | 月額・年額の固定料金 | SaaS 型 API |
| フリーミアム | 基本機能無料・高度機能有料 | 多くの開発者向けAPI |
| トランザクションフィー | 各取引に対して手数料 | 決済API |
| レベニューシェア | 取引収益を開発者と分配 | アフィリエイト・マーケットプレイス |

### パブリック API の注意点

公開APIは **メンテナンス責任** を伴う。廃止手順・通信タイムラインの遵守が必要で、一度公開した後の修正は困難。

---

## 8. プライベート・パブリック・パートナーAPI

| 種別 | 対象 | 特徴 |
|------|------|------|
| **プライベートAPI** | 組織内部のみ | アクセス制限が緩やか。内部連携・マイクロサービス間通信 |
| **パートナーAPI** | 特定パートナー組織 | 認証必須。SLA（Service Level Agreement）を締結することが多い |
| **パブリックAPI** | 外部の不特定多数 | 後方互換性・廃止通知・レートリミットが重要になる |
