# APIインフラ・金融グレードAPI・可観測性・セキュリティテスト

設計とコードレベルの対策だけでは、APIは守り切れない。本ドキュメントは、APIが実際に稼働する基盤（ゲートウェイ・ネットワーク・WAF）の堅牢化、金融グレードAPI（FAPI）による高セキュリティ要件への対応、稼働後の脅威検知を可能にする可観測性、そしてリリース前後のセキュリティテスト戦略を扱う。

---

## セキュアなAPIインフラストラクチャ

### APIゲートウェイ

*APIゲートウェイ*は、API client と API server 群の間に立ち、API govern（公開・廃止・ドキュメント整備・アクセス制御・監視）を一元化するAPI管理ソフトウェアである。複数のAPIを運用する場合ほど効果が大きい。

マイクロサービスにおけるAPIゲートウェイの本来の3つの目的:

| 目的 | 内容 |
|------|------|
| 単一エントリポイントの提供 | サービス／API discoveryを実装し、クライアントが個々のURIを直接知る必要をなくす |
| APIスタイルの変換 | REST・GraphQL・gRPC・非同期API（AsyncAPI/MQTT等）が混在するバックエンドを単一スタイル（例: REST）に統一して公開する |
| backends-for-frontends パターンの実装 | モバイル／Web等クライアント種別ごとに、必要なデータだけを集約・整形して返すゲートウェイを用意し、ラウンドトリップと過剰なデータ露出を削減する |

現代のAPIゲートウェイは、上記に加えて次の機能を提供することが多い。

- **インベントリ管理** --- 廃止すべきなのに稼働し続ける*zombie API*（非推奨だが未撤去のAPI）と、ドキュメント化されず監視もされない*shadow API*（未文書化のエンドポイント）を可視化する。全トラフィックをゲートウェイ経由に強制し、新規エンドポイントの公開を登録制にすることで、これらのリスクを緩和する
- **アクセス制御の一元適用** --- JWT検証・mTLS等の認証チェックをアウトオブザボックスでサポート
- **リクエスト/レスポンスのバリデーション** --- API仕様（OpenAPI等）を読み込み、設計時に定めた制約をランタイムで強制する
- **セキュアなレスポンス設定の適用** --- `Access-Control-Allow-Origin`（CORS）、`Strict-Transport-Security`（HSTS）、`Content-Security-Policy`（CSP）等のセキュアヘッダを一律に注入する
- **レートリミット・可観測性** --- 全トラフィックの通過点であることを活かし、アクセスログ集計や利用状況分析を標準機能として提供する

APIゲートウェイの選定は、一度導入すると運用が密結合するため後戻りが難しい意思決定になりやすい。Kong、Apache APISIX等のOSS製品や、主要クラウドプロバイダのマネージドAPIゲートウェイを含め、複数の選択肢を実際に評価してから決定することが望ましい。

### セキュアなネットワークトポロジー

*ネットワークトポロジー*（ネットワークの構造）は、APIのセキュリティ姿勢に直接影響する。ネットワークをセグメント（サブネット）に分割し、インターネットアクセスを持つ*パブリックサブネット*と持たない*プライベートサブネット*に分けるのが基本戦略である。

| 用語 | 定義 |
|------|------|
| DMZ（非武装地帯／perimeter network） | インターネットに露出するセグメント。APIゲートウェイ・リバースプロキシ・WAF等の堅牢化されたコンポーネントのみを配置し、サービス本体は配置しない |
| bastion server（jump box） | DMZ上に置く特別に堅牢化されたサーバ。SSH（ポート22）等、限定された用途・許可リストのIPからのインバウンドのみを許可し、プライベートサブネットへの踏み台となる |
| reverse proxy | クライアントリクエストをサーバへ転送するサロゲートサーバ。SSL終端、悪意あるIPのブロック、複数バックエンドへの負荷分散等に使う |

データベースのような機密性の高いコンポーネントはプライベートサブネットに置き、外部からの直接アクセスを許可しない。外部アクセスが必要な場合はbastion server経由に限定する。公開APIはDMZからAPIゲートウェイ経由でプライベートサブネットへトラフィックを転送する構成にすることで、ゼロトラストの原則（宣言していないエンドポイントは到達不能）を満たし、攻撃対象領域を常に可視化できる。

さらに、機密性の高いサービス同士を別ネットワークに分離する*マイクロセグメンテーション*は、横方向移動（lateral movement）を防ぐ。たとえば医療系サービスで、予約管理サービスと診療記録サービスを別ネットワークに分離しておけば、前者が侵害されても後者のデータには到達できず、コンプライアンス監査のスコープも限定できる。

### レイヤー3-6攻撃対策

アプリケーション層（レイヤー7）を狙う攻撃（SQLインジェクション、ファジング、mass assignment等）は本スキルの他の参考資料で扱うが、OSIモデルの下位層（ネットワーク～プレゼンテーション層、レイヤー3-6）にも独立した脅威が存在する。

| 攻撃 | 概要 |
|------|------|
| DoS/DDoS（volumetric） | ICMP flood、SYN/UDP flood、DNS/UDPアンプ攻撃、IPフラグメンテーション等で大量トラフィックを送りつけ、正規リクエストへの応答を不能にする |
| HTTP/2 rapid reset | HTTP/2の多重化ストリームを最大数まで開いては即座にリセットし、ストリーム上限を超えずに大量リクエストを送り込む。2023年に大規模DDoS事例として実際に観測された |
| ポートスキャン | 開いているポートを走査し、そのポートで動くサービス（DB、CI/CDツール等）を特定して的を絞った攻撃につなげる偵察行為 |
| IPスプーフィング/マスキング | IPヘッダを偽装し、または他のノード（VPN・Tor等）経由でリクエストを送ることで、IPベースのブロックを回避する |

対策の基本は、サーバのハードニング（不要なポートの閉鎖、不要なプロトコル/サービスの無効化、特権ユーザーの制限、ファイアウォールルールの厳格化）と、専門のDDoS対策サービスの利用である。レイヤー7も含めた全層の悪意あるトラフィックから保護する製品がWAFである。

### WAFによる保護

*WAF（Web Application Firewall）*は、インジェクションやXSSといったアプリケーション層攻撃からWebアプリケーションを保護するファイアウォールで、多くの場合リバースプロキシとしてAPIの手前に配置される。現代のWAFはゼロデイ脆弱性やレイヤー3-6攻撃への対策も提供する。

WAFの検知方式には2種類ある。

- **パターンマッチング型** --- インジェクション文字列に典型的なキーワード列を検出する。パターンが甘いと回避されやすく（例: `' or 1=1--` を `'+o//r+1=1--` のように記号でマスクして正規化処理を悪用する回避手口がある）、厳しすぎると正規リクエスト（インジェクションの解説記事等）まで誤検知する
- **確率型** --- リクエスト内容全体に対してスコアを算出し、悪意度合いを文脈から判定する

現代のWAFはAPI仕様を与えることでスキーマバリデーションを行える製品もある。ただし*ゼロデイ脆弱性*（発見から修正までの猶予がない脆弱性。例: OpenSSLのHeartbleedのように、公表から実際の全面対応まで期間を要し、長期間にわたり脆弱なサーバが残存した事例がある）への対応を含め、WAFは強力な第一防衛線ではあるが万能ではない。WAF導入を理由にセキュリティ強化を怠ることが、かえって脆弱なAPIを増やす一因になる。

## 金融グレードAPI（FAPI）

### オープンバンキングとFAPI

*オープンバンキング（open banking）*は、ユーザーの同意のもとで銀行口座データへのアクセスをサードパーティAPI経由で安全に開放する仕組みである。従来型の*スクリーンスクレイピング*（ユーザーの認証情報を預かって銀行サイトへログインし、データを取得する手法）は、認証情報をサードパーティに渡す時点で高リスクであり、銀行側も正規アクセスと悪意あるスクレイピングを区別できない。EUのPSD2（2019年施行）等の規制がオープンバンキングを後押しした。

FAPI（Financial-grade API）は、OpenID Foundationのワーキンググループが策定する、高セキュリティ要件を持つAPIのための仕様群である。FAPI 1.0は金融APIに焦点を当てていたが、FAPI 2.0は金融に限らず、医療・法務・政府・防衛等、機密性の高い分野全般に適用範囲を広げている。FAPI 2.0はOAuth 2.0（RFC 6749）、OAuth 2.0のベストプラクティス（RFC 9700）、OpenID Connectを基盤に、これらを厳格かつ限定的に利用する。

### FAPIの攻撃者モデル

FAPI 2.0の攻撃者モデルは、個別の攻撃手法ではなく、満たすべき高レベルのセキュリティ目標を3つのカテゴリで定義する。

| 目標 | 意味 |
|------|------|
| セキュアな認可 | ユーザーは自分がアクセスを許可されたリソースにのみアクセスできる。他者のアクセストークンを取得・使用されない |
| セキュアな認証 | ユーザーは他のユーザーの身元でログインできない。ID tokenは正当な所有者にのみ結びつく |
| セッション完全性 | 認可プロセス中にやり取りされるメッセージが改ざんされない（例: 認可エンドポイントへのCSRF攻撃でユーザーの認可コードを窃取される手口を防ぐ） |

### FAPI 2.0セキュリティプロファイル

FAPI 2.0の主要エージェントは、client（アプリ）・resource owner（ユーザー）・user agent（ブラウザ/モバイルアプリ）・authorization server・resource serverの5者である。攻撃者モデルの3目標を満たすため、FAPI 2.0は以下を要求する。

- **confidential clientのみ許可** --- public client（ブラウザ上で動きコードが露出する）はcredentialを保持できないため禁止し、サーバ側で動くconfidential clientのみを許可する。クライアント認証は mTLS または `private_key_jwt`（クライアント専用の署名鍵でJWTに署名する方式）で行う
- **JAR + PARの利用** --- 認可リクエストパラメータをURLに平文で含めると、盗聴・改ざん・CSRF攻撃の温床になる。JWT-secured authorization request（JAR）でパラメータをJWTにラップし、pushed authorization request（PAR）でそのJWTを事前にサーバへプッシュしてURIに変換する
- **open redirectの禁止** --- `redirect_uri` を動的に指定させず、事前登録済みURIのみを許可する
- **sender-constrained tokenの利用** --- 盗まれたアクセストークンの悪用を防ぐため、mTLSによるcertificate-bound tokenまたはDPoP tokenを使う
- **refresh token rotationの非推奨** --- confidential client前提かつバックチャネルが認証済みのため、追加の複雑性を持ち込むrotationの恩恵が薄い
- **署名アルゴリズムの制限** --- algorithm confusion攻撃やRS256の署名偽造リスクを避けるため、JWTの署名はPS256・ES256・EdDSA（Ed25519）のみを許可し、それ以外での署名は拒否する

### 認可リクエストの保護とメッセージ署名

**JAR（JWT-secured authorization request）**は、認可リクエストのパラメータ（`client_id`・`redirect_uri`・`response_type`・`scope`・`state`・`nonce`等）を*request object*と呼ばれる署名済みJWTに包む。request objectはクライアント固有の秘密鍵で署名され、改ざん・なりすましを防ぐ。送信方法には、JWTを直接クエリパラメータ `request` に載せる**by value**方式と、事前にサーバへ登録したURIを `request_uri` パラメータに載せる**by reference**方式がある。

**PAR（pushed authorization request、RFC 9126）**は、クライアントが認可リクエストパラメータ（またはJARの request object）を認証済み `POST` でauthorization serverのPARエンドポイントへ事前にプッシュし、短命・one-time useのrequest URIと引き換える方式。これによりURLにセンシティブな値が一切露出しなくなる。JARとPARを組み合わせることで、クライアント認証による正当性の担保と、URL経由の漏洩・改ざんリスクの排除を同時に達成する。

FAPI 2.0の**message signing**プロファイルは、これに加えて*nonrepudiation*（否認防止: 特定ユーザーが特定操作を行ったことを証明できる能力）を実現する高度なプロファイルで、金融・医療・法務等、ユーザー操作が法的・金銭的リスクを伴う領域で重要となる。

| 要素 | 役割 |
|------|------|
| PAR | 認可リクエストの真正性を証明する |
| JARM（JWT secured authorization response mode） | authorization serverが認可コードを署名済みJWTで返し、レスポンス改ざん・中間者攻撃を防ぐ |
| 署名付きtoken introspection | token introspectionエンドポイント（RFC 7662）を認証必須にし、レスポンスを署名済みJWT（RFC 9701）で包むことで、トークン検証結果の真正性を保証する |
| ID tokenの署名 | 発行されるID tokenに署名を必須とし、クライアントが身元情報の真正性を検証できるようにする |

これらの署名の連鎖により、認可プロセスの乗っ取りを実質的に不可能にする。発行後のトークン窃取に対しては、引き続きsender-constrained token（証明書バインドまたはDPoP）で対処する。

## APIセキュリティの可観測性

### ログ・トレース・メトリクス

*API可観測性*とは、APIから継続的にデータを生成・収集・分析する営みである。閾値ベースの従来型モニタリング（リクエスト数・レイテンシ・CPU使用率等の定義済みメトリクス監視）は、あらかじめ想定した問題の検知には強いが、「あるユーザーのデータが不正アクセス・改変されたようだ」といった非定型な問題には力不足になる。可観測性は、システムの出力（*テレメトリ*: ログ・トレース・メトリクス。OpenTelemetry(OTel)ではこれらを*シグナル*と呼ぶ）からシステムの内部状態を推定するアプローチであり、平均復旧時間（MTTR）の短縮と、規制対応で必要となる監査証跡（audit trail）の両方に寄与する。

| シグナル | 役割 |
|---------|------|
| ログ | 個別イベント（リクエスト受信・エラー発生等）の記録。単体では複数ログの関連付けが難しい |
| トレース | 同一リクエスト処理内の複数ログをtrace IDで関連付け、分散システム全体でのリクエストのライフサイクルを追跡する。サービス単体のエラーログだけでは根本原因（例: 上流UIが不正なデータを送っていた）に辿り着けないケースで威力を発揮する |
| メトリクス | リクエスト数・レイテンシ等の集計値。OSレベルの低レベル指標と、API利用状況を示す高レベル指標の双方を含み、外れ値（異常な量のリクエストを送るユーザー等）の検知に使う |

*インスツルメンテーション*（telemetryを生成・収集できるようにアプリを準備する作業）には、Webフレームワークが標準で出すリクエスト/レスポンスログや、OTel等が提供する**自動インスツルメンテーション**（コード変更なしで代表的なシグナルを収集開始できる仕組み）がある。ただし自動インスツルメンテーションはアプリの外縁（HTTPリクエストの開始・終了等）のイベントしか捉えられないため、ビジネスロジック層の詳細（外部依存呼び出しの所要時間、特定のデータ操作等）を捉えるには、tracerオブジェクトから明示的にカスタムspanを生成し、属性（ユーザーID、リクエスト内容等）を付与する手動インスツルメンテーションが必要になる。ログの属性の組み合わせがどれだけユニークかを示す**cardinality**を高く保つことは、調査時にログを見つけやすくする。

### 攻撃検知(入力ベース・エンドポイント濫用)

**入力ベース攻撃の検知**では、リクエストのヘッダ・ボディ・URL全体をできるだけ詳細にログへ残すことが、脅威の再現・調査に直結する。ただし決済情報や個人情報等の機密フィールドをそのまま記録すると、ログ自体が規制対象データとなり、アクセス制御・バックアップ等の追加コストと漏洩リスクを負う。機密フィールドはマスク・除外した上でログに残すのが基本だが、その場合「攻撃がまさにそのマスクされたフィールドに仕込まれていたら検知できない」というトレードオフが残る。WAF等の高度な検知ツールと組み合わせ、悪意ある入力を検知した際にはその判定結果自体をログへ注釈として残す運用が現実的である。ログの出力先はファイルよりも標準出力（stdout）へのストリーム出力とし、ログルーターやサイドカーコンテナで収集する構成にするとパフォーマンス影響を抑えやすい。

**エンドポイント濫用攻撃の検知**は単発のログでは捉えられない。実際に、2021年にはあるフィットネス機器メーカーのAPIで、ユーザー検索エンドポイント（`GET /users/search/{username}`のようなユーザー名検索API）を繰り返し呼び出してユーザーIDを収集し（*オブジェクト列挙*）、それを別のエンドポイント（詳細取得API）に投入して個人情報を大量取得できる脆弱性が発見・責任開示された。この種の攻撃の痕跡は「同一エンドポイントへの異常な反復呼び出し」というパターンにしか現れない。対策として、エンドポイント単位のリクエスト数をメトリクスとして収集し、平均値からの乖離をアラート条件にする。ただしアラート発火直後に自動遮断するのではなく、まず異常かどうかを見極めることが望ましい。さらに、IPアドレス・デバイスフィンガープリント・（認証済みなら）ユーザーIDをログに含めておけば、脅威が特定のアクター・地域に起因するかを切り分けられ、全ユーザー一律のレート制限ではなく、特定アクター・地域への的を絞った制限やブロックが可能になる。

## APIセキュリティテスト

### テスト戦略設計

APIのセキュリティ検証には、ペネトレーションテスト（専門家による高品質だが低頻度・高コストな検証）と自動化されたAPIセキュリティテストツール（高速だがビジネスロジックの欠陥は見逃しやすい）の二つのアプローチがあり、両者は補完関係にある。

| アプローチ | 強み | 弱み |
|-----------|------|------|
| ペネトレーションテスト | 専門家によるAPI固有の高品質な検証 | 高コスト・低頻度・スケールしない |
| 自動テストツール | 高速・継続的に実行可能 | ビジネスロジックの欠陥を見逃しやすい。ツールの品質と設定内容に精度が依存する |

効果的なテスト戦略は、脅威モデルを起点に、セットアップの手間が小さく価値の高いテスト（低い労力で大きなリスクを潰せるもの）から着手する。たとえば、あるサービスで「コンテンツへの不正アクセス」「割引/クーポンコードの濫用」「管理者ポータル経由の個人データ不正アクセス」という3つの脅威を識別した場合、それぞれ認証チェック・不正利用検知・RBAC検証という異なる粒度のテストに対応させる。

### 設計セキュリティ欠陥の発見

最も基本的なテストは、API仕様そのものの検証である。API linter（例: Spectralに代表されるOSSツールとそのOWASPルールセットプラグイン）でOpenAPI仕様を解析すると、次のようなカテゴリの指摘が得られる。

| カテゴリ | 指摘内容の例 |
|---------|------------|
| 認証・アクセス制御 | 書き込み操作にセキュリティスキームが設定されていない。管理者向けエンドポイントが非管理者と同一のセキュリティ要件になっている |
| 入力制約 | 文字列フィールドに`maxLength`/`enum`/`const`の指定がない。整数フィールドに`minimum`/`maximum`の指定がない。配列フィールドに`maxItems`の指定がない |
| API利用性 | `401`/`403`/`500`等のエラーレスポンスやレートリミットヘッダがドキュメント化されていない |

入力制約の欠如は、悪意あるペイロード・大量ページネーション要求・巨大配列レスポンス等でサーバに負荷をかける攻撃の余地を残す。ただし全フィールドを厳密に制約できるとは限らない（例: 氏名フィールドの値そのものは制約しづらい）ため、設計時の制約付けだけに頼らず、可観測性による実行時検知と組み合わせる必要がある。

### ファジングと契約テスト

API仕様と実装が乖離する現象を*API drift*と呼ぶ。仕様には存在するが実装されていないエンドポイントは*orphan endpoint*、逆に実装にはあるが仕様に存在しないエンドポイントは*shadow endpoint*と呼ばれる。API driftは開発者体験を損なうだけでなく、仕様上の制約（例: ページサイズの上限）が実装に反映されていない場合、セキュリティ上の抜け道にもなる。

*契約テスト（contract testing）*は、API仕様を「契約」とみなし、実装がそれに従っているかを検証する手法である。Schemathesis・Dredd・Microcks等のOSSツールが、仕様から自動でテストケースを生成する。この生成手法は*ファジング*（不正な形式の入力をサーバへ送り、挙動を観察する手法）に基づいており、想定外のクラッシュや挙動を通じて設計時に見落とした脆弱性を発見できる。たとえば、文字列フィールドにNULバイト（` `のようなヌル文字表現）を注入するテストケースが、データ検証層を素通りしてデータベース層でエラーを起こすといった*NULバイトインジェクション*（CWE-158）が典型例である。

### アクセス制御・ビジネスフローのテスト自動化

APIのビジネス層を検証するには、Spectralや契約テストツールでは届かない単体テストが必要になる。まず着手すべきは、次の4種類のアクセス制御テストである。

| テスト種別 | 検証内容 | 期待するステータスコード |
|-----------|---------|------------------------|
| broken authentication test | 認証情報なしのリクエストが拒否されるか | `401`（認証情報が無効・欠落） |
| BOLA test | 他ユーザーのリソースへのアクセスが拒否されるか | `403`（認証済みだが権限なし） |
| RBAC/権限昇格 test | 一般ユーザーが管理者専用エンドポイントへアクセスできないか | `403` |
| ABAC test | 特定の状態・属性（例: 本人確認済みかどうか）を満たさないユーザーの操作が拒否されるか | `403` |

これらのテストは実装コストが低いにもかかわらず、致命的な脆弱性を検出しやすい「ローハンギングフルーツ」である。SQLインジェクションやXSSと異なり、明確な足跡を残さない**ビジネスフロー脆弱性**（例: 割引/クーポンコードの再利用、他ユーザーのクーポンコードの流用）は、WAF等の汎用ツールでは検知できないため、脅威モデルに基づいた専用の単体テストが不可欠になる。典型的なテストパターンは次のとおり。

- 同一クーポンコードで同一操作を2回実行し、1回目は成功（`201`）、2回目は競合（`409 Conflict`）を期待する
- 他ユーザーが発行されたクーポンコードを使おうとした場合に`403`を期待する

これらのテストはCI上で継続的に実行し、コード変更のたびに回帰を検知できるようにする。

## 実務チェックリスト

セキュリティは開発プロセスの最後に付け足すものではなく、設計の最初期からシフトレフトで組み込む。以下は観点別のチェックリストである。

**設計**

- [ ] OpenAPIやSDL等のインターフェース記述言語でAPI設計を一元化しているか
- [ ] API linter（Spectral等）で設計の脆弱性を検査しているか
- [ ] すべての入力スキーマ・パラメータを制約しているか
- [ ] 設計を脅威モデリングしているか
- [ ] 期待するユーザーフローを標準フォーマットで記述し、フロー自体を脅威モデリングしているか

**実装**

- [ ] 堅牢なAPI開発フレームワークを使っているか
- [ ] セキュアな実装パターン（DTO、ORM、パラメータ化クエリ等）を使っているか
- [ ] サードパーティ連携を含むすべての入出力データを検証しているか
- [ ] APIファザーで仕様と実装の一致を検証しているか
- [ ] アウトバウンドプロキシで外部依存の失敗を安全にハンドリングしているか
- [ ] 脅威モデルに対応する単体テストを書いているか

**認証・認可**

- [ ] 堅牢なIDaaS（自社ホストする場合も含む）を利用しているか
- [ ] ユースケースに適したOAuthフローを使っているか
- [ ] 標準的なJWTを堅牢な署名アルゴリズム（PS256等）で使い、署名鍵を定期的にローテーションしているか
- [ ] ID tokenをアクセストークンとして使っていないか
- [ ] 高感度データを扱うAPIでsender-constrained tokenを使っているか
- [ ] すべてのエンドポイントでデフォルト認証必須とし、認証不要なものだけ明示しているか
- [ ] ユーザーロールを明確にし、RBACを全エンドポイントで一貫適用しているか

**インフラストラクチャ**

- [ ] APIゲートウェイを単一エントリポイントとし、インベントリと攻撃対象領域を管理しているか
- [ ] WAFを導入しているか
- [ ] サービス間の不要な通信経路をネットワークセグメンテーションで遮断しているか
- [ ] エンドポイント・フローの感度に応じてレートリミットを調整しているか

**可観測性**

- [ ] すべての重要イベントについてログ・トレース・メトリクスを生成しているか
- [ ] セッションID・ユーザーアカウント・地理情報・IP・デバイスフィンガープリント等、リクエストの出所を追跡しているか
- [ ] ユーザー行動を分析し、脅威検知モデルとレートリミットポリシーに反映しているか
- [ ] ビジネスフロー攻撃をランタイムで検知する仕組みを導入しているか
