# secure-by-design: 設計段階でAPIを堅牢にする

API のセキュリティ問題には大きく分けて2種類ある。実装レベルの脆弱性（SQLインジェクションのようなコーディングミス）は実行時のパッチや修正で対処できるが、**設計に組み込まれた脆弱性（vulnerable by design）**は設計をやり直さない限り根本的に解消できない。予測可能な識別子、無制約のページネーション、無制約の入力値などがこれに当たる。攻撃者が送るリクエストは「一見正当」に見えるため、WAF やレート制限だけでは検知できず、認証・認可の突破やトークン改ざんのような分かりやすい攻撃よりも見逃されやすい。だからこそ、API 開発サイクルの初期段階（設計フェーズ）でこれらのリスクを潰し込む「シフトレフト」の発想が欠かせない。

本ドキュメントでは、設計フェーズで対処すべき6つの脆弱性カテゴリ——予測可能な識別子・無制約の入力・柔軟すぎるスキーマ・サーバー側プロパティの露出・安全でないユーザーフロー——を扱う。

## 1. 脆弱なAPI設計の兆候

> **定義**: 「vulnerable by design」とは、API が設計そのものに起因して持つ脆弱性を指す。実装のバグではなく設計判断の結果であるため、実行時の場当たり的な対処ではなく設計そのものを見直さない限り解消できない。

攻撃者の多くは、レート制限のしきい値を意識的に下回りながら、正規ユーザーのように振る舞う。認証を経た正規のリクエストに偽装するため、異常検知システムや WAF をすり抜けやすい。設計上の欠陥は、こうした「正規に見える」攻撃の温床になる。

| 兆候 | 典型的な結果 |
|------|------------|
| 予測可能な識別子（連番IDなど） | リソース列挙・BOLA（Broken Object Level Authorization） |
| 無制約のページネーション | DoS・過剰な負荷 |
| 無制約のユーザー入力 | SQLインジェクション・スキーマ列挙・大量ペイロード攻撃 |
| 柔軟すぎるスキーマ（optional/additional properties） | データ破損・マスアサインメント |
| 入出力共用スキーマ | サーバー側プロパティの改ざん |
| 業務フローの未強制 | スキャルピング等の自動化された不正利用 |

ページネーションを例に、「設計の問題は設計でしか直せない」ことを確認する。以下は `page`・`perPage` に上限・下限を設けていない脆弱な例である。

```yaml
paths:
  /items:
    get:
      parameters:
        - name: page
          in: query
          schema: { type: integer }
        - name: perPage
          in: query
          schema: { type: integer }
```

`perPage=100000000` のような値を許してしまい、少数の同時リクエストだけでデータベースに過大な負荷をかけられる。実装側で「大きすぎる値はデフォルト値に丸める」ような対処も可能だが、それはセキュリティ判断を個々の実装者の裁量に委ねることになり、時間的プレッシャー下では見落とされやすい。設計時点で `minimum`/`maximum` を明示すれば、契約ベースのテストやファジングツールで実装がスキーマ通りに動くかを機械的に検証できるようになる。

```yaml
paths:
  /items:
    get:
      parameters:
        - name: page
          in: query
          schema: { type: integer, minimum: 1, maximum: 100 }
        - name: perPage
          in: query
          schema: { type: integer, minimum: 10, maximum: 100 }
```

`page` に上限を設けるかどうかは要件次第（無限スクロールを許すなら上限なし、全件スクレイピングを防ぎたいなら上限あり）だが、いずれにせよ**意図を明示すること**が重要である。

## 2. 予測可能な識別子の回避

ほとんどの API はリソース指向であり、各リソースに識別子（ID）を割り当てる。ID がデータベースの連番（auto-increment integer）をそのまま URI に露出していると、攻撃者は `1`, `2`, `3`...と ID を変えながらリクエストを送るだけで、他人のリソースを探索できてしまう（**リソース列挙**）。認可チェックが甘い場合、これは容易に BOLA（他人のリソースへの不正アクセス）に直結する。

連番 ID 自体はデータベースの主キーとして効率的で扱いやすく、問題ない。問題は、それを**そのまま API に露出すること**である。

| 対策 | 内容 |
|------|------|
| 非連番の識別子を使う | UUID/GUID（特に UUIDv4）を主キーまたは公開用IDとして採用する |
| サロゲート公開ID | 内部的には連番IDを保持しつつ、API利用者には別途発行したUUID等を公開し、内部結合には連番IDを使う |
| ID変換 | 連番IDから予測不能な短縮文字列を生成するエンコーディング手法を使い、元の値へ逆算できないようにする |

> **注意**: 予測不能な識別子にしても BOLA が根絶されるわけではない。応答時間の差・ステータスコードの違い・存在有無で異なるエラーメッセージなど、識別子以外の手がかりからもリソースの存在を推測できる。識別子の非予測化はあくまで探索コストを引き上げる対策であり、**すべてのリソースエンドポイントに堅牢な認可チェックとレート制限を適用すること**とセットで機能する。

## 3. 制約のないユーザー入力への対処

*無制約のユーザー入力* とは、クエリ/パスパラメータ・リクエストボディ・ヘッダーなど、ユーザーが触れられるあらゆるパラメータに任意の値を挿入できる状態を指す。読み取り操作（ページネーション、注文IDの指定、位置情報の指定など）も入力を受け取る点では書き込み操作と同様にリスクがある。

多くの API は Web/モバイルアプリの背後で動いており、入力の制約をフロントエンド側に委ねがちである。これは「フロントエンドが正しい値しか送らないから安全」という誤った安心感を生む。API に直接リクエストを送れる攻撃者にとって、フロントエンドの制約は存在しないのと同じである。

### よくある無制約入力パターン

| 入力箇所 | 脆弱な例 | 攻撃パターン | 対策 |
|---------|---------|------------|------|
| 数値フィールド（評価値等） | `rating: integer`（範囲制約なし） | 評価値を異常値にして集計を操作 | `minimum`/`maximum` で範囲を制約 |
| 文字列フィールド（本文等） | `review: string`（長さ制約なし） | 巨大な文字列を大量投入しリソース枯渇（大量ペイロード攻撃）を誘発 | `maxLength` で長さを制約 |
| 自由記述のクエリパラメータ | `sortBy: string`（値の制約なし） | 存在しないカラム名を試行しレスポンスの差異からスキーマを列挙（スキーマ列挙攻撃） | 許容値が有限なら `enum` で列挙値に限定 |
| 検索フィルタ等の自由記述パラメータ | `filter: string` | パラメータ化されていないクエリに対する SQL インジェクション | ビジネス要件上どうしても自由記述を許す場合は実行時に対処（パラメータ化クエリの徹底・WAF） |

`sortBy` のように許容される値が有限であれば、列挙型で制約するのが最も確実である。

```yaml
- name: sortBy
  in: query
  schema:
    type: string
    enum: [price, popularity, name]
```

一方、`filter` のように利用者が任意のキーワードで検索できることが業務要件そのものである場合、スキーマレベルでの制約は現実的でない。この種の「意図的に脆弱な設計」は正当なケースであり、リスクは実行時に緩和する。具体的には、すべてのデータベースクエリをパラメータ化し、SQL インジェクションの実行経路そのものを遮断する。WAF は保険的な追加防御として有効だが、パラメータ化クエリの代替にはならない。

## 4. 柔軟すぎるスキーマの危険性

*柔軟なスキーマ* とは、プロパティの組み合わせを自由に構成できてしまうスキーマを指す。原因は主に2つ: ①プロパティを任意（optional）にしすぎている、②未宣言のプロパティ（additional properties）を許可している。

### 4.1 オプショナルプロパティによるデータ破損リスク

似た構造を持つ複数のデータモデル（例: デジタル商品と物理商品、会員種別Aと会員種別Bなど）を、共通プロパティを流用して1つのスキーマにまとめ、差分をすべて optional にしてしまうことがある。この設計では、本来必須であるはずのモデル固有プロパティを欠いたレコードが作成可能になり、そのレコードを後続処理（表示・集計・連携）が読み込んだ時点で不整合や処理エラーを引き起こす。

対策は、**モデルごとに別スキーマを用意すること**である。共通プロパティの重複を避けたい場合は、`allOf` によるコンポジションで共通部分を継承させる。

### 4.2 追加プロパティ（マスアサインメント）のリスク

JSON Schema は既定で「未宣言のプロパティ」を許可する。この既定動作に気づかず `additionalProperties` を明示しないままにしておくと、スキーマに定義されていない値を含むリクエストもすべて有効なリクエストとして受理されてしまう。

とりわけ危険なのは、レスポンス側のスキーマに存在するサーバー管理プロパティ（`id`・`status` 等）の名前を、攻撃者がリクエストボディに含めてしまうケースである。リクエスト側スキーマが未宣言プロパティを禁止していなければ、そのプロパティはそのままモデルに書き込まれてしまう可能性がある（**マスアサインメント攻撃**）。

対策は単純で、モデルを厳格化することである。

| キーワード | 適用範囲 | 用途 |
|-----------|---------|------|
| `additionalProperties: false` | 単一モデルのローカルスコープ | コンポジションを使わないシンプルなモデルで未宣言プロパティを禁止 |
| `unevaluatedProperties: false` | コンポジション（`allOf` 等）をまたぐスコープ全体 | 継承・合成されたモデルで未宣言プロパティを禁止（`additionalProperties` はコンポジション内で矛盾を起こすため使えない） |

## 5. サーバー側プロパティの露出回避

`id`・`status`・作成日時のようなサーバー側プロパティは、サーバーだけが書き込むべき値であり、ユーザー入力として受け取ってはならない。ところが、リクエスト用モデルとレスポンス用モデルを「同じスキーマの使い回し」で表現すると、この境界が簡単に崩れる。

典型例として、決済リソースを考える。作成時に必要なのは金額と通貨だけだが、レスポンスには識別子とステータスが含まれる。この2つを同じスキーマで表現し、サーバー側プロパティを optional にして共用してしまうと、リクエストボディに直接 `"status": "completed"` のような値を含めるだけでステータスを不正に書き換えられてしまう。

secure-by-design な設計では、**入力用モデルと出力用モデルを分離**し、入力用モデルは必要な項目だけを宣言したうえで `additionalProperties: false` を設定する（データ転送オブジェクト／DTOパターン）。

```yaml
components:
  schemas:
    CreateOrderInput:        # リクエスト専用: ユーザーが指定してよい項目だけを宣言
      type: object
      required: [productId, quantity]
      properties:
        productId:
          type: string
          format: uuid
        quantity:
          type: integer
          minimum: 1
          maximum: 20
      additionalProperties: false   # id/status 等サーバー側プロパティの注入をここで拒否する

    Order:                    # レスポンス専用: サーバー管理プロパティを含む完全な表現
      type: object
      properties:
        id:
          type: string
          format: uuid
        status:
          type: string
          enum: [pending, paid, shipped, cancelled]
        productId:
          type: string
          format: uuid
        quantity:
          type: integer
```

`CreateOrderInput` には `id`/`status` が存在しないうえ `additionalProperties: false` により未宣言プロパティも拒否されるため、攻撃者がリクエストボディに `status` を紛れ込ませても弾かれる。入出力モデルを分離する設計は、4節のマスアサインメント対策とあわせて機能する。

## 6. 安全なユーザーフロー設計

単体の API 操作だけでなく、複数の操作が連なった「ユーザーフロー／業務フロー」（商品を探す→カートに入れる→決済する、のような一連の手順）の設計もセキュリティに直結する。API 攻撃のうち機微な業務フローを狙うものの割合は年々増加傾向にあり、業界的にも注目度の高い領域である。

Web/モバイルアプリの UI は、ユーザーを意図した手順に沿って誘導する。しかし API に直接アクセスできる攻撃者は、UI が強制する手順を飛び越えられる。たとえば「商品ページを見る→カートに入れる→決済する」というフローのうち、商品IDさえ分かっていれば「カートに入れる」から始めることができてしまう。このような手順の飛び越えは、在庫の一括買い占め（スキャルピング）のような自動化された不正利用を可能にする。

安全なユーザーフローを設計するための実行項目は次の2つに集約できる。

| 実行項目 | 内容 |
|---------|------|
| フローをAPIで強制する | 想定する手順（各ステップのオペレーションIDと前後関係）を仕様として明文化し、後続ステップは前段ステップの完了を要求する |
| ステップ間のレイテンシを追跡する | 人間はステップ間に数秒を要するが、ボットはミリ秒単位で全ステップを完走する。異常な速さを検知シグナルとして扱う |

具体的な実装パターン:

- **フローの形式的な記述**: OpenAPI Initiative が策定した Arazzo Specification のような、複数のオペレーションにまたがる手順を記述する仕様を使うと、期待されるフローを機械可読な形で定義できる。
- **認証の要求**: 機微な業務フローでは全ステップに認証を必須にする。これにより匿名ボットからの攻撃を排除でき、ユーザー単位での行動追跡も可能になる。
- **フロートラッカー**: セッションベースのトラッキング機構（例: フロー追跡用クッキーや状態トークン）で、ユーザーがどのステップまで到達したかを記録する。「前段ステップを経ずに後段ステップへ到達していないか」を判定条件にする。
- **逸脱の許容**: 正規ユーザーは想定フローから多少逸脱する（無関係なページを見てから戻ってくる等）。フローの強制は「順序の逸脱を一切許さない」という意味ではなく、「前段ステップを経ずに後段へ到達すること」を防ぐという意味である。過度に厳格な強制は正規ユーザーの体験を損なう。
- **追加の防御層**: 特に機微なフローのクリティカルなステップに CAPTCHA を追加したり、mTLS やメッセージ署名のような否認防止（non-repudiation）手法を組み合わせたりすることで、防御を多層化できる。

## 設計チェックリスト

- [ ] ページネーション・数値パラメータに `minimum`/`maximum` を設定したか
- [ ] 文字列パラメータに `maxLength` を設定したか
- [ ] 値の集合が有限なパラメータ（ソート順など)は `enum` で制約したか
- [ ] 自由記述を許すパラメータ（検索フィルタ等）はパラメータ化クエリで実行時に保護しているか
- [ ] リソース識別子に連番整数をそのまま露出していないか（UUID/サロゲートIDへの置き換えを検討したか）
- [ ] 異なるデータモデルを同一スキーマで表現していないか（モデルごとにスキーマを分離したか）
- [ ] スキーマに `additionalProperties`/`unevaluatedProperties: false` を設定し未宣言プロパティを拒否しているか
- [ ] リクエスト用スキーマとレスポンス用スキーマを分離し、サーバー側プロパティ（id/status等）が入力側に存在しないか
- [ ] 複数ステップから成る機微な業務フローの手順を明文化したか
- [ ] 機微な業務フローに認証・ステップ順序チェック・レイテンシ異常検知を組み込んだか
