# API セキュリティプログラムと組織アライメント

API セキュリティは技術的な対策だけでは成立しない。設計・実装レベルの原則をどれだけ整えても、組織としてセキュリティに継続的に投資し、脅威を体系的に評価し、監査に耐えうる説明責任を果たせなければ、対策は場当たり的なパッチワークに終わる。本ドキュメントは、API セキュリティを支える6つの原則と、それを組織に根付かせるための5つの実践をまとめる。

## セキュリティ原則

### Shift-Left API Security

ソフトウェアセキュリティには大きく2つのアプローチがある。開発の最後にセキュリティを後付けする「レトロフィット(bolt-on)」型と、開発の最初からセキュリティを組み込む「ビルトイン」型だ。

レトロフィット型は、ソフトウェアをブラックボックスとして扱い、TLS未設定・弱い暗号化・XSS・設定漏えい・無防備なエンドポイントといった汎用的な脆弱性だけを検査する。しかし API のセキュリティ侵害の大半は、ビジネスロジックやアクセス制御ロジックの欠陥に起因するため、この汎用チェックだけでは不十分である。

*shift-left* とは、問題を開発サイクルの早い段階で発見・修正する考え方であり、頻繁なテストと小さなバッチでの機能提供によって品質を作り込むソフトウェア開発のパラダイムである。API セキュリティにおける shift-left は「security by design」とも呼ばれ、設計段階から安全なユーザーフロー・入力パラメータを検討することから始まる。

shift-left の効果を具体例で見てみよう。EC サイトの `GET /products` エンドポイントに、価格や評価順で並べ替える `sort_by` パラメータを追加する場面を考える。

- **レトロフィット型の対応**: セキュリティテストスイートを実行して初めて SQL インジェクションの脆弱性が見つかる。正規表現でパラメータを1単語に制限するパッチを当てて修正したように見えるが、任意の単語でソートできる余地が残るため、モデルの非公開プロパティ（在庫数など）を推測される「プロパティ列挙(property enumeration)」攻撃には無防備なままになる。
- **security by design の対応**: 設計段階で `sort_by` を `price` / `review` のような列挙型に制約する。この制約だけで SQL インジェクションとプロパティ列挙の両方を設計上排除できる。

| shift-left が機能するための要件 | 内容 |
|---|---|
| 機微なユーザーフローの設計時特定 | セキュリティ担当と協働し、安全なペイロード設計・データ露出の最小化・入力制約を設計段階で決める |
| API ドキュメントの正確な整備 | 設計時点で脆弱性を検出できるよう、仕様（OpenAPI 等）を常に正確な状態に保つ |
| 設計時/実行時双方でのテストツール活用 | 導入前にツールが自分たちの要件を満たすか監査したうえで採用する |

> **警告**: shift-left は銀の弾丸ではない。SDLC 末尾での QA・セキュリティテストの代替にはならず、両者を組み合わせて初めて機能する。セキュリティインシデントは「起きるかどうか」ではなく「いつ起きるか」の問題であり、観測性・自動脅威検知・インシデント対応手順との併用が前提となる。

### ゼロトラストAPI

現代のアプリケーションは、ユーザー入力の処理・サードパーティ API 呼び出し・内部サービスとの連携・メッセージキューへのイベント発行など、複雑なデータフローで構成される。ここで問うべきは「何を信頼できるか」であり、答えは「何も信頼しない」である。

*ゼロトラスト*モデルは、信頼そのものをソフトウェアシステムの中核的な脆弱性と位置づけ、送信元にかかわらずすべてのトラフィックをデフォルトで拒否し、常に検証することを求める。NIST SP 800-207 はゼロトラストアーキテクチャの実装原則を以下のように定義している。

- すべてのデータソースをリソースとして扱う
- 送信元にかかわらずすべてのトラフィックを検証する
- 最小権限モデルに基づくユーザー単位のアクセス制御を強制する（ユーザーは自分が所有または許可されたリソースのみアクセス可能）
- 動的ポリシーによってユーザーのアクセス権をいつでも変更できるようにする
- すべてのリソースの完全性を監視し、侵害されたリソースへのアクセスを制限する
- すべてのトラフィックに期待どおりの認証・認可資格情報があるか確認する
- すべてのトラフィックを継続的に監視し、不審な挙動とセキュリティ体制を評価する

ゼロトラスト API の特性は次のように整理できる。

- すべてのエンドポイントを保護する（不要なエンドポイントや操作を公開しない）
- すべての API 呼び出しが正しい属性（ヘッダー・トークン・資格情報等）を持つか検証する
- 最小権限モデルに基づくユーザー/ロールベースのアクセス制御をすべてのリソースに適用する
- すべての API リクエスト（ペイロード・クエリパラメータ・パスパラメータ含む）を検証する
- ユーザーに返す前にすべてのデータを検証する
- サードパーティ API から受け取ったデータを処理前に検証する
- 内部サービスやデータベースから取得したデータを検証する
- ユーザーの活動を継続的に監視し、不審な挙動に反応する

認可はアクセス制御に関連する API 脆弱性の最大の温床の一つである。「認証済みユーザーは正当である」という前提でアクセス制御を緩めるのは典型的な失敗パターンであり、実際に多くの API 攻撃は認証済みユーザーによって行われる。JWT を使う API では、シグネチャ・ヘッダー・ペイロード中のすべてのクレームを期待値と照合して初めて認可が成立する。設計段階で「誰が・どの操作を・どのリソースに対して行えるか」というセキュリティプロファイルを描いておくと、アクセス制御テストの土台になる。

### すべてを検証する

「クライアントを信頼するな」はソフトウェアの原則としてよく知られている。ユーザーがアプリケーションとどう関わるかは制御できないため、サーバーに送られてくるすべてのデータに堅牢な検証とサニタイズを適用しなければならない。

- **入力検証**: データが期待するスキーマに準拠しているかを確認する
- **サニタイズ**: 悪意のある要素をデータから除去する

クライアントアプリケーション（SPA 等）は入力を制約する仕組みを持つが、ユーザーは常にクライアントを経由するとは限らない。API は CLI やその他任意のクライアントから直接呼び出されうるため、クライアント側の検証だけを信頼することはできない。受信データには「不正な形式・無効なデータ」と「悪意のあるデータ」の2種類のリスクがあり、どちらも同様に扱う必要がある。

検証すべきデータフローは、リクエスト・レスポンスの往復だけにとどまらない。

| データフロー | 検証すべき理由 |
|---|---|
| リクエストフロー | クライアントを経由しない直接呼び出しやツールからの攻撃を防ぐため |
| サードパーティ API からのデータ | 連携先が侵害されている場合、悪意のあるコード/データが自システムに侵入する「安全でない API 消費」を防ぐため |
| サービス間連携（内部サービス） | 一方のサービスの想定（例: 列挙値）と他方の実際の応答が食い違うと、カスケード的なエラーを招くため |
| 自社データベース | 手動変更・誤ったマイグレーション・スクリプトの事故等でデータが破損している可能性があるため |
| レスポンスフロー | 上記すべてのソースを集約して返すため、クライアントへ送信する直前にスキーマ準拠を再検証する必要があるため |

データベースから取得したデータは、スキーマ検証ライブラリや ORM（オブジェクト関係マッパー）を通じて列挙値などの制約を検証すること。ORM はモデルとテーブルを1対1でマッピングし、この種の検証を助けてくれる。

### 「内部API」という概念の否定

組織は、直接の収益に関わらない社内向け業務（人事・給与・案件管理・売上予測など）を自動化するために「バックオフィス API」を構築することが多い。これらは「社内利用だから」という理由で内部ネットワークに配置され、公開されないという前提のもと、認証・認可が手薄になりがちである。

*ネットワークセグメンテーション*（ネットワークを小さなサブネットに分割し、アクセス権限を段階的に管理する手法）自体は有効なセキュリティ手法だが、それだけを唯一の戦略にしてはならない。実際に、大規模な通信事業者がテスト目的で構築した顧客認証用の内部 API を、設定ミスによりインターネットに接続されたテストネットワークへ誤ってデプロイしてしまい、数百万件規模の顧客データが露出した事例が知られている。この事例から得られる教訓は次の2点だ。

- すべての API は、扱うデータの機微性に応じた堅牢な認証・認可で保護されなければならない
- 「内部 API」という区分そのものが人為的なものであり、プライベートネットワークに配置するかどうかは運用上の判断ミスに左右される。ゼロトラストモデルをすべての API に適用することが最善の防御である

内部 API に十分なアクセス制御がない場合、意図的・偶発的を問わず「インサイダー脅威」にもさらされる。悪意のある内部者によるデータの窃取・漏えいだけでなく、過剰な権限を持つユーザーが誤って機密データを変更・削除・漏えいさせるケースも同様に脅威となる。ネットワークセグメンテーションは「安全なネットワークを構築するための戦術・道具」であって「戦略」ではない、という考え方を念頭に、shift-left の原則をすべての API に一貫して適用することが根本的な解決策になる。

### APIインベントリ管理

「知らないものは守れない」がこの節の核心原則である。組織が直面する最大のリスクの一つは、ドキュメントとインベントリの欠如、すなわち *API sprawl*（API の乱立とその追跡・文書化・セキュリティ評価の失敗）である。API sprawl から生まれる代表的な類型は次の3つ。

| 類型 | 説明 |
|---|---|
| シャドー API | 適切なテスト・評価プロセスを経ずに、水面下でリリースされた API。未文書化・低セキュリティになりがちで、忘れ去られやすい |
| ゾンビ API | 廃止すべきなのに退役し忘れた、非推奨の API またはバージョン |
| ゴースト API | 自社のライブラリやインフラが暗黙に公開している API（サードパーティライブラリのバックドア、クラウドサービスのメタデータエンドポイント等） |

大規模組織ほど保有する API 数は多く、しかも増加ペースは速い。多くが未文書化のままであり、これは組織のセキュリティ上の懸念事項の上位に挙がる。

文書化には仕様書・チュートリアル・ハウツーガイド・コードサンプル・変更履歴など複数の形態があるが、攻撃対象範囲を文書化する最良の手段は **API 仕様**（REST では OpenAPI、GraphQL では SDL 等の標準記述言語による形式的な記述）である。API 仕様は曖昧さがなく機械可読なため、自動テスト・評価・カタログ化に活用できる。

数百〜数千規模の API を管理するには、個々の仕様書だけでは足りず **API カタログ**（すべての API を一元管理する仕組み）が必要になる。カタログは既存機能の重複開発を防ぎ、API sprawl と戦う組織にとって不可欠なツールになる。カタログの標準化は業界でも進んでおり、`/.well-known/api-catalog` のような well-known URI をカタログの標準的な所在地とする提案も出ている。

シャドー/ゾンビ API の発見（ディスカバリ）には、トラフィック観測ベースのアプローチとコード解析ベースのアプローチがあるが、いずれも万能ではない。観測ベースは実際に呼び出されたエンドポイントしか発見できず、コード解析ベースは実装パターンの多様性ゆえに網羅が難しい。ツールに頼り切るのではなく、運用・インフラチームとの対話を先に行うことが有効な出発点になる。

### DevSecOps for APIs

shift-left・ゼロトラスト・全データ検証・インベントリ管理の原則を、実際の開発サイクルにどう組み込むかを整理する。

1. **設計**: 提供する機能・扱うデータ・ユーザーフローを検討し、追加のセキュリティ配慮が必要な操作を特定する。成果物として API 仕様を作成する
2. **仕様のレビュー**: Git リポジトリで仕様を管理し、変更はプルリクエストで関係者がレビューする。仕様を一元管理するか、実装リポジトリに含めるかは組織の方針次第
3. **設計テスト**: 実装前に仕様そのものの脆弱性を検証し、機能テスト・セキュリティテスト・KPI を定義する
4. **並行実装**: モックサーバーで API サーバーとクライアントを並行開発し、フィードバックループを短縮する
5. **CI/CD による自動テスト**: 継続的インテグレーションパイプラインで実装を検証し、デプロイ前に不具合を検出する
6. **手動 QA/セキュリティテスト**: 自動化しにくい脆弱性（スキャルピング攻撃耐性、カタログ全体のスクレイピング耐性、ブルートフォース耐性等）を人手で確認する
7. **リリースとカタログ登録**: 問題がなければ本番リリースし、API カタログに登録する

仕様は完成した設計図ではなく「仮説」であり、実装・テストを通じて進化していくものと捉える。破壊的変更に備えてバージョニング戦略を用意し、脆弱性が見つかった場合は設計段階に戻って再評価するサイクルを繰り返す。

## 組織アライメント

技術的な原則をどれだけ理解していても、組織としての体制がなければ API セキュリティは特定の個人や一時的な取り組みに依存してしまう。ここでは、組織全体として API セキュリティを推進するための実践を扱う。

### APIセキュリティポスチャの評価

*セキュリティポスチャ*とは、脅威を予防・識別・対応・復旧する組織の能力を指す。重大なイベント（投資ラウンドのデューデリジェンスや規制監査）を待たずに、今すぐ評価を始めるべきである。評価の観点は次のように整理できる。

| 評価カテゴリ | 確認すべき問い |
|---|---|
| データインベントリ | どんなデータを収集し、どこにどれだけ保存しているか。棚卸しは定期更新されているか |
| 攻撃対象範囲 | エンドポイント数・API バージョン数を把握しているか。未知の攻撃対象範囲を防ぐ仕組みがあるか |
| 認証と認可 | ベストプラクティスと標準に従っているか。ログインエンドポイントはブルートフォース等から保護されているか。署名鍵は定期的にローテートしているか |
| アクセス制御 | すべてのエントリーポイントが保護されているか。機微なエンドポイントへの厳格なアクセス制御があるか |
| セキュアな実装 | 適切な API ライブラリでデータ検証を行っているか。依存関係は安全か。リリース前にレビュー・テストしているか |
| セキュリティテスト | 自動セキュリティテストを定期的に実行しているか。専門的なペネトレーションテストの実施経験があるか |
| リスク分析 | 主要なビジネスリスクを把握し、脆弱性の優先順位付けができているか |
| インシデント検知 | API の可観測性があり、悪意ある活動を継続的に検知・通知できているか |
| インシデント対応計画 | 詳細な対応計画があり、規制要件に準拠しているか |
| トレーニングと意識向上 | 全従業員がリスクを認識し、継続的な教育を受けているか |

各カテゴリについて「Yes/No」の二値評価、あるいは 0〜5 のスコアでより粒度の高い評価を行う。評価が曖昧なカテゴリ（例: 攻撃対象範囲の95%しかマッピングできていない）は「No」として扱い、正直な自己評価に基づいてロードマップを描くことが重要である。

> **注意**: ベンダーはしばしば「セキュリティポスチャ管理の自動化」を謳うが、トレーニング・ビジネスリスクの定義・インシデント対応計画の策定などは自動化できない。純粋に技術的なセキュリティテストであっても、ビジネスドメインの知識を要する検出は自動化ツールでは捉えきれないことが多い。すべてを自動化できると主張するベンダーには注意すること。

セキュリティポスチャ評価の基盤となるのが脅威モデリングである。

### API脅威モデリング（チームでの実施）

脅威モデリングは、システムのセキュリティ特性を分析し、脅威アクターがどう脆弱性を悪用しうるかを可視化する作業である。良い脅威モデルは、攻撃がシステムのどこを通過し、どのコンポーネントが影響を受けるかを示す。API ゲートウェイ・ロードバランサー・複数サービス・データベース・キューなど、複数コンポーネントにまたがる複雑なシステムでは、各部分を異なるチームが所有していることが多いため、**脅威モデリングはチームスポーツ**であり、エンジニア・プロダクトオーナー・QA・ビジネスステークホルダーが協働して初めて全体像が見える。

脅威モデリングが答えるべき4つの問いは次のとおり。

1. 何に取り組んでいるのか？
2. 何がうまくいかない可能性があるか？
3. それにどう対処するのか？
4. 十分な仕事ができたか？

これらの問いは、次の4ステップに対応する。

| ステップ | 対応する問い | 内容 |
|---|---|---|
| ①アプリケーション分解 | 何に取り組んでいるのか？ | データフロー図（DFD）でシステム内のデータの流れとコンポーネントを可視化する。内部サービスへの API 呼び出しの見落としに注意（SSRF 等の見逃しにつながる） |
| ②脅威の識別とランク付け | 何がうまくいかない可能性があるか？ | STRIDE 等のフレームワークで攻撃者視点の脅威を洗い出し、事業影響と発生可能性でランク付けする |
| ③対応と緩和策 | それにどう対処するのか？ | 各脅威への対応方針を決める |
| ④レビューと検証 | 十分な仕事ができたか？ | DFD の正確性・脅威の網羅性・対応の実効性をチームでふりかえる（レトロスペクティブ） |

STRIDE は脅威を6つのカテゴリで捉えるフレームワークである。

- **なりすまし(Spoofing)**: ユーザー認証情報を盗み、アカウントを乗っ取るリスク
- **改ざん(Tampering)**: データを意図せず・不正に書き換えられるリスク
- **否認(Repudiation)**: あるユーザーが特定の操作を行ったかどうかを証明できないリスク
- **情報漏えい(Information Disclosure)**: システム詳細・設定・機微データが漏えいするリスク
- **サービス拒否(Denial of Service)**: システムを利用不能にされるリスク
- **権限昇格(Elevation of Privileges)**: 攻撃者が本来持たない権限を獲得するリスク

識別・ランク付けした脅威への応答は、次の4類型に分類できる。

| 応答類型 | 内容 | 例 |
|---|---|---|
| 緩和(Mitigate) | システムを堅牢化し発生可能性を下げる | ブルートフォース対策としてレート制限を実装する |
| 排除(Eliminate) | ビジネス上不可欠でない機能そのものを削除する | 任意 URL からのデータ取得機能が SSRF の温床になるなら機能自体を廃止する |
| 移譲(Transfer) | 対応コストが高すぎる場合、責任を別の主体（顧客等）に移す | 機微データを含む共有リンク機能は、リンク管理をユーザー責任とする |
| 受容(Accept) | ビジネスにとって許容可能なリスクは対応しない | 紹介キャンペーンの不正利用リスクを、初期成長フェーズでは許容する |

脅威モデリングは一度きりの作業ではなく継続的な取り組みであり、四半期や半期ごとの定期実施に加え、大きな変更に着手する際は必ず脅威モデリングから始めるべきである。API 設計プロセスの早い段階で脅威モデリングを行うことで、セキュリティを後付けではなく作り込みとして実現できる。

### 低い枝から始める（低リスク・高効果の実践）

API セキュリティの成熟度を一夜にして高めることはできない。最も複雑な問題から着手すると、進捗が遅く感じられチームの士気を削ぐ。**組織への効果が大きく着手コストが低い「低い枝」から始め、勢いと自信を積み上げる**ことが推奨される。どこから手をつけるべきかは、「APIセキュリティポスチャの評価」の結果に応じて変わる。

| 低い枝 | 実践内容 |
|---|---|
| API を文書化する | 形式的な API 仕様がなければ、実装が正しいか検証できず、セキュリティテストも設計できない。トラフィックの傍受やコード解析からの逆生成も含め、まずは仕様を整備する |
| 認証・認可を強化する | 「認証済みユーザーは正当」という前提でアクセス制御を緩めていないか確認する。ユーザー A・B の2アカウントを用意し、A が B のリソースにアクセスできてしまわないかを検証するテスト（BOLA 検出の第一歩）を自動化し、変更のたびに繰り返し実行する |
| 適切な API ライブラリを使う | URL クエリパラメータやリクエストペイロードの処理は見た目以上に複雑であり、専用ライブラリ・フレームワークが入出力検証・シリアライズ・エラーハンドリングを肩代わりしてくれる。自前実装からの移行が必要な場合は、ビジネスロジックを外部依存から切り離すヘキサゴナルアーキテクチャ（ポート&アダプター）を使うと、API 層だけを段階的に置き換えられる |
| クラウド保護ツールを使う | API ゲートウェイはアクセストークン検証・DDoS 防御・レート制限・データ検証などを一手に引き受け、単一の入口として構成すればシャドー API の露出も抑えられる。WAF（Web Application Firewall）は SQL インジェクションや SSRF など既知の Web 脅威からアプリケーションを守る。両者は組み合わせて使うと効果的である |

> **注意**: API ゲートウェイや WAF の導入は組織に過剰な安心感を与えがちである。これらは有用だが、ユーザーが本来アクセスできないデータに到達してしまう BOLA のような脆弱性までは防げない。ゲートウェイや WAF だけを「主たる悪意ある活動の検知手段」に据えている組織は少なくないが、それだけでは不十分である。

### APIセキュリティプログラムの立ち上げ

多くの API セキュリティの取り組みは、一人の推進者から始まる。しかし API セキュリティは特定の個人やチーム一つの仕事ではなく、組織全体の仕事である。この意識を制度化する手段が **API セキュリティプログラム**（自組織の API セキュリティ戦略・ビジョン・ゴール・実現手段をまとめたもの）である。

組織レベルで API セキュリティに初めて取り組む場合、最初から野心的になりすぎないこと。実現可能な範囲を現実的に見積もり、次の3要素からなる軽量なガバナンスを整えることが成功の鍵になる。

| ガバナンスの要素 | 内容 |
|---|---|
| 標準を文書化し、軽量に保つ | 誰もが参照しやすい場所（リポジトリ等）に、簡潔でコード例付きのドキュメントを置く |
| 組織内の多様なステークホルダーから意見を集める | アーキテクト・開発者・プロダクトマネージャー等を巻き込むことで当事者意識を醸成する |
| 自動化する | 手動運用に頼るルールは形骸化しやすい。API 設計をチェックするセキュリティ意識の高い Linter や、実装を検証するファズテスターを活用する |

組織の成熟度が上がるにつれ、低い枝から高い枝へと段階的にレベルを上げていく。トレーニングプログラムの整備、インシデント対応計画の策定、高度な脅威検知技術の導入などが次の段階になる。この文脈でよく登場する体制が **レッドチーム/ブルーチーム/パープルチーム**である。

- **レッドチーム**: 脅威アクターの挙動をシミュレートし、実際に攻撃・脆弱性の悪用を試みる
- **ブルーチーム**: 脅威の検知・防止・対応に注力し、レッドチームの攻撃を発見・緩和する
- **パープルチーム**: レッドチームとブルーチームの活動を分析し、改善機会を見出す

セキュリティ体制に十分な自信が持てるようになったら、脆弱性の発見を外部の研究者に委ねる**バグバウンティプログラム**の導入も選択肢になる。ただし開始前に、既知の脆弱性を解消済みであること、報告への対応リソースがあること、報告者への公正な報酬体系が整っていることを確認する必要がある。

### 組織へのアライメント

なぜ API セキュリティは「全員の仕事」なのか。ユーザーフローの設計からクエリパラメータの実装詳細まで、API に関わるすべての決定がセキュリティポスチャに直結するからだ。従来型のアプリケーションセキュリティだけでは API を十分に保護できないため、設計・実装・アーキテクチャ・運用を横断する包括的なアプローチが必要になる。

API セキュリティへの本腰を入れた取り組みは、一見すると開発速度を落とすように見える。プロダクトマネージャーはユーザーフロー設計時にセキュリティを考慮しなければならず、開発者は適切な実装ライブラリの習得・JWT の正しい扱い・OAuth や OIDC といった標準プロトコルの理解を求められる。これをどう組織に納得してもらうか。

答えは「やらなかった場合のコスト」を提示することだ。データ侵害は高額な罰金と信頼の失墜を招き、小規模な組織にとっては事業存続に関わる規模になりうる。実際に、設定ミスによる内部 API の誤公開だけで数百万件規模の顧客データが流出し、監査と補償に巨額の費用を要した事例が実在する。

しかし security by design を正しく実践すれば、開発速度を犠牲にする必要はない。従来型の開発サイクル（設計→実装→デプロイ→手動セキュリティテスト、脆弱性が見つかれば最初からやり直し）と比較して、security by design のサイクル（設計段階から安全なユーザーフローと入力制約を組み込み、実装段階でも検証を重ねる）は、デプロイ後に見つかる問題がはるかに少なく、サイクルの繰り返し回数を削減できる。結果として、より速く、より自信を持って API をリリースできるようになる。

この効果を組織に証明する最良の手段は**メトリクス**である。

- 同じ脆弱性の修正のために設計サイクルを何回繰り返しているか
- API を設計からデプロイまで進めるのにかかる時間
- API 仕様をセキュリティ設計テストにかけたときのエラー数
- ファズテストで検出されるエラー数

こうした計測可能な成果は、組織からの支持を得るための最も効果的なコミュニケーション手段になる。

### APIセキュリティ監査への対応

理想的には組織が自発的にセキュリティポスチャの改善に取り組むが、実際には監査がきっかけになることが多い。投資家によるデューデリジェンス、買収時のセキュリティ確認、新たな事業提携先からの要求などが典型例であり、こうした場面では ISO/IEC 27001・HIPAA・SOC・PCI DSS のような認証取得を求められることも多い（特に医療・金融・防衛のような機微性の高い業界）。

監査への対応は、まず**ギャップアセスメント**（現状と合格基準との差分評価）から始める。「APIセキュリティポスチャの評価」で示した評価カテゴリを使って、自分たちがどの程度答えられるかを確認する。攻撃対象範囲を「まったく把握していない」状態と「暫定的にでもマッピング済み」の状態には大きな差があり、可能ならマッピング済みの割合を具体的に示せるようにしておく。扱うデータが機微であるほど、監査人からの精査は厳しくなる。

初めての監査であれば、完璧である必要はない。重要なのは、技術的負債への自覚・改善のロードマップ・それを実行する意思を示すことである。可能な限り、前節で述べた測定可能な成果指標も併せて提示する。

外部の専門監査サービスを活用することも有効な選択肢だが、それは自組織のセキュリティ戦略を策定しない言い訳にはならない。API を保有する以上、セキュリティポスチャ管理と security by design への取り組みそのものは自組織が主体的に持つべき責任である。
