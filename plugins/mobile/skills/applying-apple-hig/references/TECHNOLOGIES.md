# システム技術との統合（Technologies）

> 出典: Apple Human Interface Guidelines / Apple Developer Documentation（最終確認 2026-07-13）。仕様は更新されるため、実装時は必ず併記の公式 URL で最新版を確認すること。日本語版がある場合は公式の言語切替または本文の日本語 URL を使い、英語版との差分がある場合は最新側を確認する。
> 判定区分: 本文の「必須」「違反」は作業用の強度ラベルであり、HIG の推奨を一律に App Review 必須へ格上げしない。審査・API 上の必須判定は現行の Apple 公式一次資料で再確認する。

iPhone/iPad アプリがシステム技術（ウィジェット・Live Activity・決済・認証・ML/AI・AR・共有体験・地図）と統合する際の HIG 要件を、実装判断に直結する形でまとめる。

---

## ウィジェット（Widgets）

Home 画面・Today ビュー・Lock 画面・StandBy に一目で分かる情報を表示する小さな UI。アプリ本体の代替ではなく「最重要情報のグランス表示＋タップでアプリの該当画面へ」が役割。

### 🔴 必須（遵守事項）

| 項目 | 要件 |
|------|------|
| サイズファミリー | iPhone: Small / Medium / Large。iPad はさらに Extra Large。Lock 画面アクセサリは Circular / Rectangular / Inline |
| 寸法（代表値・pt） | 430×932 画面で Small 170×170 / Medium 364×170 / Large 364×382、390×844 画面で Small 158×158 / Medium 338×158 / Large 338×354。アクセサリ Circular 72–76×72–76、Rectangular 157–172×72–76 |
| マージン | 標準 16pt。グラフィックやボタンが縁に寄るレイアウトは 11pt。角丸は `ContainerRelativeShape` でコンテナに追従させること |
| テキスト | 最小 11pt 以上。テキストは必ずテキスト要素として描画すること（画像へのラスタライズ禁止 — VoiceOver が読めなくなる） |
| レンダリングモード | Full color に加え、Lock 画面/StandBy 低照度の vibrant（グレースケール）、Home 画面 tinted の accented に対応すること。`widgetAccentable()` で強調グループを指定 |
| タップターゲット | インラインアクセサリウィジェットはタップターゲット 1 つのみ。他サイズでもターゲットは確実に押せる大きさを確保 |
| 更新頻度 | リアルタイム更新は不可（システムが予算制御）。秒単位で変わる情報は Live Activity を使うこと。日時は `Text(date, style:)` でシステムに自動更新させる |

### 推奨

- 更新タイムラインは 1〜2 時間間隔を基準に設計（潮汐・天気=毎時、カレンダー=毎日など情報の鮮度に合わせる）。
- ウィジェット内インタラクションは「タップで完結する」ボタン/トグル（例: リマインダー完了）と deep link に限定し、複雑な操作はアプリへ委譲する。
- 色だけに意味を持たせない（テキスト/アイコンを併用）。フルカラー画像はメディアコンテンツに限る。
- StandBy（iPhone 横置き）では背景が除去され拡大表示される前提でデザインし、低照度の赤色ティントでもコントラストを検証する。
- プレースホルダー（データ未取得時の骨格表示）を用意する。ギャラリー用 description は「See the current weather…」のように内容を直接書く（「This widget shows…」「Use this widget to…」は冗長で不可）。
- Dynamic Type は Large〜AX5 の範囲で確認する。

### よくある違反・注意点

- ウィジェットをミニアプリ化して多段ナビゲーションや設定画面を詰め込む（グランス表示の原則違反）。
- テキストを画像として焼き込み、アクセシビリティとダークモード追従を壊す。
- タイムライン更新を乱発してシステム予算を使い切り、更新されないウィジェットになる。
- 角丸を固定値で指定し、コンテナ形状と合わない。

公式: https://developer.apple.com/design/human-interface-guidelines/widgets

## Live Activity / Dynamic Island（Live Activities）

開始と終了が定義されたタスク・イベント（配達、試合、タイマー、搭乗）の進行状況を Lock 画面と Dynamic Island で追跡する UI。

### 🔴 必須（遵守事項）

| 項目 | 要件 |
|------|------|
| 対象 | 開始/終了が明確なタスクのみ。**最大 8 時間**以内に終わるものに限る |
| 3 プレゼンテーション | Dynamic Island 搭載機では compact（leading+trailing の 2 要素）・minimal（複数 Activity 併存時）・expanded（長押しで展開）の 3 つ全てを設計すること |
| 寸法（代表値・pt） | compact leading/trailing 各 52.33–62.33×36.67、minimal 幅 36.67–45、expanded/Lock 画面 371–408×高さ 84–160（可変）。iPad の Lock 画面は幅 425–500 |
| Lock 画面 | 高さ 84–160pt、マージン 14pt（通知と揃える）。expanded と近いレイアウトにし、独自レイアウトを乱造しない |
| 広告禁止 | 広告・プロモーション表示は禁止 |
| 機密情報 | Lock 画面で第三者に見える前提。機密はタップで詳細表示・redact・ユーザー設定で制御 |
| 更新 | コンテンツが変化した時のみ更新。アラート付き更新は重要な変化に限定。アニメーションは最大 2 秒 |
| 終了処理 | タスク完了で即座に終了させる。Lock 画面残存はデフォルト 4 時間 — カスタム削除時間（推奨 15〜30 分）を設定すること |
| ダーク/常時表示 | Light/Dark 両対応必須。カスタム背景色は Always-On Display の減光状態でもコントラストを検証 |

### 推奨

- テキストは Medium 以上のウェイト（Light/Regular は Lock 画面で読みにくい）。一目で読める大きさにする。
- Dynamic Island では TrueDepth カメラ領域に要素をパディングなしで密着させ、外縁から同心円状に等マージンで配置する。角丸は「外側半径 − マージン」で計算。
- アプリアイコン全体をロゴとして再掲しない（compact ではブランドより更新情報を優先。例: タイマーは残り時間を出す）。
- インタラクティブ要素（再生/一時停止等）は誤タップ防止のため単一要素に絞る。
- compact の leading/trailing タップは同一画面へ遷移させる。
- StandBy では minimal→タップで Lock 画面表示が 2 倍スケールされる。Night Mode の赤ティントでの見え方も確認する。

### よくある違反・注意点

- プッシュ通知と Live Activity アラートで同じ内容を二重通知する。
- 変化がないのに更新し続けバッテリーを浪費する / 終了後も Dynamic Island に残す。
- 8 時間を超える長期タスク（例: 数日かかる配送）に使う — この用途はウィジェットか通知が正しい。
- compact の左右要素のサイズバランスを欠く（単位の短縮などで揃える）。

公式: https://developer.apple.com/design/human-interface-guidelines/live-activities

## App Clip（App Clips）

フル App の一部を「その場・即時」に使わせる軽量体験。iOS/iPadOS 専用。QR/NFC/App Clip Code/Smart App Banner/Maps/Messages から起動される。

### 🔴 必須（遵守事項）

- 単一タスク完結型に絞ること。Tab Bar・複雑なナビゲーション・設定画面・スプラッシュスクリーンは不可。UI はネイティブコンポーネントで構築（Web View 不可）。
- 起動したら即、文脈に合った画面を表示する（追加ダウンロードで待たせない。必要アセットは同梱し、サイズは可能な限り小さく）。
- アカウント登録・個人情報入力はタスク完了の**後**に要求する。価値提供前のサインイン強制は不可。
- 通知は起動後 **8 時間**のみ自動許可。延長はユーザーの明示同意が必要で、用途はタスク関連に限る（プロモーション通知不可）。
- App Clip Card 用画像は **1800×1200px**（PNG/JPEG・透過なし）、タイトル 30 文字以下、サブタイトル 56 文字以下、アクションは View/Play/Open のいずれか。画像へのテキスト埋め込みは不可（ローカライズ不能）。
- App Clip Code は Apple 提供の SVG のみ使用。印刷最小 **3/4 inch（1.9cm）径**、デジタル最小 **256×256px**。NFC 統合型はタグ 35mm 以上でコード 3.48cm 以上。改変（フィルター・影・回転・グラデーション・アニメーション）は全面禁止。

### 推奨

- 視距離とサイズの比は 20:1 以下（最適 10:1）。例: 1m 先から読むなら約 10cm 径。
- データはデバイス外にセキュアに保存する（App Clip は一定期間後に自動削除され、ローカルデータも消える）。認証は Sign in with Apple、決済は Apple Pay を使うと入力レス。
- フル App への誘導はタスク完了時か自然な区切りで、システム提供の App Clip Card/バナー/SKOverlay を使う。デモ型はデモを最後まで体験させてから誘導する。

### よくある違反・注意点

- フル App のインストールをタスク中に繰り返し要求して体験を損なう（HIG 明文の禁止事項）。
- App Clip 内にバックグラウンド処理を期待する実装（完全に不可）。
- App Clip Code をギャンブル・飲酒関連の媒体にロゴ付きで掲載する（規約違反）。

公式: https://developer.apple.com/design/human-interface-guidelines/app-clips

## Apple Pay

Face ID/Touch ID で完結する決済。チェックアウトの摩擦を最小化する。

### 🔴 必須（遵守事項）

| 項目 | 要件 |
|------|------|
| ボタン | API 提供の Apple Pay ボタン（`PKPaymentButton`）のみ使用。カスタムボタンに「Apple Pay」の文字や Apple Pay ロゴを入れることは禁止 |
| 最小サイズ | 幅 100pt × 高さ 30pt（Book with Apple Pay のみ幅 140pt）。周囲マージンはボタン高の 1/10 以上 |
| プライマリ表示 | Apple Pay API でカード保有を確認した場合、Apple Pay を第一の支払いオプションにすること（別フローに隔離不可） |
| 配置 | 他ボタンと横並びなら Apple Pay を**右**、縦積みなら**上**に置く |
| マーク | 支払い方法一覧には Apple Pay マークグラフィックを使用。高さは他の決済ロゴ以上、余白は高さの 1/10 以上。マークをボタンとして使う・改変・アニメーション・翻訳は禁止 |
| 表記 | 「Apple Pay」（タイトルケース・分かち書き）。「ApplePay」「Apple ペイ」等は不可 |

### 推奨

- ボタンスタイルは `automatic`（Light/Dark 自動切替）を既定にする。黒ボタンは明背景、白ボタンは濃背景、コントラスト不足時は white-with-outline。
- コーナー半径は角丸なし〜カプセルの範囲で自 UI に合わせて調整可。
- サイズ・色等の商品属性は支払いシート表示**前**に確定させ、不足時は該当フィールドへ自動ナビゲートする。
- 配送方法・受取場所は支払いシート**内**で選択させ、各選択肢に説明＋費用＋配送予定日を出す。ラインアイテム（追加料金・割引・寄付・定期課金）は 1 行で簡潔に、合計行は「Pay [事業者名]」。
- 検証エラーは具体的に書く（◯「Zip code doesn't match city」/ ✗「Address is invalid」）。128 文字以下・名詞句・句点なし。
- サブスクリプションは金額と周期、トライアルは試用額＋正規額＋開始日をラインアイテムで明示。寄付は推奨額の選択肢（$25/$50/$100 等）＋自由入力を提示。

### よくある違反・注意点

- Apple Pay をカード入力フォームの奥に隠し、主要な支払い手段として見つけにくくする。Apple Payの現行HIGと審査要件は提出時に再確認する。
- 支払いシートの複製・模倣 UI を自作する。
- 決済成功後の確認画面で Apple Pay 利用を示さない（「Paid with Apple Pay」等を表示するのが正しい）。

公式: https://developer.apple.com/design/human-interface-guidelines/apple-pay

## Wallet（パス設計）

搭乗券・チケット・クーポン・ストアカード・ID などを Wallet アプリに格納するパス（pass）の設計。

### 🔴 必須（遵守事項）

- パススタイルを正しく選ぶ: Boarding Pass（交通機関）/ Coupon / Event Ticket（poster / 非 poster）/ Store Card / Generic。用途に合わない流用は不可。
- `expirationDate`・`relevantDate`・`voided` を正確に設定し、期限切れ・無効パスを放置しない。パスの削除はユーザーの許可を得てから。
- パス更新通知は時間的に重要な情報のみ（マーケティング通知は禁止）。
- アセット寸法（pt、@2x/@3x PNG）: logo 最大 160×50 / icon 38×38 / strip 375×144（クーポン・ストアカード）/ thumbnail 最大 90×90 / 背景（イベント非 poster）343×503 / poster アートワーク 358×448 / boarding pass footer 268×15。
- 背景とテキストのコントラストを確保する（明背景×明テキスト、複雑な柄の上のテキストは不可）。ロゴへの内側ドロップシャドウ禁止。
- 注文追跡（Order Tracking）のロゴ/商品画像は 300×300px・非透過・単色背景。

### 推奨

- 複数パスは一括追加（`addPasses`）でユーザーの手間を省く。Apple Pay 購入後はシステム UI で自動追加、確認したいユーザーには `PKAddPassesViewController` プレビューを出す。
- 「Add to Apple Wallet」ボタンはデバイス対応を確認してから表示し、一度拒否したユーザーには再表示しない。
- poster スタイルでは重要コンテンツをセーフエリア内に置く（下部はバーコード領域のマテリアルストリップに覆われる）。
- ID 検証（Verify with Wallet）は実際に必要な時点でのみ要求し、目的文は能動態で具体的に書く（◯「Federal law requires this information to verify your identity」/ ✗「Your identity needs to be verified」）。データ保持期間を明示する。
- 位置・時刻の relevancy 情報を設定し、Lock 画面での自動表示を活かす。イベントチケットは Live Activity 連携を検討。

### よくある違反・注意点

- クーポンをジェネリックパスで作る等、スタイル選択ミスによりストリップ画像等の機能が使えない。
- 白余白をパディング代わりに画像へ焼き込む。
- 使用済み・期限切れパスを voided にせずユーザーの Wallet を散らかす。

公式: https://developer.apple.com/design/human-interface-guidelines/wallet

## Tap to Pay on iPhone

iPhone 自体を非接触決済端末にするマーチャント（加盟店）向け機能。iOS 専用（iPadOS 非対応）。決済サービスプロバイダー（PSP）との連携と ProximityReader API・エンタイトルメントが前提。

### 🔴 必須（遵守事項）

- 利用開始前にマーチャント（管理者ユーザー）の利用規約同意を必須とする。非管理者には管理者アクセスが必要な旨を説明。
- ボタンラベルは「Tap to Pay on iPhone」または「Tap to Pay」。SF Symbol は `wave.3.right.circle` / `wave.3.right.circle.fill` を使用。**Apple ロゴの表示は禁止**。
- 「Tap to Pay」を非決済アクション（カード照会・返金・ロイヤリティカード読取）のラベルに使わない。それらは「Look Up」「Store Card」「Verify」「Refund」「Loyalty Card」等にする。
- チェックアウト開始前に最終金額を確定させる（チップ等の顧客入力は先に処理）。
- 決済失敗時は問題の説明＋対処法＋**代替決済手段**を必ず提示する（別カード再試行、外部端末、決済リンク等）。

### 推奨

- `PaymentCardReader.prepare(using:)` をアプリ起動時とフォアグラウンド復帰直後に呼び、リーダー準備を先行させる。準備完了を待たずに決済オプション自体は常に選択可能にする。
- 進捗表示: 準備中は不定形インジケーター、ProximityReader 進捗は確定形（%）、認可中は不定形＋「Authorizing」。
- マーチャント教育は `ProximityReaderDiscovery` API のシステム提供チュートリアル（ローカライズ済・常に最新）を利用する。独自チュートリアルにはカード/ウォレットのかざし方・PIN 入力（アクセシビリティモード含む）を含める。
- 決済手段が Tap to Pay のみなら決済画面を直接開き、複数あるならスクロール不要な位置にボタンを置く。

### よくある違反・注意点

- 機能が未有効でも決済オプションとしては表示し、タップ時に規約同意フローへ誘導するのが正しい（隠すのは誤り）。
- ロイヤリティカード読取を決済フローに混在させる（明確に分離し決済用語を避ける）。

公式: https://developer.apple.com/design/human-interface-guidelines/tap-to-pay-on-iphone

## App 内課金（In-App Purchase）

IAP は Consumable（消費型）/ Non-consumable（非消費型）/ Auto-renewable Subscription（自動更新サブスクリプション）/ Non-renewing Subscription（非更新サブスクリプション）の 4 種。

### 🔴 必須（遵守事項）

- 種類を問わず、購入前に**総額**を表示すること。
- サブスクリプション登録画面には次を必ず含める: 名称・期間・提供内容、地域/通貨に正しくローカライズされた請求額、既存購入者の復元/サインイン手段。無料トライアルは試用期間と**終了後の課金額・課金開始タイミング**を明記。
- 購入確認はシステム提供の確認シートをそのまま使う（カスタマイズ・複製は不可）。
- `AppStore.canMakePayments` で支払い可否を確認し、ペアレンタルコントロール等で不可の場合はストア UI を出さない。
- 復元（Restore Purchases）手段を必ず提供する。
- Family Sharing 対応は Non-consumable と Auto-renewable Subscription のみ（最大 5 人の家族と共有）。

### 推奨

- 課金前に価値を体験させる（freemium / metered paywall / free trial）。プロンプトは無料枠の上限到達時や設定画面など適切なタイミングで。
- 価格帯は複数提示し（月額/年額）、年額の割安さは月額換算で示す。既加入者に同じサブスクを重ねて勧めない。
- 購入 UI はアプリのスタイルに統合し、製品名は短く・説明はテキストが折り返さない程度に簡潔に。
- オファーコード: 入力 UI はシステム提供（`presentOfferCodeRedeemSheet` / `offerCodeRedemption`）。カスタムコードは英数字 ASCII のみ。入手/入力方法（URL 経由かアプリ内か）を明示し、Paywall 上部・オンボーディング・設定に入口を置く。
- サブスク管理はシステム UI（`AppStore.showManageSubscriptions`）へ誘導し、次回請求日を見せる。解約フローではパーソナライズオファーや理由アンケートを提示してよいが、解約自体は妨げない。
- 返金サポートは `Transaction.beginRefundRequest` を使い、購入特定情報（画像・名称・購入日）と「Refund」ボタンをスクロール不要の位置に置く。

### よくある違反・注意点

- 無料トライアルの期間、更新、自動課金条件を曖昧にする。現行の購読表示要件とApp Review Guidelinesを提出時に再確認する。
- 解約ボタンを深い階層に埋める・Apple の解約/返金ポリシーを推測で説明する（誤案内禁止）。
- 復元ボタンの欠落。長すぎる登録フロー・不要な個人情報の先取り。

公式: https://developer.apple.com/design/human-interface-guidelines/in-app-purchase

## Sign in with Apple

Apple Account による認証。パスワードレスで、メールアドレス秘匿（Hide My Email）を選択できる。

### 🔴 必須（遵守事項）

| 項目 | 要件 |
|------|------|
| ボタン | システム API `ASAuthorizationAppleIDButton` を使用（承認済み外観・自動ローカライズ・VoiceOver 対応） |
| タイトル | 「Sign in with Apple」「Sign up with Apple」「Continue with Apple」の 3 種のみ。改変不可 |
| 最小サイズ | 幅 140pt × 高さ 30pt。周囲マージンはボタン高の 1/10 以上 |
| スタイル | Black（明背景用）/ White（濃背景用）/ White with outline（白背景でコントラスト不足時）。背景とのコントラストで選ぶ |
| プロミネンス | 他のサインインボタンと同等以上のサイズで、スクロールせず見える位置に置く |
| Hide My Email | ユーザーがプライベートリレーアドレスを選んだ場合、個人メールアドレスの入力を要求してはならない |
| パスワード | パスワードを追加で要求しない |

### 推奨

- サインイン要求はできるだけ遅延させ、要求時は理由（得られる価値）を説明する。アカウント必須なら先にその旨を説明してから手段を提示。
- カスタムボタンを作る場合: ロゴ/テキスト色は黒か白のみ、Apple ロゴは配布版のみ使用、ボタン高 ≒ フォントサイズ × 233%（例: 高さ 44pt→19pt）、テキスト右マージンはボタン幅の 8% 以上。コーナー半径・フォントウェイト・背景テクスチャ（黒/白の範囲）は調整可。
- サインイン直後にウェルカムメッセージで新規ユーザーを迎え、収集データの用途を透明に示す。必須情報と任意情報を区別する。
- リレーアドレス利用ユーザーの照合には注文番号など別の識別子を使う。

### よくある違反・注意点

- 他社ソーシャルログインより Apple のボタンを小さく/下に置く（プロミネンス違反）。
- タイトル文言の独自変更（「Apple でログイン」ボタンの自作画像等はシステム API のローカライズに任せる）。
- Hide My Email 選択者に「連絡が取れないので本当のメールを」と迫る（明示的に禁止）。

公式: https://developer.apple.com/design/human-interface-guidelines/sign-in-with-apple

## 機械学習・Apple Intelligence・生成 AI（Machine Learning / Generative AI）

ML 機能の UX は「モデル精度」ではなく「役割定義・入力・フィードバック・誤り処理」の設計で決まる。生成 AI はさらに透明性とユーザーコントロールの要件が加わる。

### 🔴 必須（遵守事項）— ML 全般

- 機能の役割を 4 軸で定義してから設計する: Critical/Complementary（無いとアプリが成立しないか）、Private/Public（データ機密性）、**Proactive/Reactive**（ユーザーが求めていない提案は低品質に不寛容）、**Visible/Invisible**（見える機能は信頼性の可視化が必要）。
- キャリブレーション（初期データ収集）は機能動作に必須の場合のみ・最小限・体験の早期に行い、進捗表示と途中キャンセルを必ず提供、成功を明示する。
- 信頼度（confidence）は、値と結果品質の相関を検証済みの場合のみ表示。生の数値（「97% match」）ではなく人が理解できる根拠（「Because you listen to pop music」）や意味分類（High/Low chance）に変換する。数値表示は天気・スポーツ等の統計分野に限る。
- 根拠表示（attribution）は客観表現にする（◯「Because you've read mysteries」/ ✗「Because you love…」の感情断定、✗「6月3日に2.34分閲覧したため」の過剰な監視感）。
- 誤りの修正（correction）手段を提供する: 既知の UI パターンで、修正は即座に反映・永続化し、自由入力より選択肢提示（ガイド型）を優先。
- 制限（limitation）は使用前（説明・プレースホルダー）・使用中・失敗時に明示する（例: Face ID の「正面を向いてください」、Memoji の「Low light」）。

### 🔴 必須（遵守事項）— 生成 AI / Apple Intelligence

- **AI の使用箇所を明確に開示**し、AI 生成コンテンツを人間作成と誤認させない。地域の AI 規制要件にも準拠する。
- 生成結果に対する **Edit / Undo / Retry / Adjust** の手段を必ず実装する（2026 年 6 月更新で必須 UI 要素として明記）。
- ハルシネーション対策: 検証済み情報がある領域にスコープを限定し、日付・人物などの事実生成を避け、誤情報が実害になる用途では使わない。「AI 生成コンテンツは誤りを含みうる」ことを明記する。
- 破壊的・取り消し困難な自動操作（例: 写真の削除）は AI に単独実行させず、確認プロセスを挟む。
- 機密データ（メッセージ・写真等）をモデルに使う前に権限を取得し、オンデバイス処理を優先、サーバー送信は最小化する。オプトアウト手段を提供する。子ども向けアプリはさらに厳格（データ制限・保護者同意）。
- Foundation Models フレームワーク利用時は Apple の許容用途要件（Acceptable Use Requirements）を確認する。

### 推奨

- 明示的フィードバックは任意参加とし、曖昧な「Dislike」ではなく具体的選択肢（「Suggest less pop music」「Mute politics for a week」）を出して即座に全体へ反映する。
- 暗黙的フィードバック（再生時間・スキップ等）は複数信号を組み合わせ、単一信号で断定しない。最新の信号を優先する（Face ID が顔の変化に追従するのと同じ原理）。
- 複数候補の提示は 1 画面に収まる数で、多様性を持たせ（Maps の複数ルートのように）、最尤候補をデフォルトにする。
- レイテンシー対策: ローディングは具体的に（◯「Finding substitutions for ingredients」/ ✗「Processing…」）、可能ならバックグラウンド生成。
- スコープ外リクエスト・曖昧入力・有害誘導・個人情報混入を戦略的にテストし、拒否時は次の良いリクエスト方法を提案する（Image Playground の「Unable to use that description」パターン）。
- バイアス対策: 性別・人種・文化属性を推測せずユーザーに入力させる。多様な属性でテストする。
- サムズアップ/ダウン等の軽量フィードバック UI を、体験を中断しない位置に置く。

### よくある違反・注意点

- Proactive（頼まれていない）提案に低信頼度の結果を流す — 提案機能は保留するのが正しい。
- 信頼度スコアの生表示でユーザーを混乱させる。
- 修正フィードバックを学習に使うことを言い訳に、低品質結果を常態化させる。
- AI 表記なしで生成文・生成画像をユーザー作成コンテンツに混ぜる。

公式: https://developer.apple.com/design/human-interface-guidelines/machine-learning / https://developer.apple.com/design/human-interface-guidelines/generative-ai

## 拡張現実（Augmented Reality）

ARKit による iPhone/iPad の AR 体験。没入的だがユーザーは現実空間に立っているため、快適性と安全性が最優先。

### 🔴 必須（遵守事項）

- ARKit 対応デバイスでのみ AR 機能を提供する。AR がオプション機能なら非対応機ではエラーではなく単に機能を非表示にする。
- **60fps を維持**する（オブジェクトのちらつき・ジャンプ防止）。
- 初期化・リローカライゼーション時はコーチング UI（`ARCoachingOverlayView` またはそれに準じたカスタム）を表示し、その間は無関係の UI を隠す。
- ユーザー向けテキストに技術用語を使わない: ◯「表面を見つけられません。少し移動してみてください」/ ✗「プレーン検出失敗。トラッキングを調整してください」。
- 中断（アプリ切替・着信）後のリローカライゼーション中は仮想オブジェクトを隠し、キャンセル/リセット手段を提供する。条件改善を強制的に待たせない。
- 安全性: 急速・広範囲・後退を誘発する動きをユーザーに要求しない（AR 中は周囲への注意が落ちる前提で設計）。
- AR バッジ/グリフは Apple 提供のものを無改変で使用し、クリアスペースはグリフ高の 10% 以上。ARKit 体験の起動以外に使わない。

### 推奨

- オブジェクト配置は「即座に置いて、表面検出の精度向上に合わせ `ARTrackedRaycast` で微調整」する（正確なデータを待って配置を遅らせない）。平面分類（floor/table 等）を活かして配置先を制御する。
- ジェスチャーは標準に従う: 1 本指ドラッグ=移動（2D 表面上に制限）、2 本指回転=単一軸回転、ピンチ=スケール。スケールは仮想空間の探索用途のみ許可し、実寸可視化（家具プレビュー等）では無効化する。
- 参照画像は 100 枚以下で運用し、超える場合はコンテキストで有効セットを切り替える。検出画像が消えても最大 1 秒は仮想オブジェクトを保持しちらつきを防ぐ。顔追跡は約 0.5 秒以上のロストで視覚インジケーターを出す。
- ヒントは 3D 空間内表示を第一候補、2D オーバーレイは必要時のみ。重要なラベル・注釈はスクリーンスペースに置き、3D テキストは距離によらず一定サイズにする。
- 長時間セッションを避け（腕の疲労）、休止できる区切りを設ける。動きの要求は単純な動作から段階的に導入する。
- 問題時のメッセージ例: 低照度→「照明を増やして移動してみてください」、動きすぎ→「もう少しゆっくり動かしてください」。

### よくある違反・注意点

- 全アイテムが AR 対応なのに全部に AR バッジを付ける（冗長・不可）。
- 表面エッジへの厳密な整列を前提にする（検出境界は概算値で変動する）。
- AR セッションを離脱しないと使えない付随機能（例: 家具の色変更）— AR 内に埋め込み中断を最小化する。

公式: https://developer.apple.com/design/human-interface-guidelines/augmented-reality

## SharePlay

FaceTime/Messages のセッション越しに、参加者全員のアプリ状態を同期して一緒に視聴・共同作業する体験。iOS/iPadOS/macOS/tvOS/visionOS 対応（watchOS 非対応）。

### 🔴 必須（遵守事項）

- 表記規則: 「SharePlay」は名詞（「Join SharePlay」）または動詞（「SharePlay Movie」）として使う。「SharePlayed / SharePlaying」等の変形、「virtual SharePlay」等の形容詞修飾は禁止。
- 全参加者に同じアプリ状態を表示する（表示モードが複数あるなら全員で統一）。新規参加者が加わっても既存参加者の体験を中断しない。
- 参加に必要な準備（サインイン・支払い・コンテンツダウンロード）はアクティビティ開始**前**に完了させ、完了後にアクティビティ UI を出す。

### 推奨

- アクティビティ（GroupActivities の共有単位）はアプリで複数定義してよい（映画視聴/番組視聴など）。各アクティビティにタイトル・短いサマリー・ポスター画像を付け、説明は切り詰められない長さにする。
- セッションがまだ無い場合は共有開始 UI を出し、システムの「共有 or ソロ実行」確認に委ねる。
- 非購読の参加者への対応: 一時/試用アクセス、購読者からのワンタイムパス、Family Sharing、SharePlay 中の簡略サインアップを検討する。
- プロフィール取得などの低優先処理は共有開始後・一時停止時に遅延実行する。
- iPhone/iPad では共有ビデオの Picture in Picture 再生をサポートする。
- 操作権の管理 UI は最小化し、「最後の変更が有効」のような単純なルールにして、音声会話での自然な調整に任せる。

### よくある違反・注意点

- 参加者ごとに異なる再生位置・状態を許してしまい「一緒に見ている」前提が崩れる。
- 参加フローの途中に長い登録・決済を挟み、グループから脱落させる。

公式: https://developer.apple.com/design/human-interface-guidelines/shareplay

## マップ（Maps / MapKit）

MapKit による地図表示・アノテーション・オーバーレイ・プレイスカードの設計。

### 🔴 必須（遵守事項）

- Apple ロゴと法的表記リンクを常に可視に保つ: パディング左右 7pt / 上下 10pt、UI の動きに追従させず固定位置に置く。**200×100px 未満**のマップでは表示されないため、それより小さい地図にしない。
- マップはズーム・パン・回転の標準操作に対応させ、非インタラクティブ要素で地図を常時遮蔽しない。

### 推奨

- 強調スタイル: 標準の地図体験なら Default（フル彩度）、カスタム要素を主役にするなら Muted（低彩度、`MKStandardMapConfiguration.EmphasisStyle`）。
- アノテーション（`MKAnnotationView`）: ティント色はアプリのカラースキームに合わせてよい。アイコン文字は 2〜3 文字まで。近接する POI はクラスタリングし、ズームで段階的に展開する。
- カスタムコントロールと地図の間はコントラストを確保する（細いストローク・ドロップシャドウ・ブレンドモード）。
- オーバーレイのレベルを意図で選ぶ: Above roads（道路の上・建物/ラベルの下 = 地形を見せつつ領域表示）/ Above labels（全要素の上 = 完全に抽象化）。
- プレイスカード（`MKSelectionAccessory`）: 小さいマップに Full callout は過剰 — Automatic / Compact / Caption（「Apple Maps で開く」リンクのみ）/ Sheet から表示面積に応じて選ぶ。選択地点が隠れないようオフセットし、既に画面にある情報を重複表示しない。
- インドアマップ: ズームアウト時は大区画のみ、ズームインで店舗・トイレ等の詳細を出す。色＋アイコンの併用、複数フロアはフロアピッカー、周辺エリアは薄色で補足扱い。Maps アプリの複製ではなく自アプリのビジュアルスタイルに合わせる。

### よくある違反・注意点

- 地図を装飾として敷き、操作不能かつロゴ/法的表記を隠す。
- 大量の POI ピンを未クラスタリングで表示し可読性を壊す。
- 選択状態のアノテーションに明確なスタイル差（アウトライン＋色変更）を付けず、何を選んだか分からない。

公式: https://developer.apple.com/design/human-interface-guidelines/maps
