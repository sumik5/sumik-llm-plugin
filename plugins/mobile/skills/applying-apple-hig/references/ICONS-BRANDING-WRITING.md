# アイコン・ブランディング・UXライティング

> 出典: Apple Human Interface Guidelines / Apple Developer Documentation（最終確認 2026-07-13）。仕様は更新されるため、実装時は必ず併記の公式 URL で最新版を確認すること。日本語版がある場合は公式の言語切替または本文の日本語 URL を使い、英語版との差分がある場合は最新側を確認する。
> 判定区分: 本文の「必須」「違反」は作業用の強度ラベルであり、HIG の推奨を一律に App Review 必須へ格上げしない。審査・API 上の必須判定は現行の Apple 公式一次資料で再確認する。

iPhone/iPad（iOS/iPadOS）アプリの視覚的アイデンティティとテキストコミュニケーションに関する実装ガイド。アプリアイコン仕様、インターフェイスアイコン、SF Symbols、ブランド表現の節度、UXライティング原則をカバーする。

## アプリアイコン（App Icons）

### 概要

アプリアイコンはホーム画面・検索結果・通知・設定・共有シートに表示される、ブランディングと UX の最重要資産。iOS/iPadOS では単一のフラット画像ではなく**レイヤード構造**（背景レイヤー + 1 つ以上の前景レイヤー）で提供し、システムが Liquid Glass 効果（スペキュラハイライト・屈折・透光性）と角丸マスクを動的に適用する。

### 🔴 必須（遵守事項）

| 項目 | 仕様 |
|------|------|
| レイアウトサイズ | **1024×1024 px** の正方形 1 枚（システムが全サイズへスケーリング） |
| 形状 | **正方形のまま提供**すること。角丸はシステムが適用する（事前マスク済みレイヤーは禁止 — ハイライト効果が破綻しエッジが欠けて見える） |
| レイヤー構成 | 背景レイヤー + 1 つ以上の前景レイヤー（Icon Composer で構成し Xcode へエクスポート） |
| 対応アピアランス | **6 種**: Default / Dark / Clear Light / Clear Dark / Tinted Light / Tinted Dark。全アピアランスを提供すること |
| カラースペース | sRGB（カラー）/ Gray Gamma 2.2（グレースケール）/ Display P3（ワイドガモット） |
| 前景レイヤーのエッジ | 明確に定義されたハードエッジを使うこと（ソフト/フェザリングされたエッジはシステムの強調表示・影と相性が悪い） |
| 主要コンテンツの配置 | 中央に置くこと（角丸適用時の切り詰め防止） |
| 禁止 | Apple ハードウェア製品のレプリカ（知的財産権侵害）／UI コンポーネントの複製・スクリーンショット（実アプリとの混同を招く） |
| アピアランス間の一貫性 | アピアランスごとに構成要素を入れ替えてはならない（切り替え時にユーザーがアプリを見失う） |

### 推奨

- **シンプルな単一コンセプト**をアプリの本質から選び、最小限のシェイプ数で表現すること。細部が多いとシステムのハイライト/影と干渉し、小サイズで潰れる。
- **ベクター形式（SVG/PDF）で制作**し、テキストはアウトライン化すること。メッシュグラデーションやラスター画像を含む場合は PNG（可逆圧縮）。
- **背景は単色またはグラデーション**を推奨。システムのライティング効果に馴染むグラデーションを選ぶ。カスタム背景画像を使う場合はフルブリード＋不透明にする。
- **前景レイヤーで不透明度を変化**させると深度と躍動感が増す。半透明レイヤーの重なりは Icon Composer でシステム効果との相互作用を必ずプレビューする。
- **写真ではなくイラストレーション**を使うこと。写真は細部が多く、Dark/Tinted アピアランスと小サイズで機能しない。
- **Dark アイコンは Light アイコンをベース**に、パレットを保ちつつ補完色を選ぶ。過度に明るい画像を避け、背景色でコントラストを確保する。
- スペキュラハイライト・ドロップシャドウ・ベベル・ブラー・グロー等の**視覚効果を自前で描き込まない**こと。システム効果が動的に適用されるため競合する。カスタム効果が必要な場合は Icon Composer と実機で厳密にテストする。
- 複数レイヤーに効果を一括適用したい場合は Icon Composer の**レイヤーグループ**を使う（グループ単位でスペキュラハイライト・屈折・透明度を制御可能）。
- 代替アイコン（Alternate App Icons）を提供する場合: 現行OSが要求する変種とApp Review上の扱いを公式資料で再確認し、アプリの内容に密接に関連させ、別アプリと誤認させるデザインを避ける。
- グリッド・テンプレートは [Apple Design Resources](https://developer.apple.com/design/resources/) を使用する。

### よくある違反・注意点

- ❌ 角丸を焼き込んだ 1024px 画像を 1 枚だけ提供する（レイヤー・アピアランス変種なし）
- ❌ アイコン内に「New」「Pro」「For iPad」等の非本質的な単語を入れる（アプリ名の頭文字などのニーモニックは可。テキストはローカライズ不可・アクセシビリティ非対応・可読性低の三重苦）
- ❌ 極細線・鋭利なコーナー多用（小サイズで消失する）
- ❌ Dark アピアランス未提供のまま Light 用アイコンを流用し、ダークモードのホーム画面で浮く

公式: https://developer.apple.com/design/human-interface-guidelines/app-icons

## インターフェイスアイコン（Icons）

### 概要

インターフェイスアイコン（グリフ）は、ボタン・タブ・リスト等で単一の概念を瞬時に伝えるグラフィック。アプリアイコンと異なり装飾を排し、形状のみで意味を定義し、色はシステム/アプリのアクセントカラーが適用される。カスタム制作・SF Symbols 利用・両者の併用のいずれも可。

### 🔴 必須（遵守事項）

- **ベクター形式（PDF/SVG）で提供**すること。PNG はスケーリング非対応で解像度ごとに複数バージョンが必要になるため避ける。
- **アクセシビリティラベル（代替テキスト）を全アイコンに付与**すること。VoiceOver ユーザーのナビゲーションに必須。
- **標準システムコンポーネント（タブバー・ツールバー・ボタン）内では選択状態を自作しない**こと。システムが選択時の外観（アクセントカラー適用）を自動処理する。
- アイコン内にテキストを含める場合（テキスト書式アイコン等、必須の場合のみ）: **個別文字はローカライズ必須**（ラテン・アラビア・ヘブライ・ヒンディー・日本語・韓国語・タイ・中国語）。RTL コンテキスト用の反転版も用意すること。
- Apple ハードウェア製品のレプリカを使わないこと（デザイン変更で陳腐化する）。Apple 製品を表す必要がある場合は Apple Design Resources の公式画像か SF Symbols のみを使う。

### 推奨

- **高度に単純化**し、アクションや内容に直結する一般的な視覚的メタファーを使うこと。細部が多いと認識性が落ちる。
- アイコン群全体で**サイズ・詳細度・ストロークウェイト・パースペクティブを統一**する。視覚的な重さに応じて個別調整すること（例: 軽く見える時計アイコンは、視覚的に重いカメラアイコンと並べるため高さを増す)。
- **隣接テキストとウェイトを一致**させ、コンテンツに一貫した強調レベルを与える。
- **光学的中央配置（optical centering）**: 非対称アイコン（例: 下部に重心があるダウンロードアイコン）は幾何学的中央ではズレて見える。視覚的中央に微調整（通常は数 px）し、その調整分をパディングとしてアセットに含めてから幾何学的に配置する。
- **インクルーシブな図像**を選ぶこと: ジェンダーニュートラルな人物表現を使い、特定の文化・言語圏でしか通じない画像を避ける。

### よくある違反・注意点

- ❌ PNG のみで提供し @2x/@3x の管理が破綻する
- ❌ 出典の異なるアイコンを混在させ、線の太さ・角の丸みがバラバラになる
- ❌ 非対称グリフを bounding box の幾何中心に置いたまま「ズレて見える」状態で出荷する
- ❌ アクセシビリティラベル欠落（VoiceOver で「ボタン」としか読まれない）
- ❌ タッチデバイスで意味の通らない文化依存メタファー（例: 特定国の郵便ポスト形状でメールを表す）

公式: https://developer.apple.com/design/human-interface-guidelines/icons

## SF Symbols

### 概要

SF Symbols は San Francisco システムフォントとシームレスに統合される数千のシンボル集。テキストとのウェイト整合・スケール調整・レンダリングモード・アニメーションをシステムが提供するため、インターフェイスアイコンの第一選択とすべき。[SF Symbols](https://developer.apple.com/sf-symbols/) で検索・カスタマイズ・書き出しを行う。

### 🔴 必須（遵守事項）

- **意味が確立されたシンボルを本来の意味で使う**こと。システム定義のシンボルを別の意味に流用してはならない。
- **SF Symbols をアプリアイコン・ロゴ・商標に使用してはならない**（ライセンス上の禁止事項）。
- **SF Symbols アプリで Info バッジが付くシンボル**（Apple が特定機能の表現として予約するもの）はカスタマイズ・転用禁止。
- Apple 製品/機能のレプリカとなるカスタムシンボルを自作しないこと。
- VoiceOver 用の**代替テキストラベルを付与**すること（システムシンボルは既定ラベルを持つが、文脈上意味が変わる場合は上書きする）。

### ウェイト・スケール・レンダリングモード

| 軸 | 選択肢 | 指針 |
|----|--------|------|
| ウェイト | **9 種**: Ultralight / Thin / Light / Regular / Medium / Semibold / Bold / Heavy / Black | 隣接テキストのフォントウェイトと一致させる |
| スケール | **3 種**: Small / Medium（デフォルト）/ Large | San Francisco の大文字高（cap height）基準。テキストサイズを変えずに強調度を調整する（SwiftUI: `imageScale(_:)`、UIKit: `UIImage.SymbolScale`） |
| レンダリングモード | **4 種**: Monochrome（単色）/ Hierarchical(単色を階層別不透明度で適用・深度表現) / Palette（レイヤーごとに 2〜3 色）/ Multicolor（固有色内在。例: 緑の `leaf`、赤の `trash.slash`） | サイズと背景コントラストごとに全モードを検証する。自動選択に任せる場合も結果確認は必須 |

- **グラデーション**（SF Symbols 7+）: 単一ソースカラーから直線グラデーションを生成。全レンダリングモード・全サイズ対応（大サイズで最も効果的）。
- **可変色（Variable Color）**: 0〜100% の値でレイヤーに段階的に色を適用。進行状況・信号強度等の表現に使う。

### iOS/iPadOS のバリアント自動選択

| コンテキスト | 自動優先されるバリアント |
|--------------|--------------------------|
| タブバー | Fill（塗り） |
| ツールバー・ナビゲーションバー | Outline（線） |
| スワイプアクション | Fill + アクセントカラー |

システムコンポーネント内ではバリアントが自動選択されるため、手動で fill/outline を固定しないこと。

### アニメーション

シンボルエフェクトとして Appear / Disappear / Bounce / Scale / Pulse / Variable Color / Replace / Magic Replace（関連形状の知的遷移・置換のデフォルト）/ Wiggle / Breathe / Rotate / Draw On・Draw Off（SF Symbols 7）等が利用できる。使用指針:

- アクション実行フィードバックには Bounce、進行中の活動には Pulse や Variable Color、状態変化には Replace を使うなど、**動きの意味と解釈を一致**させること。
- 多数のアニメーションの組み合わせは避け、目的のあるものだけ使う。
- アプリのトーンとの整合を確認する。

### カスタムシンボル作成の注意

- **既存シンボルのテンプレートから開始**し、詳細度・光学的ウェイト・配置・パースペクティブを既存シンボル群に揃えること。
- レイヤーに Primary / Secondary / Tertiary の階層を割り当て、レンダリングモード（Hierarchical/Palette）で意図通りに着色されるようにする。
- 形状は**カットアウト（切り抜き）ではなくフルシェイプ + 消去レイヤー（erase layer）**で構成すること。レイヤー情報が保持されアニメーションの挙動が予測可能になる。
- 囲み（enclosure）やバッジ等の共通バリアントは自作せず、SF Symbols のコンポーネントライブラリを使うこと。
- 全アニメーションプリセット・全レンダリングモードでテストすること。
- 自動ローカライズ（アラビア語・ヘブライ語・中国語・日本語・韓国語等のスクリプト対応シンボル）はシステムがデバイス言語に応じて切り替えるため、対応シンボルがある場合はそれを使う。

### よくある違反・注意点

- ❌ SF Symbol をラスタライズして PNG で埋め込む（ウェイト整合・Dynamic Type 連動・アクセシビリティ適応が全て失われる）
- ❌ テキストが Semibold なのにシンボルが Regular のまま（視覚的な強調不一致）
- ❌ Multicolor の固有色を無視して意味の逆転する色を強制する
- ❌ Info バッジ付きシンボル（Apple 予約シンボル）の転用
- ❌ 常時ループする装飾アニメーションで注意を奪う

公式: https://developer.apple.com/design/human-interface-guidelines/sf-symbols

## ブランディング（Branding）

### 概要

HIG のブランディング原則は一貫して「**コンテンツが主、ブランドは従**」。ブランドはロゴの露出ではなく、アクセントカラー・タイポグラフィ・ボイス＆トーン・全体の体験品質を通じて表現する。

### 🔴 必須（遵守事項）

- **Apple の商標をアプリ名・画像に含めてはならない**（Apple Trademark List / Guidelines for Using Apple Trademarks に従う）。
- **ブランド表示のためだけに画面スペースを占有しない**こと。その領域はユーザーにとって価値ある情報とコントロールに充てる。
- **ローンチスクリーン（起動画面）をブランディング機会として使わない**こと。ローンチスクリーンは一瞬で消えるため情報伝達に不適。ブランドコンテンツはウェルカム画面やオンボーディングに置く。
- **アプリ全体にロゴを常時表示しない**こと（文脈提供に本当に必要な場合を除く）。ユーザーはどのアプリを使っているか既に知っている。

### 推奨

- **アクセントカラーの指定**: システムがアイコン・ボタン・テキスト等に適用するアクセントカラーでブランドを想起させる。詳細は Color ガイドライン参照。
- **カスタムフォント**はブランドとの結びつきが強い場合のみ検討し、次を満たすこと: 全サイズで可読性を維持／ボールドテキスト・大型テキスト等のアクセシビリティ機能に対応。推奨戦略は「見出し・小見出しのみカスタムフォント、本文・キャプションはシステムフォント」（小サイズの可読性はシステムフォントが最適化済み）。
- **ボイス＆トーンの一貫性**: すべての文言でブランド固有のボイスを使う（例: 平易な言葉・時折の感嘆符・絵文字・短い文構造で楽観性を表現する等）。
- **標準パターンを維持**すること: 高度にスタイライズされた UI でも、コンポーネントを期待される位置に置き、一般的なアクションには標準シンボルを使えば親近感が保たれる。
- ブランディング要素は**洗練された控えめな方法**で統合し、ユーザーが体験そのものに集中できるようにする。
- App Store ではアプリアイコンと体験全体がブランドのマーケティング機会になる（App Store Marketing Guidelines / App Previews 参照）。

### よくある違反・注意点

- ❌ 全画面に固定ヘッダーロゴを常駐させる
- ❌ 起動画面に「ロゴ + キャッチコピー + バージョン番号」を数秒表示する演出（ローンチスクリーンは次画面の骨格を先行表示するためのもの）
- ❌ ブランドカラー至上主義でコントラスト不足・ダークモード非対応の配色を強行する
- ❌ 本文までカスタムフォントにして小サイズの可読性と Dynamic Type 対応を壊す

公式: https://developer.apple.com/design/human-interface-guidelines/branding

## UXライティング（Writing）

### 概要

オンボーディング・ボタン・アラート・エラー・設定・空状態まで、アプリのあらゆる言語要素を設計対象とする。核となる概念は **ボイス（アプリの恒常的な個性）** と **トーン（状況に応じた表現の調整）** の分離。例: 深刻な通知は直接的・簡潔に、達成の祝福は軽快に — ボイスは同一のままトーンだけ変える。

### 🔴 必須（遵守事項）

- **明確性を最優先**すること。各単語の必要性を検証し、より少ない語数で同じ意味を伝える。書いた文は音読して確認する。
- **ボタン・リンクは動詞で行動指向**にすること。「Send」は良く「Let's do it!」は不可。「Click here」はスクリーンリーダーで意味を成さないため禁止 —「Learn more about ...」のようにリンク先を語で示す。
- **タッチデバイスでは「tap」を使い「click」を使わない**こと（iPhone/iPad のジェスチャー語彙）。
- **エラーメッセージでユーザーを非難しない**こと。「Oops!」「Uh-oh」等の軽薄・不誠実な表現も禁止。問題箇所の近くに表示し、解決方法を具体的に示す。
  - ❌ `That password is too short` → ✅ `Choose a password with at least 8 characters`
  - ❌ `Invalid name` / `Don't use numbers or symbols` → ✅ `Use only letters for your name`（指示は禁止形でなく指導形で）
- **「we」を使わない**こと（誰を指すか不明確）。❌ `We're having trouble loading this content` → ✅ `Unable to load content`
- **すべてのテキストフィールドにラベルを付け**、形式例（`name@example.com`）や説明（`Your name`）のヒントテキストを提供すること。

### 大文字小文字規則（キャピタライゼーション）

| スタイル | 性格 | 使用場面 |
|----------|------|----------|
| **Title Case**（主要語の頭を大文字） | フォーマル | アラートタイトル、ナビゲーションタイトル、ボタンラベル等の格式的 UI 要素 |
| **Sentence case**（文頭のみ大文字） | 会話的 | 見出し・説明文などカジュアルな要素 |

- **UI 要素タイプ単位でルールを固定**すること（例:「アラートは常に Title Case」）。同一要素タイプ内での混在は禁止。
- どちらを基調とするかはアプリのボイスに合わせて選ぶ。ボタンラベルには個別ガイドラインがあるため該当コンポーネントの HIG を併読する。

### 推奨

- **用語リストを整備**し、頻出概念の呼び方を全画面で統一する（同じ機能を画面ごとに別名で呼ばない）。
- **言語パターンの再利用**: 一貫したパターンが「使い勝手」と直感的理解を生む。複数ステップフローでは「Get Started →（次ステップ名 or Continue/Next）→ Done」のようにラベル体系を統一する。
- **所有代名詞（my/your)は原則省く**: `Your Favorites` ではなく `Favorites`。使う場合はアプリ全体で視点を統一し、my と your を混在させない。
- **空状態（empty state）**: ユーザーを歓迎し、アプリの学び方を伝え、次のアクション（ボタン/リンク）を明示する。空状態は一時的なので、そこにしか出ない重要情報を置かない。
- **設定項目**: ラベルは簡潔に、説明は ON 時の動作を書く（例: ラベル `Handwashing Timer` + 説明「手洗いを検知して 20 秒タイマーを開始する」— OFF 時の挙動は推測できる）。設定画面への誘導は場所の説明文ではなく直接リンク/ボタンで行う。
- **配信方法の選択**: メッセージの緊急度・重要度・閲覧文脈・即時対応の要否・補足情報量に応じて、通知/アラート/アクションシートを使い分ける。
- **画面上の情報順序**: 最重要情報を先頭に。複数の概念を扱う場合は複数画面に分割して情報フローを設計する。
- **デバイス別の簡潔性**: iPhone/iPad は画面が小さいため特に簡潔に。ジェスチャー表現の正確性を保つ。
- **万人向けに書く**: 平易な語彙・専門用語回避・ジェンダー特定表現の回避・スクリーンリーダー（VoiceOver）とローカライズを前提にした文を書く（参考: Apple Style Guide の Writing Inclusively）。
- エラーが多数のユーザーで発生する場合は、文言修正ではなく**エラーを予防するインタラクション設計の見直し**を優先する。

### よくある違反・注意点

| アンチパターン | 例 | 問題 |
|----------------|-----|------|
| 曖昧な主語 | `We're having trouble...` | 「We」が誰か不明でユーザーを混乱させる |
| 冗長な所有代名詞 | `Your Favorites` | `Favorites` で十分 |
| 過度に巧妙な表現 | Send の代わりに `Let's do it!` | ボタンの目的が曖昧になる |
| 責任追及的エラー文 | `You entered a wrong password` | 非難的で不快 |
| 感情的な間投詞 | エラー時の `Oops!` `Uh-oh` | 深刻な文脈で不誠実に響く |
| デバイス語彙違反 | タッチデバイスで `click` | プラットフォーム慣習違反 |
| 要素タイプ内でのケース混在 | あるアラートは Title Case、別のアラートは Sentence case | 統一ルールの欠如が雑な印象を生む |
