# 基本パターン（起動・オンボーディング・入力・検索・設定・フィードバック）

> 出典: Apple Human Interface Guidelines / Apple Developer Documentation（最終確認 2026-07-13）。仕様は更新されるため、実装時は必ず併記の公式 URL で最新版を確認すること。日本語版がある場合は公式の言語切替または本文の日本語 URL を使い、英語版との差分がある場合は最新側を確認する。
> 判定区分: 本文の「必須」「違反」は作業用の強度ラベルであり、HIG の推奨を一律に App Review 必須へ格上げしない。審査・API 上の必須判定は現行の Apple 公式一次資料で再確認する。

iOS/iPadOS アプリのライフサイクル全体（起動 → 学習 → 入力 → 検索 → 設定 → フィードバック → 取り消し）で従うべき HIG パターン集。実装判断に直結する形（〜すること / 〜してはならない）で記述する。

## 起動（Launching）

アプリ起動の第一印象を決める Launch Screen と初回起動体験の要件。iOS/iPadOS では Launch Screen の提供が**必須**。

### 🔴 必須（遵守事項）

| 項目 | 要件 |
|------|------|
| Launch Screen 提供 | iOS/iPadOS では必須。ストーリーボードまたは `UILaunchScreen` Info.plist キーで定義すること |
| 起動速度 | 数秒以内に操作可能になること。起動の遅延を Launch Screen で覆い隠さない |
| 最初の画面との同一性 | Launch Screen は起動後の最初の画面とほぼ同一のビジュアルにし、遷移時のちらつきを避けること |
| テキスト禁止 | Launch Screen にテキストを含めてはならない（Launch Screen はローカライズされないため） |
| 広告・ブランディング禁止 | Launch Screen を広告やスプラッシュ演出の場にしてはならない。ロゴは最初の画面に固定要素として存在する場合のみ許容 |
| Orientation | デバイスの現在の向きで起動すること。単一 orientation のみ対応の場合はその向きで起動。landscape 専用アプリは左右両回転に対応すること |
| Appearance | デバイスの現在の light/dark モードに一致させること |

### 推奨

- **前回の状態を復元する**: スクロール位置・選択状態・編集中の内容など、ユーザーが前回終了した位置から継続できるようにする（`UISceneDelegate` の state restoration / SwiftUI `@SceneStorage`）。
- 最初の画面が単色から始まる場合、Launch Screen もその単色のみを表示する。
- ブランド演出（splash screen）を入れたい場合は、Launch Screen ではなくオンボーディングフローの冒頭に置くことを検討する。

### よくある違反・注意点

- ❌ ロゴ・バージョン番号・コピーライト入りのスプラッシュ画像を Launch Screen にする（テキスト禁止・広告禁止の両方に違反）。
- ❌ Launch Screen を意図的に長く表示して「ブランド体験」にする。
- ❌ 起動のたびに初期状態に戻る（状態復元の欠如はユーザーの作業を破壊する）。

公式: https://developer.apple.com/design/human-interface-guidelines/launching

## オンボーディング（Onboarding）

アプリの価値を素早く体験させる初回導入。オンボーディングは「起動体験の一部」ではなく起動完了後に行い、**高速・楽しい・スキップ可能**であること。

### 🔴 必須（遵守事項）

- チュートリアルはオプショナル（スキップ可能）にすること。スキップした内容は後から「ヘルプ／アカウント／設定」等で参照できるようにすること。
- アプリ再起動時にチュートリアルを再表示してはならない。
- ライセンス詳細・利用規約全文をオンボーディングに含めない（App Store 側で表示する。法的に必須なら体験を損なわない形で統合）。

### 推奨

- **インタラクティブに教える**: 説明を読ませるより、実際に操作させて学ばせる方が定着する。
- **コンテキスト固有の Tips を優先する**: 一括のオンボーディングフローより、機能に到達した時点で TipKit 等により状況に応じたヒントを出す方が効果的。
- **体験に集中する**: システムやデバイスの標準操作（スワイプの仕方等）は教えない。アプリ固有の価値だけを伝える。
- **権限リクエストは利点の説明とセットで**: コア機能に必須のデータアクセス許可はオンボーディング内に統合し、なぜ必要かを先に説明する。必須でない権限はその機能を使う時点まで遅延する。
- **設定・カスタマイズを強要しない**: 合理的なデフォルトで開始させ、カスタマイズは後回しにできるようにする。
- **評価依頼・購入プロンプトを出さない**: ユーザーがアプリに十分慣れてから提示する。
- 起動直後に大容量ダウンロードを挟まない。すぐ操作を始められるよう必要最小限のアセットをアプリに同梱する。

### よくある違反・注意点

- ❌ 5 ページ超のスワイプ式説明カルーセルを強制する（記憶負荷が高く、ほぼ読まれない）。
- ❌ 初回起動で位置情報・通知・トラッキングの許可を一括要求する（文脈のない権限要求は拒否率を上げる）。
- ❌ オンボーディング完了までコア機能を一切触らせない。

公式: https://developer.apple.com/design/human-interface-guidelines/onboarding

## ローディング（Loading）

コンテンツ読み込み中の空白を避け、待ち時間の体感を短くするパターン。

### 🔴 必須（遵守事項）

- 読み込みに**一瞬以上かかる場合**は、進捗インジケータ等で「読み込み中である」ことを明示すること。無反応の空白画面を見せてはならない。
- 読み込み時間が予測できる場合は determinate（進捗バー型）、予測できない場合は indeterminate（スピナー型）のインジケータを使うこと。

### 推奨

- **即座に何かを表示する**: プレースホルダ（スケルトン UI）・背景グラフィック・アニメーションを先に出し、コンテンツを順次差し込む。
- **バックグラウンドで読み込む**: 読み込み中もユーザーが他の操作を継続できるようにし、UI 全体をブロックしない。
- 長い読み込みには興味を保つコンテンツ（ゲームのヒント・新機能紹介等）を表示する。
- 大容量アセット（ゲームレベル・3D モデル・テクスチャ等）は BackgroundAssets フレームワークでインストール直後や更新時など非干渉的なタイミングに事前ダウンロードする。
- 残り時間表示は正確に。実際より短く見せて延びる表示は不信を生む。

### よくある違反・注意点

- ❌ 読み込み完了まで真っ白／真っ黒の画面のまま放置する。
- ❌ 全画面モーダルのスピナーで全操作をブロックする（部分読み込みで済む場面が大半）。
- ❌ 3 秒で終わる処理に進捗バー、30 秒かかる処理にスピナー（determinate/indeterminate の選択ミス）。

公式: https://developer.apple.com/design/human-interface-guidelines/loading

## フィードバック（Feedback）

操作の結果と現在の状態をユーザーに伝える方法。フィードバックが伝えるべきは次の 4 種: ①現在のステータス ②重要なタスクの成功/失敗 ③負の結果につながる警告 ④ミスの修正機会。

### 🔴 必須（遵守事項）

- **複数のモダリティで伝えること**: 色・テキスト・サウンド・ハプティクスを組み合わせ、消音時・画面を見ていない時・VoiceOver 使用時でも受け取れるようにする（色だけに依存するとアクセシビリティ違反）。
- **警告（alert）は予期しない・不可逆なデータ損失の場合のみ**出すこと。ユーザー自身が意図した削除など、予期された結果には警告を出さない。
- エラーメッセージには「なぜ実行できないか」を明確に記述すること（例: Maps の「現在地と同じ場所への経路は提供できません」）。

### 推奨

- **ステータスは UI に統合して表示する**: 対象の近くに配置し、ユーザーの追加操作なしで確認できるようにする（例: Mail の未読件数バッジ）。
- **alert は重要かつ実行可能（actionable）な情報に限定する**: 乱用するとユーザーが読まなくなる。割り込みの強さを重要度に一致させる。
- 重要なタスクの完了は明示的に確認する（例: Apple Pay の決済成功表示）。ただしユーザーは成功を前提とするため、多くの場面では**失敗時のみ**通知すれば足りる。
- ハプティクス・サウンドの詳細は HIG の Playing haptics / Playing audio を参照して補完する。

### よくある違反・注意点

- ❌ 全ての操作に確認アラートを出す（alert 疲れで本当に重要な警告が無視される）。
- ❌ 成功トーストを画面遷移のたびに出す（ノイズ）。
- ❌ エラーを「エラーが発生しました」だけで済ませ、原因と次の行動を示さない。
- ❌ 赤色表示のみでエラーを示す（色覚多様性・VoiceOver 非対応）。

公式: https://developer.apple.com/design/human-interface-guidelines/feedback

## モーダリティ（Modality）

親ビューとの相互作用を遮断し、明示的なアクションでのみ閉じられる専用モード。**明確な利益がある場合のみ**使う。

### 🔴 必須（遵守事項）

- モーダルを使うのは次のいずれかに限ること: ①重要情報の受信と対応が必要 ②直前アクションの確認・変更機会の提供 ③元のコンテキストを失わずに完結する独立タスク ④没入体験や複雑なタスクへの集中。
- **閉じる手段を必ず提供すること**: iOS/iPadOS ではトップツールバーのボタン（Done / Cancel / X）またはシートのスワイプダウン。アクセシビリティのため、スワイプダウンだけに頼らず明示的な閉じるボタンを必ず置くこと。
- **未保存のユーザー生成コンテンツがある状態で閉じる操作をした場合は確認を出すこと**（iOS では action sheet / confirmation dialog で「破棄」「保存」等の選択肢を提示）。
- 複数のモーダルビューを同時にスタック表示してはならない。1 つを完全に閉じてから次を提示すること。alert も同時に複数表示しない。

### 推奨: コンポーネントの使い分け（iOS/iPadOS）

| 用途 | コンポーネント |
|------|--------------|
| 重要かつ実行可能な情報の通知 | Alert |
| スコープが限定された関連タスク（作成・編集フォーム等） | Sheet（必要に応じて detent で高さ調整） |
| 直前のアクションに関連する選択肢の提示・データ損失警告 | Action Sheet（confirmation dialog） |
| 共有 | Activity View（share sheet） |
| 動画・写真・カメラ・マークアップ等の没入的/複数ステップタスク | Full-screen modal |

- モーダル内のタスクはシンプル・短時間・単一目的に保つ。タイトルと説明でタスク内容を明示する。
- 「アプリ内アプリ」化を避ける: モーダル内に多階層のナビゲーションを作らない（戻り道を見失う）。

### よくある違反・注意点

- ❌ 単なる詳細表示に full-screen modal を使う（push ナビゲーションで足りる）。
- ❌ シートの上にシートを重ね、さらに alert を出す。
- ❌ 閉じるボタンと紛らわしい位置・見た目のアクションボタンを置く。
- ❌ 入力途中のフォームをスワイプダウンで警告なしに破棄する。

公式: https://developer.apple.com/design/human-interface-guidelines/modality

## データ入力（Entering data）

入力はコストの高い操作。**ユーザーが手入力する量を最小化する**ことが大原則。

### 🔴 必須（遵守事項）

- パスワードフィールドに値を事前入力してはならない。機密入力には `SecureField`（入力を ● 表示）を使うこと。
- 必須項目が未入力の間は「Next」「Continue」等の進行ボタンを有効化しないこと（進んでからエラーで戻すのではなく、進行自体を止める）。
- 入力値は**入力中に動的にバリデーションする**こと。送信時にまとめてエラーを返すのではなく、即座に修正機会を与える。

### 推奨

- **システムから取得できる情報は入力させない**: 位置情報・カレンダー・連絡先等から（ユーザーの許可を得て）自動取得する。
- **合理的なデフォルト値で事前入力**し、意思決定と入力時間を削減する。
- **自由入力より選択を優先する**: 値の候補が有限なら、テキストフィールドではなく picker・メニュー・選択コンポーネントを使う。
- テキストフィールドには分かりやすいラベル（例:「Email」）と形式を示すプレースホルダ（例:「username@company.com」）を付ける。
- 数値入力には Number Formatter を使い、数値以外を受け付けない構造にする。
- ペースト・ドラッグ&ドロップ等、利用可能な全ての入力手段をサポートする（iPadOS ではハードウェアキーボード・ポインタ入力も考慮）。
- キーボードタイプ（`keyboardType`）・`textContentType`（AutoFill 用）を項目に応じて正しく設定する。

### よくある違反・注意点

- ❌ 都道府県や国名を自由テキストで入力させる（picker を使う）。
- ❌ 送信ボタン押下後に初めて全エラーを表示する。
- ❌ プレースホルダをラベル代わりに使う（入力開始と同時に消え、何の欄か分からなくなる）。
- ❌ メール欄に標準キーボード（`@` の出ないレイアウト）を出す。

公式: https://developer.apple.com/design/human-interface-guidelines/entering-data

## 検索（Searching）

アプリ内コンテンツの発見手段。アプリ内検索と Spotlight（システム全体検索）の両方を設計する。

### 🔴 必須（遵守事項）

- アプリ内コンテンツの検索には標準の検索フィールド（`.searchable` / `UISearchController`）を使うこと。
- 検索履歴を表示する場合は、**ユーザーが履歴をクリアできる手段を必ず提供する**こと（他人に画面を見られる可能性を考慮）。
- 現在の検索スコープを明確に示すこと: スコープバー・説明的なプレースホルダテキスト・タイトルのいずれかで「何を検索しているか」を伝える（例: Mail は検索対象のメールボックスを常に明示）。

### 推奨

- 検索がアプリの主要機能なら、目立つ位置に常設する（例: ボトムツールバーやタブとしての検索）。
- **検索場所は 1 箇所に統一する**ことを目指す（全コンテンツを横断検索できる単一の入口）。セクションが明確に分かれるアプリではローカル検索の併設も有用。
- **入力前**に最近の検索を表示し、**入力中**に予測候補を提示する（SwiftUI では `searchSuggestions(_:)`）。
- 作成日・種類等の属性によるスコープ／フィルタリングを提供する。
- Core Spotlight でアプリ内コンテンツをインデックス化し、Spotlight から直接アプリ内の該当コンテンツへ遷移できるようにする（iOS/iPadOS）。

### よくある違反・注意点

- ❌ 検索対象範囲が不明なまま結果を返す（「このタブ内だけ」なのか「全体」なのか分からない）。
- ❌ ビルトインの検索 UI を使わず独自実装し、標準の挙動（キャンセル・スコープバー・候補表示）を失う。
- ❌ 検索履歴を保存するのにクリア手段がない。

公式: https://developer.apple.com/design/human-interface-guidelines/searching

## 設定（Settings）

設定は少ないほど良い。**最多数のユーザーに最適なデフォルト**を提供し、設定に頼らないことが第一。

### 🔴 必須（遵守事項）

- **タスク固有のオプションを設定エリアに置かないこと**: そのタスクの画面内に配置する（コンテキストから切り離すと使いにくくなる）。
- **システム全体の設定を複製しないこと**: アクセシビリティ・スクロール挙動・認証方法などはユーザーが設定アプリで管理すると期待している。アプリ内に重複トグルを作らない。
- 他の手段で取得できる情報をユーザーに設定させないこと（例: ダークモードはシステム設定から自動検出、ゲームのグラフィック品質はデバイス性能から自動決定）。

### 推奨: 設定の階層

| 階層 | 置くべきもの |
|------|-------------|
| デフォルト（設定なし） | 大多数のユーザーに最適な挙動。まずこれを目指す |
| タスク画面内のオプション | その場のタスクに限定される選択（並び順・表示切替等） |
| アプリ内のカスタム設定エリア | 頻繁には変えない一般設定（アカウント関連・保存挙動・通知の粒度等） |
| 設定アプリ（Settings bundle / `UIApplication.openSettingsURLString`） | システム権限（通知・位置情報等）の変更導線。アプリ内設定から設定アプリへのリンクボタンを提供する |

- 設定項目数を最小化する。多すぎる設定は目的の項目を見つけにくくし、アプリを使いづらくする。
- iPadOS でハードウェアキーボード接続時は Command-Comma（⌘,）でアプリ内設定が開くことが期待される。

### よくある違反・注意点

- ❌ 初回起動時に大量の設定を要求する（合理的なデフォルトの欠如）。
- ❌ アプリ内に独自の「ダークモード」トグルを置き、システム設定と食い違う状態を作る（追従オプションなしの独自トグルは避ける）。
- ❌ 通知の ON/OFF をアプリ内トグルだけで完結しているように見せる（実際にはシステム権限が支配する。設定アプリへの導線を示す）。

公式: https://developer.apple.com/design/human-interface-guidelines/settings

## ヘルプ提供（Offering help）

ヘルプは「必要な人に・必要な場面で・短く」。TipKit を使ったコンテキストヘルプが iOS/iPadOS の主要メカニズム。

### 🔴 必須（遵守事項）

- プラットフォームに合った用語を使うこと: iPhone/iPad で「click」と書いてはならない（「tap」を使う）。
- 現在のタスクに直接関連するヘルプのみ表示すること（無関係な機能の宣伝をヘルプに混ぜない）。
- ヘルプコンテンツはインクルーシブに書くこと（HIG の Inclusion ガイドラインに準拠）。

### 推奨: Tips（TipKit）の設計

- **Tip は 1〜2 文・最大 3 ステップで完結する機能に限る**: それを超える説明が必要な機能はチュートリアル等の別形式にする。
- Tip タイプの使い分け: コンテンツの流れを保つなら **popover tip**、特定 UI 要素を指すなら **annotation（インライン）tip**、特定 UI に紐づかないなら **hint スタイル**。
- 動詞で始まる行動指向の文にする（例:「Restore default settings」）。プロモーション的内容を含めない。
- **表示対象をセグメント化する**: パラメータ／イベントベースの適格性ルールを定義し、既にその機能を使ったユーザーには表示しない。表示頻度も制御する（例: 24 時間に 1 回）。
- シンボルを添える場合は filled variant を優先し、Tip が指している UI 要素と同じ画像を Tip 内で繰り返さない。
- 標準コンポーネントの使い方は説明しない。非標準的な使い方をしている場合は長文説明よりアニメーション／グラフィックで示す。
- シンプルなタスクはインラインビュー、複雑・多段階のタスクはチュートリアルと、タスク種別に応じて形式を選ぶ。

### よくある違反・注意点

- ❌ 初回起動時に全機能の Tip を一斉表示する（コンテキスト外のヘルプは読まれない）。
- ❌ 同じ Tip を既習ユーザーに繰り返し表示する。
- ❌ Tip をお知らせ・キャンペーン告知に流用する。

公式: https://developer.apple.com/design/human-interface-guidelines/offering-help

## アカウント管理（Managing accounts）

アカウント作成の強制を避け、削除手段を必ず提供する。**アカウント削除機能は App Store 上の要件**でもあるため実装必須。

### 🔴 必須（遵守事項）

- **アプリ内でアカウントを削除できる機能を提供すること**（単なる非アクティブ化ではなく削除）。アプリ内で完結できない場合は、削除を実行できる Web ページへの**直接リンク**を提供すること。このリンクをプライバシーポリシーや利用規約の奥に埋没させてはならない。
- **Sign in with Apple 使用時**: アカウント削除時に関連トークンを失効させること（REST API の Revoke tokens を実装）。
- コア機能に必須でない限り、アカウント作成を強制しないこと。アカウントなしで使える範囲を作る。
- 忘れられる権利など地域の法的要件に準拠すること。法的に保持が必要な情報がある場合はその旨を明確に説明する。
- 自動更新サブスクリプションがある場合、**アカウントを削除しても Apple への課金は自動では止まらない**ことを説明し、サブスクリプションの解約手順・返金申請の情報を提供すること。

### 推奨

- **サインインを遅延させる**: 価値を体験できるまでサインインを要求しない（例: ショッピングアプリは購入直前まで不要にする）。
- サインイン画面でアカウントが必要な理由と利点を簡潔に説明する。
- Sign in with Apple を使わない場合は**パスキー（passkeys）を優先**する。パスワードを使う場合は 2FA を実装する。
- 認証方法は具体的に表示する（「Sign In with Face ID」等）。デバイスが対応していない認証方法（Face ID 非搭載機での Face ID 表記等）を表示しない（`LABiometryType` で確認）。
- アプリ内に生体認証の個別有効化設定を作らない（システムレベルで管理済み。冗長で混乱を招く）。
- 「Passcode」という語をアカウント認証に使わない（デバイスのパスコードと混同される）。
- 削除の完了時期を通知し、進捗を伝える。将来日付でのスケジュール削除と即時削除の両方の提供を検討する。Web とアプリ内で削除フローの長さ・複雑さに差をつけない。

### よくある違反・注意点

- ❌ 閲覧だけの利用にもアカウント登録を強制する。
- ❌ 削除導線を「お問い合わせ」経由のみにする（アプリ内削除または直接リンクが必要）。
- ❌ 削除フローだけ Web で極端に複雑にする（ダークパターン）。
- ❌ アカウント削除後もサブスク課金が続くことを説明しない。

公式: https://developer.apple.com/design/human-interface-guidelines/managing-accounts

## 取り消しとやり直し（Undo and redo）

ユーザーは「直近の操作は取り消せる」と期待している。結果の**予測可能性**と**視認性**が本質。

### 🔴 必須（遵守事項）

- iOS/iPadOS の標準ジェスチャを再定義してはならない:
  - **3 本指スワイプ**（左=undo / 右=redo）
  - **デバイスシェイク**で undo
- undo/redo の結果を予測できるようにすること: シェイク時のアラートには自動付与される「Undo 」「Redo 」プレフィックス（末尾スペース含む）に続けて 1〜2 語で操作内容を示す（例:「Undo Name」「Redo Address Change」）。
- 取り消し回数を不必要に制限しないこと。ドキュメントを開いてから保存するまでの全操作を遡れることが期待水準。

### 推奨

- **結果を見せる**: 画面外の操作を取り消した場合は復元箇所までスクロールし、何が起きたかを確認できるようにする。
- **バッチ undo を検討する**: 単一プロパティの連続微調整のような関連操作は 1 回の undo でまとめて戻せるようにする。
- 専用の undo/redo ボタンはシステム標準の手段で不足する場合のみ追加し、標準のシステムシンボルを使ってツールバーに配置する（iPadOS のハードウェアキーボードでは ⌘Z / ⇧⌘Z も機能させる）。
- 実装は `UndoManager` を基盤にし、SwiftUI/UIKit の標準機構と統合する。

### よくある違反・注意点

- ❌ 3 本指ジェスチャやシェイクを独自機能に割り当てる。
- ❌ undo 結果が画面外で起き、何が戻ったのか分からない。
- ❌ 直近 1 操作しか取り消せない。
- ❌ 破壊的操作の防御を確認アラートだけに頼り undo を提供しない（undo があれば確認アラート自体を減らせる）。

公式: https://developer.apple.com/design/human-interface-guidelines/undo-and-redo
