# システム連携パターン（通知・共有・ファイル・メディア・評価）

> 出典: Apple Human Interface Guidelines / Apple Developer Documentation（最終確認 2026-07-13）。仕様は更新されるため、実装時は必ず併記の公式 URL で最新版を確認すること。日本語版がある場合は公式の言語切替または本文の日本語 URL を使い、英語版との差分がある場合は最新側を確認する。
> 判定区分: 本文の「必須」「違反」は作業用の強度ラベルであり、HIG の推奨を一律に App Review 必須へ格上げしない。審査・API 上の必須判定は現行の Apple 公式一次資料で再確認する。

iOS/iPadOS アプリがシステム機能（通知・ドラッグ&ドロップ・ファイル・共有・メディア再生・ハプティクス・印刷・評価依頼・チャート）と連携する際の設計規範。いずれも「システム標準の挙動・UI を尊重し、独自実装で上書きしない」が共通原則。

## 通知の設計（Managing Notifications）

### 概要

通知はユーザーの注意を奪う操作であり、送信には必ず許可が必要。iOS は 4 段階の interruption level（割り込みレベル）で通知の緊急度を制御し、Focus（集中モード）や予定配信サマリーとの関係もレベルで決まる。

### 🔴 必須（遵守事項）

| 事項 | 内容 |
|------|------|
| 事前許可 | 通知を送信する前に必ずユーザーの許可を取得すること。許可なしの送信は禁止 |
| 緊急度の正確な申告 | 通知内容の実際の緊急度に一致する interruption level を設定すること。低優先度情報に高レベルを使ってはならない |
| マーケティング通知の明示的同意 | プロモーション通知は事前にアラート等で説明し、明示的な opt-in を得ること。**Time Sensitive の使用は絶対禁止**（Active / Passive のみ） |
| Critical のエンタイトルメント | Critical 通知（健康・安全に直結）は Apple のエンタイトルメント取得が必須。Ring/Silent スイッチを唯一無視できるレベル |
| Communication 通知の実装 | 人からの直接的なメッセージ・通話は `INSendMessageIntent`（SiriKit intents）＋ `UNNotificationContentProviding` で Communication 通知として実装すること |
| アプリ内通知設定 | アプリの Settings 画面で通知種別（特にマーケティング）の on/off を管理できるようにすること |

### interruption level の選択基準

| レベル | 用途 | 予定配信を無視 | Focus 貫通 | サイレント無視 |
|--------|------|:---:|:---:|:---:|
| Passive | 自分のペースで確認すればよい情報（おすすめ等） | ✗ | ✗ | ✗ |
| Active（既定） | 受信時に知る価値のある情報（スコア速報等） | ✗ | ✗ | ✗ |
| Time Sensitive | 「今〜1 時間以内」に関わる情報（配達到着・セキュリティ警告） | ✓ | ✓ | ✗ |
| Critical | 健康・安全上の緊急事態（要エンタイトルメント） | ✓ | ✓ | ✓ |

### 推奨

- Time Sensitive は「今起きている、または 1 時間以内に起きる」情報にのみ使う。初回使用時にシステムがユーザーへ説明し、ユーザーはいつでも無効化できる — 乱用すると即座に切られる。
- 通知はすべてユーザーがサイレント化できる（政府警報を除く）前提で設計し、通知が届かなくてもアプリ内で同じ情報へ到達できる導線を用意する。
- API: `UNNotificationInterruptionLevel` で指定。参考: WWDC21「Send communication and Time Sensitive notifications」。

### よくある違反・注意点

- ❌ マーケティング通知を Time Sensitive で送る（Focus・サマリーの回避 = 重大違反）
- ❌ 「新機能が使えます」程度の情報に Time Sensitive を割り当てる
- ❌ 通知管理画面をアプリ内に用意せず、OS 設定に丸投げする
- ❌ 起動直後にいきなり許可ダイアログを出す（価値を説明する前の許可要求は拒否率を上げる）

公式: https://developer.apple.com/design/human-interface-guidelines/managing-notifications

## ドラッグ＆ドロップ（Drag and Drop）

### 概要

選択したコンテンツをドラッグ元（source）からドロップ先（destination）へ移すシステム標準の転送手段。iPadOS ではアプリ間ドラッグ・複数アイテムのまとめ持ち（flocking）が中核体験。

### 🔴 必須（遵守事項）

- システム既定のセマンティクスに従うこと: **同一コンテナ内 = 移動、別コンテナ = コピー、アプリ間 = 常にコピー**。
- ドラッグ画像はドラッグ開始（約 3pt の移動）で即座に表示し、**半透明**にして下のドロップ先が見えるようにすること。
- ドロップ先は受け入れ可能なときのみ視覚フィードバック（挿入ポイント・ハイライト）を出すこと。受け入れ不可は無反応または `circle.slash` で示す。
- ドラッグ＆ドロップ**専用の機能にしないこと**。同じ操作をメニューコマンド（コピー/ペースト等）でも実行可能にし、アクセシビリティ API（`accessibilityDragSourceDescriptors` / `accessibilityDropPointDescriptors`）を実装する。
- アンドゥを提供すること。アンドゥ不能な操作は実行前に確認を挟む。

### 推奨

- コンテンツは複数フォーマットで提供し、忠実度の高い順（例: PDF → PNG → JPEG）に並べる。受け側は受け入れ可能な最高忠実度を選ぶ。
- 複数アイテムのドラッグをサポートする。iPadOS ではドラッグ中に別の指でアイテムを追加選択できる。
- spring loading（ドラッグ中のホバーでボタン・ナビゲーションが作動）に対応し、ドラッグしたまま画面遷移できるようにする。
- コンテナ端へのドラッグで自動スクロールさせる。
- ドロップ失敗時はアイテムを元の位置へ戻すかフェードアウトさせ、成功時はドロップ先で選択状態にする（アプリ間では元アプリの選択を解除）。

### よくある違反・注意点

- ❌ ドラッグ画像を不透明にしてドロップ位置を隠す
- ❌ 常時ハイライトを出し「どこに落ちるか」を判別不能にする
- ❌ ドラッグでしか実行できない操作を作る（キーボード・VoiceOver ユーザーを排除）
- ❌ アプリ間ドラッグで独自形式のみ提供し、標準 UTType のフォールバックを用意しない
- 注意: tvOS / watchOS はドラッグ＆ドロップ非対応。

公式: https://developer.apple.com/design/human-interface-guidelines/drag-and-drop

## ファイル管理（File Management）

### 概要

ドキュメントベースのアプリにおけるファイルの作成・保存・ブラウズの設計。iOS/iPadOS では Files アプリとの統合（document browser / File Provider）が前提となる。

### 🔴 必須（遵守事項）

- **自動保存を実装すること**。明示的な「保存」操作を要求せず、編集中・ファイルを閉じるとき・別アプリへ切り替えるときに保存する。作業損失はユーザーの責任にしない。
- 新規作成の導線として **Add（+）ボタン**を提供すること（キーボードショートカットの有無に関わらず）。
- ファイル拡張子は既定で非表示とし、表示するかはユーザーの選択に従うこと。
- File Provider 拡張を実装する場合、呼び出しコンテキストに応じたファイルのみ表示すること（例: PDF 編集アプリから呼ばれたら PDF のみ）。モーダルビューに独自ツールバーを重ねないこと。

### 推奨

- 自アプリで開けない形式も Quick Look viewer でプレビューできるようにする。逆に自アプリ形式は Quick Look generator を提供し、Files / Spotlight でプレビュー可能にする。
- **ドキュメントランチャー（iOS 18 / iPadOS 18+、`DocumentGroupLaunchScene`）**を採用する場合:
  - プライマリボタン = 新規ドキュメント作成、セカンダリボタン = テンプレート選択等に割り当てる
  - 背景は単色・グラデーション・パターン程度に留め、複雑な画像は避ける
  - アクセサリ画像はアプリ名と両ボタンの視認性を損なわない範囲で配置する
  - アニメーションは呼吸のような微細なものに限定する
- File Provider ではファイルの更新日時・サイズ・ローカル/リモート状態などの付加情報を表示し、エクスポート/移動時は宛先ディレクトリのナビゲーションを提供する。

### よくある違反・注意点

- ❌ 保存し忘れによるデータ喪失が起こり得る設計（自動保存の欠如）
- ❌ Files アプリと矛盾する独自ファイル階層概念の導入
- ❌ File Provider で全ファイルを無差別に見せる
- 注意: watchOS / tvOS にはドキュメント管理の概念がない。

公式: https://developer.apple.com/design/human-interface-guidelines/file-management

## コラボレーションと共有（Collaboration and Sharing）

### 概要

共有シート（share sheet）を起点に「コピーを送る」か「共同編集に招待する」かを選ばせ、開始後は Collaboration ボタンで参加者や更新を管理するシステム標準フロー（iOS 16 / iPadOS 16 / macOS 13+）。CloudKit・iCloud Drive・カスタムインフラのいずれでも構築できる。

### 🔴 必須（遵守事項)

- 共有の起点は **Share ボタン**（`square.and.arrow.up`）とし、ツールバー等の見つけやすい場所に置くこと。SwiftUI では `ShareLink` を使う。
- カスタムインフラでコラボレーションを実装する場合、**ユニバーサルリンク対応が必須**（Messages の通知からアプリ内の該当ビューへ直接遷移させるため）。
- 共有シートに表示する権限サマリーは 1〜2 行の簡潔なフレーズにすること（例: "Only invited people can edit" / "Everyone can make changes"）。
- コラボレーション開始後は、Share ボタンの隣に **Collaboration ボタン**を直ちに表示すること。

### 推奨

- Collaboration ボタンのポップオーバーは 3 セクション構成に従う: ①参加者リスト＋Messages/FaceTime ボタン ②アプリ固有項目（最小限に） ③管理ボタン（"Manage Shared File"。必要なら "Manage Participants" 等にカスタマイズ可）。
- 権限オプションはシンプルなセットに絞る（招待者のみ編集可 / 全員編集可 / 読み取り専用 / 参加者追加の許可）。細かい設定は管理画面へ逃がす。
- コンテンツ変更・参加者変更・メンションなどのイベントは `SWHighlightEvent`（SharedWithYou）で Messages に通知する。
- "Send Copy"（コピー送信）もサポートする。CloudKit ではファイル＋コラボレーションオブジェクトの複数アイテムを共有シートに渡す。iCloud Drive は既定で対応。

### よくある違反・注意点

- ❌ 共有シートに自アプリが対応しない共有方式の選択肢を残す
- ❌ ポップオーバーへ情報を詰め込みすぎる（Notes の「最近の更新＋詳細ボタン」程度が目安）
- ❌ 共同編集への招待とコピー送信の区別が UI 上つかない
- 注意: tvOS は共有非対応。watchOS は `ShareLink` で共有シートのみ。

公式: https://developer.apple.com/design/human-interface-guidelines/collaboration-and-sharing

## フルスクリーン（Going Full Screen）

### 概要

ゲーム・メディア視聴・集中作業のために画面全体をコンテンツへ明け渡すモード。iOS/iPadOS ではホームインジケータとシステムジェスチャの扱いが設計の中心になる。

### 🔴 必須（遵守事項）

- フルスクリーンの**終了はユーザーの明示的操作**によること。アプリ切り替えやタスク完了で勝手に終了してはならない。
- メディア再生コントロール等の必須機能へは、常時表示または単純なジェスチャ（タップ・スワイプ）で必ず戻れるようにすること。
- アプリ切り替えから復帰したとき、直前の位置・状態から再開すること。ゲームやスライドショーは復帰時に一時停止状態で戻す。

### 推奨

- ツールバー/ナビゲーションの自動非表示はコンテンツ優先の場面に限り、タップやスワイプで即復帰できるようにする。
- ホームインジケータの既定動作（操作が止まると自動で消え、下端スワイプ 1 回で復帰→もう 1 回で退出）を尊重する。誤操作が頻発するゲーム等に限り `preferredScreenEdgesDeferringSystemGestures` で「2 回スワイプで退出」に変更してよいが、明確な理由が必要。
- ゲーム以外では Dock へのアクセス（上スワイプ）を妨げない。
- レイアウトは追加領域を有効活用しつつ比率変更は微細に留める。プログラムによるウィンドウの自動リサイズは行わない。

### よくある違反・注意点

- ❌ フルスクリーン中に退出手段が見つからない（"trapped" 状態）
- ❌ 動画終了時に勝手にフルスクリーンを解除してユーザーの文脈を破壊する
- ❌ 理由なくシステムジェスチャを 2 段階化してホームへ戻りにくくする
- 注意: tvOS / watchOS は常時フルスクリーンでありこのパターン自体が不要。

公式: https://developer.apple.com/design/human-interface-guidelines/going-full-screen

## オーディオ再生（Playing Audio）

### 概要

音の再生はデバイス全体で共有される資源。`AVAudioSession.Category` の選択が「サイレントスイッチへの応答・他アプリ音とのミックス・バックグラウンド再生可否」を決める。

### 🔴 必須（遵守事項）

| 事項 | 内容 |
|------|------|
| サイレントスイッチの尊重 | サイレント時に鳴らしてよいのは**ユーザーが明示的に開始した音**（メディア再生・アラーム・音声メッセージ）のみ。効果音・ゲーム BGM・キーボードクリック等は無音化すること |
| システム音量への従属 | 音量はシステム音量に支配される。アプリが全体音量を勝手に変更してはならない（アプリ内の相対バランス調整のみ可） |
| ヘッドフォン切断時の一時停止 | オーディオルートが失われたら（イヤフォンを抜いた等）**即座に再生を一時停止**すること。接続時は中断なく自動ルーティング |
| カテゴリの正直な選択 | 実態に合う `AVAudioSession.Category` を選ぶこと。不必要に他アプリの音楽を止めるカテゴリを選ばない |

### カテゴリ選択マトリクス

| カテゴリ | 用途 | サイレントで無音 | 他音とミックス | BG 再生 |
|----------|------|:---:|:---:|:---:|
| Solo ambient | 無くても成立するゲーム BGM 等 | ✓ | ✗ | ✗ |
| Ambient | 他アプリの音楽と共存させたい効果音 | ✓ | ✓ | ✗ |
| Playback | 本質的な音（音楽・オーディオブック） | ✗ | 可変 | ✓ |
| Record | 録音 | ✗ | ✗ | ✓ |
| Play and record | VoIP・ビデオ通話 | ✗ | 可変 | ✓ |

### 推奨

- 音量 UI はシステム提供の `MPVolumeView`（スライダー＋ルート切替）を使う。
- リモートコントロールイベント（再生/一時停止等）には、実際に再生中か明確なコンテキストがあるときのみ応答する。コントロールの意味を再定義しない。
- 一時的に他アプリの音を止めた場合は `notifyOthersOnDeactivation` で再開を通知する。
- 割り込み終了時は `InterruptionOptions.shouldResume` を確認して再開判断する（電話着信のような再開可能割り込みと、他アプリの再生開始のような非再開割り込みを区別）。ゲームはユーザー期待に従い自動再開してよい。
- 短い警告音・バイブは AudioToolbox の System Sound Services を使う。
- Now Playing（コントロールセンター/ロック画面）へメタデータを供給し、バックグラウンド再生を制御可能にする。

### よくある違反・注意点

- ❌ サイレントモード中にゲーム効果音を鳴らす
- ❌ Playback カテゴリを効果音アプリに使い、ユーザーの音楽を止める
- ❌ イヤフォンが抜けてもスピーカーで音を垂れ流す
- ❌ 再生していないのにリモートコントロールイベントを横取りする

公式: https://developer.apple.com/design/human-interface-guidelines/playing-audio

## ビデオ再生（Playing Video）

### 概要

システムのビデオプレーヤー（`AVPlayerViewController`）を使えば、コントロール・PiP・アスペクト比処理・入力対応が標準で揃う。カスタムプレーヤーは真に必要な場合のみ。

### 🔴 必須（遵守事項）

- 特段の理由がない限り**システムプレーヤーを使用**すること。挙動が微妙に違うカスタムプレーヤーはユーザーを混乱させる。
- 動画は**元のアスペクト比のまま**提供すること。レターボックス/ピラーボックスを動画データ自体に焼き込んではならない（フルスクリーンで映像が縮小し、PiP やエッジツーエッジ表示が壊れる）。
- あらゆる入力手段（タッチ・キーボードの Space での play/pause・外部リモート）に応答すること。

### 表示モードの既定

| モード | 対象アスペクト比 | 挙動 | API |
|--------|-----------------|------|-----|
| Full-screen（aspect-fill） | ワイド 2:1〜2.40:1 | 画面を埋める。端はクロップされ得る | `AVLayerVideoGravity.resizeAspectFill` |
| Fit-to-screen（aspect） | 標準 4:3・16:9〜2:1、2.40:1 超 | 全映像表示。黒帯が出る | `AVLayerVideoGravity.resizeAspect` |

### 推奨

- タイトル・説明・アートワーク等のメタデータは `AVPlayerItem.externalMetadata` で供給する（プレーヤー UI とシステム連携に反映される）。
- PiP（Picture in Picture）を有効にする。PiP 移行時のオーディオは自動処理に任せ、復帰時に二重再生しないようにする。
- 二次オーディオの黙音要求（`AVAudioSessionSilenceSecondaryAudioHint` 通知）に応じて、自アプリの補助音声を制御する。
- 前回視聴位置からの再開を、確認プロンプトなしで行う（長尺コンテンツ）。
- 読み込みに 2 秒以上かかる場合のみ黒背景＋スピナーを表示する。

### よくある違反・注意点

- ❌ 4:3 動画の上下に黒帯を焼き込んだ 16:9 ファイルを配信する
- ❌ カスタムプレーヤーでシステムと異なるジェスチャ・コントロール配置を発明する
- ❌ PiP を理由なく無効化する
- ❌ 起動画面やスプラッシュを再生遷移に挟む（黒画面で遷移するのが標準）

公式: https://developer.apple.com/design/human-interface-guidelines/playing-video

## ハプティクス（Playing Haptics）

### 概要

触覚フィードバックは視覚・聴覚を補完する第三のチャネル。システム定義のパターンには学習済みの「意味」があり、意味に沿って使うことが第一原則。iPhone が主対象（iPad は Apple Pencil Pro・外部デバイス経由）。

### 🔴 必須（遵守事項）

- システム提供パターンは**文書化された意味どおり**に使うこと。意味の流用・再定義は禁止。
- 各ハプティクスと引き金となるユーザー操作・イベントの**因果関係が明確**であること。脈絡のない振動はノイズ。
- **ユーザーがオフにできる**こと。ハプティクスなしでもアプリが成立する設計にする。
- 視覚・聴覚フィードバックと調和させること（画面上の衝突と振動のタイミングがずれない等）。

### システムハプティクスの種類（`UIFeedbackGenerator`）

| 種類 | パターン | 用途 |
|------|---------|------|
| Notification | Success / Warning / Error | タスクやアクションの結果通知 |
| Impact | Light / Medium / Heavy / Rigid / Soft | UI オブジェクトの衝突・接触の物理メタファー（重さ・硬さで使い分け） |
| Selection | （単一） | ピッカー・スライダー等の値変更 |

- トグル・スライダー・ピッカーなどの標準コンポーネントは自動でハプティクスを再生する — 重ねて鳴らさない。

### 推奨

- 離散的イベントには短い（transient）ハプティクスを使う。長時間の連続振動はゲーム等の限られた文脈以外では意味を薄める。
- カスタムハプティクス（Core Haptics）は transient / continuous イベントを sharpness（有機的↔機械的）と intensity で設計し、状況に応じて動的に変える（高所からの着地は強く、足音は微かに等）。
- 音との同期は Core Haptics のオーディオ同期機能で行う。
- アプリ全体で同じ意味には同じパターンを使い続ける（一貫性）。
- 使用量はユーザーテストで調整する。「意識されないが体験が豊かになる」量が最適。

### よくある違反・注意点

- ❌ あらゆるタップに Impact を鳴らす（過剰使用で無意味化）
- ❌ Error パターンを装飾目的で使う
- ❌ カメラ・ジャイロ・マイクの使用中に振動させて計測/収録を汚す
- ❌ Apple Pencil Pro で長い連続ハプティクスを流し、描画の手元を不安定にする

公式: https://developer.apple.com/design/human-interface-guidelines/playing-haptics

## 印刷とスキャン（Printing / Scanning）

### 概要

印刷はシステム提供の印刷 UI（iOS/iPadOS: `UIPrintInteractionController`）に統合する。HIG の該当ページは印刷のみを扱う（スキャンは HIG に独立ページがなく、下記の補足はフレームワーク知識に基づく）。

### 🔴 必須（遵守事項）

- 印刷 UI は**システム提供のビュー**を使うこと。ページ範囲・部数・両面印刷（対応プリンター時）などのオプションはシステム UI に任せる。
- 印刷機能は発見しやすい場所に置くこと。iOS/iPadOS では共有シート/アクションシートを開くツールバーボタンが標準。
- **印刷できるものがあるときのみ**印刷項目を有効にすること。印刷対象がなければグレーアウト、プリンター利用不可ならアクションシートから項目を外す。

### 推奨

- 印刷対象のレンダリングには適切な `UIPrintFormatter` / `UIPrintPageRenderer` を使い、画面表示ではなく紙面に最適化した組版を渡す。
- AirPrint 対応を前提とし、独自のプリンタードライバー的 UI を作らない。

### スキャン（補足・フレームワーク知識）

- 書類スキャンは VisionKit の `VNDocumentCameraViewController`（システム標準の書類カメラ UI）を使う。独自のエッジ検出 UI を再発明しない。
- スキャン結果からのテキスト抽出は Vision / VisionKit（Live Text の `ImageAnalyzer` 等）を使い、ユーザーがテキストを選択・コピーできるようにする。
- カメラ許可の理由文（`NSCameraUsageDescription`）にスキャン用途を明記する。

### よくある違反・注意点

- ❌ カスタム印刷ダイアログでシステムのページ設定と重複・矛盾する UI を作る
- ❌ 印刷不能な状態でボタンを活性のまま残し、タップ後にエラーを出す
- 注意: tvOS / watchOS は印刷非対応。

公式: https://developer.apple.com/design/human-interface-guidelines/printing

## 評価とレビュー（Ratings and Reviews）— 評価依頼の節度

### 概要

App Store 評価の依頼はシステム標準プロンプト（StoreKit の `RequestReviewAction` / SKStoreReviewController 系 API）で行う。システムが頻度を強制し、ユーザーは全アプリ一括でオプトアウトできる。**依頼は「お願いできる」のであって「表示を保証される」のではない**。

### 🔴 必須（遵守事項）

| 事項 | 内容 |
|------|------|
| システムプロンプトの使用 | 独自の評価依頼 UI や「星を付けて」誘導画面を作らず、StoreKit の標準依頼 API を使うこと |
| 頻度上限 | システムが **365 日間に最大 3 回**へ自動制限する。この制限を回避する仕組みを作らない |
| タスク中の割り込み禁止 | 操作の途中・ゲームプレイ中に依頼しないこと。自然な区切り（レベルクリア・タスク完了直後）を選ぶ |
| 初回体験での依頼禁止 | 初回起動時・オンボーディング中に依頼しないこと。十分なエンゲージメント実績の後にのみ依頼する |

### 推奨

- 依頼間隔は最低 1〜2 週間空け、その間に追加のエンゲージメント（起動回数・機能の探索・タスク完了数など）があった場合にのみ再依頼する。
- 依頼条件はアプリ側で計測して判断する（API 呼び出し時にシステム側でも表示可否を判定し、条件を満たさなければ黙ってスキップされる — 呼び出し = 表示ではない前提で設計する）。
- 大型アップデートでの評価リセットは慎重に: 新バージョンの評価を反映できる一方、総評価数が減りダウンロード意欲を下げ得る。

### よくある違反・注意点

- ❌ 「高評価で機能解放」「星 5 なら続行」等のインセンティブ・ゲーティング（App Review ガイドライン違反）
- ❌ ネガティブなユーザーだけ問い合わせフォームへ誘導し、ポジティブなユーザーだけ評価へ流す選別ダイアログ
- ❌ 起動のたびに依頼 API を呼ぶ（システム制限に吸収されるとしても、設計として節度を欠く）
- ❌ 拒否したユーザーへ別 UI で再度促す

公式: https://developer.apple.com/design/human-interface-guidelines/ratings-and-reviews

## チャート設計（Charting Data）

### 概要

チャートはトレンド提示・状態変化の可視化・カテゴリ間比較のための手段。**データを見せること自体が目的ならリスト/テーブルで足りる** — チャートを使う積極的理由があるときだけ使う。実装は Swift Charts が標準。

### 🔴 必須（遵守事項）

- **アクセシビリティ対応は必須**。視覚表現だけで完結させず、①チャートの値・構成要素を説明するアクセシビリティラベル、②操作可能なアクセシビリティエレメント（Audio Graphs 対応を含めて検討）を提供すること。
- ユーザーが既に知っている一般的なチャートタイプ（棒・折れ線）を優先すること。独自形式を導入する場合は学習支援を設計する（例: Activity Rings は各リングを個別アニメーションで導入）。
- 同じ目的のチャートには同じタイプ・スタイルを使い、同一データセットを複数ビューで見せるときはタイプ・色・アノテーション・レイアウトを統一すること（小型チャート→拡大ビューでスタイル維持）。

### 推奨

- シンプルに始めて段階的に詳細化する。1 枚に詰め込まず、詳細レベルやデータサブセットの異なる複数チャートに分ける。
- 3 つの粒度でデータを読めるようにする: マクロ（合計・平均のサマリー）/ 中間（意味のあるサブセット）/ 個別データポイント（特定値の強調）。
- タイトル・サブタイトル・アノテーションで「何を読み取るべきか」を言語化する。一目で結論が分かるヘッドラインサマリーを添える（例: 天気アプリの「今後 1 時間は小雨の可能性」）。
- サイズは役割で決める: 大 = 詳細表示・インタラクション用、小 = グランス/プレビュー用。
- 実装は Swift Charts（全 Apple プラットフォーム対応）。設計の詳細は WWDC22「Design an effective chart」「Design app experiences with charts」が公式の一次解説。

### よくある違反・注意点

- ❌ 表で十分な数値の羅列を装飾目的でチャート化する
- ❌ VoiceOver で「画像」としか読めないチャート
- ❌ 同じ指標を画面ごとに違う色・違うチャートタイプで表示する
- ❌ 軸・単位・期間のラベルを省略し、値の解釈をユーザーの推測に委ねる

公式: https://developer.apple.com/design/human-interface-guidelines/charting-data
