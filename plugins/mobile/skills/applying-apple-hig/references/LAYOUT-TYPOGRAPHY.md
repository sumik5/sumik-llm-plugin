# レイアウトとタイポグラフィ（Layout / Typography / Right to Left）

> 出典: Apple Human Interface Guidelines / Apple Developer Documentation（最終確認 2026-07-13）。仕様は更新されるため、実装時は必ず併記の公式 URL で最新版を確認すること。日本語版がある場合は公式の言語切替または本文の日本語 URL を使い、英語版との差分がある場合は最新側を確認する。
> 判定区分: 本文の「必須」「違反」は作業用の強度ラベルであり、HIG の推奨を一律に App Review 必須へ格上げしない。審査・API 上の必須判定は現行の Apple 公式一次資料で再確認する。

## レイアウト（Layout）

### 概要

レイアウトは「セーフエリアとレイアウトガイドを尊重した適応レイアウト」を基本とする。固定座標・固定サイズで組んではならず、デバイス・向き・サイズクラス・Dynamic Type・ロケール（RTL 含む）の変化に追従するレイアウトを設計する。

### 🔴 必須（遵守事項）

| 項目 | 要求 |
|------|------|
| セーフエリア | コンテンツとコントロールは safe area 内に配置すること。Dynamic Island・カメラ領域・ホームインジケータ・タブバー/ツールバーに覆われる位置へインタラクティブ要素を置いてはならない。SwiftUI は `SafeAreaRegions`、UIKit は safe area layout guide を使う |
| レイアウトガイド | システム定義のガイド（標準マージン・readable content 幅）に従うこと。UIKit `UILayoutGuide` / SwiftUI のレイアウト修飾子で位置合わせする |
| タップ領域 | 44x44pt を快適な操作の推奨デフォルトとし、ボタンのヒット領域も一般則として 44x44pt 以上を目指す。Apple の Accessibility 資料が示す最小 28x28pt との区別を保ち、44x44pt を全要素の絶対最小値や一律の審査要件とは扱わない |
| 背景の全面化 | 背景・フルスクリーンのアートワーク・スクロールコンテンツは画面（ウインドウ）のエッジまで広げること。コントロールやナビゲーション（サイドバー・タブバー）はコンテンツの上層に浮かせる |
| サイズクラス対応 | Compact / Regular の両サイズクラスで成立するレイアウトにすること（`UserInterfaceSizeClass` / trait collection で分岐） |
| Dynamic Type 対応 | テキストサイズ変更（アクセシビリティサイズ含む）でレイアウトが破綻しないこと（詳細は Typography 節） |
| RTL 対応 | leading / trailing でレイアウトを定義し、left / right の決め打ちをしないこと（詳細は Right to Left 節） |
| iPhone の全幅ボタン禁止 | 画面端から端まで達する全幅ボタンを避け、システム定義マージンでインセットすること。全幅にする場合はハードウェアの角丸と調和させ、下部セーフエリアと整列させる |
| iPadOS ウインドウリサイズ | 現行 iPadOS はフルスクリーン表示と、自由にリサイズ・移動できるウインドウ表示を提供する。特定の分割比を前提にせず、サポートする最小幅から最大幅まで連続的に成立させ、ウインドウコントロールとツールバー項目の重なりも確認する |

### レイアウト原則（推奨）

- **グループ化**: 関連要素はネガティブスペース・背景シェイプ・色・マテリアル・セパレータで視覚的にまとめ、コンテンツとコントロールを明確に区別する。
- **優先順位**: 本質的な情報に十分なスペースを割き、読み順（上→下、leading→trailing）で重要なものを上部・leading 側に置く。二次情報は別ビューへ。
- **整列**: コンポーネント同士を整列させ、インデントで情報階層を表現する。
- **Progressive Disclosure**: 隠れたコンテンツはディスクロージャーコントロールか部分表示（ちら見せ）で存在を示す。
- **コントロールの余白**: コントロール周囲に十分なスペースを取り、論理的なセクションでグループ化する。密集させない。
- **Liquid Glass / スクロールエッジエフェクト**: コントロール層とコンテンツの区別にはシステムマテリアルとスクロールエッジエフェクトを使い、独自の不透明背景で塗り潰さない。コンテンツがウインドウ全面に届かない場合は背景拡張ビュー（SwiftUI `backgroundExtensionEffect()` / UIKit `UIBackgroundExtensionView`）でコントロール層の背後までコンテンツを見せる。
- **アートワーク**: スケール時にアスペクト比を変えない。アスペクト比が異なる画面でも重要なビジュアルが見え続けるようにする。

### 適応レイアウト（Adaptability）

対応すべき変化（すべて動的に発生しうる）:

1. デバイスごとの画面サイズ・解像度・カラースペース
2. 縦向き / 横向きの回転
3. Dynamic Island・カメラコントロール等のシステム機能
4. 外部ディスプレイ・Display Zoom・iPad のリサイズ可能ウインドウ
5. Dynamic Type のテキストサイズ変更
6. ロケール（LTR/RTL・日時/数値フォーマット・翻訳によるテキスト長変化）

実装ルール:

- コンテキスト変化（回転・任意幅へのリサイズ・フルスクリーンとウインドウ表示の切り替え）でグレースフルに適応すること。状態やスクロール位置を失わない。
- 最大レイアウトと最小レイアウトの両極から設計・テストを開始する。
- 複数デバイス・両向き・複数ロケール・複数テキストサイズでプレビューし、クリッピングを検証する。

#### iOS（iPhone）固有

- 縦・横両向きのサポートを目指す。片方向のみの場合、ユーザーは自然に両向きを試すので回転を指示するテキストは不要。横向き専用なら左右どちらの回転でも同等に動作させる。
- ステータスバーは基本的に表示を維持する。非表示にしてよいのはゲームやメディア視聴などの没入型体験のみ。
- ゲームはフルブリード（全画面）で角丸・センサーハウジング・Dynamic Island を考慮した UI にし、必要ならレターボックス表示をオプション提供する。

#### iPadOS 固有

- **コンパクトビューへの遷移は遅延させる**: フルスクリーン相当のレイアウトを可能な限り保ち、収まらなくなった時点で初めてコンパクト表示に切り替える。3 カラム構成ではまずインスペクター等の第 3 カラムを隠す。
- **コンバーティブルタブバー**: サイドバーとタブバーの両プレゼンテーションをサポートし（SwiftUI `.sidebarAdaptable`）、起動時はユーザー選択に従い、リサイズ時はビュー幅に応じて切り替える。
- 狭い・中間・広いウインドウとフルスクリーンでテストし、最小〜最大ウインドウの連続的な遷移がスムーズであることを確認する。旧 OS を対象にする場合だけ、Split View・Slide Over・Stage Manager 固有の構成を条件付きで追加する。

### サイズクラス（Size Classes）

- **Regular**: 広い寸法（iPad 全般・大型 iPhone の横向きの幅など）
- **Compact**: 狭い寸法（iPhone 縦向きの幅など）

代表パターン（幅 × 高さ）:

| デバイス | 縦向き | 横向き |
|---------|--------|--------|
| iPad（全モデル・フルスクリーン時） | Regular × Regular | Regular × Regular |
| iPhone Pro Max / Plus / Air 系 | Compact × Regular | Regular × Compact |
| その他の iPhone（標準・mini・SE） | Compact × Regular | Compact × Compact |

実装上の注意:

- 「iPhone = Compact」「iPad = Regular」と決め打ちしない。iPad のリサイズ可能なウインドウも Compact 幅になり得る。**デバイス判定ではなく利用可能領域とサイズクラスで分岐すること**。
- 大型 iPhone は横向きで Regular 幅になるため、iPad 向けの 2 カラム UI が現れうる。

### よくある違反・注意点

- ❌ セーフエリア外（Dynamic Island 裏・ホームインジケータ上）にボタンやテキストを配置する
- ❌ 画面幅 390pt を前提とした固定幅レイアウト（小型 iPhone / iPad の狭いウインドウで破綻）
- ❌ iPad で iPhone レイアウトを単純拡大しただけの UI（Regular 幅では複数カラム・サイドバー等を検討する）
- ❌ スクロールビューの背景がセーフエリア手前で途切れ、下端に不自然な帯が出る
- ❌ 回転・ウインドウリサイズで入力中の状態やスクロール位置が失われる
- ❌ ステータスバーを理由なく非表示にする

## タイポグラフィ（Typography）

### 概要

iOS / iPadOS のテキストは「システムフォント（SF Pro）＋システムテキストスタイル＋Dynamic Type」の三点セットが標準。カスタムフォントを使う場合もこの 3 つと同等の振る舞いを自前で保証する義務がある。

### 🔴 必須（遵守事項）

| 項目 | 要求 |
|------|------|
| 最小フォントサイズ | iOS / iPadOS では **11pt 未満のテキストを使ってはならない**（デフォルト本文は 17pt）。カスタムフォントは同サイズで細く見えがちなので推奨サイズより大きくする |
| Dynamic Type | システムテキストスタイル（`.body` 等）を使い、ユーザーのテキストサイズ設定に自動追従させること。カスタムフォントでも Dynamic Type スケーリングを実装すること（SwiftUI: `Font.custom(_:size:relativeTo:)` 等） |
| アクセシビリティサイズ | AX1〜AX5（Larger Accessibility Text Sizes）でもレイアウトが成立し、テキストが切り詰められないこと。最大サイズ（AX5）でテスト必須 |
| ウェイト | 本文系は Regular / Medium / Semibold / Bold を使用。**Ultralight / Thin / Light は避ける**（特に小サイズで可読性が落ちる） |
| Bold Text 設定 | システムの Bold Text（アクセシビリティ設定）に応答すること。カスタムフォントも対応させる |
| 階層の維持 | フォントサイズが変わっても情報階層（見出し > 本文 > 補足）の相対関係を保つこと |

### システムフォント

| フォント | 用途 |
|---------|------|
| **SF Pro** | iOS / iPadOS の標準サンセリフ。variable font で dynamic optical sizing（サイズに応じた光学調整）を自動適用。Ultralight〜Black のウェイト、Condensed / Expanded 等の幅バリアントあり。SF Symbols とウェイトが揃う |
| **SF Compact** | watchOS 標準のコンパクト版（iPhone/iPad では通常 SF Pro を使う） |
| **New York (NY)** | システム提供のセリフ体。SF と併用可 |
| **SF Mono** | 等幅（コード表示等） |
| 言語派生 | SF Arabic / SF Hebrew 等。RTL 言語はシステムが自動選択する |

- フォントは https://developer.apple.com/fonts/ から入手（デザインカンプ用）。実装では `Font.system` / `UIFont.preferredFont(forTextStyle:)` を使えばダウンロード不要。

### テキストスタイル階層（iOS / iPadOS・Large=デフォルトサイズ）

| スタイル | ウェイト | サイズ | 行送り (Leading) | Emphasized |
|---------|---------|-------|------------------|------------|
| Large Title | Regular | 34pt | 41pt | Bold |
| Title 1 | Regular | 28pt | 34pt | Bold |
| Title 2 | Regular | 22pt | 28pt | Bold |
| Title 3 | Regular | 20pt | 25pt | Semibold |
| Headline | Semibold | 17pt | 22pt | Semibold |
| Body | Regular | 17pt | 22pt | Semibold |
| Callout | Regular | 16pt | 21pt | Semibold |
| Subheadline | Regular | 15pt | 20pt | Semibold |
| Footnote | Regular | 13pt | 18pt | Semibold |
| Caption 1 | Regular | 12pt | 16pt | Semibold |
| Caption 2 | Regular | 11pt | 13pt | Semibold |

- Dynamic Type は標準 7 段階（xSmall〜xxxLarge、デフォルトは Large）＋アクセシビリティ 5 段階（AX1〜AX5）。
- 参考スケール感: Body は Large で 17pt → AX1 で 28pt → AX5 で 53pt。**本文が約 3 倍になっても成立するレイアウト**が要求される。
- Emphasized 列は強調時に使うウェイト（2025-12 に各プラットフォームの仕様へ追加）。強調は Symbolic Traits の Bold trait で付与する。

### 推奨

- **システムテキストスタイルを優先**する。独自のポイントサイズ指定は Dynamic Type・一貫性の両方を壊す。
- **書体は 1〜2 種類に限定**する。複数フォントの混在は可読性とまとまりを損なう。
- 階層は「ウェイト・サイズ・色」の組み合わせで表現し、色だけに依存しない。
- 行間（leading）調整: 複数行の読み物には loose leading、限られた高さには tight leading。ただし **3 行以上のテキストに tight leading を使わない**。
- トラッキングは実際のモックアップ上で確認する（variable font はサイズごとに自動調整される）。
- コントラストが不足するときは、テキストサイズを上げる・背景色を変える・背景シェイプを敷くの順で改善する。
- 意味を持つアイコン（SF Symbols）はテキストと連動してスケールさせる（`imageScale` / テキストスタイル連動）。
- 大きいテキストサイズでは、横並び（インライン）要素を縦積みに切り替え、マルチカラムを減らす。切り詰め（truncation）はスクロール領域以外では最小化する。SwiftUI では `ViewThatFits` や `dynamicTypeSize` の閾値分岐、UIKit では `numberOfLines = 0` 等で対応する。

### よくある違反・注意点

- ❌ `Font.system(size: 14)` のような固定サイズ指定の乱用（Dynamic Type 非追従）
- ❌ 11pt 未満のキャプション・注釈テキスト
- ❌ AX サイズで文字がボタン枠からあふれる / ラベルが「…」で潰れて意味を失う
- ❌ 薄いウェイト（Thin/Light）の小サイズ本文
- ❌ 見出しも本文も同ウェイト・同サイズで階層が判別できない
- ❌ カスタムフォント導入時に Dynamic Type / Bold Text 対応を省略する

公式: https://developer.apple.com/design/human-interface-guidelines/typography

## 右から左へ（Right to Left / RTL 対応）

### 概要

アラビア語・ヘブライ語などの RTL 言語では、読み方向に合わせてインターフェイス全体をミラーリングする。システム標準の UI フレームワーク（SwiftUI / UIKit の標準コンポーネント＋leading/trailing 制約）を使っていれば大部分は自動反転される。手動対応が要るのは「何を反転し、何を反転しないか」の判断が絡む箇所。

### 🔴 必須（遵守事項）

| 項目 | 要求 |
|------|------|
| leading / trailing | レイアウト制約・配置・パディングは left/right ではなく leading/trailing で定義すること（自動ミラーリングの前提） |
| テキスト整列 | 1〜2 行のテキストは現在のコンテキストの読み方向に整列。**3 行以上の段落はコンテキストではなく言語に基づいて整列**（RTL コンテキスト内の英語段落は左整列を維持） |
| リスト整列 | リスト内の全アイテムで整列を統一する |
| 数字の順序 | 電話番号・クレジットカード番号・ID 等、**特定の並びを持つ数字は反転しない**（541 は RTL でも 541） |
| 進捗系の数字 | スライダー・プログレスバー・レーティングの端点の数字は**コントロールの方向に合わせて反転する**（LTR: `1 ★★★★★ 5` → RTL: `5 ★★★★★ 1`） |
| 方向性コントロール | スライダー・プログレスバー・戻る/次へナビゲーション・順序を示すボタン群は反転すること。端点のグリフ・画像の位置も入れ替える |
| 実方向コマンド | 「右へ移動」など物理方向を指すコントロールは反転しない（常に右を指す） |

### 数字の表記

| 言語 | 使用する数字 |
|------|-------------|
| ヘブライ語 | 西アラビア数字（1, 2, 3…） |
| アラビア語 | 西アラビア数字 または 東アラビア数字（١, ٢, ٣…）。ロケール設定に従う |

### 画像・アイコンの反転判断

| 対象 | 反転 | 理由・補足 |
|------|------|-----------|
| 写真・一般アートワーク | ❌ | 意味が変わる／権利上の問題が生じうる |
| 意味のある順序で並ぶ複数画像 | ✅ | 時系列・アルファベット順・お気に入り順などの並び自体を反転 |
| 読み方向を表すアイコン（テキスト行・箇条書きグリフ） | ✅ | 左揃えバー表現は右揃えに |
| 前進/後退・音の放射など方向を示すシンボル | ✅ | スピーカー、ナビゲーション矢印など |
| ロゴ | ❌ | 常に元の形で表示 |
| ユニバーサルシンボル（チェックマーク等） | ❌ | 一貫性を優先 |
| 実世界オブジェクト（時計・鉛筆等） | ❌ | 利き手・物理との対応を保持 |
| 複雑なカスタムアイコン | 個別判断 | 構成要素ごとに方向性と視覚バランスを検討 |

- **SF Symbols を優先する**: RTL 対応済みシンボルとアラビア語/ヘブライ語ローカライズ版（署名・リッチテキスト・I-beam 等）が提供される。カスタムシンボルにも方向性（directional variant）を指定できる。
- **バッジ付きアイコン**: バッジが UI 要素を表すなら反転。装飾的なら視覚バランスで判断（例: カートの「+」バッジは右上→左上へ移動）。
- **ツール系アイコン**: ツール自体の傾き（利き手表現）は保持し、ベース画像側のみ反転（例: 虫眼鏡＋メールでは虫眼鏡の向きを維持）。

### 推奨

- アラビア文字・ヘブライ文字には大文字がなく、ラテン大文字と並ぶと小さく見える。**該当言語ではフォントサイズを約 2pt 増やす**ことを検討する。
- 実装後は必ず RTL 疑似ロケール／アラビア語ロケールで起動して検証する（Xcode の scheme 設定 App Language / 「Right-to-Left Pseudolanguage」）。
- SwiftUI では `environment(\.layoutDirection, .rightToLeft)` をプレビューに与えて開発中に確認する。

### よくある違反・注意点

- ❌ `HStack` 内の並びやアイコン位置を `.leading` ではなく座標・left 固定で組み、RTL で反転しない
- ❌ 電話番号や金額の桁を反転させてしまう（bidi 制御の誤り）
- ❌ 戻るボタン・進捗バーが LTR のままで、操作方向と読み方向が食い違う
- ❌ ロゴやチェックマークまで一括ミラーリングする
