# 入力手段（タッチ・キーボード・Apple Pencil・ポインタ・センサー）

> 出典: Apple Human Interface Guidelines / Apple Developer Documentation（最終確認 2026-07-13）。仕様は更新されるため、実装時は必ず併記の公式 URL で最新版を確認すること。日本語版がある場合は公式の言語切替または本文の日本語 URL を使い、英語版との差分がある場合は最新側を確認する。
> 判定区分: 本文の「必須」「違反」は作業用の強度ラベルであり、HIG の推奨を一律に App Review 必須へ格上げしない。審査・API 上の必須判定は現行の Apple 公式一次資料で再確認する。

本ファイルは iPhone / iPad（iOS / iPadOS）アプリの入力手段設計をカバーする。各トピックは「🔴 必須（遵守事項）」→「推奨」→「よくある違反・注意点」の順で、実装判断に直結する形で記述する。

## ジェスチャ（Gestures）

タッチスクリーン上でユーザーがオブジェクトを直接操作する物理的動作。iOS / iPadOS のインタラクションの根幹であり、標準ジェスチャの意味はシステム全体で一貫している。

### 🔴 必須（遵守事項）

- 標準ジェスチャをユーザー期待どおりの操作に割り当てること。**標準ジェスチャの意味を独自機能へ流用してはならない**（例: スワイプをスクロール以外の破壊的操作に転用しない）。
- 特定のジェスチャだけに依存せず、**代替入力手段（ボタン・メニュー・音声入力・キーボード・Switch Control）を必ず提供する**こと。
- システムが予約するジェスチャ（画面端スワイプによるホーム/アプリ切替、通知センター/コントロールセンター呼び出し等）と競合するジェスチャを定義しないこと。
- ジェスチャ実行中はリアルタイムに視覚フィードバックを返すこと（ドラッグ中の追従、スワイプ中の遷移プレビュー等）。

### 標準ジェスチャ一覧（iOS / iPadOS）

| ジェスチャ | 期待される操作 |
|---|---|
| タップ | コントロールの有効化・アイテム選択 |
| スワイプ | スクロール・アクション表示（リスト行）・ビューを閉じる |
| ドラッグ | UI 要素の移動 |
| タッチ＆ホールド（長押し） | コンテキストメニュー・追加機能の表示 |
| ダブルタップ | ズームイン/アウト・主要操作の実行 |
| ピンチ（ズーム） | コンテンツの拡大縮小 |
| 回転（2本指） | 選択アイテムの回転 |
| 3本指スワイプ | 左＝アンドゥ / 右＝リドゥ |
| 3本指ピンチ | ピンチイン＝コピー / ピンチアウト＝ペースト |
| 4本指スワイプ（iPadOS） | アプリ切り替え（システム予約） |
| シェイク | アンドゥ/リドゥ |

### 推奨

- カスタムジェスチャは「**頻繁に使う専門的タスク**」に限って追加し、①発見可能 ②実行しやすい ③他ジェスチャと区別できる ④代替手段がある、の 4 条件を満たすこと。
- カスタムジェスチャは標準操作の**補完**として追加する（例: Back ボタンを残したうえでエッジスワイプ戻りを併設）。標準経路を削除しない。
- ジェスチャが使えない状態はロック表示等で視覚的に区別し、理由を明示する。
- ゲームでは複数ジェスチャの同時認識（仮想ジョイスティック＋発射ボタン等）を検討する。
- iPadOS ではタッチと Apple Pencil 入力の統合を検討する（後述の Apple Pencil 節参照）。

### よくある違反・注意点

- 独自ジェスチャを唯一の操作経路にする（発見不能・アクセシビリティ違反になる）。
- タッチ＆ホールドに即時性の必要な主要操作を割り当てる（長押しは発見されにくい）。
- シェイクによるアンドゥを無効化したままカスタムのアンドゥ経路も用意しない。

公式: https://developer.apple.com/design/human-interface-guidelines/gestures

## ハードウェアキーボード / ショートカット（Keyboards）

iPad の Magic Keyboard / Smart Keyboard、および iPhone / iPad に接続された Bluetooth キーボードでの操作。iPad アプリではキーボード対応の品質が生産性アプリの評価を左右する。

### 🔴 必須（遵守事項）

- **Full Keyboard Access をサポートする**こと（iOS / iPadOS で利用可能）。設定 > アクセシビリティで有効化してテストする。
- 他アプリでユーザーが慣れている**標準ショートカットの意味を再定義しない**こと（例: Command–C/V/Z/F 等）。当該アクションが自アプリに存在しない場合のみ転用可（例: テキスト編集のないアプリで Command–I を「情報を見る」に使う）。
- 複数修飾キーを使う場合の表記・定義順は **Control → Option → Shift → Command** の順を守ること。

### 推奨

- カスタムショートカットは**頻繁に使われるアプリ固有コマンドのみ**に限定する（多すぎると学習できない）。
- 修飾キーの選択優先度: 第一に **Command**、関連バリエーションに **Shift** を追加、パワーユーザー向けの低頻度コマンドに **Option**。**Control はシステムが多用するため避ける**。
- キー上段の記号を使うショートカットに Shift を重ねない（Command–? であって Command–Shift–/ ではない）。
- 既存ショートカットに修飾キーを足して**無関係な**コマンドを作らない（Shift–Command–Z はリドゥの慣習があるため、Command–Z と無関係な機能に使わない）。
- ショートカットはシステムがキーボード言語に応じて自動ローカライズ・RTL ミラーリングするため、それを妨げない。
- 実装は SwiftUI `keyboardShortcut(_:)` / UIKit `UIKeyCommand`（`discoverabilityTitle` でショートカット一覧＝Command 長押し HUD に説明を出す）。

### iOS / iPadOS 固有の注意

- キーボードナビゲーション（矢印キー移動）の対象は**テキストフィールド・テキストビュー・サイドバー**が基本。ボタン・セグメンテッドコントロール・スイッチへの矢印キーナビゲーションは実装しない（それらは Full Keyboard Access に任せる）。
- フォーカス管理は `UIKeyCommand` と Focus-based Navigation API（後述「フォーカスと選択」）を併用する。

### よくある違反・注意点

- Command 長押しのショートカット HUD に載らない「隠しショートカット」を作る（発見不能）。
- ゲーム以外で Command–Q 等のシステム慣習を上書きする。
- 全コマンドにショートカットを割り当てて衝突・学習不能に陥る。

公式: https://developer.apple.com/design/human-interface-guidelines/keyboards

## 仮想キーボード（Virtual Keyboards）

オンスクリーンキーボード。入力内容に最適化したキーセットを提示でき、iOS / iPadOS ではカスタムキーボード拡張も作成できる。

### 🔴 必須（遵守事項）

- **編集中のコンテンツタイプに合ったキーボードタイプを指定する**こと（メールアドレス欄に Default キーボードを出さない）。SwiftUI `keyboardType(_:)` / `textContentType(_:)`、UIKit `UIKeyboardType` / `UITextContentType`。
- **キーボード表示時に入力中のフィールドや主要操作ボタンが隠れない**よう、`UIKeyboardLayoutGuide`（またはそれに相当する SwiftUI のレイアウト調整）でレイアウトを追従させること。

### キーボードタイプ一覧

| タイプ | 用途 |
|---|---|
| Default | 汎用 |
| ASCII capable | 標準アルファベット |
| Numbers and punctuation | 数字と句読点 |
| Number pad / Decimal pad | 数字のみ / 小数対応 |
| Phone pad / Name phone pad | 電話番号 / 名前＋電話番号 |
| Email address | @ と . を含む |
| URL | / と .com を含む |
| Web search | 検索向け |
| Twitter | @ と # を含む |
| ASCII capable number pad | 英数字パッド |

### 推奨

- Return キーのラベルをコンテキストに合わせる（検索なら「Search」）。SwiftUI `submitLabel(_:)` / UIKit `UIReturnKeyType`。
- カスタム入力ビュー（独自キーパッド）はアプリ文脈で明確な利点がある場合のみ使い、キー押下音（`UIDevice.playInputClick()`）を再生し、システムキーボードへ戻る手段を明示する。
- キーボード上部に機能を足すときは入力アクセサリビュー（`inputAccessoryView` / SwiftUI `ToolbarItemPlacement.keyboard`）を使い、標準ツールバーの外観（Liquid Glass 適用環境ではその質感）と統一する。
- カスタムキーボード拡張（iOS / iPadOS）を作る場合: Globe キー相当の切替手段を直感的に提供し、システム提供の絵文字キー・音声入力キーを重複実装せず、選択・使用・解除方法のチュートリアルを**アプリ内**（キーボード内ではなく）に置く。

### よくある違反・注意点

- パスワード欄や決済入力に不必要なカスタムキーボードを許可する（セキュリティ懸念）。
- キーボード出現でボタンが隠れ、ユーザーがキーボードを閉じないと操作継続できない画面。
- 数値入力欄で Number pad を指定せず入力ミスを誘発する。

公式: https://developer.apple.com/design/human-interface-guidelines/virtual-keyboards

## Apple Pencil と Scribble（Apple Pencil and Scribble）

**iPadOS 専用**。ピクセル精度の描画・マークアップと、テキストフィールドへの手書き入力（Scribble、オンデバイス認識）を提供する。

### 🔴 必須（遵守事項）

- Apple Pencil が画面に触れた**瞬間**にマークが付くこと（知覚可能な遅延を作らない）。
- Apple Pencil とタッチ（指）を**自由に切り替え可能**にすること。Pencil 専用モードで指入力を締め出さない。
- Scribble を機能させるため、テキスト入力には**標準テキストコンポーネント**（text field / text view / search field）を使うこと。カスタム入力欄には `UIScribbleInteraction` / `UIIndirectScribbleInteraction` で対応する。
- ダブルタップ・スクイーズの動作は**システム設定・ユーザー設定を尊重**すること。アプリ独自動作はユーザーが選択できるコントロールを提供し、デフォルトで有効にしない。

### センサー属性と用途

| 属性 | 内容 | 典型用途 |
|---|---|---|
| Altitude（高度/傾き） | ペンの傾き角 | ストローク幅の変化 |
| Pressure（筆圧） | 押下力 | インク濃度・ブラシサイズ（単純で直感的な連続量に使う） |
| Azimuth（方位角） | ペン先の向き | ブラシ方向 |
| Barrel Roll | 軸回転（Apple Pencil Pro のみ） | ハイライト角度の回転などマーク動作の変更のみ |

### ホバー / ダブルタップ / スクイーズの規則

- **ホバー**: 接地したときの結果のプレビュー（ツール寸法・色）に使う。**ホバーでアクションを実行してはならない**。高さに応じてプレビューを頻繁に更新せず、ほぼ静的に表示。中程度の値でプレビューする。ポインタ用プレビューを流用しない。
- **ダブルタップ**（デフォルト: 現在のツール ↔ 消しゴム）: 誤発火を想定し、**取り消し容易な非破壊アクションのみ**に割り当てる。データ喪失の可能性がある操作は禁止。
- **スクイーズ**（Apple Pencil Pro のみ）: 単発・離散のアクション専用。ツールパレットやコンテキストメニューの表示が典型。**破壊的操作・UI ナビゲーション・継続的操作への割り当ては禁止**。フィードバックはペン先近くに表示する。ユーザーが App Shortcut 実行を設定した場合はそちらが優先されることを前提にする。

### Scribble のベストプラクティス

- タップやモード切替なしで、書き始めたら即認識されること。
- キーボードでは自然でない場所への書き込みも許容する（例: リスト末尾の空白に直接書いて項目追加）— `UIIndirectScribbleInteraction`。
- **書き込み中に干渉しない**: 自動補完候補を出さない・プレースホルダーは書き始めで消す・フィールドの移動/リサイズ/自動スクロールをしない（リサイズは書き終わってから）。
- 手書きに十分な幅のフィールドを確保する。パスワードフィールドは Scribble 非対応。

### 描画実装（PencilKit）

- PencilKit で低遅延キャンバス＋ツールピッカーを利用する。ダークモードでの自動色調整は、既存コンテンツへのマークアップ時には無効化してシャープさを維持する。
- コンパクト環境（iPhone サイズ表示等）ではツールピッカーにアンドゥ/リドゥが載らないため、ツールバーにカスタムのアンドゥ/リドゥボタンを置き、3本指タップジェスチャもサポートする。
- 左右どちらの手でも使えるようコントロール配置を調整する。

### よくある違反・注意点

- スクイーズに「消去」「送信」など取り返しのつかない操作を割り当てる。
- Scribble 対応を確認せずカスタム描画ビューをテキスト入力に使い、手書き入力が全く効かない。
- 書き込み中にフィールドがリサイズ/スクロールされ、入力位置を見失わせる。

公式: https://developer.apple.com/design/human-interface-guidelines/apple-pencil-and-scribble

## ポインティングデバイス（Pointing Devices — iPad トラックパッド / ポインタ）

iPadOS はトラックパッド・マウスによるポインタ操作をサポートする（デフォルトは円形ポインタ、適応的に形状・エフェクトが変わる）。

### 🔴 必須（遵守事項）

- マウス/トラックパッドジェスチャへの反応を**システム全体と一貫**させること。
- **システムのトラックパッドジェスチャ（Dock・アプリ切替等）を再定義しない**こと。ゲーム中でも利用可能なままにする。
- 同一機能をタッチ・ポインタ・キーボードのどれでも実行できるようにし、入力方法をシームレスに切り替えられること。

### ヒット領域とポインタエフェクトの数値・使い分け

| 項目 | 値/基準 |
|---|---|
| ベゼル付きボタンのヒット領域パディング | ボタン周囲 **12pt** |
| ベゼルなし要素（シンボル等）のパディング | 表示領域周囲 **24pt** |
| Highlight エフェクト | 透明背景の小要素（バーボタン・タブバー・セグメンテッドコントロール） |
| Lift エフェクト | 不透明背景の小要素（アプリアイコン的な要素） |
| Hover エフェクト | 大型要素・カスタム要素（スケール/色/影を柔軟に適用） |

### 推奨

- 自動最小化するコントロール（ツールバー、全画面動画の再生コントロール等）はポインタ移動で表示/非表示を切り替える。
- 修飾キー＋ドラッグの結果はタッチ操作と一貫させる（例: Option＋ドラッグで複製）。
- ポインタアクセサリ（ポインタ脇の小さな画像）はシンプルに保ち、形状・位置の変化で状態遷移を伝える（例: plus → circle.slash で「不可」を表現）。
- iPadOS 15+ の**ポインタドラッグによる複数選択**は標準コレクションビューで自動対応。カスタムビューは `UIBandSelectionInteraction` で実装する。
- ポインタとタッチの挙動差別化は価値がある場合のみ（例: 動画スクラバーでポインタはクリック位置へ正確シーク、タッチは再生ヘッド操作）。
- ホバーエフェクトの設計: スケール＋影は要素周囲に余白がある場合のみ。テーブル行にスケールは使わない（隣接行と重なる）。影のみ（スケールなし）は浮遊感が出ないため避ける。狭いスペースでは色変化のみで可。

### よくある違反・注意点

- 実用性のない装飾的ポインタ変化（混乱の元）。複雑なポインタ形状。**ポインタ内・脇への指示テキスト表示は禁止**。
- Hover エフェクトにマグネティズム（吸着）を期待する（適用されない。吸着は Lift/Highlight のみ）。
- テキスト入力エリアの I-beam 自動変形を独自ポインタで潰す。

公式: https://developer.apple.com/design/human-interface-guidelines/pointing-devices

## アクションボタン（Action Button）

iPhone（対応機種）の側面物理ボタン。ユーザーが設定した App Shortcut またはシステム機能を実行する。iPad は非対応。

### 🔴 必須（遵守事項）

- アクションボタンに公開する機能は **App Shortcut** として実装すること（ユーザーは設定アプリで割り当てる。アプリ側から割り当てを強制する手段はない）。
- ラベルは**最大 3 語**・動詞開始・現在形・タイトルスタイルの大文字化・冠詞/前置詞なし（例: 「Start Race」○ / 「Start the Race」✗）。

### 推奨

- 「アプリを開く」機能は提供不要（システムが既に提供）。**定期的に使われる重要機能**を割り当て候補にする。
- 設定アプリでの構成フローはシステムに任せ、アプリ内で割り当て手順の重複説明をしない。
- **アプリを開かずに完結**させる: Live Activity やスニペット表示を使い、軽量に結果を返す（例: 「Set Timer」はアプリ起動なしでカウントダウンを Live Activity 表示）。

### よくある違反・注意点

- 長い説明的ラベル（切り詰められる）。
- アクションボタン押下でフルアプリ起動＋複数タップを要求する重いフロー。
- ユーザーが他機能を割り当て済みである可能性を無視した UI 文言（「アクションボタンで起動できます」と断定しない）。

公式: https://developer.apple.com/design/human-interface-guidelines/action-button

## カメラコントロール（Camera Control）

iPhone 16 系の物理カメラコントロール。軽押し（light press）でカメラオーバーレイを開き、ダブル軽押しでコントロール一覧、指のスライドで値を調整する。カメラアプリを作る場合のみ関係する。

### 🔴 必須（遵守事項）

- コントロールのアイコンは **SF Symbols（Camera & Photos カテゴリ）を使用**すること。カスタムシンボルは非対応。シンボルは状態ではなく**機能**を表す。
- オーバーレイの表示スペースを縦横どちらの向きでも確保し、**アプリ UI をオーバーレイ領域外に配置**すること。オーバーレイと同じコントロールをビューファインダー上に重複表示しない。
- コントロールの追加/削除はランタイムでは不可（実装時に固定）。モードに応じて**有効/無効を切り替える**（例: 写真モード中はビデオ用コントロールを無効化）。

### コントロール種別

| 種別 | 用途 |
|---|---|
| スライダー | 連続値（例: 露出補正） |
| ピッカー | 離散オプション（例: グリッド表示 ON/OFF） |
| システム提供 | ズームファクター・露出補正 |

### 推奨

- コントロール名は短く（Dynamic Type 拡大時にビューファインダーを隠さないため）。
- スライダー値には単位（EV・% 等）を含める — `AVCaptureSlider.localizedValueFormat`。
- よく選ばれる値・等間隔の主要値を `prominentValues` に定義してスライド時に着地しやすくする。
- よく使うコントロールを中央、低頻度を両端に配置し、最後に使ったコントロールを記憶する。
- `LockedCameraCapture` フレームワークでロック画面からの起動に対応する。

### よくある違反・注意点

- ビューファインダー上に UI を積みすぎてプレビューを狭める。
- 単位なしの生数値スライダー（何を調整しているか不明）。

公式: https://developer.apple.com/design/human-interface-guidelines/camera-control

## ゲームコントロール（Game Controls）

タッチコントロール・ゲームコントローラー・キーボード/マウスなど、ゲームの入力設計。

### 🔴 必須（遵守事項）

- **プラットフォームのデフォルト入力（iOS/iPadOS ではタッチ）を必ずサポート**し、ゲームコントローラーは補助的選択肢とすること（全プレイヤーがコントローラーを持つとは限らない）。
- タッチボタンの最小サイズ: **頻繁に使うボタン 44×44pt 以上、低頻度ボタン（メニュー等）28×28pt 以上**。
- タッチコントロールを **Home Indicator・Dynamic Island・セーフエリアと重ねない**こと。
- コントローラーの UI 操作マッピングは慣習に従うこと: A＝決定 / B＝キャンセル・戻る / 左右ショルダー＝セクション移動 / スティック・十字キー＝選択移動 / メニューボタン＝設定・一時停止。**ホーム/ロゴボタンはシステム予約**で使用不可。

### 推奨（タッチコントロール）

- 移動系は画面**左側**（プレイヤーの指の着地点に動的生成する仮想サムスティック）、カメラ系は画面**右側**（ダイレクトタッチのパン）に配置し、親指の可動域に収める。入力エリアは大きく取る。
- ボタンには押下状態の視覚フィードバック（グロー等）を必ず付け、指で隠れても分かる演出＋サウンド＋ハプティクスで強化する。
- ボタンアイコンは**アクションを絵で表現**する（攻撃なら武器の絵）。「A」「R1」のような抽象名は避ける。
- 状況依存でコントロールを表示/非表示し、実行不可能なアクションのボタンは隠す。
- 同時押し・連続押しを要求しない。ダブルタップ・タップ＆ホールドや 1 コントロールへの統合（移動＋ダッシュ等）で置き換える。
- 仮想コントロールの実装には Touch Controller 系フレームワークを使い、可能ならゲーム内オブジェクトへの直接ジェスチャ操作で仮想ボタン自体を減らす。

### 推奨（コントローラー / キーボード）

- コントローラー接続・プロファイルは Game Controller フレームワークで自動検出し、**接続中の実機に合わせたボタングリフ**（SF Symbols の Gaming カテゴリ）を UI に表示する。マルチプレイでは各プレイヤーのコントローラーに合わせる。
- キーボード操作は**単一キーコマンド優先**（I＝インベントリ、M＝マップ、Space＝メインアクション）。修飾は Control ではなく Command（W/A/S/D と併用しやすい）。W/A/S/D の近くに関連コマンドを置く。
- 合理的なデフォルトのキーバインドを用意しつつ、**カスタマイズを必ず実装**する。

### よくある違反・注意点

- タッチ UI をコントローラー前提の移植のまま放置（44pt 未満のボタン・画面上部の多用）。
- システムジェスチャ（画面端スワイプ）と重なる位置に頻用ボタンを置き誤操作を誘発する。

公式: https://developer.apple.com/design/human-interface-guidelines/game-controls

## ジャイロスコープと加速度センサー（Gyroscope and Accelerometer）

デバイスの動き・傾きを Core Motion フレームワーク経由で取得する。

### 🔴 必須（遵守事項）

- モーションデータへのアクセスには**システムの許可リクエスト**が伴う。**なぜ必要かを説明する目的文字列（purpose string）を必ず用意**すること。
- ユーザーが拒否した場合もアプリの主要機能が成立するよう設計すること。

### 推奨

- モーションデータは**ユーザーに具体的な利益がある場合のみ**使う（フィットネスの活動計測、ゲームプレイの強化等）。データ収集目的だけの取得は不可。
- モーションベースのジェスチャ（振る・傾ける）を主要操作にしない: ①正確な再現が難しい ②身体的障害のあるユーザーには実行困難 ③バッテリーを消費する。**アクティブなゲームプレイ以外での「デバイスを動かす操作」は避ける**。

### よくある違反・注意点

- シェイク/傾け操作を唯一の経路にして代替 UI を用意しない。
- バックグラウンドで不要にセンサー購読を継続しバッテリーを浪費する。

公式: https://developer.apple.com/design/human-interface-guidelines/gyro-and-accelerometer

## 近接インタラクション（Nearby Interactions）

Ultra Wideband（UWB）対応デバイス間の距離・相対方向を使った空間的体験（例: AirTag の「探す」、HomePod mini への音声転送）。NearbyInteraction フレームワークを使用。iOS では距離と相対方向の両方が得られる（watchOS は距離のみ。macOS/tvOS/visionOS 非対応）。

### 🔴 必須（遵守事項）

- **UWB 対応デバイスが前提**であることを踏まえ、非対応環境向けの代替手段を用意すること。Nearby Interaction を唯一の操作方法にしない。
- デバイス間インタラクションには**ユーザーの明示的許可**を得ること（アプリ利用中の許可）。
- プライバシー: セッション期間のみ有効な**ランダム生成デバイス識別子**を使う設計を崩さない（恒久 ID の突合をしない）。

### 推奨

- 物理世界のメタファーで設計する（近づくほど認識が鋭くなる）: 距離に応じて**段階的にフィードバックを変化**させる（例: 遠距離は脈動する円 → 近距離で方向矢印）。
- 方向・近接性は**継続的に更新**して動的な物理世界を反映する。
- 視覚・音声・触覚を組み合わせる: 画面を見て操作する場面は視覚、環境内を動く場面は音声・ハプティクス主体に。
- デバイスは**縦向き（ポートレート）での利用を推奨**（横向きは精度・可用性が低下）。視野は iPhone 11 以降の超広角カメラ相当の角度に制限されること、人・動物・大型物体による遮蔽で精度が落ちることを UI 文言で案内する。

### よくある違反・注意点

- 距離の生値（メートル表示）だけを出し、体験としての段階的フィードバックを設計しない。
- 遮蔽・視野外で精度が落ちた際に無言でフリーズする UI。

公式: https://developer.apple.com/design/human-interface-guidelines/nearby-interactions

## フォーカスと選択（Focus and Selection）

キーボード・ゲームコントローラー等でコンポーネント間を移動する際、現在の操作対象を視覚的に示す仕組み。**iPadOS 対応・iOS（iPhone）非対応**。iPadOS 15+ のフォーカスシステムは tvOS と同系だが体験は異なる。

### 🔴 必須（遵守事項）

- **システム提供のフォーカスエフェクトを使う**こと。カスタムエフェクトは真に必要な場合のみ。
- **ユーザー操作なしにフォーカスを移動しない**こと。例外は、フォーカス中の項目が消えた場合に限り、残存項目の 1 ステップ以内へ移す。
- iPadOS では Full Keyboard Access で**テキストフィールド・検索フィールド・リスト項目などコンテンツ要素**のフォーカスをサポートすること（ボタン・スライダー・トグルへの独自フォーカス実装は不要）。

### フォーカスグループとキー操作（iPadOS）

| キー | 動作 |
|---|---|
| Tab | フォーカスグループ間（サイドバー ↔ グリッド ↔ リスト等）の移動 |
| 矢印キー | 同一フォーカスグループ内の項目間移動 |

### 推奨

- 視覚表現の慣習に従う: フォーカスされたリスト項目は白テキスト＋アクセントカラー背景、非フォーカスは標準テキスト＋グレー背景。**テキスト/検索フィールドにはフォーカスリング、リスト/コレクションには行全体ハイライト**を使い分ける。
- ハロー（フォーカスリング）の形状はシステム推論が不適切な場合のみ `UIFocusHaloEffect` で角丸・ベジェパスを調整する（バッジのクリッピング対策等）。
- カスタムビューの遷移順序はリーディング順（前→後、上→下）を基本とし、`UIFocusEnvironment.focusGroupIdentifier` でグループを識別する。グループがフォーカスを受けたとき最初に選ばれるべき項目は `UIFocusGroupPriority` で優先度を上げる。

### よくある違反・注意点

- 画面遷移・データ更新のたびにフォーカスを先頭へ強制リセットする。
- カスタム描画セルでフォーカス表示を実装せず、キーボードユーザーが現在位置を見失う。

公式: https://developer.apple.com/design/human-interface-guidelines/focus-and-selection

## Siri（音声操作）

Siri・Spotlight・ショートカットからアプリの機能とコンテンツを音声/検索で呼び出せるようにする。**App Intents フレームワークでの公開が前提**（公開しない限りシステムはアプリの機能を認識しない）。

### 🔴 必須（遵守事項）

- アプリのアクションを **Intent**、コンテンツを **Entity** として App Intents で公開すること。公開すると Siri / Spotlight / ショートカットで自動的に利用可能になる。
- **Siri が配信する応答に広告・マーケティング・アプリ内課金の売り込みを含めてはならない**。
- Siri を模倣しない・Siri の機能を複製しない・Apple からの応答に見せかけない。予約フレーズ（「Hey Siri」「Call 911」等）を使わない。
- 表記規則: Siri を代名詞で参照せず常に「Siri」と書く。ローカライズ時は「Hey」のみ翻訳し **「Siri」は翻訳しない**（Apple 商標。例: ja_JP「Hey Siri」、ko_KR「Siri야」）。

### 推奨

- **最も使われるアクションを優先**して公開する。ハンズフリー文脈（運転・調理中）で価値が出るアクションを重視する。
- 用語はユーザーに親しみのある語を使う（音声で言いやすい名詞: 「トラック」「曲」等）。
- Entity は**個人的に関連性の高いもの**（最近の検索・お気に入り・ブックマーク）に絞る。メール/メッセージ系は全件提供が適切な例外。
- カスタム応答は最小限にし、ビルトイン応答で足りる場合は作らない。作る場合の設計基準:
  - 明確で具体的な問い返し（「どのスープですか？」○ / 「どれですか？」✗）
  - 簡潔（繰り返し聞かされるためユーモアや冗長さは不快になる）
  - 音声のみでも成立する応答（AirPods 等、画面がない場面を想定）
  - 不要なジェンダー代名詞を避ける（「誰に送りますか？」○）
  - 選択肢が多すぎるときは開かれた質問で絞る
  - デバイス固有表現を避ける（リクエストはどのデバイスからも来る）
  - 応答にアプリ名を含めない（帰属表示はシステムが行う）
  - エラーは原因を具体的に（「チキンヌードルスープは売り切れです」○ / 「注文を完了できません」✗）
  - 音声出力を周囲が聞く可能性と保護者設定によるコンテンツ制限に配慮する
- コンテキスト共有: 画面上コンテンツを App Entity で注釈し、Spotlight のオンデバイスインデックスへ Entity を寄付し、ユーザーの操作履歴を Intent として寄付する（「この写真を風景アルバムに追加して」のような画面文脈リクエストに応えるため）。
- 一般的な機能領域（メール・音楽・写真等）は **App Schema Domain**（プリセットテンプレート）に合わせて実装し、システムの組み込み理解を活用する。

### iOS / iPadOS 固有の注意

- Apple Intelligence 強化版 Siri は対応デバイスのみで動作するため、非対応デバイスでの体験も検証する。
- 頻用アクションは App Shortcuts として公開すると、アクションボタン（前述）や Spotlight からも実行可能になる。

### よくある違反・注意点

- App Intents を実装せず「Siri 対応」をうたう（システムは何も認識できない）。
- カスタム応答にアプリ名・宣伝文・冗長な定型句を入れる。
- 画面表示前提の応答（「下のリストから選んでください」）を音声のみのデバイスに返す。

公式: https://developer.apple.com/design/human-interface-guidelines/siri
