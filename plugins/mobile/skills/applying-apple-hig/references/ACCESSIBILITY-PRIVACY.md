# アクセシビリティ・インクルージョン・プライバシー（Apple HIG）

> 出典: Apple Human Interface Guidelines / Apple Developer Documentation（最終確認 2026-07-13）。仕様は更新されるため、実装時は必ず併記の公式 URL で最新版を確認すること。日本語版がある場合は公式の言語切替または本文の日本語 URL を使い、英語版との差分がある場合は最新側を確認する。
> 判定区分: 本文の「必須」「違反」は作業用の強度ラベルであり、HIG の推奨を一律に App Review 必須へ格上げしない。審査・API 上の必須判定は現行の Apple 公式一次資料で再確認する。

対象: iPhone / iPad（iOS / iPadOS）アプリの設計・実装。アクセシビリティは「後付け」にせず設計初期から組み込む。App Review 必須と判定する項目は、現行の App Review Guidelines または API 契約に直接根拠があるものに限る。

## アクセシビリティ（Accessibility）

### 概要

アクセシブルな UI の 3 原則:

| 原則 | 意味 |
|------|------|
| Intuitive（直感的） | 使い慣れた一貫性のあるインタラクションで操作できる |
| Perceivable（知覚可能） | 情報を単一の感覚（視覚のみ等）に依存させず、視覚・聴覚・触覚の複数チャネルで伝える |
| Adaptable（適応可能） | Dynamic Type・VoiceOver・Reduce Motion 等、ユーザーのデバイス設定に追従する |

障害は「永続的（例: 全盲）」「一時的（例: 感染症による一時的難聴）」「状況的（例: 騒がしい電車内で音が聞こえない）」のスペクトラムとして捉える。アクセシビリティ対応は障害者専用機能ではなく、全ユーザーの利用状況をカバーする設計である。

### 🔴 必須（遵守事項）

| 項目 | 要件（iOS / iPadOS） |
|------|---------------------|
| タップ領域 | iOS / iPadOS では **44x44pt を推奨デフォルト**とし、ボタンのヒット領域も一般則として 44x44pt 以上を目指す。Apple の Accessibility 資料はデフォルト 44x44pt と最小 28x28pt を区別しているため、44x44pt を全要素の絶対最小値や一律の App Review 要件とは扱わない。小さくする場合は利用頻度、標準コンポーネント、間隔、拡張ヒット領域、実機での誤操作を合わせて検証する |
| 要素間隔 | ベゼル（枠）付きコントロールは周囲に約 **12pt**、ベゼルなしコントロールは可視端から約 **24pt** の間隔を確保し、誤タップを防ぐこと |
| Dynamic Type | テキストは Dynamic Type に対応し、**200% 拡大**まで破綻なくレイアウトが追従すること。固定 frame でテキストを切り詰めてはならない |
| テキストサイズ | 本文のデフォルトは **17pt**。いかなるテキストも **11pt 未満**にしてはならない。細いウェイト（Ultralight 等）を使う場合は推奨サイズより大きくする |
| コントラスト比 | WCAG AA 準拠: 17pt 以下のテキストは **4.5:1 以上**、18pt 以上は **3:1 以上**、Bold は **3:1 以上**。Accessibility Inspector で検証する。Dark Mode 対応時はライト/ダーク両方で検証すること |
| 色への非依存 | 色**のみ**で状態・情報を伝えてはならない。形状・アイコン・テキストラベルを併用する（例: エラーは赤 + アイコン + メッセージ） |
| VoiceOver | すべての UI 要素に適切な accessibility label を付与し、VoiceOver で全機能が操作可能であること。装飾画像は読み上げ対象から除外する |
| Reduce Motion | 「視差効果を減らす」設定有効時: ズーム/スケーリング/周辺モーションを削減し、x/y/z 軸の遷移アニメーションを**フェードに置き換える**こと。スプリングアニメーションを抑制し、ぼかしのアニメーションを避ける |
| ジェスチャの代替 | カスタムジェスチャには必ず**代替のタップ操作**を用意すること（例: スワイプ削除 + 編集モードの削除ボタン）。複数指・複数段階の複雑なジェスチャを唯一の操作手段にしてはならない |
| 自動再生 | メディアの自動再生を避ける、または停止コントロールとグローバルなオプトアウト設定を提供すること |
| 点滅コンテンツ | 点滅・フラッシュ表現は Dim Flashing Lights 設定に対応し、軽減できるようにすること |
| タイマー消去 | 情報を時間経過で自動消去してはならない（明示的なユーザー操作で閉じられるようにする） |

### 推奨

- **システム定義色・システムフォント（SF Pro / text styles）を優先する**。システム色は Increase Contrast 等の設定に自動追従する。
- **聴覚コンテンツへのテキスト代替**を提供する: 動画にはキャプション（音声同期）、映像作品には字幕、重要な視覚情報には音声解説（audio description）、ポッドキャスト等にはトランスクリプト。
- オーディオキューには**ハプティクスや視覚的インジケータを併用**する（画面外コンテンツの通知等）。Music Haptics・Audio Graphs への対応も検討する。
- **Voice Control / Full Keyboard Access / Switch Control** で全機能が操作できることを確認する。iPadOS では外付けキーボードのみでの完全操作を検証する。システム標準のキーボードショートカットを上書きしない。
- **Siri / App Shortcuts** に対応し、定型タスクを音声で完結できるようにする。
- カスタムジェスチャよりも**システム標準ジェスチャを優先**し、学習コストを最小化する。
- ゲームでは難易度カスタマイズ（成功基準・リアクション時間・操作支援）を提供する。
- **Assistive Access**（認知アクセシビリティ向けの簡易モード）への最適化: コア機能に絞り、1 画面 1 インタラクション、破壊的操作（削除等）には確認を挟む。
- App Store の **Accessibility Nutrition Labels** で対応機能を申告する。
- テストは **Accessibility Inspector** と実機の VoiceOver / Dynamic Type 最大設定で行う。シミュレータだけで済ませない。

### よくある違反・注意点

- ❌ 小さいアイコンボタンを密集させ、間隔・利用頻度・実際のヒット領域を検証しない（→ 標準コンポーネントを優先し、必要に応じて contentInset や `contentShape` でヒット領域を広げ、実機で確認する）。
- ❌ Dynamic Type 対応を `UIFont.systemFont(ofSize: 17)` の固定サイズで実装する（→ `UIFont.preferredFont(forTextStyle:)` / SwiftUI の `.font(.body)` を使い、`adjustsFontForContentSizeCategory` を有効にする）。
- ❌ グラフの系列を色だけで区別する（→ 形状・パターン・直接ラベルを併用）。
- ❌ Reduce Motion 有効時にも大きなズーム遷移を再生する（→ `UIAccessibility.isReduceMotionEnabled` / SwiftUI `@Environment(\.accessibilityReduceMotion)` で分岐しフェードへ）。
- ❌ カスタムコントロールに accessibility label / traits を付けず、VoiceOver で「ボタン」とだけ読まれる、または読まれない。
- ❌ プレースホルダー色・薄いグレー文字（コントラスト 3:1 未満）を本文情報に使う。

公式: https://developer.apple.com/design/human-interface-guidelines/accessibility

## インクルージョン（Inclusion）

### 概要

インクルーシブデザインは一度きりのチェックではなく**反復的プロセス**。年齢・ジェンダー・人種・セクシュアリティ・身体/認知属性・障害・言語/文化・宗教・教育・社会経済的背景など、多角的な視点でコンテンツと UI を点検する。「不快なコンテンツがない」は必要条件にすぎず、積極的に多様なユーザーを歓迎する表現を選ぶこと。

### 🔴 必須（遵守事項）

| 項目 | 要件 |
|------|------|
| 直接的な語りかけ | ユーザーには「you / your（あなた）」で語りかけること。「the user」「the player」のような三人称は距離感を生むため UI テキストで使わない |
| 不要なジェンダー参照の排除 | 「his or her」等の二項的表現を避け、文を書き換える（例: "Subscribers can post recipes to your shared folder"）。ジェンダーは女性/男性の二項では捉えない |
| ジェンダー入力が必須の場合 | 選択肢に「nonbinary」「自己記述（self-identify）」「回答しない（decline to state）」を含め、可能なら使用代名詞の指定機能を提供すること |
| 抑圧的・文化特異的慣用句の禁止 | 差別的な歴史を持つ慣用句（例: "grandfathered in"）や、特定文化でしか通じない言い回しを使わない（翻訳不能かつ排他的） |
| ステレオタイプの禁止 | 職業と性別の固定観念（医者=男性、看護師=女性）、家族構成の固定観念（母・父・実子のみ）、民族/ジェンダーステレオタイプを強化する描写（ヒーロー vs 悪役の配役含む）を避ける |
| 前提を置いた質問の禁止 | セキュリティ質問等でユーザーの経歴を前提にしない。「大学で好きだった科目」「最初の車のメーカー」ではなく「好きな活動」「最初の友達の名前」など普遍的経験に基づく質問にする |

### 推奨

- **トーンの一貫性**: 学術的・専門的すぎるトーンは特定層だけを歓迎するメッセージになる。平易な言葉を使い、専門用語には定義を添える。
- **「we / our」の使用は最小限に**: ソフトウェアや企業を代表する文脈に限定する。過剰な使用は親密さの押し付けになる。
- **ユーモアは慎重に**: 文化間で翻訳されにくく、誤解・不快のリスクがある。
- **人物表現の多様性**: 画像・イラストには異なる人種・体型・年齢・身体能力の人物を登場させる（例: フィットネスアプリで多様な能力レベルの人がエクササイズを実演）。障害のある人を自然に含め、障害を否定的文脈で使わない。人物を語る際は障害より先に成果・目標を述べる（person-first）。
- **アバター・絵文字・人物グリフはカスタマイズ可能に**する。汎用的な人の表現には SF Symbols のジェンダーニュートラルなシンボル（`person.crop.circle`・`person.3.fill`・`figure.wave` 等）を使う。
- **富裕さの過度な演出を避ける**: 大多数のユーザーに親近感のある環境・持ち物を描く。
- **国際化を前提に設計**: RTL（アラビア語・ヘブライ語等）レイアウト対応、SF Symbols の RTL 対応グリフ・多言語グリフの活用。色の文化的意味（白 = 純潔/喪、等）はローカライズ先ごとに検証する。
- **オンボーディング**は新規ユーザーに段階的な学習パスを、経験者にはスキップ手段を提供する。

### よくある違反・注意点

- ❌ 性別選択が「男性/女性」の 2 択のみ、かつ必須入力（→ 本当に必要か再検討し、必要なら選択肢を拡張）。
- ❌ 英語 UI 文をそのまま直訳する前提の設計（慣用句・ジョーク混在）→ ローカライズ耐性のある平易な原文にする。
- ❌ デフォルトアバターが特定の人種・性別を表す画像。
- ❌ 「簡単です！」のような能力を前提とした文言（できないユーザーを疎外する）。

公式: https://developer.apple.com/design/human-interface-guidelines/inclusion

## プライバシー（Privacy）

### 概要

プライバシー設計の 2 本柱は**データ最小化**（機能に実際に必要なデータだけを、必要な時に要求する）と**透明性とコントロール**（何をなぜ使うかを明確に伝え、ユーザーが選択できるようにする）。App Store 提出時にはプライバシー詳細（App Privacy Details）の申告が必須で、ユーザーはダウンロード前にそれを見て判断する。

許可（permission）が必要な主な対象: 位置情報・健康・財務・連絡先などの個人情報、メール/メッセージ/カレンダー/写真・動画などのユーザー生成コンテンツ、カメラ・マイク、Bluetooth・HomeKit・ローカルネットワーク等の保護リソース、広告識別子（トラッキング）。

### 🔴 必須（遵守事項）

#### データ最小化

| 項目 | 要件 |
|------|------|
| 要求範囲 | 機能の動作に**実際に必要なデータ・リソースだけ**を要求すること。「あると便利」レベルで許可を求めない |
| 処理場所 | 可能な限り**オンデバイス処理を優先**すること（Neural Engine / Core ML 活用）。サーバー送信は必要最小限に |
| 機密情報の保存 | パスワード・トークン等を平文ファイル・UserDefaults に保存してはならない。**Keychain** を使うこと |
| 認証 | カスタムのパスワード管理 UI を自作しない。**Passkeys を第一選択**とし、パスワードを使う場合は 2FA + 生体認証（Face ID / Touch ID・LocalAuthentication framework）で補強。Sign in with Apple / Password AutoFill に対応する |

#### 許可要求のタイミング

| シナリオ | 要件 |
|---------|------|
| 原則 | **ユーザーがその機能を使う明確な意図を示した直後**に要求すること（例: 写真添付ボタンをタップ → 写真アクセスを要求）。起動直後に文脈のない許可ダイアログを連発してはならない |
| 起動時要求が許される場合 | その許可がアプリの中核機能に不可欠な場合のみ（例: ナビゲーションアプリの位置情報）。その場合も理由が自明であること |
| 一時的な位置情報 | 特定機能で一回だけ位置情報が要る場合は **LocationButton**（システム提供の位置情報ボタン）を使う。初回タップでシステム説明が出て、以降は「1 回のみ許可」相当の一時許可（アプリ終了で失効）になる |

#### Purpose String（許可ダイアログの説明文）

Info.plist の `NS*UsageDescription` に設定し、システムアラートに表示される文言。以下をすべて満たすこと:

| 要素 | ルール |
|------|--------|
| 態 | 能動態で書く（Active voice） |
| 内容 | 「何のために・いつ使うか」を具体的に書く |
| 形式 | Sentence case・簡潔・**末尾ピリオド必須** |

- ✅ 良い例: "The app records during the night to detect snoring sounds."（能動・具体的・理由が明確）
- ❌ 悪い例: "Microphone access is needed for a better experience."（受動・曖昧・理由不明）
- ❌ 悪い例: "Turn on microphone access."（命令文・正当性の説明なし）

#### App Tracking Transparency（ATT）

| ルール | 詳細 |
|--------|------|
| システムアラート必須 | トラッキング許可は**システム提供の ATT アラートのみ**で要求すること |
| インセンティブ禁止 | 「許可すればクレジット進呈」等、トラッキング同意を機能・報酬の条件にしてはならない（App Review 却下対象） |
| 偽装禁止 | システムアラートに似せたカスタム画面、"Allow Tracking" ボタンを持つ独自 UI、システムアラートの画像の表示・加工は禁止 |
| 誘導禁止 | アラートの「許可」ボタンへ矢印・注釈等で視覚的に誘導してはならない |
| 事前説明画面（pre-alert） | 出す場合はボタンを **1 つだけ**（"Continue" / "Next"。**"Allow" は使用禁止**）にし、その機能は「システムアラートを開く」のみとする。キャンセル/クローズ等の別アクションを付けない |

### 推奨

- 許可を求める**前に**、機能の価値が UI 上で伝わっている状態を作る（拒否されたら再要求は容易でない。システムダイアログは原則一度しか出ない）。
- 拒否された場合のフォールバックを設計する: 機能を静かに無効化しつつ、設定アプリへの導線（`UIApplication.openSettingsURLString`）を該当機能の文脈で案内する。
- 複数の許可を同時に連続要求しない。1 つずつ、それぞれの機能の文脈で。
- ユーザーのプライバシー選択（Hide My Email・Mail Privacy Protection・トラッキング拒否）を尊重し、迂回しない。
- CloudKit 等、暗号化とキー管理をシステムに任せられる仕組みを活用する。
- App Store Connect の App Privacy Details は実装変更のたびに見直し、最新に保つ。

### よくある違反・注意点

- ❌ 初回起動時に位置情報・通知・カメラ・ATT のダイアログを立て続けに表示する（拒否率が跳ね上がり、以後の再許可導線が重くなる）。
- ❌ Purpose String がテンプレ的（"This app needs camera access."）で、利用目的を具体的に説明していない。提出時は現行の App Review Guidelines と API 契約を再確認する。
- ❌ トラッキング拒否ユーザーに対して機能を制限する・再同意を執拗に迫る。
- ❌ 写真ライブラリ全体への許可を要求する（→ 用途がユーザー選択の数枚なら PHPicker を使えば許可ダイアログ自体が不要。限定アクセス（limited access）にも正しく対応する）。
- ❌ アクセストークンを UserDefaults に保存する（→ Keychain）。
- ❌ Purpose String 未設定のまま保護リソースにアクセスしてクラッシュする（iOS は該当 `NS*UsageDescription` がないと即クラッシュ）。

公式: https://developer.apple.com/design/human-interface-guidelines/privacy

## 実装チェックリスト（iOS / iPadOS）

```
アクセシビリティ
□ 44x44pt を推奨デフォルトとして、各コントロールのサイズ・間隔・ヒット領域を利用頻度に応じて検証
□ Dynamic Type 200% で全画面のレイアウト検証
□ コントラスト比 4.5:1（17pt 以下）/ 3:1（18pt 以上・Bold）
□ 色のみで情報を伝えている箇所ゼロ
□ VoiceOver で全機能操作可・全要素にラベル
□ Reduce Motion 有効時の遷移をフェード化
□ カスタムジェスチャ全てに代替タップ操作
□ 動画キャプション・音声トランスクリプト
□ Accessibility Inspector + 実機検証

インクルージョン
□ UI テキストは「you」で直接語りかけ・不要なジェンダー参照なし
□ 慣用句・文化特異表現・ステレオタイプの排除
□ 人物画像/アバターの多様性・カスタマイズ性
□ RTL レイアウト・ローカライズ耐性

プライバシー
□ 要求する許可は機能に必須のものだけ
□ 許可要求は機能使用の文脈で（起動時連発禁止）
□ 全 Purpose String が能動態・具体的・ピリオド終止
□ ATT はシステムアラートのみ・誘導/報酬なし
□ 機密情報は Keychain・オンデバイス処理優先
□ 拒否時のフォールバックと設定アプリ導線
□ App Privacy Details が実装と一致
```
