# 色・マテリアル（Liquid Glass）・モーション

> 出典: Apple Human Interface Guidelines / Apple Developer Documentation（最終確認 2026-07-13）。仕様は更新されるため、実装時は必ず併記の公式 URL で最新版を確認すること。日本語版がある場合は公式の言語切替または本文の日本語 URL を使い、英語版との差分がある場合は最新側を確認する。
> 判定区分: 本文の「必須」「違反」は作業用の強度ラベルであり、HIG の推奨を一律に App Review 必須へ格上げしない。審査・API 上の必須判定は現行の Apple 公式一次資料で再確認する。

iPhone/iPad（iOS/iPadOS）アプリの色・ダークモード・マテリアル・モーション・画像アセットに関する実装判断ガイド。

---

## 色（Color）

### 概要

色は階層・状態・インタラクティビティを伝える手段であり、装飾ではない。システムカラー（標準色）とセマンティックカラー（意味ベースの動的カラー）を使うことで、ライト/ダークモード・高コントラスト設定・将来の OS 更新に自動追従する。

### 🔴 必須（遵守事項）

| 項目 | 内容 |
|------|------|
| システム API の使用 | 色は `SwiftUI.Color` / `UIKit.UIColor` のシステム定義カラーで指定すること。RGB 値のハードコーディングをしてはならない（明暗モード・高コントラストに追従できない） |
| セマンティックカラーの用途遵守 | `label` / `secondaryLabel` / `systemBackground` 等は「見た目の色」ではなく「意味」で選ぶこと。将来 OS で実際の色値が変わりうるため、値に依存した使い方をしてはならない |
| コントラスト比 | 通常テキスト（17pt 以下相当）は **4.5:1 以上**、大きいテキスト（18pt 以上）は **3:1 以上**、アイコン等のグラフィック要素は **3:1 以上** を確保すること（WCAG 準拠） |
| 高コントラスト対応 | カスタムカラーを定義する場合、Increase Contrast 有効時のバリアント（ライト/ダーク両方）を Asset Catalog で必ず提供すること |
| 色だけに依存しない | 状態・エラー・区別を色のみで伝えてはならない。テキスト・形状・シンボルを併用すること（色覚多様性対応） |
| 意味の一貫性 | 同じ色をインタラクティブ要素と非インタラクティブな装飾の両方に使ってはならない（タップ可能に見える誤解を生む） |

### iOS/iPadOS のセマンティックカラー体系

**背景色（階層構造）** — 通常 UI とグループ化 UI（設定アプリ型のリスト）で系統が分かれる:

| 通常 UI | グループ化 UI | 用途 |
|---------|--------------|------|
| `systemBackground` | `systemGroupedBackground` | 最背面 |
| `secondarySystemBackground` | `secondarySystemGroupedBackground` | 1 段上のレイヤー |
| `tertiarySystemBackground` | `tertiarySystemGroupedBackground` | 2 段上のレイヤー |

**テキストカラー（コントラスト順）**:

| カラー | 用途 |
|--------|------|
| `label` | 主要テキスト（最高コントラスト） |
| `secondaryLabel` | 補助テキスト |
| `tertiaryLabel` | 非アクティブ・低優先情報 |
| `quaternaryLabel` | 透かし等の最低優先情報 |
| `placeholderText` | 入力フィールドのプレースホルダ |

**その他**: `separator`（背景が透ける区切り線）/ `opaqueSeparator`（不透明な区切り線）/ `link`（リンク色）/ `systemGray`〜`systemGray6`（6 段階グレー・iOS/iPadOS 専用。SwiftUI の `Color.gray` は `systemGray` 相当）。

**標準システムカラー**: `red` / `orange` / `yellow` / `green` / `mint` / `teal` / `cyan` / `blue` / `indigo` / `purple` / `pink` / `brown`。いずれもライト/ダークで別の実値に自動調整される。

### 推奨

- カスタムブランドカラーは Asset Catalog の Color Set で定義し、Light / Dark / High Contrast の各バリアントを揃える。
- 広色域を使う場合は Display P3（16-bit 推奨）で定義し、sRGB フォールバックも Asset Catalog のバリアントで併提供する（対応外ディスプレイでの色クリッピング回避）。
- P3 が効果を発揮するのは写真・動画・データ可視化。通常の UI 色は sRGB で十分。
- 屋外の直射日光下（色が沈む）・暗所（彩度が上がって見える）の両方で実機確認する。True Tone ディスプレイの影響は `UIWhitePointAdaptivityStyle` で調整できる。
- 色の文化的意味の差（例: 赤=危険/慶事、株価の上昇色の国差）をローカライズ時に考慮する。
- Liquid Glass 上の色: デフォルトは無色（背景が透ける）とし、色付け（tint）は**画面内の最重要アクション 1 つに限定**する。複数のコントロールをそれぞれ別色で tint しない。

### よくある違反・注意点

- ❌ `Color(red:green:blue:)` の直値指定 → ダークモードで背景と同化・コントラスト崩壊。
- ❌ `secondaryLabel` 相当の情報に `gray` 直指定 → 高コントラスト設定に追従しない。
- ❌ ブランドカラーをテキスト・ボタン・装飾に無差別適用 → どれがタップ可能か判別不能になる。
- ❌ 高コントラストバリアント未提供のカスタムカラー → アクセシビリティ監査で不合格。

公式: https://developer.apple.com/design/human-interface-guidelines/color

## ダークモード（Dark Mode）

### 概要

ダークモードはシステム全体の外観設定であり、ユーザーはライト/ダーク/自動（時刻で切替。アプリ実行中にも切り替わる）を選ぶ。**iOS/iPadOS アプリはダークモード対応が事実上必須**。対応しない・独自設定で上書きする挙動はバグと認識される。

### 🔴 必須（遵守事項）

| 項目 | 内容 |
|------|------|
| システム設定に従う | アプリ独自の外観切替設定を追加してシステム設定を無視してはならない（正当な例外: メディアビューア等の常時ダーク UI のみ） |
| 実行中の切替に対応 | 自動モードではアプリ実行中に外観が変わる。ハードコード色・キャッシュした色値は破綻する。セマンティックカラー/ Color Set で動的に解決すること |
| 4 条件でのテスト | ①ライト ②ダーク ③各モード + Increase Contrast ④各モード + Reduce Transparency の組合せで必ず表示確認すること |
| Base/Elevated の維持 | ダークモードの背景はベース層（純黒に近い）とエレベート層（`#1C1C1E` 程度・モーダルやポップオーバー）の 2 系統をシステムが自動管理する。`systemBackground` 系を使い、カスタム背景でこの深度表現を壊してはならない |
| コントラスト再確認 | ダーク背景上でも通常テキスト 4.5:1 以上を維持すること。小さいテキストは 7:1 を推奨 |

### 推奨

- テキストは `label` 系 4 階層で表現する（ライトで黒系→ダークで白系に自動反転）。`UIColor.black` / `UIColor.white` の直指定は禁止レベルで避ける。
- アイコンは SF Symbols を最優先で使う（自動適応 + vibrancy 対応）。
- カスタム画像・アイコンは、ライト版のままダーク背景で視認できるかを確認し、必要な場合のみ Asset Catalog にダーク専用バリアントを追加する（例: 輪郭線の追加、色値を明るく調整）。
- テキスト描画はシステムコンポーネント（`UILabel` / `UITextField` / SwiftUI `Text`）を使う。カスタム描画（`CATextLayer` 等）は vibrancy が効かないことがある。
- マルチウィンドウ（iPadOS）では、前面ウィンドウが Elevated 背景・背面が Base 背景になることで前後関係が表現される。システム背景色を使っていれば自動で得られる。

### よくある違反・注意点

- ❌ 「ダークテーマ設定」をアプリ内に独自実装 → システム設定と二重管理になりユーザー混乱。
- ❌ ライトモードのスクリーンショットだけで QA 完了 → ダーク + Increase Contrast でのコントラスト不足を見逃す。
- ❌ 白背景前提の透過 PNG をそのまま使用 → ダークで縁が消える/滲む。
- ❌ ダークモードを「色を反転するだけ」と捉える → Elevated/Base の階層・vibrancy・影の扱いが崩れる。

公式: https://developer.apple.com/design/human-interface-guidelines/dark-mode

## マテリアル（Materials）と Liquid Glass

### 概要

マテリアルは背景をぼかして前景（テキスト・コントロール）と視覚的に分離し、奥行きを生む効果。2025 年の新デザインシステムでは、ナビゲーション層に **Liquid Glass** が導入され、タブバー・サイドバー・ツールバー等の機能要素がコンテンツの上に「浮遊」する構造になった。

### 🔴 必須（遵守事項）

| 項目 | 内容 |
|------|------|
| Liquid Glass は機能層限定 | Liquid Glass はタブバー・ツールバー・サイドバー等の**ナビゲーション/コントロール層にのみ**使うこと。コンテンツ層に適用してはならない（例外: スライダー・トグル等がアクティブ操作中に一時的にまとう場合のみ） |
| 過剰適用の禁止 | カスタム要素への Liquid Glass 適用は最重要の機能要素に限定すること。多用するとコンテンツから注意を奪い視覚階層が崩壊する |
| 標準コンポーネント優先 | UIKit / SwiftUI の標準コンポーネントは Liquid Glass を自動適用する。まず標準を使い、独自再実装しないこと |
| マテリアル上は vibrant color | マテリアル上のテキスト・塗り・区切り線にはシステム定義の vibrancy スタイルを使うこと。`systemGray3` 等の通常色をマテリアルに重ねてはならない（Dark Mode・Increase Contrast・Reduce Transparency に追従しない） |
| `quaternaryLabel` の制限 | `thin` / `ultraThin` マテリアル上で `quaternaryLabel` 相当の vibrancy を使ってはならない（コントラスト不足） |
| clear + 明背景の減光 | Liquid Glass の `clear` バリアントを明るい背景上で使う場合、**不透明度 35% のダークディミング層**を背後に追加すること |
| Reduce Transparency 対応 | 透明度削減設定の有効時にも判読可能であることを確認すること（システムマテリアルは自動対応） |

### Liquid Glass の 2 バリアント

| バリアント | 特性 | 使いどころ |
|-----------|------|-----------|
| `regular` | 背景ぼかし + 輝度自動調整。判読性が高い | デフォルト。テキスト量が多い要素（サイドバー・ポップオーバー・アラート） |
| `clear` | 高い透過性 | 写真・動画などメディアの上に載るコントロール（明背景では 35% ディミング必須） |

SwiftUI では `.glassEffect(_:in:)`、標準マテリアルは `.background(Material.regular)` 等で適用する。UIKit は `UIVisualEffectView` + `UIBlurEffect` + `UIVibrancyEffect`。

### 標準マテリアル 4 種（iOS/iPadOS）

| マテリアル | 透過性 | 選択基準 |
|-----------|--------|---------|
| `ultraThin` | 最も透ける | 背景コンテキストの保持を最優先する場合 |
| `thin` | 高透過 | 中程度の分離 |
| `regular` | 中間 | デフォルト・汎用 |
| `thick` | 最も不透明 | テキストの判読性を最大化したい場合・暗い背景 |

選択は「見た目の色」ではなく**セマンティックな意味と推奨用途**で行う。厚いほど判読性が上がり、薄いほど背景の文脈が残る。

### 推奨

- マテリアル上のラベルは vibrancy の `label` → `secondaryLabel` → `tertiaryLabel` → `quaternaryLabel`、塗りは `fill` / `secondaryFill` / `tertiaryFill`（塗り 3 種と区切り線は全マテリアルで使用可）。
- ツールバー/タブバーの Liquid Glass は背景コンテンツに応じて明暗を自動調整する。カラフルな背景上ではコントロール自体はモノクロ配色を選ぶと視認性が安定する。
- Liquid Glass への色付け（tint）は主要アクション 1 つに絞る（→「色」セクション参照）。

### よくある違反・注意点

- ❌ コンテンツカード・リストセル自体に Liquid Glass を適用 → 機能層とコンテンツ層の区別が消え階層が破綻する。
- ❌ マテリアル上に通常の `UIColor` テキストを直置き → Reduce Transparency 有効時に読めなくなる。
- ❌ 判読性が欲しい場面で `ultraThin` を選ぶ → `thick` または `regular` が正解。
- ❌ 標準タブバーを不透明カスタムビューで再実装 → 新デザインシステムの外観・挙動（スクロールエッジ効果等）を失う。

公式: https://developer.apple.com/design/human-interface-guidelines/materials

## モーション（Motion）

### 概要

モーションは操作結果のフィードバック・状態遷移・空間関係の理解を助けるために使う。装飾目的の動きは価値を生まず、酔い・注意散漫・アクセシビリティ違反の原因になる。iOS/iPadOS に固有の追加制約はなく、以下の汎用原則をそのまま適用する。

### 🔴 必須（遵守事項）

| 項目 | 内容 |
|------|------|
| 目的のあるモーションのみ | すべてのアニメーションはフィードバック・関係性の提示・操作結果の理解のいずれかに寄与すること。装飾だけの動きを追加してはならない |
| モーション非依存 | 重要な情報をモーションだけで伝えてはならない。テキスト・haptics・サウンド等の代替手段を必ず併用すること |
| Reduce Motion 対応 | `accessibilityReduceMotion`（SwiftUI: `@Environment(\.accessibilityReduceMotion)` / UIKit: `UIAccessibility.isReduceMotionEnabled`）を参照し、有効時はアニメーション量を削減またはクロスフェード等の控えめな遷移へ置換すること。システムコンポーネントは自動対応するが、カスタムアニメーションは自前対応が必要 |
| 短く・キャンセル可能に | フィードバックアニメーションは短時間・正確にし、完了を待たずに次の操作へ進めること。ユーザーをアニメーション終了待ちでブロックしてはならない |
| 頻出操作に長い動きを付けない | 高頻度のインタラクション（リスト遷移・キーボード操作等）に目立つアニメーションを重ねると煩わしさが蓄積する |

### 推奨

- ジェスチャーに追従する現実的な動き（指の速度・方向と一致する遷移）を使う。ユーザーの直接操作から始まるモーションが最も自然。
- SF Symbols 5 以降のシンボルアニメーション（bounce / pulse / variable color 等）を状態変化の表現に活用する。
- ゲーム等の高負荷描画は 30〜60fps を安定維持できるデフォルト設定にし、必要ならユーザーが品質/省電力を切り替えられるようにする。
- 酔い（motion sickness）対策の一般原則: 視野の周辺部で大きく動かさない・画面全体が動く演出（全画面パララックス・自動スクロール）を避ける・大きな要素の移動はフェードで置き換える・約 0.2Hz 付近のゆっくりした揺れは特に酔いを誘発するため振幅を抑える。iOS でも全画面演出やパララックス壁紙的な効果を実装する際に適用される。
- Reduce Motion 有効時の代替: 位置移動 → クロスフェード、ズーム遷移 → ディゾルブ、自動再生アニメーション → 静止 + 明示的再生ボタン。

### よくある違反・注意点

- ❌ 起動時・画面遷移ごとの長い演出アニメーション → 2 回目以降は苦痛になる。
- ❌ Reduce Motion を無視したカスタムのバウンス/ズーム遷移 → アクセシビリティ違反。前庭障害のあるユーザーに実害。
- ❌ ローディング完了やエラーを点滅・振動アニメーションのみで通知 → モーション非依存の原則違反。
- ❌ スクロールに連動した過剰なパララックス → 酔いの主要因。

公式: https://developer.apple.com/design/human-interface-guidelines/motion

## 画像（Images）

### 概要

iOS/iPadOS の座標系は point（デバイス非依存の抽象単位）で、scale factor（@1x/@2x/@3x）が 1pt あたりのピクセル数を決める。全ビットマップ画像は対応デバイスすべての scale factor 分を提供する必要がある。

### 🔴 必須（遵守事項）

| 項目 | 内容 |
|------|------|
| 必須スケールファクタ | **iOS（iPhone）: @2x と @3x の両方**、**iPadOS: @2x** を必ず提供すること（@1x は現行 iPhone/iPad には不要） |
| 高解像度アセット提供 | アプリ内の全ビットマップ画像について、サポートする全デバイス向けの高解像度版を用意すること |
| 低解像度から設計 | ベースを 1x 相当で設計し、整数座標にコントロールポイントを置いて 2x/3x へ拡大生成すること。**高解像度から縮小して作ってはならない**（にじみ・モアレの原因） |
| 命名規則 | `image.png`（@1x）/ `image@2x.png` / `image@3x.png` の接尾辞規則に従うこと（Asset Catalog 管理を推奨） |
| カラープロファイル埋め込み | すべての画像にカラープロファイルを含めること（ディスプレイ間の色一貫性の担保） |
| 実機確認 | ピクセル化・引き伸ばし・圧縮劣化がないか、対象デバイスの実機で確認すること |

### フォーマット選択

| 用途 | 推奨フォーマット |
|------|-----------------|
| UI 用ビットマップ/ラスター | De-interlaced PNG |
| フルカラー不要な PNG | 8-bit カラーパレット（サイズ削減） |
| 写真 | JPEG（品質最適化）または HEIC |
| フラットなアイコン・インターフェースアイコン | **PDF / SVG（ベクター）を第一候補** — 単一アセットで全スケールに対応 |

### 推奨

- アイコン類は可能な限り SF Symbols → ベクター（PDF/SVG, Asset Catalog の "Preserve Vector Data"）→ ビットマップの優先順で選ぶ。
- Asset Catalog で画像を管理し、scale factor・ダークモードバリアント・高コントラストバリアントを一元化する。
- ダークモード用の画像バリアントの要否を画像ごとに判断する（→「ダークモード」セクション参照）。

### よくある違反・注意点

- ❌ @2x のみ提供して iPhone（@3x 機）でぼやける／@3x のみで iPad（@2x）にサイズ過剰。
- ❌ 大きな元画像を実行時に縮小表示 → メモリ浪費 + モアレ。表示サイズに合わせたアセットを用意する。
- ❌ カラープロファイルなしの書き出し → デバイスによって色味が変わる。
- ❌ ベクターで済むアイコンをビットマップ 3 枚組で管理 → 更新コスト増・スケール品質低下。

公式: https://developer.apple.com/design/human-interface-guidelines/images
