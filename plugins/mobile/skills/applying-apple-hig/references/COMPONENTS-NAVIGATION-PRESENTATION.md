# ナビゲーション・プレゼンテーション系コンポーネント

> 出典: Apple Human Interface Guidelines / Apple Developer Documentation（最終確認 2026-07-13）。仕様は更新されるため、実装時は必ず併記の公式 URL で最新版を確認すること。日本語版がある場合は公式の言語切替または本文の日本語 URL を使い、英語版との差分がある場合は最新側を確認する。
> 判定区分: 本文の「必須」「違反」は作業用の強度ラベルであり、HIG の推奨を一律に App Review 必須へ格上げしない。審査・API 上の必須判定は現行の Apple 公式一次資料で再確認する。

対象: iPhone / iPad（iOS / iPadOS）アプリの設計・実装。全コンポーネント共通の前提として、44x44pt を快適なタップ領域の推奨デフォルトとし、利用頻度・間隔・実際のヒット領域を検証する。44x44pt は全要素の絶対最小値や一律の App Review 要件ではない。レイアウトは Dynamic Type によるテキスト拡大へ追従させる。

## タブバー（Tab Bars）

### 概要
アプリのトップレベルセクション間を切り替えるナビゲーション。各セクション内のナビゲーション状態を保持したまま瞬時に切り替えられる。iOS では画面下部に Liquid Glass 背景でフロート表示、iPadOS では画面上部に表示される。

### 🔴 必須（遵守事項）
- タブバーは**ナビゲーション専用**とし、アクションの起動に使ってはならない（アクションは toolbar へ）
- 各タブにはアイコンと**テキストラベルの両方**を付けること（可能な限り 1 単語で説明的に）
- ナビゲーション中もタブバーを常時表示すること（例外: 一時的で自己完結したモーダルビューのみ）
- タブボタンを無効化・非表示にしてはならない。セクションが空でもタブは残し、空である理由をコンテンツ側で説明する
- バッジは赤い楕円＋白文字（数字または感嘆符）で、**重要な情報のみ**に使う

### 推奨
- タブ数は少なく保つ。デバイスサイズ・向きで表示上限を超えると末尾が「More」タブ化しアクセス性が落ちるため、More が発生しない設計にする
- アイコンは SF Symbols を使い、塗りつぶし（filled）スタイルで統一する
- 複雑な情報構造には、タブバー単体でなく sidebar か「サイドバーに変換可能なタブバー」（SwiftUI: `TabViewStyle.sidebarAdaptable`）を検討する
- iOS: MiniPlayer のような付属アクセサリがある場合、下スクロール時のタブバー最小化を検討（`TabBarMinimizeBehavior` / `UITabBarController.MinimizeBehavior`）
- iOS: 末尾（trailing）に専用の検索タブ（search tab）を配置できる（詳細は search-fields 節）
- iPadOS: タブバーのユーザーカスタマイズ（`TabViewCustomization` / `UITab.Placement`）をサポートし、**デフォルトのタブは 5 個以下**にする（コンパクト／レギュラー間の継続性維持）

### よくある違反・注意点
- タブに「投稿する」等のアクションを混ぜる（画面遷移でなく動作を起こすタブは HIG 違反）
- タブラベルと背景コンテンツに同系色を使い視認性を落とす。鮮やかなコンテンツ上ではモノクロームのタブバー外観を優先する
- オーバーフロー（More タブ）前提のタブ設計

公式: https://developer.apple.com/design/human-interface-guidelines/tab-bars

## ナビゲーションバー（Navigation Bars）→ ツールバーに統合

### 概要
HIG では従来の「Navigation Bars」トピックは **Toolbars に統合**された（navigation-bars の URL は toolbars へリダイレクトされる）。画面上部のバー＝タイトル・戻るボタン・画面単位アクションの置き場、という設計論はすべて次節「ツールバー」の規則に従う。

### 🔴 必須（遵守事項）
- 戻る・閉じるは**標準のシンボルボタン**を使い、テキストラベル「Back」「Close」を使ってはならない
- 各画面に現在地を示す**有用なタイトル**を付ける。アプリ名をタイトルにしてはならない（階層情報を提供しないため）
- タイトルが冗長になる場合はタイトル領域を空にしてよい（例: Notes）

### 推奨
- iOS では**大型タイトル**（`UINavigationBar.prefersLargeTitles`）で現在地の手がかりを強化する。スクロール開始で標準サイズへ、先頭へ戻ると大型へ自動遷移させる
- タイトルは 15 文字以下を目安にし、他のコントロール用スペースを確保する

### よくある違反・注意点
- カスタム戻るボタンで標準と異なる見た目・挙動にする（一貫性が壊れ、スワイプバックの期待も裏切る）
- 上部バーにボタンを詰め込みタイトルを圧迫する

公式: https://developer.apple.com/design/human-interface-guidelines/toolbars

注: navigation-bars は本トピックへ統合され、上記 URL へリダイレクトされる。

## ツールバー（Toolbars）

### 概要
ビューの上部または下部に水平配置されるコントロール群。①現在ビューのタイトル、②ナビゲーション（戻る・進む・検索フィールド）、③アクション（ボタン・メニュー）の 3 種のコンテンツを論理セクションにまとめる。

### 🔴 必須（遵守事項）
- タイトルは **15 文字以下**を目安に短くすること
- グループは**最大 3 つ**まで。それ以上は混雑・混乱の原因
- `.prominent` スタイル（Done / Submit 等のキーアクション用）は **1 つのみ**指定し、trailing edge に置くこと
- カスタム背景やコントロールの塗り色を避け、システム提供の背景エフェクトを妨げないこと。領域区別には `ScrollEdgeEffectStyle` を使う
- ウィンドウ幅の変化でどのアイテムがオーバーフローメニューへ移るかを定義すること。**デフォルトでオーバーフローするレイアウトは禁止**
- オーバーフローメニューを手動で追加しない（iPadOS ではシステムが自動生成する）

### 推奨
- アイテムはテキストより**枠なしの SF Symbols** を優先する（自動で色・ビブランシー・状態表現が適用される）。ただし「Edit」等シンボル化が困難なものはテキスト可
- 配置の役割分担: **leading edge** = 戻る／サイドバー開閉／タイトル／ドキュメントメニュー（カスタマイズ不可・常時表示）、**center** = 汎用コントロール（幅縮小時に自動折りたたみ・カスタマイズ可）、**trailing edge** = 常時表示すべき重要アイテム（検索・インスペクタ・More・プライマリアクション）
- 機能と使用頻度で論理グループ化し、ナビゲーションと Done/Close/Save 等の重要アクションは視覚的に分離した専用セクションに置く
- テキストラベル付きボタンとシンボルボタンを隣接させない。複数テキストボタン間には固定スペースを挿入（`UIBarButtonItem.SystemItem.fixedSpace`）
- 没入体験のためにツールバーを一時非表示にする場合は、復元手段を明確に提供する
- iOS: スペースが限られるため**最重要アイテムのみ**を置き、残りは More メニューへ
- iPadOS: 機能が豊富なアプリではツールバーのカスタマイズ許可を検討する。タブバーとツールバーは同一水平スペースに共存できる

### よくある違反・注意点
- ツールバーアイテムのラベル色と背景コンテンツ色の同系色衝突（鮮やかなコンテンツ上ではモノクローム外観を優先）
- グループ 4 つ以上、`.prominent` の複数指定、枠線付き円形シンボルの使用
- 可能な限り全アクションをツールバー上に出し、More メニューは本当に必要な場合のみにする（重要度の低いものから収容）

公式: https://developer.apple.com/design/human-interface-guidelines/toolbars

## サイドバー（Sidebars）

### 概要
アプリの主要領域やコンテンツコレクション（フォルダ・プレイリスト等）間のトップレベルナビゲーション。iPadOS のレギュラー幅で有効。

### 🔴 必須（遵守事項）
- サイドバー内の階層は**最大 2 階層**まで。3 階層以上の構造には split view（サイドバー → コンテンツリスト → 詳細）を使うこと
- サイドバーをデフォルト非表示にしてはならない（発見可能性を損なう）。ユーザーが表示／非表示を切り替えられるようにする
- サイドバー下のコンテンツが途切れて見えないよう、視覚的に拡張すること（水平スクロールまたは SwiftUI `backgroundExtensionEffect()`）

### 推奨
- iPadOS ではまず**タブバーを第一選択**にし（コンテンツ表示域が広い）、複雑なアプリでのみ `TabViewStyle.sidebarAdaptable` でサイドバー⇔タブバーの自動切替＋ユーザー切替を提供する
- 非 SwiftUI 実装では `UICollectionLayoutListConfiguration.Appearance.sidebar` で正しい外観を適用する
- アイコンは SF Symbols を優先（ビットマップ画像は不可）。色はデフォルトでアプリのアクセントカラー
- 項目が多い場合は開示コントロール（disclosure controls）で階層を畳む
- ユーザーによる内容・順序のカスタマイズを許可する
- グループタイトルは不要語を削り簡潔にする

### よくある違反・注意点
- 3 階層以上をサイドバーに直接詰め込む
- ビットマップのカスタムアイコン使用
- 回転・ウィンドウリサイズへの追従漏れ（`sidebarAdaptable` は自動対応する）

公式: https://developer.apple.com/design/human-interface-guidelines/sidebars

## スプリットビュー（Split Views）

### 概要
アプリ階層の複数レベルを複数ペインで同時表示するレイアウト。プライマリペインでの選択がセカンダリペインに反映され、**最大 3 ペイン**（例: Keynote）まで構成できる。

### 🔴 必須（遵守事項）
- 各ペインの**現在選択を永続的にハイライト**し、ペイン間の対応関係とユーザーの現在位置を明示すること
- iPhone 縦向き等の**コンパクト環境で複数ペインを同時表示しない**こと。split view はレギュラー環境専用（コンパクトでは単一カラムのプッシュ遷移に落とす）

### 推奨
- ペイン間の**ドラッグ＆ドロップ**を許可し、アプリ内のコンテンツ移動を容易にする
- iPadOS: 2 ペイン（例: Mail）／3 ペイン（例: Keynote）構成に対応し、iPad ウィンドウはリサイズ可能なため狭幅・コンパクト・中間幅など**複数の画面幅で成立するか検証**する
- 幅が狭まった際のテキスト折り返し・切り詰めによる可読性低下に注意する

### よくある違反・注意点
- 選択ハイライトの欠落（詳細ペインがどの項目に対応するか不明になる）
- サイドバーで代替できる浅い構造に 3 ペインを使う（過剰設計）
- SwiftUI では `NavigationSplitView` を使う。タブとサイドバーの両立が目的なら `sidebarAdaptable` なタブビューと役割を混同しない

公式: https://developer.apple.com/design/human-interface-guidelines/split-views

## ページコントロール（Page Controls）

### 概要
順序付きページリスト内の位置を示すドット列。現在ページは塗りつぶしドットで表示。iOS / iPadOS で利用可能（macOS 非対応）。

### 🔴 必須（遵守事項）
- ドットは**約 10 個以下**に抑える。10 ページ超はグリッド等の別レイアウトを検討すること
- ビューの**下部中央**に配置すること
- 階層的・非順序なページ関係には使わない（sidebar / split view を使う）

### 推奨
- タップとスクラビング（左右ドラッグ）の両方に対応する。ただし**スクラビング中はアニメーション付きスクロール遷移を使わない**（性能低下と視覚的ちらつきの原因。アニメーション遷移はタップ時のみ）
- 背景スタイルの使い分け: **automatic** = 操作時のみ背景表示（ページコントロールが主ナビゲーションでない場合）／ **prominent** = 常時背景表示（主ナビゲーションの場合）／ **minimal** = 背景なし（位置表示のみ・スクラビング非対応）
- カスタムインジケータ画像はシンプルな形状にし、複雑な形・ネガティブスペース・テキスト・内部線を避ける。色付けはせずシステムの自動着色に任せる

### よくある違反・注意点
- 2 種類以上の異なるインジケータ画像の混在（雑然とし、意味の暗記を強いる）
- スペースに収まらない場合は両端のドットが自動縮小して続きを示唆する（この挙動を独自実装で潰さない）

公式: https://developer.apple.com/design/human-interface-guidelines/page-controls

## 検索フィールド／検索タブ（Search Fields / Search Tab）

### 概要
検索アイコン・クリアボタン・プレースホルダーを備えた編集可能テキストフィールド。iOS ではエントリーポイントとして①タブバー内の検索タブ、②ツールバー（上／下）、③コンテンツ内インラインの 3 パターンがある。

### 🔴 必須（遵守事項）
- プレースホルダーテキストで**何を・どの範囲で検索できるか**を明示すること
- 入力中から検索を開始し、タイプに応じて結果を継続的に絞り込むこと（明示的な検索実行待ちにしない）
- 検索候補（最近の検索・予測候補）を提示すること

### 推奨
- **検索タブの 2 スタイルを使い分ける**（iOS）:
  - **Standard Tab**: タップで検索ランディングページ（上部に検索フィールド＋候補・カテゴリ・発見コンテンツ）へ遷移。探索や提案を促したい場合（例: Apple TV）
  - **Button Appearance**: タップで即座に検索フィールドへフォーカスしキーボード表示。素早く検索して前のタブへ戻る一時的体験（迅速な検索完了向け）
- ツールバー配置は**可能なら下部**（アクセスが容易。例: Settings・Mail・Notes）。下部にスペースがない／コンテンツ優先の場合は上部ツールバーのボタンとして配置（例: Wallet）
- インライン配置はフィールドとコンテンツの関係が強い単一ビュー内フィルタに使い、リスト上部に置いてスクロール時は上部ツールバーへピン留めする（例: Music のライブラリ）
- スコープバーは明確に定義されたカテゴリのフィルタに使い、**デフォルトは広い範囲**にする。トークンは頻出検索語・アイテムでのフィルタに使い、検索候補と組み合わせて使い方を学習させる
- 結果は関連度の高いものを優先し、必要ならカテゴリ分けする
- iPadOS: iPhone / Mac 間で検索体験を可能な限り統一する。多くの標準ケースではツールバー末尾に配置。ナビゲーションのフィルタが目的ならサイドバー上部へ。外付けキーボードのみの環境では、意図しないフォーカス移動を避けるため未フォーカス状態を既定にする

### よくある違反・注意点
- プレースホルダーが「検索」だけで検索対象が不明
- 検索実行までゼロフィードバック（インクリメンタルな絞り込みなし）
- 探索型コンテンツが豊富なのに Button Appearance を選ぶ（または逆）などスタイル選択のミスマッチ

公式: https://developer.apple.com/design/human-interface-guidelines/search-fields

## シート（Sheets）

### 概要
特定情報の入力や簡単なタスク完了に使うプレゼンテーション。iOS / iPadOS ではモーダル・非モーダルの両方が可能。高さは detent で制御する（**large** = 全高、**medium** = 全高の約 50%）。

### 🔴 必須（遵守事項）
- メインインターフェースから同時に表示するシートは **1 つのみ**。シートからさらにシートが必要なら、先に最初のシートを閉じる
- リサイズ可能なシートには**グラバーを必ず表示**する（ドラッグでリサイズ・タップで detent 循環。VoiceOver 利用時のリサイズ手段としても必須）
- **垂直スワイプによる dismiss をサポート**する。未保存の変更がある場合はアクションシートで確認を挟む
- ボタン配置規則: 単一ビューのシートは **Cancel を左（leading）・Done を右（trailing）**。マルチステップフローでは第 1 ステップ = Cancel＋非アクティブ Done、中間ステップ = Back（Cancel と交替）、最終ステップ = Back＋アクティブ Done
- **Done 単独は禁止**（Cancel か Back を必ず対にする）。**Cancel・Done・Back の 3 つ併記も禁止**

### 推奨
- 段階的開示には **medium detent** を活用する（例: 共有シートは最頻出項目を medium で見せ、展開で全体表示）。作成系（Messages / Mail のコンポーザ等）は十分なスペースが要るため large のみでよい
- medium のみ指定すると全高へ展開できなくなる点に注意して detent を選ぶ
- 複雑・長時間のフローにはシートでなくフルスクリーンモーダル等の代替を検討する
- iPadOS: **page または form シートスタイル**を優先する（中央配置＋背景ディム）

### よくある違反・注意点
- シートの多段重ね（シートの上にシート）
- グラバー非表示のままリサイズ可能にする（アクセシビリティ違反）
- スワイプ dismiss で未保存データを黙って破棄する

公式: https://developer.apple.com/design/human-interface-guidelines/sheets

## ポップオーバー（Popovers）

### 概要
特定の UI 要素から矢印付きで出現する一時的コンテナ。少量の情報・機能の一時表示に使う。**iPhone などコンパクト環境では使用しない**。

### 🔴 必須（遵守事項）
- **コンパクト幅（iPhone・iPad の狭い分割）では表示しない**。サイズクラスに応じてシート等の全画面モーダルに切り替えること
- 同時に表示するポップオーバーは **1 つのみ**。カスケード（ポップオーバーから別のポップオーバー）は禁止
- 矢印は**表示元の要素を正確に指す**こと。表示元要素や重要なコンテンツを覆わない
- ポップオーバーの上に他のビューを重ねない（アラートのみ例外）
- 警告・エラー表示に使わない（アラートを使う）

### 推奨
- 機能は少数の関連タスクに限定し、過度に大きくしない
- 閉じるボタンは確認・案内が必要な場合のみ（保存有無の選択等）。非モーダルなポップオーバーが自動で閉じる際は**常に作業内容を保存**する
- サイズ変更はアニメーションで滑らかに行い、別のポップオーバーに置き換わった印象を与えない
- 別のポップオーバーへは可能な限り 1 タップで切り替えられるようにする
- ユーザー向け文言で「ポップオーバー」という用語を使わず、タスクや選択内容で言及する

### よくある違反・注意点
- iPhone でポップオーバーを強行する（UIKit の adaptive presentation を殺す実装）
- 表示元と無関係な位置に矢印が刺さる・矢印なしで浮かせる
- ポップオーバー内にビュー階層を深く作り込む（それは popover でなく専用画面の仕事）

公式: https://developer.apple.com/design/human-interface-guidelines/popovers

## アラート（Alerts）— 文言規則含む

### 概要
重要な情報を即時に伝えるモーダル。タイトル＋任意のメッセージ＋**最大 3 ボタン**で構成され、ユーザーの現在のタスクを中断する。使用は最小限にすること。

### 🔴 必須（遵守事項）— 文言規則
- **タイトル**: 直接的・中立的・親しみやすいトーンで、何が・どの文脈で・なぜ起きたかを簡潔に説明する。**2 行に収める**。完全な文なら文頭大文字＋終止句読点、断片ならタイトルケース＋句読点なし。「Error」「エラー 329347 が発生しました」等の無意味な表現は禁止
- **メッセージ**: タイトルに価値を追加する場合のみ付ける。完全な文で、できる限り短く
- **ボタンラベル**: **1〜2 語の動詞／動詞句**（View All・Reply・Ignore 等）。タイトルケース・句読点なし。ボタンの動作を説明文にしない
- 「OK」は情報通知アラートのみに使う。**「Yes」「No」は禁止**。キャンセルボタンのラベルは必ず「Cancel」
- ボタン配置: デフォルト（選ばれやすい）ボタンは横並びなら右端・縦積みなら最上部。Cancel は横並びなら左端・縦積みなら最下部

### 🔴 必須（遵守事項）— 使用条件
- **取り消し可能で日常的な破壊的操作（例: メール 1 通の削除）に確認アラートを出さない**
- **稀で取り消し不能な破壊的操作にはアラートで確認する**
- 起動時にアラートを出さない（キャッシュ／プレースホルダー＋ラベル表示など、情報を発見できる別手段を設計する）
- 破壊的（赤）スタイルは「意図せず実行すると危険」な操作に限定する（例: Empty Trash はユーザーの意図的操作なので破壊的スタイル不要）

### 推奨
- 単なる情報提供にはアラート以外の手段を使う（例: Mail はオフライン時にタップ可能なインジケータを表示）
- Dynamic Type の大サイズでもスクロールが発生しないよう、タイトルは短く・メッセージは必要時のみ
- アラート内テキストフィールドは可能（入力を伴う確認に使用可）
- 意図的なアクションに続く複数選択肢の提示は、アラートでなく**アクションシート**が適切（例: Mail の下書き破棄確認は 3 択のアクションシート）

### よくある違反・注意点
- 「エラーが発生しました」だけの暗号的アラート
- ボタン 4 つ以上（3 つが上限。超えるならアクションシートへ）
- 確認アラートの乱発による警告疲れ（本当に重要なものだけに絞る）

公式: https://developer.apple.com/design/human-interface-guidelines/alerts

## アクションシート（Action Sheets）

### 概要
ユーザーが開始したアクションに関連する**複数の選択肢**を提示するモーダルビュー。SwiftUI では `confirmationDialog`。iOS / iPadOS で使用可能。

### 🔴 必須（遵守事項）
- **スクロールが発生するボタン数にしない**（スクロールは視認漏れと誤タップの原因）
- タイトルは **1 行**に収まる長さにすること
- データ喪失の可能性がある操作では**キャンセルボタンを必ず含める**（iOS / iPadOS では下部に配置。`confirmationDialog` はデフォルトで含む）
- 破壊的アクションは destructive スタイル（SwiftUI: `ButtonRole.destructive` / UIKit: `UIAlertAction.Style.destructive`）で強調し、**シート上部に配置**する

### 推奨
- アラートとの使い分け: 予期しない重大事象の通知＝アラート、**ユーザー起点のアクションに対する選択肢提示＝アクションシート**
- メッセージは通常不要（タイトル＋操作の文脈で十分な場合は付けない）
- メニューとの使い分け: ボタンタップ等の**操作の結果として自動表示**されるのがアクションシート、ユーザーが**意図的に開く**のがメニュー
- 使用頻度は必要最小限に抑える

### よくある違反・注意点
- 複数行タイトル（切り詰め・スクロールの原因）
- 破壊的ボタンを中程に埋める
- メニューで済む恒常的な選択肢提示にアクションシートを使う

公式: https://developer.apple.com/design/human-interface-guidelines/action-sheets

## アクティビティビュー／共有シート（Activity Views）

### 概要
いわゆる**共有シート（share sheet）**。現在のコンテキストで実行できるタスク（共有・コピー・プリント等）を一覧提示する。iOS / iPadOS ではデバイスと向きに応じてシートまたはポップオーバーとして表示される。

### 🔴 必須（遵守事項）
- 表示は標準の **Share ボタン**（`square.and.arrow.up`）から行うこと（ユーザーはこの導線を期待している）
- **システム既存アクションと重複するカスタムアクティビティを作らない**（Print の再実装等）。カスタム動作が必要な場合は識別可能なタイトルを付ける（例:「Print Transaction」）
- 現在のコンテキストに適したアクティビティのみ表示すること（不適切なシステムタスクは除外可、カスタムタスクは状況で出し分け）

### 推奨
- カスタムアクティビティのタイトルは**単一の動詞または短い動詞句**で簡潔に。会社名・製品名を含めない
- カスタムアクティビティのアイコンは SF Symbols を推奨。カスタムインターフェースアイコンは**約 70x70px 中央配置**
- 共有拡張（Share Extension）は一貫性のためシステム提供の合成ビューを優先。アクション拡張にはアプリ名を含める
- 拡張内の操作は数ステップで完了するよう簡潔にし、拡張の上に追加モーダルを重ねない（アラートのみ可）
- 長時間処理はメインアプリでバックグラウンド継続し、進捗確認手段をメインアプリ側に用意する。完了通知は不要（問題発生時のみ通知）

### よくある違反・注意点
- 独自デザインの「共有」ボタンや別導線で share sheet 相当を再発明する
- どのコンテキストでも全アクティビティを無差別に表示する

公式: https://developer.apple.com/design/human-interface-guidelines/activity-views

## メニュー（Menus）

### 概要
コマンド・オプション・状態を提示する汎用コンポーネント。ラベリングと構成の規則は、コンテキストメニュー・pull-down・pop-up などすべてのメニュー型に適用される。

### 🔴 必須（遵守事項）
- ラベルは明確・簡潔に。アクションは**動詞または動詞句**で始める（View・Close・Select 等）。タイトルケースを使い、冠詞（a / an / the）は削除する
- 追加入力が必要な項目には**省略記号（…）**を付ける
- 実行不可の項目は**メニューから消さずディム表示**する。全項目が実行不可でもメニュー自体は開けるようにする
- 重要・高頻度の項目を先頭に置き、関連項目はセパレータでグループ化する

### 推奨
- サブメニューは控えめに（深さは通常 1 レベル・**5 項目程度まで**）。ネスト項目が全て無効でもサブメニュー自体は有効に保つ。インデント表示でサブメニューを代用しない
- アイコンは目的をもって選び、Share・Print・Search 等の一般アクションには標準シンボルを使う。**同一グループ内では全項目にアイコンを付けるか全く付けないかで統一**する。的確に表現できない場合はアイコンなし
- トグル項目は状態が分かるラベルにする（Show Map ↔ Hide Map、Turn HDR On / Off）。属性の現在値はチェックマークで示し、複数トグル属性には解除オプション（例: Plain）を用意する
- メニューが長すぎる場合は分割かサブメニュー化を検討（History・Bookmarks 等のユーザー生成／動的コンテンツは例外）
- iOS / iPadOS の 3 レイアウト: **small** = 上部 1 行に 4 項目（シンボルのみ）、**medium** = 上部 1 行に 3 項目（シンボル＋短ラベル）、**large**（デフォルト）= 全項目リスト。高頻度の重要アクション 3 つには medium（例: Notes の Scan / Lock / Pin）、関連アクション群（Bold / Italic / Underline / Strikethrough）には small が適する

### よくある違反・注意点
- 「View the Settings」のような冠詞入りラベル
- 無関係なアイコンの装飾的使用、グループ内のアイコン有無の混在
- 多段サブメニューによるコンテンツの深い隠蔽

公式: https://developer.apple.com/design/human-interface-guidelines/menus

## 編集メニュー（Edit Menus）

### 概要
選択中のテキスト／コンテンツに対する編集コマンド（Copy・Paste 等）を提供。iOS ではタッチ＆ホールドまたはダブルタップで水平バー表示（末尾の矢印でコンテキストメニューに展開）、iPadOS ではタッチ操作時は水平・キーボード／ポインタ操作時は垂直コンテキストメニューで表示される。

### 🔴 必須（遵守事項）
- **システム提供の edit menu を優先**し、独自の編集メニュー UI を再実装しない（冗長で混乱の元）
- 表示にはシステム定義のインタラクションをそのまま使う（独自ジェスチャで開かせない）
- 現在のコンテキストに関連するコマンドのみ提示し、適用できないコマンドは削除またはディムする
- iPadOS では**水平（タッチ）・垂直（キーボード／ポインタ）両スタイルに対応**すること

### 推奨
- カスタムコマンドはシステムコマンドの近く・関連グループの後に配置する（例: 書式系はシステムの書式コマンドの後）。ラベルは動詞または短い動詞句
- **編集不可のテキストも選択・コピーを許可**する（キャプションやステータスをメッセージ・検索に貼り付けられるように）
- 可能な限り Undo / Redo をサポートする
- **Delete と Cut を区別**する（Delete = 削除のみ、Cut = ペーストボードへコピー後に削除）
- メニューが重要なコンテンツや UI を覆う場合は表示位置を調整する
- 標準コマンドは `UIResponderStandardEditActions` を参照。システムはデータ検出により関連アクションを自動追加できる（例: 住所選択時の「Get directions」）

### よくある違反・注意点
- edit menu と同じ機能のボタンを画面上に重複配置してインターフェースを混雑させる
- 選択コピーを不許可にした読み取り専用テキスト

公式: https://developer.apple.com/design/human-interface-guidelines/edit-menus

## プルダウンボタン（Pull-Down Buttons）

### 概要
ボタンの目的に直接関連する**コマンド／アイテムのメニュー**を表示するボタン。項目を選ぶとメニューが閉じ、アプリがそのアクションを実行する。ボタン自身の表示は変化しない。

### 🔴 必須（遵守事項）
- **メニュー項目は 3 つ以上**あることを目安にする。1〜2 項目ならボタン・トグル・スイッチ等の代替を使うこと
- **ビューの全アクションを 1 つの pull-down button に隠さない**。プライマリアクションは直接発見可能にする
- 破壊的アクション（赤表示）を選んだ場合は、iOS ではアクションシート・iPadOS ではポップオーバーで**意図確認**を挟む（表示位置を変え意識的操作を要求することで誤削除を防ぐ）

### 推奨
- 用途例: 「追加」ボタン → 追加するアイテム種別の選択、「ソート」ボタン → ソート属性の選択、「戻る」ボタン → 戻り先の選択
- メニュータイトルは意味を追加する場合のみ付ける（通常はボタン＋説明的な項目ラベルで文脈は十分）
- アイコンは価値がある場合のみ、SF Symbols で統一する
- **More ボタン（…）**はスペース制約で置けない項目の受け皿として有効だが発見可能性を下げるため、省スペースと発見性のバランスを取る
- iOS 14 以降はタッチ＆ホールドでのメニュー表示にも対応できる（例: Safari のタブボタン長押し）

### 使い分け（pop-up button との対比）
| 選択するもの | 使うコンポーネント |
|---|---|
| コマンド・関連アクション（実行して閉じる） | pull-down button |
| 相互排他的な選択肢・状態（選択がボタン表示に反映） | pop-up button |

### よくある違反・注意点
- 項目過多で目的の項目が探しにくいメニュー
- 破壊的アクションの即時実行（確認なし）

公式: https://developer.apple.com/design/human-interface-guidelines/pull-down-buttons

## ポップアップボタン（Pop-Up Buttons）

### 概要
**相互排他的な選択肢のフラットなリスト**からひとつを選ばせるボタン。選択するとメニューが閉じ、**ボタンの表示が現在の選択内容に更新される**。限られたスペースで多数の選択肢を提示するのに効率的。

### 🔴 必須（遵守事項）
- コマンド（アクション実行）には使わない。アクション・複数選択・サブメニューが必要な場合は **pull-down button を使う**こと
- **有用なデフォルト選択を必ず提供**する（未選択状態を作らない。多くのユーザーが望む項目をデフォルトに）

### 推奨
- 導入ラベルまたはボタンラベルで「この選択が何に影響するか」を予測可能にする
- まれにしか使わない値のために選択肢を増やさず、**Custom オプション**＋リスト下の説明テキストで対応する
- iPadOS: ポップオーバーやモーダルビュー内のリスト項目では、詳細ビューへ遷移させる開示インジケータ（>）の代わりに pop-up button を検討する（選択肢が少なく明確な場合、その場で選べて効率的）

### よくある違反・注意点
- 選択結果がボタン表示に反映されない実装（pull-down との混同）
- 排他的でない選択肢や複数選択を pop-up に押し込む

公式: https://developer.apple.com/design/human-interface-guidelines/pop-up-buttons
