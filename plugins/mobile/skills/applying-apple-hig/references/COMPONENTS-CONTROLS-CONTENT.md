# コントロール・コンテンツ系コンポーネント（HIG Components: Controls & Content）

> 出典: Apple Human Interface Guidelines / Apple Developer Documentation（最終確認 2026-07-13）。仕様は更新されるため、実装時は必ず併記の公式 URL で最新版を確認すること。日本語版がある場合は公式の言語切替または本文の日本語 URL を使い、英語版との差分がある場合は最新側を確認する。
> 判定区分: 本文の「必須」「違反」は作業用の強度ラベルであり、HIG の推奨を一律に App Review 必須へ格上げしない。審査・API 上の必須判定は現行の Apple 公式一次資料で再確認する。

iPhone/iPad（iOS/iPadOS）アプリの入力コントロールとコンテンツ表示コンポーネントの実装指針。全コントロール共通の大原則:

- タップ可能要素は **44x44pt を推奨デフォルト**とし、ボタンのヒット領域も一般則として 44x44pt 以上を目指す。Apple の Accessibility 資料が示す最小 28x28pt との区別を保ち、44x44pt を全要素の絶対最小値とは扱わない。
- 🔴 状態の区別を**色だけで表現してはならない**（形状・チェックマーク・塗りを併用）。
- 🔴 テキストを表示するコントロールは **Dynamic Type** に対応すること（システムフォントなら自動）。

## ボタン（Buttons）

ユーザーが即時アクションを実行するための基本コントロール。Style（視覚スタイル）・Content（シンボル/ラベル）・Role（意味論）の 3 要素で構成する。

### 🔴 必須

| 項目 | 規定 |
|------|------|
| ヒット領域 | 一般則として 44x44pt 以上を目指す。小さくする場合は標準コンポーネント、利用頻度、間隔、拡張ヒット領域、実機での誤操作を合わせて検証する |
| プレス状態 | カスタムボタンには必ず押下状態の視覚変化を実装すること（無いと無反応に見える） |
| Role の割当 | 破壊的操作に primary ロールを割り当ててはならない。データを失う操作は destructive ロール（システムレッド表示）を使うこと |
| 非同期アクション | 即座に完了しないアクションでは activity indicator をボタン内に表示し、ラベルを進行形に変える（例: "Checkout" → "Checking out…"） |

### Role（システムロール）

| Role | 効果 | 用途 |
|------|------|------|
| Normal | デフォルト外観 | 標準アクション |
| Primary | アクセントカラー背景。Return キー応答、シート/アラートの既定ボタン | 最有力の非破壊アクション |
| Cancel | 現在の操作の取り消し | キャンセル |
| Destructive | システムレッド | 削除・リセット等 |

### 推奨

- 優先度は**サイズではなくスタイルで表現**する。プロミネント（塗りつぶし）ボタンは 1 ビューあたり 1〜2 個まで。
- アイコンは `square.and.arrow.up`（共有）等、既知の SF Symbols を優先する。テキストラベルは動詞で始める（例: "Add to Cart"）。
- ボタン周囲に十分な余白を取り、背景と区別できるようにする。

### よくある違反・注意点

- 同格の選択肢をサイズ差で表現する（→ 同サイズで並べる）。ボタンラベルと背景を同系色にする。プレス状態のないカスタムボタン。

公式: https://developer.apple.com/design/human-interface-guidelines/buttons

## トグル（Toggles / Switches）

オン/オフなど対立する 2 状態の切り替え。

### 🔴 必須

- 2 つの**対立する状態の切り替えのみ**に使うこと。3 択以上はメニュー/ピッカーを使う。
- 影響を受ける設定・コンテンツが何かを明確に示すこと。
- 状態の区別に**色以外の手掛かり**（背景形状の有無・チェックマーク等）を必ず併用すること（色覚特性への対応）。

### 推奨（iOS/iPadOS）

- スイッチ（switch toggle）は**リスト行の中でのみ**使う。リスト行内ではスイッチ自体にラベルは不要（行のテキストが文脈を与える）。
- リスト外でトグル動作が必要な場合はスイッチではなく**トグル動作のボタン**（例: 電話アプリのフィルタボタン。アクティブ時に背景ハイライト）を使う。
- 色はデフォルトの緑を推奨。変更する場合はアプリのアクセントカラーとし、コントラストを確認する。

### よくある違反・注意点

- 複数選択肢の切り替えにスイッチを乱用する。リスト外に裸のスイッチを置き目的が不明になる。

公式: https://developer.apple.com/design/human-interface-guidelines/toggles

## スライダー（Sliders）

最小値〜最大値の連続範囲から単一値を選択する水平トラック。

### 🔴 必須

- 水平スライダーは**左端が最小値・右端が最大値**。垂直なら下が最小・上が最大。アプリ全体で一貫させること。
- 🔴 **音量調整にスライダーを使ってはならない**。iOS では音量スライダー＋出力デバイス切替を備えた **Volume View**（`MPVolumeView` / AVKit の相当 API）を使うこと。

### 推奨

- トラック色・サム（つまみ）・左右アイコンをカスタマイズして意味を伝える（例: 画像サイズなら左に小・右に大の画像アイコン）。
- 値の範囲が広い場合は**テキストフィールド＋ステッパーを併設**し、正確な値の入力/確認を可能にする。
- 値変更に対して**ライブフィードバック**（結果のリアルタイム反映）を返す。

### よくある違反・注意点

- 方向の不統一（画面ごとに最小/最大の向きが変わる）。離散的な少数の選択肢にスライダーを使う（→ segmented control / stepper）。

公式: https://developer.apple.com/design/human-interface-guidelines/sliders

## ステッパー（Steppers）

値を 1 段階ずつ増減する 2 セグメントのコントロール（`UIStepper`）。

### 🔴 必須

- ステッパー自体は値を表示しない。**隣接するラベル/フィールドに現在値を必ず表示**すること。
- どの値に影響するかを明確に示すこと。

### 推奨

- **小さな変更**（数タップで済む範囲）に限定する。大きく変わりうる値はステッパー＋テキストフィールドを併設する（例: 印刷部数）。

### よくある違反・注意点

- 広い値域をステッパーのみで操作させる（タップ回数が爆発する）。watchOS/tvOS は非対応。

公式: https://developer.apple.com/design/human-interface-guidelines/steppers

## ピッカー（Pickers）

スクロール可能な値リストから選択させるコントロール。

### 🔴 必須

- 値は**予測可能で論理的な順序**（アルファベット順等）で並べること。ユーザーが画面外の値を予測できるようにする。
- ピッカーは**編集対象の文脈内**（編集フィールドの近く・画面下部・popover 内）に表示すること。選択のために別ビューへ遷移させない。

### 推奨

- リスト長で使い分ける: 中〜長いリスト → picker / 短いリスト → pull-down button / 非常に大きな集合 → インデックス付き list/table。
- 分の粒度はデフォルト 0〜59（60 値）。60 を割り切れる間隔（例: 15 分刻み = 0, 15, 30, 45）に変更可能。

### よくある違反・注意点

- 2〜3 個の選択肢にホイールピッカーを使う。表示される値と順序がデバイスの言語・地域設定に依存する点を考慮しない。

公式: https://developer.apple.com/design/human-interface-guidelines/pickers

## 日付ピッカー（Date Pickers）

日付・時刻・カウントダウン時間の選択（HIG では Pickers ページ内で規定。`UIDatePicker` / SwiftUI `DatePicker`）。

### スタイル（iOS/iPadOS・4 種）

| スタイル | 挙動 | 使い所 |
|---------|------|--------|
| Automatic | プラットフォームとモードからシステムが選択 | 既定 |
| Compact | ボタン表示→タップでモーダルのカレンダー/時刻ピッカーを展開 | スペースが限られる場面（推奨） |
| Inline | 時刻のみ: ホイール / 日付+時刻: インラインカレンダー | 常時表示したい場面 |
| Wheels | スクロールホイール＋キーボード入力対応 | 従来型 UI |

### モード（4 種）

| モード | 表示 |
|--------|------|
| Date | 月・日・年 |
| Time | 時・分（AM/PM は設定次第） |
| Date and time | 日付＋時・分 |
| Countdown timer | 時・分（最大 23 時間 59 分）。🔴 inline / compact スタイルでは使用不可 |

### 🔴 必須・注意

- Compact スタイルはボタンにアクセントカラーで現在値を表示し、モーダル内で**確定前に複数回編集できる**（ビュー外タップで確定）。この編集フローを妨げないこと。
- 表示形式（年月日の順序・12/24 時間制）はロケール依存。ハードコードしてはならない。

公式: https://developer.apple.com/design/human-interface-guidelines/pickers

## セグメンテッドコントロール（Segmented Controls）

2 つ以上のセグメントが並ぶ線形コントロール。各セグメントはボタンとして機能する。

### 🔴 必須

- 1 つのコントロール内で**テキストとアイコンを混在させない**（どちらかに統一）。
- **選択状態を示すセグメントとアクション実行セグメントを混在させない**。
- アプリの独立した領域間の切り替えに使ってはならない（→ Tab Bar を使う）。密接に関連するサブビューの切り替え（例: カレンダーの「イベント/リマインダー」）に限定する。

### 推奨

- セグメント数: **iPhone では約 5 個以下**、幅の広い画面でも 5〜7 個程度まで。
- 全セグメントは等幅になるため、コンテンツサイズを揃える。
- ラベルは名詞または名詞句、タイトルケース。

### よくある違反・注意点

- 画面切替タブの代用にする（Tab Bar の責務）。セグメントが多すぎて各ラベルが切り詰められる。

公式: https://developer.apple.com/design/human-interface-guidelines/segmented-controls

## テキストフィールド（Text Fields）

名前・メールアドレス等、1 行の短いテキストの入力・編集領域。

### 🔴 必須

| 項目 | 規定 |
|------|------|
| 用途 | 少量の特定情報のみ。長文には text view を使うこと |
| ラベル | プレースホルダーは入力開始で消えるため、**独立したラベルを必ず併置**すること |
| 機密情報 | パスワード等は必ず **SecureField / secure text entry** を使うこと |
| キーボード | 入力内容に応じた**キーボードタイプ**（数値・メール・URL 等）を指定すること |

### 推奨（iOS/iPadOS）

- **Clear ボタン**をトレーリング端に表示する（削除キー連打を不要にする）。
- リーディング端にフィールドの目的を示す画像、トレーリング端に補助機能（ブックマーク等）のシステムボタンを置ける。
- 複数フィールドは**垂直に積み**、関連が分かる間隔・同カテゴリは同一幅にする。
- フィールドサイズは想定入力量の目安になるよう設定する。
- バリデーション: メールアドレスは別フィールドへの移動時、ユーザー名/パスワードはフィールドを離れる前に検証。数値専用フィールドは他の文字入力時に警告する。
- 数値はフォーマッター（小数桁・%・通貨）を使い、**ロケールにより表示が変わる**ことを想定する。

### よくある違反・注意点

- プレースホルダーだけで目的を説明する（入力後に手掛かりが消える）。長文入力をテキストフィールドで受ける。

公式: https://developer.apple.com/design/human-interface-guidelines/text-fields

## テキストビュー（Text Views）

複数行のスタイル付きテキストの表示（および任意で編集）。

### 🔴 必須

- 長文・編集可能テキスト・特殊フォーマットが必要な場合に使う。**短い表示のみなら label、短い編集なら text field** を使うこと。
- Dynamic Type に対応し、太字テキスト等のアクセシビリティ設定でも表示を検証すること。
- 編集可能な場合、入力内容に適したキーボードタイプを表示すること。

### 推奨

- 複数フォント・色・配置を使う場合も可読性を最優先する。
- エラーメッセージ・シリアル番号・IP アドレス等の有用な情報は**選択・コピー可能**にする。
- デフォルト挙動: 任意の高さ＋スクロール、コンテンツは先頭エッジ寄せ、システム label カラー使用。編集可能なら選択時にキーボードが出る。

公式: https://developer.apple.com/design/human-interface-guidelines/text-views

## コンボボックス（Combo Boxes）※iOS/iPadOS 非対応

テキストフィールドとプルダウンボタンを組み合わせた macOS 専用コントロール（`NSComboBox`）。

### 🔴 必須（該当時）

- **iOS/iPadOS では使用できない**。同等の「候補から選ぶ or 自由入力」が必要な場合は、テキストフィールド＋サジェスト表示、あるいはメニュー/ピッカー＋「カスタム…」項目で代替すること。
- Mac Catalyst 等で macOS 向けに使う場合: 意味のあるデフォルト値を設定し、リスト項目の幅をフィールド幅以内に収め、ラベルはタイトルケース＋コロン終端にする。カスタム入力値はリストに追加されない仕様を前提にする。

公式: https://developer.apple.com/design/human-interface-guidelines/combo-boxes

## 進捗インジケータ（Progress Indicators）

処理の進行を伝える一時的な表示。**determinate**（残り時間が判る: progress bar / 円形進捗）と **indeterminate**（判らない: activity indicator = スピナー）の 2 系統。

### 🔴 必須

| 項目 | 規定 |
|------|------|
| タイプ選択 | 可能な限り determinate を使うこと。進捗報告は正確に（90% までが 5 秒・残り 10% が 5 分のような不均等はユーザーの信頼を損なう） |
| 動きの継続 | インジケータを止めてはならない（停止＝フリーズと誤認される） |
| 切り替え | indeterminate → determinate への切り替えは可。**逆方向・circular ↔ bar のスタイル切替は禁止** |
| スピナーのラベル | activity indicator に説明ラベルを付けない。付ける場合も "loading" / "authenticating" のような無情報の文言は避ける |
| 一時性 | 操作中のみ表示し、完了したら消すこと |

### 推奨

- キャンセル手段を提供する。中断で進捗が失われる場合は確認アラートを出し、悪影響がある処理には Pause も検討する。配置はアプリ内・プラットフォーム間で一貫させる。

### iOS/iPadOS 固有: Refresh Control

- スクロールビュー/テーブルの引き下げで出る indeterminate の特化型（`UIRefreshControl`）。
- 🔴 refresh control があっても**自動更新は定期的に行うこと**（ユーザーに毎回手動更新を強いない）。
- タイトルは任意。付けるなら操作説明ではなく「最終更新時刻」等の有用情報にする。

公式: https://developer.apple.com/design/human-interface-guidelines/progress-indicators

## ラベル（Labels）

読める・コピーできるが編集できない静的テキスト。

### 🔴 必須

- 編集不可の少量テキストにのみ使う（編集可 → text field、長文 → text view）。
- システムフォントを推奨（Dynamic Type 自動対応）。カスタムフォントを使う場合はサイズ変更後の可読性を自ら保証すること。
- 重要度はシステム提供のセマンティックカラーで表現すること:

| カラー | 用途 | UIKit API |
|--------|------|-----------|
| Label | 主要情報 | `UIColor.label` |
| Secondary | 副次情報・サブタイトル | `UIColor.secondaryLabel` |
| Tertiary | 無効項目の説明等 | `UIColor.tertiaryLabel` |
| Quaternary | 透かし程度のテキスト | `UIColor.quaternaryLabel` |

### 推奨

- システムカラーはライト/ダークモードに自動追従する。カスタムカラーは両モードでコントラスト比を確認する。エラーメッセージ・IP アドレス等の有用テキストは選択・コピー可能にする。

公式: https://developer.apple.com/design/human-interface-guidelines/labels

## リストとテーブル（Lists and Tables）

行（と列）でデータを表示し、選択・追加・削除・並べ替えをサポートする。

### 🔴 必須（iOS/iPadOS）

- 行の選択（複数選択等の編集操作）を伴う場合、ユーザーは**編集モードに入ってから**項目を選択する。編集モードと並べ替えをサポートすること。
- **info ボタンをナビゲーションに使ってはならない**。階層のドリルダウンには開示インジケータ（chevron）、info ボタンは詳細情報の表示のみ。
- トレーリング側に開示インジケータがある行のテーブルに**アルファベットインデックスを併用しない**（同じ側で操作が競合する）。

### 推奨

- リスト/テーブルは主にテキスト向き。画像主体・可変サイズのアイテムは collection を検討する。
- 行テキストは簡潔にし、切り詰め・折り返しを最小化。複雑な情報は詳細ビューへ。幅が狭い場合は中央省略（…）で先頭と末尾を保持する。
- ナビゲーション行は遷移中ハイライトを維持、選択肢リストは一時ハインライト後にチェックマーク等のインジケータを表示する。
- grouped スタイルではヘッダー/フッターで文脈を与える。
- 長いリストは全件を並べず、関連度の高い項目＋「さらに表示」の形を検討する。

### よくある違反・注意点

- chevron と info ボタンの役割混同。1 行に操作要素を詰め込みすぎてタップ精度が落ちる。

公式: https://developer.apple.com/design/human-interface-guidelines/lists-and-tables

## コレクション（Collections）

順序付きコンテンツをグリッド等の視覚的レイアウトで表示（`UICollectionView`）。**画像ベースのコンテンツに最適**。

### 🔴 必須

- ユーザーが閲覧・操作している最中に**レイアウトを動的に変更してはならない**。変更は明示的なユーザーアクションへの応答としてのみ行い、追跡可能にすること。

### 推奨

- 標準の水平行/グリッドレイアウトを優先し、独自レイアウトで混乱させない。テキスト中心の情報は collection ではなく list/table の方が読みやすい。
- アイテム周囲に十分なパディングを取り、選択しやすくする（重なり禁止）。
- デフォルトジェスチャー（タップ=選択、長押し=編集、スワイプ=スクロール）を尊重し、必要な場合のみ追加ジェスチャーを実装する。挿入・削除・並べ替えにはアニメーションを付ける。

公式: https://developer.apple.com/design/human-interface-guidelines/collections

## スクロールビュー（Scroll Views）

ビュー境界を超えるコンテンツの垂直/水平移動。

### 🔴 必須

- システム標準のスクロール挙動（エラスティック等）に従うこと。カスタム実装でも期待される挙動を再現する。
- **同一方向のスクロールビューをネストしてはならない**（垂直 in 垂直は操作不能になる）。直交方向のネスト（水平 in 垂直）は可。
- ページング＋page control を使う場合、**同軸のスクロールインジケータを表示しない**（コントロールの重複）。

### 推奨

- スクロールインジケータは常時表示されないため、**端に部分的なコンテンツを見せて**スクロール可能性を示唆する。
- ページング（固定量スクロール）が内容に合うか検討する（SwiftUI `PagingScrollTargetBehavior`）。ページ間で数行分のオーバーラップを定義すると文脈が保たれる。
- 自動スクロール（検索一致の表示・入力開始時の挿入位置表示・選択ドラッグ追従）は、**文脈維持に必要な最小限だけ**動かす。
- ズームをサポートする場合は最大・最小スケールを適切に制限する（1 文字が画面いっぱいになるようなズームは不合理）。
- スクロールエッジエフェクトはフローティングのツールバー等とコンテンツの視覚的分離に使う。既定の automatic スタイルを優先し、**装飾目的で使わない**。1 ビュー 1 エフェクト。

公式: https://developer.apple.com/design/human-interface-guidelines/scroll-views

## チャート（Charts）

データから洞察を得させる可視化（Swift Charts）。構成要素は mark（bar/line/point）・plot area・axis・tick・grid line・label・legend。

### 🔴 必須

- **色だけでデータ系列を区別しない**。形状（円形/ダイヤ等）や塗りを併用する（例: 血圧チャートの収縮期/拡張期）。
- 小さな mark のためにヒット領域を**プロット領域全体へ拡張**すること（最小ターゲットの原則・運動機能への配慮）。
- 説明文・アクセシビリティラベルは客観値で書く（「急速に増加」ではなく「60 から 90 に変化」）。見た目（「赤い線が上昇」）ではなくデータの意味（「売上が増加」）を述べる。
- 曖昧なフォーマットを避ける: "6/6" → "June 6"、"60m" → "60 minutes"。

### 推奨

- mark の選択: bar=カテゴリ比較/合計、line=時系列トレンド、point=個別値・外れ値、複合=トレンド＋個別値。
- 視覚的優先度は「データ > 軸・説明 > 背景要素」。グリッドラインは多すぎず少なすぎず。tick は 0, 5, 10… のような一般的な等差にする（1, 6, 11… は読解に時間がかかる）。
- 軸範囲は用途で選ぶ: 固定（例: バッテリー 0〜100%）/ 動的（例: 歩数）。コンパクト表示ではプロット領域を最大化し、単位はタイトルへ移す。
- 重要情報はインタラクション無しで見え、詳細はドラッグ等で探索できる設計にする。タイトル/サブタイトルでメインメッセージを要約する（例: 「次の 1 時間の降水予報」）。

### アクセシビリティ（必須級）

- Swift Charts は mark ごとにアクセシビリティ要素と **Audio Graphs** を自動生成する。チャートタイトルと説明的サマリーを必ず提供すること。
- アクセシビリティラベルは「日付＋文脈＋値」を含める（"60" ではなく "June 6 の歩数は 8,543 歩"）。データ密度が高い場合は区間ごとのサマリー、極小表示では概要のみ＋詳細へのドリルダウン。
- データ変化はアニメーションに加えて `UIAccessibility.Notification` 等の代替手段でも通知する（VoiceOver・アニメーション無効ユーザー対応）。
- Full Keyboard Access / Switch Control 向けに X 軸沿い等の論理的なナビゲーション順序を定義する。

公式: https://developer.apple.com/design/human-interface-guidelines/charts

## ゲージ（Gauges）

範囲内の現在値を円形/線形パスで示すコンポーネント（SwiftUI `Gauge`）。標準スタイル（インジケータで位置表示）と容量スタイル（フィルで表示）がある。

### 🔴 必須

- 現在値と範囲の両端を伝える**簡潔なラベル**を付けること（VoiceOver が読み上げる。省略するとスクリーンリーダー利用者が値を理解できない）。

### 推奨

- グラデーションフィルで意味を補強する（例: 温度ゲージの青→赤）。
- iOS のロック画面ウィジェットには watchOS コンプリケーション風の **accessory バリアント**が適する。tvOS は非対応。

公式: https://developer.apple.com/design/human-interface-guidelines/gauges

## イメージビュー（Image Views）

単一画像（またはアニメーションシーケンス）を透明/不透明背景上に表示（`UIImageView` / SwiftUI `Image`）。PNG・JPEG・PDF 等に対応。

### 🔴 必須

- イメージビューは**表示専用**。タップ操作が必要ならボタンにすること（画像をボタンのコンテンツにする）。
- アニメーション画像は全フレームで**同一サイズ**にすること。

### 推奨

- アイコンには SF Symbols またはベクターベースのインターフェースアイコンを優先する（色・不透明度を自在にレンダリング可能、テンプレート画像はアクセントカラーに追従）。
- 画像上にテキストを重ねる場合はシャドウや背景レイヤーでコントラストを確保する。表示サイズへの事前スケーリングでシステムのスケーリング処理を減らしパフォーマンスを確保する。

公式: https://developer.apple.com/design/human-interface-guidelines/image-views

## ウェブビュー（Web Views）

アプリ内で HTML/Web コンテンツを直接表示する（`WKWebView`）。iOS/iPadOS/macOS/visionOS 対応。

### 🔴 必須

- ユーザーが複数ページを訪問しうる場合は**進む/戻るナビゲーションを有効化し、対応するコントロールを提供**すること（デフォルトでは無効）。
- **Web ブラウザを再構築してはならない**。汎用ブラウジングは Safari の責務。web view はアプリの文脈を離れずに一時的に Web コンテンツへアクセスさせる用途に限定する。

### よくある違反・注意点

- URL バー・タブ等を備えた「ミニブラウザ」化。記事表示に戻る手段が無く、ユーザーが袋小路に陥る。

公式: https://developer.apple.com/design/human-interface-guidelines/web-views
