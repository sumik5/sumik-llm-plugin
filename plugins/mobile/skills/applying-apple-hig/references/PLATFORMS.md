# iOS / iPadOS プラットフォーム特性とデザイン指針

> 出典: Apple Human Interface Guidelines / Apple Developer Documentation（最終確認 2026-07-13）。仕様は更新されるため、実装時は必ず併記の公式 URL で最新版を確認すること。日本語版がある場合は公式の言語切替または本文の日本語 URL を使い、英語版との差分がある場合は最新側を確認する。
> 判定区分: 本文の「必須」「違反」は作業用の強度ラベルであり、HIG の推奨を一律に App Review 必須へ格上げしない。審査・API 上の必須判定は現行の Apple 公式一次資料で再確認する。

本ファイルは iPhone（iOS）/ iPad（iPadOS）アプリの設計・実装判断に直結するプラットフォーム特性を扱う。対象は iOS / iPadOS のみ（macOS / watchOS / tvOS / visionOS 専用の内容は含めない）。

## iOS 向けデザイン（Designing for iOS）

### 概要

iPhone は「中程度サイズの高解像度ディスプレイ」「片手または両手での保持」「Multi-Touch 中心の入力」を前提とするプラットフォーム。利用は 1〜2 分の短時間タスク（通知確認・メッセージ送信・データ記録）と 1 時間超の長時間タスク（ブラウジング・ゲーム・メディア視聴）の両極端が混在し、ユーザーは頻繁にアプリを切り替える。この二面性に耐える「即座に価値を出し、中断・復帰に強い」設計が求められる。

### デバイス特性・エルゴノミクス

| 特性 | 内容 | 実装上の帰結 |
|------|------|-------------|
| ディスプレイ | 中程度サイズ・高解像度 | 情報を詰め込みすぎず、1 画面 1 主タスクに絞ること |
| 視距離 | 約 30〜60cm（1〜2 フィート） | この距離で判読できるテキストサイズ・コントラストを確保すること |
| 保持姿勢 | 片手 or 両手。縦向き・横向き両対応 | 親指の可動域（画面中央〜下部）に主要操作を置くこと |
| 利用時間 | 数十秒〜数分の断片利用が多い | 起動直後に主コンテンツへ到達させ、状態を保存・復元すること |

### 入力手段

iPhone アプリが想定すべき入力チャネル:

1. **Multi-Touch ジェスチャー** — 主入力。タップ・スワイプ・ピンチ・長押し
2. **仮想キーボード** — テキスト入力。表示時に画面下半分が占有される点に注意
3. **音声（Siri / 音声入力）** — ハンズフリー操作・Shortcuts 連携
4. **ジャイロスコープ・加速度計** — 傾き・動きによるインタラクション
5. **空間インタラクション**（近接デバイス連携）
6. **個人データへのアクセス**（位置情報・ヘルスケア等） — 必ず明示的なユーザー許可を得ること

### システム機能

アプリ単体で完結させず、以下の iOS システム機能に統合することでユーザーの導線に乗ること:

| 機能 | 用途 |
|------|------|
| Widgets | ホーム画面 / ロック画面で最新情報を一目で提供 |
| Home Screen quick actions | アイコン長押しから頻出タスクへ直行 |
| Spotlight | アプリ内コンテンツをシステム検索に露出 |
| Shortcuts / App Intents | ユーザー定義の自動化・Siri からの呼び出し |
| Activity views（シェアシート） | コンテンツ共有・他アプリ連携 |

### 🔴 必須（遵守事項）

- **画面上のコントロールを最小限に絞ること。** 主タスクのコンテンツに集中させ、副次的な操作・詳細は最小限のタップで発見できる場所（コンテキストメニュー・スワイプアクション・詳細画面）に退避すること。
- **外観変化に動的適応すること。** 以下の 3 点は必須対応:
  - デバイスの向き（Portrait / Landscape）の変更
  - ダークモード（Light / Dark 両対応）
  - Dynamic Type（ユーザーが選んだテキストサイズに追従。固定 pt 決め打ちで切り捨てないこと）
- **主要なインタラクティブ要素は画面の中央〜下部に配置すること。** 上部は片手保持時に親指が届きにくい。ナビゲーションバー上部にしか置けない操作は頻用アクションにしないこと。
- **44x44pt を快適なタップ領域の推奨デフォルト**とし、ボタンのヒット領域も一般則として 44x44pt 以上を目指す。全要素の絶対最小値や一律の App Review 要件とは扱わず、利用頻度・間隔・実際のヒット領域を合わせて検証する。
- **スワイプジェスチャーをサポートすること。** 画面端からのスワイプで「戻る」、リスト行の左右スワイプでアクション実行、が標準期待動作。標準ジェスチャーを別の意味に上書きしてはならない。
- **個人データ・デバイス機能（位置情報・カメラ・生体認証・決済等）はユーザーの明示的許可を得てから使うこと。** 許可要求は必要になった文脈で、理由を添えて行う。

### 推奨

- 短時間利用を前提に、起動〜価値提供までのステップ数を最小化する（オンボーディングの強制ウィザードを避ける）。
- Apple Pay・Face ID / Touch ID・位置ベース機能など、プラットフォーム標準の機能を独自実装より優先して採用する（ユーザーの学習コストゼロ・信頼性が高い）。
- 中断からの復帰時に直前の状態（スクロール位置・入力途中のテキスト・再生位置）を復元する。
- 縦向きを基本としつつ、横向きでもレイアウトが破綻しないこと（メディア系は横向き最適化も検討）。

### よくある違反・注意点

- ❌ ツールバー・画面上部にアイコンボタンを敷き詰める（コントロール過多。iOS は「少数精鋭 + 発見可能な二次アクション」が原則）
- ❌ Dynamic Type 非対応（最大アクセシビリティサイズでテキストが見切れる / レイアウト崩壊）
- ❌ ダークモードで固定色（白背景ハードコード）のまま表示する — semantic colors（`label`, `systemBackground` 等）を使うこと
- ❌ 独自の「戻る」UI だけを提供し、エッジスワイプの戻りを無効化する
- ❌ 起動直後にログイン・通知許可・トラッキング許可を連発する（許可疲れで拒否率が上がる）
- ❌ 小さい「×」ボタンやセグメントを密集させ、間隔・ヒット領域・誤タップを実機で検証しない

公式: https://developer.apple.com/design/human-interface-guidelines/designing-for-ios

## iPadOS 向けデザイン（Designing for iPadOS）

### 概要

iPad は「大型高解像度ディスプレイ」と「多様な入力手段の組み合わせ」が特徴。手持ち・机置き・スタンド設置と保持形態が多様で、視距離は iPhone より遠い（約 90cm / 3 フィート以内）。短時間タスクから、ゲーム・メディア・コンテンツ制作・生産性ワークまで長時間の没入利用が広く、複数アプリの同時運用（マルチタスキング・drag and drop）が日常的に行われる。「iPhone アプリの拡大版」ではなく、大画面と多入力を活かした再設計が求められる。

### デバイス特性・エルゴノミクス

| 特性 | 内容 | 実装上の帰結 |
|------|------|-------------|
| ディスプレイ | 大型・高解像度 | コンテンツを主役に。余った領域をサイドバー・複数カラムで有効活用すること |
| 視距離 | 約 90cm（3 フィート）以内 | iPhone より遠くから見られる前提でコンテンツサイズを決めること |
| 保持形態 | 手持ち / 机置き / スタンド | どの姿勢でも操作可能な配置（上端固定の頻用操作を避ける等）にすること |
| 利用形態 | 複数アプリ・複数ウインドウの同時運用 | 任意サイズへの適応、状態復元、drag and drop を対象範囲に応じて設計する |

### 入力手段（組み合わせ対応が必須）

iPad ではユーザーが以下を**同時に・切り替えながら**使う。単一入力前提の設計は不可。

1. **Multi-Touch ジェスチャー** — 大きめのタッチターゲットが必要
2. **仮想キーボード**
3. **外付け（接続型）キーボード** — キーボードショートカットの提供を検討すること
4. **ポインティングデバイス（トラックパッド / マウス）** — ホバー状態・ポインタ形状の適応
5. **Apple Pencil** — 描画・手書き・精密選択という Pencil 固有のインタラクション
6. **Siri（音声）**

入力手段によって最適なコンテンツ密度は変わる: タッチ主体なら大きなターゲット、キーボード + トラックパッド併用ならより高密度・精密な操作を許容できる。

### システム機能

| 機能 | 用途 |
|------|------|
| Multitasking | 複数アプリ / 複数ウィンドウの同時表示（詳細は次章） |
| Widgets | アプリ外からの情報アクセス |
| Drag and drop | アプリ間・アプリ内でのコンテンツ移動・複製。iPad では対応が強く期待される |

### 🔴 必須（遵守事項）

- **大画面を活かすこと。** モーダルインターフェースとフルスクリーン遷移を最小化し、コンテンツを優先表示すること。画面の一部で済む操作（設定・詳細編集）をフルスクリーンで覆ってはならない。ポップオーバー・サイドバー・複数カラム（split view レイアウト）を優先する。
- **オンスクリーンコントロールは「アクセスしやすく、かつコンテンツを邪魔しない」位置に置くこと。**
- **視距離（〜90cm）を前提にコンテンツサイズを決め、入力手段に応じて密度を調整すること。**
- **複数入力手段に完全対応すること:**
  - Multi-Touch ジェスチャーのフルサポート
  - 物理キーボード / トラックパッド対応（ショートカット・ポインタ対応）
  - Apple Pencil 固有インタラクションへの対応（該当アプリ）
  - 上記の併用（キーボードで入力しながらタッチでスクロール等）を壊さないこと
- **環境変化にシームレスに適応すること。** 必須対応項目:
  - デバイスの向き（Portrait / Landscape）
  - マルチタスキングによるウィンドウサイズ変化（次章）
  - ダークモード
  - Dynamic Type
  - Mac 上での実行（Mac Catalyst / iPad apps on Mac）への移行
- **iPad でも 44x44pt を快適なタップ領域の推奨デフォルト**とし、入力方式と利用頻度に応じてサイズ・間隔・ヒット領域を検証すること。

### 推奨

- サイドバー + コンテンツの 2〜3 カラム構成（`UISplitViewController` / SwiftUI `NavigationSplitView`）で大画面の情報階層を平坦化する。
- 外付けキーボード利用者向けに主要操作のキーボードショートカットを定義する（メニュー / Shortcuts 発見性も確保）。
- drag and drop をコンテンツの入出力経路として実装する（テキスト・画像・ファイルの受け入れ / 提供）。
- iPhone 版と機能パリティを保ちつつ、レイアウトだけ iPad 最適化する（iPad 版で機能を削らない）。

### よくある違反・注意点

- ❌ iPhone レイアウトを単純拡大しただけの UI（1 カラムの巨大リスト・引き伸ばされたボタン）
- ❌ フルスクリーンモーダルの多用（大画面ではコンテキスト喪失が大きい。ポップオーバー / シート / インライン編集を検討）
- ❌ タッチ専用実装（トラックパッドのホバー・右クリック・スクロールに無反応、キーボードショートカットなし）
- ❌ 横向き固定 or 縦向き固定でマルチタスキング非対応になる設計（向き固定はマルチタスキング対象外要件と衝突しやすい）
- ❌ Apple Pencil 入力をタッチと同一視して精密操作を潰す

公式: https://developer.apple.com/design/human-interface-guidelines/designing-for-ipados

## マルチタスキングとウインドウ（Multitasking / Windows）

### 概要

マルチタスキングは「複数アプリや複数ウインドウを同時に使う」「アプリを切り替えても作業が失われない」体験の総称。現行 iPadOS の中心は、フルスクリーン表示と、macOS に近い自由リサイズ可能なウインドウ表示である。アプリはユーザーが選ぶ構成名を推測せず、現在のウインドウサイズへ応答し、状態を保存・復元する。

### iPad の表示形態

| 形態 | 内容 | アプリ側の責務 |
|------|------|---------------|
| Full Screen | アプリウインドウが画面全体を占有し、App Switcher で切り替える | 全画面でも同じ中核機能と状態を提供する |
| Windowed | ユーザーがウインドウを自由にリサイズ・移動・前面化し、複数を同時表示する | 最小から最大まで連続的に適応し、システムのウインドウ UI を妨げない |
| Multiple windows | 同一アプリの複数ウインドウを表示する | scene ごとの独立状態と共有データの同期を設計する |
| Picture in Picture | ビデオや通話を別コンテンツの上へ表示する | Full Screen / Windowed の両状態で動作させる |
| 旧 OS の構成 | Split View、Slide Over、Stage Manager | 対象 OS が必要とする場合だけ条件付きで追加検証する |

iPhone（iOS）側は App Switcher による切替と Picture in Picture（ビデオ再生・FaceTime 通話を他アプリ使用中も継続）が対象。

### 🔴 必須（遵守事項）

- **コンテキストの保存と復元を実装すること。** ユーザーがいつ離脱しても、復帰時に中断点（画面・スクロール位置・入力途中データ・再生位置）から再開できること。マルチタスキング対応の土台であり全アプリ必須。
- **注意・積極的な参加が必要なアクティビティは、離脱時に自動的に一時停止すること。** ゲームはプレイを、メディアビューアーは再生を一時停止し、復帰時にシームレスに再開する。
- **オーディオ割り込みに正しく応答すること:**
  | 割り込み種別 | 例 | 対応 |
  |-------------|-----|------|
  | 恒久的割り込み | 電話着信・他アプリの音楽再生開始 | 自アプリの音声を無期限一時停止 |
  | 一時的割り込み | ナビゲーション音声案内 | 音量を下げる（ducking）or 一時停止し、終了後に自動復帰 |
- **ユーザーが開始したタスクはバックグラウンドでも完了させること。** ダウンロード・書き出し等、追加入力が不要なタスクはアプリ切替後も走らせて完遂する。
- **iPad: サポートする最小から最大まで、あらゆるウインドウサイズにレスポンシブに適応すること。** 狭い幅・中間幅・広い幅・フルスクリーンで UI が崩れないよう、利用可能領域と Size Classes（compact / regular）を使い、固定画面サイズのハードコードを排除する。UIKit / SwiftUI の scene ベースライフサイクルとマルチタスキング API を採用すること。
- **ユーザーが選ぶマルチタスキング構成をアプリから制御しない**こと。アプリは構成名を受け取る前提にせず、結果として得られるウインドウサイズへ適応する。
- **ウインドウコントロールとツールバー項目を重ねない**こと。Windowed 表示ではシステムコントロールがツールバーの leading 側へ現れるため、既存項目を内側へ移動できる構成にする。
- **ウィンドウの外観（エッジ・シャドウ・ウィンドウコントロール）を独自に変更しないこと。** 前面ウィンドウのコントロール表示やドロップシャドウはシステムが管理する。

### 推奨

- **複数ウィンドウ（multiple windows / scenes）対応**: ドキュメント型・比較作業型アプリでは同一アプリの複数ウィンドウを開けるようにする。各ウィンドウで独立した状態管理と保存 / 復元を行い、ウィンドウ間のデータは同期させる。
- **通知は慎重に使うこと**: 重要または時間制約のあるタスク完了時のみ通知を送る。日常的なタスク完了（同期完了等）で通知しない — ユーザーはアプリに戻ったときに結果を確認することを期待している。
- フルスクリーン、狭いウインドウ、中間幅、広いウインドウをテストマトリクスに含め、連続的なリサイズ中も状態と操作を保つ。旧 OS をサポートする場合だけ、従来の分割構成を追加する。
- ビデオ再生・通話機能を持つアプリは Picture in Picture をサポートする。

### よくある違反・注意点

- ❌ `UIRequiresFullScreen` に依存してマルチタスキング対応を回避し続ける（レガシー扱い。新規アプリは全サイズ対応が前提）
- ❌ ウインドウが狭くなった瞬間にレイアウトが崩れ、中核機能や入力中の状態が失われる
- ❌ アプリ切替でゲーム / 再生が走り続ける、または復帰時に進捗が巻き戻る（コンテキスト復元の欠如）
- ❌ ナビ音声等の一時的割り込み後に音声が復帰しない / 割り込み中もフル音量で鳴らし続ける
- ❌ バックグラウンド移行時にユーザー開始のアップロード / 書き出しを黙って中断する
- ❌ ありふれた完了イベントを毎回プッシュ通知する
- ❌ 複数ウィンドウ間で状態が同期されず、同じデータの異なる版が並ぶ

- https://developer.apple.com/design/human-interface-guidelines/multitasking
- https://developer.apple.com/design/human-interface-guidelines/windows

## 付録: iOS / iPadOS 設計判断の対照表

| 判断軸 | iPhone（iOS） | iPad（iPadOS） |
|--------|--------------|----------------|
| 視距離 | 約 30〜60cm | 約 90cm 以内 |
| 主入力 | Multi-Touch（片手親指中心） | タッチ + キーボード + トラックパッド + Apple Pencil の併用 |
| コントロール配置 | 画面中央〜下部（上部は届きにくい） | アクセスしやすく、かつコンテンツを遮らない位置 |
| 画面構成 | 1 カラム・1 画面 1 タスク | 複数カラム・サイドバー・ポップオーバー活用 |
| モーダル | 許容されやすい | 最小化する（フルスクリーン遷移を避ける） |
| マルチタスキング | App Switcher / PiP | Full Screen / Windowed / 複数ウインドウ / PiP |
| ウインドウサイズ | iPhone 上では端末サイズ + 回転 | 自由リサイズ可能な任意サイズへ連続的に適応 |
| 共通原則 | ダークモード / Dynamic Type / 44x44pt の推奨タップ領域 / 状態の保存・復元 | 同左 + 入力手段別の密度調整 |
