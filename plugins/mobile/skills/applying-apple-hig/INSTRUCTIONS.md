# Apple HIGをiPhone・iPadアプリへ適用する

## 1. このスキルの役割

このスキルは、Apple公式のHuman Interface Guidelines（HIG）を使い、iPhone・iPad向けの体験を設計、実装、レビューするための判断ガイドである。

対象は次の領域とする。

- iOS／iPadOSの情報設計とナビゲーション
- 画面レイアウト、タイポグラフィ、色、素材、モーション
- SwiftUI／UIKitの標準コンポーネント選択
- タッチ、キーボード、ポインタ、Apple Pencil、音声入力
- Dynamic Type、VoiceOver、色以外の識別、Reduce Motion
- iPhoneとiPadのサイズ、向き、ウインドウ、入力方式への適応
- Appleのシステム機能をUIへ統合するときの体験判断とHIGレビュー

このスキルは、HIGの推奨を自動的にApp Reviewの必須要件へ変換しない。審査必須と述べるには、現行のApp Review Guidelines、API契約、エンタイトルメント、プライバシー要件などの直接根拠が必要である。

## 2. 他スキルとの責務境界

| 判断対象 | 使用するスキル |
|---|---|
| 画面構造、レイアウト、ナビゲーション、操作、外観、アクセシビリティ | applying-apple-hig |
| SwiftUI／UIKitの技術構成、状態・データフロー、ライフサイクル、性能 | developing-ios-apps |
| テスト戦略、Privacy Manifest、Required Reason API、署名、TestFlight、配布 | developing-ios-apps |
| UIを伴うiPhone／iPadアプリの実装 | 両スキルを併用 |

たとえばNavigationSplitViewの体験判断は本スキルで行い、状態保持やルーティングの実装はdeveloping-ios-appsと合わせて決める。

## 3. Apple公式資料をsource of truthにする

### 優先順位

根拠は次の順序で扱う。

1. 現行のApple Human Interface Guidelines
2. Apple Developer Documentationの該当API・フレームワーク資料
3. App Review Guidelines、App Store Connect Help、契約・申告資料
4. Apple Design ResourcesとApple公式の開発者向け動画
5. このスキルのローカルreference

ローカルreferenceは作業用のスナップショットであり、固定仕様ではない。
Apple公式ページと矛盾するときはApple公式を採用する。

日本語の公式ページに必要な情報があれば日本語版を優先する。
日本語版が未更新、不完全、または存在しない場合は英語版を使う。

二次資料を規範根拠にしない。
書籍名、著者名、出版社名を本文やレビュー根拠へ持ち込まない。

### 鮮度確認

タスク開始時に、対象領域に対応するApple公式ページを再確認する。
ネットワークを使えない場合は、その事実と未確認範囲を報告する。

最低限、次を確認する。

- HIGの該当ページとchange log
- 対象OSとSDKにおけるAPI availability
- SwiftUIまたはUIKitの現行API契約
- 「必須」と判定する場合のApp ReviewまたはAPI上の直接根拠
- 数値、提出条件、申告項目、エンタイトルメントの現行値

次の情報は時限情報として扱い、記憶だけで断定しない。

- App Review Guidelinesの番号・文言、App Store提出時のSDK・OS条件
- Privacy Manifest、Required Reason API、権限、エンタイトルメント
- OS／SDK／Xcodeの対応とAPIの導入・非推奨・削除状況
- 寸法、上限、回数、時間、証明書、署名、配布、審査メタデータ

確認できない時限情報は「要公式再確認」と明記する。
古い数値を安全側の値として流用しない。

### 根拠の記録

重要な判断には次の4点を残す。

- 根拠区分: 必須／原則採用／条件付き
- Apple公式ページ
- 確認日
- 対象OS、SDK、デバイス、入力方式

## 4. 要求強度を3段階で判断する

### 必須

次のいずれかに直接該当するものを必須とする。

- 現行App Review Guidelinesの明示要件
- 法令、プライバシー、ユーザー同意、安全に関する要件
- 公開API、エンタイトルメント、署名、配布の契約
- システム予約操作や保護リソースの利用条件
- ユーザーまたは正式仕様が定めた受け入れ条件

必須と報告するときは、何に基づく必須かを併記する。
「HIGにあるから審査必須」と短絡しない。

### 原則採用

HIGの標準、best practice、一般則、強い推奨は原則採用とする。

- システム標準コンポーネントを優先する
- プラットフォーム慣習に沿う
- 44x44ptを快適なタッチ領域の標準目標にする
- Dynamic Typeや複数入力を初期設計から扱う
- iPadの広い画面でコンテンツと複数カラムを活用する

採用しない場合は、製品上の理由、代替策、検証結果を記録する。
単なるブランド都合や実装コストだけを理由にしない。

### 条件付き

次の条件で変わる指針は条件付きとする。

- OSとSDKのバージョン
- iPhoneかiPadか
- フルスクリーンかリサイズ可能なウインドウか
- 現在のウインドウ寸法とサイズクラス
- タッチ、キーボード、ポインタ、Apple Pencilの利用
- SwiftUIかUIKitか
- アプリの目的、コンテンツ量、利用時間
- 権限、エンタイトルメント、対応ハードウェア

条件を特定できない場合、必須へ格上げせず条件付きのまま保留する。

### 判定手順

1. Apple公式の文言とページ種別を確認する。
2. App Review、API契約、HIGのどれが根拠か分離する。
3. 対象OS、デバイス、ウインドウ、入力方式を特定する。
4. 必須、原則採用、条件付きのいずれかへ分類する。
5. 例外を採る場合は理由、代替、検証方法を記録する。

## 5. 現行のデザイン原則

Appleの現行Design principlesは、次の8原則で判断する。

| 原則 | 適用時の問い |
|---|---|
| Purpose | 誰のどの課題を解決し、中核機能と製品固有の価値へ集中しているか |
| Agency | 主タスクへ直接進め、スキップ、キャンセル、Undo、状態復元で主導権を保てるか |
| Responsibility | データ、権限、AI、課金の目的を明示し、誤用・損害を予防できるか |
| Familiarity | 既知の概念、標準パターン、ジェスチャを使い、操作結果を明確に返すか |
| Flexibility | 多様な人、デバイス、向き、ウインドウ、入力へ文脈を保って適応するか |
| Simplicity | 中核タスクに必要な要素だけを、短い文言と明確な階層で示すか |
| Craft | 外観、動き、文言、音、性能を実機で反復検証し、OS更新へ追従するか |
| Delight | 役立つ応答と滑らかな操作が、可読性やアクセシビリティを妨げず印象を作るか |

HIG全体を通じて、Hierarchy（重要度・グループ・操作順が視覚構造から分かるか）、Harmony（ハードウェア・システム外観・コンテンツ・機能が一体として感じられるか）、Consistency（プラットフォームの慣習とアプリ内の語彙・操作が一貫するか）も各画面で確認する。

## 6. タスク開始時の前提確認

実装やレビューの前に次を確定する。

- ユーザーの主目的と最重要タスク
- 対象デバイス: iPhone、iPad、または両方
- 最低OS、使用SDK、SwiftUI／UIKit
- 対応する向き
- フルスクリーン、ウインドウ表示、複数ウインドウの範囲
- 主要なコンテンツ階層とナビゲーション
- 主入力と追加入力
- Dynamic Type、VoiceOver、Reduce Motionの検証範囲
- 権限、課金、アカウント、App Store提出の関与
- 既存デザインシステムと変更できない制約

コードや既存画面から判別できる情報は先に確認する。
不明点が結果を大きく変えない場合は、合理的な仮定を明示して進める。

## 7. 実行ワークフロー

1. タスクを設計、実装、レビューのどれかに分類する。
2. 対象領域のApple公式ページを再確認する。
3. 判断を必須、原則採用、条件付きへ分類する。
4. コンテンツ階層と主タスクを先に定義する。
5. 標準ナビゲーションと標準コンポーネントを選ぶ。
6. 最小幅から最大幅までの適応戦略を決める。
7. Dynamic TypeとVoiceOverを構造へ組み込む。
8. タッチ以外の入力経路を設計する。
9. SwiftUI／UIKitのシステムAPIで実装する。
10. 検証マトリクスを実機とシミュレータで実行する。
11. 根拠区分、例外、未確認事項を報告する。

## 8. iPhoneとiPadの適応設計

### 共通原則

- デバイス名や特定解像度ではなく、利用可能領域とコンテンツで分岐する。
- safe area、レイアウトガイド、readable content widthを使う。
- 固定座標、固定高さ、画面幅の決め打ちを避ける。
- 回転やリサイズで選択、入力、スクロール、再生位置を失わない。
- 最大レイアウトと最小レイアウトの両端から検証する。
- コンパクト化で中核機能を削除せず、配置と提示方法を変える。

### iPhone

- 片手と両手、縦向きと横向きの利用を想定する。
- 頻用操作は届きやすさを確認し、上端だけに集中させない。
- キーボード表示時も入力欄と次の操作を見える状態に保つ。
- Dynamic Island、カメラ領域、ホームインジケータをsafe areaで避ける。
- 向きの制限は、ゲームやカメラ等の製品目的と対象OSの現行契約に直接理由がある場合だけ条件付きで採用し、iPadの可変ウインドウや適応レイアウトを避ける手段にしない。
- iPhone互換アプリをiPadで動かす場合も、現行環境のリサイズ挙動を公式資料で再確認する。

### iPad

- iPadは「大きな固定画面」ではなく、変化するウインドウとして設計する。
- 現行iPadOSでは、ユーザーがフルスクリーン表示とウインドウ表示を選べる。
- ウインドウ表示では、アプリウインドウを自由にリサイズ、移動、前面化できる。
- 複数ウインドウを同時表示し、同一アプリの複数ウインドウも扱える。
- アプリはユーザーが選んだマルチタスキング構成を制御しない。
- 構成名を推測せず、現在のサイズとtraitへ応答する。
- ウインドウコントロールはツールバーのleading側へ現れるため、項目の重なりを検証する。
- システムのウインドウ枠、影、コントロールを再実装しない。
- フルスクリーンでもウインドウ表示でも同じ中核機能を提供する。
- 旧OSを対象にする場合だけ、Split View、Slide Over、Stage Manager固有の挙動を条件付きで確認する。

### 幅が縮むときの優先順位

次の順で適応を検討する。

1. 余白を縮める。
2. ラベルを短くする。
3. ツールバーの低優先項目をメニューへ移す。
4. 第3カラムやインスペクタを閉じる。
5. 複数カラムを単一カラムのナビゲーションへ変える。
6. 横並びを縦積みにする。
7. 必要な方向へスクロールさせる。

重要な操作やデータを単に非表示にして解決しない。

### 状態と複数ウインドウ

- sceneごとにナビゲーション、選択、スクロール、編集中データを保持する。
- ウインドウを閉じた後の復元方針を定義する。
- 同一データを複数ウインドウで扱う場合は競合と同期を設計する。
- 新しいウインドウを既定で乱発せず、文脈保持に役立つ場面で提供する。
- ウインドウの最小・中間・最大付近で状態遷移を確認する。

### 適応検証マトリクス

| 軸 | 最低限のケース |
|---|---|
| iPhone | 小さい幅、大きい幅、縦向き、対応する横向き |
| iPad | フルスクリーン、狭いウインドウ、中間幅、広いウインドウ |
| 外観 | Light、Dark、Increase Contrast、Reduce Transparency |
| テキスト | 標準、最大の一般サイズ、アクセシビリティ最大付近 |
| 言語 | 長い翻訳、RTL、日付・数値の異なるロケール |
| 状態 | 初期、読み込み、空、エラー、オフライン、権限拒否 |

## 9. アクセシビリティ

アクセシビリティは実装後の追加作業ではない。
情報構造、コンポーネント、文言、入力方式の選択時に組み込む。

### コントロールサイズと間隔

- iOS／iPadOSでは44x44ptを快適なコントロールの標準目標とし、ボタンのヒット領域も一般則として同等以上を目指す。
- AppleのAccessibility資料は標準44x44ptと最小28x28ptを区別するため、44x44ptを全要素の絶対的なApp Review最小値とは表現しない。28x28ptは目標値ではなく下限側の例外であり、Buttons等のコンポーネント固有HIGがより大きな領域を示す場合は現行の個別指針を優先する。
- ゲームでは頻用コントロール44x44pt、低頻度コントロール28x28ptという個別指針を適用する。
- 見た目を小さくする場合も、標準コンポーネント、十分な間隔、拡張したヒット領域を検討する。
- ベゼル付き要素の周囲約12pt、ベゼルなし要素の周囲約24ptを間隔の目安として検証する。
- サイズだけで合否を決めず、誤操作、運動負荷、入力方式、利用頻度を実機で確認する。

### Dynamic Type

- システムテキストスタイルを優先し、カスタムフォントは相対スケールさせる。
- 文字を固定frameへ閉じ込めず、複数行、縦積み、スクロールを許可する。
- 重要ラベルを省略記号だけにせず、200%程度の拡大を選べることを理想目標にする。
- 一般・アクセシビリティカテゴリとBold Textで階層とレイアウトを検証する。

### VoiceOver

- 要素の役割、ラベル、値、状態を意味に沿って設定し、装飾画像は読み上げ対象から外す。
- 見た目ではなく操作対象と結果を説明し、関連要素を情報を失わない単位でグループ化する。
- 視覚順と読み上げ順を揃え、モーダル表示前後のフォーカス位置を管理する。
- 動的更新の通知を必要時に限定し、カスタムジェスチャにはaccessibility actionや可視操作を用意する。
- ローター、Switch Control、Voice Controlでも中核タスクを確認する。

### 色と知覚

- 色だけで選択、エラー、成功、系列、必須項目を表さず、テキスト、形状、シンボル、パターン、位置を併用する。
- semantic colorを意味で選び、Light／Dark／Increase Contrastで確認する。
- グラフには色に加えて形状、線種、直接ラベルを使う。

### モーション

- Reduce Motionを環境値から取得し、大きな移動、ズーム、視差、連続ループを減らす。
- 空間関係を保つクロスフェード等へ置き換え、完了やエラーを動きだけで伝えない。
- アニメーション完了まで操作をブロックせず、点滅や自動再生には停止・軽減手段を用意する。

### 音声・動画・触覚

- 音声情報へキャプション、字幕、トランスクリプトを提供する。
- 重要な音やハプティクスには別感覚の代替を用意し、自動再生には停止手段を設ける。

## 10. 入力方式

| 入力 | 設計原則 | 最低限の検証 |
|---|---|---|
| タッチ | 標準ジェスチャ、十分な領域、即時フィードバック | 片手、両手、誤タップ |
| キーボード | 標準ショートカット、フォーカス、メニュー | Full Keyboard Access |
| ポインタ | システム効果、ホバーは補助情報 | マウス、トラックパッド |
| Apple Pencil | 低遅延、Scribble、ユーザー設定尊重 | 描画、ホバー、スクイーズ |
| 音声 | App Intentsと明確な応答 | 画面を見ない操作 |
| 支援技術 | 同じ機能へ代替経路 | VoiceOver、Switch Control |

### タッチ

- システム予約ジェスチャと競合せず、スワイプ、長押し、複数指操作を唯一の経路にしない。
- 押下、ドラッグ、ドロップの状態を返し、破壊的操作にはUndoまたは適切な確認を用意する。

### キーボード

- Copy、Paste、Undo、Redo、Findなどの標準意味を変えない。
- 主要コマンドへショートカットとdiscoverability titleを与え、無効時も必要に応じて状態を示す。
- Tabと矢印キーで論理的に移動できるようにし、フォーカス表示を消さない。

### ポインタ

- タッチと同じ中核機能へ到達でき、ホバーなしでも意味が分かるようにする。
- テキスト入力のI-beamなどシステム形状を尊重し、装飾的なカスタムポインタを避ける。

### Apple Pencil

- PencilKitと標準テキスト入力を優先し、Pencil接地時の遅延を抑える。
- 指とPencilを自然に切り替えられるようにする。
- ホバーは結果のプレビューに限定し、ダブルタップとスクイーズはシステム設定を尊重する。
- 破壊的操作をPencil固有ジェスチャへ割り当てない。

### 入力の組み合わせ

iPadでは入力方式が同時利用される。キーボード入力、タッチスクロール、ポインタ選択、Apple Pencil編集を切り替えても状態を壊さない。

## 11. コンポーネント選択

標準コンポーネントを第一候補にする。
標準コンポーネントは外観だけでなく、状態、入力、Dynamic Type、
VoiceOver、Liquid Glass、将来のOS更新への適応を提供する。

カスタムコンポーネントを採用する前に次を確認する。

- 標準コンポーネントでは達成できない製品上の理由があるか
- 標準の役割、状態、入力、アクセシビリティを再現できるか
- Light／Dark、Dynamic Type、VoiceOver、キーボードで検証できるか
- 将来のOS外観変更を追従できるか

SwiftUIとUIKitを混在させる場合も、ユーザーに見える挙動を揃える。
フレームワーク固有の実装差を、画面ごとの操作差にしない。

## 15. レビュー報告形式

レビューは次の形式で報告する。

### 対象

- 画面／機能、OS／SDK:
- デバイス／ウインドウ、入力方式:
- 確認したApple公式ページ、確認日:

### 所見

所見ごとに次を記す。

1. `[必須|原則採用|条件付き] [高|中|低] 要約`
   - 根拠種別とApple公式: App Review／API契約／HIG／正式仕様
   - 影響する利用者・デバイス・入力:
   - 観測した問題と推奨する修正:
   - 検証方法:

「App Reviewで拒否される」と述べるのは、
現行App Review Guidelinesなどに直接根拠があり、観測した挙動を該当条項へ対応付けられる場合だけにする。条項番号の固定一覧をHIGから推測せず、レビュー時点の公式ページで検索する。

### 例外と未解決

- 採用しなかったHIG推奨、理由、代替策、残るリスク
- 公式再確認が必要な時限情報と、実機や権限不足で未検証の項目

### 総合判定

- 必須の未解決件数、原則採用からの逸脱件数
- 条件付きで残る検証件数と、次に行う具体的な確認

所見がない場合も、検証した軸と未検証範囲を報告する。

## 17. 禁止事項とよくある失敗

- HIGの推奨を一律に審査必須と書かない。
- 44x44ptを全コントロールの絶対最小値や自動リジェクト条件と書かない。
- iPadをSplit View／Slide Over／Stage Managerだけで説明しない。
- iPadを常にRegular幅と仮定しない。
- 特定iPhoneの画面幅をレイアウト定数にしない。
- iPhone版をそのまま拡大してiPad版としない。
- タッチだけを唯一の入力方式としない。
- 色、動き、音だけで情報を伝えない。
- Dynamic Typeを固定フォントサイズで代替しない。
- VoiceOverラベルを見た目の説明だけにしない。
- 標準ジェスチャやシステムウインドウUIを再定義しない。
- API availabilityや審査番号を記憶だけで断定しない。
- 二次資料、書籍名、著者名、出版社名を根拠にしない。
- referenceをApple公式より優先しない。
