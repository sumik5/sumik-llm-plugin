# App Store 審査ガイドライン準拠（App Store Review Guidelines）

> 出典: Apple Human Interface Guidelines / Apple Developer Documentation（最終確認 2026-07-13）。仕様は更新されるため、実装時は必ず併記の公式 URL で最新版を確認すること。日本語版がある場合は公式の言語切替または本文の日本語 URL を使い、英語版との差分がある場合は最新側を確認する。

> 鮮度補正: 本文の章・条項番号、例外、文字数、割合、地域別決済条件、審査手順は2026-07-13時点のスナップショットである。審査判断や提出メモでは本ファイルの番号をそのまま引用せず、提出時点のApp Review Guidelinesで論点を検索し、現行の条項番号・文言・適用storefrontを確認すること。

App Review ガイドラインは **Safety / Performance / Business / Design / Legal** の5章構成。アプリの設計・実装・提出物（メタデータ・IAP 設計・プライバシー実装）はすべてこのガイドラインに拘束される。違反は「リジェクト」だけでなく、重大・反復時は **App Store からの削除・Apple Developer Program 除名** に至る。実装判断の段階でガイドラインを織り込むこと（提出直前の後付け対応はリジェクトの主因）。

## 1. 安全性（Safety）

### 概要
ユーザー（特に子ども）に危害・不快を与えるコンテンツと、物理的危害・データ漏洩につながる実装を禁止する章。

### 🔴 必須（遵守事項）

| 項 | 要件 |
|----|------|
| 1.1 | 差別的・中傷的コンテンツ、リアルな暴力描写、わいせつコンテンツ、武器・危険物の使用助長、最近の事件（紛争・テロ・感染症）の収益化を含めてはならない |
| 1.2 | **ユーザー生成コンテンツ（UGC）を持つアプリは4点セット必須**: ①不適切コンテンツの投稿防止手段（フィルタリング）②ユーザーによる報告機能と迅速な対応 ③加害ユーザーのブロック機能 ④デベロッパへの連絡手段 |
| 1.2.1 | クリエイターコンテンツ配信アプリは、年齢制限を超えるコンテンツの識別手段と年齢確認メカニズムを提供すること |
| 1.3 | 「子ども向け（Kids）」カテゴリ: アプリ外リンク・課金導線は保護者ゲート（parental gate）の外に置いてはならない。サードパーティ分析・広告は原則禁止。個人識別情報・デバイス情報のサードパーティ送信禁止 |
| 1.4.1 | 医療系アプリは測定精度の根拠を明示し、デバイスセンサーのみで心電図・血圧・体温・血糖値・血中酸素の測定を主張してはならない。医師の診断が必要な旨を通知すること |
| 1.4.2 | 薬物用量計算アプリは製薬会社・病院・大学・保険会社・薬局等の認可事業体、または規制当局承認が必要 |
| 1.5 | 正確・最新のデベロッパ連絡先情報を提供すること |
| 1.6 | 収集したユーザー情報への不正アクセス・不正利用・不正開示を防止するデータセキュリティ実装を行うこと |

### 推奨
- UGC 機能の実装時は、報告 → 24時間以内対応のモデレーション運用を設計段階で組み込む（コードだけでなく運用体制も審査対象）。
- 医療・健康系は「診断」ではなく「参考情報」であることを UI 文言で明示する。

### よくある違反・注意点
- チャットルーレット型・匿名ランダムチャットは UGC 対策があってもリジェクトされる。
- 偽の位置情報生成・いたずら電話/SMS 機能はトリック機能として一律禁止（1.1.6）。
- Kids カテゴリから外しても、子どもを主対象とするアプリには同要件が引き続き適用される。

公式: https://developer.apple.com/jp/app-store/review/guidelines/

## 2. パフォーマンス（Performance）

### 概要
「完成したアプリを、正確なメタデータとともに提出する」ことを求める章。完全性は重点確認項目である。約40%という数値は確認時点の参考情報にすぎず、現在の却下率や審査結果の予測には使わない。

### 🔴 必須（遵守事項）

**2.1 アプリの完全性**
- 最終バージョンを提出すること。プレースホルダテキスト・空のWebサイト・仮画像・ダミーデータを残してはならない。
- 実機でバグ・クラッシュ・安定性をテスト済みであること。クラッシュや明白な技術的問題があれば即リジェクト。
- ログインが必要なアプリは **デモアカウント（ID/パスワード）を App Store Connect の審査メモに記載** し、バックエンドを審査期間中有効にしておくこと（法的・セキュリティ理由で不可の場合は Apple 事前承認のデモモードを内蔵）。
- IAP を含む場合、購入フローが完全・最新で正常動作すること。審査で見つけにくいアイテムは審査メモで場所を説明する。

**2.3 正確なメタデータ**
- 説明・スクリーンショット・プレビューはアプリの主要な体験を正確に反映すること。隠し機能・未記載機能は禁止（2.3.1。発覚時はアプリ削除・アカウント停止の対象）。
- アプリ名の上限は現行App Store Connect Helpとガイドラインで確認する。30文字は確認時点のスナップショットである。上限に関係なく、メタデータに他社商標・人気アプリ名・価格・無関係キーワードを詰め込んではならない。
- スクリーンショットは実際の利用中画面を使う。タイトル画面・ログイン画面・スプラッシュのみは不可（2.3.3）。プレビュー動画はアプリ内の実映像のみ（2.3.4）。
- 追加課金が必要な機能は、説明・スクリーンショットでその旨を明示すること（2.3.2）。
- カテゴリ選択・年齢制限指定（age rating）は正直に設定すること（2.3.5 / 2.3.6）。
- スクリーンショット等のメタデータは4歳以上に適切な内容にすること（2.3.8）。実ユーザーデータではなく架空アカウントを表示すること（2.3.9）。
- 他モバイルプラットフォームの名称・アイコンをメタデータに含めない（2.3.10）。

**2.4 / 2.5 ハードウェア互換・ソフトウェア要件**
- 公開 API のみ使用すること（private API 使用は即リジェクト）。現行 OS で動作すること（2.5.1）。
- アプリバンドル内で完結すること。機能を追加・変更するコードのダウンロード・実行は禁止（教育用のコード学習アプリ等の例外あり）（2.5.2）。
- 急速なバッテリー消費・過度な発熱・バックグラウンドでの無関係な処理（暗号通貨マイニング含む）は禁止（2.4.2）。
- 音量ボタン等の標準スイッチ機能やネイティブ UI の動作変更・無効化は禁止（2.5.9）。
- 録音・録画・アクティビティ記録は明示的なユーザー同意を取得し、記録中であることを視覚的/聴覚的に明示すること（2.5.14）。
- 顔認証によるアカウント認証は LocalAuthentication を使用すること（ARKit 等での代用禁止）。13歳未満には代替認証手段を用意すること（2.5.13）。
- 広告はメインアプリバイナリ内のみ。Extension・App Clip・ウィジェット・通知・キーボードに広告を入れてはならない。インタースティシャル広告には明確な「閉じる/スキップ」ボタンと「広告」表示が必要（2.5.18）。
- IPv6 のみのネットワークで完全動作すること（2.5.5）。
- ベータ配布は TestFlight のみ。TestFlight ビルドもガイドライン準拠が必要で、テスター配信の対価受領は禁止（2.2）。
- iPhone アプリは可能な限り iPad でも動作させること（2.4.1）。

### 推奨
- 「新機能（What's New）」欄には新機能・変更点を具体的に書く。軽微なバグ修正のみなら汎用文言で可（2.3.12）。
- ウィジェット・Extension・通知はアプリ本体の機能と関連させる（2.5.16）。

### よくある違反・注意点
- 「実機テストせず Simulator のみで確認」→ 実機依存クラッシュで 2.1 リジェクトの典型パターン。
- デモアカウント未記載は審査停滞・リジェクトの最頻出原因のひとつ。サインイン必須アプリは必ず記載する。
- サポート URL・プライバシーポリシー URL のリンク切れも 2.1 リジェクト対象。

公式: https://developer.apple.com/jp/app-store/review/guidelines/

## 3. ビジネス（Business）

### 概要
収益化モデルの章。**デジタルコンテンツ・機能の販売は In-App Purchase（IAP）必須** が大原則で、IAP 回避の実装はリジェクト・除名の主因。

IAPの例外、外部リンク、entitlement、storefront、サブスクリプション期間、金融条件は変更され得る。以下の条項番号・期間・率を固定実装せず、対象storefrontとビジネスモデルを提出時点のガイドラインに照合する。

### 🔴 必須（遵守事項）

**3.1.1 IAP を使わなければならないもの**
- アプリ内で消費されるデジタルコンテンツ・機能: サブスクリプション、ゲーム内通貨、追加レベル、プレミアムコンテンツ、フルバージョン解放。
- ライセンスキー・AR マーカー・QR コード・暗号通貨ウォレット等による IAP 回避（アンロック機構）は禁止。
- デジタル商品用ギフトカード・バウチャーの販売も IAP のみ。
- ゲーム内通貨・クレジットに有効期限を設定してはならない。
- **ルートボックス等のランダム型アイテムは、購入前に各アイテムの入手確率を明示すること**。
- NFT の販売・ミンティングは IAP 経由なら可。ただし NFT 所有をアプリ内機能のアンロック条件にしてはならない。

**3.1.1(a) / 3.1.3 外部購入への誘導**
- 米国ストアフロントおよび StoreKit External Purchase Link Entitlement 等の承認済みエンタイトルメント保有アプリを除き、アプリ内で IAP 以外の購入手段へ誘導（ボタン・外部リンク・CTA）してはならない。地域・エンタイトルメントごとに許可範囲が異なるため、対象ストアフロントを実装前に確認すること。
- アプリ外（メール等）でのユーザーへの案内は許可される。

**IAP が不要なもの（物理・アプリ外）**
- 物理商品・アプリ外で消費されるサービスの決済は Apple Pay・クレジットカード等を使用すること（IAP を使ってはならない）（3.1.3(e)、5.3.3 の実金ギャンブルも同様）。
- 「リーダー」アプリ（雑誌・新聞・書籍・音楽・ビデオ等の既購入コンテンツ閲覧）は外部で購入済みコンテンツへのアクセスを提供可（3.1.3(a)）。
- 1対1のパーソナルサービス（家庭教師・医療相談・不動産内覧・フィットネス指導）は IAP 以外可。ただし1対多のリアルタイムサービスは IAP 必須（3.1.3(d)）。
- マルチプラットフォームサービスは他プラットフォームで取得したコンテンツへのアクセス提供可。ただし同アイテムを IAP でも購入可能にすること（3.1.3(b)）。

**3.1.2 サブスクリプション**
- 継続的な価値提供が必要。**最低期間は7日間**、かつユーザーの全デバイスで利用可能であること。
- 価格に対して何が得られるか（発行数・ストレージ容量等）を購入前に明確に説明すること。
- アップグレード/ダウングレードはスムーズに処理し、同一コンテンツの二重購入を防ぐこと。
- 既存有料アプリをサブスク化する場合、既購入ユーザーから購入済み機能を取り上げてはならない。
- サブスク解約を意図的に困難にする、虚偽説明で加入させる等の詐欺的設計は削除・除名対象。

**3.2 その他**
- 慈善募金は承認された非営利団体のみアプリ内で直接受付可。それ以外は Safari・SMS 等アプリ外で行うこと。
- アプリの機能・コンテンツ利用の条件として、評価・レビュー投稿や他アプリのダウンロードを要求してはならない（3.2.2(x)）。
- 広告インプレッション・クリックの人為的な水増し、ランキング操作は禁止。
- 個人向けローンは全条件（最大 APR・支払期限）の明示が必須。**APR 36% 超・60日以内全額返済要求は禁止**。
- 暗号通貨: デバイス上でのマイニング禁止（クラウド処理は可）。ウォレットは組織登録デベロッパのみ。取引所は認可地域のみ。

### 推奨
- 非サブスク型の無料トライアルは「XX日間お試し」という名前の Tier 0（価格0）IAP として実装し、期間・終了後の制限・課金額を事前明示する。
- 課金 UI では価格・期間・自動更新の有無を購入ボタン近傍に明記する（詐欺的ダークパターンは 5.6 行動規範違反）。

### よくある違反・注意点
- 「Web で買えばもっと安い」等の文言・リンクをアプリ内に置く → エンタイトルメントなしでは 3.1.1(a) 違反の典型。
- デジタル機能のアンロックに Stripe 等の外部決済を直接組み込む → 即リジェクト、反復で除名。
- サブスク導入時に旧買い切りユーザーの機能を制限 → 3.1.2(a) 違反。

公式: https://developer.apple.com/jp/app-store/review/guidelines/

## 4. デザイン（Design）

### 概要
最低機能水準・独自性・Apple エコシステムとの整合を求める章。HIG（Human Interface Guidelines）準拠の粗い UI や、Web サイトの単純ラップは本章でリジェクトされる。

### 🔴 必須（遵守事項）

| 項 | 要件 |
|----|------|
| 4.1 | 他アプリのコピー・名前や UI の微修正版を提出してはならない。他社アイコン・ブランド・製品名の無断使用禁止。他アプリへのなりすましは除名対象 |
| 4.2 | **最低限の機能**: Web サイトの単純な再パッケージは不可。アプリならではの継続的な価値・有用性が必要。単なる書籍は Apple Books へ、単なる楽曲/映画は iTunes Store へ |
| 4.2.3 | 単体で動作すること。別アプリのインストールを前提にしてはならない。初回起動時に追加リソースをダウンロードする場合はサイズを開示すること |
| 4.3 | **スパム禁止**: 同一アプリを複数バンドル ID で提出しない。地域・チーム・大学別のバリエーションは1アプリに統合する。飽和カテゴリ（出会い系・懐中電灯・占い等）への差別化なき新規提出は不受理 |
| 4.4.1 | キーボード Extension は文字入力機能・「次のキーボード」への切替手段が必須で、ネットワークなしで基本動作すること。「設定」以外のアプリ起動禁止 |
| 4.5.4 | プッシュ通知を必須機能にしてはならない。通知で個人情報・機密情報を送らない。マーケティング通知は明示的オプトイン・オプトアウト手段が必須 |
| 4.7 | ミニアプリ・ミニゲーム・ストリーミングゲーム・チャットボット・エミュレータ内の全ソフトウェアについて、提供者がガイドライン準拠の責任を負う。年齢超過コンテンツの識別手段・年齢制限メカニズム必須 |
| 4.8 | **ログインサービス**: Facebook/Google 等サードパーティログインを主要認証に使う場合、①収集をユーザー名とメールに限定 ②メール非公開可 ③広告目的のデータ収集なし、を満たす別のログイン選択肢（例: Sign in with Apple）を同等の選択肢として提供すること（自社アカウントのみ・企業/教育向け等は免除） |
| 4.9 | Apple Pay 利用時は購入前に全重要情報を提示。定期支払いは更新期間・提供内容・料金・キャンセル方法の開示必須 |
| 4.10 | OS 内蔵機能（プッシュ通知・カメラ・ジャイロ等）や Apple サービスへのアクセス自体を課金対象にしてはならない |

### 推奨
- レビュー依頼は SKStoreReviewController 等の公式 API を使用する（カスタムのレビュー催促ダイアログは 5.6.1 違反）。
- ARKit 使用アプリは統合された AR 体験を提供する（単に 3D モデルを表示するだけでは 4.2.1 で不十分と判断される)。
- Extension にマーケティング・広告・IAP を含めない（4.4）。

### よくある違反・注意点
- WebView に自社サイトを表示するだけの「ガワネイティブ」は 4.2 リジェクトの定番。プッシュ通知・オフライン対応・ネイティブ UI 等のネイティブ価値を足す。
- テンプレート/アプリ生成サービス製アプリはコンテンツ提供者自身のアカウントから提出しないと 4.2.6 でリジェクト。
- サードパーティログインだけ実装して Sign in with Apple 相当の選択肢がない → 4.8 リジェクト。

公式: https://developer.apple.com/jp/app-store/review/guidelines/

## 5. 法的事項（Legal）— プライバシー中心

### 概要
配信する全地域の法令準拠と、Apple独自のプライバシー要件を定める章。2.1と5.1という番号や頻度順は確認時点の参考である。提出時は現行のプライバシー条項を検索し、番号と要件を再確認する。

### 🔴 必須（遵守事項）

**5.1.1 データの収集・保存**
- **全アプリにプライバシーポリシー必須**。App Store Connect のメタデータとアプリ内の両方からアクセス可能にすること。記載必須項目: ①収集データの種類と収集方法 ②全用途 ③サードパーティ共有先（分析・広告 SDK・親会社含む）と同等水準の保護確認 ④保持・削除ポリシー ⑤同意撤回とデータ削除リクエストの方法。
- ユーザーデータ・使用状況データの収集は（匿名データでも）同意を取得すること。同意の撤回手段を容易に提供すること。
- **最小限のデータ収集**: 中心機能に必要なデータのみ要求。写真・連絡先はフルアクセスではなくピッカー（PHPickerViewController 等）・シェアシートの利用を優先する。
- **目的文字列（purpose string）**: 機密データにアクセスする API を使う場合、Info.plist の `NS*UsageDescription` に「何のために使うか」を具体的に書くこと（「カメラを使用します」だけでは不十分。例:「プロフィール写真の撮影に使用します」）。空文字列・曖昧な文言はリジェクト対象。
- アカウント不要で使える機能はログインなしで提供すること。**アカウント作成機能を持つアプリはアプリ内でのアカウント削除機能も必須**。
- 規制産業（銀行・金融・ヘルスケア・ギャンブル・大麻・航空・暗号資産取引）のアプリは個人ではなく法人アカウントから提出すること。

**5.1.2 データの使用・共有**
- 事前のユーザー許可なくデータを使用・送信・共有してはならない。**サードパーティ AI へのデータ共有も明確な開示と事前の明示的許可が必要**。
- トラッキング（アプリ横断の広告計測等）は **App Tracking Transparency（ATT）API 経由の明示的許可** が必須。
- 収集データからの隠れたユーザープロファイル構築、「匿名化」データからの再識別は禁止。
- 連絡先・写真 API のデータで連絡先データベースを構築・販売してはならない。他アプリのインストール情報を広告目的で収集してはならない。
- HealthKit・ClassKit・深度/顔マッピング（ARKit 等）由来のデータを広告・マーケティング・データマイニングに使用してはならない。

**5.1.3 健康 / 5.1.4 子ども / 5.1.5 位置情報**
- 健康データは広告目的で使用・共有禁止。HealthKit に虚偽データを書き込まない。個人健康情報を iCloud に保存しない。健康臨床研究はインフォームドコンセントと倫理審査委員会（IRB）承認が必須。
- 子ども対象アプリは COPPA・GDPR 等に準拠。生年月日・保護者連絡先は法令準拠目的でのみ収集可。
- 位置情報はアプリの機能に直接関連する場合のみ使用可。収集・送信前に通知と同意、目的の説明が必須。

**5.2 知的財産**
- 自作またはライセンス取得済みコンテンツのみ使用可。第三者の商標・著作物・音源・動画の無断使用は削除対象。使用許諾の証憑を求められたら提示できるようにしておく。
- YouTube・SoundCloud 等のコンテンツのダウンロード/変換機能は提供元の明示的承認なしに搭載してはならない。
- Apple 製品・UI の模倣、Apple 絵文字のバイナリ埋め込み、Apple による推奨を示唆する表現は禁止。

**5.3 ギャンブル**
- 実金ギャンブル・宝くじは提供地域のライセンス必須・当該地域限定・アプリ本体は無料配信。実金ギャンブルのクレジット購入に IAP を使ってはならない。

**5.4 / 5.5 VPN・MDM**
- VPN アプリは NEVPNManager API 使用・組織デベロッパのみ・データのサードパーティ販売/共有禁止。
- MDM アプリは Apple の MDM 機能をリクエストし、営利企業・教育機関・政府機関のみ提供可。

**5.6 デベロッパ行動規範**
- カスタマーレビューへの返信は敬意をもって行い、個人情報・スパム・マーケティングを含めない。レビュー促進は公式 API のみ、カスタムプロンプト禁止。
- ランキング・検索・レビュー・おすすめの不正操作（レビュー購入・インセンティブ付き評価等）は禁止。反復すると Developer Program アカウント停止。

### 推奨
- App Store Connect の「App のプライバシー」（Privacy Nutrition Labels）とプライバシーポリシー・実装（実際に収集しているデータ）の三者を一致させる。不一致はリジェクト・信頼低下の原因。
- サードパーティ SDK 導入時は、その SDK が何を収集するか（Privacy Manifest）を確認してからラベルに反映する。

### よくある違反・注意点
- ATT ダイアログの前に独自の「許可してください」画面で誘導・強制する設計は 5.1.1(iv) の「巧みな誘導」に該当しうる。機能提供とデータ許可を交換条件にしない。
- アカウント削除機能の欠落は近年の頻出リジェクト理由。退会導線を Web に飛ばすだけの実装は原則不可（アプリ内から開始できること）。
- 公開データベース由来でも、本人の明示的同意なく個人情報を収集すれば 5.1.1(viii) 違反。

公式: https://developer.apple.com/jp/app-store/review/guidelines/

## よくあるリジェクト理由 トップパターンと対策

完全性は重点確認項目である。約40%という数値と「頻度の高い順」は確認時点の参考情報であり、現行の統計や審査優先順位として断定しない。以下は提出前に漏れやすい論点の作業用索引である。

| リジェクト理由 | 該当ガイドライン | 対策（実装・提出時にやること） |
|---|---|---|
| クラッシュ・バグ | 2.1 | 実機（最新 OS）で全フローをテスト。クラッシュレポート・デバイスログを確認してから提出する |
| 未完成・プレースホルダ残存 | 2.1 | 仮テキスト・仮画像・「Coming Soon」導線・空リンクを全て除去。未実装機能は UI から消す |
| 審査情報の不足 | 2.1 | デモアカウント・特殊な設定手順・ハードウェア依存機能のデモ動画を審査メモに必ず記載する |
| リンク切れ | 2.1 / 5.1 | サポート URL・プライバシーポリシー URL の生存を提出直前に確認する |
| メタデータ不備 | 2.3 | スクリーンショットを実画面から取得し、説明文と実機能を一致させる。キーワード詰め込みをしない |
| IAP 回避・外部決済誘導 | 3.1.1 / 3.1.1(a) | デジタル機能の解放は必ず IAP。外部購入リンクはエンタイトルメント承認と対象ストアフロントを確認してから |
| プライバシー違反 | 5.1 | 目的文字列を具体的に書く。ポリシー・Privacy Labels・実収集データを一致させる。ATT・同意フローを実装する |
| アカウント削除機能なし | 5.1.1(v) | アカウント作成があるならアプリ内削除導線を実装する |
| 最低機能不足（Web ラッパー） | 4.2 | ネイティブ機能（通知・オフライン・OS 統合）を付加し、アプリ固有の価値を作る |
| 模倣・スパム | 4.1 / 4.3 | 既存人気アプリの外観・名称に寄せない。バリエーションは1アプリに統合する |
| サードパーティログインのみ | 4.8 | プライバシー保護されたログイン選択肢（Sign in with Apple 等）を併設する |
| 規制業種を個人で提出 | 5.1.1(ix) | 金融・医療・ギャンブル等は法人の Developer アカウントで提出する |

公式: https://developer.apple.com/jp/distribute/app-review/

## 提出前チェックリスト

コードフリーズ後、App Store Connect 提出前に全項目を確認すること。

**機能・品質**
- [ ] 最新 OS の実機で全主要フローをテストし、クラッシュ・バグを修正済み
- [ ] プレースホルダコンテンツ（仮画像・仮テキスト・ダミーデータ）が一切ない
- [ ] サポート URL・プライバシーポリシー URL を含む全リンクが機能する
- [ ] IPv6 のみのネットワークで動作する
- [ ] IAP の購入・リストア（restore purchases）が正常動作する

**メタデータ**
- [ ] アプリ名は現行の文字数上限を確認し、商標・他アプリ名・無関係キーワードを含まない
- [ ] スクリーンショットは実際の利用中画面で、対応デバイスタイプ分用意した
- [ ] 説明文が実機能と一致し、追加課金の必要性を明示している
- [ ] カテゴリと年齢制限指定（age rating）を正直に設定した

**審査向け情報（App Store Connect）**
- [ ] サインインが必要な場合、デモアカウントの ID/パスワードを審査メモに記載した
- [ ] 特別な構成・ハードウェアが必要な機能は手順またはデモ動画を提供した
- [ ] 新機能・非自明な機能の説明を審査メモに書いた
- [ ] 連絡先情報が最新である

**プライバシー**
- [ ] プライバシーポリシーが現行のプライバシー条項に必要な収集内容・用途・共有先・保持/削除・同意撤回を網羅している
- [ ] 機密データアクセス API すべてに具体的な目的文字列（`NS*UsageDescription`）を設定した
- [ ] Privacy Nutrition Labels が実際の収集データ・SDK の収集内容と一致している
- [ ] トラッキングを行う場合、ATT の許可フローを実装した
- [ ] アカウント作成機能がある場合、アプリ内アカウント削除を実装した

**必要書類（該当する場合）**
- [ ] 子ども向け + サードパーティ広告 → 広告サービスのポリシー文書
- [ ] 医療機器連動 → 規制当局の認可証
- [ ] 著作権/商標コンテンツ使用 → 使用許諾の証憑
- [ ] ギャンブル・VPN・実金取引 → ライセンス/認可証

公式: https://developer.apple.com/jp/distribute/app-review/

## 審査への対応（Resolution Center・再審査・アピール）

### 審査フロー
1. **提出** → App Store Connect の「App Review」ページでステータスを確認できる（ステータス変更時はメール通知あり）。
2. **審査時間**: 通常は迅速だが、複雑・新規性の高い問題を含むアプリ、ガイドライン違反を繰り返した履歴のあるアカウントは長くなる。
3. **承認後**: 未来のリリース日を設定した場合はその日まで非公開。全ストアフロントへの反映には最大24時間かかる。

### 🔴 リジェクトされたときの対応手順
1. **理由の精読**: App Store Connect でリジェクト理由と該当ガイドライン条項を確認する。推測で直さない。
2. **App Review チームへの返信（Resolution Center / App Review メッセージ）**: 理由が不明確な場合は App Store Connect からレビューチームに直接質問できる。修正方針の妥当性を再提出前に確認することも可能。
3. **修正して再提出**: 指摘事項を完全に解決してから再提出する。部分対応での再提出は再リジェクトと審査時間増加を招く。
4. **異議申し立て（Appeal）**: ガイドラインに準拠していると考える明確な根拠がある場合、App Review Board にアピールできる。1案件につきアピールは1件のみ。レビューチームからの追加情報リクエストにすべて回答してから提出すること。

### 優先審査（Expedited Review）
- 重大なバグ修正（再現手順を添える）や日時が確定したイベント関連リリースなど、時間的制約が重大な場合のみリクエスト可能。
- 濫用するとリクエスト自体が却下されるようになるため、本当に必要な場合に限定すること。

### バグ修正版の救済ルール
- 既に配信中のアプリのバグ修正提出は、法的・安全性の問題を除き、ガイドライン違反を理由に配信を遅延されない。バグ修正を先に通し、指摘事項は次回提出で対処する旨をレビューチームに伝えられる。

### 推奨
- リジェクトへの返信は事実ベースで簡潔に。感情的・攻撃的な応答は 5.6 行動規範の観点でも不利。
- ガイドライン自体への改善提案チャネルも用意されている（承認可否とは別枠）。
- 再現困難な機能（特定ハードウェア・地域限定・IoT 連携）は最初からデモ動画を添付すると往復回数が減る。

### よくある注意点
- リジェクト後に**別の新規アプリとして提出し直す**のは審査プロセスの不正操作とみなされ、以後の審査が遅くなる要因になる。同一提出物内で修正・返信すること。
- アピールは「修正が面倒」な場合の迂回路ではない。ガイドライン解釈の相違を論証できる場合のみ使う。

公式: https://developer.apple.com/jp/distribute/app-review/

## 実装エージェント向けサマリー（最短の遵守判断）

1. **デジタル販売はIAPを原則とし、現行の例外を確認する**。外部決済や誘導は、対象storefront、アプリ区分、承認済みentitlement、現行ガイドラインを確認してから実装する。
2. **データ利用前に必要な開示と同意を確定する**。目的文字列・ATT・プライバシーポリシー・Privacy Labelsを実装とセットで揃え、同意が必要な条件は現行公式で確認する。
3. **完成した提出物を用意する**。プレースホルダ・リンク切れ・デモアカウント欠落は審査上の重大な問題になる。
4. **メタデータは実物どおり**。誇張・キーワード詰め込み・実画面でないスクリーンショットは書かない。
5. **アカウント、ログイン、UGCの現行条件を確認する**。アカウント削除、同等のログイン選択肢、UGCのフィルタ・報告・ブロック・連絡先を、該当する現行条項に照合する。

公式: https://developer.apple.com/jp/app-store/review/guidelines/
