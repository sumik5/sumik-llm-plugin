# 計測フレームワーク実践ガイド

プロダクト成功を定量化するための計測設計・メトリクス選択・ダッシュボード構築の実践知識。

---

## 1. HEART Framework 実践ガイド

HEART Frameworkは、ユーザー体験を5つの次元で定量化する計測体系。**各次元に対してGoal（目標）→ Signal（観測指標）→ Metric（計算式）** の順で定義する。

### HEART 5次元とKPI対応表

| 次元 | 意味 | 代表KPI | 計測方法 |
|------|------|---------|---------|
| **Happiness** | ユーザーの主観的満足度 | 事後NPS・平均評価スコア | イベント後アンケート（1〜5段階評価） |
| **Engagement** | アクティブ利用度 | DAU、WAU、DAU/WAU比率 | ログイン・操作イベント集計 |
| **Adoption** | 新規取り込み速度 | 新規アクティブユーザー数/週 | 初回アクション完了率 |
| **Retention** | 継続利用率 | Day-1/Day-7/Day-30 リテンション | コホート別返還率 |
| **Task Success** | コアタスク完遂率 | タスク完了率、エラー率、所要時間 | ファネル完了追跡 |

### HEART 運用の4原則

1. **全5次元をセットで監視する** — 1次元だけ改善して他を悪化させる施策を検出できる
2. **Happinessは定性データで補完する** — スコアだけでなく自由記述を必ずセットで収集
3. **TaskSuccessはコアジャーニーに限定する** — 離脱ファネルと抱き合わせで設計
4. **四半期ごとに閾値を見直す** — 成長フェーズによって「正常値」が変わる

### HEART ダッシュボード構成例

```
┌─────────────────── HEART Dashboard ──────────────────────┐
│  Happiness    │  4.2 / 5.0    │ ▲ 0.1 vs 前週             │
│  Engagement   │  DAU/WAU 38%  │ ▼ 2pt（要注意）           │
│  Adoption     │  +1,240 新規  │ ▲ 12% vs 先月             │
│  Retention    │  Day-30: 42%  │ → 横ばい                  │
│  Task Success │  完了率 81%   │ ▲ 3pt vs 先月             │
└──────────────────────────────────────────────────────────┘
```

## 2. コア vs 虚栄メトリクスの判断基準

### 判断フレームワーク：3つの問い

メトリクスを採用する前に3つの問いで分類する。

| 問い | コアメトリクス（◎） | 虚栄メトリクス（✗） |
|------|-------------------|-------------------|
| 施策の変更でコントロールできるか？ | YES — UIの変更で直接動く | NO — 外的要因で上下する |
| ビジネス目標（収益・リテンション・成長）に直結するか？ | YES — 相関が実証されている | NO — 見栄えは良いが成果と無関係 |
| セグメント別（デバイス・コホート・実験variant）に分解できるか？ | YES — 原因の特定が可能 | NO — 集計値のみで改善行動を取れない |

### よく混同されるメトリクスの対比

| 虚栄メトリクス（避ける） | コアメトリクス（使う） | 理由 |
|----------------------|---------------------|------|
| 総ダウンロード数 | インストール後7日間のアクティブ率 | インストールだけでは実際の利用を測れない |
| MAU（月次アクティブユーザー） | DAU/WAU スティッキネス比率 | MAU急増でもDAU/WAUが下がれば習慣形成に失敗 |
| 総コンテンツ投稿数 | 投稿→シェア変換率（SSR: Share-to-Save Ratio） | 量より質・継続エンゲージメントが重要 |
| ページビュー数 | セッション深度（訪問あたりの平均アクション数） | PVはコンテンツの質と無関係に増減する |
| 掲示板投稿数 | 返信スレッド深度・返信率 | 一方的な投稿の量は議論の質を示さない |

### 診断シナリオ

**シナリオA（機能リリーススパイク）**: 施策投入後に虚栄メトリクスが急増したが、コアメトリクスが微動 → 表面的利用であり継続機能の整備が必要

**シナリオB（マーケティングキャンペーン）**: WAUが30%増加したがDAU/WAUが低下 → 好奇心流入であり習慣化に失敗している。エンゲージメント機能（通知・推薦）の強化を検討

## 3. 先行/遅行指標のキャリブレーション

### 定義

| 種別 | 定義 | 特徴 | 典型例 |
|------|------|------|-------|
| **遅行指標（Lagging）** | 過去の結果を反映 | 変化を遅れて検出。成果確認に使う | Day-30リテンション率、LTV、NPS |
| **先行指標（Leading）** | 将来の結果を予測 | 早期に行動変容を示す。介入判断に使う | 初回48時間のSCR、初セッション中の複数コアアクション |

### 先行指標の発見プロセス（4ステップ）

1. **仮説列挙**: 「〇〇行動が早期に起きたユーザーは長期リテンションが高い」という仮説を3〜5本立てる
2. **コホート分割**: 行動あり/なしでコホートを分けて30日後のリテンション率を比較
3. **相関係数検証**: R² > 0.65 を目安に統計的相関を確認（全セグメント横断で保持されることを確認）
4. **ダッシュボードに昇格**: 確認された先行指標をプライマリ指標として配置し、遅行指標はセカンダリに降格

### 先行指標の相関パターン例

```
早期行動（D0〜D2）                    → 遅行指標（D30）
─────────────────────────────────────────────────────
初回48h内のSCR成長 +1.5pt             → D30リテンション +7%  (R²=0.78)
初回セッション中の複数コアアクション実行 → 翌週のシェア数 +20%  (R²=0.65)
W1のBED（掲示板エンゲージメント深度）+10% → D30平均セッション時間 +12% (R²=0.71)
```

### 定期キャリブレーションのタイミング

| タイミング | アクション |
|-----------|-----------|
| 四半期ごと | 先行指標と遅行指標の相関係数を再計算。R²が0.5を下回れば指標を入れ替える |
| 大型機能リリース後 | 新機能がユーザー行動パターンを変えた場合、先行指標の有効性を再評価 |
| 成長フェーズ転換時 | グロース期→成熟期など、KPIの優先順位自体を見直す |

## 4. Time-to-First-Value (TTFV) メトリクス設計

「最初の価値実感」を数値化するメトリクス群。オンボーディング最適化の核になる。

### TTFV の2種類の定義

| 定義 | 開始点 | 終了点 | 用途 |
|------|--------|--------|------|
| **登録起点 TTFV** | アカウント作成時刻 | 最初のコアアクション完了時刻 | アクティベーションフロー全体の改善 |
| **初回インタラクション起点 TTFV** | 初回ダッシュボードログイン時刻 | 最初のコアアクション完了時刻 | ログイン後UXの改善（メール認証遅延を除外） |

> **使い分け**: 2種類を同時に計測することで「アカウント活性化の摩擦」と「実際の利用摩擦」を切り分けられる。

### TTFV が改善すべき指標として機能する条件

1. コアアクション（PR提出・初注文・初送金など）がプロダクト価値と直結している
2. ユーザーが自律的にコアアクションに到達できる（誘導フローが存在する）
3. データ収集に必要なイベント（登録タイムスタンプ・コアアクションタイムスタンプ）が計装されている

### TTFV 短縮のための介入パターン

| 問題 | 原因 | 介入例 |
|------|------|-------|
| 登録起点 TTFV が長い | メール認証・権限設定の摩擦 | ソーシャルログイン導入、段階的権限要求 |
| 初回インタラクション起点 TTFV が長い | UI上でコアアクションへの導線が不明確 | ウェルカムウィザード、プログレスバー |
| セグメント間で TTFV 差大 | 特定デバイス・OS のバグ or UXギャップ | セグメント別ファネル分析でボトルネック特定 |

## 5. 計測スキーマ設計手法

### イベント命名規則

一貫した命名体系により、分析クエリの複雑さを低減し、全チームが共通言語で話せる。

```
フォーマット: <Category>.<Action>.<Context>
                  ↑          ↑         ↑
              機能エリア  ユーザー行動  オプション詳細

例:
  Pin.Clicked.Button        # ピンのボタンをクリック
  Share.Initiated.Popover   # シェアポップオーバーを起動
  Board.Viewed.MainPage     # ボードのメインページを閲覧
  User.Registered.Email     # メール登録完了
  Onboarding.Completed.Step3 # オンボーディング3ステップ完了
```

### 標準イベントペイロード構造

すべてのイベントに共通のペイロード構造を強制することで、BI ツールとの統合を簡素化する。

```json
{
  "event": "<Category>.<Action>.<Context>",
  "timestamp": "ISO 8601 UTC",
  "userId": "一意ユーザーID（認証済み）",
  "sessionId": "セッションID",
  "properties": {
    // イベント固有のメタデータ（イベントタイプごとに明示的スキーマを定義）
    "entityId": "操作対象のID",
    "pageLocation": "発生画面・ページ",
    "referrerEntityId": "参照元エンティティID（任意）",
    "device": {
      "type": "mobile | tablet | desktop",
      "os": "プラットフォーム + バージョン",
      "appVersion": "アプリバージョン"
    }
  },
  "context": {
    "language": "ロケール",
    "experimentVariant": "A/Bテストvariant名（実験参加時）",
    "trackingVersion": "スキーマバージョン"
  }
}
```

### スキーマ設計の6つのベストプラクティス

| プラクティス | 内容 |
|------------|------|
| **動詞駆動の Action 命名** | `Clicked`, `Created`, `Viewed`, `Errored` など過去形動詞を使う |
| **スキーマバージョン管理** | `trackingVersion` フィールドで後方互換性を保証 |
| **A/B variant タグ付け** | `experimentVariant` を全イベントに乗せてセグメント分析を可能にする |
| **Nullable フィールドの明示** | NULL を許容するフィールドを明記し、欠損を「データなし」と区別する |
| **フロントエンドとバックエンドのスキーマ共有** | 1つの定義から両側の検証ルールを生成（JSON Schema 推奨） |
| **コラボレーションチェックポイント** | スキーマ変更は PM・エンジニア・アナリストの3者レビューを必須とする |

## 6. エンゲージメントダッシュボード設計原則

### ダッシュボード3パターンと用途

| パターン | 目的 | 主要チャート | 読み手 |
|---------|------|------------|-------|
| **ファネル分析** | 離脱ポイントの特定 | 縦型ファネル + 各ステップ間の離脱率 | PM・デザイナー |
| **コホート保持マトリクス** | 施策効果のコホート別検証 | ヒートマップグリッド（コホート × 経過日数） | PM・アナリスト |
| **時系列 + 実験マイルストーン** | 施策投入と指標変化の因果関係確認 | ローリング7日平均 + 実験開始日の縦マーカー | ステークホルダー・リーダーシップ |

### ダッシュボード設計の7原則

1. **スキャナビリティ優先** — 最重要KPIをページ上部に配置。折り返し以降は深掘り用
2. **コンテキスト付き数値** — 実数値だけでなく「前週比」「目標値対比」を必ず添える
3. **インタラクティブフィルタ** — 期間・デバイス・コホート・実験variantで動的絞り込み
4. **閾値による条件付き書式** — KPIが目標を超えたら緑、下回ったら赤に自動着色
5. **実験アノテーション** — 施策投入日をチャートに縦線マーカーで記録
6. **ロール別ビュー** — PM用（深掘り）・マネジメント用（サマリ）を分けて管理
7. **データ鮮度の明示** — 「最終更新: ○○分前」を常に表示し古いデータでの意思決定を防ぐ

### コホートリテンションマトリクスの読み方

```
           Day-1   Day-7   Day-30
Cohort W1:  48%     31%     22%    ← ベースライン
Cohort W2:  52%     35%     24%    ← 機能改善後。全期間で+2〜4pt改善
Cohort W3:  55%     38%     28%    ← さらなる改善

色分けルール:
  ■ 緑：リテンション率 > 50%（短期）/ SCR > 15%
  ■ 黄：目標の80〜100%
  ■ 赤：目標の80%未満
```

## 7. メトリクス階層設計（North Star → Core → Leading）

```
Tier 1: North Star Metric（北極星指標）
  └─ プロダクトの長期健全性を最も正確に反映する1指標
     例: 月次アクティブ課金ユーザー数 / 取引成立件数 / 解決チケット数

Tier 2: Core Metrics（コアメトリクス）
  ├─ Retention（Day-7/Day-30リテンション）
  ├─ Engagement（DAU/WAU比率、セッション深度）
  └─ Monetization（コンバージョン率、ARPU）

Tier 3: Leading Indicators（先行指標）
  ├─ 初回セッション内のコアアクション数
  ├─ 48時間以内のエンゲージメント深度
  └─ オンボーディング完了率

Tier 4: Input Metrics（施策入力指標）
  ├─ A/Bテスト metric（機能別転換率・操作完了率）
  └─ ファネルステップ別完了率
```

**設計の鉄則**: 下位指標の改善が上位指標の改善につながることを、コホート分析で事前に検証してから採用する。

## クイックリファレンス：よくある計測の落とし穴

| 落とし穴 | 症状 | 対処 |
|---------|------|------|
| 指標の多すぎ | ダッシュボードに100以上のチャートが並ぶ | Tier構造で「今見るべき指標」を5〜7本に絞る |
| 遅行指標しか見ていない | 問題が起きてから2〜4週間後に気づく | 先行指標を特定し、48時間以内に異常検出できる仕組みを作る |
| セグメント未分解 | 全体平均が良くても悪化セグメントを見逃す | デバイス別・コホート別・Variant別の常時監視を徹底 |
| 実験Variantの未タグ付け | 施策効果とオーガニック変動を切り分けられない | 全イベントに `experimentVariant` を付与 |
| 計測スキーマの未整備 | 指標が計算できず後から計装し直す | リリース前にイベントスキーマレビューを必須化 |
