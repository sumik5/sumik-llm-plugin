# デザインシステム コンテンツ策定ガイド

DSに含める各種コンテンツ（ガイドライン）の策定プロセス。

---

## 1. コンテンツ分類と優先順位

デザインシステムに含めるコンテンツは大きく4種類に分類できる。着手順序は組織の課題に応じて変わるが、一般的な優先度を以下に示す。

| ガイドライン種別 | 含まれるコンテンツ | 着手推奨度 |
|-----------------|-------------------|-----------|
| **ブランドガイドライン** | ロゴ・タイポグラフィ・色・パーソナリティ・イラストレーション | ★★★ 最優先（DSの基盤） |
| **UIガイドライン** | デザイン原則・デザイントークン・コンポーネント・デザインパターン | ★★★ プロダクト開発に即効 |
| **コンテンツガイドライン** | 文章・用字用語・UIテキスト・マイクロコピー | ★★ 品質安定に直結 |
| **運用ガイドライン** | デザインレビュー・ユーザビリティテスト・ブランドコンポーネント | ★ 組織成熟後に整備 |

**優先順位の判断フロー:**

```
プロダクトが複数ある・チームが拡大している
    → UIガイドライン（デザイントークン・コンポーネント）を先行

ブランドのばらつきが問題になっている
    → ブランドガイドライン（色・タイポグラフィ）を先行

文章やUIテキストが統一されていない
    → コンテンツガイドライン（用字用語・UIテキスト）を先行
```

## 2. ブランドガイドライン策定

### 2-1. 運営理念の言語化

DSを始める際の「宣言」。「このDSがどういう目的で作られたか」を言葉にする。

| 含める要素 | 具体例 |
|-----------|--------|
| 誰のためのものか | 対象者（デザイナー・エンジニア・全従業員など） |
| 運営スタンス | WIP精神・オープンな更新方針など |
| 組織との接続 | 会社のミッション・バリューとの関連付け |
| 利用者への期待 | フィードバックを得られる関係の構築 |

**作成のポイント:**
- 抽象度が高い文章になりがち → 第三者レビューで解釈のズレを確認する
- 社内外のステークホルダーが「自分ごと」にできる表現を選ぶ
- 完成を待たずWIPで公開する勇気を持つ

### 2-2. パーソナリティ定義の手法

ブランドが醸し出す雰囲気・印象を「人柄を表す言葉」で定義する。

**定義プロセス:**

```
1. 経営戦略（ミッション・ビジョン）との接続を確認
2. 競合・市場環境（外部環境）を踏まえた自社のポジショニング把握
3. 「ありたい姿」を言語化（単語レベル: 誠実・ポジティブ・わかりやすいなど）
4. 各単語に解釈の幅が出ないよう説明文を添える
5. 対象範囲を明示（プロダクト・コミュニケーション全般など）
6. 周期的に見直し（経営戦略の更新タイミングに合わせる）
```

**浸透施策の例:**

| 施策 | 効果 |
|------|------|
| ブランドをリードするデザイナーが実例を共有 | 「らしさ」の実例として機能 |
| デザインレビューにパーソナリティ観点を追加 | 日常業務に組み込む |
| トンマナ可視化ワークショップ（半年ごと）| 認識のズレを早期に発見 |
| ブランドブック・ブランドサイトの制作 | 解像度の高い共通認識を形成 |

### 2-3. ロゴガイドライン

| 対応レベル | 含める内容 |
|-----------|-----------|
| **最低限** | アイソレーション・配色・禁止事項 |
| **充実版** | 上記 + 媒体別バリエーション・NG事例・印刷仕様 |

**策定のスタンス:**
- ガイドラインは「法律」ではなく「守るとロゴが適切に伝わる状態になるもの」
- 対応できない例外が発生したら、その都度ロゴ管理者と相談して調整する
- FigmaなどにOK/NG事例を蓄積する仕組みを最初から作る

### 2-4. タイポグラフィ選定基準

フォント選定で考慮する3つの観点:

| 観点 | チェックポイント |
|------|----------------|
| **1. ブランド印象** | パーソナリティとの一致（和らかさ・クールさなど） |
| **2. 媒体汎用性** | ウェブ・印刷・スライドすべてで使えるか |
| **3. アクセシビリティ** | 読みやすさ・文字サイズを考慮しているか |

**実装時の追加考慮点:**
- ウェブフォントの読み込みコスト（代替フォント・総称フォントの方針）
- 多言語サポートが必要な場合の文字種カバレッジ
- 全従業員が利用できるか（システムフォントの検討）

### 2-5. カラーシステム設計（役割ベース）

**カラーセットの構成例:**

| カラーセット | 役割 | 策定の起点 |
|------------|------|-----------|
| Primary Brand Color | ブランドの中心色（1色） | すでにある「ブランドカラー」を基準に |
| Secondary Brand Colors | Primaryを引き立てる色群 | ムードボード → 世界観の言語化 → 数値化 |
| Extended Colors | 汎用カラーパレット | 用途を想定 → 必要な色数・トーン数を決定 |

**アクセシビリティ対応:**
- 背景色と文字色の組み合わせすべてでWCAG基準のコントラスト比を確認
- 背景がグレーになるパターンも忘れずに検証
- 段階的に移行する場合はお試し期間を設けてフィードバックを収集

**ブランドカラーをプロダクトに展開する際のポイント:**
- コミュニケーション用パレットとプロダクト用パレットは並行して設計する
- デザイントークンを介して変更が全プロダクトに自動反映される仕組みにする
- 色の変更は影響範囲が大きいため、段階的移行計画を立てる

### 2-6. イラストレーション基準

**システム化の効果:**

| 効果 | 具体的な価値 |
|------|------------|
| 情報処理速度の向上 | 見る側が情報をイメージしやすくなる |
| ブランドイメージの解像度向上 | 印象の安定・一貫性 |
| 全員が強度ある資料を作れる | 非デザイナーでも品質を維持できる |

**作成のステップ:**

```
1. 最小限のモチーフを決める（人物・建物・アイコンなど用途を絞る）
2. 汎用バリエーションを展開（特定の用途に収まるラインを設定）
3. テイストを複数パターン出してブランドに合うものを選択
4. 作成ガイドラインを定める（新規作成時の基準を明文化）
5. 権利関係をイラストレーターと確認・契約
6. 利用ガイドラインを明示（著作権・利用範囲・READMEファイル）
```

## 3. UIガイドライン策定

### 3-1. デザイン原則の作り方（チームワークショップ）

**作成プロセス（7ステップ）:**

| ステップ | 内容 |
|---------|------|
| 1. 関係者を集め役割分担 | 作成者（少数精鋭）と、レビュアー（第三者）に分ける |
| 2. 方向性・目的を確認 | スタンス型・特性型・価値観型など方向性を統一 |
| 3. 観点・キーワードを洗い出す | 付箋ワーク（Miroなど）で発散 |
| 4. グルーピング・収れん | 4〜6項目に絞り込む |
| 5. 標語化 | 短く印象的なフレーズにする |
| 6. 説明文を作成 | キャッチ + ボディコピーの2段構成 |
| 7. 第三者レビューでブラッシュアップ | 意図が伝わるか確認・修正を繰り返す |

**作成時の注意:**
- 折衷案にならないよう合議ではなく少数精鋭で初稿を作成する
- 会社のバリューとの差別化（バリューで言えないプロダクト固有の観点を扱う）
- 「意思決定に迷ったときのものさし」になるか常に確認する

### 3-2. デザイントークン体系

**2種類のトークンの役割:**

| トークン種別 | 性質 | 用途例 |
|------------|------|-------|
| **プリミティブトークン** | 具体的な値。最も原子的 | `blue-500 = #3B82F6`、`16px` |
| **セマンティクストークン** | 意味・役割を表す | `color/text/primary`、`spacing/md` |

**カラートークン設計の観点:**

```
色名 + 番号（blue-500など）でプリミティブを定義
    ↓ 参照
メイン・アクセント・テキスト・コンポーネントなどの役割で分類
（セマンティクストークン → Single Source of Truth）
```

**タイポグラフィトークン:**
- フォントサイズ: ブラウザデフォルト（16px）を基準に調和数列で設定
- Tシャツサイズ（S/M/L）でセマンティクストークン化すると直感的
- 値は`px`ではなく`rem`でブラウザのフォントサイズ変更に対応

**余白トークン:**
- 基準値（8pxや16pxなど）の倍数で設定すると整合性が取りやすい
- プリミティブ（1/4〜4文字分など）とセマンティクス（Tシャツサイズ）の二層構造
- 余白には意図があるべき → 情報の関係性を視覚的に表現する

**行送りトークン:**
- WCAG 2.1「テキストの間隔」達成基準を最低ラインとして設定
- TIGHT（見出し）/ NORMAL（本文）/ RELAXED / NONE（ラベル）など用途別に定義

### 3-3. コンポーネント設計

**カテゴリ分類例:**

| カテゴリ | 代表コンポーネント | 設計の注意点 |
|---------|-----------------|------------|
| ボタン系 | Button・Dropdown | variantで役割を切り替える設計が拡張性高い |
| 表示系 | Card・Table・Badge | 状態（空・エラー・ローディング）のパターン定義 |
| フォーム系 | Input・Checkbox・Select | HTMLネイティブ要素の挙動から乖離させない |
| レイアウト系 | Stack・Cluster・Grid | 余白トークンを内部で使い一貫したスペーシング |
| ナビゲーション系 | Header・TabBar・Pagination | プロダクト横断で使えるか慎重に判断 |
| オーバーレイ系 | Dialog・Tooltip | 濫用を避けるガイドラインも同時に整備 |
| 状態系 | Loader・NotificationBar | 状態ごとの視覚的フィードバックを明確に定義 |

**命名規則の原則:**
- ユーザーの「行動・目的」を反映した名前を付ける（見た目・コンテンツタイプではない）
- チーム内で暗黙に使われていた独自名称はドキュメント化してSingle Source of Truthにする

**早すぎる抽象化を防ぐ:**
- 実際のプロダクト開発の課題を起点にコンポーネントを追加する
- 見た目が似ているだけで役割が違うものを共通化しない
- デザインツール上の机上設計に閉じず、実装での再利用可能性を常に確認する

### 3-4. デザインパターン

**2種類のパターンの定義:**

| パターン種別 | 定義 | 例 |
|------------|------|-----|
| **レイアウトパターン** | 頻出する画面レイアウト | 一覧テーブル・詳細ビュー・フォーム画面 |
| **インタラクションパターン** | 頻出する操作フロー | 一括選択・権限による表示制御・ページング |

**パターン定義のポイント:**
- 社内で口伝・暗黙知になっているものをドキュメント化することに大きな価値がある
- 「名前をつける」ことで認識のズレを防ぎコミュニケーションコストを削減できる
- 実態に即した名前 + 印象的で呼びやすい言葉の選択が重要

## 4. コンテンツガイドライン策定

### 4-1. 文章ガイドライン

**ライティングガイドラインの構成要素:**

| 要素 | 内容 |
|------|------|
| トーン&ボイス | ブランドパーソナリティに基づく文章のトーン |
| 用字用語集 | 表記揺れを防ぐ用語リスト（漢字・ひらがな・カタカナの使い分け） |
| 文章の基本ルール | 一文の長さ・主語の省略可否・読点の使い方など |
| チェックリスト | 文章完成後の自己レビュー用 |
| Before/After実例 | 具体的な改善例（理解しやすさが格段に上がる） |

**用字用語集の作り方:**

```
1. 社内で揺れやすい表現を洗い出す（スプレッドシートで収集）
2. 対象範囲を決める（プロダクトUI・ヘルプ・マーケティング等）
3. 議論が必要なものは定例で合意形成
4. textlint等の校正ツールに組み込み自動化
5. DSのガイドラインとして公開（社外公開も有効）
```

### 4-2. UIテキストガイドライン

**ラベル・エラーメッセージ・マイクロコピーの策定ポイント:**

| UIテキスト種別 | 策定のポイント |
|--------------|--------------|
| **ラベル（入力項目名）** | 曖昧さをなくす。「名前」より「姓名」など具体的に |
| **エラーメッセージ** | 「何が」「なぜ」「どうすれば」を含める。行き止まりにしない |
| **マイクロコピー** | ユーザーの不安を先回りして解消する |
| **空状態のテキスト** | 次のアクションを促す。「〇〇はまだ登録されていません」+ 追加ボタン |

## 5. 運用ガイドライン策定

### 5-1. デザインレビュープロセス

**レビューの種類と目的:**

| レビュー種別 | 対象 | 観点 |
|------------|------|------|
| 開発プロジェクトレビュー | 機能開発中の画面 | 仕様・UX・パーソナリティとの整合 |
| グループレビュー | デザイナー定例 | 職能横断のフィードバック |
| コンポーネントレビュー | DS本体の更新 | 実装との整合・影響範囲の確認 |

**レビュー依頼テンプレートの必須項目:**
- 関連リンク（デザインツール・チケットなど）
- レビュー概要・背景
- レビューの目的（何を判断してほしいか）
- レビュー完了条件
- レビューしてほしい点 / しなくてよい点

### 5-2. ユーザビリティテスト計画と実施

**計画フロー（8ステップ）:**

| ステップ | 内容 |
|---------|------|
| 1 | テストの「目的」を確認（何を明らかにしたいか） |
| 2 | プロダクトを理解する（全体像・開発意図を把握） |
| 3 | 検証内容を具体化する（観察したい操作を絞る） |
| 4 | 被験者の条件を検討する（役職・習熟度・会社規模など） |
| 5 | テスト設計（シナリオ・タスク・イントロダクション） |
| 6 | テスト環境を用意する（モックアップ・本番環境など） |
| 7 | 役割分担を決める（ファシリテーター・記録係など） |
| 8 | 被験者のリクルーティングと連絡 |

**タスク設計のポイント:**
- スタートとゴールを明確に定義する（判定基準になる）
- 「ボタンを押してください」などの具体的操作指示にしない
- タスクは3工程程度に収める

### 5-3. ブランドコンポーネント（非開発者向けアセット管理）

**ブランドコンポーネントとは:**
デザイナー以外（営業・マーケティング・カスタマーサポートなど）が使う、スライド・資料・名刺などのテンプレートやアセット群。

| 提供方法 | 適した規模 |
|---------|-----------|
| ダウンロード可能なファイル（Figma・Keynote等） | 中小規模 |
| パスワード付き社内限定ページ | センシティブな社内ブランドアセット |
| 全社員に配布するオフライン媒体 | ブランドカラーなど普及促進したいもの |

## 6. アクセシビリティ統合

### 6-1. a11y方針の策定

| 考慮ポイント | 内容 |
|------------|------|
| **達成レベルの設定** | WCAG 2.1 AA（一般的な推奨）またはAAA（医療・行政など） |
| **プロダクト特性との整合** | toBかtoC・対象ユーザーの特性（高齢者・障害者の比率） |
| **組織のミッションとの接続** | a11y = ミッション実現の手段として位置付ける |

### 6-2. 簡易チェックリスト

```
□ カラーコントラスト比（テキスト/背景の組み合わせすべてで確認）
□ 代替テキスト（画像・アイコンに適切なalt/aria-labelがあるか）
□ キーボード操作（マウス不使用での全操作が可能か）
□ フォーカス状態の視覚的明示（フォーカスリングが見えるか）
□ エラーメッセージのアクセシブルな提供（色だけに頼らない）
□ 行送り（WCAG 2.1「テキストの間隔」達成基準1.4.12準拠）
```

### 6-3. 実装時の考慮事項

- デザイントークンのカラー設計段階でコントラスト比を確認する
- コンポーネントにa11yプロパティ（aria-*）を標準で組み込む
- 定期的なa11yレビューをデザインレビューに組み込む

## 7. 各コンテンツの「最低限 vs 充実版」

| ガイドライン | 最低限やるべきこと | 余裕があればやること |
|------------|-----------------|-------------------|
| **運営理念** | 目的・誰のためかを1ページに | ミッション・バリューとの関連図 |
| **パーソナリティ** | 3〜5単語 + 簡単な説明文 | トンマナ可視化ワークショップ・ブランドブック |
| **ロゴ** | アイソレーション・禁止事項 | 媒体別バリエーション・OK/NG事例集 |
| **タイポグラフィ** | 推奨フォント・代替フォント | ウェブフォント方針・タイポスケール |
| **カラー** | Primary + Secondary + 簡単な使い方 | Extended Colors・アクセシビリティ検証・印刷色見本 |
| **デザイン原則** | 3〜5項目の標語 + 説明文 | 判断フローチャート・適用事例 |
| **デザイントークン** | カラー + フォントサイズ | 余白・行送り・影・アニメーション |
| **コンポーネント** | 最頻出コンポーネント10〜20個 | 全コンポーネント + Storybook |
| **文章ガイドライン** | 用字用語集 + 基本ルール | Before/After実例・textlint辞書 |
| **UIテキスト** | エラーメッセージの型 | マイクロコピー集・空状態パターン |
| **デザインレビュー** | 依頼テンプレート | 種別別ガイドライン・完了条件定義 |
| **ユーザビリティテスト** | 計画書テンプレート | シナリオ・タスク作成ガイド |
| **アクセシビリティ** | コントラスト比チェック方針 | チェックリスト・実装ガイド |
