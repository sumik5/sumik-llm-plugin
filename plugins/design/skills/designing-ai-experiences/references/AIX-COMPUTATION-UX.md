## 主要用語解説

AIシステムを設計する際に知っておくべき基本的な技術概念。エンジニアとの会話や処理プロセスの分析に必要な共通語彙。

| 用語 | 定義 |
|------|------|
| **Tokenization** | テキスト入力を「トークン」と呼ぶより小さな単位（単語・部分語・文字）に分割するプロセス。モデルはトークンを数値IDに変換して処理する |
| **Context Window** | モデルが一度に処理できるトークンの最大数。この上限が、会話履歴や入力情報のうちどれだけを考慮できるかを決定する |
| **Latent Space** | モデルがデータを表現する高次元の圧縮空間。単なる内容ではなく「他のデータとの関係性」で表現される。意味的に似た入力は空間上で近くに配置される |
| **Inference** | 学習済みモデルを新しい入力に適用して出力を生成する行為。データ駆動の推測・予測プロセス |
| **Routing** | リクエストを処理するモデル・ツール・ステップを決定するプロセス。分類・ヒューリスティクス・学習済み決定木が使われる |
| **RAG（Retrieval Augmented Generation）** | 生成前に外部知識ソースを検索し、取得した情報をモデルの内部知識と組み合わせる技術 |
| **Agentic Workflow** | AIシステムが複雑なタスクを自律的に計画・実行・監視する多段階プロセス。ツール使用と長時間の意思決定を含む |
| **Model Switching** | タスク要件・ユーザー設定・パフォーマンス最適化に基づき、異なる専門モデルを選択する計算プロセス |
| **Temperature** | 生成時のランダム性を制御するパラメータ。低値は一貫性・保守性、高値は創造性・多様性をもたらすが誤りのリスクも増す |

## 処理パイプライン3ステップ

ユーザーがプロンプトを送信してから応答が生成されるまでの3段階プロセス。大部分はユーザーには不可視であり、デザイナーにも不透明な場合がある。

```
ユーザー入力
    ↓
Step 1: 入力処理と準備（Input Processing）
    ↓
Step 2: ルーティング（Routing）
    ↓
Step 3: 生成と推論（Generation & Inference）
    ↓
出力
```

### Step 1: 入力処理と準備

入力がモデルに届く前に、様々な前処理が行われる。

| 処理 | 内容 |
|------|------|
| **テキストのTokenization** | テキストを部分語に分割し数値IDに変換（例: "Summarize" → ["Summ", "arize"] → [15234, 8567]） |
| **Vector Embedding** | トークンを高次元空間に配置。意味的に近い概念が幾何学的に近い位置に置かれる |
| **画像エンコード** | Convolutional Neural Network（CNN）が階層的にパターンを検出（エッジ→形状→テクスチャ→オブジェクト）し、意味的ベクトルに変換 |
| **音声エンコード** | 音声をスペクトログラム（時間×周波数×強度の視覚的グラフ）に変換後、ベクトル化 |
| **動画エンコード** | フレームごとの空間次元と、フレーム間の時間次元を組み合わせて処理 |

**設計への示唆**: ユーザーが入力するテキストはそのままモデルに渡されるのではなく、数学的に変換される。この変換で意味の微細な差異が失われる可能性があることを念頭に置いた入力UI設計が重要。

### Step 2: ルーティング

システムはリクエストをどのモデルやツールに振り分けるかを決定する。

**ルーティングの例（PDF+プレゼン生成の場合）:**

```
PDFアップロード
    ↓ ドキュメントパーサーでテキスト抽出
    ↓ 要約モデルで内容を圧縮
    ↓ スライド生成システムへ受け渡し
    ↓ デザインモデルで視覚フォーマット
    ↓ 必要に応じてグラフ/画像用の別モデルを呼び出し
```

| ルーティングタイプ | 例 | 注意点 |
|------------------|-----|--------|
| **内部知識から生成** | 「フランスの首都は？」→ 学習データで回答 | レスポンスが速い |
| **外部検索(RAG)** | 「今日のパリの天気は？」→ 外部データを取得 | より最新・引用可能だが遅延増加 |
| **マルチモデル連携** | 複数ステップのタスク → 各専門モデルが分担 | 各ステップが失敗の可能性を持つ |

**設計への示唆**: 外部検索を伴う場合、レイテンシーが大幅に増加する可能性がある。ルーティング判断をユーザーに透明化する設計（「外部ソースを検索中...」等）が信頼構築に有効。

### ルーティングのUX設計への影響

**ルーティング透明性の設計**

ルーティングはユーザーには不可視だが、その結果はレイテンシー・情報品質・エラー率として体験される。ルーティングタイプに応じた透明性の設計で信頼を構築する：

| ルーティングタイプ | ユーザーへの表示 | 設計目的 |
|-----------------|--------------|---------|
| **内部知識生成** | 表示不要（高速・シームレス） | 速度の印象を維持 |
| **外部検索（RAG）** | 「外部ソースを検索中...」のインジケータ | レイテンシー増加の予期と情報の新鮮さの価値を伝える |
| **マルチモデル連携** | ステップ別の進捗表示 | 各ステップの完了感と複雑性の提示 |

**マルチステップルーティングの設計原則**

PDF→プレゼンテーション生成のような複数ステップのルーティングでは、各ステップが独立した失敗点を持つ：

```
PDFアップロード
    ↓ テキスト抽出（失敗例：フォーマット非対応PDF、パスワード保護）
    ↓ 要約モデル処理（失敗例：コンテキストウィンドウ超過）
    ↓ スライド生成システム（失敗例：構造化データの変換エラー）
    ↓ デザインモデル（必要な場合・グラフ/画像処理）
```

各ステップを可視化することで：ユーザーはどのステップが失敗したかを特定できる。部分的な成功が保存でき最初からやり直す必要がない。長いパイプラインへの信頼感が醸成される。

**外部検索（RAG）のUXパターン**

外部検索を伴うルーティングでは、クエリ変換→複数データベース検索→フィルタリング→内部知識との統合という複合処理が行われる：

- **ソース引用の表示**: 「参照したソース」を明示し、情報の出所への透明性を確保
- **内部知識 vs. 外部検索の区別**: 鮮度と信頼性のトレードオフをユーザーが理解できるよう示す
- **検索失敗時のフォールバック**: 「リアルタイム情報が取得できませんでした。学習データに基づいてお答えします」のような明示的なフォールバック表示

### Step 3: 生成と推論

実際の出力生成プロセス。モデルタイプによって方法が大きく異なる。

| モデルタイプ | 生成方法 | 特徴 |
|-------------|---------|------|
| **言語モデル（LLM）** | 1トークンずつ確率分布からサンプリング | 即興演奏に近い。coherenceは文脈全体から生まれるが、事実的正確さは保証されない |
| **拡散型画像モデル** | ランダムノイズから段階的に精緻化（彫刻家が石から形を掘り出すように） | 安定性と制御性が高い。段階的プロセスのため進捗状況の可視化が可能 |
| **その他の画像モデル** | 単一パスまたは視覚トークンの逐次生成 | 各モデルでレイテンシー特性が異なる |

> LLMは「最も確率の高い次のトークン」を逐次選択して生成する。これが「自信満々の誤情報（ハルシネーション）」を生む根本原因。モデルは事実的正確さではなく、「それらしさ」を最適化している。

## コンテキスト組み立て

現代のAIシステムは入力単独で処理せず、追加コンテキストを組み合わせてから処理する。

**コンテキストに含まれる情報:**

| 要素 | 説明 |
|------|------|
| 会話履歴 | 直近のやり取りの記録 |
| ユーザー設定 | 過去セッションから保存された嗜好 |
| 知識ベース | 関連ドキュメントや外部データソース |
| 環境情報 | 時刻・場所・デバイスタイプ等 |
| システムプロンプト | 製品側が設定する動作指示 |

**設計への示唆**: コンテキスト組み立ては計算負荷と出力品質の両方に大きく影響する。ユーザーには不可視だが、同じプロンプトでも文脈の違いで出力が変わることを、UI設計で適切に示す必要がある。

### コンテキスト組み立てのUX設計パターン

コンテキスト組み立てはユーザーには不可視だが、同じプロンプトで異なる結果が生じる根本原因となる。この不可視性をUIで扱うことが信頼構築の核心。

**コンテキスト可視化の3段階**

| 可視化レベル | 手法 | 適した場面 |
|------------|------|---------|
| **最小開示** | 「現在のドキュメントを参照中」のステータス表示 | 一般向け製品・コンテキストが明白な場合 |
| **部分開示** | コンテキストソースのラベル（「メモリから」「ドキュメントから」「Web検索から」） | プロフェッショナルツール・引用の重要な場面 |
| **完全開示** | コンテキストウィンドウの内容を展開可能な形で表示 | デバッグ・開発者ツール・高信頼性が必要な場面 |

**コンテキストの矛盾への対応**

ユーザー設定・ドキュメント内容・会話履歴が矛盾する場合（例：以前の「フォーマルで」という指示 vs. 現在の「カジュアルに書いて」）、システムはどちらを優先するかをユーザーに示す必要がある：

- 矛盾を検出した場合は確認クエリを提示（「以前の設定ではフォーマル指定がありますが、今回はカジュアルで生成しますか？」）
- 以前の指示を上書きする場合はその旨を明示し、ユーザーが意図的に変更できる状態を保つ

**コンテキストウィンドウ上限の設計対応**

コンテキストウィンドウには上限があり、長い会話や大きなドキュメントでは古い情報が失われる：

- セッションの「記憶の崖」をユーザーに事前に伝える（「この会話は非常に長くなっています。早い段階の内容が参照されにくくなっている可能性があります」）
- 重要な情報のピン留め機能（コンテキストに常に含める情報として優先度付け）を提供する
- 「このセッションでは文書の最初の部分のみを参照しています」のような透明な通知で、ユーザーが出力の限界を正しく理解できるよう支援する

## レイテンシー管理戦略

ユーザーは待つことが嫌い。しかし研究によると、「なぜ待つのか」と「進捗が見える」の2点が理解できると、許容度が大幅に向上する。

### レイテンシーの知覚閾値

| 閾値 | ユーザー体験 | 設計対応 |
|------|-----------|---------|
| **0.1秒以下** | 即座に感じる | 特別なフィードバック不要。結果をそのまま表示 |
| **1秒以下** | 思考の流れを保てる | 遅延を認識するが集中は継続。厳密なフィードバックは不要 |
| **10秒以下** | 注意維持の限界 | 進捗インジケータか時間見積もりが必要 |
| **10秒超** | 他タスクに移行 | 非同期化 + 通知機能が必要 |

### タスクタイプ別の設計優先度

| タスクタイプ | 優先事項 | 設計戦略 |
|------------|---------|---------|
| **即時制御・トランザクション** | 高速性 | 遅延を最小化。即座の確認フィードバック |
| **クリエイティブ・オープンエンド** | 関与継続 | 多少の遅延を許容。進捗の楽しさを演出 |
| **検索・学習・意思決定** | 明瞭性 | 速さより分かりやすさを優先 |

## レイテンシーの視覚的手法

待機中の沈黙そのものがユーザーへのメッセージ。どのように「待つ」を見せるかが製品の信頼性の印象を形成する。

| 手法 | 最適ユースケース | 注意点 |
|------|--------------|--------|
| **即座のフィードバック** | 全ケース共通の基本原則 | アクション確認後、意味のある進捗更新に速やかに移行すること |
| **スピナー（不確定インジケータ）** | 短時間・時間不明のプロセス | 活動を示すが進捗情報は提供しない |
| **プログレスバー（確定インジケータ）** | 時間見積もりが可能なプロセス | 不正確・停止したプログレスバーは満足度を大きく下げる |
| **スケルトンスクリーン** | 予測可能な構造化コンテンツ | レイアウトを先示しすることで知覚待機時間を短縮 |
| **非同期（ノンブロッキング）** | 長時間の処理 | ユーザーが他のタスクを継続できる。完了通知システムが必要 |
| **結果の段階的開示** | 複数情報の取得 | 部分的な結果から先に表示することで即時価値を提供 |
| **リアルタイムストリーミング** | LLM テキスト生成 | トークン逐次表示で知覚レイテンシーを大幅に削減 |
| **待機理由の説明** | 長時間かつ複雑な処理 | 「何をしているか」を言語化することでユーザーの信頼を維持 |

> **過剰なレイテンシー表現に注意**: ストリーミング・タイミング調整・プログレスインジケータは戦略的に使用する。過多だと体験が煩雑になり、AIの能力に対する印象を薄める。

## 位置・速度の設計パターン

### 位置の設計原則

| 状況 | 推奨パターン |
|------|-----------|
| 出力領域が空の場合 | 大きく中央に配置したローダーで待機を明示 |
| 推論ステップが可視化される場合 | 微細な状態インジケータに抑制（他のUI要素が進捗を示しているため） |
| レスポンスが部分的に開始された場合 | ユーザーが読んでいる内容を妨げない非侵入的な配置 |
| 詳細な推論が展開される場合 | アコーディオン要素で折り畳み。アクティブなステップのみ可視化 |

**ブランドコンシステンシー**: 異なるモデル・ツール呼び出しに応じたレイテンシー表現のビジュアルファミリーを定義しておく。「このプロダクトは考えているのか？反省しているのか？生成しているのか？」という問いへの一貫した答えを持つこと。

### 速度の設計原則

| ケース | 推奨アプローチ |
|--------|-------------|
| 処理時間が予測可能 | 「通常15秒かかります」のような時間明示 |
| 単一タイプの出力 | モデル処理時間の告知が有効 |
| オープンエンドな入力 | 15秒〜2分いずれの場合も同等に許容できるアニメーションを設計 |

## 通知と3層通知設計

タスクに複数のステップがある場合、ユーザーが離れて戻ってきても状況を把握できる通知設計が必要。

### 通知の基本原則

通知はユーザーがオプトインできる設計が望ましい。

| 原則 | 内容 |
|------|------|
| **最上位情報のみ** | 最も関連性の高い情報だけを含める |
| **見つけやすさ** | アプリ内・システムトレイ・スマホ画面等、ユーザーが自然にいる場所に表示 |
| **簡潔性** | 一目で意味が分かる brevity |
| **アクション要否の明確化** | 一時停止タスク（承認が必要）vs 進行中タスク（確認のみ）を明示 |

### 3層通知設計

| 層 | 目的 | 含む情報 | ユーザー状態 |
|----|------|---------|-------------|
| **Tier 1: Overview** | 現状把握の維持 | 現在のステージ・経過時間・アクティブなリソース | 離れて概要確認 |
| **Tier 2: Detail** | 理解と検証 | 完了/進行中/残タスクの詳細・推論の説明・使用ツールとデータソース | 詳細確認・方向性の検証 |
| **Tier 3: Full Record** | 学習・デバッグ・説明責任 | すべてのアクションの完全な記録・グループ化と平易な言語での整理 | 深く探求・コンプライアンス確認 |

**設計の鍵**:
- Tier 1は「選択」—必須でない情報は省略し、後で深掘りできるよう残す
- Tier 3は使うユーザーが少くても、存在するだけで信頼性への印象を高める
- 各Tierは異なる注意レベルに対応し、ユーザーが好きな深さで関与できる設計

## 4つのプロセスタイプ別UX設計

ユーザーがどのような認知状態にあるかによって最適なレイテンシーUXが異なる。

### トランザクション（支払い・認証・予約確認）

即座の確認が必要。ユーザーは高集中・目的志向の状態にある。

| 要素 | 設計指針 |
|------|---------|
| **フィードバック速度** | 即座の確認が必須。空白があるとユーザーが行動を繰り返す可能性がある |
| **UI戦略** | Optimistic UI（サーバー確認前に成功表示）+ 段階的フィードバック（"Step 1/3: カード確認中"） |
| **長時間処理時** | 推定待機時間または段階的ステータス更新 |

### 会話エージェント（チャットボット・バーチャルアシスタント）

会話のリズムが重要。速すぎても遅すぎてもダイアログの幻想を壊す。

| 要素 | 設計指針 |
|------|---------|
| **自然な応答タイミング** | 2〜3秒が多くのユーザーに自然に感じられる |
| **必須要素** | タイピングインジケータ（ユーザーをインタラクションに繋ぎ止める） |
| **複雑なタスク** | 部分的なレスポンスのストリーミング + 確認エコー（"了解、調べますね"）でユーザーを維持 |

### クリエイティブコンテンツ生成（テキスト・画像・音楽・コード）

探索的マインドセット。意味のある進捗感があれば遅延を許容できる。

| 要素 | 設計指針 |
|------|---------|
| **スピナー回避** | 長時間タスクには静的スピナーを避け、プログレスバーを優先 |
| **固定時間の提示** | 拡散モデルなど計算が確定的な場合は合理的な時間見積もりを提供 |
| **拡張表現の機会** | 処理中のナレーションやゲーム風ウェイト画面でブランド個性を表現可能（文脈を選ぶこと） |

### 自律タスク（エージェントAI・バックグラウンド処理）

ユーザーが離れて戻ってくることを前提とした設計。継続的な信頼関係の構築が必要。

| 要素 | 設計指針 |
|------|---------|
| **セットアップ段階** | 想定待機時間と複雑さの告知が必須 |
| **定期的なチェックイン** | 一定間隔で進捗更新または次ステップの承認を求める |
| **ストリーミング進捗** | ドキュメントスキャン・メール整理等のバックグラウンド処理の可視化 |
| **自律モードの提供** | 常時承認不要の自律モード + 完了時通知のオプション |

> **研究の知見**: AIの処理遅延は逆説的な効果を持つことがある。ユーザーが複数の問いを同時に考える時間が生まれ、生成される質問量と制御感が向上した事例がある。「待機設計」は単なる時間の穴埋めではなく、思考促進の機会でもある。

## エラー設計

AIエラーには「検出可能なエラー」と「検出不可能なエラー」という2種類があり、それぞれ異なる設計アプローチが必要。

### エラー設計の基本原則

| 原則 | 内容 |
|------|------|
| **理解可能にする** | 平易な言語で「何が問題か」「次に何をすべきか」を説明。技術的ジャーゴンや曖昧な表現（"Something went wrong"）を避ける |
| **回復可能にする** | 修正・取り消し・リトライが可能な設計。ユーザーの作業やコンテキストを保全 |
| **予防する** | 適切なデフォルト値・バリデーション・インタラクションフレームで、ありがちなエラーをそもそも起きにくくする |
| **優雅に劣化させる** | システムの一部が利用不可能な場合でも、縮退機能や代替経路を提供 |
| **システム状態を明確化** | 障害が一時的か永続的か、そもそも起きているのかをユーザーが把握できるようにする |

### AIエラーの2分類

| 分類 | 説明 | 設計アプローチ |
|------|------|-------------|
| **検出可能なエラー** | システムが障害を認識できる（タイムアウト・レート制限・ファイル処理失敗等） | 明確なエラーメッセージ + 具体的な次のステップ |
| **検出不可能なエラー（ハルシネーション等）** | システムが問題に気づかず自信を持って誤情報を出力 | 出力への疑念を促すUI摩擦・検証手段・ヒューマンレビューの導線 |

> **重要な設計転換**: 検出不可能なエラーが存在するため、「エラー処理」だけでなく「批判的評価を促す設計」が必要。美しいフォーマット・流暢な文章・自信ある口調であっても誤りの可能性があることを、UIが常に示唆する必要がある。

### エラーメッセージの実践

| 避けるべき表現 | 改善例 |
|-------------|--------|
| "Error 413: Request Entity Too Large" | "ファイルをアップロードできませんでした。PDFまたはJPEGをお試しください" |
| "CORS policy violation" | "接続に問題が発生しました。少し待ってから再試行してください" |
| "Something went wrong" | "応答の生成中にエラーが発生しました。入力を確認して再試行してください" |

### ミッドストリームエラーとリカバリー

AIの生成中断はトラディショナルなエラー回復とは異なる。

| 従来のエラー回復 | AIのエラー回復 |
|---------------|-------------|
| 問題を特定して修正→同じ操作を再試行 | 同じ入力を再実行しても異なる出力が生成される |
| オートセーブで確定的に復元可能 | 生成済みの出力は確定的ではなく、完全な復元は不可能 |
| 修正 → 再送信のシンプルなフロー | プロンプト・会話履歴・部分的な出力を保全し、新しい方向性を探索できる足場を提供 |

**チェックポイントパターン**: Gitのコミットに相当する「チェックポイント」機能を提供することで、生成物の安定バージョンを保存し、実験を安全に行えるようにする。

### モメンタムの維持

エラー後の復帰をスムーズにするためのUI戦略。

- 入力の再編集を容易にするプロンプト履歴（↑キーで前のコマンドに戻るmacOSターミナルに相当）
- 異なるモデルや機能でのクイックリトライオプション
- エラーパターンの学習を支援するエラー情報の可視化（どの入力タイプが繰り返しエラーになるか）

## ユーザー確認の原則（AskUserQuestion）

**判断分岐がある場合、推測で進めず必ずAskUserQuestionツールでユーザーに確認する。**

確認すべき場面:
- どのプロセスタイプ（トランザクション/会話/クリエイティブ/自律タスク）に設計するか
- レイテンシーのトレードオフ（速さ vs 明瞭さ）をどう優先するか
- エラー処理に非同期パターンを採用するかどうか
- 3層通知設計のどの層まで実装するか

確認不要な場面:
- 即座のアクション確認（すべてのインタラクションに基本原則として適用）
- エラーメッセージに平易な言語を使うこと（常にベストプラクティス）
