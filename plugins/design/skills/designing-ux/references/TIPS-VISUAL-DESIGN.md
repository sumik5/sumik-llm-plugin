## 赤の使い方パターン

赤は「危険・注意・活気」の印象を持つ。定番的に使われるだけに、意図を取り違えられるリスクも高い。

### いつ赤を使うか / 避けるか

| 場面 | 判断 | 理由 |
|------|------|------|
| **数値がネガティブな方向に動いた**（直帰率増加、支出超過） | ✅ 使う | 増減ではなくポジティブ/ネガティブで色を決める |
| **支出と収入の区別が業務上重要**（会計ソフト） | ✅ 使う（支出=赤） | 誤入力を防ぐための区別。間違えると業務上問題になる |
| **支出がほぼすべて**（家計簿など、赤が画面を占める） | ❌ 避ける | 画面全体が赤くなりユーザーの気分を下げる |
| **削除確認ダイアログの最終ボタン** | ✅ 使う | 破壊的行為の直前に限定してのみ注意を促す |
| **削除アイコン（一覧・常時表示）** | ❌ 避ける | 削除を促したいわけではない。目立たせる必要がない |
| **インジケータ：容量・予算が上限に近づいている** | ✅ 使う（多い→赤） | 「増えすぎ」を警告するパターン |
| **インジケータ：バッテリー・ガソリンが下限に近づいている** | ✅ 使う（少ない→赤） | 「足りなくなる」を警告するパターン |
| **エラーメッセージ** | ✅ 使う | 注意喚起の定番。ただし色だけに頼らない（次節参照） |

### 赤を使う際の注意事項

- **色だけで情報を伝えない**: 赤と緑は色弱の方に区別されにくい。アイコン・ラベル・形状も併用する
- **使う範囲を限定する**: 赤は「ここに注目してほしい」シグナル。広い面積や高頻度で使うと効果が薄まる
- **ユーザー体験の文脈を優先する**: 同じ「支出」でも、厳格な会計（赤=適切）か日常の家計管理（赤=不要）かで判断が変わる

## 配色の実装テクニック

### テーマカラー決定時に同時に決める色

テーマカラーだけを先に決めてそれ以外を後から雑に決めると、全体の調和が崩れる。
以下の色を**テーマカラーと同じタイミング**でトーンを揃えて決める。

| 色の用途 | 例 | 注意点 |
|---------|-----|--------|
| エラーメッセージ色 | 赤系 | テーマカラーとの親和性を確認 |
| 成功・完了メッセージ色 | 緑系 | 目的: 「問題なし」の伝達 |
| 操作不可 (disabled) の色 | グレー系 | 前景と背景のコントラスト比を確保 |
| 警告・注意色 | 黄系 | エラー色と区別できる色相を選ぶ |

### 構成と配色をセットで検討する

- カラーパレットだけを抽出して検討しない
- 実際の画面構成（一覧・詳細・フォーム等）の**ワイヤフレームに色を乗せた状態**で検討する
- 色の面積・隣接する色・コンテンツとの対比も同時に確認できる

### 無彩色の使い方

| 状況 | 推奨パターン | 理由 |
|------|------------|------|
| 背景色に明るいグレーを使いたい | テーマカラーをごく少量混ぜた有彩色 | 純粋な無彩色は無機質で寂しく見える |
| disabled状態・ステップナビ（非アクティブ）を表現したい | 背景色に合わせた**透過**（不透明度を下げる） | 単独の灰色にするより背景に自然になじむ |
| 色数を増やしたくない | 既存テーマカラーに透明度を調整して再利用 | 色の体系を壊さずに調和を保てる |

**透過 vs グレー の使い分け**:
- 背景が白/単色 → グレーでも問題なし
- 背景が有彩色・グラデーション → 透過を使うと自然に馴染む

## 説明UIの3パターン

ユーザーへの説明をどのUI形式で伝えるかは、「いつ・どれくらいの頻度で読む必要があるか」で決まる。

### パターン比較テーブル

| パターン | 表示タイミング | 向いている場面 | 注意点 |
|---------|--------------|--------------|--------|
| **❶ 条件付き表示**（コーチマーク・初回モーダル） | 特定条件でのみ（初回起動・機能初利用時） | 画面をシンプルに保ちたい / 最初に一度伝えれば十分 | あとから見返せない。繰り返し確認が必要な情報には不向き |
| **❷ 常時表示**（インライン説明文） | 常に表示 | 必ず目に入れてほしい / 操作と同時に読む必要がある | 文章量が多いと画面が複雑になる。分量に注意 |
| **❸ オンデマンド表示**（ツールチップ・ヘルプアイコン） | ユーザーが能動的に操作したとき | 全員に見せる必要はないが困ったときに参照できれば十分 | 「どこをタップすれば表示されるか」を分かりやすくする必要がある |

### パターン選択の判断フロー

```
この説明を見逃した場合、操作が破綻するか？
  ├─ Yes → ❷ 常時表示
  └─ No → 操作後に振り返ることがあるか？
              ├─ No → ❶ 条件付き表示（初回のみ）
              └─ Yes → ❸ オンデマンド表示（ツールチップ等）
```

### 補足: 条件付き表示 vs 常時表示の使い分け例

選択数に制限がある場合（例: 都道府県3つまで）:
- 選択数が**少ない（3は十分少ない）** → 4つ目を選んだときにアラート（❶）で十分
- 選択数が**多くなりやすい**（市区町村 / ジャンル選択等）→ 制限を常に画面に表示（❷）

## アニメーション・トランジションの判断基準

動きはユーザーの「楽しさ」体験に直結する。ただしやりすぎると逆効果になるため、目的を持って使う。

### コンテンツ変化による活気の演出

| 手法 | 効果 | 採用すべきコンテキスト |
|------|------|----------------------|
| **リアルタイム数値の表示**（ユーザー数・クチコミ数） | サービスの賑わいを伝える | UGCサービス / マーケットプレイス |
| **日次更新コンテンツの掲示**（今日の見どころ等） | 毎日訪れる理由を作る | メディア / キュレーションサービス |
| **他ユーザーの行動表示**（「今X人がカートに入れています」） | 社会的証明で行動を促す | EC / 予約サービス |

注: 変化する情報を増やすとナビゲーション導線が下に押し下げられる。ハンバーガーメニューへの移行とセットで検討する。

### タップ・クリック時のフィードバックアニメーション

| 場面 | 推奨アニメーション | 避けること |
|------|-----------------|----------|
| **いいね・反応ボタン** | 縮む→弾む + 放射状ライン | 単純な色変化のみ（物足りない） |
| **破壊的アクション（削除・送信）** | アニメーション最小化 or なし | 楽しい演出（目的と逆行する） |
| **操作完了フィードバック** | 軽微なスケール変化 + チェックアイコン | 長すぎるアニメーション（操作の邪魔になる） |

**原則**: アニメーションに「意味」を持たせる。何が起きたかを伝えながら楽しさを両立させる。

### 画面遷移トランジション

| 遷移の種類 | 推奨パターン | 設計の意図 |
|-----------|-----------|-----------|
| **下層への遷移**（詳細画面） | 右からスライドイン | 「進む」方向の認知モデルと一致 |
| **タブ切り替え** | 横スライド / ページめくり風 | タブの順序・空間的位置関係を伝える |
| **モーダル表示** | 下から上へのスライドアップ | 「現在の文脈の上に重なる」ことを表現 |
| **完了後の画面戻り** | 逆方向スライド（右→左） | 「戻る」方向の認知モデルと一致 |

**共通の注意点**:
- トランジション時間は短く（200〜350ms が目安）。長いと操作のテンポが悪くなる
- アニメーションはアクセシビリティ設定（`prefers-reduced-motion`）を尊重する
- プラットフォームの標準トランジションを無視した独自表現は認知コストを上げる

## まとめ: 判断の優先順位

1. **ユーザー体験の文脈を最優先**: 同じ手法でもサービスの目的・ユーザー層によって適否が変わる
2. **色・動きに頼りすぎない**: 色弱対応・motion 設定を考慮し、視覚以外の手段も組み合わせる
3. **全体の調和で判断**: 個別要素の美しさより、画面・サービス全体のトーン統一を優先する
4. **少なく・厳選して使う**: 赤もアニメーションも「ここぞ」という場面でのみ効果を発揮する
