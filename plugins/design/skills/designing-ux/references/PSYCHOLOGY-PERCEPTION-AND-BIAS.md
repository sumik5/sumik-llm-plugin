# 知覚とバイアス

ユーザーの知覚は客観的な事実をそのまま反映しない。過去の経験・現在のコンテキスト・近未来の目標という3つの力が常に知覚を歪める。UIデザイナーはこの「歪み」を理解し、それを逆用することでより直感的なインターフェースを構築できる。

---

## 1. 認知バイアス

### 1.1 知覚的プライミング

直前に与えられた刺激（プライム）は、その後の知覚に強く影響する。同じ視覚情報でも、先に「建物」という文脈を与えられた人と「文字」という文脈を与えられた人では、まったく異なるものを見る。

**UIデザインへの影響:**
- ユーザーは「今のコンテキスト」に沿ったものを知覚しやすく、コンテキストと不一致な要素は見落とされやすい
- エラーメッセージや重要通知は、ユーザーの現在のタスクフローと文脈的に一致した位置・タイミングで表示する
- オンボーディング画面では、後続UIに対するプライミングを意図的に設計できる

### 1.2 知覚フレーム（なじみのパターン）

人は繰り返し経験した状況に対してフレーム（期待パターン）を持つ。そのフレームに基づいて「あるはずのもの」を知覚し、「ないもの」を見落とす。

**主な現象:**
- ウィザードの最後のページだけボタン配置が変わっても、ユーザーは前ページと同じと思いこむ
- フォーム内で「送信」ボタンの色や形が他フォームと異なると見つけにくくなる
- Webサイトに「ナビゲーションバー・ロゴ・検索ボックス」がない場合、ユーザーは混乱する

| フレームの種類 | 具体例 | UIデザインへの示唆 |
|-------------|--------|----------------|
| プラットフォームフレーム | iOS/Androidの操作慣習 | OS標準パターンから逸脱する際は明示的な説明が必要 |
| サービスカテゴリフレーム | ECサイトはカート・検索・商品一覧が前提 | 業種別の標準レイアウトを踏襲する |
| アプリ内フレーム | 同アプリ内で統一された操作体験 | 画面間で一貫したコントロール配置を保つ |

**設計原則:** ユーザーの既存フレームに合わせる（「現実」をフレームに寄せる）。フレームを破る場合は明確なサインを提供する。

### 1.3 習慣化

同じまたは酷似した刺激を繰り返し受けると、その刺激への感度が低下する。

**UIでの具体例:**
- 「本当によいですか？」という確認ダイアログを繰り返し表示すると、ユーザーは内容を読まずにボタンをクリックするようになる
- 常時表示されるバナーや警告は徐々に無視されるようになる
- SNSの通知バッジも多すぎると習慣化し、チェックされなくなる

**設計指針:**
- 重要なメッセージは頻度を絞る（警告の乱用禁止）
- 本当に危険な操作の確認ダイアログは、内容を毎回変化させるか、操作コストを高める
- 定常表示する要素は重要度の低いものに限定する

### 1.4 注意の瞬き

何か重要なものに気づいた直後の約0.15〜0.45秒間、他の視覚・聴覚刺激をほとんど知覚できなくなる現象。

**UIでの具体例:**
- 画面が素早く連続して変化すると、後続の画面情報が見落とされる
- 重要な通知の直後に別の通知を表示しても、後者は認識されない可能性が高い

**設計指針:**
- 重要情報を連続して表示しない（間隔を空ける）
- 自動再生スライドショーの切り替え間隔は十分に長く取る（数秒以上）
- 操作フィードバックは単発で明確に示す

### 1.5 コンテキスト効果

周囲の状況（コンテキスト）が知覚を変える。同じ視覚要素でも、周囲の配色・形・文脈によって異なって見える。

**代表的現象:**
- 錯視: 周囲の形が中心要素のサイズ・向きを歪んで見せる
- マガーク効果: 視覚情報が聴覚認識に影響する（口の動きが聞こえる音を変える）
- 文字認識: 同じ文字でも前後の文脈で異なる文字として読まれる

**設計への応用:**
- 配色は単独ではなく、周囲との関係で決定する（コントラスト比の確認）
- エラー状態・成功状態は周辺要素も含めて区別可能にする
- ボタンの「押せる」感はボタン単体でなく周囲との対比で決まる

### 1.6 目標指向バイアス

ユーザーは目標に関連する情報を優先的に知覚し、無関係な情報は（目の前にあっても）認識しない傾向がある。

**主なメカニズム:**
1. **誘導**: 目標達成に必要な刺激に向かって知覚が自動的に引き寄せられる
2. **ふるい分け**: 目標と無関係な刺激は前意識的に除去され、意識に上がらない

**具体例:**
- 「奨学金の情報を探す」タスク中は、「100ドルプレゼント」のバナーは目に入らない
- 工具箱から「ハサミ」を取り出すよう頼まれたユーザーは、他の工具を覚えていない
- カクテルパーティーでは会話相手の声だけを選択的に聴取できる

| 状況 | バイアスの影響 | 設計対応 |
|------|-------------|---------|
| 検索・タスク実行中 | 目標外の情報は無視される | 重要情報はタスクフロー内に埋め込む |
| 大人vs子供のユーザー | 大人のほうが目標絞り込みが強い | 周辺情報（クロスセル等）は子供向けに有効 |
| 初回利用vs熟練ユーザー | 熟練ほど目的に直線的に動く | 初回：探索を促す。熟練：効率的な直線経路を提供 |

### 1.7 UIデザインへの応用（第1章まとめ）

| 設計原則 | 理由 | 実装例 |
|---------|------|-------|
| あいまいさを排除する | 複数解釈可能な表示はバイアスで誤解される | ボタンの状態を影・色・形で明確に区別する |
| 一貫性を保つ | フレームによりユーザーは前ページと同じを期待する | 同じ機能のコントロールは全画面で同位置・同デザイン |
| ユーザー目標を把握する | 目標外の情報は認識されない | 主要アクションをページのメインフローに配置する |
| 光源の慣例に従う | 陰影のプライミングで立体感が変わる | ボタンの光源方向は画面左上が標準慣例 |

## 2. ゲシュタルト原則

人の視覚系は個別の要素ではなく「全体のまとまり」を自動的に知覚する。UIデザインでは、この傾向を利用してグループ・関係・階層を視覚的に表現できる。

### 2.1 近接（Proximity）

**原則:** 距離が近いオブジェクト同士は同じグループに属するように見える。

**適用:**
- フォームの項目名とフィールドを近くに配置（離すと関連が見えにくくなる）
- ボックスや区切り線を使わなくても、間隔差だけでグループ化できる
- 関連するボタン群は隣接させる（確認・キャンセルなど）

```
悪い例: 項目名  ────────────  フィールド（離れすぎ）
良い例: 項目名 フィールド（近接）
```

### 2.2 類同（Similarity）

**原則:** 見た目が似ているオブジェクト同士は同じグループとみなされる。

**適用:**
- 同種の操作（全プルダウンメニュー等）は外観を統一する
- 異なるカテゴリの要素には視覚的差異をつける（色・形・サイズ）
- ラジオボタン・チェックボックスは種別ごとに形を統一する

### 2.3 連続（Continuity）

**原則:** 途切れた部分があっても連続した形として知覚しようとする。

**適用:**
- スライダーは「1本の溝にハンドルが乗る」として認識される（分割した2つの溝ではない）
- 分断された視覚要素でも流れが感じられると1つの形として見える
- プログレスバーは連続性により「進捗」を直感的に伝える

### 2.4 閉合（Closure）

**原則:** 不完全なオブジェクトも完全体として補完して知覚しようとする。

**適用:**
- 「積み重なったオブジェクト」のアイコンは輪郭の一部を見せるだけで複数を示せる
- ページ下部でコンテンツが少し見えていると（スクロール示唆）、ユーザーは続きを補完して認識する
- アイコンのシルエットは完全な輪郭がなくても認識可能

### 2.5 対称性（Symmetry）

**原則:** 複雑な光景を最もシンプルで対称性のある解釈に変換する。

**適用:**
- 2Dの図形で3Dオブジェクトを表現できる（立方体の線図など）
- レイアウトは対称性を活用してシンプルに整理する
- 過度に非対称なデザインは認知負荷を高める

### 2.6 図と地（Figure & Ground）

**原則:** 脳は視野を「図（前景）」と「地（背景）」に自動的に分離する。

**特性:**
- 小さく別の色の領域は「図」、大きな領域は「地」として認識される傾向
- 図と地の切り替えで多義的な認識が生じることがある

**適用:**
- メインコンテンツの背後にテーマ画像を配置してコンテキストを伝える
- モーダル・ポップアップは下のコンテンツを「地」に変えて注意を誘導する
- 背景色と前景色のコントラストは図と地の区別を明確にする

```
活用例:
- 背景に薄い透かし模様    → テーマ・ブランド伝達
- モーダル表示時のオーバーレイ → 前景を図として強調
```

### 2.7 共通運命（Common Fate）

**原則:** 同時に同じ動作をするオブジェクトはグループとして認識される。

**適用:**
- 複数アイテムを選択してドラッグすると、全てが一体として動き「グループ」を表現
- アニメーションで時間変化の相関関係を示す（グラフの複数系列が連動する等）
- 同期してフェードイン/アウトする要素は関連性があると認識される

### 2.8 組み合わせと確認のプロセス

複数のゲシュタルト原則は同時に働き、相互に影響する。意図しない視覚的関係性が生じることがある。

**デザインレビュー手順:**
1. デザイン完成後、各法則を1つずつ念頭に置いて画面を確認する
2. 意図していないグループ・関係が生じていないか確認する
3. 特に「近接」と「類同」の組み合わせは誤ったグループ化を引き起こしやすい

| 法則 | 主な用途 | よくある失敗例 |
|------|---------|-------------|
| 近接 | フォームのグループ化 | 項目名とフィールドが離れすぎ |
| 類同 | 同種コントロールの統一 | 異なる機能のボタンが同じ色 |
| 連続 | スライダー・プログレスバー | 色変化でスライダーが分断して見える |
| 閉合 | アイコン・積み重ね表現 | 輪郭が不明瞭で何かわからない |
| 対称性 | 3D表現・レイアウト整理 | 非対称すぎて認知負荷が高い |
| 図と地 | モーダル・背景画像 | 背景と前景のコントラスト不足 |
| 共通運命 | 選択・アニメーション | 無関係要素が同タイミングで動く |

## 3. 構造の利用

視覚システムは秩序立った構造を知覚するよう特化している。情報を構造化して提示することで、ユーザーの読み取り速度と理解度を大幅に向上させられる。

### 3.1 簡潔化と繰り返しの排除

散文（連続した文章）より構造化された表示のほうが、速く・楽に情報を読み取れる。

**悪い例（散文）:**
> あなたが予約なさったのは、10月15日火曜日の14:30にオークランドを出発し、10月15日火曜日の11:40にサンフランシスコに到着するユナイテッド航空237便です。

**良い例（構造化）:**
```
便名:   UA237 オークランド → サンフランシスコ
出発:   10月15日（火） 14:30
到着:   10月15日（火） 11:40
```

**繰り返し排除の原則:**
- ナビゲーションリンクで毎回同じ動詞を繰り返さない
- 選択肢の共通部分を見出しに出す（スキャン速度向上）
- 検索結果では同じメタデータを各アイテムに繰り返し表示しない

| 問題 | 対策 |
|------|------|
| リンクに同じ動詞が繰り返す | 見出しに動詞を括り出し、選択肢のみリスト化 |
| 検索結果に同じプレフィックスが並ぶ | 共通部分を除去し差分情報を前面に出す |
| フォームに「〜を入力してください」が毎行ある | プレースホルダーまたは見出しで一括説明 |

### 3.2 数字・コードの分割表示

桁数の多い数字や長いコードは、適切に分割することで読み取り精度と入力精度が向上する。

| データ種別 | 推奨分割形式 | 理由 |
|----------|------------|------|
| 電話番号 | (03) 1234-5678 | 慣例パターンに従い認知負荷低減 |
| クレジットカード | 1234 5678 9012 3456 | 4桁区切りで確認しやすい |
| 郵便番号 | 123-4567 | ハイフンで区別が容易 |
| 日付 | YYYY/MM/DD | 区切り文字で意味の区切りを明示 |

**入力フィールドの設計:**
- 複数フィールドに分割（各部が独立入力）
- または単一フィールドでスペース・ハイフンを許容し内部変換
- 「スペースや記号を入れないでください」という制約はユーザビリティの失敗

### 3.3 データ専用コントロール

特定データの入力・表示に特化したコントロールを使うことで、視覚的構造が明確になり、入力ミスが減る。

| データ種別 | 汎用テキスト | 専用コントロール |
|----------|------------|---------------|
| 日付 | テキスト入力（形式ミス多発） | カレンダーピッカー |
| 時刻 | テキスト入力（AM/PM混乱） | 時刻セレクター |
| 選択肢（少数） | テキスト入力 | ラジオボタン |
| 選択肢（多数） | テキスト入力 | ドロップダウン/セレクト |
| 数値範囲 | テキスト入力 | スライダー |
| メールアドレス | 単一フィールド | `type="email"` でバリデーション付与 |

### 3.4 視覚的な階層化

情報を階層化して表示することで、ユーザーは目的外の情報を読み飛ばし、必要な情報だけを素早く発見できる。

**階層化の3ステップ:**
1. 情報を明確な「節」に分割し、大きい場合はさらに小節に分ける
2. 各節・小節に内容を端的に示すタイトルをつける
3. 階層が上の節ほど視覚的に目立たせる（サイズ・太さ・位置）

**視覚的手段:**
```
見出し（H1）      ← 最大・最太・ページ最上部
  見出し（H2）    ← 次に目立つ
    本文          ← 読まなくてよい情報は一段下げる
    箇条書き      ← スキャン対応
```

| 適用場面 | 効果 |
|---------|------|
| 長文テキスト | 見出しでジャンプ読みを可能にする |
| 設定ダイアログ | セクション分けで目的の設定を即発見 |
| ナビゲーションメニュー | 近接・類同で親子関係を表現 |
| フォーム | フィールドグループに見出しをつける |
| 検索結果 | タイトル・説明・メタデータの視覚的差異をつける |

## 4. クイックリファレンス: 知覚バイアス対策チェックリスト

### デザインレビュー時

- [ ] コントロール・情報の位置は全画面で一貫しているか？（フレーム対策）
- [ ] 確認ダイアログは本当に必要な場合だけ表示しているか？（習慣化対策）
- [ ] 重要な通知は連続して表示していないか？（注意の瞬き対策）
- [ ] ボタン・リンクのデザインはユーザーの目標フローに沿っているか？（目標バイアス活用）
- [ ] 各ゲシュタルト原則で意図しないグループ化が発生していないか？
- [ ] 情報は散文でなく構造化（箇条書き・テーブル・階層）されているか？
- [ ] 長い数字・コードは適切に分割されているか？

### 代表的なUIバイアス対策パターン

| バイアス | 問題症状 | 対策パターン |
|---------|---------|------------|
| フレーム | ボタンが「予期した位置」にないと見落とされる | 業界標準・自アプリの慣例に従う |
| 習慣化 | 確認ダイアログが無視される | 重要操作の確認はコスト増加（テキスト入力等）で対応 |
| 目標バイアス | キャンペーンバナーがスキップされる | プロモーションはタスクフロー内か完了後に配置 |
| 近接違反 | フォームの項目名とフィールドの対応が不明 | 名前とフィールドを近接させ、グループ間に間隔を設ける |
| 類同違反 | 異なる機能のボタンが同じ見た目 | プライマリ/セカンダリ/危険を視覚的に区別 |
| 階層なし | 設定画面で目的の項目が見つからない | セクション見出し＋近接グループ化 |
