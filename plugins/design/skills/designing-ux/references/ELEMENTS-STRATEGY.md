# 戦略段階 — 製品目標とユーザーニーズの定義

UXデザインの成否は、最初の1行のコードが書かれる前に決まる。
戦略段階の問いは2つだけ：「自分たちはこの製品から何を得たいか」「ユーザーはこの製品から何を得たいか」。

---

## 1. 製品目標

### 1.1 ビジネスゴール

組織内の戦略目標を指す。「収益を増やす」「コストを削減する」のような一般的な表現では不十分。かといって「テキストによるリアルタイム通信ツールを提供する」のような実装レベルでも駄目。

| レベル | 問題点 |
|--------|--------|
| 一般的すぎる（例：「収益向上」） | 製品が目標達成にどう貢献するか説明できない |
| 具体的すぎる（例：「チャット機能を追加する」） | 戦略的な懸念を十分に説明できない |
| **適切** | 問題の全体像を理解したうえで、達成条件を明確にする |

**If X then Y**:
- If 目標が「何をするか」の記述になっている → then 「なぜそれをするか」「それで何が達成されるか」を問い直す
- If 抽象レベルが人によって違う → then 全員が共通認識を持てる戦略記述書を作成する

---

### 1.2 ブランドアイデンティティ

ブランドは視覚（ロゴ・カラー・タイポグラフィ）だけでなく、ユーザーがインタラクションを通じて組織に持つ「概念的な関連性と感情的な反応の集合体」である。

- ブランドの印象は「たまたまそうなった」か「意識的にデザインした」かどちらかしかない
- 非営利・政府・個人を含む、ウェブサイトを持つすべての組織はブランドを持つ
- ブランドアイデンティティを具体的な目標として成文化することで、ポジティブな印象を作れる確率が上がる

---

### 1.3 成功測定基準

**定義**: 製品発売後、目標やユーザーニーズを満たしているかを確認するための具体的な指標。

成功測定基準は以下の2つの目的を持つ：
1. プロジェクト期間中の意思決定に影響を与える
2. UXの取り組みの価値を証明する具体的な証拠になる

#### 測定基準の選択指針

| 目標タイプ | 測定候補 |
|-----------|---------|
| ユーザーエンゲージメント向上 | 登録ユーザー1人あたりの月間訪問数 |
| すばやいアクセス提供 | 訪問1回あたりの滞在時間（短くなること） |
| 情報提供型サポート | カスタマーサポートへの電話件数（減ること） |
| 営業支援 | 契約締結までの時間（短縮） |

**注意事項**:
- 測定基準を単独で見ると誤解を招くことがある。競合の広告キャンペーンや組織の評判など、ウェブサイト外の要因も把握する
- 測定基準は**デザインの選択によって形作られるユーザー行動**と明確に結びついている必要がある
- 広告収入最大化のためにナビゲーション階層を増やすような「ユーザーニーズと相反する設計」は長期的にはマイナスになる

---

## 2. ユーザーニーズ

「自分たちと同じような理想的なユーザーのためにデザインしている」という考えに陥りがちだが、作ったものを使うのは他の人々。誰のためにデザインするかを知ることが出発点。

---

### 2.1 ユーザーセグメンテーション

大量かつ多様なユーザーニーズを扱いやすくするため、共通の特性でユーザーを小さなグループ（セグメント）に分割する。

#### セグメント分割の軸

| 分類軸 | 内容 | 特徴 |
|--------|------|------|
| **デモグラフィック** | 年齢・性別・収入・学歴など | 一般的だが、同属性でも世界観が異なる場合がある |
| **サイコグラフィック** | 製品テーマへの態度・受け取り方 | デモグラフィックでは得られない洞察を提供 |
| **テクノロジー親和性** | ウェブ利用時間、最新ガジェットへの関心度 | パワーユーザーとテクノ恐怖症ではデザイン要件が異なる |
| **知識・経験レベル** | 対象ドメインの専門度 | 初心者と熟練者では情報の見せ方が逆になることも |
| **社会的・職業的役割** | 職種、家族内の役割 | 同じ情報でも立場によって必要なことが異なる |

**If X then Y**:
- If 2つのセグメントのニーズが**正反対**（例：初心者はステップバイステップ / 熟練者は全機能まとめたUI） → then 対象セグメントを一方に絞るか、同じタスクへの2経路を提供するか戦略的に決断する
- If 調査でセグメント内の差異が大きい → then サブセグメントに分割する
- If 2つのセグメントのニーズがほぼ同じ → then 統合してよい

セグメントの数は「異なるニーズのセット数」で決まる。恣意的に多くする必要はない。

---

### 2.2 ユーザビリティとユーザー調査

ユーザーニーズを理解するには、ユーザーが誰かを知り、データを集め、その行動・態度・知識を把握する必要がある。

#### 調査手法の選択マトリクス

| 手法 | 適用場面 | 深さ | コスト |
|------|---------|------|--------|
| **サーベイ** | 一般的な態度・認識の収集 | 浅い | 低 |
| **フォーカスグループ** | 複数ユーザーの意見収集 | 浅〜中 | 中 |
| **インタビュー** | 個人の考え方の深掘り | 中 | 中 |
| **タスク分析** | 特定タスクの実行プロセス調査 | 中 | 中 |
| **ユーザーテスト（完成品）** | UIの問題点発見 | 中〜深 | 中〜高 |
| **プロトタイプテスト** | 開発途中のデザイン検証 | 中 | 低〜中 |
| **コンテクスト探求** | 日常生活の文脈でユーザーを理解 | 深い | 高 |
| **カードソーティング** | ユーザーの情報分類方法の把握 | 中 | 低〜中 |

**調査設計の原則**:
- 「何を知りたいか」を明確にしてから手法を選ぶ
- 個々のユーザーに費やす時間と参加ユーザー数はトレードオフ
- 行動の「何を」を知りたいか、「なぜ」を知りたいかで手法が変わる

---

### 2.3 ペルソナの作成

ユーザー調査のデータに「顔と名前」を与えた架空のキャラクター。実際のユーザーのニーズを代表する。

**ペルソナの目的**:
- バラバラな調査データを人間として統合することで、デザイン判断時に「この人に役立つか？」と自問できる
- チームが常にユーザーを念頭に置いてデザインできる

**作成プロセス**:
1. ユーザー調査でセグメントを定義する
2. 各セグメントを代表するキャラクターに名前・年齢・職業・目標・背景を与える
3. 詳細は調査データと一致させる（キャラクターの細部は作り上げてよい）
4. 写真付きのプロフィールを作成してオフィスに掲示する

**If X then Y**:
- If セグメント間でニーズが大きく異なる → then それぞれのペルソナを作成する
- If デザイン判断に迷ったとき → then 「このペルソナはどう反応するか」と自問する
- If ペルソナが多すぎる → then セグメントを見直す（ニーズが似ているなら統合）

---

## 3. チームの役割とプロセス

### 3.1 ステークホルダーの巻き込み

戦略策定に関与すべき人々：

| 関与者 | 役割 | 注意点 |
|--------|------|--------|
| **ステークホルダー（上級意思決定者）** | 製品の最終的な戦略的方向性に影響 | 組織の意思決定構造によって誰が含まれるかが変わる |
| **一般従業員** | 現場から見た「何がうまくいくか」の知見 | 無視されがちだが最も実態を知っている |
| **ユーザー自身** | 外部からの要求の源 | 調査を通じて声を集める |

### 3.2 戦略記述書

製品目標とユーザーニーズを文書化したもの。単なる目標リストではなく、目標間の関係と組織全体のコンテクストを分析する。

**効果的な戦略記述書の条件**:
- 簡潔で要点が明確（ボリュームより明確さ）
- プロジェクトメンバー全員（デザイナー・開発者・PMなど）がアクセスできる
- 定期的に見直し、洗練させていく

**アンチパターン**:
- ❌ 重役だけに共有して担当チームが見られない
- ❌ 一度書いたら更新しない
- ❌ 誰も読めないほど長い

### 3.3 戦略と開発の関係

戦略が固まらないとプロジェクトが動けない、ではなく：
- **戦略は進化するもの**。プロジェクトを通じて洗練され続ける
- 体系的な見直しと改良を続けることで、継続的なインスピレーションの源になる
