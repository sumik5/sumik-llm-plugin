## API設計

Web API設計の2つの根本原則:
1. **仕様が決まっているものは仕様に従う** (HTTP仕様 RFC 9110等)
2. **仕様がないものはデファクトスタンダードに従う** (JSON, RESTful URI設計, OAuth 2.0)

### エンドポイント設計

優れたURIの6特徴: 短い・人間が読める・小文字統一・Hackable・実装非反映・ルール統一

基本パターン:

| 目的 | エンドポイント | HTTPメソッド |
|------|--------------|-------------|
| 一覧取得 | `/v1/resources` | GET |
| 新規登録 | `/v1/resources` | POST |
| 個別取得 | `/v1/resources/:id` | GET |
| 更新（全体） | `/v1/resources/:id` | PUT |
| 更新（部分） | `/v1/resources/:id` | PATCH |
| 削除 | `/v1/resources/:id` | DELETE |

- リソース名は複数形の名詞 (`/users`, `/orders`)
- URIに動詞を含めない
- ネストは2階層まで (`/users/123/orders`)

### レスポンス設計

- JSONをデフォルト (`Content-Type: application/json`)
- 不要なエンベロープで包まない（直接配列またはオブジェクトを返す）
- フラットな構造を維持
- 命名規則はAPI全体で統一 (camelCase or snake_case)

エラーレスポンス必須フィールド:
```json
{
  "code": "VALIDATION_ERROR",
  "message": "Invalid input parameters",
  "details": [{"field": "email", "issue": "Invalid email format"}]
}
```

### HTTPステータスコード（主要）

```
200 OK              - GET/PUT/PATCH成功
201 Created         - POSTでリソース作成成功
204 No Content      - DELETE成功
400 Bad Request     - 入力パラメータ不正
401 Unauthorized    - 認証が必要/失敗
403 Forbidden       - 権限不足
404 Not Found       - リソース不存在
422 Unprocessable Entity - バリデーションエラー
429 Too Many Requests - レートリミット超過
500 Internal Server Error - サーバ内部エラー
```

### バージョニング

URIパスにメジャーバージョンを埋め込む方式を推奨:
```
https://api.example.com/v1/users
```

バージョンを上げる変更: フィールド削除・型変更・必須パラメータ追加・エンドポイント削除
バージョンを上げない変更: フィールド追加・新エンドポイント追加・オプションパラメータ追加

### セキュリティ基本

- **HTTPS必須**: `Strict-Transport-Security: max-age=31536000; includeSubDomains`
- **OAuth 2.0**: Authorization Code Flow + PKCE (Web/SPA/モバイル), Client Credentials (M2M)
- **レートリミット**: `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset` ヘッダ
- **入力バリデーション**: 型・範囲・フォーマット・ホワイトリスト検証を必須実装

詳細は `references/DESIGN-*.md` を参照してください。

### ユーザー確認が必要な場面

認証方式の選択、バージョニング戦略、ページネーション方式、CORS設定スコープ、レートリミット値はAskUserQuestionで確認してください。

## Spec First開発

API仕様を先に記述し、E2Eテストフレームワークで検証しながら実装を進める開発方法論。

### 優れたAPI仕様の3要件

| 要件 | 説明 |
|------|------|
| **理解が容易** | 実装コードを読まなくても機能・動作・エラーが理解できる |
| **変更が容易** | API仕様の修正が容易に実装へ反映でき、変更結果を自動テストで確認できる |
| **テストが容易** | 自動テストが整備されており、ローカルPC上で容易に実行できる |

API仕様に書くべきこと: サービス概要・各エンドポイントの説明・**エラーの説明**（必須）

> エラーを仕様に書くことは実装での防御的プログラミングに直結する。エラー仕様なしではE2Eテストを書けない。

### 開発サイクル（新規）

```
1. API仕様記述（PRとして独立させてレビュー）
2. E2Eテストフレームワーク実装
3. テストコード作成（エラーケース先行）
4. 機能実装
5. リファクタリング＆カバレッジ確認
6. PRレビュー
```

**エラーケースを先に実装する理由**: コードの前半がバリデーション・後半が正常処理という自然な構造になり、可読性が向上する。

### E2Eテストフレームワーク原則

1. **本番同一バイナリの使用**: テストコードをサービス本体に挿入しない
2. **カバレッジ取得**: テスト実行後に行単位で実行コードを確認できる

2プロセスモデル:
```
Target Service (本番バイナリ) ◄──► Test Suite Process
                                   ├─ テストコード
                                   └─ フェイクサービス群（依存サービスのスタブ/モック）
```

フェイクサービスの動作は単純に: テストコードから指示されたレスポンス/エラーを返す・設定外は panic する。

### 技術的負債の返済

3つのケースで段階的に返済:
- **既存エンドポイント修正時**: 仕様記述 → 既存テスト作成 → 修正実装
- **新規エンドポイント追加時**: Spec Firstサイクルに従う（新たな負債を作らない）
- **不具合修正時**: 再現テスト作成 → 実装修正の順序を守る

### 防御的プログラミング

API仕様にエラーを書く → E2Eテストでエラー検証を書く → 実装で防御的プログラミングが強制される

この連鎖により、API仕様・テスト・実装が整合した堅牢なバックエンドサービスになる。

詳細は `references/SPEC-*.md` を参照してください。

## APIテスト

### テスト戦略の全体像

品質とリスクを3ステップで定義:
1. 品質特性の特定 (ルックアンドフィール・セキュリティ・正確度・可用性等)
2. リスクの特定 (ヘッドラインゲーム・オブリーク・リスクストーミング)
3. 優先順位付け (発生確率 × 深刻度)

**TaTTa判断基準**: 「APIをテストするのか（統合テスト）、APIを介してテストするのか（ユニットテスト）」を問う。

### API設計テスト（コード前）

5W1H による問い立て:
```
Who（誰が）     — 誰がこのAPIを使うか
What（何を）    — 誤った値を送ったら何が起きるか
Where（どこで） — データはどこに保存されるか
When（いつ）    — バージョン変更のタイミング
Why（なぜ）     — なぜこの設計判断をしたのか
How（どのように）— どのような仕組みで動くか
```

OpenAPI 3スキーマで設計を可視化し、思い込みを排除する。

### 自動化設計

3層構造でフレームワークを構築:
```
checks/    — テストの意図とアサーション（読みやすさ最優先）
requests/  — APIごとのHTTPリクエストをメソッドとして抽象化
payloads/  — リクエスト/レスポンスのデータモデル
```

自動化対象の選択原則:
- **リスクを尺度に** — コードカバレッジではなくリスク優先順位順に自動化
- **変化検出器として使う** — 変化の意味の解釈は人間が行う

### ATDD（受け入れテスト駆動開発）

Given-When-Then形式でビジネス要件を表現し自動チェックに変換:
1. 受け入れ基準をGherkinで定義
2. 失敗する自動チェックを作成
3. チェックが通るまで本番コードを実装
4. リファクタリング

### テスト戦略確立の判断軸

| 判断軸 | 内容 |
|--------|------|
| 品質特性の優先度 | ユーザーにとって何が最重要か |
| リスクの発生確率 | そのリスクが実際に起きる可能性 |
| リスクの深刻度 | 起きた場合のダメージの大きさ |
| テスタビリティ | そのリスクのテストがどれだけ困難か |

詳細は `references/TEST-*.md` を参照してください。
