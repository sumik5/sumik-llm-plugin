# 設計・評価手法リファレンス

UXデザインプロセス③〜⑥段階で用いる設計・評価手法の実践ガイド。
設計手法はアイデアをコンセプトへ変換し、評価手法は品質を検証する。

## 手法の概観

| 手法 | 種別 | 主な段階 | ポイント |
|------|------|---------|---------|
| UXコンセプトシート | 設計 | ③発想 | ペルソナ×体験価値×シーン×シナリオ |
| 構造化シナリオ法 | 設計 | ③発想→④視覚化 | 3層（バリュー/アクティビティ/インタラクション） |
| ストーリーボード | 設計 | ④視覚化 | 絵コンテ形式、6〜9コマ |
| 体験談型バリューストーリー | 設計 | ③④ | 利用経験者がコンセプトを語る形式 |
| 9コマシナリオ | 設計 | ④視覚化 | コンパクトなシナリオ表現 |
| コンセプトテスト | 評価 | ⑤プロトタイプ | 受容性・魅力度の確認 |
| ユーザビリティテスト | 評価 | ⑤⑥ | タスク操作による問題発見 |
| ヒューリスティック評価 | 評価 | ⑤ | 専門家が10原則で評価 |
| アイデア・タスク展開 | 設計→評価 | ⑤ | シナリオからタスクへ変換 |
| UX評価尺度（AttrakDiff等） | 評価 | ⑥ | 主観的UXの定量測定 |
| SUS | 評価 | ⑥ | 10項目のユーザビリティ測定 |
| NEM | 評価 | ⑥ | 初心者/熟達者比率による分析 |

## 設計手法

### UXコンセプトシート

**目的**: アイデアを体験価値・ペルソナと結びつけてコンセプトとしてまとめる

**適用段階**: ③アイデア発想とコンセプト作成

**テンプレート構造**:

| フィールド | 内容 |
|-----------|------|
| ペルソナ | 氏名・特徴・この体験で強調する特徴 |
| 実現すべき体験価値 | アイデアが目標とする体験価値・本質的ニーズ |
| UXDコンセプト：キーフレーズ | 「どんな体験ができるか」「どんなものか」を端的に表現 |
| 使い続けた時の心の声 | 累積的UXとしてのペルソナの声（KA法の価値と対応） |
| シーン・利用文脈 | 典型的な利用シーンを複数（最初に最も典型的なものを） |
| バリューシナリオ | 「いつ・どのように使うと・どんな嬉しいことがあるか」 |
| 時間軸の心の声（使用前） | 予期のUX：期待と不安の両方 |
| 時間軸の心の声（使用中） | 瞬間的UX：機能的な嬉しさを含む |
| 時間軸の心の声（使用後） | エピソード的UX：使い心地・性能の感想 |

**実施ポイント**:
- 「使い続けた時の心の声」をKA法で解釈すると体験価値が導出できる関係にする
- 機能の詳細化は後回しにし、先にユーザーの体験を記述する
- ユニークなアイデアでも「心の声」に納得感があれば体験価値を実現できる

**よくある失敗と対策**:
- 機能中心の記述になる → 「その仕組みを使うと」のように機能を抽象化して表現
- 体験価値と心の声が対応しない → KA法で心の声→価値の導出関係を確認して修正

### 構造化シナリオ法

**目的**: アイデアを3層のシナリオで段階的に具体化し、ビジョン提案型の仕様を作成する

**適用段階**: ③アイデア発想 → ④視覚化 → ⑤プロトタイプ反復

**3層シナリオ構造**:

| シナリオ | 扱う内容 | 記述しない内容 | 評価観点 |
|---------|---------|--------------|---------|
| バリューシナリオ | 体験価値 | 製品・ユーザーの行為 | 満足度（魅力性・新規性） |
| アクティビティシナリオ | ユーザーの活動・行動 | 製品の操作・機能 | 有効さ（有用性） |
| インタラクションシナリオ | 具体的な操作手順 | 画面デザイン | 効率（効率性） |

**各シナリオの記述ポイント**:
- **バリューシナリオ**: シーンが感じさせる価値のみ。UXコンセプトシートのバリューシナリオと同位置づけ
- **アクティビティシナリオ**: 製品は登場するが操作は書かない。「その仕組みが〜してくれた」と機能を効果に変換
- **インタラクションシナリオ**: タスクごとにユーザーと製品の操作を時系列で記述

**実施ポイント**:
- 必ずしも①→②→③の順番でなくてよい。構築・再構築を繰り返しながら進める
- アクティビティシナリオに操作・機能が混入しやすい。「その仕組み」と置き換えて書き直す
- 各シナリオに対応するユーザビリティ指標（満足度・有効さ・効率）で評価できる

**よくある失敗と対策**:
- エンジニアはアクティビティに操作を混在させがち → 機能を「その仕組みが〜する効果」に変換
- バリューシナリオが体験価値と結びつかない → UXDコンセプトシートのバリューシナリオで確認

### ストーリーボード

**目的**: 提案するUXを時系列の絵コンテで視覚化し、利用文脈・タイミング・ユーザーの動機を確認する

**適用段階**: ④UXと利用文脈の視覚化

**実施ポイント**:
- アクティビティシナリオを骨格にして作成すると効果的
- 含めるべき要素：ペルソナの目的・動機、主要な利用シーン・文脈・環境、製品挙動とユーザーのふるまい、ゴール達成と継続利用への動機
- コマ数：1タスクにつき6〜9コマ（ペルソナを含まず）が一般的
- スケッチが難しい場合はアクティングアウト（実際に演じた写真）でも代替可能

**3つの活用法**:
- **ウォークスルー評価**: ストーリーボードそのものの内容を時系列で評価・改善
- **コンセプトテスト**: ユーザーに提案UXを提示する評価材料として使用
- **ペーパープロトタイプ評価**: ストーリーボードの下にインタラクションシナリオ/ペーパープロトタイプを配置して評価

**よくある失敗と対策**:
- 利用文脈・環境の表現に誤解が生じる → 現実の利用文脈に基づいたエッセンスを含める
- コマ数が少なすぎる → 理想のUXを表現するのに十分なコマ数で表現する

### 体験談型バリューストーリー

**目的**: 体験価値レベルの抽象度でコンセプトを評価可能な形で表現する

**適用段階**: ③発想 → ④視覚化（コンセプトテストの提示材料として）

**概要**:

利用経験のあるペルソナが、未経験のサブ・ペルソナに製品・サービスの体験談を語るシーンを描く手法。
基本パターン（テンプレート）に沿ってコンセプト内容を当てはめるだけでストーリーを構築できる。

**テンプレート骨格**（9コマ構成）:

| コマ | 内容 |
|-----|------|
| 1 | コンセプトタイトル |
| 2 | 提案物の紹介 |
| 3 | できない理由（現状の課題） |
| 4 | 提案物のイメージ |
| 5〜7 | 提案物の利用シーン1〜3 |
| 8 | 提案物を継続利用したイメージ |
| 9 | 提案物への賛同 |

**実施ポイント**:
- UXDコンセプトシートより具体的で、評価者が想像で補う余地が少ない
- ペルソナの話を聞く第三者としてストーリーを聞き、共感できるかで評価する
- 累積的UXとして製品への受け入れ方を確認できる

---

### 9コマシナリオ

**目的**: アクティビティシナリオを9コマの構造化されたフォーマットでコンパクトに表現する

**適用段階**: ④UXと利用文脈の視覚化

**9コマの割り当て**:

| コマ | UXの位置づけ | 内容 |
|-----|------------|------|
| 1 | ペルソナ | ペルソナの紹介 |
| 2 | コンテキスト | 現状の問題・状況 |
| 3 | 予期のUX | 提案への期待 |
| 4〜7 | 瞬間的UX | 利用シーン（1〜4） |
| 8 | エピソード的UX | ゴール達成の振り返り |
| 9 | 累積的UX | 継続利用による変化 |

**実施ポイント**:
- ストーリーボードの平均コマ数（6〜9コマ）に対応した簡易版として使いやすい
- 複数アイデアをポイントを絞って比較検討するのに適している
- 9コマが制約になる場合はストーリーボードに切り替えること

## 評価手法

### コンセプトテスト（シナリオ共感度評価）

**目的**: 製品・サービスのコンセプトを想定ユーザーに提示し、受容性・魅力度・利用意欲を把握する

**適用段階**: ⑤プロトタイプの反復

**評価尺度（コンセプトボードに対する5段階評定）**:

| 評価項目 | 評価内容 |
|---------|---------|
| わかりやすさ | 商品説明は充分にわかりやすいか |
| ニーズ合致度 | あなたのニーズに合っているか |
| 共感性 | 利用シナリオにどの程度共感するか |
| 魅力度 | どの程度魅力を感じたか |
| 新規性 | 目新しく感じるか |
| 経験意欲 | シナリオの利用経験をしてみたいか |
| 購入意向 | 妥当な価格なら買ってみたいか |

**効果的な2段階評価方法**:
1. **第1段階**: バリューシナリオ提示 → ニーズ合致度・新規性・魅力度（1回目）・利用意欲（1回目）を評定
2. **第2段階**: シーン＋アクティビティシナリオ提示 → 共感度・魅力度（2回目）・利用意欲（2回目）を評定

*魅力度と利用意欲の1回目と2回目の差を分析することで、アクティビティの魅力度を評価できる*

**典型的な進め方**:
1. 対象行為について現在の状況をインタビュー
2. ペルソナについて説明
3. コンセプトボードを提示・説明（質問は「なぜそう思いましたか？」と理由を返す）
4. 評価尺度で評定後、理由と想定した利用文脈をインタビュー
5. 複数案に対して(2)〜(4)を繰り返す
6. 全案の評定後に相対的な順位づけと理由をたずねる

**よくある失敗と対策**:
- 評価値だけを数量化してしまう → コンセプトの理解のされ方をインタビューで必ず確認する
- 単一案のみ評価 → 複数案（既存製品を混ぜた比較含む）で相対評価を行う

### ユーザビリティテスト

**目的**: プロトタイプ・製品でのタスク操作を通じてユーザビリティ上の問題を発見・測定する

**適用段階**: ⑤プロトタイプ反復（形成的評価）・⑥実装レベル評価（総括的評価）

**実施人数の考え方**:
- 形成的評価（問題発見）: 5〜8名で主要な問題の大部分が発見できる（5名で約85%の問題発見）
- 総括的評価（測定）: より多くの参加者が必要（統計的分析を行う場合）

**タスク設計の手順**:
1. テスト目的・狙いを明確にし、主要なタスクに絞る
2. ユーザー視点で操作の目的・意図・状況を想定する
3. タスクのスタートとゴールを定義する
4. 状況設定としてのシナリオを作成する
5. タスクを紙に印字し口頭でも説明する

**タスク設計の禁止事項**:
- タスク文に製品用語や操作手順を含めない
- ユーザーの視点でなくシステムの視点で書かない
- ゴールを曖昧にしない

**発話思考法（think aloud method）**:
操作中に考えていること・理解したこと・感じたことをすべて声に出してもらう手法。
認知的問題を把握する手がかりとなる。記録したものを「発話プロトコル」と呼ぶ。

**分析の焦点**:

| 分析項目 | 対応するユーザビリティ指標 |
|---------|------------------------|
| 操作の誤り・タスク達成度 | 有効さ |
| 操作の無駄・戸惑い・タスク実行時間 | 効率 |
| 調査協力者の心理（いらだち・不満） | 満足度 |

**よくある失敗と対策**:
- モデレーターが操作方法を教えてしまう → 「思ったとおりにやってみてください」「そうするとどうなると思いますか？」と返す
- タスクにシステム用語を使う → ユーザーの行為の目標（Be/Do Goal）レベルで記述する

### ヒューリスティック評価（専門家評価）

**目的**: HCDの専門家が経験則（ヒューリスティック）に基づいてインターフェースのユーザビリティ問題を発見する

**適用段階**: ⑤プロトタイプ反復（開発初期〜中期）

**ニールセンの10ヒューリスティックス**:

| # | 原則 | 内容 |
|---|------|------|
| 1 | システム状態の視認性 | 適切なフィードバックでユーザーが何を実行しているか常に知らせる |
| 2 | システムと現実世界の調和 | ユーザーのなじみのある言葉・慣習で情報を提示する |
| 3 | ユーザー・コントロールと自由度 | 誤操作から抜け出すための「非常出口」（Undo/Redo）を提供する |
| 4 | 一貫性と標準化 | 同じことを同じ表現で。プラットフォームの慣習に従う |
| 5 | エラーの防止 | エラーメッセージより問題の発生を防止するデザインを優先する |
| 6 | 記憶しなくても見ればわかるように | オブジェクト・動作・オプションを可視化する |
| 7 | 柔軟性と効率性 | ショートカット等で上級者の操作をスピードアップする |
| 8 | 美的で最小限のデザイン | 関連のない情報・めったに使わない情報は含めない |
| 9 | エラーの認識・診断・回復のサポート | エラーメッセージは平易な言葉で問題と解決策を示す |
| 10 | ヘルプとマニュアル | 探しやすく・タスクに焦点を当てた・具体的・簡潔なヘルプを提供する |

**実施手順**:
1. 3〜5名の専門家が同じ評価対象・範囲・ガイドラインで独立して評価
2. 「評価者ミーティング」で各自の結果を持ち寄る
3. 問題点・原因・改善の方向性をレポートにまとめる

**実施ポイント**:
- ユーザー参加による評価（ユーザビリティテスト）の前に実施して基本的な問題を潰す
- 最近は「エキスパートレビュー」と呼ぶことが増えている（10ヒューリスティックスに限定しない）
- Web・スマートフォンアプリでは10ヒューリスティックスで対応できない問題も増加

**よくある失敗と対策**:
- 実際にはユーザーにとって大きな問題でない箇所を問題視してしまう → ユーザビリティテストとセットで実施し相補的に使う

### アイデア・タスク展開

**目的**: シナリオ・ストーリーボードに描かれた理想のUXから、プロトタイプ開発のためのタスクを導出する

**適用段階**: ⑤プロトタイプ反復

**概要**:

9コマシナリオやストーリーボードで表現されたアクティビティシナリオレベルのUXを、
アイデアを修正しながら具体的な製品・サービスのタスクへ変換する手法。

**2段階のプロセス**:

1. **アイデア展開**: 9コマシナリオのコマごとにユーザーが予想するネガティブなポイント（不安・障壁）を挙げ、それを良い意味で裏切るようにアイデアを修正する
2. **タスク展開**: 修正したアイデアをもとに、各利用シーンで必要なタスクを導出する

**実施ポイント**:
- 構造化シナリオ法のインタラクションシナリオと同様に、アイデアと具体的仕様の橋渡しをする
- 利用文脈を考慮したシナリオを基点にするため、ユーザー視点のタスクになりやすい

### UX評価尺度

**目的**: ユーザーの主観的な利用経験に基づいて製品のUXを定量的に評価・比較する

**適用段階**: ⑥実装レベル評価

**主要な評価尺度の比較**:

| 尺度 | 特徴 | 評価因子 | 適用場面 |
|-----|------|---------|---------|
| **UX評価尺度（日本）** | 長期利用経験に基づく評価、家電製品向け | 主観的ユーザビリティ・ブランドイメージ・不満感・使う喜び・愛着感（37項目・6件法） | 家電・コンシューマ製品 |
| **AttrakDiff** | 実用的品質（Pragmatic）と快楽的品質（Hedonic）を分離評価 | PQ（実用的）・HQ-S（刺激性）・HQ-I（同一性）・ATT（魅力性） | 製品印象の比較 |
| **meCUE** | 製品知覚の多次元評価 | 有用性・使いやすさ・審美性・感情・コミットメント | デジタル製品全般 |

**UX評価尺度（日本）の実施上のポイント**:
- 因子分類は調査協力者には見せずに6件法で回答させる
- 「不満感」因子の得点は逆転して計算する（不満が高いほど得点が低くなる）
- 他カテゴリの製品には質問項目を目的に合わせて変更して使う（学術的には非推奨だが実務では許容）

### SUS（System Usability Scale）

**目的**: 10項目の質問でシステムのユーザビリティを素早く定量測定する

**適用段階**: ⑥実装レベル評価（ユーザビリティテスト後に実施）

**SUS 10項目**:

| # | 質問項目 | スコア方向 |
|---|---------|---------|
| 1 | このシステムは、たびたび使ってみたい | 奇数（正方向） |
| 2 | このシステムは、不必要なほど複雑であると感じた | 偶数（逆方向） |
| 3 | このシステムは、容易に使えると思った | 奇数（正方向） |
| 4 | このシステムを使うのに、技術に詳しい人のサポートを必要とするかもしれない | 偶数（逆方向） |
| 5 | このシステムは、さまざまな機能がよくまとまっていると感じた | 奇数（正方向） |
| 6 | このシステムでは、一貫性のないところが多くあった | 偶数（逆方向） |
| 7 | ほとんどの人は、このシステムの使い方について、とても素早く学べると思う | 奇数（正方向） |
| 8 | このシステムは、とても扱いにくいと思った | 偶数（逆方向） |
| 9 | このシステムを使うのに自信がある | 奇数（正方向） |
| 10 | このシステムを使い始める前に、多くのことを学ぶ必要があると思った | 偶数（逆方向） |

**スコア計算方法**:
1. 奇数項目（1,3,5,7,9）: `回答値 − 1`
2. 偶数項目（2,4,6,8,10）: `5 − 回答値`
3. 10項目の計算値の合計 × 2.5 → **0〜100点**に変換

**SUSスコアのグレード基準（Jeff Sauro, 2011）**:

| グレード | スコア範囲 | 評価 |
|---------|---------|------|
| A | ≥ 80.3 | 優秀 |

### NEM（Novice Expert ratio Method）

**目的**: 初心者と熟達者の操作時間比から、ユーザビリティ上の問題のある手順を特定する

**適用段階**: ⑥実装レベル評価

**概要**:

ノーマンの概念モデル理論（設計者の概念モデル vs ユーザーの概念モデル）から着想を得た手法。
熟達者の操作 ≈ 設計者の意図した操作、初心者の操作 ≈ ユーザーが実際に行う操作 と仮定して分析する。

**NE比の計算式**:

```
NE比 = Tn / Te

Tn: 初心者ユーザーが要した操作の平均時間（複数名の平均）
Te: 熟達者ユーザーが要した操作の平均時間（複数名の平均）
```

**結果の解釈**:
- NE比が高い操作ステップ → 初心者が戸惑いやすい問題箇所
- 操作ステップごとにグラフ化することで問題箇所を視覚的に特定できる

**実施上の注意**:
- WebサイトのようにタスクへのルートがBranchする場合は、正解手順以外の「迷い」時間の扱いを工夫する（正解画面到達までの迷い時間を合算するなど）
- 複数の初心者・熟達者のデータ平均を使うこと

## AskUserQuestion: UX評価の焦点

```
UX評価を計画する際に確認すること:
1. 何を評価したいか？
   - ユーザビリティ（操作性の問題発見）→ ユーザビリティテスト + SUS/NEM
   - UX全体（印象・感情・ブランド）→ UX評価尺度 + AttrakDiff
   - コンセプトの受容性 → コンセプトテスト
2. 開発の段階は？
   - 初期プロトタイプ → ヒューリスティック評価 + コンセプトテスト
   - 中期プロトタイプ → ユーザビリティテスト（形成的評価）
   - 実装レベル → SUS/NEM/UX評価尺度（総括的評価）
3. リソース（人数・時間・コスト）は？
   - 少ない → ヒューリスティック評価（専門家3〜5名で完結）
   - 標準 → ユーザビリティテスト（5〜8名）
   - 大規模 → 複数手法の組み合わせ
```
