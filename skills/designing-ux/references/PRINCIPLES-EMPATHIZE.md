# UX実践原則：Empathize（原則 18–37）

フェーズ「Empathize」はユーザーを深く理解するための視点。
アクセシビリティ・多様なユーザー層・認知・操作のコンテキストを把握する。

---

### 18. アクセシビリティを最初から設計する

**核心**: アクセシビリティは後付けではなく、設計の出発点にする。

- 視覚障害・聴覚障害・運動障害・認知障害を持つユーザーへの配慮はすべての人の体験を向上させる（カーブカット効果）
- WCAG 2.1 AA準拠を最低ラインとして設定する
- スクリーンリーダー・キーボード操作・拡大表示での動作を設計時に確認する

**If/Then**: If 新しいUIコンポーネントを作るなら then キーボードナビゲーションとスクリーンリーダー対応を同時に実装する

---

### 19. デジタルリテラシーの差を前提にする

**核心**: ユーザーはデジタルツールの習熟度が千差万別。高リテラシーを前提にした設計は多くのユーザーを排除する。

- 「スワイプで削除」「長押しでメニュー」などの隠れたジェスチャーは発見されにくい
- アイコンだけのUIは初心者を迷子にする
- ヘルプ・ツールチップ・ガイダンスは「必要な人だけ見る」レイアウトにする

**If/Then**: If 非自明なインタラクションを実装するなら then 初回使用時のコーチマークまたはツールチップを追加する

---

### 20. 高齢ユーザーを考慮する

**核心**: 加齢によって視力・手の巧緻性・記憶力・認知速度が変化する。これを前提にした設計が将来の自分への投資にもなる。

- タップターゲットは最低44×44px（Apple HIG）以上
- 高コントラスト・大フォントのオプションを用意する
- 複雑な手順は段階的に分割し、一度に要求する記憶量を減らす

**If/Then**: If モバイルUIを設計するなら then インタラクティブ要素のタップターゲットを最低44×44pxに設定する

---

### 21. 子どもは小さな大人ではない

**核心**: 子ども向けUIは成人向けを縮小したものではなく、発達段階に応じた独自設計が必要。

- 読み書き能力・抽象思考・時間概念は年齢によって大きく異なる
- 親の監督なしの購入・個人情報入力・年齢確認を設計する
- ゲーミフィケーション・キャラクター・アニメーションは子どもの学習意欲を高める

**If/Then**: If 子ども向けアプリを設計するなら then 対象年齢の認知・読解レベルを調査し、そのレベルに合わせたUIを設計する

---

### 22. ラーナビリティ（習得のしやすさ）

**核心**: 初めて使うユーザーが学習コストをかけずに操作を理解できるよう設計する。

- 一般的なUIパターン・標準アイコン・慣習的な配置を守ることで学習コストを下げる
- 新しいインタラクションパターンを採用するときは教育コストを見積もる
- オンボーディングフローでの「やってみる」体験が理解を定着させる

**If/Then**: If 独自のインタラクションパターンを導入するなら then インタラクティブチュートリアルで体験させる

---

### 23. 初心者とパワーユーザーの両立

**核心**: 同一製品内でノービスとエキスパートを同時に満足させる設計が必要。

- 初心者にはシンプルなデフォルトフロー。パワーユーザーにはキーボードショートカット・詳細設定・バッチ操作
- 段階的な開示（Progressive Disclosure）で複雑な機能を隠す
- ノービスが成長してパワーユーザーになるパスを設計する

**If/Then**: If 高度な機能を持つツールを設計するなら then 基本操作と上級操作を視覚的に分離し、上級機能はワンアクションで展開できるようにする

---

### 24. ヒックの法則（選択肢と決断時間）

**核心**: 選択肢の数が増えると意思決定にかかる時間が対数的に増加する。

- メニュー項目・製品バリエーション・フォームオプションは最小限にする
- 選択肢が多い場合はカテゴリ分け・検索・フィルターで絞り込みを可能にする
- 緊急時のUI（エラー画面・警告ダイアログ）は選択肢を1〜2個に絞る

**If/Then**: If ドロップダウンに7項目以上あるなら then カテゴリ分けまたは検索UIへの変更を検討する

---

### 25. 多様なチームが多様なユーザーを救う

**核心**: デザインチームの多様性（年齢・性別・文化・障害の有無等）は製品の包括性に直結する。

- 同質なチームは自分たちに似たユーザーのためにしかデザインできない
- ユーザーリサーチで補完するとしても、チーム内多様性が盲点を減らす
- アクセシビリティの課題は障害当事者がチームにいると早期発見される

**If/Then**: If デザインレビューを実施するなら then チームの属性に偏りがあることを認識し、多様なユーザー層の視点を意識的に代弁する

---

### 26. 画面サイズより利用文脈が重要

**核心**: レスポンシブデザインは画面幅の適応だけでなく、利用文脈（状況・目的・注意量）への適応が本質。

- モバイルユーザーは移動中・片手操作・高い干渉（通知・話しかけ）の中で使う
- デスクトップユーザーは集中モード・大量データ閲覧・複数ウィンドウ操作が前提
- 同じ情報でも、文脈に応じて優先表示する要素が変わる

**If/Then**: If モバイル版を設計するなら then 片手操作・短い集中時間・低帯域環境を前提に情報優先度を見直す

---

### 27. タッチターゲットの不器用なデザイン

**核心**: 小さすぎるタップ領域・密集したボタンはすべてのユーザーにとって障壁になる。

- 親指は人差し指より太く、スマートフォン下部の操作が最も自然
- リストアイテムはテキスト部分だけでなく行全体をタップ可能にする
- ボタン間の余白は最低8pxを確保する

**If/Then**: If モバイルでインタラクティブ要素が密集するなら then タップターゲットサイズと余白を増やし、誤タップを減らす

---

### 28. 現実世界のメタファーを活用する

**核心**: 物理世界に存在するオブジェクト・動作を模倣したUIはユーザーの学習コストを大幅に削減する。

- フォルダ・ゴミ箱・書類などのメタファーはPCの普及を加速した
- ただしメタファーは制約も引き継ぐ（フォルダは階層構造に縛られる）
- 現実に対応物がない概念（クラウド同期・API等）は独自の説明UIが必要

**If/Then**: If 新しい概念をUIで表現するなら then まず現実世界の類似体を探し、有効なら活用する

---

### 29. 常識を打破する正しいタイミング

**核心**: UXの慣習を破ることで突出した体験を生む場合があるが、タイミングと根拠が重要。

- 慣習破りが成功するのは、ユーザーにとってより明確・自然・効率的な場合のみ
- テストなしに「革新的」と称して慣習を無視するのはリスクが高い
- ユーザーが新しいパターンに適応する学習コストを上回る価値があるか検証する

**If/Then**: If 標準UIパターンから逸脱するなら then A/Bテストまたはユーザーテストで学習コストと価値のバランスを検証する

---

### 30. 強要でなく説得する

**核心**: ユーザーを強制的に誘導するより、納得してもらう方が長期的な信頼を構築する。

- 強制的なニュースレター登録・自動課金・デフォルトONの通知はユーザーの信頼を損なう
- 価値を先に提供し、それと引き換えに情報やアクションを求める
- ダークパターン（→原則05）の反対が「説得的デザイン」

**If/Then**: If メール登録や通知許可を求めるなら then 先に価値を示し、その後に明示的な同意を求める

---

### 31. IoTデバイスのUX設計

**核心**: 画面のないまたは小さい画面のデバイスでは、UIパラダイムが根本的に変わる。

- 物理ボタン・LED・音声フィードバック・振動がUIの主要要素になる
- 「接続中」「エラー」「完了」を光のパターンや音で表現する
- スマートフォンアプリはIoTデバイスの補助UIとして設計する

**If/Then**: If IoTデバイスのUXを設計するなら then 画面なしでの操作フローを先に設計し、アプリは補完として位置づける

---

### 32. ユーザーの目的はツールを使うことではない

**核心**: ユーザーがアプリを開く理由は「アプリを使うこと」ではなく、「何かを達成すること」。

- 目的（旅行の計画）と手段（旅行アプリの操作）を混同しないデザイン
- ユーザーがアプリを「意識せずに目的を達成できる」ことが理想
- 機能中心ではなくタスク中心の情報設計

**If/Then**: If 機能リストを設計するなら then 「ユーザーが何を達成したいか」にリマップし、達成を妨げる不要な機能を除く

---

### 33. 邪魔は必要なときだけ

**核心**: 通知・確認ダイアログ・アップセル表示は、ユーザーのフローを中断させる。最小化する。

- 割り込みは緊急かつユーザーに即座の判断が必要な場合のみ
- バッジ通知は数が増えると無視されるようになる（通知疲れ）
- コンテキストを外れたタイミングの通知（例：使用中の別機能のお知らせ）は最悪

**If/Then**: If 通知・ポップアップを追加するなら then 「今すぐこのユーザーが知る必要があるか」を問い、NOなら延期または削除する

---

### 34. 価値ある通知だけを送る

**核心**: 通知はユーザーの時間と注意という貴重なリソースを消費する。価値がなければ信頼を失う。

- プッシュ通知のopt-in率が低いのは、過去の経験でゴミ通知が届いたから
- パーソナライズされた通知（行動トリガー・タイミング最適化）は開封率が高い
- 通知設定の細かなコントロールをユーザーに与えると信頼感が高まる

**If/Then**: If プッシュ通知を設計するなら then ユーザーが設定できるきめ細かな通知カテゴリと頻度コントロールを実装する

---

### 35. フォームを最小化する

**核心**: フォームはユーザーが最も嫌うUIの一つ。必要最小限の項目に絞ることで完了率が上がる。

- 各項目に「これは絶対に必要か？」を問い、任意項目は削除する
- ソーシャルログイン・自動入力・住所補完などで入力量を削減する
- 複数ステップに分割し、各ステップの完了感を演出する

**If/Then**: If フォームに10項目以上あるなら then 必須・任意・後で収集できる項目を分類し、初回フォームから任意項目をすべて除く

---

### 36. 簡素で速いデザインが最高のUX

**核心**: ページ速度・操作の軽快さ・シンプルなレイアウトは、すべての要素の中で最もUXに影響する。

- Googleの調査では、ページ読み込みが1秒から3秒になると直帰率が32%増加する
- アニメーション・重い画像・複雑なレイアウトは「リッチに見える」が体験を損なうことが多い
- パフォーマンスバジェットを設定し、デザインの意思決定に組み込む

**If/Then**: If 新しいビジュアル要素を追加するなら then パフォーマンスへの影響を確認し、許容範囲内か検証する

---

### 37. ルールは破るためにある

**核心**: UXの原則はガイドラインであり絶対法則ではない。文脈と根拠があれば破ることが正しい場合もある。

- 「3クリックルール」「Fパターン」などの古い原則を盲目的に適用しない
- 原則を破るときは「なぜ破るのか」「何を得るのか」を明確にする
- 破ったことをテストで検証し、仮説が正しかったか確認する

**If/Then**: If 既存のUX原則に反するデザインを採用するなら then 具体的な根拠と検証計画を持って判断する
