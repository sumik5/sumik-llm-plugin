# 共感ペルソナ 実践ガイド

ユーザー調査・分析データをもとに「共感図」を使ってユーザー像を精緻に描き、プロジェクトメンバー全員の目線を揃えるための手法。

---

## 仕事としての始めやすさ

| こっそり練習 | 一部業務でトライアル | クライアント巻き込み |
|---|---|---|
| ★★ | ★★★ | ★★★★ |

**こっそり練習**: まず既存の調査データを使って共感図に分類してみる。エリアが埋まらない部分は「追加調査が必要な箇所」と捉える。

**一部業務でトライアル**: 小規模プロジェクトで、ユーザーリサーチ（Ch5）と親和図法の分析が完了した後に実施する。複数人でのワークショップ形式が理想。

**クライアント巻き込み**: ワークショップにクライアントを参加させることで「自分事」としてとらえてもらえる。多人数での実施を前提とした上級者向けの手法だが、効果は顕著。

---

## 概要

「共感ペルソナ」は、実際の調査・分析データを「共感図（Empathy Map）」のテンプレートに落とし込み、具体的なユーザー像を描く手法。

**簡易ペルソナとの違い**:

| 項目 | 簡易ペルソナ | 共感ペルソナ |
|---|---|---|
| 別名 | プラグマティックペルソナ | データドリブンペルソナ |
| データ基盤 | 調査や分析に基づかない仮のもの | 実際のデータを利用 |
| 見直し頻度 | アジャイル開発等での頻繁な見直し | 長く利用することを前提に緻密に作成 |
| 主な用途 | 早期の仮説検証 | 設計の根拠・チームの共通認識 |

どちらもペルソナ作りが重要であることに変わりはない。プロジェクトの規模・目的・フェーズに応じて使い分ける。

---

## 7-1 プロジェクトメンバーの目線を揃える

### ユーザー像ほど伝わりにくいものはない

調査資料があっても、それだけでは次のアクションに結びつかない。人によって解釈が異なり、認識のズレが生じやすい。このズレは工程が進むほど大きくなり、軌道修正が難しくなる。

**翻訳作業としてのユーザーモデリング**

工程が進むごとに表現方法・言葉遣い・分量が変わる。そのメンバーにとって最適な表現で伝えることが必要。ユーザーモデリング = ユーザー像をしっかり次工程に伝える「翻訳作業」。

翻訳ミスが起きると:
- メンバーによって見ているユーザー像が異なる
- 勝手な解釈でユーザーに間違った共感をする
- ユーザーのことを分かった気になっているだけ

→ 工程が進むほどブレが拡大し、最終的に「出てきたものを見てビックリ」という事態が生じる。

## 7-2 ペルソナとは

### ペルソナを作る2つの目的

1. **現在を描く**: ユーザーがいま何をしているかを捉える
2. **未来を描く**: ユーザーがこれからどうなっていくかを捉える

この2つを、プロジェクトメンバー全員が一緒に考えられる状態にすることがペルソナ作りのゴール。

語りかける対象としての「人間」としてのペルソナが必要。どこにいるか分からない架空のユーザーをでっちあげるのが目的ではない。

### 共感図（Empathy Map）について

共感図は、複数人でワークショップを行い、ユーザーの内面を探り共感を得るためのグループワーク手法。

**6つのエリア**:

```
        ┌─────────────────────┐
        │  考えていること（THINKING）  │
        └──────────┬──────────┘
                   │
   ┌───────────────┼───────────────┐
   │               │               │
聞こえていること    │          見えているもの
 （HEARING）       │           （SEEING）
   │               │               │
   └───────────────┼───────────────┘
                   │
        ┌──────────┴──────────┐
        │   言っていること（SAYING）   │
        └──────────┬──────────┘
                   │
        ┌──────────┴──────────┐
        │   していること（DOING）     │
        └──────────┬──────────┘
                   │
        ┌──────────┴──────────┐
        │  感じていること（FEELING）  │
        └─────────────────────┘
```

| エリア | 意味 |
|---|---|
| 見えているもの（SEEING） | ユーザーが目にしているもの・環境 |
| 言っていること（SAYING） | ユーザーが実際に発言していること |
| していること（DOING） | ユーザーが取っている行動・振る舞い |
| 感じていること（FEELING） | ユーザーの感情・情動 |
| 聞こえていること（HEARING） | ユーザーが耳にしていること（他者の声・メディア等） |
| 考えていること（THINKING） | ユーザーが心の中で思っていること（表に出さない部分） |

## 共感ペルソナの作り方

### 前提: データ収集と分析の完了

共感ペルソナの作成は、以下が完了している段階で実施する:
- ユーザーリサーチ（感情曲線インタビュー・弟子入りインタビュー）
- 親和図法による分析

### 手順

**① 調査データを共感ペルソナの6エリアに分類する**

分析で得られたデータを付箋に書き、6エリアに分類しながら貼っていく。

- 「考えていること（THINKING）」以外のエリアに、多くのデータが貼られるはず
- 埋まらないエリアがあれば → 「追加調査が必要な部分が見つかった」という良い気づき

**② 「考えていること」エリアを深掘りする**

ユーザーが心の中で思っていたであろうこと、特に行動に影響を与えたであろうことを追加する。

- データに基づく部分と「私たちの推測」を付箋の色で区別する（例: 推測は別色）
- 推測は追加調査で確認する

**③ 文章量・背景の深さを確保する**

共感ペルソナは要約・省略をせず、ありのままを描く。良い記述の基準:

❌ 「趣味は自転車で、毎週100km走っている」（表面的な事実）

✅ 「毎週100km走るようなトレーニングを欠かないストイックさが、このユーザーの行動の根拠になっている。そのため、△△のような状況では〇〇という反応をすることが予想できる」

→ 将来あるシナリオに置かれた時の行動を予想できる記述が必要。

**④ プロジェクト目的との関連性を確認する**

プロジェクトの今後の作業において役立つ記述か確認する。設計に使えない表面的な事実の記載は避ける。

## 3つの品質基準

共感ペルソナの記述・作成で気をつけること:

| 基準 | 説明 |
|---|---|
| 実際の調査・分析データに基づいているか | 参加者の経験・想像だけでなく、実際のデータを入れ込む |
| 文章量が豊富で行動の背景や心理が記述されているか | 要約・省略せず、ありのままを描く。シナリオの根拠として立ち返れる状態にする |
| プロジェクトの目的や調査テーマとの関連性が明白か | 次のアクションに結びつく記述になっているか確認する |

## 7-3 最終確認：ユーザーに共感できるか

### 共感の確認項目

ペルソナ作りの成功条件: 「私たちが向かい合うユーザーはこんな人なのか」という共通の認識・共感がプロジェクトメンバーに浸透し、それを土台として具体的な設計の根拠とできること。

完璧なペルソナをいきなり作り込む必要はない。「少し足りない部分」があることで議論が活発になることもある。

### 振り返りチェック

プロジェクト終了後に以下を振り返る（チームメンバーへのヒアリングも含めて）:

- **ユーザーのゴール・価値**: ユーザーは何を達成することを求めていたのかをユーザーの言葉で理解できたか
- **ユーザーの不満足**: ユーザーは何ができないでイライラしているのかを理解できたか
- **Webサイトの果たすべき役割**: このWebサイトはユーザーのゴール達成・不満解消にどんな役割を果たせるのかを把握できたか
- **ユーザーの制約事項**: ユーザーが置かれている環境・モノによる制約事項は何かを把握できたか

**究極のゴール**: 「いままでユーザーのことをまるで分かってなかった」という気づきと反省が得られること。

### 成長するペルソナ

共感ペルソナは一度作って終わりではない。プロジェクトとともに更新していく。

- Webサイトの公開・運用が開始されたら、実際に何が起こったかを追跡する
- 「解決策を提案したことでユーザーの行動がどう変わったか」という視点で見る
- 追加調査で新しいデータを得て、ペルソナのイメージをより具体的に膨らませる

「時間軸を持ったペルソナ」を設計してクライアントに説明できれば、制作作業だけでなく継続的な運用の仕事も提案できる。
