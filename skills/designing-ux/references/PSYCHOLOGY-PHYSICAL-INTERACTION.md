# 物理的インタラクション

物理的操作に関する定量的法則と、人間の知覚・認知の時間的制約を扱う。
ターゲットサイズ、操作経路、応答性の設計に直接活用できる。

---

## フィッツの法則

### 公式と意味

```
T = a + b × log₂(1 + D/W)
```

| 変数 | 意味 |
|------|------|
| T | 動作開始からターゲットを指すまでの所要時間 |
| D | 動作開始点からターゲット中心までの距離 |
| W | ターゲットの幅 |
| a | ポインタ移動の開始・停止時間（デバイス・人依存） |
| b | ポインタの移動速度（デバイス・人依存） |

**法則の本質:** ターゲットが大きいほど、近いほど、到達時間は短くなる。
ただし関係は **対数（非線形）** であり、効果は逓減する。

### 弾道的動作と微調整

ポインタ移動は 2 フェーズで構成される：

1. **弾道的動作フェーズ**: 開始直後は大雑把で速い動き（砲弾の弾道に類似）
2. **微調整フェーズ**: ターゲット接近後、手と目のフィードバックで精密な位置合わせ

→ ターゲットが小さいほど微調整の難易度が上がり、到達時間が長くなる

### 対数関係がもたらす逓減効果

| 変化 | 効果 |
|------|------|
| 極小ターゲットを 2 倍にする | 到達時間が大幅に短縮される |
| さらに 2 倍にする | 効果は最初より小さい |
| ある大きさを超えてさらに拡大 | ほぼ効果なし |

**実践的な含意:** 小さすぎるターゲットは優先的に大きくする。すでに十分な大きさのターゲットを拡大しても投資対効果が低い。

### 画面端の特性（デスクトップ限定）

デスクトップ・ノートPCでは、画面の端にポインタを勢いよく動かすだけで止まる。
それ以上動かしてもカーソルは移動しないため、**画面端のターゲットは実質的に幅が無限大**になる。

- **適用範囲:** デスクトップ・ノートPCのみ
- **除外:** スマートフォン・タブレット（物理的な縁がなく、指を止める境界がない）

### UIデザインへの応用

| ガイドライン | 理由 |
|-------------|------|
| ボタン・リンク・メニュー項目に十分なサイズを確保する | 小さすぎるとフィッツの法則により到達時間が急増する |
| クリック可能領域をテキストラベルだけでなく要素全体にする | クリック可能領域 = ターゲットのW（幅）に直結する |
| チェックボックス・ラジオボタンはラベルテキストもクリック可能にする | クリック可能領域を拡大できる |
| ターゲット間の余白を十分にとる | 意図しないターゲットへの誤クリックを防ぐ |
| 重要なターゲットを画面端付近に配置する | 画面端の無限幅効果でクリックしやすくなる（デスクトップのみ） |
| プルダウンより**ポップアップメニュー**や**パイメニュー**を検討する | ポインタからの距離が均等に短く、多方向に選択肢が展開する |

**判断基準テーブル: ターゲットサイズ設計**

| 状況 | 優先アクション |
|------|--------------|
| 「クリックしにくい」という報告がある | まずサイズを 2 倍にする（効果が最大） |
| すでに大きいが操作ミスが多い | ターゲット間の余白・配置を見直す |
| モバイルで操作ミスが多い | タッチターゲットは最低 44px × 44px を確保する |
| 画面密度が高くサイズを確保できない | クリック可能領域をビジュアルより大きくとる（padding 活用） |

## ステアリングの法則

### 公式と意味

```
T = a + b × (D/W)
```

フィッツの法則と異なり **線形関係**。経路の幅 W が 2 倍になると所要時間は半分になる。

**法則の本質:** ポインタを狭い経路に沿って移動させる場合、経路が広いほど速く移動できる。

### UIデザインへの応用

**問題が生じやすい UI パターン:**

| UIパターン | 問題 | 改善策 |
|-----------|------|--------|
| プルライトメニュー（ウォーキングメニュー） | サブメニューを開くまで狭い経路内を移動しなければならない | できる限り避ける。必要なら経路を広くする |
| 幅の狭いルーラー | タブをドラッグ中に経路からはみ出すと操作失敗 | ルーラーの幅を広くする |
| 幅の狭いスクロールバー | スクロール中にバーからはみ出すと操作が止まる | スクロール中はバー外への動きを無視する（縦なら上下のみ感知） |

**スクロールバーの設計教訓:** 現代のスクロールバーはドラッグ中にポインタがバー外にはみ出しても動作を継続する。これは「操作中の有効幅を画面全体に拡大する」という設計であり、ステアリングの法則の応用例。

**判断基準テーブル: 経路設計**

| 状況 | 推奨アクション |
|------|--------------|
| メニューに階層がある | プルライトメニューを避け、別画面・サブセクションで代替する |
| プルライトメニューが必須 | メニュー項目の幅を最大化する |
| ドラッグ操作がある | ドラッグ中は操作軸以外の方向への誤移動を許容する設計にする |

## 時間の要件

### 応答性の定義

**応答性 ≠ パフォーマンス（処理速度）**

| 概念 | 測定基準 |
|------|---------|
| パフォーマンス | 単位時間当たりの計算量 |
| 応答性 | 人間の時間的要求への対応度・ユーザー満足度 |

**応答性が最重要な理由:** 50年以上の研究で一貫して「ユーザー満足度を決定する最重要要素」と示されている。学習のしやすさや使いやすさより上位。

**応答性の高いシステムの条件:**
- 入力を受け取ったら即座にフィードバックを返す
- 処理にかかる時間の目安を表示する
- 待機中にユーザーが他の操作をできるようにする
- 優先度の低いタスクをバックグラウンドで処理する
- よく要求される操作を先読みして準備する

### 脳の時定数テーブル

人間の知覚・認知の時間スケールと UI 設計への含意。

| 時間スケール | 主な知覚・認知機能 | UIデザインへの含意 |
|------------|-------------------|------------------|
| **0.001秒** | 音の途切れを感知できる最短時間 | 音のフィードバックの最大許容遅延 |
| **0.01秒** | サブリミナル知覚の閾値、スタイラス軌跡の途切れを感知する最短時間 | タッチペン入力の「インク」追従の最大遅延。これを超えると書きにくく感じる |
| **0.1秒** | 因果関係の知覚閾値、手と目の協調（知覚運動フィードバック）、サッカード（眼球運動）の持続時間、「一瞬」として知覚できる時間上限 | クリック・タップへのフィードバック返却期限。0.1秒を超えると「自分の操作の結果」と感じられなくなる |
| **1秒** | 会話の「間」の最大値、予期しない出来事への視覚運動反応時間 | 1秒以内に処理完了かプログレスインジケータ表示が必要。超えるとユーザーは「何か起きた」と感じる |
| **10秒** | ユニットタスクへの集中力の持続時間 | ウィザードの 1 ステップ、1 回の編集操作などを 10 秒以内に収める |
| **100秒** | 緊急時の重要な意思決定にかかる時間 | 意思決定に必要な情報が 100 秒以内に得られる設計が必要 |

### インタラクション制限時間テーブル

設計現場で使う「大きさのオーダー」別の設計基準。

| 制限時間 | 設計基準 |
|---------|---------|
| **0.1秒以内** | ボタン押下・クリックへのフィードバック表示。処理が完了しなくてもよいが「受け付けた」を示す必要がある |
| **0.1秒以内** | 手と目の協調作業（ドラッグ・リサイズ・スクロール）のフィードバック。遅れると操作精度が落ちる |
| **1秒以内** | ビジーインジケータ（処理中表示）の表示。0.1秒で表示できない場合は 2 段階（0.1秒で受付確認→1秒でビジー表示） |
| **2〜3秒以上かかる処理** | プログレスインジケータを必ず表示する |
| **10秒以内** | 1 つのユニットタスク（1 回の編集操作、ウィザードの 1 ステップなど）の完了目安 |

### 応答性改善ガイドライン

#### プログレスインジケータの設計

| ガイドライン | 良い例 | 悪い例 |
|-------------|--------|--------|
| 残り作業量を表示する | 「4件中3件コピー済み」 | 「3件コピー済み」 |
| 全体の進行状況を表示する | 「あと15秒で終了」 | 「あと5秒でこのステップを完了」 |
| 開始時は 1% から始める | 1% → ... → 100% | 0% のまま数秒停止 |
| 100% は完了直前に一瞬だけ表示する | 完了時に瞬間表示 | 100% のまま数秒停止 |
| バーを滑らかに伸ばす | 一定速度で進む | 急に跳ぶ |
| 人間が理解しやすい単位を使う | 「約4分」 | 「240秒」 |

#### 重要情報の先行表示

処理が完了する前から有用な情報を表示することで、体感速度を上げられる：

- 文書を開くとき: まず最初のページを表示し、残りを後からロードする
- 検索結果: 検索しながら見つかったものをリアルタイムに表示する
- 画像: 低解像度で全体を先に表示し、高解像度に置き換えていく（テキストには不適用）

#### 遅延の許容場所

**ユニットタスク内部の遅延 > ユニットタスク間の遅延**（有害度の差）

| タイミング | ユーザーへの影響 |
|-----------|---------------|
| ユニットタスクの**途中**での遅延 | 作業記憶の内容が乱れ、何をしていたか忘れる。非常に有害 |
| ユニットタスクの**間**での遅延 | リラックスのタイミングと重なり、比較的許容される |

→ どうしても遅延が発生する場合は、**ユニットタスクの境界**に配置する

#### 入力処理の優先順位

入力を受付順（FIFO）で処理せず、**重要度順**で処理する：

- 「キャンセル」や「戻る」の操作は現在の処理を即中断して応答する
- よく使われる操作への応答を先読み計算しておく（アイドル時間を活用）
- 手と目の協調作業（ドラッグ等）のフィードバックを最高優先度にする

#### 品質と速度のトレードオフ

制限時間を守れない場合は**処理の質を一時的に低下させる**選択が有効：

- アニメーション: 20fps を維持できない場合、テキストラベルや 3D 効果を省略して描画を簡略化する
- 画像: ドラッグ中はアウトラインのみ表示し、ドロップ後に完全描画する
- **判断基準**: 一時的な品質低下は「フレームレートが落ちて動きがカクつく」より害が少ない

## よくある設計ミスと対策

| ミス | 問題 | 対策 |
|------|------|------|
| 処理中にビジーインジケータを表示しない | 「処理が始まったか」わからずユーザーが重複操作する | 0.1秒で応答できない処理にはすべてインジケータを表示 |
| プログレスバーが 0% のまま長時間停止する | システムがハングしたと誤解される | 1% からスタートし、常に滑らかに前進させる |
| 処理完了まで操作を受け付けない | ユーザーの作業ペースを乱す | 処理をバックグラウンド化し、UIはインタラクション可能に保つ |
| ユニットタスクの途中で応答が遅くなる | 作業記憶が乱れ、操作ミスが増加する | 重い処理をユニットタスクの境界にスケジューリングする |
| 応答性の問題をハードウェア交換で解決しようとする | 応答性はデザインの問題であり性能の問題ではない | インジケータ表示・先読み・優先度付けなどの設計で対応する |
