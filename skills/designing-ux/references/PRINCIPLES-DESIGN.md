# UX実践原則：Design（原則 64–87）

フェーズ「Design」は構造・視覚・インタラクションを具体化する段階。
IA・ナビゲーション・タイポグラフィ・アニメーション・データ可視化まで幅広い設計判断を扱う。

---

### 64. ブレインストーミングを効率化する

**核心**: 個人の発散思考の後にグループで収束させる2段階が最も効果的。

- 個人作業（1時間）→ グループ共有・発展（1時間）の構成が最も多様なアイデアを生む
- 全員が同時に話す旧来のブレインストーミングは声の大きい人のアイデアに収束する
- 「判断を保留する」「量を優先する」「他者のアイデアを発展させる」の3ルールを守る

**If/Then**: If ブレインストーミングを設定するなら then 個人での発散時間を先に確保し、その後グループで共有・発展させる

---

### 65. コンセンサスを確立する（ワイヤーフレーム承認）

**核心**: 詳細なビジュアルデザインに進む前に、ワイヤーフレームでレイアウト・フロー・コンテンツの合意を取る。

- ワイヤーフレームの承認なしに進むと、後の「好みの違い」によるやり直しが増える
- ステークホルダーの承認を書面（メール等）で記録する
- 承認は「変更なし」か「具体的な変更内容」のみ受け付け、「とりあえずOK」は避ける

**If/Then**: If ワイヤーフレームを提出するなら then 明示的な承認（YES/変更要求）を得た証跡を残す

---

### 66. 現実のナビゲーションから学ぶ

**核心**: 物理空間のナビゲーション設計（標識・地図・案内）の原則がデジタルUXに直接応用できる。

- 「現在地の表示」「行き先への道筋」「戻り道の確保」はデジタルでも同様
- パンくずナビ・アクティブ状態のメニュー強調・URLの可読性が「現在地感」を与える
- 迷子になったユーザーを最速で目的地に戻す仕組みを設計する

**If/Then**: If 多階層のナビゲーションを設計するなら then 「今どこにいるか」「どこから来たか」「どこへ行けるか」の3点をUIで常に示す

---

### 67. 情報アーキテクチャ（IA）を構築する

**核心**: コンテンツの構造・分類・命名を設計することで、ユーザーが情報を予測可能に発見できる。

- サイトマップはIAの設計ツール。コンテンツ量が増える前に全体構造を設計する
- ナビゲーションラベルはユーザーの言葉で命名する（社内用語・業界用語を避ける）
- IA設計にはカードソーティング（→原則63）の結果を反映する

**If/Then**: If 新しいセクションを追加するなら then 既存のIA全体への影響を評価してから配置を決定する

---

### 68. サイトマップを作る

**核心**: 全ページ・全画面の階層構造を図示したサイトマップが、IA実装の共有基盤になる。

- サイトマップは設計の出発点。後から追加されるページの配置判断の根拠にもなる
- 階層は最大3レベルを目安に。深すぎるとナビゲーションが複雑になる
- 開発・コンテンツ・SEOチームとの共通言語として機能する

**If/Then**: If プロジェクトのページ数が10を超えるなら then 正式なサイトマップを作成し、ステークホルダー全員と共有する

---

### 69. ナビゲーションの10原則

**核心**: 優れたナビゲーションが満たすべき10の基準。

1. **発見可能**: ユーザーが探さなくても目に入る
2. **明確**: ラベルが行き先を正確に表す
3. **一貫性**: 全ページで同じ構造・位置・スタイル
4. **フィードバック**: 現在地を強調表示
5. **効率性**: 目的地への最短パス
6. **柔軟性**: 複数の経路（検索・パンくず・関連リンク）
7. **予測可能**: リンクをクリックする前に行き先がわかる
8. **拡張性**: コンテンツ増加に対応できる構造
9. **アクセシビリティ**: キーボード操作・スクリーンリーダー対応
10. **モバイル対応**: タッチ操作・小画面での操作性

**If/Then**: If ナビゲーションをレビューするなら then この10原則をチェックリストとして使い、満たしていない項目を改善する

---

### 70. サイドドア（直接アクセス）を設計する

**核心**: ユーザーの大半はトップページではなく、検索・SNS・外部リンクから内部ページに直接入る。

- 各ページがスタンドアロンで意味をなすコンテキスト情報を持つべき
- パンくずナビ・グローバルナビが全ページに存在することでサイドドア入口でも迷子にならない
- OGPタグ・メタディスクリプションは「サイドドア」のための見た目

**If/Then**: If コンテンツページを設計するなら then トップページからの流れを知らないユーザーが見ても意味がわかる構成にする

---

### 71. プレゼン前に文脈を説明する

**核心**: デザインをステークホルダーに見せる前に、「誰のため・何の問題を解く設計か」を説明する。

- コンテキストなしにビジュアルを見せると「好みの話」に終始する
- 「ユーザーは〜で困っていた。この設計は〜で解決する」という形式で提示する
- デザインの評価基準（ユーザー目標の達成度）を先に合意する

**If/Then**: If デザインレビューを実施するなら then スライドの1枚目にプロブレムステートメントを置く

---

### 72. フィデリティを使い分ける

**核心**: プロジェクトフェーズとフィードバックの目的に応じて、プロトタイプの忠実度を選ぶ。

- **低フィデリティ**: 紙・スケッチ。フローと構造の検証。コスト低い
- **中フィデリティ**: グレースケールワイヤーフレーム。情報設計とインタラクションの検証
- **高フィデリティ**: ビジュアルデザイン完成版。視覚的印象・ブランドの検証
- 高フィデリティでしか発見できない問題があるが、早すぎる高フィデリティは変更コストが高い

**If/Then**: If プロトタイプを作成するなら then 今何を検証したいかを先に決め、それに最低限必要なフィデリティを選ぶ

---

### 73. アノテーションでデザイン意図を伝える

**核心**: デザインファイルに動作・状態・ルールの注記（アノテーション）を追加し、開発者との齟齬を防ぐ。

- 「ホバー時・エラー時・空状態・最大入力時」などのエッジケースをアノテーションで示す
- アニメーションの速度・イージング・トリガーを記述する
- アノテーションのないデザインは「完成していないデザイン」

**If/Then**: If デザインを開発に渡すなら then すべてのインタラクティブ要素のステート（通常・ホバー・押下・無効・エラー）をアノテーションで明示する

---

### 74. インタラクションはブランドである

**核心**: アプリの動き方・操作の感触・フィードバックの質がブランドの個性を形成する。

- AppleのSpring Animationは「弾力・応答性・活力」のブランドを伝える
- 遅いアニメーション・不自然なトランジションは製品品質への不信感を生む
- マイクロインタラクション（いいねアニメーション・送信確認等）が感情的記憶を作る

**If/Then**: If アニメーションを設計するなら then ブランドの個性（活発・落ち着き・遊び心等）をインタラクションで表現する機会を探す

---

### 75. タイポグラフィを制する

**核心**: フォント選択・サイズ・行間・文字間隔・コントラストが可読性と雰囲気を決める。

- 本文テキストは最低16px（Webでは1rem）を基準にする
- 1ページで使うフォントファミリーは2つまで（見出し・本文）
- コントラスト比：本文4.5:1以上、大きいテキスト3:1以上（WCAG AA）
- 行長は45〜75文字（英語）が最適な可読性

**If/Then**: If タイポグラフィシステムを設計するなら then 見出し・本文・補助・キャプションの4レベルで規定し、コントラスト比を全て確認する

---

### 76. スクローリーテリング

**核心**: スクロールの動きに連動してコンテンツが現れる手法で、物語を体験させる。

- ランディングページやプロモーションサイトで価値を段階的に開示するのに有効
- スクロール量に応じたアニメーションは操作感と達成感を生む
- アクセシビリティ（視差酔い・reduce-motion設定）に配慮する

**If/Then**: If スクロールトリガーのアニメーションを実装するなら then `prefers-reduced-motion`メディアクエリを設定し、アニメーションを無効化するフォールバックを用意する

---

### 77. アニメーションの責任

**核心**: アニメーションには明確な目的が必要。装飾的なアニメーションはパフォーマンスと認知負荷を増やす。

- **機能的アニメーション**: 状態変化の視覚化・操作フィードバック・注意誘導
- **装飾的アニメーション**: ブランド演出・楽しさ・驚き（二次的）
- 過剰なアニメーションは操作の邪魔になり、前庭障害を持つユーザーに不快感を与える

**If/Then**: If アニメーションを追加するなら then 「このアニメーションがなければユーザーが何を失うか」を問い、答えがなければ削除する

---

### 78. データ可視化の原則

**核心**: データをグラフや図で表示するときは、誤解を招かない・必要な情報を強調する設計が必須。

- チャートタイプの選択：比較→棒グラフ、時系列→折れ線、割合→帯グラフ（円グラフより）
- Y軸はゼロから始める（切り取ると変化が誇張される）
- データインクレシオ（Tufte）：不要な要素を排除し、データの量とインク量の比率を最大化する

**If/Then**: If ダッシュボードを設計するなら then 各グラフの「ユーザーが知りたい問い」を先に定義し、その問いに最適なチャートタイプを選ぶ

---

### 79. ダークモードは選択肢として提供する

**核心**: ダークモードはアクセシビリティ・目の疲れ・電力節約に有効だが、全員に向くわけではない。

- OSのダークモード設定を`prefers-color-scheme`で検出してデフォルトに使う
- 独自の切り替えオプションを提供し、ユーザーが上書きできるようにする
- ダークモードでのコントラスト・シャドウ・アイコンの可視性を別途検証する

**If/Then**: If ダークモードを実装するなら then ライトモードと独立してコントラスト比・アイコン・画像の視認性をすべて確認する

---

### 80. コントロールを制限する

**核心**: ユーザーに与える設定・オプション・カスタマイズの量は最小限に抑える。

- 選択肢が多すぎると意思決定疲れが起き、何もしない（デフォルトを使い続ける）になる
- 製品がベストな設定を判断できるなら、ユーザーに判断させない（スマートデフォルト）
- 上級ユーザー向けの詳細設定は「詳細設定」セクションに隔離する

**If/Then**: If 設定画面を設計するなら then ユーザーが変更すべき設定と変更不要な設定を分類し、後者はスマートデフォルトで自動設定する

---

### 81. パーソナライズは諸刃の剣

**核心**: ユーザーに合わせたコンテンツ・機能・レイアウトは体験を向上させるが、プライバシーリスクとフィルターバブルを伴う。

- パーソナライズには明示的同意とプライバシーポリシーの透明性が必要
- アルゴリズムによる推薦は「知らなかった情報」への露出を減らす（フィルターバブル）
- ユーザーがパーソナライズ設定を確認・修正できるコントロールを提供する

**If/Then**: If パーソナライズ機能を実装するなら then 「なぜこのコンテンツが表示されているか」をユーザーが確認できる仕組みを設ける

---

### 82. アイコンの限界を知る

**核心**: アイコン単体では意味が伝わらないことが多い。テキストラベルとの組み合わせが基本。

- ユニバーサルに理解されるアイコンは非常に少ない（ハンバーガー・ホーム・検索・共有程度）
- 新しいコンセプトを表すアイコンはほぼ必ずラベルが必要
- アイコンとラベルの組み合わせはどちらか単体より認識速度が速い

**If/Then**: If アイコンのみのナビゲーションを設計するなら then 少なくともモバイルではアイコン+テキストラベルの組み合わせに変更する

---

### 83. B2BとB2Cでファネルが違う

**核心**: 個人消費者向けと企業向けでは意思決定プロセス・関係者・購入サイクルが根本的に異なる。

- **B2C**: 短い購入サイクル・感情的決定・1〜2人の意思決定者・低価格
- **B2B**: 長い購入サイクル・合理的決定・複数の承認者・高価格・ROI重視
- B2B UXでは製品デモ・ROI計算機・ケーススタディが重要なコンバージョン要素

**If/Then**: If B2B製品を設計するなら then 複数の意思決定者（エンドユーザー・管理者・購買担当）それぞれのニーズを個別に分析する

---

### 84. デバイス選択の重要性

**核心**: どのデバイスを一番最初に・最も力を入れて設計するかが体験の質を決める。

- 最多使用デバイスから設計し、他デバイスに適応させる（モバイルファーストの哲学）
- 特定デバイスでしか実現できない体験（カメラ・GPS・ウィジェット等）を活かす
- デバイスを横断した一貫性と、各デバイスの固有の強みのバランスを取る

**If/Then**: If マルチデバイス製品を設計するなら then ユーザーの主要デバイスをデータで特定し、そのデバイスで最高の体験を作ることを優先する

---

### 85. デザインシステムは大企業向け

**核心**: デザインシステムの構築・維持にはコストがかかる。規模に見合う場合にのみ投資する。

- 5人以下のデザイナー・1製品・単一チームにはデザインシステムより共有コンポーネントライブラリで十分
- 大規模チーム・複数製品・外部開発者がいる場合は投資対効果が高い
- デザインシステムは「作ったら終わり」ではなく継続的なメンテナンスが必要

**If/Then**: If デザインシステム構築を検討するなら then 「何チームが何製品で使うか」と「メンテナンスリソース」を評価し、投資判断する

---

### 86. モジュール化されたデザイン

**核心**: UIを独立した再利用可能なコンポーネントとして設計することで、一貫性と開発効率が向上する。

- アトミックデザイン（Atoms→Molecules→Organisms→Templates→Pages）で設計を構造化する
- コンポーネントは1つの責務を持ち、親コンテキストに依存しない
- 変更が局所化されるためリファクタリングが容易になる

**If/Then**: If UIを設計するなら then 同じ要素を2箇所以上で使う場合はコンポーネントとして定義する

---

### 87. 不測の事態を予測する

**核心**: ハッピーパスだけでなく、エラー・空状態・タイムアウト・オフライン・低帯域などのエッジケースを設計する。

- 「最初の使用」「エラー発生時」「データゼロ時」「読み込み中」の4状態は必ず設計する
- エラー状態のデザインが貧弱な製品は、問題が起きたときにユーザーを最も失望させる
- ローディング中・エラー時・空状態を設計せずにリリースするのは未完成の製品

**If/Then**: If 新しい機能を設計するなら then 「通常・読み込み中・エラー・空状態」の4ステートを必ず設計してからデザインファイルを閉じる
