# 構造段階 — インタラクションデザインと情報アーキテクチャ

要件段階が「何を作るか」を明確にするとすれば、構造段階は「それらのパーツをどのように組み合わせて全体を作るか」を定義する。
抽象的な要求を概念的な構造（ユーザーが経験するパターンと順番）に変換する段階。

---

## 1. 構造段階の2つの軸

| 軸 | 適用範囲 | 起源 |
|----|---------|------|
| **インタラクションデザイン** | 機能性重視の製品。タスクを実行・完了するための選択肢のパターンと順番 | ソフトウェア工学 |
| **情報アーキテクチャ** | コンテンツ重視の製品。ユーザーへの情報伝達にかかわる選択肢 | 図書館学・ジャーナリズム・テクニカルコミュニケーション |

**本質**: どちらも「人を理解すること」—— 人々の行動や考え方を理解し、それを製品の構造に組み込む分野。

---

## 2. インタラクションデザイン

**定義**: ユーザーがどのような行動を取りうるかを示し、その行動にシステムがどのように対応するかを定義すること。

ユーザーと製品は「ダンス」をしている。ユーザーが動けばシステムが応え、またユーザーがシステムに応じて動く。ダンスを成功させるには、それぞれの参加者が相手の動きを予測しなければならない。

### 2.1 概念モデル（Conceptual Models）

**定義**: インタラクティブコンポーネントの振舞い方に対してユーザーが抱く印象の体系。「これはコンテナだ」「これは場所だ」「これは物だ」といった理解の枠組み。

#### 概念モデルの役割

- デザイン上の判断を**一貫させる**（あるときは「場所」、別のときは「物」として扱うのはNG）
- ユーザーの暗黙の期待とシステムの振舞いが一致することで、直感的な理解が生まれる
- 明示的に説明する必要がない（むしろ説明すると混乱する場合もある）

#### 具体例

| 概念モデル | メタファー | 典型的なアクション |
|-----------|-----------|----------------|
| ショッピングカート（コンテナ） | 物理的なカート | 入れる / 取り出す / チェックアウト |
| カタログオーダーフォーム | 注文書 | 編集する / 送信する |
| 雑誌（Slateの事例） | 紙の雑誌 | ページをめくる（ウェブでは効果的でなかった失敗例） |

**慣習（Convention）の重要性**: ウェブ上で広く使われている概念モデルは慣習化している。ユーザーが他のサイトでよく使う概念モデルなら、それに従うと適応が容易。

**If X then Y**:
- If 慣習から外れた概念モデルを採用したい → then そうするだけの理由があり、代わりの概念モデルがユーザーニーズを十分満たすことを確認する
- If 現実世界のメタファーにもとづいた概念モデルを作る → then 文字どおりに捉えすぎない（電話のメタファーを使っても実際に電話を表示する必要はない）
- If コンポーネントの扱いが一貫していない → then 概念モデルが混在している証拠。統一する

---

### 2.2 エラーハンドリング

インタラクションデザインの大部分は、ユーザーエラーへの対応で構成される。

#### エラーハンドリングの階層（優先度順）

| レイヤー | 内容 | 例 |
|---------|------|-----|
| **1. エラー防止** | エラーが起こらないようシステムを設計する | オートマチック車は変速中に発進できない |
| **2. エラー発生困難化** | できるかぎりエラーが起きにくくする | 入力フォームの制約、選択肢の絞り込み |
| **3. エラー理解支援** | ユーザーがエラーを理解し修正できるよう支援する | わかりやすいエラーメッセージ、解釈しやすいUI |
| **4. システムによる自動修正** | システムがエラーを修正する（善意だが注意が必要） | 自動補完、スペルチェック（ユーザーが望まない修正も起きる） |
| **5. エラー回復** | ユーザーがエラーから回復できる手段を提供する | Undo機能、確認ダイアログ |

**確認ダイアログの注意点**: 「よろしいですか？」を何度も表示すると、本当に重要な警告が見落とされる。助けられるユーザーより迷惑をかけるユーザーが多くなることもある。

**If X then Y**:
- If 特定のエラーが頻発する → then まずエラーを「不可能にする」設計を検討する
- If システムが自動修正する → then ユーザーが意図して行ったことまで修正しないか確認する
- If 回復不能なエラーがある → then 警告を早期かつ明確に出し、確実にユーザーに届くようにする

## 3. 情報アーキテクチャ

**目的**: ユーザーが情報を効率的かつ効果的に閲覧・発見できるよう、組織やナビゲーションの仕組みを作ること。人が情報を認知的にどう処理するかに関係する。

### 3.1 コンテンツの構成アプローチ

| アプローチ | 開始点 | 特徴 | リスク |
|-----------|-------|------|--------|
| **トップダウン** | 製品目標とユーザーニーズ → 大分類 → サブカテゴリー | 戦略と直結した構造 | コンテンツ自体の重要な細部が見落とされる |
| **ボトムアップ** | 既存コンテンツ → 低レベルカテゴリー → 高レベルカテゴリー | 実際のコンテンツを反映した構造 | 既存コンテンツ向けに固定化され、変更・追加への柔軟性が低い |

**推奨**: 両アプローチのバランスを取ることが、落とし穴を回避する唯一の方法。

### 3.2 情報構造の4タイプ

情報の基本単位は**ノード（node）**（ページ・セクション・コンテンツ要素など抽象化された単位）。ノードの配置パターンは主に4種類。

| 構造タイプ | 特徴 | 最適なユースケース |
|-----------|------|-----------------|
| **階層型（ツリー／ハブ&スポーク）** | 親子関係。すべてのノードにルートへの親がある | ほとんどのウェブサイト。ユーザーが理解しやすく、ソフトウェアも階層的に機能する傾向がある |
| **マトリクス** | 2つ以上の次元に沿ってナビゲート可能 | 異なるニーズを持つユーザーが同じコンテンツを別の軸で探す場合（色別・サイズ別など）。4次元以上は人間が視覚化できない |
| **有機（オーガニック）** | 一貫したパターンなし。ケースバイケースの接続 | 自由な探索を楽しんでほしいエンタメ・教育サイト。ただし、同じコンテンツへの確実な再アクセスは困難 |
| **シーケンシャル（順次的）** | 直線的な流れ | 記事・音声・動画などの単一コンテンツ。操作手順など「順番が重要」なコンテンツ |

**構造の品質評価**: ステップ数より「各ステップがユーザーにとって意味があるか」「前のステップから自然に続いているか」が重要。わかりにくい3ステップより明確な7ステップを好む。

### 3.3 構造の適応性

サイトは生き物。構造は**成長と変化に対応できる**ことが重要な品質指標。

**If X then Y**:
- If 新しいコンテンツが追加され続ける → then 現在の組織化原則が長期的に機能するかを定期的に見直す
- If 戦略目標やユーザーニーズが変わった → then 構造の見直しを検討する（必要性に気づく頃にはユーザーがすでに不便を経験している可能性）

## 4. 組織化原則

**定義**: ノードをグループ化する基準。「どのノードをまとめ、どのノードを別にするか」を決める論理。

### 4.1 階層レベルと組織化原則の関係

| アーキテクチャのレベル | 組織化原則の影響要因 |
|----------------------|-------------------|
| **上位レベル** | 製品目標・ユーザーニーズと密接に結びついた原則（例：対象ユーザー別・地域別） |
| **中間レベル** | コンテンツのドメイン特性（例：ニュースなら時系列が最重要） |
| **下位レベル** | コンテンツや機能要求に特有の問題（例：スポーツ→競技種別、国際ニュース→地域別） |

### 4.2 ファセット（facets：分類の切り口）

ファセットはほぼどんなコンテンツでも使えるシンプルで柔軟な組織化原則。

| 例（自動車情報サイト） | 消費者向け | プロ（海外輸送業者） |
|----------------------|-----------|------------------|
| **メーカー・モデル・車種** | 最適 | あまり重要でない |
| **重量** | ほぼ不要 | 非常に重要 |
| **燃費・エンジンタイプ** | 重要 | あまり重要でない |

**アンチパターン**: 「すべてのファセットをユーザーに選ばせる」はアーキテクチャを混乱させる。ファセットを選ぶ負担はユーザーではなく設計者が負う。

**If X then Y**:
- If 複数の組織化原則が考えられる → then 製品目標とユーザーニーズに合った原則を選ぶ。設計者がユーザーの期待を予測し説明できることがUX成功の鍵
- If 間違ったファセットを使っている → then 何も使わないより悪い

## 5. 言語とメタデータ

### 5.1 制限語彙（Controlled Vocabulary）

構造が完璧でも、ユーザーが命名法（サイトで用いる用語・ラベル・説明）を理解できなければ、アーキテクチャを活かせない。

**制限語彙**: サイトで用いる標準的な用語を集めたもの。

**目的と効果**:
- ユーザーの言葉を反映させ、組織内ジャーゴンの混入を防ぐ
- コンテンツ制作担当者全員が同じ用語を使えるようになる（場所が異なるチームでも）
- 一貫性によってコンテンツ間のつながりの定義が自動化できる

**シソーラス（類語集）**: 制限語彙のより洗練されたバージョン。承認された用語に加え、禁止用語（内輪の略語・俗語・頭字語）を承認済み用語にマッピングし、広義・狭義・関連用語も記録する。

### 5.2 メタデータ

**定義**: 「情報についての情報」—— コンテンツを描写するための構造化されたアプローチ。

#### メタデータの例（製品活用事例の記事）

```
・執筆者名
・投稿日時
・コンテンツタイプ（ケーススタディ / 記事 等）
・製品名
・製品タイプ
・顧客の業種
・関連情報（地方自治体、救急サービス 等）
```

#### メタデータがもたらす価値

| 価値 | 内容 |
|------|------|
| **構造の柔軟性** | 詳細なメタデータがあるほど、コンテンツ構造を柔軟に変更できる |
| **動的リンク** | 特定トピックのページを自動的にリンクさせることが可能 |
| **検索強化** | シソーラスと組み合わせることで、禁止用語の検索を優先用語にマッピングし、ゼロヒットを回避 |
| **新市場対応** | 戦略転換時に、既存コンテンツをメタデータで新セクションにすばやく配置できる |

**If X then Y**:
- If メタデータを収集するシステムを作る → then データ自体の一貫性がなければ機能しない。制限語彙で一貫性を確保してから
- If 検索エンジンをより賢くしたい → then シソーラスと連携させ、関連テーマのお勧めも提供できるようにする

## 6. チームの役割とドキュメント

### 6.1 構造を担う役割

| 役割名 | 専門領域 | 備考 |
|--------|---------|------|
| **情報アーキテクト** | コンテンツの組織化・ナビゲーション構造 | インタラクションデザインの経験も積むことが多い |
| **インタラクションデザイナー** | タスクフロー・エラーハンドリング | 情報アーキテクチャの経験も積むことが多い |
| **ユーザーエクスペリエンス・デザイナー** | 上記両方を包含する一般的な肩書き | — |

**重要**: 専任者の有無より、**誰かが責任を持つこと**が重要。計画された構造を持つサイトは、点検頻度が少なくて済み、ユーザーニーズを満たし、成果を上げる。

### 6.2 構造のドキュメント化

構造の複雑さをテキストで伝えようとすると誰も読まなくなる。ビジュアルによる表現が最も効率的。

| ドキュメントタイプ | 適用場面 |
|------------------|---------|
| **テキストアウトライン** | 階層構造に大量コンテンツを持つシンプルなプロジェクト |
| **スプレッドシート / データベース** | 複雑なアーキテクチャのニュアンスを把握したい場合 |
| **アーキテクチャダイアグラム** | ほとんどのプロジェクトで最適。分岐・グループ・相互関係をビジュアルに表現 |

**ダイアグラムの注意点**: 全ページの全リンクを記録する必要はない。詳しすぎると重要な情報がぼやける。概念的な関係（どのカテゴリーが一緒になるか、インタラクションのステップがどうまとまるか）のドキュメント化が目的。
