# UX実践原則：Research（原則 52–63）

フェーズ「Research」は仮説を検証し設計判断に根拠を与える段階。
リサーチの限界と有効性を理解した上で、適切な手法を選択・組み合わせる。

---

### 52. 客観的なデザインは不可能

**核心**: デザイナーが作るすべての選択は主観を含む。それを前提にプロセスを設計する。

- デザイナーの経験・文化・先入観はデザインに無意識に反映される
- 「客観的なデータが証明している」という主張も、データの選択と解釈は主観的
- 多様なステークホルダーのレビューと継続的なユーザーテストが主観バイアスを緩和する

**If/Then**: If デザイン判断を下すなら then 「自分のバイアスがこの判断に影響していないか」を自問し、反証となる情報を探す

---

### 53. リサーチは万能ではない（科学主義批判）

**核心**: UXリサーチは有用だが、すべての問いに答えられるわけではない。過剰な信仰は危険。

- ユーザーは自分の行動を正確に報告できない（言ったことと実際の行動は異なる）
- アンケートは「理想の自分」への回答を集めやすく、実態と乖離する
- リサーチの価値は「完全な真実」ではなく「仮説の棄却・強化」に限定して活用する

**If/Then**: If リサーチ結果を設計判断に使うなら then 結果の限界（サンプル数・調査法の制約）を明示した上で判断する

---

### 54. リサーチの適量を見極める

**核心**: リサーチへの過少投資と過剰投資はどちらも問題。適切な量とタイミングが重要。

- 小さな決定に大規模なリサーチを投資するのは費用対効果が低い
- リサーチなしに大きな設計変更を行うのはリスクが高い
- 「この判断が間違った場合のコスト」と「リサーチのコスト」を比較して判断する

**If/Then**: If リサーチ実施を判断するなら then 「この決定が間違った場合の影響」を評価し、それに見合うリサーチ規模を選ぶ

---

### 55. エコシステムマッピング

**核心**: 製品が存在するエコシステム（関係者・デバイス・サービス・外部要因）全体を可視化する。

- 直接ユーザー以外のステークホルダー（管理者・保護者・承認者等）の影響を把握する
- 製品が依存する外部サービス・APIのリスクを洗い出す
- 競合製品・代替手段（Excelで管理等）をエコシステムの一部として把握する

**If/Then**: If 新サービスを設計するなら then ステークホルダーマップを作成し、直接ユーザー以外の関係者の期待・制約を可視化する

---

### 56. データ分析でパターンを発見する

**核心**: 行動データ（クリック・スクロール・離脱）はユーザーの実態を表す最も信頼性の高い情報源の一つ。

- ヒートマップ・ファネル分析・セッションリプレイで実際の操作パターンを把握する
- 数値は「何が起きているか」を教えるが、「なぜそうなのか」は教えない
- 定量データの異常値は定性リサーチの出発点になる

**If/Then**: If デザイン改善を検討するなら then まず現在の行動データを確認し、定性調査が必要な仮説を立てる

---

### 57. 定性調査が本質を明らかにする

**核心**: 数値では見えないユーザーの動機・感情・コンテキストを定性調査で深掘りする。

- ユーザーインタビュー・文脈的調査（コンテキスチュアルインクワイアリー）は「なぜ」に答える
- 「5つのなぜ」で表面的な問題の根本原因を探る
- 5〜7名のインタビューで全体の傾向の約80%が現れると言われている

**If/Then**: If 定量データで異常な離脱ポイントを発見したなら then 5名以上のインタビューでその理由を深掘りする

---

### 58. 定量テストの適切な規模（25〜100人）

**核心**: 統計的に有意な結論を得るには適切なサンプルサイズが必要。少なすぎると誤った結論を導く。

- A/Bテストは最低でも各バリアント100〜500セッションが必要（コンバージョン率による）
- 統計的有意差がなければ「どちらでもよい」ではなく「まだわからない」
- サンプルサイズを事前に計算し、達する前に結論を出さない（ピーキングバイアス）

**If/Then**: If A/Bテストを実施するなら then 事前にサンプルサイズ計算を行い、統計的有意差が得られる数に達するまでテストを続ける

---

### 59. ペルソナは思い込みを防ぐツール

**核心**: ペルソナはリサーチデータから作成する架空のユーザー像。チームの思い込みに対抗する。

- 「自分がユーザーだったら」という思考が設計を歪める。ペルソナがその代替
- リサーチなしのペルソナは「デザイナーの理想像」に過ぎない
- ペルソナは生きている文書。リサーチのたびに更新する

**If/Then**: If ペルソナを作成するなら then 最低5名以上のユーザーインタビューデータを基に作成し、「仮定」と「実データ」を明確に区別する

---

### 60. 競合分析（4〜7社）

**核心**: 4〜7社の競合製品を体系的に分析し、業界標準・差別化ポイント・ユーザー期待を把握する。

- 直接競合（同機能）・間接競合（同ユーザー課題を別手段で解決）・ベストインクラス（異業種の優れたUX）を分析する
- 競合のUI慣習を知ることで、破っていい慣習と守るべき慣習を判断できる
- 競合調査は模倣の許可証ではなく、戦略的差別化の根拠を見つける活動

**If/Then**: If 新機能を設計するなら then 同機能を提供する競合4〜7社の実装を試し、自社設計の根拠と差別化ポイントを文書化する

---

### 61. ヒューリスティック分析（専門家評価）

**核心**: Nielsenの10ヒューリスティックなど確立された原則を使い、専門家がUIを評価する手法。

- ユーザーテストより低コストで素早く問題を発見できる
- 主要な10ヒューリスティック：システム状態の可視化・実世界との一致・ユーザーの自由・一貫性・エラー防止・記憶より認識・柔軟性・エレガントなデザイン・エラーリカバリー・ヘルプ
- ヒューリスティック分析はユーザーテストの代替ではなく補完

**If/Then**: If 既存製品の改善点を探すなら then ヒューリスティック分析を実施し、重大度（1〜4）と影響範囲でソートした問題リストを作成する

---

### 62. メンタルモデルに合わせる

**核心**: ユーザーは世界の動き方についての内的なモデルを持っている。UIがそのモデルに反すると混乱する。

- Gmailのスター機能はユーザーが「重要マーク」と理解する（実際は検索フィルター）
- メンタルモデルを変えるのは高コスト。既存モデルに合わせた設計が学習コストを下げる
- 新技術（AR・AI等）はユーザーのメンタルモデルが未形成なため教育が必要

**If/Then**: If 既存のUIパターンを変更するなら then ユーザーの現在のメンタルモデルを調査し、変更コストを見積もる

---

### 63. カードソーティングで情報構造を検証する

**核心**: ユーザーがコンテンツをどのように分類・命名するかを調査し、IA（情報アーキテクチャ）の根拠とする。

- オープンカードソーティング：ユーザーがカテゴリを自由に作る（発見的）
- クローズドカードソーティング：既存カテゴリへの分類（検証的）
- ツリーテストで階層ナビゲーションの有効性を検証する

**If/Then**: If ナビゲーション構造を設計するなら then 20〜30名のカードソーティングでユーザーの分類モデルを把握し、それを情報アーキテクチャに反映する
