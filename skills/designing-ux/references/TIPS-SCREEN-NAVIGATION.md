# UIデザイン実践TIPS: 画面設計・遷移・実践知

## 画像の配置判断

### リスト内の画像位置と情報優先度

| 配置 | ユーザーの認識順序 | 適したケース |
|------|------------------|------------|
| **左配置** | 画像 → タイトル → 説明 | 写真が直感的・商品が主役・画像ありが前提 |
| **右配置** | タイトル → 説明 → 画像 | テキストが主役・画像有無が項目ごとに変わる |

**鉄則**: 画像の有無が項目によって異なる場合は**右配置**を選ぶ。左配置すると画像がない行でテキスト位置がずれ、比較しづらくなる。

### 感性訴求レイアウト

| パターン | 特徴 | 向く用途 |
|---------|------|---------|
| **全面写真** | 写真大きく・文字は画像上に最小限 | 宿泊/グルメ予約・なんとなく眺める体験 |
| **不揃い2ペイン** | 写真サイズに合わせた可変高さ | 画像コレクション・探索体験・雑誌的体験 |
| **整列リスト** | 統一高さ・比較しやすい | 目的の商品/記事を素早く見つける用途 |

### 一覧→詳細の情報量差

一覧をタップした先には「さらなる情報」があることを早期に示す。タップ直後の詳細画面が一覧と情報量が同じだとユーザーは価値を感じにくい。
→ 2枚目以降の写真がチラリと見える構成が効果的。

## 長いコンテンツの整理テクニック

### 要素を追加する前に確認すること

1. 類似機能がないか → あれば削除または統合できないか
2. 既存要素と面積を「半分ずつ」にできないか
3. 追加後に既存要素のクリック数が奪われないか（効果の最大化）

### 面積比率のルール化

```
本来あるべき要素: 70%
関連情報・広告等: 30%
```

このルールを超えない範囲でのみ要素を追加できるようにすると、無制限な肥大化を防げる。

### 長くなった場合の対処法

| 手法 | 説明 | 効果 |
|------|------|------|
| **キー要素の直前に配置** | ユーザーが必ず見る「口コミ」「地図」等の手前に追加要素を置く | スクロール途中で視野に入りやすい |
| **要素を分散させる** | 一覧画面で5件に1件ほど別要素を挟む | 目的探しの視点を変えられる |
| **一部を隠す** | 必要なユーザーだけタップして展開 | 画面はシンプルに保ちつつ情報提供 |

## メニュー項目が多いときの対処法

### 横配置メニューのデメリット

- **表示件数が少ない**: スマートフォンの横幅では最大5件程度
- **階層構造が表現しにくい**: 縦配置なら階層をインデントで表現できる

### 件数増加時の4パターン比較

| パターン | ユーザーへの認識 | タップのしやすさ | 適した状況 |
|---------|---------------|---------------|----------|
| **スクロール** | 気付かれにくい | ◎ | カテゴリ等、多数あることをユーザーが推測できる場合 |
| **最後に「その他」** | ◎（確実に伝わる） | 1タップ増 | 多様なメニューを認識してほしい場合 |
| **複数段** | ◎（全量が見える） | ◎ | 画面途中への配置、固定でない場合 |
| **長押しで表示** | × （気付かれにくい） | 慣れれば便利 | ツール型・習熟ユーザーが多いアプリ |

## 「もっと見る」の設計パターン

### 使用シーン

| シーン | 目的 | 代替手段 |
|--------|------|---------|
| **複数切り口をコンパクトに** | サービス全体像の目次的提示 | カルーセル |
| **連続する情報を途中で切る** | 検索結果・一覧の続き読み込み | ページャ・オートページャ |

### 配置位置と効果

| 配置 | 長所 | 短所 |
|------|------|------|
| **タイトル横（右寄せ）** | 枠の高さを節約できる | タップ率が低い傾向 |
| **コンテンツ下** | ユーザーの目線に入りやすい | スペースをとる |

> 「もっと見る」へ誘導したいなら**下配置**。コンテンツ直接タップを促したいなら**右寄せ**。

### 「画面遷移」vs「その場で開く」

| 方式 | 実装コスト | 位置記憶 | 向く用途 |
|------|----------|---------|---------|
| **その場で展開** | 高（戻り時の状態復元が必要） | 失いやすい | 繰り返し閲覧 |
| **別ページへ遷移** | 低 | ページとして記憶できる | 情報探索・比較 |

### 「もっと見る」vs「カルーセル」

| 観点 | もっと見る（縦スクロール） | カルーセル（横スクロール） |
|------|--------------------------|--------------------------|
| 情報の快適な閲覧 | ◎（慣れ親しんだ操作） | △（スワイプしないユーザーが多い） |
| 複数切り口の横断 | △（遷移が増える） | ◎（画面遷移なしに気軽にアクセス） |

## 入力フォームの分割判断

### 分割するかの基準テーブル

| 状況 | 推奨 | 理由 |
|------|------|------|
| **CV重視（登録完了率を上げたい）** | **分割しない** | 画面数が多いと離脱ポイントが増える |
| **モバイル + 項目が多い** | **分割する** | キーボード出現+スクロールで操作しにくい |
| **じっくり編集・後から更新が多い** | **分割する** | カテゴリ分けで目的の項目を見つけやすくなる |
| **B2Bサービスの詳細設定** | **分割する** | 業務情報は種類ごとに整理すると管理しやすい |

### 分割しないときの工夫

| 工夫 | 説明 |
|------|------|
| **チャットUI** | 質問に答えるだけで完了。スクロール不要・受動的に入力させる |
| **入力タイミング分割** | 必須のみ先行登録→任意項目は後から促す |

### 分割する場合の2方式

| 方式 | 説明 | 向く状況 |
|------|------|---------|
| **種類で分ける** | 入力内容のカテゴリ（場所/料金/詳細等）で分割 | 項目が多岐にわたる・後から更新しやすい |
| **重要度で分ける** | 全員必須の項目 → 任意項目の順で分割 | 習熟度差が大きい・項目を減らせない状況 |

## 画面遷移の改善手法

### 画面単位改善の落とし穴

- 小さな改善の積み重ねで**部分最適**が起きる
- 改善した箇所以外が**改悪**になることがある（前後の文脈の違和感）
- 施策が小粒でユーザーが変化を実感できないレベルにとどまる

### 改善手順

**Step 1: ストーリー単位で考える**
ユーザー目線で「1回の利用で完結するストーリー」を文章で書き出す。
短すぎると画面単位と大差なく、長すぎると改善点がぶれる。

**Step 2: すべての画面変化を書き出す**
ストーリーに沿って通過する画面を書き出す。
ダイアログ・インジケータ・トランジションも含めて可視化する。
→ 目に見えにくい課題（特定画面の遅延・不要なインジケータ・フローが長い）が見えてくる。

**Step 3: 課題と解決案を分けて記載**
```
課題 (Issue): ここに違和感がある
解決案 (Suggestion): ここをこう改善したい
```
1つの課題に複数の解決案があることも多い。分けて記録することで実装コストが変わっても課題は失われない。

## デザインガイドライン（HIG/Material Design）との付き合い方

### 現状の変化

- iOS/Android 双方のガイドラインが年々収束している（Bottom Navigation追加等）
- クロスプラットフォーム開発（Flutter/React Native）の普及でUI統一が加速
- 人気アプリ（Instagram等）はほぼ共通構造、細部のみプラットフォーム差が残る

### ガイドライン準拠の判断基準

| 状況 | 判断 |
|------|------|
| 一般的なツール型アプリ | ガイドラインに従う（学習コスト低減） |
| 動画・エンタメ系没入型アプリ | 体験に合わせて逸脱可（コンテンツを隠さない設計） |
| 画面が大型化したスマートフォン | 上部ナビを下部に移動するなどガイドライン外の工夫が必要 |
| コンポーネントが体験を損なう場合 | ガイドラインより体験を優先する判断も有効 |

**基本方針**: ガイドラインは「理解した上で適用する」もの。目的・ユーザー・端末サイズを考慮した上で準拠可否を判断する。

## 業務UIの設計ポイント

### ウィンドウ幅への対応

- **テーブルの最小幅**を定義し、それ以下での表示を確認する
- 任意列（備考等）は**2段目に配置**すると必須列がウィンドウ縮小の影響を受けにくい
- 重要な操作ボタン・ステータスは**行頭**に配置（行末は縮小時に隠れる）
- サイドバーは**可変幅**に設計する（アイコンのみ/省略/フル表示の3段階等）

### 文字情報の工夫

| 工夫 | 説明 |
|------|------|
| **ボタンを動詞にする** | 「登録」→「登録する」。ひらがな混じりで印象が柔らかくなり、ボタンも大きくなる |
| **説明を惜しまない** | ツールチップ/バルーンヘルプで専門用語を補足。サポートコストを削減できる |
| **漢字をひらがな/カタカナに** | 「検索」→「さがす」のように柔らかく表現することも可 |

## デザインの説得力を高める実践知

### エンジニア連携で事前確認すべき項目

| 確認内容 | 理由 |
|---------|------|
| 実装フレームワーク（ネイティブ/Flutter/React Native） | フレームワークによってデザイン表現に制限が生まれる |
| 利用するグラフ・ダッシュボードライブラリ | カスタマイズ範囲がデザインを左右する |
| 負荷が高いUI表現の可否 | リアルタイム絞り込み等はサーバー負荷で実現不可の場合がある |
| 再現精度のチェックポイント | 揃え・余白・フォント・行間は口頭伝達だけでは再現しにくい |

### 初期リリース戦略

| 情報量が少ない段階 | 推奨対応 |
|------------------|---------|
| フリーワード検索 | 情報1,000件未満では不要。カテゴリ/タグで誘導する |
| 絞り込み機能 | 結果ゼロになるリスクがある。まず**並び替え**機能を優先する |
| 自動表示（新着/人気順） | 重複が起きる。初期は**手動キュレーション**でカバーする |
| 1画面あたりの件数 | 情報が少ない場合は1件あたりの情報量を増やして件数を減らす |

### 「○○っぽいデザイン」のエッセンス抽出

「○○っぽさ」は3パターンに分解できる:

| パターン | 内容 | 対処法 |
|---------|------|------|
| **UIデザイン** | 色・レイアウト・見た目の特徴 | 「ベージュと緑」→「ナチュラルな色使い」にエッセンスを抽象化する |
| **体験そのもの** | 操作感・速さ・使い続けたくなる仕組み | UIだけでなく導線設計・コンテンツ設計も含む |
| **利用者の傾向** | 対象ユーザーが使い慣れているサービス | メーラー/Excelユーザー向けなら、そのエッセンスを直接引用 |

→ 見た目をそのまま模倣せず、**エッセンスを抽出して目的に合わせて適用**する。

### アナリティクス活用の流れ

```
仮説を立てる
  ↓
計測項目を定義（カテゴリ/アクション/ラベル）
  ↓
データを収集（1週間 or 1ヶ月）
  ↓
仮説が正しかったか検証
  ↓
新しい仮説を立てて改善サイクルへ
```

見るべき基本データ: デバイス比率・画面解像度・集客元・よく見られているページ

### 説得力のあるデザインにする

- **依頼主の好みを読む**: こだわるポイント（色・文言・情報量等）を把握してデザインに反映する
- **言語化を習慣にする**: 新規デザインなら影響した要素を、リニューアルなら変更点と理由を言葉で添える
- **事例を参照する**: A/Bテスト結果・ユーザーテスト結果など外部事例を添付すると説得力が増す
- **示差性を活用する**: 「すべてを目立たせようとすると何も目立たない」。ベースデザインの中で何が際立つかを設計する

## 長押しの設計活用

### 長押しが有効な用途

| 用途 | 説明 | 例 |
|------|------|-----|
| **ショートカットメニュー** | 通常タップとは別の隠しメニューを表示 | Twitter投稿ボタン長押し→下書き/GIF/スペース |
| **頻度低・重要な操作の割り当て** | 誤操作防止が必要な機能に割り当て | Bluetoothペアリング・電源再起動 |
| **選択モード** | 長押しで複数選択モードへ移行 | Material Designのガイドライン記載例 |

### 長押しの注意点

- **視覚フィードバックが必要**: 押し続けていることをプログレス表示等で伝える
- **万人向けではない**: タップし続ける高齢者には誤操作リスクがある
- ガイドライン（HIG/Material Design）では「追加コントロール表示」として推奨
- タップと長押しの2操作を1ボタンに割り当てると混乱しやすい

## メンタルモデルを考慮した設計

### メンタルモデルとは

ユーザーが経験から築いた「当然こうなるだろう」という無意識の期待値。
UIがこれを裏切ると**苛立ち・離脱**につながる。

### 判断基準

| 状況 | 判断 |
|------|------|
| メンタルモデルに沿った挙動 | 学習コストが低く安心感を与える。基本はこちらを選ぶ |
| メンタルモデルを覆す挙動 | 上回る体験を提供できると確信があれば有効。ただし徹底した検証が必要 |
| メンタルモデルをずらした驚き演出 | 「THANK YOU!!」等の微小な違いでサービスの味付けになる |

> **実装原則**: 技術的な都合（配線・パフォーマンス等）よりもユーザーのメンタルモデルを優先した設計を選ぶ。メンタルモデルは慣れによってアップデートされることもあるため、固執しすぎないことも重要。
