# 骨格段階：インターフェース・ナビゲーション・情報デザイン

> 骨格段階は「機能がどのような形をとるか」を定義する。構造段階が大規模なアーキテクチャとインタラクションを扱うのに対し、骨格段階は個別コンポーネントとその関係という小規模な問題を扱う。

---

## 骨格段階の3要素

| 要素 | 定義 | 機能性/情報側 |
|------|------|-------------|
| インターフェースデザイン | ボタン・フィールド等のコンポーネントをユーザーが扱える形に配置する | 機能性側 |
| ナビゲーションデザイン | 情報空間の構造をユーザーが見て移動できるレンズを提供する | 情報側 |
| 情報デザイン | アイデア・情報をユーザーに効果的に伝える表現を決定する | 両方 |

**重要な原則**: 3つの境界は曖昧になりやすい。しかし「それぞれの問題領域が何か」を区別して認識することで、本当に解決できたかどうかを正確に評価できる。ナビゲーションが優れていても、情報デザインが悪ければカバーできない。

---

## 慣例とメタファー

### 慣例の役割

人間は習慣と反射で行動する。慣例があるからこそ反射的な行動を異なる状況でも適用できる。

**テレフォンパッドの事例**
- 「3×4」配列への人々の馴染みは、電子レンジ・リモコン・ATMにまで波及
- 慣例と違うだけで電話1回あたり3秒のロスが発生 → これは時間の問題でなくフラストレーション体験

```
If 慣例から外れる選択をする
Then 明確な利点がある場合のみ行う（理由が必要）
```

### 一貫性の2層構造

| レベル | 内容 |
|--------|------|
| 製品内部 | 同じ概念モデルを持つフィーチャーには同じ慣例を使う |
| 製品外部 | ユーザーが既に馴染んでいる他製品との一貫性を持たせる |

「スタート」「終了」「戻る」「保存」といった汎用概念は、コンテクストが違っても一貫した扱いが求められる。

### メタファーの限界

```
If メタファーを使って機能を表現したい
Then フィーチャーの性質を明確にするか確認する（曖昧にしていないか）
```

- **有効なメタファー**: 電話帳アイコンで電話番号ディレクトリを表す（直接対応）
- **問題のあるメタファー**: コーヒーショップの絵でチャットエリアを表す（推測が必要）

異なる文化背景のユーザーには同じ絵でも全く異なる解釈が生まれる。ユーザーが推測しなければならないものは排除した方がシンプルになる。

## インターフェースデザイン

> インターフェースデザインの核心: ユーザーがとるであろう行動を認識し、その要素に最もアクセスしやすく、使いやすく作ること。

### プログラマー思考との違い

| 思考様式 | アプローチ | 結果 |
|---------|-----------|------|
| プログラマー | すべてのエッジケースを等しく扱う | 極端なケースと多数のニーズを同じように扱う |
| UXデザイナー | ユーザーがとるであろう行動を重視 | 頻度の高い行動に最もアクセスしやすくする |

少数の極端なケースと大多数のユーザーニーズを同じ扱いにしたインターフェースは、どちらも満足させられない。

### デフォルト選択の重要性

良いデフォルト選択はユーザーの認知負荷を下げる最も簡単な手法の一つ。

```
If ユーザーの多くが詳細表示を好むとわかっている
Then 「詳細表示」チェックボックスをデフォルトでチェック済みにする
```

### 標準コントロールと選択基準

| コントロール | 機能 | 適した状況 |
|------------|------|-----------|
| チェックボックス | 互いに独立したオプションの複数選択 | 選択肢が少ない（6つ以下程度） |
| ラジオボタン | 相互排他的なオプションから1つ選択 | 選択肢が少なく全て表示したい |
| ドロップダウンリスト | ラジオボタンと同機能、コンパクト | 選択肢が多い、スペース節約優先 |
| リストボックス | チェックボックスと同機能、コンパクト | 大量の選択肢を複数選択 |
| テキストフィールド | テキスト入力 | 自由入力が必要 |
| アクションボタン | 他の要素の情報を受け取りアクション実行 | フォーム送信・確定処理 |

**トレードオフの例**:
- ドロップダウン → スペース節約だが選択肢が隠れる
- ラジオボタン → 全選択肢が見えるがスペースを使う
- テキスト入力 → DBへの負荷が減るがユーザー負担が増える

```
If 技術的制約でインターフェースオプションが限られる
Then 標準コントロールの範囲内でユーザーエクスペリエンスを最適化する
（標準コントロールの制約は、ユーザーが知識を転用できる利点でもある）
```

## ナビゲーションデザイン

### ナビゲーションデザインの3つのゴール

すべてのナビゲーションデザインは同時に以下を達成しなければならない:

1. **移動手段の提供**: サイトのある地点から別の地点へ移動できる手段
2. **関係性の伝達**: ナビゲーション要素同士の関係（重要度・違い）を伝える
3. **現在地との関連性**: ナビゲーション内容と現在見ているページの関係を伝える

> 物理空間では方向感覚に頼れるが、情報空間ではその脳のメカニズムは機能しない。だからこそ「今どこにいるか」「どこへ行けるか」を明確にすることが重要。

**前提として守るべき原則**: 「ユーザーはページを移動するとそれまでのことを覚えていない」と仮定する（検索エンジンによってどのページも入り口になりうるため）。

### ナビゲーションシステムの種類

| 種類 | 目的 | 特徴 |
|------|------|------|
| グローバルナビゲーション | サイト全体へのアクセス | 主要アクセスポイントをまとめる |
| ローカルナビゲーション | 「すぐ近く」のコンテンツへのアクセス | 親・兄弟・子ページへのリンク |
| サブナビゲーション | 関連コンテンツへの近道 | ファセット分類の利点を活かす |
| コンテクストナビゲーション | ページコンテンツ内に埋め込まれたリンク | 読み中の関連情報へのアクセス |
| 優先ナビゲーション | 定期的に必要ではないが便宜上提供 | 連絡先・フィードバック・方針文 |

```
If コンテクストナビゲーションを使う
Then ユーザーのタスク・ゴールを明確にサポートしているか確認する
（ハイパーリンクだらけで目的のリンクが選べない状態は「乱雑」と見なされる）
```

**グローバルナビゲーション vs 不変のナビゲーション**:
- グローバル: サイトの端から端まで移動するための主要アクセスポイント
- 不変（Persistent）: すべてのページに存在するナビゲーション要素（グローバルと異なる概念）

### リモートナビゲーションツール

他のナビゲーションシステムに不満があるユーザー、または「わかろうとさえしないほうがいい」と感じたユーザーが使う独立したツール。

| ツール | 説明 | 適した状況 |
|--------|------|-----------|
| サイトマップ | サイト全体のアーキテクチャを1ページで表示 | 上位2階層まで。それ以上必要なら上層に問題あり |
| インデックス | トピックをアルファベット順に並べリンク | 大量コンテンツの広範テーマサイト |

## 情報デザイン

> 情報デザイン: 情報をどのように見せれば、人々がより使いやすく、より理解しやすくなるかを判断すること。他のデザイン要素をくっつける「接着剤」として機能する。

### 情報デザインの2つの側面

**ビジュアル面**:
- データの表現形式選択（円グラフ vs 棒グラフ）
- アイコンの選択（双眼鏡 vs 虫眼鏡で「検索」を表す）

**情報のグループ化・配置面**:
ユーザーの考え方を反映し、タスク・ゴールをサポートするように情報要素を整理する。

```
悪い例（ランダム配置）:
州 / 役職 / 電話番号 / 番地 / 名前 / 郵便番号 / 組織名 / 市町村 / メールアドレス

良い例（ユーザーのメンタルモデルに沿ったグループ化）:
【個人情報】名前 / 役職 / 組織名
【住所】番地 / 市町村 / 州 / 郵便番号
【その他連絡先】電話番号 / メールアドレス
```

これは「ミクロレベルの情報アーキテクチャ」であり、情報デザインはその構造をページ上で伝える。

### インターフェースデザインとの交差

インターフェースはユーザーから情報を集めるだけでなく、ユーザーに情報を伝える必要がある。エラーメッセージ・操作指示情報はインターフェースデザインの問題であり、同時に情報デザインの問題でもある。

```
If システムがユーザーに情報提供しなければならない場面
   （エラー・初回利用案内・操作指示）
Then これは情報デザインの問題として扱う
```

### 経路探索（Wayfinding）

情報デザインとナビゲーションデザインが連携して行う、「自分がどこにいるか・どこへ行けばよいか」をユーザーに伝える機能。物理空間（公園・空港・駐車場）のデザイン原則がウェブにも適用できる。

**経路探索ナビゲーション要素**: ナビゲーションシステム（移動手段の提供＋視点の伝達）

**経路探索情報デザイン要素**: ナビゲーション機能を持たないページ要素
- 色分け（セクションの視覚的区別）
- アイコン
- ラベリングシステム
- タイポグラフィー

優れた経路探索により、ユーザーは以下を即座に把握できる:
1. 自分たちがどこにいるか
2. どこへ行けるか
3. どの選択肢を選べば目標に近づけるか

## ワイヤーフレーム

> ワイヤーフレーム: ページを構成するすべての要素と、それらをどのように組み合わせるかを必要最低限示した文書。

ページレイアウトは、インターフェースデザイン・ナビゲーションデザイン・情報デザインが一体となって、まとまりのある骨格を形成する場。これら3要素のバランスを一度にとるのは大変なため、ワイヤーフレームで詳細に記述する。

### ワイヤーフレームの内容

- **インターフェースデザイン**: インターフェース要素の配置と選択
- **ナビゲーションデザイン**: 核となるナビゲーションシステムの特定と定義
- **情報デザイン**: 情報要素の配置と優先順位づけ

通常は注釈が付けられ、アーキテクチャダイアグラム・コンテンツ要求・機能仕様書を参照できる。

### ワイヤーフレームの適切な粒度

```
If チームが同じ空間に集まった小規模チーム
Then 鉛筆スケッチ + 付箋でも十分

If 地球の裏側にいる大規模チーム
Then より詳細な文書化が求められる
```

文書化は目的ではなく、目的を達成するための手段。「ニーズに合った適切なレベルの文書」を作成する。

### ワイヤーフレームの活用者

| 活用者 | 用途 |
|--------|------|
| 戦略・要件・構造の責任者 | 最終製品が期待に応えられるか確認 |
| 実装責任者 | 「サイトがどのように機能するか」の質問に答える参照 |
| ビジュアルデザイナー | ビジュアルデザイン作業のリファレンス |

**UXデザイナーとビジュアルデザイナーの協業**: ワイヤーフレームは両者の協力なしには成功しない。互いの視点から問題を見ることで、プロセスの早い段階で未解決の問題を発見できる。

### ワイヤーフレームの位置づけ

- 構造段階のアーキテクチャダイアグラム = 遠大なビジョン
- 骨格段階のワイヤーフレーム = そのビジョンをどのように実現するかを示す詳細な文書
- 表層段階への道筋を示すと同時に、概念構造の上に構築される骨格を定義する

すべての画面にワイヤーフレームが必要なわけではない。アーキテクチャ上のコンテンツクラスと同様に、機能性とナビゲーションの多様性に基づいて比較的少数の標準的な画面パターンが明らかになる。
