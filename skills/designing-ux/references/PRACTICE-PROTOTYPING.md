# プロトタイピング実践ガイド

## 仕事としての始めやすさ

| こっそり練習 | 一部業務でトライアル | クライアント巻き込み |
|:---:|:---:|:---:|
| ★★ | ★★ | ★★★（条件付き） |

ペーパープロトタイピングは一人でも始められる。チームで実施すると効果が高い。クライアント承認は規模・コストによる。

---

## プロトタイピングとは

製品・サービスの完成版ができる前に**試作品（プロトタイプ）を作り、評価・改善を繰り返す**こと。

**従来の開発フロー**
```
キックオフ → ─────────────────────────── → 公開
                                             ↑ここで初めて失敗が分かる
```

**プロトタイピングを取り入れたフロー**
```
キックオフ → [作成→評価→改善] → [作成→評価→改善] → 公開
               ↑小さく何度も軌道修正
```

### なぜプロトタイピングをするのか

- 実際に作ってみると分かることが非常に多い
- 情報設計の基本骨格に関わる修正は後になるほど大変
- HTMLコーディング後にワイヤーフレームレベルの修正が入るコストを考えれば、早期に問題発見するほうが費用対効果が高い

---

## プロトタイピングの種類と選択基準

### フェーズ別プロトタイピング手法

| フェーズ | 内容 | 主な手法 |
|---------|------|---------|
| **①コンセプト・企画** | まだ具体的なモノが存在しない | コンセプトテスト・ストーリーボード・MVP |
| **②情報設計・画面フロー** | 骨格を作り上げる段階 | **ペーパープロトタイプ・オズの魔法使い** |
| **③ビジュアル・インタラクション** | 装飾・アニメーション設計 | ユーザビリティテスト・プロトタイピングツール |

> この章では**②情報設計フェーズ**のプロトタイピングを主に扱う。Webの現場で最も費用対効果が高い。

### 忠実度（Fidelity）の選択

| 忠実度 | 作り込み | 向くシーン |
|--------|---------|-----------|
| **低（ペーパー・スケッチ）** | 最小限の要素のみ | 設計初期・アイデア探索 |
| **中（ワイヤーフレーム）** | 主要要素と構造 | 設計中盤・チーム共有 |
| **高（ツールプロト）** | 動き・ビジュアル | 設計後半・ユーザビリティテスト |

## ワイヤーフレームを評価する：オズの魔法使い

設計途中のワイヤーフレーム（PowerPoint・Excelなど）でもユーザビリティ評価を行える手法。

### オズの魔法使いのしくみ

```
[被験者] ──指で操作→ [紙のワイヤーフレーム]
                              ↑
                    [ブラウザー役の人間が
                     次のページに差し替える]
```

被験者はすぐに「紙の画面」に慣れ、実際の利用状況さながらに一喜一憂してタスクを実行してくれる。

### オズの魔法使い 対応表

| 通常のブラウジング | オズの魔法使いでの代替 |
|------------------|----------------------|
| マウスでクリック | 被験者の指でタップ |
| 画面遷移 | ブラウザー役がワイヤーフレームを差し替え |
| モジュール切り替え | 切り替えパーツを乗せ替え |
| テキスト入力 | 紙とペンで書いてもらう |

### 実施メリット

- 実装前の早い段階で問題を発見できる
- ビジュアルデザイン前のシンプルな状態で、見た目に惑わされない問題に気づける
- 設計者自身が紙のプロトタイプを作る過程で、矛盾・過不足に気づく副次効果がある

## ペーパープロトタイピング

紙で作ったプロトタイプを使い評価と改善を行う手法。

### なぜ紙で作るのか

- ソフトウェア・プログラミング知識が不要
- 誰でも手早く作成・修正できる
- 付箋を貼り替えるだけで変更できる
- 捨てることへのストレスが低い → 反復設計が活性化する
- 複数名でアイデアを出し合い、その場で改善案を即座に反映できる

### ペーパープロトタイピングの流れ

```
1. 範囲を定める
   ↓
2. プロトタイプを作成する
   ↓
3. 試用する（メンバーがユーザーになりきる）
   ↓
4. ブラッシュアップする（その場で付箋で改善）
   ↓
（3→4を繰り返す）
```

### Step 1：プロトタイピングの範囲を定める

**設計初期の場合**
- 最も重要なコンバージョン経路のみを作成
- 記載する情報は導線を支える最低限のものに絞る
- 余計なものを描かないことで「コンバージョンの流れ」が浮き彫りになる

**設計中盤の場合**
- 再利用・推奨意向を高めるフックを含める
- より詳細な画面要素を記載する

### Step 2：プロトタイプを作成する

**作成方法の選択**

| 方法 | 特長 |
|------|------|
| **手書き（推奨）** | 短時間・捨てやすい・反復設計が活発になる |
| パソコン（デザインツール） | 精度が高い・繰り返し使いやすい |

> **はじめは手書きを推奨**: ストレスなく作り直せる感覚が重要。

### Step 3：試用する

チームで集まり、各メンバーが**ユーザーになりきって「見る」のではなく「試用」する**。

**具体的な手順**

1. コンバージョン導線上の画面のプロトタイプを作成
2. ユーザーが操作する順に並べる
3. ターゲットユーザーのペルソナ・シナリオを読んでおさらいする
4. ターゲットユーザーになりきり、サービス利用のきっかけとなるシーンを想像する
5. ランディングページからコンバージョンページまでプロトタイプを試用する（指でタップ、ペンで入力）
6. 気づいた点をメモ（悪い点だけでなく良い点・ジャストアイデアも）
7. 改善する → 再試用

### Step 4：ブラッシュアップする

その場でプロトタイプに改善案を反映 → 再度試用。

- 付箋を貼って変更
- 折って隠す（不要要素の削除）
- 新しい紙に書き直す

**ポイント**: 論点をホワイトボードに書いておき、議論が脱線しないようにする。

---

## ペーパープロトタイピング 必要ツール

| ツール | 用途 |
|--------|------|
| **付箋（複数サイズ）** | オブジェクトを描く・貼り替えで差し替え可能 |
| **コピー用紙** | 台紙・大きめのオブジェクト用（スマホ：B5、PC：B4程度） |
| **貼って剥がせるのり** | コピー用紙のオブジェクトを台紙に貼る |
| **ペン（サインペン）** | はっきり書ける太めのもの |
| **ハサミ** | 紙・付箋を適切なサイズに切る |
| **厚手の紙** | デバイス・ブラウザの枠を描く（可視領域を表現） |

---

## プロトタイピングツールの活用

昨今のプロトタイピングツールは急速に進化しており、紙より早く反復できる場合もある。

**ツール活用の注意点**
- 表現力が高いため、必要以上に見た目を作り込みやすい
- 評価・改善のためのプロトタイピングなら、**過剰な作り込みは意識的に抑制する**
- モックアップ（提案用の見た目重視のもの）とは目的が違う

---

## 実施時の注意点：「作り込みすぎ」を避ける

プロトタイピングに慣れると陥りやすい「作り込みすぎ」問題。

| 弊害 | 内容 |
|------|------|
| **リソースのムダ** | 見た目を作り込む時間・稼働が無駄になる |
| **評価・改善速度の低下** | 修正のたびに手間がかかり、サイクルが遅くなる |
| **余計な話題の混入** | きれいなプロトタイプは見た目への指摘を誘発し、情報設計の議論が減る |

**対策**
- 必要最低限の粒度・機能だけを作る
- 「今話すべき論点」をホワイトボードに明示する
- 有用なアイデアが浮かんでも論点外のものはメモだけして後回し

## 機会の作り方：3段階アプローチ

### こっそり練習
ワイヤーフレームを書く設計業務をやっているなら、その延長でペーパープロトタイピングを試す。一人でやると「紙になっただけ」に感じるため、複数メンバーで試用することが重要。

### 一部業務でトライアル
プロトタイピングは「おまかせしたいライン」（設計者の裁量範囲）にある。クライアント承認なく、設計プロセスの一環として取り込める。

- 実施しても追加外部コストはかからない（被験者を呼ばない場合）
- 後工程の手戻りが減るため、トータルで帳尻が合うことが多い

### クライアントを巻き込む
以下のケースでは積極的に提案：
- クライアントが設計へのこだわりや質問が多い場合
- 後から改修が大変なシステム・高トラフィックサイト
- 全く新しい分野で設計チームにノウハウがない場合

## 事例紹介

### Light Case（0.5週間・3人日）

**体制**: UXデザイナー × 1、ディレクター × 1
**目的**: 印面デザイン入力UIの設計妥当性を短期間で検証
**工夫**:
- プロトタイピングによるスケジュール影響：0営業日
- 評価設計をUXデザイナーに委任し、クライアントレビュー省略
- UXアーキテクト + ディレクターのペアで設計・評価・改善を短時間で実施

### Heavy Case（2週間・15人日）

**体制**: UXアーキテクト × 1、ディレクター × 2、プロデューサー × 1
**目的**: ゴルフ場予約サイトの「地図から探す」UI改善
**プロセス**:
1. 評価範囲を「地図検索トップからアクセス方法確認まで」に限定
2. さまざまな操作モジュールをペーパーで作成（乗せ替え可能な設計）
3. 被験者3名で評価
4. 問題点・改善案レポート作成 → クライアントとブラッシュアップ

## チェックリスト：プロトタイピング実施前確認

**計画**
- [ ] プロトタイピングの目的が明確か
- [ ] 評価範囲（導線）が決定しているか
- [ ] ペルソナ・シナリオが準備されているか

**作成**
- [ ] 必要最低限の粒度に抑えているか（作り込みすぎていないか）
- [ ] 試用に必要な画面は揃っているか
- [ ] 操作に必要な付箋・差し替えパーツが準備されているか

**試用**
- [ ] 参加者がペルソナ・シナリオを理解しているか
- [ ] 「見る」ではなく「試用」の意識で進めているか
- [ ] 気づきを記録する準備があるか

**改善**
- [ ] 論点がホワイトボード等に見える形で整理されているか
- [ ] その場で改善案を反映できる資材が揃っているか
