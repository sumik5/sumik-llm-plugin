# ユーザビリティ評価 実践ガイド

## 仕事としての始めやすさ

| こっそり練習 | 一部業務でトライアル | クライアント巻き込み |
|:---:|:---:|:---:|
| ★★★ | ★★★ | ★★★ |

アウトプットに近い業務のため最も始めやすい。認知的ウォークスルーであれば被験者不要で一人からでも練習できる。

---

## ユーザビリティ評価とは

WebサイトやアプリのユーザビリティをユーザーやUX専門家が評価すること。

### ユーザビリティの定義

ISO9241-11: 「ある製品が、特定のユーザーによって、特定の利用状況において、指定された目標を達成するために用いられる際の、有効さ・効率・ユーザーの満足度の度合い」

| 評価軸 | 問い | 問題のレベル感 |
|--------|------|---------------|
| **有効さ** | タスクを完遂できたか | 最重要（離脱・成約に直結） |
| **効率** | 効率良く完遂できたか | 重要（体験品質） |
| **満足度** | 不満なく完遂できたか | 中程度（推奨意向・再利用に影響） |

### 評価手法の比較

| 手法 | コスト | 被験者 | 有効さ | 効率 | 満足度 |
|------|:---:|:---:|:---:|:---:|:---:|
| **被験者評価（ユーザビリティテスト）** | 高 | 必要 | ◎ | ○ | ○ |
| **専門家評価（認知的ウォークスルー）** | 中 | 不要 | ○ | ○ | × |
| **ヒューリスティック評価** | 低 | 不要 | △ | △ | × |

---

## 5ステップ実施手順

```
① 計画 → ② 評価設計 → ③ 実査準備 → ④ 実査 → ⑤ 分析
```

---

## ① 計画

### ①-1 目的・目標の明確化

**悪い例（曖昧な目的）**
> 「今のサイトの使いやすさを知りたい」

**良い例（明確な目的）**
> 「Webサイトのターゲットをサービス理解の少ないユーザーにまで拡げたい。その人達にとっての現状の使い勝手に関する課題を4週間後にスタートするサイト改修プロジェクトのインプットとしたい」

**目標の書き方テンプレート**
```
評価対象: [誰が] [どんな状況で] [何をするシーン]
評価目標: [何を明らかにするか]
活用方法: [結果をどう使うか]
```

### ①-2 評価用簡易ペルソナの作成

ユーザビリティは「使う人・状況・タスク」の3つで変わる。ペルソナはこのうち「使う人」を見える化するもの。

**ペルソナ記述テンプレート**

| 項目 | 内容 |
|------|------|
| 氏名・属性 | 山田 花子（45歳/女性/パート） |
| 一言 | 「難しいと思ったら諦めちゃう」 |
| インターネット利用 | 検索・閲覧はOK。入力が多い画面は苦手 |
| 課題（困りごと） | ○○したいが△△でできない |
| 環境 | スマートフォン・自宅リビング |
| 利用背景 | ○○をきっかけに利用を検討 |

**ペルソナ項目の選択基準**

| 項目 | 内容 |
|------|------|
| ユーザーの特性 | 知識・経験・スキル・好み・習慣 |
| ユーザーの仕事 | 作業手順・役割・利用目標・頻度 |
| 環境要因 | デバイス・物理環境・周囲の人との関係 |

> **ポイント**: 設計者にヒアリングしてペルソナを作成後、管理者に確認する手順が効率的。専任の調査チームがなければ社内の知識ベースで簡易作成でよい。

### ①-3 日程・場所・体制・被験者の決定

**スケジュール目安（標準的な進行）**

| フェーズ | 期間 |
|---------|------|
| 計画・リクルーティング開始 | 4〜6週前 |
| 評価設計（タスク・観察ポイント・進行） | 3〜4週前 |
| 実査準備・プレテスト | 1〜2週前 |
| 実査 | 当日 |
| 分析・レポーティング | 1〜2週後 |

> **現実的なスケジュール**: プロジェクト全体への影響は1〜3週間が多い。承認不要・クライアント参加で0日に近づけることも可能。

## ② 評価設計

### ②-1 評価範囲の決定

評価目標から「どの導線を評価するか」を決める。

```
スタート → [ページA] → [ページB] → [ページC] → ゴール（コンバージョン）
```

- 一通りではない実際の導線から、最も重要な経路を選択
- 範囲外に迷い込んだ場合はモデレーターが誘導

### ②-2 タスク・シナリオの設計

**シナリオ（利用背景）**:「なぜタスクをするのか」の背景設定。これによりユーザーの行動・我慢の度合いが変わる。

**タスク（目標）**: 完了状態が明確な課題。「完遂できたか」がはっきり判断できるよう書く。

**タスク設計の例**

```
ペルソナ: スマートフォンではメールしかしない45歳主婦
シナリオ: 楽しみにしていたドラマを見逃した。DMに新サービスの案内がある。
タスク①: DMを見て、このサービスがどんなサービスか分かったら教えてください。
タスク②: パソコンから[サービス名]のサイトで[ドラマ名]の第○話を見られるようにしてください。
```

**タスク設計の注意事項**

- [ ] 完了状態が明確か（「見つけてください」より「見られるようにしてください」）
- [ ] 使い方のヒントを与えていないか（製品名・手順を先に教えない）
- [ ] ユーザーが実際に使う言葉で書いているか
- [ ] 複数ステップは分けて設計しているか

**操作のシナリオ**（評価後の分析用）

ユーザーがタスクを完遂するために必要な操作を具体的に列挙したもの。被験者の行動とのギャップから問題を発見する。

| 画面 | 期待される操作 |
|------|---------------|
| トップページ | 「ドラマ」ナビをクリック |
| カテゴリ一覧 | ドラマタイトルで絞り込む |
| 詳細ページ | 「視聴する」ボタンをクリック |

### ②-3 観察ポイントの設定

評価中に特に注目すべき箇所を事前に洗い出す。

- 仮説として問題がありそうなUI要素
- 操作のシナリオとのギャップが起きやすいステップ
- 目に見えない理解の問題（用語・フロー理解等）

### ②-4 進行・タイムテーブル設計

| パート | 目安時間 |
|--------|---------|
| イントロ（説明・許可取得） | 5分 |
| 事前インタビュー | 5分 |
| 思考発話法の練習 | 5分 |
| タスク実行 | 30〜45分 |
| 事後インタビュー | 10〜15分 |
| 謝礼・見送り | 5分 |
| 次セッション準備（被験者入れ替え） | 30分 |

## ③ 実査準備

### 必要な資材チェックリスト

**機材**
- [ ] パソコン・スマートフォン（評価対象デバイス）
- [ ] 画面録画ソフト（または画面キャプチャ手段）
- [ ] ボイスレコーダー（またはスマートフォン）
- [ ] カメラ（被験者の表情・行動記録用）

**ドキュメント**
- [ ] テスト設計書（シナリオ・タスク・観察ポイント）
- [ ] タイムテーブル
- [ ] モデレーター用スクリプト
- [ ] 被験者用タスクシート（印刷）
- [ ] 入力用ダミーデータシート
- [ ] 機密保持同意書（NDA）・個人情報同意書

**記録用**
- [ ] 模造紙・付箋（発見問題の整理用）

### プレテストの実施

本番前に**評価内容を知らない社内の人**に被験者役をやってもらう。

確認ポイント：
- タイムテーブル通りに進むか
- シナリオ・タスクが理解できるか
- 資材に不足はないか
- 問題が発見できるか

## ④ 実査（テスト当日）

### ④-1 設営・配置

```
[パーティション]

被験者 ── ノートPC      |  記録係（行動・発話）
  ↑                    |  見学者
モデレーター             |
（斜め後方・表情見える位置）
```

被験者がリラックスできる配置。見学者は被験者の視界に入らない位置へ。

### ④-2 被験者への説明事項

1. **調査の目的**: 製品の問題を発見すること（被験者のスキル評価ではない）
2. **やってもらうこと**: Webサイトの利用・操作
3. **取り組み姿勢**: 普段通りに使ってほしい。うまくやろうと頑張らなくてよい
4. **思考発話法の依頼**: 考えていること・感じたことを声に出しながら使ってほしい

> **ラポール形成**: 事前インタビューで被験者に安心してもらう関係を作る。「テスト」「被験者」という言葉は避け「Webサイトを使ってみる」「ユーザー」と言い換える。

### ④-3 思考発話法（Think Aloud）

「今どこを見ていますか？」「今何を考えていますか？」と適宜促す。

**練習例**: ボイスレコーダーを渡して「録音してください」→「えーと Rec とか書いてあるのかな？赤いボタンがある、これかな？」のように話し始めるまで練習。

### ④-4 タスク実行中の観察

**記録は2種類**

| 種類 | 方法 | 信頼性 |
|------|------|--------|
| **客観的記録** | 画面録画・音声・行動動画 | 事実（解釈混入なし） |
| **主観的記録** | モデレーター・記録係のメモ | 背景（解釈の参考） |

**分析優先度**: 行動（事実）＞ 発話（背景・解釈）

**質問のタイミング**
- 原則：タスク後の事後インタビューでまとめて聞く
- 例外：「今聞かないと忘れる」無意識の行動が出た直後

**クローズドクエッション禁止** → **オープンクエッション推奨**
- ❌「○○だからですか？」
- ✅「なぜそうされましたか？」

### ④-5 事後インタビュー

タスク実行後にまとめて「なぜ」を聞く。

- 該当ページを見せながら聞く（記憶想起しやすい）
- 録画がある場合は再生しながら聞く
- 複数タスクの場合は1タスクごとに実施
- 時間目安：タスク実行時間の1/4（60分のタスク → 15分）

### ④-6 次セッション準備（被験者入れ替え）

- ブラウザの履歴・キャッシュ・Cookieを完全削除
- 入力フォームの初期化
- 資材・ドキュメントのリセット

## ⑤ 分析

### ⑤-1 問題の重さ分類

| 重さ | 定義 | 優先度 |
|------|------|:---:|
| **有効さの問題** | タスクを完遂できない | 最高 |
| **効率の問題** | 完遂できるが効率が悪い | 高 |
| **満足度の問題** | 効率も問題ないが不快・不安 | 中 |

### ⑤-2 問題の要因分析

ドナルド・ノーマンの「良いデザインの4原則」で要因を分析：

| 原則 | 要因 | 例 |
|------|------|-----|
| **可視性** | 見ただけで何をすべきか分からない | ボタンがボタンに見えない |
| **良い概念モデル** | 仕組みのイメージと実際がずれている | メール認証の意味が分からない |
| **良い対応付け** | 操作と結果の対応が分からない | どれをクリックすれば何が起きるか不明 |
| **フィードバック** | 操作後に結果が分からない | 送信が完了したか確認できない |

### ⑤-3 解決優先度の決定

| | 解決が易しい | 解決が中程度 | 解決が難しい |
|---|:---:|:---:|:---:|
| **有効さの問題** | 最高 | 高 | 高 |
| **効率の問題** | 高 | 中 | 中 |
| **満足度の問題** | 中 | 中 | 低 |

## 専門家評価（被験者なしの評価）

### 認知的ウォークスルー

設計者がペルソナになりきってタスクを実行する方法。

**手順**
1. ペルソナ・シナリオ・タスクを設定
2. ペルソナになりきって実際に操作する
3. 操作のシナリオ通りに使えないところをメモ
4. 使えるが迷うところをメモ
5. 使えるが不満を感じるところをメモ

**向いている場合**: 設計者とターゲットユーザーのスキル・知識・文化的背景が近い場合

**被験者評価が必要な場合**: デバイスに不慣れな高齢者・専門知識を持つ薬剤師・子ども・外国人など、設計者がなりきることが難しい場合

### ヒューリスティック評価

UIガイドラインに基づいてWebサイトを網羅的にチェックする手法。特定のコンバージョンに焦点を当てる評価よりも、全体的な問題発見に適している。

## 定量評価手法

### NEM（Novice Expert ratio Method）

初心者と熟練者がタスクを行ったときの**所要時間を比較**し、問題箇所を特定する定量的手法。

**NE比の計算式**
```
NE比 = 初心者の所要時間 ÷ 熟練者の所要時間
```

**分析例**
- NE比5.0 → 初心者は熟練者の5倍かかっている（問題あり）
- しきい値の例：NE比4.5以上を「要改善」と設定

**注意点**
- 思考発話法と併用できない（時間計測に影響する）
- 極端に早い・遅い被験者は除外した平均値も算出する

### タスク完遂率

```
タスク完遂率 = 完遂した被験者数 ÷ 実施した被験者数 × 100%
```

完遂率0%は重大な問題のサイン。問題の深刻度を即座に伝えるのに有効。

### カードソート（情報設計の評価）

コンテンツ名のカードをカテゴリ別に分類してもらい、情報設計の妥当性を評価する手法。

| 種類 | 方法 | 用途 |
|------|------|------|
| **クローズドカードソート** | カテゴリ名を用意して分類してもらう | 設計した分類・ラベルの評価 |
| **オープンカードソート** | カテゴリ名もユーザーが命名 | ユーザーの思考モデル調査 |

**実施手順**
1. コンテンツ名カードをカテゴリに分類
2. 思考発話法を必ず実施（分類の「なぜ」が重要）
3. 一通り完了後に事後インタビュー

## 被験者の人数について

**推奨人数**: 1回につき3名が実践的

- 1名：その問題が一般的かどうか判断しにくい
- 2名：逆の結果が出たとき判断に迷う
- 3名：多くの人に出そうな問題とそうでない問題を判別しやすい

**回数のほうが重要**: 5名×1回よりも3名×複数回の方が設計精度が上がる（評価→改善→評価のサイクルを回せるため）。

## 事例紹介

### Light Case（1週間・4人日）

- 体制：UXデザイナー×3名 + ディレクター + プロデューサー
- 計画・評価設計はミーティングのみで短時間作成
- プレテストはUXデザイナー自身で実施
- 被験者：社内から3名・同日連続
- レポート：メモ書き程度

### Heavy Case（3週間・30人日）

- 体制：UXデザイナー×4名 + ディレクター + プロデューサー
- 既存調査データからペルソナ/シナリオを作成（クライアントレビュー含む）
- 4シナリオ × 4タスク
- 被験者：調査会社モニター5名
- NEM（定量評価）を含む50ページのレポート
