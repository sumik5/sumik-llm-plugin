# UIデザイン実践TIPS: 機能表現・UIコンポーネント

---

## ライトフィードバック（いいね等）の設計

### 設計の目的

ライトフィードバックの本質は、「考えずにすぐ押せる」こと。コメントより低コストなフィードバック経路を設けることで、コンテンツ作者への反応量を増やし、質の高いコメントとの棲み分けを実現する。

### ボタン設計の原則

- コンテンツの近くに配置し、小さすぎ・隠れすぎを避ける
- 「いいね！」という呼称を変える前に、まず標準的な実装でユーザー反応を確認する
- 個性を出すのは反応を見てからでも遅くない

### 単一軸 vs 多軸フィードバックの判断

| 判断基準 | 単一軸（いいね1種類） | 多軸（複数の絵文字等） |
|---------|---------------------|----------------------|
| コンテンツの性質 | 写真投稿など感情が単純 | 「悲しい出来事」等コンテキストが多様 |
| 目的 | 数・存在を示すだけで十分 | 賑わいの演出・感情の多様性 |
| 実装コスト | 低 | 高（選択UIの設計が必要） |
| 推奨タイミング | 初期実装 | 反応を見てから拡張 |

### 感情の強弱表現

段階的なフィードバック価値を持たせることで、コンテンツへの貢献度を可視化できる。

- **希少性**: 色や種類ごとに稀少性を変え、貴重なフィードバックを目立たせる
- **ユーザー可視化**: 誰がアクションしたかを表示することで社会的証明を強化

### ネガティブフィードバックの設計方針

| 表示方針 | 説明 | リスク |
|---------|------|--------|
| 否定評価数を公開 | 不適切投稿の抑止効果 | 誹謗中傷を加速させる可能性 |
| 否定評価数を非公開 | ユーザーへの心理的ストレス軽減 | 投稿者には管理画面で確認可能にする |
| 否定評価ボタン自体を非表示 | 最もリスクが低い | フィードバックの粒度が落ちる |

> **設計原則**: アクションを受け付けることと、その数値を公開することは分けて考えてよい。

### いいね vs 保存の明確な分離

ライトフィードバック（いいね）と保存（ブックマーク）は異なる機能として設計する。

| 機能 | 性質 | 公開可否 | 目的 |
|------|------|---------|------|
| いいね | 相手への表現 | 第三者に見える | コンテンツ作者へのリアクション |
| 保存/ブックマーク | 自分のための記録 | 非公開が望ましい | あとで自分が参照する |

混在させると「保存したいが公開したくない」というニーズに応えられなくなる。

### コンテンツ作者と読者の良い循環

```
投稿 → フィードバック → 満足感 → 再投稿 → サービス活性化
```

いいね機能の評価軸はフィードバック数だけでなく、この循環が継続するかどうか。まずシンプルに実装し、作者・読者双方の反応を観察してから拡張を検討する。

## 保存パターンの判断基準

### 3種類の保存UIと使い分け

| 保存パターン | 特徴 | 向いているケース | 向かないケース |
|------------|------|----------------|--------------|
| **自動保存** | 操作不要、ユーザーが意識しない | ドキュメント・画像編集など作業消失リスクが高い場面 | 慎重な変更が必要なデータ（権限・報酬等） |
| **行ごとに保存ボタン** | 行単位でこまめに保存可能 | 行単位で完結する操作が想定される場合 | 列をまたいだ比較・調整が必要な場合 |
| **すべて保存ボタン** | 1回のアクションで全体を保存 | 複数項目を俯瞰しながら調整する場合 | ページをまたいで項目が分散している場合 |

### 選択フローチャート

```
データを慎重に変更する必要がある？
├─ Yes → 自動保存は避ける（行ごと or すべて保存）
│         操作が行単位で完結する？
│         ├─ Yes → 行ごとに保存ボタン
│         └─ No  → すべて保存ボタン
└─ No  → 自動保存が適している（消失リスクが高い作業）
```

### 自動保存を使う際のUX配慮

- 保存されたことをユーザーに伝えるフィードバックを表示する（"Saved!" 等）
- プレビューが必要な変更（デザイン変更等）は自動確定せず、確認後に保存できるようにする

### すべて保存のアンチパターン

ページをまたいで項目が存在する場合、「すべて保存」では何が変更されたかユーザーが把握できなくなる。1つのページで保存が完結するよう設計すること。

### 未保存状態での離脱対策

フォームから離脱しようとするとき、変更が破棄される旨をダイアログで伝える（ブラウザ標準機能やアプリ内実装）。これにより意図しないデータ消失を防げる。

## 未読・既読の表現設計

### 未読表現の種類

| 表現方法 | 強さ | 特徴 |
|---------|------|------|
| バッジ（件数あり） | 強 | CTRが上がりやすいが、多用すると機能しなくなる |
| バッジ（件数なし） | 中 | 未読の存在は伝えるが注意を引きすぎない |
| 背景色・テキスト変化 | 中 | リスト上で一目で状態がわかる |
| 別画面に分離 | 弱 | 既読情報を隠すことで未読の重要性を高める |

### 未読数を抑制すべき判断基準

過剰な未読数表示は逆効果になる。以下の条件を満たす機能にのみ件数バッジを使用する。

- [ ] ユーザーが能動的に確認しようとする情報か
- [ ] 件数が現実的な範囲に収まるか（膨大な数になっても意味をなすか）
- [ ] 未読を消化しないとサービスが機能不全になるような情報か

### 複数カテゴリの数字表示ルール

異なる意味を持つ複数の数字（未読数・カート数・未完了タスク数）を同じデザインで表示しない。配色・位置・形状で差をつけて意味を分ける。

### 既読タイミングの2パターン

| パターン | 既読になる条件 | メリット | デメリット |
|---------|--------------|---------|-----------|
| **一覧タップで既読** | カテゴリ画面に入った時点 | 数字が溜まりにくい | コンテンツごとの既読を追跡できない |
| **コンテンツごとに既読** | 各コンテンツをタップした時点 | 厳密な既読管理が可能 | 未読数が増え続けると放置される |

**補完策**: コンテンツごとの既読管理を採用する場合、「すべて既読にする」ボタンで一括消化できるようにする。

## 情報更新の通知パターン比較

非同期で情報が更新されるサービスで、変更をユーザーに伝える6つのアプローチ。

| パターン | 方法 | 強制度 | メリット | デメリット |
|---------|------|--------|---------|-----------|
| **a. お知らせ画面** | サービス内の通知一覧ページ | 低 | 既存UIに影響を与えない | お知らせ動線の設計が別途必要 |
| **b. モーダル** | 画面の大部分を使った強制表示 | 高 | 確実に情報を届けられる | 複数更新時にモーダル連打になりストレス大 |
| **c. 詳細画面に更新欄** | 該当コンテンツ画面内にスペース | 中 | コンテンツとの関連が明確 | その画面を訪れないと気づかない |
| **d. 一覧・詳細にアイコン** | 一覧から更新をアイコンで示す | 中 | 一覧画面で気づける | 更新箇所の詳細まで一覧で伝えにくい |
| **e. 履歴管理機能** | 更新履歴を参照可能にする | 低 | 差分確認・ロールバックができる | 能動的に見にいかないと気づかない |
| **f. メール・通知** | サービス外のチャネルで伝える | 中 | 既存画面に変更不要、開発コスト低 | 開封率依存・継続的な計測が必要 |

### パターン選択の判断軸

```
更新を「必ず」見せる必要がある？
├─ Yes → b（モーダル）か f（通知）を検討
│         更新頻度が高い？
│         └─ Yes → モーダルは避け、d（アイコン）を推奨
└─ No  → サービス画面への影響を最小化したい？
          ├─ Yes → f（メール・通知）で検証してから画面実装
          └─ No  → a（お知らせ）か d（アイコン）
```

> **プロセスの工夫**: 推し案を1つ決めつつ、他の案のメリット・デメリットも整理して提示すると、意思決定者が判断しやすくなる。

## ボタンの設計と使い分け

### 状態バリエーションの設計ルール

| 状態 | デザイン指針 |
|------|------------|
| 通常 | ベースデザイン |
| マウスオーバー / 押下時 | 背景と文字のコントラストを**上げる** |
| 利用不可（Disabled） | 画面背景とボタン、ボタンと文字の両方のコントラストを**下げる** |

> **アンチパターン**: 入力完了前にボタンをDisabledにして押せない状態にすると、なぜ押せないかユーザーが理解しにくい。常に押せる状態にし、未入力項目があればエラーメッセージで理由を示す。

### 形状バリエーションと用途

| 形状 | 用途 | 対象範囲 |
|------|------|---------|
| 大きいボタン | 画面全体に対するアクション（保存・送信） | 画面全体 |
| 小さいボタン | 画面の一部に対するアクション | セクション・行単位 |
| テキストボタン | インライン配置のアクション | 文章中・リスト内 |
| アイコンボタン | スペース制限があり意味が自明な場合 | ツールバー・コンパクトUI |

### 同一アクション・異なる形状の使い分け例

同じ「閉じる」アクションでも状況により最適な形状が変わる。

| 状況 | 推奨ボタン形状 | 理由 |
|------|-------------|------|
| 写真拡大モーダル | アイコン（×）| コンテンツ表示面積を最大化したい |
| 入力フォームモーダル | テキストボタン（「キャンセル」） | 意図が明確で誤操作リスクを下げる |
| セクション内クリア | 小さいテキストボタン | 画面内での存在感を抑える |
| 画面全体のクリア | 大きいボタン | スコープが広いことをサイズで示す |

### プライマリ・セカンダリの使い分け

| 原則 | 詳細 |
|------|------|
| プライマリボタンは画面に1つ | デスクトップでは特に複数のプライマリが乱立しないよう管理する |
| セカンダリは背景に馴染む配色 | 複数並んでも主張が強くなりすぎないよう設計する |
| 強弱でサービスの優先度を表す | どのアクションが重要かをビジュアルで伝える |

## 数値入力UIの選択基準

### 4コンポーネント比較テーブル

| コンポーネント | 入力方式 | メリット | デメリット | 最適な用途 |
|-------------|---------|---------|-----------|-----------|
| **テキストフィールド** | キーボード自由入力 | 使い慣れた操作 | 制限かけにくい | 電話番号・自由な数値入力 |
| **プルダウンメニュー** | 選択肢から選択 | 制限をかけやすい、親しみやすい | 選択肢が多いと面倒 | 個数制限・決められた候補 |
| **スライダー** | ドラッグで範囲選択 | 連続値を直感的に操作 | 正確な値を指定しにくい | 音量・明るさ・温度（感覚的調整） |
| **ステッパー** | ＋/－で1ずつ増減 | 確実に値を変更できる | 大幅な変更は手間がかかる | 温度・数量（変更幅が小さい） |

### 選択フローチャート

```
ユーザーが自由に数値を入力してよい？
├─ Yes → テキストフィールド
└─ No  → 選択肢は有限の候補から？
          ├─ Yes → 選択肢の数は少ない（〜10程度）？
          │         ├─ Yes → プルダウンメニュー
          │         └─ No  → テキストフィールドも検討
          └─ No  → 値の変更幅は大きい（0〜100等）？
                    ├─ Yes → スライダー
                    └─ No  → 変更幅が小さい（±5程度）？
                              └─ Yes → ステッパー
```

### スライダー vs ステッパーの場面別判断

| 調整対象 | 推奨 | 理由 |
|---------|------|------|
| 音量 | スライダー | 感覚的・大幅な変更（一気に0にする等）が多い |
| 温度（エアコン・湯沸かし） | ステッパー | 変更幅が小さく正確な値が重要 |
| 照明の明るさ | スライダー | 感覚的な調整が主目的 |
| 購入数量（ECサイト） | プルダウン | 上限制限をかけたいケースが多い |

> **補足**: 生年月日のようにユーザーが正確な値を知っている場合は、選択肢が多くても自由入力（テキストフィールド）の方が使いやすいことがある。

## カードUIの判断基準

### カードUIが効果的な4つの場面

| 場面 | 説明 |
|------|------|
| 不均一な情報の整理 | 高さ・情報量が異なるコンテンツをまとまりよく見せる |
| 個々のコンテンツの強調 | レビュー・記事など1件ずつしっかり読ませたい場合 |
| レスポンシブデザイン | テーブルをスマートフォン対応させる際の代替手段 |
| 視覚的な強弱づけ | フラットなデザインの中に立体感を持たせ存在感を出す |

### カードUI vs テーブル vs リストの選択基準

| 要件 | 推奨コンポーネント |
|------|-----------------|
| 情報を横断的に比較したい | テーブル |
| 各コンテンツを個別に読ませたい | カードUI |
| 短い情報を高速でスキャンさせたい | リスト |
| 高さが均一な情報を並べる | リスト |
| 高さが不均一な情報をまとめる | カードUI |
| スマートフォンでテーブルを代替する | カードUI |

### カードUIのアンチパターン

| アンチパターン | 問題 | 対策 |
|-------------|------|------|
| カードの入れ子（ネスト） | 余白が増えコンテンツ密度が低下 | 外側の要素はカードを使わず横幅いっぱいに配置 |
| 比較が必要なデータにカード | 横断比較がしにくくなる | テーブルレイアウトを維持するか、比較UIを別途設ける |
| 短文のタイムラインにカード | 一画面の情報量が減り、スクロールが増える | リスト形式を検討 |

### レスポンシブ対応時の注意点

テーブル → カードUI変換はモバイルの縦スクロールに最適化できる反面、PC閲覧時のデータ比較体験が失われる。PC・モバイルで異なるコンポーネントを使い分けるか、カードUI内にソート・フィルター機能を加える検討が必要。

## メッセンジャーUIの設計ポイント

### 基本的な画面構成

```
┌──────────────────────┐
│  やりとり先の情報       │ ← 相手が誰か・何の画面かを示す
├──────────────────────┤
│                       │
│  メッセージログ         │ ← 最新が下、古いものが上に流れる
│  （相手のアイコン付き）  │   自分の発言にはアイコン不要
│                       │
├──────────────────────┤
│  操作スペース           │ ← 入力フォーム + テキスト以外の選択肢
└──────────────────────┘
```

### 状態設計の4パターン

| 状態 | 操作スペースの表示 | 目的 |
|------|-----------------|------|
| **a. 操作スペース非表示** | ログが全画面 | ログの読み込みに集中させる |
| **b. 基本表示（フォーカスなし）** | 入力欄＋テキスト以外の候補アイコン | 文字・写真・絵文字を選択可能にする |
| **c. フォーカスあり** | 入力欄のみ（他のアイコンを非表示） | 文字入力に集中させる |
| **d. テキスト入力済み** | 入力欄＋送信ボタン出現 | 送信アクションを明確に示す |

### テキスト以外の情報要素の優先度設計

入力フォーカス前に表示するアイコンの数は最大6つ程度が限界。その中の優先順位はサービスの性質によって変わる。

| サービス特性 | 優先する要素 |
|------------|------------|
| プライベートコミュニケーション（SNS等） | 写真・絵文字・スタンプ |
| ビジネスコミュニケーション（Slack等） | メンション・書式装飾・リンク |
| マッチングサービス | プロフィール参照・写真共有 |

### 操作スペースの切り替えUX

| アプローチ | 特徴 | 例 |
|----------|------|-----|
| **キーボードと同サイズで切り替え** | コンテキストから離れずに選択 | LINE（絵文字・写真がキーボード領域で切り替わる） |
| **全画面モーダルで選択** | 選択に集中でき、元の画面との距離感を作る | Slack（メディア選択時にログがグレーアウト） |

どちらが正解ではなく、「ライトなコミュニケーション」では前者、「丁寧な入力が求められるビジネス利用」では後者が馴染みやすい傾向がある。
