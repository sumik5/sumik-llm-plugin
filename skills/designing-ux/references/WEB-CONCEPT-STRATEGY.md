# WEB-CONCEPT-STRATEGY.md
# デザインコンセプト立案プロセス

Webデザインの機能性・情緒性フレームワーク（→[WEB-FUNCTIONALITY.md](WEB-FUNCTIONALITY.md)、[WEB-EMOTIONALITY.md](WEB-EMOTIONALITY.md)）を実プロジェクトに適用し、デザインコンセプトを立案するための4ステッププロセス。

---

## 4プロセス概要

```
① 調査・分析
        ↓
② デザイン基本方針立案
        ↓
③ デザイン詳細方針立案
        ↓
④ 可視化
```

| ステップ | 目的 | 主な成果物 |
|---------|------|-----------|
| ① 調査・分析 | デザイン作業の前提・制限を整理 | デザイン前提条件表 |
| ② 基本方針立案 | 機能性・情緒性の方向性を策定 | デザイン基本方針（3〜4行の要約） |
| ③ 詳細方針立案 | 8要素ごとの具体的な方針を策定 | デザイン詳細方針（要素別指針） |
| ④ 可視化 | 方針をビジュアルに落とし込み合意形成 | ビジュアルサンプル + 解説資料 |

---

## Step 1: 調査・分析

デザイン作業に入る前の準備段階。対象サイト/アプリが「誰が・いつ・何を・どこで・どのように・何のために使うか」を抜け漏れなく整理する。

### デザイン前提条件表テンプレート

| 項目 | 内容 |
|------|------|
| **対象クライアントとその特徴** | 組織名・理念・公式に打ち出している方針・価値観・企業文化・主要サービスの方針 |
| **対象の業界** | 飲食 / 流通 / 金融 / 宿泊 / IT / 教育 / 医療 / 自治体 など |
| **サイト/アプリの種類** | コーポレートサイト / ECサイト / ポータルサイト / 業務アプリケーション など |
| **対象ユーザー** | 性別・年齢層・職種・ペルソナ（既存があれば参照）など |
| **対象デバイス** | PC / スマートフォン / レスポンシブ / 特定デバイス |
| **デザインの目的・背景・課題** | プロジェクト立ち上げ経緯 / 現行サイトの課題 / リニューアルコンセプト / KPI |
| **その他** | 担当者の要望 / 承認フロー / 素材の提供方法 / バリエーション数 |

### 参照すべき資料

- 現行コーポレートサイト・関連サイト（企業情報・経営方針・採用方針）
- 企業パンフレット・VI/CI
- RFP・プロジェクト計画書
- カスタマージャーニーマップ・ペルソナなどUX関連資料
- ヒューリスティック評価表

> **ポイント**: この段階では具体的なデザイン方向性は記載しない。あくまでデザイン制限・前提を整理する作業。不足情報はクライアントへのヒアリングで補完する。

---

## Step 2: デザイン基本方針立案

デザイン前提条件表を踏まえ、機能性7軸・情緒性6軸それぞれの方針を策定し、最終的に3〜4行のデザイン基本方針としてまとめる。

### 2-1: 機能性の方針策定

機能性7軸（詳細: [WEB-FUNCTIONALITY.md](WEB-FUNCTIONALITY.md)）ごとにプロジェクト固有の方針を記述する。

| 機能性軸 | 方針記述の観点 |
|---------|--------------|
| 誘目性・視認性の向上 | 対象ユーザーの年齢・利用状況に応じた見つけやすさ基準 |
| 情報理解の促進 | 情報の組織化・構造化方針、ビジュアルコミュニケーション活用度 |
| 操作性の実現・向上 | デバイス・環境を踏まえた使いやすさの基準 |
| 習熟度の向上 | 共通ルール・一貫性の水準 |
| アイデンティティの体現 | ブランド要素の取り込み方針 |
| 一般常識・社会的習慣への準拠 | 業界標準・競合と同等の慣習への対応方針 |
| 可変性の維持 | 運用・拡充を見越した柔軟性の水準 |

**記述ポイント**
- 一般的な定義に加え、デザイン前提条件に固有の条件・注力点を追記する
- トレードオフが生じる場合は「〇〇はそこまで考慮しない」と明示することで意思決定を可視化する

### 2-2: 情緒性の方針策定

情緒性6軸（詳細: [WEB-EMOTIONALITY.md](WEB-EMOTIONALITY.md)）ごとに、軸のどちら方向にどの程度振るかを検討する。

| 情緒性軸 | 左端 | 右端 | 方針設定イメージ |
|---------|------|------|----------------|
| 装飾性 | シンプル | デコラティブ | `[●--------]` |
| 成熟度 | 爽やか・若々しい | 円熟した・伝統的な | `[●--------]` |
| 態度 | 真面目・フォーマル | ポップ・カジュアル | `[----●----]` |
| 愛嬌 | スタイリッシュ・スマート | キュート・プリティ | `[●--------]` |
| 品性 | アクティブ・ダイナミック | 静か・落ち着いている | `[●--------]` |
| 温度感 | 冷たい・無機質 | 暖かい・開放的 | `[------●--]` |

**記述ポイント**
- 「どちらでもない」設定も有効。中間位置を意図的に選ぶこともある
- リニューアル案件では現行サイトの位置も合わせて記載すると変化量が明確になる
- 情緒性の方針はデザイナー・アートディレクターが客観的に判断・提案し、ステークホルダーで合意する。クライアントに「どちらがよいか」と委ねるのは避ける

### 2-3: デザイン基本方針のまとめ

機能性・情緒性の方針を総括し、3〜4行に端的に要約する。重複する要素は統合し、プロジェクト固有の優先事項を前面に出す。

**フォーマット例**

```
[デザイン基本方針]
- [情緒性の軸Aと軸Bを踏まえた方向性]
- [機能性の重要軸1と軸2の両立方針]
- [背景条件・制約を踏まえたトーンの補足]
```

**まとめのポイント**
- デザイン基本方針はプロジェクト全体を通じて立ち返るべき指針として機能する
- 機能性と情緒性のバランス・優先順位も明示すると判断基準が明確になる（例: 「〇〇なトーンを優先するが、最低限の視認性は担保する」）

## Step 3: デザイン詳細方針立案

デザイン基本方針を具現化するために、8つのデザイン要素ごとに具体的な方針を策定する（各要素の詳細: [WEB-DESIGN-ELEMENTS.md](WEB-DESIGN-ELEMENTS.md)）。

### 詳細方針の記述構造

各デザイン要素について「具体的な属性方針 → 紐づく機能性/情緒性ラベル」の形式で記述する。

```
[デザイン要素名] 属性名
・方針の記述
機能性：〇〇の向上 / 情緒性：〇〇（軸名）
```

### 8要素別の主要観点

| デザイン要素 | 主要な属性・観点 |
|------------|----------------|
| **配色** | 色相・色数 / 彩度・明度 / アイデンティティカラーの取り込み / コントラスト設定 |
| **タイポグラフィ** | 出力方法（Webフォント/デバイスフォント） / サイズ・ウェイトのコントラスト / 標準フォントサイズ / 書体の種類 |
| **イメージ（写真・イラスト）** | 被写体の選定基準 / トリミング・アングルの方向性 / 比率・黄金比 |
| **アイコン** | 出力方法（SVG/Webフォント） / デフォルメ度（シンプル↔リッチ） / 配置方針 |
| **シェイプ** | 出力方法（CSS/画像） / サイズ基準（タッチターゲット） / ボーダー有無 / 角丸の大きさ |
| **デコレーション** | 採用/非採用の範囲（グラデーション・シャドウ・テクスチャ） / 一貫したルール |
| **インタラクション・アニメーション** | アクション種別ごとのフィードバック方針 / イージング・速度設定 / 情緒性との紐づけ |
| **レイアウト** | 種類（固定/リキッド） / 余白・距離の水準 / 整列方針 / 反復・リズム設計 |

### 詳細方針立案のポイント

- 基本方針を実現するための**具体的で細かい指針**を策定する
- OK例・NG例（参考事例）を添付するとイメージが共有しやすい
- 可能であればデザインガイドラインとして流用できる粒度で記述する

---

## Step 4: 可視化

詳細方針に基づきビジュアルサンプルを作成し、ステークホルダーと合意形成する。

### 可視化のポイント

**評価の軸**: 「デザイン基本方針・詳細方針が概ね反映されているか」

ビジュアルサンプルの評価をその軸に置くことで、感覚的・主観的なフィードバックを構造化できる。方針に反映されていない点があればデザイナーにフィードバックし、ブラッシュアップする。

**方針とビジュアルが乖離する場合のチェックポイント**

```
1. デザイン前提条件表の内容にズレがないか
2. 機能性・情緒性の方針設定に誤りがないか
3. 詳細方針の解釈に齟齬がないか
```

### 解説資料の構成

| セクション | 内容 |
|-----------|------|
| デザイン基本方針（再掲） | 3〜4行の基本方針を冒頭に再掲 |
| 情緒性まとめ（再掲） | 6軸のイコライザー設定を再掲 |
| デザイン要素別解説 | 各要素の方針がビジュアルにどう反映されているかを記述 |

### 解説資料の記述ポイント

- 各解説項目に「機能性: 〇〇」「情緒性: 〇〇（軸名）」のラベルを付与し、基本方針との紐づけを明示する
- 詳細方針の内容に加え、ビジュアル作成時に考慮したことや実現のために取った方策を追記する
- 冒頭に基本方針・情緒性を再掲することで、評価段階で全員が同じ指針を参照できる状態を作る

## プロセス全体チェックリスト

### ① 調査・分析
- [ ] デザイン前提条件表の7項目がすべて記載されている
- [ ] 不足情報のヒアリング・資料収集が完了している
- [ ] この段階ではデザインの具体的な方向性を記述していない

### ② デザイン基本方針立案
- [ ] 機能性7軸すべてについて方針を記述した
- [ ] 情緒性6軸すべてについてイコライザー位置を設定した
- [ ] 3〜4行のデザイン基本方針にまとめた
- [ ] 機能性と情緒性のトレードオフがある場合は明示した

### ③ デザイン詳細方針立案
- [ ] 8つのデザイン要素すべてについて方針を記述した
- [ ] 各方針に機能性/情緒性のラベルを付与した
- [ ] OK例・NG例を添付している（可能な範囲で）

### ④ 可視化
- [ ] ビジュアルサンプルを作成した
- [ ] 解説資料の冒頭にデザイン基本方針・情緒性まとめを再掲した
- [ ] デザイン要素別の解説を記載した
- [ ] ステークホルダーで合意を取り付けた

## よくある落とし穴と対処

### 調査・分析フェーズ

| 落とし穴 | 対処 |
|---------|------|
| 前提条件が曖昧なままデザインに進む | ヒアリングリストを事前に用意し、不明点をすべて解消してから進む |
| デザイン方向性を前提条件表に混入する | 「制限・前提の整理」と「方向性の検討」を明確に分離する |

### 基本方針立案フェーズ

| 落とし穴 | 対処 |
|---------|------|
| 情緒性の方針をクライアントの好みに委ねる | アートディレクターが客観的に提案し、根拠を説明した上で合意する |
| 全軸を均等に扱い優先度が不明瞭になる | プロジェクト固有の重要軸を明示し、トレードオフを意識的に記述する |
| 基本方針が抽象的すぎて判断軸にならない | 「〇〇と〇〇を両立」「〇〇を優先するが〇〇は担保」など具体的な記述にする |

### 詳細方針立案フェーズ

| 落とし穴 | 対処 |
|---------|------|
| 基本方針との紐づけが不明確 | 各方針に「機能性: 〇〇」「情緒性: 〇〇」ラベルを必ず付与する |
| 要素が多く途中で力尽きる | 優先度の高い要素（配色・タイポグラフィ・レイアウト）から着手する |

### 可視化フェーズ

| 落とし穴 | 対処 |
|---------|------|
| ビジュアルが「好き嫌い」で評価される | 冒頭にデザイン基本方針・情緒性まとめを再掲し評価軸を共有する |
| 方針とビジュアルの乖離に気づかない | 要素別解説で方針ラベルを明示し、根拠のある評価を促す |
| フィードバックが感覚的なものになる | 「デザイン基本方針に対してどう見えるか」という問いかけを軸にする |
