# UXデザインの組織導入ガイド

## 仕事としての始めやすさ

| こっそり練習 | 一部業務でトライアル | クライアント巻き込み |
|:---:|:---:|:---:|
| ★★★ | ★★ | ★（条件付き） |

ステークホルダー分析は一人で始められる。導入活動自体はチームの合意が前提。クライアント巻き込みは社内実績ができてから。

---

## 組織的なUXデザイン導入とは

個人やチームでUXデザインを試すことと、組織として継続的に取り組むことは別次元の活動である。組織への導入には、手法の習得だけでなく以下4つの視点が必要になる。

### 組織導入に必要な4つの視点

| 視点 | 内容 |
|------|------|
| **手法** | UXデザインチームが社内に提供可能な手法の明示・実施 |
| **プロセス** | 既存の業務プロセスへの統合・実務での継続的な実施 |
| **文化** | 取り組みの目的・価値・背景の理解と共有。UXデザインの有効範囲の共通認識化 |
| **組織** | 活動を担うチームの編成。育成・評価・採用の仕組み作り |

> どの視点から着手するか、何に注力すべきかは組織によって異なる。4つをすべて同時に取り組む必要はない。

## UXデザインステージ：組織の5段階

UXデザインへの取り組みに対する組織の成熟度を5段階で表したモデル。現在地と目標を確認するために使う。

### 5段階の定義

| ステージ | 名称 | 状態 |
|---------|------|------|
| **第1段階** | 未活動 | 個人レベルの関心・情報収集のみ。組織的な取り組みはない |
| **第2段階** | 有志活動 | 社内合意なし。有志メンバーによる勉強会・非公認活動の段階 |
| **第3段階** | 実験・検証 | パイロットプロジェクト等で実験的・検証的にUXデザインを開始 |
| **第4段階** | 展開 | マネジメントも関与し、自社の重要テーマとして扱われる段階 |
| **第5段階** | 統合 | 組織・制度の両面が整備され、標準的なプロセスとして機能する段階 |

```
[第1段階 未活動]
      ↓ ハードル1「有用性の立証」
[第2段階 有志活動]
      ↓
[第3段階 実験・検証]
      ↓ ハードル2「既存業務への適用」
      ↓ ハードル3「稼働の有償化」
[第4段階 展開]
      ↓
[第5段階 統合]
```

> ボトムアップ型の導入活動では、主に第1〜第3段階が実践の主戦場となる。第4段階以降はトップダウン型の活動スタイルが必要になる。

---

## 3つのハードル

受託業務の現場でボトムアップ型にUXデザインを導入する際の典型的な障壁とその突破策。

---

### ハードル1｜有用性の立証（第2→第3段階）

**ぶつかる壁**：「UXデザインって役に立つの？」

**なぜ難しいか**：実績のない段階で周囲の協力を得るのは困難。活動が立ち消えになるリスクがある。

**突破のポイント**
- 小さくても「成功事例」を早期に作り出す
- 「今まで悩みだったことが解決できた」「したかったことが実現できた」が伝わる事例にする
- 成功体験を積み上げながら社内認知活動を行い、有用性を継続的に立証する

---

### ハードル2｜既存業務への適用（第3→第4段階）

**ぶつかる壁**：「スケジュールや予算を圧迫する取り組みでは？」「自分の仕事がやりづらくなる」

**なぜ難しいか**：変化への抵抗は合理的な反応。改革を迫るアプローチは逆効果になりやすい。

**突破のポイント**
- スケジュール・プロセスをなるべく変更せずにUXデザインを試せるようにする
- リソース（人・モノ・金）をなるべくかけずに実施できる方法を探る
- 実践例：ユーザビリティテストを部分導入 / ワイヤーフレームとペーパープロトタイプを並行実施 / 知り合いにカフェでインタビューして結果を口頭で共有

---

### ハードル3｜稼働の有償化（第3→第4段階）

**ぶつかる壁**：「UXデザインって仕事になるの？」（受託業務における収益化の問題）

**なぜ難しいか**：UXデザインは結果の予測値・費用対効果の事前提示が難しい。クライアントには「やってみないと分からない」ものとして映る。

**突破のポイント**
- 今までのやり方に限界を感じているクライアント、新しいアプローチを模索しているクライアントに向けて提案する
- 自社事例がない段階でも、他プロジェクトの中間成果物（プロトタイプ・テストビデオ・付箋群）を活用して、プロセスと手法の効果を具体的に示す
- パイロットプロジェクトとして無償や値引きで実績作りは有効だが、いつまでも無償のままにしない

## ステークホルダーマップ

導入活動に関わる関係者を4タイプに分類し、それぞれのペイン・ゲインと背景を整理するツール。

### 4タイプの関係者

| タイプ | 定義 |
|--------|------|
| **推進者** | 導入活動を主体的に進めていくメンバーおよびその候補 |
| **協力者** | 主体的には関わらないが、活動への一定の理解を示し支援・協力してくれるメンバー |
| **影響者** | 社内の情報発信力・伝播力が強く、その言動が多くのメンバーに影響するメンバー |
| **抵抗者** | 導入活動に対して抵抗を示す立場をとるメンバーおよびその候補 |

> **影響者は「人」だけでない**：社内メルマガ・掲示板・注目されているプロジェクトなど「モノ」が影響者になる場合もある。

### マップの構成要素

```
外部環境（市場・競合・PEST動向）
  └── 背景（所属組織のミッション・目標・状況）
        └── 関係者（役割・目標・導入活動に対するペイン／ゲイン）
```

**ペイン**: 苦労・悩み・避けたいこと
**ゲイン**: 報酬・動機・得たいこと

### ステークホルダーマップの作成手順

1. 4タイプの各ボックスに関係者（候補を含む）の名前を書く
2. 各人の役割・目標・導入活動に対するペイン/ゲインを記載する（本人や周囲から情報収集。情報がない場合は仮説を書く）
3. 関係者が所属する組織の背景（ミッション・目標・状況）を記載する
4. 導入活動に影響を与える外部環境を記載する

**作成のポイント**
- 複数名で話し合いながら作成すると精度が上がる
- 関係者の表面的な肩書きではなく、**導入活動に関わることで生じるペイン/ゲインに着目**する
- 現在の抵抗者も何かのきっかけで協力者・推進者に変わりうる。その「キー」を事前に検討しておく

### タイプ別記入のポイント

**推進者**
同じ志でも各人のモチベーションは異なる（「ユーザーのため」vs「スキル獲得」など）。個別の背景の違いを整理しておく。

**協力者**
協力理由はUXデザイン自体と無関係な場合もある（新サービス商材化・競合差別化など）。協力者の視点から整理する。直接的な協力が難しければ「名前を借りる」などの間接的な協力から始める。

**影響者**
どうすれば良い方向で情報発信してもらえるか、または誤った方向の発信を防げるかを考える。

**抵抗者**
抵抗者の視点から、導入活動のどこにデメリットがあると考えているかを整理する。「自分のやり方を否定された」「オリジナリティが発揮できない」という心理的な抵抗にも配慮する。

## UXデザイン導入シナリオ

ステークホルダーマップ（現状）とUXデザインステージ（目指す状態）をつなぐための具体的な計画ツール。

### シートの構成

| 左側 | 右側 |
|------|------|
| **ミッション** — 導入活動の意義・価値観・一貫した指針（ステージが変わっても不変） | **アクションプラン** — 4タイプの関係者ごとの具体的な活動計画 |
| **目標** — 次のステージで目指す状態（理想目標と必達目標に分ける） | |
| **強み** — 調整・交渉に際して推進メンバーがもつ強み | |

### 目標の設定方法

```
形式: 「《対象》を《期限》までに《状態》にする」

例:
  理想目標 | 活動時間を来週以降、週4時間以上確保する
  必達目標 | 活動時間を来月以降、月8時間以上確保する
```

**ルール**
- 一度に設定する目標は最大3つ
- 理想目標（望みうるベスト）と必達目標（ここより低いと目指す姿が満たせない下限）に分ける
- 目標に幅を設けることで、関係者との調整・交渉での判断基準と譲歩可能範囲を準備できる

### アクションプランの考え方（タイプ別）

**推進者向け**
新たなメンバーを探す・活動時間/予算の確保・承認の取り付け・参加のハードルを下げる・モチベーション維持。
露出と接触の機会を増やす（出入り口付近への告知掲示・オープンスペースでの活動）ことで、声をかけやすい雰囲気を作る。

**協力者向け**
どの観点から協力を仰ぐかを考える。直接的な協力が難しければ「オーナーとして名前を借りる」だけでも周囲への見え方は変わる。

**影響者向け**
導入活動に関する情報をどう発信してもらうかを考える。影響者が注目するプロジェクトで実績を出すことで、その影響力を活用する。

**抵抗者向け**
導入活動と抵抗者のペイン/ゲインを両立させる方法を探る。「重要なこと」のリストを共に作り、UXデザインがプラスになる部分とマイナスになる部分を一緒に確認する。必ずしもマイナスの影響だけではないことに気づいてもらえる場合がある。

### 導入シナリオの使用サイクル

```
1. UXデザインステージで目指すべき状態を明確にする
   ↓
2. ステークホルダーマップで関係者のタイプと背景を整理
   ↓
3. UXデザイン導入シナリオで具体的なアクションプランを立てる
   ↓
4. 活動から得られたフィードバックをもとに2と3をアップデート
   ↓
（以降繰り返し）
```

## Light Case / Heavy Case

### Light Case（こっそり練習〜有志活動）

- 担当プロジェクトの納品直後に自主的に簡易ユーザビリティ評価を実施
- 評価結果をレポートして社内に共有
- 注目プロジェクトでの実績が自然に問い合わせ・相談を呼ぶ副次効果

### Heavy Case（実験・検証〜展開）

- ステークホルダーマップを複数名で作成し、関係者の背景を体系的に整理
- 導入シナリオを推進メンバーで策定し、全員でミッション・目標を共有
- パイロットプロジェクトでの実績を社内発表会で共有し、次の案件への展開につなげる
