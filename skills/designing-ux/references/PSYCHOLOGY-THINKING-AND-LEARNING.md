# 思考と学習

UIデザインにおける認知心理学の基盤：人間の思考モード・学習特性・意思決定バイアスと設計への応用。

---

## 脳の構造と思考モード

### 3層脳構造

| 層 | 別名 | 主な機能 |
|----|------|---------|
| 旧脳（脳幹） | 古い脳 | 生存本能・自律機能（呼吸・消化・反射）。周囲を「食べられるか・危険か・性交相手か」で分類 |
| 中脳（大脳辺縁系） | 真ん中の脳 | 感情をつかさどる。喜び・恐れ・攻撃性・不安・怒りを引き起こす |
| 新脳（大脳皮質） | 新しい脳 | 意識的・計画的・論理的思考。意思決定・問題解決・自己認識 |

### デュアルプロセス理論（システム1 / システム2）

| 特性 | システム1 | システム2 |
|------|-----------|-----------|
| 脳の層 | 旧脳＋中脳 | 新脳（大脳皮質） |
| 速度 | 高速（システム2の10〜100倍） | 低速 |
| 処理方式 | 並列・自動・無意識 | 逐次・制御・意識的 |
| 精度 | 大ざっぱ・直観・ヒューリスティクス | 緻密・論理的 |
| 短期記憶消費 | ほぼゼロ | 多大 |
| 主導権 | **実際の行動の大半を統制** | 必要時のみ介入 |

**重要な含意：** システム2は「自分が主導権を握っている」と信じているが、実際に介入できるのはまれ。人間の行動は主としてシステム1が決定する。

#### システム2が介入するタイミング

- 厳密・正確な情報を必要とするとき
- システム1が認知できない状況が生じたとき
- システム1が複数の矛盾する反応を示したとき

#### システム1の特徴的な振る舞い

- **問題すり替え**：解きにくい問題を、より容易な問題にすり替えて答えを出す
- **見えないものは存在しない**：目の前にない情報を考慮しない
- **フィルタリング**：システム2の目標と合わない情報を自動的に除去
- **バイアス耐性**：システム2が誤りと認識したあとも、システム1の見解を覆すのは困難

## 経験による学習

### 学習の容易さ

人間は経験から一般論を引き出すことが得意で、学習の多くはシステム1が無意識に行う。

**学習を制約する4要因：**

| 要因 | 内容 |
|------|------|
| 複雑な状況 | 変化する要素が多すぎる、予測困難 |
| 身近な体験への偏重 | 統計データより家族・友人の体験談を重視 |
| 誤った教訓 | 過ちからいつも正しい学習ができるとは限らない |
| 過度の一般化 | 不十分なデータから早急に一般化する |

### 熟練技能の再現

やり慣れた行動はシステム1が管理し、**認知資源をほぼ消費しない**。そのため他の作業と同時並行できる。

- 例：熟練ドライバーが運転しながらラジオを聴く
- 例：慣れたメールソフトでメール確認しながら内容に集中する

**自動化への道：** 練習を繰り返すことで神経回路が形成され、行動がシステム1の管理下に移行する。

### 慣れないことは難しい

慣れない作業はシステム2を使い、作業記憶（平均3〜5項目）に大きな負荷をかける。複数の慣れない要素が同時に発生すると容量を超えてミスが増える。

**UIへの含意：** 馴染みのない操作が重なるシステムでは、ユーザーの認知負荷が急増する。

## 問題解決と計算の苦手さ

| 得意 | 苦手 |
|------|------|
| 経験から学ぶ | 初めての問題を解く |
| 熟練技能の再現 | 複数ステップの計算 |
| 並列処理（システム1） | 統制処理（システム2）の連続実行 |
| パターン認識 | 作業記憶の容量を超える問題 |

**新しい脳があっても、計算は依然として苦手。** 数や計算の概念は人類が獲得したのは紀元前3400年頃であり、人間の脳は計算に最適化されていない。

### UIデザインへの応用

| ルール | 意図 |
|--------|------|
| システム状況と目標の達成度を目立つ形で表示 | ユーザーが現状把握のために認知資源を使わずに済む |
| 目標へ向けてユーザーを誘導する（暗示的・ウィザード形式） | システム1の「自動処理」が働くように誘導 |
| ユーザーが知るべきことを明確・正確に伝える | 推測・消去法の強要を避ける |
| システムの問題をユーザーに診断させない | 技術的訓練のないユーザーには不可能 |
| 設定の数と複雑さを最低限に抑える | 相互作用する設定の組み合わせは人間が苦手 |
| 計算させるより知覚を使ってもらう | グラフィックスで直観的な判断を可能にする |
| 親近感のあるシステムを作る | 業界標準・過去の製品・メタファーを活用 |
| 計算はコンピュータにやらせる | 入力済みデータから算出できる情報をユーザーに再入力させない |

## 学習を促進するUI設計

### 練習の3要素

| 要素 | 原則 |
|------|------|
| **頻度** | 使用頻度に合わせたUIをデザインする。低頻度（ATM等）は毎回ガイド必須。高頻度（メール等）はパワーユーザー向け機能を提供 |
| **規則性** | 定期的に使いたくなるデザインにする。不規則な使用は習慣化を遅らせる |
| **正確さ** | 正確な操作を促す仕掛け（目印・グリッド・ガイド）を設ける。不正確な練習は不正確さを強化する |

**習慣化の目安：** 18日〜254日（複雑な活動ほど長くかかる）

### タスク指向の概念モデル

```
タスク分析 → タスク指向の概念モデル → タスク指向のUI
```

**「実行における隔たり（Gulf of Execution）」を最小化する**

ユーザーの目的とUI操作の差が小さいほど、認知負荷が下がり、習熟が早くなる。

| 概念モデルの条件 | 内容 |
|----------------|------|
| タスク指向 | ユーザーがやりたいことに直接対応する概念のみ |
| シンプル | 概念が少ないほど良い。概念が増えると複雑性は幾何級数的に増大 |
| 一貫性 | 予測可能な操作で、一度学んだ知識を転用できる |

### 一貫性の2レベル

| レベル | 内容 | 影響 |
|--------|------|------|
| **概念レベル** | オブジェクト・アクション・特性の対応付け | ユーザーが「このシステムがどう働くか」を素早く学習 |
| **キーストロークレベル** | 同タイプの操作には同じ物理アクション | 「筋肉の記憶」（筋運動習慣）の定着を促進 |

**一貫性の鉄則：** キーストロークレベルでは業界慣習を踏襲し、革新は概念レベルとタスクレベルで実現する。

### タスク指向の用語

| 悪い例 | 良い例 | 理由 |
|--------|--------|------|
| 「データベース」（共有保存） | 「共有」「パブリック」 | ユーザーのタスク語彙に合わせる |
| 「ローカル」（個人保存） | 「非公開」 | 技術用語ではなくタスク指向の表現 |
| 「UIテーマ」 | 「表示スタイル」 | 開発者専門用語を避ける |

**用語の一貫性ルール（キャロライン・ジャレット）：**
> **名前が同じなら同じもの、名前が違うなら別のもの**
>
> 1概念 = 1用語の厳密な1対1対応。同じ概念に異なる用語を使ったり、異なる概念に同じ用語を使ったりしない。

### 低リスク環境の重要性

高リスクなシステム（ミスが起きやすく・訂正が困難）では、ユーザーが探索や練習を避けるようになり、習熟が妨げられる。

**低リスク設計の実践：**

- エラーを極力防止する（無効なコマンドを非アクティブに）
- エラーが起きたとき、ユーザーが気づきやすくする
- 容易に元に戻せる・取り消せる・訂正できるようにする

## 意思決定

### 人は非論理的に意思決定する

従来の経済学モデル（「人間は合理的」）は、実際の行動予測には役立たない。実際の意思決定の大半はシステム1が行い、システム2はほとんど介入しない。

### 損失回避

損失から感じる苦痛は、同等の利得から得る喜びより**大きく感じられる**。

**リスク行動の四分割パターン：**

| | 利得 | 損失 |
|---|------|------|
| **高い確率** | リスク回避的（確実な少額を選ぶ） | リスク追求的（損失回避のため冒険） |
| **低い確率** | リスク追求的（大きな利得を期待） | リスク回避的（大きな損失を恐れる） |

**UIへの含意：** 有料への誘導や解約抑止には「失うものを強調するフレーミング」が効果的（ただし倫理的運用が前提）。

### フレーミング効果

同じ情報でも、表現の仕方によって選択が変わる。

| 表現 | 反応 |
|------|------|
| 「生存率90%」 | 朗報と感じる |
| 「死亡率10%」 | 悲報と感じる |
| 利得フレーム | リスク回避的になる |
| 損失フレーム | リスク追求的になる |

**「アンカリング」：** あらかじめ基準値（アンカー）を提示することで、それ以降の判断にバイアスをかけられる。

### イメージと記憶の影響

| バイアス | 内容 |
|---------|------|
| 利用可能性ヒューリスティクス | 思い出しやすい出来事の確率を過大評価 |
| 鮮烈なイメージ | 感情的に強い記憶がシステム1の判断を左右 |
| 身近な体験談 | 統計的事実より友人・家族の体験談を重視 |
| サンプルサイズ無視 | 少数例からも大胆に一般化してしまう（30人の調査は信用するが3人では疑う） |

## 認知特性をUIデザインに活かす

### システム2を支援する：意思決定支援

ユーザーが合理的な選択ができるよう、システム2の参加を促す設計。

| 原則 | 実装 |
|------|------|
| すべての選択肢を提示 | 多すぎる場合は親カテゴリーで要約 |
| 代案を見出す手助け | ユーザーが見逃しやすい選択肢を提示 |
| 偏りのないデータ提供 | 客観的・再現可能な方法で収集したデータ |
| 計算をシステムに委譲 | ユーザーに計算させない |
| 主張・前提をチェック | 重要な決定では健全性チェックを実行 |

### システム1を活用する：データ可視化

人間の自動的な視覚プロセス（ゲシュタルト・動体知覚・顔認識等）を動員して、複雑データの理解を補助する。

| 活用する視覚能力 | 可視化の例 |
|---------------|-----------|
| 構造の知覚（ゲシュタルト） | グループ化・近接・閉合 |
| 動きの感知 | 時系列アニメーション |
| 顔の認識 | チャーノフの顔グラフ（多次元データ） |
| 3次元物体の認識 | 3Dデータビジュアライゼーション |
| 境界線・コントラスト | ヒートマップ・カラースケール |

**データ可視化の原則：** 計算・比較をユーザーに課さず、視覚的な直観判断ができる形でデータを提示する。

### システム1に訴える：説得設計

システム2を回避し、直接システム1に訴えかけて行動を促す。

| 効果的な要素 | 内容 |
|------------|------|
| 説得力のある物語 | 統計より個人の体験談・ストーリー |
| 感情的トリガー | 恐怖・希望・満足・喜びを喚起 |
| 具体的なイメージ | 人の顔・具体的な場面 |
| 損失フレーミング | 「得られるもの」より「失うもの」を強調 |

**意思決定支援 vs 説得の使い分け：**

| 目的 | アプローチ |
|------|-----------|
| ユーザーが合理的な選択をできるようにしたい | 意思決定支援（システム2を関与させる） |
| ユーザーに特定の行動を促したい | 説得（システム1に直接訴える） |

どちらが正しいかは倫理・文脈による。意図を明確にしてデザインする。
