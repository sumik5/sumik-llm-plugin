# Webサイトのモーションデザイン実践ガイド

> モーションデザインの理論的基盤（4分類・身体延長・構造認知・状態表現・イージング）は `PHILOSOPHY-MOTION-THEORY.md` を参照。
> 本ファイルはWebサイト制作における**実践パターンと実装判断**を扱う。

---

## 1. Webサイトにモーションを持たせる意味

| 目的 | 説明 |
|------|------|
| **遷移のつなぎ** | 画面・パーツの変化をアニメーションで繋ぎ、「今何が起きたか」をユーザーに自然に伝える |
| **質感・個性の表現** | イージング・物理的な動きでブランドの世界観を表現する。等速でなく自然現象を模した動きが効果的 |
| **注目誘導・操作促進** | 新機能の告知やスクロール誘導など、静止画では伝えにくい機能をアニメーションで示す |

**モーション追加の判断基準**

| 状況 | 判断 |
|------|------|
| 画面全体の内容が切り替わる | 遷移アニメーション必須 |
| ページ内アンカーで位置移動 | スクロールアニメーション必須 |
| クリック可能要素が平坦に見える | ホバー・クリックフィードバック推奨 |
| コンテンツが縦に長い | スクロールフェードイン推奨 |
| すでにアニメーションが多い | 削減を優先（ユーザーを疲弊させない） |

---

## 2. 遷移アニメーションの種類と役割

### ページ内遷移（アンカーリンク）

アニメーションなしだと表示コンテンツが瞬時に切り替わり混乱を招く。

- **持続時間**: 1秒以内（待たせすぎると不快）
- **イージング必須**: 等速移動は慣性のない不自然な動きになる。ease-in-out が標準

### スライドイン（メニュー展開）

| スライド方向 | 適切な場面 |
|-------------|----------|
| 右から | 右上にハンバーガーボタンがある場合（空間的に一致） |
| 上から | ページ上部のメガメニュー |
| 下から | デザインの前後関係がわかりにくくなるため**避ける** |

### ページ遷移とローディング

- **フェード**: 別ページ遷移ではフェードアウト→フェードインが標準。コンテンツが完全に置き換わることを明示
- **ローディングアニメーション**: 重いコンテンツのロード中に使用

| ローディングパターン | 効果 |
|-------------------|------|
| プログレスバー（等速） | 「毎回待たされている」と感じさせる |
| プログレスバー（前半加速） | 体感待ち時間を短くする（推奨）|
| 2〜3秒の演出アニメーション | 「待ち」→「世界観の演出」として認識させる |

> **前半加速の原理**: 0→70%を速く、70→100%をゆっくり進める。「もうすぐ終わる」という期待感が体感待ち時間を短くする。

## 3. マイクロインタラクション

ボタン・アイコン等のパーツ操作時に、操作を促し・結果をフィードバックする動きや機能。

### ホバー（マウスオーバー）

フラットデザインではボタンがクリック可能かどうか視覚的にわかりにくい。ホバーエフェクトで補う。

| パターン | 印象・用途 |
|---------|----------|
| 色の反転 | シンプルで汎用性が高い |
| 半透明の白を重ねる | ソフトな印象、画像上のボタンに向く |
| 少し浮かせる（shadow） | 立体感・プレミアム感 |
| キラッと光らせる | エンターテインメント・CTAボタン |

### クリックアクション・入力支援・状態通知

| 場面 | 推奨実装 |
|------|---------|
| ボタンクリック | 押下状態の色変化（最低限）。ブランドに合うアニメーションでユーザー感情を動かす |
| 入力フォームのフォーカス | プレースホルダを左上に縮小してラベル化（省スペース・リッチな見た目） |
| パスワード間違い | 入力部分が左右に振動（首振りの擬人化。初回ユーザーでも直感的に理解） |
| バリデーションエラー | 入力中にリアルタイムで赤枠等で囲む（ネガティブな印象を軽減） |
| スクロール誘導・NEW通知 | さりげないアニメーションで他要素を阻害せずに機能の存在を示す |

## 4. 動画を利用した表現

### ヒーロービデオ（背景全面動画）

ファーストビューで画面全体の背景に使う動画。世界観・メッセージを一瞬で伝える。

| 項目 | 推奨値・注意事項 |
|------|---------------|
| ループ処理 | 動画の終わりと始まりが繋がるよう編集する |
| テキスト・字幕 | 動画内に入れない（背景上に別要素が乗るため可読性が低下） |
| 目標ファイルサイズ | 1MB以内が理想、最大3〜4MB |
| 動画の尺 | 10秒程度 |
| 画質劣化のカバー | 動画の上にストライプやドットのパターンを重ねると粗さが目立たない |

### フレームアニメーション・文字アニメーション

| 形式 | 特徴 |
|------|------|
| GIF / PNG連番（JavaScript） | 背景透過が可能。長方形の輪郭を意識せずパーツとして扱える |
| 文字アニメーション | 1文字ずつ表示で世界観を伝達。手書き筆致の臨場感・和風レイアウトとの相性も良い |

## 5. スクロールアニメーション・エフェクト

### パララックス（視差効果）

複数レイヤーをスクロール速度が異なる形で動かし、奥行き・浮遊感を表現する。

| レイヤー | スクロール速度 | 知覚 |
|---------|-------------|------|
| 前景（手前） | 速い | 近い |
| 背景（奥） | 遅い | 遠い |

**向いている場面**: 空間・浮遊感を表現したい。画像に自然に視線を集めたい（マスクとの組み合わせ）

### カットイン・フェードイン

スクロールに合わせてコンテンツが外から滑り込む・フェードで現れる汎用表現。

- アニメーション速度はデザインのトーン&マナーに合わせる
  - 落ち着いた高級感 → ゆったり（0.9秒程度、ease-in-out）
  - ポップ・ストリート感 → 勢いのあるスライドイン

### 横スクロール

| 判断基準 | 対応 |
|---------|------|
| テーマに横方向の必然性がある | 採用を検討（SPは縦スクロールで別実装） |
| 単なるデザイン上の差別化目的 | 避ける（SPユーザビリティが著しく低下） |
| 採用する場合 | 縦スクロール操作でも横スクロールが進む実装を行う |

## 6. インタラクティブコンテンツ

| 種類 | 説明 | 主な用途 |
|------|------|---------|
| **主観視点体験** | 360°映像・疑似体験コンテンツ | 物件・施設・旅行の事前体験 |
| **ゲームコンテンツ** | シンプルなミニゲーム（スマホセンサー活用も可） | ブランドプロモーション・SNS拡散・集客 |
| **動画組み込み型** | ユーザー操作に応じて動画が再生される | 世界観体験・ストーリーテリング |
| **ジェネレーター** | ユーザーが操作してコンテンツを生成 | SNSシェア促進・商品・サービスの体験提供 |

## ユーザー確認の原則（AskUserQuestion）

確認すべき場面:
- ヒーロービデオを使うかどうか（パフォーマンスとの兼ね合い）
- パララックスの採用可否（SEO・スマートフォン対応コストの考慮）
- スクロールアニメーションの演出レベル（ミニマル〜リッチ）
- インタラクティブコンテンツの技術要件（WebGL・ジャイロ等の対応確認）
- ローディングアニメーションのアプローチ（シンプルバー vs. 演出アニメーション）

確認不要な場面:
- ホバー・クリックフィードバックの実装（UXのベストプラクティスとして必須）
- イージングの使用（等速アニメーションは違和感を生むため必ず使用）
- ページ内遷移時のスクロールアニメーション（なしでは混乱を招くため必須）
