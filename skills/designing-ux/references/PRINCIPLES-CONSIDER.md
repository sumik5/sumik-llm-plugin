# UX実践原則：Consider（原則 01–17）

フェーズ「Consider」はプロジェクト着手前の哲学的準備。
UXの本質・倫理・時間軸・認知効果を把握し、正しい視点でデザインに臨む。

---

### 01. ユーザーを最優先にする

**核心**: デザインはユーザーのためにある。ビジネス目標はユーザー体験を通じて達成する。

- ビジネス要件とユーザーニーズが衝突したとき、ユーザー側に立って解決策を探る
- 「ユーザーに何をさせたいか」ではなく「ユーザーが何を達成したいか」を問う
- ユーザーの怒り・混乱・不満はデザインの失敗シグナル

**If/Then**: If ビジネス要件とユーザーニーズが対立するなら then ユーザーの利益を優先した代替案をステークホルダーに提案する

---

### 02. UXとUIは別物だが同時進行する

**核心**: UX（体験設計）とUI（視覚実装）は役割が異なるが、分離して進めると整合性が失われる。

- UXはユーザーの目標達成プロセスを設計する。UIはそれを視覚的に実現する
- UXなきUIはただの見た目。UIなきUXは実現されない構想
- 両方の視点を持つデザイナーは、フローと見た目の矛盾を早期に発見できる

**If/Then**: If ワイヤーフレームを作成するなら then 同時にビジュアルデザインの方向性も検討し、後で齟齬が出ないようにする

---

### 03. ユーザビリティはUIに大きく左右される

**核心**: どれほど優れたUX設計でも、UIが複雑・不明確なら使いにくい製品になる。

- ユーザビリティ問題の大半はUI層で発生する（ラベル・配置・フィードバックの不足）
- 美しいUIは信頼感を生み、ユーザーの忍耐力を高める（→原則08 美的ユーザビリティ効果）
- アクセシビリティはUIレベルで実装される（色・フォントサイズ・コントラスト等）

**If/Then**: If ユーザビリティテストで問題が出たなら then まずUI要素（ラベル・ボタン配置・フィードバック）から原因を探る

---

### 04. ユーザーの期待を超えることを目指す

**核心**: 最低限の要件を満たすだけでは、ユーザーに記憶される製品にはならない。

- カノモデルの「魅力品質」がある製品だけが口コミで広がる
- ユーザーが気づいていないニーズを先取りすることで感動を生む
- 過剰な演出ではなく、「ちょうど必要なタイミングで役立つ」ことが感動の源泉

**If/Then**: If 基本機能の実装が完了したなら then 「ユーザーが驚く瞬間」を1つ加えられないか検討する

---

### 05. デザインは中立ではない

**核心**: すべてのデザインの選択は、特定の行動・感情・価値観を促進または阻害する。

- デフォルト値・オプションの順序・ボタンの色はユーザーの意思決定に影響する
- 「ダークパターン」は意図的にユーザーを誤誘導するデザイン。倫理的に許容できない
- デザイナーは意図せず偏見を製品に埋め込む可能性を常に意識する

**If/Then**: If UIを設計するなら then 「このデザインが誰かを不当に不利にしていないか」を問う

---

### 06. 言葉がUXを作る

**核心**: コピー（ラベル・ガイダンス・エラーメッセージ）はUXの最重要要素の一つ。

- UXライターがいないチームは、エンジニアかデザイナーがコピーを書く。質は著しく下がりやすい
- 専門用語・否定形・受動態はユーザーを混乱させる
- エラーメッセージは「何が問題で、どうすれば解決できるか」を明確に伝える

**If/Then**: If エラーメッセージを書くなら then 「問題の説明」「原因」「解決手順」の3要素を含める

---

### 07. 視覚メタファーに頼りすぎない

**核心**: アイコンやビジュアル比喩は文化・世代・経験によって解釈が異なる。

- フロッピーディスクの「保存」アイコンを知らない世代が増えている
- ハンバーガーメニューは認知率が高いが、発見可能性は低い
- ラベルのないアイコンはユーザーに学習コストを強いる（→原則72 アイコンの限界）

**If/Then**: If アイコンのみでUIを構成するなら then 少なくとも重要なアクションにはテキストラベルを併記する

---

### 08. 美的ユーザビリティ効果

**核心**: 美しいデザインはユーザーに「使いやすい」と感じさせ、問題への耐性を高める。

- 視覚的に洗練されたUIはユーザーの信頼感を高め、エラーを許容しやすくなる
- この効果は諸刃の剣：問題を隠蔽することで、重大なUX欠陥の発見が遅れる
- テスト参加者に高フィデリティプロトタイプを見せると評価が甘くなる傾向がある

**If/Then**: If ユーザビリティテストを実施するなら then ビジュアルデザインが判断を歪めないよう低フィデリティで行う

---

### 09. フォン・レストルフ効果（孤立効果）

**核心**: 均質な要素の中で際立つ要素は記憶に残り、注目を集める。

- 重要なCTAは周囲と異なる色・サイズ・形で強調する
- 強調要素が多すぎると効果が消える（すべてが重要＝何も重要でない）
- ランディングページのCTAボタンは1つに絞るのが最も効果的

**If/Then**: If 複数のボタンを配置するなら then 最重要アクション1つだけをビジュアル的に際立たせる

---

### 10. 系列位置効果

**核心**: ユーザーはリストの先頭（初頭効果）と末尾（新近効果）の要素を最も記憶する。

- ナビゲーションメニューの最重要項目は先頭か末尾に配置する
- フォームの最初と最後の質問は最も印象に残る
- 中間に埋もれた情報は見落とされやすい

**If/Then**: If ナビゲーションを設計するなら then 最重要リンクをリストの先頭または末尾に配置する

---

### 11. 少ないほど良い（Less is More）

**核心**: 機能・選択肢・情報量を削減することで、ユーザーの認知負荷が下がり使いやすくなる。

- 使われていない機能は製品を複雑にする害悪
- 削除できる要素を探し続けることがUXの継続的改善
- シンプルさはユーザーが自分で追加できないが、複雑さはデザイナーが後から削除しにくい

**If/Then**: If 機能追加を検討するなら then 先に「削除できる既存機能」を探す

---

### 12. 少ないほどつまらない（Less can be Boring）

**核心**: 過剰なミニマリズムは冷たく無機質な体験になる。感情的な豊かさも必要。

- 完全に装飾を排除したUIは効率的だが、ブランドの個性や温かみを失う
- 適切なマイクロインタラクション・イラスト・空白の活用が体験を豊かにする
- 原則11との均衡：削減すべきは「認知負荷を増やす要素」であり「感情的価値」ではない

**If/Then**: If UI簡素化を行うなら then 機能的でない要素でも感情的価値があるものは残す

---

### 13. フィードバック速度（0.1秒 / 1秒 / 10秒）

**核心**: ユーザーの知覚に合わせたシステム応答速度の設計が体験品質を左右する。

- **0.1秒以内**: 即時反応。ユーザーはシステムが自分の操作に直接反応していると感じる
- **1秒以内**: 思考の流れが維持される。ローディングインジケーターは不要
- **10秒以内**: ユーザーの注意が他に向く。プログレスバー・説明テキストが必要
- 10秒超: ユーザーは別タブを開くか離脱する

**If/Then**: If API応答に1秒以上かかるなら then スケルトンUIまたはプログレス表示を実装する

---

### 14. フリクション（良いフリクションと悪いフリクション）

**核心**: すべての摩擦が悪いわけではない。意図的な摩擦が誤操作・後悔・危険を防ぐ。

- **悪いフリクション**: 不要な確認ダイアログ・複雑な登録フォーム・隠れた機能
- **良いフリクション**: 削除前の確認・高額決済の二重確認・危険な操作の警告
- フリクションの適否は「その行動をユーザーが本当にやりたいか」で判断する

**If/Then**: If 破壊的操作（削除・送金・公開等）を実装するなら then 意図的な確認ステップを設ける

---

### 15. 第一印象は10秒で決まる

**核心**: ユーザーは製品を初めて開いた10秒で継続利用するかどうかを判断する。

- オンボーディングフローは「価値の証明」を最優先にする
- ログイン・権限要求・チュートリアルは価値体験の後に置く
- 初回起動時の空状態（empty state）はユーザーの離脱ポイントになりやすい

**If/Then**: If オンボーディングを設計するなら then ユーザーが価値を体感できるまでの手順を最小化する

---

### 16. 時代を問わないUXは存在しない

**核心**: UXのベストプラクティスは技術・文化・社会の変化とともに常に更新される。

- スマートフォン普及前のUI標準（マウスオーバー前提等）は今では通用しない
- AIの台頭でプロンプトUI・音声UIなど新しいパラダイムが生まれている
- 過去の成功パターンへの依存が革新を阻む

**If/Then**: If 5年以上前のUXガイドラインを参照するなら then 現在のコンテキストに適合するか批判的に検証する

---

### 17. 永遠に続くものはない

**核心**: 製品・機能・デザインシステムはいつか廃止される。それを前提に設計する。

- 成功した製品は必ず競合・技術変化・ユーザー嗜好の変化に直面する
- 技術的負債を蓄積しすぎた製品は刷新できず消えていく
- 「完成したデザイン」は存在しない。継続的改善を組み込んだ設計が長寿命の秘訣

**If/Then**: If 新機能を実装するなら then 将来の廃止・置換を容易にするモジュール設計を意識する
