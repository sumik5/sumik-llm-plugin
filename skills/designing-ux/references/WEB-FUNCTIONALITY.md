## 1. Webデザインの定義と構造概念

### マテリアル・コンテナ・コンテンツの関係

Webデザインを理解するための3つの基本概念。

| 概念 | 定義 | 例 |
|------|------|-----|
| **マテリアル** | コンテンツの素になる原材料 | 写真・原稿・動画素材 |
| **コンテナ** | マテリアルを加工・整形する行為・枠組み | 配色・レイアウト・タイポグラフィ |
| **コンテンツ** | マテリアルにコンテナを与えて生まれる成果物 | Webページ上の各要素 |

```
マテリアル → [コンテナ（デザイン行為）] → コンテンツ
                        ↕
              機能性 + 情緒性を付与
```

### Webデザイン = 機能性・情緒性の設計

デザインによってユーザーに引き起こされる変化・反応は2種類に大別される。

| 変化の種類 | 内容 | 例 |
|-----------|------|-----|
| **機能性** | 使いやすさ・読みやすさ・分かりやすさに関わる変化 | 「見やすい」「使いやすい」「分かりやすい」 |
| **情緒性** | 感情・情緒・印象に関わる変化 | 「かっこいい」「おしゃれ」「スマート」 |

> 情緒性の詳細フレームワーク（6軸イコライザーモデル）は `WEB-EMOTIONALITY.md` を参照。

## 2. 機能性の定義と7軸体系

### 機能性の定義

文字が認識できる、内容が理解できる、操作がスムーズに行える——これらをまとめて**機能性**と定義する。機能性を細分化すると以下の7項目に区分される。

### 7軸クイックリファレンス

| # | 機能性 | 一言要約 | 優先度（常時） |
|---|--------|---------|--------------|
| ① | 誘目性・視認性の向上 | 見つけやすくする | 最高 |
| ② | 情報理解の促進 | 分かりやすくする | 最高 |
| ③ | 操作性の実現・向上 | 使いやすくする | 最高 |
| ④ | 習熟度の向上 | 覚えやすくする | 高 |
| ⑤ | アイデンティティの体現 | 自分らしくする | 中（プロジェクト依存） |
| ⑥ | 社会的習慣への準拠 | 受入れやすくする | 中（文脈依存） |
| ⑦ | 可変性の維持 | 変えやすくする | 高（運用フェーズ） |

> **重要**: 7軸は「どの程度まで達成するか」がプロジェクト・ユーザーセグメントによって変わる。達成すべき水準の設定方法は `WEB-CONCEPT-STRATEGY.md` を参照。

## 3. 機能性① 誘目性・視認性の向上

### 定義

- **誘目性**: 視線を誘導すること
- **視認性**: 目でその存在を認めること

画面上のどこに何があるかをユーザーに正しく認識させることが、Webデザインで解決すべき最初の課題。

> ⚠️ 誘目性・視認性は「存在の認識」のみ。認識したものを理解・操作できるかは②③の課題。

### 実践方法

| 実践方法 | 内容 | 具体的手法 |
|---------|------|----------|
| **視線の誘導** | 目に留まるよう変化を起こす | サイズ拡大・アイコン配置・点滅/移動アニメーション・矢印指示 |
| **コントラストの調整** | 前景と後景の差を明確にする | 色相・彩度の差をつける・枠線を施す |

### 判断基準テーブル

```
視認性を強化すべきタイミング:
- ターゲットが高齢層 → 強いコントラスト・大きなサイズ
- 重大な情報を即座に伝達 → 画面アクセス直後に目に入る位置・サイズ
- 緊急アラート・通知 → 動き・明度コントラスト・強調色

視認性を抑えてもよいタイミング:
- 情緒的な世界観を優先するプロモーションサイト → あえてコントラスト抑制
- ハイリテラシーユーザー向け → シンプルで情報密度が高い設計
```

→ 配色によるコントラスト調整の具体実装は `WEB-COLOR.md` を参照。

## 4. 機能性② 情報理解の促進

### 定義

視認した要素が「どういう情報か」「何を表現しているか」を理解させること。情報を構造化し、ユーザーの認知負荷を下げる。

### 実践方法

| 実践方法 | 内容 | 具体的手法 |
|---------|------|----------|
| **統一感・類同性の演出** | 関係性の強い情報をグルーピング | 整列（グリッドに沿わせる）・接近/離隔・包囲（枠線・背景色） |
| **要素間距離の最適化** | 行間・マージン・パディングの適正化 | 行間を広げ圧迫感を解消・余白を設計する |
| **情報の抽象化・視覚化** | 文字情報をビジュアルに変換 | 図版・グラフ・ラベル・アイコン・リスト・テーブル |
| **コントラストの調整** | 情報の階層構造を意匠で表現 | サイズ・配色・太さで重要度の差をつける |
| **安定感の演出** | 視線の流れと構図で調和をつくる | Z字の視線流・左右対称・反転配置によるリズム |

### グルーピングの3手法

```
1. 整列（Alignment）
   └ 基準線（グリッド）に沿わせることで統一感を生む

2. 接近・離隔（Proximity）
   └ 関係性が強い要素は近く、弱い要素は遠く配置

3. 包囲（Enclosure）
   └ 枠線・背景色でひとまとまりを視覚的に明示
```

> **参考**: 「近接・整列・コントラスト・反復」はグラフィックデザイン一般の情報理解原則として知られる。本フレームワークはこれをWeb向けに5項目に拡張・再分類したもの。

→ レイアウトの整列・グリッド実装は `WEB-LAYOUT.md` を参照。

## 5. 機能性③ 操作性の実現・向上

### 定義

情報閲覧だけでなく、クリック・タップ・入力といった操作を快適にし、利用満足度を上げること。

### 操作の5種類

| 操作種別 | 説明 | 代表的UIコンポーネント |
|---------|------|---------------------|
| **自由入力（記入）** | 非定型データの入力 | テキストフィールド |
| **定型入力（選択）** | 複数選択肢から選ぶ | ドロップダウン・ラジオボタン・チェックボックス・スライダー |
| **出現・消失** | 要素の表示/非表示 | モーダルウィンドウ・カルーセル・ツールチップ |
| **遷移** | 別ページ・別箇所への移動 | リンク・タブ・ナビゲーション・パンくず |
| **決定** | データの送信 | ボタン |

> UIコンポーネントの実装パターンは `WEB-UI-COMPONENTS.md` を参照。

### 実践方法（時間軸別）

操作の前後と操作中・全般の4段階で考える。

#### 操作前

| 実践方法 | 内容 |
|---------|------|
| **期待感の演出** | 操作可能であることをビジュアルで示す（立体的なボタン・枠線で囲まれたフィールド） |
| **選択の容易化** | タップ・クリックに不自由ないサイズ・位置への配置 |

#### 操作中

| 実践方法 | 内容 |
|---------|------|
| **レスポンスの向上** | 表示速度の軽量化（画像削減・圧縮・CDN活用） |
| **誤作動の防止** | 適切なラベル・誤解のないフォーム・ミスタッチ防止レイアウト（決定/キャンセルの差別化） |

#### 操作後

| 実践方法 | 内容 |
|---------|------|
| **復元・復帰への補助** | 簡単に元の状態に戻れる設計（×ボタン・入力値の自動復元・トグル操作） |

#### 操作全般

| 実践方法 | 内容 |
|---------|------|
| **ステータスの伝達** | 今どの状態かをフィードバック（ステータスバー・プログレス表示） |
| **構造の可視化** | 処理内容や操作結果をビジュアルで間接表現（ローディング画面・構造アニメーション） |

### 操作性の優先度判断

```
最重要実践: レスポンスの向上
→ どんなに洗練されたUIでも挙動が重ければストレスになる
→ パフォーマンス最適化が操作性の土台

次に重要: 期待感の演出・誤作動の防止
→ 「押せるか」が分からない、「間違えた」が起きる設計は基礎的な欠陥
```

## 6. 機能性④ 習熟度の向上

### 定義

繰り返し利用する中で、学習コストをかけずに自然とインターフェースの使い方を習熟できるようにすること。

### 設計思想：「覚えやすい」より「覚えなくていい」

| アプローチ | 内容 |
|-----------|------|
| **覚えにくいものを覚えやすく** | 視覚的一貫性でパターンを予測可能にする |
| **覚えなくていいようにする** | 同じ役割には必ず同じ意匠・インタラクションを適用する |

### 実践方法：一貫性の保持

同じ役割を持つパーツ・エリア・ページには、**同じ意匠・インタラクション・配置ルール**を必ず踏襲する。

| 一貫性を欠く例（NG） | 一貫性を保つ例（OK） |
|--------------------|-------------------|
| 同じ種類の見出しでもページごとにサイズ・色が変わる | 同じ階層の見出しは常に同じスタイル |
| 同じ色のボタンが別ページで別機能を持つ | ボタン色と機能は1対1で対応 |
| 同じコンポーネントなのに箇所によってデザインが異なる | 同一コンポーネントは全箇所で同じ見た目 |

### 副次効果

一貫性の保持はユーザーの習熟度向上だけでなく、以下の効果も生む。

- **ガバナンス強化**: Webサイト・アプリケーション全体の設計ルールが明確化
- **ブランディング強化**: 一貫したビジュアル言語がブランド認知を高める
- **メンテナンスコスト削減**: 変更時の影響範囲が予測しやすくなる

## 7. 機能性⑤ アイデンティティの体現

### 定義

クライアント・プロダクト・ブランド固有のクリエイティブやレギュレーションをWebデザインに取り入れ、他社と差別化すること。

### 体現するメリット

| メリット | 内容 |
|---------|------|
| **識別性の向上** | 他社との区別が容易になり、効率的にブランドを選別できる |
| **満足度の向上** | ブランドに愛着を持つユーザーの所有・利用満足度が上がる |

### 実践方法

| 実践方法 | 内容 | アプローチ |
|---------|------|----------|
| **帰属の明確化** | 画面上の目立つ位置にロゴを配置 | 直接的（多くは左上） |
| **マスコット・キャラクターの採用** | 公式キャラクターをWebデザインにも配置 | 直接的（ファン向け誘引効果） |
| **サービス・理念のモチーフ化** | 企業文化・ソリューションをモチーフとした画像・シェイプを配置 | 間接的 |
| **レギュレーションの適用** | テーマカラー・レイアウトテンプレート・オリジナルフォントの適用 | 直接的（VI準拠） |

### アイデンティティと情緒性のバランス

```
判断ポイント:
アイデンティティ体現 ≠ 常に100%適用

例:
- 公式キャラクターを配置 → キュートさが増す
- スタイリッシュなトーンが求められるプロジェクト → 露出を控えめに調整

プロジェクトのデザイン方向性を優先し、アイデンティティ体現の強度を調整する
```

## 8. 機能性⑥ 社会的習慣への準拠

### 定義

多くの人が「そうである・そのほうが好ましい」と認識している慣習や常識をデザインに取り入れることで、広く受け入れられる・違和感のない表現を実現すること。

### 実践方法

| 実践方法 | 内容 | 例 |
|---------|------|-----|
| **審美性の追求** | 人が概ね好意的に受け止める「美しさ・整い」を具現化 | 人体写真の健康的な補正・適切なトリミング・黄金比の採用 |
| **文化的背景への依拠** | 社会・文化の中で形成された連想を利用 | 女性向け → ピンク系・子供向け → 明るく鮮やかな配色 |
| **モデルケースの想起** | 慣れ親しんだ体験を想起させ、ストレスを軽減 | ラジオボタン（選択ボタンの物理的類似物）・アコーディオンメニュー |
| **多数派への追従** | 「よく見かける・多くが使っている」信頼感を活用 | 大手サービスが採用するナビゲーション構造・UIパターンを踏襲 |

### 準拠レベルの判断基準

```
積極的に準拠すべき場面:
- 広くリーチするコーポレートサイト・ECサイト
- 高齢層・Webリテラシーが低いユーザーが対象
- 信頼性・安心感の訴求が優先課題

準拠を意図的に崩してよい場面:
- プロモーションサイト・ブランドキャンペーン
- 革新的・先進的なイメージの訴求が強く求められる
- ターゲットが感度の高いユーザー層

⚠️ 「それっぽさ」は受け入れやすさを生む反面、
   全く新しいアイデア・表現を阻害する側面もある
```

## 9. 機能性⑦ 可変性の維持

### 定義

運用・拡充において変更・削除・追加が発生した際も、機能性と情緒性を損なわず、効率的に作業を行えるよう柔軟な設計にしておくこと。

### なぜWebデザインに可変性が求められるか

| 理由 | 内容 |
|------|------|
| **頻繁な更新** | Webは一度作って終わりではなく、開発・運用で変更・複製・展開が頻繁に発生する |
| **環境の非一定性** | ブラウザ・デバイス・解像度が一定でない（印刷物やファッションとの根本的な違い） |

### 運用で発生する変化パターン

```
- マテリアル（写真・ラベル・原稿）が変わる
- コンポーネントの数が変わる
- コンポーネントの順番・組み合わせが変わる
- コンポーネントの位置が変わる
- 閲覧デバイスが変わる
```

### 実践方法

| 実践方法 | 内容 | 効果 |
|---------|------|------|
| **画像の不使用** | テキスト・装飾をCSSとデバイスフォントで実装 | グラフィックツール不要で変更可能 |
| **相対的なサイズ設定** | ブラウザ・デバイスのサイズ変更に自動吸着する設計 | マルチデバイス対応の安定化 |
| **伸縮性の担保** | コンポーネントが増えてもレイアウトが破綻しない設計 | コンテンツ追加に耐える構造 |
| **入れ替え・組み換えの容易化** | マテリアル変更時に閲覧・操作に支障がでない設計 | 様々なバリエーションに耐えうる柔軟性 |

### 可変性を犠牲にしてよい例外ケース

```
以下の条件が重なる場合は可変性よりも表現の自由・設計スピードを優先してよい:
1. 運用頻度が低い（LP・キャンペーンサイト）
2. 閲覧環境が特定デバイスに限定されている

例: スマートフォン限定の短期キャンペーンLP
→ レスポンシブ対応を省略し、ビジュアル表現に注力する選択もあり
```

## 10. 機能性と情緒性のトレードオフ

### 基本的な関係

機能性を高めることで情緒性が失われる場合がある。これはデザインの本質的なトレードオフ。

| 機能性の高い設計 | 失われやすい情緒性 |
|-----------------|-----------------|
| 鮮やかな配色・大きなパーツ → 見やすい | スタイリッシュさ・洗練された印象 |
| 操作への丁寧なフォロー → 使いやすい | シンプルさ・ミニマルな印象 |
| 大量のフィードバック・ガイド → 分かりやすい | ハイリテラシーユーザーへの親和性 |

### トレードオフの判断プロセス

```
Step 1: ターゲットユーザーのリテラシー水準を定義する
Step 2: プロジェクトで求められる情緒性の方向性を定義する（→ WEB-EMOTIONALITY.md）
Step 3: 機能性7軸それぞれの「達成水準」を設定する
Step 4: 設定した水準内で情緒性を最大化するデザインを探る
```

> 機能性水準の具体的な設定方法とデザイン基本方針の立案プロセスは `WEB-CONCEPT-STRATEGY.md` を参照。

## 11. 機能性レベル最適化の判断テーブル

ユーザーセグメントとプロジェクト特性に応じた機能性優先度の調整指針。

### ユーザーセグメント別

| ユーザー特性 | 強化すべき機能性 | 調整可能な機能性 |
|------------|---------------|---------------|
| **高齢層** | ①視認性（大・高コントラスト）・③操作性（大きなタップ領域） | ⑥社会的習慣（馴染みのあるUIのみ使用） |
| **低Webリテラシー** | ①視認性・②情報理解・③操作前の期待感演出 | ⑦可変性（変更頻度低ければ許容） |
| **高Webリテラシー（プロ向け）** | ③操作中レスポンス・⑦可変性 | ③操作前演出（過剰なガイドを省略） |
| **ブランドファン** | ⑤アイデンティティ・④習熟度（ブランド体験の一貫性） | ⑥社会的習慣（独自表現を許容） |

### サイト種別別

| サイト種別 | 最優先機能性 | トレードオフ対象 |
|-----------|------------|---------------|
| **コーポレートサイト** | ②情報理解・④習熟度・⑥社会的習慣 | ⑦可変性（大規模更新なら投資価値あり） |
| **ECサイト** | ①視認性・③操作性・⑦可変性（商品追加） | ⑤アイデンティティ（機能性を優先） |
