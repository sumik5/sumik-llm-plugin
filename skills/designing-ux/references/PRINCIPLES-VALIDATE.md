# UX実践原則：Validate（原則 88–100）

フェーズ「Validate」はリリース前後の検証・改善・継続の段階。
テスト手法の限界・メトリクスの使い方・ローンチ後の姿勢を扱う。

---

### 88. 音声アシスタントの限界（10〜40%エラー率）

**核心**: 音声UIは10〜40%のエラー率が現実。完璧な自然言語理解を前提にした設計は破綻する。

- 音声UIは補完手段として設計し、常にビジュアルフォールバックを用意する
- 認識エラー時の回復フロー（「もう一度言いましたか？」「こう聞こえました」）が体験を決める
- 音声アシスタントはアンビエント操作（手がふさがっている・視線が使えない場面）で価値を発揮する

**If/Then**: If 音声UIを設計するなら then エラー率20%を前提に、エラー回復フローをハッピーパスと同等の優先度で設計する

---

### 89. デフォルト値を活用する（判断疲れ防止）

**核心**: ユーザーはほとんどの設定を変更しない。デフォルト値が製品体験を決める。

- 初期設定は「大多数のユーザーに最も適した値」を設定する
- デフォルトONのオプトイン（メルマガ同意等）は倫理的に問題がある
- デフォルトの変更には強い根拠が必要。変更されなかった場合の体験を最優先に設計する

**If/Then**: If 新機能のデフォルト値を設定するなら then 「変更しないユーザーが最も多い。そのユーザーにとって最適な値は何か」を基準にする

---

### 90. エラー対処はデザイナーの責任

**核心**: ユーザーのミス・システムエラー・予期しない状況への対処はデザイナーが設計すべき。

- 「ユーザーが間違えた」ではなく「そのミスを誘発するUIがある」という視点で分析する
- エラーメッセージは問題・原因・解決方法の3要素を含む（→原則06）
- エラーが発生した時点でユーザーが入力した内容は保持する（フォームの入力が消えるのは最悪）

**If/Then**: If エラーメッセージを設計するなら then 「何が起きたか」「なぜ起きたか」「どうすれば解決できるか」の3点を含める

---

### 91. 入力の自由度（ポステルの法則）

**核心**: 送信では厳格に、受信では寛容に。ユーザーの多様な入力形式を受け入れる。

- 電話番号の入力はハイフンあり・なし・スペースありを全て受け付ける
- 日付入力は複数のフォーマットを解釈する
- クレジットカード番号はスペース区切りをそのまま入力できるようにする

**If/Then**: If フォーム入力バリデーションを実装するなら then まず入力を正規化（ハイフン削除・スペース除去等）してから検証する

---

### 92. 行動を確認させる（取消可能にする）

**核心**: 重要な操作は確認ステップを入れるか、取消機能（Undo）を提供することでエラーコストを下げる。

- 削除・送信・公開の後にUndoを短時間提供する方が確認ダイアログよりスムーズ（Gmailの送信取消等）
- 確認ダイアログは「本当に重大なリスクがある操作」のみに使う
- 「削除します」より「削除しました。元に戻す」の方がユーザー体験が良い

**If/Then**: If 破壊的操作を実装するなら then まず「取消（Undo）機能」を検討し、それが難しい場合のみ確認ダイアログを使う

---

### 93. 404ページをデザインする

**核心**: 404ページは離脱ポイントではなく、ユーザーを正しい場所へ案内する機会。

- 「ページが見つかりません」だけの404は設計の放棄
- サイト内検索・人気ページへのリンク・ナビゲーションを提供する
- 人間的なコピーとビジュアルで、エラーの不快感を和らげる

**If/Then**: If 404ページを設計するなら then 検索フォーム・主要ページへのリンク・連絡先を含め、ユーザーが次の行動を取れるようにする

---

### 94. プロトタイプは印象確認用

**核心**: プロトタイプはすべての機能を検証するためのものではなく、特定の仮説を検証するための道具。

- プロトタイプの目的を明確にする：フローの検証・視覚的印象・インタラクションの感触
- 完全に動作するプロトタイプにこだわると時間がかかりすぎる
- 「何を学びたいか」を先に決め、その仮説を検証できる最小限のプロトタイプを作る

**If/Then**: If プロトタイプを作成するなら then 「このプロトタイプで答えたい問い」を1〜2つに絞り、その部分だけ高精度に作る

---

### 95. メトリクスを基準にしたデザイン批判に注意

**核心**: 数値指標（CTR・コンバージョン率等）がデザインの善悪を決めるわけではない。短期指標最適化が長期体験を壊す場合がある。

- クリック率を最大化するためにミスリーディングなタイトルを使うのはダークパターン
- 指標はユーザー体験の代理変数。代理変数を最適化することが本来の目標とずれる場合がある
- 「この指標の改善はユーザーの生活をより良くするか」を問い続ける

**If/Then**: If KPIを設定するなら then ビジネス指標だけでなく「ユーザーの満足度・達成感・推薦意向」など体験品質の指標も含める

---

### 96. ユーザビリティテストの現実

**核心**: ユーザビリティテストは完璧ではないが、設計の重大な問題を発見する最も信頼性の高い手法。

- 5名のテストで重大なユーザビリティ問題の約80%が発見できる（Jakob Nielsen）
- モデレートありのテストとモデレートなしのテストは異なる問題を発見する
- テスト後はすぐにチームで問題を共有し、優先度をつけて改善計画を立てる

**If/Then**: If 設計を検証するなら then 完璧なテスト計画が揃うまで待つより、5名でのラピッドテストを素早く実施し早期にフィードバックを得る

---

### 97. 自己採点禁止

**核心**: 自分のデザインを自分で評価することは客観性がなく、重大な問題を見落とす。

- ヒューリスティック分析（→原則61）は専門家の視点だが、「作った本人」は使えない
- チームメンバーによるクロスレビューを最低限として設定する
- 最終的な評価者は常に「リアルユーザー」であることを忘れない

**If/Then**: If デザインを完成させたなら then 自己レビューは行わず、別のデザイナーまたは実際のユーザーに評価を依頼する

---

### 98. パレートの法則（80:20）

**核心**: ユーザーの80%は機能の20%しか使わない。コアユースケースへの集中投資が効果的。

- アクセスログを分析し「どの機能・ページが80%のアクティビティを占めるか」を把握する
- 残り80%の機能は「あって困らないが投資対効果が低い」候補
- リソース制約があるときは、コア20%の体験を完璧にすることを優先する

**If/Then**: If 機能改善の優先度を決めるなら then 使用頻度データで上位20%の機能を特定し、そこへの投資を最優先にする

---

### 99. ローンチ後も継続する

**核心**: ローンチは終わりではなく始まり。実際のユーザーデータが設計改善の最大の情報源になる。

- リリース後の最初の2週間はユーザーの行動データと問い合わせを特に注意深く監視する
- A/Bテスト・フィードバックループ・定期的なユーザーインタビューを継続プロセスに組み込む
- 「リリースしたら完了」の文化がある組織はUX品質を向上させられない

**If/Then**: If 機能をリリースするなら then リリース後30日間の成功指標（KPI）と監視計画を事前に定義する

---

### 100. 期待値を管理する

**核心**: ユーザーが期待することと製品が提供できることのギャップが不満の主要原因。期待値のコントロールがUXの核心。

- マーケティング・オンボーディング・UIのコピーが形成した期待と実際の体験を一致させる
- 過剰な期待を生む誇張表現・ミスリーディングな機能名は長期的な信頼を損なう
- 「期待を超えたとき」の喜びは「期待通りのとき」の満足より記憶に残る（→原則04）

**If/Then**: If プロダクトの説明・マーケティングコピーを書くなら then 実際の機能・体験と照合し、形成される期待と提供できる体験が一致しているか確認する
