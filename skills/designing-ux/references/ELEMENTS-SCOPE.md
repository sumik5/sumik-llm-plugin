# 要件段階 — 機能仕様とコンテンツ要求の定義

戦略段階が「**なぜ**この製品を作るか」を問うとすれば、要件段階は「**何を**作るか」を問う段階だ。
抽象的な目標を、具体的なフィーチャー（機能・コンテンツ）の集合に変換する。

---

## 1. 要件定義の意義

### 1.1 プロセスとしての価値

要件定義は2つの意味で価値がある：

1. **プロセスの価値**: まだ仮説の段階で、製品の潜在的な矛盾や粗い部分への対処を余儀なくされる。今すぐ取り組めることと後回しにすべきことを見極められる
2. **成果物の価値**: チーム全体に「プロジェクトを通じて行うべき全作業の基準点」と「共通言語」を与える

### 1.2 なぜ要件を定義するか

| 理由 | 内容 |
|------|------|
| **構築するものを知るため** | 何を達成するか、いつ達成されたかを全員が知ることができる。プロジェクトが「伝言ゲーム」になるのを防ぐ |
| **構築しないものを知るため** | スコープクリープ（要件増大）を防ぎ、優れたアイデアを今後のリリースに適切に組み込む |

**スコープクリープのリスク**: 追加要求ひとつひとつを見れば小さな負担でも、重なり合うと「雪だるま」のようにプロジェクトが手に負えなくなる。納期・予算の見積りが崩れ、最終的なクラッシュへと突き進む。

## 2. 機能性とコンテンツ

要件段階では機能（ソフトウェアとして何をするか）とコンテンツ（何を伝えるか）の両方を扱う。2つはまったく異なるものに見えるが、要件を定義する際には非常に似た方法で対処できる。

| 要件の種類 | 出力文書 | 内容 |
|-----------|---------|------|
| **機能要求** | 機能仕様書 (Functional Specifications) | 製品が何をするか |
| **コンテンツ要求** | コンテンツ要求文書 | どんな情報を含める必要があるか |

**Note**: 機能とコンテンツは相互に影響する。CMSの機能要件はコンテンツ要求（言語・フォーマット・ワークフロー）によって変わり、機能要件はエラーメッセージなどコンテンツへの影響もある。

---

## 3. 要求の定義プロセス

### 3.1 要求の3つのカテゴリー

人々と話して要求を集めると、以下の3種類が出てくる：

| カテゴリー | 内容 | 対応 |
|-----------|------|------|
| **明示的な要求** | 人々が「欲しい」と言っているもの | そのまま採用できるものも多い |
| **真の要求** | 「欲しい」と言っているが実際には症状への対処であり、背後に別の問題がある | じっくり検討して潜んでいる本当の問題を見つける |
| **潜在的な要求** | 欲しいと気づいていないフィーチャー | ブレインストーミングで発掘する |

### 3.2 要求の発掘手法

| 手法 | 特徴 |
|------|------|
| **ユーザー調査**（戦略段階の手法） | 最も生産的な情報源はユーザー自身 |
| **ステークホルダーへのヒアリング** | 組織内の発言権を持つ人々。フィーチャーへの固執に注意し、背後の戦略目標に焦点を当てる |
| **ペルソナ + シナリオ** | ペルソナが「ユーザーニーズをどうやって満たそうとするか」の短い物語。潜在的な要求を発見できる |
| **グループブレインストーミング** | エンジニア・カスタマーサービス・マーケティング等の多様な部門を集める。製品に深く関わるほど新方向を想像しにくいため、外部視点が重要 |
| **競合分析** | 同じユーザーニーズを満たす競合がどんなフィーチャーを使っているかを調査。直接競合以外からの参考も有効 |
| **ハードウェア要件の確認** | デバイスにカメラ・GPS・センサーが搭載されているかどうか。機能的可能性と制約の両方を生む |

## 4. 機能仕様の書き方

### 4.1 機能仕様の原則

「文書で問題は解決しないが、**定義**では解決する」。ボリュームや詳細の問題ではなく、**明確さと正確さ**が重要。

#### ルール1: ポジティブに書く

システムが「してはいけないこと」ではなく、「それを防ぐためにシステムが何をするか」を書く。

```
❌ このシステムでは、ユーザーは凧糸なしで凧を購入できない。
✅ このシステムでは、ユーザーが凧糸なしに凧を購入しようとしたら、
   凧糸のページに導く。
```

#### ルール2: 具体的に書く

解釈の余地を残さないことが、要求を達成できたかどうか判断する唯一の方法。

```
❌ 最も人気のある動画がハイライト表示される。
   （「人気」とは？「ハイライト」とは？）

✅ 直近一週間の再生回数が多い動画がリストの上位に表示される。
```

#### ルール3: 主観的な表現を避ける

要求は**検証可能**でなければならない。「ヒップ」「派手」「高品質」などは人によって定義が異なる。

```
❌ このサイトは、ヒップで派手なスタイルにする。
✅ このサイトの外観は、会社のブランドガイドライン文書に準拠する。
```

定量化できる要求は数値で表現する：

```
❌ システムはハイレベルのパフォーマンスを有すること
✅ システムは最低でも1000人の同時ユーザーをサポートするよう設計されていること
```

## 5. コンテンツ要求の定義

### 5.1 コンテンツのタイプ

「コンテンツ＝テキスト」ではない。以下のすべてを要件として扱う：

- テキスト（文字数を概算）
- 画像（解像度・サイズを概算）
- 音声・動画（ファイルサイズを概算）
- ダウンロード可能な独立コンテンツ（PDF等）
- 異なるタイプを組み合わせた複合コンテンツ

### 5.2 コンテンツ要求の落とし穴

**フォーマットと目的を混同しない**

「FAQが必要」はフォーマット（質問と答えの羅列）の指定にすぎない。FAQの**目的**（人々が共通して必要とする情報にすぐアクセスできること）を理解せずに作ると、コンテンツ提供者が「考えた質問」への回答になってしまう。

**If X then Y**:
- If コンテンツ要求が「フォーマット名」になっている → then そのフォーマットが果たすべき**目的**に立ち返る
- If 想定サイズが不明 → then サムネール vs フルスクリーンのような差がデザインに大きく影響するため早期に確認する

### 5.3 コンテンツ管理の要件

各コンテンツ要素について以下を明確にする：

| 項目 | 内容 |
|------|------|
| **責任者** | 誰が作成・維持するか。仮定のままにしない |
| **更新頻度** | 製品目標・ユーザーニーズ・利用可能なリソースのバランスで決める |
| **対象ユーザー** | 子供向け・親向け・両方で表現が変わる |
| **コンテンツインベントリー** | 既存コンテンツの棚卸しリスト。大規模プロジェクトでは必須 |

**アンチパターン**:
- ❌ コンテンツを投稿したら放置（定期的なメンテナンスがないとサイトが劣化する）
- ❌ 誰が担当するか決めないまま開発を進める（後で「作業がきつすぎる」となり穴が開く）

## 6. 要求の優先順位付け

### 6.1 優先順位の基準

要求の評価軸：

| 軸 | 問い |
|----|------|
| **戦略目標との整合** | 製品目標とユーザーニーズを満たすか |
| **技術的実現可能性** | 現在のテクノロジーで実装できるか |
| **リソース** | 人・金・時間の制約内に収まるか |
| **他フィーチャーとの関係** | 依存・競合・トレードオフはないか |

### 6.2 戦略目標との1対1対応はほぼない

- ひとつの要求が複数の戦略目標に適用されることがある
- ひとつの目標が複数の要求に関係することもある
- 独立して存在するフィーチャーはほとんどない（周囲のフィーチャーへの依存がある）

**If X then Y**:
- If 戦略目標に沿わないフィーチャー提案がある → then 定義上、要件外。ただし、よいアイデアに思える場合は戦略目標を再検討するサイン
- If ステークホルダーが特定フィーチャーにこだわる → then そのフィーチャーが満たす戦略目標に焦点を当て、「別の方法でも同じ目標を達成できる」と伝える
- If スケジュール上の制約がある → then フィーチャーを次のマイルストーンとして先送りする。今後の開発サイクルの基盤になる
- If 複数の戦略目標の相対的な重要度が不明 → then 優先度の決定は「企業政治」の問題になるため、戦略記述書で事前に階層を定義しておく
