# 業務システム設計パターンと移行テクニック

AWS業務システム構築における、VPCネットワーク全体設計・9つの設計パターン・インフラ自動化・移行テクニックの実践ガイド。

---

## VPCネットワーク全体構成

### 通信要件の整理

AWSに設置するシステムとオンプレミスネットワーク基盤との間で必要となる通信内容を整理する。

**通信要件の記録項目**

| 項目 | 内容 |
|------|------|
| 通信元アドレス | オンプレミスのサーバ群 |
| 通信先アドレス | VPC内リソース / インターネット上のAWSサービス |
| プロトコル / ポート | TCP, UDP, ICMP など |
| 許可・拒否 | セキュリティポリシーに基づく |
| 通信量・頻度・時間帯 | ピーク見積り用 |

**AWSとの通信は2種類に分類される**

- **VPCとの通信**: 従来LAN内で完結していた通信が数ms以上の遅延を伴う。スループット・遅延に厳しい要件（バッチ処理・DBクエリ等）は特に注意
- **インターネット上のAWSサービスとの通信**: S3・CloudWatch・DynamoDB・マネジメントコンソールはインターネット上で稼働。VPN/DirectConnect接続でもインターネット経路が必要

### 物理構成設計

**VPCの接続先選択**

| 接続先 | メリット | デメリット |
|--------|---------|----------|
| 既存WAN（遠隔拠点接続） | 利用者に近い構成。新規回線不要な場合あり | コアネットワーク回線増強が必要。キャリア選択肢が限られる |
| コアネットワーク（データセンタ/本社） | 大容量回線を専用に確保しやすい。広帯域回線の入手性良好 | 遠隔拠点ユーザーとの通信遅延増大。新規回線/機器のコスト・リードタイム |

**インターネット接続の配置選択**

| 方式 | メリット | デメリット |
|------|---------|----------|
| オンプレミス基盤のインターネット接続を利用 | 既存監視・制御ポリシーを再利用できる | S3等のデータ送信増加に合わせた性能確保・ポリシー整備が必要 |
| VPC内にインターネット接続環境を構築 | 業務システムにインターネット非接続ポリシーとの折り合いが可能 | NAT・プロキシの設計・運用が必要 |

**VPCエンドポイント**: S3・DynamoDBへのインターネット不要なアクセス経路。セキュリティポリシーとの整合に有効。PrivateLinkで拡張サービスにも対応。

### 論理構成設計

**デフォルトVPCは削除推奨** — CIDRブロックが変更不可で企業システムに不適合。誤設定のリスクあり。

## アドレス設計（CIDR/サブネット）

### CIDRブロック設計原則

- **AWS全体にまとまったCIDRを事前割り当て**: 例: `10.1.0.0/16` をAWS全体に確保。個々のVPCにはその範囲内でサブレンジを割り当て
- **オンプレミスネットワークのルーティングが集約可能**: 「10.1.0.0/16宛はVGW経由」という1エントリで済む
- **RFC 1918プライベートアドレスを使用**: 割り当てられていないグローバルアドレスは禁止（AWSのサービスアドレスと重複するリスク）

```
例: 10.1.0.0/16 をAWS全体に割り当て
  VPC-A: 10.1.0.0/21
    サブネット: 10.1.0.0/24, 10.1.2.0/24, 10.1.3.0/24
    拡張余地:   10.1.4.0～10.1.7.255
  VPC-B: 10.1.8.0/22
    サブネット: 10.1.8.0/24, 10.1.9.0/24, 10.1.10.0/25, 10.1.10.128/25
    リザーブ:   10.1.11.0～10.1.11.255
```

### サブネット設計のポイント

- **同種の役割を持つサーバを同一CIDRにまとめる** — ルートテーブル・ネットワークACLの適用単位になるため
- **AZ指定は作成時に確定、変更不可** — AZ分散配置には複数サブネットが必須
- **ELB専用サブネット推奨** — ELBはスケールアウト時にランダムIPを使用。EC2インスタンスと共用するとIPアドレスの競合リスクがある
- **ELB要件**: /27より大きく(少なくとも28bit)、作成時点で8個以上の空きアドレスが必要
- **ブロードキャスト・マルチキャストは使用不可** — VPCでは非サポート
- 各サブネット先頭4アドレス+末尾1アドレス(計5個)は予約済みで使用不可

## ルーティング設計

### AWSのルーティング要素

| ゲートウェイ | 役割 | 備考 |
|------------|------|------|
| local | VPC内サブネット間通信（自動設定・削除不可） | ロングゲストマッチより優先される特例 |
| インターネットゲートウェイ (IGW) | VPC〜インターネット間。VPCに1つのみアタッチ可 | パブリックIPまたはNAT GW経由が必要 |
| 仮想プライベートゲートウェイ (VGW) | VPN/DirectConnect接続の外部接続ルータ | 内部ルートテーブルを持つ。1VPCに1つ |
| VPCピアリング接続 | VPC間直接接続（同一リージョン・CIDRが非重複） | トランジット不可。スター型トポロジ不可 |
| VPCエンドポイント (ゲートウェイ型) | S3/DynamoDBへのインターネット不使用接続 | ルートテーブルに論理エントリが追加される |
| NATゲートウェイ | プライベートサブネットからインターネットへのアウトバウンド | AWSマネージド、冗長性済み |
| EC2インスタンス (送信元/先チェック無効化) | カスタムルータ・NATインスタンス用途 | レガシー用途 |

### VGW（仮想プライベートゲートウェイ）の設計上の制約

> ⚠️ **重大な制限**: 1つのBGPピアから100以上の経路を広報受信できない。超過するとBGPピアが切断されVPCとの通信が全断する。

**対策: ルート集約が必須**
- デフォルトルート `0.0.0.0/0` を広報するのが簡単かつ有効
- ただし異なるAS番号の複数VPN/DirectConnectから同じデフォルトルートを広報するとAS Path長に基づく分散やActive-Active競合が発生するため注意

**VGWのルートテーブル優先順位**

1. ロングゲストマッチ（最優先）
2. DirectConnect > VPN（同一経路の場合）
3. BGP AS Pathのメトリック（AS Path長が短い方を優先）

**複数接続の冗長構成パターン**

- Active-Active: 複数接続から等しいAS Path長を広報
- Active-Standby: 異なるAS Path長を広報（短い方がActive）

### ルーティングによるフィルタリング

通信が不要な経路を最初から設定しないことで、無駄なリソース消費を防ぐ。パブリックサブネットはIGW向け、プライベートサブネットはVGW向けにデフォルトルートを設定する方式が典型例。

## フィルタリング設計（SG/NACL）

### ネットワークACL (NACL)

| 特徴 | 内容 |
|------|------|
| 適用単位 | サブネット |
| ルール種別 | 許可・拒否両方定義可能（ステートレス） |
| 制限 | 1NACLあたり20ルール（拡張申請で最大40） |
| 欠点 | ルール数制限が厳しく、主役にはなれない |

### セキュリティグループ (SG)

| 特徴 | 内容 |
|------|------|
| 適用単位 | ネットワークインターフェイス（ENI） |
| ルール種別 | 許可のみ（暗黙の拒否）。ステートフル |
| ルール数 | 1SGあたり50ルール。1ENIに最大5SG適用可能（合計250ルール） |
| 強み | ENI単位の細かい制御。SGをネクストホップとして指定可能 |

### 使い分け原則

- **原則はSGでフィルタリングを行う** — AWSもNACLのデフォルト全許可でSG主体を誘導
- **NACLが有効な例外**: 「許可すべき大きなCIDRの中に拒否すべきアドレスがある」場合など、拒否定義が必要なケース
- **オンプレミスFWの活用**: AWSへ向かう全通信が通過するFWがあれば、そこで定義する方が効率的な場合もある

## ELB設計（ALB）

### ELBのメリット・デメリット

| | ELB（ALB） | 仮想アプライアンス |
|--|-----------|--------------|
| 調達・構築 | API操作で即時 | ライセンス/インストール作業が必要 |
| 維持管理 | AWSが担当（パッチ・スケーリング） | 自社対応が必要 |
| SSL処理 | AWS Certificate Managerで無償・自動更新 | 自社証明書管理 |
| VIP（仮想IP） | 非サポート（スケーリングでIPが変わる） | オンプレでは利用可能 |
| 固有機能 | コンテンツベース負荷分散（ALB）など | 製品固有機能に依存 |

**CLB vs ALB**

| | CLB（Classic） | ALB（Application） |
|--|--------------|-----------------|
| レイヤー | L4 + 一部L7 | L7（HTTP/HTTPSのパスベース分散） |
| 推奨度 | レガシー | 後発で機能・料金とも優位 |
| TCP用途 | 対応 | 非対応（HTTP以外はCLB） |

### ELB利用時の重要ポイント

**スケーリングと暖機運転**

- ELBは段階的にスケールアウトするため、急激なアクセス増（5分間で50%以上増）に対してはPre-Warming（暖機運転）申請が必要
- 申請にはビジネスレベル以上のサポート契約が必要

**スティッキーセッション**

- Cookieを用いたスティッキーセッションをサポート（HTTP/HTTPSのみ）
- Auto Scalingとの組み合わせ時に問題が発生しやすい → インメモリキャッシュでセッション情報を管理するステートレス設計が推奨
- ALBではALB生成Cookieのみサポート（Webサーバ生成Cookieはスティッキー不可）

**接続元IPアドレスの記録**

- バックエンドへの通信はELBのIPにNATされる
- HTTP/HTTPSでは `X-Forwarded-For` ヘッダに接続元IPが記録される → ログ設定を変更して記録

**フィルタ設計**

- ELBにはSGを適用できる（稼働サブネットのNACLも適用される）
- ELB傘下のEC2インスタンスは「ELBが稼働するサブネットのアドレスからの接続のみ許可」が原則

**Connection Draining**

- ヘルスチェック失敗時に処理中リクエストが終わるのを待ってから切り離す機能
- スケールイン時のリクエスト破棄を防止

### パターン全体像

| # | パターン名 | 対象システム | 主要AWSサービス |
|---|-----------|------------|--------------|
| 5-1 | ECサイト（オンプレミス連携なし） | ECサイト基本構成 | EC2, VPC, RDS, S3 |
| 5-2 | Web可用性向上パターン | ECサイト | ELB (ALB), Auto Scaling |
| 5-3 | コンテンツキャッシュパターン | ECサイト | CloudFront, S3 |
| 5-4 | DB可用性向上パターン | ECサイト | RDS Multi-AZ, リードレプリカ |

| 5-7 | ハイブリッド利用パターン | 基幹システム | VGW, DirectConnect, VPN |
| 5-8 | ファイルサーバ利用パターン | 基幹システム | Storage Gateway |

### パターン5-1: ECサイト（オンプレミス連携なし）

**構成概要**

```
インターネット
  └── CloudFront (CDN)
        └── ALB (ELB)
              ├── Web/APサーバ × 4 (2AZ分散, Auto Scaling)
              └── [Private Subnet]
                    ├── ElastiCache (CACHE)
                    ├── RDS Multi-AZ (M/S)
                    └── [Batch Private Subnet]
                          └── バッチサーバ → NATゲートウェイ → S3 (VPCエンドポイント経由)
```

**サブネット設計**
- パブリックサブネット × 2（異なるAZ）: Webサーバ配置用
- プライベートサブネット × 2（異なるAZ）: DB用
- プライベートサブネット × 1: バッチサーバ用

**基本原則**: 1システム1環境 = 1AWSアカウント

---

### パターン5-2: Web可用性向上パターン（ELB + Auto Scaling）

**課題**: ピーク時のサーバ数見積りが困難 → 過剰または不足が生じる

**解決**: ELB + Auto Scalingで需要に応じてEC2インスタンス数を自動増減

**Auto Scalingの主要設定**

| 設定 | 内容 |
|------|------|
| 希望台数 (Desired Capacity) | 初期台数 |
| 最小台数・最大台数 | スケール範囲の上下限 |
| ターミネーションポリシー | 削除優先順位（最古起動・最新・次課金タイミングが近いもの等） |

**Scalingポリシーの種類**

| 方式 | 適用場面 |
|------|---------|
| 手動スケーリング | 希望台数を手動変更 |
| オートヒーリング | 最小台数=希望台数で、最低台数を常時維持（障害時の自動補充） |
| スケジュールベース | 時間帯が予測できる場合（「17:45にN台」等） |
| ルールベース | 「CPU 70%超えで2台追加」等の動的条件。ピーク予測困難な場合に有効 |

**AMI提供方式の選択**

| 方式 | 特徴 |
|------|------|
| Golden AMI | 事前に必要設定済みのAMIを作成。起動が速い |
| ユーザーデータ | 起動時スクリプトでインストール・設定。AMI管理不要 |

> ⚠️ スケールアウト/インの閾値を同じ値にすると、インスタンス増減が繰り返される（ハンティング）リスクがある。スケールアウトとスケールインの閾値に差をつけること。

### パターン5-3: コンテンツキャッシュパターン（CloudFront）

**課題**: 静的コンテンツへのリクエストでサーバ負荷増大 + レスポンス遅延

**解決**: CloudFront（CDN）をALBの前段に配置して静的コンテンツをキャッシュ

**構成**

```
インターネット
  └── CloudFront Distribution
        ├── オリジン: ELB (動的コンテンツ)
        └── オリジン: S3バケット (静的コンテンツ + Sorryページ)
```

**CloudFront設定のポイント**

| 設定項目 | 内容 |
|---------|------|
| Origin Domain Name | S3エンドポイントまたはELBのDNS名 |
| Path Pattern | パスごとにキャッシュ設定を分けられる（例: `img/*`） |
| TTL (最小/最大/デフォルト) | デフォルト86400秒（1日） |
| Object Caching | Use Origin Cache Headers / Customize から選択 |

**S3バケットポリシー例（静的ホスティング）**

```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": "*",
    "Action": "s3:GetObject",
    "Resource": "arn:aws:s3:::バケット名/*"
  }]
}
```

**Sorryページ配信**: Webサーバが全台ダウン時に、S3から代替ページを配信するフォールバック構成

### パターン5-4: DB可用性向上パターン（RDS Multi-AZ）

**課題**: データベース障害による機会損失。Webサーバと異なりDBのスケールアウトは困難

**解決**: RDS Multi-AZ構成（マスター・スレーブ）

**構成**

```
[Private Subnet AZ-A]  [Private Subnet AZ-C]
  RDS Master (M)   ←→   RDS Slave (S)
  ElastiCache (CACHE)     ElastiCache (CACHE)
```

**RDSの可用性機能**

| 機能 | 内容 |
|------|------|
| Multi-AZ | 異なるAZにスタンバイを自動配置。フェイルオーバーは自動（エンドポイントDNSが切り替わる） |
| リードレプリカ | 読み取り負荷分散用。レプリケーション遅延に注意 |
| 自動バックアップ | スナップショット（日次）+ トランザクションログで特定時刻へのPoint-In-Time復元が可能 |

**スケールアップ戦略**: DBはスケールアップ（インスタンスタイプ変更）が基本。スケールアウトはリードレプリカで読み取り分散。

---

### パターン5-5: インメモリキャッシュパターン（ElastiCache）

**課題**: 頻繁に同じDBクエリが発行される → DB負荷増大。セッション情報のサーバ間共有問題

**解決**: ElastiCacheでクエリ結果とセッション情報をキャッシュ

**キャッシュ処理フロー**

```
Webサーバ
  ① ElastiCacheにキャッシュ確認
  ② ヒット → キャッシュからデータ取得（DBアクセス不要）
     ミス  → DBに問い合わせ → 結果をElastiCacheに保存 → 返却
```

**ElastiCacheエンジン比較**

| | Memcached | Redis |
|--|-----------|-------|
| スレッド | マルチスレッド | シングルスレッド |
| データ型 | 文字列のみ | 文字列・リスト・ビットマップ等 |
| 永続化 | なし（揮発性） | スナップショット・.rdbインポート/エクスポート |
| クラスタ構成 | CacheCluster + AutoDiscovery | リードレプリカ + 自動フェイルオーバー |
| 障害時の挙動 | ノード障害でそのキャッシュは失われる | プライマリ障害時にリードレプリカが自動昇格 |

**ElastiCacheの制限（EC2上のRedis/Memcachedと比較）**

- `config`・`migrate` コマンドが使用不可（Redis）
- パスワード設定不可 → アクセス制御はSGで行う

**導入ポイント**

- **キャッシュ対象の選定**: 「更新頻度が低く、アクセス頻度が高いデータ」が効果的（例: 商品マスター情報。在庫数は不向き）
- **インスタンスタイプ**: 小さめから始めて負荷テスト後に最適化する（ElastiCacheは高性能なため過剰スペックになりやすい）
- **耐障害性**: AZをまたがるリードレプリカ構成でAZ障害に備える。アプリはElastiCache障害時にDBフォールバックできるようエラーハンドリングが必要

### パターン5-6: ジョブサーバパターン（NAT GW + VPCエンドポイント）

**課題**: バッチサーバをプライベートサブネットに配置したいが、外部API・AWS APIを呼び出す必要がある

**アンチパターン**: バッチサーバをパブリックサブネットに配置 → セキュリティ設計として不適切

**解決**: NAT Gateway + VPCエンドポイントの組み合わせ

```
[プライベートサブネット]
  バッチサーバ
    ├── → NATゲートウェイ → インターネットゲートウェイ → 外部API
    └── → VPCエンドポイント → S3 (インターネット経由不要)
```

**NATゲートウェイ vs NATインスタンス**

| | NAT Gateway | NATインスタンス |
|--|------------|--------------|
| 冗長性 | AWSが管理・冗長構成済み | 自分で冗長構成が必要 |
| パフォーマンス | 高い | インスタンスタイプ依存 |
| 管理コスト | 低い | 高い（パッチ・監視等） |
| 推奨度 | 推奨 | レガシー |

**NATゲートウェイ作成手順**

1. パブリックサブネットを指定
2. Elastic IPを割り当て
3. プライベートサブネットのルートテーブルに `0.0.0.0/0 → NAT GW` を追加

**VPCエンドポイントとネットワークACL併用時の注意**

- VPCエンドポイントへの通信はS3/DynamoDBのグローバルIPアドレスにルーティングされる
- NACLで「グローバルIPとの通信を拒否」している場合、VPCエンドポイントも疎通不可になる
- 解決策: NACLではなくSGで制御する（プレフィックスリストIDを使用）か、ルーティングテーブルでIGW経路を除去

**可用性向上**: AZ障害時に対応するため、別AZに同一構成（バッチサーバ + NATゲートウェイ）を用意してフェイルオーバー設計する

### パターン5-7: ハイブリッド利用パターン（VPN/DirectConnect）

**前提**: エンタープライズでは全システムの一括移行が困難 → オンプレミスとAWSのハイブリッド利用

**ネットワーク接続方式の選択**

| 方式 | 適合用途 | メリット | デメリット |
|------|---------|---------|----------|
| インターネットVPN | 評価用途・接続VPC数少・コスト重視 | 安価・手軽 | 帯域不安定・遅延大 |
| DirectConnect | 安定スループット必要・長期利用・多数VPC接続 | 安定・低遅延・データ転送料安価 | 初期コスト・調達リードタイム |

**DirectConnect Gateway**: 1つのDirectConnectで複数のVPC・クロスリージョン接続が可能（2017年11月追加）

**移行パターン**

1. **システムの一部をAWSに移行**: DBのみRDSに移行し、オンプレミスWebサーバからVPN/DirectConnect経由でアクセス
2. **システム全体をAWSに移行**: WebAP層 + DB層をまとめて移行。ELB/Auto ScalingでランダムIPが付与されるため、Private Subnet（ランダムIP用）に配置

**サブネット構成（ハイブリッド時）**

| サブネット | ルートテーブル |
|-----------|-------------|
| Private Subnet（ランダムIP用） | オンプレミスNW → VGW、S3 → VPCエンドポイント |
| Private Subnet（固定IP用） | オンプレミスNW → VGW |
| Private Subnet（NATゲートウェイ接続用） | オンプレミスNW → VGW、0.0.0.0/0 → NAT GW |
| NAT Gateway Subnet | 0.0.0.0/0 → IGW |

> ⚠️ オンプレミスからELB/RDSに接続する際はFQDN名前解決が必要。イントラネット内でDNS疎通ができない環境では `amazonaws.com` の外部DNS連携が必要。

### パターン5-8: ファイルサーバ利用パターン（Storage Gateway）

**課題**: オンプレミスのファイルサーバ維持コスト削減。バックアップデータの安価な保管

**解決**: Storage Gateway（VMware/Hyper-V仮想アプライアンス または EC2 AMI として提供）

**Storage Gatewayの3タイプ**

| タイプ | プライマリ保存先 | クライアント接続 | S3 APIアクセス | 主な用途 |
|--------|--------------|--------------|--------------|---------|
| ファイルゲートウェイ | S3 | NFS v3/v4.1 | 可 | S3をNFSマウントして使うファイルサーバ代替 |
| ボリュームゲートウェイ（キャッシュ型） | S3（ローカルキャッシュあり） | iSCSI | 不可 | データ量増大に対応。ローカルキャッシュで高速アクセス |
| ボリュームゲートウェイ（保管型） | ローカルディスク（S3はスナップショット） | iSCSI | 不可 | バックアップが主目的。アクセス速度低下なし |
| テープゲートウェイ | ローカルディスク → S3/Glacier | iSCSI (VTL) | 不可 | 物理テープの代替。既存バックアップアプリと連携可 |

**データ移行時のStorage Gateway活用手順**

```
① 保管型ボリュームゲートウェイでS3にデータ転送
② S3スナップショットからEBSを作成
③ EC2インスタンスにEBSをマウントして移行データを利用
```

**ファイルゲートウェイの注意点**: データ読み書き速度がローカルディスクより遅い。高速アクセスが要件の場合は事前の性能テストが必要。

### パターン5-9: 大規模データ分析パターン（Redshift + BI）

**課題**: オンプレミスでは初期投資が大きくて実現困難だった大規模データ分析

**解決**: Redshift（列指向カラム型DWH）+ BIツール（Tableau/QuickSight）

**Redshiftの構成**

```
SQLクライアント / BIツール (JDBC/ODBC)
  └── Redshiftクラスター
        ├── Leaderノード (クエリ受付・分散)
        ├── Computeノード 1 (並列処理)
        ├── Computeノード 2
        └── Computeノード N
```

**Redshiftの特徴**

- PostgreSQL互換（JDBC/ODBC接続可）
- 数百GB〜数PB規模のデータを並列処理
- ノードタイプはdc2.large（SSD）から選択。後から変更可能

**分析パターンの構成**

```
オンプレミス
  └── Storage Gateway → S3 (分析データ格納)

AWS VPC
  ├── Private Subnet: Redshiftクラスター (VPCエンドポイント経由でS3アクセス)
  ├── Private Subnet: BIツールインスタンス (Tableau等)
  ├── Private Subnet: Storage Gateway (NATゲートウェイ経由)
  └── NAT Gateway Subnet: NATゲートウェイ → IGW
```

**導入手順**

1. VPC内にRedshiftクラスター作成
2. Storage Gateway経由でS3にサンプルデータをアップロード
3. JDBC/ODBC接続設定
4. Redshiftにテーブル作成・データロード
5. BIツール（Tableau等）から接続・分析

## インフラ自動構築

### 3サービスの比較

| サービス | 対象レイヤー | 難易度 | 主な用途 |
|---------|------------|--------|---------|
| Elastic Beanstalk | アプリ層 + インフラ（定番構成のみ） | 低 | 新規プロダクトのスモールスタート。トライアンドエラー重視 |
| CloudFormation | AWSリソース全体（OSより下） | 高 | 大規模企業システム。複雑な構成を自由に設計したい場合 |
| OpsWorks | OS〜ミドルウェア層（Chef） | 中 | Chefレシピ資産がある場合。CloudFormationとの組み合わせ |

**使い分けの判断**
- **CloudFormation推奨**: 大規模システム・本番環境・マルチ環境構築（開発/テスト/本番 × リージョン）
- **Beanstalk推奨**: AWSメンバーが少ない・スモールスタート・素早いリリース優先
- **OpsWorks**: 既存Chef資産の再利用が可能な場合

### CloudFormation

**基本概念**

- **テンプレート**: JSON または YAML形式のインフラ定義ファイル
- **スタック**: テンプレートから作成されたAWSリソースの集合。スタック単位で一括作成・削除が可能
- **リソース依存関係**: CloudFormationが自動判断して構築順序を決定

**テンプレートの主要セクション**

```yaml
Parameters:        # 実行時に利用者が選択する値の定義
  InstanceType:
    Type: String
    Default: t2.medium
    AllowedValues: [t2.medium, t2.large]

Mappings:         # Map形式の変数定義（例: リージョン別AMI IDなど）
  RegionMap:
    ap-northeast-1:
      hvm: "ami-xxxxxxxx"

Resources:        # 構築するAWSリソースの定義（必須セクション）
  SampleVPC:
    Type: 'AWS::EC2::VPC'
    Properties:
      CidrBlock: '192.168.0.0/16'
```

**主要な組み込み関数**

| 関数 | 用途 |
|------|------|
| `!Ref` | パラメータやリソースの値を参照（動的に決まるIDなど） |
| `Fn::GetAtt` | リソースの属性値を取得 |
| `Fn::Select` | リストから値を選択（AZ選択等） |
| `Fn::FindInMap` | Mappingsから値を取得 |

**CloudFormation採用の推奨理由**

1. 同一環境を複数構築する際のコスト回収が早い（3環境以上で効果的）
2. 別リージョンへの展開が容易（DR/海外展開）
3. テンプレートがGit管理できる → 構成管理が常に最新状態を維持

### Elastic Beanstalk のデプロイ方式

| 方式 | 特徴 | 向いている環境 |
|------|------|-------------|
| All at Once | 全インスタンスに一括デプロイ。一時停止あり | 開発環境 |
| Rolling | 段階的にデプロイ。縮退構成あり | テスト環境 |
| Rolling with additional batch | 新インスタンスを追加してからデプロイ。縮退なし | ステージング |
| Immutable | 全インスタンスを新規作成してデプロイ | 本番（切り替えリスク最小） |
| URL Swap / Route 53切り替え | 新環境で事前確認後、DNSを切り替え | 本番（最もリスク少） |

## 移行テクニック

### データ移行にかかる時間の目安

| 回線速度 | データ量 | 移行時間（約・使用率80%） |
|---------|---------|----------------------|
| 100Mbps | 100GB | 1時間30分 |
| 100Mbps | 10TB | 7日半 |
| 1Gbps | 10TB | 17時間 |
| 1Gbps | 1PB | 74日 |

> テラバイト規模以上ではネットワーク移行が現実的でない場合も。物理デバイス（Snowball）の利用を検討する。

### データ移行方式の選択

| データ量 | 推奨方式 |
|---------|---------|
| 数GB以下・停止可能 | OSコマンド（tar/rsync/mysqldump） |
| 数十GB以下 | AWS CLI + S3 sync |
| 数TB以下（時間余裕あり） | Storage Gateway |
| テラバイト〜ペタバイト級 | Snowball / Snowball Edge |
| 100PB超 | Snowmobile（セミトレーラトラック） |

### OSコマンドによるデータ移行

**tarによる一括移行**

```bash
# 圧縮
cd /path/from && tar zcvf data_migration.tar.gz ./data/
# 解凍
tar zxvf data_migration.tar.gz -C /path/to
```

**rsyncによる差分同期（本番切り替えまで定期実行で差分を最小化）**

```bash
nohup rsync -auzh -e "ssh -i /home/user/.ssh/id_rsa" \
  --delete --progress --partial \
  --append /path/from/migration/ user@移行先IP:/path/to/migration/ &
```

### AWS CLI + S3 によるデータ移行

```bash
# S3への同期
aws s3 sync /path/from/migration/ s3://my-bucket/path/to/migration/ --delete
```

**S3の制約**

- 1オブジェクトの最大サイズ: 5TB
- 1回のPUT操作: 最大5GB

**マルチパートアップロード（大容量ファイル向け）**

```bash
# 1. ファイル分割（split コマンド等）
# 2. アップロード開始
aws s3api create-multipart-upload --bucket my-bucket --key large-file.tar.gz
# 3. 各パートをアップロード
aws s3api upload-part ...
# 4. アップロード完了・結合
aws s3api complete-multipart-upload ...
```

**CLIのS3パフォーマンス設定** (`~/.aws/config`)

```ini
[default]
s3 =
  max_concurrent_requests = 20     # デフォルト: 10
  max_queue_size = 10000           # デフォルト: 1000
  multipart_threshold = 64MB       # デフォルト: 8MB
  multipart_chunksize = 16MB       # デフォルト: 8MB
```

### Snowball / Snowball Edge / Snowmobile

| サービス | 容量 | 特徴 |
|---------|------|------|
| Snowball | 80TB | 専用クライアントツール。データをS3に転送後 |
| Snowball Edge | 100TB（クラスタ構成でPB級） | S3 API/NFS対応。40Gbps光ファイバー。Lambda前処理可能 |
| Snowmobile | 100PB/台 | セミトレーラトラックで輸送 |

**Snowball利用フロー**

```
AWSから機材配送 → CLIでデータ書き込み → 機材返送 → AWS内高速回線でS3転送
```

### 仮想サーバ移行（AWS Server Migration Service）

**AWS Server Migration Service (SMS)** — VMware上の仮想マシンをAWSに移行するサービス

**VM Import/Export との比較**

| | SMS | VM Import/Export |
|--|-----|-----------------|
| 対象 | VMware仮想サーバ | OVA/VMDK/VHD/RAW形式イメージ |
| 差分転送 | 可能（スケジューリング対応） | 不可 |
| 操作インターフェイス | GUI（マネジメントコンソール） | CUI |
| イメージ変換 | SMS が自動実施 | S3→AMI変換が必要 |

**SMS移行手順**

```
① オンプレミス準備:
   - vCenterにAWS移行用ユーザー作成
   - VMwareホストに仮想アプライアンス(SMC)をセットアップ
   - SMCの初期設定（接続情報・ユーザー情報）

② AWS準備:
   - IAMロール/ユーザーの作成と権限付与
   - マネジメントコンソールからSMCに接続して仮想サーバ情報を収集
   - コピー設定（実行タイミング・頻度）を定義
```

**差分転送機能の活用**: 事前にイメージをAMI化しておき、切り替え時に最新の差分だけを転送 → ダウンタイムの最小化

**OSライセンス（BYOL）**: 既存ライセンスをAWS上で継続利用するBYOL（Bring Your Own License）が可能。OSやライセンス種別によっては非対応の場合があるため事前確認が必要。

### データベース移行（DMS）

**AWS Database Migration Service (DMS)** — データベースをAWSに移行するサービス。同一RDBMS間・異種RDBMS間の両方に対応

**接続経路の選択**

| 接続経路 | 備考 |
|---------|------|
| インターネット経由 | 双方グローバルIPが必要。SSL証明書による暗号化が必要 |
| VPN接続 | 時間的余裕がある場合に現実的な手法 |
| DirectConnect | DB移行専用敷設はコスト高 |

**DMSの移行対象範囲**

| 移行対象（DMS） | 移行対象外（要SCT/手動） |
|--------------|----------------------|
| テーブル定義 | View |
| Primary Key | Procedure |
| Trigger | Index |
| データ | 各種制約 |

**異種RDBMS間移行 (Oracle → Aurora等) のフロー**

```
① SCT (Schema Conversion Tool) でスキーマを変換
   - カラム型の差異を吸収（例: Oracle Number型 → MySQL Double/INT型）
   - View・Procedure・Trigger・Index・制約の変換候補を提示
② DMS でデータを移行
③ トランザクションログ（REDO/BINログ）による継続的差分同期
④ 切り替え時のダウンタイムを極力短縮
```

**DMS利用の前提条件**: 移行元または移行先のどちらかがAWS上に存在すること（オンプレミス間には使用不可）

**差分同期設定**: 移行元DBでトランザクションログ出力を有効化することで、初回移行後の差分をDMSが継続的に同期可能。切り替え直前の差分を最小化できる。
