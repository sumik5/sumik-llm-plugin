# Bedrock GenAIスケーリングリファレンス

GenAIアプリケーションを本番環境でスケールさせるための5柱フレームワーク。
パフォーマンス最適化の実装詳細は [GENAI-ARCHITECTURE.md](GENAI-ARCHITECTURE.md) を参照。

---

## スケーリング5柱フレームワーク

| 柱 | 主な目的 | 代表的な手法 |
|----|---------|------------|
| **Reliability（信頼性）** | ステートレス設計・非同期化 | SQSデカップリング、EDA、Lambda/ECS |
| **Performance（性能）** | スループット最大化 | 水平スケーリング、コンピュート選択 |
| **Cost（コスト管理）** | トークンコスト最適化 | LLMカスケード、プロンプト圧縮、予算管理 |
| **Observability（可観測性）** | 多層監視 | CloudWatch、プロンプトテレメトリ |
| **DR（障害設計）** | 障害の予測・回復 | Circuit Breaker、クロスリージョン、Route 53 |

## Reliability（信頼性）

### ステートレスアーキテクチャ

Bedrock の `InvokeModel` / `InvokeModelWithResponseStream` はステートレス設計。
各リクエストは完全に独立しており、サーバーサイドにセッション情報を持たない。

**アーキテクチャ原則**:
- コンピュート層（Lambda/ECS）にセッション状態を持たせない
- 状態は専用のマネージドサービスに外部化する

| 状態の種類 | 推奨サービス |
|-----------|------------|
| 会話履歴 | Amazon DynamoDB（低レイテンシKV） |
| 短期セッションキャッシュ | Amazon ElastiCache for Redis |
| エージェント会話管理 | Amazon Bedrock Session Management API |

ステートレス設計により「セッションアフィニティ（特定ユーザーを同じサーバーに固定）」が不要になり、
コンピュート層を水平スケールに対応させることができる。

### 非同期デカップリング（SQSパターン）

即時応答が不要なワークロードには、API Gateway + SQS + Lambda のパターンを採用する。

```
クライアント → API Gateway → SQS キュー → Lambda → Bedrock → 結果保存（S3/DynamoDB）
                              ↑
                         HTTP 202 即時返却
```

**フロー詳細**:
1. API Gateway がリクエストを SQS に直接エンキュー
2. HTTP 202（Accepted）をクライアントに即時返却
3. Lambda が SQS ポーリングしてバックグラウンド処理
4. 処理結果を S3 または DynamoDB に保存

**利点**:
- **トラフィックスパイクの吸収**: SQS がショックアブソーバーとして機能
- **独立スケーリング**: Lambdaの並行数を調整してBedrockへのスループットを制御
- **耐障害性**: 処理失敗時はDLQに移動、メッセージロストなし

### マイクロサービス分解

GenAIアプリケーションを機能別に分離することで、各コンポーネントを独立してスケールできる。

| サービス | 責務 |
|---------|------|
| **Inference Service** | Bedrockへの全呼び出し、プロンプト整形、モデル選択 |
| **Data Ingestion Service** | RAG用ドキュメント処理（チャンキング、ベクトル化、インデックス） |
| **Prompt Management Service** | プロンプトテンプレートのバージョン管理・動的取得 |
| **User Feedback Service** | モデル品質評価データの収集・保存 |

Inference Serviceをカプセル化することで、モデル変更が他サービスに影響しない。

### イベント駆動アーキテクチャ（EDA）

RAGパイプラインのような多段ワークフローには、EventBridge を使ったEDAが適している。

```
DocumentUploaded
    ↓ (EventBridge)
Chunking Service
    → ChunkingComplete
        ↓
    Vectorization Service
        → VectorizationComplete
            ↓
        Indexing Service
```

EDA により、新機能（例: 処理完了通知）の追加は既存サービスを変更せず新しいConsumerをデプロイするだけで実現できる。

### Lambda vs ECS の選択基準

| 次元 | AWS Lambda | Amazon ECS（Fargate） |
|------|-----------|----------------------|
| スケーリングモデル | リクエスト毎の自動スケール、ゼロからスタート | タスク追加/削除、メトリクスベース |
| コストモデル | リクエスト数×実行時間（ミリ秒単位） | 割り当てvCPU/メモリ（秒単位） |
| 最大実行時間 | 15分 | 制限なし |
| メモリ/CPU | 最大10GB / CPU比例割当 / GPU非対応 | 最大120vCPU・244GiB / GPUあり |
| 推奨用途 | イベント駆動・スパイク負荷・短時間処理 | 長時間処理・大規模ML依存・高負荷定常 |

**ハイブリッドパターン**: ユーザーAPIは Lambda、長時間ドキュメント処理は Lambda が ECS/Fargate に委任する。

## Cost（コスト管理）

> Provisioned Throughput・バッチ推論・料金テーブルの詳細は [GENAI-ARCHITECTURE.md](GENAI-ARCHITECTURE.md) を参照。

### トークンプロファイル分析

コスト最適化の出発点は「入出力トークン比率（token profile）」の把握。

- **Input-heavy（RAGアプリ）**: 大量のコンテキストをプロンプトに含む → 入力トークンコストが支配的
- **Output-heavy（創作支援）**: 短い指示から長文を生成 → 出力トークンコストが支配的

「どのモデルが最安か」ではなく「自アプリのトークンプロファイルに合う料金体系はどれか」が正しい問いかけ。

### スマートモデルルーター（LLMカスケード）

複雑さに応じてモデルを動的に選択するパターン。全リクエストを高コストモデルに送らない。

```
ユーザーリクエスト
    ↓
Haiku（安価・高速）でリクエスト複雑度を判定
    ├─ 単純クエリ → Haiku が直接応答
    └─ 複雑クエリ → Sonnet/Opus にルーティング
```

**注意点**:
- ルーティングロジック自体の評価コスト（ベンチマーク・人間レビュー）が発生する
- ルーター構築・評価・維持コストがルーティング節約額を上回らないか事前試算すること

### プロンプト圧縮

**直接圧縮**: 冗長な表現を削除する。同じ意図を半分のトークンで表現できる。

```
悪い例: "Hello Claude, I was hoping you could take a look at the following
         text and provide a summary. Please make it about three sentences long,"
良い例: "Summarize the following text in three sentences:"
```

**自動圧縮**: ユーザーの冗長な入力を小型LLMで事前圧縮してから大型モデルに渡す。数百万リクエスト規模では数千ドルの削減につながる。

### セマンティックキャッシング

プロンプトキャッシング（完全一致）の限界を超えるために、意味的類似度でキャッシュヒットを判定する。

```
ユーザークエリ
    ↓
Titan Embeddings でベクトル変換
    ↓
ElastiCache for Redis でベクトル近傍探索
    ├─ キャッシュヒット（類似度閾値以上）→ 保存済みレスポンスを返却
    └─ キャッシュミス → Bedrock 呼び出し → レスポンスとベクトルをキャッシュに保存
```

> 注: Bedrockネイティブのプロンプトキャッシング（cache_control）とRedisキャッシュの実装は
> [GENAI-ARCHITECTURE.md](GENAI-ARCHITECTURE.md) を参照。

**コスト効果の条件**: ElastiCacheの時間課金 + Embeddingコスト < Bedrockキャッシュヒット節約額。キャッシュヒット率が十分高い場合のみ有効。

### ストリーミングによるコスト制御

ストリーミングは単なるUX向上ではなく、**アクティブなコスト管理手段**でもある。

```python
# ストリームをリアルタイム解析し、有効な応答が完成したら早期クローズ
response = bedrock.invoke_model_with_response_stream(...)
for event in response['body']:
    chunk = process_chunk(event)
    if is_complete_response(chunk):
        break  # 不要なトークン生成を中断 → コスト削減
```

有効なJSONが形成された、または特定のストップシーケンスが検出されたタイミングで接続をクローズすることで、
余剰トークンの生成費用を防ぐ。

### AWS Cost Explorerによる分析

**Bedrockコストを詳細分析する手順**:
1. AWS Cost Explorer コンソール → Service: `Bedrock` でフィルタ
2. Group by: `Usage Type` に変更（モデル別・トークン種別に分解される）
3. Usage Type 例: `anthropic.claude-3-5-sonnet-20240620-v1:0-InputTokens`

**マルチテナント/マルチプロジェクトのコスト配賦**:
- Application Inference Profiles を使ってモデルにタグを付与
- モデルARNの代わりにプロファイルARNで呼び出す
- タグが自動的にUsageデータに付与されCost Explorerに反映される

### AWS Budgets によるプロアクティブ管理

```
Bedrockサービスの月次予算設定
    ↓
実績が予算の80%超過 → SNS → Slack通知
予測が予算の100%超過 → SNS → Slack通知
    ↓
アクション: キャッシュ層追加 / モデルダウングレード など
```

事後の請求確認ではなく、予算超過前に対処できる体制を構築する。

## Observability（可観測性）

GenAIの監視はインフラの稼働確認だけでは不十分。非決定的な出力品質を監視する多層アプローチが必要。

### 3層可観測性モデル

| 層 | 何を監視するか | 答える問い |
|----|--------------|----------|
| **モデルレベル** | Bedrockとの直接インタラクション | システムは動いているか |
| **パイプラインレベル** | 周辺AWSインフラ | 処理フローは健全か |
| **ユーザーフィードバック** | エンドユーザーの評価 | 出力品質は十分か |

### モデルレベルメトリクス

Amazon Bedrock は CloudWatch に以下のメトリクスをネイティブ出力する。

| メトリクス | 説明 | 活用例 |
|-----------|------|-------|
| `Invocations` | モデル呼び出し総数 | スパイク検出・不正使用検知 |
| `InputTokenCount` | 入力トークン数 | コスト帰属・プロンプト肥大化検出 |
| `OutputTokenCount` | 出力トークン数 | コスト帰属・出力長監視 |
| `InvocationLatency` | モデル応答時間 | UX劣化・非効率プロンプト検出 |
| エラーレート | 呼び出し失敗数 | サービス信頼性の基本指標 |

### パイプラインレベルメトリクス

Bedrock呼び出しは単独では発生しない。周辺インフラも同時に監視する。

**Lambda**:
- `InitDuration`（CloudWatch LogsのREPORTログ）: コールドスタートのレイテンシ影響を追跡
- `ConcurrentExecutions`, `Errors`, `Duration`: 基本的なオーケストレーション健全性

**SQS / EventBridge**:
- `ApproximateNumberOfMessagesVisible`（SQS）: キュー積滞 → Lambdaが処理できていない兆候

**スロットリングアラーム**:
- ログに `ThrottlingException` が出現したらアラームを発火
- クォータ上限の80%でServiceQuotasへの増加申請を促すアラートを設定

### ユーザーフィードバック収集

技術メトリクスは「システムが動いているか」しか答えない。「良いか」はユーザーが判断する。

| フィードバック種類 | 実装方法 | データの価値 |
|----------------|---------|-----------|
| **明示的フィードバック** | 👍/👎ボタン、自由記述コメント | 品質の即時バイナリシグナル |
| **暗黙的フィードバック** | ユーザーが応答を編集した「変更前後」のペア | ファインチューニング用の高品質データセット |
| **差分フィードバック** | A/Bテスト: 2つの応答を提示して選択させる | モデル/プロンプト比較の根拠データ |

### CloudWatch監視の構築手順

1. **Invocation Logging 有効化**: Bedrockコンソール設定 → CloudWatch Log Groupに詳細ログ（トークン数・モデルID・プロンプト）が自動送信
2. **メトリクスダッシュボード**: `AWS/Bedrock` ネームスペースの `Invocations`, `InputTokenCount`, `OutputTokenCount`, `InvocationLatency` をグラフ化
3. **コールドスタート追跡**: Logs Insights でREPORTログの `initDuration` フィールドを抽出
4. **ThrottlingExceptionアラーム**: メトリクスフィルターで ThrottlingException を計測し、閾値超過でSNS通知

### 監視ツール比較

| 機能 | Amazon CloudWatch | Datadog LLM Observability | Prometheus + Grafana |
|------|-----------------|--------------------------|---------------------|
| 主な用途 | AWSインフラ・基本モデル監視 | LLMアプリのE2E品質・セキュリティ監視 | 汎用オープンソースメトリクス |
| セットアップ | ネイティブ統合、設定最小 | SDKインテグレーション必要 | サーバー・Grafanaの自己管理が必要 |
| AI特化機能 | レイテンシ/トークン/エラー基本監視 | エージェントトレーシング、ハルシネーション検出、プロンプトインジェクション監視 | AI特化機能なし、カスタム計装が必要 |
| コスト | メトリクス/ログ/アラーム毎従量課金 | ホスト/データ量サブスクリプション | インフラコストのみ（OSS） |

### プロンプトレベルテレメトリ

高品質なGenAIシステムは、APIコールごとに詳細メタデータをログに記録する。

**記録すべき情報**:
- **プロンプトバリアント追跡**: `version_id: v3.2` など使用テンプレートをタグ付け → どのバージョンがエラーを多発するか特定
- **トークン数トレンド**: エンドポイントごとの平均入出力トークン数を経時追跡 → 徐々に増加していれば「プロンプト肥大化」の兆候
- **品質メタデータ**: `is_truncated`, `quality_score`, `flagged_for_review` を応答に付与 → 劣化パターンを特定
- **request_id**: 問題報告時に正確なプロンプト・パラメータ・応答を再現できる

このテレメトリがシステムの継続的改善とファインチューニングデータセット構築の基盤になる。

## DR（障害設計）

### GenAI特有の障害分類

従来ソフトウェアと異なり、GenAIシステムは「技術的に動作しているが品質的に失敗している」状態が存在する。

**ハード障害（従来型）**:
- APIタイムアウト、`ThrottlingException`
- プロバイダーレベルの停止（大規模ネットワーク障害など）

**ソフト障害・サイレント障害（GenAI特有）**:
| 障害種別 | 説明 |
|---------|------|
| **ハルシネーション** | 事実に反する情報を生成。HTTP 200 が返るため監視で検出不可 |
| **性能崩壊** | 既知タスクで突然精度が激減。入力の微細な変化で発生 |
| **行動ドリフト** | モデル更新後に出力形式・トーンが変化。構造依存アプリが静かに壊れる |

ソフト障害はリージョン切り替えでは解決しない。同じモデルバージョンがセカンダリにも存在すれば同じ障害が再発する。**モデルレベルのフォールバック**（異なるモデルへの切り替え）が必要。

### リトライ戦略

**指数バックオフ**:
- 1回目失敗後: 1秒待機 → 2回目失敗後: 2秒 → 3回目: 4秒 → ...
- 過負荷の下流サービスに回復時間を与える

**ジッターの重要性**:
指数バックオフだけでは、多数のクライアントが同時障害→同期的にリトライ→再び過負荷（Thundering Herd問題）が発生する。
バックオフ時間にランダム値を加えることで、リトライのタイミングを分散する。

```python
from botocore.config import Config

retry_config = Config(
    retries={
        'max_attempts': 10,
        'mode': 'adaptive'  # 指数バックオフ+ジッター+クライアントサイドレート制限
    },
    connect_timeout=5,   # 接続タイムアウト
    read_timeout=60      # ストリーミング応答用の読み取りタイムアウト
)

bedrock_runtime = boto3.client(
    'bedrock-runtime',
    config=retry_config,
    region_name='us-east-1'
)
```

`adaptive` モードはAWS SDKに組み込まれており、カスタム実装不要。

### サーキットブレーカーパターン

リトライが有効なのは一時的障害のみ。サービスが完全に停止している場合、リトライし続けると
クライアントリソースを無駄に消費し、障害中のサービス回復を妨げる。

**状態遷移**:
```
[Closed] ─失敗閾値超過→ [Open] ─タイムアウト→ [Half-Open]
   ↑ リセット ─成功────────────────────────────────┘

Closed:     正常状態。リクエストを通過させながら失敗数をカウント
Open:       リクエストを即時拒否（フォールバックを返す）。タイムアウト後に Half-Open に遷移
Half-Open:  試験リクエストを通過。成功すれば Closed に戻る、失敗すれば Open に戻る
```

**サーバーレス環境での実装**:
Lambdaのメモリはインスタンス間で共有されないため、サーキット状態は外部ストアに永続化が必要。

```
Step Functions（Express Workflow）
    ↓ サーキット状態を確認
DynamoDB（CircuitBreakerState テーブル）
    ├─ Circuit Closed → Lambda → Bedrock 呼び出し
    │       成功: FailureCountをリセット
    │       失敗: FailureCountをインクリメント → 閾値超過でOpen
    └─ Circuit Open → 即時FailFastエラーを返却
```

DynamoDB の TTL 機能で `OpenUntil` タイムスタンプを管理すると、タイムアウト後に自動的に Half-Open に遷移するシンプルな実装が可能。

### フォールバック戦略

| 戦略 | 説明 | 適用場面 |
|------|------|---------|
| **モデルフォールバック** | プライマリ失敗時に別モデルに切り替え（Opus→Sonnet、Anthropic→Cohere） | プロバイダー障害・セマンティック障害 |
| **キャッシュフォールバック** | リアルタイム生成失敗時に過去の類似応答を返却 | リアルタイム性が高くないクエリ |
| **グレースフルデグラデーション** | AI生成不可時に静的メッセージを返す | すべての障害の最終手段 |

グレースフルデグラデーションの例:
```
"AI アシスタントは現在、高負荷状態です。数分後に再度お試しください。"
```
ユーザーに生の `500 Internal Server Error` を見せるよりも信頼感を維持できる。

### クロスリージョン推論

Bedrock ネイティブ機能。モデルIDの代わりに **Inference Profile ID** を指定することで有効化。

```python
# 通常のモデルID
modelId = 'anthropic.claude-3-sonnet-20240229-v1:0'

# Inference Profile ID（複数リージョンへの自動ルーティング）
modelId = 'us.anthropic.claude-3-sonnet-20240229-v1'
```

**動作**: ソースリージョンで容量制約やパフォーマンス問題を検出した場合、
AWSが自動的に同一ジオグラフィ内の別リージョンにルーティング。
データ転送はAWSグローバルネットワーク経由で追加コストなし。

**制限**: アプリケーションレベルのDRではない。ソースリージョンのコンピュート層自体が
障害になった場合は、別途 Route 53 フェイルオーバーが必要。

### Route 53 DNSレベルフェイルオーバー

完全なマルチリージョンDR（Warm Standby / Active-Active）の制御プレーン。

**構成**: Route 53 ヘルスチェック + フェイルオーバールーティングポリシー

- Primary と Secondary の2つのリソースレコードセットを同一ドメイン名で作成
- Primaryのヘルスチェック失敗 → Route 53が自動的にSecondaryのIPを応答

**ヘルスチェック推奨**: CloudWatchアラーム監視（Lambdaエラーレート・レイテンシ等の複合指標）

**レジリエンスの多層化**:
```
Layer 1（リクエストレベル）: リトライ + ジッター
Layer 2（コンポーネントレベル）: サーキットブレーカー
Layer 3（サービスレベル）: クロスリージョン推論（Inference Profile）
Layer 4（リージョンレベル）: Route 53 DNSフェイルオーバー
```

### 障害シナリオ別クイックリファレンス

| 問題/障害シナリオ | 推奨パターン | 主要AWSサービス | 実装複雑度 |
|----------------|------------|----------------|-----------|
| 散発的なAPIタイムアウト・スロットリング | リトライ（指数バックオフ＋ジッター） | AWS SDK（Boto3）、Lambda | 低 |
| Bedrockモデルが数分間完全無応答 | サーキットブレーカー | Step Functions、DynamoDB | 中 |
| プライマリモデルがハルシネーション発生中 | モデルフォールバック | Lambda、Amazon Bedrock | 中 |
| AWSリージョン全体の障害 | マルチリージョン（Warm Standby / Active-Active）＋DNSフェイルオーバー | Route 53、API Gateway | 高 |
| 反復クエリの高レイテンシ・高コスト | Read-Asideキャッシング | DynamoDB（TTL）、Lambda | 低 |
| 単一リージョンの容量超過 | クロスリージョン推論 | Bedrock（Inference Profile） | 低 |

## UX（ユーザー体験）

### レイテンシ許容度の設計

GenAIのレスポンスレイテンシは予測困難。UXはそれを前提に設計する。

- **ストリーミング必須**: 全体の完了を待つより、最初のトークンが届く時点（TTFT）でユーザーは「動いている」と感じる
  → ストリーミング実装でユーザー体感レイテンシを50-80%削減（詳細は [GENAI-ARCHITECTURE.md](GENAI-ARCHITECTURE.md)）
- **プログレスインジケーター**: 処理中状態を可視化。「処理中…」よりもリアルタイムストリームが望ましい
- **タイムアウト許容値の設計**: UX目標に合わせてタイムアウト値（connect_timeout/read_timeout）を決定

### エラーハンドリングのUXガイドライン

| 状況 | 悪い対応 | 良い対応 |
|------|---------|---------|
| スロットリング | `ThrottlingException: Rate exceeded` を表示 | 「応答が遅延しています。自動的に再試行中です」 |
| サーキット Open | `CircuitOpenError` を表示 | 「現在、AIサービスが高負荷です。しばらくお待ちください」 |
| 全フォールバック枯渇 | 500エラー | 「AI機能は現在利用できません。チームが対応中です」+ 代替フローの提示 |

エラーメッセージはユーザーの信頼を維持しながら、状況と代替手段を明確に伝える。

### ユーザーフィードバックループの構築

品質監視のためのフィードバック機構をUX設計に組み込む。

```
AIレスポンス表示
    ├─ 👍 / 👎（明示的フィードバック）
    ├─ 編集機能（暗黙的フィードバック）→ 「変更前後」ペアをキャプチャ
    └─ 「別の回答を表示」（差分フィードバック）→ A/Bでモデル比較
```

蓄積されたフィードバックはプロンプト改善・モデル選択・ファインチューニングデータセット構築に活用する。
