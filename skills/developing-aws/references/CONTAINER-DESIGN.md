## コンテナ技術の概要

### 仮想マシン vs コンテナ

| 比較軸 | 仮想マシン | コンテナ |
|--------|-----------|----------|
| 分離単位 | ハードウェアレベル（仮想マシン） | OSプロセスレベル |
| カーネル | ゲストOSがそれぞれ占有 | ホストOSのカーネルを共有 |
| 起動速度 | 分単位 | 秒単位 |
| リソース効率 | 低い（ゲストOS分のオーバーヘッド） | 高い（プロセス単位の最小構成） |
| ポータビリティ | 低い（OS依存が残る） | 高い（依存関係をイメージ内に閉じ込め） |

**コンテナ利用のメリット（WHY）**:

- **環境依存からの解放**: ランタイム・ライブラリをイメージ内に閉じ込め、環境差異を排除
- **再現性とポータビリティ**: 一度ビルドしたイメージはどの環境でも同一動作
- **リソース効率と起動速度**: ゲストOS不要で軽量、かつプロセス起動のみで高速デプロイ

### Dockerの基本概念

| 用語 | 説明 |
|------|------|
| **Dockerfile** | イメージを構築するためのテキスト定義ファイル |
| **イメージ** | コンテナ実行に必要なビルド済みパッケージ |
| **タグ** | イメージに割り当てるラベル（主にバージョン管理用） |
| **レジストリ** | イメージを保管・配布するサービス |
| **コンテナ** | イメージから生成された実行主体（アプリ+依存関係） |

**Build, Ship, Run ワークフロー**:
```
Dockerfile + ソースコード → (docker image build) → イメージ
→ (docker image push) → レジストリ
→ (docker image pull + docker container run) → コンテナ
```

### コンテナオーケストレータの役割

複数コンテナのプロダクション運用で必要になる管理機能:

| 機能 | 内容 |
|------|------|
| **配置管理** | ホスト負荷に応じた自動分散・障害時の再配置 |
| **負荷分散** | 複数コンテナへのリクエスト振り分け |
| **状態監視と自動復旧** | 異常検知・切り離し・コンテナ数維持 |
| **ローリングデプロイ** | 稼働を維持しながら新旧コンテナを自動入れ替え |

## AWSコンテナサービス選定

### コントロールプレーン（オーケストレータ）

| サービス | 特徴 | 選定基準 |
|---------|------|---------|
| **Amazon ECS** | AWSネイティブ独自オーケストレータ。AWSサービスとの親和性が高い。月間SLA 99.99% | AWSサービスを中心とした構成、運用コスト重視、小〜中規模 |
| **Amazon EKS** | Kubernetesマネージドサービス。コントロールプレーンをAWSが管理。月間SLA 99.99% | Kubernetes既存資産の活用、マルチクラウド移植性重視、大規模 |

### データプレーン（コンテナ実行環境）

| サービス | 特徴 | トレードオフ |
|---------|------|------------|
| **AWS Fargate** | サーバーレスコンピューティングエンジン。ホスト管理不要 | EC2より料金高め。ただしTCO（運用人件費含む）では有利。OS介入不可 |
| **Amazon EC2** | 仮想マシン。カーネルパラメータ等を詳細チューニング可能 | OSのセキュリティパッチ・キャパシティ管理が利用者責務 |

### コンテナアーキテクチャ4パターン

| パターン | コスト | 拡張性 | 信頼性・サポート | エンジニア確保 |
|---------|--------|--------|-----------------|--------------|
| **ECS on Fargate** | 利用料高め。運用コスト大幅削減（TCO有利） | デプロイ遅め（ENI生成+イメージキャッシュ不可）。水平スケール容易 | ECS純正のため手厚いAWSサポート | ECS/Fargateを知る人材は増加中 |
| **ECS on EC2** | 利用料抑えられる。EC2運用コストが高い | デプロイ速い（イメージキャッシュ活用）。リソース拡張が容易 | EC2障害はSSH/SSMでダイレクト調査可能 | 枯れた技術。有識者が多い |
| **EKS on Fargate** | Fargate料金 + EKSクラスター料金 | Fargate制約あり（DaemonSet不可等） | Kubernetesの複雑性。AWSサポート範囲限定 | Kubernetes人材が必要 |
| **EKS on EC2** | EC2利用料。Kubernetes管理コスト高 | 最高の柔軟性（GPU・DaemonSet対応） | EKSコントロールプレーンはAWS管理 | Kubernetes有識者が必要 |

**新規コンテナ構築の推奨**: まず **ECS on Fargate** を検討する。ホスト管理から解放されビジネスに集中でき、PCI DSS等コンプライアンス準拠にも有利。

### その他コンテナ関連サービス

| サービス | 用途 |
|---------|------|
| **Amazon ECR** | フルマネージドなプライベートコンテナレジストリ。ECSとの連携が容易。イメージはS3に保存（イレブンナイン耐久性） |
| **AWS App Runner** | WebアプリをECRイメージやGitHubソースから即デプロイ。インフラ設計不要で開発者が手軽に利用 |
| **AWS Lambda** | コンテナを意識せずコードに集中できるサーバーレス。Firecracker上で動作。実行時間制限あり |

## ECS on Fargateアーキテクチャ

### 主要コンセプト階層

```
ECSクラスター (Cluster)
  └─ ECSサービス (Service)  ← タスク数維持・ELB連携・デプロイ制御
       └─ ECSタスク (Task)  ← コンテナの実行単位（Fargateマイクロ仮想マシン）
            ├─ コンテナ (アプリ)
            ├─ コンテナ (サイドカー: FireLens, X-Ray等)
            └─ ENI (タスクごとに独自のElastic Network Interface)
```

### タスク定義の重要パラメータ

| パラメータ | 説明 | 設計ポイント |
|-----------|------|------------|
| `cpu` / `memory` | タスクレベルのリソース割り当て | コンテナの合計+空き分を考慮。コンテナイメージ取得からタスク終了まで課金 |
| `networkMode` | `awsvpc`固定（Fargate） | タスクごとに独自ENIでプライベートIP割り当て |
| `taskRoleArn` | タスク内コンテナがAWSリソースにアクセスするためのIAMロール | 最小権限原則を遵守 |
| `executionRoleArn` | ECSエージェントがECRからイメージ取得・CloudWatch Logsに書き込むためのロール | タスクロールと混同しないこと |
| `logConfiguration` | ログドライバー（`awslogs` or `awsfirelens`） | コンテナごとにログ出力先は1つのみ |

### ECSタスク起動時の動き（Fargate）

```
ECSサービス → タスク起動指示
  → ENI生成・アタッチ（ここに時間がかかる）
  → ECRからコンテナイメージをダウンロード（キャッシュなし）
  → コンテナ起動
```

**Fargateの制約事項**:
- タスクに割り当てられるエフェメラルストレージ上限: 200GB（変更不可。永続化にはEFS利用）
- 最大リソース割り当て: 4vCPU / 30GBメモリ
- カーネルパラメータのチューニングは不可（ホストOSはAWS管理）

## Well-Architected × コンテナ設計

### 運用設計（運用上の優秀性）

#### オブザーバビリティの3要素

| 要素 | 目的 | AWSサービス |
|------|------|------------|
| **ログ** | 「なぜ」を突き止める（定性的なイベント記録） | CloudWatch Logs / FireLens |
| **メトリクス** | 「何が起きているか」を把握（定量的な計測情報） | CloudWatch Metrics / Container Insights |
| **トレース** | 「どこで」を特定（サービス間リクエストの追跡） | AWS X-Ray |

#### ログ設計

**CloudWatch Logs（シンプル構成向け）**:
- ECSタスク定義で `awslogs` ログドライバーを指定するだけで収集
- サブスクリプションフィルターでエラーを抽出→Lambda→SNS通知が容易
- ログ取り込み量に応じた課金（200GB/月で約$152）

**FireLens（本番推奨）**:
- Fluent Bitコンテナをサイドカー構成でECSタスクに追加
- CloudWatch LogsとS3への同時出力が可能（運用監視 + 長期保管）
- データ分析基盤（Redshift・OpenSearch Service）への転送も容易
- ログドライバー制約（1コンテナ1ドライバー）をFluent Bitで回避

| 設計パターン | 適用場面 |
|------------|---------|
| awslogs のみ | シンプルなアプリ、少量ログ、ログ分析不要 |
| FireLens + S3 + CloudWatch | 監査ログ長期保管 + 障害通知が両方必要 |
| FireLens + Firehose + Redshift/OpenSearch | データ分析基盤への取り込みが必要 |

#### メトリクス設計

**基本CloudWatchメトリクス**（デフォルト取得）:
- `CPUUtilization` / `MemoryUtilization`（ECSサービス単位、1分間隔）

**CloudWatch Container Insights**（オプトインが必要、追加課金）:
- ECSタスク単位のCPU・メモリ・ネットワーク・ストレージメトリクス
- `TaskCount` / `DeploymentCount` 等の運用監視メトリクス
- タスクレベルの詳細分析が可能（特定タスクの負荷偏り検知等）

#### トレース設計（X-Ray）

3つの設計ポイント:
1. **サイドカー構成**: X-RayコンテナをECSタスク定義に同梱
2. **IAMロール**: タスクロールに `AWSXRayDaemonWriteAccess` を付与（タスク実行ロールではない）
3. **ネットワーク経路**: プライベートサブネットからX-Rayエンドポイントへ、VPCエンドポイントまたはNATゲートウェイが必要

#### CI/CD設計

**AWSマネージドCI/CDサービス（Codeシリーズ）**:

| サービス | 役割 |
|---------|------|
| **CodeCommit** | マネージドGitリポジトリ（IAM統合でセキュア） |
| **CodeBuild** | ビルド・テスト・コンテナイメージ作成をYAMLで定義 |
| **CodeDeploy** | ECSへのBlue/Greenデプロイ（ダウンタイムなし切り替え） |
| **CodePipeline** | ソース→ビルド→デプロイをパイプライン化。承認アクション等も組み込み可能 |

**マルチ環境CI/CD設計**:
- CodeCommitを共有リソースとして集約（環境ごとにブランチ分離）
- CodeBuild・CodeDeployは環境ごとに分離（ビルド内容・デプロイ対象が異なるため）
- ステージング環境でビルド済みイメージをプロダクション環境で再利用（品質担保）

**ECRイメージタグ管理**:
- タグプレフィックスで環境識別（例: `stg_2be3a41`, `prod_20210101`）
- コミットIDをタグに含めることでデプロイ済みソースバージョンを追跡可能
- ライフサイクルポリシーをタグプレフィックスごとに設定してプロダクション用イメージを保護

### セキュリティ設計

#### ECS/Fargateの責任共有モデル

| レイヤ | AWS責務 | 利用者責務 |
|-------|---------|-----------|
| **ホストOS（Fargate）** | OSパッチ適用・セキュリティ維持 | 対策不要 |
| **コンテナランタイム** | ECSコントロールプレーン管理 | 対策不要 |
| **レジストリTLS接続** | HTTPS通信保護 | 対策不要 |
| **コンテナイメージ** | — | 脆弱性スキャン・マルウェア対策・秘密情報の管理 |
| **ネットワーク（VPC）** | — | セキュリティグループ・NACL設計 |
| **IAM** | — | 最小権限原則・アクセス制限 |
| **アプリケーション** | — | 脆弱性対応・ルートファイルシステムの読み取り専用化 |

#### NIST SP800-190対応マトリクス（ECS/Fargate）

| 観点 | 主な対策項目 | 対策手段 |
|------|------------|---------|
| **イメージ** | 脆弱性・マルウェア・設定不備 | ECRプッシュ時スキャン、Trivy（CI/CD組み込み）、Dockle（設定チェック） |
| **イメージ** | 平文秘密情報の埋め込み | Secrets Managerで実行時注入、イメージへのハードコーディング禁止 |
| **イメージ** | 信頼できないイメージの使用 | ECRプライベートリポジトリ利用、パブリック化禁止ポリシー（SCP）設定 |
| **レジストリ** | 古いイメージの残存 | ECRライフサイクルポリシー（世代数・期間ルール）、タグIMMUTABLE設定 |
| **レジストリ** | 認証・認可の不備 | リポジトリポリシーでCodeBuildのみpush許可、ECSタスク実行ロールのみpull許可 |
| **オーケストレータ** | 無制限の管理アクセス | IAM最小権限原則、ECSクラスター単位のアクセス制限 |
| **コンテナ** | 無制限ネットワークアクセス | セキュリティグループ、VPCエンドポイント（プライベート通信）、WAF |
| **コンテナ** | アプリケーション脆弱性 | ルートファイルシステムの読み取り専用化（タスク定義で設定） |

#### 継続的なイメージスキャン設計

```
CodePipeline
  → CodeBuild（ビルド + Trivyスキャン実行）
  → スキャン結果に問題あり → パイプライン停止（デプロイ阻止）
  → スキャン問題なし → ECRプッシュ（プッシュ時スキャンも実行）
  → CloudWatch Events（定期スキャン結果通知）
  → Lambda → SNS（脆弱性アラート通知）
```

### 信頼性設計

#### マルチAZ構成

ECS/Fargateのマルチ可用性設計の基本:
- ECSサービスを複数AZにタスクを分散配置（ECSサービスの設定で制御）
- ALBを複数AZのサブネットに配置し、クロスゾーン負荷分散を有効化
- Auroraはマルチ-AZ構成（プライマリ+レプリカを異なるAZに配置）
- 各AZ障害時においても同等のパフォーマンスを維持するタスク数設計が必要

#### 障害時の切り離しと自動復旧フロー

```
ECSタスク異常発生
  → ALBヘルスチェック失敗
  → ターゲットグループからECSタスクを登録解除（切り離し）
  → ECSサービスが新しいタスクを起動（自動復旧）
  → CloudWatch Alarmsでタスク停止イベントを検知
  → SNS/Chatbotで運用担当者に通知
```

**SIGTERM ハンドリング（安全なタスク停止）**:
- ECSがタスク停止前にSIGTERMシグナルを送信
- アプリケーション側でSIGTERMをハンドリングし、処理中リクエストを完了してから停止
- Fargate Spot中断時（AWSによる強制停止）でも同様に2分前に通知される

#### サービスクォータへの考慮

| クォータ | デフォルト値 | 注意点 |
|---------|-----------|-------|
| クラスター数 | 10,000 | — |
| クラスターあたりのサービス数 | 5,000 | — |
| Fargateオンデマンドタスク数（リージョン×アカウント） | **1,000** | スパイク時にAuto Scalingが上限に到達するリスク |
| Fargate Spotタスク数（リージョン×アカウント） | **1,000** | 同上 |

**クォータ対策**:
- 上限緩和リクエストはAWSサポートへ（自動引き上げなし）
- CloudWatchメトリクスでタスク数使用率を継続的に監視
- タスク数が80%を超えたらアラート設定（Trusted Advisorは対象外のため自前実装）

### パフォーマンス設計

#### 適切なリソース設計の流れ

```
Step 1: ビジネス上のパフォーマンス要件の把握
  → リクエスト数/秒、ピーク倍率、AZ障害時の維持基準等

Step 2-1: ECSタスクのリソース割り当て（CPU/メモリ仮決め）
  → 余裕を持って初期設定。テスト結果で見直し

Step 2-2: スケール戦略の選択（以下の3種から選択）

Step 3: 負荷テスト実施

Step 4: CloudWatchメトリクスで結果確認

Step 5: リソース割り当てとスケール戦略を見直し（繰り返し）
```

#### スケール戦略3種

| 戦略 | 説明 | 推奨場面 |
|------|------|---------|
| **スケールアップ** | タスクのCPU/メモリを増やす。タスク再起動が必要 | 一時的なリソース不足の根本解決時 |
| **ステップスケーリングポリシー** | CloudWatchアラーム条件でタスク数をステップ的に増減 | 閾値ベースで段階的にスケールしたい場合 |
| **ターゲット追跡スケーリングポリシー** | 指定したメトリクス目標値（例: CPU使用率70%）を維持するよう自動調整 | シンプルで最もおすすめ。Auto Scaling設定が容易 |

**スケールアウト推奨の理由（WHY）**:
- スケールアップはFargate最大リソース（4vCPU/30GB）の上限に当たりやすい
- スケールアウトは稼働中タスクの停止が不要（可用性を維持）
- Auto Scalingとの組み合わせで自動化が容易
- パフォーマンス効率だけでなく可用性と耐障害性が向上

**注意点**: Fargate上のすべてのタスクにENIが割り当てられる。VPCのサブネットCIDR設計でIP枯渇にも注意。

### コスト最適化設計

#### ECS/Fargate料金の基本構造

- **課金単位**: タスクに割り当てたvCPU・メモリ × 稼働時間（秒単位、最低1分）
- **課金開始**: コンテナイメージ取得時点〜タスク終了まで
- **VPCエンドポイント**: ECRからのイメージダウンロード量もデータ処理料金として計上

#### コスト削減手段一覧

| 手段 | 効果 | 適用条件 |
|------|------|---------|
| **Compute Savings Plans** | 最大47%割引（3年・全額前払い）。1年でも22%削減 | 安定したワークロード。1年または3年コミット |
| **Fargate Spot** | 約68%割引（オンデマンド比） | タスク中断が許容できる環境（開発・ステージング） |
| **開発環境のタスク停止** | 夜間・休日のタスク稼働ゼロ | 24時間稼働不要な開発・ステージング環境 |
| **コンテナイメージ最小化** | データ転送料削減 + 起動時間短縮 | 全環境。alpineベースイメージ・マルチステージビルド活用 |
| **ECRライフサイクルポリシー** | 不要イメージの保管コスト削減 | 常時適用推奨 |
| **タスクリソース最適化** | 過剰割り当てを排除 | 負荷テスト後に設定値を見直し |

#### Compute Savings Plansの活用指針

| コミット期間 | 支払い方法 | 割引率 |
|------------|---------|-------|
| 1年間 | 前払いなし | 15% |
| 1年間 | 一部前払い | 20% |
| 1年間 | 全額前払い | 22% |
| 3年間 | 前払いなし | 40% |
| 3年間 | 一部前払い | 45% |
| 3年間 | 全額前払い | 47% |

コミット量を超過した分は通常料金が適用されるため、安定稼働タスク数を把握した上でコミット量を決定する。

#### Fargate Spotの設計パターン

キャパシティプロバイダ戦略でオンデマンドとSpotを混在させる:

```
キャパシティプロバイダ戦略の例:
  FARGATE: ベース=2, ウェイト=1  ← 最低2タスクはオンデマンドで確保
  FARGATE_SPOT: ベース=0, ウェイト=3 ← それ以上はSpotを優先

→ 6タスク起動時: FARGATE×2(ベース) + FARGATE×1 + FARGATE_SPOT×3
```

**Spot中断時の注意**: AWSがタスクを停止する2分前にSIGTERMが送信される。アプリケーション側でのハンドリング実装が必須。プロダクション環境では中断が許容できないためオンデマンドを推奨。

#### ECRライフサイクルポリシー設計

マルチ環境でのイメージ競合を防ぐためタグプレフィックスで分離:

```
ライフサイクルポリシー1 (優先度高):
  タグプレフィックス [prod] → 過去10世代を保持

ライフサイクルポリシー2 (優先度低):
  タグプレフィックス [stg] → 過去3世代を保持
```

これによりステージング環境でのビルドが、プロダクション参照イメージを削除するリスクを排除できる。
