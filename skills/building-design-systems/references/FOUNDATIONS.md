# デザインシステムの基礎知識

デザインシステム（DS）の定義・歴史・構成要素・用語階層・導入判断・失敗パターンを体系的にまとめたリファレンス。
INSTRUCTIONS.mdで参照される詳細版。

---

## デザインシステムの定義

**デザインシステムとは、デジタルプロダクトの目的を達成するために、一貫性をもって組織化された「相互接続されたパターンの集合」と「共有されたプラクティスの集合」である。**

| 構成要素 | 内容 |
|---------|------|
| **パターン** | インターフェースを構成する繰り返し要素（ユーザーフロー、インタラクション、ボタン、テキストフィールド、アイコン、カラー、タイポグラフィ、マイクロコピー） |
| **プラクティス** | パターンをどのように作り、収集し、共有し、使用するか（特にチームで協働する場合の方法） |

### 効果的なデザインシステムの2条件

1. **デザインプロセスのコスト効率** — パターンを再発明せず、既存のものを効率的に使い回せる
2. **プロダクト目的に対するUXの効率と満足度** — インターフェースがプロダクトの目的達成を助ける

## 歴史的背景とDSの必要性

### デザインにおける「システム」の起源

| 時代 | 動向 |
|------|------|
| 1919〜33年 | バウハウス運動: "形は機能に従う"。比率・グリッド・カラー理論を重視。装飾より機能的シンプルさ |
| 1950〜60年代 | スイス・スタイル（国際タイポグラフィ）: グリッドをさらに発展。非対称・クリーンなサンセリフ書体 |
| 1990年代 | WWWの台頭。CSSなし→テーブルレイアウト→Flash。Webデザインの実験時代 |
| 2000年代前半 | CSS普及・JavaScriptの発展。ページベースからコンポーネントベース設計へ移行 |
| 2001年〜 | Agileの誕生。高速デリバリーが求められ、コンポーネント単位の設計・管理の必要性が高まる |

### なぜ今デザインシステムが必要か

コンポーネントベースのアーキテクチャとAgile開発の組み合わせにより、デザイン作業は「ページ全体」から「部品」に焦点が移った。その結果、全体像を見失いやすく、一貫性の維持が困難になった。デザインシステムは：

- プロダクトの目標と駆動ビジョンの**必要不可欠な概観**を提供
- デザイン・エンジニアリング・プロダクト間の**共通言語**を定義
- インターフェースの**反復的なスナップショット**として機能するウェイファインディングツール

## デザインシステムと関連用語の違い

### 3つの主要概念の比較

| 用語 | 定義 | 特徴 |
|------|------|------|
| **スタイルガイド** | ブランドをインターフェース要素にスタイリスティックに適用する方法を定義した静的ドキュメント | カラー・タイポグラフィ・アイコノグラフィ等の高レベル詳細。多くは静的 |
| **コンポーネントライブラリ** | チーム間で共有・使用できるスタイルとコンポーネントのセット | アプリ全体で使われる共通UI要素。デザインツール同期が可能な場合もある。ライブコードを含む場合と含まない場合がある |
| **デザインシステム** | デザインと実装の両方のガイドラインを含む、ドキュメント化された要素・コンポーネント・リージョンのシリーズ | ライブコード例を含み、複数プロダクトへの再利用を可能にする。デザイン原則・ルール・ガイドラインを含む |

> **重要**: コンポーネントライブラリとパターンライブラリという用語は業界でしばしば混同されて使われる。組織内で一貫した用語定義を持つことが大切。

### DS関連リソースの完全な用語階層

```
デザインシステム（上位概念）
├── コンテンツ系
│   ├── エディトリアルスタイルガイド
│   │   └── トーン・ワード選択・文法・句読点の標準
│   └── ボイス&トーンガイドライン
│
├── ビジュアル系
│   ├── ブランドガイド / ブランドキット
│   │   └── ロゴ・カラーパレット・タイポグラフィ・写真ガイドライン
│   └── デザインガイドライン / ビジュアル言語
│       └── デザイン思想・コンセプト・デザイン原則
│
├── コード系
│   ├── フロントエンドスタイルガイド
│   │   └── コードスニペット・デザインアセット付きのパターン集
│   ├── リビングスタイルガイド（Living Style Guide）
│   │   └── 本番環境と同期するガイド。変更が即時に反映
│   ├── コードスタイルガイド
│   │   └── インデント・命名規則等のコード品質標準
│   └── コンポーネントライブラリ / UIライブラリ
│       └── フロントエンドコードで実装されたUIコンポーネント集
│
└── コンテンツ生産支援
    └── テンプレート / CMS
        └── 既存デザイン知識なしでコンテンツを量産するためのボイラープレート
```

> **パターンライブラリとデザインシステムの違い**: パターンライブラリはデザインパターンを収集・保存・共有するツール。デザインシステムはその背景にある哲学・方法論・プラクティス全体を指す。パターンライブラリはシステムを強力にするツールだが、システムそのものではない。

## 6構成要素モデル

デザインシステムを構成する6つの相互連動する領域：

| 構成要素 | 内容 | 例 |
|---------|------|-----|
| **Layout** | スペーシングとグリッドシステムを構成する定義された寸法 | 8px グリッド、スペーシングスケール |
| **Styles** | ビジュアル言語のコアアスペクト | カラー・アイコノグラフィ・タイポグラフィ |
| **Components** | インターフェースのコア要素 | ボタン・フォームフィールド |
| **Regions** | プロダクト全体に使われる大きなデザインパラダイム | ナビゲーション・検索エリア |
| **Content** | ボイス&トーン・句読点ガイドライン・用語集 | マイクロコピー・エラーメッセージ |
| **Usability** | アクセシビリティと国際化のルール | WCAG基準・i18n対応 |

これらは互いに依存し、積み上がる関係にある：
- コンポーネント ＝ レイアウト + スタイルの組み合わせ
- リージョン ＝ 複数コンポーネントの集合
- コンテンツ ＝ コンポーネント内のコピーに影響
- ユーザビリティ ＝ コンテンツをアクセシブルに保つ

## 要素の階層構造

インターフェースを構成する単位の階層：

```
リージョン（Region）
    └── 複数のコンポーネントグループから構成される大きなパラダイム
        例: ナビゲーションバー全体、検索エリア全体

コンポーネントグループ（Component Group）
    └── 複数のコンポーネントが集まり大きなコンポーネントを形成
        例: ボタングループ（セグメントコントロール）、入力フィールド+ボタン（フォーム）

コンポーネント（Component）
    └── 1つ以上のエレメントが組み合わさって全体として機能する
        例: フォーム、カード、ドロップダウン

エレメント（Element）
    └── 最低レベルのオブジェクト。これ以上分解できない
        例: ラベル、アイコン、テキスト
```

## 導入タイミングの判断基準

### 3つの主要判断軸

| 判断軸 | 推奨しない状況 | 推奨する状況 |
|--------|-------------|-------------|
| **組織の年齢** | 早期スタートアップ: 全てが流動的で、厳格なガイドラインは時間の無駄になる | 製品が安定し、ピボットの頻度が下がった後 |
| **チームサイズ** | デザイナー・エンジニアが数名以下: 時間が不足しがち | 5名以上に成長した、または成長が見込まれる |
| **作業量と種類** | — | 複数プロダクト・複数チーム・高頻度の繰り返し作業がある場合 |

### デザインシステムを使うべき状況

以下の複数の状況に当てはまるなら、DSを導入する価値がある：

- 会話が非効率または不可能なほど多くの関係者がいる
- タイムゾーン・時短・パートタイム等の会話の障壁がある
- 別チームへのハンドオーバーが発生する（代理店→社内チーム等）
- 新入社員のオンボーディングを迅速化したい
- 多くのコンテンツ寄稿者（ライター・画像担当・動画編集）がいる
- DSを構築する時間・スキル・リソースがある
- UIパターンの名称が決まっておらず混乱を招いている
- 一貫したデザインの信頼できる単一情報源（Single Source of Truth）が欲しい

> **小規模組織への注意**: 少人数チームであれば、新しいデザイン要素についての10分の会話の方が、全員が文書を書いて読む時間より速い場合もある。完全なDSが本当に必要か見極めること。

## DS失敗パターン5つ

デザインシステムが失敗する主な理由を理解しておくことは、成功と同じくらい重要。

### ❶ 初期の賛同不足（Lack of Initial Buy-In）

**症状**: 組織のサポートなしにDSの構築を試みる

DSには多くのリソースと時間が必要。全組織レベルでの理解と支持なしに始めると、デザイン専用の取り組みとして見なされ、必要なリソースを失いやすい。

**対策**:
- プロダクト管理・エンジニアリング・営業・マーケティング・エグゼクティブに各自への具体的なメリットを伝える
- 問題解決の視点から会話を始める: 「このシステムで解決できる痛点は何か?」

### ❷ 過度・過速な取り組み（Trying to Do Too Much, Too Soon）

**症状**: 興奮に任せて全てを一度に構築しようとする

組織のユニークなニーズと課題を理解する前に全時間とリソースを投入すると、燃え尽き症候群のリスクがある。

**対策**: ペースを守り、本当に必要なものを理解してからコミットする。多大な成果の後に何もしない状況は、ステークホルダーとチームの信頼を失う。

### ❸ 完璧主義（Perfectionism）

**症状**: 全ての細部にこだわり、システムを完成させようとする

**対策**: 改善事項の実行リストを維持しつつ、すぐに方向転換しないこと。他のプロジェクトと同様に計画・実行・イテレーションを繰り返す。「完成する」ことは決してないという考えに慣れること。

### ❹ メンテナンス不足（Maintenance）

**症状**: 静的な画像・繰り返しのテキストなど、スケールしない要素でシステムを構築する

スケールするにつれてより複雑になるシステムに対して、定期的な更新が追いつかなくなる。

**対策**: 自己文書化（Self-documentation）を取り入れる。システムが成長・複雑化しても関連性を保てる構造を最初から設計する。

### ❺ 他社DSへの羨望（Design System Envy）

**症状**: 著名組織の洗練されたDSと自分たちのDSを比較して意欲を失う

**対策**: 多くの有名DSは、フルタイムの専任チームが数年かけて構築したもの。組織のニーズに集中し、自分たちの進歩を喜ぶこと。他社DSはインスピレーションや内容分類の参考として使うのが賢明。
