# 実装・運用リファレンス

デザインシステムの実装から運用・維持まで、実践的なガイドラインを提供するリファレンス。

---

## 実装前の組織評価

### 評価すべき3つの軸

| 軸 | 検討事項 |
|----|---------|
| **組織タイプ** | 単一プロダクト vs 複数クライアント（エージェンシー） |
| **ステークホルダーの支持** | 強い支持あり / なし（段階的に構築が必要） |
| **チームサイズ** | 小規模（時間確保が課題）/ 大規模（蓄積した不整合への対処が課題） |

### 組織タイプ別の始め方

**単一プロダクトの場合**:
1. まずレイアウトとスタイルを定義する
2. これが全コンポーネントの基盤になる

**エージェンシー（複数クライアント）の場合**:
1. 全プロダクトで使用される中核コンポーネントを特定する
2. CSS変数・デザイントークンを使ったテーマ対応レイアウトシステムを構築する
3. プロダクトごとにカスタマイズ可能なスタイル層（テーミング）を設計する

### 支持なしで始める方法

ステークホルダーの支持がなくても実装を始められる。

1. 現状把握から着手する（インベントリ作成）
2. 要素定義と規則文書化を少しずつ進める
3. 速度・品質向上のデータを蓄積してビジネスケースを構築する

## インターフェースインベントリ

既存プロダクトの現状を把握するための体系的な棚卸し手法。

### 実施手順

1. **キャプチャ**: 製品内の全UIエレメント・コンポーネントのスクリーンショットを取得する（全インスタンスではなく、ユニークなデザインを記録）
2. **カテゴリ分類**: タイポグラフィ・入力・ボタン・ラジオボタン等を種別ごとにグループ化する
3. **重複の特定**: 同じ目的を持つ異なるデザインを並べて不整合を可視化する
4. **インベントリの活用**: コンポーネントライブラリ構築の起点とする

**新規プロダクトの場合**: インベントリは不要。設計段階から系統的に構築する。

---

## デザインシステムの6カテゴリ構成

システムを以下6つの相互連携するカテゴリに分解する。

| カテゴリ | 内容 |
|---------|------|
| **Layout** | スペーシングシステム・グリッドシステム |
| **Styles** | カラー・アイコノグラフィ・タイポグラフィ |
| **Components** | ボタン・フォームフィールド等の中核UIエレメント |
| **Regions** | ナビゲーション・検索等、複数コンポーネントで構成される領域パターン |
| **Content** | ボイス＆トーン・句読点・プロダクト固有の用語集 |
| **Usability** | アクセシビリティ・国際化規則 |

### カテゴリ構成のカードソート

新規コンテンツの分類に迷ったとき、カードソートで予測可能なアーキテクチャを決定する。

**手順**:
1. 既存・追加予定のコンポーネントをカード化する（30〜50枚）
2. 7〜10名の参加者にカードを自由にグループ分けさせる
3. 同じカードが同じグループに入った頻度を分析する
4. 分析結果をもとにカテゴリ名・構造を調整する

## レイアウト設計

### スペーシングシステム

- **基本単位**: 8pxを基準とした数学的級数（8, 16, 24, 32, 48...）が多様な解像度に対応しやすい
- **代替案**: 選択肢を絞りたい場合は幾何級数（8, 16, 32, 64...）も有効
- padding（内部余白）と margin（外部余白）の使い分けを定義する

### グリッドシステム

| グリッドタイプ | 特性 | 適用場面 |
|--------------|------|---------|
| **カラムグリッド** | 不連続コンテンツを均等分割・レスポンシブ対応 | 一般的なWebページ |
| **ベースライングリッド** | テキストを水平グリッドに整列・可読性向上 | テキスト中心のアプリ |
| **測定システム** | 高密度情報の厳格なスペーシング制御 | 複雑なWebアプリ |

ベースライングリッドの基本値はスペーシングシステムの派生値とする（例: 8px基準なら4px）。

## スタイル設計

### タイポグラフィシステム

**2種類のタイポグラフィを分けて定義する**:

| 種類 | 用途 | 特性 |
|------|------|------|
| **UIタイポグラフィ** | ボタン・ラベル・ドロップダウン等 | line-heightはスペーシングシステムの倍数 |
| **長文タイポグラフィ** | 本文・説明文 | より広いline-heightで可読性向上 |

### カラーシステム

- アクセシビリティ最優先: WCAGコントラスト比基準を満たす組み合わせのみ採用する
- カラーパレットは段階（primary, secondary, tertiary）で管理する
- 多数の色を使う場合は優先度番号付きシステム（フォントウェイトの概念を応用）を実装する

### アイコノグラフィ

- アイコングリッドとコアシェイプを確立し、比率・視覚的一貫性を維持する
- チーム全員がエクスポート・実装方法を理解できるようドキュメント化する
- 検索・コピー可能なアイコンビューアーの実装を検討する

## 技術実装

### シングルソースオブトゥルース

- デザインシステムをプロダクトの**依存関係**として位置づける
- HTML・CSS・JavaScriptのコンポーネント実装をデザインシステムと共有する
- コード変更が自動的にデザインシステムとプロダクト双方に反映される仕組みを構築する

### インクリメンタル実装

既存プロダクトへの導入は段階的に行う。

| 方法 | メリット | デメリット |
|------|---------|-----------|
| **製品全体で一括置換** | 一貫性を即座に確保 | 大規模プロダクトでは負荷が高い |
| **ページごとに順次置換** | リスクを分散 | 移行中に不整合が発生する |

**推奨**: 両方を組み合わせる。シンプルなコンポーネントは全体一括、複雑なシステム（タイポグラフィ）はページごと。

### バージョン管理

- バージョン管理システムを使ってコンポーネントの変更履歴を追跡する
- コンポーネントのステータス（upcoming / in-progress / stable）を文書化する
- 新規追加されたコンポーネントには「new」タグを付ける

### ドキュメント

各コンポーネントの文書は以下を含める:
- **What**: このコンポーネントとは何か
- **When**: いつ使うべきか
- **Where**: どこで使うべきか
- **Why**: なぜこの設計なのか
- クラス名・コードスニペット
- ライブコードサンプル（静的画像ではなく）

## 目的指向インベントリ（パターン体系化）

視覚的整合性ではなく**目的（行動）**によってパターンをグループ化する手法。

### 従来のインベントリとの違い

| | 従来のインベントリ | 目的指向インベントリ |
|--|------------------|-------------------|
| グループ化の基準 | 外観（同じボタンをまとめる等） | ユーザー行動（「本を予約する」等） |
| 目標 | 視覚的整合性の確保 | 本質的なパターン定義と相互理解 |
| 参加者 | デザイナー中心 | 多職種（デザイナー・開発・PM・コンテンツ）|

### 実施手順

**準備**:
1. プロダクトの根幹となる10〜12画面を特定する（これなしでは製品が成立しない画面）
2. 各画面を2部印刷する（1部: ユーザージャーニー順で壁に貼る / 1部: 切り取り用）
3. 4〜8名の多職種チームを編成する

**ステップ1: 主要行動の特定**

各画面セグメントで支援すべきユーザー行動を洗い出す。

例（図書館サイト）:
- 発見（Discovery）: 読みたい本を見つける
- カタログ（Catalog）: 特定の本を検索する
- ウィッシュリスト（Wish list）: 候補リストを管理する

**行動の命名における注意**: 「Promotion」（ビジネス視点）ではなく「Discovery」（ユーザー視点）とする。ユーザーにとっても価値がある言葉を選ぶ。

行動を具体的なアクションに細分化する:
- 本のリストをスキャンする
- リストの絞り込み条件を変更する
- 本の詳細を確認する

**ステップ2: 目的別グループ化**

1つの行動を取り上げ、全画面からその行動を支援する要素を見つける。
切り取った2部目から関連要素を切り出し、グループ化してラベルを付ける。
グループ内の要素は**同じ粒度**になるよう揃える（モジュールと単体ボタンを混在させない）。

**ステップ3: パターン定義**

グループ化した要素を「統合する」か「分離する」かを判断する。

### 特異性スケール（Specificity Scale）

パターン定義の粒度を決める際の考え方。

| 粒度 | 特性 | トレードオフ |
|------|------|------------|
| **高特異性**（Specific） | コンテキストに最適化 | 再利用性低い・保守コスト高 |
| **低特異性**（Generic） | 再利用しやすい | 汎用的すぎてデザインが平凡になる |

**判断の問い**: 「このモジュールを変更したとき、他の要素も同様に変更されるべきか？」
- Yes → 統合してパターン化
- No → 分離して独立したパターン、またはバリアントとして定義

### コンテンツ構造の分析

パターンを統合・分離する際の判断ツール。

1. そのモジュールが機能するために必要な必須コンテンツスロットをリストアップする
2. オプション要素を特定する（画像は必須か？ラベルは常に必要か？）
3. 要素の階層とグループ化を決定する
4. スケッチで構造を可視化する

**同じ構造を共有する要素 → 統合候補**
**構造が異なる要素 → 分離またはバリアント**

### バリアント（Variants）

同一パターンのコンテキスト別変形。

- **デフォルト**: コアスタイルを持つ基本パターン
- **バリアント**: 追加スタイルを持つ派生パターン

どの特徴がコアで、どの特徴がバリアント固有かを明確にする。バリアント側の変更がコアに影響するか予測できるようになる。

### パターン命名の原則

- 名前だけでパターンの用途が伝わるものが理想
- 迷ったときは特異性の高い名前から始める（後で汎用化可能）
- コンテンツタイプだけでなく、目的を反映した名前にする
- デザイナーと開発者が協調して命名を決定する

例: 特定領域のみで使う → 「Course tabs」（限定名）/ 他でも使えると判断したら → 「Page tabs」（汎用名）

## ボタンとリンクの一貫性設計

システム内でボタンとリンクを一貫して定義するためのガイド。

### CTA（Call to Action）による分類

従来の「ボタン vs リンク」の技術的区分ではなく、CTAとリンクという目的的区分を採用する。

| 要素 | 定義 | 実装 |
|------|------|------|
| **CTA ボタン** | 現在ページ内でのアクション | `<button>` |
| **CTA リンク** | 現在コンテキストからの離脱アクション | `<a>` |
| **標準リンク** | 追加情報へのオプションパス | `<a>` |

### 視覚的階層の設計

| スタイル | 意味 | 使用条件 |
|--------|------|---------|
| Primary | 最も重要なアクション | 1画面に1つを原則 |
| Secondary | 必要だが補助的なアクション | 複数配置可能 |
| Ghost / Subtle | オプション・頻度の低いアクション | Primary の隣 |

### 特殊ケースの扱い

ブランドの核心機能（「予約」「完了マーク」等）は一般ルールの例外として特別扱いを許可する。ただし特殊ケースは少数にとどめ、一般ルールの存在を損なわないようにする。

## パターンライブラリの構築

### コンテンツ優先アプローチ

ツール選定よりコンテンツ定義を先行させる。まずGoogleドキュメント等のシンプルなツールでパターンを書き下ろし、チームが参照できるMVPライブラリとして機能させる。

### パターンの組織化方法

| 方法 | 特性 | 適用 |
|------|------|------|
| **アルファベット順** | 分類議論が不要・シンプル | 少〜中規模のパターン数 |
| **階層順（Atomic Design）** | 粒度による整理（Atom/Molecule/Organism） | 大規模・構造重視のシステム |
| **目的別** | チームの行動指針と一致 | 目的指向設計を実践するチーム |
| **カードソート結果** | チームのメンタルモデルに基づく | 継続的なユーザーリサーチ可能なチーム |

**重要**: パターン数の増加に合わせて構造を変更することを前提とする。最初の構造が完璧である必要はない。

### パターンドキュメントの基本要素

#### 機能パターン（Functional Patterns）のドキュメント

**1. 名前（Name）**
- パターンの目的が名前だけで伝わること
- 見出しとして目立つように配置する

**2. 目的（Purpose）**
- 1〜2文で「何であるか」「何のためのものか」を明記する
- 曖昧な説明例: 「Showcase は複数タイプの情報をメディアファイルと表示する」（NGパターン）
- 具体的な説明例: 「Fact Grid は事実の短縮リスト。読者に次のコンテンツの印象を与えるために使用する」

**3. 使用例（Example）**
- 目的を強調するUIコピーを使ったライブサンプルを優先する
- インタラクション・アニメーションはライブサンプルで示す
- 特定の状態はGIF・静止画で補足する

**4. バリアント（Variants）**
- バリアントをスイートとして並べて表示する
- 各バリアントの違いを明文化する（「Optional、Infrequent、Subtle なアクション」等）

**追加で文書化できる事項**:
- バージョン履歴・廃止要素と代替パターン
- パターン作成者・担当者
- 関連パターン・代替パターン

#### 知覚パターン（Perceptual Patterns）のドキュメント

**色**:
- パレット（変数リスト）だけでなく、用途（テキスト・リンク・背景等）を明記する
- Do/Don't 形式で誤用を防ぐ

**タイポグラフィ**:
- 文字のペアリングと推奨用途を示す
- コンテキスト内（実際のUIに近い状態）でサンプルを表示する

**コンポーネントレベルでのスタイル参照**:
- コンポーネントドキュメント内にも使用スタイルを記載する（色のみの独立ページでは不十分）
- インタラクティブステート（hover/focus/active/disabled）を全コンポーネントで統一的に定義する

## 運用・ガバナンス

### 新規パターン追加のワークフロー

**基本的な3ステップ**（シンプルなアプローチ）:
1. 提案者がデザインをレポジトリ（またはDropbox等）に提出する
2. チームで包含の是非を議論する（定期会議または非同期レビュー）
3. 承認されたパターンをパターンライブラリに追加・公開する

### パターン追加の基準

| アプローチ | 内容 | リスク |
|-----------|------|------|
| **自動追加** | 新UIは自動的にライブラリに追加 | 重複パターンが増える可能性 |
| **再使用時追加** | 2〜3回使われた時点で追加 | ライブラリ外の要素が見えにくくなる |
| **再使用ポテンシャル** | 汎用的に定義されていれば追加 | 定義の厳密さが求められる |

**推奨**: 用途特化の一時的コンポーネント（季節プロモ等）は例外として管理し、ライブラリに追加しない。

### チームモデル：キュレーターとプロデューサー

| モデル | 特性 | 適した組織 |
|-------|------|-----------|
| **Curator（キュレーター）** | 各チームから提案を受け付け、基準設定とレビューを担当 | 分散型チーム・柔軟なシステム |
| **Producer（プロデューサー）** | システムチームが主要パターンを作成、最終決定権を持つ | 集中型チーム・厳格なシステム |

**共通の原則**: どちらのモデルでも、システムチームは「警察」ではなく「パートナー」として機能する。早期の協力関係を築き、承認・拒否のみの関係にならないようにする。

### デザイン・コード・ライブラリの同期

3つのファシットを概念的に整合させる:
- **コンポーネントライブラリ（コード）**
- **デザインファイル（Figma/Sketch）**
- **パターンライブラリ（ドキュメント）**

命名規則・フォルダ構造・パターンの目的理解を3者間で統一する。完全な同期は困難なので、「常に少し差異がある」ことを前提として運用ルールを設計する。

## 測定・維持

### OKR（目標と主要結果）による測定

デザインシステムの有効性をOKRフレームワークで追跡する。

**Objective例1: UXサイクル時間を削減し、市場投入速度を向上する**

| Key Result | 内容 |
|-----------|------|
| KR1 | Priority 1 UXコンポーネントのシステム導入ロードマップを定義する |
| KR2 | 月8件のガイドラインを確定・公開する |
| KR3 | コードレビューに要するUXレビュー時間を50%削減する |

**Objective例2: アクセシビリティをすべての実装に組み込む**

| Key Result | 内容 |
|-----------|------|
| KR1 | 新規コンポーネントの100%がアクセシビリティ基準チェックリストを満たす |
| KR2 | 3ヶ月以内に既存コンポーネントをすべてアクセシビリティ基準に更新する |

**Objective例3: デザインシステムを組織横断のリソースにする**

| Key Result | 内容 |
|-----------|------|
| KR1 | 非デザインチームメンバーのシステムチャンネル参加を月10名増加させる |
| KR2 | 非デザインチームからのコントリビューションを月5件増加させる |

### アンケートによる効果測定

定期的なアンケートでシステムの有効性を定量化する。

**測定すべき指標**:
- コンポーネントのスタイル・使用方法に関する質問頻度
- デザインシステムを参照した回数
- ドキュメントで求める情報を見つけられた割合

ベースラインデータを導入前に収集し、定期的に再測定して変化を追跡する。

## 成長に伴うシステム維持

### 組織成長時の対応

| 対応策 | 内容 |
|-------|------|
| **新メンバーのオンボーディング** | 全デザイナーにシステムの使用方法・コントリビューション方法を指導する |
| **部門横断での普及** | AMAs・ウェビナーでプロダクト管理・開発チームへの認知を広げる |
| **専任チームの組成** | 組織の成長に合わせてデザインシステム専任チームを設立する |

### UX Debt（UXデット）の管理

UX Debt = 文書化されたスタンダードとプロダクト実装の間の不整合蓄積。

**対処戦略**:

| 戦略 | 内容 |
|------|------|
| **段階的実装** | 一度に全修正を試みず、機能単位で順次改善する |
| **設計者のオーナーシップ** | デザイナーがシステムに対して責任を持ち、スプリントごとに改善を組み込む |
| **部門横断コラボレーション** | プロダクト管理・エンジニアリングと協調し、優先度と複雑度を共有する |

### イテレーション原則

| 原則 | 内容 |
|------|------|
| **完璧主義を避ける** | 長い議論より動くものを早く出して改善する |
| **一元管理** | 全員がアクセスできる集中したシステムを維持する |
| **プロセス設計** | 追加・レビューのためのチェック機構を仕組みとして設ける |
| **フォーカス維持** | デリバリー圧力に流されずシステムへの投資時間を守る |

### 変更管理

コンポーネントやガイドラインを改定する際の原則:

1. 変更はユーザーニーズや組織要件の変化に基づいて行う
2. 変更内容と理由をドキュメント化する
3. ロードマップとして組織内に事前周知する
4. コンポーネントのステータス変更（deprecated等）を明示する

システムを固定されたものではなく、製品・組織と共に成長する生きたリソースとして扱う。
