# UIパターンカタログ

デザインシステムで活用できるUIパターンの定義・分類・組み合わせ手法を提供する実践リファレンス。

---

## パターンの基本構造

### パターンの3要素

| 要素 | 説明 |
|------|------|
| **Named Solution（命名された解法）** | 設計判断に名前を与える。共通語彙として機能し、チーム間の設計議論を加速する |
| **Problem（問題）** | このパターンが解決しようとしている設計課題 |
| **Context（文脈）** | パターンが効果を発揮する条件・状況 |

### 追加の記述要素（充実したパターン）

- **Why**: このパターンが問題を解決する理由
- **Examples**: 実際の適用例・スクリーンショット
- **How**: 実装の詳細と注意点
- **Consequences**: トレードオフ・副作用
- **Related**: 関連パターン・代替パターン

---

## コンテキスト分析の5要素

パターンを選択する前に、以下5つの観点からコンテキストを分析する。

| 要素 | 分析内容 |
|------|----------|
| **User journeys** | ユーザーがどのような経路でここに到達するか |
| **User tasks** | ユーザーがこの画面で達成しようとしていること |
| **Personas** | 対象ユーザーの知識レベル・目的・状況 |
| **Constraints** | 技術的・ビジネス的・法的な制約 |
| **Content & Data** | 扱うコンテンツの性質・量・変動性 |

## カタログ：オンボーディング系

### Walkthrough（ウォークスルー）

**問題**: 複雑なワークフローや新しい概念を持つアプリで、初回ユーザーが迷子になる。

**文脈**:
- UIを見ただけでは目的が伝わらない機能がある
- ユーザーに事前知識が必要な概念が存在する
- 登録後の最初の体験が製品の価値を決定する

**How**:
- ステップ数を画面に表示してユーザーに進捗を伝える
- 各ステップは1つの概念またはアクションに絞る
- スキップオプションを提供する
- ユーザーが読んだことを確認できるインタラクションを含める

**避けるべき場面**: 直感的に使えるシンプルなアプリ、再訪問ユーザー

---

### Playthrough（プレイスルー）

**問題**: ユーザーがアプリの仕組みを把握する前に、実際の操作を体験したい。

**文脈**:
- インタラクティブな操作が製品の核心である
- 「やってみないとわからない」性質のUI
- Setup Wizard または Sandbox 環境が提供できる場合

**How**:
- 実際のUIと同一または近い環境で操作させる
- エラーリスクのないSandbox（サンプルデータ活用）を用意する
- 完了時に本番環境へ誘導する

**Walkthrough vs Playthrough**:
| | Walkthrough | Playthrough |
|--|-------------|-------------|
| 学習方式 | 説明を読む | 実際に操作する |
| 適した内容 | 概念説明・ワークフロー紹介 | インタラクション習得 |

---

### Coachmarks（コーチマーク）

**問題**: 新機能や隠れた機能を、既存ユーザーに気づいてもらえない。

**文脈**:
- 新機能のリリース時
- 重要だが見落とされやすいUI要素の案内
- ユーザーが最初に操作する瞬間

**How**:
- 対象要素をオーバーレイでハイライトし、簡潔な説明を付ける
- 説明は1〜2文以内。詳細リンクを提供する
- 閉じるボタンを目立つ位置に配置する

**⚠️ 注意**: 一度に複数のCoachmarkを表示しない。ユーザーの注意を分散させ、体験を壊す。

---

### Progressive Disclosure（段階的開示）

**問題**: 全情報を一度に表示すると認知負荷が高く、重要な情報が埋もれる。

**文脈**:
- 初心者と上級者が混在するユーザー層
- 主要アクションと補助的オプションが存在する
- 設定画面・フォームの複雑な選択肢

**How**:
- デフォルトで最重要情報のみ表示する
- ユーザー操作（展開ボタン・タブ）で追加情報を表示する
- 「詳細設定」「高度なオプション」等のラベルで誘導する

**Staged Disclosure との違い**:
| | Progressive Disclosure | Staged Disclosure |
|--|------------------------|-------------------|
| トリガー | ユーザーが能動的に選択 | システムが自動的に次ステップへ |
| フロー | 非線形・ユーザー主導 | 線形・ガイド付き |
| 適用 | 設定・詳細情報 | 登録・設定ウィザード |

---

### Staged Disclosure（段階的フロー）

**問題**: 複数ステップを要するプロセスを1画面に詰め込むと圧倒感を与える。

**文脈**:
- 会員登録・支払い・設定のウィザード
- 情報を段階的に収集する必要があるフォーム
- ステップ間で依存関係がある入力項目

**How**:
- 各ステップは単一の目的に絞る
- 進捗インジケーター（「3/5」等）を表示する
- 戻るボタンを提供し、入力済み情報を保持する

---

### Progressive Reduction（段階的削減）

**問題**: ユーザーが慣れた後も、初心者向けのヘルプ・ラベルが邪魔になる。

**文脈**:
- 頻繁に使用するパワーユーザー向け機能
- ツールバー・コンテキストメニュー
- 操作回数や習熟度を追跡できる場合

**How**:
- 使用回数・スキルレベルに応じてUI要素を段階的に簡略化する
- テキストラベルをアイコンのみに変更する
- 既知のショートカットを案内して習慣化を促す

---

## カタログ：認証系

### Login Form（ログインフォーム）

**問題**: ユーザーが自分のアカウントにアクセスするために認証が必要。

**必須構成要素**:

| 要素 | 役割 |
|------|------|
| 識別子フィールド | メールアドレスまたはユーザー名 |
| 認証情報フィールド | パスワード（マスク表示） |
| Submit ボタン | 明確なラベル（「ログイン」「サインイン」） |
| エラーリカバリー | 「パスワードを忘れた」リンク |

**How**:
- エラーメッセージは「パスワードが間違っています」ではなく「メールアドレスまたはパスワードが正しくありません」とする（セキュリティ対策）
- 入力フィールドにautocomplete属性を適切に設定する
- 失敗回数に応じたCAPTCHAやレート制限を実装する

---

### Social Signup（ソーシャルサインアップ）

**問題**: 新規登録フォームへの入力摩擦が登録完了率を下げる。

**文脈**:
- ユーザーが既に対応するSNSアカウントを持つ場合
- 信頼性の高い認証プロバイダーが利用可能な場合
- 素早い登録体験を最優先する場合

**How**:
- Google・GitHub・Apple等、ターゲットユーザーが使用するプロバイダーを選定する
- 「Google でサインイン」等、動詞を含む明確なラベルを使用する
- OAuth のスコープは最小限にする

**Consequences**: プロバイダーへの依存が生まれる。アカウント統合の複雑さが増す。

---

### Lazy Signup（遅延登録）

**問題**: 登録を強制すると、価値を知る前にユーザーが離脱する。

**文脈**:
- 製品の価値体験を登録前に提供できる場合
- ゲストとして一定の操作を許可できるアーキテクチャ
- コンバージョン率の改善が優先目標の場合

**How**:
- まずゲストとして使用させ、保存・共有等の段階で登録を促す
- 「登録するとXX機能が使えます」と具体的なメリットを提示する
- 入力した情報をセッション保存し、登録後に引き継ぐ

---

## カタログ：フォーム・フィードバック系

### Newsletter Signup（メールマガジン登録）

**必須構成要素**:

| 要素 | 役割 |
|------|------|
| Label | 何のメールマガジンか明示 |
| Input (email) | メールアドレス入力欄 |
| Submit | 登録ボタン |
| List description | 配信内容・頻度の説明 |
| Validation | 入力エラーのフィードバック |
| Legal | プライバシーポリシーへの同意 |

**How**:
- 送信後の成功メッセージを明示する（「登録完了。確認メールをお送りしました」）
- 配信解除が容易であることを事前に伝える

---

### Validation Feedback（バリデーションフィードバック）

**問題**: ユーザーがフォームを送信するまでエラーを知れない。または、エラーメッセージが曖昧で修正方法がわからない。

**フィードバックの種類**:

| 種類 | タイミング | 用途 |
|------|-----------|------|
| **Warning** | 入力中・フォーカスアウト | 潜在的な問題（強力でないパスワード等） |
| **Error** | 送信試行後 | 修正必須の問題 |
| **Confirmation** | 入力完了 | 入力が正しいことの確認 |

**How**:
- エラーメッセージは「入力が無効です」ではなく「メールアドレスの形式が正しくありません（例: user@example.com）」と具体的に書く
- エラーはフィールドの近くに表示する
- 色のみに依存しない（アイコン・テキストを併用する）

---

### Good Defaults（適切なデフォルト値）

**問題**: 多数の設定項目を空欄のまま表示すると、ユーザーが途方に暮れる。

**文脈**:
- ユーザーの大多数が選択するオプションが分かる場合
- 過去のデータや一般的な傾向から最適値を推定できる場合
- 空の状態（Empty State）を避けたい場合

**How**:
- 多数派に適したデフォルト値を事前入力する
- デフォルト値を変更しやすいUIにする（ロック感を与えない）
- 例: 国のデフォルトをIPアドレスから推定、日付を本日に設定

---

## カタログ：通知系

### Notifications（通知）

**問題**: ユーザーに重要な情報を適切なタイミングで届けたい。しかし、過剰な通知はユーザーを疲弊させる。

**時間感度による分類**:

| 種類 | 時間感度 | コンテナ |
|------|----------|----------|
| Toast / Snackbar | 即時（数秒） | 一時的オーバーレイ |
| Inline notification | 中程度 | コンテンツ内埋め込み |
| Modal alert | 緊急 | フルスクリーンモーダル |
| Notification center | 低（後で確認） | 専用パネル |

**Dismiss（閉じる）のルール**:
- Toast: 一定時間後に自動消去 + 手動閉じるを提供
- Modal: ユーザーが明示的に操作するまで消えない
- Inline: 関連コンテンツが解決するまで表示維持

**Opportune Moment（適切なタイミング）**:
- タスクの区切りや自然な中断点で通知する
- 集中を要する操作中は通知を遅延させる

---

## カタログ：検索系

### Autocomplete（オートコンプリート）

**問題**: ユーザーが正確なクエリを入力しなければならず、タイプミスや表記揺れで目的の結果に到達できない。

**文脈**:
- 検索対象のアイテム数が多い（100件以上）
- ユーザーが正確なアイテム名を知らない可能性がある
- 既存データから補完候補が生成できる

**How**:
- 入力開始（2〜3文字）から候補を即時表示する
- キーボード操作（矢印キー・Enter）で候補を選択できるようにする
- 候補が0件の場合でも「見つかりませんでした」とフィードバックする

**Autosuggest との違い**:
| | Autocomplete | Autosuggest |
|--|--------------|-------------|
| 目的 | 既存アイテムへの入力補完 | 関連クエリ・代替案の提案 |
| 結果 | 確定したアイテムリスト | 検索クエリの候補 |

---

### Search Filters（検索フィルター）

**問題**: 検索結果が多すぎてユーザーが目的のアイテムを見つけられない。

**フィルターの実装方式**:

| 方式 | 特性 | 適用 |
|------|------|------|
| **Live filter** | 選択と同時に結果が絞り込まれる | シンプルなフィルター、高速API |
| **Batch filter** | Apply ボタンで一括適用 | 複数条件、重い処理 |

**複数フィルターのUX**:
- 適用中のフィルターを一覧表示する
- 個別・一括のクリアオプションを提供する
- フィルター適用後の件数を表示する（「〇件の結果」）

---

## カタログ：コンテンツ表示系

### Activity Feed（アクティビティフィード）

**問題**: ユーザーが自分や他者の最近のアクティビティを把握できない。

**スキャナビリティ設計**:
- アイテム間に明確な区切りを設ける
- タイムスタンプは相対表示（「3分前」）と絶対表示（「12月3日」）を使い分ける
- 各アクティビティに関連するアクション（いいね・返信）を近くに配置する

**パフォーマンス**:
- 初回ロードは20〜30件程度に制限し、スクロールで追加ロードする
- Infinite Scroll または「もっと見る」ボタンを選択する

---

### Pagination（ページネーション）

**問題**: 大量のアイテムを一度に表示するとパフォーマンスが低下し、ユーザーの認知負荷も高まる。

**文脈**:
- 特定のアイテムを後で参照したい（URL共有・ブックマーク）
- ユーザーが「〇ページ目」という感覚を持つことが有用
- データ全体の量が把握できる場合

**設計のポイント**:
- 前へ・次へボタンに加え、ページ番号を表示する
- 現在のページを明示する（視覚的・aria-current）
- 1ページあたりのアイテム数は5〜25件程度（コンテンツの性質による）
- 総件数・総ページ数を表示する

**Pagination vs Infinite Scroll**:
| | Pagination | Infinite Scroll |
|--|------------|-----------------|
| URL共有 | ✅ 容易 | ❌ 困難 |
| 特定位置への復帰 | ✅ 可能 | ❌ 困難 |
| ゴールなき探索 | ❌ 中断感あり | ✅ 継続しやすい |
| フッターへのアクセス | ✅ 可能 | ❌ 埋もれる |

---

### Infinite Scroll（無限スクロール）

**問題**: ページ切り替えのインタラクションがコンテンツ探索の流れを中断する。

**適した文脈**:
- ソーシャルフィード・ニュースフィード
- 特定のゴールなく探索するコンテンツ（画像ギャラリー）
- アイテムの順序が時系列で、ページの区切りに意味がない場合

**避けるべき場面**:
- 特定アイテムの検索・比較
- フッターへのアクセスが重要な場合
- アイテム数が有限で全件表示したい場合

**実装上の注意**:
- ローディング中のスケルトンスクリーンを表示する
- スクロール位置をブラウザ履歴と同期する（戻ると同じ位置に戻る）

---

## カタログ：ソーシャル系

### Favorites（お気に入り）

**問題**: ユーザーが後で参照したいアイテムを保存する手段がない。

**設計のポイント**:
- ハート・星等のアイコンはユーザーが直感的に理解できるものを選ぶ
- 追加・削除のフィードバックは即時に（アニメーション効果が有効）
- 公開（他者が閲覧可能）か非公開（個人のみ）かを明確にする
- リスト管理機能（複数リストへの分類）は高度な使用者向け

---

### Follow（フォロー）

**問題**: ユーザーが特定のユーザー・トピック・タグの更新を継続的に受け取りたい。

**特性**: 一方向の購読関係。相手の承認なしに成立する。

**設計のポイント**:
- フォロー済み状態を明確に表示する（「フォロー中」ボタン）
- フォロー解除を誤操作しにくいUI設計
- フォロワー数・フォロー数の表示（オプション）

---

### Friend's List（フレンドリスト）

**問題**: 双方向の関係性（友達申請と承認）を管理する手段が必要。

**特性**: 双方の承認が必要な相互関係。Follow とは異なり、非対称ではない。

**状態管理**:

| 状態 | 表示 |
|------|------|
| 未申請 | 「友達申請」ボタン |
| 申請中 | 「申請取り消し」または「申請中」 |
| 承認待ち | 「承認」「拒否」ボタン |
| 友達 | 「友達」表示・メッセージボタン |

---

## パターン組み合わせ手法

複数のパターンを組み合わせて複雑なUIを構築する際の8つのアプローチ（スコープドサーチを例として）。

### アプローチ1：要素の再利用
既存コンポーネント（Input、Button、Autocomplete等）を再利用する。新規パターンの作成は最後の手段。

### アプローチ2：重複の削除
複数箇所で独立して実装されている同じパターンを統合し、単一の実装に集約する。

### アプローチ3：効率的な組み合わせ
関連するパターンを論理的にグループ化し、ユーザーの操作フローを最適化する。

### アプローチ4：中間ステップ（インタースティシャル）
長いフローの途中に自然な確認・説明ステップを挿入し、ユーザーを誘導する。

### アプローチ5：視覚的な結合と区別
関連するUI要素を視覚的にグルーピングし、無関係な要素を明確に分離する。

### アプローチ6：データの保持と破棄
ユーザーの入力・選択をどの範囲で保持するかを明示的に設計する（フィルター解除後の状態など）。

### アプローチ7：繰り返しパターンの明確化
同じ操作が繰り返し発生する箇所を特定し、一貫したインタラクションモデルを適用する。

### アプローチ8：トレードオフの評価
各アプローチの利便性・実装コスト・ユーザー体験への影響を比較し、最適な組み合わせを選択する。

## パターン破りの判断基準

標準パターンから逸脱する前に確認すべき5段階の判断プロセス。

### 段階1：Design Smell の調査
- ユーザーテストでパターンに問題があるかを確認する
- 分析データで問題箇所を特定する
- 感覚的な違和感だけでは判断しない

### 段階2：パターン原則のレビュー
- そのパターンが守ろうとしている設計原則を理解する
- 原則を維持しつつ問題を解決できないか検討する

### 段階3：問題と文脈の再確認
- 現在の文脈でそのパターンが本当に適切かを再評価する
- 別のパターンの方が文脈に合っている可能性を検討する

### 段階4：予測可能性の維持
- ユーザーが標準パターンに基づいた期待を持っている
- 逸脱することでユーザーを混乱させないかを評価する

### 段階5：革新の必要性
- 既存パターンより明らかに優れている（2倍以上の改善）場合のみ逸脱を検討する

### デザインシステムでの判断基準テーブル

| 状況 | 判断 |
|------|------|
| 一貫性が予測可能性を損なっている | パターン破りを検討 |
| 一貫性がユーザーに有害な体験を生んでいる | パターン破りを検討 |
| 文脈によって動作が本質的に異なる | バリアント作成を検討 |
| 単に見た目が古い・飽きた | パターン維持 |
| チームの好みの問題 | パターン維持 |

## Information Architecture の8原則

（Dan Brown による分類）

| 原則 | 説明 |
|------|------|
| **Objects** | コンテンツを生きたもの・属性・ライフサイクルを持つオブジェクトとして扱う |
| **Choices** | ページごとの選択肢を意味のある範囲に絞る |
| **Disclosure** | ユーザーが何かを決める際、必要十分な情報のみを表示する |
| **Exemplars** | カテゴリの内容を例示してユーザーの理解を助ける |
| **Front doors** | どのページからも訪問者が来ることを想定し、コンテキストを提供する |
| **Multiple classification** | 異なる視点からコンテンツを分類する複数のパスを提供する |
| **Focused navigation** | 1つのナビゲーションシステムで複数の目的を混在させない |
| **Growth** | コンテンツが増えることを想定して設計する |
