# デザインシステム 実践パターン集

複数組織のデザインシステム運用事例から、横断的なパターンを抽出・整理したリファレンス。
特定の組織・個人・書籍に依存しない汎用的な知見として再構成している。

---

## 1. DS立ち上げ動機パターン

| 動機パターン | 概要 | 典型的なトリガー |
|------------|------|----------------|
| **UI統一** | サービス間・チーム間でUIがバラバラになっている | プロダクト数増加、長期運用による老朽化 |
| **開発速度向上** | デザイン・実装の重複作業を排除したい | 同じ議論が何度も繰り返される状態 |
| **スケール対応** | 人員増加後も一貫性を保ちたい | チーム分割、採用拡大のタイミング |
| **技術負債解消** | レガシーUIの刷新を体系的に進めたい | 大規模リニューアル、メンテナー不在 |
| **デザイナー補完** | デザイナーがアサインされないプロジェクトの品質担保 | デザイナー1名体制、新事業立ち上げ |
| **アクセシビリティ戦略** | 草の根的な取り組みを超えて、共通基盤から浸透させたい | 法的要件、ブランド品質基準の策定 |
| **リソース集約** | 各部門がバラバラに進めているDS開発を統合したい | マルチブランド・複数プロダクト運営 |

> 複数の動機が重なるケースが多い。特に「UI統一 + 開発速度向上」「スケール対応 + デザイナー補完」は頻出の組み合わせ。

## 2. 組織規模・開始タイミングパターン

| 規模感 | 特徴 | 注意点 |
|--------|------|--------|
| **〜30名（超小型）** | 1名のデザイナーが立ち上げ。速度優先 | オーバーエンジニアリングに注意 |
| **30〜100名（小型）** | デザイナー+エンジニアの少人数チーム | 兼務が前提。専任確保は困難 |
| **100〜500名（中型）** | 複数チームが存在し始め、一貫性問題が顕在化 | 浸透施策が必要になる段階 |
| **500名〜（大型）** | プロダクト横断でのガバナンスが求められる | ステークホルダー調整コストが増大 |

**開始の判断基準（パターン）**

- プロダクト開発のアジリティ向上が目的 → 早期に着手（10〜50名規模でも有効）
- UI統一・技術負債解消が主目的 → 大規模リニューアルのタイミングに合わせる
- 公式な施策として推進 → 組織改編・新部門設立のタイミングを活用

## 3. チーム構成パターン

### 立ち上げ期の構成

| パターン | 構成 | 向いているケース |
|---------|------|----------------|
| **デザイナー単独** | デザイナー1名 | 超小型組織、PMF前のスタートアップ |
| **デザイン中心** | デザイナー2〜4名（エンジニアは外部協力） | デザインガイドライン先行型 |
| **混成小チーム** | デザイナー2〜3名 + エンジニア1〜2名 | トークン・コンポーネント実装を並走 |
| **多職種チーム** | PM + デザイナー + エンジニア計8名前後 | 大規模組織の横断プロジェクト |

> **全員兼務が当たり前** — 専任メンバーを置いている組織は少数派。プロダクト開発と並走する前提で設計する。

### 成熟期の構成（よく見られるパターン）

| 役割 | 担当 |
|------|------|
| DS専任リード（1〜2名） | 方針策定・優先順位管理・全体統括 |
| デザイナー（兼務、複数） | UIパターン定義・Figmaライブラリ管理 |
| エンジニア（兼務、複数） | コンポーネント実装・Storybook管理 |
| アクセシビリティ担当（1名） | a11yチェック・ガイドライン整備 |
| 各プロダクトコントリビューター | 提案・実装・フィードバック |

## 4. 初期コンテンツパターン

### スタートセットの類型

| 類型 | 初期コンテンツ | 特徴 |
|------|--------------|------|
| **トークンファースト** | カラー・タイポグラフィのデザイントークン | 実装との接続が早い。エンジニア協力が必要 |
| **コンポーネントファースト** | ボタン・フォームなど最小限のUIコンポーネント | 即日適用できる反面、設計思想が後付けになりやすい |
| **原則ファースト** | デザインビジョン・デザイン原則のみ | 共通言語の確立を優先。具体的な成果物は後工程 |
| **ブランドファースト** | ブランドガイドライン + 基本スタイル | コミュニケーションデザイン領域も含む組織向け |
| **段階型** | トークン → コンポーネント → ガイドライン | 着実に拡張できる。長期プロジェクト向き |

### 初期コンテンツの選定観点

| 観点 | 問い |
|------|------|
| 最も困っている問題はどれか | UI統一 / 実装重複 / 共通認識不足 のどれが深刻か |
| デザイナーとエンジニアの連携度 | 実装まで手が届くか、デザインファイルのみか |
| 既存リソースは活用できるか | CSSフレームワーク・Sketchライブラリ等の移行コスト |

## 5. ツール選定パターン

| カテゴリ | よく選ばれるツール | 採用理由 |
|---------|-----------------|---------|
| **デザインツール** | Figma | チーム共同編集、コンポーネント管理、バリアント機能 |
| **コンポーネントカタログ** | Storybook | コンポーネント単位の開発・閲覧・インタラクションテスト |
| **ドキュメント管理** | Notion / 自社構築静的サイト | ガイドライン・原則・意思決定ログの管理 |
| **トークン管理** | Style Dictionary / Figma Tokens Plugin | トークンの変換・配布の自動化 |
| **バージョン管理** | GitHub | コード・変更履歴の管理、PR レビュープロセス |

**ツール選定の注意点**

- Figmaはほぼ必須だが、ブランチ運用の設計（誰が編集できるか）を事前に決めること
- ドキュメントサイトは「自社構築 vs zeroheight 等の SaaS」を比較検討する
- トークンの実装連携ツールは、フレームワーク（React / Vue / Native）に依存する

## 6. 運用課題と対策パターン

### 頻出する課題

| 課題 | 根本原因 | 対策パターン |
|------|---------|------------|
| **浸透しない** | 存在を知らない、使い方がわからない | オンボーディング整備・定期共有会・Slack監視 |
| **専任不足** | 全員兼務で優先度が下がる | CDO/リードによる経営層へのコミット・専任枠確保 |
| **コントリビューション不足** | プロダクト側がバグを報告しない | コントリビューターを「Spindler」等で名前付け・役割明確化 |
| **ガイドライン品質のばらつき** | 言語化スキルの差 | テンプレート整備・ペアライティング・レビュー体制 |
| **フィードバックループが回らない** | 更新の透明性不足 | 定期アップデートのリリースノート・変更ログ公開 |
| **プラットフォーム間の差異** | Web / iOS / Android での対応状況が異なる | プラットフォーム優先度を明示・ロードマップで管理 |

### 浸透施策の類型

| 施策 | 効果 | コスト |
|------|------|-------|
| わかりやすいガイドライン + 理由の明記 | 自己解決率向上 | 中（初期作成コスト） |
| 定期共有会・横断定例 | 認知維持・議論の場 | 低（継続的なミーティング） |
| 社内ツールへの積極導入 | 日常的な接触機会 | 中（実装工数） |
| オンボーディング（有人） | 新メンバーの早期習熟 | 高（担当者の時間） |
| コントリビューター制度 | 自走者の育成・エンゲージメント向上 | 中（仕組み設計） |
| Slackエゴサーチ・積極介入 | 潜在的な疑問の早期解消 | 低（担当者の意識） |

## 7. 論点別の多様な見解

### 「デザイン原則は必須か」

複数組織への調査結果:

| 回答 | 割合 | 主な見解 |
|------|------|---------|
| **必要** | 約 85% | 共通認識・判断基準として機能する。事業フェーズによらず、あると良い |
| **不要 / 課題による** | 約 15% | 解決したい課題が明確であれば原則がなくても動ける |

**「必要」派の理由（抽象化）**

- デザインレビュー時に属人化した議論を防ぐ言語化された基準になる
- 職種を問わず全員が製品を形作る共通認識を持てる
- 新メンバーがプロダクトの思想を学ぶ入口になる

**「不要」派の理由（抽象化）**

- 抽象的な原則よりも、具体的な実装判断基準の方が現場で使われる
- 課題が実装品質・速度向上であれば、コンポーネントで解決できる

> **結論**: デザイン原則は「必須」ではないが、チームが大きくなるほど・プロダクトが複雑になるほど価値が増す。

### 「コンポーネント集は必須か」

| 回答 | 割合 | 主な見解 |
|------|------|---------|
| **必要** | 約 69% | 実装効率化・品質統一の中核。エンジニアとの協働に不可欠 |
| **不要 / 課題による** | 約 31% | ブランドガイドラインやビジョン重視の組織では優先度が低い |

**「不要」派の事例**

- ブランドコミュニケーション（ロゴ・印刷物等）が主な適用範囲の場合
- 「デザイン原則 + 共通言語」でプロセス統一を優先するアプローチ
- 各プロダクトのUI多様性を尊重し、コンポーネントで縛らない方針

> **結論**: プロダクト開発組織にとってはほぼ必須。ただしコンポーネント整備より先に「デザイントークン + 原則」を揃える方が長期的に健全。

## 8. DSの命名パターン

DS の名前は組織の姿勢・カルチャーを反映する。命名は任意だが、名前があることで「誰もが指し示せる共通の存在」になる。

| 命名タイプ | 傾向 | 向いている組織 |
|-----------|------|--------------|
| **名前なし** | シンプルに「デザインシステム」と呼ぶ | 小規模・内部限定・実用優先 |
| **機能名・役割名** | "デザインシステム" そのまま、または機能記述型 | 政府・公共機関など中立性重視 |
| **抽象コンセプト名** | 組織の思想・世界観を込めた造語 | ブランドアイデンティティを重視する組織 |
| **料理・素材・道具** | プロダクトドメインと関連した隠喩 | 料理・生活系サービス |
| **生き物・自然** | 有機的な成長・連携をイメージ | ヘルスケア・医療系 |
| **建築・構造物** | 堅牢性・基盤性を表現 | インフラ・BtoB系 |

**命名のタイミング**

- 「必ず名前が必要」ではない。名前がなくてもDS は機能する
- 組織が拡大し、複数チームに展開する段階で名前があると浸透しやすくなる
- 名前を決めるプロセス自体がチームのDS観・ブランド観を整理する機会になる

## 9. DS効果の実感パターン

### 「役に立った」と感じる場面（類型）

| 効果カテゴリ | 具体的な場面 | 関係者 |
|------------|-----------|--------|
| **実装効率化** | HTMLやCSSに詳しくないエンジニアが高品質な画面を速く実装できた | エンジニア |
| **レビュー効率化** | デザインレビューで言語化されたルールを提示できた。属人的な議論が減った | デザイナー |
| **a11y浸透** | コンポーネントを使うだけでアクセシビリティチェックをパスできた | エンジニア・デザイナー |
| **新メンバー定着** | オンボーディングでDSを案内するだけで、新メンバーが即日戦力化できた | 全職種 |
| **コンセンサスコスト削減** | ステークホルダーが多い場面で、既存パターンを示すだけで合意が取れた | PM・デザイナー |
| **非デザイナーの活躍** | マーケター・ディレクターがDSを使って自分でデザインを完結させた | 非デザイン職 |
| **データ管理の統一** | デザイントークンを営業ドキュメントにも展開し、全社で統一デザインを実現 | ビジネスチーム |

### 効果測定の観点

DSの効果を定量的に示すのは難しいが、以下の観点でアプローチできる:

| 測定観点 | 指標例 |
|---------|--------|
| 開発効率 | 特定機能の実装工数の変化（Before/After比較） |
| 一貫性 | デザインレビューで指摘された「不整合」件数の推移 |
| a11y品質 | アクセシビリティチェック通過率 |
| 浸透度 | Figmaライブラリの利用ファイル数・コンポーネント呼び出し数 |
| 依存度 | 「DSがないと不便」という声の頻度 |

> Figmaのデザインシステムアナリティクス機能を活用すると、どのコンポーネントがどの程度使われているかを定量把握できる。利用ゼロのコンポーネントは削除候補として検討する。
