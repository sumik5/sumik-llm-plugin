# コード連携 リファレンス

FigmaデザインシステムとコードベースをつなぐJSON出力・Style Dictionary・Storybookの実装詳細。

---

## 1. FigmaからのJSONエクスポート

FigmaのバリアブルはJSON形式でエクスポートできる。エクスポートしたJSONをStyle Dictionaryで各プラットフォームのトークンファイルに変換するパイプラインを構築する。

**エクスポートされるJSONの例**

```json
{
  "color": {
    "text": {
      "default": { "$value": "#1A1A1A", "$type": "color" },
      "subtle": { "$value": "#6B6B6B", "$type": "color" }
    },
    "background": {
      "default": { "$value": "#FFFFFF", "$type": "color" }
    }
  },
  "spacing": {
    "4": { "$value": "16px", "$type": "dimension" }
  }
}
```

## 2. Style Dictionary

Style Dictionaryはデザイントークン（JSON）を複数のプラットフォーム向けファイルに変換するビルドツール。

**変換フロー**

```
Figma バリアブル
    ↓（JSONエクスポート）
tokens.json
    ↓（Style Dictionary）
├── CSS カスタムプロパティ（Web）
├── Swift / Objective-C 定数（iOS）
└── XML / Kotlin 定数（Android）
```

**CSS変換例（ライトモード）**

```css
:root {
  --color-text-default: #1A1A1A;
  --color-text-subtle: #6B6B6B;
  --color-background-default: #FFFFFF;
  --color-background-primary-action-enabled: #2563EB;
  --spacing-4: 16px;
  --border-radius-md: 8px;
}
```

**ダークモードの切り替え**

```css
:root {
  --color-text-default: #1A1A1A;
  --color-background-default: #FFFFFF;
}

[data-theme="dark"] {
  --color-text-default: #F5F5F5;
  --color-background-default: #1A1A1A;
}
```

セマンティックカラートークンのみをCSS変数として公開することで、ダークモード切替を1箇所のdata属性変更で実現できる。

## 3. Storybook連携

StorybookはコンポーネントのUIカタログであり、デザインシステムのドキュメントサイトとして機能する。

**主な役割**

| 機能 | 内容 |
|------|------|
| UIカタログ | コンポーネントを独立した環境で確認 |
| ドキュメント | PropTypes / TypeScriptの型から自動生成 |
| バリアント確認 | Controls機能でプロパティをリアルタイム変更 |
| 視覚的回帰テスト | スナップショット比較による変更検知 |

**Storybookとデザイントークンの連携**

1. CSS変数をStorybookのグローバルスタイルに注入
2. ダークモードはStorybookのthemeアドオンでdata属性を切替
3. デザイントークンのリファレンスページをドキュメントとして追加

**コンポーネントストーリーの例**

```tsx
// Button.stories.tsx
export default {
  title: 'Components/Button',
  component: Button,
};

export const Primary = {
  args: {
    variant: 'primary',
    size: 'md',
    children: 'ボタン',
  },
};

export const AllVariants = () => (
  <div style={{ display: 'flex', gap: 'var(--spacing-2)' }}>
    <Button variant="primary">Primary</Button>
    <Button variant="secondary">Secondary</Button>
    <Button variant="ghost">Ghost</Button>
    <Button variant="danger">Danger</Button>
  </div>
);
```

## 4. トークン更新のワークフロー

```
1. Figmaでバリアブルを変更
2. JSONをエクスポート（手動 or Figma Plugin自動化）
3. Style DictionaryでCSS/コード生成
4. プルリクエスト作成
5. Storybookで視覚的差分を確認
6. デザイナー・エンジニアの承認後マージ
```

Figmaバリアブルをトークンのsource of truthとし、コードは常にFigmaから生成されたトークンを参照する運用が理想形。この一方向の流れにより、デザインとコードの乖離を防ぐ。

## 5. エンジニアとのスムーズな連携

認識の齟齬を防ぐことがエンジニア連携のカギ。**データの作り方**と**コミュニケーション**の両面で工夫できることを押さえておこう。

### 5-1. データの作り方のルール化

#### パターンと差分が分かるようにまとめる

エンジニアはコンポーネントを実装するとき、まず共通化できる部分を探し、次に差分を把握してコード設計を組み立てる。コンポーネントライブラリにはパターンと差分が網羅されており、見やすくまとまっていることが重要。

- バリアントを活用してプロパティの値を記載しておくと、エンジニアは「共通構造・パターン数・状態バリエーション」をひと目で把握できる
- 例: ボタンであれば「ボックス＋テキストが共通」「primary/normal/text の3パターン」「2サイズ展開」「hover/focus/disabled の状態あり」が一覧できる状態にする

#### コンポーネントの命名を実装に合わせる

Figmaの右サイドパネルに表示される値はエンジニアがコードにコピー&ペーストして使うケースがある。コンポーネント名・バリアントのプロパティ名・カラー値がコードと乖離していると、参照ミスが発生する。

- コンポーネントの名称はエンジニアと相談して決める
- 実装済みコンポーネントの命名規則（例: `icon_clip`）とFigma上の命名（例: `icon-heart`）が異なるとコードとデザインが一致しなくなる
- 命名規則はデザインパターン（CEV命名等）を参照して統一する

#### 固定・可変の箇所を明示する

Auto Layoutを正しく設定することで、コンポーネントの寸法特性がエンジニアに伝わる。

| Auto Layout設定 | エンジニアが読み取れる情報 |
|---------------|----------------------|
| 幅 `100%`・高さ `Auto` | 幅は親要素に依存、高さはコンテンツに依存 |
| 固定サイズ指定 | 幅・高さが固定（テキスト量に関わらず変わらない） |

Auto Layoutを指定しないと、テキスト量が増減した際の挙動をエンジニア側で判断できなくなる。さらに、固定・可変かどうかを補助的にコメントで記載しておくと、だれが見てもコンポーネントの仕様をすぐ把握できる。

### 5-2. コミュニケーションの進め方

データの工夫で事前に防げることもあるが、予期せぬ課題は必ず生まれる。よくあるケースと解決策を把握しておこう。

#### ケース1: デザインルールの変更が伝わっていなかった

**状況の例**:
- 既存コンポーネント（例: テキストリンク）にパディングなしで設計されていた
- 後からパディング追加のデザインルールが生まれたが、Figmaの更新と周知が漏れた
- 担当者・画面によってパディングの有無にバラつきが生じた

**解決アプローチ**:
1. デザイナーとエンジニア担当者どうしで今後の対応方針を合意する（例: 修正工数確保まで既存画面はそのままにする）
2. 新しいデザインルールとその目的・対応方針をチーム内で再周知する
3. Figmaコンポーネントとデザインガイドラインを修正する
4. 未反映箇所は同じ画面で更新が入るタイミングに併せて対応する

#### ケース2: コンポーネント変更の影響範囲を考慮できていなかった

**状況の例**:
- アイコン変更をエンジニアに依頼したところ、コンポーネント管理の都合で意図しない画面の画像まで全て差し替わってしまった

**解決アプローチ**:
1. エンジニア側で古いアイコンを残し、新しいアイコンは別の名称で追加する
2. エンジニアチーム内ドキュメントに「古いアイコンは使わない」旨を記載する
3. デザイナーもアイコンのガイドラインに経緯を記載し、本番で古いアイコンを見た人が疑問を持たないようにする

#### ケース3: 後になってデザインルールの考慮漏れに気づいた

**状況の例**:
- モーダルの高さ上限をデザインルールとして定めていなかった
- コンテンツが多い場合に画面から見切れることに後から気づいた

**解決アプローチ**:
1. デザイナーとエンジニアで会話し、そのケースを機にルールを決める
2. デザインガイドラインにルールを追記し、チーム内で周知する

---

**共通の原則**: どのケースも地道に会話を重ねてひとつずつ解決していくことが基本。複数メンバーが同じ認識を持ち続けることは難しく、特効薬のような解決策はない。データの工夫もコミュニケーションも、小さな積み重ねを続けることがもっとも重要。普段からエンジニアの意見も取り入れ、信頼関係を築くことで解決策も見つけやすくなる。
