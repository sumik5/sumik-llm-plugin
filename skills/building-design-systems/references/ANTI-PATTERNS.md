# アンチパターン・Design Smell・デザイン負債

UIパターンの落とし穴と品質管理手法のリファレンス。「良く見えるが実は問題のある」解決策を識別し、回復する方法を体系的にまとめる。
INSTRUCTIONS.mdで参照される詳細版。

---

## アンチパターンとは何か

**アンチパターンとは、表面的には魅力的に見えるが、実際には解決しようとした問題より多くの問題を生み出す、繰り返し使われる「解決策」のことだ。**

UIアンチパターンは単なる「悪いデザイン」ではない。通常の悪いUI選択と異なり、アンチパターンは**表面上は良い解決策のように見える**。また、アンチパターンの問題範囲はUIそのものだけにとどまらず、コンポーネントがどのように更新されるかを取り巻くガバナンスやビジネスプロセスにまで及ぶ場合がある。

### アンチパターンの4要素

アンチパターンを正確に理解・記録するには、以下の4つの視点が必要：

| 要素 | 説明 |
|------|------|
| **症状（Symptoms）** | アンチパターンが*何を*するかの説明。良い結果より悪い結果の方が多い状態 |
| **誘惑理由（Tempting Reason）** | アンチパターンに引き込まれる*理由*。解決しようとしていたユーザー問題を含む |
| **文脈欠落（Missing Context）** | アンチパターンが有効だったかもしれない*条件の欠如*。代替パターンの示唆を含む |
| **回復手順（Recovery Steps）** | 問題から*どのように*回復するか。代替となる成功しているパターンを含む |

> **アンチパターンに積極的な意図を仮定する**: アンチパターンを使う人は意図的に難解なUIを作ろうとしているのではない。アンチパターンは表面上は魅力的に見える。問題に気付いたときは責める前に背景を理解すること。

## 代表的なアンチパターン

### 1. Mystery Meat Navigation（謎肉ナビゲーション）

**何をするか**: リンク先が明確でないリンクを使用する。ユーザーはリンク先を確認するためにホバーするか、盲目的に辿る必要がある。

**典型的な症状**:
- 「こちらをクリック（Click here）」リンク: 文脈なしに読むと意味が分からない（スクリーンリーダー、ページスキャン）
- ラベルなしアイコン: ホバーしないと機能が分からない
- クリック確信度の低下・完了時間の延長・放棄率の増加

**誘惑理由**:
- 「Click here」は初期Webの定番でユーザーへの行動指示として明確に見えた
- ラベルなしアイコンはスペース節約・視覚的ノイズ削減・翻訳コスト削減ができる

**文脈欠落（アイコンが有効な例外）**: 高頻度で使い慣れたユーザーが少数の特定アイコンを毎日使う場合（例: 毎日使うFacebookのメインナビゲーション）

**回復手順**:
- リンクテキストは内容そのものをリンクにする: ~~「詳細はこちら」~~ → 「**蜂蜜の詳細**を見る」
- アイコンにはラベルを付ける、またはテキストのみにする

---

### 2. Hamburger Basement（ハンバーガー地下室）

**何をするか**: ハンバーガーボタン（☰）の背後にナビゲーション選択肢を隠すことで、ユーザーが使う機能の発見性を損なう。

**誘惑理由**: 小さな画面でコンテンツを優先し、没入体験を作りつつプライマリナビゲーションへのアクセスを維持しようとした。

**症状リスト**:

| 問題 | 説明 |
|------|------|
| 発見性の低下 | ユーザーがプロダクトの他の機能を偶然に見つけにくい |
| 位置シグナルの喪失 | 「現在地」を示す別のナビゲーション要素を除去 |
| インタラクションコスト増加 | 即座にアクセスできるリンクと比較して1タップ余分に必要 |
| 親指の伸ばし強制 | 小さな画面で最も届きにくい場所に配置されることが多い |
| 謎肉ナビゲーションの一形態 | タップするまでメニューの中身が分からない |
| 分析麻痺 | 文脈の少ない長いリンクリストで選択が困難。誤ったリンクを選ぶと試行錯誤を繰り返す |
| プラットフォーム間の混乱 | デスクトップでは認識されにくい。iOS では「戻る」ボタンと競合 |

> **研究結果**: 隠れたナビゲーションはディスカバビリティを半減させ、エンゲージメントを著しく低下させる。

**回復手順（地下室をメニューに変える）**:
- セカンダリナビゲーション（設定・ヘルプ・プライバシーポリシー等）のみを格納する
- メインインターフェースからプライマリナビゲーションを露出させる（例: 下部タブバー）
- ハンバーガーボタンを親指の届く位置に移動する
- アイコンにラベルを付ける、またはボーダーで囲んでボタンらしく見せる
- デスクトップでの使用は避ける

## ダークパターンと悪意ある設計

**ダークパターン**（またはEvil Design / Black Hat UX）は、ユーザーよりも作成者に利益をもたらす欺瞞的なパターン。ユーザーが意図しない行動（隠れた費用を含む長期契約への登録など）を取るよう誘導したり、意図した行動（退会など）を阻止したりする。

デザインは本来、コミュニケーションと説得のツール。ダークパターンはそのツールを使って**人を騙す**点で根本的に異なる。

### Manipulinks と Confirmshamers

**Manipulink（操作的リンク）**: キャンセル・閉じる・却下などのリンクに、ユーザーを否定的な価値観の表明に追い込むテキストを使用するダークパターン。

**例**:
- ~~「いいえ、収益を増やしたくありません」~~
- ~~「結構です。割引は不要です」~~
- ~~「いいえ、かわいい赤ちゃんは好きじゃありません」~~

**問題点**:
- ラベルとして機能していない（クリックすると何が起きるかを説明していない）
- スクリーンリーダーユーザーにはモーダルの操作が困難になる
- 短期的なコンバージョン増加が長期的なブランド信頼・NPS低下につながる
- ネガティブな自己評価を促し、ユーザーのメンタルヘルスに影響する可能性がある

**Confirmshamer（確認による羞恥）**: Exit-intent pop-up（ユーザーがブラウザを閉じようとする意図を検知して表示するモーダル）によく現れる形態。

**回復手順**: 行動を正確に説明するオプトアウトリンクを使用する。例: 「いいえ、このガイドは不要です」は**ユーザーが実際にしていること**を正確に表現している。

## Design Smell（デザイン臭）

ソフトウェア開発における「コードスメル」の概念をUIデザインに転用したもの。**デザイン臭は、より深い問題の存在を示唆する可能性のある設計上の兆候**だ。

- 火事（根本問題）を発見する前に見る煙（表面的症状）のようなもの
- 臭いは*必ずしも*深刻な問題を意味しない（煙が煙幕機から来ている場合もある）
- しかし臭いを発見したときは、その背後を調査する価値がある

デザイン臭は**デザイン負債の蓄積**を識別する助けになる: 今すぐ問題を解決するために将来を担保にすること。デザイン負債の問題は、放置するほど対処にかかるコストが増大すること。

---

### Design Smell 1: TMI（Too Much Information / 情報過多）

**症状**: コンテンツが限界まで詰め込まれている状態。

| 形態 | 例 |
|------|-----|
| 巨大な設定リスト | 500項目のリスト・未分類で無関係な項目が「Settings」にまとめられている |
| ペジネーションの引き伸ばし（Pagination Stretch） | ページ数が何百・何千にも及ぶペジネーション |

**ペジネーションの引き伸ばし**は特に重要なTMIのシグナル: ページ数が多くなりすぎると、検索フィルタリング等のさらなる絞り込みが必要であることを示している。ユーザーが10ページ目以降に関連コンテンツを見つけることはほぼない。

**対策**: オートコンプリート・ページネーション・フィルタリング・検索等のパターンを活用してコンテンツを整理する。

---

### Design Smell 2: Modal Design Smell（モーダルのデザイン臭）

**モーダル（ダイアログ/オーバーレイ/ライトボックス）** は、フローを一時的に変更して別のコンテンツを表示する構造。

モーダルを使う場面: フロー内容やインタラクションを変更する必要があるとき。

しかしモーダルはしばしばアンチパターンになる。**モーダルの本質的な目的（フローを中断して一時的にモードを変える）は、UIに何か不十分なものがあることを示唆している**——なぜフローを中断する必要があるのか?

#### Pop-up アンチパターンの症状

| 症状 | 詳細 |
|------|------|
| アクセシビリティの問題 | キーボード・スクリーンリーダー・支援デバイスでの操作が困難 |
| モバイルで画面全体を覆う | 少ないコンテンツなら折りたたみ可能な要素内にインライン表示が適切 |
| システム発生による中断 | ユーザー起動でないポップアップはユーザーの期待に反する |
| 慣れによる反射的却下 | 多くのユーザーがコンテンツを読まずに閉じる |
| スクロール困難 | モバイルで大きなマージンがあると、タッチ可能なスクロール領域が狭い |
| モードエラー | ユーザーが別の場所に入力している最中にポップアップが現れ、誤入力を誘発 |
| カスケードポップアップ | 複数のポップアップが重なり、元のタスクからさらに離れる |

**モーダルを避けるための代替策**:
- インラインでアクションを即時提示（ボタングループ等）
- コンテンツをインライン表示（折りたたみ可能な「disclosure」要素）
- 別ページへのリダイレクト
- ツールチップ（画面端への対応が必要）
- スナックバー・トースト・ページ固定通知バー

**どうしても使う場合の最小化手順**:
- フォーカスを奪わない（他に入力中ならポップアップに入力させない）
- ユーザー起動のオンデマンド表示のみ（プログレッシブディスクロージャー）
- 簡単な退出手段: Escキー・オーバーレイクリック・明確な「閉じる」リンク
- アクセシビリティ対応: `role="dialog"`, `aria-labelledby`, `aria-describedby` + フォーカス管理
- ボタンラベルは動詞+名詞のペアで（例: 「プロジェクト作成」「招待を送信」）

---

### Design Smell 3: "Overall Pattern" Smell（全体パターン臭）

**症状**: デザインシステム内に、複数の異なるパターンを束ねる「親パターン」や「全体パターン」が存在する。

例: 「プロパティ・機能・プロセス・掲示」などのモーダルのカテゴリが全て「モーダル」という一つの親パターンとして文書化されている。

**問題点**: これらが全て同じスタイル・インタラクション・ビヘイビアを持つなら、実装するエンジニアにとって区別は無意味になる。

**解決の選択肢**:

| 状況 | 対応 |
|------|------|
| バリアントが意味のある違いを持つ | 親なしの個別の名前付きパターンに分割（例: 「benefits-modal」）。文書が重複しても良い |
| バリアントの差異が小さく意味がない | 差異を一つのパターンに統合。製品の一貫性・予測可能性が向上 |
| 本当はオプションや状態の違い | 一つのパターンのオプション（例: 通常スタイル vs. 警告スタイル）として表現 |

## 悪いパターンのライフタイム

アンチパターンとダークパターンは短期的な利益や表面的な勝利を優先するが、数年以内にその代償は明らかになる：

1. **消費者の反乱**: ユーザーは経験から賢くなり、より使いやすく信頼できるプロダクトに乗り換える
2. **業界の学習**: デザイナーが長期コストを認識し、新プロジェクトでの使用をやめる
3. **規制・法的措置**: 政府が禁止（例: EUでの「Sneak into Basket」禁止）、企業によるペナルティ（例: Googleによる侵入的インタースティシャル広告のペナルティ）

> **未来の傾向**: 新しいダークパターンが現れるたびに、規制・罰則・法的措置が迅速に追随するようになっている。

## パターン品質チェックリスト

新しいパターンを評価する際に、以下を確認する：

### アンチパターンかどうかの判定

- [ ] 短期的には有効に見えるが、長期的に悪い結果をもたらすか?
- [ ] ストップギャップとして意図されているのに、想定外の理由で長期化しそうか?
- [ ] 別の文脈では適切だが、現在の状況には合っていないか?
- [ ] UI問題の外に（ガバナンス・プロセス等の）根本原因があるか?

### ダークパターンかどうかの判定

- [ ] ユーザーより作成者・ビジネスに利益をもたらすか?
- [ ] ユーザーが意図しない行動を取るよう誘導するか?
- [ ] ユーザーが意図した行動を取ることを妨げるか?
- [ ] ユーザーを欺く要素が含まれているか?

### Design Smell の確認

- [ ] コンテンツが整理されないまま詰め込まれていないか?（TMI）
- [ ] ページ数・項目数が現実的な使用範囲を超えていないか?（Pagination Stretch）
- [ ] フローを中断させる必要性の根本原因を調査したか?（Modal Smell）
- [ ] 親パターンの下にサブパターンが束ねられていないか?（Overall Pattern Smell）

### パターンとして記録する前の確認

- [ ] 問題・解決策・コンテキストが明確に定義されているか?
- [ ] 少なくとも複数の場面で繰り返し使われているか?（1回のみの使用はパターンでない）
- [ ] 代替パターンと比較してトレードオフが評価されているか?
- [ ] アクセシビリティへの影響が検討されているか?
