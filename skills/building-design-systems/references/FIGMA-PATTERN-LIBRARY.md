# パターンライブラリ リファレンス

レイアウトシステム・グリッド設計・コンポーネント構成の詳細リファレンス。

---

## 1. レイアウトシステム

### 1-1. ブレイクポイント定義

| 名称 | 最小幅 | 想定デバイス |
|------|-------|------------|
| xxl  | 1400px以上 | 大型デスクトップ |
| xl   | 1200px以上 | デスクトップ |
| lg   | 992px以上  | 小型デスクトップ・横向きタブレット |
| md   | 768px以上  | タブレット |
| sm   | 576px以上  | 大型スマートフォン |
| xs   | 576px未満  | スマートフォン |

### 1-2. カラムグリッド設計

| ブレイクポイント | カラム数 | ガター幅 | マージン |
|---------------|---------|---------|---------|
| xxl / xl | 12 | 24px | 40px |
| lg       | 12 | 16px | 24px |
| md       | 8  | 16px | 16px |
| sm       | 4  | 16px | 16px |
| xs       | 2  | 8px  | 8px  |

カラムグリッドはFigmaの「レイアウトグリッド」機能でフレームに適用する。フレームサイズをブレイクポイントの最小幅に設定し、グリッドが正確に敷けているかを確認する。

### 1-3. バーティカルグリッド

垂直方向の余白管理には8pxグリッドを使用する。

- 基準単位: **8px**
- 細かい調整: 4px（基準の半分）
- すべてのスペーシングは8の倍数または4の倍数を原則とする

`spacing` トークンはこの8pxグリッドに基づいて定義されている（DESIGN-TOKENS.md参照）。

## 2. Figmaでのグリッド実装

### 2-1. レイアウトグリッドの設定手順

1. フレームを選択し、右パネルの「レイアウトグリッド」を追加
2. 種類を「カラム」に設定
3. カラム数・ガター幅・マージンをブレイクポイントに合わせて設定
4. 色を薄い赤等にして視認性を確保

### 2-2. バリアブルモードによるブレイクポイント連携

`Breakpoint` バリアブルコレクションのモードを切り替えることで、同一フレーム内でブレイクポイントごとのレイアウトを確認できる。

```
フレームを xxl モードに切替 → 1400px幅でのレイアウトを確認
フレームを md モードに切替  → 768px幅でのレイアウトを確認
```

コンポーネントのパディング・マージン・フォントサイズを `Breakpoint` バリアブルで管理すると、この切替が自動反映される。

## 3. コンポーネント設計

### 3-1. バリアント設計

Figmaのコンポーネントは「バリアント（Variant）」と「コンポーネントプロパティ（Component Property）」を使って状態と見た目を管理する。

**バリアントで管理するもの（外見が変わる）**

- Size（サイズ）: `sm` / `md` / `lg`
- Variant（種別）: `primary` / `secondary` / `ghost` / `danger`

**コンポーネントプロパティで管理するもの（内容が変わる）**

- State（状態）: `enabled` / `hovered` / `pressed` / `disabled` → Boolean or Instance Swap
- Icon（アイコン有無）: Boolean Property
- Label（ラベルテキスト）: Text Property
- Leading Icon / Trailing Icon: Instance Swap Property

### 3-2. ボタンコンポーネントの例

```
Button
  ├── Variant: primary / secondary / ghost / danger
  ├── Size: sm / md / lg
  ├── State: enabled / hovered / pressed / disabled
  ├── hasLeadingIcon: Boolean
  └── hasTrailingIcon: Boolean
```

バリアントと状態を掛け合わせると組み合わせが膨大になる。「バリアント = 外見の種類」「状態 = インタラクション上の変化」と切り分けることで、コンポーネントセットを整理しやすくなる。

### 3-3. ベースコンポーネントとコンポジット

ファイル構成はモノリシック→ライブラリ化→モジュール化の順に段階的に成熟させる。

| 段階 | 構成 | 特徴 |
|-----|------|------|
| モノリシック | 1ファイルに全コンポーネント | 小規模向け・シンプル |
| ライブラリ化 | 共有ライブラリファイル分離 | チーム利用開始時 |
| モジュール化 | カテゴリ別ファイル分割 | 大規模・複数チーム |
| ベースコンポーネント | 基底コンポーネントとバリアントを分離 | 再利用性の最大化 |

ベースコンポーネント方式では、スタイル定義を持つ「基底コンポーネント」と、それを参照する「テーマ付きコンポーネント」を分けることで、テーマ変更を局所的に管理できる。

## 4. テンプレートと画面設計

### 4-1. テンプレートの役割

テンプレートはコンポーネントを配置したページレイアウトの雛形。具体的なコンテンツは入れず、構造のみを定義する。

```
Template: Dashboard
  ├── Header（NavigationBar）
  ├── Sidebar（NavigationList）
  └── Main Content Area
        ├── PageHeader
        ├── DataTable / Chart
        └── Pagination
```

### 4-2. 画面（Page）との違い

| 種類 | 内容 | 用途 |
|------|------|------|
| Template | 構造のみ・コンテンツなし | レイアウト確認・デベロッパー説明 |
| Page | 実際のコンテンツを含む | ステークホルダーレビュー・ユーザーテスト |

## 5. アイコノグラフィ

### 5-1. アイコン管理指針

- ベクターデータ（SVG）で管理
- Figmaコンポーネントとして登録し、Instance Swap Propertyで切替可能にする
- サイズバリアントを統一する（例: `16px` / `20px` / `24px`）
- 線幅はデザイントークン `border-width` と一致させるか、専用のアイコン線幅トークンを定義する

### 5-2. アイコンセットの選定

外部アイコンライブラリを使う場合は以下を確認する:

- ライセンス形式（商用利用可否）
- スタイルの一貫性（Outlined / Filled / Rounded）
- SVGフォーマットでのダウンロード可否
- Figmaプラグインによるインポート対応

## 6. エレベーションとレイヤー

UIのレイヤー構造は視覚的な奥行きを示す。エレベーショントークン（`elevation/0`〜`elevation/4`）をコンポーネントのシャドウに適用して一貫性を保つ。

| エレベーション | 代表的なコンポーネント |
|-------------|-------------------|
| 0 | フラットなカード・インプット |
| 1 | ホバー時のカード・チップ |
| 2 | ドロップダウン・ツールチップ・ポップオーバー |
| 3 | モーダル・ダイアログ |
| 4 | スナックバー・トースト通知 |

エレベーションが高いほど手前に表示されるという視覚的な一貫性を維持する。

## 7. ガイドライン作りの3原則

わかりやすいガイドラインはチームやプロジェクトによって異なる。利用メンバーの習熟度・チームの規模・プロジェクト固有の視点を考慮したうえで、どんなデザインシステムにも共通する基本的な整備方法を押さえておこう。

### 7-1. 取り扱い方法を明記する

デザインシステムの冒頭に**取り扱い説明書**ページを用意する。記載すべき内容は以下のとおり。

- **目的と使用方法**: だれが・どのように利用するのかを明確化
- **作られた背景・きっかけ**: 組織内外の関係者にデザインシステムの存在意義を伝える
- **公開範囲**: チーム内のみか、外部パートナーにも閲覧させるかを定義

Figmaで整備する場合は、「ページ」機能の1ページを取り扱い説明書専用にする。新規参画者がどう活用すべきかをすぐ把握できる導線になる。

### 7-2. アクセスしやすく構成する

ドキュメントやリソースは構成要素ごとにまとめ、初見でも目的の情報にすぐたどり着けるようにする。推奨ページ構成の例:

| ページ | 内容 |
|-------|------|
| 取り扱い説明書 | 目的・使用方法・背景 |
| 理念 | ブランド方針・デザイン原則 |
| デザインコンセプト | ビジュアル方向性 |
| スタイルガイド | カラー・タイポグラフィ・スペーシング等 |
| コンポーネントライブラリ | UIコンポーネント一覧 |
| アセットとリソース | アイコン・イラスト・素材 |
| ワークフロー | 運用フロー・更新プロセス |

スタイルガイドやコンポーネントライブラリ内でも、カテゴリごとにフレームを分けて更新日を記載すると、アップデート箇所が一覧でわかりやすくなる。FigmaではAuto Layoutを活用すると更新も容易になる。

### 7-3. 設計思想を伝える

デザインシステムの方針（堅牢さ vs 柔軟さ）はチームによって異なる。一貫性を維持するには、メンバー全員が方針を正しく理解して使う必要がある。ガイドラインを整備する際のポイント:

**UIパーツのドキュメント化**
- 分類の概要・デザイントークンの説明・使用用途をドキュメントに残す
- Figmaであればデータ内にドキュメントを含められるため、デザインシステム利用時に同時確認できる

**正誤例の提示**
- 注意すべき使用方法には正例・誤例を併記する
- 誤った使用方法に対する代替案がある場合は記載する（より丁寧でわかりやすいガイドラインになる）

**命名の統一**
- 各パーツの名称をガイドラインに記載して統一することで、チーム内の認識のズレを防ぐ
- 未定義・暫定的な定義がある場合は、ガイドライン上での扱い方をあらかじめ決めておく

**変更履歴の記録**
- 外部参照をもとに定義したパーツは参照元を記載（更新時・新規参加者の確認のため）
- 非推奨になったパーツやルールがある場合も、その経緯を残す
