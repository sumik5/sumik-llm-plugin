# 進化的アーキテクチャとマイクロサービス入門

## 進化的アーキテクチャの考え方

### アーキテクチャは「スナップショット」である

ソフトウェアアーキテクチャは静的な設計図ではなく、**現時点での理解の最善解**を表すスナップショットである。

```
❌ 誤解：アーキテクチャは最初に完璧に設計すべきもの
✅ 正解：アーキテクチャはシステムと共に継続的に進化するもの
```

この認識の転換が、硬直したシステムと適応性の高いシステムの違いを生む。

### なぜ進化的アプローチが必要か

| 問題 | Big Upfront Design | 進化的アプローチ |
|------|--------------------|----------------|
| 要件の変化 | 再設計コストが高騰する | 小ステップで継続的に調整 |
| 知識の不足 | 誤った前提に基づいた設計 | 学習に基づいてアーキテクチャを更新 |
| 予測不能な変更 | 設計が陳腐化・固定化 | 柔軟性を維持しながら対応 |
| 過剰設計 | YAGNI違反・複雑性の増加 | 現在の要件に集中 |

## インクリメンタル開発の実践

### 作業方法の原則

**変更しやすい状態を保つこと**が最優先。これにより、誤りを修正し実験できる自由が生まれる。

| 原則 | DO | DON'T |
|------|----|-------|
| **小ステップで作業する** | 現在の理解に基づいて最小限の変更を加え、フィットを継続的に評価する | 一度に大きな変更を加え、後から修正しようとする |
| **インクリメンタルな設計** | 現在の要件を満たすシンプルな設計から始め、新要件が出たら段階的に拡張する | 将来のすべての要件を予測した完全な設計を最初から作ろうとする |
| **マインドセット** | 自分のアイデアや設計決定に疑問を持ち、欠点やより良い代替案を探し続ける | 初期設計に執着し、批判に防衛的になる |

### スケーラビリティ・保守性・セキュリティの扱い

初日からすべての品質特性を完璧にしようとしない。代わりに、**段階的に改善することを意図して設計する**。

```
Day 1: 現在の要件を満たす最小限の設計
Day N: 実際の問題が発生したタイミングで対処
```

## EAテクニック

不確実な部分には**抽象化をプレースホルダー**として使用する。これにより、確信のある部分で先に進みながら、不確実性を局所化できる。

### テクニック一覧

#### 1. 関心の分離・モジュール性の強調

| | DO | DON'T |
|--|----|-------|
| **設計** | システムを明確な責務を持つ独立したモジュールに分割し、インターフェースと抽象化でコンポーネント間のやり取りを管理する | 異なるシステム部分を密結合させる。相互依存はシステムを壊れやすくし変更を困難にする |

#### 2. テスト可能性の優先

| | DO | DON'T |
|--|----|-------|
| **設計** | テストしやすいようにシステムを設計する。TDDなどのプラクティスを使ってテスト可能なユニットにコードを整理する。テスト可能性はモジュール性・凝集性・関心の分離・抽象化・疎結合を促進する | テストを後回しにしたり軽視したりする。テスト可能性の欠如は設計上の欠陥を隠し、脆いアーキテクチャを招く |

> **重要**: テスト可能性が高いシステムは、良いアーキテクチャの副産物として現れることが多い。

#### 3. コンテキストへの適応

| | DO | DON'T |
|--|----|-------|
| **判断** | 「万能なアーキテクチャ」は存在しないことを認識する。最善のアプローチはプロジェクト固有の要件・目標・制約に依存する | アーキテクチャパターンやトレンドを、それがプロジェクトに適しているかを考慮せずに盲目的に従う |

#### 4. 変化と不確実性を前提とする

システムは変化し、新要件は必ず出てくる。この前提に基づいてアーキテクチャを設計する。

**ホワイトボードモデル（Tourist Map）の活用**:
- システムのキーとなるアーキテクチャを表す高レベルの概要を維持する
- **意図的にシンプル**にする——チームの誰もがそらで再現できるレベル
- 細部を詰め込みすぎない（モデルが壊れやすくなり頻繁に変更が必要になる）
- 十分に正確だが、ある程度の変動を許容できる抽象度を保つ

| | DO | DON'T |
|--|----|-------|
| **変化への対応** | 柔軟で適応可能なアーキテクチャを設計し、リファクタリングと拡張を容易にする | すべての将来要件を予測しようとして過剰設計する（不必要な複雑性を招く） |

**Big Upfront Designを避ける**: 最初にすべての詳細を計画することに時間をかけすぎない。代わりに、広い設計から始め、フィードバックと学習に基づいてインクリメンタルに洗練させていく。

#### 5. 継続的な学習と改善

| | DO | DON'T |
|--|----|-------|
| **継続改善** | 新たな知識・新興のベストプラクティス・ステークホルダーからのフィードバックに対してアーキテクチャを定期的に評価する | アーキテクチャを「完成した」と考えることは避ける。現状維持は停滞と改善機会の喪失につながる |

> アーキテクチャの制約はシステムの進化する仕様の一部として扱う。これを**自動化テスト**として具現化することで「まだ十分な速度か・セキュアか・耐障害性があるか」を各変更後に検証できる。

## マイクロサービス入門：採用判断

### マイクロサービスとは

マイクロサービスは**分散システムアーキテクチャ**であり、**分散開発アーキテクチャ**でもある。ソフトウェア開発を多数の小さな独立チームに分散させるため、開発組織を非常にスケールしやすくなる。

しかし、単なるサービス分割以上の複雑さを持つ。サービスを互いに独立してリリースできるレベルまでに疎結合させる高い設計スキルが前提条件となる。

### マイクロサービスの3つの核心特性

| 特性 | 内容 |
|------|------|
| **Small（小さい）** | 1〜2週間で書き直せる規模。1つのタスクに集中してうまくこなす。大きなシステムを任意に分割するのではなく、問題ドメインに沿った自律的なサービスを作る |
| **Independently Deployable（独立デプロイ）** | 疎結合で他のサービスへの依存なしにデプロイできる。これにより各チームが並行して進捗できる。ただしすべてを一緒にビルド・テスト・デプロイするわけではないためコストが生じる |
| **Bounded Context対応** | 各サービスをBounded Contextに対応させることで、自律性と独立デプロイを実現しやすくなる（詳細は [INSTRUCTIONS.md](INSTRUCTIONS.md) のDDD統合セクション参照） |

### 採用判断基準

```
Q1: 開発チームを独立してスケールさせる必要があるか？
  No → モノリスで十分（CA適用でコードは綺麗に保てる）
  Yes ↓

Q2: 独立デプロイの高い設計スキルがチームにあるか？
  No → まずモノリスで設計スキルを磨く
  Yes ↓

Q3: サービス境界を安定的に定義できる見通しがあるか？
  No → モジュラーモノリスから始めてBounded Contextを実験する
  Yes ↓

→ マイクロサービスへの移行を検討する
```

| シグナル | 推奨アプローチ |
|---------|--------------|
| チーム 1〜5人 | モノリス + Clean Architecture |
| チーム 5〜20人 | モジュラーモノリス |
| チーム 20人以上 + 独立リリース要件 | マイクロサービス |
| ドメイン境界が不明確 | モノリスで探索・学習してから分割 |

## マイクロサービス：開始方法

### よくある間違い：最初から別リポジトリに分割する

```
❌ よくある誤ったスタート
→ 各サービスに別リポジトリをすぐに作成する
→ 問題: サービス・インターフェース・メッセージングを事前に包括的に定義することはほぼ不可能
→ 結果: 学習・進化がはるかに複雑で危険になる
```

### 推奨する開始方法

```
1. シングルリポジトリから始める
   ↓
2. 問題を深く考える。異なるモデルを試す
   ↓
3. サービスの最初のベストゲスを作成し、候補概念を特定する
   ↓
4. Event Stormingでドメイン探索を行い、Bounded Contextを発見する
   ↓
5. 実際のインタラクションをコーディングして何が本当に機能するかを確かめる（抽象化を洗練する）
   ↓
6. 単一のDeployment Pipelineでコードと複数サービス間のインタラクションをテストする（速いフィードバック）
   ↓
7. 抽象化の設計が安定してきたら初めて別リポジトリを作成する
```

### Bounded Contextの特定

同じ概念でも異なるコンテキストでは別のモデルになることに注意する：

```
例: 書籍販売システムの「book」
  販売コンテキスト: 価格・表紙画像・レビュー
  物流コンテキスト: 重量・寸法・在庫場所

→ 同じ現実の書籍を指していても、別のBounded Context = 別のモデル
→ マイクロサービスをBounded Contextに合わせて設計する
```

**Event Storming**はこの創造的探索を始めるのに良い方法だが、アイデアを検証するために実際のコードも書くこと。

## マイクロサービス：メッセージング設計

### 疎結合メッセージングの原則

サービス間通信には必ず何らかの結合が生じる。メッセージを内部実装詳細と密接に結びつけないことでこの結合を最小化する。

### Ports & Adapters パターン

**メッセージを別のBounded Contextとして扱う**——サービスそのものとは別のものとして扱う。

```
Service A (実装)
    ↕ PORT（インターフェース）
  ADAPTER（変換層）
    ↕ MESSAGE（境界コンテキスト）
  ADAPTER（変換層）
    ↕ PORT（インターフェース）
Service B (実装)
```

入力を変換し出力を組み立てるコードをサービスの内部ロジックとは別にする。これによって：
- メッセージはサービス実装とより独立して変更できる
- 異なる変更率で進化できる
- 同じサービス内に複数バージョンのメッセージに対応する複数のトランスレーターを持つことも可能
- メッセージの消費者と生産者をより独立してデプロイできる

### メッセージング設計のルール

| ルール | 内容 |
|--------|------|
| ✅ **反復的に会話を設計する** | 初期段階では全部まとめて高速に評価・学習する |
| ✅ **Bounded Context遷移時は必ず変換する** | 境界を越える概念は必ず変換する。変換なしで共有すると密結合になりすべての変更で障害が発生する |
| ✅ **技術的なメッセージを避ける** | メッセージが表す「会話」を問題ドメインのレベルでモデル化する（非技術者が理解できる水準） |
| ✅ **境界を特別に扱う** | サービス間の境界はシステム設計の重要部分。デプロイ独立性と疎結合を維持するために特別な注意を払う |
| ✅ **Contract Testingを検討する** | サービス間の会話における破壊的変更を検出するために利用する。実際のサービス間インタラクションではなく、コントラクトの期待値（メッセージの使われ方の期待）を検証する |

### Contract Testingの位置づけ

```
❌ Integration Test（実サービス間の実際のインタラクションをテスト）
✅ Contract Test（メッセージのコントラクト期待値を検証）
```

よく抽象化されたメッセージはContract Testingを大幅に助ける。

## 判断フローまとめ

### 進化的アーキテクチャ適用フロー

```
現在のアーキテクチャが問題を抱えているか？
  No → 継続的改善のため定期レビューを計画
  Yes ↓

問題は要件変更への対応困難か？
  Yes → 関心の分離・モジュール性を強化する
  No ↓

問題はテストしにくいことか？
  Yes → テスト可能性を設計に組み込む（TDD・依存性逆転）
  No ↓

問題は変化の予測不可能性か？
  Yes → ホワイトボードモデルを整備 + Big Upfront Designを削減
```

### マイクロサービス移行フロー

```
大規模チームのスケーリングが必要か？
  No → モノリス + Clean Architecture で十分
  Yes ↓

Bounded Context境界は明確か？
  No → Event Stormingでドメイン探索 → シングルリポで試行
  Yes ↓

チームに独立デプロイの設計スキルがあるか？
  No → モジュラーモノリスで段階的にスキルを獲得する
  Yes ↓

単一パイプラインでの学習フェーズを経たか？
  No → シングルリポ + 単一パイプラインで最初の設計を検証する
  Yes ↓

→ 抽象化が安定したら個別リポジトリへ移行
```
