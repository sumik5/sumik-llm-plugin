# コンポーネント設計原則

## コンポーネントとは

コンポーネントはデプロイの単位。言語によってはライブラリ、パッケージ、モジュール、`jar`、`dll`、gem 等に対応する。

---

## コンポーネント凝集性の3原則

「何をコンポーネントに含めるか」を決める原則。

### REP：Reuse/Release Equivalence Principle（再利用・リリース等価原則）

> **リリースの粒度 = 再利用の粒度**

コンポーネントに含まれるクラスやモジュールは「一緒にリリースできる」凝集したグループでなければならない。リリース管理されていないものは再利用できない。

**チェック：**
- このコンポーネントのユーザーは変更通知を必要としているか？
- リリースドキュメント（CHANGELOG等）が意味をなすか？
- 含まれるすべてのモジュールが同じ目的を持つか？

### CCP：Common Closure Principle（共通閉鎖原則）

> **同じ理由・同じタイミングで変わるクラスを同じコンポーネントに集める。異なる理由・タイミングで変わるクラスは別コンポーネントに分ける。**

SRP のコンポーネント版。変更が単一コンポーネントに閉じていれば、そのコンポーネントだけ再デプロイすれば済む。

**実践例：**
```
❌ 悪い：UI ロジックと DB スキーマが同一コンポーネント
✅ 良い：UI 変更理由と DB 変更理由で別コンポーネントに分離

UI コンポーネント      DB コンポーネント
  [UserView]    │     [UserRepository]
  [OrderForm]   │     [OrderRepository]
    変更理由: UI仕様   変更理由: DBスキーマ
```

**OCP との関係：** CCP は OCP の「閉鎖」に対応。同じ種類の変更に対して閉じているクラスを集める。

### CRP：Common Reuse Principle（共通再利用原則）

> **一緒に再利用されるクラスをコンポーネントに集める。互いに強く結合していないクラスは同じコンポーネントに置かない。**

ISP のコンポーネント版。依存するコンポーネントの不要なクラスの変更でも再コンパイル・再デプロイが強制される。

**「一緒に使われるか」チェック：**
- Container クラスと Iterator クラス → 同コンポーネント（常に一緒に使う）
- HTTP ライブラリと PDF ライブラリ → 別コンポーネント（独立に使う）

## テンションダイアグラム

```
           REP（リリース単位）
           /              \
          /                \
[不要なリリースが        [不要なコンポーネントに
 多数含まれる]             依存させられる]
        /                    \
      CCP ─────────────────── CRP
（変更の局所化）           （再利用性）
      [変更時に多数のコンポーネントに影響]
```

| フェーズ | 重視する原則 | 理由 |
|---------|------------|------|
| 開発初期 | CCP > CRP > REP | 変更頻度が高く、再利用より保守性が重要 |
| 成熟期 | REP ≈ CRP > CCP | ライブラリ化・再利用が重要になる |
| ライブラリ開発 | REP > CRP > CCP | 外部ユーザーへのリリース管理が最重要 |

## コンポーネント結合の3原則

「コンポーネント間の関係」を決める原則。

### ADP：Acyclic Dependencies Principle（非循環依存原則）

> **コンポーネントの依存グラフに循環を許してはならない。**

循環依存の症状：
- 一つの変更が波及して複数コンポーネントのビルドが壊れる
- テスト実行時に大量の他コンポーネントが必要になる
- ビルド順序が決まらない

```
❌ 循環依存の例
A → B → C → A  （A,B,C は実質的に1コンポーネント）

✅ DAG（有向非循環グラフ）の例
Entities → Interactors → Controllers → Main
                      ↘ Presenters ↗
```

**循環依存の解消法：**

方法1：**DIP を適用する**
```
A → [Interface I] ← C   （C が I を実装）
A は I に依存。C は I に依存しなくなる → 循環解消
```

方法2：**新コンポーネントを作る**
```
A と C が依存している共通クラスを新コンポーネント NEW に移動
A → NEW
C → NEW
（A と C の循環が消える）
```

### SDP：Stable Dependencies Principle（安定依存原則）

> **安定した（変化しにくい）コンポーネントに依存せよ。不安定な（変化しやすい）コンポーネントに依存するな。**

**安定度（I：Instability）の計算：**

```
I = Fan-out / (Fan-in + Fan-out)

Fan-in  = このコンポーネントに依存している他コンポーネントの数
Fan-out = このコンポーネントが依存している他コンポーネントの数

I = 0: 最安定（誰にも依存していないが多くから依存される）
I = 1: 最不安定（誰も依存しないが多くに依存している）
```

**SDP チェック：**
- 依存の向きが常に安定側（I が低い方向）を指しているか？
- `I(依存元) > I(依存先)` が成立しているか？

```
❌ 不安定な A に安定な B が依存
B（I=0.1）→ A（I=0.9）

✅ 安定した A に不安定な B が依存
B（I=0.9）→ A（I=0.1）
```

### SAP：Stable Abstractions Principle（安定抽象原則）

> **コンポーネントは安定しているほど抽象的であるべき。**

安定したコンポーネント（I≈0）は変更しにくい。変更しにくいものが具体実装だと、拡張が困難になる。解決策：安定コンポーネントは**抽象クラス・インターフェース**で構成する。

**抽象度（A：Abstractness）の計算：**

```
A = 抽象クラス数 / 全クラス数

A = 0: 完全に具体的（コンクリート）
A = 1: 完全に抽象的
```

## 主系列（Main Sequence）ライン

`I + A = 1` が理想ライン。

| ゾーン | 状態 | 問題 |
|-------|------|------|
| **苦痛ゾーン** A≈0, I≈0 | 安定だが具体的 | 変更困難・拡張困難 |
| **無駄ゾーン** A≈1, I≈1 | 抽象だが不安定 | 誰も使わない抽象コンポーネント |
| **主系列付近** | 理想状態 | バランス取れた設計 |

```
A
1 |     .
  |   .   .
  | .       .
0 |____________ I
  0             1
  苦痛ゾーン: (0,0)付近
  無駄ゾーン: (1,1)付近
  理想: A+I=1 のライン上
```

## 実践：依存グラフの管理

### 定期的な依存分析

```bash
# TypeScript プロジェクトの場合
# dependency-cruiser 等でコンポーネント依存グラフを可視化
npx depcruise --include-only "^src" --output-type dot src | dot -T svg > dependency-graph.svg

# Python プロジェクトの場合
# pydeps や import-graph で依存関係を可視化
pydeps mypackage --show-deps
```

### コンポーネント境界のレビューポイント

- [ ] 循環依存がないか（ツールで定期チェック）
- [ ] 変化しやすいコンポーネントに安定コンポーネントが依存していないか
- [ ] 安定コンポーネントは抽象クラス/インターフェース中心か
- [ ] コンポーネントの凝集性（変更理由が一つか）は維持されているか
