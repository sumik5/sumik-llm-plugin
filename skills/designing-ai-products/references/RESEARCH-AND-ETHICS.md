## 1. MUSE/Disciplined Brainstorming（規律あるブレインストーミング）

### Bookending（ブックエンディング）とは

「ブックエンディング」は、アイデアを特定の設計方向に沿って可能な限り遠くまで追求し、その「ブックエンド（本棚の端）」でスケッチを描く技法。

- **本棚の比喩**: 特定カテゴリの本が並ぶ棚の末端に新しい本を追加するイメージ
- **目的**: 単一アイデアへのアンカリングを避け、複数の設計方向を素早く探索する
- **実施形態**: 一人でも可能だが、小規模なタイガーチームで行うと効果的

### Bookendingの手順

| ステップ | 内容 |
|---------|------|
| 1. 方向設定 | 既存製品・デザインパターンを参考に設計方向を選ぶ |
| 2. 追求 | その方向のアイデアを「これ以上新しいことがない」と感じるまで展開 |
| 3. スケッチ | ブックエンドとなる設計案を簡単にスケッチ（紙で十分）|
| 4. 離脱の判断 | 色やラベルの議論に入ったら「転換タイミング」のサイン |
| 5. ピボット | 全く異なる方向に切り替え、ステップ1から繰り返す |

### 「転換タイミング」の見分け方

> アイデアが完璧なのは「これ以上追加するものがないとき」ではなく、「これ以上削除するものがないとき」— Saint-Exupéry

パネルの位置（左右・下部）の話や、ボタンの色・テキストの議論が始まったら転換のサイン。

### ブレインストーミングのシードアイデア

新しい設計方向を見つけるためのヒント：

- 類似製品・インターフェースで同じ問題がどう解決されているか
- 有名ブランド（Apple、Facebook、Amazon）ならどう解決するか
- SF・神話的キャラクター（IronMan/Jarvis、スタートレック技術）の視点
- 異なるモダリティ（おもちゃ、キオスク、ウェアラブル、コマンドライン）
- スティッキーノートを再利用してアイデアを素早くスケッチ・移動

## 2. AI-Inclusive User-Centered Design Process（AI包括型UCD）

### 従来UXプロセスの限界

**線形プロセス図**（定義→発想→プロトタイプ→テスト→リリース）は産業時代の産物：
- 「アイデア担当者」と「実装担当者」が分離している前提
- AIでは技術が100%確定していないため、この前提が崩壊

### UI + AI + Data トライアングル

AIプロジェクトで「最初に何をするか」に答えはない。**3つを同時に並行して反復する**：

```
        UI（インターフェース）
            ↙        ↘
  AI（モデル）  ↔  Data（データ）
            ↖        ↗
          顧客（中心）
```

- **UI反復** → AIモデルへの新要件が生まれる
- **AIスパイク** → 新しいデータが必要になる
- **データ制約** → AIモデルの実現可能性に影響し、UX設計の修正が必要になる

### Spike（スパイク）とは

アジャイルにおけるスパイクは「特定のアイデアが実現可能かを示す超軽量な概念実証プロジェクト」：

- **形態**: Pythonノートブック形式の簡易I/O「体験」が最適
- **目的**: AI駆動システムの最も基本的なPoC。問題定義を素早く固め、解決策を証明する
- **重要原則**: 本番コードとは正反対の「超軽量・粗削り」が正しい

### AIプロジェクトにおけるUXの役割変化

UXの役割は伝統的に3つだった：

1. 顧客のバリア除去（ユーザビリティテスト）
2. イノベーションと競合分析
3. 顧客・ビジネス・技術を繋ぐ「接着剤」

AI時代に最も重要になるのは**「接着剤」の役割**。理由：AIという新技術は未知で予測不能なため、技術自体を研究するUXが不可欠。

### 実践的なAI-Inclusive UCDの進め方

1. **紙プロトタイプ + Pythonノートブック**の組み合わせが「現代版ウィザード・オブ・オズ」
2. 評価者にUI（紙プロトタイプ）を見せてユースケースを理解させる
3. 評価者が「AIに聞きたいこと」をブレインストーミング
4. Pythonノートブックの基本AIモデルで実際に答えを試す
5. データについて評価者と一緒に議論（不足データ、学習データの入手方法）

### 最終ハンドオフの特徴

AI-Inclusive UCDにおける開発へのハンドオフは明確な区切りとして存在：
- 探索的な「スパイク開発」と本番化のための開発を分離
- ハンドオフ時には開発者を「顧客」として扱い、実装を円滑にする詳細仕様を提供

## 3. AIリサーチにおける技法の分類

### 自動化される技法

| 技法 | 自動化の状況 |
|-----|------------|
| ルーティンユーザビリティスタディ | スクリプト作成〜初期プロトタイプ生成まで自動化 |
| ルーティンNPS・サーベイ | 質問作成〜データ分析〜提言レポートまで対応可能 |
| 研究データの収集・整理 | タグ付け・ラベリング・録画管理・アフィニティ図等が自動化 |
| 定性・定量洞察の三角測量 | 複数年にわたる洞察DBへの自然言語クエリが可能に |

### AI拡張（オーグメント）される技法

| 技法 | 変化の方向性 |
|-----|------------|
| 競合分析 | AIが映像・文書から機能を逆算、数週間→数時間・数分に短縮 |
| 新規ユースケース特定 | 競合分析・市場非効率性に基づくビジネス機会の自動指摘 |
| RITEスタディ | 抜本的かつ永続的に変革（次節で詳述） |

### 価値が高まる技法（AIに代替困難）

| 技法 | 理由 |
|-----|------|
| **コアスキル**（合意形成・交渉・共感） | 高度に専門化した職種間の協力が不可欠 |
| ワークショップファシリテーション | 相反する意見からコンセンサスを導く能力 |
| フォーマティブリサーチ・フィールドスタディ | ロボット視覚や多感覚統合による洞察生成は未実現 |
| ビジョンプロトタイピング | 人間の共感から生まれる創造的スパークの再現は困難 |
| エグゼクティブ戦略補佐 | 技術・ビジネス・市場・消費者需要・価値観の多角的分析 |

### AI「ブルシット」（死に体のアイデア）

| NG技法 | 問題点 |
|-------|--------|
| AIによる新規ビジネスアイデア代替 | 経験・共感・人間理解の代わりにはならない |
| AIヒューリスティクス分析による設計代替 | ヒューリスティクスは「問い」の指針に過ぎず、答えではない |
| **「Synthetic Users」** | ロボットを設計対象とした製品になる（次節で詳述） |
| AIペルソナ生成 | 最重要プロセス（チームの合意形成・議論・教育）をスキップ |

## 4. Synthetic Users（合成ユーザー）批判

### 核心的問題

> AIで本当のユーザー調査を代替することは「ロボットのために製品を設計する」ことを保証する。

- AIはユーザーの行動を「予測」できるが、実際の顧客の反応は予測できない
- AIは「典型的な人間」の行動を模倣するが、実際のユーザーグループは独自の背景・ニーズ・動機を持つ
- **最も重要な点**: 「LLMは製品を買えない」

### AIが苦手なリサーチスキル

| スキル | AIの限界 |
|-------|---------|
| コンテキスト理解 | 状況の含意（親同伴でのティーン、業界の忖度等）を見落とす |
| 観察 | 言動不一致（「簡単」と言いながら眉をひそめる等）を検出できない |
| アハモーメント | 予想外の発見から仮説を立て、オンザフライでスクリプトを変える能力 |

### 「合成ユーザー」の唯一有効な活用例

- **研究者トレーニング「Kobayashi Maru」**: 経験不足の研究者に対し、AIが「試験を絶対に失敗させようとする意地悪な顧客」を演じる訓練ツール

### AIリサーチで有用なタスク

| タスク | 活用効果 |
|-------|---------|
| 包括的文献レビュー | プロジェクト開始前に大量の先行研究を素早く整理 |
| ビデオインタビュー書き起こし蒸留 | 大量の文字データからテーマを抽出 |
| サーベイ自由記述のテーマ整理 | 数千件の回答をカテゴライズ |
| 日記研究の画像スキャン・分類 | 提出画像の自動分類 |
| サーベイ設問の代替表現生成 | 設問のバリエーション作成 |

## 5. RITE Methodology（ラピッド反復テスト・評価法）

### RITE vs. Usability Test: 3つの根本的違い

| 観点 | Usability Test | RITE Study |
|-----|---------------|-----------|
| **位置づけ** | プロセスの後半で行うQA的検証 | デザインプロセスのコアに組み込まれた継続的学習 |
| **プロトタイプ** | 高忠実度の洗練されたプロトタイプ（変更困難） | 最もシンプルな粗削りプロトタイプ（変更容易） |
| **成果物** | ユーザビリティレポート（問題と推奨事項） | 改善されたデザイン解決策そのもの |

> 粗削りなプロトタイプは変更を招く。洗練されたプロトタイプは変更を禁じる。

### RITEスタディの構造

- **参加者数**: 9〜12名、3〜4ラウンドに分けて各3〜4名
- **セッション時間**: 1セッション10〜15分が理想（2時間セッションは不要）
- **ラウンド間**: プロトタイプを更新し、前ラウンドで発見した問題を修正
- **プロトタイプ形式**: スティッキーノート、シンプルなFigmaクリックスルー ＋ Pythonノートブック（AI実出力）

### RITE実施手順（ステップバイステップ）

1. **ユースケース設定**: 最初から1〜3スクリーンで十分。ウェルカムフローではなく最重要ユースケースに集中
2. **シーン設定の台本例**:
   > 「Life Copilotアプリをインストール・パーソナライズ済みだと想像してください。今はお昼時で、何を食べるか決めようとしています。アプリを開いたとき、このスクリーンが見えます。最初の印象を教えてください。」
3. **オープン誘導質問**: 「次に何をしたいですか？」→ 予め決めたフローへの誘導を避ける
4. **リアルタイム共創**: スクリーンが不足したら、その場で評価者と一緒にスケッチ
5. **ミリオンダラー質問**: 「このアプリにお金を払いますか？いくら払いますか？」
6. **偏りの防止**: 複数デザイン案を提示する場合、必ずランダム順で見せる（初見効果バイアス対策）

### ラウンドを重ねた後の成果物

| 成果 | 内容 |
|-----|------|
| ユースケース検証 | 一次ユースケースの妥当性（またはピボット先）確認 |
| デザイン選択肢の絞り込み | 3〜5案を1〜2案に絞り込み |
| 追加方向性 | ユーザーが自ら提案・触発した新しい設計方向 |
| マネタイズ検証 | 支払い意欲と価格帯の確認 |
| デジタルツイン情報 | AIに必要なデータの種類・期待される出力の把握 |

### 完全なRITEスタディ（2〜3週間）の成果物

- [ ] 単一最良デザイン方向の検証
- [ ] 主要スクリーンの粗削りフォーマットでの完成
- [ ] 必須機能・入力・ボタンの特定・ラベリング・配置
- [ ] PM・ユーザーとのマネタイズ戦略検証
- [ ] ユーザー・開発リードとのMVPスコープ検証
- [ ] データサイエンスリードとのデジタルツイン・AIモデル設計検証

### RITE スタディの付随的効果

- アジャイル/スクラムとの相性が良い（ソリューション志向のため非対立的）
- 4-in-the-boxチーム（UX・PM・Dev・AIリード）のリアルタイム共同プロセス
- 冗長なレポートではなく「プロセスの改善」が成果となる

### AI-Assisted RITE（未来の姿）

近い将来、AIによる支援で実現する変化：

- 参加者のフィードバックをリアルタイムで検出し、オンザフライでデザインフロー変種を生成
- 研究者が次に見せるバージョンをAI生成の複数案から選択
- 研究セッション中に直接、**Figmaをスキップして**動作するReactコードへ変換
- 1日のセッションで及第点のデザイン解決策に到達可能

## 6. AIバイアス認識

### 基本原則

> **すべてのAIはバイアスを持つ。** バイアスが体験にどう影響するかを常に考えること。

### サバイバーバイアスの理解

**第二次大戦の飛行機の例**：

- エンジニアは「帰還した飛行機」の弾痕を分析し、弾痕が多い箇所の装甲を強化しようとした
- 統計学者Abraham Waldの質問：「なぜエンジン・操縦室・尾翼には弾痕がないのか？」
- 答え：その部分に当たった飛行機は**帰還できなかった**からデータに含まれない

**AIへの応用**: AIが生成する結果を見たとき、まず**「そこにないもの」を探す**。それがバイアスの核心。

### 画像生成AIのバイアス事例と対策

| クエリ | バイアスの現れ方 | 対策クエリ例 |
|-------|---------------|-----------|
| "biologist" | 白人男性25:女性1（実際は女性54%） | "black trans biologist" |
| "basketball player" | 黒人男性が大多数 | "Indian woman basketball player" |
| "depressed person" | 白人女性が大多数 | "depressed older Asian man" |

### バイアス検出チェックリスト

- [ ] AIが出力した結果に**誰がいないか**を確認した
- [ ] 労働統計・人口統計と比較して表現を検証した
- [ ] 72以上ある性別アイデンティティへの配慮を検討した
- [ ] クエリに多様性を示すキーワードを追加してテストした
- [ ] 別のAIモデルを「審査員」として使い、出力の公平性を監査した

### Algorithm Aversion（アルゴリズム忌避）

AIが出力する結果に対する2種類の認知バイアス：

| バイアス | 内容 | 危険性 |
|---------|-----|--------|
| **過信（Overreliance）** | 「機械が言うなら正しい」という中立性の幻想 | エラーが下流でより高コストになる |
| **忌避（Underreliance）** | 「見慣れない」だけでAI提案を拒絶 | 価値ある革新を見逃す |
| **デスキリング** | AI依存によって人間のスキルが徐々に失われる | 長期的な人間の能力低下 |

## 7. AI Ethics（AI倫理）

### AI Humanifesto: 5つの中核概念

プロ人間型AI製品設計のためのガイドフレームワーク：

#### 1. Control（制御）― ガバナンス・説明責任

| 側面 | 要件 |
|-----|------|
| ユーザー自律性 | AI機能の変更・オプトアウト能力を提供 |
| アルゴリズム透明性 | 意思決定プロセスの明確な説明 |
| データ管理 | 保存・処理・共有方法の開示、削除権の保障 |

#### 2. Trust（信頼）― 透明性・信頼性・説明可能性

| 側面 | 要件 |
|-----|------|
| 信頼性・一貫性 | 期待通りのパフォーマンスの継続的提供 |
| エラーの所有権 | 誤り発生時の透明な報告と解決プロセス |
| データ責任 | 倫理的データ取り扱いと明確な説明 |
| 説明責任 | AIが害を与えた場合の組織的責任と救済手段 |

#### 3. Diversity（多様性）― セキュリティ・プライバシー・ロバスト性

| 側面 | 要件 |
|-----|------|
| 包括的データ | 多様な人間体験を反映するデータセット |
| バイアス・欠落の認識 | 定期的な評価と欠落集団の特定・修正 |
| 社会との共進化 | 多様性概念の変化に合わせてモデルを更新 |
| 障壁の除去 | リテラシー・年齢・社会経済的格差を超えた対応 |

#### 4. Safety（安全性）― 包摂・公平性

| 側面 | 要件 |
|-----|------|
| プライバシー・データ保護 | 個人データの安全な取り扱いと侵害防止 |
| 同意と教育 | リスク・メリット・バイアスの明確な説明 |
| 身体・感情的ウェルビーイング | 依存性・孤立・精神的ストレスの回避 |
| リスク管理 | 潜在的危害の特定と予防的安全策の内包 |

#### 5. Balance（バランス）― 持続可能性・調和・人間エンパワーメント

| 側面 | 要件 |
|-----|------|
| 人間エンパワーメント | AI拡張ではなく人間のポテンシャルを増幅 |
| 人間 vs. AI貢献の明示 | どの部分がAI生成かを明確に区別 |
| 効率 vs. 創造性 | 生産性最適化と人間の創造性・柔軟性のバランス |
| 環境影響 | エネルギー使用の最小化、不要なAI使用の回避 |
| 社会・人間関係 | 人間関係の強化、依存性行動の防止 |

### Practical Ethics: デザイナーのための実践的倫理

#### インセンティブの現実を理解する

- ビジネスモデルは設計のグリッドを形成する—これを理解した上で倫理を実践する
- 自分の上下・横の関係者のインセンティブを把握し、倫理的考慮をそれと整合させる
- 説明責任の原則: ユーザーがAIの意思決定を見られ、フィードバックを提供できる設計を行う

#### 倫理的影響の継続的テスト

- [ ] 多様なユーザーグループ・ステークホルダーとのテストを実施した
- [ ] Value Matrixにおける偽陽性・偽陰性の倫理的含意を評価した
- [ ] 会話エージェントに「解決不能な問題」でストレステストを行った
- [ ] 「All AI Is Biased」を念頭に多様なグループでのバイアスを確認した
- [ ] 別のAIモデルを審査員として出力の公平性を監査した

#### 環境影響への配慮

100語のメールをAIで送ることは、水1本消費または14個のLED電球を1時間点灯させることに相当。

- [ ] AIコールの最小化を検討した
- [ ] そもそもAIが必要かを問い直した
- [ ] 誰も使わないAI製品に貴重な資源を無駄遣いしないよう確認した

## 8. UXリサーチにおけるAI代替の問題

### 「Synthetic Users」批判

AIをユーザー代替として使用することへの批判：

| 問題 | 詳細 |
|-----|------|
| ロボット向け設計 | AIのみに話しかけると、ロボット向けシステムが生まれる |
| 文脈の欠如 | AI は人間固有の状況的ニュアンスを理解できない |
| 予測不能性の喪失 | 人間は非常に予測不能—それがリサーチの価値の源 |
| ターゲット特性の無視 | 「典型的人間」を模倣し、実際のユーザーグループの特性を反映しない |
| 最大のポイント | LLMは製品を「買えない」 |

> 「ユーザーリサーチをアウトソーシングするのはバケーションをアウトソーシングするようなもの。そのリサーチをAIに置き換えるのは…バケーションをロボット軍団にアウトソーシングするようなもの。」

## 9. Vision Prototyping（ビジョンプロトタイピング）

### 9つのベストプラクティス

| # | プラクティス | 内容 |
|---|-----------|------|
| 1 | **Play（遊び）** | プレイのフリをして不快な質問を投げかける。ユーモアと楽しさで聖域を揺さぶる |
| 2 | **シナリオ探索** | アイデアをできる限り先へ推し進め、違う方向でもやり直す（Bookending） |
| 3 | **エゴを脇に置く** | 政治的機転と敬意を示しながら、好奇心でライトに探索をリード |
| 4 | **現実的制約を前提に** | 利用可能なデータ・技術の制約を制約として受け入れ創造性の燃料に |
| 5 | **全能AIゲーム** | 「AIが全能だったら何ができる？」から始め、後でトレーニング方法を考える |
| 6 | **「何が価値を生むか？」を問い続ける** | 毎機会に問い、「情報の出所を追え」で価値の連鎖を辿る |
| 7 | **データを追え** | 「誰がモデルのトレーニングに必要なデータを持っているか？」を問う |
| 8 | **フィールドに出る** | 現場調査に代わるものはない。実際の移動・検査・ワークフローを観察する |
| 9 | **チームワーク** | 複数のSME・部門・顧客を集め、共創・参加型デザインを活用する |

### 7つの典型的ミス

| # | ミス | 説明 |
|---|-----|-----|
| 1 | **近すぎる目標** | ビジョンプロトタイプは1〜2年〜「永遠」が時間軸。2〜3スプリントは戦術プロトタイプ |
| 2 | **画面の羅列** | ユースケースを活用し、最終スクリーンまで顧客の課題解決を追いかける |
| 3 | **Figmaのみ** | 理想の提供形式は1〜2分のビデオ（ユースケース→画面説明→価値の明示） |
| 4 | **Lorem Ipsum** | コンテンツは重要。数字・名前・計測値をリアルに、AIツールを活用して現実的に |
| 5 | **全コーナーケースを網羅** | 1〜2ユースケースが目標。網羅性ではなくビジョンが重要 |
| 6 | **プロトタイプと実装の混同** | 夢は大きく、実装は小さく始める。最初のMVPをできるだけシンプルに |
| 7 | **結果への執着** | 85%のAIプロジェクトは失敗する。結果よりプロセスへのコミットを優先する |

## 10. AI時代のUXの将来

### 「AIは我々のために起きている、我々に対してではない」

- AI時代のUXデザイナーはかつてない機会・発見・成長・社会貢献の可能性を持つ
- 「4-in-the-box」モデル（PM・UX・Dev・データサイエンティスト/AIスペシャリスト）がスタンダードになる

### AI時代に価値が高まるUXスキル

| スキル | 説明 |
|-------|------|
| 合意形成・交渉 | AI時代もAIに代替不能。相反する意見の調整能力 |
| ビジョンプロトタイピング | 新製品の方向感覚を見つける人間の創造的スパーク |
| AI倫理・バイアス認識 | ミスインフォメーション・ディープフェイクへの対処 |
| エグゼクティブ戦略補佐 | 技術・ビジネス・市場・人間的価値の多角的合成 |
| コア「人間スキル」（共感・観察・創造的傾聴） | UX専門職の人間と機械を分かつ一線 |

### AIの最良の姿は「拡張知性（Augmented Intelligence）」

- **機械が得意なこと**: 大量計算・パターン認識
- **人間が得意なこと**: 共感・創造性・調和・喜び
- この交差点でUXが最も輝く

> AI is just too important to be left to data scientists and business people alone.
> — UXがAIの全概念・開発プロセスに関与する必要性
