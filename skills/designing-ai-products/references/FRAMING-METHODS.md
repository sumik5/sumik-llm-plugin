## 1. AIプロジェクト失敗パターン（5つのFail）

85%のAIプロジェクトが失敗する。技術的な失敗よりもUX設計・リサーチ・プロセスの失敗が原因となることが多い。

### 失敗パターン一覧

| Fail # | パターン名 | 内容 | 判断シグナル |
|--------|-----------|------|------------|
| **Fail #1** | 専門家をAIで代替しようとする | 訓練された現場専門家をAIが直接置き換える設計 | ウォーク → 専門家より高コストなら **Run** |
| **Fail #2** | コスト/ベネフィット分析の省略 | 予測の正誤ごとの金銭的影響を定量化しない | 偽陰性1件のコスト ＞ 年間利益 → **Run** |
| **Fail #3** | MLトレーニングデータ不足 | データがない、または装置ごとにカスタムAIが必要 | インストールごとにデータが異なる → **Run** |
| **Fail #4** | AIが答えている質問がズレている | 利益最大化ではなく「データサイエンス的に答えやすい問い」を選ぶ | 街灯の下で鍵を探す状態 → **Walk** |
| **Fail #5** | SMEだけに頼ってユーザーリサーチを省く | 現場を直接見ないため、重要な変数（「ガラス窓センサー」等）を見落とす | 1回も現地訪問なし → **Run** |

### 重要原則：「The Best AI Is No AI」

> ユーザーはAIが欲しいのではなく、特定の成果・問題解決が欲しい。AIはその手段に過ぎない。

- 非AI（ルールベース・決定木・線形回帰など）で解決できるなら、迷わずそちらを選ぶ
- 次善策は「退屈なAI（Boring AI）」—— シンプルなML手法から始めてベースラインを確立する

### Resilient Core の原則

AIコアは「失敗しても機能が完全停止しない」設計にする：

| 設計方針 | 例 |
|---------|-----|
| AIがあれば速い、なくても動く | オートコンプリート、レコメンデーション、プリフィル |
| ユーザーが意図を部分的に伝えられる | ファセット検索フィルター＋AIランキング |
| ビジネスロジックでAI出力をラップ | 入札価格の上限設定、スパムフィルターの例外ルール |

## 2. AI Storyboarding（5要素フロー）

ストーリーボードはAI駆動ユースケースを視覚化し、ギャップ・矛盾・非現実的な想定を早期発見するツール。

### なぜストーリーボードが必要か

- 文章で読むと成立しそうなユースケースも、視覚化すると「話がつながっていない」ことが露わになる
- 4〜6パネルが理想（簡潔さが重要）
- AIの存在は「もう一人の登場人物」として描く

### 5要素フロー

#### 要素①：Establishing Shot（場面設定）

| 役割 | ポイント |
|------|---------|
| 主人公と読者をストーリーの環境に引き込む | 最初のパネルに時間をかける価値がある |
| 「私はどこにいるのか？」を問いかけて描き始める | カフェ、病院、路上—— 具体的な場所が没入感を生む |

#### 要素②：Things（物）

- 建物・家具・ガジェット等の無生物
- 四角と丸で描ける（コンピューターも電話も同様）
- 描くのが難しければ「Things」に集中する

#### 要素③：People（人）

| スタイル | 特徴 |
|---------|------|
| 棒人間（推奨） | 速く、意味を損なわない。読者が自分を投影しやすい |
| ボックス型 | 少しリアルな表現に |
| ヒトデ型（Starfish） | ダイナミックなポーズに向く。スケッチノートアーティストに人気 |

> **品質は確信度に合わせる** — プロジェクトの初期段階ほど、荒削りな絵のほうがストーリーに集中できる

#### 要素④：Faces（顔・表情）

- スティックフィギュアの顔に **眉毛を追加** するだけで微妙なニュアンスが表現できる
- 眉で区別できる表情の例：

| 眉の向き | 表情 |
|---------|------|
| 外側上がり | 驚き（ポジティブ） |
| 内側寄り + 下がり | 驚き（ネガティブ）、怒り |
| 水平 | 満足・無関心 |
| 片方だけ上がり | 懐疑的 |

#### 要素⑤：Transitions（遷移）

| 遷移タイプ | 説明 | 使用例 |
|-----------|------|--------|
| **Action-to-Action** | 同じ主人公が連続した行動を取る | 最も基本的なデフォルト遷移 |
| **Subject-to-Subject** | 場面内で視点が切り替わる | 主人公A → 主人公B |
| **Scene-to-Scene** | 時間・空間を大きく飛ぶ | キャプションが唯一許される遷移 |
| **Subject-to-AI** | 主人公がAIと対話する | AI UXストーリーボード専用の遷移型 |

> **キャプション使用のルール** — Scene-to-Scene遷移のみキャプション使用可。それ以外はナレーション禁止（読者がストーリーから引き離される）

#### ストーリーボードの結論パネル（Conclusion）

- 最も重要なパネル
- AIソリューションが提供するベネフィットを現実的な規模で示す
- 「大げさすぎる結末」より「自然な幸福（Natural Bang）」を目指す

**結論パネルの良い例・悪い例：**

| NG例 | OK例 |
|------|------|
| 「うつから即座に回復！大喜び！」 | 「会話のきっかけをつかみ、初対面の人と話せた」 |
| 「請求書支払いで踊り狂う」 | 「手間が減り、時間ができた。犬とフリスビーで遊べた」 |

### AI向けストーリーボードの特別ルール

1. **UIの詳細を省く** — AIの実装方法は早期に決定しない（チームの創造性を保護）
2. **Subject-to-AI 遷移を活用** — AIを「もう一人の登場人物」として扱う
3. **「何を」「なぜ」に集中** — 「どのように」はあとで決める

## 3. Digital Twin 構築手順

Digital Twinとは、物理システムの重要な側面をモデル化した図で、AIが分析・予測すべき変数を明確にするツール。

### Digital Twinの定義と目的

| 目的 | 内容 |
|------|------|
| システムの理解とモデリング | AIが何を計測し、何を無視するかを明確にする |
| 最も価値ある予測の特定 | 顧客が本当に気にしている問いに答えるAIを設計する |
| 見落としているデータの発見 | 「ガラス窓センサー」のような盲点を早期に発見する |

### 構築ステップ（10分演習）

```
1. 中央に物理システムの絵を描く
2. 左側：AIが収集するセンサーデータをラベリング
   （自己申告データ・外部DBから取得するものも明記）
3. 下部：計算できる追加変数を書く
   （例：体重 + GPS座標 = 消費カロリー）
4. 右側：AIが予測したい変数をラベリング
5. 制御「ノブ」があれば記載（ユーザーが結果に影響できる変数）
6. 不足しているデータを特定し、入手方法を議論
```

### 風力タービン（ヨーモーター）のDigital Twin例

**入力（センサーデータ）：**
- 入力電流（モーターへの電力供給）
- モーター温度

**物理システム（中央）：**
- 7つのヨーシステムモーターのうちの1つ（EN4）
- モーターコイルの熱的状態

**予測（出力）：**
- 残りモーター寿命

> **キー洞察** — 「モーターが稼働中にどれだけ温度が上昇するか」が劣化の指標。古い車が坂道で過熱しやすいのと同じ原理

### Life Clock アプリのDigital Twin例

**1段目（食品認識レイヤー）：**
- 入力：ラベル付き食品画像DB（訓練データ） + ユーザーが撮影した食品画像
- 処理：画像分析AIモジュール
- 出力：食品の種類・提供量 → 栄養DBルックアップ → カロリー/マクロ数値

**2段目（健康・寿命予測レイヤー）：**
- 入力：栄養データ + 個人データ（体重・年齢・性別・運動・睡眠）
- 処理：健康・寿命分析AIモデル
- 出力：即時健康影響（D&D風表記：+1筋力、−1敏捷性）+ 予測寿命への影響（例：−5分）

### Digital Twinを作る際の問い

| チェック項目 | 問い |
|------------|------|
| データの完全性 | 「ガラス窓センサー」のような見えない重要変数はないか？ |
| データ入手可能性 | 必要な追加データをどこで、どのように取得するか？ |
| 倫理的考慮 | 予測していることは倫理的か？保険料・融資適格性への影響は？ |
| 潜在的な悪用 | このシステムはどのように悪用される可能性があるか？ |

### チーム編成の原則

Digital Twinは単独作業ではなく、**4職種チームでの共同作業**が必須：

| 役割 | 貢献 |
|------|------|
| Product Manager | ビジネスユースケースと優先度 |
| UX Designer | ユーザーリサーチに基づく実際の使用状況 |
| Developer Lead | 実装可能性とデータ取得方法 |
| Data Scientist | モデル化可能な変数と予測精度の限界 |

> **盲目の男と象の法則** — 1人では象の部分しか触れない。4職種のチームで初めて全体像が見えてくる

### 適用判断基準

```
Digital Twinが特に有効なケース：
✅ 物理システムや実世界プロセスをモデリングする場合
✅ 複数のデータソースを統合する必要がある場合
✅ AI予測の「入力→出力」関係が不明確な場合
✅ チームが「何を測定すべきか」について合意できていない場合

Digital Twinが不要なケース：
❌ 単純な分類タスク（入出力が明確）
❌ 既存データパイプラインが十分に文書化されている場合
```

## 4. Value Matrix 設計

### コアコンセプト：「AIの精度（Accuracy）は現実世界ではナンセンス」

データサイエンスのメトリクス（accuracy/precision/recall）は、現実のビジネスROIとは直接対応しない。Value Matrixは、Confusion Matrixを金銭的価値に変換することで、この問題を解決する。

### Confusion Matrix の理解

AIの予測には4種類の結果がある（車のメンテナンス検知AIの例）：

```
                    │ 予測：問題なし │ 予測：問題あり
────────────────────┼───────────────┼───────────────
実際：問題なし（80件）│  TN（真陰性） │  FP（偽陽性）
                    │   79件        │   1件
────────────────────┼───────────────┼───────────────
実際：問題あり（20件）│  FN（偽陰性） │  TP（真陽性）
                    │   11件        │   9件
```

| 略語 | 名称 | 意味 |
|------|------|------|
| **TP** | True Positive（真陽性） | 問題あり → AIが正しくアラート |
| **TN** | True Negative（真陰性） | 問題なし → AIが正しくスルー |
| **FP** | False Positive（偽陽性） | 問題なし → AIが誤ってアラート（空振り） |
| **FN** | False Negative（偽陰性） | 問題あり → AIがアラートを出し損ねる（見逃し） |

**Accuracy の計算式：**
```
Accuracy = (TP + TN) / 全予測数 × 100%
例：(79 + 9) / 100 × 100% = 88%
```

**88%の精度でも「半分以上の問題を見逃す」ことがある** — これがデータサイエンスメトリクスの限界。

### 3種のAIモデルの比較

| モデルタイプ | データサイエンス特性 | 傾向 | リスク |
|------------|-------------------|------|--------|
| **Conservative（保守的）** | 精度・Accuracy が高い | アラートを出し渋る | 問題を見逃す（FN多発） |
| **Balanced（バランス型）** | いずれのメトリクスも中程度 | 現実的なROI最大化 | 特定のメトリクスでは見劣り |
| **Aggressive（積極的）** | Recall（再現率）が高い | あらゆるシグナルでアラート | 空振りが多くなる（FP多発） |

> **例：** TP=$1,000、TN=$100の場合
> Conservative AI: $5,800 ROI
> Balanced AI: $15,000 ROI（158%以上高い）
> Aggressive AI: $9,000 ROI

### Confusion Matrix → Value Matrix 変換手順

```
Step 1: Confusion Matrix を作成（TP/TN/FP/FNの件数を埋める）

Step 2: 各結果の金銭的価値を設定
  TP の価値：正しく問題を特定したときの節約額・収益
  TN の価値：正しく「問題なし」と判断したときのコスト節約
  FP のコスト：空振りアラートの調査コスト・顧客不満
  FN のコスト：見逃しによる損失・事故コスト・信頼損失

Step 3: 件数 × 価値で各セルの合計値を計算

Step 4: 全合計がプラスになるモデルを選択
```

### 偽陽性/偽陰性のUXコスト分析

ビジネスROIだけでなく、**ヒューマンコスト（顧客・ユーザーへの影響）**も必ず考慮する：

| ケース | ビジネスROI | ヒューマンコスト | 最適AIタイプ |
|--------|------------|---------------|------------|
| FPが非常に高コスト（不正告発等） | Conservative を選ぶ | 無実の人への影響甚大 | Conservative |
| FNが壊滅的（テロ防止・癌診断等） | Aggressive を選ぶ | 見逃しは人命に関わる | Aggressive |
| FP/FNが均衡しROI最大化が目的 | Balanced が最善 | 過剰アラートで信頼損失 | Balanced |
| ユーザーの処理能力が限られる | 件数/日の上限を設定 | アラート疲れ（Alert Fatigue） | 処理容量に合わせる |

### Human Cost/Benefit の考え方

> **TSAの例：** AIが100%の人を二次検査しても数学的にはプラスになるが、それをしない理由は「ヒューマンコストが高すぎるから」

UXデザイナーが問うべき問い：

```
✅ AIの判断は顧客に不当な不便をかけないか？
✅ 長期的な顧客ロイヤルティへの影響は？
✅ 同じ状況で人間ならどう判断するか？
✅ このAIモデルはどのように悪用される可能性があるか？
```

### Value Matrix 適用の判断基準

| 状況 | Value Matrixが特に有効 |
|------|----------------------|
| TP と FN のコストが大きく異なる | ✅ 必須 |
| 複数のAIモデル候補がある | ✅ 必須 |
| ステークホルダーがaccuracyでAI評価している | ✅ 必須（意識改革ツールとして） |
| ユースケースが連続変数（価格・時間）の予測 | ✅ 適用可能（概念は同じ） |
| 単純なルールで解決できる | ❌ AI不要を検討 |

## 5. Design Exercise テンプレート

### Exercise A：AI Storyboardを作成する（10分）

**準備物：** 付箋（正方形）、鉛筆

1. **ユースケースを選ぶ** — 「夜も眠れない顧客の悩み」から逆算して選定
2. **Establishing Shot** — 「私はどこにいるのか？」を問いかけて最初のパネルを描く
3. **4〜6パネルで展開** — Subject-to-AI遷移を必ず含める
4. **Conclusion** — 「自然な幸福（Natural Bang）」でAIのベネフィットを現実的に示す

**チェックリスト：**
- [ ] AIが登場するパネルがあるか（Subject-to-AI遷移）
- [ ] UIの詳細を描きすぎていないか（抽象的でよい）
- [ ] Conclusionのベネフィットは現実的か（過大でも過小でもなく）
- [ ] キャプションはScene-to-Scene遷移のみか

---

### Exercise B：Digital Twinを作成する（10分）

1. **中央** — 物理システムの絵を描く
2. **左** — 入力データをラベリング（センサー / 自己申告 / 外部DB）
3. **下** — 計算できる追加変数を列挙
4. **右** — AIが予測したい変数をラベリング
5. **ノブ** — ユーザーが制御できる変数があれば記載

**振り返り問い：**
- [ ] 重要な変数が見落とされていないか（「ガラス窓センサー問題」）
- [ ] 不足データの入手方法は？
- [ ] 予測していることは倫理的か？
- [ ] 悪用される可能性は？

---

### Exercise C：Value Matrixを作成する（10分）

**4つの価値を設定する：**

| 問い | 金額設定 |
|------|---------|
| True Positiveの価値は？ | $___（問題を正しく特定したときの節約額） |
| True Negativeの価値は？ | $___（問題なしを正しく判断したときのコスト節約） |
| False Positiveのコストは？ | -$___（空振りの調査コスト + 顧客不満） |
| False Negativeのコスト は？ | -$___（見逃しによる損失 + 信頼低下） |

**AIモデル選択の判断：**

```
FN コスト >> FP コスト → Aggressive（高Recall）モデルを選択
FP コスト >> FN コスト → Conservative（高Accuracy）モデルを選択
FN コスト ≈ FP コスト → Balanced モデルで ROI 最大化
```

**ヒューマンコストの振り返り：**
- [ ] AIの誤判断によるユーザーへの不便・不満はどの程度か？
- [ ] ユーザーの1日の処理容量（アラート件数など）に合っているか？
- [ ] 判断は人間の倫理感覚と一致しているか？

## クイックリファレンス：3ツールの使い分け

| ツール | 使用タイミング | 成果物 | 所要時間 |
|--------|--------------|--------|---------|
| **AI Storyboard** | ユースケース検証の最初期 | 4〜6パネルの絵 | 10分 |
| **Digital Twin** | ユースケース選定後・データ検討前 | 入力→システム→出力の図 | 10〜15分 |
| **Value Matrix** | AIモデル選定前・ステークホルダー説得時 | ROI比較表 | 10分 |

**推奨実施順序：**
```
ユースケース候補 → Storyboard（可視化・検証）
                → Digital Twin（データ・変数の明確化）
                → Value Matrix（AIモデル選択・ROI評価）
```
