## 1. AI Search UX

### LLM検索UIの革命

従来の検索エンジン（キーワードマッチング）では、「怖くない謎解きもの（Mysteries That Are Not Scary）」のような**ファジークエリ**に対応できない。

**従来検索の失敗パターン:**
- 否定的ニュアンスを理解しない（"Not Scary" を無視）
- 意味的類似性ではなくキーワードで一致させる
- 結果が脈絡のないジャンル混在になる

**LLM検索の革新ポイント:**
- クエリの意図と感情的ニュアンスを理解する
- 自然言語で結果の解釈文を生成する（「眠れない夜なしで楽しめるミステリー」等）
- 既存のファセット（絞り込み）をクエリに応じて動的に拡張できる

### LLM検索UIデザインの判断基準

| 判断項目 | 選択肢A | 選択肢B | 推奨基準 |
|---------|---------|---------|---------|
| LLM検索のトリガー | 自動（通常検索と同一UI） | 手動（スイッチで切替） | ユーザーが「なぜ違う結果が出たか」を理解できるかで判断 |
| ファセットの扱い | 従来ファセットを維持 | LLMが動的生成 | 両方提供が最善。既存ファセットは価値がある |
| 結果の表示形式 | 通常リストと同一 | LLM特有フォーマット | クエリの複雑さに応じて切替 |
| クエリ解釈テキスト | 非表示 | 画面上部に表示 | 常に表示を推奨（透明性と信頼感のため） |

### 否定クエリの設計原則

否定クエリ（「〜ではない」）はLLMが最も得意とするが、UIで以下を考慮すること:

1. **クエリの再解釈を表示**: AIが何を理解したかをユーザーに示す
2. **スカリネス評価など独自メトリクスを追加**: クエリ固有のフィルタリング軸を動的生成
3. **フォールバックを設計**: LLM結果がゼロの場合の代替表示

## 2. Eye Meat（視覚的情報）とDOI Sort Algorithms

### ダイナミックダッシュボード（Eye Meat）の概念

**Eye Meat（視覚的な「肉」）**: ユーザーが次に見たいと思うコンテンツを視覚的に提示するダッシュボード。Edward Tufteは「視覚的な複合物（visual confections）」と呼んだ。

**設計の根本的難しさ:**
> 特定のユーザーが次に何を見たいかを把握することは、非常に困難な問題である。

ダッシュボードの各セクション（「ルーブリック」）は、それぞれ独立したアルゴリズムで構成される。そのため:
- セクション間で重複コンテンツが発生する
- アルゴリズムの失敗がユーザーに見える（猫フードを犬派に推薦する等）

### AIバイアスへの警戒

ダイナミックダッシュボードのAIアルゴリズムは**バイアスを反映する**:
- 政治的・文化的な偏りが検索順位に現れる
- 「エコーチェンバー」を強化するアルゴリズムは企業収益と直結している
- UXデザイナーはAIが何を優先しているかを深く理解する必要がある

### DOI（Degree of Interest）ソートアルゴリズム

**定義**: コンテンツの「関心度」に基づく並べ替えアルゴリズム。何がページ1に表示され、何がページ6に埋もれるかを決定する。

**DOIアルゴリズムの典型的な構成要素:**

| 要素 | 説明 | 注意点 |
|------|------|--------|
| 人気度（Popularity） | 閲覧数・購入数 | 既存コンテンツに有利 |
| 新しさ（Recency） | 新着コンテンツを優遇 | スタートゼロからでも「高成長」に見える |
| トレンド（Trending） | 成長速度（スロープ）で判定 | 最小閾値の設定が重要 |
| ユーザー適合性 | 過去の行動との一致度 | バイアスの温床になりやすい |
| シェア数 | ソーシャル拡散 | フェイクエンゲージメントに弱い |

**UXデザイナーの責任:**
- 自社のソートアルゴリズムの仕組みを理解する
- どのアルゴリズムが収益に貢献しているかを把握する
- アルゴリズムの倫理的リスクを評価に含める

## 3. AI-First 情報アーキテクチャ（IA）

### Canvasパターンの定義と限界

**Canvasパターン**: ChatGPTが普及させた、AIが生成したコンテンツをポップアウトウィンドウで表示するデザインパターン。

**Canvasの限界:**
- バーチャルアシスタントには依然としてプロフィール編集・サブスクリプション設定・支払い履歴等が必要
- **チャットだけでは完結したアプリケーションにならない**
- 他社のUIを盲目的にコピーすることは推奨されない

> **原則**: 自社ユースケースが最優先。ChatGPTにとって良いものが、あなたのSaaSアプリに必要とは限らない。

### IAは死んでいない

AIチャット時代において「IAは死んだ」という論調があるが、これは誤りである。
チャットだけでは:
- ユーザーがアプリの価値を理解しにくい
- 有益なクエリを思いつくことができない
- 時間的に重要な情報（アラート、セール等）を認知できない

> **核心原則**: AI-Firstを正しく実装するためには、情報アーキテクチャが必要である。

### 従来型IA vs AI-First IA 比較テーブル

| ページタイプ | 従来型IA | AI-First IA |
|------------|---------|------------|
| ホームページ | 汎用的なレコメンド（精度低い） | **Analysis Overview**: LLMテキストサマリー＋文脈的ストーリー告知 |
| カテゴリページ | 固定カテゴリ一覧 | **Category Analysis**: ユーザー購入履歴からLLMが推測した関心カテゴリ＋ヒーロー画像 |
| 検索結果 | キーワードマッチング | **LLM Search**: クエリ解釈テキスト＋クエリ固有メトリクス（スカリネス評価等）＋動的ファセット |
| アイテム詳細 | 複数の独立した検索ボックス | **Item Detail**: コンテキスト化されたオールインワンQ&A＋アイテム固有LLMサマリー |
| メンテナンス | 時系列リスト（LIFO） | **Maintenance**: 自然言語Q&A対応＋AI駆動のガイダンス＋トラブルシューティング |

### AI-First IAフレームワーク実装原則

1. **Analysis Overview（旧ホームページ）**: 時節に応じたLLMサマリー・ユーザーへの理由説明（Why Shop Now）・特定カテゴリへの誘導
2. **Category Analysis**: ユーザーの過去注文から関心カテゴリを自動推測・アイテム固有の購買提案
3. **LLM Search**: クエリ解釈の表示・クエリ固有評価軸・スコアリングによる精度向上
4. **Item Detail**: コンテキスト化されたAsk Bar・アイテム固有サマリー・Next Steps質問タグ
5. **Maintenance**: 会話型Q&A・LLMによる設定ガイダンス・独立エージェント的対応

**重要**: これらはテンプレートではなく、**自社ユーザーのユースケースに合わせて変形するための参考例**である。

### Generative UI（Jakob Nielsen視点）

- **第1世代Generative UI**: 設計段階でAIがUIを生成し、人間がレビュー後に固定する
- **第2世代Generative UI（近未来）**: アクセスのたびにUIがリアルタイムで生成。ユーザーごとに異なるUI

UXデザイナーの役割シフト: **ピクセルを配置する**から**AIがUIを生成するためのルールとヒューリスティクスを設計する**へ

## 4. Forecasting（予測）UI設計

### 予測グラフの基本構成

| 視覚要素 | 意味 | 実装推奨 |
|---------|------|---------|
| 実線 | 実際のデータ | 必須 |
| 破線 | 予測値 | 必須 |
| 垂直「nowライン」 | 現在日時（「あなたはここにいます」マーカー） | 推奨 |
| 信頼区間コーン | 予測の不確実性の範囲 | 予測精度が重要な場合は必須 |

### 線形回帰と R²

**線形回帰**: データ点を通る直線を引き、未来の値を予測する最も基本的な手法。

**R²（決定係数）の設計的意味:**
- 0〜1の値で予測フィットの良さを表す
- **R²が1に近い**: データ点が線に近い → 予測の信頼度が高い → 信頼区間コーンが細い
- **R²が0に近い**: データ点がばらついている → 予測の信頼度が低い → 信頼区間コーンが太い
- R²は**線形スケール**: 0.8は0.4の2倍の精度

**ユーザーへの表示推奨:**
- 信頼区間コーンで予測の不確実性を直感的に示す
- R²の数値そのものは一般ユーザーには難解 → 視覚的なコーンで代替する

### R² vs R（方向性の考慮）

| 指標 | 特徴 | 使用場面 |
|------|------|---------|
| R² | 方向を無視した絶対距離 | 過大/過少の影響が同等な場合 |
| R | 方向（over/under）を考慮 | 過大予測と過少予測でコストが異なる場合 |

**例**: 食料予測でのオーバーシュート（余分な食料を運ぶ）とアンダーシュート（飢餓）では影響が非対称 → Rを使用すべき

### 非線形回帰

自然界のほとんどの現象は**真に線形ではない**。非線形回帰が必要な場合:
- 化学物質の劣化・腐食
- 成長曲線・飽和曲線
- 複雑な機械的振動

**注意点**: 複数の非線形モデルが統計的にほぼ同等に見えても、物理的現実と一致するモデルを選ぶ必要がある。例:「冷蔵庫内の塩素が時間とともに増加する」という曲線は統計的には適合しても物理的に誤りである。

**UXデザイナーの役割**: データサイエンティストに「このモデルは実際の物理/ビジネス現象と一致しているか」を問う質問力を持つこと。

### 季節性（Seasonality）

**定義**: 曜日・月・年単位で繰り返す周期的なパターン。

| 季節性タイプ | 例 | 対処法 |
|------------|-----|-------|
| 日次 | 深夜はトラフィック低下、平日ピーク | MLモデルで曜日パターンを学習 |
| 週次 | 土日に需要減少 | MLモデルで週末調整 |
| 年次 | ブラックフライデー・クリスマス | 複数年のデータでMLモデルを訓練 |

**設計上の重要な非対称性問題:**
- Cyber Mondayトラフィックを**過大予測**: AWSの請求が少し増える（影響小）
- Cyber Mondayトラフィックを**過少予測**: サイトがクラッシュ、ショッピングカート消失（影響甚大）

### 集約変数の予測（Aggregate Variable Forecasting）

**定義**: 日次合計のように「集計値」を予測する場合は折れ線グラフより**棒グラフが適切**。

**適用例:**
- 日次の水道使用量予測
- 日次カロリー摂取量
- 日次注文数

**棒グラフ予測の視覚的区別:**
- 実績値: 濃い色の棒
- 予測値: 薄い色（淡い）棒
- nowラインで実績と予測を分離

## 5. Anomaly Detection（異常検知）UI設計

### 異常検知の主要ユースケース

| ユースケース | 異常のシグナル | ビジネス価値 |
|------------|-------------|------------|
| 製造品質管理 | 測定値の外れ値 | 不良品の早期発見・Six Sigma対応 |
| セキュリティ・不正検知 | 短時間での高額取引が複数国から | カード不正利用のリアルタイム遮断 |
| 予兆保全 | 振動・温度の異常上昇 | 計画外ダウンタイムの防止 |
| Webトラフィック管理 | 季節外れのトラフィック急増 | マーケティング施策への即時対応 |
| コンプライアンス | 製造プロセスの逸脱 | 規制違反・罰金回避 |

### 4種の異常タイプと判断基準テーブル

| 異常タイプ | 定義 | 検知方法 | UIでの表現 | アラートのトリガー |
|-----------|------|---------|-----------|------------------|
| **Point Anomaly（単発異常）** | 値が一時的にスパイクし、すぐに正常値に戻る | 静的または動的閾値 | 異常点を赤丸でマーク | 1分間に3回以上発生した場合 |
| **Change Point Anomaly（変化点異常）** | 値が高い水準に変化し、**持続的に維持**される | 静的または動的閾値 | 異常開始点を赤でハイライト | 1分以上継続した場合 |
| **Contextual Anomaly（文脈依存異常）** | 値自体は正常範囲でも、**時間帯・季節のパターン**に対して異常 | AI/MLによる形状認識（季節性モデル） | 予測ライン（シャドウ）からの逸脱を可視化 | 予測値から所定%以上逸脱した場合 |
| **Curve Shape Anomaly（波形異常）** | 値の波形形状そのものが、既知の正常パターンから変化 | AI/MLによる形状分類 | 正常波形と異常波形を並列表示 | 形状一致信頼度が閾値以上の場合 |

### 静的閾値 vs 動的閾値の選択基準

| 条件 | 推奨 | 理由 |
|------|------|------|
| CPU使用率・エラー数 | **静的閾値** | 値が高い = 常に問題。システムに「慣れ」させてはいけない |
| コンプライアンス・SLA指標 | **静的閾値** | 法規制上の上限は動的に変えてはいけない |
| ネットワークトラフィック量 | **動的閾値（Bollinger Bands）** | 曜日・時間帯で正常値が変動するため |
| 注文量・セッション長・収益 | **動的閾値（Bollinger Bands）** | 外部要因（キャンペーン等）で正常範囲が変動する |

### Bollinger Bandsの仕組み

**Bollinger Bands** = 動的閾値の最もシンプルな実装:

1. N日間の単純移動平均（SMA）を計算する
2. SMAからの標準偏差を計算し、2倍にする
3. 上部バンド = SMA + 2σ、下部バンド = SMA − 2σ
4. バンドを「突き抜けた」ときに異常を検知する

**特長**: 複雑なAIなしで実装でき、株価・Webトラフィックなど値が動的に変化する指標に有効。ただし、**コンプライアンス指標や絶対的上限がある指標には不適切**。

### Point / Change Point 異常の統合UIデザイン

ショートハンド記法を使った統合フォーム例:
```
[Critical \/] アラートをトリガーする条件:
[X] 値が [90 %] を [1 minute \/] 間超過した場合
[X] [3 times \/] / [1 minute \/] の頻度で発生した場合
```

### Point / Change Point UI設計の5原則

1. **発生タイマーを設ける**: 1回の異常でアラートを出すと誤報が多発する。Point異常は「N分間にM回」、Change Point異常は「N分間継続」で制御する
2. **過去データのプレビューグラフを提供する**: 設定変更が過去データに何件のアラートを出したかをリアルタイムでプレビューできるようにする
3. **Mad Lib（穴埋め文）パターンを使う**: 設定UIを英語/日本語の自然文として読めるよう、文中にドロップダウンやテキストフィールドを埋め込む
4. **AIによるデフォルト最適化**: 数百〜数千のメトリクスに対して手動設定は非現実的。AI/MLが偽陰性・偽陽性を最小化するパラメータを自動設定する
5. **手動オーバーライドを常に残す**: AI/MLが自動調整している場合でも、手動での上書きを必ず可能にする

### Contextual Anomaly（季節性異常）の改善チェックリスト

標準的な季節異常検知UIの主な問題点と推奨対策:

| 問題 | 推奨対策 |
|------|---------|
| 単一周期しか設定できない（日次 or 週次 or 年次） | AI/MLが最適な複合周期を自動提案する |
| アルゴリズムが自己調整しない（高需要期に誤報が多発） | 異常が多すぎる場合に自動感度調整するオプションを追加 |
| 感度設定がAdvancedメニューに隠れている | 感度スライダーをメイン画面に配置（デフォルト: Auto） |
| 感度のパーセンタイル表示が難解 | スライダー（←少ない ─── ─── →多い）に置き換え |
| CopilotのQ&Aが不在 | 設定Copilotで「何故この設定？」に自然言語で回答できるようにする |

### Curve Shape Anomaly（波形異常）のUI設計

**ホースヘッド型石油ポンプ（Downhole Dynacard）の例**:

AI/MLが「ポンディング（油面打撃）」などの異常波形を認識し、自動でモーターを停止する。UIはシンプルでよい:

```
[Fluid Pound \/] カーブ解析を毎 [10 \/] ポンプ回転ごとに実行
一致信頼度 >= [90% \/] のとき:
[X] ポンプモーターを [10分 \/] 停止
[ ] [Warning \/] アラートを発報
[[Update]]
```

**UIのポイント**: 検知メカニズムが高度でも、UIはシンプルで良い。複雑な波形分類はAIに任せ、ユーザーが理解できる「アクション設定」だけを表面化させる。

## 6. Shorthand UX Design Notation（AIプロンプトとしてのUXノーテーション）

### 概念

UXデザイナーが遠隔会議中でもリアルタイムにUIを記述できる**テキストベースの速記記法**。さらに現在では、この記法を**LLMへのプロンプト**として使い、React/HTMLコードを直接生成できる。

### 主要なノーテーション記号

| 記号 | 意味 | 例 |
|------|------|-----|
| `[値]` | テキスト入力フィールド | `[Fluid Pound]` |
| `[値...]` | テキストエリア | `[説明...]` |
| `[値 \/]` | ドロップダウン | `[10分 \/]` |
| `[値1 \| 値2 \| 値3 \/]` | 選択肢付きドロップダウン | `[10分 \| 20分 \| 30分 \/]` |
| `[X]` | チェック済みチェックボックス | `[X]` |
| `[ ]` | 未チェックチェックボックス | `[ ]` |
| `[[ボタン名]]` | プライマリボタン | `[[Update]]` |
| `{要素名}` | 非フォーム要素（画像・グラフ等） | `{Anomaly graph}` |
| `(ヘルパーテキスト)` | ヘルプテキスト | `(詳細情報)` |
| `<リンク名>` | リンク | `<モデルを再訓練>` |
| `(0)ON ( )OFF` | ラジオボタン | `(0)ON ( )OFF` |
| `(0   )` | スイッチ | `(0   )` |
| `--/\--` | スライダー | `--/\--` |

### AIプロンプトとしての活用パイプライン

```
1. ノーテーション記号の定義を「seed」としてLLMに渡す（RAGの一形態）
2. UIフォームをノーテーションで記述する
3. LLMがデザインシステム準拠のReact/HTMLを出力する
4. LLMが誤字・略語・フォーマットを自動修正する
```

**「Robot Monkey Work」の自動化**: テーブル・単純フォームの生成はLLMに任せ、UXデザイナーは**戦略・共感・イノベーション・ステークホルダー連携**に集中する。

## 7. Agentic AI UX

### AIエージェントとは何か

**アリの巣メタファー**: アリの巣には、働きアリ・兵隊アリ・女王アリ等の**専門役割**がある。個々のアリは賢くないが、**集合知として複雑な問題を解決する**。

AIエージェントも同様:
- 個々のワーカーエージェント: 特定タスクに高度に特化
- スーパーバイザーエージェント（推論エージェント）: ワーカーを指揮・結果を統合
- 人間オペレーター: 一般的なリクエストを出し、アウトプットをレビューする

**AIとAIエージェントの最大の違い**: **自律性（Autonomy）**。精密な指示を与えなくても、エージェントは問題を探索し、仮説を立て、解決策を見つける。

### マルチステージ・エージェントワークフロー

| ステップ | アクター | アクション |
|---------|---------|-----------|
| 1 | 人間オペレーター | 問題を発見し、スーパーバイザーエージェントに調査を依頼 |
| 2 | スーパーバイザー | ワーカーエージェントを並列起動。各エージェントがシステムの異なる部分を調査 |
| 3 | ワーカーエージェント | 「提案された観察」（Suggested Observations）を返却 |
| 4 | 人間オペレーター | 関連する観察を「Accept」。スーパーバイザーが次フェーズへ進む |
| 5 | スーパーバイザー | 仮説（Hypothesis）を提示。人間がAcceptすると修正アクションを提案 |

### CloudWatch Investigation UIの例

AWSのCloudWatch調査フローは現実的なエージェントUXの好例:

**左ペイン（ケースファイル）**: Acceptされた観察が蓄積される「捜査ファイル」
**右ペイン（提案）**: エージェントが非同期に返してくる提案が随時追加される

**UX設計上の重要ポイント**:
- 右ペインは最初「空」で始まり、エージェントが結果を返すたびに更新される（**非同期UX**）
- 人間が観察をAcceptするたびにエージェントが調査の深度を変える（**フィードバックループ**）
- スーパーバイザーが「観察提案」から「仮説提案」にモードを切り替える（**フェーズ遷移の可視化**）

### エージェントAI UXの設計原則

| 原則 | 詳細 | 設計的含意 |
|------|------|----------|
| **Flexible, Adjustable UI** | エージェントは複数ステージにわたる継続的インタラクションを必要とする | 固定フォームではなく、フロー型・ループ型のUIを設計する |
| **Autonomy（自律性）** | エージェントは「試すためにアクション」をとることがある。false positiveのコストを考慮しない場合も | 重大なアクション前には必ず確認フローを設ける |
| **New Controls Required** | ほとんどのエージェントアクションは**非同期** | start/stop/pauseボタン、進行状態インジケーターが必須 |
| **"Hire" not "Use"** | エージェントはツールではなく、推論する実体 | ロールベースアクセス制御（RBAC）・バージョン管理・機密保護が必要 |
| **Continuously Learning** | エージェントは使用するほど「専門家」になる | 学習の透明性・モデルのバージョン管理UIが必要 |

### 非同期UXのパターン集

エージェントとの非同期インタラクションに必要なUIパターン:

| パターン | 説明 | 実装例 |
|---------|------|--------|
| **Pending State** | エージェントが処理中であることを示す | スピナー・「調査中...」テキスト |
| **Incremental Update** | 結果が揃ったものから表示する | 右ペインに観察が随時追加される |
| **Accept/Reject Flow** | 人間がエージェントの提案を評価する | Accept/Rejectボタン付きカード |
| **Phase Indicator** | エージェントのフェーズを明示する | 「証拠収集中」→「仮説立案中」のプログレス |
| **Circuit Breaker** | 暴走を防ぐ停止制御 | Stop/Pause/Resume ボタン |
| **Hypothesis Card** | 最終的な仮説を強調表示する | 「The Sorcerer's Apprentice」的な暴走を防ぐ確認ダイアログ |

## クイックリファレンス

### AI-First IA ページ種別 判断フロー

```
ユーザーが来訪
    ↓
初回訪問・季節イベント → Analysis Overview（LLMサマリー + Why Now）
    ↓
特定カテゴリへの関心が高い → Category Analysis（個人化コンテンツ + ヒーロー画像）
    ↓
検索クエリを入力 → LLM Search（解釈テキスト + 動的ファセット + クエリ固有メトリクス）
    ↓
アイテムを閲覧 → Item Detail（コンテキスト化Ask Bar + 個人化サマリー）
    ↓
注文・設定管理 → Maintenance（自然言語Q&A + AI駆動ガイダンス）
```

### 異常タイプ 選択チェックリスト

```
□ 値が急激にスパイクし、すぐ戻る → Point Anomaly
□ 値が変化し、そのまま高い水準が持続する → Change Point Anomaly
□ 値は正常範囲だが、時間帯・曜日パターンに対して異常 → Contextual Anomaly
□ 値の波形形状そのものが変化する → Curve Shape Anomaly
```

### 予測UI 選択チェックリスト

```
□ 連続値の予測 → 折れ線グラフ（実線＋破線＋nowライン）
□ 集計値の予測（日次合計等） → 棒グラフ（濃色＋淡色＋nowライン）
□ 予測精度の可視化が必要 → 信頼区間コーンを追加
□ 過大/過少が非対称 → R²ではなくRを使用するよう要求する
□ 週次・年次の繰り返しパターンがある → 季節性対応MLモデルを採用
```

### エージェントUX 設計チェックリスト

```
□ 非同期インタラクションに対応したUIを設計したか
□ Start/Stop/Pause 制御を実装したか
□ エージェントのフェーズ（調査中→仮説立案中等）を可視化したか
□ Accept/Reject フィードバックループを設計したか
□ 重大アクション前の確認フローがあるか
□ エージェントのバージョン・学習状態を管理できるか
```
