# designing-ai-products — 実践UXガイド

## 1. 概要・使用タイミング

このスキルは **AIプロダクトのUXデザイン** に特化したフレームワークを提供する。
フレーミングからデザインパターン、ユーザーリサーチ、バイアス・倫理まで体系的にカバーする。

### このスキルを使う場面

| 場面 | 適用する章 |
|------|-----------|
| AIプロダクトのユースケース選定・フレーミング | §2 フレーミング |
| SaaS CopilotやLLM UIの設計 | §3 デザインパターン |
| AI検索・予測・異常検知UIの設計 | §4 AI-First IA & データUX |
| AIプロダクトのユーザーリサーチ計画 | §5 リサーチ手法 |
| AIバイアス・倫理レビュー | §6 バイアス・倫理 |

### AIプロジェクト失敗の5パターン（5 Fails）

AIプロジェクトの85%が失敗する。主な原因:

1. **Fail #1: 間違ったユースケース選択** — 価値がないところにAIを適用
2. **Fail #2: ビジョンの欠如** — 戦術的UIしか作らず、長期ビジョンがない
3. **Fail #3: データ品質の無視** — AI精度を過信し、データのバイアスを見過ごす
4. **Fail #4: ユーザーリサーチ省略** — 「誰が何を困っているか」を調査しない
5. **Fail #5: 倫理・バイアス無視** — AIの偏りがユーザーに害を与える

> **重要原則**: AIプロダクトで問うべき最初の問いは「AIで何を予測すべきか？」ではなく、**「どこに価値があるか？」** である。

## 2. AIプロジェクトフレーミング

### 2-1. AIストーリーボード（5要素）

UIモックより先にストーリーボードを描く。5要素を必ず含める:

| 要素 | 役割 | ポイント |
|------|------|---------|
| **Establishing Shot** | 設定シーン（場所・文脈） | ユーザーが誰で、どこにいるかを示す |
| **Things** | 物理的オブジェクト・アーティファクト | ユーザーが操作する物・情報を明示 |
| **People** | 登場人物 | 主役ユーザーだけでなく、影響を受ける人も描く |
| **Faces** | 表情・感情 | AIのアウトプットが感情にどう影響するかを示す |
| **Transitions** | 画面遷移・ワークフロー | AIが介入する前後の状態変化を描く |

**ストーリーボードの終わり方**: 必ず「ユーザーが価値を得た瞬間」で締める。
途中で終わらせない（Attack of the Clones問題 = スクリーン羅列で終わる失敗）。

#### ストーリーボード作成フロー

```
1. ユースケースを1文で書く（As [役割], I need [目標]）
     ↓
2. Establishing Shot を描く（付箋/スケッチ）
     ↓
3. Things / People / Faces を追加
     ↓
4. AIが介入するTransitionを描く
     ↓
5. 価値が実現した最終シーンで締める
     ↓
6. 1〜2分の動画 or ポスターに仕上げる
```

### 2-2. デジタルツイン（Digital Twin）

物理・業務システムのAIを設計するとき、デジタルツインモデルで全体像を可視化する。

**描き方の原則**:
- **左側**: 入力データ（センサー・ユーザー操作・外部データ）
- **中央**: AIシステム（モデル・予測ロジック）
- **右側**: 出力（アクション・アラート・推奨）

**手順**:
1. 対象の物理システム（または業務フロー）を一枚の絵に描く
2. 各コンポーネントに「どんなデータを持つか」をメモ
3. AIが「予測/分類/推奨」できる部分を特定
4. その予測が「どんな価値」を生むか確認 → Value Matrixへ接続

> **問い**: 「AIは何を予測すべきか？」
> → 正しい問いは「AIが正確にこれを予測できたとして、それは誰にとってどれだけ価値があるか？」

### 2-3. Value Matrix（価値マトリクス）

AIの「精度」よりも「価値」を最大化することが重要。Confusion MatrixをValue Matrixに拡張する。

#### Confusion Matrix → Value Matrix

| | AIが Positive と予測 | AIが Negative と予測 |
|--|---------------------|---------------------|
| **実際は Positive** | TP（True Positive）✅ | FN（False Negative）⚠️ |
| **実際は Negative** | FP（False Positive）⚠️ | TN（True Negative）✅ |

**各セルに金額・コスト・ユーザー体験コストを入れて評価する**:

| セル | 典型的なコスト/価値 | UXが問うべきこと |
|------|-------------------|----------------|
| TP | 収益・コスト削減 | この価値を最大化するUIは？ |
| FP | 無駄な作業・信頼低下 | 誤報をユーザーはどう処理するか？ |
| FN | 見逃しリスク・安全問題 | 見逃した場合の被害は許容範囲か？ |
| TN | 現状維持 | ユーザーは通知不要で安心するか？ |

> **AIの精度向上よりも**: FPとFNのビジネス・人的コストを定量化し、**ROIを最大化するUX設計**を選ぶ。

#### Value Matrix設計の問い

1. FPが起きたとき、ユーザーはどんな追加作業をするか？（コスト）
2. FNが起きたとき、どんなリスクが発生するか？（安全・法的問題）
3. TPが多いほど、どんな価値が生まれるか？（ROI）
4. モデルの閾値をどこに設定するとROIが最大化されるか？

## 3. AI UXデザインパターン

### 3-1. SaaS Copilot配置パターン

> **AskUserQuestion**: Copilotを設計する場合、タスクの重要度と画面占有面積の要件を確認する。
> 「このCopilotは補助的ですか？それとも主要タスクですか？」

| パターン | 説明 | 使用場面 |
|---------|------|---------|
| **Side Panel** | 画面右側にパネル表示 | 既存UIの補助として常時参照したい場合 |
| **Large Overlay** | 画面中央に大きくオーバーレイ | Copilotが主要ワークフローの一部の場合 |
| **Full Page** | Copilot専用ページ | Copilotがプライマリユースケースの場合 |

**判断フロー**:
```
タスク中に常時参照が必要？
  ├─ Yes → Side Panel
  └─ No → Copilotが主要タスクか？
              ├─ 補助的 → Large Overlay
              └─ 主要 → Full Page
```

**Stateful Copilot設計原則**:
- ユーザーの操作履歴・選択を記憶する
- セッションをまたいで文脈を引き継ぐ
- Fine-Tuned Model + Plug-Ins でドメイン特化

### 3-2. 7つのLLMデザインパターン

> **AskUserQuestion**: どのLLMパターンを実装するか確認する前に、
> ユーザーが「AIの提案」をどの場面で求めているかをヒアリングする。

| # | パターン名 | 説明 | 最適な場面 |
|---|-----------|------|-----------|
| 1 | **Restating** | 入力を言い換えて確認 | 曖昧な入力が多い、誤解リスクが高い場面 |
| 2 | **Auto-Complete** | 入力補完・次の単語予測 | 繰り返し入力が多い、コマンド入力画面 |
| 3 | **Talk-Back** | 質問形式でガイド | ユーザーが目標を言語化できていない場面 |
| 4 | **Initial Suggestions** | 空白時の初期提案表示 | 空ページ症候群（Blank Page Problem）の解消 |
| 5 | **Next Steps** | 次のアクション提案 | タスク完了後の次行動を促したい場面 |
| 6 | **Regen Tweaks** | 再生成オプション提供 | AI出力の微調整が必要な場面 |
| 7 | **Guardrails** | 禁止・警告・制約 | 安全制約、コンプライアンス要件がある場面 |

**LLMパターン選択フロー**:
```
ユーザーの入力が曖昧？ → Restating
空白/初期状態？ → Initial Suggestions
繰り返し入力が多い？ → Auto-Complete
目標が不明確？ → Talk-Back
タスク完了後？ → Next Steps
出力の調整が必要？ → Regen Tweaks
安全制約が必要？ → Guardrails（常に最後の砦として設ける）
```

#### Guardrails設計原則
- 禁止トピックを明示した上で「代替を提案」する
- 拒否するだけでなく、ユーザーを正しい方向へ導く
- 安全制約をUX的に嫌なものにしない（ユーモア・親切なトーンを使う）

### 3-3. AI Search UX

**従来検索 vs LLM検索の違い**:

| 軸 | 従来検索 | LLM検索 |
|----|---------|---------|
| クエリ形式 | キーワード | 自然言語 |
| 結果 | リスト表示 | 統合された回答 |
| 曖昧さへの対応 | 苦手 | 得意 |
| 「怖い謎」問題 | 発生しやすい | 解消できる |

**「Mysteries That Are Not Scary」問題**:
ユーザーは「こんなこと聞いていいのか？」「どう検索すればいいかわからない」という心理的障壁を持つ。
LLM検索UIは、この障壁を下げる設計が必要:
- 検索ボックスに「自然な言葉で聞いてください」と添える
- 類似質問の例を表示する
- 「こんなことも聞けます」の例示

**動的ダッシュボードのソートアルゴリズム**:

| アルゴリズム | 概念 | 用途 |
|------------|------|------|
| **Eye Meat** | 最も重要な情報を視線の中心に | ダッシュボードの優先順位付け |
| **DOI Sort** | Degree of Interest スコアで並べ替え | パーソナライズされた情報提示 |

## 4. AI-First情報アーキテクチャ & データUX

> **AskUserQuestion**: 既存のIA（情報アーキテクチャ）を持つプロダクトを改善する場合、
> 「現在のユーザーの主な導線はカテゴリブラウジングですか、それとも検索ですか？」を確認する。

### 4-1. AI-First IAのCanvasパターン

従来型IAとAI-First IAの比較:

| 従来型IA | AI-First IA |
|---------|------------|
| カテゴリナビゲーション中心 | 分析オーバービュー中心 |
| ユーザーが迷子になりやすい | AI が適切な情報を先回り表示 |
| 検索は補助的 | LLM検索が主要導線 |
| 静的ページ構成 | コンテキスト適応型レイアウト |

**AI-First IA 4層モデル**:
```
1. Analysis Overview（分析オーバービュー）
   → AIが全体像・異常・推奨を要約
     ↓
2. Category（カテゴリ）
   → ドリルダウン先の分類
     ↓
3. LLM Search（自然言語検索）
   → 「〇〇を探している」に回答
     ↓
4. Item Detail（詳細）
   → 個別アイテムの詳細ビュー
```

### 4-2. Forecasting UI設計

予測UIを設計するときの判断基準:

| 要素 | 設計上の問い | 実装ガイド |
|------|------------|-----------|
| **トレンドライン** | 線形か非線形か | R²値を表示して信頼度を示す |
| **信頼区間** | 予測のブレ幅をどう表現するか | シェーディングで範囲を表示 |
| **季節性** | 周期パターンがあるか | 前年同期との比較を追加 |
| **外れ値** | 異常値は除外するか含めるか | 除外理由を明示する |
| **シナリオ比較** | 複数の未来を比較させるか | 「悲観/中立/楽観」の3ライン |

**R²スコア（決定係数）の表示**:
- R² ≥ 0.9: 「高信頼度の予測」として表示
- 0.7 ≤ R² < 0.9: 「中程度の信頼度」として注意書き
- R² < 0.7: 「参考値」として警告を表示

### 4-3. Anomaly Detection UI（異常検知UI）

> **AskUserQuestion**: 異常検知UIを設計する前に、
> 「どの種類の異常を検知しますか？単発のスパイクですか、それともトレンド変化ですか？」を確認する。

| タイプ | 定義 | UI設計のポイント |
|--------|------|----------------|
| **Point Anomaly** | 単発の外れ値 | マーカーで点を強調、クリックで詳細 |
| **Change Point Anomaly** | トレンドの転換点 | 区間を色分け、「ここから変化」を示す |
| **Contextual Anomaly** | 文脈依存の異常 | 比較対象（前月・前年）を並べて表示 |
| **Curve Shape Anomaly** | パターン形状の変化 | 「正常パターン」のテンプレートを重ねる |

**Static vs Dynamic Threshold**:

| 方式 | 説明 | 使う場面 |
|------|------|---------|
| **Static Threshold** | 固定値（例: 100以上で警告） | ルールが明確で変動しない場合 |
| **Dynamic Threshold** | 統計的範囲（Bollinger Bands等） | データが季節性・変動性を持つ場合 |

**Bollinger Bands（動的閾値）**:
- 中央線: 移動平均
- 上限/下限: 標準偏差 ×2（通常95%信頼区間）
- この帯から外れたとき初めてアラートを発する

### 4-4. Agentic AI UX

複数のAIエージェントが協調して動作するシステムのUX設計:

**Supervisor + Worker Agentモデル**:
```
ユーザー → Supervisor Agent（タスク分解・調整）
              ├─ Worker Agent A（サブタスク1）
              ├─ Worker Agent B（サブタスク2）
              └─ Worker Agent C（サブタスク3）
              ↓
ユーザーへの統合結果
```

**非同期UI（Async UI）の設計原則**:
- タスク開始後すぐに「処理中インジケーター」を表示
- バックグラウンド処理中もユーザーは別作業ができる
- 完了通知はプッシュ型（ユーザーがポーリングしない）

**必須コントロール**:
| コントロール | 役割 |
|------------|------|
| **Start** | エージェント起動、進捗の可視化開始 |
| **Stop** | 即時停止、ロールバック可否を明示 |
| **Pause/Resume** | 一時停止と再開、長時間タスク向け |

**Shorthand UX Notation（AI Promptとして使う）**:
- UIコンポーネントを記号で表現し、LLMに設計意図を伝える
- 例: `[Input: user_query] → [AI: search] → [List: results] → [Detail: item]`
- プロトタイプ前の高速設計検討ツールとして活用

## 5. リサーチ手法

### 5-1. MUSE / Disciplined Brainstorming

**MUSE**（Mindful, Unusual, Surprising, Evolving）ブレインストーミングの手順:

1. **Mindful**: 現在の課題を付箋に書き出す（判断なし）
2. **Unusual**: 常識外れのアイデアを意図的に出す
3. **Surprising**: チームが驚く組み合わせを探す
4. **Evolving**: 有望なアイデアを発展させる

**Bookending手法**:
```
アイデアAを限界まで展開して描く
     ↓
アイデアBを同様に限界まで展開して描く
     ↓
両端を比べて中間点を探す
     ↓
「文書化してから次へ」（一つに固執しない）
```

> **ポイント**: ブレインストーミング中は「記録して次へ」が鉄則。一つのアイデアに固執すると他の可能性が潰れる。

### 5-2. AI-Inclusive UCD（ユーザー中心設計）

AIプロダクト開発において、UXの役割は「UI+AI+Data」の三角形全体を設計すること:

```
         UI（使いやすさ）
           /\
          /  \
         /    \
        /      \
      AI ────── Data
（予測精度）   （品質・多様性）
```

**Spike概念**:
- 実装前に「AIが動くか？」を素早く検証するUXスパイク
- 本物のモデルではなく、Wizard of Oz方式（人が裏で動かす）でUIを先に検証
- **Paper Prototype**: 紙UIで操作感をテスト
- **Python Prototype**: LLMにAPIでシンプルなロジックを実装して検証

**UXリサーチャーの役割変化**:
- 従来: ユーザーインタビュー・ユーザビリティテスト
- AI時代の追加役割: データバイアス調査・AI出力品質の評価・合成ユーザーの監査

### 5-3. RITE vs 従来ユーザビリティテスト

> **AskUserQuestion**: リサーチ計画を立てる前に、
> 「今回のAI機能は反復的に修正できますか、それとも大規模リリースですか？」を確認する。

**3つの根本的違い**:

| 軸 | 従来ユーザビリティテスト | RITE（Rapid Iterative Testing） |
|----|------------------------|--------------------------------|
| **修正タイミング** | 全テスト完了後 | 各セッション後に即時修正 |
| **参加者数** | 5〜8人（一度に） | 3〜4ラウンド × 3〜4人 = 9〜16人 |
| **目的** | 問題発見・一覧化 | 問題発見 → 修正 → 再検証のサイクル |

**RITE実施手順**:
```
Round 1: 3〜4人でテスト
     ↓
即時修正（1〜2日以内）
     ↓
Round 2: 3〜4人でテスト（修正後）
     ↓
即時修正
     ↓
Round 3〜4: 同様に繰り返す
     ↓
9〜12人完了時点で「完成度チェック」
```

**AI-Assisted RITE**:
- セッション録画をLLMで自動書き起こし・要約
- 複数セッションの共通問題をLLMで抽出
- 修正案のブレインストーミングにLLMを活用

**合成ユーザー（Synthetic Users）の限界**:
- AI生成の「架空ユーザー」はリアルなユーザーの代替にならない
- 合成ユーザーは既知のパターンしか再現できない
- 必ず実際のユーザーとRITEセッションを行うこと

## 6. バイアス・倫理チェックリスト

### 6-1. AI Humanifesto（5原則）

AIプロダクトを設計するとき、以下の5原則を確認する:

| 原則 | 核心 | チェック項目 |
|------|------|------------|
| **Control（制御）** | ユーザー自律性・ガバナンス | ユーザーはAIのオン/オフを制御できるか？ データの使われ方が透明か？ |
| **Trust（信頼）** | 透明性・信頼性・説明可能性 | AIの判断根拠をユーザーに説明できるか？ 誤りを認め修正できるか？ |
| **Diversity（多様性）** | 包括性・プライバシー・堅牢性 | 訓練データは多様か？ 誰が「表現されていないか」を確認したか？ |
| **Safety（安全）** | 公平性・同意・ウェルビーイング | プライバシー保護は十分か？ 依存性・感情的害のリスクは評価したか？ |
| **Balance（バランス）** | 持続可能性・人間の力の増幅 | AIは人を置き換えるか、補完するか？ 環境コストは最小化されているか？ |

### 6-2. バイアス認識チェックリスト

**「すべてのAIはバイアスを持つ」** を前提に、以下を確認する:

#### 生成AI出力のバイアスチェック

- [ ] 生成結果に「誰が欠けているか」を確認したか？（サバイバーバイアス視点）
- [ ] 職業・役割のステレオタイプが含まれていないか？（例: 「科学者」→白人男性）
- [ ] ネガティブな状態（例: うつ）が特定の人口統計と結びついていないか？
- [ ] クエリに多様性を明示することで結果が改善されるか試したか？

#### バイアス対策のアプローチ

```
Step 1: 生成結果を「デフォルト出力」として観察
Step 2: 欠けているグループ・属性を特定
Step 3: クエリに多様性を明示して再生成
   例: "biologist" → "Black woman biologist", "Asian male biologist"
Step 4: 結果のバリエーションを文書化
Step 5: 製品に使用する前にチームでレビュー
```

#### サバイバーバイアス（見えないものを見る）

戦時中の航空機分析（被弾箇所の研究）から学ぶ原則:
「データに表れているものではなく、**データに表れていないもの**に着目せよ」

AIの文脈で:
- 「よく使われる機能」だけを改善するのではなく、「使えていないユーザー」を調査する
- 「成功したAI予測」だけを見るのではなく、「見逃した失敗」を分析する

#### 倫理的テストの実践

- [ ] 多様なユーザーグループでAIのアウトプットをテストしたか？
- [ ] 別のAIモデルをジャッジとして使い、バイアスを相互チェックしたか？
- [ ] Value MatrixでFP/FNの人的コストを評価したか？
- [ ] 環境コスト（エネルギー消費）を最小化する設計を検討したか？

#### 倫理をデザインに組み込む実践

- **アカウンタビリティUI**: AIの決定根拠をユーザーが確認できる画面を設ける
- **フィードバックループ**: ユーザーがAI出力を「良い/悪い」と評価できる仕組み
- **代替提案型Guardrails**: 禁止するだけでなく、適切な代替を提案する
- **コンセント設計**: データ収集・利用についての明確な同意フロー

> **詳細**: 包括的なバイアス・倫理ガイドは `references/RESEARCH-AND-ETHICS.md` を参照。
