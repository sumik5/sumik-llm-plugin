## 1. AIの歴史がデザインに与える影響

ユーザーがAIプロダクトへ持ち込む誤解や前提は、AIの歴史的文脈から形成されている。この経緯を理解することがデザインの出発点になる。

### 技術進化の三段階

| 時代 | 代表例 | ユーザーへの影響 |
|------|--------|---------------|
| 記号主義（1950〜70年代） | ELIZA、初期チャットボット | 「ルールに従う機械」という認識 |
| 統計モデル台頭（1980〜2010年代） | PageRank、T9、SmarterChild | 「賢い推測ツール」という認識 |
| 大規模言語モデル（2017〜現在） | GPT、Claude、Gemini | 「人間のようなもの」という誤認識 |

### なぜ今のユーザーは混乱するのか

LLMが登場した際、ユーザーはすでに「検索エンジン」「スマートアシスタント」「チャットボット」の経験を持っていた。LLMの入力欄は見た目がGoogleの検索窓と似ているが、その裏にある技術は根本的に異なる。

**デザイン原則**: インターフェースの視覚的類似性がユーザーの誤解を助長することがある。入力欄の設計時には、既存の類似体験との違いを明示する機会を設ける。

## 2. ELIZA効果と擬人化の罠

1960年代に開発された会話プログラムELIZAは、単純なパターンマッチングでユーザーの言葉を反射するだけだったが、ユーザーはそこに「理解」「感情」「知性」を見出した。開発者自身も、自分の秘書がELIZAと1対1の会話を望んだことに衝撃を受けた。

このような**ELIZA効果**（言語的出力に人間らしい資質を投影する傾向）は、LLMの時代においてもより強力な形で再現される。

### ELIZA効果がデザインに与える影響

| リスク | 設計的対応 |
|--------|-----------|
| ユーザーが「理解されている」と過信する | システムの動作原理を適切に開示する |
| 出力を事実として無批判に受け入れる | 検証を促すUI（引用・免責事項）を設ける |
| 感情的な依存関係が生まれる | 長時間/個人的な会話への注意喚起を実装する |
| エラーに強い拒否反応を示す | エラーを「機械の限界」として正常化する |

**重要な洞察**: 言語はもともと人間だけが使うものだった。脳はテキストを受け取ると、人間レベルの他の能力（推論・感情・意図）の存在を自動的に予期する。これは進化的な反応であり、意図的なデザインでのみ緩和できる。

## 3. ユーザーの5つの誤った前提

研究によってAIプロダクトのユーザーが持つ典型的な誤解が特定されている。

| # | 誤った前提 | 実際 | デザインへの影響 |
|---|-----------|------|----------------|
| 1 | **人間的理解** AIは文脈・感情・ニュアンスを人間と同様に理解する | 統計的パターンマッチング。文脈の「理解」はない | モデルの動作をユーザーが実験できる機会を設ける |
| 2 | **事実性の仮定** AIの出力は正確で信頼できる | ハルシネーション（もっともらしい誤り）を生成する | ファクトチェックを促す設計；引用・根拠の表示 |
| 3 | **一貫性の期待** 同じプロンプトで常に同じ結果が出る | 確率的生成。毎回結果が異なる可能性がある | 出力の変動性をユーザーに周知する |
| 4 | **完全な知識** AIはすべての情報にアクセスできる | 学習データのカットオフ日以降の情報を持たない | 知識の限界を明示；リアルタイム検索ツールを提供 |
| 5 | **中立的な出力** AIの出力は偏りがない | 学習データに含まれる偏りを反映・増幅する | バイアスについての透明性；多様な視点の提示 |

### 誤解がプロダクト体験に与える悪循環

```
誤った前提を持つユーザー
        ↓
不適切なタスクにAIを適用
        ↓
期待はずれな結果が出る
        ↓
「AIは役に立たない」という誤った結論
        ↓
本来有用な機能の未発見
```

**ビジネス的文脈**: 年間数兆円規模のAI投資により、企業は機能を素早くリリースする圧力を受けている。この圧力がユーザー教育よりも機能誇示を優先させ、誤解を悪化させることがある。

## 4. デザイナーに必要な技術理解

AIデザインに必要な技術知識は「すべてを知ること」ではない。以下の3つの能力の組み合わせが実践的な出発点となる。

### 技術理解の適切なレベル

| 能力 | 必要な理由 | 実践方法 |
|------|-----------|---------|
| **ユーザー理解** 誰が使い、何を求めているか | AIは汎用的すぎるため、パーソナ設計が特に重要 | ユースパターン分析（チュートリアル探索/編集/創作等のクラスタリング） |
| **技術的流暢さ** モデルの能力・限界の把握 | 設計判断の品質が左右される | AIプロトタイピングの実践；エンジニアパートナーとの協働 |
| **評価スキル** 出力品質の判断 | システムが実際に機能するか確認できる | ベンチマーク（MMLU・GSM8K等）の概念理解；評価に参加 |

### プロトタイピングの重要性

静的ワイヤーフレームは、AIプロダクトの設計において機能しにくい。なぜなら：

- AIの出力は動的で文脈依存
- 実際に動かすまでわからない振る舞いがある
- 異なるモデル・パラメータで結果が大きく変わる

**推奨アプローチ**: vibe coding（自然言語でAI機能を記述し、生成されたコードを迭代する手法）などを活用して、設計者自身がAI機能の動作を早期に体験する。

### 組織内コミュニケーション

| ロール | AI知識の流通責任 |
|--------|---------------|
| PM | モデル能力の変化を設計・エンジニア・マーケティングに翻訳・配信 |
| エンジニア | 制約・信頼性・システム挙動を設計に共有 |
| デザイナー | ユーザーの声を評価・開発プロセスに持ち込む |
| リサーチ | ペルソナ・ユースケースを全ディシプリンで共有 |

## 5. モデル能力の理解：トレーニングと評価

### トレーニングの2段階

**事前学習（Pretraining）**

大量のテキストを使って「次のトークンを予測する」ことを学習する段階。膨大な言語パターン・知識・推論様式を獲得する。ただし、この段階のモデルは「指示に従う」ことを学んでいない。

**ファインチューニング（Fine-tuning）**

特定の行動（指示への従順、質問への回答、要約など）に向けてモデルを調整する段階。RLHF（人間フィードバックによる強化学習）では、人間のレビュアーが出力をランキングし、そのランキングで行動を調整する。

**デザインへの示唆**: モデルは「知っている」のではなく「学習データに基づいてもっともらしいものを生成する」。この本質的な違いをユーザーに伝える設計が重要。

### 評価（Evals）の種類

| ベンチマーク | 測定対象 | デザイナーへの意味 |
|------------|---------|-----------------|
| **MMLU** 57分野の多肢選択 | 一般知識と言語理解 | 幅広いトピックを扱う機能の信頼性の目安 |
| **GSM8K** 数学的推論 | 多段階論理 | 計算・分析機能の信頼性 |
| **MT Bench** 人間評価 | オープンエンドのタスク品質 | 実際のユーザー体験に最も近い評価 |

**デザイナー向けアドバイス**: 高いベンチマークスコアは現実のユーザー体験を保証しない。エンジニアパートナーに「どのモデルをどのタスクに使うか」を尋ね、設計スプリントに技術パートナーを含める。

## 6. 組織AI成熟度3レベルモデル

組織のAI成熟度は、情報がどのように流通し、責任がどのように定義され、設計がどの程度安定した基盤の上で行われるかによって決まる。

### Level 1: 個人探索段階

| 項目 | 状態 |
|------|------|
| モデル知識の所有 | 個人が持つ（共有構造なし） |
| 評価（Evals）の実施 | 未実施；プロダクト判断に影響しない |
| 責任あるAI対応 | 未割り当て；問題発生時に個別対応 |
| 能力共有 | 定例なし；アドホックな会話のみ |
| プロトタイピング | 個人の主導による；スクリプトやペルソナなし |
| 新機能の発案元 | 組織上位から下降；ユーザーニーズ検証なし |

**設計への影響**: 設計者は情報不足のまま動き、技術更新に追随できない。モデルの変更が設計決定に遅れて反映される。

### Level 2: 共有理解段階（手動メンテナンス）

| 項目 | 状態 |
|------|------|
| モデル知識の所有 | PM/エンジニアが担当を始める |
| 評価（Evals）の実施 | 担当者が存在するが配布は手動 |
| 責任あるAI対応 | 初期的なフェアネス・安全基準が形成 |
| 能力共有 | 散発的（関心主導、定例なし） |
| ペルソナ・プロトタイプ | 初期段階；軽い調査に基づく |
| コンテンツ警告 | 初期基準が存在 |

**設計への影響**: 一貫性が高まるが、手動の労力・頻繁な調整が必要。設計者は情報にアクセスできるが、能動的に調整しなければならない。

### Level 3: 統合AI組織段階

| 項目 | 状態 |
|------|------|
| モデル知識の所有 | 明確な担当者がいる；変更は確立されたチャネルで配信 |
| 評価（Evals）の実施 | 専任チームが所有；結果が設計・エンジニア・リーダーに自動共有 |
| 責任あるAI対応 | 安定した拠点があり、計画・レビュー・リリースに組み込まれる |
| 能力共有 | 定期的なケイデンスで高参加率 |
| ペルソナ・プロトタイプ | 検証済みペルソナ＋リアルタイムモデルデータ活用 |
| 新機能の発案元 | リサーチとユーザーニーズから発生（ボトムアップ） |

**設計への影響**: 設計者は安定した基盤の上で意思決定できる。ユーザーはAI機能を「後付け」ではなく直感的に感じる。変化への適応がスムーズ。

## 7. 4つの発見パターン

ユーザーがAI機能を発見するパターンは4つに分類できる。各パターンは「ユーザーのIntent（意図）」と「システムのInitiative（主導性）」の組み合わせによって異なる設計アプローチを必要とする。

### 発見設計のための2×2マトリクス

```
                    システムInitiative
                  低                    高
            ┌─────────────────┬──────────────────┐
    低      │  自然発生的      │  アンビエント/    │
ユーザー    │  （社会的共有）  │  プロアクティブ   │
Intent     │                  │  （文脈認識型）   │
            ├─────────────────┼──────────────────┤
    高      │  入力ベース     │  プログレッシブ   │
            │  （コマンド等）  │  （段階的開示）   │
            └─────────────────┴──────────────────┘
```

### パターン1: 入力ベース発見

ユーザーが自分のアクションを通じて機能を発見する。最もユーザー主導的なパターン。

| 手法 | 例 | 設計ポイント |
|------|-----|------------|
| コマンドトリガー | スペースバーでAIパレットを開く（Notion） | 入力中に候補を表示；新規ユーザーにも探索可能に |
| ジェスチャー/インタラクション | テキスト選択でコンテキストメニューにAIオプション | 既存の操作パターンに溶け込ませる |
| 空状態の活用 | 空のドキュメントに「予算を作成しますか？」 | ブランクページの不安を解消するタイミングで提示 |

### パターン2: コンテキスト認識発見

システムがユーザーの環境を読み取り、適切なタイミングで機能を提案する。

| 手法 | 例 | 設計ポイント |
|------|-----|------------|
| コンテンツ分析 | 議事録フォーマットを検出し「アクション項目を要約」を提示 | ユーザーの作業文脈への理解を示す |
| 行動パターン認識 | 繰り返し操作に「ルールを作成しますか？」 | 無意識な非効率に気づかせる |
| 位置情報ベース | カーソルが数式セルの上にあるとき「数式を追加」 | 注意が集中している場所でのみ表示 |

### パターン3: アンビエント/プロアクティブ発見

システムがユーザーの要求なしに先手を打つ。背景で動く「安全ネット」的な役割。

| 手法 | 例 | 設計ポイント |
|------|-----|------------|
| 異常検出 | カレンダー重複・添付忘れの警告 | 問題が発生する前に介入 |
| 予測的サーフェシング | 外国語ページで翻訳を提案 | ユーザーが気づく前にニーズを充足 |

### パターン4: プログレッシブ発見

機能を段階的に公開し、信頼構築のペースに合わせる。

| 手法 | 例 | 設計ポイント |
|------|-----|------------|
| 機能の昇格 | 文法修正が提案から常設メニューへ | 採用実績に基づいて露出を高める |
| 文脈的チュートリアル | 特定のコンテンツ種類を初めてアップロードした時だけ新機能を紹介 | 即時関連性が高い瞬間に絞る |

## 8. 採用とモメンタム行動

機能を「発見した」だけでは採用につながらない。ユーザーが自分のワークフローに機能を組み込むまでには心理的・設計的な障壁がある。

### モメンタム行動の仕組み

ユーザーは一度ワークフローを決めると、より良い方法が存在しても既存の方法に固執する（**モメンタム行動**）。

> インターフェースのデザインが「必要な時に呼び出せる強さ」を持たない場合、ユーザーはトンネルビジョンに入り、有用な機能を見過ごす。

**実践的データ**: モバイルアプリのオンボーディングが1分以内に完了しない場合、多くのユーザーが離脱する。2分を超えると継続利用者は大幅に減少する。

### 採用促進の設計戦略

| 戦略 | 手法 |
|------|------|
| **戦略的介入** | モメンタムが最も崩れやすい瞬間（初回体験・習慣変化時）に機能を提案 |
| **包括的教育** | ワークフロー外でもアクセスできるドキュメント・チュートリアルを提供 |
| **コミュニティ活用** | パワーユーザーのワークフロー共有を促し、公式ドキュメントと統合 |

### AIの高速イノベーションが引き起こす課題

AIの機能は継続的に変化する（**plastic state**）。これは従来のソフトウェア（機能が数年間変わらない）とは異なり、以下の問題を生む：

- ユーザーが学習したメンタルモデルが頻繁に無効化される
- **アルゴリズム転移**：過去のAI製品での経験（誤解を含む）を新製品に持ち込む
- 変更に対する不安・混乱を管理するための追加メッセージングが必要

## 9. オーケストレーション概念

ユーザーがAIツールを使い始める前に、多くの重要な設定判断が行われる。この「オーケストレーション層」の設計がユーザー体験全体の土台となる。

### オーケストレーション設計の3原則

| 原則 | 内容 |
|------|------|
| **意図的** | 各選択肢が何を有効/制限するかをユーザーが理解できる状態で提示 |
| **透明** | モデル切替・権限付与・プラグイン有効化などの変化を可視化 |
| **認識可能** | 設定変更後もユーザーが自分のメンタルモデルを維持できる；一貫したラベリング |

### モデル・エージェント・ツールの3層構造

```
    ┌─────────────────────────────────┐
    │           エージェント           │ ← 計画・自律実行・複数ステップ
    ├─────────────────────────────────┤
    │             ツール              │ ← Web検索・ファイル読取・DB接続
    ├─────────────────────────────────┤
    │            モデル              │ ← テキスト生成（反応的；自律性なし）
    └─────────────────────────────────┘
```

| 層 | 設計上の主な考慮事項 |
|---|-------------------|
| **モデル** | 入力・出力フォーマット；生成フィードバック；速度/品質のトレードオフ表示 |
| **ツール** | 権限管理；アクセス理由の可視化；エラー処理 |
| **エージェント** | システムの動作の表示；一時停止・修正の機会；複雑なワークフローへの信頼構築 |

### モデル選択のUXパターン

技術仕様（モデル名）ではなく、結果志向の記述でモデルを提示する：

| ❌ 技術的記述 | ✅ 結果志向の記述 |
|-------------|----------------|
| GPT-4 / Claude Sonnet | 詳細な分析に最適 |
| Claude Haiku | 素早い返答に最適 |
| GPT-4o with vision | 画像を含むタスクに対応 |

**自動ルーティング**（クエリ内容に基づいて最適モデルを自動選択）は認知負荷を下げるが、透明性のトレードオフがある。モデルバッジやツールチップで「現在使用中のモデルとその理由」を表示することで、速度・品質の変動をユーザーが理解できる。

### メータリング（使用量管理）のUX

AIの計算コストは変動が大きい（単純な補完 → 複雑な推論で数百倍の差がある）。

| メータリング要素 | 設計のポイント |
|----------------|-------------|
| 残量の可視化 | セッション開始前に制限を開示する（不意の中断を防ぐ） |
| コスト抽象化 | 金額よりもトークン・ポイントなどの独自単位に変換 |
| 再開可能性 | クレジット切れ時に作業状態を保存；復帰時に再開できる設計 |
| 低優先タスクの抑制 | 高価な処理への「クイック回答を取得」等のエスケープハッチ |

### 権限設計のポイント

従来のアプリと異なり、AIは動的に複数データソースへのアクセスを必要とする。

- **粒度の細かい制御**: 「この会話のみ」「プロジェクトフォルダ」「全ファイル」など段階的な権限
- **取り消しの容易さ**: 権限付与と同等の簡単さで取り消せる
- **変更の可視化**: 新しい権限が有効になった際の明確な表示
- **ゲートキーパーではなくパートナー**: 権限設定をユーザー教育として機能させ、信頼を構築する
