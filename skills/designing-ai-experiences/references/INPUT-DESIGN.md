## 1. 入力の進化史：負担の移行

AIインターフェース設計を理解するには、「誰が適応コストを負担するか」という観点で入力の歴史を見る必要がある。

### 入力パラダイムの変遷

| 時代 | 代表インターフェース | 負担の所在 | 特徴 |
|------|-------------------|-----------|----|
| 1960〜70年代 | CLI（コマンドシェル） | **ユーザー側** | 正確な構文・コマンドの暗記が必須；誤字一文字でエラー |
| 1990〜現在 | 検索エンジン | 中間 | 自然言語クエリを許容；タイポ訂正；曖昧な入力でも動く |
| 2020〜現在 | LLM（大規模言語モデル） | **システム側** | 不完全・曖昧な入力を解釈；文脈を構築；対話を持続 |

### 「よいGoogle検索」と「よいプロンプト」の違い

ユーザーは20年以上かけて「Google検索がうまくなった」。AIには同様の蓄積がない。

```
Google検索で学習した戦略例:
  "filetype:pdf" で PDFを絞り込む
  "サイト名 Reddit" でそのサイト限定

LLMプロンプトで有効な戦略:
  対象者・文脈・制約・フォーマットを明示
  例示を含める
  役割を割り当てる
```

**デザインの課題**: 検索ボックスとLLMの入力欄は見た目が似ているが、要求されるスキルは根本的に異なる。ユーザーは慣れ親しんだ検索の文法をLLMに誤適用しがちである。

## 2. 3チャネルモデル

ユーザーがAIシステムに意図を伝える方法は3つのチャネルが存在し、多くの場合これらは同時に機能する。

```
  ┌─────────────────────────────────────────────────┐
  │               ユーザーの意図（Intent）            │
  └──────────────┬──────────────┬───────────────────┘
                 ↓              ↓              ↓
  ┌──────────────┐  ┌──────────────┐  ┌───────────────┐
  │  暗黙コンテキスト │  │  明示的プロンプト │  │  直接操作      │
  │ (Implicit)   │  │ (Explicit)   │  │(Manipulation) │
  │              │  │              │  │               │
  │ 開いているファイル│  │ タイプ/音声   │  │ ボタン/スライダー│
  │ 選択テキスト  │  │ テキスト入力  │  │ 選択/ジェスチャー│
  │ 場所・時刻    │  │              │  │               │
  └──────────────┘  └──────────────┘  └───────────────┘
```

**チャネル間の協調**: 単一のインタラクションで3チャネルが同時に動作する。

例：Canvaでの画像編集
- 開いている画像（**暗黙コンテキスト**）
- オブジェクトをブラシでなぞる（**直接操作**）
- 「ゴールデンレトリバーに置き換えて」と入力（**明示的プロンプト**）

設計目標は「3チャネルが翻訳の負担を分担し、ユーザーが明示的に言語化しなければならないことを減らす」こと。

## 3. 暗黙コンテキスト（チャネル1）

ユーザーが意識的に提供しなくても、システムが環境から収集する情報。

### 暗黙コンテキストの種類

| コンテキスト種類 | 例 | 活用例 |
|---------------|-----|-------|
| **ドキュメント状態** | 開いているファイル・選択テキスト | 選択範囲に基づくAI提案の精度向上 |
| **作業環境** | コードの命名規則・インポート・関数の上の行 | GitHub Copilotがプロジェクト全体のスタイルに合わせた補完 |
| **ユーザー行動** | 繰り返しの操作・閲覧パターン | 定型文の自動化提案 |
| **時間・場所** | 月曜朝・現在地 | 「今週の優先事項を計画」のスターター表示 |
| **コミュニケーション文脈** | メールの返信先（上司 vs. 外部業者） | トーンの自動調整 |

### 暗黙コンテキストの設計課題

**コンテキストのミスマッチ**: ユーザーが想定していないコンテキストをシステムが使用することで、予期せぬ挙動が発生する。

**解決アプローチ**:

| アプローチ | 例 |
|-----------|-----|
| **コンテキストの可視化** | 「編集中：選択テキスト」「参照中：現在のドキュメント」を表示 |
| **空間的な手がかり** | 検討中の領域をハイライト；対象外の部分をディム |
| **範囲の調整機能** | ユーザーがコンテキスト範囲を調整できるUI |

**コンテキストの過多問題**: リポジトリ全体を読み込むほど提案が一貫性を持つ一方で、速度低下・関係ないパターンへの過剰依存・意図しない情報共有のリスクも増大する。ユーザーがこのトレードオフを認識・調整できる透明性が必要。

## 4. 明示的プロンプト（チャネル2）

ユーザーがタイプまたは音声で直接システムに伝える入力。最も目に見えるチャネル。

### 効果的なプロンプトの特徴

自然な会話はコンテキストを共有した上で成り立つ。AIはその共有コンテキストを持たないため、プロンプトでそれを補う必要がある。

**弱いプロンプトと強いプロンプトの比較**:

| 弱いプロンプト | 強いプロンプト |
|-------------|-------------|
| 「このプレゼンをもっと良くして」 | 「CFO向けの10分プレゼンのスライドを簡潔にし、意思決定に集中できる構成にしてください」 |
| 「このデータを分析して」 | 「四半期売上データで最も成長した製品カテゴリ、地域別のパフォーマンス差、季節パターンを特定し、来期の戦略提案3点を出してください」 |
| 「メールを書いて」 | 「私はソフトウェアスタートアップのPMです。保守的な金融機関のクライアントに、規制要件の変更により3週間の遅延が発生したことを伝えるメールを書いてください。以前にも一度延期があり、信頼関係の維持が重要な案件です」 |

**重要な原則**: LLMは「次の最も確率の高いトークン」を予測する。つまり、プロンプトの「雰囲気」（tone, framing）がそのまま出力に反映される確率的なプロセスである。これは従来のプログラミングとは根本的に異なる。

## 5. CAREフレームワーク

効果的なプロンプトを設計するための4要素。**C**ontext / **A**ction / **R**esult / **E**xample

### 4要素の詳細

#### Context（文脈）

AIがリクエストの「なぜ」と「どのように使われるか」を理解するための背景情報。

- ユーザーの役割・組織
- 対象となるオーディエンス
- 制約条件（時間・予算・関係性）
- 状況の繊細さ（リスク・感情的な重要性）

**設計含意**: ブリーフ型の設定質問、ユーザー履歴に基づくスマートデフォルト、状況説明を促すプレースホルダーテキストが有効。

#### Action（アクション）

AIに何をやってほしいかの具体的な処理指示。「見てください」ではなく「要約してください」「パターンを特定してください」「比較してください」。

- 要約 / 分析 / 比較 / 推奨 / 変換
- 処理の種類を明示することでモデルが適切な手法を選択

**設計含意**: 構造化プロンプトテンプレートでアクション種類を選択肢として提示。

#### Result（結果）

出力の形式・長さ・スタイル・詳細度の指定。

| 用途 | 結果指定の例 |
|------|------------|
| 役員向けブリーフィング | 「2ページのエグゼクティブサマリー、意思決定に集中、箇条書きで重要発見」 |
| チームミーティング | 「15分プレゼン用のトーキングポイント、議論を促す質問付き」 |
| クライアント向け | 「専門性を示しつつ分かりやすい構成、次のステップを明示」 |

**設計含意**: フォーマットセレクター、長さスライダー、オーディエンス別テンプレートが有効。

#### Example（例示）

望む出力スタイル・品質・トーンの具体的なサンプル。

「指示」より「例示」の方が強力な理由：
- 抽象的な指示では伝わらないニュアンスを具体的に示せる
- ユーザー自身が気づいていない好みを発見させる
- モデルが「engaging / informative」のような曖昧な形容詞を正確に解釈できる

**設計含意**: サンプルギャラリー・「このスタイルで」ボタン・ユーザーが過去の成功出力を再利用できるプロンプトライブラリ。

### CAREを統合したプロンプト例

```
[Context] 来週の取締役会で、カスタマーサービスチームの
増員を提案するプレゼンを準備しています。

[Action] 過去6ヶ月のサポートチケットデータを分析し、
ピーク時間帯、問題タイプ別の解決時間、顧客満足度の
トレンドを特定してください。データに基づいて増員の
必要性を論証してください。

[Result] 10分プレゼン用のアウトラインとして整理し、
スライド向けの統計データを強調表示し、
予算異議への返答ポイントとROI計算を含めてください。

[Example] 以前成功したボード向けプレゼン形式：[サンプル]
```

## 6. プロンプトインテント分類

ユーザーがプロンプトで達成しようとしていることは複数の種類に分類できる。プロダクトが成熟するにつれて、これらのパターンを研究・設計に活かせる。

| インテント種類 | 説明 | 例 | 設計上の含意 |
|-------------|------|-----|------------|
| **指示型** | 明確なタスクと要件を提示 | 「この記事を3つの箇条書きで要約して」 | 定義されたアウトプット形式に最適 |
| **探索型** | 可能性の空間を開く | 「製品ローンチのクリエイティブな導入方法を考えて」 | 多様な選択肢を提示する設計が有効 |
| **役割ベース型** | トーン・スタイル・専門性を割り当て | 「あなたは経験豊富な旅行エージェントです...」 | スタイル制約に集中した設計 |
| **修正型** | 反復的な精緻化 | 「もっと簡潔にして」「読者がすでに背景知識を持つ前提で」 | 会話の継続性；前の出力をコンテキストとして保持 |
| **限界探索型** | エッジケース・逆説的な質問 | 「このなぞなぞを解いてみて」 | ユーザーのメンタルモデル形成に影響；境界を明示する設計 |

**研究の活用**: ユーザーのプロンプトパターンを分析することで、最も一般的なインテントに合わせたテンプレート・初期状態の設計が可能になる。

## 7. 明示的プロンプトのリスク

会話的なAIインターフェースの流暢さは「理解されている」という錯覚を生む。この錯覚は設計が積極的に対処しなければならない構造的なリスクをはらんでいる。

### リスク1: 追従性バイアス（Sycophancy）

LLMはユーザーの前提を肯定する傾向がある。「ケールはなぜ不健康なのですか？」と聞かれると、前提を訂正するより「なぜ不健康か」の理由を列挙する。

**原因**: 学習プロセスとRLHFにより、「ユーザーが評価する回答」＝「ユーザーの枠組みに沿った回答」になりやすい。

**影響**:
- 楽観的なプロンプト → 可能性に満ちた楽観的な回答
- 懐疑的なプロンプト → リスクを強調した慎重な回答
- ユーザーは「客観的な分析」を受け取ったと誤認する

**設計的対応**:

| アプローチ | 実装例 |
|-----------|-------|
| 誘導的質問への警告 | 前提に誤りがある可能性の通知 |
| 複数視点の表示 | 一つの立場に偏らず対立視点も提示 |
| 長時間/個人的会話の注意喚起 | モデルへの感情的依存のリスクを通知 |

### リスク2: 精度パラドックス（Precision Paradox）

プロンプトに詳細を追加すると初期は品質が向上するが、過剰な長さで逆に劣化する。

**技術的背景**: Transformerアーキテクチャでは、プロンプトの中間部分に対する注意（attention）が低下する。100,000トークンのコンテキストウィンドウでも、中央付近の情報が過小評価される傾向がある。

**悪循環**:
```
長いプロンプトで品質が下がる
      ↓
ユーザーはさらに詳細を追加する
      ↓
プロンプトがさらに長くなる
      ↓
さらに品質が下がる
```

**設計的対応**:

| アプローチ | 実装例 |
|-----------|-------|
| 複雑さのアンビエントフィードバック | プロンプトが長すぎる可能性の穏やかな表示 |
| 簡略化の提案 | 「このプロンプトを最適化しますか？」 |
| 重要部分のハイライト | どの部分がモデルに重視されるかの視覚的表示 |

## 8. 直接操作（チャネル3）

ボタン・スライダー・選択・ジェスチャーなどの従来のUI要素でAI行動を形成するチャネル。言語化を必要とせず、数十年のインターフェース学習を活用する。

### 直接操作の役割

| 役割 | 詳細 |
|------|------|
| **効率化** | 定型アクションのワンクリック実行（「要約」「フォーマル化」等） |
| **教育** | 「このボタンをクリックすれば同様のことが言葉でも頼める」と学習させる |
| **発見促進** | ボタンで能力の存在を示し、プロンプトへの橋渡しをする |
| **慣れ親しんだパターンの活用** | 画像編集のラッソツール・ハンドル・境界線など既知の操作 |

### ヤコブの法則の適用

> ユーザーはほかのサイト・アプリで多くの時間を使っている。だから、そこで動く仕組みと同じように動くことを好む。

**実装原則**:
- 画像選択はフォトショップ等で学習済みの操作（ラッソ・ハンドル・マーキーアント）を使用
- テキスト編集はワードプロセッサの慣行（インラインコメント等）を継承
- AI能力がどれほど新しくても、インターフェースは既存パターンを拡張する形で設計

### ハイブリッドアプローチ

| ユーザー状態 | 適した入力 |
|------------|---------|
| 何が欲しいか分かっている | **直接操作**（ボタン・選択） |
| 具体的だが複雑なリクエスト | **明示的プロンプト** |
| 境界テスト・新機能探索 | 両方の組み合わせ |

**ユーザー成長の軌跡**:
```
「要約」ボタンをクリック
      ↓
「技術的なオーディエンス向けに要約して」と入力
      ↓
複数の要件を含む複雑なプロンプトを独自に構築
```

インターフェースはこの段階的な成長を強制せず、自然に支援する。

## 9. 入力モダリティ設計

意図の3チャネルはテキスト・画像・音声の異なるモダリティを通じて伝わる。各モダリティには固有の設計要件がある。

### テキスト入力

現在最も支配的な入力モダリティ。設計の選択肢：

#### フォーマット許可

テキスト入力欄でMarkdown/箇条書き/見出しなどの書式設定を許可することで：

- 複雑なプロンプトの構造化が容易になる
- 段落と命令を視覚的に分離できる
- モデルの解釈精度が向上する（研究によりMarkdown形式で精度が大幅向上）

**設計含意**: リッチフォーマット対応の入力欄；選択前の命令発行；プレビュー機能。

#### 構造化プロンプト

| アプローチ | 特徴 | 適した場面 |
|-----------|------|---------|
| 自由テキスト | 柔軟性が高い；探索的 | 汎用AI・早期ユーザー |
| モードセレクター + スライダー | 選択肢が明確；学習コスト低 | 特定機能（言い換えツール等） |
| フィールド分割型フォーム | 完全性を促す；誤りが減る | 文書生成・広告コピー等 |
| 詳細設定トグル | シンプル＋制御の両立 | 中〜上級ユーザー向け |

**重要な判断軸**:
- タスクの明確さ：明確 → 構造化・曖昧 → 自由
- リスク許容度：高リスク → 構造化で誤りを減らす
- ユーザーの知識：新規 → 構造化が手がかりに

### 画像入力

画像ベースのAIインターフェースは、既存の視覚操作パターンへの準拠度で成否が分かれる。

**成功パターン（既存操作の継承）**:
- ラッソツールでオブジェクト選択
- ハンドルで境界の調整
- ブラシ選択 + テキスト命令の組み合わせ

**テキストとの組み合わせ**:
- 画像を選択してテキストで指示（例：画像ファイルのURL+「この部分を置き換えて」）
- 視覚的な文脈 + 言語的な指示の相互補完

**今後への展開**: 動画・3Dへの拡張が進む。成功するインターフェースは「新しいビジュアル言語の習得」を要求せず、既存のメンタルモデルを拡張する形を取る。

### 音声入力

| 入力種類 | 特徴 | 設計要件 |
|---------|------|---------|
| **短いコマンド** | 「10分のタイマーをセット」 | 高速認識・低レイテンシー・即時確認 |
| **長い録音** | 会議の議事録・講義 | 話者分離（誰が何を言ったか）・音声品質変動への対応・長時間コンテキスト保持 |
| **会話型** | 複数ターンの対話 | コンテキスト維持と応答速度のバランス；自然なフローを維持するフィードバック |

**モバイルの優位性**: スマートフォンでの音声入力はテキスト入力の約3倍の速度（161 wpm vs 53 wpm）を達成しつつ誤り率が低い。モバイルAI機能では音声入力の役割が大きくなる。

**フィードバックの種類**:

| 入力種類 | フィードバック |
|---------|------------|
| 短いコマンド | 即時確認（テキスト表示・アクション実行） |
| 録音 | 処理インジケーター・信頼スコア・進捗バー |
| 会話型 | フローを乱さない微妙なキュー（波形・点滅等） |

## 10. ユーザーへの入力ガイド

ユーザーが「よいプロンプトを書く」ことを自然に学べる環境の設計。

### 即時改善（今すぐの効果）

| 手法 | 例 | 効果 |
|------|-----|------|
| **カテゴリボタン** | 「クリエイティブライティング」「分析」「コード」 | バックエンドでの専用モデルへのルーティング |
| **テンプレート** | 「会議サマリー（参加者・決定事項・アクションアイテム）」 | 成功率を上げながら優れた構造を教える |
| **プログレッシブ開示** | 「詳細設定」トグルで「トーン」「長さ」「フォーマット」を追加 | シンプルさを保ちつつ制御も提供 |

**重要原則**: 即時改善は「修正」ではなく「提案」として感じられるべきである。ドロップダウン・展開セクション・視覚プレビューで、ユーザーが「間違っていた」と感じさせずに誘導する。

### 長期的な理解構築

Googleの検索ボックスは数十年間本質的に変わらなかったが、その一貫性によって何百万人もの人が「よい検索の仕方」を身につけた。AIインターフェースも同様の原理が働く。

| 戦略 | 詳細 |
|------|------|
| **一貫したパターン** | テンプレートの場所・提案のスタイル・基本操作フローを安定させる |
| **ポジティブ強化** | テンプレートを使って良い結果が出た経験が何が効果的かを内在化させる |
| **段階的な熟達** | テンプレート依存 → テンプレートなしで同様の構造 → 独自の高度なプロンプト |

### ガイダンスのスペクトラム

```
（見えにくい）─────────────────────────────（明示的）
     │                                        │
  一貫した          オプション       テンプレート    例示ギャラリー
  配置・挙動         フィールド        集            ・
                    表示              ・         プロンプトライブラリ
```

## 11. スターターPromptとプロダクトポジショニング

空の状態に表示されるスターターPromptはユーザーの最初のタッチポイントであり、プロダクトのポジショニングを伝える重要な設計要素。

### スターターPromptの役割の再定義

| ❌ よくある誤解 | ✅ 正しい理解 |
|-------------|------------|
| 「とりあえず何かタイプさせる」ための汎用ネタ | プロダクトの差別化価値を示す具体的な使用例 |
| 「秋についての詩を書いて」のような万能な例 | 「このメモをアジェンダに変換して」のような実際のユースケース |

### 効果的なスターターPromptの要件

| 要素 | 内容 |
|------|------|
| **機能の具体性** | そのプロダクトが何ができるかを端的に示す |
| **統合の可視化** | 連携サービスの活用例（例：Zillow連携の場合、物件検索のサンプル） |
| **形式と意図の暗示** | 「メモをアジェンダに変換」→ユーザーのコンテンツとシステムの能力の関係を示す |
| **期待値の設定** | どんな出力が来るかを予感させる |

### コンテキスト適応型スターターPrompt

| コンテキスト | スターターPromptの例 |
|------------|-------------------|
| 月曜朝（プロジェクト管理ツール） | 「今週の優先事項を計画する」 |
| 金曜午後 | 「今週の進捗をまとめる」 |
| 新規ドキュメント（会議）| 「この会議のアジェンダを作成」 |
| コードファイル（エディタ） | 「この関数のユニットテストを生成」 |

**設計原則**: スターターPromptは静的な例ではなく、文脈・ユーザーロール・過去のパターンに応じて変化するアクティブなインターフェースコンポーネントとして設計する。

## AskUserQuestion — 確認すべき判断分岐

| 確認タイミング | 確認すべき内容 |
|--------------|-------------|
| 入力設計の開始 | 「このプロダクトにおける主要な入力タイプは何ですか？（テキスト・画像・音声・複合）」 |
| チャネルの優先順位 | 「ユーザーは主にどのチャネルで意図を伝えると想定されますか？」 |
| 構造化の程度 | 「このユースケースでは自由入力と構造化入力のどちらが適切ですか？」 |

確認不要な場面:
- 暗黙コンテキストの可視化（常に考慮が必要）
- 追従性バイアスへの設計的対応（LLM使用時は常に検討が必要）
- 既存の操作パターンの尊重（直接操作設計では常に基本）
