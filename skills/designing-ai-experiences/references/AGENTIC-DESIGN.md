## 1. エージェントAIとは何か

エージェントAIは通常のAIとは異なり、**自律性（Autonomy）**を持つ。精密な指示を与えなくても、問題を探索し、仮説を立て、解決策を見つける能力がある。

**AIとエージェントAIの最大の違い:**

| 観点 | 通常のAI | エージェントAI |
|------|---------|--------------|
| 動作モード | 質問→回答（単一交換） | 目標→計画→実行→適応（多段階） |
| 自律性 | なし（指示依存） | あり（目標から行動を自律決定） |
| ツール使用 | なし | データベース・API・外部サービス等を呼び出し |
| エラー処理 | ユーザーが気づく | 自己評価・修正のループを持つ |
| 適切な制御 | 不要 | 権限設計・チェックポイント・ロールバックが必須 |

**マルチエージェントの構造（アリの巣メタファー）:**
- **ワーカーエージェント**: 特定タスクに高度に特化
- **スーパーバイザーエージェント（推論エージェント）**: ワーカーを指揮・結果を統合
- **人間オペレーター**: 一般的なリクエストを出し、アウトプットをレビューする

## 2. 5つのAgenticデザインパターン

5つのパターンは相互排他的でなく、実際のシステムはこれらを組み合わせる。

### 2-1. Reflection（自己評価）パターン

**概念**: エージェントが自己の出力を評価・修正するループを持つ。「生成→評価→改善」のサイクル。

**仕組み:**
- 初期出力を生成した後、同じまたは別のモデルが基準に基づいてその出力を批評する
- 批評に基づいて出力を再生成または調整する
- 高品質な出力のみをユーザーに提示する

**ユーザーが体験する形:**
- 「レビュー中...」「計算を確認中...」といったステータスメッセージ
- 処理時間の延長（反省サイクルごとに時間が増加する）
- 改善された最終出力（内部の試行錯誤は見えない）

**UX設計ガイド:**

| 課題 | 設計アプローチ |
|------|-------------|
| **レイテンシーの説明** | 「品質確認中...」等のメッセージで待機理由を明示する。期待値を最初から設定する |
| **透明性レベルの選択** | 高ステークス作業: 確認ステップを表示（「Q3合計でエラーを発見、修正・再検証済み」）。低ステークス作業: 改善された出力のみ提示 |
| **Progressive Disclosure** | デフォルトでは最終出力のみ表示。Chain of Thought（詳細確認オプション）で内部プロセスをオプトイン提供 |
| **最終出力のみ vs 変更差分** | 変更差分の表示は価値あるが認知負荷を増やす。デフォルト: 最終出力のみ。オプション: 何が確認・変更されたかのサマリー |

**高ステークスな活用場面**: 財務計算・コード生成・コンプライアンスチェック（エラーのコストが高い領域）。

---

### 2-2. Tool Use（ツール使用）パターン

**概念**: エージェントが外部システムを呼び出し、実際にアクションを実行する。「アドバイザー」から「オペレーター」への変換。

**ツールの種類:**
- 情報取得ツール: Webサーチ・データベースクエリ・文書アクセス
- 計算ツール: 電卓・コード実行環境
- 統合ツール: カレンダー・メール・CRM等の外部サービス

**ユーザーが体験する形:**
- 「カレンダーを確認中...」「顧客記録を更新中...」等のステータス更新
- 「作業カレンダーへのアクセスを許可しますか？」等の権限リクエスト
- 取られたアクションの確認メッセージ

**UX設計ガイド:**

| 課題 | 設計アプローチ |
|------|-------------|
| **権限設計** | ツールタイプ別に承認要否を分ける（読み取り専用: 自動許可、ファイル変更: 都度確認、シェル実行: 都度確認）。永続的許可オプションも提供 |
| **可視性と制御** | エージェントが何にアクセスしているか・なぜかを示す。ユーザーが実際・認知両面での制御感を持てるよう設計 |
| **エラーハンドリング** | 「APIが応答していません。再試行するか、このステップをスキップしますか？」等、方向性を与えるエラーメッセージ |
| **スコープ設定** | コンテキストに応じてツールアクセスを制限する（汎用AIは本番DBを変更すべきでない等） |

**権限フレームワーク例（ティア制）:**

| ツールタイプ | 例 | 承認要否 | 「今後確認しない」動作 |
|-----------|-----|---------|------------------|
| 読み取り専用 | ファイル読込・検索 | 不要 | N/A |
| シェルコマンド | コード実行 | 必要 | プロジェクト・コマンドごとに永続 |
| ファイル変更 | 編集・書込 | 必要 | セッション終了まで |

### 2-3. Planning（計画立案）パターン

**概念**: 複雑なゴールをサブタスクの構造化シーケンスに分解する。人間が問題をToDoリストで整理するのと同様。

**計画の構成要素:**
- **線形シーケンス**: ステップ1→2→3の順次実行
- **依存関係**: 先行タスクの完了を待つ条件
- **条件分岐**: 「テストが通れば X、失敗すれば Y」等の条件ロジック
- **階層構造**: 高レベルのステップが詳細なサブステップを含む

**ユーザーが体験する形:**
- 実行前の計画プレビュー（「ステップ1: データ収集、ステップ2: 分析、ステップ3: 可視化、ステップ4: サマリー草案。このアプローチでよいですか？」）
- 実行中の進捗インジケーター（「4ステップ中2ステップ完了: Q3レポートのトレンド分析中...」）
- フェーズ別の状態更新

**UX設計ガイド:**

| 課題 | 設計アプローチ |
|------|-------------|
| **計画の可視性** | 高ステークス・長時間タスク: 実行前に計画を表示。定常タスク: 内部で計画し結果のみ提示。両方のモードをサポートする |
| **計画の粒度** | 1分未満: 3ステップ程度。数分以上: フェーズ分けし詳細はオンデマンドで展開 |
| **介入ポイント** | 計画をユーザーが評価・方向修正できるチェックポイントを設ける |
| **条件分岐の表示** | 主要パスを表示し、分岐は実際に発生した時点で説明する（過剰な可能性の事前開示を避ける） |
| **タスクカード表示** | カード・行・ノード形式でタスクを表示。担当エージェント・依存関係・完了状況を含める |

### 2-4. Multiagent Collaboration（マルチエージェント協調）パターン

**概念**: 複雑な問題を専門化されたエージェントに分散させ、オーケストレーターが調整する。人間組織の専門分業と同様。

**協調モード:**

| モード | 説明 | 適用場面 |
|-------|------|---------|
| **Sequential Handoff** | 1エージェントが完了後、次へ引き継ぐ（リレー方式） | 各ステップが前の結果に依存する場合 |
| **Parallel Collaboration** | 複数エージェントが同時並行で作業し、出力を統合 | 独立したサブタスクがある場合 |
| **Supervised Collaboration** | マネージャーエージェントがワーカーを監督・方向付け | 複雑な調整と品質管理が必要な場合 |

**UX設計ガイド:**

| 課題 | 設計アプローチ |
|------|-------------|
| **可視性の判断** | シームレスに動作する場合: タスクレベルの進捗のみ表示（内部アーキテクチャを隠す）。問題発生時: どのコンポーネントが失敗したかを明示する |
| **エラー表示** | 「文書調査エージェントが共有ドライブにアクセスできませんでした。手動で文書を提供するか、アクセスを許可しますか？」のように具体的に示す |
| **ステータスダッシュボード** | 各エージェントの状態（作業中・完了・待機中・失敗）と現在のタスクの概要を表示 |
| **競合解決** | エージェント間の矛盾を透明化する（「コスト見積もりエージェントは10時間と算出。履歴データエージェントは類似プロジェクトで15時間が典型的と指摘。どちらを採用しますか？」） |

**CloudWatch調査フロー（参考実装例）:**

マルチステージ・エージェントワークフローの実践例:
- **左ペイン（ケースファイル）**: Acceptされた観察が蓄積される
- **右ペイン（提案）**: エージェントが非同期に返す提案が随時追加される
- 人間の承認（Accept）ごとにエージェントが調査深度を変える（フィードバックループ）
- スーパーバイザーが「観察提案」→「仮説提案」にモードを切り替える（フェーズ遷移の可視化）

### 2-5. ReAct（推論と行動の適応的組み合わせ）パターン

**概念**: 推論と行動を交互に繰り返す適応的アプローチ。「考える→行動する→観察する→考える...」のループ。固定計画ではなく、現実に応じて継続的に適応する。

**ReActループ:**
```
推論: 「現在の天気情報が必要だ」
行動: weather_api(location='Tokyo') を呼び出す
観察: 気温・天候データを受け取る
推論: 「APIが返した結果から完全な回答を生成できる」
行動: 気象データを含む回答を生成する
```

**他パターンとの組み合わせ:**
- ReAct + Reflection: 各推論ステップで自己チェック
- ReAct + Tool Use: アクションフェーズで外部ツールを呼び出し
- ReAct + Multiagent: 特化したサブタスクに他エージェントに委任

**UX設計ガイド:**

| 課題 | 設計アプローチ |
|------|-------------|
| **プロセスの透明性** | エージェントの推論を表示することで信頼を構築する（「Xを試みてYを観察した。次にZを試みる」） |
| **進捗の予測不可能性** | 固定ステップ数は示せない。代わりに反復カウンター（「試み3回目: 代替アプローチ中」）や経過時間を表示 |
| **エラーとエスカレーション** | エージェントが行き詰まりを認識し人間に引き渡せるよう設計する。「ログと診断で調査したが、ハードウェアレベルの検査が必要そうです。発見内容:」のような明確な引き渡しメッセージ |
| **適切な場面** | 探索的タスク・トラブルシューティング・調査など、出発点から完全な経路が見えない作業に特に適している |

## 3. エンジニアリングから体験へ: 処理の可視化

エージェントAIでは、計算が「単一の不透明な瞬間」から「可視化された一連の操作」へと変換される。

### 3-1. 処理ステージ

**分解（Decomposition）:**

ゴールを実行可能な部分に変換する。ユーザーへの表示タイミングが重要: リクエスト後・実行前に表示することで、エージェントの理解を確認し修正の機会を与える。

| 表示形式 | 適用場面 |
|---------|---------|
| 番号付きアウトライン | シンプルな線形タスク |
| シンプルなダイアグラム | ステップに依存関係がある場合 |
| タスクカード（カード・行・ノード） | 複雑な作業、進捗追跡が必要な場合 |
| 非表示（サイレント） | シンプルな単一ステップリクエスト |

**順序付け（Sequencing）:**

何が先行し、何が並列で、どこで自然な一時停止があるかを決定する。ユーザーがエージェントの推論を追い、次のステップを予測できるようにする。

複雑さへの対応:
- 3ステップ: 番号付きリストまたはプログレスバーで十分
- 15ステップ: 要求に応じて展開できる広いフェーズにグループ化する

**委譲（Delegation）:**

エージェントが意図の記述から実行へと移行する瞬間。どの部分が自律的に進行し、どの部分が他のシステムやコラボレーターに依存するかを明示する。複数エージェントが稼働する場合、誰が何に責任を持つかを明確に示す。

### 3-2. 明確化（Clarification）

複雑・主観的・計算コストの高いタスクでは、実行前に短い確認ステップを挟む。

**目的:**
- 無駄な処理を防ぐ
- 共同作者意識を生む（盲目的委任ではなく）
- ユーザーに何がアウトカムに影響するかを示す

**設計のポイント:**
- 2〜3の焦点を絞った質問のみ（「学術的か会話的か」「簡潔か探索的か」「優先するソースは？」）
- Tab操作等で素早く回答できるインタラクション設計
- すべてのタスクに確認を挟まない（シンプルなリクエストは直接実行）

## 4. 進捗通信の階層設計

エージェントが自律的に作業を進める間、ユーザーが方向感覚を保てるようにする。

### 通知（Notifications）

最外層。ユーザーが別の作業に集中していても、重要な変化（完了・一時停止・エラー・承認リクエスト）を届ける。継続的な注意を必要とせずに委任を可能にする。

### 層1: 概要（Overview）

ユーザーが最小限の詳細で方向感覚を保てる高レベルのステータス画面。「作業はまだ進行中か」「どの段階にあるか」「依存関係や権限がブロックしているか」等の簡単な質問に答える。

### 層2: 詳細（Detail）

より詳しく知りたいユーザーのための展開ビュー。完了したステップ・進行中の作業・近いアクションを表示し、完全な内部記録を露出しない。

### 層3: 完全な記録（Full record）

エージェントのアクションの完全な記録。レビュー・デバッグ・学習・コンプライアンスをサポートする。定期的に参照するユーザーは少ないが、その存在が信頼と説明責任に貢献する。

**層の選択基準:**

| タスクタイプ | 推奨アプローチ |
|-----------|--------------|
| 長時間バックグラウンドタスク | 主に通知 + 概要の確認 |
| 高ステークス・不慣れなタスク | 詳細または完全記録の確認 |
| 一般的な定常タスク | 概要のみ |

### ブラウザ / Computer Use 可視化パターン

エージェントがリモートブラウザやデスクトップで作業する際の画面共有型可視化。

**長所:**
- エージェントの能力を具体的に示す（「知ってるアプリを使っている」という即時理解）
- ツール使用の具体化（「顧客データベースにアクセス中」→実際の操作が見える）
- 観察学習によるより現実的なメンタルモデル形成

**限界:**
- 帯域幅要件が高く、ラグが生じる場合がある
- 「今の瞬間」しか見えない（完了済み・残り作業が不明）
- 長時間タスクの継続監視は非実用的

**推奨使用場面:**
- 特定ソフトウェアに閉じた環境でのトラスト構築
- セキュリティ・監査要件がある環境
- 単独ではなく、階層型進捗ビューと**組み合わせて**使用する

## 5. チェックポイント設計

チェックポイントは多段階タスクの途中でエージェントが停止し、進捗を示し、前提を確認し、続行前に承認を求めるタイミング。

### 5-1. チェックポイント（Checkpoints）

| 設計要素 | ガイドライン |
|---------|------------|
| **発生タイミング** | 計画生成後・データセット分析後・元に戻しにくいアクション前 |
| **自動 vs 設計的** | システムの確信度が低下したとき自動生成。または設計者が意図的に配置 |
| **ユーザーへの表示** | 何を待っているか・次に何が起こるか・どんな選択肢があるかを常に明確に |

### 5-2. ロールバック（Rollback）

バージョン管理（GitHub / Google Docs の履歴）と同様、各チェックポイントが保存状態を作成し、後で復元できる。

| 設計要素 | ガイドライン |
|---------|------------|
| **UI パターン** | 「履歴を見る」リンク・保存状態のリスト・バージョン間の差分表示 |
| **復元動作** | 現在の状態を削除せず、新しいブランチまたはスナップショットを作成する |
| **伝達** | 「どこまで戻るか」「何が保持され何が失われるか」を明確に示す |

### 5-3. 中間出力（Intermediate outputs）

チェックポイント間に現れる部分的な結果（グラフ・草案テキスト・データサマリー等）。

| 設計要素 | ガイドライン |
|---------|------------|
| **ラベリング** | 暫定的であることを明示し、どのステップが生成したかにリンクする |
| **編集のハンドリング** | ユーザーが編集した場合、エージェントはそれを次ステージへの新しいインプットとして扱う |
| **レビューパターン** | 一貫した「プレビュー→承認または修正」フローを提供する |

### 5-4. 権限設計（Permissions）

エージェントが新しい機能（カレンダー接続・文書読込・メール送信等）を必要とするとき、チェックポイントで表示されることが多い。

| 設計要素 | ガイドライン |
|---------|------------|
| **説明** | 何にアクセスするか・なぜ必要か・後でどう取り消せるかを説明する |
| **一貫性** | すべての権限リクエストで同じ配置・アイコン・文言を使用する |
| **外部サービス** | OAuth スタイルのウィンドウ等、標準的で透明な認可フロー |
| **拒否後の処理** | 「共有ドライブへのアクセスが拒否されました。権限が付与されるか、スキップされるまでタスクは一時停止します」のように次のステップを案内する |
| **管理UI** | 「Connected apps」や「Authorized access」等の予測可能な場所で権限を確認・管理できるようにする |

### 5-5. 編集とエラー（Edits and errors）

| 設計要素 | ガイドライン |
|---------|------------|
| **編集フロー** | 現在の実行を停止→リクエストを調整→最後のチェックポイントから再開。Pause and Editパターン |
| **エラー表示** | エラーをシステム中断としてではなく、進行中の作業の一部として表示する |
| **エラー後の選択肢** | 原因の説明 + 「再試行」「このステップをスキップ」「入力を修正」の選択肢を提供する |

### 5-6. ソース表示（Sources）

エージェントが使用・参照した外部データ・リンク・ツールの透明化。

| 設計要素 | ガイドライン |
|---------|------------|
| **表示内容** | ソース名・アクセス時刻・確認方法へのリンク |
| **配置** | チャット: 関連ステートメントの下に展開可能なリストまたはインラインリンク。キャンバス: チェックポイントサマリーの末尾に「参照ソース」セクション |
| **目的** | 証拠と推論の関係を実用的かつわかりやすく保つ |

## 6. 3つの新原則

エージェントAIインターフェース設計を統治する3つの核心原則。

### 原則1: 計画を見せる（Reveal the Plan）

**問題**: エージェントはユーザーのリクエストを解釈してアクションを決定するが、ユーザーにはその解釈が見えない。作業が始まったり完了した後に誤解が発覚すると、修正コストが高い。

**解決策**: 実行前（まだ影響を与えられる間）に計画を可視化する。

**実装:**
- リクエスト後・実行前のタイミングで計画を表示する
- 形式より**タイミング**が重要
- 計画への明示的な承認（またはコメントなし）が作業スコープの合意を形成する
- 計画変更が生じた場合も変更を表面化して説明する

**計画可視性のキャリブレーション:**

| タスクの特性 | 推奨 |
|-----------|------|
| シンプル・低ステークス | 計画なしで実行（サイレント計画） |
| 複雑・高ステークス | 実行前に計画を提示・承認を得る |
| 長時間・コスト大 | 計画プレビューを必須とする |
| 「Just do it」モード | 計画を内部で処理し結果のみ提示 |
| 「Show your work」モード | 計画を承認のために提示 |

---

### 原則2: 重要なものを優先する（Prioritize What Matters Most）

**問題**: エージェントAIは通常のAIよりはるかに多くの情報を生成する（計画・進捗・中間結果・ツール出力・ステータスメッセージ・権限リクエスト・最終成果物）。すべてを表示するとユーザーが圧倒される。

**解決策**: 各瞬間にユーザーが実際に必要な情報のみを積極的に優先する。

**重要度の時間的変化:**

| タスクフェーズ | 最も重要な情報 |
|------------|-------------|
| 開始時 | エージェントがリクエストを正しく理解したか |
| 実行中 | 進捗は正常か、何かアテンションが必要か |
| 完了時 | 結果と次のステップ |

**マルチエージェント時の情報圧縮例:**
「研究エージェントが12件の関連文書を発見。分析エージェントが処理中。可視化エージェントがデータ待ち」—膨大な活動を、一瞥で理解できる状態に圧縮する。

**エラーメッセージの優先度:**
即時優先: 何が起きたか・回復可能か・ユーザーが取るべきアクション。技術的な詳細（なぜエラーが発生したか）は1クリック下の階層に配置する。

---

### 原則3: 共有制御を設計する（Design for Shared Control）

**問題**: 完全な自律性は不安を生む。委任が制御の喪失を意味するように感じさせてはならない。

**解決策**: エージェントに独立性を与えつつ、常に人間が介入・リダイレクト・制御を取り戻せるようにする。

**共有制御の要素:**

| 要素 | 設計アプローチ |
|------|-------------|
| **事前の境界設定** | 何をエージェントが自律的に行えるか vs 何に承認が必要かを実行前に明確化する |
| **中断可能性（Interruptibility）** | 進行中の作業を壊したり進捗を失ったりすることなく、いつでも停止できる。一時停止時に「完了済み・進行中・残り」を即座に表示する |
| **ロールバック** | どの以前の状態にも戻れることを可視化する。何が保持され何が失われるかを明確に示す |
| **判断ポイントの表面化** | すべての判断をユーザーに流さない。主観的なトレードオフ・不確実性・結果が永続するアクションについてのみ人間の入力を求める |
| **時間的境界** | いつエージェントが行動できるかの境界設定（即時 vs 営業時間内 vs 前提条件が満たされた後等） |

**共有制御を支えるUIパターン:**
- 明示的なハンドオフ（「分析が完了しました。提案に進む前にレビューしますか？」）
- スコープと期間を説明する権限リクエスト
- エージェントが独立して作業中 vs 入力待ちを示すステータスインジケーター
- 作業の所有権がユーザーに戻るタイミングの明確なマーカー

## 7. クイックリファレンス

### エージェントUX 設計チェックリスト

**パターン設計:**
```
□ Reflectionのレイテンシーをユーザーに事前説明したか
□ Tool Useの権限をティア別に設計したか
□ Planningの計画表示タイミングを判断したか（高ステークスは必須）
□ Multiagentのステータスダッシュボードを設計したか
□ ReActのエスカレーション経路を設計したか
```

**処理可視化:**
```
□ 分解（Decomposition）の表示形式を選択したか
□ 明確化ステップの必要性を判断したか
□ 明確化フォームを2〜3質問以下に抑えたか
```

**進捗通信:**
```
□ 通知・概要・詳細・完全記録の4層を設計したか
□ ブラウザ可視化を使用する場合、階層型ビューと組み合わせたか
```

**チェックポイント:**
```
□ 不可逆アクション前のチェックポイントを配置したか
□ ロールバック機能と履歴表示を設計したか
□ 中間出力への「プレビュー→承認/修正」フローを設計したか
□ 権限リクエストのデザインパターンを統一したか
□ エラー時の「再試行/スキップ/修正」オプションを提供したか
□ ソース表示の配置を決定したか
```

**3原則:**
```
□ 計画をどのタイミングで表示するかを判断したか（タスク複雑度で分岐）
□ 各フェーズで最も重要な情報を定義したか
□ 任意の時点でユーザーが停止・修正・引き継ぎできるようにしたか
```

### パターン選択ガイド

| 状況 | 最適なパターン |
|------|------------|
| 高品質な出力が重要・エラーコストが高い | Reflection |
| 外部システムへのアクションが必要 | Tool Use |
| 複数ステップ・長時間タスク | Planning |
| 異なる専門知識が必要 | Multiagent Collaboration |
| 探索的・経路不明確 | ReAct |
| 複雑な実世界タスク | 複数パターンの組み合わせ |

### エージェント自律度の判断フロー

```
エージェントの自律度レベルを決める
    ↓
アドバイザー（提案のみ）
    → すべてのアクションに確認が必要
    → 低信頼・学習段階・高ステークス領域に適切
    ↓
オペレーター（定義されたスコープで自律実行）
    → ルーティンタスクは自動、判断が必要なものは確認
    → 最も一般的な本番エージェントのモード
    ↓
完全自律（ほぼ人間の介入なし）
    → 確認済みで低リスクな反復タスクのみ
    → 常に非常停止制御を維持する
```
