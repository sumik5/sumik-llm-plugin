## 出力の3サブフェーズ

入力処理・ルーティング・生成を経た後、出力は3つのサブフェーズを経てユーザーに届く。

| フェーズ | 内容 |
|---------|------|
| **1. Internal Generation** | モデルが内部で計算を完了し、出力内容を決定 |
| **2. Post-Processing** | システムが出力をクリーンアップ・フォーマット・強化。言語モデルでは「detokenization（数値列を人間語に戻す）」とガードレールスキャンが実行される |
| **3. Delivery / Presentation** | 出力をユーザー・システム・環境に送信または描画 |

> **核心的な設計転換**: AIが出力を生成し終えた時点で、責任はインターフェースに移る。出力が何であるかよりも、どう届けるかがユーザー体験を決定する。

## 出力5原則

効果的なAI出力が満たすべき5つの特性。

| 原則 | 意味 | ユーザーの問い |
|------|------|-------------|
| **Clear（明瞭）** | 即座に理解可能 | "これは何を言っているのか？" |
| **Verifiable（検証可能）** | 根拠付きで確認できる | "これは本当か？" |
| **Grounded（文脈根拠あり）** | コンテキスト対応の情報源 | "これは私の状況に合っているか？" |
| **Actionable（行動可能）** | タスク指向の次のステップ | "これで何ができるか？" |
| **Adjustable（調整可能）** | ユーザーが編集・精緻化できる | "自分のニーズに合わせられるか？" |

## 明瞭性設計

AIが「それらしい」出力をデフォルト生成する一方、ユーザー中心の構造・トーン・可読性はデザイナーが介入して設計する必要がある。

### Raw から Refined へ

AIモデルは構造化された良い文章が何かを本質的には「知らない」。指示なしには均質なテキストを生成することが多い。

**Before（指示なし）:**
```
To set up an MCP server, first install the Java Development Kit (JDK) and
download the necessary server software. After extracting the files, create a
script file to start the server. Setup varies slightly based on your operating
system. If you run into issues, check the documentation.
```

**After（構造化指示あり）:**
```
Setting Up an MCP Server
Introduction: ...（目的の明示）
Key Concepts: ...（箇条書き）
Steps: 1. Install Java / 2. Download MCP / 3. Extract Files...
Notes: Beginner Tip... / Advanced Tip...
Sources: [hyperlinks]
```

構造化された出力は「スキャンしやすく・行動しやすく・信頼しやすい」。

### フォーマット指定の実践

| 要素 | 設計指針 |
|------|---------|
| **階層** | H2（主要セクション）/ H3（サブセクション）のように役割を明示的に指定 |
| **リスト** | 順序なし情報には箇条書き・手順には番号付きリスト |
| **段落長** | 2〜3文を上限とすることで可読性向上 |
| **太字** | 重要なアクションや用語にのみ使用 |
| **欠損情報** | 内容がない場合は「情報がありません」と明示させる（情報の発明を禁止） |

### トーンと構造のバランス

| 注意点 | 理由 |
|--------|------|
| 過剰なテンプレート化を避ける | 硬直した出力テンプレートはハルシネーション率を上げ、機械的に感じさせる |
| リーンな出力スタイルを保つ | 過多なスタイリング指示は計算コストを増加させる |
| 優先順位のヒューリスティクス | 「網羅性より可読性を優先」のような判断ルールを持たせる |

### 他フォーマットへの適用

テキストと同じ原則が画像・動画にも適用される。

| フォーマット | 設計介入の内容 |
|------------|-------------|
| **テキスト** | スキャナビリティ・明瞭性・構造 |
| **画像** | 焦点・ストーリーテリング・感情的引力 |
| **動画** | ペーシング・トランジション・感情的アーク |

> AIが生成したコンテンツは常に原材料（raw material）である。ユーザビリティは自動的に備わるのではなく、思慮深いデザインの結果として生まれる。

## 検証可能性設計

AI出力は確率的であり、同じ入力でも異なる結果になりうる。ハルシネーション（自信を持った誤情報）は見た目で判断できない。

### AI過信（Overreliance）問題

AI過信（Overreliance）は、ユーザーがAIシステムに過度に依存し、誤った推奨を受け入れ、自身の批判的思考能力を低下させる現象。60以上の研究を分析した研究が、その心理的メカニズムを明らかにしている。

| 問題 | 説明 |
|------|------|
| **スキルの萎縮** | AIへの継続的依存が人間のスキルを弱体化させる（GPS→空間認識力の低下等） |
| **自動化バイアス** | AIの提案を自分の専門知識より優先してしまう傾向。文脈的に不適切でも |
| **確認バイアス** | 自分の意見に同意するAIをより信頼してしまう |
| **順序効果** | 最初のポジティブなAI体験が、その後の過信を生む |

**逆説的な知見**: 詳細な説明を提供することが過信を**増加させる**場合がある。直感に反するが、詳細な理由付けはユーザーに「より信頼できる」という印象を与え、かえって批判的検討を減らす。

**チームパフォーマンスの低下**: 過信が発生すると、「人間+AI」チームが人間単独またはAI単独よりも**パフォーマンスが低くなる**ことがある。これはAIの誤りを人間が訂正できなくなるためで、インターフェース設計が特に重要な理由の一つ。

**設計介入（4パターン）:**

| 介入タイプ | 内容 | 効果 |
|----------|------|------|
| **Cognitive forcing functions** | 自動的な受け入れを中断させる摩擦の意図的な導入（例: 「本当に正しいか確認しましたか？」という確認ステップ） | 反射的な受け入れを防ぐ |
| **透明なケイパビリティ告知** | AIが得意なこと・苦手なことをユーザーが理解できるよう設計（「このAIは法律の専門家ではありません」） | 適切な信頼水準の形成 |
| **Progressive disclosure** | 現実的なメンタルモデルを段階的に構築するUI（最初は簡単な例から始め、徐々に限界を見せる） | 過信の防止 |
| **リアルタイムフィードバック** | 判断品質へのフィードバック（「あなたはAIの提案を何回修正しましたか？」等） | 自己調整の促進 |

> **設計の本質**: 目標は「AIを信頼しないこと」ではなく「**適切な信頼水準**の形成」。ユーザーをパッシブなコンシューマーではなく、効果的なAIコラボレーターにする設計。

### 信頼度スコアの問題

AIの「信頼度パーセンテージ」は見た目の精密さと異なる意味を持つ。

| 比較 | 説明 |
|------|------|
| **スペルチェッカーの93%** | 辞書照合によるカウント。ロジックが透明・再現可能 |
| **LLMの93%** | 数十億パラメータが潜在空間で決定した次トークン確率。ロジックは基本的に検査不能 |

> **設計の指針**: 抽象的な信頼度スコアより「ユーザーが独立してこの情報を検証できるか？」を問うこと。検証できない場合、システムは引用・証拠・ヒューマンハンドオフへのルートを提供するか、権威ある口調で提示することを避ける。

### 最適なAI活用の適用範囲

| 適したタスク | 適さないタスク |
|------------|-------------|
| 主観的タスク（校正・リライト・ブレスト） | 深い専門知識・調査が必要な客観的事実 |
| 素早く検証できる情報 | 医療・法律・財務での高精度判断 |
| ユーザビリティが正確性より重要な場面 | 誤りの影響が大きい意思決定 |

### Human-in-the-Loop（HITL）設計

人間が意思決定プロセスに積極的に関与し続ける設計アプローチ。

| パターン | 内容 | 効果 |
|---------|------|------|
| **ルーティング型** | AIが分類・人間がエッジケースをレビュー | 量を効率的に処理しつつ品質を維持 |
| **ライブラリ型** | 人間が承認済み回答を事前に用意・AIが選択 | 高リスク領域での安全な自動化（例: メンタルヘルス支援） |
| **監査型** | AIが処理・人間が定期的に結果を確認 | 自動化効率と品質保証のバランス |

> **研究知見**: 適切に設計されたHITLシステムは、特にクリエイティビティ・倫理的判断・エッジケース処理において、完全自動化アプローチを上回るパフォーマンスを発揮することが多い。

## グラウンディング設計

出力がどのコンテキストで生成されたかを開示する設計。検証可能性が「情報を確認できるか」であるのに対し、グラウンディングは「なぜそう答えたか」に焦点を当てる。

### グラウンディングの構成要素

| 要素 | 説明 | 設計例 |
|------|------|--------|
| **モデル名・バージョン** | トーン・スコープ・信頼性に影響 | "GPT-4o で生成" の表示 |
| **担当エージェント/ツール** | マルチエージェント環境でどのコンポーネントが生成したか | ソース表示 |
| **地理・ロケール** | 地域設定の誤りが重大な誤りに繋がる（州法の違い等） | 管轄地域のユーザー指定UI |
| **タスク/モードのグラウンディング** | クリエイティブ vs 分析的、ドラフト vs 最終などのモード | モード表示 |
| **セッション・会話コンテキスト** | AIが以前の会話を参照した場合の明示 | "あなたがQ2計画を話していたことを基に..." |

### 「なぜこれを提案したか」アフォーダンス

出力にインライン・ホバー・情報アイコンで「推論の根拠」を提示する設計パターン。

**含まれる情報の例:**
- 参照したデータやファイル
- ユーザー設定・履歴・過去のクエリの影響
- 出力を生成したモデルまたはエージェント

メインの体験を妨げない軽量な説明として機能し、ユーザーが出力を調整・精緻化する起点となる。

### AI開示（Disclosure）の設計

| 規制 | 内容 |
|------|------|
| カリフォルニア州ボット開示法（SB 1001, 2019） | 商業・政治的文脈でのボットは自動化されていることを明示 |
| EU AI Act（2024） | AIと対話していることの透明性を義務化。人間らしすぎるインターフェースは開示を免除されない |
| 米国FTC ガイダンス | AI使用の未開示が欺瞞的広告表現として規制される可能性 |

**設計原則**: AI開示を警告として後付けするのではなく、ユーザーの理解を支援するツールとして早期かつ自然にインタラクションに組み込む。

## アクション可能性設計

出力は最終製品ではなく、ユーザーの次のアクションのための「踏み台」として設計する。

### フォワードアクション設計

出力の周りにある「動詞」——保存・購入・予約・エクスポート・送信・引用・ナビゲート——がユーザーの実際の価値になる。

**設計の問い:**
- ユーザーはここから何をしたいか？
- 即時の質問の先にある下流アクションは何か？
- 取り消し不可能なアクション（購入・送信）はレビューの機会が与えられているか？

| アクションタイプ | 設計指針 |
|---------------|---------|
| **探索的** | 「さらに詳しく」「関連情報を見る」等 |
| **変更前レビュー** | ドラフト保存 vs 即時送信の明示 |
| **外部ドメイン遷移** | 別ドメインへの遷移を明示 + 戻れることの暗示 |
| **既訪問リンク** | 訪問済みリンクの色変更（ウェブのベストプラクティスをAIに適用） |

### Canvas UX

Canvas（インタラクティブな編集可能ワークスペース）はチャットインターフェースや静的出力パネルとは異なる永続的な協創空間。

**Canvas の主要特性:**

| 特性 | 内容 |
|------|------|
| **編集・合成可能** | ユーザーがAI出力を直接修正（段落リライト・チャート調整・コンポーネント追加等） |
| **空間レイアウト** | チャット（線形）と異なり、コンテンツのグルーピング・並び替え・レイヤリングが可能 |
| **永続状態** | コンテンツが会話履歴に埋もれず、常に表示・編集可能 |
| **マルチモーダル統合** | テキスト・画像・コード・データ・UIブロックの混在をサポート |

**Canvasへの移行タイミング**: 共有・保存・インタラクティビティ・発見性が必要になった時点で、出力をCanvasに移行させる。この移行で出力は「機械生成コンテンツ」から「人間主導の協創物」になる。

**バージョン管理の設計:**
- どのイテレーションを作業しているかを明示
- 特定バージョンをお気に入りや保存として固定する機能
- 発行後も会話に戻って繰り返せる設計

## 調整可能性設計

ユーザーはゼロから再プロンプトするより既存の出力を洗練させることを好む。この「調整志向」を支援する設計。

### なぜ調整が再スタートより好まれるか

ユーザーが出力を全削除して再プロンプトするのではなく「調整」を選ぶ理由は、**コンテキストウィンドウ**にある。会話ログや文書履歴（コンテキストウィンドウ）は、モデルが次の応答を生成する際に参照する。このウィンドウにはすでに確立されたトーン・意図・以前の指示が蓄積されており、再スタートするとこれらのシグナルが失われる。

| 調整を選ぶ理由 | 詳細 |
|------------|------|
| **既存の成果を保持** | うまく機能している部分を破棄せずに済む |
| **コンテキスト継続** | トーン・意図・以前の指示が引き継がれる |
| **進捗感の維持** | 再スタートは「進捗を捨てる」感覚を与える |
| **試行錯誤の低コスト** | 小さな変更で結果を検証できる |

### インライン調整パターン

テキスト選択時に軽量な調整UIを提供するパターン（例: Notionのインラインリファインメント）。

| アクション | 動作 |
|----------|------|
| **Elaborate（詳述）** | 選択テキストを展開・詳細化（Prompt augmentationで元プロンプトに"elaborate"を付加） |
| **Simplify（簡略化）** | 選択テキストをより簡潔に |
| **Rewrite（リライト）** | 異なるトーンや文体で再生成 |
| **逐次リファイン** | 複数回クリックで段階的に出力を深化させる |

### Prompt Augmentation（プロンプト拡張）

インライン調整の技術的実装パターン。ユーザーが「Elaborate」をクリックした際、システムは元の指示を**修正版プロンプト**として再構成し、モデルに渡す。

**仕組み:**
1. ユーザーがテキストを選択して「Elaborate」をクリック
2. システムが元プロンプトに "elaborate on this section" / "add more detail" 等を付加
3. 修正プロンプトでドキュメント全体ではなく**選択箇所のみ**を再生成
4. 周囲のコンテキストと整合性を保ちながら展開される

**逐次的深化（Iterative Refinement）:** 「Elaborate」を複数回クリックすることで、毎回直前の出力を基に段階的に内容を深化させる。ユーザーはペースと詳細度をコントロールできる。

**重要な設計原則**: 調整ツールが機能するためには、ユーザーが**そのツールの存在を認識し、使用を許可されていると感じる**必要がある。ツールを発見しやすいUI配置と、「選択して調整できる」という明示的なサインが必要。

### バリエーション提供

同一入力に基づく複数の代替出力を提示することで、ユーザーが最適なものを選べる設計。

**Temperature制御の可視化:**
- 低Temperature → 予測可能・一貫した出力
- 高Temperature → 創造的・多様だが誤りのリスク増
- UIでは「再生成」ボタンのたびに微妙に異なる出力が得られるような実装が一般的

### コンテキスト保持の重要性

セッションベースのツールでは、再起動すると以前に確立したトーン・意図・指示のシグナルが失われる。調整のたびにゼロからプロンプトするコストが高いため、文脈を保持した継続的な洗練を支援する設計が重要。

## マルチターン出力設計

ユーザーが長時間にわたってAIと協働するケース（文書編集・段階的問題解決・半自律プロセス等）では、出力が「会話の一部」ではなく「セッション・スレッド・ワークフロー」として機能する。

### メンタルモデルの転換

マルチターンインターフェースの設計では、「入力→出力」という単発モデルから、**セッション・スレッド・ワークフロー**という長期的な協働モデルへの転換が必要。

| シングルターン思考 | マルチターン思考 |
|----------------|--------------|
| 質問 → 回答 | セッション / 作業スレッド |
| 出力は最終成果物 | 出力はマイルストーン（途中地点） |
| 会話履歴は二次的 | 文脈・記憶が体験の中核 |
| 1回の交換で完結 | 一時停止・再開・分岐・中断が可能 |

出力はただの回答ではなく、長期的な対話の「監査可能・中断可能・方向転換可能な」マイルストーンとして設計する。

### 設計テンション: 柔軟性 vs 継続性

マルチターン設計では、2つの要求が衝突する場面が生じる。

| ユーザーの要求 | 設計上の課題 |
|------------|------------|
| **柔軟性**: アイデアをコミットせずに試したい | どこまで「undo」できるかを明確にする |
| **継続性**: 以前の文脈・意図が引き継がれてほしい | 明示的・暗黙的な記憶手がかりのUI |
| **分岐・改訂**: ペナルティなく方向を変えたい | バージョン管理・ブランチ設計 |

**インタラクションのリズム**: マルチターン出力では、内容と同じくらい「やり取りのリズム」が重要になる。計画→提案→改訂→精緻化というシーケンスが自然に流れるよう設計する。

### マルチターン設計の要素

| 要素 | 内容 |
|------|------|
| **ターンマーカー** | 新しいフェーズや段階を視覚的に区切るセパレーター |
| **変更注釈付きインライン編集** | 前のステップに対してどう変化したかを示す |
| **ローリングサイドパネル / タイムライン** | 進行中タスクの全体像を常に表示 |
| **名前付きスレッド/タスク** | 後で戻れるよう命名・保存できる作業単位 |
| **差分ビジュアライゼーション** | 変更内容を元の状態と並べて表示 |

**設計の鍵**: ユーザーは現在の出力と以前の出力の関係を常に理解できる必要がある。「第2段落を修正する」という指示には、何が第2段落か・どう変わるか・他への影響が明示されなければならない。

> **設計の目標**: ユーザーがIDEやプロジェクト管理ツールを扱っているような感覚を与えず、自然に「会話のスレッド」をたどれるようにすること。

### 変更の権限委譲

| 変更タイプ | 設計原則 |
|----------|---------|
| **1回の変更** | その変更への明示的な許可を求める |
| **複数ターンの変更** | 継続的な許可を与える機能（"今後10ターンは承認不要"等） |
| **重要な変更** | 実行前に必ず確認を求める |

## ウォーターマークと検出

AI生成コンテンツの透明性・出所確認・誤用防止のための技術的・デザイン的アプローチ。

### AIミスユースのスペクトラム

ウォーターマーク技術の必要性を理解するには、AI生成コンテンツの「ミスユース」が連続スペクトラムであることを認識する必要がある。

| スペクトラム | 例 |
|------------|-----|
| **無自覚な規範違反** | 学生がAIで作成したエッセイを開示なしに提出（意図的ではないが倫理的問題あり） |
| **グレーゾーン** | ジャーナリストがAIで調査を補助するが記事で未開示 |
| **意図的な欺瞞** | 政治的・商業的利益のためにAI生成コンテンツを人間製作として提示 |

ウォーターマークシステムは、このスペクトラム全体に対応する必要がある。透明性の社会規範の確立（無自覚なケース向け）と、不正行為の法的追跡（意図的欺瞞向け）の両方を視野に入れて設計する。

### AI生成コンテンツのリスク

| リスク | 内容 |
|--------|------|
| **スケール問題** | 数分で数千件の説得力あるコンテンツを生成可能。事実確認が困難 |
| **学術的誠実性** | 学生・研究者によるAI生成物の未開示提出 |
| **専門的信頼性** | AI生成の法的・財務・医療文書の偽装 |
| **クリエイティブ権** | AI生成コンテンツによる著作権・帰属の問題 |
| **人格欺瞞** | AIペルソナがユーザーを人間と思わせて操作する |

### テキストウォーターマークの仕組み

生成時に統計的な偏りを埋め込み、後から検出可能なパターンを作成する手法。

| 手法 | 内容 |
|------|------|
| **統計的バイアス** | 「quick」と「fast」の選択時に特定の単語を優先するよう誘導し、暗号署名を作成 |
| **構文レベル** | 特定の文構造・句読点パターンへの微妙な傾向 |
| **意味レベル** | アイデアの構成方法に検出可能なパターンを埋め込む |

**主な制限:**
- テキストの長さが短いと検出精度が下がる
- 事実的回答では単語選択の自由度が低くウォーターマーク挿入が困難
- プロバイダーが異なる場合、他のプロバイダーのウォーターマークを検出できない（協調問題）

### SynthIDと生成時ウォーターマーク

後処理による検出（事後的パターン分析）の限界を受けて、**生成時にウォーターマークを埋め込む**アプローチが注目されている。

**テキストSynthIDの仕組み（Google DeepMind）:**
1. 言語モデルが次の単語を選択する際、確率スコアを微調整
2. テキストの意味は変えず、**単語の選択肢間の統計的なパターン**に署名を埋め込む
3. 個々の単語ではなく、「同等に有効な複数の選択肢の間の選び方」に痕跡を残す

**SynthIDの限界:**
- **テキスト長依存**: 短いテキストは統計的証拠が少なく信頼性が低下
- **エントロピー制約**: 事実的回答（日付・名前・数値）では単語選択の自由度が低く、署名を埋め込む余地がない
- **プロバイダー間協調問題**: SynthIDはGeminiのテキストのみを検証可能。他のプロバイダーのコンテンツに対しては無効（鍵と設定が共有されないため）

### 画像・動画ウォーターマーク

| タイプ | 内容 |
|--------|------|
| **明示的ウォーターマーク** | 視覚的に表示（ロゴ等）。無許可使用を明確化 |
| **不可視ウォーターマーク** | 特定ピクセル値を微調整して数学的署名を埋め込む（人間の目には見えない） |
| **メタデータ標準** | C2PA（2021年）等の標準が複数のマニフェストとメタデータを埋め込む |
| **法的義務化** | 中国では、ディープフェイク対策としてAI生成動画への「顕著なマーキング」が法的に義務付けられている |

### AI検出システムの限界

| 問題 | 内容 |
|------|------|
| **低精度** | ある著名なツールはAI生成テキストを26%しか正確に検出できず、人間のテキストを9%誤検出 |
| **非英語話者への偏り** | 非ネイティブ英語話者の文章がAI生成に似ているとして誤フラグ |
| **歴史的テキストへの誤検出** | 独立宣言やシェイクスピアがAI生成と誤判定された事例がある |
| **回避可能性** | 軽微な編集で検出を回避できる |

### 将来展望: 多層的アプローチ

現状のAIウォーターマーク・検出技術は、**初期のスパムフィルタリング**に似ている。制御された環境ではある程度機能するが、実世界の複雑性・敵対的な使用には課題が多い。

スパムフィルタリングが「技術フィルター + ユーザー教育 + 社会規範」の多層アプローチへと発展したように、AI生成コンテンツの識別も同様の複雑さが必要と考えられる。

| アプローチ | 内容 |
|----------|------|
| **技術的手法** | ウォーターマーキング（最も有望な技術的アプローチ） |
| **社会規範の醸成** | AI使用の開示を当然とする文化的規範の確立 |
| **ユーザー教育** | AI能力と限界の理解促進 |
| **プロバイダー間協調** | 業界標準のウォーターマーク仕様（現在は断片化） |

> **現状の評価**: AI検出技術は制御された環境では合理的に機能するが、実世界の複雑性・敵対的な利用には課題が多い。技術的な解決策と並行して、透明性の社会規範の醸成・ユーザー教育・AIシステムへの説明責任構築が必要。

> **AskUserQuestion対象**: ウォーターマーク実装の要否はコンテンツ種別・法的要件・業界標準に大きく依存する。実装を検討する場合は必ずユーザーに要件を確認すること。

## 問題出力の管理

生成AIはどれほど慎重に学習させても、偏った・誤解を招く・有害な出力を生成することがある。デザイナーはユーザーがこれらをどう受け取り・解釈し・回復するかを設計する必要がある。

### 多層的な問題出力低減アプローチ

| 層 | 手法 |
|----|------|
| **学習段階** | 人間フィードバックによる強化学習（RLHF）・Constitutional AI（原則ベースのガイダンス） |
| **評価段階** | 最悪ケースユーザーを模倣してモデルの限界を探るRed Teaming |
| **製品統合段階** | コンテンツフィルター・セーフガード |
| **ドキュメント** | 既知の問題・リスク・用途制限を記したSafety/System Card |

### 拒否応答（Refusal）のUI設計

モデルが応答を断る場合、会話のリズムを保ちながら理解可能に伝える。

| 要素 | 指針 |
|------|------|
| **口調** | 懲罰的・謎めいた表現ではなく、製品のボイスに統合した自然な会話表現 |
| **説明** | 短い説明（"それはお手伝いできません"）+ 穏やかな理由（"有害になる可能性があります"） |
| **継続性** | 拒否でやり取りが終わるのではなく、会話が続けられることを示す |

### 設計者の役割

技術的なモデル訓練と政策レベルの対策に加え、インターフェースデザインがユーザーの認識・解釈・回復に重要な役割を果たす。

| 設計責任 | 内容 |
|---------|------|
| 批判的評価の促進 | 完璧に見える出力でも検証を促すUI摩擦の意図的設計 |
| 不確実性の可視化 | 確信度が低い・検証が必要な場面でのヘッジ言語（"〜と思われます"）の使用 |
| 回復経路 | 問題ある出力からユーザーが建設的に前進できる選択肢の提供 |

## ユーザー確認の原則（AskUserQuestion）

**判断分岐がある場合、推測で進めず必ずAskUserQuestionツールでユーザーに確認する。**

確認すべき場面:
- 出力の主要ユースケース（主観的タスク vs 客観的情報提供）の特定
- HITL設計の適用範囲（どのステップで人間のレビューが必要か）
- Canvas UXを採用するかどうか
- ウォーターマーク・AI開示の要件（法規制・業界標準の確認）
- マルチターン設計における権限委譲の粒度

確認不要な場面:
- AI開示を含める（すべてのAI出力に適用するベストプラクティス）
- エラー・拒否応答に平易な言語を使用する
- フォワードアクションを出力に含める（常に推奨）
