## §1 概要・使用タイミング {#1}

このスキルは **AIインターフェースのUXデザイン** に特化したフレームワークを提供する。
フレーミングからエージェント設計、ユーザーリサーチ、バイアス・倫理まで体系的にカバーする。

### Input-Computation-Outputフレームワーク

AIインターフェースの設計は3段階で捉える:

```
Input（入力設計）
  → ユーザーが意図を伝える3チャネル（暗黙・明示・直接操作）

Computation（処理過程）
  → 処理パイプライン・レイテンシーUX・通知設計

Output（出力設計）
  → 明瞭性・検証可能性・グラウンディング・アクション可能性
```

### AIプロジェクト失敗の5パターン（5 Fails）

AIプロジェクトの大多数が失敗する主な原因:

| # | パターン | 内容 |
|---|---------|------|
| 1 | **間違ったユースケース選択** | 価値がないところにAIを適用 |
| 2 | **ビジョンの欠如** | 戦術的UIしか作らず、長期ビジョンがない |
| 3 | **データ品質の無視** | AI精度を過信し、データのバイアスを見過ごす |
| 4 | **ユーザーリサーチ省略** | 「誰が何を困っているか」を調査しない |
| 5 | **倫理・バイアス無視** | AIの偏りがユーザーに害を与える |

> **重要原則**: 問うべき最初の問いは「AIで何を予測すべきか？」ではなく、**「どこに価値があるか？」** である。

### このスキルを使う場面

フレーミング→§2、入力UI→§3、Copilot/LLMパターン→§4、処理中UX→§5、出力設計→§6、AI検索/予測→§7、エージェント設計→§8、リサーチ→§9、倫理レビュー→§10

## §2 ユーザーとシステムの理解 {#2}

### ユーザーのメンタルモデル

ユーザーはAIに対して以下の誤った前提を持ちやすい:

| 誤った前提 | 実態 |
|-----------|------|
| 人間のような理解をする | 統計的パターンマッチング |
| 出力は常に正確 | ハルシネーション（確信を持った誤り）が起こる |
| 同じ入力→同じ結果 | 確率的な出力。再現性なし |
| 最新情報を持つ | 学習データのカットオフ以降は不明 |
| バイアスがない | 学習データのバイアスを反映する |

設計者の役割は、ユーザーが **正確なメンタルモデル** を形成できるよう透明性を設計すること。

### 組織AI成熟度

> **AskUserQuestion**: フレーミング開始前に確認する。
> 「このプロジェクトのユースケース段階は？（初期探索 / データ検討 / AIモデル選定）」

| 段階 | 特徴 | 設計上の注意 |
|------|------|------------|
| **初期探索** | ユースケース未確定 | 5 Failsチェック → Value Matrixで価値検証 |
| **データ検討** | データ品質・バイアス分析中 | デジタルツインで全体像可視化 |
| **AIモデル選定** | モデル能力の評価中 | 能力を正直に伝える発見UI設計 |

### AIプロジェクトフレーミング

フレーミング手法の詳細は `references/FRAMING-METHODS.md` を参照:

- **AIストーリーボード（5要素）**: Establishing Shot / Things / People / Faces / Transitions
- **デジタルツイン**: 入力データ → AIシステム → 出力の全体図可視化
- **Value Matrix**: Confusion MatrixにビジネスコストをマッピングしROIを最大化

### 能力・発見性・オーケストレーション

AIプロダクトが成功するには3要素が揃う必要がある:

| 要素 | 定義 | 設計上のポイント |
|------|------|---------------|
| **Capability** | モデルができること | 能力の正直な開示。過大評価しない |
| **Discovery** | ユーザーが何を知っていて何を試す気があるか | 発見しやすく・教育的に設計 |
| **Orchestration** | 意図とモデル能力をつなぐ設計層 | スキャフォールド・UX選択で橋渡し |

**発見性の2×2マトリクス**（ユーザー意図の高低 × システム主導性の高低）:
- 高意図 × 低主導 → 明確なアフォーダンスと命名 / 低意図 × 高主導 → タイミング・確信度・終了パスの設計
- 低意図 × 低主導 → 教育的オンボーディング・サンプル / 高意図 × 高主導 → プロアクティブな提案（邪魔にならない設計）

## §3 入力設計 {#3}

### 3チャネルモデル

ユーザーの意図は同時に3つのチャネルで伝達される:

| チャネル | 説明 | 設計のポイント |
|---------|------|--------------|
| **暗黙コンテキスト（Implicit Context）** | ユーザーが意識せず提供する情報（開いている文書・選択テキスト・場所・時間） | 何が使われているかを可視化。誤推論の修正手段を提供 |
| **明示的プロンプト（Explicit Prompting）** | タイプまたは発話するリクエスト | 良いプロンプトを書きやすいUIを設計。ユーザーにプロンプトエンジニアリングを強いない |
| **直接操作（Direct Manipulation）** | スライダー・選択・ジェスチャーでAI動作を整形 | 言語化なしにAI挙動を調整できる手段を提供 |

**設計の核心**: 「より良いプロンプトを書かせること」ではなく、3チャネルが翻訳負荷を分担できるよう設計する。

### CARE Framework

効果的なプロンプト設計の4要素（UIテンプレートや初期入力支援に活用）:

| 要素 | 内容 | 例 |
|------|------|-----|
| **C – Context** | 役割・目的・制約など背景情報 | 「マーケティング担当として、B2B向けに…」 |
| **A – Action** | 実行してほしいこと | 「提案書の要約を書いてください」 |
| **R – Results** | 期待する成果・形式 | 「3点の箇条書き、200字以内」 |
| **E – Examples** | 参考例・スタイル | 「以下のサンプルのようなトーンで」 |

プロダクトで実装する際は、ユーザーが意識せずCAREを満たせるUIを設計する（セットアップ質問・スマートデフォルト・スターターPrompt等）。

### 入力モダリティとスターターPrompt設計

| モダリティ | 特徴 | 留意点 |
|-----------|------|--------|
| **テキスト** | 最も一般的。精度高・コンテキスト豊富 | プロンプト長のバランス（過不足どちらもNG） |
| **画像** | ビジュアル指示・参照イメージ | モデルの視覚能力の限界を開示 |
| **音声** | ハンズフリー・自然なインタラクション | 文字起こしの誤りに対するUXを設計 |

**スターターPrompt設計原則**:
- 空白ページ症候群を防ぐ例示プロンプトを用意
- ユーザーのコンテキストを事前に収集し初期プロンプトに埋め込む
- CAREの要素が自然に含まれるテンプレートを提供

## §4 AIデザインパターン {#4}

### SaaS Copilot配置パターン

> **AskUserQuestion**: Copilotを設計する場合は必ず確認する。
> 「このCopilotは補助的ですか？それとも主要タスクですか？」

| パターン | 説明 | 使用場面 |
|---------|------|---------|
| **Side Panel** | 画面右側にパネル表示 | 既存UIの補助として常時参照したい場合 |
| **Large Overlay** | 画面中央に大きくオーバーレイ | Copilotが主要ワークフローの一部の場合 |
| **Full Page** | Copilot専用ページ | Copilotがプライマリユースケースの場合 |

**Stateful Copilot設計原則**: ユーザーの操作履歴・選択を記憶し、セッションをまたいで文脈を引き継ぐ。

### 7つのLLMデザインパターン

| # | パターン名 | 説明 | 最適な場面 |
|---|-----------|------|-----------|
| 1 | **Restating** | 入力を言い換えて確認 | 曖昧な入力が多い、誤解リスクが高い場面 |
| 2 | **Auto-Complete** | 入力補完・次の単語予測 | 繰り返し入力が多い、コマンド入力画面 |
| 3 | **Talk-Back** | 質問形式でガイド | ユーザーが目標を言語化できていない場面 |
| 4 | **Initial Suggestions** | 空白時の初期提案表示 | 空ページ症候群（Blank Page Problem）の解消 |
| 5 | **Next Steps** | 次のアクション提案 | タスク完了後の次行動を促したい場面 |
| 6 | **Regen Tweaks** | 再生成オプション提供 | AI出力の微調整が必要な場面 |
| 7 | **Guardrails** | 禁止・警告・制約 | 安全制約・コンプライアンス要件がある場面 |

**Guardrails設計原則**: 禁止するだけでなく「代替提案」でユーザーを正しい方向へ導く。

### AI Search UX

| 軸 | 従来検索 | LLM検索 |
|----|---------|---------|
| クエリ形式 | キーワード | 自然言語 |
| 結果 | リスト表示 | 統合された回答 |
| 曖昧さへの対応 | 苦手 | 得意 |

**設計のポイント**: ユーザーの「こんなこと聞いていいのか？」という心理的障壁を下げる設計（例示質問・入力欄ヒント）。

発見パターン（能動的発見 / 文脈的提案 / 空白状態活用 / 段階的露出）の詳細は `references/COPILOT-PATTERNS.md` を参照。

## §5 処理過程の設計 {#5}

> **AskUserQuestion**: レイテンシー設計前にプロセスタイプを確認する。
> 「プロセスタイプは？（トランザクション / 会話 / クリエイティブ / 自律タスク）」

### 処理パイプラインの理解

プロンプト送信後、3ステップで処理が行われる（すべてユーザーには不可視）:

```
Step 1: 入力処理・前処理
  → トークン化・ベクトル埋め込み・コンテキスト組み立て

Step 2: ルーティング
  → 適切なモデル・ツール・RAGへの振り分け

Step 3: 生成・推論（Inference）
  → 確率的なトークン選択で出力生成
```

### レイテンシーUXパターン

| タスクタイプ | 許容レイテンシー | 推奨UXパターン |
|------------|----------------|--------------|
| **リアルタイム操作** | 即時（<100ms） | 即時ビジュアルフィードバック・スケルトンローダー |
| **情報検索** | 中〜低速（数秒） | プログレッシブディスクロージャー・自動補完 |
| **会話・生成** | 中速 | ストリーミングレスポンス・処理中メッセージ |
| **自律タスク** | 低速（分〜） | 通知設計・バックグラウンド実行UI |

**ストリーミング表示**: テキストを逐次表示することで体感速度を向上。長い回答でも最初のトークンが即座に届く体験を提供。

### 3層通知設計

複数ステップのタスクや自律エージェントには3層の通知を設計する:

| 層 | 目的 | 内容 |
|----|------|------|
| **Tier 1: Overview** | 離席中でもトラッキング | 最小限の状態表示（処理中 / 完了 / 要確認）|
| **Tier 2: Detail** | 方向確認・信頼構築 | サブタスク分解・ステップ別説明・依存関係 |
| **Tier 3: Full Record** | 学習・デバッグ・コンプライアンス | 全アクションのログ・理由・ツール使用履歴 |

ユーザーが注意をどの程度向けるかに応じて適切な層を選べるよう設計する。

### プロセスタイプ別設計

| タイプ | 特徴 | UX重点 |
|--------|------|--------|
| **トランザクション** | 明確な入出力・短時間 | 即時フィードバック・確認ダイアログ |
| **会話** | 多ターン・文脈継続 | 履歴の可視化・文脈リセット手段 |
| **クリエイティブ** | 探索的・反復的 | Regen Tweaks・変数スライダー |
| **自律タスク** | 長時間・マルチステップ | Tier1-3通知・中断/再開コントロール |

詳細は `references/COMPUTATION-UX.md` を参照。

## §6 出力設計 {#6}

### 出力5原則

効果的なAI出力は以下の5原則を満たす:

| 原則 | 内容 |
|------|------|
| **Clear（明瞭）** | 即座に理解できる構造・トーン・フォーマット |
| **Verifiable（検証可能）** | 根拠・出典・証拠にアクセスできる |
| **Grounded（根拠付き）** | どのコンテキスト・地域・時点のデータかが明示されている |
| **Actionable（行動可能）** | 次のステップが明確・タスク指向 |
| **Adjustable（調整可能）** | ユーザーが出力を編集・再生成できる |

### 明瞭性設計

Raw出力 → Refined出力: 見出し構造（H2/H3）・箇条書き・太字強調・トーン仕様をシステムプロンプトで明示。
**注意**: 過剰なテンプレート化はハルシネーション率を上げるため柔軟性を残す。

### 検証可能性とAI過信防止

ユーザーはAI出力を過信しやすい（AI overreliance）。設計で防ぐ:

- **ソースリンク**: 主張ごとに参照元を添付（Perplexity AI方式）
- **ヘッジング言語**: 「〜と思われます」「確認を推奨」等の表現
- **Human-in-the-Loop**: 高リスク出力（法律・医療・財務）は必ず人間レビューを経由
- **確信度スコアへの注意**: LLMの信頼度は uninspectable。スコア表示より出典提示を優先

### グラウンディング

出力が「どの文脈に基づいているか」を明示する:

| グラウンディング要素 | 例 |
|------------------|-----|
| モデル名・バージョン | 「GPT-4o で生成」 |
| 地理・ロケール | 「日本の法律に基づく」 |
| 時点・データ範囲 | 「2024年Q2データより」 |
| タスクモード | 「ドラフト作成モード」 |
| セッション文脈 | 「先ほどの Q2 計画に基づいて…」 |

詳細は `references/OUTPUT-DESIGN.md` を参照。

## §7 AI-First情報アーキテクチャ & データUX {#7}

> **AskUserQuestion**: 既存プロダクトのIA改善時に確認する。
> 「現在のユーザーの主な導線はカテゴリブラウジングですか、それとも検索ですか？」

### AI-First IA 4層モデル

```
1. Analysis Overview（分析オーバービュー）
   → AIが全体像・異常・推奨を要約
     ↓
2. Category（カテゴリ）
   → ドリルダウン先の分類
     ↓
3. LLM Search（自然言語検索）
   → 「〇〇を探している」に回答
     ↓
4. Item Detail（詳細）
   → 個別アイテムの詳細ビュー
```

### Forecasting UI設計

| 要素 | 設計上の問い | 実装ガイド |
|------|------------|-----------|
| **トレンドライン** | 線形か非線形か | R²値を表示して信頼度を示す |
| **信頼区間** | 予測のブレ幅をどう表現するか | シェーディングで範囲を表示 |
| **季節性** | 周期パターンがあるか | 前年同期との比較を追加 |
| **シナリオ比較** | 複数の未来を比較させるか | 「悲観/中立/楽観」の3ライン |

### Anomaly Detection UI（異常検知UI）

| タイプ | UI設計のポイント |
|--------|----------------|
| **Point Anomaly** | マーカーで点を強調、クリックで詳細 |
| **Change Point Anomaly** | 区間を色分け、転換点を示す |
| **Contextual Anomaly** | 前月・前年を並べて比較 |
| **Curve Shape Anomaly** | 「正常パターン」テンプレートを重ねる |

**Static vs Dynamic Threshold**:
- Static（固定値）: ルールが明確で変動しない場合
- Dynamic（Bollinger Bands等）: データが季節性・変動性を持つ場合

詳細は `references/AI-FIRST-UX.md` を参照。

## §8 エージェントAI設計 {#8}

> **AskUserQuestion**: エージェント設計前に自律度を確認する。
> 「エージェントの自律度は？（アドバイザー / オペレーター / 完全自律）」

### 5つのAgenticデザインパターン

| # | パターン | 概要 | UX上の課題 |
|---|---------|------|-----------|
| 1 | **Reflection** | 自己評価・エラー修正ループ | レイテンシー増加の透明化 |
| 2 | **Tool Use** | 外部API・DB・ツール呼び出し | どのツールを使ったかの可視化 |
| 3 | **Planning** | 複雑な目標をステップ分解 | 計画の可視化・修正手段の提供 |
| 4 | **Multiagent** | 専門エージェントへの委任・協調 | 誰が何をしているかのトラッキング |
| 5 | **ReAct** | 推論と行動を交互に繰り返す適応型 | 中間状態の提示と軌道修正 |

パターンは組み合わせて使用可能（例: Planning + Tool Use + Reflection）。
詳細は `references/AGENTIC-DESIGN.md` を参照。

### 進捗可視化・チェックポイント・ロールバック

エージェントタスクのUXには以下のコントロールが必須:

| コントロール | 役割 |
|------------|------|
| **Start** | エージェント起動、進捗の可視化開始 |
| **Stop** | 即時停止、ロールバック可否を明示 |
| **Pause/Resume** | 一時停止と再開、長時間タスク向け |
| **Checkpoint** | 中間成果物の確認・承認 |

**進捗表示の原則**: 計画を見せる（show the plan）、重要なものを優先表示（prioritize what matters）、共有制御（shared control）。

### 共有制御の3原則

1. **計画を見せる**: エージェントが何をしようとしているかを事前に開示
2. **重要なものを優先**: 全ステップを見せるより、重要な分岐点のみ提示
3. **共有制御**: ユーザーが介入・修正できるタイミングを設ける

### 非同期UXパターン

長時間タスクで必要な設計: タスク開始直後に「処理中インジケーター」→ バックグラウンド処理中も別作業可能 → 完了通知はプッシュ型（ユーザーがポーリングしない）

## §9 リサーチ手法 {#9}

### MUSE / Disciplined Brainstorming

**MUSE**（Mindful, Unusual, Surprising, Evolving）ブレインストーミングの手順:

1. **Mindful**: 現在の課題を付箋に書き出す（判断なし）
2. **Unusual**: 常識外れのアイデアを意図的に出す
3. **Surprising**: チームが驚く組み合わせを探す
4. **Evolving**: 有望なアイデアを発展させる

**Bookending手法**: アイデアAとBを限界まで展開し、両端を比べて中間点を探す。「文書化してから次へ」が鉄則。

### AI-Inclusive UCD（ユーザー中心設計）

UXの役割は「UI + AI + Data」の三角形全体を設計すること:

- **Spike**: Wizard of Oz方式でAIが動くかを素早く検証（Paper Prototype / Python Prototype）
- **合成ユーザーの限界**: AI生成の架空ユーザーはリアルなユーザーの代替にならない

### RITE（Rapid Iterative Testing）

> **AskUserQuestion**: リサーチ計画を立てる前に確認する。
> 「今回のAI機能は反復的に修正できますか、それとも大規模リリースですか？」

| 軸 | 従来ユーザビリティテスト | RITE |
|----|------------------------|------|
| 修正タイミング | 全テスト完了後 | 各セッション後に即時修正 |
| 参加者数 | 5〜8人（一度に） | 3〜4ラウンド × 3〜4人 |
| 目的 | 問題発見・一覧化 | 発見 → 修正 → 再検証のサイクル |

**AI-Assisted RITE**: セッション録画のLLM自動書き起こし・共通問題抽出・修正案ブレストに活用。

詳細は `references/RESEARCH-AND-ETHICS.md` を参照。

## §10 バイアス・倫理チェックリスト {#10}

### 7カテゴリ倫理リスク分類

AIプロダクト設計前に確認すべき倫理リスク:

| カテゴリ | 主なリスク |
|---------|-----------|
| **バイアス・公平性** | 訓練データの偏り、特定グループへの不均等な影響 |
| **プライバシー** | 個人情報の学習・再現・漏洩 |
| **透明性** | 意思決定根拠の不可視化 |
| **安全性** | 有害コンテンツ生成・悪用リスク |
| **自律性侵害** | ユーザーの判断を代替・操作 |
| **環境負荷** | 大規模モデルのエネルギー消費 |
| **依存性・ウェルビーイング** | 過度な依存・感情的操作 |

### AI Humanifesto（5原則）

| 原則 | チェック項目 |
|------|------------|
| **Control（制御）** | ユーザーはAIのオン/オフを制御できるか？データの使われ方が透明か？ |
| **Trust（信頼）** | AIの判断根拠をユーザーに説明できるか？誤りを認め修正できるか？ |
| **Diversity（多様性）** | 訓練データは多様か？誰が「表現されていないか」を確認したか？ |
| **Safety（安全）** | プライバシー保護は十分か？依存性・感情的害のリスクは評価したか？ |
| **Balance（バランス）** | AIは人を置き換えるか、補完するか？環境コストは最小化されているか？ |

### バイアス認識チェックリスト

- [ ] 生成結果に「誰が欠けているか」を確認した（サバイバーバイアス視点）
- [ ] 職業・役割のステレオタイプが含まれていないか確認した
- [ ] クエリに多様性を明示することで結果が改善されるか試した
- [ ] Value MatrixでFP/FNの人的コストを評価した
- [ ] 別のAIモデルをジャッジとして使い、バイアスを相互チェックした

詳細は `references/RESEARCH-AND-ETHICS.md` を参照。
