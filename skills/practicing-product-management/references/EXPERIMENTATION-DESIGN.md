# 実験・A/Bテスト設計ガイド

実験フレームワークの構築から統計的意思決定まで、プロダクトマネージャーが現場で使える実践知識をまとめる。

---

## 1. 実験フレームワークの構築

### 背景：なぜ構造化実験が必要か

場当たり的な機能リリースは矛盾するフィードバックを生む。あるユーザーは改善を称賛し、別のユーザーはリグレッションを報告する。直感だけでは意思決定の根拠にならない。

**解決策**: 仮説駆動の実験フレームワークを導入し、アドホックなトライアルを構造化されたテストに置き換える。

### 実験の対象範囲

| 対象 | 変更の種類 |
|------|-----------|
| オンボーディングメール | 文言・CTA・件名 |
| アプリ内ガイダンス | ツールチップのタイミング・位置 |
| インタラクティブチュートリアル | ステップ構成・視覚要素 |
| ボタン・UI要素 | 色・配置・アイコン |

---

## 2. 仮説設計フレームワーク

### 仮説テンプレート

```
When we apply [変更X] in [コンテキストY],
then [メトリクスZ] will move by [目標Δ],
because [ユーザーフィードバック or データからの洞察].
```

**実例**:
```
When the onboarding email includes a one-click Git Setup Wizard link,
then the click-through rate will climb by 25%,
because users have indicated difficulty finding setup instructions.
```

### 仮説ドキュメントの必須要素

- **変更内容**: 何を変えるか（具体的に）
- **対象コンテキスト**: どのユーザーセグメント・画面・フロー
- **主要メトリクス**: 変動を観測する指標（1つを優先）
- **期待効果量 (Δ)**: 最小検出可能効果の推定値
- **根拠**: ユーザーインタビュー・行動データ・先行研究

> **重要**: 仮説はコードを書く前に文書化する。プロダクト・エンジニアリング・アナリティクスチームの三者が同じ理解を持つことが前提条件。

## 3. A/Bテスト設計手法

### テストバリアントの構成

実験ごとに3〜5バージョンを設定する:

| バージョン | 内容 |
|-----------|------|
| Control | 既存エクスペリエンス（変更なし） |
| Variant A | 文言・CTA変更 |
| Variant B | 視覚要素変更（ボタン色・配置） |
| Variant C | 機能変更（ツールチップタイミング等） |
| Variant D | 複合変更（A+Bの組み合わせ効果確認） |

Feature flagにより、フルデプロイ不要でバージョンのON/OFFが可能。ターゲティングルールで「新規サインアップのみ」「特定コホート」等の対象絞り込みを行う。

### 隔週テストリズム（Biweekly Cadence）

学習速度と統計的厳密性を両立するため、2週間サイクルで実験を回す:

| フェーズ | 期間 | 内容 |
|---------|------|------|
| Planning | 1日目 | 仮説確定・バリアント設計・サンプルサイズ計算 |
| Execution | 2〜7日目 | 実験実施・データ収集 |
| Mid-check | 4日目 | 中間評価（Early Stopping判定） |
| Analysis | 8〜12日目 | 統計分析・結果解釈 |
| Decision | 13〜14日目 | 採用/棄却/再試験の意思決定 |

**制約**: 1チームあたり同時実験数は最大2件。重複による影響とユーザー疲弊を防ぐ。

## 4. サンプルサイズ計算

### 必要な4パラメータ

| パラメータ | 説明 | 典型値 |
|-----------|------|--------|
| p₀（Baseline Rate） | 計測対象メトリクスの現在値（例: 12% CVR） | 実績データから取得 |
| Δ（Desired Lift） | 検出したい最小効果量（相対値） | 10〜20% |
| 1-β（Statistical Power） | Type II誤り（見逃し）を最小化する確率 | 80% |
| α（Significance Level） | 許容するType I誤り（誤検知）率 | 5%（α=0.05） |

### Python計算例

```python
from statsmodels.stats.power import NormalIndPower

analysis = NormalIndPower()
required_n = analysis.solve_power(
    effect_size=0.012 / 0.12,  # Δ/p₀ = 相対効果量
    power=0.8,
    alpha=0.05,
    ratio=1
)
print(int(required_n))  # ~10,200 per variant
```

p₀=0.12、Δ=10%（相対）の場合、各バリアントに約10,200ユーザーが必要。

### テスト期間の決め方

- **最低2ビジネスサイクル**（2週間）を確保
- 平日・週末の行動パターン差を吸収するため
- 週間アクティブユーザー数からアームあたりの充足期間を逆算

## 5. データ収集と計測設計

### トラッキング実装パターン

**メールテスト**:
- 各リンクに一意のUTMパラメータを付与
- 開封・クリック・フォローアップアクションをユーザーIDと紐付け

**アプリ内テスト**:
- Feature flagを参照して表示バリアントを決定
- ユーザーの「表示・クリック・閉じる」をセッション詳細（ロール・経過時間）とともにログ記録

**チュートリアルフロー**:
- 全ステップで「表示・クリック・完了」を記録
- バージョンタグを各イベントに付与

### データ品質の確保

- 全テスト結果を**単一のアナリティクスウェアハウス**に集約
- 実験間のコホート重複を防ぐため、一意のランダム化キーを実験ごとに割り当て
- 集中ダッシュボードで全アクティブ実験のステータスとサンプルカバレッジを追跡

## 6. 統計的意思決定ルール

### 二重検定アプローチ

| 手法 | 判定内容 |
|------|---------|
| **頻度論（Frequentist）** | p値と信頼区間でバリアント間差の統計的有意性を確認 |
| **ベイズ（Bayesian）** | バリアントがコントロールを意味ある差で上回る確率を推定 |

### 採用/棄却/保留の基準

| 判定 | 条件 |
|------|------|
| **採用（Adopt）** | 頻度論で95%信頼度 **または** ベイズで90%事後確率 |
| **棄却（Reject）** | 頻度論で95%信頼度でコントロール未満 |
| **保留（Inconclusive）** | 上記以外 → 再設計または廃止 |

> p値 < 0.05かつ観測リフト ≥ 目標Δ → 採用。p値 ≥ 0.05 → 証拠不足で次の実験へ。

### Early Stopping ルール

4日目（計画参加者の約半数が経験済み）に中間評価を実施:

- **暫定p値 < 0.005** かつ最小検出可能効果以上のリフト → 実験を即時終了し勝者バリアントをプロモート
- コントロールが同基準で有利 → 実験を中断してリソースを節約
- 判断不能 → 2週間サイクルの終了まで継続

この逐次設計（Group Sequential Design）により、科学的厳密性を損なわずに素早い意思決定が可能になる。

## 7. 仮説と信頼区間

### 仮説文書化のベストプラクティス

実験開始前に全前提条件をプレイブックに明記する:

1. **測定対象の定義**: 何をカウントするか（例: 1セッション = 1クエリ群）
2. **サンプリング前提**: 各サンプルは独立していること（属性: タイムスタンプ・地域・デバイス）
3. **中心極限定理の適用条件**: サンプルサイズが十分大きければ推定分布は正規分布に近似 → CIの計算が可能

### 95%信頼区間の実務的解釈

> 「同じ方法を同じ条件で繰り返した場合、20回中19回は真の値がこの区間内に入る」

信頼区間は「推定の揺れ幅」を定量化する。大きなサンプルほど揺れが小さく（狭い区間）、小さなサンプルほど揺れが大きい（広い区間）。

**計算手順（比率推定の場合）**:

```
1. サンプル比率: p̂ = 観測数 / サンプルサイズ
2. 標準誤差: SE = √[p̂(1-p̂) / n]
3. 95% CI: p̂ ± 1.96 × SE
4. 全体推定値へのスケール: 推定数 = p̂ / サンプリングレート
5. 不確実性バンド: 推定数 ± 1.96 × (SE × N)
```

### シナリオベースの感度範囲（Sensitivity Analysis）

標準的なCIに加え、前提条件の変化が推定に与える影響を「what-if」シナリオで評価する:

| シナリオ | 影響 |
|---------|------|
| サンプリングレート低下（1% → 0.8%） | CI幅が拡大（例: ±1M → ±1.24M） |
| バースト性イベント（バイラルニュース等） | データ分布の偏りで不確実性増大 |
| インジェストレート誤差（±1%） | 中央推定値がシフト、CIも連動して変化 |

エグゼクティブにはベスト・ワーストケースの範囲を提示し、異常需要時の意思決定を支援する。

## 8. サンプリング戦略と外挿法

### 均一サンプリングの問題点

全ログを均一にサンプリングすると、高トラフィック地域が推定を支配し、小規模セグメントの信号が埋もれる。

### 層化ランダムサンプリング

3軸でログを分割し、各層内でランダムサンプリングを実施:

| 層 | 区分 |
|----|------|
| **地理** | Americas / EMEA / APAC / LATAM |
| **時間帯** | 00-06 / 06-12 / 12-18 / 18-24 UTC |
| **デバイス** | Web / iOS / Android / Voice |

```python
# サンプリングの疑似コード（各層内で実施）
if rand() < 0.01:  # 1%サンプリングレート
    emit_to_sample_stream(log_entry)
```

各エントリに0〜1の一様乱数タグを付与し、タグ < サンプリング率のもののみをサンプルストリームへ書き込む。層ごとの母集団重みに比例してサンプルサイズを維持することで代表性を確保する。

### データクレンジング：サンプリング前フィルタ

サンプリング前にノイズを除去して精度を確保する:

| フィルタ | 対象 |
|---------|------|
| ボット・クローラー除去 | user_agentブロックリストマッチ + データセンターIPレンジ判定 |
| 異常セッション除去 | 統計的外れ値（クエリ数/秒が極端に高いセッション） |
| 重複排除 | 同一user_id + タイムスタンプの重複エントリ |

### 外挿（Extrapolation）の手順

```
推定総量 = サンプルカウント / サンプリングレート
例: 24.8M サンプル / 0.01 = 2.48B 推定総量
```

この規則はスライス（地域・時間帯・デバイス）ごとに独立して適用し、スライス推定値を合算することで全体推定値を得る。

## 9. 統計的信頼の可視化と伝達

### 非技術者向けの言葉の置き換え

| 統計用語 | 言い換え |
|---------|---------|
| Margin of Error（MoE） | 「不確実性バンド」（uncertainty band） |
| P-value | 「リスクスコア」（risk score） |
| Power Analysis | 「信頼ゲージ」（confidence gauge） |

### 意思決定への統計の接続

| 受け手 | 使い方 |
|-------|-------|
| **エンジニアリング** | 不確実性バンドの上限でサーバーリソースを確保（過少プロビジョニング防止） |
| **マーケティング** | リスクスコアで施策継続/異常調査を判断 |
| **プロダクト** | サンプルサイズ要件をロードマップに組み込み、実験の信頼目標を事前確保 |

### エグゼクティブレポートの構成

1日次ブリーフィングに含める要素:
- **推定中央値 ± 不確実性バンド**（例: 2.48B ± 98K）
- **地域別内訳**（主要マーケットのシェアと変動）
- **リスクスコア**（有意な変化の有無）
- **感度プロット**（前提変化時のシナリオ範囲）

数式の詳細は付録に格納し、ブリーフィングの流れを阻害しない。
