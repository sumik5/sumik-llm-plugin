# 根本原因分析・問題解決リファレンス

定量データと定性データを組み合わせてドロップオフの根本原因を特定し、
フィッシュボーンダイアグラムを活用してクロスファンクショナルチームで
構造的に原因を分解・優先順位付けする手法。

---

## 1. ドロップオフファネル分析

### ファネルステージの設計

ユーザージャーニーを明確なマイルストーンに分解する。例：開発者オンボーディング：

| ステージ | イベント名 | 意味 |
|---------|-----------|------|
| Account | Email Verified | 登録完了・メール認証 |
| Dashboard | Dashboard Viewed | 初回ログイン・UI探索 |
| Discovery | Repo Viewed | コンテンツ探索 |
| Clone | Repo Cloned | ローカル環境セットアップ |
| First Local | Local Commit | 初回変更保存 |
| First PR | PR Created | メインブランチへの初回貢献 |

**設計原則:** 各ステージは「次の行動への転換点」を表す。段階間の離脱率が最も高い箇所が優先介入ポイントになる。

### インストルメンテーション設計

フロントエンド（UIコンポーネント）とバックエンド（APIエンドポイント）の両端にイベントを計装し、ジャーニー全体を把握する。

```
イベントスキーマの必須フィールド:
- userId（ユーザー識別子）
- event（イベント名: 'Email Verified', 'Repo Cloned'など）
- timestamp（発生時刻）
- contextual metadata（デバイス種別、セッションIDなど）
```

補完として **ヒートマップ** を活用：クリック・ホバー・スクロール行動を視覚化し、主要CTA（「メール認証」「Clone in IDE」「PR作成」ボタン等）のエンゲージメントを把握する。

### コンバージョン率の計算クエリ（SQL例）

```sql
WITH onboarding AS (
  SELECT userId,
    MAX(CASE WHEN event='Email Verified'  THEN 1 ELSE 0 END) AS activated,
    MAX(CASE WHEN event='Dashboard Viewed' THEN 1 ELSE 0 END) AS explored,
    MAX(CASE WHEN event='Repo Viewed'      THEN 1 ELSE 0 END) AS discovered,
    MAX(CASE WHEN event='Repo Cloned'      THEN 1 ELSE 0 END) AS cloned,
    MAX(CASE WHEN event='Local Commit'     THEN 1 ELSE 0 END) AS committed,
    MAX(CASE WHEN event='PR Created'       THEN 1 ELSE 0 END) AS pr_created
  FROM events
  WHERE timestamp >= CURRENT_DATE - INTERVAL '30 days'
  GROUP BY userId
)
SELECT
  ROUND(100.0 * SUM(explored)  / SUM(activated),  2) AS activation_to_explore,
  ROUND(100.0 * SUM(discovered)/ SUM(explored),   2) AS explore_to_discover,
  ROUND(100.0 * SUM(cloned)    / SUM(discovered),  2) AS discover_to_clone,
  ROUND(100.0 * SUM(committed) / SUM(cloned),      2) AS clone_to_commit,
  ROUND(100.0 * SUM(pr_created)/ SUM(committed),   2) AS commit_to_pr
FROM onboarding;
```

**ウィンドウ:** ローリング7日・30日の両方を並べて傾向変化を検出する。

### 定性診断との統合

定量データだけでは「どこで」は分かっても「なぜ」は分からない。補完的な定性調査を組み合わせる：

| 手法 | 規模 | 目的 |
|------|------|------|
| 1-on-1ユーザーインタビュー | 30〜50セッション | 離脱理由の深堀り |
| セッション録画レビュー | ファネル上位ステージ重点 | UIの混乱箇所の特定 |
| インアプリサーベイ | 離脱直前のトリガーポイント | 即時フラストレーション把握 |

**典型的な発見パターン:**
- UIラベルとチュートリアルの命名規則のミスマッチ
- CLIエラー時の代替導線の欠如
- SSH設定手順の不明瞭さ

## 2. データ収集とイベントインストルメンテーション

### イベントソースの特定

複数システムにまたがる問題を分析するには、すべての関連イベントソースを特定して計装する：

- **コアシステムイベント**: 作成・確認・キャンセル・返金処理メッセージ
- **フロントエンドSDK**: ユーザー操作（クリック・ダイアログ操作・フロー遷移）
- **通知サービス**: メール配信・開封イベント
- **サポートプラットフォーム**: チケット作成・チャット・通話ログ
- **外部パートナーAPI**: スケジュール変更・ステータス更新
- **決済システム**: 試行成功・失敗・タイムアウト

### イベントスキーマレジストリ

各イベントタイプについて以下を記録・管理する：

```
レジストリ構成要素:
- JSONスキーマ定義
- 所有チーム情報
- 発火頻度
- 用途（分析/アラート/監査）
```

検索可能なインターフェースで公開することで、パイプライン開発時の重複計装を防ぎ、クロスチームでの一貫性を保つ。

### タギング規則

複数システム間でイベント名とメタデータを統一する。異なるシステムが同じ概念を異なる名前で表現していると分析時に結合が困難になる。

**統一タギングの利点:**
- ダッシュボードでのリアルタイム集計が容易
- アラートルールの一元管理
- 根本原因分析時の横断的なクエリが可能

## 3. フィッシュボーンダイアグラム実践

### 概要

フィッシュボーンダイアグラム（Ishikawa図・特性要因図）は、問題の根本原因をクロスファンクショナルに構造化するための視覚化ツール。「なぜこの問題が起きているか」を複数の観点から体系的に探る。

**4つの主幹ブランチ（PPTE）:**

| ブランチ | 対象領域 |
|---------|---------|
| People（人） | 対応速度、スキルギャップ、人的ミス |
| Process（プロセス） | フロー設計、ポリシー、ハンドオフ |
| Technology（技術） | SDK不具合、API障害、バグ |
| Environment（環境） | 外部要因、規制変更、天候・スケジュール変動 |

### ワークショップ設計（2時間）

**参加者:** データエンジニア、PM、UXリサーチャー、パートナー担当、サポートリード

**アジェンダ:**

```
00〜10分  Introduction
  - 問題データの概況（指標・スパイク・UIフリクションポイント）を共有
  - ファシリテーターがブレームレス・探求的マインドセットを設定

10〜30分  Category Brainstorm（サイレント付箋出し）
  - テーブルごとに担当ブランチを割り当て
  - 各自が潜在原因を付箋に無声で記入
  - 例: People→エージェント応答時間、Process→返金フロー、
        Technology→モバイルSDKバグ、Environment→外部天候アラート

30〜40分  Dot-Voting（ドット投票）
  - 参加者がクリティカルと判断する原因にドットシールを貼る
  - ブランチごとに上位5件を次のフェーズへ

40〜80分  Drill-Down RCA（詳細分析）
  - 上位原因ごとにサブブランチを展開
  - 例: Technology → Mobile SDK Bugs
          → Retry Logic Failure
          → Missing Backoff
          → Schema Mismatch
  - UXチームが未処理例外による画面遷移問題を追跡

80〜90分  Cross-Branch Dependencies（横断依存の可視化）
  - ブランチをまたぐ因果チェーンをマッピング
  - 例: Environment（API障害）→ People（サポートスパイク）→ Process（ポリシー文言の不備）

90〜110分 Action Mapping（対応策ペアリング）
  - 各根本原因に対策候補を紐付け
  - 例: 「返金ポリシー不明瞭」 → 文言改善 + A/Bテスト配置
  - 「API障害」 → フォールバックサービス設計
  - 「UX例外ループ」 → モバイルSDKパッチ

110〜120分 所有権の割り当て
  - エンジニアリング: SDKエラー対応
  - プロダクト: ポリシー文言改善
  - オペレーション: APIフォールバック構築
  - サポート: SLAコミットメント見直し
```

### ワークショップ後のデジタル化

1. Miro等のツールで図をデジタル化し、内部wikiに共有
2. 各根本原因を **Jiraエピック** にリンク（依存関係とSprint Zeroの見積もりを含む）
3. 成果物を共有タイムラインとして整理：
   - **Quick Wins**: ポリシー文言更新、UX例外パッチ（〜2週間）
   - **Medium-term**: API冗長化、SDKバージョン監査（〜1〜2ヶ月）

## 4. 仮説検証とデータドリブン意思決定

### 根本原因から仮説へ

フィッシュボーンで特定した原因をもとに、検証可能な仮説を設定する：

```
仮説テンプレート:
「[根本原因] が原因で [問題] が発生している。
[対策] を実施することで [指標] が [数値] 改善すると予測する。
[期間] ・[サンプルサイズ] で検証する。」
```

**例:**
- 「返金ポリシーの文言が不明瞭なため、確認ダイアログでのキャンセルが増加している。簡潔な文言への改善で、48時間パイロット（5,000ユーザー）においてキャンセルリンクのクリック率を10%削減できると予測する。」

### 成功基準の事前定義

| Spike | 成功条件 |
|-------|---------|
| コピー明確性A/Bテスト | 48時間でキャンセルリンククリック -10% |
| SDKエラーハンドリングパッチ | ステージング環境で5日間連続クラッシュ0件 |
| フォールバックAPIプロトタイプ | 99%のリクエストで500ms以下の応答 |

**スプリント毎のデイリースタンドアップ確認:** 「このスパイクは仮説を確認したか、それとも反証したか？」

## クイックリファレンス: RCAの選択フロー

```
問題発生
    ↓
定量データでファネルを可視化
    ↓
最大ドロップオフポイントを特定
    ↓
定性調査で「なぜ」を補完（インタビュー/録画/サーベイ）
    ↓
フィッシュボーンWSで根本原因を構造化（PPTE）
    ↓
ドット投票で上位原因を絞り込み
    ↓
所有権を割り当て + Jiraエピック化
    ↓
Spikeタスクで仮説検証（2〜3日 / 明確な成功基準）
    ↓
実験結果を踏まえた本格実装判断
```
