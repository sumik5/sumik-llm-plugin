## 1. プロダクトアナリティクス基礎

### データへの没入

優秀なPMは「データに浸かって暮らす」ことを習慣とする。ダッシュボードを眺めるだけでなく、数値を掘り下げてピボット処理・細分化し、定性的に調査する価値のある手がかりを引き出す。

**What → Why → Howの変換プロセス:**

| ステップ | 内容 | 担当 |
|---------|------|------|
| What（なに）| 数値の異変を特定 | データ分析 |
| Why（なぜ）| 異変の原因を探る | 定性調査（UX強み） |
| How（どのように）| 実行可能な改善策を立案 | PM主導 |

> UXバックグラウンドを持つPMは、定量データの「What」から定性調査でしか発見できない「Why」へと会話を進める独自のポジションにある。

### インスツルメンテーションの原則

**設計時から計測を組み込む鉄則:**
- 機能仕様書にインスツルメンテーションを標準項目として追加する
- 達成目標と紐づけた成功指標を事前に定義する
- エンジニアと協力してローンチ時に計測基盤を実装する
- 命名規則と分類法を統一してシステム全体で一貫性を保つ

**注意:** 何でもかんでも追跡しようとしない。主要なユーザーとシステムイベントに絞る。

### ファネル分析

ファネルは「最初のステップを開始した人が最後のステップまで減少していく」形を可視化する手法。

**分析アプローチ:**
1. **離脱率の異常値を特定** — 他のステップと比べて大幅に離脱が多い箇所を探す
2. **時系列モニタリング** — 同じステップの週次・月次変化でパフォーマンス改善を追跡
3. **ファネルの短略化** — 本当に必要なステップか検討し、不要ステップを削除する
4. **Moment of Truth** — 価値判断が行われる瞬間（例: 決済入力）は必ず大幅な離脱が発生する

**経験則:** プロセスに1ステップ追加するごとに約10%が離脱すると想定する。

## 2. AARRR海賊指標

成長を段階的に促進する5段階フレームワーク。長期的なファネルとして計測できる。

| 指標 | 意味 | 最適化のポイント |
|------|------|----------------|
| **Awareness** | 認知 — プロダクトの存在を知ること | プロダクトマーケティング・チャネル分析 |
| **Acquisition** | 獲得 — ユーザー基盤に「獲得」すること | サインアップフロー最適化 |
| **Activation** | 活性化 — 意味ある体験をしてアクティブになること | Aha!モーメント設計 |
| **Retention** | 継続 — 繰り返し戻ってくること | コホート分析・リテンション改善 |
| **Referral** | 紹介 — 既存ユーザーが新規ユーザーを呼ぶこと | バイラル係数（K）の向上 |
| **Revenue** | 収益化 — 持続可能な収益を得ること | 収益モデルの最適化 |

### Activation（活性化）の詳細

**アクティブユーザーの定義パターン:**

| 定義 | 説明 | 用途 |
|------|------|------|
| 訪問ベース | サイト/アプリを訪れた全員 | 広義のアクティブユーザー数 |
| イベントベース | 特定のイベントをトリガーしたユーザー | より正確な計測 |
| エンゲージドユーザー | 厳選されたイベントリストをトリガーしたユーザー | エンゲージメント品質の把握 |

**主要メトリクス:**
- `DAU` — 日間アクティブユーザー数
- `WAU` — 週間アクティブユーザー数
- `MAU` — 月間アクティブユーザー数
- `DAU/MAU` — 月次スティッキネス（一般的に40%以上で良好、50%以上で優良）

### Referral（紹介）の計算式

**バイラル係数（K）:**

```
K = i × c%
```

- `i` = 各顧客が送った招待数
- `c%` = 招待が顧客獲得につながった割合
- `K > 1` の場合、有機的な成長が起こる
- `K = 1` が損益分岐点
- `K < 1` の場合、追加獲得なしにはユーザー基盤が縮小

## 3. リテンション分析とコホート分析

### リテンション計測

**基本的なリテンション計測方法:**

```
リピーター率 = (全ユーザー - 新規ユーザー) / 全ユーザー
```

リピーター率が増加していれば良い傾向。ただし新規獲得が停滞しているサインにもなりうる。

**コホート分析のポイント:**
- 特定期間に初訪問したユーザーグループ（コホート）の行動を長期追跡する
- 0週目が100%（基準週）で、時間の経過と共に減少するロングテールパターンを描く
- 異なるコホート間のリテンション率を比較して改善効果を検証する

**リテンション計測の2方式:**

| 方式 | 定義 | 特徴 |
|------|------|------|
| 厳密計算 | 特定の日に戻ってきた人 | 精度高 |
| 大まかな計算 | 特定日までのいずれかの日に戻ってきた人 | 大局的な把握向き |

### コホートの分類軸

コホートは時間軸だけでなく、以下の軸でも分類可能:
- 左利き/右利き（デバイス使用特性）
- 年齢層
- 設定言語
- 登録チャネル
- 地域

異なる属性のコホート間でリテンションを比較することで、どの顧客セグメントに投資すべきかの洞察が得られる。

## 4. メトリクスの罠

### ダークメトリクス（Dark Metrics）

**定義:** 人の不幸・欺瞞・操作によって増強されたデータを最適化すること。

| ダークパターン例 | 問題 |
|----------------|------|
| 購読キャンセルを意図的に難しくする | 解約率を下げるが顧客満足は損なう |
| 許可なくアドレス帳をスクレイピングする | 短期的な獲得増だが信頼を破壊する |
| 自動継続課金の通知を見づらくする | 収益は上がるが顧客は怒る |

> **原則:** データを盾に人を操作したり傷つけたりしてはいけない。

### プロキシメトリクス（Proxy Metrics）の罠

**定義:** 本来の目標を代替するために選んだ指標が、根本的な状況を改善しないまま数値だけが変わること。

**対策:**
- 指標を選んだ理由を常に文書化する
- 1つの指標だけでなく、複数の指標で検証する
- 定量メトリクスと定性調査（ユーザーインタビュー）の両方で実態を確認する

**判断分岐（AskUserQuestion）:**

> この指標はビジネス目標の本質を正しく反映しているか？
> 数値が改善したとき、実際のユーザー体験も改善しているか確認できているか？

## 5. 仮説駆動開発と実験

### ビルド / フィックス / チューニングの分類

| 活動 | 定義 | 実験の役割 |
|------|------|-----------|
| **ビルド** | まだ存在しない機能の構築 | 何を誰向けに作るか・ペインポイントの特定 |
| **フィックス** | バグ・欠陥の修正 | どのバグを優先するかの仮説検証 |
| **チューニング** | リリース済み機能の最適化 | 成果改善のための実験（ほぼ実験そのもの） |

### 仮説の立て方

**良い仮説の条件:**
1. 検証可能であること
2. 具体的で明確であること
3. 成功・失敗を判断できる基準があること

**仮説と実験の優先順位付け指標:**
- 実験に使用可能なリーチ数（トラフィック量）
- 成功した場合の潜在的な影響
- 必要なエンジニアリング工数
- 仮説の妥当性に対する信頼度

### ABテストの運用ルール

**必須チェックリスト:**
- [ ] 終了日をテスト開始前に決定する（チェリーピッキング防止）
- [ ] 各バケットに最低2,000人のサンプルを確保する（統計的有意性）
- [ ] AAテストで計測システムの信頼性を検証する
- [ ] 局所的最大値への最適化に気をつける（より大きな改善機会を見逃さない）
- [ ] 「What」だけでなく「Why」を理解するために定性調査を併用する

**ABテスト以外の実験手法:**

| 手法 | 説明 | 適用場面 |
|------|------|---------|
| コンシェルジュ | 人力でサービスを提供し、価値を検証 | 自動化の前に需要を確認したい場合 |
| オズの魔法使い | 完成版UIを見せつつ、バックエンドは人力 | MVP検証・スタートアップ初期 |
| フェイクドア | 架空の機能を置いてクリック率で需要を測る | 開発前の需要テスト |
| 段階的ロールアウト | 10% → 20% → 50% → 全体と徐々に展開 | 大幅変更のリスク軽減 |
| ホールドオーバー | 旧版を一部ユーザーに維持して長期比較 | 長期パフォーマンスの検証 |
| ベータプログラム | コアユーザーグループに先行提供 | リリース前の品質確認 |

## 6. 収益モデルと損益

### 収益ライフサイクル

プロダクトの収益状況はライフサイクルに応じて変化する:

```
研究/開発期 → ローンチ期 → 損益分岐点 → 黒字化 → 成熟/停滞期 → 廃止
     ↓              ↓            ↓            ↓           ↓
  コストのみ   収益開始     収支均衡     利益創出   最適化維持   撤退計画
```

### 収益モデルの種類

**AskUserQuestion（収益モデル選択）:**

> どの収益モデルが自社プロダクトに最適か？以下の軸で判断する:
> - 顧客のニーズ・使用目的・制約は何か？
> - BtoB / BtoCのどちらか？
> - アーリーステージか成熟フェーズか？
> - 潜在顧客の支払い意欲はどの程度か？

| 収益モデル | 説明 | 適合シナリオ |
|-----------|------|------------|
| ベータ無料 → 正規課金 | ベータ期間後にフル料金 | 新機能・新サービスの立ち上げ |
| 無料トライアル → サブスク | 期間終了後に定期課金 | SaaS・中長期利用サービス |
| フリーミアム | 基本無料、機能追加で課金 | 大規模ユーザー基盤獲得後の収益化 |
| アプリ内課金 | 無料提供 + コンテンツ課金 | ゲーム・メディア |
| シートライセンス | ユーザー数で課金 | BtoB SaaS |
| 使用量課金 | 使用量に比例した課金 | API・クラウドサービス |

### P&L（損益）の理解

**プロダクトPMが把握すべき財務の基本:**
- **収益（Revenue）**: サービスから得る売上
- **費用（Cost）**: サービス提供にかかるコスト（開発費・インフラ費・人件費）
- **損益（P&L）**: 収益 - 費用

**損益分岐点について:**
損益分岐点の達成は重要なマイルストーンだが、必ずしも唯一の成功指標ではない。市場シェア獲得を優先するロスリーダー戦略を採用する場合、利益より先に規模拡大を目指すことも戦略的選択として有効。

**判断分岐（AskUserQuestion）:**

> 現在のフェーズでの主要な成功指標は何か？
> - 損益分岐点の達成を優先するか？
> - 市場シェア獲得を優先するか（ロスリーダー戦略）？
> - ユーザー成長数を潜在収益の代替指標として扱うか？

## 7. ロードマップ管理

### ロードマップの本質

**ロードマップ ≠ ローンチ計画**

| ロードマップ | ローンチ計画（ガントチャート） |
|------------|----------------------|
| 今後の方向性と優先事項を示す | 具体的な機能とリリース日程を確約する |
| アウトカム（成果）中心 | 機能・タスク中心 |
| 曖昧さを許容する | 確定した内容のみ含む |
| 定期的に変化する | リリースサイクルごとに確定 |

### Now-Next-Later フレームワーク

```
|---- Now ----|------ Next ------|------------- Later -------------|
|  現在の四半期  |  今後6ヶ月       |  その後9ヶ月（合計約18ヶ月先まで） |
```

| 時間枠 | 内容 | 精度 |
|--------|------|------|
| **Now（今）** | 現在チームが実際に取り組んでいること | 高（確定事項） |
| **Next（次）** | Nowが終わったら次に取り組むこと | 中（方向性は明確） |
| **Later（その後）** | 中長期的に実現したいアイデア | 低（変更可能性大） |

**ロードマップの中身はアウトカムで書く（機能リストにしない）:**

| ❌ 機能ベース（悪い例）| ✅ アウトカムベース（良い例）|
|---------------------|------------------------|
| 「TikTok共有ボタンを追加」 | 「コラボレーション方法を増やすことでリテンションを向上させる」 |
| 「プロフィールページを改修」 | 「ユーザー満足度を50%向上させる」 |
| 「検索機能を改善」 | 「パフォーマンスを改善して直帰率を下げる」 |

### テーマによる整理

ロードマップのアウトカムをテーマ別に整理することで整合性が高まる。テーマの例:
- ユーザー成長
- リテンション向上
- 収益最大化
- パフォーマンス改善
- コラボレーション機能

### ロードマップの維持管理

**更新頻度:**
- **月次（またはスプリントごと）:** ロードマップと現在の作業量を照合する
- **四半期ごと:** 全ステークホルダーとロードマップを見直し、Now/Next/Laterを再配置する

**ロードマップへのインプット4要素（優先順位付けの根拠）:**
1. プロダクトビジョン
2. ユーザーフィードバック
3. データからの洞察
4. チームメンバーのアイデア

**判断分岐（AskUserQuestion）:**

> 機能をロードマップのどの時間枠に配置するか迷う場合、以下を確認する:
> - ビジネス価値: 組織の目標にどう貢献するか？
> - 技術リスク: 現時点での実現可能性は？
> - 顧客インパクト: どれだけ多くのユーザーに影響するか？

## 8. 優先順位付けフレームワーク

### 影響度 vs. 工数マトリクス

最も基本的な優先順位付けの手法。

```
          低工数           高工数
高影響 |  ★即実行          見直し・段階実施  |
低影響 |  時間があれば      ✕排除（脳力の無駄） |
```

**4象限の対処法:**
1. **高影響 × 低工数** → 最優先で即実行
2. **高影響 × 高工数** → 長期投資として計画的に実行
3. **低影響 × 低工数** → メンテナンス時に対応
4. **低影響 × 高工数** → 排除して他の優先事項に集中する

### RICEフレームワーク

影響度 vs. 工数に2軸を加えたより精緻なフレームワーク:

| 要素 | 説明 |
|------|------|
| **Reach（リーチ）** | アイデアが潜在顧客に与える影響範囲 |
| **Impact（影響度）** | 成功した場合の定量的な改善効果 |
| **Confidence（確度）** | 見積もりへの信頼度（%） |
| **Effort（工数）** | 実現に必要なエンジニアリング等の労力 |

```
RICEスコア = (Reach × Impact × Confidence) / Effort
```

### アイゼンハワーマトリクス

緊急性と重要性を区別するフレームワーク:

| | 緊急 | 非緊急 |
|-|------|--------|
| **重要** | 即座に実行 | スケジュールを立てて実行 |
| **非重要** | 他に委任 | 排除（忘れる） |

**重要な洞察:** 重要だが緊急でない事柄こそ、先送りにされやすく最も計画が必要。

> **フレームワークの本質:** フレームワークは完璧な決断を下すためのツールではなく、決断に至った経緯をステークホルダーに透明性を持って伝えるためのコミュニケーションツールでもある。

## 9. 「ノー」の伝え方

### 「ノー」を伝えるべき相手

顧客・事業パートナー・営業部・マーケティング部・エンジニアリング部・UXデザイナー・CEO など、あらゆるステークホルダーへの「ノー」が求められる。

### 「ノー」を伝える7ステップ

1. **感謝** — アドバイスをくれたことに感謝する
2. **質問** — どんな問題を解決しようとしているのか、期待するアウトカムは何か、他の解決策は検討したかを確認する
3. **探索の提案** — 問題領域をより深く調査することを提案する
4. **現在の優先事項を確認** — Nowにある項目が会社戦略にどう関係しているか説明する
5. **トレードオフを明示** — 新たな優先事項を追加するなら、何を後回しにするかを尋ねる
6. **データで主張** — 優先順位を裏付けるデータを示す
7. **成功指標の設定** — 圧力に負けて追加した場合でも、成功指標と影響測定を確約させる

> **重要:** 「ノー」と言う前に必ず「なぜ」を問う。幹部が新しいアイデアを持ってくる背景には必ず理由がある。その理由を理解することが先決。

## 10. PM-UX協働関係

### 基本的な役割分担

| 役割 | 担当 |
|------|------|
| プロダクトマネージャー | **What** — 何を作るか |
| UXデザイナー | **How** — どう作るか |

ただしこの分担はスタートラインであり、重複する「グレーゾーン」が必ず存在する。重要なのは、特定のタスクの「最終決定権」を誰が持つかを明確に合意すること。

### チームタイプの判定テーブル

| タイプ | 特徴 | 対処法 |
|--------|------|--------|
| **良いチーム** | 役割明確・協働機会豊富・決定権の合意あり | 文化を維持しながらスケールする |
| **悪いチーム** | 善意あるが行き違い多い（多数派） | ワーキングアグリーメントを結ぶ |
| **不愉快なチーム** | 有害な関係・指示と服従のみ | 戦う場を選び、指標を定義し、脱出計画も準備する |

### ワーキングアグリーメントの結び方

1. 互いの不満について率直に話し合う機会を設ける
2. 「何がどうなってほしかったか」「何に懸念を抱いているか」を全員から聞く
3. 共有目標・関心事を合意する
4. 基本ルールを決める（完璧でなくていい。大まかな合意で十分）
5. レトロスペクティブで定期的に見直し、イテレーションする

### PM-UXの境界線と組織図

**組織的な立ち位置の判断分岐（AskUserQuestion）:**

> UXはプロダクトに報告するのか、同列か？この問いに唯一の答えはない。チームの状況・組織文化・会社がデザインをどう位置づけているかによって判断する。

**UX出身PMへの注意:** PMになったら「チームのデザイナー」ではなくなった。デザイン提案は意見として提示できるが、決定権はUX担当者にある。

**ハイブリッドロール（PM兼UX）のリスク:**
- コンテキスト切り替えのオーバーヘッドが大きい
- 自分自身と反対意見を戦わせるのが難しい
- チームが成長したら早めに専門家を分離する

## 11. プロダクトリーダーシップ

### プロダクトヘッドの主要な責務

1. **プロダクト全体の一貫性の管理** — プラットフォームをまたいで一貫したUXを確保する情報アーキテクトの役割
2. **ステークホルダーとのコミュニケーション** — 過剰なほど意図的に行う
3. **顧客の声を代弁するパートナーとの連携** — カスタマーサポート・カスタマーサクセス・コミュニティマネージャー
4. **プロダクトチームの構築と育成** — スキルヒストグラムで強化領域を特定し採用する

### エグゼクティブとのコミュニケーション

**経営幹部に響く話し方の原則:**
- ビジネス言語を話す（利益・コスト・市場シェア・人材確保）
- 長い前置きや言い訳を省く
- 相手が知りたいことに答える（自分が答えたい質問ではなく）
- アクティブリスニングを超えた「ディープリスニング」を実践する
- スルーライン（一貫したテーマ）を明確にする

### チームをプロダクトとして売り込む

組織内の他チームに対しても、顧客に対するのと同じマインドセットを適用できる:
- 「顧客」= 組織内の隣接チーム・リーダー
- 「市場」= 社内の需要
- 「プロダクト」= チームが提供するサービス・ツール・ナレッジ

> 成功と失敗の分かれ目は、往々にして技術スタックの選択ではなく、チームのパフォーマンスと対人力学にある。

## ユーザー確認の原則（AskUserQuestion）

**このスキルを使う際、判断分岐がある場合は推測で進めず必ずAskUserQuestionでユーザーに確認する。**

確認すべき場面:
- 収益モデルの選択（サブスクリプション/フリーミアム/トランザクション等）
- ロードマップの時間枠への機能配置（Now/Next/Laterの判断）
- 損益分岐点vs市場シェア獲得の優先順位
- PM-UXの線引き（チーム状況によって異なる）
- プロキシメトリクスが本質的な目標を反映しているかの確認

確認不要な場面:
- ダークメトリクスを避けるべき原則（明確なベストプラクティス）
- インスツルメンテーションを実装時から組み込む原則
- ABテスト終了日を事前に決める原則
