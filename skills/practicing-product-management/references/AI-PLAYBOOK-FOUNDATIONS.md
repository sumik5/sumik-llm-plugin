# AI/MLリテラシー基礎：PMが持つべきAI知識

一般PMがAI製品開発を主導するために必要な基礎知識をまとめたリファレンス。技術者になる必要はないが、「AIの言語を話せる」ことが目標。

---

## AI vs ML の区別：PMにとっての意味

**AI（人工知能）**はゴール：機械が人間のように知的に振る舞うことを目指す広い概念。
**ML（機械学習）**は手段：データから学習することでAIを実現する特定の技術。
**DL（深層学習）**はMLのサブセット：多層ニューラルネットワークを使った特殊なML。

```
AI（人工知能）
 └── ML（機械学習）
      └── DL（深層学習）
```

### PMへの実践的含意

| 問い | AI（ルールベース） | ML（学習ベース） |
|------|-------------------|-----------------|
| 開発速度 | 速い（ルール定義で実装） | 遅い（データ収集・学習が必要） |
| データ要件 | 少なくて良い | 大量の高品質データが必要 |
| 適応性 | 静的（手動更新が必要） | 動的（データで自動改善） |
| 予測の性質 | 決定論的（同じ入力→同じ出力） | 確率論的（不確実性あり） |

**PM判断基準**：FAQチャットボットはルールベースで十分。パーソナライズ推薦はML必須。まずシンプルな手法で解決できないか問い直す。

## PM向けAI/ML用語集

| 用語 | 一言説明 | PM的意味 |
|------|---------|---------|
| **Algorithm** | 問題を解く手順のセット | レシピ。どのアルゴリズムを選ぶかは精度・解釈性・コストのトレードオフ |
| **Model** | データから学習した予測器 | 「AI脳」。精度だけでなく誤りのコストで評価する |
| **Training Data** | モデルを学習させるデータ | 質と量の両方が重要。「Garbage In, Garbage Out」 |
| **Features** | モデルへの入力となる特徴量 | 何をインプットするかがモデル精度の鍵。PMがドメイン知識を提供できる |
| **Labels** | 正解データ | 教師あり学習に必要。ラベリングコストを計画に入れる |
| **Inference** | 学習済みモデルで予測すること | 本番運用フェーズ。レイテンシ・コストを考慮 |
| **Neural Network** | 人間の脳を模した計算モデル | 複雑なパターン認識に強い。解釈性が低い傾向あり |
| **NLP** | 自然言語処理 | テキスト・音声を扱う技術全般 |
| **Generative AI** | 新しいコンテンツを生成するAI | テキスト・画像・コード生成。幻覚リスクに注意 |
| **Prompt Engineering** | LLMへの指示を最適化する技術 | プロダクト品質に直接影響。反復テストが必要 |

## 4大誤解（Myths vs Reality）

| 誤解 | 現実 | PMの対応 |
|------|------|---------|
| 「AIは魔法のブラックボックス」 | データ品質・アルゴリズム・問題定義に依存する応用数学 | 問題定義とデータ戦略に積極的に関与する |
| 「完璧なデータが揃うまで始められない」 | 小さな高品質データセットから始め反復改善できる | 「AI MVP」で早期スタート。データは継続収集 |
| 「AIがあればユーザーリサーチは不要」 | AIは「何を」しているかを示すが「なぜ」かは示さない | 定性リサーチとAI分析を組み合わせる |
| 「AIフィーチャーは一度作れば完了」 | モデルはデータのドリフトで性能劣化する（Model Drift） | ロードマップに監視・再学習コストを組み込む |

## MLの学習プロセス：Training / Validation / Testing

データセットを3分割してモデルを開発する基本プロセス。

```
全データセット
├── Training Set（〜70%）：モデルがパターンを学習
├── Validation Set（〜15%）：ハイパーパラメータ調整・過学習防止
└── Testing Set（〜15%）：最終的な実世界性能を測る「本番テスト」
```

### PMへの含意

- **データ計画**：ラベリング・収集コストを早期に見積もる
- **性能評価**：訓練精度が高くても、テスト精度が低ければNG（過学習）
- **反復を前提に**：バリデーションは「モデルの練習テスト」。複数回のチューニングを計画に含める

## Confusion Matrix と主要メトリクス

分類モデルの性能を4象限で表す。PMが意思決定に使うための基本ツール。

```
                  予測: Positive  予測: Negative
実際: Positive    TP（正しく陽性）  FN（見逃し）
実際: Negative    FP（誤検知）     TN（正しく陰性）
```

### 主要メトリクスとPMへの含意

| メトリクス | 計算式 | いつ優先するか |
|-----------|--------|--------------|
| **Accuracy** | (TP+TN)/(全体) | クラスバランスが均等な場合の概況把握 |
| **Precision** | TP/(TP+FP) | 誤検知のコストが高い場合（例：重要メールをスパム扱い） |
| **Recall** | TP/(TP+FN) | 見逃しのコストが高い場合（例：疾患検出・不正検知） |
| **F1-Score** | 2×(Precision×Recall)/(P+R) | Precision/Recallの両立が必要な場合 |

**PMの問い**：「FP（誤検知）と FN（見逃し）のどちらのコストが高いか？」これが最適メトリクスを決める。

## 過学習 / 未学習：PMの役割

| 状態 | 症状 | 原因 | PMの対応 |
|------|------|------|---------|
| **過学習（Overfitting）** | 訓練精度は高いが、本番で失敗 | モデルが複雑すぎる・ノイズを暗記 | 「テストデータでの性能は？」と必ず確認。多様なデータ追加を推進 |
| **未学習（Underfitting）** | 訓練・テスト両方で精度が低い | モデルが単純すぎる | より複雑なモデルの検討をデータサイエンスチームと議論 |
| **理想（Generalization）** | 未知データでも安定した予測 | 適切なモデル複雑度とデータ量 | これを目標設定する |

**PMの重要な問い**：「このモデルは本番ユーザーが経験するような新しいシナリオで、どれだけうまく機能するか？」

## Human-in-the-Loop（HITL）

AIの出力に人間の判断を介在させる設計パターン。自動化の限界を補い、信頼性と精度を高める。

### HITL が有効なシナリオ

| シナリオ | AIの役割 | 人間の役割 | PMへの恩恵 |
|---------|---------|---------|----------|
| コンテンツモデレーション | 問題のある投稿をフラグ | 境界ケースを最終判断 | FPリスク低減、コミュニティ安全 |
| 医療診断支援 | 異常箇所をハイライト | 医師が最終診断 | 高精度と人間監視の両立 |
| カスタマーサポート | 定型問い合わせに自動回答 | 複雑・感情的な案件をエスカレーション | 満足度向上、コスト効率 |
| 採用スクリーニング | 候補者をスコアリング | 採用担当者が最終判断 | 時間短縮、バイアス軽減 |

### HITL 実装のPM判断基準

- **介入ポイントの特定**：リスクが高いか、モデルの確信度が低い箇所
- **ワークフロー設計**：人間が効率よくレビューできるUIを設計
- **成功指標**：介入率・解決時間・ユーザー満足度を定義

## ML 4種類の概要

| 種類 | 仕組み | 代表的なPM活用場面 | 主な注意点 |
|------|-------|-----------------|----------|
| **教師あり学習** | ラベル付きデータで学習 | 顧客チャーン予測、不正検知、レコメンド | ラベリングコスト、データバイアス |
| **教師なし学習** | ラベルなしデータからパターン発見 | 顧客セグメンテーション、異常検知 | 「正解」がなく、解釈が主観的 |
| **強化学習** | 報酬・ペナルティで行動最適化 | 広告配置最適化、通知送信タイミング | 報酬関数の設計が難しい、計算コスト大 |
| **Generative AI** | パターン学習から新コンテンツ生成 | コード生成、コンテンツ作成、要約 | 幻覚リスク、バイアス、著作権問題 |

## LLMの制約とリスク：PM向けチェックリスト

| リスク | 内容 | PM対応策 |
|--------|------|---------|
| **幻覚（Hallucination）** | 流暢だが事実と異なる出力を生成 | 医療・法律・金融での事実確認機能を必須化。免責事項を表示 |
| **バイアス・有害コンテンツ** | 訓練データのバイアスを反映・増幅 | 出力モニタリング、フィルタリング、定期評価を実施 |
| **実世界理解の欠如** | 物理的世界・感情・常識的判断が苦手 | 言語操作で解決できるタスクに限定して活用 |
| **コンテキスト忘却** | 長い会話・文書での一貫性維持が困難 | 長期記憶が必要なユースケースには別途設計が必要 |
| **ブラックボックス性** | なぜその出力をしたか説明が困難 | 高リスク判断には説明可能性要件を設ける |
| **プロンプト感受性** | 微小な表現変化で出力が大きく変わる | プロンプトを反復テストし、バージョン管理する |
| **過依存・スキル低下** | 人間の判断力が退化するリスク | AIは補完ツールと位置づけ、批判的評価を促す |

## AIライフサイクル8段階：PMの役割

| 段階 | 内容 | PMの主な役割 |
|------|------|-------------|
| **1. 問題定義** | ユーザー課題とビジネスゴールを明確化 | 🔴 **最重要**：「なぜAIか？」を問い続ける。測定可能な目標を設定 |
| **2. データ収集・探索** | データソースを特定し理解する | データの偏り・欠損を把握。倫理的調達を確認 |
| **3. データ前処理** | 欠損処理・正規化・形式統一 | データ品質とプライバシーへの影響を監視 |
| **4. 特徴量エンジニアリング** | 予測に有効な特徴量を設計・生成 | ドメイン知識を提供。重要特徴量から製品意思決定の示唆を得る |
| **5. モデル選択・学習** | アルゴリズム選択とデータで学習 | 精度・解釈性・コストのトレードオフを議論。成功指標を再確認 |
| **6. モデル評価・チューニング** | テストデータで性能検証・改善 | メトリクスの意味を理解し優先課題を判断。反復を前提に計画 |
| **7. デプロイ・モニタリング** | 本番統合とデータドリフト監視 | デプロイ要件を定義。モニタリングダッシュボードとアラートを設計 |
| **8. 再学習・メンテナンス** | 変化するデータへの継続対応 | 再学習スケジュールを計画。ロードマップにメンテコストを組み込む |

> **PMの根本原則**：AIのローンチはゴールではなくライフサイクルの「始まり」。製品ロードマップにモデルの継続運用コストを必ず計上する。

## AIプロダクトPMの5つの指針

1. **問題から始める、技術からではなく**：「AIを使おう」ではなく「最も価値ある課題は何か？AIはその最善策か？」
2. **データは製品の原材料**：コードではなくデータが核心。データ戦略をPMが主導する
3. **確率論的に設計する**：AIは「スパムの確率98%」と言う。モデルが外れた時のUXを設計する
4. **Day 1から倫理・信頼を設計する**：バイアス・公平性・プライバシーは後付けではなく要件
5. **ライフサイクルで考える**：ローンチ後も監視・改善・再学習が続く。これを計画に含める
