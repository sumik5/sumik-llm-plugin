## RADフレームワーク（グロースPMの核心）

**RAD = Retention（保持）・Acquisition（獲得）・Development（顧客開発）**

グロースPMの役割はこの3領域を戦略的に組み合わせて持続可能な成長エンジンを構築すること。

### Retention（保持）— 最優先領域
```
なぜ最優先か:
  - 新規獲得コストは既存維持コストの5-7倍
  - リテンション率1%改善 ≈ 収益7%改善（LTV効果）
  - 高いリテンションは口コミ・バイラリティの基盤

主要メトリクス:
  - 7日/30日/90日リテンション率
  - Churn Rate（解約率）
  - 再訪問率（Return Rate）
  - フィーチャー採用率（Feature Adoption Rate）
```

### Acquisition（獲得）
```
PMのフォーカスポイント:
  - チャネル別CAC（顧客獲得コスト）の最適化
  - プロダクト主導のAcquisition（PLG）の設計
  - アクティベーション率の改善（サインアップ→Aha Moment）

主要メトリクス:
  - MAU/DAU成長率
  - サインアップ→アクティベーション率
  - チャネル別コンバージョン率
  - Organic vs Paid 比率
```

### Development（顧客開発）
```
拡張収益の発掘:
  - ユーザーニーズの継続的発見（Customer Development）
  - アップセル・クロスセル機会の設計
  - ネットワーク効果の設計

主要メトリクス:
  - NRR（Net Revenue Retention）
  - ARPU/ARPAU/ARPMAU
  - Expansion MRR
  - Feature Stickiness（機能定着率）
```

## PLG（Product-Led Growth）4モデル詳細

### Model 1: Product Engagement Model
**メカニズム**: 高い利用頻度が習慣化 → 離脱困難な状態（Product Stickiness）を作る

**設計のポイント**:
- 「Aha Moment」を早期に体験させる（Time to Value最小化）
- ループ設計：使えば使うほど価値が増す仕組み
- 通知・リマインダーによるエンゲージメント維持

**代表例**: Slack（毎日使うことでチームのコミュニケーション文脈が蓄積）

**KPI**: DAU/MAU比（30%以上が健全）、Session Length、Feature Usage

### Model 2: Viral Loop Model
**メカニズム**: 既存ユーザーが他ユーザーを招待することで自己増殖する

**設計のポイント**:
- 招待インセンティブの設計（双方に価値）
- コラボレーション機能そのものが招待をトリガーする
- Viral Coefficient（K-factor）> 1 を目指す

**代表例**: Dropbox（ストレージ容量プレゼントで友人招待を促進）

**KPI**: Viral Coefficient（K-factor）、招待→登録コンバージョン率

### Model 3: Expansion Model
**メカニズム**: ユーザー数・利用量が増えるほど価値が増加 → 自然な拡大

**設計のポイント**:
- フリーミアム → プレミアム の自然なアップグレード設計
- 使用量ベースの課金（成功とともに課金増加）
- チーム内伝播（個人→チーム→部門→全社）

**代表例**: GitHub（個人→チームリポジトリ → オルガニゼーション契約）

**KPI**: Expansion MRR、PQL（Product Qualified Lead）数、Seat Growth

### Model 4: Platform Ecosystem Model
**メカニズム**: 外部開発者・パートナーがエコシステムを構築 → プラットフォーム価値向上

**設計のポイント**:
- API・SDK・マーケットプレイスの整備
- 開発者向けDX（Developer Experience）の最適化
- パートナーエコノミクスの設計（収益分配モデル）

**代表例**: Salesforce（AppExchangeエコシステム）

**KPI**: API呼び出し数・成長率、サードパーティアプリ数、エコシステム収益貢献率

| モデル | 最適な条件 | 注意点 |
|--------|-----------|--------|
| **Freemium** | 製品で価値証明できる・CAC高い | Free→Paid転換率1-5%が目安 |

## 実験設計の8ステップ

```
Step 1: 仮説を立てる
  形式: 「[変更]を行うことで、[指標]が[方向]する（なぜなら[理由]）」
  例: 「オンボーディングフローを3ステップ→2ステップに短縮すると、
       7日リテンション率が向上する（なぜなら初期摩擦が減るから）」

Step 2: 成功指標（OEC）を定義する
  - Primary Metric（主指標）: 仮説で変動が期待される指標
  - Guardrail Metrics（ガードレール指標）: 悪化させてはいけない指標
  - 例: Primary=7日リテンション、Guardrail=Signups数・NPS

Step 3: 実験設計
  - サンプルサイズ計算（統計的有意性のために必要なユーザー数）
  - テスト期間の設定（最低1週間・完全なビジネスサイクルを含む）
  - セグメント分割方法（Random・ユーザーID hash等）

Step 4: バイアス排除
  - ランダム化（Randomization）
  - ブラインド評価の設計
  - Novelty Effect対策（新奇性バイアス）

Step 5: 実験実行
  - フィーチャーフラグの設定
  - ログ・計測の確認
  - 途中経過モニタリング

Step 6: 統計分析
  - 統計的有意性（p-value < 0.05 が一般的基準）
  - 実際の効果量（Effect Size）の評価
  - 信頼区間の確認

Step 7: ビジネスインパクト評価
  - 統計的有意 ≠ ビジネス的有意
  - 実際の収益・コストインパクトを試算
  - Long-term effect vs Short-term effectの考慮

Step 8: 意思決定と文書化
  - ロールアウト・停止・継続の判断
  - 学習の記録（実験ログ）
  - 次の実験への知見還元
```

| **ARPAU** | Average Revenue Per Active User | アクティブユーザーの収益化効率 |
| **ARPMAU** | Average Revenue Per Monthly Active User | 月次アクティブユーザーの収益化 |

### 拡張収益コア戦略

```
1. 価格モデル選択:
   - 座席/シート課金（ユーザー数で従量）

2. パッケージセグメンテーション:
   - ユーザーグループ別バンドル設計

3. Add-on モジュール:
   - コアプロダクトを補完する個別機能
   - 統合・コネクター（API連携）

4. Upsell タイミング:
   - 使用量上限到達時
   - 機能利用の阻害ポイント
   - ビジネス成長のマイルストーン到達時

5. Cross-sell:
   - 関連プロダクトへの誘導
   - 課題の広がりに応じたソリューション拡張
```

## North Star Metric 設計ガイド

North Starは「プロダクトがユーザーに届ける中核的価値」を最もよく代理する単一指標。

### 良いNorth Star Metricの条件

```
□ ビジネス価値と相関している（長期収益予測につながる）
□ ユーザーが価値を感じていることを反映している
□ 先行指標である（結果指標ではなく行動指標）
□ チームがアクションを取れる（コントロール可能）
□ 長期的に安定している
```

### 業界別North Star Metric例

| 業界 | North Star Metric |
|------|------------------|
| SNS | Weekly Active Users |
| SaaS (コラボレーション) | Seats × 週次アクティブ率 |
| EC | 月次購買ユーザー数 |
| コンテンツ | 1ユーザーあたり視聴分数 |
| マーケットプレイス | GMV（総流通額） |
| AI製品 | AI機能の月次能動的利用ユーザー数 |

### OKR との接続

```
North Star Metric: [AI機能を活用したタスク完了数]
  │
  ├── O1: オンボーディング完了率を80%に向上
  │    KR1.1: AI機能の初回利用率70%（現状45%）
  │    KR1.2: Time to first value 5分以内（現状12分）
  │
  ├── O2: 月次能動的AI利用ユーザーを30%増加
  │    KR2.1: AI機能の利用率 55%（現状40%）
  │    KR2.2: 週3回以上利用するユーザー率 25%（現状15%）
  │
  └── O3: AI機能によるチャーン削減
       KR3.1: AI heavy users（週5回以上）のチャーン率5%以下
       KR3.2: AI機能未利用ユーザーの活性化30%
```

## 混同行列（Confusion Matrix）— AI PM必須理解

### なぜAccuracyだけでは騙されるのか

**落とし穴の例**: スパムフィルターのデータ分布が「正常メール99%・スパム1%」の場合、「すべて正常」と予測するだけでAccuracy 99%を達成できる。しかしこのモデルはスパムを1件も検出できない完全に役立たずなモデルだ。

混同行列はこの欺瞞を即座に暴露する。

### 4象限の定義（スパムフィルターを例に）

| | 予測:スパム | 予測:正常 |
|---|---|---|
| **実際:スパム** | **True Positive (TP)** ✅ スパムをスパムと正しく検出 | **False Negative (FN)** ❌ スパムを見逃した（危険なミス） |
| **実際:正常** | **False Positive (FP)** ⚠️ 正常メールをスパムと誤検知（迷惑なミス） | **True Negative (TN)** ✅ 正常メールを正常と正しく識別 |

### PMが選ぶべき指標とその使い分け

| 指標 | 計算式 | PMとしての使い時 | ビジネス判断 |
|------|--------|----------------|------------|
| **Accuracy** | (TP+TN)/(全体) | 概要把握のみ | 不均衡データでは危険 |
| **Precision** | TP/(TP+FP) | 誤検知コストが高い場合 | 例: 不正取引検知で誤判断が信頼を損なう |
| **Recall** | TP/(TP+FN) | 見逃しコストが高い場合 | 例: 疾患検知で見逃しは生命に関わる |
| **F1-Score** | 2×(P×R)/(P+R) | PrecisionとRecallのバランスが必要 | 医療・金融以外の一般的AIシステム |

**PMの問いかけ**: 「誤検知（FP）と見逃し（FN）のどちらのコストが高いか？」— この問いへの答えが最適指標を決定する。

## AI/ML A/Bテストが特に重要な3理由

一般的な製品A/Bテストとは異なり、AI/MLには固有の理由がある:

### 理由1: AIは確率論的、ルールベースではない
従来ソフトウェアは決定論的（同じ入力→同じ出力）だが、AIは確率に基づいて予測する。False Positive・False Negativeが常に存在し、A/BテストはこれらのReal Worldインパクトを定量化する唯一の手段。

### 理由2: 意図しない副作用（Unintended Consequences）
AIモデルは訓練データと異なるReal Worldデータに晒されたとき、予期しない振る舞いをすることがある。A/Bテストで全ユーザーに影響が広がる前に副作用を発見・対処できる。

### 理由3: モデル性能 ≠ ユーザー体験
精度・再現率がテストデータセットで高くても、それは実際のユーザー行動改善を保証しない。A/Bテストはユーザーの行動変化とビジネス成果を直接測定する。

### A/BテストはAI/MLの「安全網かつ最適化エンジン」
- **Validate**: AI機能が実際に望ましいインパクトをもたらしているか証明
- **Optimize**: モデルとUI統合を最大効果に向け細調整
- **Mitigate Risk**: 広範な被害の前にネガティブな結果を早期発見
- **Learn**: ユーザーとAI機能の関わり方に関するインサイト獲得

## AI ROI計算 — エンドツーエンド例（チャーン予測）

### 前提条件とベースライン

```
プラットフォーム規模: 10,000サブスクライバー
月額費用: $50/ユーザー
現状チャーン率: 月次20%（手動ルールベース処理で改善率30%しか捕捉できない）
```

### A/Bテスト結果

| 指標 | コントロール群（AI未導入） | トリートメント群（AI介入） |
|------|----------------------|----------------------|
| チャーン率 | 20%（ベースライン） | 15%（5%ポイント改善） |
| 月次解約ユーザー数 | 2,000人 | 1,500人 |

### 財務インパクト換算

```
維持できたユーザー数: 500人/月
月次収益保全額: $25,000（500人 × $50）
年間収益保全額: $300,000

完全なROI計算には開発・デプロイ・介入コストを差し引く必要があるが、
このスケールの収益インパクトはほぼあらゆるAIプロジェクト投資を正当化する。
```

### 指標選択4ステップガイド

```
Step 1: ビジネスゴールを定義する
  「AI機能なし」→「あり」で何がどう変わるか？
  例: 「チャーン率を20%→15%に削減する」

Step 2: AIアプリケーションの種類を特定する
  分類問題 → Precision/Recall/F1-Score
  回帰問題 → MAE/RMSE
  推薦システム → クリック率/エンゲージメント率

Step 3: エラーコストのトレードオフを評価する
  「FP（誤検知）のコスト」vs「FN（見逃し）のコスト」を比較
  コスト非対称の場合はその方向に合わせた指標を優先

Step 4: 指標候補を絞り込む
  Primary Metric（1つ）: 仮説と直結した主指標
  Secondary Metrics（2-3つ）: ガードレール・副次効果の監視
  Business KPI（1つ）: ビジネス成果への変換（収益・コスト・NPS等）
```
