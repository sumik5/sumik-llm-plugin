# AIケーススタディ集（PM実践視点）

## 概要

本ドキュメントでは、4つのAIプロダクト実装事例をPM視点で解説します。各ケーススタディは**ビジネス課題 → AI問題への翻訳 → PM判断ポイント → 成果 → 教訓**という構造で整理しています。

### AI Case Study 文書化テンプレート：セクション別責任者

| セクション | 主担当 | PMの関与度 |
|-----------|--------|----------|
| Executive Summary | PM | ◎ 主担当 |
| Solution Approach | ML Engineer / Data Scientist | △ レビュー |
| Data Collection & Preprocessing | ML Engineer / Data Engineer | △ レビュー |
| Feature Engineering | ML Engineer | × 任意 |
| Model Development | ML Engineer / Data Scientist | × 任意 |
| Model Performance | ML Engineer + PM | ○ 共同 |
| Deployment | MLOps Engineer + Software Engineer | △ レビュー |
| Challenges and Limitations | PM（エンジニアの協力を得て） | ◎ 主担当 |
| Business Impact | PM（analytics/financeと連携） | ◎ 主担当 |

**PMの主担当セクション**: 「なぜこの問題をAIで解くのか」「何をもって成功とするか」「どんな限界があるか」の3点を明確にすること。

## Case Study 1: AIエクササイズ認識 × Computer Vision

### ビジネス課題と背景

医療系ヘルスケアプラットフォームが、ライブジム授業のスケーラビリティ問題に直面していた。1人のインストラクターが対応できるユーザー数が限られており、成長停滞と収益性低下が課題だった。

**AIへの問題翻訳:**
- ビジネス課題: 「トレーナー1人あたりのキャパシティ制約」
- AI問題定義: 「カメラ映像から体のキーポイント座標を抽出し、理想フォームとのポーズマッチングを実現するモデル」

### PM判断ポイント

| 判断項目 | 決定内容 | 判断の理由 |
|---------|---------|----------|
| North Star Metric | トレーナー費用の月次削減率 | AIの直接的ビジネス価値を測定 |
| Counter Metric | ユーザーリテンション率 | コスト削減のみ最適化してユーザー体験を毀損しないための防護 |
| MLモデル精度指標 | Pose Matching Accuracy（類似ポーズ>80%、非類似ポーズ<30%） | ビジネス要件から逆算した閾値設計 |
| スケーラビリティ要件 | 新種目追加時に再トレーニング不要 | ライブラリ拡張スピードが事業成長に直結 |

**PMのポイント**: Counter Metricを設定することで「コスト削減」と「ユーザー体験」のトレードオフを可視化した。

### ソリューション概要

- Computer Vision（Mediapipe）で体のキーポイント座標を抽出
- Triplet Loss学習戦略でポーズマッチングを一般化（新種目追加のたびにモデル再学習不要）
- サーバーレスクラウド関数 + FAISS Vector DBで推論を実装
- オンデバイス処理（端末側でキーポイント抽出） + クラウド推論の分散アーキテクチャ

### 成果とPMの教訓

**ビジネス成果:**
- 月次トレーナー費用の大幅削減（North Star達成）
- ユーザーリテンション維持（Counter Metric維持）
- 新種目追加の高速化によるサービス拡張性向上

**PMの教訓:**

| 学び | 実践への適用 |
|------|------------|
| モデルのスケーラビリティはビジネス要件から定義する | 「新しいエクササイズを毎週追加したい」→ モデル要件に「追加学習コスト最小化」を明記 |
| Counter Metricなしの最適化は危険 | AIによるコスト削減施策には必ずユーザー体験系のCounter Metricを設定する |
| データ品質の課題は事前に特定する | カメラアングルのバリエーション不足、ポーズ定義の揺れ → データ収集計画に組み込む |
| エッジ推論とクラウド推論の分割設計はPMが関与する | レイテンシー要件、コスト、プライバシーのバランスをPMが定義する |

## Case Study 2: GenAI医療トリアージチャットボット

### ビジネス課題と背景

オンラインフィットネスプラットフォームで顧客離脱率30%の問題が発生。調査の結果、筋骨格系（MSK）の痛みを抱えるユーザーが、トレーナーへの問い合わせに対するレスポンス遅延を理由に離脱していることが判明した。

**AIへの問題翻訳:**
- ビジネス課題: 「MSK問題ユーザーへの24/7対応ができない」
- AI問題定義: 「ドメイン特化（MSK）のGenAIチャットボットによるユーザー自己ケア支援」

### PM判断ポイント

| 判断項目 | 決定内容 | 判断の理由 |
|---------|---------|----------|
| North Star Metric | ユーザーリテンション率 | 離脱防止という根本的なビジネス価値と直結 |
| MLモデル精度指標 | Content Quality Score（1〜10スケール） | ユーザーの信頼に関わる医療ドメインの特性 |
| モデル選択 | GPT-4o（GPT-4o miniより高品質） | レスポンスフォーマット遵守率がGPT-4o miniで50%失敗 → UX破綻のリスク |
| UXデザイン方針 | 短文レスポンス + クイックリプライ提案 | スマートフォン利用前提、ユーザーの入力負荷を最小化 |

**PMのポイント**: GenAIの評価は「客観的指標が存在しない」問題を持つ。PMはカスタム評価フレームワーク（AIエージェントによる自動評価）の設計にオーナーシップを持つ必要がある。

### ソリューション概要

- GPT-4o + Prompt Engineeringによるドメイン限定チャットボット（MSK以外のトピックは拒否）
- カスタム評価フレームワーク: AIエージェントがコンテンツ品質を1〜10でスコアリング
- レスポンスフォーマットの自動検証: ルールベースのチェックで形式適合率を測定
- プロダクションモニタリング: 週次でランダムサンプルの手動レビュー + AIエージェント評価

### 成果とPMの教訓

**ビジネス成果:**
- ユーザーリテンション20%改善（North Star達成）
- ユーザー1セッションあたりの平均利用時間増加（エンゲージメント向上の指標）
- 今後の拡張計画：食事・栄養・ライフスタイル管理チャットボットへの横展開

**PMの教訓:**

| 学び | 実践への適用 |
|------|------------|
| GenAIの「評価インフラ」はプロダクト要件の一部 | QAフレームワーク設計をMLチームに丸投げせず、PMが評価基準を定義する |
| フィードバックループのないAIは改善できない | 個別レスポンスへのユーザーフィードバック機能は初期から設計に含める |
| モデルの汎用性とドメイン適合のトレードオフを設計する | Prompt Engineeringによるスコープ制限がUXとリスク管理に有効 |
| コスト効率と品質のバランスはPMが判断する | GPT-4o vs GPT-4o miniの選択は純技術判断ではなく事業判断 |

## Case Study 3: AI需要予測による動的価格設定

### ビジネス課題と背景

Eコマースプラットフォームで手動価格設定が収益機会損失を生んでいた。需要変動への対応が遅く、高需要時の最適価格設定ができていなかった。

**AIへの問題翻訳:**
- ビジネス課題: 「手動価格設定ではリアルタイム需要変化に追いつけない」
- AI問題定義: 「過去の販売データと価格履歴から翌日需要を予測し、収益最大化価格（Price × Demand）を算出するリグレッションモデル」

### PM判断ポイント

| 判断項目 | 決定内容 | 判断の理由 |
|---------|---------|----------|
| North Star Metric | 日次収益（Revenue per day） | プライシング施策の直接指標 |
| Counter Metric | 顧客離脱率 | 高値設定がユーザー離れを招かないための防護 |
| 実験設計 | ユーザー×時間軸のダブルバリアントA/B | ステークホルダーが通常のA/Bテストを拒否 → 時間バイアスとユーザー選択バイアスを除去するカスタム設計 |
| ビジネスルール統合 | 価格は仕入れ値以上・最大販売価格以下に制約 | モデル出力がビジネス制約を常に尊重する設計 |

**PMのポイント**: ステークホルダーが「同一商品を異なるユーザーに異なる価格で見せたくない」という懸念を持った場合のA/Bテスト設計が鍵。バイアス除去の方法論をPMが主導する必要がある。

### ソリューション概要

- LightGBMリグレッションモデルで翌日需要を予測
- Airflow DAGによる毎朝5時の全商品価格自動更新
- 異常検知: 3σを超える予測誤差の商品はAI価格設定を自動停止し手動レビューへ
- 自動再学習パイプライン: 需要ドリフト検知時にモデル自動再トレーニング

### 成果とPMの教訓

**ビジネス成果:**
- 平均日次収益3%増（95%信頼区間でマニュアル価格設定を上回る）
- 手動価格設定コストの削減
- 在庫全体へのスケールアウト実現

**PMの教訓:**

| 学び | 実践への適用 |
|------|------------|
| 「収益比較」は本質的に確率論的 | AI施策のROI評価は点推定ではなく信頼区間で報告する習慣をステークホルダーに根付かせる |
| ビジネスルールとAI出力の統合設計はPMの仕事 | 価格制約（仕入れ値下限・上限）はAIシステム設計に組み込む要件として定義する |
| ステークホルダー懸念は実験設計の質を上げる | 「ユーザーに差別化価格を見せたくない」という懸念→ より堅牢なバイアス除去A/B設計へ |
| モデルの誤差傾向（過大/過小推定）を活用する | 体系的な過小推定商品は価格を高め、過大推定商品は低めに設定する補正ロジックでモデルを強化 |

## Case Study 4: Eコマース検索広告のAI最適化

### ビジネス課題と背景

Eコマースプラットフォームの検索結果ページ（モバイル）で、広告スロットの60〜70%しか埋められていなかった。既存の完全一致キーワード手法は高CTR（2.8%）だが在庫が限られ、部分一致では CTRが1.5%に低下しユーザー体験と広告収益が悪化していた。

**AIへの問題翻訳:**
- ビジネス課題: 「広告スロットが常に未充填で収益機会を損失」
- AI問題定義: 「検索クエリと商品タイトルのセマンティックマッチングにより、完全一致以外の関連商品も高CTRで表示するモデル」

### PM判断ポイント

| 判断項目 | 決定内容 | 判断の理由 |
|---------|---------|----------|
| North Star Metric | 広告収益（Ads revenue） | スロット充填率向上によるマネタイゼーション改善 |
| Counter Metric | 顧客満足度（Customer satisfaction） | 広告品質低下がユーザー体験を毀損しないための防護 |
| MLモデル精度指標 | CTR by Match Score Bucket | モデルのマッチスコアとビジネス成果（CTR）を直接接続 |
| 精度評価方針 | カテゴリマネージャーによる手動テスト（False Negative重視） | 関連商品を表示しないリスクのほうが無関係商品を表示するより大きい |

**PMのポイント**:「False NegativeとFalse Positiveのどちらが事業に深刻か」という非対称リスクをPMが定義し、モデル評価閾値に反映させること。

### ソリューション概要

- Siamese深層学習モデルで検索クエリと商品のセマンティックマッチングスコアを算出
- FastText Word Embeddingsで語彙外クエリ（タイポ等）にも対応
- 高トラフィック対応: 最大4台のGCPサーバー（フラッシュセール時）+ Milvus Vector DB
- Redisキャッシュで繰り返しクエリの処理負荷を削減
- A/Bテスト: コントロール（完全一致のみ）vs バリアント（AI + 完全一致）でCTR比較

### 成果とPMの教訓

**ビジネス成果:**
- 広告収益20%増
- 広告スロット充填率95%以上達成（完全一致のみでは60〜70%）
- AIモデルのCTR 2〜2.5%（完全一致の2.8%に近い水準を維持）
- RoAS（広告費用対効果）への悪影響なし

**PMの教訓:**

| 学び | 実践への適用 |
|------|------------|
| マッチスコアとCTRの直接相関を確認する | モデルスコア（>90%: CTR 2.5%、70〜90%: 2.0%、50〜70%: 1.5%）→ スコア閾値の設定がUXを決める |
| 非対称リスクはPMが定義する | 「関連商品を見逃す」vs「無関係商品を表示する」の深刻度を事業観点から定義してモデル評価に反映 |
| スケーラビリティ要件は事前にトラフィックピークから逆算する | フラッシュセール時の2,000req/secを想定したインフラ設計が必要 → プロダクト要件に明記 |
| 既存システムとの共存戦略を設計する | AI推奨と既存キーワードマッチングの重複排除、表示優先度ルールをPMが設計する |

## AI Case Study 文書化フレームワーク

### PMが必ず定義すべき4項目

```
1. North Star Metric  → AIが最終的に動かすべきビジネス指標
2. Counter Metric     → North Star最適化の暴走を防ぐ指標
3. MLモデル精度指標   → ビジネス要件から逆算した技術的評価基準
4. 非対称リスク定義   → False NegativeとFalse Positiveの事業影響の大小
```

### セクション別責任者テーブル（詳細版）

| セクション | 主担当 | PMが確認するポイント |
|-----------|--------|-------------------|
| Executive Summary | PM | ビジネス課題の明確さ、成功指標の妥当性 |
| Solution Approach | ML Engineer | AIへの問題翻訳が正しいか、指標設計は適切か |
| Model Performance | ML + PM | ビジネス要件の閾値を満たしているか |
| Deployment | MLOps | SLAとコスト制約を満たしているか |
| Challenges and Limitations | PM | ユーザーへの影響、ステークホルダーへのリスク開示 |
| Business Impact | PM | North Star / Counter Metricの達成状況 |

### ユーザー確認の原則（AskUserQuestion）

**判断分岐がある場合、推測で進めず必ずAskUserQuestionツールでユーザーに確認する。**

確認すべき場面:
- North Star MetricとCounter Metricの選択（複数候補がある場合）
- A/Bテスト設計の方針（特にステークホルダー制約がある場合）
- モデル評価閾値の設定（False Negative vs False Positiveの優先度）
- GenAI評価フレームワークの設計（主観的品質の定義）

確認不要な場面:
- ソース出典情報の除去
- セクション責任者テーブルの標準的な適用
- 一般的なAIモニタリング指標（ドリフト検知、精度追跡）
