# ソフトウェアサプライチェーンのオブザーバビリティ

ソフトウェアサプライチェーンとは「開発からCI/CDパイプラインを経て、本番環境にデプロイされるまでのソフトウェアに入るもの、あるいは影響を与えるものすべて」です。本番環境と同様に、ビルドシステムにも分散システムとしてのオブザーバビリティが不可欠です。

---

## 1. なぜCI/CDにオブザーバビリティが必要なのか

### 内部ツールの特性

| 特性 | 内容 |
|------|------|
| **複雑性** | コードのテスト・デプロイワークフローは一部の本番サービスより複雑になることがある |
| **依存関係の変化** | インフラ種類・ランタイムバージョンが予期せず変化し意図しない不具合が発生する |
| **カーディナリティ** | 本番環境より大幅に低いが、単一イベント・スパンの重要度は大幅に高い |
| **影響の連鎖** | サプライチェーン初期のボトルネックが後続全体のボトルネックになる |

### 「遅い」と「不安定」が最難問

```
症状: "遅くなっている" / "不安定だ"
↓
課題: 相互作用する高複雑度システムにおける問題の相関付け
↓
解決: 分散トレースによるオブザーバビリティ
```

ビルドシステムがある程度複雑になると、**オブザーバビリティは必須のデバッグツール**となります。不十分な統合、目に見えないボトルネック、問題の原因特定不能は、本番環境と同様にビルドシステムを苦しめます。

## 2. 共有クライアントライブラリとディメンション設計

### ディメンションは「問題調査のパンくずリスト」

CI/CDパイプラインの各ステージに追加のディメンションを付与することで、問題発生時にコンテキストを辿れる「パンくずリスト」として機能します。

```
コミットプッシュ → Checkpoint（オーケストレーション）
                 → Jenkins ビルダー
                 → Jenkins テストエグゼキューター
                 → QA環境
各ステージに共通ディメンションを伝播させる
```

### 共通ディメンションの設計例（Slack Hackコードベースより）

| ディメンションカテゴリ | 具体例 | 活用用途 |
|----------------------|--------|---------|
| **インフラ** | ホスト名、Auto Scalingグループ | 特定ホストの異常検出 |
| **実行環境** | Jenkinsワーカーグループ・ラベル | 並行性問題の特定 |
| **コード変更** | コミットヘッド、コミットメインブランチ | 壊れたビルドの原因コミット特定 |
| **テスト** | test_suite名、テストケースID | スイート単位のパフォーマンス追跡 |
| **リソース** | メモリ使用量（P50/P95/P99） | リソース劣化の時系列追跡 |

### TraceContextシングルトンパターン

```
TraceContext（シングルトン）
├── 共通ディメンションで初期トレースを構築
├── 新しいトレースを構築
└── 各トレースには複数スパン＋ディメンション配列
    └── 個々のスパン（例: runner.test_execution）
        └── 元のリクエストコンテキスト ＋ ルートスパンへのディメンション追加
```

**ディメンション追加 = 問題調査のコンテキストと豊かな情報の追加**

## 3. Slack事例: Checkpoint・Jenkins・QA環境のトレース

### Slackのウェブアプリ CI エコシステム構成

```
GitHub（コミットプッシュ）
    ↓ Webhook
Checkpoint（CI/CDオーケストレーター）
    ↓ ビルド・テストリクエスト
Jenkins（ビルダー・テストエグゼキューター）
    ↓ CIBotがスクリプト実行
テスト環境（QA環境 x 複数サービス）
```

### トレース適用による具体的な問題解決実績

| 時期 | 問題 | 計装後の解決 |
|------|------|------------|
| **2019 Q2** | Gitチェックアウトの異常な実行時間（エラーなしで遅い） | トレースデータが出てから**数分で**ASGで未更新のホストを特定・デプロビジョニング |
| **2019 Q3** | Git LFSによるシステム全体のスループット低下・複数チームのインシデント | CIランナーへの簡単な計装追加で**2時間以内に**Git LFSから成果物を取得できないホスト群を特定・解決 |
| **2020後半** | E2Eテストのフレーク率15%、ピーク時週10万時間の廃棄計算時間 | Cypressのランタイムパラメータにトレース追加→数日以内に設定差分と相関するディメンションを発見→フレーク率**0.5%以下**に改善 |
| **2021年8月** | バックエンドユニットテストのOOMによる大量失敗 | メモリ使用量の時系列トレースで変曲点を特定、原因コミットを切り戻し |

## 4. アクション可能なアラートの設計

### ステークホルダーごとのSLO設計

同一のテスト実行でも、チームによって関心事が異なります。それぞれのSLOに基づいたアラートを設計します。

| ステークホルダー | 関心事 | SLO候補 |
|----------------|--------|---------|
| **テストスイートオーナー** | 不安定性・信頼性・メモリ使用量 | フレーク率 < 1%、P95実行時間 |
| **テストインフラチーム** | Docker操作パフォーマンス・コスト効率 | 操作レイテンシ・テストあたりコスト |
| **デプロイメントオーナー** | テスト内容・ホットフィックスCIの進捗 | デプロイパイプライン完了時間 |
| **社内ツールチーム** | テスト結果処理スループット | 処理キュー長・スループット |

### アクション可能なアラートフロー

```
異常検出アラート（ビジネスメトリクス）
    または
スイート固有アラート（P50を超える実行時間増加など）
    ↓
オブザーバビリティツールへのリンク付き通知（Slackメッセージ）
    ↓ リンクをクリック
test_suiteディメンションでフィルタされたビューコレクション
    ↓
REDダッシュボード（Rate・Error・Duration）でドリルダウン
    ↓
対応者が仮説検証 → アクション実施 → テレメトリーで健全性確認
```

### アラート設計のチェックリスト

- [ ] アラートにオブザーバビリティツールへの直接リンクを含める
- [ ] リンク先は問題に関連したディメンションで事前フィルタされたビュー
- [ ] 各チームのユースケースに基づいてダッシュボードを個別作成
- [ ] 高レベルの異常検出アラートと、スイート固有アラートの両方を用意
- [ ] 仮説→アクション→検証のループをリアルタイムで完結できるクエリーを準備

## 5. 変更の追跡とデプロイメント相関

### 変更追跡が重要な理由

大規模なコードベースでの劣化原因特定は困難です。その理由：

- **変化速度が高い**: コードベースとインフラの両方で多くの変数が同時に変化
- **複数の相互作用**: コード変更・インフラ変更・プラットフォームランタイムにまたがる複合的影響
- **時間的なずれ**: 変更から劣化の顕在化まで時間差が生じる場合がある

### デプロイメントと障害の相関付け手順

```
1. メモリ使用量・レイテンシ等の時系列データをトレースで記録
2. P50・P95・P99の変曲点を時系列で特定
3. 変曲点の時刻に紐付くプルリクエスト・コミットを特定
4. 原因候補コミットを切り戻し
5. テレメトリーで健全性を確認（変更後の待機は必須）
```

### CAN（Conditions/Actions/Needs）プロセスへの組み込み

インシデント対応においてオブザーバビリティデータを構造化するフレームワーク：

```
Conditions（状態記述）:
  - 現在の症状（例: OOMで高頻度失敗）
  - 確認されている変曲点・傾向
  - 現在調査中の仮説

Actions（実施中のアクション）:
  - トレースデータを用いた仮説検証
  - 原因候補の切り戻し

Needs（必要なもの）:
  - 追加の計装・テレメトリー
  - 専門知識・担当チームのエスカレーション
```

## 6. 導入パターンと実践ガイド

### サプライチェーン計装の優先順位

| 優先度 | 計装対象 | 期待される効果 |
|--------|---------|--------------|
| **高** | CI/CDオーケストレーター（Checkpointなど） | パイプライン全体の可視化 |
| **高** | ビルドランナー・テストエグゼキューター | ボトルネック特定 |
| **中** | テスト環境・QA環境 | フレーキーテストの根本原因特定 |
| **中** | 依存サービス（Git、成果物ストレージ等） | 依存関係の障害検出 |
| **低** | 個々のテストケース | 詳細なリソース使用量追跡 |

### 共有クライアントライブラリ設計のポイント

1. **TraceContextシングルトン**を採用し、すべてのサービスで共通ディメンションを自動付与
2. **段階的に拡張**: 基本ディメンション（ホスト名・コミット）→詳細ディメンション（メモリ・並行数）
3. **可逆的な実験**: ディメンションを追加して相関を確認→設定変更→効果測定のループ
4. **フレーキー検出に注目**: フレーク率・実行時間の統計（P50/P95/P99）を必ず記録

### よくある落とし穴と対策

| 落とし穴 | 対策 |
|---------|------|
| フレーキーの原因をテストコードのみに帰属させる | インフラ・設定・ランタイムパラメータも計装して相関を確認 |
| 中央チームだけが問題を把握 | 各テストスイートオーナーがアクセスできるダッシュボードとアラートを整備 |
| 変更後すぐに健全性を判断する | テレメトリーが安定するまで待機してから判断する |
| ディメンションが少なすぎる | 「後から追加する」より「最初から広く取る」——ディメンションは調査の選択肢を広げる |

## 7. 効果測定の指標

| 指標 | 計測方法 | 改善目標 |
|------|---------|---------|
| **テストフレーク率** | 同一コードで異なる結果になる割合 | < 1%（Slackは15%→0.5%以下を達成） |
| **テストターンアラウンドタイム（P95）** | コミットプッシュ～全テスト完了時間 | 組織のベースラインから50%削減 |
| **廃棄計算時間** | フレーキー失敗による無駄な実行時間 | ピーク時比で大幅削減 |
| **MTTD（検出時間）** | 問題発生から検出までの時間 | アラート→担当者通知を分単位で |
| **MTTR（解決時間）** | 問題検出から解決までの時間 | トレースデータを活用し時間単位以内 |

## 参考: Slack事例の教訓

> 「ソフトウェアサプライチェーンを計装することは、競争力を高めることになります。より速い開発＋より速いリリースサイクル＝顧客へのより良いサービス」
> — Frank Chen（Slack シニアスタッフソフトウェアエンジニア）

**主な教訓:**
- CI/CDパイプラインは本番環境と同等の分散システムとして扱う
- 計装の開始から**数時間～数日以内**に問題を特定できる場合が多い
- エラーが出ていない「ただ遅い」問題こそトレースが威力を発揮する
- アクション可能なアラートは「気づき」と「調査ツールへの直接リンク」をセットで提供する
