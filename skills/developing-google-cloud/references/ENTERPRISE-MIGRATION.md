# エンタープライズ移行設計（移行パターン・ステップ・移行方法）

オンプレミスから Google Cloud への移行設計の全体像を扱う。移行パターン選択、4 ステップの移行プロセス、サーバー・データ・データベースの各移行方法、クラウドネイティブへのモダナイゼーション戦略まで、エンタープライズ移行に必要な知識を体系化する。

## 移行パターン選択

### 4 つの移行パターン

Google Cloud への移行パターンは、移行後にアプリケーションが目指す状態によって 4 種類に分類される。

| パターン | 日本語名 | アプリへの変更 | 移行難易度 | マネージドレベル | 主な移行先プロダクト |
|---------|---------|-------------|---------|---------------|-----------------|
| **Lift & Shift** | リフト＆シフト | なし（そのまま移行） | ⭐ | 低 | Bare Metal Solution、Google Cloud VMware Engine |
| **Lift & Optimize** | リフト＆最適化 | 最小限（プラットフォームのみ変更） | ⭐⭐ | 中 | Compute Engine、Cloud SQL |
| **Improve** | 改良 | 機能は変えずコードをクラウドネイティブに書き換え | ⭐⭐⭐ | 高 | GKE、Anthos、App Engine、Cloud Run |
| **Rebuild** | 再構築 | 要件定義から再設計・新規開発 | ⭐⭐⭐⭐ | 最高 | Cloud Run、Cloud Functions、マネージドサービス全般 |

### パターン別トレードオフ

| パターン | メリット | デメリット | 推奨ケース |
|---------|---------|----------|----------|
| **Lift & Shift** | 最短期間で移行完了。運用手順の変更が最小 | クラウドのメリット（スケーラビリティ・コスト最適化）を完全には享受できない | まず迅速にクラウド移行し、後から最適化するアプローチ |
| **Lift & Optimize** | マネージドプロダクトにより運用を一部委任 | L&S より学習コスト・移行作業が増加 | 変更量を抑えつつオートスケール・コスト管理等の恩恵を得たい場合 |
| **改良** | GKE/Cloud Run 等の高マネージドサービスが利用可能 | アプリの書き換えが必要。スキル習得も必要 | L&S では効果不十分、または再構築では時間がかかりすぎる場合 |
| **再構築** | クラウド移行の効果を最大化。最新技術を自由に採用 | 最も時間・コスト・スキル投資が大きい | 既存アプリが GCP 非対応、または維持コストが高すぎる場合 |

### レガシーアプリ vs クラウドネイティブアプリ

| 特性 | レガシーアプリ | クラウドネイティブアプリ（Twelve-Factor App） |
|-----|-------------|------------------------------------------|
| **アーキテクチャ** | モノリシック | マイクロサービス（小さな独立したサービスの集合） |
| **スケーリング** | 垂直スケール（手動） | 水平スケール（自動） |
| **状態管理** | ステートフル | ステートレス |
| **設定管理** | 設定ファイル内に記述 | 環境変数で外部化 |
| **マネージドサービス** | 活用困難 | 積極活用 |
| **インフラ担当** | 構成変更・障害対応に時間を費やす | 改善活動に集中できる |

### 移行先プロダクト選択のディシジョンポイント

| 確認項目 | Yes の場合 | No の場合 |
|---------|----------|----------|
| Cloud へ移行できる（コンプライアンス・依存関係等をクリア）| 次の確認へ | コロケーションサービスを検討 |
| サーバーレスプロダクトの制約要件をクリアできる | Cloud Run / App Engine / Cloud Functions | コンテナ適合性を確認 |
| コンテナ化に適している（Linux アプリ等） | GKE / Anthos | Compute Engine を検討 |
| ハイブリッド/マルチクラウド要件がある | Anthos | GKE |
| vSphere のまま移行したい | Google Cloud VMware Engine | Compute Engine |

## 移行ステップ（4 フェーズ）

Google Cloud が推奨する移行プロセスは以下の 4 ステップ。

| ステップ | 名称 | 主な作業内容 |
|---------|------|------------|
| **1** | ワークロード評価 | 移行ビジョンの明確化。全ワークロードの情報整理。最初に移行するアプリの決定 |
| **2** | 移行計画の策定 | 移行後アーキテクチャの設計。アカウント・セキュリティ・ネットワーク・監視設計を並行して実施 |
| **3** | 移行の実施 | 移行方式に応じた作業（VM移行・DB移行・アプリ書き換え）の実施 |
| **4** | 移行後環境の最適化 | マネージドサービスへの移行等、クラウドの恩恵をさらに拡大する最適化を継続 |

### クラウド適用フレームワーク（組織の成熟度評価）

移行前に組織のクラウド成熟度を評価することが重要。Google Cloud 導入フレームワークでは 4 つの能力と 3 フェーズを定義。

| 能力 | 評価観点 |
|-----|---------|
| **Learn（学習）** | クラウド技術習得・トレーニング体制 |
| **Lead（統率）** | 経営層のコミットメント・組織変革 |
| **Scale（拡大）** | 標準化・自動化・ガバナンス |
| **Secure（セキュリティ）** | セキュリティ・コンプライアンス体制 |

| フェーズ | 特徴 |
|---------|------|
| **戦術フェーズ** | 基本的なクラウド移行。コスト削減が主な目的 |
| **戦略フェーズ** | クラウドの機能を活用して新しいビジネス価値を創出 |
| **変革フェーズ** | クラウドファーストの組織へ完全移行 |

## サーバー移行方法

### 移行ユースケース別の手法選択

| 移行元 | 移行先 | 推奨手法 |
|-------|-------|---------|
| VMware（そのまま） | Google Cloud VMware Engine | VMware HCX（L2延伸・vMotion） |
| VMware / AWS / Azure | Compute Engine | Migrate for Compute Engine（M4CE） |
| 仮想マシン（OVF/OVA形式） | Compute Engine | 仮想アプライアンスのインポート |
| 仮想ディスク（VMDK/VHD等） | Compute Engine | 仮想ディスクのインポート（自動インポート推奨） |

### VMware HCX による移行

| 特性 | 内容 |
|-----|------|
| **ライセンス** | Google Cloud VMware Engine ライセンスに含まれる（追加コスト不要） |
| **前提条件** | Cloud Interconnect による VPC への接続が必要 |
| **特徴** | オンプレミスの L2 ネットワークをクラウドまで延伸。IP アドレス変更なしで移行可能 |
| **無停止移行** | vMotion 機能で仮想マシンを稼働させたまま移行可能（ネットワーク遅延への考慮は必要） |

### Migrate for Compute Engine（M4CE）

Google Cloud が提供する無料の VM 移行ツール。

| 特性 | 内容 |
|-----|------|
| **対応移行元** | vSphere 環境、AWS、Azure の IaaS |
| **エージェントレス** | 移行元へのエージェントインストール不要 |
| **ストリーミング** | アクセス頻度の高いデータを優先的にストリーミング。数分で GCE 上での稼働が可能 |
| **コスト最適化** | 移行前にリソース使用状況を確認してサイジングを推奨 |
| **テスト移行** | 事前に移行テストが可能。移行時の問題発生時は移行元への書き戻しも可能 |
| **一括移行** | 複数 VM の一括移行または分割移行に対応 |
| **OS アップグレード** | Windows Server 2008 R2 から 2012 への自動アップグレードに対応 |

**M4CE 移行手順:**
1. Google Cloud 環境を準備（IAM・ネットワーク・Marketplace から M4CE Manager デプロイ）
2. 移行元の準備（M4CE 用権限付与・ソースとして設定）
3. Cloud Extension を設定（small: 最大 20 VM / large: 最大 50 VM）
4. ランブック（CSV）を作成して移行ジョブを実行

## データ移行方法

### データ移行方式の選択基準

| 方式 | 特徴 | 適したケース |
|-----|------|------------|
| **Cloud Storage オンライン転送** | gsutil コマンドで直接アップロード | ファイル数が多い場合・定期同期 |
| **Storage Transfer Service（クラウド用）** | 他社クラウド（AWS S3等）からの転送・スケジュール設定 | 他クラウドからの移行・定期転送 |
| **Transfer Service for On-Premises Data** | オンプレミスから帯域幅制限付きで大量転送 | 数十億ファイル・ペタバイト規模 |
| **Transfer Appliance** | 大容量ストレージサーバーを物理輸送 | 期限内に転送できる帯域幅がない場合（米国・シンガポールのみ） |
| **仮想ディスクインポート** | VMDK/VHD 等をインポート | ファイル数が少なく I/O 停止可能な場合 |
| **OSコマンド（rsync等）** | 慣れ親しんだコマンドでの移行 | 小規模・ファイル数が少ない場合 |

### gsutil コマンド活用

```bash
# マルチスレッドで複数ファイルをコピー
gsutil -m cp -r top-level-dir gs://example-bucket

# ディレクトリの継続的同期（-d: 同期元にないファイルを同期先で削除）
gsutil -m rsync -r -d local-dir gs://example-bucket
```

| オプション | 効果 |
|----------|------|
| `-m` | マルチスレッドを有効化（大量ファイル転送を高速化） |
| `-r` | サブディレクトリを再帰的に処理 |
| `-d` | ⚠️ 同期元にないファイルを削除（注意して使用）|

### データ転送時間の目安

ネットワーク帯域幅とデータ量から転送時間を事前に試算することが重要。転送時間短縮のポイント:
- ファイルを圧縮してから転送
- 帯域幅を最大限に確保
- gsutil の並列複合アップロードを活用（大ファイルを分割して並列アップロード）

## データベース移行方法

### 移行タイプの分類

| タイプ | 定義 | 特徴 |
|-------|------|------|
| **同種間移行** | MySQL → Cloud SQL（MySQL）等、同じ DB テクノロジー間 | スキーマ変換不要。DB 固有のエクスポート/レプリケーション機能が使える |
| **異種間移行** | Oracle → Cloud Spanner 等、異なる DB テクノロジー間 | スキーマ変換・データ変換が必要。CSV/Avro 等の共通形式を使用 |

### 5 つのデータベース移行方法

| 方法 | ダウンタイム | データ変換 | 同種/異種 | 推奨ケース |
|-----|-----------|----------|----------|----------|
| **エクスポート/バックアップ** | 長い（書き込み停止が必要） | 別途実施 | 両方 | ダウンタイムを十分確保できる場合 |
| **DBレプリケーション機能** | 最小（切り替え時のみ） | 不可 | **同種のみ** | ダウンタイムがほぼ取れない同種移行 |
| **データベース移行システム** | 最小（長期同期後に切り替え） | 可能 | 両方 | ダウンタイム不可 かつ データ変換が必要 |
| **差分クエリ** | 中程度 | 可能 | 両方 | トランザクションログがない DB |
| **BigQuery Data Transfer Service** | なし | - | BigQuery 移行 | Teradata/Redshift/S3 → BigQuery |

### エクスポート/バックアップ移行の流れ

```
1. ソース DB への書き込み停止
2. ソース DB からデータをエクスポート
3. エクスポートデータをクラウドへ移動（必要に応じてデータ変換）
4. ターゲット DB にインポート
5. データ整合性を確認
6. アプリの接続先を新 DB に変更
7. 新 DB の書き込みを有効化
8. 新 DB のパフォーマンスを確認
```

**移行時間短縮のポイント:**
- 差分エクスポート機能を活用（最後のエクスポート以降の変更分のみをエクスポート）
- インポート時のインデックス一時削除
- 並列インポートの活用

### Database Migration Service（DMS）

Google Cloud が提供する GUI ベースのデータベースレプリケーション管理サービス。

| 対応 DB | 状況 |
|--------|------|
| MySQL | オンプレミス / 他クラウド から Cloud SQL（MySQL）へ |
| PostgreSQL | オンプレミス / 他クラウド から Cloud SQL（PostgreSQL）へ |
| SQL Server | プレビュー対応（ネイティブサポートへ移行中） |

### データベース移行システム（CDC方式）

変更データをリアルタイムにキャプチャ（CDC: Change Data Capture）する方式を採用し、ダウンタイムを極限まで抑制する。

| CDCの方式 | 仕組み | 推奨度 | 条件 |
|---------|-------|-------|------|
| **トランザクションログ** | DB のトランザクションログから変更を取得 | ✅ 推奨 | DB がトランザクションログ機能を持つ場合 |
| **差分クエリ** | タイムスタンプを基に一定間隔でクエリ | 代替手段 | トランザクションログがない場合 |

**主なデータベース移行システム:**

| ツール | 提供元 | 特徴 |
|-------|-------|------|
| **Datastream** | Google Cloud | CDC ベースのリアルタイムレプリケーション（プレビュー） |
| **Striim** | サードパーティ | エンタープライズ向け高機能 CDC ツール |
| **Cloud Data Fusion** | Google Cloud | Apache Beam ベースのデータ統合・変換パイプライン |

## モダナイゼーション戦略

### 移行後の継続的最適化

クラウド移行は「一度やったら終わり」ではなく、継続的に最適化するサイクルが重要。

| フェーズ | アプローチ | 目標 |
|---------|----------|------|
| **移行初期** | Lift & Shift でとにかくクラウドへ | 固定資産の削減、データセンター費用の解消 |
| **短期最適化** | Compute Engine → マネージドサービスへの移行 | 運用負荷の軽減、オートスケールの活用 |
| **中長期** | コンテナ化・サーバーレス化 | クラウドネイティブの恩恵を最大限享受 |

### マネージドレベル vs 移行難易度のトレードオフ

```
移行難易度（↑高）     再構築（Rebuild）
                    改良（Improve）
                    Lift & Optimize
移行難易度（↓低）   Lift & Shift

               マネージドレベル（低←→高）
```

**指針:** フルマネージドサービス（Cloud Run、Cloud Functions 等）を優先的に検討。インフラ・ミドルウェア管理の負担を削減し、アプリ開発に集中できる環境を目指す。

### 移行後のコスト削減効果（Lift & Shift の場合）

| 削減対象 | 内容 |
|---------|------|
| データセンター | 確保スペース・入館管理費用 |
| ハードウェア | ラック・電源・サーバー・仮想化基盤 |
| データ分析 | クラウド上の BigQuery 等を即座に活用可能 |
| AI/ML | GCP の AI/ML プロダクトで予測処理を導入しやすい |

## 移行設計チェックリスト

| フェーズ | 確認項目 |
|---------|---------|
| **パターン選択** | 移行目的・現状のシステム構成に応じて適切な移行パターンを選択したか |
| **評価** | 全ワークロードの情報を整理し、最初に移行するアプリを決定したか |
| **設計** | アカウント・セキュリティ・ネットワーク・監視設計を移行計画と並行して実施したか |
| **サーバー移行** | ライセンス要件（BYOL vs 払い出し）を確認したか |
| **サーバー移行** | M4CE を使用する場合、事前テスト移行を実施したか |
| **データ移行** | ネットワーク帯域とデータ量から転送時間を見積もったか |
| **DB移行** | 同種/異種を判定し、ダウンタイム要件に合う移行方式を選択したか |
| **DB移行** | データ整合性の確認手順を定義したか |
| **最適化** | 移行後の継続最適化計画（マネージドサービスへの段階的移行）を定義したか |
| **組織** | クラウド適用フレームワークで組織の成熟度を評価したか |
