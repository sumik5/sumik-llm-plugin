# ゲームインフラ on GCP（設計・移行・アーキテクチャ）

モバイルゲーム・オンラインゲーム・コンソールゲームのバックエンドインフラをGCPで構築・運用するための実践ガイド。ゲームインフラの特性・スパイクトラフィック対策・常時接続型アーキテクチャ・DB設計・グローバル展開・移行事例を解説する。Aiming社・grasys社・シリコンスタジオ社の実運用事例を含む。

---

## ゲームインフラの特性と要件

### 一般的なシステムとの比較

ゲームインフラは企業向けシステムと比較して本質的に異なる特性を持つ。

| 特性 | 一般的なシステム | ゲームインフラ |
|-----|----------------|-------------|
| **サーバ規模** | 基幹系を除いて10台程度 | 数十〜数百台が通常 |
| **負荷パターン** | 比較的安定・予測可能 | スパイクが大きく予測困難 |
| **スケーリング方向** | スケールアップ中心 | OSSを使ったスケールアウト中心 |
| **サイジングの難しさ** | 比較的予測可能 | ヒット/失敗で1〜100倍の差 |
| **インフラ規模の変化** | 安定 | ユーザ数に応じて急増・縮退 |

### ゲームインフラに求められる要件

```
1. 大規模サーバ管理の仕組み（数十〜数百台の効率的管理）
2. インフラコストの効率的管理（固定費の最小化）
3. ピーク対応能力（平常時の数倍〜数十倍のスパイク処理）
4. 柔軟なスケーラビリティ（需要増減への即応）
5. 需要予測困難への対応（事前計画なしで増減できる柔軟性）
```

### 中長期的インフラ戦略の3要素

昨今のゲーム市場のトレンド（モバイル化・グローバル化・マルチクラウド）から、戦略的に考えるべき要素：

| 要素 | 内容 | なぜ重要か |
|-----|------|----------|
| **インフラの柔軟性** | スケールアップ/ダウンが迅速にできる | 需要予測が困難なゲームでは固定コストを最小化 |
| **グローバルインフラ** | 複数リージョンへの展開が容易 | 国内ゲーム市場の成熟でグローバル展開が必須化 |
| **技術的中立性（ベンダニュートラル）** | 特定ベンダへの過度な依存を避ける | 乗り換え・海外展開・開発ベンダの選択肢を確保 |

## GCPがゲームインフラに向いている理由

### GCPのゲームインフラとしての優位点

| 優位点 | 内容 | ゲームインフラへの効果 |
|-------|------|---------------------|
| **価格の安さと柔軟性** | 単価が安く、支払いスキームが柔軟 | 他社クラウドからの移行で20〜30%のコスト削減が期待できる |
| **ライブマイグレーション** | ハードウェアメンテナンス中も無停止 | 計画停止不要 → メンテナンスウインドウなし → 機会損失削減 |
| **高速インスタンス起動** | SSH接続まで概ね20〜30秒 | 障害対応・オートスケール時の即応性 |
| **グローバルプライベートネットワーク** | リージョン間がVPN不要でプライベート接続 | グローバルゲームの構築コスト・レイテンシを削減 |
| **グローバルロードバランシング** | IPベースで世界中のリクエストを最適転送 | グローバルタイトルで有利 |

### ゲームタイプ別のGCP活用パターン

| ゲームタイプ | 特徴 | 推奨構成 | GCPの強み |
|------------|------|---------|---------|
| **モバイルゲーム** | スパイクしやすい・収益とアクセス数が連動 | GCE（IaaS） or GAE（PaaS） | スケール柔軟性・コスト最適化 |
| **コンソール/PCゲーム** | パッケージ販売主体・手持ち資産活用も有効 | グローバル展開の場合はGCE | グローバルネットワーク・ライブマイグレーション |
| **アーケードゲーム** | 営業時間中のメンテナンスが特に困難 | GCE | ライブマイグレーション（無停止が最重要） |
| **パズル/ライトゲーム** | レイテンシ許容 | GAE | オートスケール自動化・メンテナンス不要 |

### GAE活用のトレードオフ

| 項目 | GAEのメリット | GAEのポイント（注意） |
|-----|------------|-------------------|
| スケーラビリティ | アクセス負荷に応じて自動スケール | レイテンシがGCEより大（数百ms） |
| 可用性 | 計画停止なし | PaaSのためネイティブライブラリ使用不可の場合あり |
| 運用コスト | ミドルウェアまでのメンテ不要 | GAE固有の設計クセを理解する必要がある |
| **推奨用途** | ソーシャルゲーム・シンプルなパズルゲーム | リアルタイム性の高いゲームには不向き |

## スパイクトラフィック対策

### スパイク特性の理解

モバイルゲームのアクセス負荷は「ボラティリティ（変動の不確実性）」が高い。

- リアルメディア・ソーシャルメディアでの口コミで突発的にアクセス増加
- リリース直後・大型イベント・コラボ施策でピーク
- 人気低下時には急速な縮退が必要

### GCEを使ったスパイク対策パターン

```
【オートスケール設定の推奨値】
- 最小インスタンス数: ベースライン負荷から算出
- 最大インスタンス数: ピーク予測の2〜3倍（余裕を持つ）
- スケールアップ判断: CPU使用率60〜70%程度
- クールダウン時間: ゲームのセッション時間を考慮

【GCEの起動速度を活かす】
- インスタンス起動 → SSH接続まで20〜30秒
- Startup Scriptでの初期化処理を事前に最適化
- カスタムイメージを使って起動時間をさらに短縮
```

### 長期予約契約（リザーブドインスタンス）を避けるべき理由

AWSやAzureのリザーブドインスタンス相当の契約は、ゲームインフラには不向き。

| リスク | 内容 |
|-------|------|
| **需要予測困難** | ゲームのヒット/失敗で必要リソースが大きく変わる |
| **ムーアの法則** | コンピューティング単価は継続的に低下するため、長期先払いは機会損失 |
| **技術革新リスク** | 魅力的な新サービス登場時に乗り換えが困難 |

**GCPの割引体系の優位性:** Sustained Use Discount（継続使用割引）は自動適用されるため、予約不要でコスト削減できる。

## 常時接続型アーキテクチャ（WebSocket / リアルタイム）

### ゲームサーバの基本構成

オンラインゲーム・コンソールゲームの常時接続型は、2種類のサーバで構成する。

```
【標準的な2層構成】
┌─────────────────────────────────┐
│  ロビー系サーバ（ログイン処理）   │
│  - HTTP/HTTPS                   │
│  - ユーザ認証・マッチング        │
│  - GCEインスタンスグループ       │
└─────────────────────────────────┘
              ↓
┌─────────────────────────────────┐
│  ゲーム系Socket サーバ（常時接続）│
│  - WebSocket / 独自プロトコル    │
│  - 固定グローバルIP              │
│  - Redis（Pub/Sub・セッション）   │
└─────────────────────────────────┘
```

### WebSocketアーキテクチャの実装

grasys社の実績ある構成例（Python Django + socket.io を使ったケース）：

```
【WebSocketサーバの構成要素】
- Nginx（リバースプロキシ）
- Python Django + Gunicorn + socket.io
- Supervisord（プロセス管理）
- MaxScale（DBプロキシ）
- HAProxy（内部LB）
- Redis（同時接続データ保持）
- Fluentd（ログ収集）
```

**重要設計判断: RedisをインスタンスA内部に配置する**
- 外部Redisを使うとレイテンシが悪化（ゲーム内部ネットワークでも影響が出る）
- WebSocketサーバ内部にRedisを配置してPub/Subで通信
- 冗長性は下がるが、ゲーム仕様上の接続時間要件（1〜2分）で許容

### WebSocketのヘルスリスト管理

ロードバランサがWebSocketサーバのHTTP振り分けを直接行えないため、別途ヘルス管理の仕組みが必要。

```
【Consul + Redisによるヘルスリスト実装】
1. Consul HTTP APIからHealth Endpointの各サービス状態を取得
2. 全サービスのステータスをRedisのリストに保存
3. アプリケーションがRedisのヘルスリストを参照
4. ユーザのマッチングと接続先サーバの割り当てを管理
```

### GCPで常時接続を実装する際の注意点

| 注意点 | 内容 | 対策 |
|-------|------|------|
| **ライブマイグレーション遅延** | GCPのライブマイグレーション発生時に遅延が生じる | ヘルスリスト管理をGCEインスタンスに乗せる際は遅延を考慮 |
| **LocalSSD性能への過信禁止** | オンプレのPCIe型SSDほどの性能は期待できない | 性能要件を再確認 |
| **外部Redis/Memcachedのレイテンシ** | 内部ネットワーク経由でも密なアクセスではレイテンシに影響 | セッションデータは可能な限りインスタンス内部に保持 |
| **1インスタンス集約禁止** | サービスに影響を与えるサーバを1インスタンスに集約しない | ロールごとにインスタンスを分散 |
| **自動起動設定必須** | hostError（HW障害）による突発的再起動が稀に発生 | rc3（runlevel 3）による自動起動を必ず設定 |

### Photonサーバ（Windows + GCE）

GCEはWindows Server（2008R2/2012R2）が利用可能。Exit Games社のPhotonと組み合わせた常時接続型ゲームの実績あり。

| 注意点 | 内容 |
|-------|------|
| Photonライセンス費用 | 別途ライセンス購入が必要 |
| Windowsコスト | 包括ライセンスの持ち込み不可 → 台数次第でコスト増 |
| 監視の課題 | Cloud MonitoringがPerformance Counter未対応 → 別途実装が必要 |
| カスタムイメージ非推奨 | gcesysprepが時間がかかる → 新インスタンスから自動構成を推奨 |

```powershell
# インスタンスメタデータからグローバルIPを取得（Photon config自動生成に使用）
Invoke-Request -Uri "http://metadata.google.internal/computeMetadata/v1/instance/network-interfaces/0/access-configs/0/external-ip"
```

## データベース構成

### DB選択の判断基準

| DB | 特性 | GCPでの実装 | ゲームユースケース |
|-----|------|------------|---------------|
| **Cloud SQL (MySQL)** | マネージドMySQL | Cloud SQL第2世代 | シンプルなRDBMS要件 |
| **Percona XtraDB Cluster + MaxScale** | MySQLクラスタ・高可用 | GCE上に自前構築 | 高性能・細かいチューニングが必要 |
| **Cloud Spanner** | グローバル分散RDBMS | フルマネージド | グローバルタイトルで強整合性が必要 |
| **Redis** | インメモリKVS | GCE上に自前構築 or Memorystore | セッション・ランキング・キャッシュ |
| **MongoDB** | ドキュメント指向NoSQL | GCE上に自前構築 | 柔軟なスキーマが必要な場合 |

### Cloud SQL vs Percona XtraDB Cluster

grasys社はCloud SQLを使わず、GCE上でPercona XtraDB Cluster + MaxScaleを構築している。

| 比較項目 | Cloud SQL | Percona XtraDB Cluster（GCE） |
|---------|----------|------------------------------|
| **構築コスト** | 低（マネージド） | 中（自前構築） |
| **チューニング** | 制限あり | 完全自由 |
| **可用性** | マネージドで自動フェイルオーバー | 自前でMaxScaleによるRead/Write分離 |
| **コスト** | 使用量に応じた課金 | GCEインスタンスコストのみ |
| **推奨ケース** | シンプルな要件 | 高性能・細かい制御が必要 |

**Percona XtraDB Cluster + MaxScaleの利点:**
- 構築が簡易で運用が安定
- ある程度安心できる性能
- MaxScaleによるRead/Write自動分離

### Percona XtraDB Clusterのバックアップ設計

```bash
#!/bin/bash
# wsrep_local_index=2 のノード（最後尾）でバックアップを取得

gsutil=/usr/local/bin/gsutil
wsrep_local_index=`mysql -s --skip-column-names \
  -e "show status like 'wsrep_local_index'" | awk '{print $2}'`

if [ $wsrep_local_index -eq 2 ]; then
  date=`date +%Y%m%d`
  backupdir=/usr/local/var/backup/mysql/${date}/${HOSTNAME}
  mkdir -p $backupdir
  mysqldump --single-transaction --routines \
    --default-character-set=binary \
    --set-gtid-purged=OFF --all-databases > $backupdir/db.dump
fi

# バックアップ階層（grasys推奨）:
# - 直近1日: GCEアタッチ Persistant Disk
# - 直近3日: GCS
# - それ以前: Nearline（コスト半額以下）
```

### Redisの冗長化設計（ゲームインフラ）

```
【Redis Sentinel構成】
- redis-sentinel: 別インスタンスで起動
- HAProxy（各サーバ内）: マスタ/スレーブ監視、更新系/参照系をポート番号で分離
- 更新系: HAProxy → マスタ
- 参照系: HAProxy → スレーブ

注意: レプリケーション遅延についてアプリ開発者と事前に合意が必要
→ スレーブ参照のみの要件がなければシンプルなマスタ切り替えで十分
```

## グローバル展開パターン

### マルチリージョン構成の選択肢

| パターン | 構成 | メリット | デメリット |
|---------|------|---------|---------|
| **グローバルLB** | HTTP(S) LBを前段に配置 | 地域別に最適なリージョンへ自動転送 | 80/443/8080のみ対応 |
| **リージョン独立型** | 各リージョンに独立したゲームサーバ | 設計がシンプル | クロスリージョンのユーザ間対戦が困難 |
| **プライベートネットワーク活用** | VPN不要でリージョン間接続 | 低レイテンシ・低コスト | GCP固有の強み（他社では追加設定が必要） |

### グローバルLBの設計（grasys社実績）

```
【HTTP(S) ロードバランシング】
- リージョン: all（グローバル）
- インスタンスをゾーン単位でインスタンスグループに集約
- バックエンドサービスとしてフォワーディングルールに登録
- US圏からのリクエスト → USリージョンのインスタンスへ転送
- URLマップで Host / PATH 単位での転送先振り分けが可能

【ネットワークLB（TCP/UDP）】
- リージョン固定（asia-east1でやるとasia-east1-a/b/cのみPoolに参加可能）
- 独自プロトコル・特殊ポートに対応
- UDPロードバランシングも可能（オンラインゲームのP2P通信等で活用）
```

### 静的ファイル配信（アセット配信）

GCSのエッジキャッシュを活用してゲームアセット（テクスチャ・音声等）を配信：

```bash
# GCSバケット作成とCDN設定
bucket=gs://[bucketname]
gsutil mb -c STANDARD -l ASIA $bucket
gsutil defacl set public-read $bucket
gsutil web set -m index.html -e error.html $bucket
gsutil -m rsync -r [SourcePath] $bucket  # -m で並列化して高速化

# アクセスURL
# HTTP:  http://[bucketname].storage.googleapis.com/
# HTTPS: https://storage.googleapis.com/[bucketname]/
# CNAME: c.storage.googleapis.comへCNAME設定
```

**注意:** 従量課金のため大量配信時はコスト試算が必要（定額CDNとは異なる）

## GCPと他社クラウドの比較（ゲームインフラ観点）

### 構築・運用視点での比較（grasys社 2年以上の運用経験）

| 評価項目 | GCPの優位点 |
|---------|-----------|
| **設定のシンプルさ** | gcloudコマンドがわかりやすく、混乱が少ない |
| **インスタンス起動速度** | SSH接続まで20秒以内 → 障害復旧が高速 |
| **ライブマイグレーション** | 定期的な再起動なし → メンテナンス工数がほぼゼロ |
| **HW障害時の自動再起動** | 落ちたら落ちっぱなしという状況がない |
| **ロードバランシング** | グローバルLBによりDNS依存なしのバランシングが可能 |
| **ストレージ** | GCS（エッジキャッシュ自動付き）+ Nearline（バックアップ用、半額以下） |
| **BigQueryの存在** | 「どのログを取るか」ではなく「すべてのログを取る時代」を実現 |
| **LocalSSD** | インフラエンジニアの最終手段としてPCIe型SSD代替が利用可能 |
| **Cloud Monitoring** | 統合監視・インシデント管理・多様な通知先 |
| **GKE（Kubernetes）** | Docker本番運用の複雑さをKubernetesで補完 |

### GCPの課題点

| 課題 | 内容 |
|-----|------|
| **日本語情報** | 書籍・Web上での日本語情報がAWSと比較して少ない |
| **ハードウェアベースのチューニング** | ディスクIOなど細かいチューニングはオンプレには及ばない |
| **マネージドサービスの種類** | AWSに比べてマネージドサービスのバリエーションが少ない |

### コスト比較まとめ

- AWSからGCPへの移行で、同一負荷想定でも全体コストは**20〜30%削減**が一般的
- ゲームインフラの需要は予測困難なため、3年リザーブドのような長期契約は不向き
- GCPの自動割引（Sustained Use Discount）は予約なしで自動適用される点が優位

## 監視・運用パターン

### ゲームサーバの推奨監視スタック

```
【grasys推奨の監視構成】
- Consul: プロジェクト内メタデータ管理・ヘルスチェック
- InfluxDB + Grafana: メトリクス保存・可視化
- Cloud Monitoring（Stackdriver）: GCP統合監視（サブシステムとして併用）
- Cloud Logging: Nginxエラーログ・MySQLスロークエリ・Syslogを一元収集
- Serf + Cinnamon: オーケストレーション・ロールとタスクの管理

【Cloud Monitoringで監視する対象】
- Nginx, Redis, MySQL（XtraDB）の各種メトリクス
  → collectd + stackdriver-agent + stackdriver-extractor で収集
```

### オペレーション構成

```
踏み台サーバ（Jump Host）の役割:
- SSHログインの入口（セキュリティ境界）
- 各種オペレーション（Serf/Cinnamon経由）
- Chef/Ansible等のデプロイ管理
  → Fluentdの設定更新など、多台数への一括配布に必須
```
