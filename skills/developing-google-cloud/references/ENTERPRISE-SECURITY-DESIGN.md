# エンタープライズセキュリティ設計（責任共有・Googleインフラ・プロダクト）

エンタープライズ企業が Google Cloud を安全に利用するためのセキュリティ設計の全体像を扱う。責任共有モデルから始まり、Google のインフラセキュリティ、従来型セキュリティ対策（ファイアウォール・WAF・暗号化）、クラウド時代のセキュリティ対策（ゼロトラスト・VPC SC・SCC）まで体系的に解説する。

## セキュリティ責任モデル

### 責任共有モデル

クラウド利用では、Google とユーザーが責任を分担する。クラウドサービスの形態によって責任範囲が異なる。

| 責任範囲 | オンプレミス | IaaS（Compute Engine等） | PaaS（App Engine等） | SaaS |
|---------|------------|------------------------|---------------------|------|
| **物理インフラ** | ユーザー | **Google** | **Google** | **Google** |
| **ハードウェア** | ユーザー | **Google** | **Google** | **Google** |
| **ネットワーク** | ユーザー | **Google** | **Google** | **Google** |
| **OS/ランタイム** | ユーザー | ユーザー | **Google** | **Google** |
| **データ・アプリ** | ユーザー | ユーザー | ユーザー | ユーザー |
| **IAM設定** | ユーザー | ユーザー | ユーザー | ユーザー |

**運命共有モデル（Google の新しいアプローチ）:** 設計・構築・デプロイ・実行の各フェーズを Google とユーザーが共同で実施し、ユーザーが包括的なリスク管理プログラムを構築できるよう支援する取り組み（2021年3月開始）。

## Google インフラストラクチャのセキュリティ

Google が管理する層でのセキュリティ対策の概要を理解することは、エンタープライズにおけるクラウド採用の判断材料になる。

### セキュリティカテゴリ一覧

| カテゴリ | 概要 |
|---------|------|
| **オペレーションセキュリティ** | 24時間365日体制での脅威検出・対応。厳しい基準でのソフトウェア開発・リリース |
| **インターネット通信** | パブリッククラウドとの通信は転送時に暗号化。多層防御による DoS 攻撃保護 |
| **ストレージサービス** | 保存データをデフォルトで暗号化。複数ロケーションへのデータ分散配置 |
| **ユーザー識別** | Googleスタッフのアクセスに複数要素認証。フィッシング耐性のあるセキュリティキー使用 |
| **サービスデプロイ** | ソフトウェアの信頼性を継続検証。マルチテナント設計を初期段階から組み込み |
| **ハードウェアインフラ** | 物理DC・専用設計サーバー・ネットワーク機器・独自設計セキュリティチップまですべて Google が制御 |

### コンプライアンス対応

Google Cloud は世界各地の基準に対応した認証・証明書・監査レポートを取得している。

| カテゴリ | 主な認証・法令 |
|---------|--------------|
| **国際認証** | ISO 27001（ISMS）、ISO 27017（クラウドセキュリティ）、ISO 27018（プライバシー）、SOC 2/3 |
| **グローバル法令** | GDPR（欧州）、HIPAA（米国医療）、PCI DSS（カード業界） |
| **日本固有** | マイナンバー法、CSVガイドライン、FISC、NISC、3省2ガイドライン、ISMAP |

## 従来型セキュリティ対策（ネットワーク境界型）

### ファイアウォール

Google Cloud のファイアウォールは **ステートフル** 型（応答パケットは暗黙的に許可）。

#### VPC ファイアウォールルール

| 特性 | 内容 |
|-----|------|
| **適用対象** | 特定プロジェクト・VPC ネットワーク内の VM |
| **方向** | 上り（Ingress）と下り（Egress）を個別に設定 |
| **ターゲット指定** | ネットワークタグ または サービスアカウントで対象 VM を指定 |
| **推奨** | ネットワークタグより**サービスアカウント**を使用（IAM による厳密な権限管理が可能） |

#### 階層型ファイアウォールポリシー

| 特性 | 内容 |
|-----|------|
| **適用スコープ** | 組織またはフォルダ単位 |
| **評価順序** | 上位レベルのルールから順番に評価。下位ルールで上書き不可 |
| **ユースケース** | 組織全体に一貫したセキュリティポリシーを適用（特定ポート以外の外部アクセスを全 VM で拒否等） |

#### ファイアウォール管理ツール

| ツール | 機能 |
|-------|------|
| **ファイアウォールインサイト** | 利用されていないルールの検出、設定ミスの特定 |
| **ファイアウォールルールロギング** | ルール適用状況を Cloud Logging に記録。BigQuery 連携で監査対応 |

### WAF（Cloud Armor）

Cloud Armor は Google Cloud Load Balancing の機能として提供される WAF。オンプレミスや他クラウドのアプリにも、LB を前段に置くことで適用可能。

| 機能 | 内容 |
|-----|------|
| **IP・位置情報制御** | IPv4/IPv6/CIDR に基づくアクセス制限 |
| **WAFルール** | 基本モード（IP指定）と詳細モード（ルール言語）で設定 |
| **OWASP対策** | OWASP Top 10 リスクに対応した事前設定ルール（XSS、SQLi、LFI、RFI、RCE等）|
| **名前付き IP リスト** | CDN プロバイダ等のサードパーティ IP リストを参照 |
| **DDoS 保護** | Managed Protection（スタンダード/プラスティア）による保護 |
| **プレビューモード** | 本番適用前にルールの影響範囲を確認できる |

### パケット解析（パケットミラーリング）

| 特性 | 内容 |
|-----|------|
| **仕組み** | VPC ネットワーク内の特定インスタンスのトラフィックをクローンして転送 |
| **対象指定** | サブネット・ネットワークタグ・インスタンス名で絞り込み |
| **注意点** | 対象インスタンスの帯域幅を消費する（全ミラーリング時は下り帯域が 3 倍になる）|
| **統合** | Palo Alto VM-Series 等のサードパーティセキュリティツールと組み合わせ可能 |

### 暗号化戦略

Google Cloud では **エンベロープ暗号化**（DEK をさらに KEK で暗号化）を採用。

#### 暗号化のポートフォリオ

| オプション | 説明 | コントロール範囲 | 管理負荷 |
|---------|------|--------------|---------|
| **Google管理キー（デフォルト）** | 完全に Google が管理 | 最小 | 最小 |
| **CMEK（Cloud KMS生成キー）** | KEK を Cloud KMS で生成・管理 | 中 | 中 |
| **CMEK（インポートキー）** | ユーザーが作成した KEK を Cloud KMS にインポート | 中 | 中〜高 |
| **外部管理キー（Cloud EKM）** | KEK を Cloud 外部の KMS で管理 | 最大 | 最大 |
| **CSEK** | ユーザーが KEK を直接 Cloud Storage/GCE に提供 | 最大 | 最大 |

#### Cloud KMS の設定ポイント

| 設定項目 | 選択肢 | 用途 |
|---------|-------|------|
| **鍵の種類** | 生成した鍵 / インポートされた鍵 / 外部管理 | 企業のキー管理ポリシーに応じて選択 |
| **保護レベル** | ソフトウェア / ハードウェア（Cloud HSM: FIPS 140-2 Lv3） | 高セキュリティ要件には HSM を選択 |
| **目的** | 暗号化・復号化用 / 署名用 | 用途に応じて選択 |

**Confidential Computing:** AMD EPYC CPU の SEV 技術を活用し、データ処理中（メモリ上）のデータも暗号化。アプリコード変更不要。

## クラウド時代のセキュリティ対策（ゼロトラスト型）

従来のネットワーク境界型では不十分な理由:
- 設定ミスによる意図しない機密データ漏洩
- 正規ユーザーによる悪意ある内部からのデータコピー
- 漏洩した ID/SA でインターネット経由のアクセス

### VPC Service Controls（VPC SC）

機密データに**サービス境界**を設定し、データ漏洩リスクを軽減。

| 機能 | 内容 |
|-----|------|
| **サービス境界** | 境界内の API サービス間通信を許可。境界をまたぐ通信はデフォルト拒否 |
| **アクセスレベル** | 送信元 IP 等の属性に基づき、境界外からの一部アクセスを許可する条件定義 |
| **ドライランモード** | 実際のトラフィックをブロックせずに影響範囲を確認してから本番適用 |

**VPC SC の適用範囲と注意点:**

| 注意点 | 内容 |
|-------|------|
| 対象は API サービス間通信 | VM ↔ VM 間通信は別途ファイアウォールで制御が必要 |
| Cloud Shell は非対応 | Cloud Shell からは VPC SC 保護リソースへのアクセス不可 |
| プロダクトのサポート確認 | 全プロダクトが対応しているわけではない。事前確認必須 |
| IAM Conditions との使い分け | VPC SC 適用前に IAM Conditions での対応可否を検討 |

### Security Command Center（SCC）

組織全体のセキュリティ・コンプライアンス情報を一元管理するプラットフォーム。

| ティア | 特徴 |
|-------|------|
| **スタンダード** | 基本機能（アセット検出、Security Health Analytics 手動スキャン等） |
| **プレミアム** | 全機能（自動スキャン、ETD、Web Security Scanner マネージドスキャン等） |

#### SCC の主要機能

| 機能 | 説明 | 検出例 |
|-----|------|--------|
| **Security Health Analytics** | 設定ミス・コンプライアンス違反をリアルタイム検出 | 公開バケット、過剰なFWルール、GKEのモニタリング無効 |
| **Anomaly Detection** | 外部動作シグナルによる認証情報漏洩・VM不正利用の検出 | 自動有効化 |
| **Cloud DLP 統合** | 機密データ（クレジットカード番号・電話番号等）の検出・分類 | 個人情報の特定 |
| **Event Threat Detection（ETD）** | VPC フローログ等を監視してほぼリアルタイムで脅威を特定 | マルウェア通信、認証情報窃取 |
| **Web Security Scanner** | GKE/GCE/GAE 上のアプリを自動発見してスキャン | XSS 等の OWASP Top 10 |

**SCC 自動対応の実装例:**
```
SCC 検出 → Cloud Logging エクスポート → Pub/Sub → Cloud Functions → 自動修復
（例: 公開バケットを自動でプライベートに戻す）
```

### BeyondCorp Enterprise（BCE）/ ゼロトラストアクセス

VPN に頼らない、ユーザー・デバイス・コンテキストに基づいたアクセス制御。

| VPN の課題 | BCE での解決策 |
|----------|--------------|
| アクセス元デバイスの多様化への対応困難 | デバイス状態（OS バージョン、セキュリティパッチ等）をアクセス条件に組み込む |
| VPN の逼迫・パフォーマンス劣化 | VPN 不要のゼロトラストアクセスへ移行 |
| ユーザー種別ごとのアクセス粒度制御 | コンテキストに応じたアプリレベルのアクセス制御 |

**BCE の主要コンポーネント:**

| コンポーネント | 役割 |
|-------------|------|
| **Cloud Identity** | ユーザーアイデンティティの管理・認証（外部 ID も利用可） |
| **Access Context Manager** | デバイス状態・IP・地理情報等のアクセスレベルを定義 |
| **Identity-Aware Proxy（IAP）** | アプリへのアクセスをユーザー ID とアクセスレベルで制御 |
| **Chrome Enterprise** | マネージドブラウザによるデバイスシグナルの収集 |

### reCAPTCHA Enterprise

Web 上の不正行為対策として 10 年以上の実績を持つ reCAPTCHA をエンタープライズ向けに再設計。

| 機能 | 内容 |
|-----|------|
| **大規模アクセス対応** | 月 100 万回制限なし（無料版 reCAPTCHA との差異） |
| **細粒度リスクスコア** | 0〜1 のスコアでリスクを判定。Webサイト特性に合わせたカスタマイズ可能 |
| **ネイティブアプリ対応** | iOS/Android アプリへの組み込みも可能 |
| **SLA/SLO 付き** | エンタープライズ向けの Google Cloud SLA 適用 |

## セキュリティ設計の全体構成

| 脅威レイヤ | 対応プロダクト | 設計ポイント |
|----------|-------------|------------|
| **ネットワーク境界** | VPC FW、階層型 FW ポリシー | サービスアカウントをターゲットに使用 |
| **アプリケーション層** | Cloud Armor（WAF） | OWASP 事前設定ルール + プレビューモードで検証 |
| **データ漏洩対策** | VPC Service Controls | ドライランモードで影響確認後に本番適用 |
| **暗号化** | Cloud KMS（CMEK）、Cloud HSM | 規制要件に応じて保護レベル（SW/HW）を選択 |
| **脅威検出・統合** | SCC、ETD、Cloud DLP | SCC を有効化して自動修復フローを構築 |
| **ゼロトラストアクセス** | BeyondCorp Enterprise、IAP | VPN 依存を段階的に削減 |
| **監査ログ** | Cloud Logging、Audit Logs | 管理アクティビティ・データアクセスログを有効化 |

## セキュリティ設計チェックリスト

| カテゴリ | 確認項目 |
|---------|---------|
| **責任範囲** | IaaS/PaaS/SaaS の別で自社の責任範囲を明確化しているか |
| **ファイアウォール** | 階層型ポリシーで組織全体のベースラインルールを定義しているか |
| **ファイアウォール** | サービスアカウントをターゲットに使用し、ネットワークタグ依存を排除しているか |
| **WAF** | Cloud Armor の OWASP 事前設定ルールを適用しているか |
| **暗号化** | BigQuery・Cloud Storage 等の機密データに CMEK を設定しているか |
| **データ漏洩防止** | 機密データを扱うプロジェクトに VPC SC を適用しているか |
| **脅威検出** | SCC を有効化し、検出結果の自動対応フローを設定しているか |
| **監査ログ** | 管理アクティビティログ・データアクセスログを有効化しているか |
| **ゼロトラスト** | リモートアクセスに BeyondCorp / IAP の採用を検討しているか |
| **コンプライアンス** | 業界規制（PCI DSS、FISC 等）に対応した認証・監査資料を確認しているか |
