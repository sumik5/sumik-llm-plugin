# BI・データ可視化リファレンス

## 概要

Google Cloud が提供するBI・データ可視化プロダクト群と、それらを支えるBigQuery機能（BI Engine・マテリアライズドビュー）を解説する。データ分析の成熟度に応じたツール選択から、エンタープライズ要件を満たすLookerのアーキテクチャまでを網羅する。

---

## BIとBIツールに求められる要件

### BIとは

ビジネスインテリジェンス（BI）とは、企業活動における意思決定をデータに基づいて行うこと。経営層から個人の行動計画まですべてのレベルにおいて、経験と勘に依存しない客観的な意思決定を支援する。

### BIツールに求められる5つの要件

| 要件 | 概要 |
|------|------|
| **大量データの処理** | モダンなDWH（BigQuery等）と接続できることが必須 |
| **容易な可視化・分析操作** | フィルタリング機能など使いやすいインターフェース |
| **ダッシュボードレポーティング** | 組織での共有・動的な分析機能 |
| **データ管理のセキュリティ** | アクセス制御の詳細かつ効率的な管理機能 |
| **データガバナンス** | 組織内での指標定義の一元管理・透明化 |

### Google Cloudで利用できるおもなBIツール

- **コネクテッドシート** — スプレッドシートUIで大量データのアドホック分析
- **Looker Studio / Looker Studio Pro** — 本格的なダッシュボード・レポーティング（無料〜）
- **Looker** — セマンティックモデリングレイヤーによるエンタープライズBI

サードパーティBIツール（Tableau・Power BI等）もBigQueryと組み合わせて利用可能。

## コネクテッドシート

### コンセプト

Googleスプレッドシートで大量データを扱える機能。データ処理をBigQueryに委ねることで、Excelや通常スプレッドシートの100万行程度の上限を解消する。

- データはスプレッドシートに抽出せず BigQuery 上のデータを直接参照
- DWHの最新データにいつでも更新可能
- BIツールの学習コストなく大量データを分析できるため、データ活用推進の入口になる
- 2024年10月時点では、個人利用の Gmail アカウントを含む全エディションで利用可能

### 主な機能

**ピボットテーブル：** ピボットテーブルエディタで値・行・列を設定して [適用] するとBigQueryにクエリが発行される。通常はSQLスキルが必要な多軸クロス集計を、データ専門外のメンバーでも実施可能。

**グラフ作成：** ドーナツグラフ等の任意のグラフ種別を作成。複数グラフをシートに配置してダッシュボード的に運用できる。グラフ共有はGoogleスプレッドシートの共有機能を活用。

**制限事項：** 複数グラフへの共通フィルタやレポート送信機能はないため、閲覧参照用途に留まる。動的な分析やアクション連携が必要な場合は Looker Studio 以上を検討。

### データ更新

| 更新方法 | 操作 |
|---------|------|
| 手動更新 | グラフ・テーブルをマウスオーバー → [更新] ボタン |
| 定期更新 | [更新オプション] → [更新スケジュール] で間隔設定 |

更新のたびにBigQueryのクエリ料またはスロットが消費されるため、更新頻度はコストと要求鮮度のバランスで決定する。

## Looker Studio / Looker Studio Pro

### コンセプト

ブラウザベースかつサーバーレスなデータ分析レポートツール。以前は「データポータル」という名称で提供。2022年10月にLookerファミリーとして統合。

- 無料（Looker Studio）と有償（Looker Studio Pro）の2ティア
- BigQueryを含むさまざまなデータソースへ接続可能

### Looker Studio Pro の追加機能

| 機能 | 概要 |
|------|------|
| 組織所属のコンテンツ | レポートが個人ではなく組織に属し、退職者対応が不要。IAMで権限管理 |
| チームワークスペース | チームで共有できるワークスペース・アクセス管理の一元化 |
| 高度なレポート配信 | 担当者ごとにフィルタされたレポートの個別配信（Google Chatへの配信・条件通知も可） |
| 個人レポートリンク | 元のレポートを変更させずに他ユーザーが自由にデータ探索 |
| Cloudカスタマーケアサポート | 本番環境でのテクニカルサポート |

### グラフとダッシュボード機能

**ディメンションと指標：**
- **ディメンション** — 分析の軸（日時型・文字列型等）。選択した値ごとにデータを分けて可視化
- **指標** — グラフで見る数値。レコード件数、金額の合計・平均など

**特徴的なビジュアライゼーション：** Googleマップグラフで緯度経度を用いた地理的な可視化が可能。

**インタラクティブフィルタ（クロスフィルタリング）：** グラフの [Interactions] → [フィルタを適用] を有効にすると、そのグラフの選択が同一レポート内の他グラフに連動する。

**ドリルダウン：** 階層的なディメンション（月別 → 日別、カテゴリ → 品目等）を動的に深掘り・ドリルアップできる。

**共有：** ユーザー/グループへのメール指定共有、組織内リンク共有、組織外公開の設定が可能。グループを活用した権限管理が推奨。

## Looker

### コンセプト

Google Cloud が提供するエンタープライズデータプラットフォーム。最大の特徴はビジネス指標の定義・計算ロジックを**中央集権的にキュレート・管理**できるセマンティックモデリングレイヤー（LookML）の提供。

#### セマンティックモデリングレイヤーとは

データウェアハウスのスキーマと分析者が必要とするディメンション・指標の間のギャップを埋める層。

- 注文テーブル × 顧客セグメントテーブルの結合
- UTCからJSTへの変換
- キャンセル注文の除外

これらのビジネスロジックを事前定義することで、誰もが統一された方法でデータを分析できる。セルフBIツールの普及で個人依存になりがちな指標定義・計算方法を、コードで一元管理することがガバナンスの核心。

### Lookerのその他の特徴

| 特徴 | 説明 |
|------|------|
| **マルチクラウド** | BigQuery以外のDWH・クラウドDB（Redshift・Databricks等）にも接続可能 |
| **埋め込み** | ポータルサイトや業務ツールにダッシュボードをiFrameで埋め込み可能 |
| **ヘッドレスBI** | Looker Studio・Tableau・Power BIからLookerのセマンティックモデルを利用可能 |
| **他サービス連携** | 分析結果からマーケティングツール・MLへのアクションをスムーズに実行 |

### Looker (Google Cloud コア) エディション

2023年3月から、Google Cloudコンソールから直接Lookerインスタンスを作成できる Looker (Google Cloud コア) が利用可能に。

| エディション | 主な特徴 |
|------------|---------|
| Standard | 基本機能 |
| Enterprise | VPC Service Controls・プライベートIP・CMEK対応 |
| 埋め込みエディション | 埋め込み特化の機能セット |

### LookML によるモデリング

LookMLで定義するファイル：

**モデルファイル（`.model.lkml`）：**
- 接続するデータベースのコネクション名
- モデルに含めるビューファイル（パス指定）
- **Explore** — 分析者がアクセスできる分析単位（単一ビューまたは複数ビューの結合）

**ビューファイル（`.view.lkml`）：**
- ビュー名と参照するBigQueryテーブル（`<データセットID>.<テーブルID>`形式）
- **ディメンション** — 分析軸。型（string/yesno/number/date等）とSQL定義。`sql`にビジネスロジック（条件式・結合等）を記述可能
- **メジャー** — 分析指標。count/sum/average等のタイプ指定。`drill_fields`でドリルダウン先を指定

### Git によるバージョン管理

開発モードとプロダクション環境が分離されており、安全な変更管理が可能：

1. **開発モード**でLookMLを編集（他ユーザーへの影響なし）
2. [Commit Changes & Push] で個人ブランチにコミット・プッシュ
3. [Deploy to Production] でmainブランチにマージ・本番反映
4. プルリクエストを使った承認プロセスも構築可能

「指標の算出ロジックが気づいたら変わっていた」という事態を防止し、変更追跡とデータガバナンスを実現。

### Explore によるデータ分析

LookMLで定義したディメンション・メジャーを使って、グラフや可視化を実行するインターフェース。

設定項目：
- **ディメンション** — 分析の軸
- **メジャー** — 分析対象の数値
- **フィルタ** — 特定条件でのデータ絞り込み
- **ピボット** — ディメンションのピボット集計

Exploreで作成・保存したグラフは **Look** と呼ばれ、ダッシュボードの構成要素として利用可能。

### ダッシュボードの構築と配信

**フォルダ管理：**
- **共有フォルダ** — 組織で共有するダッシュボード・Lookを格納
- **ユーザー用フォルダ** — 個人専用・作成中のダッシュボード
- **LookMLダッシュボード用フォルダ** — LookMLで管理されたダッシュボード

**スケジュール配信：** PDFや画像形式でダッシュボードを定期配信。配信先：メール・Slack・Teams・Google Cloud Storage・Google Drive・Amazon S3・SFTP・Dropbox

LookMLで記述・管理されたダッシュボードはGitで変更管理可能。

### アクセス制御

3層構成でアクセスを管理：

**1. データアクセス：** LookMLモデル単位。モデルセットをロールに割り当て。**ユーザー属性**（メールアドレス・タイムゾーン・任意属性）を用いた行レベルアクセス制御も可能。

**2. 機能アクセス（権限セット）：** `access_data`（データアクセス）・`download_with_limit`（ダウンロード）・`develop`（LookML編集）・`deploy`（本番反映）等の権限セットをロールに割り当て。

デフォルトの権限セット：管理者・開発者・ユーザー・ビュア・LookMLダッシュボードユーザー

**3. コンテンツアクセス：** フォルダ単位でView権限（参照）とManage Access/Edit権限（編集・管理）をユーザー/グループに付与。グループ単位の運用を推奨。

ロール = モデルセット（データアクセス）+ 権限セット（機能アクセス）として構成。

### アクションへの連携

**埋め込み（iFrame）：**

| 方式 | アクセス制御 |
|------|------------|
| プライベート埋め込み | Lookerでのログインを前提 |
| 署名付き埋め込み | ユーザーサイト独自の認証機構（SSO）を利用 |
| 公開埋め込み | 認証なし（Lookのみ対応） |

**アクション（Looker Action Hub）：**
分析結果をサードパーティツールと連携する機能。

- **Action Hub提供済みサービス例：** Google Cloud Storage・Google Drive・Slack・Teams・SendGrid
- **カスタム開発：** 対象ツールがAPIを持っていれば独自アクションを開発可能

ユースケース：ロイヤリティの高いユーザーをセグメント化してマーケティングツールへ自動インポート、施策評価レポートをSlackに定期送信など。

**アラート：** ダッシュボードタイルのメジャーが設定条件を満たした際にメール等で通知（単一値・時系列・カテゴリの3種類）。

## BigQuery BI Engine

### 概要

高速なメモリベースの分析機能。BigQueryにあるデータを通常のクエリ実行よりも高速に返し、BIツールでの高速な分析を実現する。

**解決する課題：**
- データマート事前生成 → パイプライン運用コスト増・分析柔軟性の低下
- 汎用データセット直接利用 → パフォーマンス問題
- **BI Engine** がこの双方を解決

### 設定と確認

- BigQueryコンソールから有効化し、**予約メモリ容量**を指定
- クエリ実行後のジョブ情報でBI Engineの利用状況を確認
  - `FULL_INPUT` — すべての入力ステージがBI Engineで高速化済み
  - `BI_ENGINE_DISABLED`（理由付き）— BI Engineが適用されなかった場合

**適用されない主なケース（`INPUT_TOO_LARGE`等）：**
- 予約メモリが不足している場合
- BI Engineでサポートされていない関数・クエリを利用している場合
- 対処：パーティション分割・クラスタリングでスキャンデータ量を削減

### コスト考慮

- 予約メモリ量に応じてコストが発生
- BigQueryエディションのコミットメント契約ではスロット数に応じて一定量が無料
- BI Engineのメモリ利用量はCloud Monitoringで監視可能
- 同時利用数の多いダッシュボードでは BigQueryのスロット消費を削減し、総コスト削減に貢献する場合あり

## Gemini in Looker

Google Cloud Next '24でアナウンスされた生成AI活用によるLookerの新機能群：

| 機能 | 概要 |
|------|------|

これらにより、BIツールを使うユーザーの範囲が拡大し、企業内のデータ分析民主化が加速する。

## BIツール選定ガイド

### 成熟度に応じた選択

```
データ分析の成熟度
セルフアドホック分析 ─── ダッシュボード分析 ─── エンタープライズBI
    ↓                          ↓                        ↓
コネクテッドシート         Looker Studio /           Looker
                         Looker Studio Pro
                                          ↑ ガバナンスの強化 ↑
                                   セマンティックモデリングレイヤー
```

### 各ツールの選択基準

**コネクテッドシート：**
- SQLスキルがない営業・経理などが個人・チーム単位でセルフ分析
- データアナリストへの都度依頼を削減したい場合
- 小規模組織での迅速なデータ活用の第一歩として

**Looker Studio：**
- チームや部署単位でのダッシュボード共有・運用
- 無料から始めて、エンタープライズ要件が出てきたらProへ移行可能
- ガバナンスをそれほど厳しく求めない状況での積極的選択が可能
- **注意：** 組織規模が大きくなると指標の不整合・ダッシュボード乱立等のガバナンス課題が発生

**Looker：**
- 組織全体でのデータガバナンスとアクション連携が必要な段階
- 指標定義の不一致や、可視化から一歩進んだアクション展開を組織的に推進したい場合
- 他BIツール導入済みでも、ヘッドレスBIとして既存ツールを継続利用しながらガバナンス強化が可能
- コストを考慮し、PoCでの効果検証を推奨

## Apigee API管理概要

### コンセプト

APIマネジメントプラットフォームサービス。2016年にGoogleが買収、2021年にGoogle Cloudネイティブなアーキテクチャーに刷新。APIの構築・運用にとどまらず、**APIをビジネス活用するうえでの課題解決を総合サポート**するプラットフォーム。

主要機能：
- **API Proxy（ゲートウェイ）** — トラフィック管理・認証/認可・フォーマット変換・キャッシュ
- **API Monitoring/Analytics** — 使用状況の監視・分析・レポーティング
- **Developer Portal** — API利用者（開発者コミュニティ）向けポータル
- **収益化機能** — APIコールへの課金管理

### 主なユースケース

1. **APIエコシステムの構築** — 自社サービス・データをAPIとして外部提供し課金
2. **社内APIプラットフォームの整備** — バックエンドサービス群のAPI管理レイヤーを一本化
3. **データアクセスのAPI化** — データ基盤のデータ利用を円滑に広めるための一元管理

### 提供形態

| 形態 | 概要 |
|------|------|
| **Apigee X** | Google Cloud ネイティブマネージドサービス。Cloud CDN・Cloud Armorと直接連携可能。Google管理VPCにプロビジョニングし、ユーザーVPCからVPCピアリングでアクセス |
| **Apigee hybrid** | Anthosを使ったオンプレミス/マルチクラウド環境での運用。ユーザー自身でRuntimeを管理 |

### サブスクリプション

| | Standard | Enterprise | Enterprise Plus |
|---|---|---|---|
| APIコール数（年間） | 1.8億回 | 12億回 | 120億回 |
| データ転送量（年間） | 1TiB | 10TiB | 150TiB |
| アナリティクス保持期間 | 30日 | 3カ月 | 14カ月 |
| Runtime SLA | 99% | 99.9%〜99.99% | — |
| Developer Portal | なし | 含まれる | 含まれる |

本番環境での使用にはSLAの観点からEnterprise以上が推奨。

**Cloud Endpoint / Cloud API Gatewayとの違い：** これらはGoogle Cloud上のバックエンドアプリケーションをAPI化する機能に特化したサービス。APIライフサイクル全般・収益化・Developer PortalまでカバーするApigeeとはカバー範囲が異なるため、選定時に注意が必要。
