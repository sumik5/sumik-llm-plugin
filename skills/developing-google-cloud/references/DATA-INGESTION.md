# DATA-INGESTION — BigQueryへのデータ集約

## 概要

BigQueryを中心としたデータ基盤では、様々なデータソースからBigQueryへ効率的にデータを集約することが重要。BigQueryの強みを最大限発揮するため、目的に応じた取り込み方法の選択が必要。

### BigQueryへのデータ集約のメリット

- 大規模データに対する高速クエリ実行
- 統計関数・ウィンドウ関数など分析用関数の充実
- UDF（ユーザー定義関数）による強力な分散ELT処理
- BigQuery ML・GIS・生成AIを用いた発展的な分析
- Looker等BIツールやコネクテッドシートとのシームレスな連携

---

## データ取り込み方法の分類（8種類）

| 方法 | 概要 | 主なユースケース |
|------|------|----------------|
| ① ローカルファイル バルクロード | ローカルPCからデータをバルク取り込み | 小規模・スポットのデータ取り込み |
| ② Cloud Storage バルクロード | Cloud StorageからBigQueryへバルク取り込み | 大量ファイルの一括取り込み |
| ③ 外部データソース（フェデレーション） | 外部データソースへのクエリ結果をBigQueryテーブルとして取り込み | 外部DBとの連携クエリ |
| ④ Storage Write API | ストリーミング/バッチでデータ取り込み | ログ等のリアルタイム投入 |
| ⑤ BigQuery DTS | 対応サービスからスケジュール転送 | 定期的なサービス間データ転送 |
| ⑥ Datastream for BigQuery | CDC方式によるリアルタイムレプリケーション | DBの変更をBigQueryにリアルタイム同期 |
| ⑦ データパイプライン構築 | Dataflow・Dataproc・Data Fusionを活用したカスタムパイプライン | 複雑なETL/ELT処理 |
| ⑧ サービス間連携（エクスポート） | Firebase・GA4等からの自動データエクスポート | サービス内データのBigQuery活用 |

## BigQuery Data Transfer Service (BigQuery DTS)

### 概要

- 対応データソースから設定されたスケジュールでBigQueryへデータ転送するマネージドサービス
- GUI上のみで設定完結。転送コードの記述不要
- データ加工（前処理）はできない。フォーマット変換・フィルタリングはデータソース側で実施するか、取り込み後にBigQuery側で実施
- BigQueryと同等のSLA（月間稼働率99.99%以上）と24時間以内のデータ配信SLO

### 対応データソース

- **Googleサービス**: Google Ads、YouTube等のGoogleが提供するサービス
- **Cloud Storage**
- **Amazon S3**
- **Azure Blob Storage**
- **Amazon Redshift**
- **Teradata**

### サードパーティコネクタ

- Cloud Marketplace（Marketplace）上でパートナー企業提供のコネクタを確認・登録可能
- 登録後、BigQuery DTSの設定メニューから当該コネクタが利用可能になる
- **注意**: サードパーティコネクタには追加設定や別途コストが発生する場合がある

### Amazon S3からのデータ転送設定

転送設定に必要な情報:
- Amazon S3上のファイルのURI
- IAMのアクセスキーID
- シークレットアクセスキー

主な設定項目:
- `[データセット]`: 転送先BigQueryデータセットID
- `[Destination table]`: 転送先テーブル名
- `[Amazon S3 URI]`: 転送元ファイルのS3パス
- `[File format]`: CSV, JSON, Avro等から選択
- `[Header rows to skip]`: CSVにヘッダー行がある場合は1を設定

### 通知設定

- **メール通知**: 転送失敗時のみ通知（転送管理者向け）
- **Pub/Sub通知**: 転送成功・失敗の両方を通知
  - Pub/SubメッセージをトリガにCloud Functions/Cloud Runで後続処理をキックするパイプライン構築が可能

### 定期実行とランタイムパラメータ

- `[即時開始]`を選んだ場合でも、実際にはその後毎日同時刻に実行するようスケジューリングされる
- **注意**: 実験目的で利用した場合、転送完了後は`[転送の無効化]`または`[削除]`を忘れずに実施
- ランタイムパラメータを利用すれば、実行時の日付に基づいたファイル名パターンを動的に指定した定期転送が可能

### コストと注意事項

**ロケーション制限**:
- BigQueryデータセットのロケーションは一度設定すると変更不可（変更にはデータセットコピーが必要）
- ロケーションをまたいだデータセット間のJOINは不可（クエリジョブはデータセットのロケーションで実行されるため）
- BigQuery DTSのCloud Storageコネクタはソース・宛先のロケーションが同一である必要がある

**コスト**:
- Google Cloud上での転送コストは一部のデータソース以外は無料
- **注意**: 転送元プラットフォームのエクスポートコストは別途発生（例: Amazon S3からのインターネット転送コスト）

## Datastream for BigQuery（CDC方式データレプリケーション）

### CDCとバッチ方式の比較

**バッチ方式の課題**:
- SELECT文によるデータベースの断面取得が基本
- 更新頻度が高く一部のみ更新されるテーブルをバッチ転送すると、ソースDBへの読み取り負荷が増大
- マスター情報などの最新データを常時分析に利用したい場合には不適

**CDC（Change Data Capture）**:
- データソースの変更ログ（MySQLのbinlog、PostgreSQLのWAL等）を読み取り変更をキャプチャ
- SELECT文を発行しないため、ソースデータベースへの負荷が低い
- 変更差分をリアルタイムに検知・転送可能

### Datastream for BigQueryの特徴

- CDCを用いてソースDBの情報をBigQueryにリアルタイム同期
- キャプチャした変更ログをBigQuery Storage APIのUPSERT/DELETEオペレーションに変換
- わずか数ステップで設定完了。一度設定すれば継続的な同期が行われる
- オンプレミスや他クラウドのDBへの接続対応（ハイブリッドクラウド構成でも利用可能）

### 対応データソース

- MySQL
- PostgreSQL（AlloyDB含む）
- Oracle
- SQL Server

### PostgreSQLの事前準備

```sql
-- パブリケーションの作成
CREATE PUBLICATION publication_all_tables FOR ALL TABLES;

-- レプリケーションスロットの作成
SELECT PG_CREATE_LOGICAL_REPLICATION_SLOT('rep_slot', 'pgoutput');
```

Cloud SQL for PostgreSQLの場合: `[cloudsql.logical_decoding]`フラグを`on`にした後、上記クエリを実行。

### 設定手順

**①ソース接続プロファイルの作成**:
- Datastreamのトップ画面 → `[接続プロファイル]` → `[プロファイルの作成]` → `[PostgreSQL]`
- 接続方法の選択: `IP許可リスト`、`フォワードSSHトンネル`、`VPCピアリング`から選択
- `[接続プロファイルのテスト]`で接続確認

**②BigQuery宛先接続プロファイルの作成**:
- `[プロファイルの作成]` → `[BigQuery]`
- リージョンを指定して作成

**③ストリームの作成**:
- ストリーム名、ソースタイプ（PostgreSQL）、宛先（BigQuery）を設定
- ソースの構成: パブリケーション名・レプリケーションスロット名・対象テーブル/スキーマを指定
- 宛先の構成: BigQueryリージョン・書き込みモードを設定

### 書き込みモード

| モード | 動作 | 用途 |
|--------|------|------|
| **結合** | ソースと同じデータを持つテーブルをBigQueryに維持。変更履歴は保存されない | データレプリケーション（最新データが必要な場合） |
| **追加のみ** | データソースの変更履歴がBigQueryに追記される | 変更履歴の保持（UPDATE/DELETEの履歴を残す場合） |

### ストリームの開始

- 初回起動時は既存データの取得（バックフィル）が行われる
- バックフィル中はソースDBに負荷がかかる場合があるため注意
- 開始後しばらくするとBigQueryにソースDBと同じスキーマが作成されデータが連携される

## BigQueryへのデータパイプライン構築

### ELTアプローチの基本考え方

クラウド環境のスケーラビリティにより「とにかくBigQueryにデータを集めて、加工はあとで考える」ELTアプローチが有効。BigQueryのストレージは安価なためデータを積極的に集約することが推奨される。

### 簡易パイプラインとその課題

**簡易構成の例**: BigQuery DTS（Extract + Load） → BigQueryスケジュールクエリ（Transform）

**課題**: データ取り込みと加工の依存関係が管理できない

| パターン | 発生状況 | リスク |
|----------|---------|--------|
| パターン1 | データ取り込み失敗時、加工クエリは正常実行されるが結果が意図と異なる | 発見が遅れる（クエリエラーにならないため）|
| パターン2 | 日付指定クエリで対象テーブル・パーティションが存在しないためエラー | クエリ自体がエラーを返す |

**対策**: BigQuery DTSはPub/Subに転送完了・失敗を通知できる機能を持つ。この通知をトリガにCloud Functions/Cloud Runで後続処理を実行し、依存関係を管理するパイプラインを構築可能。

### データパイプライン構築のソリューション

| ソリューション | 特徴 | 使い分け |
|--------------|------|---------|
| **Dataform** | SQLXでパイプライン全体を表現。BigQuery上で完結。運用負荷が最も低い | BQ中心のELT処理全般 |
| **Dataflow** | Apache Beamのマネージド実行環境。コードでパイプラインを表現。ジョブ管理機能は別途必要 | 複雑な変換処理・ストリーミング処理 |
| **Dataflow + Cloud Scheduler** | シンプルなパイプラインの定期実行 | 単純な依存関係 |
| **Dataflow + Cloud Composer** | 複数ジョブの依存関係管理が必要な場合 | 複雑な依存関係 |
| **Cloud Data Fusion** | GUI操作によるETLパイプライン構築 | ノーコード・ローコードETL |

## サービス間連携によるBigQueryへのデータ連携

### Googleアナリティクス4（GA4）からBigQueryへのエクスポート

**設定手順**:
1. エクスポート先BigQueryプロジェクトを準備
2. GA4の管理画面から当該Google CloudプロジェクトをリンクするだけでOK

**機能**:
- 日次エクスポート: 前日分のデータが毎日エクスポート
- **ストリーミングエクスポート**: 当日分のデータを数分遅れで取得可能（別途設定が必要）

**コスト**:
- GA4→BQへのエクスポート自体は無料
- エクスポート後のBigQueryストレージ料金・クエリ料金は通常通り発生
- ストリーミングエクスポートはBigQueryのストリーミング挿入料金が追加発生

**データ確認**:
- BigQueryの一般公開データセット `bigquery-public-data.google_analytics_sample` でサンプルデータが確認可能（難読化済み）

### FirebaseからBigQueryへのエクスポート

**エクスポート対応Firebaseサービス**:
- Google Analytics for Firebase
- Cloud Messaging
- Crashlytics
- Performance Monitoring
- A/B Testing
- Remote Config personalization

**設定手順**:
1. Firebaseのプロジェクト設定 → `[統合]`タブ
2. BigQueryを選択してリンク設定
3. エクスポート対象サービスを選択して完了

**エクスポートの種類**:
- 毎日の定期エクスポート（日次データ）
- 当日分のリアルタイムエクスポート（ストリーミングオプション選択時）
  - GoogleアナリティクスとCrashlyticsはストリーミングオプション選択可能

**コスト**:
- リアルタイムエクスポート（ストリーミング挿入）分のコストが発生
- エクスポート後はBigQueryのストレージ料金が発生

**サンプルデータ確認**:
- `firebase-public-project` プロジェクトにサンプルデータが公開（非公開データセット）
- BigQueryコンソールの`[名前を指定してプロジェクトにスターを付ける]`でアクセス可能

### FirebaseとBigQueryの活用ユースケース

| ユースケース | 内容 |
|------------|------|
| A/Bテストの詳細分析 | Firebase以外のデータも統合して多面的な効果検証が可能 |
| ネイティブアプリクラッシュ分析 | Crashlyticsデータを細かい条件でクエリ。ストリーミングエクスポートで新バージョンリリース直後のエラー率監視 |
| プッシュ通知のセグメント最適化 | BigQueryで作成したユーザーセグメントをCloud MessagingにインポートしてターゲティングPUSH通知を実施 |
| リアルタイム分析との連動 | Pub/Sub→Cloud Functions→Cloud Messagingの構成でリアルタイムイベントに応じた通知配信 |

## サービス選択ガイド

| 状況 | 推奨サービス |
|------|------------|
| 定期的にファイルをBigQueryに転送したい（コーディング不要） | BigQuery DTS |
| Amazon S3やRedshiftのデータをBigQueryに定期転送したい | BigQuery DTS |
| RDBの変更をBigQueryにリアルタイム同期したい | Datastream for BigQuery |
| マスター情報・ユーザー情報を常に最新の状態でBigQueryで利用したい | Datastream for BigQuery |
| BigQueryに集めたデータを変換・加工するパイプラインを構築したい | Dataform（SQLで完結）/ Dataflow+Composer（複雑な処理） |
| Webサイトのアクセス解析データをBigQueryで分析したい | GA4→BigQuery エクスポート |
| モバイルアプリのユーザー行動・クラッシュ情報をBigQueryで分析したい | Firebase→BigQuery エクスポート |
