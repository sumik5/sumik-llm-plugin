# レイクハウス構築ガイド（BigLake・Dataplex）

データウェアハウスとデータレイクの利点を統合したレイクハウスアーキテクチャの概念、Google Cloud上での実装パターン、BigLakeによるストレージ統合、Dataplexによるデータガバナンスの実践を解説する。

---

## レイクハウスの概念

### データウェアハウスとデータレイクの違い

| 項目 | データウェアハウス | データレイク |
|------|------------------|------------|
| **データ種別** | 独自フォーマットの構造化データ | オープンフォーマットの半構造化・非構造化データ |
| **データサイズ** | テラバイト級 | ペタバイト級 |
| **主な利用用途** | ビジネスインテリジェンス（BI） | 大量データ蓄積・処理、機械学習処理 |
| **データ処理エンジン** | SQL | SQL からプログラミング言語による機械学習まで |
| **主な分析ユーザー** | データアナリスト・ビジネスユーザー | データサイエンティスト・データアナリスト |
| **スキーマ方式** | スキーマオンライト（書込時定義） | スキーマオンリード（読取時定義） |
| **データ品質・セキュリティ** | 比較的容易 | 比較的困難 |

**データウェアハウスの課題**: スパースなデータや半構造化データ（JSON形式のアクセスログ等）の格納が非効率。スキーマドリフト（上流アプリのスキーマ変更によるETLジョブ異常停止）が発生する。

**データレイクの課題**: Hadoopエコシステム（HDFS・Hive・Spark・Ranger・Atlas）の複数の専門的ソフトウェアを複雑に組み合わせた運用コスト・セキュリティリスク。データが増えると検索性の低下とガバナンスの困難が顕在化する。

### DWHとデータレイクを別々に運用する課題

1. **データの二重管理コスト**: 重要データが両基盤に存在し、整合性担保のためのパイプラインが複雑化する
2. **セキュリティとガバナンスの低下**: 異なるアクセス制御方式を並行運用することで管理コストとセキュリティリスクが増大する
3. **分析効率の低下**: データとツールが異なる基盤に存在するため、データ移動か使用ツールの妥協を強いられる

### レイクハウスの登場と利点

レイクハウスはデータウェアハウスとデータレイク両者の機能を1つの基盤上で実現する新しいデータ基盤アーキテクチャ。以下の2つの機能によって実現する。

1. **統合的なアクセスレイヤー（ストレージエンジン）**: データの種類にかかわらず一元的なアクセスレイヤーを多様なクエリエンジン・アプリケーションに提供する。データのコピー・変換を最小化し、データのリアルタイム性を向上させる

2. **データの自動発見と統合メタデータ管理**: データレイク・データウェアハウスのメタデータを自動でデータカタログ化し、横断検索を実現する。データ品質の追跡やガバナンスをDWHと同等レベルで一元管理する

## Google Cloudでのレイクハウスアーキテクチャ

Google Cloudのレイクハウスアーキテクチャは3層で構成される。

| 層 | サービス | 役割 |
|----|---------|------|
| **ストレージ層** | BigLake | マルチクラウド・マルチフォーマット統合ストレージ管理 |
| **データ処理エンジン層** | BigQuery | SQL分析・BigQuery ML・リモート関数・サーバーレスSpark |
| **データガバナンス層** | Dataplex | メタデータ管理・品質・リネージ・セキュリティ |

BigLakeがCloud StorageとBigQueryストレージを統合し、Dataplexがその上でガバナンスを提供するという2層の役割分担が核心。

## BigLake

### 概要

BigLakeは、マルチクラウド・マルチストレージ・マルチフォーマットに対応したストレージエンジン。BigQueryストレージ以外のデータもBigQueryのテーブルとして扱うことで、データウェアハウスとデータレイクを統合的に管理・分析できる。

```
[BigLakeアーキテクチャ]

SQL / Hadoop/Spark / その他アプリ
          ↓
  BigQuery Storage API（gRPC通信）
          ↓
BigLakeテーブル  ┃  オブジェクトテーブル
    ↓           ↓           ↓
BigQueryストレージ  Cloud Storage  Amazon S3 / Azure Blob
（構造化データ）   （半構造化・非構造化）  （マルチクラウド）
```

### 主要機能

#### マルチクラウド・マルチフォーマット

対応クラウドストレージ: Cloud Storage、Amazon S3、Azure Blob Storage

対応ファイルフォーマット:
- **列指向**: Parquet、ORC
- **行指向**: Avro、CSV、JSON
- **テーブルフォーマット**: Apache Iceberg、Delta Lake、Apache Hudi

既存のHadoop/Spark環境からのスムーズな移行が可能。Presto・Sparkなどのオープンソースクエリエンジンも利用できる。

#### 細かいアクセス制御

BigLakeテーブルではCloud Storageへの直接アクセス権限をユーザーに付与する必要がない。外部接続に関連付けられたサービスアカウントがCloud Storageへのアクセスを代行する仕組み。

```
従来の外部テーブル: ユーザー → BigQueryテーブル権限 + Cloud Storage権限（両方必要）
BigLakeテーブル: ユーザー → BigQueryテーブル権限のみ（CS権限不要）
```

通常のBigQueryテーブルと同様に、**カラム単位・行単位のアクセス制御**、**動的データマスキング**を設定できる。データレイク上のデータにデータウェアハウスと同等のセキュリティレベルを実現する。

#### メタデータキャッシュ

Cloud Storage上のファイルパス（テーブルメタデータ）をキャッシュするオプションを提供し、クエリのパフォーマンスを向上させる。ファイル数やパーティション数が多いテーブルで効果が大きい。

#### BigQuery Storage API

BigLakeテーブルとBigQueryテーブルはBigQuery Storage APIを通じてアクセスできる。

- **従来のDWH**: 外部クエリエンジン → DWHクエリエンジン（ボトルネック） → ストレージ
- **BigQuery Storage API**: 外部クエリエンジン → BigQuery Storage API → ストレージ（直接アクセス・gRPCで高速）

対応アプリケーション: Apache Spark / Apache Hive（BigQueryコネクタ経由）、Pythonライブラリ（`google-cloud-bigquery`等）

### BigLakeテーブルの作成

```bash
# 1. BigQuery Connection APIの有効化
gcloud services enable bigqueryconnection.googleapis.com

# 2. 外部接続の作成（Cloud Storageアクセスを委任するサービスアカウント）
bq mk --connection --location=us \
  --project_id=$(gcloud config get-value project) \
  --connection_type=CLOUD_RESOURCE biglake-conn

# 3. サービスアカウントにバケットの読み取り権限を付与
SERVICE_ACCOUNT=$(bq show --connection \
  --project_id=$(gcloud config get-value project) \
  --location=us --format=json us.biglake-conn | jq -r '.cloudResource.serviceAccountId')
gcloud storage buckets add-iam-policy-binding gs://YOUR-BUCKET \
  --member=serviceAccount:${SERVICE_ACCOUNT} \
  --role=roles/storage.objectViewer
```

```sql
-- BigLakeテーブルの作成（CSVファイルをテーブルとして定義）
CREATE OR REPLACE EXTERNAL TABLE `dataset.table_name`
(
  column1 STRING,
  column2 INTEGER
)
WITH CONNECTION `us.biglake-conn`
OPTIONS (
  format = 'CSV',
  skip_leading_rows = 1,
  uris = ['gs://your-bucket/path/*.csv']
);
```

BigLakeテーブルは**読み取り専用**（DMLは不可）。通常のSELECTクエリや複雑な分析クエリは実行可能。

### オブジェクトテーブル

Cloud Storageの非構造化データ（画像・音声・動画・ドキュメント）のメタデータを自動的にBigQueryのテーブルとして管理する機能。

```sql
-- オブジェクトテーブルの作成
CREATE OR REPLACE EXTERNAL TABLE `dataset.image_files`
WITH CONNECTION `us.biglake-conn`
OPTIONS (
  object_metadata = 'SIMPLE',
  uris = ['gs://your-bucket/images/*']
);
```

自動生成されるスキーマ:
- `uri`: Cloud Storage上のオブジェクトファイルパス
- `content_type`: Content-Type（例: `image/jpeg`）
- `size`: オブジェクトデータのサイズ（バイト）
- `updated`: オブジェクトの最終更新日時
- `md5_hash`: ハッシュ値

**行レベルアクセス制御の適用例**（ファイル拡張子による制限）:

```sql
CREATE ROW ACCESS POLICY jpg_only
ON dataset.image_files
GRANT TO ("user:analyst@example.com")
FILTER USING (ENDS_WITH(uri, '.jpg'));
```

**主な活用シナリオ**:
- SQL + BigQuery MLで非構造化データに対してモデル推論を実行
- リモート関数で任意の処理（画像認識・音声文字変換等）を適用
- 署名付きURLの発行（`EXTERNAL_OBJECT_TRANSFORM`関数）
- 非構造化データへのきめ細かいアクセス制御

## Dataplex

### 概要と解決する課題

DataplexはGoogle Cloudのレイクハウスにおけるデータガバナンス層を担うサービス。BigLakeのストレージ統合だけでは解決できない以下の課題を解決する。

| 課題 | Dataplexの解決策 |
|------|----------------|
| データの発見性が低い（テーブルが多くなると必要なデータを探せない） | データカタログ機能 |
| ビジネスコンテキストが理解できない | アスペクト（ビジネスメタデータ）管理 |
| 分散したアセットの一括権限管理が困難 | レイク・ゾーンによるドメイン管理 |
| データレイクのファイルを手動でテーブル化する手間 | データディスカバリ（自動検知） |
| データ間の依存関係が不透明 | データリネージ |
| データの品質劣化を検知できない | データプロファイリング・品質チェック |

### データカタログ（Dataplex Catalog）

**Dataplex Catalog**（2024年7月GA）: 従来のData Catalogを統合・刷新したカタログ機能。新規利用はDataplex Catalogを推奨。

#### テクニカルメタデータ（自動連携）

BigQuery、Pub/Sub、Spannerなど一部のGoogle CloudサービスはリアルタイムでDataplex Catalogに自動連携される。管理漏れが防げ、データエンジニアの作業負担も減少する。

含まれる情報の例:
- BigQueryデータセット・テーブルの名前・ID・説明
- BigQuery/BigLakeテーブルのスキーマ情報・カラム説明
- Dataplexのレイク・ゾーンの名前・ID・説明

#### ビジネスメタデータ（アスペクト）

組織のデータガバナンスポリシーに基づいて定義するカスタムメタデータ。**アスペクトタイプ**を作成し、BigQueryテーブル・カラムに関連付ける。

```
アスペクトタイプ「Data Governance」の定義例:
  - Data Owner（テキスト型）
  - Freshness（列挙型: daily/weekly/monthly）
  - Stage（列挙型: Development/Staging/Production）
  - Has PII（ブール型）
  - Sensitivity Level（列挙型: LOW/MEDIUM/HIGH）
```

#### ビジネス用語集（Glossary）

業界用語・コード値の意味など解釈に揺れが出やすい用語を定義・管理する機能。BigQueryテーブルのスキーマ情報のカラムに関連付け、分析者のデータ理解を支援する。関連する用語（`関連する用語`）や同義語（`類義語`）の設定も可能。

#### キーワード検索

アスペクト・テクニカルメタデータに含まれるキーワードでテーブル等のアセットを検索できる。プロジェクト・アスペクト種別等でフィルタリングも可能。

### ドメインに基づくデータ管理とセキュリティ

**レイク**: Google Cloud内で複数プロジェクト・異種データストアに分散したアセットを、ビジネスドメイン（販売・商品・人事等）単位でまとめるグループ。

**ゾーン**: レイク配下のデータ処理ステージ別グループ。

| ゾーン種別 | 格納データ | アクセス対象ユーザー |
|-----------|----------|-------------------|
| **未加工ゾーン（Raw）** | 任意の形式の生データ。Cloud Storageに任意フォーマット可 | データエンジニア |
| **キュレートされたゾーン（Curated）** | 分析利用可能なデータ。Cloud StorageはAvro/Parquet/ORC形式+Hive互換ディレクトリ構成が必要 | データアナリスト・ビジネスユーザー |

**権限の伝搬**: Dataplexの[安全]ビューでゾーンにポリシーを設定すると、配下のアセット（BigQueryデータセット・Cloud Storageバケット）に自動的に権限が伝搬する（伝搬に最大30分かかる）。複数データストアの権限をまとめて管理でき、個別設定の手間・ミスによるセキュリティリスクを防ぐ。

### データディスカバリ（Dataplex Discovery）

データレイクに置かれたファイルを自動検知し、BigLakeテーブルを自動登録する機能。

**探索対象**: CSVやParquetなどのオープンフォーマットファイル、画像等の非構造化データ

**設定方法**:
1. Dataplexコンソールでレイクを作成（例: `discovery-handson`）
2. ゾーンを作成（タイプ: 未加工ゾーン、[メタデータの検出を有効にする]をチェック）
3. ゾーンにCloud StorageバケットをアセットとしてAdd（[マネージドにアップグレード]チェックでBigLakeテーブルとして登録）
4. スキャン完了後、対応するBigQueryデータセットにBigLakeテーブルが自動生成される

**自動スキャンスケジュール**: 初回のみ即時スキャン後、設定したスキャンスケジュールにしたがって定期実行。定期スキャンはコストが発生するため使用後はアセットを削除する。

**キュレートゾーンでの追加条件**:
- ファイルフォーマット: Avro、Parquet、またはORCのいずれか
- Hive互換のディレクトリ構成

### データリネージ

データアセット間の依存関係を自動検知・可視化する機能。**Data Lineage APIを有効化**すると自動的に依存関係が構築され、コンソールから参照できる。リネージ情報は**30日間保持**。

**活用シナリオ**:
- データパイプライン変更・テーブルスキーマ変更・テーブル削除の影響範囲把握
- テーブルの信頼性判断のためのデータ生成元確認
- センシティブなデータの組織内利用状況把握
- テーブル問題発生時の上流テーブルの調査

**自動リネージ構築の対象サービス**: BigQuery、Cloud Data Fusion、Cloud Composer、Dataproc、Vertex AI

外部のリネージ情報や対象外サービスのリネージはAPIで任意に追加して統合管理も可能。

### データプロファイリング

BigQueryテーブルの各カラムに含まれるデータ内容を把握するためのスキャン機能。

**提供される統計情報（カラムごと）**:
- 最大値・最小値・平均（数値型）
- NULL値の割合
- ユニーク値の割合
- 出現頻度の高い上位10個の値とその割合
- 文字列の最大長・最小長・平均長

**スキャン設定項目**:
- スキャン対象テーブル・カラム
- スキャン範囲: 全体 or 増分
- フィルタ: 特定行・列を対象に指定
- サンプリングサイズ: スキャン対象データの割合（%）
- スケジュール: 定期実行 or 1回実行
- BigQueryへの結果出力: スキャン結果をBigQueryテーブルに出力（ダッシュボード活用等）

スキャン結果は過去**300回または1年間**のいずれか早い方の履歴を参照可能。履歴参照でデータドリフトや性質の変化を分析できる。

コスト削減のポイント: スキャン範囲・フィルタ・サンプリングを活用して必要最低限のスキャンに抑える。

### データ品質チェック

予期しないデータの混入・欠損をチェック・モニタリングする自動データ品質検証機能。BigQueryおよびCloud Storageのデータに適用可能。

**データ品質低下の代表的な要因**:
- データソースの故障による異常データ発生
- データソース・アプリケーションの仕様変更
- データ変換ロジックの変更や不具合

**ルールの種類**:

| ルール種別 | 評価単位 | 例 |
|-----------|---------|-----|
| 範囲チェック | 行レベル | カラム値が指定範囲内か |
| NULLチェック | 行レベル | NULL値が含まれていないか |
| 値セットチェック | 行レベル | 指定された複数の値のいずれかか（例: A, AB, B, O） |
| 一意性チェック | テーブル単位 | 値がユニークか |
| 集計統計チェック | テーブル単位 | 平均/最大/最小の集計値が一定の範囲内か |
| 正規表現チェック | 行レベル | 指定した正規表現にマッチするか |

**ルール作成の3通りの方法**:
1. **データプロファイルの結果から推奨ルールを生成**: 最も簡単。プロファイル結果の値の傾向から自動生成
2. **事前定義ルールを利用**: GUIで手動設定
3. **SQLカスタムルール**: 他テーブルのカラムを参照するルール等

```sql
-- SQLカスタムルールの例（参照整合性チェック）
-- 「注文」テーブルの user_id が「ユーザー」テーブルに存在するか
user_id IN (SELECT id FROM my_project.dataset.users)
```

**チェック結果の確認**: Dataplex・BigQueryコンソールの[データ品質]タブでOverall Score・ディメンション別スコア・ルールごとの合否が表示される。失敗ルールを選択すると詳細情報とNGレコード取得クエリも閲覧可能。

**アラート連携**: 検査ログはCloud Loggingに出力されるため、ログベースのアラートでメール通知等が可能。

**注意**: データプロファイルとデータ品質スキャンの設定時に「結果をBigQueryとDataplex Catalog UIに公開する」を有効化が必要。

## レイクハウスの統合運用まとめ

| 機能 | 対象 | 主な役割 |
|------|------|---------|
| BigLakeテーブル | Cloud Storage上のオープンフォーマットファイル | BigQuery SQLからのアクセス統合・細かいアクセス制御 |
| オブジェクトテーブル | Cloud Storage上の非構造化データ | SQLによる非構造化データ管理・ML推論 |
| Dataplex レイク・ゾーン | 複数プロジェクト横断のアセット | ドメイン単位の権限管理・一括セキュリティポリシー適用 |
| Dataplex Discovery | Cloud Storage上のオープンフォーマットファイル | BigLakeテーブルの自動登録 |
| データカタログ | BigQuery・BigLake・外部ストア | メタデータ検索・ビジネスコンテキスト管理 |
| データリネージ | BigQuery等のデータアセット間 | 依存関係の可視化・影響範囲の把握 |
| データプロファイリング | BigQueryテーブル | カラム統計情報・データ理解の促進 |
| データ品質チェック | BigQuery・Cloud Storage | データ品質ルール定義・異常検知・アラート |

**重要**: リネージ・データプロファイル・データ品質チェックはBigQueryテーブル・BigLakeテーブル・オブジェクトテーブルのいずれにも適用可能。データ形式を問わない横断的なデータガバナンスを統一的な方法で実現できる。DataplexとBigQueryのUIは統合が進んでおり、各機能の結果はBigQueryコンソールのテーブル情報からも参照できる。
