# コンピューティングプロダクト選択ガイド（GCE/GKE/GAE/Cloud Run/Cloud Functions）

Google Cloud のコンピューティングプロダクトは IaaS・コンテナ・PaaS・FaaS の 4 層で構成され、プロダクトごとに Google が管理する範囲（インフラ〜ランタイム）とユーザーが管理する範囲が大きく異なる。本リファレンスでは 3 冊の書籍知見を統合し、ワークロード特性・運用要件・コストに基づくプロダクト選択フローを解説する。

## コンピューティングプロダクト横断比較

### サービスモデル別プロダクト概要

| プロダクト | サービスモデル | Googleが管理する範囲 | インスタンスライフサイクル | 主なユースケース |
|------------|--------------|-------------------|------------------------|----------------|
| **Bare Metal Solution** | ベアメタル | ネットワーク・電力のみ | 専用物理サーバー | 認定ハードウェア要件・複雑ライセンス制約 |
| **Google Cloud VMware Engine** | VMware 移設 | インフラ全般 | VMware vSphere | オンプレミス VMware 環境をそのまま移設 |
| **Compute Engine（GCE）** | IaaS | ハードウェア・ハイパーバイザー | ユーザー管理 | 仮想マシンが必要なすべてのワークロード |
| **Google Kubernetes Engine（GKE）** | IaaS（コンテナ） | ハードウェア・OS・K8s コントロールプレーン | Pod/Nodeライフサイクル | マイクロサービス・コンテナ化アプリ |
| **App Engine（GAE）** | PaaS | インフラ・ランタイム・スケーリング | リクエスト駆動 | Webアプリ・モバイルバックエンド |
| **Cloud Run** | サーバーレスコンテナ | インフラ・ランタイム全般 | リクエスト駆動（0スケール対応） | コンテナ化サーバーレスアプリ |
| **Cloud Functions** | FaaS | インフラ・ランタイム全般 | イベント駆動（1関数1イベント） | イベント駆動の軽量ロジック |

### GCE・GKE・GAE の詳細比較（プログラマー視点）

| 比較軸 | GCE（IaaS） | GKE（コンテナ） | GAE（PaaS） |
|--------|------------|----------------|------------|
| **仮想化技術** | 仮想マシン | コンテナ（Docker/Kubernetes） | 独自コンテナ（Standard）/ Docker（Flexible） |
| **OS の選択** | Linux / Windows Server を自由選択 | Linux（コンテナイメージで指定） | ランタイムで固定（Standard）/ 自由（Flexible） |
| **スケーリング** | 手動 or MIG による自動スケール | Pod/Node の多段自動スケール | 自動スケーリング（秒単位） |
| **デプロイ単位** | VM イメージ / 起動スクリプト | コンテナイメージ（Dockerfile） | アプリケーションコード / コンテナ |
| **インフラ管理** | ユーザーが全責任 | K8s 制御プレーンは Google 管理 | インフラは Google が完全管理 |
| **ステートフル対応** | ◎（永続ディスク活用） | ○（PersistentVolume） | △（外部データストア前提） |
| **コールドスタート** | なし | ほぼなし（Pod 起動は数十秒） | あり（Standard は数十ミリ秒） |
| **学習コスト** | 低（既存サーバー知識が活用できる） | 中〜高（Kubernetes の知識が必要） | 低〜中（PaaS 制約の理解が必要） |

## プロダクト選択フロー

### ワークロード別選択ガイド

```
ワークロードの特性は何か？
│
├── 認定ハードウェア / 複雑ライセンス制約あり
│   └── → Bare Metal Solution
│
├── VMware vSphere ベースの既存システムを維持したい
│   └── → Google Cloud VMware Engine
│
├── 仮想マシンを使いたい
│   ├── コンテナ化したくない / レガシーアプリ
│   │   └── → Compute Engine（GCE）
│   └── コンテナを使いたい
│       ├── Kubernetes の制御が必要（マイクロサービス・複雑な構成）
│       │   └── → Google Kubernetes Engine（GKE）
│       └── サーバー管理不要でコンテナを動かしたい
│           └── → Cloud Run
│
├── PaaS で開発効率を優先したい
│   ├── サポート言語・ランタイムの制約を許容できる
│   │   └── → App Engine Standard（高速スケール・低コスト）
│   └── 任意の言語 / Docker コンテナが必要
│       └── → App Engine Flexible or Cloud Run
│
└── イベントドリブンの軽量処理
    └── → Cloud Functions
```

### 用途別推奨プロダクト早見表

| 用途 | 推奨プロダクト | 理由 |
|------|--------------|------|
| オンプレミス VM のリフト&シフト | GCE | インフラ構成を維持したまま移行可能 |
| マイクロサービスアーキテクチャ | GKE | Pod 単位のデプロイ・スケール・障害分離 |
| スケーラブルな Web フロントエンド | GAE Standard / Cloud Run | 自動スケール・インフラレス |
| バッチ処理・スポットジョブ | GCE（プリエンプティブル） / Cloud Run Jobs | コスト最優先 |
| API バックエンド（コンテナ化済み） | Cloud Run | ゼロスケール・リビジョン管理 |
| Webhook / Storage イベント処理 | Cloud Functions | 軽量・イベント駆動・シンプル |
| データベースサーバー（商用 DB） | GCE + 永続ディスク | OS/ライセンス制御が必要 |
| ML トレーニング | GCE（A2 GPU ファミリー） | GPU インスタンス直接制御 |

## 料金体系（GCE 中心）

### 継続利用割引（SUD: Sustained Use Discounts）

GCE は**コミットなし・前払いなし**で月間の使用時間に応じて自動割引が適用される。これは AWS Reserved Instances との大きな差別化ポイント。

| 月間稼働割合 | 支払い率 | 割引率 |
|------------|---------|-------|
| 0〜25%（0〜7.5日） | 100% | 0% |
| 25〜50%（7.5〜15日） | 80% | 20% |
| 50〜75%（15〜22.5日） | 60% | 40% |
| 75〜100%（22.5〜30日） | 40% | 60% |

**24時間365日稼働（本番環境）の実効コスト:** 加重平均 70%（= 30% 割引が自動適用）

**同一タイプの複数インスタンスを断続稼働させた場合:** 合計稼働時間を基に割引計算されるため、各インスタンスを短時間だけ使っていても効率的に割引が適用される。

### 確約利用割引（CUD: Committed Use Discounts）

1年または3年の使用コミットメントで最大57%（メモリ最適化ファミリー）の割引。SUD と異なり事前コミットが必要だが、さらに大きな割引を得られる。

| コミット期間 | 割引率（汎用ファミリー） |
|------------|----------------------|
| 1年 | 最大37% |
| 3年 | 最大55% |

### プリエンプティブル VM / Spot VM

| 特性 | 内容 |
|------|------|
| **割引率** | 通常料金の約60〜91%オフ（最大91%割引） |
| **最大稼働時間** | 24時間（Preemptible） |
| **稼働保証** | なし（ゾーン余剰リソース） |
| **ライブマイグレーション** | 対象外 |
| **SLA** | 対象外 |
| **主なユースケース** | バッチジョブ、負荷テスト、CI/CD ワーカー |

**設計のポイント:** ジョブのステータスを Cloud Spanner や Cloud SQL などの外部データベースで管理し、停止後の再開を設計段階で考慮する。

## GAE Standard vs Flexible の比較

| 比較軸 | Standard 環境 | Flexible 環境 |
|--------|--------------|--------------|
| **実行環境** | Google 独自のコンテナインフラ | Compute Engine 上の Docker コンテナ |
| **対応ランタイム** | Python/Java/Go/Node.js/PHP/Ruby（指定バージョン） | 任意の言語・ランタイム |
| **スケールアウト** | 秒単位（非常に高速） | 分単位 |
| **最小インスタンス** | 0（完全ゼロスケール可） | 1（常時 1 インスタンス稼働） |
| **ディスクアクセス** | /tmp のみ（メモリ上） | 永続ディスクも接続可 |
| **コスト** | インスタンス時間ベース（低コスト） | GCE ベース（Standard より高め） |
| **推奨用途** | トラフィック変動が大きい Web アプリ | カスタムランタイム・長時間バックグラウンド処理 |

**選択指針:** まず Standard を検討し、言語制約・ライブラリ要件が満たせない場合のみ Flexible または Cloud Run を選択する。

## Cloud Run vs Cloud Functions の比較

| 比較軸 | Cloud Run | Cloud Functions |
|--------|----------|----------------|
| **デプロイ単位** | コンテナイメージ | 関数（ソースコード） |
| **同時実行** | 最大 250 リクエスト/インスタンス | 1 リクエスト/インスタンス |
| **最大実行時間** | 60分（デフォルト）〜60分 | 540秒（gen1）/ 60分（gen2） |
| **言語** | 任意（コンテナ化されていれば何でも） | Node.js/Python/Go/Java/PHP/Ruby/.NET |
| **gRPC 対応** | ◎（HTTP/2 オプション） | △（HTTP トリガーのみ） |
| **Websocket** | ◎ | ✕ |
| **コールドスタート** | あり（最小インスタンス設定で軽減） | あり |
| **推奨用途** | API サーバー・複雑なマイクロサービス | イベント処理・Webhook・シンプルなトリガー |

## エンタープライズ設計ポイント

### コンピューティングプロダクト選択時の非機能要件チェック

| 非機能要件 | GCE | GKE | GAE | Cloud Run |
|-----------|-----|-----|-----|-----------|
| **高可用性（マルチゾーン）** | MIG でマルチゾーン構成 | リージョンクラスタ | リージョン内自動複製 | リージョン内自動複製 |
| **SLA** | 単一ゾーン 99.5% / マルチゾーン 99.99% | コントロールプレーン 99.95% | 99.95% | 99.95% |
| **スケーリング速度** | 分単位（MIG） | Pod は秒単位 | 秒単位 | 秒単位 |
| **ステートフル対応** | ◎ | ○ | △ | △ |
| **コスト最適化の柔軟性** | ◎（SUD/CUD/プリエンプティブル） | ○（プリエンプティブルノード） | ○（使用分のみ） | ◎（0スケール） |
| **セキュリティ制御粒度** | ◎（OS レベルまで制御） | ◎（Pod/Node/Network ポリシー） | ○（IAM/Firewall） | ○（IAM） |
