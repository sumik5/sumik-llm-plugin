## AI Lifecycle 8段階 — 詳細ガイド

### Stage 1: Problem Definition & Business Understanding（問題定義）
> *「なぜ作るか？」— ビジネス問題の本質を明確化する*

**PMの核心タスク**:
- ビジネスゴールをAI/MLで解決可能な形に変換する
- 「AI機能を作りたい」ではなく「ユーザーのXという問題を解決したい」から始める
- 成功の定義（Success Criteria）を定量的に設定する

**重要な問い**:
```
□ どのビジネス問題を解決するか？
□ AI/MLなしでは解決できないか？
□ 成功したとき何が変わるか（ビジネス指標で）？
□ ステークホルダーは誰で、何を期待しているか？
```

**アウトプット**: Problem Statement・Success Metrics・制約条件一覧

---

### Stage 2: Data Collection & Exploration（データ収集・探索）
> *「どんなデータが必要か？」— データの食材を見極める*

**PMの核心タスク**:
- 必要なデータ種類・量・品質の要件定義
- 既存データソースの棚卸しとギャップ分析
- データプライバシー・規制要件の確認

**データ品質チェック**:
- 完全性（Completeness）: 欠損値の割合
- 正確性（Accuracy）: ラベルの信頼性
- 一貫性（Consistency）: 異なるソース間の整合性
- 適時性（Timeliness）: データの鮮度

**PMへの警告**: 「完璧なデータが揃うまで待つ」は誤り。
小規模・高品質なデータでAI MVPを作り、イテレーションで改善する。

---

### Stage 3: Data Preprocessing（データ前処理）
> *「食材を調理する」— データをモデルが扱える形にする*

**主要タスク**:
- データクリーニング（欠損値・外れ値・重複の処理）
- 正規化・標準化（数値スケールの統一）
- エンコーディング（カテゴリ変数→数値変換）
- 訓練・検証・テストセットの分割

**PMの役割**:
- データエンジニアリング工数をロードマップに適切に反映する
- データ前処理パイプラインの品質保証プロセスを要求する

---

### Stage 4: Feature Engineering（特徴量エンジニアリング）
> *「モデルへの入力を設計する」— 問題を解くための情報を抽出する*

**概念**:
- Feature（特徴量）: モデルが学習に使う入力変数
- Feature Engineering: 生データから意味のある特徴量を作り出すプロセス

**PMの役割**:
- ドメイン知識（ビジネス理解）をエンジニアに提供する
- 「このユーザー行動が重要」という仮説をデータチームと共有する

---

### Stage 5: Model Selection & Training（モデル選択・訓練）
> *「どのアルゴリズムを選ぶか？」— 問題に適したツールを選択する*

**モデルタイプとPMへの含意**:

| タイプ | 特徴 | 代表的なユースケース |
|--------|------|------------------|
| 線形回帰 | シンプル・説明しやすい | 価格予測・需要予測 |
| 決定木/ランダムフォレスト | 解釈可能・ロバスト | チャーン予測・分類 |
| ニューラルネットワーク | 高精度・複雑 | 画像認識・自然言語処理 |
| 大規模言語モデル（LLM） | 汎用・生成 | チャットボット・コンテンツ生成 |

**PMの役割**:
- 精度 vs 解釈可能性のトレードオフを事業要件に基づき判断
- ハイリスク領域（医療・融資）では解釈可能モデルを優先

---

### Stage 6: Model Evaluation & Tuning（モデル評価・チューニング）
> *「品質を確認し磨く」— ビジネス目標に対して十分に機能するか検証する*

**主要評価指標（PMが知るべき）**:

| 指標 | 意味 | PMとの関連 |
|------|------|-----------|
| **Accuracy（正解率）** | 全体的な正確さ | 平均的な性能把握 |
| **Precision（適合率）** | 陽性予測の正確さ | 誤検知コストが高い場合（スパム・詐欺検知） |
| **Recall（再現率）** | 実際の陽性の捕捉率 | 見逃しコストが高い場合（病気検知） |
| **F1 Score** | PrecisionとRecallの調和平均 | バランスが必要な場合 |
| **AUC-ROC** | 全体的な判別性能 | モデル比較・閾値設計 |

**GenAI評価手法（3種）**:
- **Human Evaluation**: 評価者が品質を直接判断（Gold Standard・コスト高）
- **Automated Metrics**: BLEU/ROUGE/Perplexity等（スケーラブル・客観的）
- **Hybrid Approaches**: 自動フィルタリング → 人手評価（バランス型・推奨）

---

### Stage 7: Deployment & Monitoring（デプロイ・監視）
> *「本番稼働と健全性維持」— AIを生きたシステムとして管理する*

**デプロイ戦略**:

| 戦略 | 内容 | 使いどころ |
|------|------|-----------|
| **Shadow Deployment** | 本番影響なしにモデル評価 | 新モデルの安全な検証 |
| **A/B Testing** | モデルバージョン比較 | インパクト測定 |
| **Canary Deployment** | 一部ユーザーに段階的展開 | リスク最小化 |

**監視すべき指標**:
```
技術的指標:
  - モデルの予測精度（継続的）
  - レイテンシ・スループット
  - エラー率・可用性

ビジネス指標:
  - North Star Metricへの貢献
  - ユーザー行動の変化
  - 収益・コストへのインパクト

Model Health:
  - Data Drift（入力データ分布の変化）
  - Concept Drift（問題の性質変化）
  - Feature Importance変化
```

---

### Stage 8: Retraining & Maintenance（再訓練・保守）
> *「モデルを最新状態に保つ」— living systemとして継続管理する*

**Model Driftへの対応**:
- **データドリフト**: 入力データの分布が変化 → 再訓練トリガー
- **コンセプトドリフト**: 問題の性質が変化 → 特徴量・モデル再設計
- **パフォーマンス劣化**: 精度指標の低下 → 調査・再訓練

**再訓練トリガーの種類**:
- **スケジュールトリガー**: 月次/四半期定期再訓練
- **パフォーマンストリガー**: 精度が閾値を下回ったとき
- **イベントトリガー**: 重大なビジネス変化時

**PMのロードマップへの組み込み方**:
```
✅ 初回リリース後の保守工数を最低30%見積もる
✅ モデル再訓練サイクルをスプリント計画に含める
✅ モデル劣化のアラートシステムを要件に含める
```

## NPD（新製品開発）の7段階とAI製品

出典: *AI Product Manager's Handbook 2nd Ed.* Irene Bratsis

一般的なNPDプロセスとAI製品特有の考慮点：

| ステップ | 内容 | AI製品特有の考慮点 |
|---------|------|-----------------|
| 1. Discovery | ユーザーリサーチ・市場分析 | データ可用性の調査を追加 |
| 2. Define | 要件定義・成功指標設定 | AI評価指標（MLメトリクス）の定義 |
| 3. Design | UX/UIデザイン | AI特有のUX（不確実性表示・説明可能性）設計 |
| 4. Implementation | 開発・訓練 | AI Lifecycle 8段階の実行 |
| 5. Marketing | GTM準備 | AI機能のポジショニング・リスク開示 |
| 6. Beta Testing | 限定公開テスト | 人間評価・多様なユーザーでのバイアステスト |
| 7. Launch | 本番リリース | 段階的デプロイ（Shadow→Canary→Full） |

## AI/MLドリームチームの構成

出典: *AI Product Manager's Handbook 2nd Ed.* Ch.6

| ロール | 主要責任 |
|--------|---------|
| **AI PM** | 戦略・優先順位・ステークホルダー管理 |
| **Data Engineer** | データパイプライン・インフラ構築 |
| **Data Scientist** | モデル設計・実験・評価 |
| **ML Engineer** | モデルの本番実装・MLOps |
| **UX Designer/Researcher** | AI体験設計・ユーザーリサーチ |
| **QA/Testing Engineer** | AIテスト・品質保証 |
| **Data Analyst** | メトリクス分析・インサイト抽出 |

## 製品化チェックリスト（AI機能リリース前）

```
技術的準備度:
  □ モデル精度が定義した閾値を満たしているか
  □ レイテンシ要件を満たしているか
  □ 障害時のフォールバック機能があるか
  □ 監視・アラートシステムが稼働しているか

品質・安全性:
  □ バイアス評価を完了したか
  □ 多様なユーザーグループでテストしたか
  □ エッジケース・敵対的入力のテスト済みか

ビジネス準備度:
  □ KPIと測定方法を定義したか
  □ 再訓練サイクルをロードマップに組み込んだか
  □ ユーザーへのAI利用開示を準備したか
  □ データ収集・利用のプライバシーポリシーが整備されているか
```
