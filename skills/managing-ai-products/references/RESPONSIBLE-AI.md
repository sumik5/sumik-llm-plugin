## AIバイアスの種類と対策

### バイアスが発生する5つの場所

```
1. Data Collection Bias（データ収集バイアス）
   問題: 収集されたデータが実際の分布を代表していない
   例: 男性優位な採用データで訓練した採用AIが女性候補を低評価
   対策: データソースの多様性確保・サンプリング手法の見直し

2. Historical Bias（歴史的バイアス）
   問題: 過去の不公平な意思決定パターンを学習・再現する
   例: 歴史的に不当に高いリスクスコアをつけられた地域への融資モデル
   対策: 歴史的バイアスのある特徴量の除外・修正

3. Measurement Bias（測定バイアス）
   問題: 特定グループに対してデータの測定・収集方法が異なる
   例: 医療診断AIが特定人種の痛み評価を過小評価（研究から実証済み）
   対策: グループ別の測定方法一貫性確認

4. Aggregation Bias（集計バイアス）
   問題: 異なるグループを同一モデルで扱うことによる不公平
   例: 全人種共通の糖尿病診断モデルが特定人種で精度低下
   対策: サブグループ別モデル評価・必要に応じたモデル分離

5. Evaluation Bias（評価バイアス）
   問題: 評価データセット自体が特定グループに偏っている
   例: 英語のみのNLPベンチマークで非英語話者への性能を見落とす
   対策: 多様なグループを含む評価セットの構築
```

### Protected Classes（保護される属性）

法的・倫理的に差別が禁止されているカテゴリ：

| カテゴリ | 例 |
|---------|-----|
| 人種・民族 | 人種・皮膚の色・出身国 |
| 性別・性自認 | 男性・女性・ノンバイナリー |
| 年齢 | 特に高齢者・若年者 |
| 障害 | 身体・精神・認知障害 |
| 宗教 | 信仰・宗教的実践 |
| 性的指向 | LGBTQコミュニティ |
| 妊娠・育児状況 | 妊娠・育児休業取得者 |

**AI PMのAction**: これらの属性でモデル性能を定期的にサブグループ評価する

## バイアス軽減のPM実践手法

### プロアクティブなバイアス対策（開発フロー統合）

```
Discovery・要件定義フェーズ:
  □ ユースケースに関わるProtected Classesを特定
  □ データ収集・ラベリング計画にDEIレビューを組み込む
  □ 「Do No Harm」原則の確認（高リスクユースケースの識別）

開発・訓練フェーズ:
  □ サブグループ別の訓練データ分布を確認
  □ 公平性指標（Fairness Metrics）をSLAに組み込む
  □ 多様な評価者によるデータアノテーション

評価フェーズ:
  □ サブグループ別の精度・エラー率を評価
  □ Disparate Impact（格差的影響）テストを実施
  □ 人間評価者の多様性を確保

デプロイ・監視フェーズ:
  □ 本番でのバイアスモニタリングを設定
  □ 異常時の自動アラートと人手介入フロー
  □ 定期的なバイアス監査スケジュールを設定
```

## 透明性・説明可能性（XAI）

### なぜ透明性が重要か

1. **ユーザーの信頼構築**: ブラックボックスは信頼を損なう
2. **法的要件**: EU AI Act・GDPR等で説明責任が求められる
3. **モデル改善**: 問題の根本原因特定が容易になる
4. **リスク管理**: 高リスク決定（融資・医療・採用）での透明性確保

### 説明可能性の実装手法

| 手法 | 内容 | 適用場面 |
|------|------|---------|
| **LIME** | 個別予測の局所的説明 | 顧客への個別説明 |
| **SHAP** | 特徴量重要度の可視化 | モデルデバッグ・監査 |
| **Attention Visualization** | LLMの注意機構可視化 | テキスト分類の根拠 |
| **Counterfactual Explanation** | 「Xが違えばYだった」形式 | 融資否決・採用落ちの説明 |

### PMのXAI実装判断基準

```
XAI必須のユースケース:
  ✓ 融資・保険の審査
  ✓ 採用・評価システム
  ✓ 医療診断・治療推薦
  ✓ 刑事司法・リスクスコアリング
  ✓ 公共サービスの配分

XAI推奨のユースケース:
  ✓ 高額商品の推薦
  ✓ コンテンツモデレーション
  ✓ ユーザーアカウント停止・制限

XAIが任意のユースケース:
  ✓ コンテンツ推薦（低リスク）
  ✓ 予測テキスト入力
  ✓ 最適化・効率化タスク
```

## プライバシー・データガバナンス

### AI製品のプライバシー設計原則

```
Privacy by Design（7原則）:
1. プロアクティブ（事後対応ではなく予防）
2. プライバシーをデフォルトに
3. プライバシーをアーキテクチャに組み込む
4. 完全な機能性（ゼロサムではなくポジティブサム）
5. エンドツーエンドのセキュリティ
6. 可視性・透明性
7. ユーザー中心主義
```

### AI学習データのプライバシー要件

| 要件 | 内容 |
|------|------|
| **Consent（同意）** | ユーザーデータをAI学習に使う明示的同意 |
| **Purpose Limitation** | 収集目的以外への利用禁止 |
| **Data Minimization** | 最小限必要なデータのみ収集 |
| **Right to Erasure** | 「忘れられる権利」の実装（モデル学習データからの削除） |
| **Data Portability** | ユーザーが自分のデータをエクスポートできること |

## 倫理的AIチェックリスト（完全版）

### リリース前必須確認事項

```
バイアス・公平性:
  □ Protected Classes への差別的影響を評価したか
  □ サブグループ別の性能評価を実施したか
  □ 多様なユーザーによるユーザーテストを実施したか

透明性・説明可能性:
  □ ユーザーにAI使用を開示しているか
  □ 高リスクな決定に対して説明機能を実装しているか
  □ AIの限界・不確実性をUIで適切に表示しているか

プライバシー:
  □ 学習データの収集・利用に適切な同意を得ているか
  □ データ最小化の原則に従っているか
  □ データセキュリティ（暗号化・アクセス制御）を実装したか

説明責任:
  □ AI意思決定の監査ログが存在するか
  □ 問題発生時のエスカレーションフローがあるか
  □ ヒューマンオーバーライド機能があるか（高リスク領域）

継続監視:
  □ バイアスの継続モニタリング体制があるか
  □ 定期的な倫理監査のスケジュールを設定したか
```

## 主要AI規制対応

### EU AI Act（2024年施行）対応ポイント

```
リスク分類:
  □ 許容不可能なリスク（禁止）:
      - リアルタイム生体認証による公共監視
      - ソーシャルスコアリング

  □ 高リスク（厳格な要件）:
      - 雇用・採用システム
      - 教育・職業訓練
      - 医療機器
      - 重要インフラ管理
      → 適合性評価・登録・透明性・人間監視が必要

  □ 限定リスク（透明性義務）:
      - チャットボット（AI使用の開示）
      - ディープフェイク（開示義務）

  □ 最小リスク:
      - スパムフィルター等（自主的行動規範）
```

### GDPR（データ保護規則）× AI

- **Art.22**: 完全自動化された意思決定からの保護権
- **Art.13-14**: AIを使ったデータ処理の開示義務
- **Data Protection Impact Assessment (DPIA)**: 高リスクAI処理の事前評価必須

## AIインシデント対応フロー

```
Level 1: 性能劣化（通常の監視内）
  → 自動アラート → エンジニアチーム対応 → 24時間以内に改善計画

Level 2: バイアスインシデント（特定グループへの不公平な影響）
  → PM・倫理チームにエスカレーション → 48時間以内に緊急評価
  → 必要に応じてロールバック → ユーザーコミュニケーション

Level 3: 重大な倫理違反・法的問題
  → 法務・経営層への即時報告 → 機能停止を検討
  → 外部監査・当局への報告（規制要件に応じて）
  → 再発防止策の策定・公開
```
