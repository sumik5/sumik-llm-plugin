# Figmaデザインシステム構築 実装ガイド

## 第1章: デザインシステムとは

### 1-1. 目的と役割

デザインシステムとは、プロダクトのデザインと実装を支える**再利用可能な部品と判断基準の集合体**。単なるUIコンポーネントライブラリではなく、チームがより速く・一貫性を持って開発するための「共通言語」として機能する。

主な構成要素は以下の4つ：

| 構成要素 | 内容 |
|---------|------|
| **デザイン原則** | 意思決定の指針となるブランドの価値観・哲学 |
| **スタイルガイド** | カラー・タイポグラフィ・スペーシング等の視覚的ルール |
| **パターンライブラリ** | UIコンポーネント・レイアウト・テンプレートの集合 |
| **実装コード** | フロントエンド実装・デザイントークン・ドキュメント |

### 1-2. デザインシステムのメリット

- **変更容易性**: トークン1箇所を変えるだけで全コンポーネントに反映
- **チームコラボレーション**: デザイナーとエンジニアが共通の命名で会話できる
- **拡張性**: 新機能追加時に既存のルールを踏まえた設計ができる
- **オンボーディング**: 新メンバーが判断基準を参照しやすくなる
- **バリエーション対応**: ダークモード・多言語・アクセシビリティ対応が一元管理できる

### 1-3. いつ始めるか

プロダクトが以下の状態になったら検討するタイミング：

- 同じUIが複数ページで少しずつ違う形で実装されている
- デザイナーとエンジニアで「このボタンの色は何？」という会話が繰り返されている
- 新機能のデザインに既存UIの再確認が必要になっている

```
【AskUserQuestion】
デザインシステムを新規に立ち上げますか、それとも既存のコンポーネント群を整理して標準化しますか？
```

## 第2章: Figmaファイル構成

### 2-1. ファイル分割の段階

```
Stage 1: モノリシック
  └── 1ファイルにすべて（小規模・スタート時）

Stage 2: ライブラリ分離
  ├── Library ファイル（コンポーネント・スタイル）
  └── Design ファイル（実際の画面）

Stage 3: モジュール化
  ├── Foundation（カラー・タイポ・アイコン）
  ├── Components（ボタン・入力・カード等）
  ├── Patterns（グリッド・テンプレート）
  └── Design Files（各プロジェクトの画面）
```

```
【AskUserQuestion】
現在のチーム規模はどれくらいですか？（デザイナー人数・プロジェクト数）
これによりファイル分割の段階を決定します。
```

### 2-2. ライブラリ管理

Figmaのチームライブラリ機能を使い、コンポーネントとスタイルを共有する。

- **公開する**: スタイル・コンポーネントを他ファイルで使えるようにする
- **スワップライブラリ**: 旧ライブラリから新ライブラリへの一括移行
- **バージョン管理**: Figmaのバージョン履歴を定期的に保存（命名規則: `v1.2.0 - 変更内容`）

## 第3章: デザイントークン設計

トークンの詳細は [DESIGN-TOKENS.md](references/DESIGN-TOKENS.md) を参照。

### 3-1. 階層の決定

```
【AskUserQuestion】
トークンは何層構成にしますか？
- 2層（Primitive + Semantic）: シンプルで小規模向け
- 3層（Primitive + Theme/Alias + Semantic）: テーマ切替やブランド複数対応向け
```

### 3-2. カラーパレット設計

```
【AskUserQuestion】
ブランドのプライマリカラーは決まっていますか？
また、カラーパレットはどのように生成しますか？
- 既存ブランドカラーを起点にスケール生成
- ゼロから設計
- Figmaプラグイン（Oklch Palette等）を使用
```

カラーパレットは **プライマリ・ニュートラル・フィードバック（danger/success/warning）** の3グループを最低限用意する。それぞれ9段階（/10〜/90）のスケールを定義する。

### 3-3. ダークモード対応

```
【AskUserQuestion】
ダークモードは必要ですか？
必要な場合、実装タイミングはいつですか？
- 最初から両モード設計（後から変えにくい設計決定）
- まずライトモードのみ、後でダークモード追加
```

ダークモードを後から追加する場合、セマンティックカラートークンの設計が鍵になる。ライトモード専用の名前（例: `color/background/white`）をつけると後から変えにくくなるため、最初から用途ベースの名前（例: `color/background/default`）にしておく。

## 第4章: カラーシステム実装

### 4-1. Figmaバリアブルコレクションの作成順

1. `_PrimitiveColor` コレクション作成 → 全カラー値を登録
2. `_ThemeColor` コレクション作成 → プリミティブを参照するエイリアスを登録
3. `SemanticColor` コレクション作成（Light / Darkモードあり）→ テーマカラーを参照
4. `SemanticColor` の各モードでライト/ダークの参照先を切り替える

### 4-2. 非公開バリアブルの設定

コレクション名の先頭に `_` を付けることで、ライブラリ公開時に他ファイルから直接参照できなくなる（内部実装の隠蔽）。

- `_PrimitiveColor` → 非公開（内部実装）
- `_ThemeColor` → 非公開（内部実装）
- `SemanticColor` → **公開**（エンジニアが参照）

### 4-3. アクセシビリティの確認

セマンティックカラーのすべてのテキスト/背景の組み合わせで WCAG 2.1 レベルAA（通常テキスト4.5:1以上）を達成していることを確認する。Figmaのコントラストチェックプラグインを使う。

## 第5章: タイポグラフィシステム

### 5-1. フォントの決定

```
【AskUserQuestion】
使用するフォントは決まっていますか？
- 和文フォント・欧文フォントそれぞれの指定
- Webフォントの使用可否（Google Fonts / Adobe Fonts）
- フォントウェイトのバリエーション（Regular/Medium/Bold/SemiBoldのどれを使うか）
```

### 5-2. スケールの定義

フォントサイズは9段階（`font-size/10`〜`font-size/90`）を基本とする。具体的な値はプロダクトの文字量・対象ユーザー・使用デバイスに合わせて調整する（詳細値は DESIGN-TOKENS.md 参照）。

### 5-3. Figmaへのテキストスタイル登録

コンポジットタイポグラフィトークン（`typography/{context}/{size}/{weight}`）をFigmaのテキストスタイルとして登録する。スタイル名はそのままコード側のトークン名と一致させることで、エンジニアとの会話コストを下げる。

## 第6章: その他の拡充スタイル

### 6-1. スペーシング・角の半径・線幅

8pxグリッドに基づくスペーシング、角の半径、線幅を `Token` コレクションとして管理する（詳細は DESIGN-TOKENS.md 参照）。

```
【AskUserQuestion】
UIのデザイントーンはどちらに近いですか？
- シャープ（角の半径が小さい → 機能的・業務系）
- ラウンド（角の半径が大きい → 親しみやすい・コンシューマー系）
これにより border-radius のスケール設定が変わります。
```

### 6-2. エレベーションとシャドウ

エレベーショントークン（`elevation/0`〜`elevation/4`）をFigmaのエフェクトスタイルとして登録する。コンポーネントのz-index管理と連動させることで、モーダルとドロップダウンの重なり順を一貫して管理できる。

## 第7章: パターンライブラリ

パターンライブラリの詳細は [PATTERN-LIBRARY.md](references/PATTERN-LIBRARY.md) を参照。

### 7-1. レイアウトグリッドの設定

```
【AskUserQuestion】
ターゲットデバイスはどれですか？
- デスクトップのみ（12カラムグリッド）
- モバイルファースト（xs〜xl全対応）
- デスクトップとタブレットのみ（xs不要）
```

### 7-2. コンポーネントの優先順位

デザインシステム構築時にすべてのコンポーネントを一度に作る必要はない。以下の順で優先度をつける：

1. **アトム**: ボタン・入力・テキスト・アイコン・バッジ（最頻出）
2. **モレキュール**: フォームグループ・カード・ナビゲーションアイテム
3. **オーガニズム**: ヘッダー・サイドバー・モーダル・テーブル
4. **テンプレート**: ページレイアウトの雛形

## 第8章: コード連携

コード連携の詳細は [CODE-INTEGRATION.md](references/CODE-INTEGRATION.md) を参照。

### 8-1. トークンエクスポートの自動化

```
【AskUserQuestion】
デザイントークンの更新をどのように管理しますか？
- 手動エクスポート（小規模・シンプル）
- Figmaプラグインによる自動同期（Tokens Studio等）
- Figma REST APIを使ったCI/CD連携（大規模）
```

### 8-2. Storybookの整備

コンポーネントライブラリとデザインシステムをStorybookで統合することで、デザイナーとエンジニアが同じカタログを参照できる。デザイントークンのリファレンスページをStorybookのDocsとして追加することを推奨する。

### 8-3. 運用の仕組み

- デザインシステムの変更はプルリクエストで管理する
- デザイン変更（Figma）とコード変更（PR）は対になることを明示する
- 変更影響範囲のレビューをStorybookのビジュアル差分で行う
