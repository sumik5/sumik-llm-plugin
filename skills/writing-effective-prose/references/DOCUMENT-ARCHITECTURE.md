# 文書アーキテクチャ リファレンス

文書の構造設計と、その背後にある認知心理学的根拠を体系化する。
「なぜこの構造が読み手に伝わるのか」を理解することで、設計の判断が根拠に基づいたものになる。

出典:
- 書籍1: 仲田尚央『エンジニアが一生困らない ドキュメント作成の基本』（ソシム）
- 書籍4: 倉島保美『論理が伝わる 世界標準の「書く技術」』（講談社ブルーバックス）

## 1. 文書の5要素階層構造（書籍1）

### 概要

ドキュメントは階層構造を持つ。大きなテーマがサブテーマに分解され、さらに個々の話題まで分解される。この階層構造を形成するのが**5つの要素**である。

```
タイトル（全体テーマ）
│
├── リード文（タイトルを補足）
│
├── 見出し（サブテーマ）
│   │
│   ├── リード文（見出しを補足）
│   │
│   ├── パラグラフ（話題1）
│   │   └── 中心文（言いたいこと） + 支持文（理由・説明）
│   │
│   └── パラグラフ（話題2）
│       └── 中心文 + 支持文
│
└── 見出し（サブテーマ）
    └── ...
```

### 各要素の定義と役割

#### タイトル — 全体テーマを示す

タイトルはドキュメント全体のテーマを示す。示し方は2種類ある。

| 種類 | 例 | 読み手の行動 |
|------|-----|------------|
| **何が書かれているか** |「ユーザーマニュアル」「○○機能の仕様」 | 自分が必要とする情報があるか「当たりを付ける」 |
| **何が言いたいか** | 「伝え方が9割」「イシューからはじめよ」 | 全体の要点を掴んでから読み進む |

#### 見出し — サブテーマを示す

タイトルと同様に「何が書かれているか」または「何が言いたいか」を示す。

- **当たりを付ける機能**（辞書形式ドキュメントで重要）: 必要な情報がどこにあるか判断する手掛かり
- **要点を掴む機能**（読み物形式ドキュメントで重要）: 内容を先に把握してから詳細を読む

> 「1つの見出しに1つのテーマ」原則。プログラミングの1関数1責任と同じ。

#### リード文 — タイトル・見出しを補足する

タイトルや見出しの直後に置かれ、本文の内容を補足する。役割は2つ:

1. **先読みの補助**: タイトルや見出しだけでは情報が不足するとき、本文の内容に当たりを付ける手掛かりになる
2. **接続の補助**: 前の見出しとのつながりを良くする橋渡し

> リード文は省略可能。タイトル・見出しだけで十分な情報が伝わる場合は不要。

#### パラグラフ — 1つの話題を扱う文の集まり

1つの話題（= 言いたいこと + 理由・説明）が1つのパラグラフに対応する。

```
パラグラフ
├── 中心文（言いたいこと）    ← Topic Sentence
└── 支持文（理由・説明）      ← Supporting Sentences
    ├── 理由を述べる（「なぜなら〜だからです」）
    ├── 説明を述べる（「つまり〜ということです」）
    └── 具体例を挙げる（「たとえば〜があります」）
```

> **段落との違い**: 段落は改行で区切られた文の固まり（構造が自由）。パラグラフは「1つの話題を1つの固まりにする」ルールがある。ドキュメント（実用文）にはパラグラフが適している。

#### 中心文 — パラグラフの言いたいことを示す文

英語では Topic Sentence。パラグラフごとに1つ存在し、そのパラグラフで何が言いたいかを端的に述べる。

- 中心文を意識して読む = 「言いたいこと」と「理由・説明」に分けて読む
- 中心文だけを拾い読みすれば、文書の骨格が把握できる

## 2. 辞書形式 vs 読み物形式（書籍1）

ドキュメントの形式によって、読み手の目的・読み方・重要な要素が異なる。

### 比較表

| 観点 | 辞書形式 | 読み物形式 |
|------|---------|-----------|
| **目的** | 特定の情報を調べる（目的が明確） | 要点を掴みながら読む（目的が曖昧なことも多い） |
| **読み方** | 当たりを付けて必要な部分だけ読む | 流れに沿って先頭から読む |
| **ゴール** | 必要な情報が得られたら終了 | 読み終えるか、十分な情報を得て満足したとき |
| **主な文書** | マニュアル・仕様書・API リファレンス | 報告書・企画書・技術ブログ・チュートリアル |
| **最重要要素** | タイトルと見出し（当たりを付ける手掛かり） | リード文と中心文（要点を掴む手掛かり） |

### 辞書形式のドキュメントを書く際の指針

1. **見出しが機能するかを優先する**: 見出しを見るだけで必要な情報がどこにあるかわかるか
2. **キーワードで探しやすくする**: 検索されるキーワードを見出しに含める
3. **自己完結した構成にする**: 前後の文脈を知らなくても読める節を作る

### 読み物形式のドキュメントを書く際の指針

1. **リード文で要点を先に示す**: タイトルの後に何が書かれているかを補足する
2. **中心文を先頭に置く（PREP法）**: 各パラグラフで言いたいことを冒頭に述べる
3. **流れを意識する**: 前の見出しとのつながりを見出しのリード文でブリッジする

## 3. 認知心理学的基盤（書籍4）

なぜこの文書構造が読み手の理解を助けるのか、認知心理学的な根拠を示す。

### 3-1. メンタルモデル

#### 原理

認知心理学によると、人は入力情報を短期メモリに保存しながら、長期メモリの関連情報と組み合わせて処理する。この処理を高速化するため、人は**メンタルモデル**（自分なりの理解の世界）を構築して次の情報を予測する。

```
情報の入力
    ↓
メンタルモデル作成（「次はこう展開されるはずだ」）
    ↓
関連情報を長期メモリからあらかじめ活性化
    ↓
予測どおりの情報が入力される → 高速処理・理解
予測が外れる              → 低速処理（モデルの修正が必要）
```

#### 文章設計への応用

| 設計判断 | メンタルモデルとの関係 |
|---------|---------------------|
| 総論パラグラフで始める | 読み手が「A→B→Cの順に展開される」という予測モデルを先に作れる |
| 総論と同じ順序で各論を展開する | 予測を裏切らないことで処理コストを下げる |
| リード文で次の内容を予告する | 読み手が次のパラグラフに必要な関連情報を活性化できる |
| 見出しで内容を先に示す | 詳細を読む前にメンタルモデルが形成され、理解が促進される |

**短期メモリの制約**: 短期メモリには 7±2 個の情報しか約20秒しか保存できない。この制約が「情報の絞り込み（3の力）」の根拠になる（→3-3参照）。

### 3-2. 強調のポジション

#### 原理

文章の固まりの**最初と最後**が記憶に残りやすい「強調のポジション」である。人は最初と最後で緊張（注意力が高まる）するからである。

この強調のポジションは、すべての階層に存在する。

```
文章全体:   [最初のパラグラフ] ←強調  ...  [最後のパラグラフ] ←強調
階層（章）:  [最初のパラグラフ] ←強調  ...  [最後のパラグラフ] ←強調
パラグラフ: [最初の文]         ←強調  ...  [最後の文]         ←強調
文:         [最初の単語]       ←強調  ...  [最後の単語]       ←強調
```

#### 文章 vs プレゼンテーション

| 形式 | 優先する強調ポジション | 理由 |
|------|---------------------|------|
| **文章** | 最初（冒頭） | 読み手が最後まで読む保証がない。重要な情報ほど先に書く |
| **プレゼンテーション** | 最後（締め） | 聴衆は最後まで聞くのが前提。最後に話した内容が記憶に残る |

#### 設計への応用

- **重要情報は冒頭に**: 結論・要約・最重要メッセージをパラグラフの最初の文（中心文）に置く
- **文末にも意味を持たせる**: 1文の中で最も強調したい語句を文末に置く（「〜が原因だ」より「原因は〜だ」）
- **文書全体の冒頭を丁寧に作る**: 総論パラグラフは読み手が唯一確実に読む部分

### 3-3. 情報の絞り込み（3の力）

#### 原理

情報は数を絞ると記憶に残りやすい。短期メモリの制約（7±2）から、人が一度に処理できる情報量には限りがある。

**3つが黄金律**である理由:
- 多すぎると記憶できない
- 少なすぎると手を抜いた印象になる・内容が物足りない
- 3は多くも少なくもない、ちょうどいい数

例: "government of the people, by the people, for the people"（リンカーン）

#### 設計への応用

| 場面 | 推奨 |
|------|------|
| 根拠・ポイントの列挙 | 3つに絞る（または3つにグループ化） |
| セクション構成 | 3〜5の見出しに分ける |
| 強調したいメッセージ | 1つだけに絞る（最多でも3つ） |
| どうしても10個ある場合 | 3つのカテゴリに分類して提示する |

> **例外**: 網羅性・忙しさをアピールしたいときは意図的に数を多く出す。このときは「記憶」ではなく「量の多さ」を伝えることが目的。

## 4. 文書の3つの目的と形式の選択

書籍1では、エンジニアが書くドキュメントを3つの目的に分類している。目的と形式の関係を把握することで、適切なアーキテクチャを選べる。

| 目的 | 形式 | 代表的な文書 |
|------|------|------------|
| **説明型**（概念や手順を説明） | 辞書形式 | 要件定義書・仕様書・マニュアル・API リファレンス |
| **報告型**（知見や活動を報告） | 読み物形式 | 報告書・技術ブログ・論文 |
| **説得型**（意見・提案で行動を促す） | 読み物形式 | 企画書・提案書 |

> 初心者向けのチュートリアルは「説明型」だが、読み物形式を取ることが多い。

## 5. 構造設計の実践フロー

### 書く前の構造設計（5ステップ）

```
①  読み手とテーマを選定する
    └── 誰に何を伝えるか（目的・知識レベル・立場で読み手を絞る）
        ↓
②  テーマを分解する（Why-What-How）
    └── なぜ（背景・目的）/ 何を（内容・対象）/ どうやって（方法・手順）
        ↓
③  ドキュメントの構成を組む（アウトライン）
    └── 分解したテーマ → 見出しに割り当て
        ↓
④  文章の構成を組む（パラグラフ計画）
    └── 各見出しの中で「言いたいこと」を1文ずつ書き出し → 並べ替え
        ↓
⑤  文を書く
    └── 言いたいこと（中心文）に対して支持文を書き足す
```

### 形式の判断フロー

```
文書の目的は？
├── 特定の情報を調べるために読む  → 辞書形式
│   └── 見出しの「当たりやすさ」に注力
└── 流れを追って読む             → 読み物形式
    └── リード文・中心文の「要点先出し」に注力
```
