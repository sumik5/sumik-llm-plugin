# エンジニア設計書 リファレンス

ソフトウェア開発で必要な設計書（要件定義書、仕様書、機能設計書、運用設計書）の作成ガイド。
「エンジニアが一生困らない ドキュメント作成の基本」（仲田尚央）および「エンジニアなら知っておきたい システム設計とドキュメント」（梅田弘之）の知識を体系化。

---

## A. エンジニア設計書の全体像

### なぜ設計書を作成するのか（4つの理由）

1. **設計の効率化と品質向上**
   設計書は設計の「見える化」。書くことで思考が整理され、レビューが可能になる。頭の中だけで考えた設計は、矛盾や抜け漏れに気づきにくい。

2. **設計書の信頼性確保**
   設計書がないと「口頭伝承」になり、人が変わると知識が失われる。「あの仕様はなぜそうなっているのか」が誰にも説明できない状態を防ぐ。

3. **保守運用を楽にする**
   運用・保守フェーズで設計書がないと、変更影響分析ができない。「どこを直せばいいか」「この変更は他に影響するか」が即座にわかるようにする。

4. **アジャイル開発でも必要**
   ドキュメントを書かないのではなく、「必要十分」なドキュメントを書く。アジャイル宣言も「包括的なドキュメントよりも動くソフトウェアを」であり、ドキュメント不要とは言っていない。

### 設計書の3大原則

#### 原則1: 「なぜ・何を」の構成

仕様（何を）だけでなく、目的（なぜ）を必ず併記する。

- 「なぜ」がわかると仕様変更時の判断基準になる
- 目的を知らずに実装すると、手段が目的化する
- 例: 「ログイン失敗3回でロック」という仕様だけでは不十分。「不正アクセスを防ぐため（なぜ）、連続ログイン失敗3回でアカウントをロックする（何を）」と書く

#### 原則2: 「全体から部分へ」の展開

まずシステム概要・構成図を示し、次に機能一覧、個別機能の詳細へとドリルダウンする。

- 読者が「自分が今どこを読んでいるか」を常に把握できる
- 詳細だけを見せられても、文脈なしでは理解できない
- コンポーネント図 → モジュール一覧 → 各モジュールの詳細の順で展開する

#### 原則3: One Fact One Place（ワンファクト・ワンプレイス）

同じ情報を複数箇所に書かない。

- 変更時に1箇所だけ直せば良い状態を保つ
- 「概要に書いたから詳細には書かなくていい」ではなく、「概要には参照先を書き、詳細で定義する」
- 設計書間の参照関係（「詳細は○○.md参照」）を明確にする

## B. システム開発ドキュメントの体系

### V字モデルに沿ったドキュメントフロー

| 工程 | 作成ドキュメント | 対応するテスト |
|------|-------------|-----------|
| 要件定義 | 要件定義書 | システムテスト |
| 基本設計（外部設計） | 外部仕様書（機能設計書） | 結合テスト |
| 詳細設計（内部設計） | 内部仕様書（詳細設計書） | 単体テスト |
| 実装 | ソースコード | - |
| 運用設計 | 運用設計書 | 運用テスト |

各ドキュメントは左のフェーズで作成し、対応するテストの「何を確認するか」の基準となる。

### 令和時代のドキュメントの変化

- **Excelベースの設計書 → 設計ツール・Wiki・Markdownへの移行**
  梅田弘之の指摘「システム開発だけCADを使わない不思議」。建築や機械設計では専用ツールで設計するのに、ソフトウェアだけExcelで設計する文化は変わりつつある。

- **データ中心設計（DOA）の重要性**
  処理ではなくデータ構造から設計する発想。正規化されたデータモデルは設計書の核になる。

- **リポジトリで設計書を管理する流れ**
  ソースコードと同じリポジトリに設計書を置き、PRレビューで品質を担保する。「コードは変わったが設計書が古いまま」を防ぐ。

## C. 要件定義書テンプレート

### 表紙・更新履歴

```
表紙: システム名、版番号、発行日、部署名と名前
更新履歴: 版番号、日付、更新者、更新内容
```

### 目次構成

```
要件定義書
├── はじめに
│   ├── 開発の目的          ← システムの必要性、解決したい課題（なぜ）
│   └── 定義                ← 用語・略語の定義
├── 業務要件
│   ├── 業務実施手順        ← 業務の概要とフロー、業務の一覧、業務の体制、扱う情報
│   ├── 規模                ← ユーザー数、処理件数
│   ├── 時期・時間          ← 業務時間、繁忙期
│   ├── 場所                ← 業務実施場所・設備
│   └── システム化の範囲    ← 対象となる業務
├── 機能要件
│   ├── システム概要        ← システムの概要説明
│   │   └── システム構成図  ← 全体の構成図
│   ├── 機能                ← 機能の一覧、各機能の説明と利用目的、機能同士の連携
│   ├── 画面                ← 各画面のイメージ、説明と利用目的、画面遷移図
│   ├── 帳票                ← 各帳票のイメージ、説明と利用目的
│   ├── データ              ← データの一覧、利用者、利用目的、公開範囲、処理内容
│   └── 外部インタフェース  ← 一覧、接続先、送受信データ、タイミング、方式
└── 非機能要件
    ├── ユーザビリティとアクセシビリティ ← ITリテラシー、利用設計
    ├── システム方式        ← ハードウェア、ネットワーク、技術スタック
    ├── 規模                ← サーバー数、データ量、処理件数、ユーザー数
    ├── 性能                ← 応答時間、スループット
    ├── 信頼性              ← 稼働率、バックアップ
    ├── 継続性              ← 障害発生時の復旧
    ├── 上位互換性          ← OS・ミドルウェアのバージョンアップ対応
    ├── 情報セキュリティ    ← セキュリティレベル
    ├── 稼働環境            ← システム稼働の環境要件
    ├── テスト              ← テスト種類、確認事項
    ├── 教育                ← ユーザー教育要件
    ├── 運用                ← メンテナンス、サポート体制
    └── 保守                ← 障害対応、不具合修正
```

### 記述のポイント

- **「はじめに」の開発目的**で、システムの必要性と解決したい課題を記載する。ここが薄いと、後続の仕様がなぜ必要かわからなくなる
- **業務要件では「全体から部分へ」**の流れでユーザーの業務を説明する。業務フロー図を先に示し、各ステップの詳細に進む
- **業務と機能の紐付きを明確に**。機能の使われ方を開発メンバーが理解できるようにする
- **機能が多い場合は大区分→小区分に分類**して一覧にする。フラットな100機能一覧より、カテゴリ別5×20のほうが把握しやすい
- **各機能の利用目的も業務に紐付けて記載**し、その機能がなぜ必要かがわかるようにする

## D. 外部仕様書（機能設計書）テンプレート

### 表紙・更新履歴

```
表紙: 「外部仕様書」、システム名、版番号、発行日、部署名
更新履歴: 版番号、日付、更新者、更新内容
```

### 目次構成

```
外部仕様書
├── はじめに
│   ├── 開発の目的          ← なぜ
│   └── 定義                ← 用語
├── システムの概要
│   ├── システム構成図
│   └── 機能の一覧          ← 機能一覧と概要
├── 機能の詳細              ← 機能一覧の階層構造に沿って展開
│   └── （機能名）
│       ├── 機能の概要
│       ├── ユースケース    ← 利用シーン・ユースケース（なぜ）
│       ├── 画面仕様        ← モックアップ、画面遷移、各項目の名前・種別・データ型・必須/任意・入力制限
│       ├── 項目仕様        ← 各項目の詳細仕様
│       ├── エラー          ← エラー一覧（発生場所・条件・メッセージ）
│       └── ログ            ← ログ一覧（出力条件・ログレベル・メッセージ）
└── 非機能要求の詳細
    └── セキュリティ
```

### 記述のポイント

- **仕様を定義するだけでなく、それぞれの仕様を「なぜ」と結び付ける**。仕様（何を）とユーザーシナリオ（なぜ）が結び付くことで、目的に合った仕様になっているかがわかりやすくなる
- **アジャイル開発では「なぜ」の認識を開発メンバーと合わせておく**ことが仕様の継続的な見直しにつながる
- **ユーザーストーリーフォーマット**:「（機能の利用者）として、（達成したいこと）をしたい。なぜなら（理由）だからだ。」
- Figmaなどのツールでモックアップを描く場合はリンクを記載する

## E. 機能設計書の標準化

### 機能設計書の標準フォーマット

| 要素 | 内容 | ポイント |
|------|------|---------|
| **表紙** | 設計書の全体像を把握させる「表紙」ページ | 各ページの関連性を示す。ないと読者が文脈を失う |
| **概要説明** | 機能の目的・背景を1ページにまとめる | これがないと読者が文脈を理解できない |
| **画面レイアウト** | ワイヤーフレーム→モックアップ→プロトタイプ | 順を追って精度を上げる |
| **コントロール一覧** | 各コントロールの項目名・型・制約を一覧化 | テキストボックス、ボタン等の詳細定義 |
| **イベント定義** | 各イベント（ボタンクリック等）で実行される処理 | ユーザー操作と処理の対応表 |
| **ロジック（モジュール）** | 処理の具体的なロジックをモジュール単位で記述 | 実装者が迷わない粒度で書く |

### One Fact One Place の徹底

- **同じ情報を複数箇所に書かない**
- 変更があったときに1箇所だけ直せば良い状態を保つ
- 設計書間の参照関係を明確にする（「詳細はX機能設計書のY章参照」）
- コピペで複製した仕様は、どちらかが更新された時点で不整合が生まれる

## F. 非機能要件の定義

非機能要件は後回しにされがちだが、要件定義時に定めないとテスト・運用フェーズで問題が噴出する。

### 定義する項目

| カテゴリ | 定義項目 |
|---------|--------|
| **可用性** | 稼働率（99.9%等）、計画停止ウィンドウ、復旧目標時間（RTO）、復旧目標ポイント（RPO） |
| **性能・拡張性** | レスポンスタイム（3秒以内等）、スループット（件/秒）、同時接続数、将来の拡張性 |
| **運用・保守性** | 監視要件、バックアップ方式・世代数、メンテナンスウィンドウ |
| **移行性** | データ移行方式、移行期間、切り替え方式（段階的 or 一括） |
| **セキュリティ** | 認証・認可、暗号化方式、監査ログ、脆弱性対応方針 |
| **環境・エコロジー** | サーバー設置環境、電源、空調（オンプレミスの場合） |

### 記述のポイント

- **定量的に記述する**。「速い」ではなく「3秒以内」、「信頼性が高い」ではなく「稼働率99.9%（月間ダウンタイム43分以内）」
- **優先順位をつける**。すべてをMAXにはできない。性能と費用はトレードオフであることを明示する
- **チェックリストで漏れを防ぐ**。非機能要件は「考えればわかる」ではなく、チェックリストを使って網羅する

## G. 運用設計書

運用設計書は「システムが本番稼働してから使うドキュメント」。開発フェーズで作成し、運用チームに引き継ぐ。

### 3部構成

```
運用設計書
├── I. 基本方針・構成管理・運用体制
│   ├── 運用設計書の目的と内容
│   ├── 構成管理（ハードウェア、ソフトウェア、ネットワーク構成）
│   └── 運用体制（役割分担、エスカレーションフロー）
├── II. セキュリティ・スケジュール・稼働監視
│   ├── セキュリティ設計（認証、暗号化、ゼロトラスト）
│   ├── 運用スケジュール（定期メンテナンス、計画停止）
│   ├── 稼働監視（監視項目、閾値、アラート、キャパシティ管理）
│   └── IDS/IPS
└── III. 障害・災害対応・バックアップ・ジョブ管理
    ├── 障害対応（インシデント管理、障害管理、人為的ミス対策）
    ├── 災害対応（BCP: Business Continuity Plan）
    ├── バックアップ（方式、世代管理、リストア手順）
    └── ジョブ管理（バッチ処理スケジュール）
```

### 運用設計書で特に重要な項目

- **エスカレーションフロー**: 誰に・何を・いつ報告するか。これが曖昧だと障害対応が混乱する
- **監視項目と閾値**: 何を監視し、どのレベルでアラートを出すかを明示する
- **リストア手順**: バックアップが取れていても、リストアできなければ意味がない。手順を実際にテストする

## H. アジャイル開発の設計書

アジャイル開発は「ドキュメントを書かない」のではなく「必要十分なドキュメントを書く」。

### アジャイルとドキュメントの関係

| 規模 | 推奨ドキュメント量 |
|------|----------------|
| 小規模（数人・数ヶ月） | README + ADR + スプリントバックログ |
| 中規模（10人・1年程度） | 上記 + 機能設計書（軽量版）+ 非機能要件定義 |
| 大規模・請負契約 | 従来型の設計書体系（要件定義書、外部仕様書）+ アジャイル実践 |

### 反復型開発におけるドキュメントの進化

- **最初は薄く、イテレーションごとに充実させる**
- スプリント0でアーキテクチャの骨格ドキュメントを作成する
- 各スプリントで変更があった部分のドキュメントを更新する

### アジャイル開発で有効なドキュメント形式

- **ユーザーストーリー**: 「（利用者）として、（達成したいこと）をしたい。なぜなら（理由）だからだ。」
- **受け入れ基準**: ユーザーストーリーが「完了」と判断できる条件を明示
- **ADR（Architecture Decision Records）**: なぜその設計を選んだかの決定記録。「なぜ○○ではなく△△を選んだか」を残す

### 請負契約とアジャイルの折り合い

日本の商習慣では、請負契約に基づき成果物（ドキュメント）を契約書に明記することが多い。この場合:
- 要件定義書・外部仕様書は契約上の成果物として作成する
- 内部ドキュメント（ADR、技術メモ等）はアジャイルの軽量形式を採用する

## I. 設計書のアンチパターン

| アンチパターン | 問題 | 対策 |
|-------------|------|------|
| **「なぜ」がない設計書** | 仕様変更の判断基準がわからない | 各機能にユースケース・目的を併記 |
| **いきなり詳細から書く** | 読者が全体像を把握できない | 「全体から部分へ」の原則で構成 |
| **コピペの繰り返し** | 変更時に不整合が発生 | One Fact One Place |
| **設計書が信頼できない** | 実装と設計書の乖離が起きる | 設計書をリポジトリで管理、レビュープロセスを設ける |
| **非機能要件の後回し** | テスト・運用フェーズで問題噴出 | 要件定義時にチェックリストで網羅 |
| **概要説明の省略** | 文脈なしで詳細だけ見せられても理解できない | 表紙・概要ページを必ず作成 |
| **設計書を書かないアジャイル** | 保守・引き継ぎが困難 | 「必要十分な」ドキュメントを継続更新 |
| **定性的な非機能要件** | テスト基準が定まらない | 「速い」ではなく「3秒以内」と定量化 |

## J. 設計書のチェックリスト

要件定義書・仕様書作成時の確認事項:

### 全体構成
- [ ] 「はじめに」に開発目的（なぜ）が記載されている
- [ ] 用語・略語が定義されている
- [ ] 更新履歴が管理されている

### 業務要件
- [ ] 業務要件が「全体から部分へ」の流れで説明されている
- [ ] 業務と機能の紐付きが明確

### 機能要件
- [ ] 機能が階層構造（大区分→小区分）で整理されている
- [ ] 各機能にユースケース（利用シーン・なぜ）が記載されている
- [ ] 画面仕様に項目定義（名前・型・必須/任意・制約）がある
- [ ] エラー一覧が定義されている

### 非機能要件
- [ ] 非機能要件が定量的に記述されている（「速い」ではなく「3秒以内」）
- [ ] セキュリティ要件が明記されている
- [ ] 可用性・性能・運用・移行の各カテゴリが網羅されている

### 一貫性
- [ ] One Fact One Place が守られている
- [ ] 設計書間の参照関係が明示されている
