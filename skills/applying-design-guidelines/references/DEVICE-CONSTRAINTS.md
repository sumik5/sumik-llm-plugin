# デバイス制約リファレンス

マルチデバイスUI設計における物理的制約のリファレンス。
デバイスの特性を理解し、入力手段・画面・プラットフォームの違いに応じた設計判断を行うための指針。

---

## 1. デバイスバリエーション

### 4デバイスの特性マトリクス

| 特性 | スマートフォン | タブレット | PC | TV |
|------|-------------|-----------|-----|-----|
| 入力手段 | タッチパネル | タッチパネル | キーボード + マウス/トラックパッド | リモコン |
| 画面サイズ | 小（4〜7インチ） | 中（7〜13インチ） | 大（11〜27インチ） | 超大（32インチ〜） |
| 画面の可変性 | 固定（フルスクリーン） | 固定（フルスクリーン） | 可変（ブラウザ） | 固定（フルスクリーン） |
| 主な向き | 縦（時に横） | 縦・横どちらも | 任意 | 横のみ |
| 画面との距離 | 至近（〜40cm） | 近い（〜50cm） | やや近い（〜70cm） | 遠い（1.5m〜） |
| 代表アスペクト比 | 16:9、18:9 | 4:3、16:10 | 16:9、16:10 | 16:9（固定） |
| 文字入力 | ▲ やや難（ソフトキーボード） | ▲ やや難 | ● 容易（ハードキーボード） | × 難（リモコン） |
| 操作精度 | ▲ やや粗（指） | ▲ やや粗（指） | ● 細かい（マウス） | × とても粗（フォーカス移動） |
| デザイン要素の大きさ | やや大きめ | 大きめ | 小さめ可 | 非常に大きめ |
| プライバシー性 | 個人利用 | 個人・共用 | 主に個人 | 多人数で共視聴 |

**設計判断の出発点**: デバイスの入力手段が決まれば、要素サイズ・ホバー有無・直接選択の可否が自動的に決まる。

## 2. 入力手段の違い

### 2-1. マウス / トラックパッド（PC）

| 特性 | 詳細 |
|------|------|
| 操作精度 | 高い（1px単位の操作が可能） |
| スクロール | **縦が得意**（ホイール・2本指スクロール）。横スクロールは不快 |
| ホバー | **使用可能**（クリック前に状態フィードバック可） |
| 選択方式 | 直接選択（1アクションで確定） |

**PCデザイン指針:**
- 要素を小さくレイアウトしてよい（マウスの高精度を活かせる）
- ホバーで操作可否・状態を補助的に伝えられる（視覚デザインの自由度が広がる）
- 縦に長いレイアウトで情報密度を高められる
- 横スクロールは避ける（縦スクロールのみ設計を基本とする）

### 2-2. タッチパネル（スマートフォン・タブレット）

| ジェスチャー | 操作 | 設計上の注意 |
|------------|------|------------|
| タップ | 要素を選択・確定 | タップ領域は最小44×44px（iOS HIG基準）以上 |
| ダブルタップ | ズームイン・特殊操作 | 誤操作が多い。必須操作には使わない |
| スワイプ | 画面遷移・スクロール | 方向（縦/横）はアプリ内で一貫させる |
| ピンチ | ズーム操作 | 地図・画像閲覧などで有効 |
| ロングプレス | コンテキストメニュー | 発見性が低い。補助機能として使う |

**タッチデザイン指針:**
- **ホバーは存在しない**。操作可否は「見た目だけで判断できる」ことが必須
- ボタン・リンクは指で触れられる十分なサイズ（最低44dp / 48dp）を確保
- 要素間の余白を大きくとり、誤タップを防止
- テキスト入力はソフトキーボードが画面の下半分を占有することを前提にレイアウトを設計

#### 親指ゾーン（片手操作の設計）

スマートフォンを片手で持ったとき、握り直しなしで親指が届く範囲は限られる。

```
【親指可動範囲の目安】
┌────────────────────┐
│ × 届かないゾーン（画面上部） │
├────────────────────┤
│ △ 届きにくいゾーン（画面中央上）│
├────────────────────┤
│ ● 届くゾーン（画面中央〜下部）│
└────────────────────┘
```

**適用ルール:**
- 主要なナビゲーション・CTA ボタンは画面下部（届くゾーン）に配置
- 重要でない情報・補助的な要素は上部に配置してよい
- 横幅が狭い端末ほど片手操作しやすい（狭額縁設計の背景）

### 2-3. リモコン（TV）

| 特性 | 詳細 |
|------|------|
| 基本操作 | 十字キー（上下左右）+ 決定ボタン + 戻るボタン |
| 選択方式 | **間接選択**（フォーカスを移動してから確定）。直接選択は不可 |
| フォーカス移動 | 連続的に隣の要素へ移動。要素が孤立していると「瞬間移動」が発生 |
| 音声入力 | 多くのリモコンがマイク搭載。文字入力の代替手段として有効 |

**TVデザイン指針:**
- フォーカスの現在位置を明確に示す（枠・背景色反転・サイズ変化を2つ以上組み合わせる）
- 要素を**連続してレイアウト**する（孤立した要素はフォーカス移動が困難）
- フォーカス移動を「縦のみ」または「横のみ」の1方向に単純化すると操作負担が減る
- 重要なUI要素は**セーフゾーン（画面内側90%）**に配置（メーカー固有の表示と干渉防止）
- 文字入力が必要な箇所には音声入力を優先的に提供する

### 2-4. 入力手段別 設計指針まとめ

| 設計要素 | PC（マウス） | スマホ/タブレット（タッチ） | TV（リモコン） |
|---------|------------|------------------------|--------------|
| 最小タップ/クリック領域 | 特に制約なし | 44dp × 44dp 以上 | 要素間の連続性を重視 |
| ホバー活用 | ✅ 積極的に活用 | ❌ 使用禁止 | △ フォーカス状態で代替 |
| 選択方式 | 直接選択 | 直接選択 | 間接選択（フォーカス移動） |
| スクロール方向 | 縦推奨（横は避ける） | 縦推奨（横は機能次第） | 縦か横の1方向に統一 |
| 要素サイズ | 小さくても可 | 大きめ必須 | 非常に大きめ必須 |
| 文字入力 | 容易 | ソフトキーボード対応 | 最小化・音声入力推奨 |
| 操作の発見性 | ホバーで補助可 | 見た目だけで判断可能に | フォーカス時の視覚差別化 |

## 3. 画面の違い

### 3-1. 画面サイズと可変性

| 要素 | 考慮すべきポイント |
|------|----------------|
| 可変性 | PCのみブラウザでリサイズ可能。他3デバイスはフルスクリーン固定 |
| アスペクト比 | デバイス固有で変更不可。レイアウト設計の出発点になる |
| サイズ | 大きいほど情報量を増やせるが、距離の影響で体感サイズが変化 |

### 3-2. 画面との距離と視覚的体感サイズ

距離が異なると、画面上の物理的なサイズと体感サイズが変わる。

| デバイス | 代表的な視聴距離 | デザイン要素の大きさ |
|---------|--------------|-----------------|
| スマートフォン | 30〜40cm | 小さめでも明瞭に見える |
| タブレット | 40〜60cm | スマホより少し大きめに |
| PC | 50〜80cm | 中程度 |
| TV | 1.5〜3m | 非常に大きく設計必須 |

> **視角の原則**: 画面が大きくても距離が遠ければ体感サイズは小さい。TVの文字・UIは極端に大きくしないと読めない。

### 3-3. アスペクト比とレイアウト設計

| 比率 | 代表デバイス | レイアウトの特徴 |
|------|-----------|--------------|
| 4:3 | iPad（一部） | 正方形に近い。縦横どちらにも余裕がある |
| 16:9 | スマートフォン、TV | 横に広い。横向きコンテンツ（動画等）に適合 |
| 16:10 | 一部タブレット・PC | 16:9より少し縦長。作業用途に適合 |
| 18:9以上 | 狭額縁スマートフォン | 縦に長い。縦スクロールコンテンツに適合 |

**設計ポイント:**
- スマートフォンは縦向きがデフォルト。縦スクロールを基本設計に据える
- タブレットは縦横どちらでも使われる。両方の向きでレイアウトを検討する
- TVは横向き固定・16:9固定。横幅をフルに活用したレイアウト設計が前提

### 3-4. 画面の向きと機能設計

| 向き | スクロール方向 | 代表的な活用 |
|-----|-------------|-----------|
| 縦向き（portrait） | 縦スクロールが自然 | フィード閲覧・記事読了・設定画面 |
| 横向き（landscape） | 横スクロールor縦スクロール | 動画視聴・横幅を活かした2カラム表示 |

- 同じアプリでも縦横で**提供する機能ごと変えてよい**（縦：イベント一覧 / 横：タイムスケジュール 等）
- 縦横で大きくレイアウトを変化させることは、「レイアウトが収まらない問題」の解決策になる

## 4. Android vs iPhone

### 4-1. ナビゲーション設計の根本的差異

2プラットフォームの最大の違いは「戻る手段」の違いにある。

| 要素 | Android | iPhone (iOS) |
|------|---------|-------------|
| 戻るボタン | OS標準で画面下部にハードウェアボタン（または画面端スワイプ） | ❌ ハードウェアボタンなし |
| ホームに戻る | ホームボタン or ジェスチャー | ホームボタン or 画面下端フリックアップ |
| 「戻る」の実装 | OSの戻るボタンに委ねる設計が可能 | アプリ内に「← 戻る」ボタンを常設する必要あり |
| タブ位置の慣習 | 画面上部（上タブバー） | 画面下部（下タブバー） |

### 4-2. 設計への影響

**ヘッダーの設計:**
- **iOS**: 左上に「← 戻る」ボタンを常設（ないと前画面に戻れない）
- **Android**: 左上に「戻る」ボタン不要。ハンバーガーメニューなど他機能を配置できる

**タブ・ナビゲーションの位置:**
- **iOS**: タブは画面下部（Bottom Navigation）。親指が届く位置に主要操作を集める
- **Android**: タブは画面上部も一般的。OS標準の戻るボタン（画面下部）を活用できる

**ジェスチャー:**
- **iOS（iPhone X以降）**: ホームへ戻る = 画面下端フリックアップ。アプリ切替 = 画面下端スワイプ
- **Android**: 端末・バージョンによってジェスチャー設定が異なる。多様性を前提に設計する

### 4-3. クロスプラットフォーム設計の判断テーブル

| 状況 | 推奨設計 |
|------|---------|
| 両OS共通ナビゲーションを作る | iOS基準（戻るボタン常設）で設計し、Android側でもそれを使う |
| OSネイティブ操作を優先する | 各OSのデザインガイドライン（HIG / Material Design）に準拠 |
| タブ配置を決める | iOS配慮で**下タブ推奨**（Androidでも下タブは問題なく使える） |
| ジェスチャーに機能を割り当てる | iOS/Androidで同一ジェスチャーに異なる意味を持たせない |

## 5. まとめ：デバイス制約チェックリスト

設計開始前に以下を確認する:

- [ ] 対象デバイスは何か？（PC/スマートフォン/タブレット/TV）
- [ ] 入力手段は何か？（マウス/タッチ/リモコン）
- [ ] ホバーを使えるか？（PC: Yes / タッチ・TV: No）
- [ ] 直接選択か間接選択か？（PC/タッチ: 直接 / TV: 間接）
- [ ] 最小タップ領域を確保しているか？（タッチ: 44dp以上）
- [ ] 親指ゾーンに主要操作を配置しているか？（スマートフォン）
- [ ] フォーカス移動の連続性を確保しているか？（TV）
- [ ] セーフゾーン（内側90%）内に重要要素があるか？（TV）
- [ ] 画面の向き（縦/横）による機能変化を設計しているか？
- [ ] AndroidとiOSの「戻る手段」の違いに対応しているか？
