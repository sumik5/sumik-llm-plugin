# プラットフォームの影響と人間の認知特性

マルチデバイスUI設計における「ソフトウェア制約」と「人間の認知特性」の実践ガイド。

---

## 1. ソフトウェアレイヤーの基本構造

ハードウェア → OS → Web（ブラウザ）/ アプリ → コンテンツ（テキスト/画像/動画）

OSの違いよりも、**ハードウェア構成の違いのほうがUIデザインへの影響が大きい**。
TVとスマートフォンはOSが異なっていても、入力手段が同じなら設計は近くなる。

---

## 2. Web vs ネイティブアプリ

### 技術基盤と特性の比較

| 観点 | Web（ブラウザ） | ネイティブアプリ |
|------|---------------|----------------|
| 実装技術 | HTML / CSS / JavaScript | Swift (iOS) / Kotlin (Android) |
| ディストリビューション | URLで即アクセス（インストール不要） | ストア経由でインストール |
| マルチデバイス対応 | レスポンシブデザインで1コードベース対応可 | OS別に個別実装が必要 |
| OS間の差異 | ブラウザが差異を吸収 → ほぼ同一UI | ハードウェア差異が直接UIに影響 |
| インタラクション | ジェスチャー対応が困難。操作が重い | タッチジェスチャー（スワイプ/フリック等）が豊富 |
| パフォーマンス | 操作速度・レスポンスが劣る | 動作が軽快。通信量を最小化できる |
| 画面端スワイプ | 実装困難（ブラウザのUIと競合） | 画面端からのスワイプ、TinderUIなど実現可能 |
| 中断・再開 | タブを閉じると状態消失しやすい | 中断・再開が容易 |
| 更新 | サーバー更新で即反映 | ストアレビュー + ユーザーのアップデート待ち |

### PWAの位置付け

**PWA (Progressive Web Apps)**: Webの技術基盤でアプリのような体験を実現する中間形態。

| 特性 | 説明 |
|------|------|
| インストール | ホーム画面に追加（ストア不要） |
| オフライン対応 | キャッシュ活用でオフライン動作可 |
| プッシュ通知 | Webサイトでは不可だが PWA では可能 |
| 制約 | アプリほどのジェスチャーインタラクションは不可。iOS では「戻る」操作の設計に注意が必要 |

### プラットフォーム選択の判断基準

| 判断軸 | Web を選ぶ | アプリを選ぶ |
|--------|-----------|------------|
| 展開範囲 | マルチデバイス・マルチOS | 特定デバイス専用機能 |
| インタラクション | シンプルな操作で十分 | 豊富なジェスチャーが必要 |
| ユーザー獲得 | まず試してもらいたい（インストール障壁なし） | 継続利用が前提（ホーム常駐） |
| 開発コスト | 1コードベースで広くカバーしたい | デバイス最適化を優先したい |
| パフォーマンス | 許容できる速度で十分 | 操作の軽快さが重要 |

> **注意**: アプリとWebで最適なインターフェースデザインは根本的に異なる。アプリで最適化したUIをWebに流用しても同一には実現できない。設計初期段階でプラットフォームを確定させること。

## 3. メディア最適化

### テキスト

解像度に依存せず、OSとブラウザが常に最適表示する。**最も安定したデータ形式。**

| メリット | 詳細 |
|---------|------|
| 解像度自由 | @1x〜@4x まで全解像度で鮮明表示 |
| アクセシビリティ | コピー可・機械翻訳対応・スクリーンリーダー対応 |
| 耐久性 | 技術進化に対しても経年劣化しない |
| ファイルサイズ | 極めて軽量 |

**設計指針**: テキストで表現できるものは画像化しない。特にボタン・メニュー・ラベルは必ずテキストで実装する。

### 画像

| 種別 | 形式 | 適用場面 | 解像度対応 |
|------|------|---------|-----------|
| ラスター | JPEG / PNG / GIF / WebP | 写真・複雑なグラフィック | @1x/@2x/@3x/@4x の複数バリエーション必要 |
| ベクター | SVG | アイコン・ロゴ・シンプルな図形 | 解像度不問。環境に応じて最適表示 |

**解像度対応の現実的アプローチ**:
- 最低限 @2x 画像を用意し、等倍は縮小・3倍以上は拡大で対処（差異は人間に認知困難）
- アイコン・ロゴ・図形は積極的に SVG を採用
- `<img srcset="@1x.png 1x, @2x.png 2x, @3x.png 3x">` でレスポンシブイメージに対応

### 動画

- 大多数は 16:9 の横長固定。縦長動画はスマートフォン撮影コンテンツに限定的
- デバイスが大きいほど視聴頻度が高く、大画面ほど複数人での視聴に適する
- **バックグラウンドムービー**: 補助的演出として有効だが、パフォーマンスへの影響に注意
- 自動再生は通信量と注意力を消費するため、デフォルト OFF + ミュートが原則

## 4. パフォーマンスと体感速度

表示速度の低下は離脱率上昇・ユーザー満足度低下に直結する。**表示速度はUIの一部**として設計する。

| 改善手法 | 説明 |
|---------|------|
| HTTP/2 | 複数リクエストの非同期並列処理（HTTPS 必須） |
| リソースの先読み | Resource Hints APIで次ページのファイルを事前取得 |
| SVG活用 | ラスター画像の複数バリエーション管理コストを削減 |
| コンテンツの最小化 | 不要なJavaScript削減・画像の最適化 |

**プラットフォーム別パフォーマンス特性**:
- アプリはネイティブ実装のため、Webより操作速度で有利
- Webのリッチインタラクションはパフォーマンスを悪化させやすい（特にスマートフォン）
- 体感速度の向上にはローディングアニメーション・スケルトンスクリーンの活用も有効

## 5. 色の認知特性（プリアテンティブ処理）

### 瞬間的な把握力

人間は色の違いを**瞬間的かつ並列**に処理する（プリアテンティブ処理）。
要素数が25個でも100個でも、色で区別された要素を探す時間はほとんど変わらない。

これに対して、**色の数（バリエーション）が増えると効果は急速に低下する**。
複数の色が混在すると、特定の色を探す時間は要素数に比例して増加してしまう。

### 色数制限の原則

| 原則 | 説明 |
|------|------|
| 無彩色 + 少数のキーカラー | 基本は無彩色で構成し、強調・分類に少数の色を使う |
| 用途ごとに色を固定 | 現在地・警告・強調など、意味と色の対応を一貫させる |
| 増やすより減らす | 色は後から減らすのが困難。最初から絞り込む |
| デザインガイドラインで管理 | 色の増殖を防ぐために運用ルールを文書化する |

### 効果的な色彩設計の用途

| 用途 | 効果 |
|------|------|
| 現在地の表示 | ナビゲーションの選択状態を即座に把握できる |
| カテゴリ分類 | 同種の要素を同色にすることで情報の塊を認知させる |
| フォーカス・選択状態 | 色反転で「今どこにいるか」を明示（特にTV/リモコン操作に重要） |
| 強調・警告 | 単一の強調色で重要情報を目立たせる |

## 6. 形の認知特性

形状の違いも瞬間的に認知できるが、**十分に異なる形状の場合のみ有効**。

| 条件 | 認知 |
|------|------|
| 類似した形状（R と P など） | 1つずつ順番に確認が必要（順列的処理） |
| 十分に異なる形状（R と Z など） | 瞬間的に特定できる（プリアテンティブ処理） |

**設計原則**:
- 選択状態・ホバー・フォーカス時に**色と形の両方**を変化させると認知が強化される
- 色覚障害ユーザーにも形状の差異は有効 → アクセシビリティの観点でも形状変化は重要
- アイコンセットは「十分に異なる形状」で設計する。類似アイコンの多用は判別困難を招く

## 7. 動きの認知特性

動きや変化には意識せずとも注目してしまう本能的特性がある。

### 動きの3つの役割

| 役割 | 説明 | 注意点 |
|------|------|--------|
| **注目のための動き** | 要素にアテンションを集める（点滅・スライド・バウンス等） | 多用すると煩わしくなる。「理解のための動き」と組み合わせると効果的 |
| **理解のための動き** | 「A → B」の状態変化を体感的に理解させる（ページ遷移アニメーション等） | UIの動きの中核。あるとないとでは分かりやすさが大きく異なる |
| **装飾のための動き** | 演出・ギミック・パララックス等。存在しなくても機能する | 魅力づくりには有効だが、パフォーマンスへの影響に注意 |

### デバイス別の動き設計

| デバイス | 推奨アプローチ |
|---------|--------------|
| スマートフォン | 短いトランジションで遷移の方向・階層を示す（iOSの左スライド等） |
| PC | ホバー時の色変化・下線変化で「押せる対象」を明示 |
| TV | フォーカス移動時の強調アニメーションで現在位置を明確化 |

**使いすぎのリスク**: アニメーションを多用すると認知阻害・パフォーマンス低下が起きる。
常に「このアニメーションは理解を助けているか？」を問う。

## 8. インタラクションコスト

インタラクションコスト = **精神的負荷（認知コスト）+ 身体的負荷（操作コスト）**

ユーザビリティの指標であり、この合計を最小化することがデザインの目標。

### 精神的負荷（頭のコスト）

読む・理解する・覚える・判断する、すべてが精神的負荷。

| 増加要因 | 例 |
|---------|-----|
| 小さすぎる文字・長すぎる行 | 読みにくいテキスト |
| ボタンかどうか判別しにくい | 見ただけでは押せるか分からないUI |
| 選択肢が多すぎる | 選ぶのに時間がかかる |

### 身体的負荷（体のコスト）のデバイス差

同じ操作でも、デバイスによって身体的コストは**桁違いに異なる**。

| 操作 | PC | スマートフォン | TV（リモコン） |
|------|----|--------------|--------------:|
| アイコン選択 | マウス移動 + クリック（小） | 指を伸ばしてタップ（中） | 方向キー連打 + 決定（大） |
| テキスト入力 | キーボード（小） | フリック入力（中） | 一文字ずつ決定（非常に大） |
| ページ内移動 | スクロールホイール（小） | フリック（小〜中） | 方向キー繰り返し（大） |

**TV設計の重要原則**: リモコン操作のコストは他デバイスより圧倒的に高い。
テキスト入力・検索・複雑な選択を要求しない設計を優先する。

### コスト最小化の3原則

「ゴールまでのステップ数を減らす」のではなく、「ゴールまでの認知・身体的負荷の合計を最小化する」が正しい考え方。

| 原則 | 説明 |
|------|------|
| **一貫性** | 予測可能な操作。学習済みパターンの再利用でコスト低減 |
| **シンプルさ** | 認知リソースを有効活用。余計な要素を削ぎ落とす |
| **共通概念** | 既知のパターン（慣習）を活用して学習コストをゼロに近づける |

## 9. 一貫性の原則

### 内部一貫性 vs 外部一貫性

| 種別 | 説明 | 効果 |
|------|------|------|
| **内部一貫性** | サービス内のルール・ビジュアルの統一 | ひとたび学習すれば新機能の操作コストが低い |
| **外部一貫性** | 業界標準・他サービスとの共通パターン | 初回から「知っている操作」でサービスを使える |

一貫性のメリット:
1. ユーザーが操作を予測できる（認知コスト低減）
2. 機能追加・改善コストの削減（既存コンポーネント再利用）
3. 安心感・信頼性の向上

### 一貫性と柔軟性のバランス

一貫性（ルール）と柔軟性（自由度）はトレードオフ。100% の一貫性は硬直し、100% の自由は無法地帯になる。

**設計指針**:
- ルールは**最初に大きなものから**決める（後から変更するほど難しい）
- 例外は「枝葉」にとどめ、根幹のルールは動かさない
- マルチデバイス展開時: デバイス間で共通ルールを持つと、デバイス切替時の学習コストが低減される

### ヒックの法則との関連

選択肢の数が増えるほど決定までの時間は増加する（ヒックの法則）。
一貫性が高いUIは「すでに知っているパターン」を使えるため、選択肢が多くても認知負荷が低い。

## 10. シンプルさの原則

**シンプル ≠ 単純。シンプル = 明快・意図が即座に伝わる状態。**

| 誤解 | 正しい理解 |
|------|-----------|
| 要素を削ればシンプルになる | 削ること自体がリスク。削る根拠となる意図の理解が先 |
| シンプルは誰にでも分かりやすい | 習熟度が高いユーザーにはある程度の複雑さが効率的 |
| シンプルは簡単に作れる | 本来とても難しい。引き算の設計は高度な技量が要る |

**複雑さが有効な場面**: 専門業務・反復操作・即時性が求められる操作（医療機器・制御パネル・音楽編集等）。
利用者の習熟度によってシンプルと複雑の優劣は逆転する。

## 11. 共通概念（シグニファイア・アフォーダンス）

既知のパターンを活用することで、新しい学習コストをゼロに近づける。

### 定着した共通概念の例

| 概念 | 意味 |
|------|------|
| 青文字・下線テキスト | リンク |
| 虫眼鏡アイコン | 検索 |
| 星アイコン | お気に入り・評価 |
| サービスロゴ | トップページへのリンク |
| フロッピーディスクアイコン | 保存（実物が廃れても概念として定着） |
| 下向き三角 | プルダウン・アコーディオン |

### 共通概念に逆らうリスク

共通概念に反するデザインは**認知負荷を大幅に増加**させる。
例: 「リンクのない下線テキスト（強調目的）」→ ユーザーはクリックして失望する。
　　物理的な手間 + 余計な認知負荷の両方が発生する。

### ユニバーサルデザインの配慮

国・文化によって共通概念は異なる場合がある。

| 文化 | ○の意味 | ✔の意味 |
|------|--------|---------|
| 日本 | 正解・OK | チェック・完了 |
| 欧米圏 | 間違い・空白・否定 | 正解・OK |

グローバル展開時は文化差を考慮したアイコン・サインの選定が必要。
標準的なMaterial Icons / SF Symbolsなどを活用することでユニバーサルデザインに近づける。

### 直感的とは

「直感的 = ユーザーの予測とその結果が完全に一致すること」。
それはユーザーが積み上げてきた**経験と共通概念**に依存する。
シンプルさ・一貫性・共通概念の3つが揃って初めて、直感的なデザインが生まれる。

## まとめ：設計判断のクイックリファレンス

| 設計場面 | 適用する原則 |
|---------|------------|
| プラットフォーム選択 | 機能要件・インタラクション深度・ユーザー獲得方針で判断 |
| アイコン・画像の形式選択 | 写真→ラスター（WebP推奨）、アイコン・ロゴ→SVG |
| 色の使い方 | 無彩色ベース + 少数のキーカラー。増やすより減らす |
| 要素の強調 | 色と形の両方を変化させる（アクセシビリティも向上） |
| アニメーションの追加 | 「理解のための動き」を中核に。装飾は節制する |
| TV向けデザイン | リモコンコストを最優先に考慮。テキスト入力を最小化 |
| 新機能のUI設計 | 既存の共通概念を最大限活用。新パターン学習を強いない |
| 複雑な操作フロー | ステップ数ではなく「精神的+身体的負荷の合計」で評価 |
