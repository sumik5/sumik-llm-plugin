# README.md 作成ガイド

## Quick Start（最小限テンプレート）

```markdown
# プロジェクト名

[![License](https://img.shields.io/badge/license-MIT-blue.svg)](LICENSE)

一言説明（高校生でも理解できるレベルで、専門用語を最小限に）。

## インストール

\`\`\`bash
npm install your-package
\`\`\`

## クイックスタート

\`\`\`bash
your-package --help
\`\`\`

## ドキュメント

詳細ドキュメントは [docs/](docs/) を参照。

## ライセンス

MIT
```

---

## 対象読者の特定

README作成前に対象読者を決定する。

| 対象 | 主なニーズ | 優先セクション |
|------|-----------|--------------|
| **ユーザー** | インストール・使い方・トラブルシューティング | Getting Started・クイックスタート・リファレンス |
| **開発者** | ローカル環境構築・テスト・アーキテクチャ | セットアップ・テスト・コントリビューション |
| **両方** | 概要・バッジ・ライセンス | プロジェクト概要・コミュニティ |

**主要閲覧者の言語**でREADMEを書く。OSS国際プロジェクトは英語、社内ツールは日本語が一般的。

## 必須セクション構成

### 1. プロジェクト概要（名前 + 説明）

- H1（`# プロジェクト名`）は文書に1つだけ
- 説明は1〜3文で完結させる。高校生レベルの読みやすさを目指す
- エコシステムでの位置づけ、類似ツールとの差異を明記する
- ロゴがある場合は説明の直前に配置

```markdown
# my-cli

軽量なJavaScript bundler。esbuildベースでwebpackより5倍高速。
```

### 2. バッジ

バッジは過剰に使わない。**推奨は3〜5個まで**。

```markdown
[![npm version](https://img.shields.io/npm/v/your-package.svg)](https://www.npmjs.com/package/your-package)
[![CI](https://github.com/user/repo/actions/workflows/ci.yml/badge.svg)](...)
[![License](https://img.shields.io/badge/license-MIT-blue.svg)](LICENSE)
```

shields.io 技術スタックバッジフォーマット:
```
https://img.shields.io/badge/-{name}-{color}.svg?logo={logo}&style=for-the-badge
```

### 3. インストール・セットアップ

- 前提条件（Node.js バージョン等）を明記
- コマンドにはコードブロックと言語指定を必ず付ける
- 複数の環境（npm/yarn/pnpm）がある場合はタブで切り替えを示す

```markdown
## インストール

**前提条件**: Node.js 18+

\`\`\`bash
npm install your-package
\`\`\`
```

### 4. クイックスタート / 使い方

- READMEに複数のチュートリアルを含めない（1つに絞るかリンクにする）
- 最短で動作確認できる例を示す
- 期待出力があれば示す

### 5. ドキュメントリンク

外部ドキュメントへのリンクをまとめる。

```markdown
## ドキュメント

- [APIリファレンス](docs/api.md)
- [設定ガイド](docs/configuration.md)
- [移行ガイド](docs/migration.md)
```

### 6. 開発者向け情報

```markdown
## 開発

\`\`\`bash
git clone https://github.com/user/repo
cd repo
npm install
npm test
\`\`\`
```

アーキテクチャの説明が長い場合は `ARCHITECTURE.md` に分離してリンクする。

### 7. コミュニティ / コントリビューション

- バグ報告・機能要望のリンク（GitHub Issues等）
- `CONTRIBUTING.md` へのリンク
- 行動規範（`CODE_OF_CONDUCT.md`）へのリンク

### 8. ライセンス

```markdown
## ライセンス

[MIT](LICENSE) © 2024 Your Name
```

## プロジェクトタイプ別ガイド

### ライブラリ / パッケージ

必須: パッケージ名・バッジ・説明（1〜3文）・インストール・セットアップ・クイックスタート・ドキュメントリンク・コミュニティ・引用

追加推奨:
- 類似ライブラリとの比較表
- ブラウザ/Node.jsサポート表
- Peer dependenciesの明示

### Webアプリケーション

必須: スクリーンショット/デモURL・環境変数テーブル・ローカル起動手順・デプロイ手順

環境変数テーブルの例:

| 変数名 | 目的 | デフォルト値 | 開発環境値 | 注意事項 |
|--------|------|------------|----------|---------|
| `DATABASE_URL` | DB接続文字列 | - | `postgres://...` | 本番は必須 |
| `PORT` | サーバーポート | `3000` | `3000` | - |

### CLIツール

必須: インストール方法（npm/brew/binary）・コマンドリファレンス・設定ファイル説明

コマンドリファレンスの例:

| コマンド | 操作内容 | 等価コマンド |
|---------|---------|------------|
| `make build` | 本番ビルド | `npm run build` |
| `make test` | テスト実行 | `npm test` |

### OSS プロジェクト

追加推奨:
- Shields.ioの動的バッジ（スター数・フォーク数・コントリビューター数）
- GitHub Template repository機能の活用
- 引用セクション（学術・研究用途）

## Markdown 書式規範

### 見出し

- H1は文書に1つのみ（プロジェクト名）
- 見出し階層は4レベル以内（H2〜H4まで）
- ATX形式（`#`記号）で統一
- `#`の後に半角スペース1つ
- 見出しの前後に空行を入れる

```markdown
## セクション名

本文...

### サブセクション
```

### 段落・文章

- 段落は3〜5文、100〜200文字程度
- アクティブな語調で書く（「〜することができます」ではなく「〜します」）
- スペースのみ使用（タブ不可）
- 行の長さ: 英語80文字、日本語100文字を目安

### 強調

- 太字: `**text**`（アスタリスク統一）
- 強調は段落の10%以下に留める
- 必要以上に太字・斜体を使わない

### リスト

- ハイフン（`-`）で統一
- インデントは2〜4スペース
- リスト項目は並列な内容のみ

### コードブロック

言語指定を必ず付ける:

````markdown
\`\`\`bash
npm install package
\`\`\`

\`\`\`typescript
const x: string = "hello";
\`\`\`
````

### 画像

Altテキストを全画像に付ける:

```markdown
![スクリーンショット: ダッシュボード画面](docs/images/screenshot.png)
```

### テーブル

- ヘッダー行を必ず含める
- 各カラムのパイプ（`|`）を揃える
- 内容が長い場合は箇条書きへの変換を検討

### 目次

- 1000文字未満の文書には省略可
- 導入セクションの後・本文前に配置

```markdown
## 目次

- [インストール](#インストール)
- [使い方](#使い方)
- [API](#api)
```

## バッジ活用ガイド

### 推奨バッジ種類

| カテゴリ | バッジ例 | 用途 |
|---------|---------|------|
| バージョン | npm version, PyPI version | パッケージバージョン表示 |
| CI | GitHub Actions status | ビルド・テスト状態 |
| カバレッジ | codecov, coveralls | テストカバレッジ |
| ライセンス | license badge | ライセンス種別 |
| サイズ | bundle size | パッケージサイズ |

### 過剰バッジの回避

避けるべきバッジ:
- 更新が止まったバッジ（古いCI/CDサービス等）
- 意味が重複するバッジ
- 情報密度が低いデコレーション的バッジ

## トラブルシューティングセクションの設計

よくあるエラーと解決策をテーブル形式で示す:

```markdown
## トラブルシューティング

| エラー | 原因 | 解決策 |
|-------|------|--------|
| `ECONNREFUSED` | DBが起動していない | `docker compose up db` を実行 |
| `MODULE_NOT_FOUND` | 依存関係未インストール | `npm install` を再実行 |
```

## 品質チェックリスト

### 公開前チェック

- [ ] H1見出しが1つのみ
- [ ] 全リンクが有効（404なし）
- [ ] コードブロックに言語指定がある
- [ ] 全画像にAltテキストがある
- [ ] テーブルのフォーマットが正しい
- [ ] モバイル表示で読みやすい（行が長すぎない）
- [ ] インストール手順を実際に試した
- [ ] 古いバージョン情報が残っていない

```yaml
title: プロジェクト名
version: "1.0.0"

```

## アンチパターン

| アンチパターン | 問題 | 改善策 |
|--------------|------|--------|
| 複数チュートリアルをREADMEに詰め込む | 長くなり読みにくい | 1つに絞るかdocs/に分離 |
| バッジを10個以上使う | 視覚的なノイズ | 3〜5個に絞る |
| インストール手順がない | ユーザーが使えない | 必ず最短手順を記載 |
| コードブロックに言語指定なし | シンタックスハイライトなし | 必ず言語を指定 |
| 「このREADMEは随時更新されます」 | 不信頼感を与える | 削除。実際に更新する |
| 全文大文字・太字の乱用 | 強調の意味が失われる | 重要な箇所のみに限定 |
| リリースノートをREADMEに書く | 肥大化する | CHANGELOG.mdに分離 |
