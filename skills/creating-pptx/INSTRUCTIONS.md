# creating-pptx — ワークフロー＆手順

## スキルの目的

雑なメモ・箇条書き・既存スライドを「構造」で組み直し、**So What（だから何が言えるか・何をすべきか）まで届けるコンサル品質の1枚スライド**を生成する。

このスキルが行うのは「単なる清書」ではない。与えられた情報から読み手の意図を読み取り、論理構造を選び直し、示唆を引き出すことで、初見の人が 3 秒で主張を理解できるスライドを作る。

**原則：足すのではなく削る。**
余白・情報の差・構造の強さで意思決定を一目で促す。

---

## 入力ハンドリング

ユーザーは `<input>` に以下のいずれかの素材を渡す前提で動作する。

- 箇条書き・メモ・テキスト
- 既存スライドの画像（スクリーンショット等）
- 数値データや表

**素材が空または曖昧な場合**: `AskUserQuestion` を使い、伝えたいこと・対象の読み手・場面を確認してから作業を開始する。

**数値の出典確認**: 数値やデータに出典（出所・期間）が明記されていない場合、「出典のない数字は載せない」原則に従い、出所と対象期間をユーザーに確認する。確認が取れない数値はスライドから除外するか「要確認」として明示する。

---

## ワークフロー

以下の 8 ステップを**この順序で**思考し実行する。

### Step 1：解読

入力から以下の 3 要素を抽出する。

1. **一番伝えたいこと（主張）** — 読み手に何を信じてほしいか、何を決めてほしいか
2. **要素** — 項目・数値・関係性（事実の集合）
3. **読み手と場面** — 誰に・どんな文脈で・何のために

注意点：
- 事実は足さない・落とさない
- ただし、与えられた事実から**導ける示唆（So What）は積極的に引き出す**
- 入力が既存スライド画像の場合は、設計原則・デザイン規律違反の診断も行う

### Step 2：構造を選ぶ

`references/STRUCTURES.md` の「6つの構造」から最適なものを 1 つ選ぶ（必要に応じて入れ子にする）。

「代表レイアウト」に当てはまる場合はそれを優先する。当てはまらない場合は内容に最適な別の構造・図解を選ぶが、**箇条書きの羅列のままにしない（必ず構造化する）**。

### Step 3：グラフを選ぶ

数値データがある場合のみ、`references/STRUCTURES.md` の「7つのグラフ」から 1 つ選ぶ。数値がなければグラフは使わない。

### Step 4：レイアウトを選ぶ

`references/STRUCTURES.md` の「代表レイアウト」に該当するパターンがあれば採用する。

### Step 5：ヘッドメッセージ化

スライド上部に主張を**一文**で置く。

ルール：
- 結論の一文（名詞止め・「〜について」は禁止）
- 可能なら示唆（だから何をすべきか）まで踏み込む
- 事実はボディ、示唆はヘッドまたは別枠に配置する
- 主役メッセージは 1 枚に 1 つ

### Step 6：デザイン規律を適用する

`references/DESIGN-RULES.md` の「4つの設計原則」と「デザイン規律」全体を適用する。

### Step 7：生成

`skills/creating-pptx/scripts/pptx_kit.py` の **DualSlide ヘルパー**を使い、ビルドスクリプト（例：`build.py`）を書いて 16:9 の .pptx と プレビュー .png を生成する。

- **API の詳細・各メソッドの引数は `references/SCRIPT-API.md` を参照**（このファイルでシグネチャを再定義しない）
- 実行コマンド：`uv run --with python-pptx python <build>.py`
- 出力：`<出力名>.pptx` と `<出力名>.png`

### Step 8：出力前チェック＋最終磨き

`references/CHECKLIST.md` の**全項目**を通す。

- 生成した .png プレビューをユーザーに提示する
- 「初見の人が 3 秒で主張を言えるか」を自問する
- 言えなければヘッドメッセージと構造を組み直して再生成する
- 日本語が正しく描画されているか確認する
- 最終的に **.pptx（編集可能な成果物）** と **.png（プレビュー）** を納品する
