## 1. Constraints（レイアウト制約）

Constraintsは、親Frameのサイズが変わったとき子要素がどう振る舞うかを定義する。画面サイズ対応UIの基盤。

### 設定値と挙動

| 水平/垂直の設定 | 意味 |
|----------------|------|
| Left / Top | 左端/上端からの距離を固定（デフォルト） |
| Right / Bottom | 右端/下端からの距離を固定 |
| Left and right | 左右の距離を固定（親が広がると子も伸びる） |
| Top and bottom | 上下の距離を固定（親が縦に伸びると子も伸びる） |
| Center | 中心点からの距離を固定（中央配置を維持） |
| Scale | 親の拡縮に比例してサイズを変更 |

### 典型的な使い分けパターン

```
ヘッダー（横幅全体に伸びるUI）
  水平: Left and right  → 画面幅に追従
  垂直: Top             → 上部に固定

フッター（画面下部に固定するUI）
  水平: Left and right  → 画面幅に追従
  垂直: Bottom          → 下部に固定

ヘッダー内の中央タイトル
  水平: Center          → 横幅が変わっても中央を維持

ヘッダー内の右側ボタン
  水平: Right           → 右端からの距離を固定
```

### Auto LayoutとConstraintsの関係

- Auto Layoutを適用したFrameでは、一部のConstraints設定（Left and right / Top and bottom / Scale）が無効になる
- `Hug contents`設定のFrameはサイズが子要素依存のため、これらを指定できない
- Auto Layout未適用の場合は従来のConstraintsで制御する

## 2. Auto Layout設計パターン

Auto Layoutは「コンテンツに応じてサイズが変化するFrame」を実現する。テキスト量やコンポーネント数が動的に変化するUIに不可欠。

### Resizingプロパティ（親Frame視点）

| 設定値 | 意味 | 適用場面 |
|--------|------|---------|
| `Hug contents` | 子要素を包むようにFrameがリサイズ | ボタン、ラベル、バッジなど可変サイズのコンポーネント |
| `Fixed` | サイズを固定 | 固定サイズのコンテナ |
| `Fill container` | 親Frameを埋めるようにリサイズ | リスト項目、カード内の伸びる領域 |

### 子要素のResizing

| 設定値 | 意味 |
|--------|------|
| `Fill container` | 親Frameが伸びるとそのスペースを埋める（Paddingを維持） |
| `Fixed` | 親Frameが変わっても子要素のサイズは変わらない |
| `Hug contents` | 子要素がFrameの場合にも選択可能（子要素のコンテンツに依存） |

### テキストのResizing対応表

| テキスト設定 | 水平方向 | 垂直方向 | 用途 |
|------------|---------|---------|------|
| `Auto width` | Hug contents | Hug contents | 短い1行テキスト |
| `Auto height` | Fixed width | Hug contents | 折り返しあり可変高さ |
| `Fixed size` | Fixed width | Fixed height | 固定枠内のテキスト |

### 入れ子Auto Layoutのパターン

複雑なレイアウトも、Auto Layoutの入れ子で実現できる。

```
親Frame（垂直方向 Auto Layout、Fixed幅）
  ├── 上段Frame（水平方向 Auto Layout）
  │    ├── 画像（Fixed）
  │    └── テキストエリア（Fill container）
  ├── 中段Frame（水平方向 Auto Layout）
  │    └── 説明文（Hug contents）
  └── 下段Frame（水平方向 Auto Layout、Space between）
       ├── 価格テキスト（Hug）
       └── 購入ボタン（Hug）
```

### 分布（Distribution）

`Space between`を使うと子要素を親Frame内に均等分布できる。

- 前提: 親FrameのResizingが`Fixed`または`Fill container`
- 設定: Advanced layoutパネルの `Spacing mode: Space between`
- 結果: `Space between items`が`Auto`となり自動計算される

### Auto Layout適用の判断基準

| 適用すべき場面 | 適用不要な場面 |
|--------------|--------------|
| テキスト量が動的に変わるUI | 固定サイズのアイコンのみ |
| リスト、カード、ボタン | 1回しか使わない装飾要素 |
| 複数デバイスに対応するレイアウト | 厳密なピクセル配置が必要な場合 |
| コンポーネントの内部構造 | — |

## ブーリアン演算とマスク

ベクターオブジェクト合成の基礎技術。アイコン制作で頻用する。

| 演算 | 効果 | 主な用途 |
|------|------|---------|
| 結合（Union） | 全形状を1つに合成 | 複合アイコン作成 |
| 型抜き（Subtract） | 下から上を切り抜く | 穴あき形状・ロゴ |
| 交差（Intersect） | 重なり部分のみ残す | クリッピング効果 |
| 中マド（Exclude） | 重なり部分を除去 | リング・ドーナツ形状 |
| 統合（Flatten） | 演算結果を単一パスに変換 | 配布用最適化 |

**マスク**: 上レイヤーが表示範囲を決定。グループ内マスクは効果をグループ内に限定（`⌘ + option + M`）。

## 3. コンポーネント設計

### コンポーネント作成の基準

```
繰り返し使う要素 → コンポーネント化
  例: ボタン、ナビゲーションアイコン、リストアイテム、カード

1回しか使わない要素 → 通常のFrameのまま
  例: 特定画面のイラスト、一意の装飾
```

### 命名規則

- 半角スラッシュ（`/`）で階層化: `Checkbox/Default`、`Checkbox/Checked`
- 階層化すると、インスタンス置換時に同じ階層のコンポーネントが候補表示される
- プロパティ名と値は一貫性を持たせる（`Large` と `Big` を混在させない）

### インスタンスの原則

| 操作 | 説明 | 注意点 |
|------|------|--------|
| コンポーネントの変更 | すべてのインスタンスに反映 | — |
| インスタンスのプロパティ上書き | 上書き側が常に優先 | コンポーネント変更が反映されなくなる |
| `Reset all overrides` | 上書きをリセット | 右パネルの詳細メニューから実行 |
| `Detach instance` | インスタンスを通常オブジェクトに変換 | デザインの一貫性が崩れるリスク大 |

### 入れ子コンポーネント

コンポーネントの内部に別のコンポーネントのインスタンスを配置できる。

```
Button（コンポーネント）
  └── Checkbox/Checked（インスタンス）← 入れ子

インスタンスを配置した時、右パネルから
Checkbox/Checked → Checkbox/Default に置き換え可能
```

**インスタンス置換の活用:**
- `Checkbox/Default` と `Checkbox/Checked` を同じ階層にネーミング
- インスタンス選択後、右パネルのコンポーネント名をクリックして候補一覧を表示
- ワンクリックで状態切り替えが可能

### Componentsページの分離

- コンポーネントは専用の `Components` ページに集約
- `Design` ページには配置しない（コンポーネントはそこに直接使用しない）
- Move to page で後から移動可能（右クリック → Move to page → Components）

### グループとフレームの使い分け

| 機能 | グループ | フレーム |
|------|---------|---------|
| Auto Layout | ❌ | ✅ |
| Constraints | ❌ | ✅ |
| クリッピング（Clip content） | ❌ | ✅ |
| スクロール（Overflow scrolling） | ❌ | ✅ |
| Inner shadow | ❌ | ✅ |

**判断基準**: ただまとめるだけ → グループ。レイアウト制御・クリッピング・スクロールが必要 → フレーム。コンポーネント内部は必ずフレーム。

## 4. バリアントとコンポーネントプロパティ

### バリアント

外観が大きく異なるバリエーションを管理する仕組み。コンポーネントセット（紫の点線）にまとめられる。

**バリアントが適する場面:**
- ボタンのDefault / Pressed / Disabled
- チェックボックスのOn / Off
- カードのExpanded / Collapsed

**バリアント設計の注意点:**

```
プロパティ名と値の統一
  ❌ State: Large / State: Big    （同義語の混在）
  ✅ State: Default / Pressed / Disabled

組み合わせ爆発に注意
  State (3種) × Rounded (2種) = 6バリアント
  State (3種) × Rounded (2種) × Size (3種) = 18バリアント
  → 複雑すぎる場合はコンポーネントプロパティを検討
```

**ブール値プロパティ:**
値が `true` / `false` の場合、バリアント切り替えのUIがスイッチに変わる。
→ アイコンの表示/非表示など二値切り替えに最適。

### コンポーネントプロパティ（3種類）

| プロパティ種別 | 用途 | 設定方法 |
|--------------|------|---------|
| ブール値（Boolean） | レイヤーの表示/非表示 | レイヤー選択 → Layer > `①` アイコン |
| インスタンス置換（Instance swap） | 子コンポーネントの差し替え | コンポーネント名隣の `⬡` アイコン |
| テキスト（Content） | テキストの上書き | テキストレイヤー選択 → Content > `①` アイコン |

**バリアントvsコンポーネントプロパティの選択:**

| 選択肢 | 適した場面 |
|--------|-----------|
| バリアント | 見た目やサイズが大きく異なるバリエーション |
| コンポーネントプロパティ | 自由度の高いカスタマイズ（アイコン差し替え、テキスト変更、表示切替） |

## 5. スタイル管理

スタイルは「再利用可能なプロパティ」。コンポーネントがオブジェクトの雛形であるのに対し、スタイルはカラー・テキスト・エフェクト・グリッドの定義。

### スタイルの種類

| スタイル種別 | 登録できるプロパティ |
|------------|------------------|
| カラースタイル | Fill（単色・グラデーション・画像含む） |
| テキストスタイル | フォント、サイズ、ウェイト、行高、字間など |
| エフェクトスタイル | Drop shadow、Inner shadow、Blur |
| グリッドスタイル | Layout gridの設定 |

### 命名規則（カラースタイル）

階層化命名（`/` 区切り）で管理する。

```
推奨パターン（ライトモード/ダークモード対応）:
  Light/Background/Primary
  Light/Text/Secondary
  Dark/Background/Primary
  Dark/Text/Secondary

または（カテゴリ + セマンティック名）:
  Background/Primary
  Background/Secondary
  Text/Primary
  Text/Disabled
  Border/Default
  Brand/Main
  Destructive/Default
```

### テキストスタイルの命名

```
推奨パターン:
  Heading/Large     → 大見出し
  Heading/Medium    → 中見出し
  Body/Regular      → 本文
  Body/Small        → 小さい本文
  Label/Default     → ラベル
  Caption/Regular   → キャプション
```

### チームライブラリ

チームライブラリを使うと、スタイルとコンポーネントを複数ファイル間で共有できる。

- 無料プラン: スタイルのみ共有可能
- 有料プラン: コンポーネントも共有可能（チームライブラリ経由でリンクを維持）

## ファイル構成戦略

大規模デザインでは、Figmaファイルを役割別に4ページに分離する。

| ページ名 | 役割 | 配置するもの |
|---------|------|------------|
| `Design` | 画面デザイン | 各画面のFrame、ワイヤーフレーム |
| `Components` | コンポーネント定義 | Master Component（本体）のみ |
| `Styles` | スタイル見本 | カラーパレット・タイポグラフィの確認用 |
| `UI Kit` | 外部アセット | OS標準コンポーネント、サードパーティKit |

**コンポーネント移動**: 右クリック → `Move to page` → `Components`

**命名のポイント:**
- ページ名は英語・シンプルに（`Design`, `Components`, `Styles`）
- 役割が明確で、チームメンバー全員が迷わない構成を作る
- Components専用ページで分離すると検索ノイズを削減できる

## 6. ワイヤーフレーム制作

### 8ptグリッドシステム

**8pt（8dp）を基準にレイアウトを決定する手法。**

| メリット | 理由 |
|---------|------|
| デザインの一貫性 | 複数デザイナーが同じルールに従える |
| 意思決定の高速化 | 「10ptより8pt」「15ptより16pt」が明確 |
| 画像書き出しに有利 | @0.75x時に整数サイズになる（48×48→36×36） |

```
8の倍数でよく使う値:
  8, 16, 24, 32, 40, 48, 56, 64, 72, 80...

グリッドからはみ出してよい場合:
  全体の一貫性を失わない範囲ならルールを破っても良い
  例: テキストの文字サイズ（12, 14, 17など）は厳密でなくてよい
```

### Safe Area / Cutout Area

OSが提供するUIが占有する領域。**アプリのUIはSafe Areaの内側に配置する。**

| デバイス種別 | 上部の余白 | 下部の余白 |
|------------|-----------|-----------|
| iPhone SE / 8（ホームボタンあり） | 20pt | 0pt |
| iPhone X以降（ノッチあり） | 44pt | 34pt |
| iPhone 12/13シリーズ | 47〜50pt | 34pt |
| Android（Cutout Area） | デバイスにより異なる | デバイスにより異なる |

### 外部UI Kitの活用

FigmaのCommunityに公開されているUI Kitを利用することで、Status Bar、Home Indicator、OS標準コンポーネントをすぐに使える。

**活用手順:**
1. Figma Community で対象OS向けのUI Kitを検索・Duplicate
2. 必要なコンポーネント（Status Bar & Home Indicator等）を自プロジェクトにコピー
3. 専用ページ（例: `UI Kit`）に格納し、デザインページから参照

**チームライブラリとの違い:**
- Duplicate（複製）: ファイルが独立し、元ファイルとのリンクが切れる
- チームライブラリ経由: リンクを維持し、UI Kitの更新が自動反映（有料プラン）

### ワイヤーフレームのレイヤー構造

```
Home（トップレベルFrame）
  ├── Status Bar（FIXED - スクロール固定）
  ├── Home Indicator（FIXED - スクロール固定）
  ├── Grid（レイアウトガイド用Frame、ロック推奨）
  ├── App Header（スクロール固定設定を別途行う）
  └── コンテンツ（スクロール対象）
```

## 7. プロトタイプ設計

### FlowとStarting point

| 概念 | 定義 |
|------|------|
| **Flow** | 画面から画面へ移動する一連の流れ |
| **Starting point** | Flowの開始点となる画面 |

**設計の原則:**
- ユーザーフローごとにFlowを分割する（Flow 1: 投稿フロー、Flow 2: ストーリー閲覧フロー）
- Flowが複雑化するとメンテナンスが難しくなるため、機能ごとの分割を推奨

### スクロールの設定

| 設定 | 説明 |
|------|------|
| `Overflow scrolling: Horizontal scrolling` | 水平スクロール（横スクロールカルーセル等） |
| `Overflow scrolling: Vertical scrolling` | 垂直スクロール（デフォルトはトップレベルFrameに適用） |
| `Fix position when scrolling` | スクロール時に位置を固定（ヘッダー、ステータスバー等） |

**スクロール設定の条件:**
- 子要素がFrameからはみ出していること（はみ出しがなければスクロールは発生しない）

### インタラクション設定

**Trigger（発火条件）:**

| Trigger | 用途 |
|---------|------|
| On tap | ボタンタップ（モバイルの基本） |
| On drag | スワイプ操作 |
| After delay | 自動遷移、スプラッシュ画面 |
| While hovering | Webのホバー状態 |
| Key/gamepad | キーボードナビゲーション |

**Action（アクション）:**

| Action | 用途 |
|--------|------|
| Navigate to | 指定画面へ遷移 |
| Back | 直前の画面に戻る（逆アニメーションが自動適用） |
| Open overlay | 画面の上にモーダルを重ねる |
| Swap overlay | 重ねた画面を差し替える |
| Close overlay | 重ねた画面を閉じる |
| Scroll to | 同一画面内でスクロール |

### アニメーション設定

**Transitionの種類と用途:**

| Transition | 効果 | 推奨用途 |
|-----------|------|---------|
| Instant | アニメーションなし | プロトタイプ確認の初期設定 |
| Dissolve | フェードイン | モーダル、オーバーレイ |
| Smart animate | レイヤー名一致で自動アニメーション | 共通要素を維持したい画面遷移 |
| Slide in / out | 前後の画面が動く | iOS/Androidのページ遷移の標準パターン |
| Move in / out | 行き先の画面が重なる | — |
| Push | 行き先の画面に押し出される | — |

**Easingの選択指針:**

| Easing | 特徴 | 自然な用途 |
|--------|------|-----------|
| Ease out | 動き出し速く、ゆっくり終わる | 画面に入ってくるUI（ポジティブな動作） |
| Ease in | ゆっくり始まり、最後に速くなる | 画面から出るUI（ネガティブな動作） |
| Ease in and out | 最初と最後はゆっくり | 移動・変形アニメーション |
| Ease out back | 行き過ぎてバウンド | バネ感のあるUI（iOS的表現） |

### Smart Animate活用パターン

Smart animateは前後の画面で**同一レイヤー名**のオブジェクトを自動でアニメーションする。

```
活用例:
  Page Header が2画面で同じ名前 → スライドアニメーション中に固定表示される

落とし穴:
  意図しないアニメーションが発生する場合、前後画面で
  偶然に同じレイヤー名が使われていないか確認する

背景の透け問題:
  Smart animate有効時、行き先FrameのFillが無視される
  → 行き先FrameにBackground用Rectangleを追加して解決
```

### インタラクティブコンポーネント

バリアント間にインタラクションを設定することで、コンポーネント内部で状態変化するUIを作れる。

```
例: Toggle Switch
  State: Off（バリアント1）
  State: On（バリアント2）
  両者をインタラクションでつなぐ → タップで状態が切り替わる

使用場面:
  スイッチ、チェックボックス、いいねボタン、アコーディオン
```

### 実機プレビュー（Figma Mirror）

Figma Mirrorアプリを使うと、PCのデザインをスマートフォンでリアルタイム確認できる。同一アカウントでサインインし、PCで画面を選択すると自動反映される。

**確認ポイント**: タッチ領域のサイズ感、テキストの実際の読みやすさ、遷移・アニメーションの違和感。
