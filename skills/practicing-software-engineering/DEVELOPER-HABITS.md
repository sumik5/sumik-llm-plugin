# 開発者の習慣と落とし穴（DEVELOPER-HABITS）

偉大な開発者は「生まれつきの天才」ではなく、「優れた習慣を持つ開発者」である。
ここでは、プロフェッショナルな開発者として身につけるべき習慣と、
陥りやすい落とし穴・悪習慣を体系化する。

> 「専門家とは、その分野で犯しうる最悪の間違いを知っており、
>  それを避ける方法を知っている人のことである。」— Werner Heisenberg

---

## GREAT Developer Habits

7つの習慣で構成される開発者の行動指針。

---

### G: Good Communication（コードで伝える）

**コードの対象は「コンピュータ」ではなく「人間」**

コードは将来の自分を含む他の人間に読まれる通信手段。

#### 明確なコードのための実践

| 実践 | 説明 |
|------|------|
| **明確で説明的な名前** | モジュール・関数・変数の名前で意味の通る文を作れるように |
| **小さく集中した関数** | 関数は1つの責務に集中。目安: 5〜10行以内 |
| **誰でも読める** | 数秒で「何をしているか」がわかるコードを目指す |

#### ゴール

「このコードは何をしているか」を見た人が即座に理解できること。
将来の自分へのプレゼントだと思って書く。

---

### R: Reasoned Engineering（エンジニア的思考）

**エンジニアリングの本質: 「自分はおそらく間違っている」という前提から始める**

意見を持つことと、それが正しいことは別物。

#### エンジニア思考の実践

| 実践 | アクション |
|------|----------|
| **前提を疑う** | すべてのアイデアが同等の価値を持つわけではない |
| **エビデンスを求める** | 「何があれば考えを変えるか？」を問う。個人的好みではなくデータで判断 |
| **代替案を試す** | 比較なしに「Aの方が優れている」とは言えない |
| **説明できる理由を構築する** | なぜこのプラクティスが機能するかのモデルを持つ |
| **実験する** | 「このアイデアが間違いであることを示すには何が必要か？」を問う |

これは個人的好みやファッションの問題ではなく、何が実際に優れた結果をもたらすかの問題。

---

### E: Engineering Design（設計としてのコーディング）

**ツールや技術に囚われず、システムの「形」に注目する**

#### 設計の本質

- 扱っている **情報** と、システムの各部分間での **情報交換** に注目
- **複雑さの管理** がゴール
- **低複雑性のシステム = 良い設計**
- **良い設計の測定基準: 変更のしやすさ**

#### 複雑さを管理するための設計原則

| 原則 | 意味 |
|------|------|
| **Modularity（モジュール性）** | 独立した部品に分割する |
| **Cohesion（凝集性）** | 関連するものをまとめる |
| **Separation of Concerns（関心の分離）** | 異なる責務を混在させない |
| **Abstraction（抽象化）** | 詳細を隠蔽し、本質を露出する |
| **Reduced Coupling（疎結合）** | 部品間の依存を減らす |

#### フレームワークへの注意

| リスク | 対策 |
|--------|------|
| フレームワークがコードの構造を強制する | 自分の要件に合ったフレームワークを選ぶ |
| フレームワーク変更でコード変更が強制される | サードパーティコードを自分の抽象化の後ろに隠す |
| 特定フレームワークへのロックイン | どの程度フレームワークの制約を受けているか常に意識する |

---

### A: Always Improve（品質優先）

**品質を犠牲にして機能を出すことは、長期的には遅くなる**

#### 品質と速度の関係

> **速度と品質のトレードオフは存在しない。**
> 機能を速く届ける唯一の方法は、**高品質なソフトウェアを作ること**。
> 今品質に投資することで、将来の手直しが激減する。

#### 実践

- 変更しやすいソフトウェアを作り、**ずっとそれを維持する**
- Continuous Delivery・Agile 開発でインクリメンタルに進む
- コードを改善できる機会を見つけたら、積極的に改善する
- 「将来の自分がこのコードを読んでわかるか？」と問う → わからなければリファクタリング

---

### T: Talk More（社会的活動）

**偉大な開発者はより多く人と話す。最高の開発者は偉大なコミュニケーター**

#### ソフトウェア開発はチームスポーツ

- プロのソフトウェア開発の大半は **チーム** で行われる
- 明確にアイデアを表現する能力を磨く（シンプルなものから複雑なものまで）

#### コミュニケーション力向上の実践

| 方法 | 効果 |
|------|------|
| **ペアプログラミング** | コードについての会話を強制し、双方向の学習が生まれる |
| **アイデアを他人に説明する** | 説明している途中で自分で問題を解決することが多い |
| **不慣れなタスクでのコラボレーション** | 新しい視点と学習の機会 |
| **テックブログ執筆** | 思考の整理と知識の定着 |
| **コミュニティ参加** | ユーザーグループ・Meetup・オンラインフォーラム |

#### コードレビューよりペアプログラミング

従来の非同期コードレビューより、ペアプログラミングの方が:
- 「古くなった」コードではなく「書きながら」レビューできる
- プロセスを遅らせない
- チームメンバーが変更時点で慎重に考えるようになる

---

### 小さなステップ（Work in Small Steps）

**最高の開発者は小さなステップで進み、各ステップを確認しながら前進する**

#### 実践

- **TDD**: 小さな変更 → テストがすべてパスすることを確認
- **リファクタリング**: 小さな変更 → 各ステップ後にリファクタリング

#### 小さなステップの価値

小さなステップで進むことで:
- **より速く学習**できる
- **より自信を持って**前進できる
- **解決策を探索**しやすくなる
- **継続的に判断・精錬・改善**できる

## よくある落とし穴（悪習慣の排除）

---

### 落とし穴1: Happy Path 思考

**「正常系しか考えない」という思考の罠**

#### 問題

「自分のシステムはそんなことをしない」という魔法的思考。
高品質なシステムは、**問題が起きたときに耐性を持つ**。

#### 考えるべきネガティブシナリオ

- 関数が予期しない値を受け取ったら？
- ディスクが満杯または対象ファイルが存在しなかったら？
- サードパーティライブラリが例外をスローしたら？
- 並行処理での競合状態は？
- 分散システムでのキューが溢れたら？
- セキュリティ侵害が起きたら？
- 予算が無限に消費され始めたら？

**対処を後回しにする場合でも、リスクを無視しないこと。**

---

### 落とし穴2: コードオーナーシップ

**「自分のコード」という意識の罠**

#### 問題

コードを個人の所有物と考えると:
- 他人が変更しにくくなる
- 知識がサイロ化する
- バスファクターが低下する

#### 解決策

| 問題 | 解決策 |
|------|--------|
| コードを「自分のもの」と思っている | **共有コードオーナーシップ** モデルに移行 |
| 一人だけが知っている部分がある | **ペアプログラミング**または定期的なローテーション |
| 特定の人が離脱すると進捗が止まる | 誰もが全部分を扱える状態を目指す |

---

### 落とし穴3: 「自分の問題じゃない」

**責任の範囲を狭く定義する罠**

#### 「誰かのジョブ」ではないこと

- 「何を書くか」を指示してもらうことを待つ
- コードベースをきれいに保つ「許可」をもらうことを待つ
- テストやリファクタリングの「許可」をもらうことを待つ

#### あるべき姿

プロフェッショナルとして、ユーザー・チーム・会社への **Duty of Care（ケアの義務）** を持つ。
自分の作業に責任を持ち、誇りを持って取り組む。

**全員がオーナーシップを持つ。**

---

### 落とし穴4: コード変更への恐怖

**「コードを変えるのが怖い」という状態は深刻な問題**

#### 原因と対策

| 原因 | 対策 |
|------|------|
| テストがない | Approval Tests（特性テスト）を追加する |
| テストが壊れやすい | 「何をすべきか」を記述するテストに改善（「どうするか」ではなく） |
| コードが複雑すぎる | リファクタリングでコードを改善 |
| スキル不足 | リファクタリング手法を練習し、IDE のリファクタリング機能を活用 |

#### 考え方の転換

- コードは「現時点でのベストな推測」であり、**おそらく間違っている**
- コードを見直し・改善することは良いことであり、**学習を深める**
- 安全に・インクリメンタルに変更できる能力を常に維持する

> **偉大なコードとは、安全かつ容易に変更できるコードのこと。**

---

### 落とし穴5: ツール崇拝

**「ツールが目的」になってしまう罠**

#### 本質

本当に重要なのは: **優れたソフトウェアをユーザーに素早く・効率的に届けること**

これは問題を理解し解決することを意味する。ツールはその手段に過ぎない。

**まず問題を理解してから、ツールを選ぶ。**

---

### 落とし穴6: よくある「神話」への執着

これらの思い込みがより良い方法を試すことを妨げる。

| 神話 | 反論 |
|------|------|
| **「本格的な作業にはフィーチャーブランチが必要」** | CI が最も安全で効率的。CI とフィーチャーブランチの両立には、ブランチ期間を1日未満にする必要がある |
| **「品質より速度を選ぶよう指示された」** | **速度と品質のトレードオフは存在しない**。CD 実践チームは未実践チームより44%多い時間を新機能に費やす（DORA研究） |
| **「テストは時間の無駄」** | テストは時間を**節約する**。問題は「どのテスト方法が最も効率的か」だけ |
| **「CI と Pull Request は共存できない」** | できる。ただし、古いコードのレビューを待つよりペアプログラミングで即時レビューした方が効率的 |
| **「これはシンプルな Web アプリとは違う複雑なシステムだ」** | 軍事システム・金融システム・大規模 SaaS も同じプラクティスを使っている |
| **「ずっとこうやってきて、うまくいっている」** | 改善しなければ、他の誰かに追い越される |

---

## まとめ: エンジニア思考で全てを測る

**判断基準:**

1. 「このアプローチが間違っていることを示すには何が必要か？」を問う
2. 代替案を常に試す
3. **Throughput**（スループット）と **Stability**（安定性）で評価する
4. 主張を裏付けるエビデンスを求める
5. 「なぜこのアイデアが機能するか」の説明を構築できるか確認する

> **問題が痛ければ痛いほど、もっと頻繁に取り組む。痛みが消えて習慣になるまで。**
