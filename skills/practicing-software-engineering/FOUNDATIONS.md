# プロジェクト基盤（FOUNDATIONS）

新規プロジェクトを成功させるための基盤プラクティス。
プロジェクト開始時は「最も情報が少ない時期」であることを前提に、
学習を最大化し変化に対応できる構造を作ることが最優先課題。

---

## 基本原則

> プロジェクト開始時の決定（設計・技術・アプローチ）は、ほぼ確実に不完全である。
> だからこそ、素早く学習し、早期に軌道修正できる仕組みが必要。

**新規プロジェクトのトップ原則:**

| 優先順位 | 原則 |
|---------|------|
| 1 | Fast Feedback を得る |
| 2 | 学習を最適化する |
| 3 | ビジョンを作る |
| 4 | 小さなステップで働く |
| 5 | Deployment Pipeline を構築する |
| 6 | 計測を導入する |
| 7 | 変化を受け入れる準備をする |
| 8 | 開発環境を整える |

---

## Fast Feedback

### 目的

学習と適応のために、高速かつ高品質なフィードバックループを確立する。

### フィードバックの種類

| フィードバック源 | 内容 | 活用方法 |
|---------------|------|---------|
| Build | コンパイルエラー・依存関係エラー | 即時修正 |
| Test | ユニット/受け入れテスト結果 | リグレッション検知 |
| Deploy | デプロイ成否・環境エラー | インフラ問題の早期発見 |
| Production | ユーザー行動・エラー率・パフォーマンス | 意思決定の根拠 |

### 実践

- テスト・メトリクスのデータをチーム全体で共有する
- 「どこにいるか」「どう改善できるか」を常にチーム全員が把握できる状態を維持
- フィードバックが遅い場合、そのボトルネックを特定して排除する

---

## 学習の最適化

### SW開発 = 学習プロセス

ソフトウェア開発の本質は学習。プロジェクト全体を通じて、継続的な学習サイクルを回す。

### 実践方法

**レトロスペクティブの早期導入**
- プロジェクト開始時からレトロスペクティブを習慣化する
- チームの進捗を定期的にレビューし、アイデアと専門知識を共有する
- インクリメンタルかつ協調的な働き方のスキルを構築する

**オープンマインドの維持**
- 過去の経験・実績ある技術から始めるのは合理的
- しかし「このプロジェクトは同じではない」という前提を持つ
- 教条主義や「昔からそうしてきた」という理由ではなく、エビデンスに基づいて決定する
- 実験的アプローチを取り、仮説を立て、検証する

**オープンな問題共有**
- 知らないことを正直に認める
- 問題を隠さず、チームでオープンに議論する

## 開発環境のセットアップ

### 4つの要素

| 要素 | 説明 |
|------|------|
| **Quality Tooling** | 高速で効率的なフィードバックを実現するツール群 |
| **Good Connectivity** | ネットワークアクセスと情報リソースへのアクセス（リモートでも同等に） |
| **Automated Setup** | 新メンバーのオンボーディングを自動化（OS・言語・VCS・権限等） |
| **Team Relationships** | 作業方法の合意（Pairing・TDD・コーディング規約等） |

### 開発環境のセットアップ指針

- セットアップの完璧さを求めず、まず決め、繰り返しの中で洗練させる
- 新メンバーが初日から貢献できるレベルを目指す
- チームの作業方法（ペアプロ・TDD・コーディング規約）を最初に合意する

## Deployment Pipeline の構築

### 原則: Walking Skeleton から始める

最初のフィーチャーと同時に最小限のパイプラインを構築し、徐々に拡充する。

```
最初の機能
  → Unit Tests + Acceptance Tests 追加
  → テスト環境へのデプロイ自動化
  → パイプライン拡充（遅いコンポーネントを特定し高速化）
```

### ステップ

1. システム全体を代表する最も単純なフィーチャーを選択
2. そのフィーチャーに対してコード・テスト・インフラを構築
3. TDD + ATDD（BDDスタイルの受け入れテスト）を最初から採用
4. 自動化できるものをすべて自動化（テスト・デプロイ）
5. 最も遅いコンポーネントを特定し、高速化する

> CI/CD パイプラインの詳細設定は `practicing-devops` スキルを参照

## 目標設定（ビジョン）

### ビジョンの性質

- **方向性を示す**が、詳細ではない
- 「このソフトウェアはユーザーの何をよくするか？」に焦点を当てる
- 最小成功基準（Minimum Success Criteria）は何か？

### 計画のルール

| ルール | 説明 |
|--------|------|
| **詳細計画を作らない** | 最初から詳細な完成計画を作ることは不可能 |
| **TIME か SCOPE を固定** | 期限があるなら SCOPE を可変に、機能が固定なら TIME を可変に |
| **両方を固定しない** | TIME と SCOPE の両方を同時に固定するのは失敗のレシピ |

### 進捗の確認方法

各ステップでこう問う:「このステップは目標に近づいているか、遠ざかっているか？」

## 小さなステップで働く

### 原則

「最小のフィーチャー」から始め、インクリメンタルに進める。

### メリット

| メリット | 理由 |
|---------|------|
| 高速フィードバック | 小さなステップ = 素早い検証 |
| 素早いリバート | 問題発見時の損失が最小 |
| ミスの影響を限定 | 間違いが小さい範囲に留まる |
| 「未知」を遅延させられる | 不明な部分を抽象化して後回しにできる |

### 実践

- 「未知の部分」は抽象化して後で詳細を決める
- 理解している部分から先に進める
- システム全体を最初に設計しようとしない

---

## 計測（メトリクス）

### なぜ計測が必要か

小さなステップが目標に向かっているかどうかを知るため。「十分に良い」「速い」「ボトルネックがない」を客観的に判断するために計測が必要。

### 推奨メトリクス: DORA Metrics（最低限）

| メトリクス | 定義 |
|-----------|------|
| **Throughput（スループット）** | 新フィーチャーのビルド・デプロイの効率性 |
| **Stability（安定性）** | そのスループットレベルにおける出力品質 |

### 計測の導入タイミング

プロジェクト早期に「適切なメトリクスは何か」を確立し、データ収集を開始する。
計測が遅れるほど、意思決定の根拠が弱くなる。

## 変化への準備

### 防御的設計

- **モジュラーアーキテクチャ**: 一部の変更が他に影響しないように
- **関心の分離**: モジュール性を高め、変更を局所化
- **問題に強くフォーカス、現在のソリューションへの執着を弱く**

### 心構え

- 何かが機能していないことを認める勇気を持つ
- 悪いアイデアを素早く捨てることを恐れない
- 技術にこだわらない。より良いアイデアが出れば方向転換する
- プロダクト全体が機能しない場合、方向転換する可能性も排除しない
