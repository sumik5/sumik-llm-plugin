# 統合テストの原則

> 出典: Vladimir Khorikov, *Unit Testing: Principles, Practices, and Patterns* (Manning, 2020)
> 第8章「なぜ統合テストを行うのか」/ 第9章「モックのベストプラクティス」/ 第10章「データベースに対するテスト」

---

## 統合テストの役割と4本の柱との関係

### 単体テストとの役割分担

```
コードの複雑さ
      ↑
  ドメインモデル    ← 単体テスト対象
  アルゴリズム
────────────────→ 協力者オブジェクトの数
  コントローラ     ← 統合テスト対象（プロセス外依存との連携）
```

| | 退行保護 | リファクタリング耐性 | 速度 |
|--|---------|------------------|------|
| 単体テスト | 中（ドメイン内のみ） | 高 | 高速 |
| 統合テスト | 強（外部依存含む） | 高 | 中速 |

### テストピラミッドとテストケース配分

```
    E2E: 最長ハッピーパス1件のみ
   ────────────────────────────
  統合: ハッピーパス1件 + 単体では検証不可な異常ケース
 ───────────────────────────────────────────────
単体: すべての異常ケース（エッジケース）を網羅
```

**シンプルなアプリ**（ドメインモデルがほぼない）では単体と統合がほぼ同数になり、四角形に変形する。それでも統合テストの価値は変わらない。

### 早期失敗（Fail Fast）との関係

事前条件チェックをコントローラが忘れても、ドメインモデルが例外をスローするなら統合テストは不要。そのバグは「早期失敗」で検出・停止される。**質の悪いテストより、テストを書かないほうがまし。**

## プロセス外依存のモック判断基準

### 2種類の依存

| 依存の種類 | 外部から観察可能? | 典型例 | テストでの扱い |
|-----------|---------------|--------|--------------|
| **管理下にある依存** | ❌ 不可（自チームのみ） | 自チームのDB | 実際のインスタンスを使う |
| **管理下にない依存** | ✅ 可能 | メールサービス、メッセージバス | モックに置き換える |

**核心**: 管理下にある依存とのコミュニケーションは「実装の詳細」→ モックすると退行保護を失う。管理下にない依存とのコミュニケーションは「観察可能な振る舞い（契約）」→ 後方互換を検証する。

### 判断フローチャート

```
プロセス外依存を発見した
         ↓
他のアプリケーションからも観察可能なコミュニケーションか?
    ↙ Yes                      ↘ No
管理下にない依存               管理下にある依存
  ↓                              ↓
モックに置き換える              実際のインスタンスを使う
（コミュニケーションの仕様を検証）  （最終的な状態を検証）
```

### 両方の性質を持つDB（共有テーブルがある場合）

- 他アプリと共有されるテーブル → 管理下にない依存 → **モック**（実質的なメッセージバスとして扱う）
- 自チームのみのテーブル → 管理下にある依存 → **実際のDBを使う**

### 実際のDBを使えない場合

DBをモックして統合テストを書いてはならない（リファクタリング耐性が失われる）。**ドメインモデルの単体テストに専念し、価値のない統合テストは作らない。**

## インターフェースを使った依存の抽象化

### いつインターフェースを使うべきか

| 状況 | インターフェース | 理由 |
|------|---------------|------|
| 管理下にない依存（モック対象） | **使う** | モックを作るため |
| 管理下にある依存 | **使わない** | 具象クラスを直接注入すれば十分 |
| 同一プロセス内ドメインクラス | **使わない** | ドメインクラス間のやり取りをモックすると耐性低下 |
| 実装クラスが2つ以上ある | **使う** | 本当の抽象化が存在する |

**YAGNI原則**: 実装クラスが1つしかないのにインターフェースを作ることは設計の無駄。将来の拡張性のために現時点でコストを払うべきでない。

**例外の理由**: 管理下にない依存に限り、「モックを作れるようにするため」という実践的な理由でインターフェースを使う。

## 統合テストのベストプラクティス

### ドメインモデルの境界明確化
ドメインロジックを単一の明確なグループに集約し、コードベース全体に散在させない。これにより単体テスト（ドメイン対象）と統合テスト（コントローラ対象）の区別が容易になる。

### アプリケーション層の削減
多すぎる間接参照層はコントローラとドメインの境界を曖昧にし、「各層を個別に検証する」モック多用の価値の低いテストを量産する。バックエンドは**ドメイン層・アプリサービス層・インフラ層の3層**で十分。

### 循環依存の除去
インターフェースで循環依存を「隠す」だけの解決は避ける。コンパイルエラーは消えても実行時に循環は残り、認知的負荷が増える。依存の方向を一方向にするようリファクタリングする。

### 複数Actフェーズの限定的使用
通常は1テスト1Actを守る。**例外**: プロセス外依存の状態を制御できない場合（呼び出し回数制限のある外部APIなど）。実質的にE2Eテストでのみ使われる。

## ログテストの原則

| ログ種別 | 対象者 | テスト要否 |
|---------|-------|----------|
| **サポートログ** | システム管理者・サポートスタッフ | テストする（ビジネス要求） |
| **診断ログ** | 開発者のみ | テストしない（実装詳細） |

**実装方針**: サポートログは汎用 `ILogger` を直接モックせず、ドメイン専用の `IDomainLogger`（`userTypeHasChanged()` などのドメイン用語のメソッドを持つ）を通して検証する。

## モックのベストプラクティス4原則

### 原則1: アプリケーション境界のコミュニケーション検証のみ

モックの置き換え対象は、管理下にない依存とのコミュニケーションフローの**最末端**（境界に最も近い箇所）にする:

```
コントローラ → EventDispatcher → MessageBus → [IBus] → 外部メッセージバス
                                              ↑
                                     ここをモック（境界に最も近い）
```

境界に近いほど退行保護が強まり（経由するコードが増える）、実装詳細から離れてリファクタリング耐性も高まる。

### 原則2: モックからスパイへの置き換え

境界クラスのテストダブルには、モックライブラリよりも**手書きのスパイ**が優れている。確認ロジックをスパイ内に閉じ込め、複数テストから再利用できる:

```typescript
class BusSpy implements IBus {
  private sentMessages: string[] = []
  send(message: string) { this.sentMessages.push(message) }

  // 流れるようなインターフェースで検証
  shouldSendNumberOfMessages(count: number): this {
    expect(this.sentMessages).toHaveLength(count); return this
  }
  withEmailChangedMessage(userId: string, email: string): this {
    expect(this.sentMessages).toContain(`Type: EMAIL CHANGED; Id: ${userId}; Email: ${email}`)
    return this
  }
}

// テストでの使用
busSpy.shouldSendNumberOfMessages(1).withEmailChangedMessage(userId, 'new@example.com')
```

### 原則3: 呼び出し回数の常時確認

後方互換維持のため「想定する呼び出しが行われること」と「想定外の呼び出しが行われないこと」の両方を検証する:

```typescript
// ❌ 存在確認のみ（余分な呼び出しを見逃す）
expect(busMock).toHaveBeenCalledWith(expectedMessage)

// ✅ 回数も確認
expect(busMock).toHaveBeenCalledTimes(1)
expect(busMock).toHaveBeenCalledWith(expectedMessage)
```

### 原則4: モック対象の型は自プロジェクト所有のみ

サードパーティライブラリを直接モックせず、**アダプターを自作してそれをモックする**:

```typescript
// ✅ 自プロジェクトのアダプターインターフェースをモック
interface IBus { send(message: string): Promise<void> }
// サードパーティSDKはAdapterクラスの内部に隠す
```

**理由**: ライブラリの内部動作を信頼できない。ライブラリ変更の影響をアダプターに局所化できる。

## データベーステスト原則

### スキーマのソースコード管理

- スキーマ（テーブル・ビュー・インデックス）はSQLスクリプトとしてGitで管理する
- 参照データ（アプリ動作に必要な事前データ）もINSERT文として同じく管理する
- 「モデルDB」を真の情報源にするのはアンチパターン（変更履歴が失われる）

### 状態ベース vs 移行ベース

| アプローチ | DBの状態 | データ移行（data motion） | 本番リリース後の実用性 |
|-----------|---------|--------------------------|---------------------|
| 状態ベース | 明確 | ❌ 困難（ツールはデータ変換を推測できない） | △ |
| **移行ベース** | 暗黙的 | ✅ 容易（開発者がドメイン知識で記述） | ◎ |

本番リリース後は必ず移行ベースを使う。間違えた場合も直接データを修正せず、取り消し移行 → 正しい移行の順で対処する。

### テストデータのライフサイクル（後始末の方法）

| 方法 | 速度 | 本番との乖離 | 推奨度 |
|------|------|------------|--------|
| バックアップから復元 | 遅い | なし | △ |
| テスト後クリーンアップ | 速い | なし | △（途中中断で失敗） |
| ロールバック（1トランザクション） | 速い | **あり** ❌ | ✗ |
| **テスト前クリーンアップ** | 速い | なし | ◎ |

```typescript
// ✅ 推奨: 準備フェーズでクリーンアップ（参照データは削除しない）
abstract class IntegrationTests {
  protected async setup() {
    await db.query('DELETE FROM users; DELETE FROM companies;')
    // 参照データ（user_types等）は削除しない
  }
}
```

**ロールバック方式が駄目な理由**: Arrange/Act/AssertのすべてのフェーズでDB接続が共有されるため、「ビジネスオペレーションごとに個別トランザクション」という本番動作と乖離する。

### インメモリDB使用の問題点

インメモリDB（SQLite等）は機能面で本番DBと異なり、偽陽性・偽陰性を生む。**テスト環境と本番環境は同じDBベンダーを使う**（バージョン・エディションは違ってもよい）。

### フェーズごとに独立したトランザクション

```typescript
// ✅ 3フェーズそれぞれ別トランザクション（本番と同じ動作）
it('should change email', async () => {
  // Arrange: 別トランザクション
  const user = await createUser('user@corp.com', UserType.Employee)

  // Act: 別トランザクション
  const result = await execute((ctrl) => ctrl.changeEmail(user.id, 'new@example.com'))
  expect(result).toBe('OK')

  // Assert: 別トランザクション（DBから再取得 = 本番と同じ読み込み経路）
  const userFromDb = await queryUser(user.id)
  expect(userFromDb.email).toBe('new@example.com')
})
```

### テストコードの再利用（Object Motherパターン）

Arrangeフェーズはデフォルト値付きファクトリメソッドで共有し、重要なパラメータだけを明示する:

```typescript
// Object Motherパターン: デフォルト値で「このテストで重要な違い」が際立つ
async function createUser(
  email = 'user@example.com',
  type: UserType = UserType.Employee,
  isEmailConfirmed = false
): Promise<User> { /* DB保存して返す */ }

// 呼び出し側: email だけが重要なパラメータであることが明確
const user = await createUser('user@corp.com')
```
