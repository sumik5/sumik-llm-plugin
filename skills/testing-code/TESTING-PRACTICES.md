## TDDの本質：設計ツールとしてのTDD

TDDは「テストを書く技法」ではなく、**より良い設計を生み出す技法**です。

テスト可能なコードを書こうとすると、以下5つの設計品質が自然に向上します：

| 設計品質 | TDDによる改善メカニズム |
|---------|----------------------|
| モジュール性 | 小さく独立したユニットが必要になる |
| 疎結合 | 依存性注入を自然に採用する |
| 関心の分離 | テスト対象を明確に絞る必要がある |
| 高凝集 | 単一の振る舞いをテストするため |
| 抽象化 | フェイク依存の注入点が抽象化を促す |

### Red-Green-Refactorの3つのマインドセット

| フェーズ | 主目的 | 意識すべきこと |
|---------|--------|--------------|
| **Red** | 外部設計 | テストは公開インターフェースを設計する行為。HOWではなくWHATを表現する |
| **Green** | 戦術的最速実装 | 安全な状態（通過）に戻ること優先。設計は後回し |
| **Refactor** | 内部設計改善 | テスト通過を守りながら、可読性・抽象化・保守性を向上させる |

> **重要**: リファクタリングはGreenのときのみ行う

### TDD実践の判断基準

| 状況 | 判断 |
|------|------|
| テストが書きにくい | ✅ コードの設計が複雑なシグナル — コードを改善する |
| テストが大きく複雑 | ✅ TDDを実践していないか設計が悪いシグナル |
| テストを後で書いた | ❌ それはTDDではない |
| カバレッジ向上を目標にしている | ❌ カバレッジは副産物。振る舞いを駆動することが目標 |

### テスト命名の原則

- テスト名は必ず「SHOULD（すべき）」で始まる
- 「何をするか（WHAT）」を表現し、「どうするか（HOW）」には触れない
- 実装を別のものに置き換えてもテストは依然として有効であるべき

## ATDD：受け入れテスト駆動開発

### ATDDの位置づけ

Acceptance TestsはContinuous Deliveryにおける**リリース可能性の判定**が役割です。

```
役割分担:
  Acceptance Tests = ビジネス視点のリリース判定
  Unit Tests(TDD)  = 技術視点のコード品質確保
```

### 効果的なAcceptance Testの5条件

1. **外部ユーザーの視点** で記述されている
2. **本番に近いシナリオ** で評価される
3. **本番に近いテスト環境** で実行される
4. **公開インターフェース経由** でのみSUTと対話する
5. **WHATのみ**を評価し、HOWには一切触れない

### 4層アーキテクチャ

```
┌─────────────────────────────────────┐
│     Test Case (Executable Spec)      │  ← ドメイン言語で記述
├─────────────────────────────────────┤
│      Domain Specific Language (DSL)  │  ← 共通語彙の抽象化レイヤー
├──────────────────┬──────────────────┤
│  Protocol Driver │  External Stubs   │  ← 技術詳細の隔離
├──────────────────┴──────────────────┤
│         System Under Test (SUT)      │  ← 本番と同等の構成
└─────────────────────────────────────┘
```

| レイヤー | 責務 |
|---------|------|
| Test Case | ドメイン言語で「何をすべきか」を表現 |
| DSL | テストケースとプロトコルの橋渡し。省略可能なデフォルト値を活用 |
| Protocol Driver | DSLをシステムの言語に変換。UIやAPIの技術詳細をここに封じ込める |
| SUT | IaCで管理された本番同等環境 |

### テストデータの原則

- **Atomic**: 各テストケースは独立したデータから開始する
- **共有書き込みデータ禁止**: テスト間でミュータブルデータを共有しない
- 一意識別子を生成してデータ競合を防ぐ

## テスト戦略：いつ何をテストするか

### 4フェーズテスト戦略

| フェーズ | 目的 | 主なテスト種別 |
|---------|------|--------------|
| **Commit** | FAIL FAST（高速フィードバック） | ユニットテスト、静的解析、コーディング規約 |
| **Acceptance** | リリース可能性の判定 | ATテスト、性能テスト、セキュリティ、コンプライアンス |
| **Release** | リリース支援 | スモークテスト、カナリーリリース、ヘルスチェック |
| **Production** | プロダクト設計への学習 | A/Bテスト、ビジネスメトリクス、技術監視 |

> 各フェーズは「何を確認するか」が本質的に異なる — ツールではなく目的で分類する

## Testing in Production

### 基本原則

Testing in Productionは**開発中のテストの代替ではない** — より広いテスト戦略の一部です。

- 開発フェーズでは発見できないことを学ぶ手段
- Deployment Pipelineで十分にテストされた後に行う
- 単独では高品質を保証できない

### Productionで収集すべき情報

| カテゴリ | 具体例 |
|---------|--------|
| **技術情報** | メモリ/CPU/ディスク使用率、エラーログ、キュー深度、応答時間 |
| **機能情報** | ユーザージャーニー、機能利用状況、エラー率 |
| **ビジネス情報** | AARRR指標（獲得・活性・継続・収益・紹介） |

### Production Testing手法

| 手法 | 目的 | 判断基準 |
|------|------|---------|
| **Smoke Test** | デプロイ直後の環境確認 | 設定と可用性の検証 |
| **Canary Release** | 低リスク環境から段階的展開 | Pass/Fail基準を事前定義 |
| **A/B Test** | バージョン比較 | 統計的に有意な差を検証 |
| **Monitoring** | 継続的なシステム健全性確認 | 閾値アラートで自動通知 |

### E2E Testing（ゲートキーピング方式）を避ける理由

E2E Testing（全変更をまとめてステージング環境で手動テスト）の問題点：

- 変更量が多くなるにつれ、問題の特定が指数的に困難になる
- 結果が不明瞭で意味が薄くなる
- フィードバックが遅すぎて修正コストが高い

## フレーキーテスト排除

### フレーキーテストの原則

> **フレーキーテストは失敗として扱う** — 再実行で通過しても、失敗の方が有益な情報

| 考え方 | 適切なアプローチ |
|--------|---------------|
| 「たまたま失敗した」と再実行 | ❌ 根本原因を調査する |
| フレーキーテストを無視する | ❌ リリース可能性の判定を破壊する |
| 失敗を原因別に分類する | ✅ 環境・データ・並行性の問題を特定 |

### フレーキーテスト根本原因と対策

| 根本原因 | 症状 | 対策 |
|---------|------|------|
| **環境の不一致** | 実行環境によって結果が変わる | 環境設定をバージョン管理し、セットアップを自動化 |
| **テストデータの競合** | 実行順序によって結果が変わる | テストデータを分離（Functional Isolation） |
| **タイミング依存** | 速い環境では通過、遅い環境では失敗 | タイミングに依存せず、条件完了を待機 |
| **並行性・Race Condition** | 並列実行時に失敗 | 同期化を実施。並行性のコントロールをテストインフラに委譲 |
| **リソース不足** | 負荷状況によって失敗 | リソース監視を設定し、割り当てを最適化 |

### テスト安定化の実践規則

- **VERSION CONTROL EVERYTHING**: コード・テスト・環境設定をすべてバージョン管理
- **テストデータを原子的に**: テストケースごとにデータを新規作成し、一意IDを使用
- **CIに即座に統合**: コミットごとにテストを実行し、失敗は即座に対処
- **スコープを絞る**: テストの範囲が狭いほど変数をコントロールしやすい

## BDD：振る舞い駆動開発

### BDDの本質

BDDはCucumberなどのツールではなく、**アプローチとマインドセット**です。

> TDDの改良版として、「テストカバレッジ」ではなく「振る舞いの仕様化」を目的とする

| 比較 | TDD | BDD |
|------|-----|-----|
| 主眼 | コード品質 | システムの振る舞い仕様 |
| 表現方法 | テスト関数 | 実行可能仕様（Executable Specifications） |
| 起点 | 開発者の視点 | ユーザーの視点 |
| 言語 | コード寄り | ドメイン言語 |

### Given-When-Then形式

```
Given: システムが特定の状態にある（前提条件）
When:  特定のアクションが発生する（操作）
Then:  システムは特定の振る舞いをすべき（期待結果）
```

### WHATとHOWの分離

| 例 | 評価 | 理由 |
|----|------|------|
| `shopping.placeOrder(book)` | ✅ WHAT | ユーザーの意図を表現 |
| `driver.findElement(By.id("buy-button")).click()` | ❌ HOW | UI実装の詳細を含む |
| `shopping.payByCreditCard(card)` | ✅ WHAT | 何をするかを表現 |
| `form.fillField("card-number", "1234...")` | ❌ HOW | フォーム操作の詳細 |

> WHATを記述した仕様はUIが変わっても仕様自体は正しいまま — テスト失敗時も何をすべきかを示し続ける

### ユビキタス言語の確立

- 開発者・テスター・ステークホルダー全員が同じ語彙を使う
- 仕様・コード・テスト・ドキュメントで言語を統一する
- 技術的ジャーゴンをドメイン言語に置き換える

### BDDの翻訳プロセス

```
Wish（漠然としたアイデア）
  → User Story（ユーザー視点の要求）
    → Concrete Examples（具体的な例）
      → Acceptance Criteria（受け入れ基準）
        → Executable Specifications（実行可能仕様）
          → Working Software（動くソフトウェア）
```

### BDDのよくある落とし穴

| 落とし穴 | 回避策 |
|---------|--------|
| ツール（Cucumber等）がBDDだと思う | ツールは補助。アプローチとマインドセットが本質 |
| 仕様に実装詳細を含める | 実装変更でテストが壊れ、仕様が形骸化する |
| 開発者だけで仕様を書く | BDDはコラボレーション。全ステークホルダーが参加 |
| 要件が曖昧なまま実装する | User Story → Exampleで段階的に精緻化する |

## パフォーマンステスト

### 基本概念

| 概念 | 定義 |
|------|------|
| **Throughput** | 単位時間あたりのトランザクション処理量（帯域幅） |
| **Latency** | アクション開始から結果観測までの時間遅延 |
| **Responsiveness** | 高Throughput + 低Latencyによる一貫した快適な応答 |

### ユーザー体感のLatency基準

| レベル | 範囲 |
|--------|------|
| 優秀 | 0〜150 ms |
| 良好 | 〜300 ms |
| 許容 | 〜450 ms |
| 不許容 | 450 ms以上 |

### Pass/Fail テスト戦略

パフォーマンステストはPass/Fail基準を事前定義し、**解釈不要の客観的判定**にする：

- **PASS**: リリース可能（定義したThroughput/Latency閾値を満たす）
- **FAIL**: リリース禁止（調査・改善が必要）

> 「どう見える」ではなく「閾値を満たすか否か」で判断する

### コンポーネントvs全体システムテスト

| テスト種別 | 用途 | 特徴 |
|-----------|------|------|
| **コンポーネントテスト** | 予測される性能問題の発見 | 変数コントロールが容易、再現性高い |
| **全体システムテスト** | 予期しない性能問題の発見 | 変数コントロールが難しい、本番に近い |

> 高性能システムは両方をDeployment Pipelineに組み込む

### 変数コントロールの戦略

- **専用テスト環境**: 本番環境をミラーした専用環境を使用
- **Infrastructure as Code**: 環境設定をコードで管理し一貫性を保証
- **ネットワーク分離**: 外部トラフィックの影響を排除
- **Test First**: スタブに対してテストを先に書き、計測インフラを検証してから本番コードをテスト
