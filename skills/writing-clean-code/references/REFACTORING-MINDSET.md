# リファクタリングマインドセット

## 概要

リファクタリングはスキルであると同時に、**組織文化・マインドセット**でもある。本書の核心は、ルールやパターンの機械的適用だけでなく、「コードを恐れない姿勢」「構造を発見する目」「必要最小限のシンプルさ」という3つの態度にある。

本リファレンスは「何をするか（how）」ではなく「いつ・なぜ・どんな姿勢で（when/why/attitude）」に焦点を当て、REFACTORING-CATALOG.md と補完関係を成す。

---

## 恐れずにコードを変更する（Ch.10ベース）

### コーディング恐怖症の兆候

以下のいずれかに当てはまる場合、チームがコードを恐れているサインである：

- 実際のコードより周辺ツール（CI/CD、テスト環境、監視基盤）の方が洗練されている
- 設計や議論に時間を使いすぎ、手を動かさない
- 完璧なコードを書こうとして先延ばしが続く

**Why（なぜ問題か）**

Googleの大規模調査により、チームの生産性を最もよく予測するのは**心理的安全性**であることが判明している（リスクを取っても安全だとチームメンバーが感じているか）。恐怖は非合理的な先延ばしを生み、最も不確実な領域——すなわち最も学ぶべき領域——を避ける行動を引き起こす。

### スパイク（探索的プロトタイプ）

**スパイクとは**、最終的にメインコードに取り込まれないことを前提とした探索的コードである。

- 目的は「コード」ではなく**「知識」の獲得**
- 品質を気にせず実装することで、恐怖が和らぐ
- スパイクの成果は「3つのポイント＋スクリーンショット」のスライドに集約し、関係者に共有する

**厳守すべきルール**: スパイクで書いたコードは本番に持ち込まない。持ち込んだ瞬間に、スパイクのコードに品質を求めてしまうという悪循環が始まる。

### 割合の固定（ツール過剰投資の回避）

非機能要件（CI/CD、ツール整備、技術的負債）に割り当てる開発時間を **20%** に固定する。

- 残り80%が機能開発に充てられ、複雑さの比率が健全に保たれる
- 効果的な実践例: **金曜日を非チケット作業日**とし、大規模リファクタリングやツール改善に充てる
- 5スプリントごとの「リファクタリングスプリント」は避けること（作業が集中しすぎ、直前4スプリントで結合度が高まる）

### 重複コードの戦略的利用

コード共有とコード複製には、それぞれ異なるトレードオフがある：

| 特性 | コード共有（DRY） | コード複製 |
|------|-----------------|-----------|
| グローバル変更の速度 | 速い | 遅い |
| ローカル変更の速度 | 遅い | 速い |
| システムの脆弱性 | 高い | 低い |

**スパイク中は積極的に複製する**。コードが落ち着いたら、「これはコピー元と結合すべきか？」を問い、必要なら第5章のリファクタリングパターンで統合する。

**判断基準**: 変更時にコピー元も変更すべきか？統合コードは自分たちのチームが管理すべきか？どちらかが"No"なら分離したままにする。

### 機能トグルによるリスク軽減（Ch.10.8）

「コードのデプロイ」と「機能のリリース」を分離することで、未完成コードを安全にメインブランチに統合できる。

```typescript
// Step 1: FeatureToggleクラスを作成
class FeatureToggle {
  static paymentV2(): boolean { return false; } // 初期値はfalse
}

// Step 2: 変更箇所をトグルで囲む
class PaymentService {
  processPayment(order: Order) {
    if (FeatureToggle.paymentV2()) {
      // 新実装（まだ空でもOK）
      processPaymentV2(order);
    } else {
      // 既存実装（そのまま維持）
      processPaymentV1(order);
    }
  }
}
```

トグルが不要になったら: 不要クラスを削除 → インターフェース削除 → メソッドをインライン化 → クラスを削除の順で痕跡をゼロにする。

**導入ステップ**: まず「変更箇所を `if` 文で囲む」習慣を作り、それが自然にできるようになってから本番で活用する。

## コード内の構造を発見する（Ch.11ベース）

### 空行が示す構造的境界（Ch.11.6.1）

開発者は、処理の塊を頭の中でグループ分けし、**空行でそれを表現する**。これは「コードに潜む構造の地図」として機能する。

```typescript
// Before: 空行で区切られた2つのブロック → 2つの責務が存在している
function subMin(arr: number[]) {
  let min = Number.POSITIVE_INFINITY;
  for (let x = 0; x < arr.length; x++) {
    min = Math.min(min, arr[x]);
  }
                                      // ← この空行が「メソッド境界」を示す
  for (let x = 0; x < arr.length; x++) {
    arr[x] -= min;
  }
}

// After: 空行が示した構造をメソッドとして具現化
function subMin(arr: number[]) {
  const min = findMin(arr);
  subtractFromEach(min, arr);
}

function findMin(arr: number[]): number {
  let min = Number.POSITIVE_INFINITY;
  for (let x = 0; x < arr.length; x++) {
    min = Math.min(min, arr[x]);
  }
  return min;
}

function subtractFromEach(min: number, arr: number[]): void {
  for (let x = 0; x < arr.length; x++) {
    arr[x] -= min;
  }
}
```

**フィールドの空行も同様**：フィールド間の空行は、データの関連性を示す。空行で区切られたフィールド群は、別クラスへのカプセル化候補である（`データをカプセル化する` パターンを適用）。

### データによる動作表現（Ch.11.2）

コードに動作を組み込む方法は3つある：

| 方法 | 特徴 | 変更しやすさ |
|------|------|------------|
| 制御フロー（if/for） | 文を移動するだけで大きな変更が可能 | グローバル変更が容易 |
| データ構造（クラス/型） | 型安全性・局所性が高い | ローカル変更が安全 |
| データそのもの | ハードコーディング | 変更不要なドメインに最適 |

「タイプコードをクラスで置き換える」「ストラテジーパターンを導入する」は、制御フローからデータ構造への移行を行うリファクタリングである。

### 構造を明確にするコード追加（Ch.11.3）

リファクタリングの方向性は、コードが**今後どのように変化するか**によって決まる：

- **変化しないコード** → 何もしない（チェスのルールのように500年変わらないなら、ハードコーディングが最善）
- **予測不可能に変化するコード** → 脆弱性回避のためのみリファクタリング
- **特定パターンで変化するコード** → 過去に行ったような変更を受け入れやすい構造にリファクタリング

基盤となる構造が定まっていない段階では、クラスより列挙型・ループを優先し、まず正しさに重点を置く。

### 経験的手法: 予測するのではなく観察する（Ch.11.4）

「賢くあろう」とする罠を避ける：

> コードがどのように変化していくのかを**観察**し、確信が持てない汎用性は実装しない。

**悪い例**:
- チェスの実装でインターフェースを使う（500年変わっていないのに拡張性を持たせる）
- 「きっと使われるだろう」という憶測で機能を追加する

**良い例**:
- 実際に変更が発生してから、その変更に対応できる構造にリファクタリングする
- スパイク＆スタビライズ: 6週間後に実際に使われていないコードは削除する

## 汎用性を最小限にする（Ch.12ベース）

### 汎用性の具体的害悪（結合度の増大）

汎用的なコードは**より多くのものが結合する可能性を生む**。呼び出し元が増えるほど、変更時に考慮すべきパターンが増え、認知負荷が高まる。

```typescript
// 汎用的すぎる関数 → どの Tile 値で呼ばれるか不明で最適化できない
function remove(tile: Tile) {
  for (let y = 0; y < map.length; y++) {
    for (let x = 0; x < map[y].length; x++) {
      if (map[y][x] === tile) {
        map[y][x] = new Air();
      }
    }
  }
}

// 特定用途に特殊化 → 最適化・テスト・削除がしやすくなる
function removeStone() {
  for (let y = 0; y < map.length; y++) {
    for (let x = 0; x < map[y].length; x++) {
      if (map[y][x] instanceof Stone) {
        map[y][x] = new Air();
      }
    }
  }
}
```

### 不要な汎用性の除去手順（Ch.12.2.3）

1. **「削除しコンパイルする」パターン**: コードを削除 → コンパイルエラーが出た箇所だけが実際の使用箇所
2. **引数の監視**: 実行時に渡される引数をログに出力し、常に同じ値が渡されていないか確認する → 特殊化の候補
3. **「メソッドを特殊化する」パターン**: 常に同じ値が渡される引数をハードコーディングして汎用性を除去

### 安定性による統合の判断（Ch.12.2.2）

新しいコードと古いコードをすぐに統合してはならない。**同程度の安定性を持つのを待つ**。

> 新しい実装の2番目のバージョンは非常に早く安定し、3番目のバージョンはさらに早く安定する。

最小限の実装を貫くための指針:

> **行わなくてよい作業を最大化せよ。** — Kent Beck

「注文されたものだけを作る」: 想像上の問題ではなく、今持っている問題を解決する。

## 根底にある3つの原則（Ch.14ベース）

### より小さなステップを探す（Ch.14.2.1）

リファクタリングの本質は**「動くもの」から「別の動くもの」へ**、常にグリーンな状態を保ちながら小さく進むことにある。

- 大きな変更 = 小さな変更の積み重ねで実現できる
- コンテキストスイッチは**グリーンな状態の間**でのみ行う（リファクタリング途中での切り替えはエラーリスクを急増させる）
- 緊急修正時: 最新のグリーン状態まで `git reset` して戻れる——これが小さなステップの真の価値

**機能トグルの導入も小さなステップで**: まず「変更を `if` 文で囲む習慣を作る」→ 自然にできるようになったら本番で活用する。

### 基盤となる構造を探す（Ch.14.2.2）

ミケランジェロの言葉がリファクタリングの哲学を完璧に表現している：

> **すべての石の塊の中には影像が宿っており、彫刻家の仕事はそれを見つけ出すことである。**
> — ミケランジェロ

コードも同様に、良い設計を「作る」のではなく、すでにコードに潜む構造を「発見する」。

構造を発見する実践的アプローチ:
1. **行の並び** → どこにメソッドがあるかを判断（空行・コメントが境界を示す）
2. **メソッドの集まり** → どこにクラスがあるかを判断
3. **クラスの集まり** → どこに名前空間・パッケージがあるかを判断

コツは「内側から始めて外側の抽象層へ連鎖させる」こと。メソッドが1つ多い方が良い——そのメソッドが、別のクラスを作るかどうかの判断材料になるからだ。

### コラボレーションのためにルールを活用する（Ch.14.2.3）

**ルールは法律ではなく、ガイドラインである。**

- ルールを「チームメンバーを監視・非難するための道具」として使ってはならない
- ルールは「コード品質に関する議論の出発点」「どこから手をつけるべきかを示す経験則」として機能する
- 開発チームにとって最も重要なのは**心理的安全性**: ルールが安心感を与えるなら良いことだが、萎縮を生むなら悪いことだ

## リファクタリングの組織文化（Ch.1ベース）

### いつリファクタリングを行うべきか（Ch.1.3）

**リファクタリングは、シャワーを浴びるようなものだ。** — Kent Beck

推奨する6ステップワークフロー:

```
1. 探索する   → スパイクでプロトタイプ、要件を明確化
2. 仕様化する → 何を作るかを明示的に定義（自動テストの形が理想）
3. 実装する   → コードで実装
4. テストする → 仕様を満たすことを確認
5. リファクタリングする → 次の人（自分かもしれない）が作業しやすくする
6. リリースする → コードをユーザに届ける
```

リファクタリングのステップ5の中身:
```
メソッドを選ぶ
  → ルールに違反していないか？
    → Yes: 対応するリファクタリングパターンを見つけ、指示に従う
    → No: 次のメソッドへ
コンパイルエラーを修正 → 再チェック
```

### レガシーコードとの共存戦略（Ch.1.3.1）

> **まず変更しやすく、その後、しやすくなった変更を行う。** — Kent Beck

全ての作業を止めてコード全体をリファクタリングする必要はない。新しい機能を実装する際に、**まずそのコードを追加しやすい状態にしてから機能を追加する**。

### リファクタリングをすべきでないのはいつか（Ch.1.3.2）

以下の3つのケースでは、リファクタリングはコストに見合わない：

1. **スパイクコード**: 1回しか実行せず削除予定のコード（エクストリームプログラミングでいうスパイク）
2. **廃止予定コード**: メンテナンスのみで廃止が決まっているコード
3. **パフォーマンス最優先コード**: 組み込みシステムやゲームの高性能物理エンジンなど

## 最適化に対する原則（Ch.12.3）

### 「推定無罪」の原則

> コードは**効率的でないと証明されるまで**、効率的であるとみなされる。

最適化を行う動機付けとして、自動パフォーマンステストを作成し、テストが通らなかった場合にのみ最適化を検討する：

- **ベンチマークテスト**: 「このメソッドは14ミリ秒以内に完了しなければならない」
- **負荷テスト**: 「このサービスは1秒間に1000件のリクエストを処理できること」
- **パフォーマンス確認テスト**: 「前回の実行時間より10%以上遅くなってはならない」（外部要因の影響を受けない）

### 最適化前にリファクタリングする（Ch.12.3.1）

リファクタリングの目的の1つは**不変条件を局所化すること**。最適化は不変条件に依存するため、適切にリファクタリングされたコードの方が最適化しやすい。

**手順**:
1. コードが適切にリファクタリングされているか確認
2. スレッドスケジューリングに無駄がないか確認
3. ボトルネックを特定
4. プロファイリングで最適化対象メソッドを絞り込む
5. 適切なデータ構造・アルゴリズム選択またはキャッシュ導入
6. 最適化コードをカプセル化して隔離する
