# 変更容易なコードの設計原則

レガシーコードへの対処戦略と、変更しやすいコードを維持するための設計原則をまとめたガイド。

---

## コード品質の定義

> **高品質なコードとは、変更しやすいコードである。**

変更容易性は、コードの読みやすさ・保守性・拡張性・堅牢性の副産物でもある。

| 変更しやすいコードの特性 | 変更しにくいコードの特性 |
|----------------------|----------------------|
| 意図が一目で理解できる | 理解に時間がかかる |
| テストが書けている | テストが書けない・存在しない |
| 依存関係が明確 | 依存関係が複雑に絡み合っている |
| モジュールが独立している | 変更が他の場所に波及する |

---

## レガシーコードへの5ステップアプローチ

レガシーコードは一括書き換えではなく、段階的な改善で対処する。
作業が必要な箇所に絞って集中的に改善することで、リスクを最小化しながら品質を上げていく。

### ステップ順序は厳守する（スキップ不可）

```
1. Approval Testing   ← 現在の振る舞いを固定する
2. Remove Clutter     ← 不要なコードを削除する
3. Reduce Complexity  ← 複雑性を下げる
4. Compose Methods    ← メソッドを構成する
5. Refactor to Testability ← テスト可能な構造にする
```

---

### Step 1: Approval Testing（承認テスト）

**定義**: 現在の振る舞いをキャプチャしてアサートするテスト（Characterisation Testとも呼ばれる）。

初回実行時は失敗する。その出力を「承認」として保存し、以降の実行で同じ出力が得られるかを検証する。

**目的と効果:**

| 目的 | 効果 |
|------|------|
| 現在の振る舞いを固定する | 意図しない変更を検出できる |
| 変更が振る舞いを保存することを保証する | 自信を持ってリファクタリングできる |
| どんなコードにも追加しやすい | すぐに安全網を作れる |

> Refactoringは**常に振る舞いを保存する**。承認テストはその保証手段。

---

### Step 2: Remove Clutter（不要物の削除）

**削除対象:**
- コメントアウトされたコード
- 使われていない関数・変数
- 不要なコメント（「何をするか」を説明するコメント）
- 到達不能なコード（dead code）

**原則**: バージョン管理があれば、dead codeを残す理由はない。削除することで構造と意図が見えやすくなる。

**効果:**
- コードの構造が可視化される
- 承認テストでカバーされないコードが消えるため、テスト対象が明確になる

---

### Step 3: Reduce Complexity（複雑性の削減）

Cyclomatic Complexity（実行パス数）を下げることで、コードを変更しやすくする。

**具体的なアプローチ:**

| 手法 | 説明 |
|------|------|
| インデント階層を減らす | `for`/`while`/`if`のブロックをExtract Methodする |
| `break`/`continue`の排除 | コードの流れを理解した上で除去する |
| 単一の出口点 | 各メソッドのreturnは1箇所を目標にする |
| Extract Method | 長いパラメータリストをオブジェクトでグループ化する |
| 条件式の単純化 | 複雑な条件・ループは独立したメソッドに抽出する |

---

### Step 4: Compose Methods（メソッドの構成）

コードブロックに名前をつけ、「関数がストーリーを語る」状態にする。

**関数の理想的な形:**
```
// NG: 何をしているか全部書いてある
function processOrder() {
  // 100行のインライン処理...
}

// OK: サブメソッドの呼び出しがストーリーになっている
function processOrder() {
  validateInput();
  calculatePrice();
  applyDiscount();
  saveToDatabase();
  sendConfirmationEmail();
}
```

**手順:**
1. 関連するコードをグループ化する
2. 非関連コードを分離する
3. 各ブロックを説明的な名前でExtract Methodする
4. IDEのリファクタリング機能を活用する（ミスが減る）
5. 繰り返す（サブメソッドも同様に構成する）

**基準**: 元のメソッドが「サブメソッドへの呼び出しのみ」になれば完成。

---

### Step 5: Refactor to Testability（テスト可能な構造へ）

テストが書けない = 変更が怖い。テスト可能な構造こそが変更容易性の土台。

**2つの核心原則:**

```
関連するコードは近くに   → Cohesion（凝集性）の向上
非関連コードは遠くに    → Modularity（モジュール性）の向上
```

**効果:**
- ユニットテスト・統合テストが書けるようになる
- 保守性・信頼性・品質が向上する
- テストカバレッジが上がる

---

## 変更容易なコードの設計原則

レガシーコードだけでなく、新規コードを書く際にも適用する原則。

### 可読性を優先する

| すべきこと | 避けること |
|----------|----------|
| 意図を明確に表現する変数名・関数名を使う | 解読が必要な短縮名・汎用名（`data`, `temp`等）|
| コードを自己説明的にする | 「何をするか」を説明するコメント |
| 「なぜ」の判断背景のみコメントに残す | コードと乖離する可能性があるコメント |

> コメントは真実ではないかもしれない。コードが唯一の真実。

---

### 複雑性をデザインで管理する

変更しやすいコードの4つの設計特性:

| 特性 | 定義 | 変更容易性への貢献 |
|------|------|-----------------|
| **Modularity** | コードを小さな独立した部品に分ける | 1つのモジュール変更が他に波及しない |
| **Cohesion** | 関連する概念を近くに配置する | 変更箇所が明確になる |
| **Separation of Concerns** | 各部分が1つのことに集中する | 変更の影響範囲を限定できる |
| **Abstraction** | 内部実装を隠蔽する | 内部を変えても外部に影響しない |

**疎結合（Loose Coupling）** をデザインの目標にする:
- ある場所の変更が別の場所に漏れ出ないようにする
- 抽象化とその他の設計パターンで実現する

---

### Separation of Concernsの実装パターン

| パターン | 説明 |
|---------|------|
| Layered Architecture | UI層・ビジネスロジック層・データアクセス層を分離する |
| Modular Design | 特定のドメインや機能をカプセル化したモジュールに分割する |

---

### 自動テストとCI

変更を安全に行うための安全網:

| 手法 | 目的 |
|------|------|
| Unit Test | 各コンポーネントが意図通り動くことを保証する「ミニ仕様書」 |
| Continuous Integration | コミット時に自動でテストを実行し、早期にエラーを検出する |

---

## まとめ: 変更容易性チェックリスト

新しいコードを書く際、またはリファクタリング時の確認項目:

- [ ] 関数名・変数名が意図を明確に表現しているか
- [ ] 各関数が「1つのこと」をしているか（Compose Methods）
- [ ] テストが書けるか（Separation of Concerns, 依存の注入）
- [ ] dead codeが存在しないか（Remove Clutter）
- [ ] 深いネスト・複雑な条件式がないか（Reduce Complexity）
- [ ] 変更が1箇所に留まるか（疎結合・情報隠蔽）

> 最初から完璧にする必要はない。コードはソフトウェアのライフサイクルを通じて継続的に改善できる。
