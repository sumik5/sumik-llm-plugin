# 認知科学・理解設計リファレンス

聴衆の脳の処理メカニズムを踏まえたプレゼンテーション設計の実践リファレンス。「分かりやすい」の正体は、脳の作業負担の事前代行である。

---

## 脳内関所モデル（藤沢晃治）

### 原理

脳の短期記憶を「脳内関所」、長期記憶を「脳内整理棚」と定義する認知モデル。情報は脳内関所で5段階の処理を経て、長期記憶の整理棚に格納される。

| 処理段階 | 内容 | 詰まるケース |
|---------|------|------------|
| **① サイズチェック** | 情報が小さな口（関所）を通れるか | 専門用語・複文・長文 |
| **② 整理棚選定** | どの引き出しに入れるか | 文脈がない・カテゴリが不明 |
| **③ 構造分析** | 情報の骨格を把握する | 構造が不明・箇条書きの羅列 |
| **④ 論理性審査** | 主張に矛盾がないか確認する | 根拠が薄い・飛躍した論理 |
| **⑤ 意味確定** | 何を言いたいのかを確定する | 結論が不明・余白がない |

**核心的定義**: 分かりやすい説明 ＝ 脳内関所の作業負担の事前代行

### プレゼン設計への応用

作り手がすべき仕事は「5段階の処理を聴衆の代わりに先回りして行うこと」である。スライドを見た聴衆が何も考えなくても自然に理解できる状態を目指す。

### 実践のポイント

- **①のために**: 1スライド1メッセージ・文章を短く切る・専門用語は即定義
- **②のために**: 最初に「今日話すカテゴリ」を宣言しておく
- **③のために**: 図解・見出し構造で骨格を視覚化する
- **④のために**: 根拠→結論の順、または結論→根拠の順を一貫させる
- **⑤のために**: スライドの末尾にSo What?を明記する

## タイムラグの法則（藤沢晃治）

### 原理

話し手は内容を熟知しているが、聴衆は初見である。新しい話題が提示されてから聴衆がそれに慣れるまでには、必ず**時間差（タイムラグ）**が生まれる。話し手はこのタイムラグを「ないもの」として扱いがちである。

```
話し手の認識  ━━━━━━━━━━━━━━━━━━━━━━━━→ 新スライド
聴衆の認識    ━━━━━━━━━━━━━━━━━━━━━━━━━━━━→ 数秒後にようやく追いつく
```

### プレゼン設計への応用

| 状況 | タイムラグ短縮策 |
|------|--------------|
| 新しいスライドに切り替えた直後 | 2〜3秒間、無言の間を置く |
| 新概念を初めて提示するとき | 「これから〇〇について説明します」と概要を先に示す |
| 複雑な図表を見せるとき | まず全体像を説明してから個別要素に入る |
| セクションが変わるとき | 「ここからは〇〇の話です」とブリッジを入れる |

### 実践のポイント

- スライドを切り替えたら「すぐ話し始める」衝動を抑制する
- 概要→詳細の順番で提示し、聴衆のメンタルマップを先に形成させる
- 慣れていない言葉は最低3回以上登場させる（初見・復習・定着）

## ビール瓶の原理（藤沢晃治）

### 原理

脳内関所はビール瓶の小さな口のように容量が限定されている。大量の情報を一度に注ぎ込もうとすると、受け取れずに溢れる。脳の短期記憶が保持できる情報量は**約10文字・7±2チャンク**程度とされる。

```
❌ 悪例: ドバドバ注ぐ
[情報]→→→→→→→→→→→→→→→→[脳内関所の小さな口]→溢れる

✅ 良例: しみ入るように注ぐ
[情報]→（少量）→[脳内関所]→格納 → （少量）→[脳内関所]→格納 → 繰り返し
```

### プレゼン設計への応用

| 設計原則 | 実装 |
|---------|------|
| 1スライドに詰め込まない | 伝えたいことは1枚1つ |
| 箇条書きは5項目以下 | それ以上はスライドを分割 |
| 文字数を絞る | キーワードだけ残し、詳細は口頭で補足 |
| 図解で圧縮する | テキスト情報を視覚情報に変換してチャンク数を減らす |

### 実践のポイント

- スライドから文字が溢れているなら、それは「ビール瓶を傾けすぎ」のサイン
- 資料として手元で読むドキュメントと、スクリーンに投影するスライドは別物として設計する
- 「削ることは付け足すこと」 — 情報を削るほど残った情報が際立つ

## 情報共有の錯覚（藤沢晃治）

### 原理

話し手が「聴衆も当然知っている」と錯覚し、前提情報の説明を省略してしまう現象。自分にとって「当たり前すぎること」が最大の盲点になる。特に専門家・熟練者ほど陥りやすい（専門家の呪い）。

### プレゼン設計への応用

```
情報共有の錯覚が起きるプロセス

① 作り手が長年積み上げた暗黙知
　↓ 「これくらい知っているだろう」
② 前提説明を省略してスライドに記述
　↓ 聴衆は前提を持っていない
③ 聴衆が「ついていけない」と感じる
　↓ 興味が薄れる・信頼が下がる
```

| 対策 | 具体的な方法 |
|------|------------|
| **第三者チェック** | そのテーマを知らない人に見せて「わからない言葉」をマークしてもらう |
| **視点移動** | 「自分が初めてこれを聞く立場だったら何を知りたいか」を書き出す |
| **用語リスト** | プレゼン内に登場する専門用語を列挙し、定義を準備する |
| **ゼロ前提設計** | 聴衆が何も知らないと仮定してスライドを作り、後から削る |

### 実践のポイント

- 「この言葉を最初に聞いたとき、自分はどう感じたか」を思い出す
- 業界外の人に30秒で説明できるかをテストする
- インタビューで「聴衆が実際に使う言葉」を収集し、それをそのままスライドに使う

## 3-5-7ルール（ロジック構築の技術）

### 原理

認知心理学の知見（Miller の 7±2、Cowan の 4±1）を実践的に整理した記憶・理解の制約ルール。情報の種類によって「適切な数」が異なる。

| 用途 | 上限 | 根拠 |
|------|------|------|
| **記憶させたい情報**（Take Away） | **3つ** | 翌日まで確実に記憶されるのは3項目まで |
| **理解させるロジック**（論拠・根拠） | **5つ** | 論理的理解の限界点 |
| **複雑な場合の最大値** | **7つ** | Miller の短期記憶上限。これを超えると全体把握不能 |

### プレゼン設計への応用

```
設計チェックフロー

Q1. このスライドで「覚えてほしい」ことは何個？
　→ 3個以下 → OK / 3個超 → 削る or 分割

Q2. このスライドの論拠（根拠）は何個？
　→ 5個以下 → OK / 5個超 → グループ化 or 分割

Q3. 全体を通じて最も重要なメッセージは3つに絞れているか？
　→ Yes → OK / No → 再優先度付け
```

### 実践のポイント

- 「言いたいことは10あるが伝えるのは3だけ」が設計の基本姿勢
- 5を超える根拠が必要なときはグループ化してラベルを付ける（例: 3つのメリット・2つのリスク）
- セクション終わりに「このセクションの3点」でまとめスライドを入れる

## メンタルモデル活用（ロジック構築の技術）

### 原理

人間は新しい情報を受け取る前に「全体像の枠組み（メンタルモデル）」が用意されているほど、詳細情報を正確に理解・記憶できる。概略を先に提示することで、後続の詳細情報が「適切な引き出し」に格納される。

### プレゼン設計への応用

| タイミング | メンタルモデル形成の実装方法 |
|-----------|--------------------------|
| プレゼン冒頭 | 「今日は A・B・C の3点を説明します」と全体構造を宣言 |
| 各セクション開始 | 「ここでは○○について、3つの観点から説明します」 |
| 複雑な図表の前 | 「この図は〇〇を表しています。全体は3ゾーンに分かれています」と説明してから詳細へ |
| 新概念の導入時 | 身近なたとえ（メタファー）で概念の輪郭を先に作る |

```
✅ メンタルモデル先行の構成例

[全体宣言] 「本日は課題・原因・解決策の3点でお話しします」
　↓
[課題] 詳細説明
　↓
[原因] 詳細説明
　↓
[解決策] 詳細説明
　↓
[まとめ] 「つまり課題は〇〇、原因は〇〇、解決策は〇〇です」
```

### 実践のポイント

- 目次スライドはメンタルモデル形成の最も基本的な実装
- 「これから言うことの地図」を常に聴衆に持たせ続ける
- 同じ目次スライドを節目ごとに繰り返し挿入し、現在地を示す（全体位置インジケータ）

## 情報の非対称性（ロジック構築の技術）

### 原理

発信者（プレゼンター）は結論・全体像・文脈をすべて知っているが、受信者（聴衆）はその一部しか知らない。この「知っていること」の差が「分かりにくい」という体験の根本原因の一つ。

```
発信者が知っていること
  ┌─────────────────────────────────────┐
  │  結論   全体像   文脈   前提   経緯  │
  └─────────────────────────────────────┘

聴衆が知っていること（プレゼン開始時）
  ┌─────────┐
  │  文脈だけ │  ← 大きなギャップ
  └─────────┘
```

### プレゼン設計への応用

| 非対称性の症状 | 解消策 |
|-------------|-------|
| 「なぜ突然この話になったのか」と感じさせる | セクション遷移ごとに接続ブリッジを入れる |
| 「スライドがどこに向かっているか」が分からない | 離れたスライド間の因果関係を図示する |
| 「これは何のためのデータか」が分からない | データ提示前に「目的のスライド」を入れる |
| 前のセクションとのつながりが分からない | 前スライドのキーワードを縮小して次スライドに配置する |

### 実践のポイント

- 「聴衆は自分が何を知らないかを知らない」 — 作り手が差分を埋める責任を持つ
- プレゼン全体の「情報の流れ図」を裏紙に描き、飛躍がないか確認する
- 第三者レビューで「ここ、なんで急にこの話になったの？」と感じた箇所を全てブリッジで補完する

## 強調のポジション（ロジック構築の技術）

### 原理

人間の集中力は「最初と最後」が最も高まる。これはプレゼン全体・各セクション・各スライド・各文のどのレベルでも同様に機能する（初頭効果・親近効果）。

| レベル | 強調の「最初」 | 強調の「最後」 |
|-------|-------------|-------------|
| プレゼン全体 | 冒頭（つかみ・核の提示） | 締め（最重要メッセージの繰り返し） |
| 各セクション | セクション見出しの直後 | セクションまとめスライド |
| 各スライド | トピックセンテンス（見出し） | ポイント（So What?） |
| 各文 | 文頭キーワード | 文末の結論語 |

### プレゼン設計への応用

```
スライド設計の基本パターン

[見出し（強調したい主張を書く）]

  ↓ 支持情報・根拠

[末尾のポイント（So What? を明記する）]
```

### 実践のポイント

- 「最も伝えたいこと」をスライドの先頭の見出しに書く（中ほどに埋めない）
- スライドの末尾に「まとめの一文」を必ず置く
- 冒頭から3スライド以内に「このプレゼンの核（最重要メッセージ）」を登場させる
- 「最後だから印象に残りやすい」を利用して、Q&A前に最重要メッセージをもう一度見せる

## 問いかけの二重効果（藤沢晃治）

### 原理

キーポイントを伝える直前に問いかけを挿入すると、次の2つの効果が同時に発生する。

| 効果 | メカニズム |
|------|-----------|
| **① 重要情報の予告** | 「この問いに答えるのが次の情報だ」と事前に告知される |
| **② 概要説明** | 問いの構造自体が「どんな情報が来るか」の概要になる |

脳内関所の「全作業員」が問いに注目し、次の情報を受け取る準備態勢を整える。

### プレゼン設計への応用

```
問いかけの効果的な使い方

❌ 悪例（いきなり結論）
  「当社の売上は昨年比150%になりました」

✅ 良例（問いかけ先行）
  「昨年と比べて売上がどう変化したか、ご存知でしょうか？」
  　→ 「実は昨年比150%を達成しました」
```

| 問いかけのタイプ | 使用タイミング | 例 |
|--------------|-------------|-----|
| 修辞疑問（答えを求めない） | 重要な主張の直前 | 「なぜ、この市場が重要なのでしょうか？」 |
| 選択式 | 対比・比較のとき | 「AとBのどちらが効果的だと思いますか？」 |
| 数字の推測 | インパクトある数字の前 | 「この割合、何%だと思いますか？」 |
| 経験への問い | 共感を引き出すとき | 「こういう経験、したことはありませんか？」 |

### 実践のポイント

- キーポイントのスライドの直前に「問いかけスライド」を1枚挿入する習慣をつける
- 問いかけは短く（15文字以内が理想）、答えを想起させやすい形にする
- 答えを言う前に「2〜3秒待つ」ことで聴衆が考える時間を作る

## まとめ言葉によるダメ押し（藤沢晃治）

### 原理

「要するに…」「つまり…」「一言で言えば…」といったまとめ言葉は、脳内関所で「住所がほぼ決定された情報」の格納先を確定させる働きをする。まとめ言葉は情報理解の「後押し（ダメ押し）」として機能する。

```
まとめ言葉の脳内処理イメージ

[詳細情報を受け取り処理中]
　↓
「要するに…」← ここで格納先が決定される
　↓
[長期記憶の適切な引き出しへ格納完了]
```

### プレゼン設計への応用

| まとめ言葉 | 使用場面 |
|-----------|---------|
| 「要するに」「つまり」 | 複数の情報を受け取った後の集約 |
| 「一言で言えば」 | 複雑な概念を簡潔に定義するとき |
| 「言い換えると」 | 難しい表現を平易にするとき |
| 「結論として」 | 根拠を積み上げた後に結論を出すとき |
| 「まとめると」 | セクション終わりの要約 |

### 実践のポイント

- 長い説明の後には必ず「つまり一文」を用意する
- スライドのまとめページの見出しに「つまり〇〇だ」と書く
- 口頭の言葉でも「要するに…」を意識的に使う — 聴衆が「ここが結論だ」と分かる
- まとめ言葉の後の文章は15文字以内に圧縮する

## 要約力と階層見出し構造（藤沢晃治）

### 原理

情報には「大見出し・中見出し・小見出し」の3段階の粒度があり、持ち時間や聴衆の理解レベルに応じて、どの粒度で説明するかを選択できる。この「粒度コントロール」がプレゼンの柔軟性と分かりやすさを生む。

| 粒度 | 内容 | 使用場面 |
|------|------|---------|
| **大見出し**（抽象度高） | カテゴリ・テーマ | 時間が短い・概要のみ伝えたい |
| **中見出し**（中間） | 各要素の主張 | 通常のプレゼン・意思決定者向け |
| **小見出し**（具体度高） | 詳細・データ・手順 | 時間が十分・実務者向け・操作説明 |

### プレゼン設計への応用

```
時間×聴衆別の粒度選択

5分プレゼン  → 大見出し（3点）だけ
15分プレゼン → 大見出し + 中見出し
30分以上    → 大見出し + 中見出し + 小見出し（一部）

エグゼクティブ向け  → 大見出し中心・詳細はAppendix
実務担当者向け     → 中〜小見出しまで
混合聴衆         → 大→中→詳細の順で層状提供
```

### 実践のポイント

- スライドを作り始める前に「大見出しレベルの目次」を確定させる
- 目次が完成したら中見出し・小見出しを付け足していく（トップダウン設計）
- Q&Aで詳細を聞かれたら、小見出しレベルで回答するためのAppendixを準備する
- 「このスライドは大・中・小のどの粒度か」を意識して設計する

## Silence is Goal（浅田すぐる）

### 原理

プレゼンの理想形は「資料を見せるだけで理解できる状態（スタンドアロン型）」。できるだけ話さなくてもOKな状態を目指すことで、資料の自走力が高まり、後から参照した人にも伝わるプレゼンになる。

```
プレゼン資料の自走度スペクトル

[要説明型]                          [スタンドアロン型]
「口頭で補足しないと                 「資料を見るだけで
全く伝わらない」                      内容が理解できる」
     ◀━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━▶
                              ↑ここを目指す
```

### プレゼン設計への応用

| スタンドアロン化の実装 | 具体的な方法 |
|---------------------|------------|
| スライドにナレーション文を入れる | 各スライドにスピーカーノートを充実させる |
| 図解に全ての情報を含める | 図のキャプションで意図と結論を補足する |
| 接続詞・遷移ロジックを視覚化する | 「だから」「しかし」をアイコンや矢印で表現 |
| 用語を定義する | 専門用語は初出時に括弧内で定義 |

### 実践のポイント

- 「自分が不在でもこの資料だけで相手が動けるか」をチェック基準にする
- 資料を見た第三者が「要点は〇〇」と言えるかをテストする
- 後日メール転送されることを前提に設計すると自走度が上がる
- 「話さなくてもわかる資料」を作ると、プレゼン当日は強調に集中できる

## 糖衣錠テクニック（藤沢晃治）

### 原理

薬を飲みやすくするために砂糖でコーティングした「糖衣錠（コーティング錠）」のように、拒絶されそうな内容（苦い薬）を伝える際、まず聴衆が受け入れやすい内容（甘いコーティング）で包んでから本題に入る説得技法。

```
糖衣錠の構造

[ポジティブ（同意・共感・賞賛）]  ← 甘いコーティング
　↓
[ネガティブ（課題・問題・懸念）]  ← 苦い薬の本体
　↓
[提案（解決策・代替案・アクション）] ← 飲み込みやすくする
```

### プレゼン設計への応用

| 場面 | 糖衣錠の実装例 |
|------|-------------|
| 反対意見が予想される提案 | 「現在の方法の成果を認めた上で、さらなる改善策として…」 |
| 予算削減を伝えるとき | 「チームの努力で達成してきた実績を踏まえ、新たな優先事項として…」 |
| ライバルの良さを認めてから差別化 | 「競合他社のAという点は優れています。その上で、私たちは…」 |
| 厳しいフィードバックを伝えるとき | 「〇〇という部分の設計は素晴らしい。加えて〇〇を改善すると…」 |

### 実践のポイント

- ポジティブ部分は本気で探す（お世辞では聴衆に見透かされる）
- ネガティブ部分は事実ベースで短く・具体的に
- 提案部分で終わる — 批判で終わると関係が悪化する
- 「ライバルを魅力的に描いてから転じる」技法と組み合わせると効果倍増

## 画像優位性効果（Picture Superiority Effect）

### 原理

人間の記憶において、テキスト情報よりも画像情報の方が大幅に記憶されやすいという実証された認知現象。

| 提示形式 | 記憶率（72時間後） | 根拠 |
|---------|----------------|------|
| 音声のみ | **約10%** | 単一チャンネルでの処理 |
| 音声＋画像 | **約65%** | 複数チャンネルでの相互強化 |
| 画像単独（大量実験） | **90%以上** | 2500枚の写真で実施した実験で90%以上が想起された |

**核心的メカニズム**: 脳は言語情報と視覚情報を**別々のシステム**で処理する。画像で提示された情報は両システムに同時記録されるため、想起経路が倍増する。

### プレゼン設計への応用

- テキスト箇条書きを**1枚の強烈な写真**に置き換える
- グラフや図解を積極的に活用し、数字を「見える形」にする
- キーメッセージに対応するビジュアルを必ず添える（テキストだけのスライドは記憶定着が低い）

### 実践のポイント

- 「写真1枚 + キャプション1文」がテキスト10行より強い
- 使用写真は「感情的に反応できるもの」を優先する（中立的な画像は効果が低い）
- ストック写真より、実際の現場・人物・製品の写真が記憶定着を高める

## デュアルコーディング理論（Allan Paivio）

### 原理

認知科学者Allan Paivoが提唱した理論。人間の記憶システムは**言語システム**と**視覚（非言語）システム**の2つが独立して存在し、同じ情報を両方でコード化すると想起・理解が著しく向上する。

```
テキストのみ
  → 言語システムのみに格納
  → 想起経路: 1つ

テキスト＋画像
  → 言語システム＋視覚システムに格納
  → 想起経路: 2つ → 想起確率が倍増
```

### プレゼン設計への応用

| 情報タイプ | デュアルコーディングの実装 |
|----------|----------------------|
| 概念・定義 | 説明テキスト ＋ 概念図・アイコン |
| 数値データ | 数値テキスト ＋ グラフ・棒の視覚化 |
| プロセス・手順 | テキスト箇条書き ＋ フロー図 |
| 比較・評価 | 文章説明 ＋ マトリクス表・対比スライド |

### 実践のポイント

- スライドは「読む資料」ではなく「見て聞く体験」として設計する
- 口頭説明（言語）＋スライドビジュアル（視覚）の組み合わせが最強のデュアルコーディング
- 視覚情報は「テキストの補足」ではなく「記憶の別チャンネル」として意図的に設計する
- 画像が「意味を持っている」こと（ランダムなイラストでなく内容と対応）が前提条件

## 18分ルール（TED設計原則）

### 原理

TEDが講演時間を最大18分に制限する背景には、**認知的バックログ（Cognitive Backlog）**という神経科学的な根拠がある。長時間の情報摂取は脳の情報処理能力を超え、記憶の定着どころか既に処理した情報まで失われる。

```
認知的バックログのメカニズム

[情報入力] → [作業記憶] → [長期記憶]
                ↑
         限界を超えると
         「バックログ（積み残し）」が発生
         → 処理されない情報が増加
         → 全体の定着率が下落
```

**「制約がクリエイティビティを生む」**: TEDが18分という制約を課した理由は単に「長い=悪い」ではない。制約を設けることで話者は「本当に伝えるべき1つのことは何か」を強制的に問われ、メッセージが明確化される。

| 実験 | 結果 |
|-----|------|
| 45分のプレゼンを18分に凝縮した場合 | コアメッセージの明確度が上昇。冗長部分が除去されることでインパクトが増大 |

### プレゼン設計への応用

- 準備段階で「このプレゼンが18分しかできないとしたら何を話すか」を問う
- 全体の構成は「18分相当の密度」を基準に設計し、時間があれば付録を追加する形をとる
- セクションごとに「このセクションは本当に必要か」を18分ルールで問い直す

### 実践のポイント

- 長さと価値は比例しない —「短く濃い」ほど記憶に残る
- 削ることへの恐怖を克服する: 削った情報は消えるのではなくAppendixに移動する
- 「1時間の予定で30分で終わった」は成功。逆は失敗

## 3点ルール（Rule of Three）

### 原理

人間の短期記憶が同時に保持できる情報量は**3〜4チャンク**が上限であるという認知科学の知見（Millerの7±2の更新研究）。「3」という数字は認知的に最も安定した構造を持ち、記憶・説得力・完結感を最大化する。

| 認知研究 | 結論 |
|---------|------|
| Miller（1956年）「マジカルナンバー7±2」 | 短期記憶の容量は5〜9チャンク |
| Cowan（2001年）・最新研究 | 実際の限界は**3〜4チャンク**が更に正確 |

**身の回りの「3」の法則**:
- アメリカ独立宣言: 「生命・自由・幸福の追求」（3項目）
- 三銃士（Dumas）
- 光の三原色（赤・緑・青）
- 仏教の三宝（仏・法・僧）

「なぜ3が強いのか」: 2は単純すぎる（対比しかできない）、4以上は多すぎる（記憶への負担増）。3は「少なすぎず多すぎない」完全な均衡点。

### メッセージマップとの関連

「3点ルール」はメッセージマップの設計原則と直結する。ヘッドライン（全体メッセージ）を3つのキーメッセージで支え、それぞれを3つの補強情報で裏付けることで、聴衆が整理・記憶しやすい論理ツリーが完成する。

### プレゼン設計への応用

| 要素 | 3点ルールの適用 |
|------|-------------|
| テイクアウェイ | 「今日覚えてほしいことは3つ」で設計 |
| 箇条書き | 1スライドの項目は3つを基本に（最大5つ） |
| 根拠 | 主要な論拠は3本柱で構成 |
| セクション分け | プレゼン全体を3部構成（序論・本論・結論）に |

### 実践のポイント

- 「4つ言いたいことがある」場合は2グループに分け、それぞれ2+2で見せるより3+1に再構成する
- 「3つ目は何か」と常に問う: 2点しかなければ根拠不足、4点以上はグループ化が必要
- スライドのクロージング: 「今日お伝えした3点は…」で締めると記憶定着率が上がる
