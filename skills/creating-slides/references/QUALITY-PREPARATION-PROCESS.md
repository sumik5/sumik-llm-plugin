# プレゼン準備プロセスリファレンス

プレゼン資料を作り始める「前」の準備手法。質の高いプレゼンは準備段階で決まる。

---

## インタビュー駆動のプレゼン構築プロセス

プレゼンの最初のステップは、スライドを開くことではなく、聴衆と同じ属性の人に話を聞くことから始まる。

### インタビューの2原則

| 原則 | 内容 | 実践方法 |
|-----|------|---------|
| **リスペクト** | 相手の時間・経験・意見を尊重する | 「教えてください」の姿勢で臨む。自分の仮説を押しつけない |
| **具体的に聴く** | 「一般的にどう思いますか？」ではなく「先週、どんな場面で困りましたか？」 | 5W1Hで深掘りする。特に「いつ・どこで・どうやって」 |

### なぜインタビューがプレゼンを変えるか

- 生の声をそのままプレゼンに引用することで、聴衆が「自分のことだ」と感じる説得力が生まれる
- 作り手の思い込みによる「ズレ」を事前に排除できる
- 聴衆の言語（彼らが実際に使う言葉）でプレゼンを語れるようになる

**原則**: 「10人にインタビューすればプレゼンは10倍よくなる」。1人でも0人より格段に効果的。

## 「小出しアウトプット→フィードバック」の反復構築法

完成度を上げてから見せるのではなく、荒削りの段階からアウトプットを繰り返すことでプレゼンの質を高める。

### 3段階フィードバックサイクル

| 段階 | 形式 | 目的 |
|-----|------|------|
| **ゼロベース相談** | 「こういうプレゼンを考えているんですが」と話す | 構成の方向性確認。大きな修正はここで行う |
| **質疑応答** | ラフなドラフトを見せながら質問を受ける | 理解できない部分・気になる部分の洗い出し |
| **プチ本番** | 小さな場で一通り話してみる | 実際の反応・時間配分・流れの確認 |

### 「つもり」を浮き彫りにする

プレゼン作成者が陥りやすいのは「わかっているつもり」の罠。自分の頭の中では論理が完結していても、初めて聞く人には伝わっていないことが多い。小出しアウトプットはこの「つもり」を早期に発見するための装置として機能する。

**実践**: 思いついたアイデアはSNSや同僚に即座にアウトプット。完璧主義を避け、「ラフスケッチから段階的に精緻化する」プロセスを意識的に設計する。

## ホワイトボードディスカッション起点の作成プロセス

スライドツールを開く前に、アナログのホワイトボードから始めることで、論理構造を自由に発展させられる。

### プロセスの流れ

```
ホワイトボード（アナログ）
├── バッティングピッチャー役との対話
│   └── 問い返し・反論によって論点が磨かれる
├── 全体を写真撮影またはプリントアウト
│   └── 各まとまりを四角で囲む
└── 四角1つ = スライド1枚 の材料へ変換
```

### バッティングピッチャーの役割

| 役割 | 効果 |
|-----|------|
| 聞き手の代理人として素朴な疑問を投げかける | 「当たり前すぎて書いていなかった前提」が可視化される |
| 反論・別視点を提示する | 「弱い論点」「飛躍している論理」が発見される |
| このプロセス自体がリハーサルの代替になる | 本番前に言葉が整理された状態になる |

**注意**: バッティングピッチャーは「批判する人」ではなく「より良くするための協力者」として依頼する。

## 情報収集＝情報発信サイクル

プレゼン準備における情報収集は、同時に情報発信でもある。この等式を意識することでインプットの質と量が飛躍的に向上する。

### 情報発信が情報収集を強化するメカニズム

| ステップ | 内容 |
|---------|------|
| アンテナに引っかかった情報を即発信 | 「面白い」と思ったらその場でSNSや社内チャットに投稿 |
| 発信のために言語化が必須 | 「なぜ面白いか」「誰に役立つか」を言語化する作業が深い理解を生む |
| 発信自体が「メモ」になる | 後から検索・参照できる形で蓄積される |
| 反応がフィードバックになる | 他者のリアクションが「刺さるポイント」の指標になる |

**原則**: 「情報は熱いうちに出せ」。時間が経つと「なぜ面白いと思ったか」の感覚が薄れ、言語化の質が下がる。

## QCD期待値コントロール

プレゼン制作を依頼された場合も、自主的に作成する場合も、QCDの3軸を依頼者（または自分自身）と事前にすり合わせることで後工程の手戻りを防ぐ。

### QCD3軸の定義

| 軸 | 内容 | 確認すべき問い |
|---|------|--------------|
| **Q（Quality/品質）** | どのレベルの品質を求めるか | 概要レベルか詳細か。体裁（デザイン）のレベルは？ |
| **C（Cost/費用）** | 投入できるリソースはどのくらいか | 調査予算・投入人員・優先度はどのくらいか？ |
| **D（Delivery/納期）** | いつまでに完成させるか | 品質とのトレードオフをどこに設定するか？ |

### すり合わせの実践

**手書きイメージの活用**: 文字で説明するより、手書きのラフスケッチを見せながらイメージをすり合わせる方が早く・正確に合意できる。

```
QCDすり合わせのタイミング
├── 作業着手前（必須）: ゴール・制約の合意
├── 中間確認（任意）: 方向性のズレ検出
└── 完成前（推奨）: 最終調整の要否確認
```

**Tip**: 「品質を上げるなら時間がかかる」「時間を短縮するなら品質が下がる」というトレードオフを最初に可視化し、依頼者に選択させることで後からの「なぜこのクオリティなのか」という不満を防ぐ。

## 思考整理ファースト（浅田すぐる）

### 原理

スライド作成においていきなりPCを開くことは禁止。まず手書き3色プロセスで思考を整理し、その後資料化（転記）という2段階で進める。デジタルツールは「清書」であり、「思考」はアナログで行う。

### 手書き3色プロセス

| 色 | 役割 | 使い方 |
|----|------|-------|
| **緑** | 枠（構造） | テーマ・目的・ゴール・大見出しを書く。全体の器を先に決める |
| **青** | 発散（発想） | 枠の中に思いつくことを何でも書き出す。批判禁止・質より量 |
| **赤** | 収束（選択） | 青で書いたことから「伝えるべき核」を絞り込む。重要度ランク付け |

```
3色プロセスの流れ

Step 1（緑）: A3用紙に大見出し/目的/ゴールを書く
　↓
Step 2（青）: 思いつく内容・根拠・事例を全部書き出す
　↓
Step 3（赤）: 伝えるべき順序・核心・削る情報を決定
　↓
Step 4: 赤でマークした内容だけをPCに転記してスライド作成
```

### プレゼン設計への応用

| 手書きフェーズ | PC転記フェーズ |
|-------------|-------------|
| 自由に書き直せる（消しゴム・修正テープ不要） | 確定した構造を清書する |
| 全体像を一覧できる（A3用紙推奨） | スライド単位に分解する |
| 論理の飛躍・漏れが一目で分かる | ビジュアル・フォントを整える |
| 「削る判断」が感情的に楽 | 追加・変更は最小限にとどめる |

### 実践のポイント

- 緑（枠）が曖昧なまま青（発散）に進まない — ゴールが不明なままでは発散が散漫になる
- 青の段階では「くだらないアイデア」も書く — 選ぶのは赤の段階の仕事
- 赤で「残す」ものを選ぶのではなく「削る」ものを選ぶ（残す方が自然と絞られる）
- 手書き終了後、写真を撮って保存してから転記する

## ブレイントラスト（菅野誠二／ピクサー流）

### 原理

「手強い身内」にプレゼン案を建設的に叩いてもらい、壊す→再構築のサイクルを最低1回経由する。ピクサーが映画品質を高めるために実施したクリエイティブレビューの手法をプレゼン準備に応用する。

```
ブレイントラストのサイクル

[初稿]
　↓
[ブレイントラストセッション]
　├── 徹底的な問いかけ・反論・欠点指摘
　└── 「なぜ？」「本当に？」「代替案は？」
　↓
[壊れた状態]（一旦バラバラになる）
　↓
[再構築]（より強固な構造で組み立て直す）
　↓
[完成版]（初稿より格段に品質が上がる）
```

### ブレイントラストの役割設計

| 役割 | 選び方 | 行動指針 |
|------|-------|---------|
| **レビュアー（手強い身内）** | 内容を知っているが発表者ではない人・率直に言える関係 | 「いい」は言わない。「なぜ？」「分からない」「弱い」を言う |
| **発表者** | プレゼン作成者本人 | 防御しない。全て聞いた後に取捨選択する |
| **ファシリテーター** | 第三者（いない場合は発表者が兼任） | 時間管理・議論を発散させすぎない |

### プレゼン設計への応用

- **対象**: 完成前の中間稿（70〜80%完成度のとき実施が効果的）
- **頻度**: 最低1回。重要なプレゼンは2回以上
- **時間**: 30〜60分。時間を決めて集中する
- **アウトプット**: 指摘リストを記録し、翌日に優先度をつけて反映する

### 実践のポイント

- 「壊れる」ことを恐れない — 壊れるのは初稿の弱さであり、プレゼンターの弱さではない
- 「批判してほしい」ではなく「強くしてほしい」という依頼フレームで招集する
- バッティングピッチャー（1人）と組み合わせて段階的にレビューを重ねる
- ブレイントラスト後は24時間以上間を置いてから再構築に取り掛かる

## 六次元フレームワーク発想法（高田貴久）

### 原理

プレゼンのネタ・論点の漏れを防ぐために、物事を「6次元」で網羅的に発想する思考ツール。3次元（物理空間）だけでなく、時間・見えない流れ・人の気持ちを加えることで、普通の発想では見落とす論点を発掘できる。

| 次元 | 内容 | 発想の問いかけ |
|------|------|-------------|
| **X軸（幅）** | 物理的な横の広がり | 「幅・範囲・カバレッジは？」 |
| **Y軸（高さ）** | 物理的な縦の高さ | 「高さ・階層・規模は？」 |
| **Z軸（奥行）** | 物理的な奥行き | 「深さ・内部構造・裏側は？」 |
| **時間軸** | 過去→現在→未来の変化 | 「いつ変わった？今後どう変わる？」 |
| **見えない流れ** | 情報・お金・権限の流れ | 「誰が誰にお金を払う？情報はどこへ？」 |
| **人の気持ち** | 感情・動機・価値観 | 「誰が何を嬉しいと感じる？誰が困る？」 |

### プレゼン設計への応用

```
六次元チェックの使い方

テーマ: 「新サービス導入の提案」

X軸: どのくらいの範囲/ユーザー数をカバーするか
Y軸: 経営レベル〜現場レベルへの影響は？
Z軸: 内部処理・データフロー・セキュリティは？
時間: 導入前・導入中・3年後の状態は？
見えない流れ: 費用負担者・利益享受者・決定権者は誰か
気持ち: 導入推進者・反対者それぞれの感情は？
```

### 実践のポイント

- 6次元すべてを埋める義務はない — 「抜け漏れがないか確認する」目的で使う
- 「見えない流れ」と「人の気持ち」は見落とされがちな最重要次元
- ホワイトボードの6ゾーンに分けて、各次元のアイデアを書き出す
- 発想後に「このプレゼンで触れるべき次元」と「Appendixに回す次元」を選別する

## 会議設計の3視点立方体モデル（高田貴久）

### 原理

プレゼンが行われる「会議・場」の位置を3軸で定めることで、プレゼンの目的と設計指針が明確になる。3軸のどの座標にいるかによって、スライドの構成・内容・トーンが大きく変わる。

| 軸 | 定義 | 端点A | 端点B |
|----|------|------|------|
| **仮説検証軸** | この会議は何を確かめる場か | 仮説を立てる | 仮説を検証する |
| **コミュニケーション軸** | 誰とどんな関係性で話すか | 情報共有（一方向） | 意思決定（双方向） |
| **問題解決軸** | どの段階の問題解決をする場か | 問題発見 | 解決策決定 |

```
立方体モデルの座標例

「新規事業の方向性を決める経営会議」
  仮説検証: 仮説立て（まだ確証がない）
  コミュニケーション: 意思決定（トップが判断する）
  問題解決: 解決策決定（次のアクションを決める）

→ 設計指針: 複数案を提示し、判断材料を揃える構成
```

### プレゼン設計への応用

| 立方体の位置 | 最適なプレゼン構成 |
|-----------|----------------|
| 情報共有 × 問題発見 | 現状報告型（ファクト中心・結論は開放） |
| 情報共有 × 解決策決定 | 提案型（結論先出し・根拠で補強） |
| 意思決定 × 仮説立て | ブレスト型（たたき台提示・発散を促す） |
| 意思決定 × 解決策決定 | 承認型（推奨案１点・比較はAppendix） |

### 実践のポイント

- プレゼン作成前に「この会議は立方体のどこにあるか」を明示的に決める
- 参加者によって期待する軸が異なる場合、キーパーソンの期待軸を優先する
- 「情報共有」の場なのに「意思決定」を求めるプレゼンを持ち込まない（会議の位置のズレがプレゼン失敗の原因になる）
- 3軸の位置をプレゼン冒頭で明示すると「この場の目的」を共有できる

## 準備プロセスチェックリスト

### 着手前

- [ ] ターゲット聴衆と同じ属性の人にインタビューを実施したか
- [ ] QCD（品質・費用・納期）を依頼者または自分自身とすり合わせたか
- [ ] ホワイトボードまたはアナログ手段で論点の初期設計を行ったか
- [ ] 会議設計の3視点立方体モデルで「この場の位置」を特定したか（仮説検証×コミュニケーション×問題解決）
- [ ] 六次元フレームワークで論点の漏れ（時間・見えない流れ・人の気持ち）を確認したか

### 構築中

- [ ] 小出しのフィードバックサイクル（ゼロベース相談→質疑応答→プチ本番）を回したか
- [ ] バッティングピッチャー役との対話で論点を磨いたか
- [ ] アンテナに引っかかった情報はリアルタイムで発信・蓄積したか
- [ ] いきなりPCではなく手書き3色プロセス（緑:枠→青:発散→赤:収束）を実施したか（思考整理ファースト）
- [ ] ブレイントラスト（手強い身内によるレビュー）を最低1回実施し、壊す→再構築サイクルを回したか

### 完成前

- [ ] インタビューで得た生の声を引用または反映しているか
- [ ] 「つもり」が残っていないか（初見の人に見せてフィードバックを得たか）
- [ ] エピソードネタの収集・言語化を行ったか

## TED分析駆動の準備プロセス（実践式）

インタビュー駆動・ホワイトボードセッション・ブレイントラストとは別に、TEDプレゼンを手本とした準備パスがある。国際学会向けの英語プレゼンに固有の課題——スライド設計・話速・アイコンタクト・スクリプト活用——に対して、実証済みの手本から体系的に学ぶアプローチだ（参考: Ch2, Ch15）。

### TED動画の選定と模倣手順

（参考: Ch2.2）

1. TED検索画面で関心分野の発表を選ぶ。英語字幕を表示すると、センテンスの長さと発音の対応を同時に確認できる
2. まず**ジェイ・ウォーカー "The world's English mania"** から始める。5分未満・明瞭な短文・適度な話速が手本として分かりやすい
3. 書き起こし（transcript）をダウンロードし、気に入った表現を記録する
4. スクリプトを声に出しながら動画を再生し、発音・抑揚・間の取り方を体で覚える

### スライドの使い方をTEDから学ぶ

（参考: Ch2.3-4）

TEDプレゼンターが大きく動かなくても引きつけられる理由は、スライドの設計にある。

- スライド1枚は1〜2ワード中心。最大でも19ワード程度
- 統計データは「整数・百分率・具体的な比喩」の3角度で示す（例：「インドの人口の4倍」）
- 背景色と文字色のコントラストを高く保ち、聴衆がプレゼンターの声に集中できる状態を設計する

「いつ・どの場面でスライドを減らせるか」を分析する習慣が、自分のスライドの精度を上げる（参考: Ch2.7）。

### TED評価シートの活用

（参考: Ch2.11-2.12）

TED動画を数本視聴した後、次の問いに答える。

- 内容・プレゼンター・スタイルについて、今覚えていることは何か？
- 自分の集中力が途切れたのはどの時点か、なぜか？

1週間後に同じ問いに答えると、情報量より印象・スタイルの記憶が長く残ることに気づく。このギャップが「大量データを詰め込む意味はない」という設計判断の実感につながる。主要参考文献が提供する評価シートでは、構成・スライド・声・アイコンタクト・フォーマリティを項目ごとに採点し、自分のプレゼンに転用する観点を整理できる。

### リハーサルと自己評価

（参考: Ch15）

| ステップ | 実践 |
|---------|------|
| スクリプトを口になじませる | 声に出して読み、その後メモだけで話す |
| 録画して確認する | アイコンタクト・不自然な身振り・間投詞（um/er）を観察する |
| 同僚との相互評価 | チェックリストを共有し、発音・視線・熱意の3軸でフィードバックする |
| 指導教授へメールで送付 | 発表前に添付し、特定スライドへの意見を求める |
| 重要な内容を前半に置く | 最悪の場合、時間切れでも重要なことが伝わるよう設計する |
| オープニングとエンディングを反復する | この2箇所は即興にしない。第一印象と最終印象は記憶に残りやすい |

録画は「音声のみ→映像のみ→音声+映像」の順で再生すると、それぞれ異なる問題点が見えてくる（音声のみ→論理の流れ、映像のみ→ボディランゲージ）。

### ソフトウェア故障への備え

（参考: Ch15.10）

発表資料は会場のPCへ余裕を持ってアップロードし、動画再生・フォント表示の互換性を確認する。練習中に一度、スライドなしで通す——スライドがなくてもプレゼンできる状態を経験しておくだけで、本番の心理的余裕が変わる。ホワイトボードに図を描く選択肢も準備しておくと、不測の故障に対応しやすい。
