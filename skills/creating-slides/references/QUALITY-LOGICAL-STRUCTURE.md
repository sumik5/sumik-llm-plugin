# ロジカル構成・メッセージ設計リファレンス

プレゼンテーションの論理構造と説得力あるメッセージ設計の実践ガイド。

---

## 1. Why-What-Howフレームワーク

プレゼンテーション全体の設計を3階層で捉えることで、聴衆の思考に沿った構成が生まれる。

### フレームワークの構造

```
Why（なぜ？）
└── この問題・機会はなぜ重要か？
    聴衆の動機・関心・課題認識と結びつける

What（何を？）
└── 達成すべき目標・提案の骨子・選ぶべき方向性
    聴衆が意思決定・行動するために必要な情報

How（いかに？）
└── 具体的な実行計画・手順・資料化・実演
    意思決定後に必要な行動レベルの情報
```

### Why-What-Howの適用例

| プレゼン種類 | Why（動機） | What（骨子） | How（行動） |
|------------|------------|------------|------------|
| 提案型 | 現状課題の深刻さ | 解決策の選択肢 + 推奨案 | 実行スケジュール・必要リソース |
| 報告型 | 報告が必要な理由 | 現状・結果・課題 | 次のアクション・依頼事項 |
| 啓発型 | 変化の必要性 | 新しい視点・洞察 | 今日からできる実践方法 |
| ピッチ型 | 問題の大きさ | ソリューションの優位性 | 投資・協力の具体的依頼 |

### 設計の順序

**逆算で考える**：Howから設計すると情報過多になりがち。Whyから始め、聴衆の動機を確認してからWhatを絞り込み、最後にHowを具体化する。

## 2. プレゼンの種類分類と構成選択

### 4つの基本タイプ

| タイプ | 主目的 | 最重要要素 | 構成の傾向 |
|-------|-------|-----------|-----------|
| **提案型** | 意思決定を促す | 根拠の明確さ・代替案比較 | 帰納法 or 演繹法 + 推奨の明示 |
| **報告型** | 状況・結果を伝える | 事実の正確さ・簡潔さ | 時系列 or 重要度順 |
| **啓発型** | 気づき・視点転換を促す | ストーリー性・問いかけ | 問題提起→洞察→示唆 |
| **ピッチ型** | 支持・投資を獲得する | 信頼性・実現可能性 | 課題→解決→トラクション |

### タイプ選択の判断チェック

- [ ] 聴衆は「何を決める（または考える）」のか？
- [ ] 発表後に期待するアクションは何か？
- [ ] 聴衆の知識レベル・関心はどの程度か？
- [ ] 利用可能な時間・形式は？

## 3. 聴衆分析に基づく構成調整

### 聴衆プロファイル設計

プレゼン構成を決める前に、「1人の具体的な聴衆像」を明確にする。

```
聴衆プロファイルの質問
├── 立場: 意思決定者か？情報収集者か？反対勢力か？
├── 専門性: この分野をどの程度知っているか？
├── 関心: 最も気にしている課題・KPIは何か？
├── 制約: 時間的・予算的・組織的な制約は？
└── 期待: この場に何を期待して来ているか？
```

### 聴衆タイプ別の調整方針

| 聴衆タイプ | 調整ポイント |
|-----------|------------|
| 意思決定者（経営層） | 結論先行、詳細は付録、数値・ROIを強調 |
| 技術専門家 | 詳細な根拠・手法を示す、専門用語OK |
| 一般参加者 | 平易な言葉、具体例多用、「なぜ自分に関係するか」を明示 |
| 懐疑的聴衆 | リスク・懸念を先に言語化し、正面から答える |

## 4. 論理構成の3パターン

### 4.1 帰納法（Bottom-Up）

複数の事実・事例から共通パターンを導き出し、結論へ向かう。

```
事実A ─┐
事実B ─┼─→ 共通パターン → 結論・提言
事実C ─┘
```

**使う場面**: 聴衆が結論を受け入れにくい場合、段階的に説得したい場合
**注意点**: 事実の選定に恣意性がないか確認。反論となる事実も提示すると信頼性が上がる

### 4.2 演繹法（Top-Down / PREP法）

前提→論理→結論の順に組み立てる。

```
Point（結論・主張）
├── Reason（理由）
├── Example（根拠・証拠）
└── Point（結論の再確認）
```

**使う場面**: 時間が限られている場合、結論を先に示して信頼を得たい場合
**注意点**: 前提が正しくないと全体が崩れる。前提の妥当性を検証する

#### 拡張PREP法（EをEvidence+Explanationに分割）

理由だけでは不十分な場面では、EをEvidence（データ裏付け）とExplanation（論理をつなぐ説明）に分割する。

```
Point     （結論・主張）
├── Reason      （理由）
├── Evidence    （根拠データ・事実）
├── Explanation （データと結論をつなぐ論理）
└── Point       （結論の再確認）
```

**使う場面**: 懐疑的な聴衆・データが重要な提案型プレゼン。「理由はわかったが根拠が薄い」と感じさせないための構造強化。
**注意点**: Explanationを省くと「データは示したが、だからどう結論につながるのか」が伝わらない。データと主張の「橋渡し」を明示する。

### 4.3 関係論証法（Pyramid Structure）

トップの主要メッセージの下に、相互に支持し合う論点を並べる。

```
【主要メッセージ（So What?）】
         │
    ┌────┼────┐
 論点1  論点2  論点3
（それぞれに根拠）
```

**使う場面**: 複雑な提案を構造的に整理したい場合
**注意点**: 各論点が主要メッセージを独立して支持しているか確認（MECE: 漏れなく重複なく）

### 4.4 三幕構成（1:2:1）

物語の黄金律を活用した構成。序論・本論・結論の**比率 1:2:1** が聴衆の集中力を維持する。

| 幕 | 役割 | 内容 | 時間比率 |
|----|------|------|---------|
| **オープニング（序論）** | 期待形成・フック | 問題提起・共感・本日のゴール宣言 | 1 |
| **ボディ（本論）** | 主張と根拠 | メインメッセージ + サブメッセージ × 3 + 証拠 | 2 |
| **クロージング（結論）** | 記憶定着・行動喚起 | 要約・メッセージの強調・次のステップ | 1 |

**時間配分の目安**:

| 総時間 | 序論 | 本論 | 結論 |
|-------|------|------|------|
| 5分 | 1分 | 3分 | 1分 |
| 15分 | 3分 | 9分 | 3分 |
| 30分 | 6分 | 18分 | 6分 |
| 60分 | 10分 | 40分 | 10分 |

**よくある失敗**: 序論に業界動向・会社説明を詰め込みすぎて、本論に辿り着く前に聴衆の集中力が切れる。

### 4.5 AIDMA フレームワーク（心理段階設計）

マーケティングの心理プロセスをプレゼン構成設計に応用する。

| 段階 | 心理状態 | 設計の技法 |
|------|---------|-----------|
| **A: Attention（注意）** | 「おや、これは何だ？」 | 短い言葉・謎フック・スッキリしたスライド |
| **I: Interest（関心）** | 「なるほど、聞いてみよう」 | 結論＋根拠＋たとえば の三角構造 |
| **D: Desire（欲求）** | 「いいな、自分にも当てはまる」 | エピソード・心のキャンバス・ビジュアル |
| **M: Memory（記憶）** | 「あの一言、忘れられない」 | 超ひとこと・核・繰り返し |
| **A: Action（行動）** | 「やってみよう / 賛成しよう」 | 情熱・具体的な次のステップ・締めの言葉 |

**使用タイミング**: 提案型・啓発型・ピッチ型。意思決定を促したい場面で特に効果的。

### 4.6 空・雨・傘型（事実→解釈→行動）

報告型・提案型プレゼンに向いた、シンプルな因果構成。

```
空（事実）: 客観的なデータ・現状・観察結果を述べる
　「今年の市場成長率は15%です」

雨（解釈）: その事実が何を意味するかを説明する
　「成長が鈍化傾向にあると言えます」

傘（行動）: とるべきアクションを提示する
　「よって、次の主力製品を今期中に打ち出すべきです」
```

| よくある問題 | 原因 | 解決 |
|------------|------|------|
| 「で、何が言いたいの？」 | 空だけで終わっている | 雨（解釈）を追加 |
| 「結論が唐突」 | 傘から始まっている | 空→雨の根拠を先に示す |
| 「データが多すぎる」 | 複数の「空」が混在 | 最重要の空を1つに絞る |

### 4.7 ストーリー展開の3部構成（エピソード→ショッキングファクト→結論）

聴衆の集中力の波に合わせた感情設計。

| 部 | 内容 | 設計の意図 |
|----|------|-----------|
| **序盤** | 「へえ～」「あるある！」エピソード | 共感で引きつけ、聴く姿勢を作る |
| **中盤** | ショッキングなファクトの突きつけ | 中だるみを防ぎ、集中度を再び高める |
| **終盤** | 結論＋クロージング（未来への希望） | 前向きな行動意欲で締める |

**アンチパターン**: 「ご静聴ありがとうございました」で締めるのはNG。「自分はどうすればいい？」で終わる設計にする。

### 4.8 主観→客観→主観の「右・左・右」思考法

感情（主観）とロジック（客観）の黄金比。

```
主観（直感・想い） → 客観（データ・ロジック） → 主観（感情で押す）
```

| フェーズ | 役割 | 比率の目安 |
|---------|------|-----------|
| 主観（1回目） | 感情的な課題・共感を提示 | 7 |
| 客観 | データ・論理で納得感を確保 | 3 |
| 主観（2回目） | 感情で行動へ押し出す | 7 |

**身近な例**: テレビショッピング「つらい腰痛…（主観）→臨床データで効果実証（客観）→お電話はいますぐ！（主観）」

### 4.9 態度変容の三幕構成（解凍→変化→再凍結）

クルト・レヴィンの態度変容モデルをプレゼン構成に適用。変革提案・意識改革系に最適。

| 幕 | 役割 | 設計の狙い |
|----|------|-----------|
| **第1幕（解凍/Unfreeze）** | 既存の信念・態度に揺らぎを与える | 「自分の考えは正しいのか？」と問わせる |
| **第2幕（変化/Change）** | 新たな方向性を示し、変化の主体性を促す | 「確かにそっちの方がいいかも」と感じさせる |
| **第3幕（再凍結/Refreeze）** | 芽生えた変化を定着・新しい行動を喚起する | 「やってみよう」で終わらせる |

**黄金比率**: 第1幕 : 第2幕 : 第3幕 = 1 : 2 : 1

### 4.10 ピッチの10ステップ構造

短時間プレゼン（5〜30分）に特化した構造。投資・協力を獲得する場面で有効。

| # | ステップ | 内容 |
|---|---------|------|
| 1 | オープニング | キーメッセージ提示 |
| 2 | 顧客と課題 | 痛みの特定（誰が・何に困っているか） |
| 3 | 解決策 | プロダクトの価値 |
| 4 | 競争優位性 | 差別化（他との違い） |
| 5 | Why Now? | なぜ今この解決策が必要か |
| 6 | マーケットサイズ | 市場規模・成長性 |
| 7 | 将来の展望 | 3〜5年ビジョン |
| 8 | 実績 | トラックレコード（信頼性の担保） |
| 9 | Big Ask | 大胆な要望・投資額の提示 |
| 10 | クロージング | キーメッセージの再提示 |

**使う場面**: ピッチ型（投資家・経営層向け）。Why-What-Howのうち「Why」と「What」に比重を置く。

### 4.11 ストーリー4型の使い分け基準（深化版）

既存の展開パターン（分解型・積上型・選択肢型・意外性展開型）に対する、選択基準の精緻化。

**判断の2軸**:
1. 相手の「必要性の認識度合い」（認識済み ↔ 未認識）
2. 相手の「意思決定の段階」（まだ決められない ↔ 受け入れ準備済み）

| 展開パターン | 選択すべき状況 |
|-----------|-------------|
| **分解型** | 聴衆が結論受容の準備ができているとき |
| **積上型** | 反論が予想される場合・慎重な聴衆 |
| **選択肢型** | 必要性は認識済みだが「どれにすべきか」決められない相手（推奨案＋積極案＋消極案の3案比較） |
| **意外性展開型** | まだ必要性を感じていない相手 |

## 5. 1メッセージ原則

### 原則の定義

> 「このプレゼンで聴衆に持ち帰ってほしい、たった1つのことは何か？」

全ての構成要素はこの1メッセージを支持するために存在する。

### 1メッセージの作り方

**悪い例**（メッセージが複数）:
> 「コスト削減と品質向上と顧客満足度の改善について説明します」

**良い例**（1つの核心）:
> 「プロセス改善により6ヶ月でコストを20%削減できる」

### テスト方法

- 「このプレゼンは何についてのプレゼンですか？」と聞かれた時に1文で答えられるか
- 聴衆が翌日同僚に説明するとしたら、何と伝えるか想像する
- スライドタイトルの集合が「1つの主張」を形成しているか

### 戦略的プレゼン設計：「略す」ことの徹底

> 戦略の本質は「略」すること ── 何を言わないかを決めること。

情報を徹底的に集め、**枝葉を除去し、幹だけに絞り込む**意思決定がプレゼン設計の核心。

| 設計の原則 | 実践方法 |
|-----------|---------|
| 「何を言わないか」を決める | 話したいことを全部書き出し、最重要1つ以外を「付録」に回す |
| 幹に絞る | 「この情報がなくても結論は変わるか？」でフィルタリング |
| 枝葉の扱い | 削除ではなく付録・補足スライドに移動（Q&A対応に活用） |

### メッセージ20文字以内の原則

各スライドのメッセージ（タイトル）は**10〜20文字以内**に凝縮する。

| メッセージの長さ | 特徴 |
|---------------|------|
| 20文字以内 | 一瞬で理解・記憶定着・伝播しやすい |
| 20〜40文字 | 読める、だが記憶しにくい |
| 40文字超 | 読むのに努力が必要・プレゼン中には不向き |

**実践**: スライドを作る際、まずメッセージのみ（文字だけ）で全スライドを並べ、20文字以内で意味が通るかを確認してから、グラフや図を追加する。

### 1メッセージ決定のフロー

```
1. 話したいことをすべて書き出す
2. 「So What?（だからどうした？）」で本質を問い続ける
3. 聴衆の立場から「なぜこれが重要か？」を考える
4. 動詞を含む1文に圧縮する
5. その1文でプレゼン全体を組み立て直す
```

## 6. 思考整理のフレームワーク

### 「1枚に収める」技術

複雑な内容を整理する際、まず1枚の紙（またはスライド）に収めることを制約として使う。

**なぜ有効か**:
- 情報の優先順位付けが強制される
- 何が本質的でないかが見える
- 「考え抜く」プロセスそのものが資料の質を高める

### 思考整理の3類型

| タイプ | 用途 | 構造 |
|-------|------|------|
| **WHAT型** | 現状・事実の整理 | 3列: 問い / 事実 / 示唆 |
| **WHY型** | 原因分析 | 原因1 → 原因2 → 根本原因 |
| **HOW型** | 解決策設計 | 課題 / アクション / 効果 |

### 思考整理のプロセス

```
Step 1: 問いを立てる
　「このプレゼンで答えるべき問いは何か？」

Step 2: 情報を出し切る（発散）
　思いついたことを全部書き出す

Step 3: グルーピング（収束）
　関連するものを束ね、カテゴリ名をつける

Step 4: 優先度付け
　聴衆にとって最も重要なものを選ぶ

Step 5: メッセージに変換
　情報を「主張・提言」の形式に変える
```

## 7. 構成設計チェックリスト

プレゼン設計完了時の最終確認。

### 全体構成
- [ ] Why-What-Howが明確に分離されている
- [ ] プレゼンのタイプ（提案型/報告型/啓発型/ピッチ型）が決まっている
- [ ] 聴衆プロファイルに合わせた調整ができている

### メッセージ
- [ ] 1メッセージが1文で言える
- [ ] 全てのスライドが1メッセージを支持している
- [ ] 「So What?」に答えられる

### 論理構造
- [ ] 論証パターン（帰納法/演繹法/関係論証法）が選択されている
- [ ] 結論と根拠の接続が論理的に正しい
- [ ] 反論・懸念点に対する回答が含まれている
- [ ] ロジックエラー（プレゼンター視点でのみ成立する論理）がないか聴衆視点で確認した
- [ ] 展開パターン（分解型/積上型/選択肢型/意外性展開型）が聴衆の必要性認識度合いに合っている
- [ ] 縦の論理に3つの原因パターン（隠れた前提の不一致 / 異質なものの同質化 / 偶然の必然化）がないか確認した
- [ ] 横並びの要素が表現ふぞろい（動詞形・主語・時間場所）でないか確認した
- [ ] ファクトの強度（定量×一次×第三者が最強）を確認した
- [ ] SCQA導入が冒頭の「つかみ」として適切に使われているか確認した

### 聴衆設計
- [ ] 「聴衆は何を知っているか？」を踏まえた情報選択ができている
- [ ] 専門用語の使用が聴衆レベルに合っている
- [ ] アクションを促す「次のステップ」が明確
- [ ] ゴール設計の3層（行動・理解・心理状態）が明確になっている
- [ ] プレゼンの3ゴール（ハッピー・行動・言いふらし）を意識した設計ができている

### スライド枚数・構成
- [ ] 1スライドあたりの標準時間を元にスライド枚数を算出した
- [ ] プレゼン本体と質疑応答の時間配分を事前に決めた
- [ ] ビジネス文書タイプ（活動報告/解決策提案/調査報告/企画提案）に適した筋道構成を選択した

### ロジック品質・構造設計
- [ ] 並列項目が「5つの揃い」（主語/構成/抽象度/比較対象/時間場所）を満たしているか（ロジック3力：横並べの力）
- [ ] スライド間の「だから」の接続が縦論理として成立しているか（ロジック3力：縦つなぐ力）
- [ ] 各セクションの項目がMECE（ダブり/モレ/粒度違いの3パターンなし）になっているか
- [ ] MECE切り口（層別/因数分解/時系列/フレームワーク）を選択して適用したか
- [ ] 各スライドでポイント（解釈・結論）を「データの前」に示しているか（ポイント先行提示）
- [ ] メッセージマップ（140字ヘッドライン → 3つのキーメッセージ → アウトライン）を作成したか
- [ ] 総論（目的+要約）→各論（詳説）→結論（総論の繰り返し・新情報なし）の構造になっているか

## 9. プレゼンの3ゴール・フレームワーク

プレゼンの成否を3つのゴール達成で測定する。

| ゴール | 内容 | 設計の観点 |
|-------|------|-----------|
| **ゴール1** | 聴いた人がハッピーになる | 聴衆にとっての「価値」を中心に設計する |
| **ゴール2** | 聴いた人から行動（決断）を引き出す | 具体的なアクションを設計に組み込む |
| **ゴール3** | 聴いた内容を他人に言いふらしたくなる（伝播性） | 「誰かに話したくなる」フック・一言を仕込む |

**特にゴール3の意義**: 「言いふらしたくなる」は、プレゼン終了後の波及効果を設計段階から意識させる。口コミ・社内共有・SNS拡散まで想定すると、「簡潔で記憶に残るメッセージ」の重要性がさらに高まる。

## 10. ゴール設計の3層構造（行動・理解・心理状態）

プレゼンの目的を3層で分解し、逆算で設計する。

```
第3層: 心理状態（「これなら自分でもできそう」など）
  ↓ 逆算
第2層: 理解（行動のために理解してもらいたいこと — 1つに絞る）
  ↓ 逆算
第1層: 行動（相手にしてほしい具体的アクション）
```

**設計への応用**:

| 層 | 設計時の問い | 例 |
|----|-----------|-----|
| 第1層（行動） | 「プレゼン後に相手に何をしてほしいか？」 | 「プロジェクト参加を承諾してほしい」 |
| 第2層（理解） | 「その行動のために最も重要な1つの理解は？」 | 「このプロジェクトが自分のKPIに直結する」 |
| 第3層（心理） | 「説明後の理想の感情状態は？」 | 「自分ごととして動けそうな気がする」 |

**効果**: 心理状態から逆算することで、専門用語の量・資料ボリューム・表現トーンが自然に決まる。

## 11. ロジックエラーの聴衆視点チェック

### ロジックエラーの定義

> プレゼンター視点では論理が成立しているが、聴衆視点では「そうかな？」となるポイント。

**典型例**:
- 「主食と言えばご飯かパンですよね」→ 聴衆「パスタもラーメンも主食では？」
- 「このコストは高いですが許容範囲です」→ 聴衆「誰の許容範囲？」

### ロジックエラーが起きると

ロジックエラーが発生した瞬間、聴衆の思考は「本当にそうか？」の検証モードに切り替わり、プレゼンへの集中が途切れる。一度失った信頼は取り戻しにくい。

### 対策

| 方法 | 内容 |
|------|------|
| **準備段階での「質疑応答」** | 想定される反論を自ら列挙し、事前に答えを準備する |
| **プチ本番** | 実際の聴衆に近い人に事前発表し、「そうかな？」と感じた箇所をフィードバックしてもらう |
| **言い切り前の検証** | 「〜ですよね？」形式の前提は、聴衆全員が同意するか確認してから使う |

## 12. スライド枚数の設計基準

### 標準時間からの逆算

自分の過去実績から「1スライドあたりの平均時間」を算出し、それを元に枚数を計算する。

```
スライド枚数 = プレゼン本体の持ち時間 ÷ 1スライドあたりの標準時間
```

**参考値**:

| プレゼンター属性 | 1スライドあたりの目安時間 |
|---------------|------------------------|
| 孫正義（ビジョン型） | 約1分/枚 |
| 一般ビジネスパーソン | 約3分/枚 |
| 専門外聴衆向け | 4分以上/枚 |

### 時間配分を先に決める

| 項目 | 例（30分の場合） |
|------|----------------|
| プレゼン本体 | 20分 |
| 質疑応答 | 10分 |
| → スライド枚数（3分/枚の場合） | 20 ÷ 3 ≒ 6〜7枚 |

**注意**: 枚数はあくまで目安。1枚で複数分使う「見せ続けるスライド」も存在する。

## 14. SCQA導入モデル

バーバラ・ミントが提唱し菅野誠二が実践紹介した、冒頭の「つかみ」として信頼訴求を実現する構成パターン。

### SCQAの構造

```
Situation（状況）: 聴衆が認める事実・現状を確認する
　「現在、○○の市場は年率15%で成長しています」

Complication（複雑化/問題）: その状況に生じた問題・変化を示す
　「しかし、当社の市場シェアは逆に2ポイント下落しています」

Question（問い）: 問題から生まれる課題・問いを立てる
　「どうすれば競合に対してシェアを回復できるか？」

Answer（答え）: 結論・提言を予告する（詳細は本論で）
　「今日は3つの施策を提案します」
```

### 適用ガイドライン

| 適用場面 | 活用方法 |
|---------|---------|
| 提案型プレゼンの冒頭 | Sで聴衆の事実認識を確認→Cで問題→Qで課題→Aで結論予告 |
| 報告型の問題提起 | Sで前提共有→Cで異常値→Qで対策を問う→A: 本日の報告内容 |
| 啓発型の導入 | Sで常識→Cで矛盾・反例→Qで「なぜか」→A: 新視点の提示 |

**効果**: Sで聴衆が「そうだ」と同意する事実から始めることで、信頼感を確立した上でCの問題提起が刺さりやすくなる。

## 15. 論理検証の高度技法

論理の破綻を発見・修正するための実践的チェック手法。

### 15.1 縦の論理を崩す3つの原因パターン（高田貴久）

「AならばB」の縦の接続（因果・根拠）が崩れる場合、以下の3パターンのいずれかに該当することが多い。

| パターン | 定義 | 検出の問い |
|---------|------|-----------|
| **① 隠れた前提条件の不一致** | 「AならばB」が成立するために必要な前提が明示されていない | 「この接続は、何を前提とした場合にのみ成立するか？」 |
| **② 異質なものの同質化** | 性質の異なるA・Bを同じカテゴリとして扱っている | 「AとBは本当に同じ種類のものか？」 |
| **③ 偶然の必然化** | 因果関係がない相関を「だから」でつなげている | 「AがなくてもBは起きないか？」 |

**適用方法**: 各「AならばB」の接続ごとに上記3パターンを照合する。いずれかに該当する場合、前提を明示するか接続を修正する。

### 15.2 縦接続の言葉重ね確認法

前の論理単位（スライド・段落）の末尾キーワードと、次の単位の冒頭キーワードがつながるかを確認する手法。

```
確認の手順:
1. スライドAの末尾に登場するキーワードを書き出す
2. スライドBの冒頭に登場するキーワードを書き出す
3. 「AのキーワードがあるからBのキーワードが出てくる」と言えるか確認する
4. つながらなければロジック破綻のサイン → 橋渡しスライドまたは文章を追加
```

**例**:
- スライドA末尾: 「顧客獲得コストが増加している」
- スライドB冒頭: 「新製品の開発を加速する」
- → 「なぜコスト増加→新製品加速なのか」が欠けている。中間ロジックを補う

### 15.3 横並びの表現ふぞろい検出法

並列項目（箇条書き・複数の論点）の表現統一を検証する4テスト。

| テスト | 確認内容 | NG例 → OK例 |
|-------|---------|------------|
| **①動詞形テスト** | すべての項目が同じ動詞形か | 「コスト削減」「品質を上げる」「顧客対応」→「コストを削減する」「品質を向上させる」「顧客対応を強化する」 |
| **②主語統一** | 暗黙の主語が統一されているか | 「営業チームが〜」「マーケが〜」「経営層が〜」→ 主語を揃えるか明示 |
| **③時間場所統一** | 異なる時間軸・場所が混在していないか | 「昨年の実績」「現在の課題」「将来のビジョン」→ 時制を統一するか軸を明示 |
| **④重要順並べ替えテスト** | 優先度の高いものが先頭にあるか | 聴衆の関心・インパクト順に並べ直す |

### 15.4 過剰一般化の回避（中川功一）

「すべての」「必ず」「常に」などの絶対的表現は、1つの反例で論理全体が崩れる危険がある。

| 過剰一般化の表現 | 限定語句への修正 |
|---------------|---------------|
| 「すべての顧客が〜」 | 「弊社の主要顧客層（企業規模50名以上）では〜」 |
| 「必ず効果が出る」 | 「過去12ヶ月の導入事例ではXX%で効果を確認」 |
| 「業界全体の課題」 | 「国内製造業の中堅規模（100-500名）において」 |

**実践**: 母集団（誰に対して）と条件（どんな前提で）を常に明示する。反例があることを先に認めると信頼性が増す（「例外はありますが、全体傾向として〜」）。

## 16. 根拠の強化技法

証拠・事実の信頼性を高め、主張の説得力を増す手法。

### 16.1 ファクトの強度3軸分類（高田貴久）

根拠として使用する事実・データを3軸で評価し、最も強い根拠を優先的に使う。

| 軸 | 強い（+） | 弱い（−） |
|----|---------|---------|
| **定量 vs 定性** | 定量（数値・割合） | 定性（印象・感触） |
| **一次 vs 二次** | 一次（自ら収集・測定） | 二次（既存文献・他社データ） |
| **第三者 vs 当事者** | 第三者（中立機関・外部調査） | 当事者（自社・関係者の証言） |

**最強の組み合わせ**: 「定量 × 一次 × 第三者」（例: 第三者機関が実施した定量調査の生データ）

**評価表の活用**:

| 根拠候補 | 定量/定性 | 一次/二次 | 第三者/当事者 | 強度評価 |
|---------|---------|---------|------------|---------|
| 自社売上データ | 定量 ○ | 一次 ○ | 当事者 △ | 中〜高 |
| 業界団体の調査レポート | 定量 ○ | 二次 △ | 第三者 ○ | 中〜高 |
| 「顧客が喜んでいた」 | 定性 △ | 一次 ○ | 当事者 △ | 低 |

**活用指針**: 強度の低いファクトを使う場合は、複数の低強度ファクトを組み合わせるか、強度の高いファクトで補強する。

### 16.2 トライアンギュレーション（中川功一）

主張を単一の根拠ではなく、異なる種類の3つの根拠で裏付けることで、1つの根拠が崩れても主張が成立する構造を作る。

```
主張: 「この施策は有効である」

根拠1【データ】: 導入企業のNPSが平均15pt向上（定量×第三者調査）
根拠2【事例】  : A社での導入事例（6ヶ月でROI 130%を達成）
根拠3【理論】  : 行動科学の習慣形成理論に基づく設計
```

**3種類の組み合わせ例**:

| 種類 | 内容 | 特徴 |
|-----|------|------|
| **データ** | 統計・数値・調査結果 | 定量性・再現性を示す |
| **事例** | 具体的な成功例・失敗例 | リアリティ・共感を生む |
| **理論** | 学術的根拠・原理原則 | 普遍性・信頼性を担保する |

**使う場面**: 重要な意思決定を求める場面（予算承認・方針変更）で、特に懐疑的な聴衆に対して有効。

## 17. 論点適合性の診断

「正しい問いに答えているか」を検証するフレームワーク。優れた根拠と構成でも論点がずれていれば成果は出ない。

### 17.1 論点を外す4パターン診断（高田貴久）

プレゼンが「良い内容だが刺さらない」場合、以下の4パターンのいずれかに論点のズレが生じている。

| パターン | 症状 | 確認の問い |
|---------|------|-----------|
| **① 議論スタンス違い** | 聴衆は「Yes/No」を問うているのに、こちらは「How」を語っている | 「聴衆は今、何を決めようとしているのか？」 |
| **② 要望の汲み違い** | 聴衆が求めている答えの粒度・範囲が違う | 「聴衆が本当に必要としている情報の範囲は？」 |
| **③ 具体的判断項目の不足** | 「方向性はわかった、でも何で判断すればいいのか」という状態 | 「聴衆が判断するために必要な基準・指標を示しているか？」 |
| **④ 相手が既に答えを持っている** | 聴衆はすでに結論を持っており、説明ではなく「確認・根拠」を求めている | 「この聴衆はすでに心の中で答えを持っていないか？」 |

**早期発見の方法**: プレゼン設計の最初に「聴衆が今どの状態か」を4パターンで診断し、ミスマッチを事前に解消する。

### 17.2 仮説検証5ステップ（高田貴久）

完璧な答えが出せない場面でも「示唆」として方向性を示すための構造的なアプローチ。

```
Step 1: 目的理解
　「このプレゼンは何のために行われるのか？」
　「意思決定者は何を決めようとしているのか？」

Step 2: 論点把握
　「本質的に答えるべき問いは何か？」
　「議論のスタンス・範囲・粒度は適切か？」

Step 3: 仮説構築
　「現時点でのベストアンサーは何か？」
　「なぜそれが答えだと言えるか？（仮の根拠）」

Step 4: 検証実施
　「仮説を支持・否定するデータ・事例はあるか？」
　「反証を探し、仮説を修正する」

Step 5: 示唆抽出
　「検証結果から導ける結論・方向性は何か？」
　「完璧な答えでなくても『示唆』として方向性を示す」
```

**「示唆」の活用**: 分析が不完全でも「現時点のデータが示す方向性は〜」「確証はないが、〜の可能性が高い」という形で示唆を提示することで、意思決定の助けになる。

## 18. 追加の構成パターン

目的・場面別に使い分ける3つの追加構成テンプレート。

### 18.1 STAR構成（中川功一）

行動報告・実績発表・ケーススタディに最適な構成。「何をしたか」より「なぜ・どんな成果か」を際立たせる。

```
Situation（状況）: 当時の背景・文脈・課題
Task（タスク）    : 自分/チームに課されていた役割・目標
Action（行動）    : 実際に取った具体的な行動・施策
Result（結果）    : 達成した成果・数値・学び
```

**使う場面**: 営業報告・プロジェクト振り返り・採用面接・実績発表
**注意点**: ResultはできるだけS（状況）の数値と比較して示す（「コスト30%超過の状態から→期限内コスト5%削減で完了」）

### 18.2 仮説検証形式（中川功一）

研究発表・データドリブン提案に向いた構成。論理的一貫性と透明性が高い。

```
仮説設定 : 「○○である」という仮説を立てた
検証方法 : どのようなデータ・実験・調査で検証したか
結果     : 検証で得られたデータ・観察結果（事実のみ）
考察     : 結果が仮説を支持/否定する理由・含意
（結論）  : 仮説の採否と今後の方向性
```

**使う場面**: 社内分析報告・製品改善提案・学術発表・エビデンスを重視する意思決定層向け
**注意点**: 結果と考察を混在させない。「データはこう言っている（結果）、だからこう解釈できる（考察）」を分離する

### 18.3 完結ストーリーライン（菅野誠二）

What（何をすべきか）のピラミッドとHow（どうやるか）のピラミッドを並列で構築し、1つのストーリーに統合する手法。

```
【What ピラミッド】          【How ピラミッド】
 主要提言                     実行計画
 ├── 提言1                    ├── アクション1
 ├── 提言2                    ├── アクション2
 └── 提言3                    └── アクション3
        ↓                             ↓
        └──────── 統合ストーリーライン ──────────┘
                「何をすべきか×どうやるか」の一体化された主張
```

**効果**: 「方向性は正しいが、実現可能性が見えない」という聴衆の不安を解消する。Whatだけでは「で、どうするの？」、Howだけでは「なぜそれをするの？」という疑問が残るが、統合することで完結したナラティブになる。

## 19. ロジック構築の3つの力（木山泰嗣）

論理的なプレゼンを作るために必要な3つの基本能力。この3力を習得することで、あらゆる構成の問題を診断・修正できる。

### 3つの力の定義

| 力 | 内容 | スライドへの適用 |
|---|------|----------------|
| **① 縦につなぐ力（因果関係）** | 「AだからB」の因果・根拠の連鎖を正確に構築する力 | スライド間の「だから」でつながる論理チェーン |
| **② 横に並べる力（並列の揃い）** | 並列項目が5つの基準で統一されているか確認する力 | 箇条書き・比較表・複数スライドの対称性 |
| **③ 複雑な状況で活用する力** | ①②を組み合わせて複雑な論点に対応する力 | 多層ピラミッド・長期計画・組織提案 |

### 横並びの「5つの揃い」（並列の厳格基準）

並列項目は以下の5軸が揃っていなければ「構造的欠陥」となる。

| 揃いの軸 | 確認の問い | NG例 |
|---------|----------|------|
| **①主語の揃い** | 各項目の暗黙の主語は同じか | 「営業が〜」「経営判断で〜」「コストが〜」 |
| **②構成要素の揃い** | 各項目が同じ内部構造を持つか | 「メリット3点」「デメリット1点のみ」 |
| **③抽象度の揃い** | 具体度・詳細度が統一されているか | 「コスト削減」「品質向上施策の実施（3項目）」 |
| **④比較対象の揃い** | 比較の基準軸が統一されているか | 「A社とB社の比較」に「C社の昨年データ」を混在 |
| **⑤時間・場所の揃い** | 同じ時制・地域・文脈で語られているか | 「昨年の実績」と「将来予測」を並列扱い |

---

### 19.1 3つの接続関係の体系（倉島保美）

あらゆる情報は、以下3種類の接続関係のいずれかで結びついている。この分類がロジック構築の起点となる。

| 接続関係 | 定義 | 並べ替え可否 | 明示の要否 |
|---------|------|------------|----------|
| **縦接続** | 原因→結果のように論理的なつながりがある状態。接続順に並ぶ | ❌ 順序変更不可 | 必須（矢印・接続詞） |
| **横並び** | 同じ種類で互いに独立した状態。重要度順などに並べ替え可能 | ✅ 並べ替え可能 | 羅列のみでOK |
| **包含** | 上位概念が下位概念を含む階層構造 | ❌ 上下関係は固定 | 必須（階層・字下げ） |

**羅列は横並びを意味する**: 接続関係を明示せずに情報を列挙した場合、聴衆は横並びと判断する。縦接続・包含の関係があるなら、プレゼンターがその関係を明示しなければならない。縦接続や包含の情報を横並びの情報と一緒に羅列してはならない。

---

### 19.2 3レベル接続診断

論理の接続は、スライド内・隣接スライド間・離れたスライド間の3レベルすべてで確認が必要。

| レベル | 対象 | チェックポイント |
|-------|------|----------------|
| **スライド内** | 1枚のスライド内の情報同士 | すべての情報が縦接続/横並び/包含のいずれかで明確に接続されているか |
| **隣接スライド** | 前後のスライド | 2枚の間の論理的な接続関係が明示されているか。キーワードとレイアウトを共通化して縦接続を可視化する |
| **離れたスライド** | 章をまたぐスライド | 問題点と対策、現状と改善後など、対応する情報が一致しているか。目次レベルで縦接続を設計する |

**診断の順序**: スライド内 → 隣接 → 離れた、の順で確認する。離れたスライドの接続ミスは、目次（章構成）を見直すと発見しやすい。長いプレゼンほど離れたスライドの接続が崩れやすい。

---

### 19.3 横並びの3条件（倉島保美）

横に並べてよい情報は、以下の3条件をすべて満たす必要がある。既存の「5つの揃い」（§19）は条件③の具体的な検証軸。

| 条件 | 内容 | 違反例と対処 |
|-----|------|------------|
| **① 同じ種類** | 同じカテゴリに属する情報であること | 工程別の問題と全体共通の問題を並列 → 別階層に分離する |
| **② 独立** | 縦接続・包含の関係にある情報を混在させない | 原因と結果を横並びにする → 縦接続として明示する。上位概念と下位概念を並べる → 階層構造にする |
| **③ 揃い** | 表現・構成・抽象度・比較対象・時間場所が統一されている | 数値あり項目と数値なし項目の混在 → 全項目に同じ構成要素を揃える |

> **§15.3「横並びの表現ふぞろい検出法」との接続**: ③の揃いを確認する具体的テスト（動詞形・主語・時間場所・重要順）は§15.3を参照。§19の「5つの揃い」は③を構成要素の揃いと比較対象の揃いの観点でさらに細分化したもの。

---

### 19.4 3つの基本ロジック図（倉島保美）

すべてのロジックは、以下3パターンの組み合わせで構築される。まずブロック図で全体骨格を設計し、必要な箇所をテーブルまたはピラミッドで詳細化する。

| パターン | 構造 | 典型的な用途 |
|---------|------|------------|
| **ブロック図** | 縦接続＋横並びの組み合わせ。情報が縦に流れながら横に展開する | ほぼすべてのプレゼンの全体骨格（問題→原因→対策→効果） |
| **テーブル** | 横並び（列）×縦接続・横並び・包含（行）の二次元マトリクス | 複数の選択肢・戦略・事例の比較対照 |
| **ピラミッド** | 包含関係を階層構造（ロジックツリー）で表現 | 情報の分類・網羅的リストアップ・MECE確認 |

```
ブロック図の例（縦接続＋横並び）:
  問題の把握
      ↓
  原因A ── 原因B    ← 横並び
      ↓
  対策の実行
      ↓
  効果の検証

テーブルの例（横並び×縦):
  　　   | 戦略A | 戦略B | 戦略C |  ← 横並び
  概略   |      |      |      |
  コスト |      |      |      |
  効果   |      |      |      |

ピラミッドの例（包含の階層）:
  分類テーマ
  ├─ カテゴリA
  │   ├─ 要素1
  │   └─ 要素2
  └─ カテゴリB
      ├─ 要素3
      └─ 要素4
```

**ブロック図→テーブル・ピラミッドの関係**: ブロック図がプレゼン全体の骨格。ブロック図の特定部分を「比較したい」ならテーブル、「分類したい」ならピラミッドで詳細化する。

> 出典: 倉島保美『論理が伝わる 世界標準のプレゼン術』（ブルーバックス）第2部「ロジックを組む」

## 20. MECEのスライド適用（Mutually Exclusive, Collectively Exhaustive）

論理の「漏れなく重複なく」原則をスライド構成に適用することで、聴衆に「すべてを網羅している」という信頼を与える。

### MECE失敗の3パターン

| 失敗パターン | 内容 | 例 |
|-----------|------|-----|
| **① ダブり** | 複数の項目が同じ内容を指している | 「コスト削減」と「予算最適化」を別項目にする |
| **② モレ** | 重要な視点・選択肢が抜けている | 顧客セグメント分析で「法人」「個人」のみ → 「政府・自治体」が漏れる |
| **③ 粒度違い** | 大きな概念と詳細な概念が混在 | 「マーケティング」と「SNS広告のCTR改善」を並列にする |

### MECEの切り口選択

| 切り口 | 説明 | 適用場面 |
|-------|------|---------|
| **層別（セグメント）** | 構成要素を重複なく分類 | 顧客・市場・製品ラインのセグメント分析 |
| **因数分解** | KPIを構成要素の積に分解 | 売上 = 客数 × 客単価 × 購入回数 |
| **時系列** | 時間軸で分割（過去・現在・未来） | プロジェクト工程・変化の流れ |
| **フレームワーク** | 既存の整理枠組みを活用 | SWOT / 3C / 4P / バリューチェーン |

### 項目数の最適値

MECEを維持しつつ認知負荷を下げる項目数の目安は**3〜4個**。

- 2個: MECEは保ちやすいが説明が薄い
- **3〜4個: 認知的に最適（3点ルールと一致）**
- 5個以上: MECE維持が難しく、グループ化が必要

## 21. 総論-各論-結論構造（木山泰嗣）

提案書・報告書型プレゼンで最も信頼性の高い構成フォーマット。

### 3部構成の定義

```
【総論】: 目的と要約を示す
  「本日は〇〇について、3点から説明します。結論は△△です」
  → 聴衆のメンタルモデルを事前形成する

【各論】: 時系列・論理順で詳説する
  「まず①〇〇について...次に②〇〇について...最後に③〇〇について...」
  → 総論で示した3点を順番に根拠とともに展開する

【結論】: 総論の繰り返し（新情報は一切なし）
  「以上3点をまとめると、△△です」
  → 重要: 結論スライドに新しい情報を入れない
```

**なぜ総論と結論は同じ内容でよいのか**: 人間の記憶は「最初と最後に聞いたこと」が最も残る（初頭・親近効果）。同じメッセージを最初と最後に配置することで、記憶定着が確実になる。

### よくある失敗

| 失敗 | 問題 | 修正 |
|-----|------|------|
| 結論スライドに新情報を追加する | 「え、そんな話もあったの？」と混乱する | 結論は総論の完全な繰り返しにする |
| 総論が長すぎる | 各論を聞く前に集中が切れる | 総論は1〜2スライド・要点のみ |
| 各論の順番が論理的でない | 「なぜこの順番？」という疑問が生じる | 因果関係・重要度・時系列で統一する |

## 22. ポイント先行提示（Point-First Presentation）

データや根拠を見せる**前に**、そのスライドのポイント（解釈・結論）を先に述べる原則。「データファースト」の習慣を意識的に改善する。

### Before / After の比較

```
❌ データファースト（失敗パターン）

スライド: 複数のグラフを次々と表示
→ 「こちらが昨年のデータです...次にこちらが今年...比較すると...」
→ 最後に: 「ということで、コスト削減が必要です」

聴衆の体験: データを見ながら「何が言いたいの？」と混乱

✅ ポイント先行（成功パターン）

スライドタイトル（または冒頭の一文）: 「コスト削減が急務—前年比20%増の根拠」
→ その後: グラフ・データで裏付け
→ 末尾: 「つまり、コスト削減施策を今期中に実施する必要があります」

聴衆の体験: 「なぜこのデータを見せているのか」が最初から明確
```

### ポイント先行の実装方法

| 要素 | 実装 |
|------|------|
| スライドタイトル | 「売上推移（2020-2024）」→「5年連続増収—デジタル領域が牽引」 |
| 冒頭の一文 | データを見せる前に「今から〇〇を示すデータをお見せします」 |
| グラフキャプション | 「出典: 〇〇調査」→「出典: 〇〇調査。ポイントは〇〇」 |

## 23. メッセージマップ（Message Map）

プレゼンの全体構造を「ヘッドライン→キーメッセージ×3→補強情報」の3層で整理するテンプレート。

### 3ステップで作成する

```
Step 1: 140字ヘッドライン
  「このプレゼンで一番伝えたいことを140字（ツイート1本）に凝縮する」
  例: 「デジタル投資を今期2倍にすることで、3年後の売上は現在の150%に達し、
       競合との差別化が実現できる」

Step 2: 3つのキーメッセージ
  ヘッドラインを支える3本柱を設定する
  KM1: 市場環境（なぜ今デジタル投資か）
  KM2: ROI見込み（なぜ2倍か）
  KM3: 実行可能性（どうやって達成するか）

Step 3: ストーリーでアウトライン
  3つのキーメッセージを「流れ」としてつなぎ、プレゼン全体の骨格にする
  「まず市場環境を示し（KM1）、次にROI根拠を示し（KM2）、
   最後に実行計画で締める（KM3）」
```

### メッセージマップの効果

| 効果 | 内容 |
|-----|------|
| **準備の効率化** | スライド作成前に全体構造が決まるため、無駄なスライドが生まれない |
| **一貫性の担保** | 全スライドが「ヘッドライン」の裏付けになっているか検証できる |
| **Q&A対応力** | マップから外れた質問に「今日の議題の範囲外ですが…」と対応できる |
| **繰り返し強化** | ヘッドラインを冒頭・中盤・末尾に繰り返すことで記憶定着を高める |

**使う場面**: 経営層向け変革提案・コンサルティング型プレゼン・重要な方針決定を促す場面

## 24. トランジション設計（Wallwork式）

### アジェンダスライドの要否判断 (Ch7.2-3)

アジェンダスライドは「あれば良い」ではなく、**目的に応じて使い分ける**。

| 発表形式 | アジェンダ | 理由 |
|---------|-----------|------|
| 10〜15分の学会発表 | 不要なことが多い | 時間が短く、構成が単純なため |
| 30分以上の講演・教育セッション | 有効 | 聴衆が全体像を把握しやすい |
| 複数のメッセージがある複合トピック | 有効 | どのセクションにいるかが分かる |

アジェンダを使う場合でも、箇条書きを羅列するだけでなく、発表の「流れの意図」を1文で示すと効果的（Wallwork Ch7.3）。

### セクション間トランジションの種類とシグナル機能 (Ch7.6-8)

トランジションには「ここで話題が変わる」という合図機能がある。聴衆を迷子にしないために明示的なシグナルを使う。

**言語的シグナル**

| タイプ | 例 |
|-------|---|
| 要約型 | "So far I have shown that …" |
| 先読み型 | "Now I'd like to turn to …" |
| 対比型 | "In contrast to what I've just presented …" |
| 問いかけ型 | "So what does this mean in practice?" |

**視覚的シグナル**

- セクション区切りスライド（タイトルのみ）を挿入する
- アジェンダスライドを再表示し現在位置をハイライトする
- スライドの背景色・レイアウトを変えてフェーズの転換を示す

（Wallwork Ch7.6-7）

### スライド内トランジションの省略原則 (Ch7.9)

**同一セクション内で次のスライドに進む場合、口頭トランジションは原則不要**。

次のスライドに進む際に "The next slide shows …" や "As you can see here …" と言う習慣は時間の無駄で聴衆のリズムを乱す。スライドを切り替えた後、すぐに内容を話し始めるだけでよい。

トランジションが必要なのは「セクション（話題）が変わるとき」だけ。

### トランジションのバリエーション化 (Ch7.11)

同じパターンを繰り返すと単調になる。1つの発表で異なるタイプを組み合わせる。

```
冒頭     → 問いかけ型でオープニング
第1→第2節 → 要約型（前半を締める）
第2→第3節 → 視覚的シグナル（区切りスライド）
結論へ   → 先読み型（"What does this mean for us?")
```

バリエーションを持たせることで、聴衆が「また同じ言い方だ」と飽きることを防げる（Wallwork Ch7.11）。
