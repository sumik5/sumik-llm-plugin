# コンテンツ処理・フォーマットルール詳細

フラッシュカード作成時のコンテンツ自動分析ヒューリスティクスとフォーマット変換ルールの詳細リファレンス。

## pandocアーティファクトのクリーニング

pandocはEPUBをMarkdown変換する際に以下のアーティファクトを生成する。すべて除去すること。

| アーティファクト | 例 | 除去パターン |
|----------------|-----|------------|
| CSSクラスマーカー | `{.class_s74W}`, `{.heading_s3M-0}` | `\{[.#][^}]*\}` |
| インラインスパン | `[text]{.class_sXXX}` | テキストのみ抽出 |
| ページマーカー | `[]{#c3W.xhtml}` | 完全除去 |
| divマーカー | `::: heading_s6MV` | 行ごと除去 |
| ゼロ幅スペース付きリストマーカー | `[​- ​]`, `[​1. ​]` | 通常のリストマーカーに変換 |
| Google URL リダイレクト | `[text](https://www.google.com/url?q=...)` | テキストのみ抽出 |
| コードブロックdiv | `::: class_sBD ... :::` | base64退避→パース後復元（後述） |
| エスケープ文字 | `\``, `\*`, `\#`, `\'`, `\~`, `\<`, `\>`, `\--\>` | エスケープ除去（順序注意） |
| 空ブラケット | `[]` | 除去 |
| 誤記ヘッダー | `### 解説 {.heading_s6S}`（正解用クラスなのにテキストが解説） | CSSクラスベースで`正解`に修正 |
| 残存bare bracket | `[テキスト]` | テキストのみ抽出（最終クリーンアップ） |
| **inline style属性** | `{style="font:7.0pt 'Times New Roman'"}` | `\{style="[^"]*"\}` で除去 |
| **inline style付きスパン** | `[text]{style="font-size:11.0pt"}` | テキストのみ抽出 |
| **ネストブラケットリンク** | `[[url]](url)` | `\[\[([^\]]*)\]\]\([^)]*\)` でテキスト抽出 |

### 🔴 クリーニング処理順序の罠

**inline spans (`[text]{.class_sXXX}`) は CSS class markers (`{.class_s...}`) より先に除去すること。**

順序を誤ると、CSS class markers の除去 (`\{[.#][^}]*\}`) が先に `{.class_sXXX}` 部分を消し、inline span regex (`\[text]\{\.class_s...\}`) がマッチしなくなる。結果として bare bracket `[text]` がテキスト中に残留する。

**正しい処理順序:**

```python
# 1. ページマーカー除去（空ブラケット + アンカーID）
text = re.sub(r"\[\]\{#[^}]*\}", "", text)
# 2. inline spans 除去（テキスト抽出）★ CSS class markers より先
text = re.sub(r"\[([^\]]*)\]\{\.class_s[^}]*\}", r"\1", text)
# 3. CSS class markers 除去（残りの単独マーカー）
text = re.sub(r"\{[.#][^}]*\}", "", text)
# 4. ... 他のクリーニング処理 ...
# 5. 最終クリーンアップ: 残存bare bracket除去
text = re.sub(r"\[([^\]\[]*)\]", r"\1", text)
```

**失敗例**: CSS class markers を先に除去 → `[拒否]{.class_s6U0}` が `[拒否]` になる → inline span regex がマッチせず `[拒否]` が残る。

### `<pre>`ブロック保護

`\{[^}]*\}`パターンでpandocクラスを除去する際、**`<pre><code>`ブロック内の`{`や`}`も誤ってマッチする**危険がある（`[^}]`は改行にもマッチするため、遠くの`}`まで横断マッチする）。

**対策**: クリーニング処理の前に`<pre>...</pre>`ブロックをプレースホルダーに退避し、処理後に復元する:

```python
pre_blocks = {}
counter = [0]
def save_pre(m):
    key = f'\x00PRE{counter[0]}\x00'
    pre_blocks[key] = m.group(0)
    counter[0] += 1
    return key
text = re.sub(r'<pre[^>]*>.*?</pre>', save_pre, text, flags=re.DOTALL)
# ... クリーニング処理 ...
for key, value in pre_blocks.items():
    text = text.replace(key, value)
```

### 誤記ヘッダーの自動修正

EPUBソースでセクションヘッダーのテキストが誤っていることがある（例: 「正解」セクションのヘッダーテキストが「解説」になっている）。pandocのCSSクラスで本来の意図を推定できる:

```python
# heading_s6S = 正解用クラス、heading_s6V = 解説用クラス
block = re.sub(
    r'^(#{2,4})\s*解説\s*(\{\.heading_s6S\})',
    r'\1 正解 \2',
    block,
    flags=re.MULTILINE,
)
```

### エスケープ文字の除去順序

pandocのエスケープ文字を除去する際、**複合エスケープを先に処理する**こと。順序を誤ると部分置換が発生する:

```python
# ⚠️ 順序が重要: 複合エスケープ → 単純エスケープ
text = text.replace(r'\--\>', '-->')   # 複合を先に
text = text.replace(r'\>', '>')         # 単純を後に
text = text.replace(r'\<', '<')
text = text.replace(r'\~', '~')
text = text.replace(r'\#', '#')
text = text.replace(r'\*', '*')
text = text.replace("\\'", "'")         # アポストロフィ
text = text.replace('\\-->', '-->')     # フォールバック（部分置換の残骸対策）
```

> ⚠️ **`&nbsp;` / ` ` の行内フィラー混入**: 紙面の段組み調整で挿入された non-breaking space が、Markdown変換後も `&nbsp;` HTMLエンティティまたは生の ` ` 文字として行内に残留する。例: `**第 1 問** &nbsp;&nbsp;&nbsp; 難易度 ★★★`。これを残したまま正規表現マッチを行うと、`第\s*\d+\s*問` の `\s` は ` ` にマッチするが `&nbsp;` 文字列にはマッチせず検出漏れを起こす。**クリーニング段階の早期で** `text = text.replace('&nbsp;', ' ').replace(' ', ' ')` を実行し、両形式を半角空白に正規化すること。

**失敗例**: `\>` → `>` を先に実行すると、`\--\>` が `\-->` になり、その後の `\--\>` → `-->` 置換がマッチしなくなる。

## pandoc div形式コードブロック（`::: class_sBD`）

pandocはEPUBの一部のコードブロックを `::: class_sBD ... :::` のdiv形式で出力する。この形式はスパン注釈形式とは異なり、コード全体がdivブロックで囲まれる。

### 構造

```
::: class_sBD
resource "aws_instance" "example" {
  ami           = "ami-12345"
  instance_type = "t2.micro"
}
:::
```

### base64退避パターン

divブロック内のコードは問題テキスト・選択肢テキストの一部として解析する必要があるが、行ベースのパーサーを混乱させる。**パース前にbase64エンコードでプレースホルダーに退避**し、パース後に復元する:

```python
import base64, re

def preprocess_code_blocks(text):
    def replace_block(m):
        code = m.group(1).strip()
        code = re.sub(r'\\\s*$', '', code, flags=re.MULTILINE)  # pandoc行継続除去
        encoded = base64.b64encode(code.encode('utf-8')).decode('ascii')
        return f'{{CODE:{encoded}}}'
    return re.sub(r'::: class_sBD\n(.*?)\n:::', replace_block, text, flags=re.DOTALL)

def decode_code_block(marker):
    m = re.match(r'\{CODE:([A-Za-z0-9+/=]+)\}', marker)
    return base64.b64decode(m.group(1)).decode('utf-8') if m else marker
```

**重要**: この退避処理は`clean_pandoc()`よりも**前に**実行すること。`clean_pandoc`の`:::`行除去がコードブロックの内容を破壊するのを防ぐ。

## pandocスパン注釈コードブロック

pandocはEPUBのコード部分を`[text]{.class_sXXX}`形式のスパン注釈に変換する。このパターンはプレーンテキストのコードブロックとは異なる処理が必要。

### スパン注釈の構造

```
[variable]{.class_s3CW}[ ]{.class_s3C5}["instance_type"]{.class_s3CX}[
{]{.class_s3C5}
```

| クラス種別 | 役割 | 例 |
|-----------|------|-----|
| `class_s3CW` | キーワード | `variable`, `resource` |
| `class_s3C5` | 通常コード | `{`, `}`, スペース |
| `class_s3CX` | 文字列リテラル | `"instance_type"` |
| `class_s3E0` | 空白 | スペース、改行 |
| `class_s3CF` | 演算子 | `=` |
| `class_s3C9-0` | ハイパーリンク（コードではない） | URL |

### 処理上の3つの罠

#### 罠1: スパンの行分割

pandocはスパンを行をまたいで分割する。ブラケット数を数えてバランスが取れるまで行を結合する:

```python
def join_split_spans(text):
    lines = text.split('\n')
    result = []
    i = 0
    while i < len(lines):
        line = lines[i]
        temp = line.replace('\\[', '').replace('\\]', '')
        while temp.count('[') > temp.count(']') and i + 1 < len(lines):
            i += 1
            line = line + ' ' + lines[i].strip()
            temp = line.replace('\\[', '').replace('\\]', '')
        result.append(line)
        i += 1
    return '\n'.join(result)
```

#### 罠2: エスケープされたブラケット

コード内容に`\[`、`\]`、`\"`が含まれる場合、スパンのブラケットと混同される。プレースホルダーで保護する:

```python
def extract_code_text(text):
    text = re.sub(r'\{\.class_s[\w-]+\}', '', text)  # クラス除去
    text = text.replace('\\[', '\x00LB\x00')          # 保護
    text = text.replace('\\]', '\x00RB\x00')
    text = text.replace('\\"', '\x00DQ\x00')
    text = text.replace('[', '').replace(']', '')      # スパンブラケット除去
    text = text.replace('\x00LB\x00', '[')             # 復元
    text = text.replace('\x00RB\x00', ']')
    text = text.replace('\x00DQ\x00', '"')
    return text
```

#### 罠3: コード行間の空行

pandocはコード行間に空行を挿入する。次の非空行もコード行か先読みして、コードブロックの範囲を正しく判定する:

```python
elif stripped == '' and in_code:
    # 先読み: 次の非空行がコード行なら継続
    next_is_code = False
    for j in range(i + 1, min(i + 4, len(lines))):
        if lines[j].strip() == '':
            continue
        if re.search(r'\{\.class_s\w+\}', lines[j].strip()):
            next_is_code = True
        break
    if next_is_code:
        continue  # コードブロック継続
```

### ハイパーリンクとの区別

コードスパンのクラス名は`\w+`パターン（例: `class_s3C5`）、ハイパーリンクは`class_s3C9-0`のようにハイフンを含む。検出時に`\{\.class_s\w+\}`（ハイフンなし）を使うことでコードのみを対象にできる。

## pandoc行折り返しの結合

pandocは長い行を任意の位置で改行する。以下のルールで結合する。

### 結合すべきケース

| ケース | 元の形式 | 結合後 |
|-------|---------|--------|
| 選択肢（文字のみ行） | `A.\n選択肢テキスト` | `A. 選択肢テキスト` |
| 選択肢（テキスト折り返し） | `D. テキストが\n次の行に続く` | `D. テキストが次の行に続く` |
| 不正解解説（文字のみ行） | `B:\nテキスト` | `B: テキスト` |
| 不正解解説（折り返し） | `D: テキストが\n次の行に続く` | `D: テキストが次の行に続く` |
| 段落テキスト | `Cloud\nRunサービス` | `Cloud Runサービス` |
| バレット選択肢（インデント継続） | `- テキスト\n  続きの行` | `- テキスト 続きの行` |

### 結合しないケース

以下のマーカーで始まる行は新しいブロックの開始と見なし、前の行とは結合しない:

- 選択肢マーカー: `A.` 〜 `I.`（最大9選択肢まで対応）
- セクションマーカー: `解説:`, `不正解の選択肢の解説:`, `正解の選択肢:`, `参考資料:`
- 不正解解説マーカー: `A:` 〜 `I:`
- 見出し: `#`, `##`, `###`
- 空行

## コンテンツ構造の自動検出

### 検出すべきマーカー

#### 問題マーカー

| マーカー種別 | 例 | 検出パターン |
|------------|-----|------------|
| 問題番号（日本語） | 問1、問題1、第1問、第一問 | `問\d+`、`問題\d+`、`第\d+問` |
| 問題番号（英語） | Q1、Question 1 | `Q\d+`、`Question\s+\d+` |
| 括弧番号 | (1)、【1】、[1] | `\(\d+\)`、`【\d+】` |
| 見出し付き番号 | `## 問題 1`、`# Q1` | 見出しレベル + 問題キーワード |
| 見出し埋め込み型 | `##### 問題 1. [問題文全体]` | 問題番号と問題文が同一見出し内 |
| インラインテキスト型 | `[Question 1]{.class}: question text` | 見出しではなく本文中に埋め込み |
| Bold装飾型（日本語） | `**問題1 ★★**`、`**問題 5** ★★★`、`**問題1** ★ **重要!**` | `\*\*問題\s*\d+\*\*` または `\*\*問題\s*\d+\s*★+(?:\s*\*\*重要[!！]\*\*)?\*\*` |
| Bullet prefixed 番号型 | `- **095** 問題文…`、`- **100** 問題文…`（章末・最終ページのみ箇条書きスタイルに切り替わる場合） | `^(?:-\s+)?\*{0,2}\d{1,3}\*{0,2}\s+` のように `(?:-\s+)?` を先頭に許容 |
| ○×問題本文（選択肢なし） | 問題番号の直後に平叙文1つのみが続く | 問題ブロック内に `A.` `B.` 等の選択肢マーカーが存在しない |

> ⚠️ **`第N問` マーカーの装飾内部混入と空白フィラー**: pandoc変換時に同一書籍内で次の5変種が混在するケースがある:
>
> | 変種 | 例 |
> |------|-----|
> | 標準 | `第3問` |
> | 内部空白 | `第 3 問` |
> | 全体bold | `**第 3 問**` |
> | 数字のみbold | `第 **3** 問` |
> | 見出し化 | `### 第13問` / `#### 第38問` |
>
> 単一の `第\d+問` パターンでは検出漏れが多発する。検出用正規表現は `第\s*\*{0,2}\s*(\d+)\s*\*{0,2}\s*問` のように **`*` の0〜2個許容 + 数字前後の空白許容** を必須とする。マッチ前に `text = text.replace(' ', ' ')`（NBSP→半角空白）の正規化を実施すること。


> ⚠️ **書籍末尾セクションのみ bullet list 形式に変わる罠**: ○×型一問一答セクション（例: 100問中の最終10問）では、章末・最終ページのレイアウト都合により bold装飾型 `**095** 問題文` から **bullet 形式** `- **095** 問題文` に切り替わるケースがある。問題行と解答行のどちらでも発生する。ファイル全体を bullet なし regex でパースすると最終 5〜10 問が systematic に欠落する。**問題番号 regex の先頭に `(?:-\s+)?` を必ず許容**し、`has_answer_markers` 判定（`^\d{3}\s+[○×✕]` 等）にも同じ prefix を加えること。検出漏れの兆候: パース直後の `q_num` ソート出力で**末尾連番だけがごっそり欠ける**。

#### 解答マーカー

| マーカー種別 | 例 |
|------------|-----|
| セクション見出し | `## 解答`、`# 答え`、`## Answers` |
| インライン見出し | `**解答**`、`**正解**`、`**Answer**` |
| ラベル付き | `解答:`、`答え:`、`Answer:` |
| 括弧付き | `【解答】`、`【答え】`、`【正解】` |
| 解説付き | `解説:`、`Explanation:` |
| 解答例見出し | `##### 解答例` |
| 数字レスポンス型 | `Correct Response: 4` |
| Bold装飾型（番号付き○×） | `**解答1 ○**`、`**解答2 ✕**`、`**解答 3 ×**` |
| ○×判定マーカー | `○` (U+25CB), `✕` (U+2715), `×` (U+00D7), `〇` (U+3007) | 正規化: 全て `○` または `✕` に統一 |

> ⚠️ **ローマ数字ページ番号と○×判定マーカーの衝突**: pandoc変換後のMarkdownにはローマ数字のページ番号（`ii`, `iii`, `iv` 等）が残ることがある。これらを検出する正規表現で `[ivxIVX]{1,5}` のように1文字以上を許容すると、○×型問題の判定マーカー `X` (U+0058) や `V` を「ローマ数字ページ番号」として誤検出して食い潰してしまう。**対策: ローマ数字ページ番号の検出は2文字以上を必須とすること**（`[ivxIVX]{2,5}`）。

> ⚠️ **解答マーカーの複合前置記法**: `**解答N**` 形式の前に複数のMarkdown装飾・記号が付くケースがある（箇条書きプレフィックス `- `、verdict プレフィックス `○ ` / `✕ ` / `× `、三重アスタリスク装飾 `***`、見出し記号 `### ` 等）。これら複合前置を許容するパターン例:
>
> ```
> ^(?:-\s+)?(?:[○〇◯×✕XxOo]\s+)?(?:\*\*\s*|###\s+)?解\s*答
> ```

**正解の数字→レターマッピング:**

| 数字 | レター |
|------|--------|
| 1 | A |
| 2 | B |
| 3 | C |
| 4 | D |
| 5 | E |
| 6 | F |
| 7 | G |
| 8 | H |
| 9 | I |

#### 境界マーカー

| マーカー種別 | 例 | 意味 |
|------------|-----|------|
| 水平線 | `---`、`***` | ページ区切りの可能性 |
| ページ番号 | `- 1 -`、`Page 1` | 元のページ区切り |
| 大きな空行 | 3行以上の空行 | セクション区切りの可能性 |

> ⚠️ **3桁ゼロパディング問題番号とページマーカーの混同**: 「短い数字 + 短文」パターンでページマーカー（例: `10 最重要項目チェック`）を検出するヒューリスティクスを実装する場合、**ゼロパディングされた問題番号**（例: `029 使用者責任の成立要件として誤っているものは…`）を誤ってページマーカーとして除去してしまう罠がある。
>
> **対策**:
> - 許容する数字桁数を **1〜2桁のみ** に限定する（`^\d{1,2}\s+\S`）
> - ゼロ始まり数字は問題番号と推定して除去対象から除外（`^0\d+` を許容しない）
> - 数字直後の文字列長で識別する（短い見出し語のみ許容、長文が続けば本文と判断）

### 構造パターンの判定ロジック

以下の順序で判定する。先に該当したパターンを採用する。

#### 判定1: 前半後半分離型

**判定条件:** ファイルの中盤〜後半に「解答」「答え」等の大きなセクション見出しが1つだけ存在し、前半に問題、後半に解答が並ぶ。

**パース方法:** 解答セクション見出しで2分割 → 問題番号でマッチング。

#### 判定2: 交互配置型

**判定条件:** 問題マーカーと解答マーカーが交互に出現する。

**パース方法:** マーカーの出現位置をリスト化 → 交互にペアリング。

#### 判定3: 同一セクション型

**判定条件:** 各セクション内に問題マーカーと解答マーカーの両方が含まれる。

**パース方法:** セクション内で解答マーカーを境界に分割。

#### 判定4: 章末解答型

**判定条件:** 複数の章に分かれ、各章の末尾に解答セクションがある。

**パース方法:** 章単位で分割 → 章内の問題・解答をマッチング。

> ⚠️ **章末集中問題集型サブパターン（要点解説テキスト主体）**: テキスト本文の各テーマ節は要点解説テキストのみで構成され、問題マーカーを含まない。問題と解答は章の末尾に一括集約される構造（`# 第N章 → Themeセクション群 → 章末問題集（**問N**） → 答え合わせ（**問N 正解：①**）`）のサブパターン。パース戦略: **テーマ単位**ではなく**章単位**を抽出アンカーとし、章末から `\*\*問\d+\*\*` 起点で問題ブロック、`\*\*問\d+\s+正解\s*[：:]` 起点で解答ブロックを切り出す。Theme節のテキストは問題と紐付けず、章見出し情報としてタグ化のみに利用する。

> ⚠️ **インライン埋め込み型解答マーカー（`**問N 正解：①**` 形式）の致命的失敗パターン**: 解答情報が `**問1 正解：①**` のように bold 装飾の中にインラインで埋め込まれているケースでは、既存の `^解答|^正解|^Answer` 等の行頭マッチ正規表現では**検出件数ゼロ**になる。検出に成功する正規表現例: `^\*\*問\s*\d+\s+正解\s*[：:]\s*[①-⑳\d]+\s*\*\*`。解答ブロックの境界は「次の `\*\*問\d+\s+正解` 行」または「次の `\*\*問\d+\*\*` 行（新問題開始）」または「章末」とする。

> ⚠️ **番号正解（丸数字 ①〜⑩）方式の追加対応**: 正解表示がレター方式（A./B./C./D.）・数字方式（1./2./3./4.）だけでなく、丸数字（① U+2460〜⑩ U+2469）を使用するパターンがある。これは「組み合わせを①〜⑥の中から1つだけ選びなさい」型のメタ問題形式と組み合わせて使用される。Front の選択肢は `<ol>` の連番付与が可能。Back の正解表示は `<b>正解:</b> ① [組み合わせ内容]` のように丸数字を保持して表示する。

> ⚠️ **ア・イ・ウ・エ小問 + 組み合わせ番号選択型のフォーマット指針**: 複数小問（ア/イ/ウ/エ）の正誤判定を組み合わせた番号（①〜⑥等）を1つ選ぶ問題形式。既存スキルでは複数小問の存在に言及しているが、「組み合わせ番号を1つ選ぶ」型のフォーマットは未明文化。Front: 主問題文 + 「次のア〜エの記述のうち〜組み合わせを①〜⑥の中から1つだけ選びなさい」+ 小問ア〜エの平叙文を `<ol>` で表示 + 組合せ選択肢 ①〜⑥ を別 `<ol>` で表示する。Back: 正解番号 + 各小問ア〜エの○×判定 + 解説を記載する。

> ⚠️ **難易度マーカー ★☆ 混在形式への対応**: 難易度マーカーが `★★☆`（中）/ `★☆☆`（易）/ `★★★`（難）のように ★ と ☆ が混在するパターンがある。既存の `★+` パターンでは ☆ も含むマーカー文字列に対して「★の数だけ難易度」を正しく算出できない。抽出ロジック: `re.findall(r"★", marker_line)` で★の個数を集計するか、`★(?=[★☆\s」$])` 等の lookahead で連続★数をカウントする。値域: ★☆☆→易（`難易度::★`）/ ★★☆→中（`難易度::★★`）/ ★★★→難（`難易度::★★★`）。マーカー不明・未記載は「難易度不明」タグを付与する。

> ⚠️ **半角コロン `:` と全角コロン `：` の混在**: 同一書籍内でも `正解：` と `正解:` が混在するケースがある。すべての解答マーカー検出正規表現で `[：:]` 文字クラスを使用し、半角・全角どちらも許容すること。例: `^\*\*問\s*\d+\s+正解\s*[：:]\s*` のように文字クラスで統一対応する。

> ⚠️ **解答マーカー4種表記揺れの正規化順序**: 同一EPUB内でも章ごとに `**問N 正解：X**` / `**問N** 正解：X` / `# 問N 正解：X` / `問N 正解：X` の **4形式が混在**することがある。pandoc変換時のヘッダーレベル推定や太字装飾のスコープ判定が章ごとに揺れるため、単一パターンでは検出漏れが発生する。
>
> **3段階の事前正規化（順序厳守）**:
>
> ```python
> # 段階1: 行頭の見出し記号 # / ## / ### を除去（`# 問N 正解：X` → `問N 正解：X`）
> text = re.sub(r'^#{1,6}\s+(?=問\s*\d+\s+正解\s*[：:])', '', text, flags=re.MULTILINE)
> # 段階2: 単独装飾 `**問N**` を除去（後続に正解インラインが続かない問題開始行のクリーンアップ）
> text = re.sub(r'^\*\*\s*問\s*(\d+)\s*\*\*\s*$', r'問\1', text, flags=re.MULTILINE)
> # 段階3: `**問N 正解：X**` の外側 `**` を除去して内側のインライン構造を平坦化
> text = re.sub(r'^\*\*(問\s*\d+\s+正解\s*[：:][^*]+?)\*\*', r'\1', text, flags=re.MULTILINE)
> ```
>
> 段階3 を段階1・2より先に実行すると、`# 問N 正解：X`（見出し+インライン未装飾）が検出対象外となり、章ごとの抽出件数に偏りが出る。**段階1→段階2→段階3 の順序が必須**。正規化後は単一の `^問\s*\d+\s+正解\s*[：:]\s*(\S+)` パターンで全形式を検出できる。

> ⚠️ **「答え合わせ」見出しを解答集セクション開始マーカーとして追加**: 既存の解答マーカーキーワード（解答/答え/Answer/正解/解説）に加え、`答え合わせ` を登録する。章末で `^##?\s*答え合わせ` を検出した場合、それ以降を解答ブロック領域として認識する。これを未登録のままにすると、章末解答型のパーサーが解答セクション開始を検出できず、解答件数ゼロになる。

> ⚠️ **Theme番号の表記揺れ（`# Theme N` / 単独数字見出し / 改行分離型）**: 同一書籍内でもテーマ見出しの形式が揺れるケースがある。(a) `# Theme 1 タイトル` 形式、(b) `# 1`（単独数字見出し）形式、(c) 前行が `Theme` のみでその次行に番号タイトルが来る改行分離型。パース時に「Theme番号＋タイトル」を正規化して連結する。改行分離パターンの検出: 前行が `^Theme\s*$` または `^テーマ\s*$` の場合、次の見出し行と論理的に連結する。既存の「テーマ単位ページ分離型」（判定6）では `**テーマ\d+` 形式しか想定していないため、半角英字 `Theme` 形式も許容するよう注記する。

#### 判定5: その他（フォールバック）

上記に該当しない場合、利用可能なマーカーを最大限活用して推測抽出 → サンプル確認でユーザー判断。

#### 判定6: テーマ単位ページ分離型

**判定条件:**

- ファイルが `---` の水平線で頻繁に分割されている（書籍のページ区切り由来）
- `**問題N**` 形式のbold装飾問題番号が連続して並ぶ「問題ページ」と、`**解答N**` 形式が連続する「解答ページ」が**ペアで繰り返される**
- 問題ブロックと解答ブロックの間に章・節・テーマ見出しが挟まることがある
- ○×判定マーカー（`○` / `✕` / `×`）が解答ブロックの先頭に存在する

**パース方法:**

1. **テーマセクション抽出**: `**テーマ\d+`、`## 第\d+節`、`# 第\d+章` の見出しでファイルを階層分割
2. **問題ブロック検出**: 各テーマ内で `^\*\*問題\s*\d+` で始まる行を起点として、次の `^\*\*問題\s*\d+` または `---` までを1ブロックとして切り出す
3. **解答ブロック検出**: 同様に `^\*\*解答\s*\d+` で始まる行をブロック単位で切り出す
4. **番号マッチング**: 問題N と 解答N を**問題番号で対応付け**（出現順ではなくテーマ内での番号一致）
5. **メタ情報抽出**: 各問題ブロックから難易度（`★+`）、重要マーカー（`\*\*重要[!！]\*\*`）を抽出してタグ化
6. **○×正規化**: 解答先頭の判定マーカーを `○` または `✕` に統一（`×`→`✕`、`〇`→`○`）

> ⚠️ **目次・要点整理セクションの除外**: ファイル冒頭の「# 目次」、各章末の「要点整理」、書籍紹介ページ（はしがき、傾向と対策等）は問題ブロックを含まない。テーマ見出しが存在しないブロックはスキップする。

> ⚠️ **テーマ見出しの欠落補完**: 一部のテーマブロックではpandoc変換でテーマ見出しが欠落することがある。直前のテーマを継承するか、解答ブロックに含まれるテーマ見出しから逆引きする。

> ⚠️ **コンテキスト参照渡しの罠**: 章/節/テーマのコンテキストを `Context` インスタンスやdictで保持しながらパースする場合、問題関数と解答関数で**同一インスタンスを共有**すること。コピー（`dict.copy()` / `dataclasses.replace()` 等）を解答関数に渡すと、解答ページ内の見出しから取得した章情報が次の問題ランに伝播せず、解答ページが章境界をまたぐケースで章情報が古いまま固定される。

> ⚠️ **冒頭プレビュー問題の重複**: 資格試験書籍の冒頭には「サンプル問題」「お試し問題」として本編章節の最初の数問を再掲するパターンがある。問題文の**完全一致で重複検出し、後出現を残してデダプ**すること（先出現を残すとサンプルセクションには章コンテキストが付与されておらず章タグが欠落する）。後出現の本編側を採用することで、章・節・テーマ情報が正しく付与されたカードが生成される。

> ⚠️ **OCR変換ログの混入**: 画像ベースEPUBを `recognize-image.py` 等のOCR LLMでMarkdown化したファイルでは、変換LLMの「思考プロセス」（`1. **Analyze the Request:**`、`2. **Scan the Image Content:**`、`*Self-Correction during transcription:* ...`、`*Wait, looking at the first block again.* ...` 等の英語メタ文）がページ単位で大量に混入することがある。同じ文字判定を繰り返す無限ループ（「`*Wait, I think I see it now.*` `The character is "被" (hi) and "者" (sha).`」を50回反復等）も発生する。これらは**ページ全体を汚染するだけでなく、本来のQ&Aコンテンツも同ページ内に部分的に再構成されている**ため、安易にページ全体を破棄すると本物のQ&Aを失う。
>
> **検出と対応**:
> - `1.\s+\*\*Analyze the Request:\*\*` / `Scan the Image` / `Transcribe Section by Section` / `Final Formatting` / `Self-Correction` をボイラープレートパターンとして登録
> - **OCRモデルがシステムプロンプトを復唱するパターン**も追加登録: `The user wants the text` / `\*\*Analyze the Request:\*\*`（行頭斜体無し）/ `\*\*Scan the Image Structure:\*\*` / `\*\*思考プロセス:\*\*` / `ユーザーは(.+)?画像(の|内の|から)`（日本語版「ユーザー意図再構成」マーカー）。これらは復唱直後から同一フレーズの無限ループに突入する兆候として強い
> - 行単位フィルタ: **CJK文字を含まず英字で始まる2語以上の行**は英語メタとして除去（`is_likely_english_meta` 関数）
> - ただし**単独の `O` / `X` / `○` / `×` 行は除外しない**（判定マーカーである可能性が高い）。RE_OXMARK_ONLY を先にチェックすること
> - OCR-thinking内にネストされたQ&Aコンテンツ（例: `* Label: 解答 1 / * Symbol: ○ / * Text: 使用者責任の成立要件として...`）は手動抽出で復元する
> - パーサーはこれらの汚染ページを許容し、対象問題の解答が空欄になったケースを後段で個別補正する戦略を取ること

> ⚠️ **OCR debug 構造化記述からの完全復元**: OCR LLM の思考ログには大きく分けて 2 種類のパターンが観測される: (A) 同一文字を延々と反復するだけの**完全破壊型**（リカバリ困難）、(B) 画像の構造を構造化された Markdown リスト（`1. **ヘッダー部分**: ...` / `2. **問題文**: "..."` / `3. **選択肢**: ① "..." ② "..." ③ "..." ④ "..."` / `4. **解答**: ...` 等）で**整然と記述したまま無限ループに突入する記述型**。後者は思考ループのノイズ行を除けば**問題文・選択肢・解答が原文ほぼそのまま埋め込まれている**ため、ページ全体破棄ではなく**手動復元による完全カード化が可能**。
>
> **検出パターン**: ブロック先頭が以下のような構造化マーカーで始まり、後続行に引用符（`"..."` / `「...」`）で囲まれた本文・選択肢が並ぶ:
>
> - `ユーザーは提供された画像のテキストを Markdown 形式で出力することを求めています。`
> - `The user wants the text from the image transcribed into markdown format.`
> - `**画像の(分析|構造)**:` / `**Identify the main title:**`
> - 続いて `①: "..."` `②: "..."` `③: "..."` `④: "..."` または `a. "..."` `b. "..."` `c. "..."` `d. "..."` の選択肢列挙
>
> **復元手順**:
>
> 1. パーサーで取れなかった問題番号を warning として列挙し、ソース md の該当ブロックを目視確認
> 2. 構造化記述ブロックが見つかった場合、その記述に含まれる引用符内テキストを問題文・選択肢として手動抽出
> 3. 直後の `---` 区切り後にある解答ブロック（こちらは比較的クリーン）から正解番号と解説を取得
> 4. 別 Python スクリプト（例: `recover_missing.py`）で `addNotes` API 経由で個別カード追加
> 5. 復元カードには `OCR汚染復元` タグを付与し、Step 8 完了報告で件数を提示
>
> **実観測ケース**: 8セクション92問抽出中、4問（複数章にまたがる）がこのパターンに該当し、すべて完全復元できた（合計 96問 → 246カード）。OCR debug の構造化記述パターンは内容がほぼ無損失で残るため、自動化スキップ → 手動補完のフォールバックが有効。

> ⚠️ **OCR汚染ブロックからの段階的サルベージ**: ハルシネーション・思考ループでページ全体が汚染されていても、汚染ブロックの内部に**断片的に正しい構造**が残っていることが多い。汚染を理由にページ全体を破棄する前に以下の優先順位でサルベージを試みること:
>
> 1. **解答ヘッダ式救出**: `第\s*\d+\s*問\s*解答\s*[①-⑳]` パターンで番号と正解だけ救出する（解説は失われるが正解は得られる）
> 2. **○×サマリー行救出**: `^\d{3}\s+[○✕×〇◯]\s+\S+` のような「3桁問題番号 + 判定 + 短文」パターンが解答リストとして残存している場合がある
> 3. **答え合わせ集計表フォールバック**: 章末・解答セクション冒頭に `\| 第\s*\d+\s*問 \| [①-⑳\d] \|` 形式のサマリー表が出力されていることがある。これを最終フォールバックとして利用し、最低限「正解番号のみ」のカード（解説欠落）を生成する
>
> 救出したカードには `OCR汚染復元` `解説欠落` タグを付与し、Step 8 完了報告で件数を提示する。

> ⚠️ **OCR LLM の日本語思考ループ汚染（英語メタとは別系統）**: 画像ベースEPUBを生成AIでOCR化する場合、英語の「Analyze the Request」系メタログだけでなく、**日本語による自問自答の思考ループ**が大量に混入することがある。代表例: `画像の分析を行います。` / `いや、待て。` / `よし、「[文字]」と判断する。` / `もう一度、画像の文字を忠実に読む…` / `The character is "被" (hi) and "者" (sha).` の混在反復（同一範囲で10回以上ループするケースを実観測）。
>
> **検出と対応**:
> - 既存の `is_likely_english_meta`（英字始まり2語以上 + CJKなし）では**捕捉できない**（思考ループは日本語と英語が混在し、CJK文字を含むため）
> - 日本語OCR思考ログ専用パターンを追加: `^(画像の(分析|文字)|いや、待て|よし、「.+?」|もう一度|Self-Correction)` 等のアンカーで行を検出
> - **連続反復検出**: 同一文字列が3回以上連続して出現する範囲は思考ループとみなしてブロック単位で除去
> - 同範囲に埋もれた本物のQ&Aは手動再構築でリカバリ（思考ループ範囲は本来コンテンツも破壊されているため、自動抽出は精度が出ない）
> - ボイラープレートパターン管理表に「OCR_JP_THINKING」カテゴリを追加して国際化対応する

> ⚠️ **OCR LLM 思考ログの追加パターン — Mental Draft / 視覚特徴説明文**: 既存のメタログパターン（`Analyze the Request` / `Scan the Image` / `ユーザーは(.+)?画像(の|内の|から)` / 日本語自問自答系）に加え、以下の2系統の汚染が観測される。
>
> | パターン種別 | 例 | 検出キーワード |
> |------------|-----|--------------|
> | Mental Draft マーカー | `*Mental Draft:*` / `Mental Draft 1:` / `# Mental Draft 2` | `(?i)^(?:\*+\s*|#+\s*)?Mental\s+Draft` |
> | 視覚特徴説明文の反復 | `実際には右から書かれている` / `白抜き文字` / `縦書きで表示` / `太字で表示されている` / `赤い丸で囲まれている` | `実際には.*?(右から|左から)` / `白抜き(文字|の)` / `縦書きで` / `太字で表示` / `赤い(丸|枠)` |
>
> **同一フレーズの連続反復**: 視覚特徴説明文は同一範囲で **数十行（10〜50行規模）にわたって同一・類似フレーズを反復**する実観測ケースがある（既存の「3回以上連続」検出基準で捕捉可能だが、ブロック単位除去時のスキャン範囲は最大50行先まで広げること）。
>
> **検出と対応**:
> - ボイラープレートパターン管理表に `OCR_MENTAL_DRAFT` カテゴリと `OCR_VISUAL_DESCRIPTION` カテゴリを追加
> - 視覚特徴説明文は単独行では判定困難（本物のQ&A解説でも `白抜き` 等の語彙が出現しうる）。**同一語彙が3行以上連続出現する場合のみ汚染ブロックとして除去**する語彙密度フィルタを採用
> - 汚染ブロック内に埋もれた本物のQ&Aは Step 8 完了報告で件数を提示し、手動補完を促す

> ⚠️ **章フッターのページ内位置不定混入**: 紙面の章フッター（章番号 + 章タイトルの2行構造）は、通常はページ区切り `---` の直前に出現する想定だが、OCR時のレイアウト崩れにより**問題文の中盤・選択肢の合間・解説段落の途中**にも混入する。位置に依存した単純な「`---` の直前のみ削除」戦略では検出漏れが発生し、抽出された問題文に章フッター断片が残留する。
>
> **対策**: 章フッターの語彙を**明示リスト化**してファイル全体をスイープする:
>
> ```python
> CHAPTER_FOOTERS = [
>     # コンテンツ構造分析（Step 3）で目次見出しから動的に抽出した章タイトル群
>     "章タイトル1",
>     "章タイトル2",
>     # ...
> ]
> for footer_title in CHAPTER_FOOTERS:
>     # 章番号(1〜N) + 改行 + 章タイトル の2行ペアを除去
>     text = re.sub(rf'^\d{{1,2}}\s*\n{re.escape(footer_title)}\s*$', '',
>                   text, flags=re.MULTILINE)
> ```
>
> 章フッター語彙は書籍ごとに異なるため、コンテンツ構造分析（Step 3）の段階で目次見出しから抽出して動的に構築する。**位置依存ではなく語彙依存**にすることで、ページ内位置に関わらず確実にスイープできる。

> ⚠️ **箇条書き形式の解答ブロック**: 一部のページでは解答が箇条書きで列挙される（`*   **解答1 ○**\n    元本が確定すると…`）。`clean_line` で先頭の箇条書きマーカー（`*  ` / `-  ` / `+  `）を除去するとともに、続く**インデント行を継続行として連結**すること。これを処理しないと、解答1の本文がパーサーから消える。

> ⚠️ **Unicode ○×表記揺れの追加**: `○` (U+25CB) / `×` (U+00D7) / `〇` (U+3007) / `✕` (U+2715) に加え、**`◯` (U+25EF BIG CIRCLE) と `❌` (U+274C CROSS MARK)** も判定マーカーとして使われることがある。正規化マップに追加すること。

> ⚠️ **省略された判定マーカーの推論**: 同じ書籍シリーズでも、章・節・テーマ毎にレイアウト方針が異なり、一部の解答ブロックでは ○/× の判定マーカーが**完全に省略され、解説文のみ**が表示されるケースがある（読者は解説文から判定を推論することが暗黙的に要求される）。この場合の処方は2段階:
> 1. **明示マーカー検出を尽くす**: 解答ヘッダの直前・直後・同一行・次行のすべてを ○/× 候補としてスキャンする（O/Latin O / X/Latin X / ○ / × / ◯ / ❌ / 〇 / ✕、半角・全角両方）
> 2. **残った欠落ケースの意味推論**: 解答テキストの末尾に「誤り」「誤って」「設問前段が誤り」「設問後段が誤っている」等の否定キーワードがあれば ✕。質問文の主張を再述・補強する解説（「〜できる」「〜認められる」「本問のとおり」）であれば ○。両義的な場合は Claude が個別に質問と解答を読み比べて判定する
> 3. 判定不能のものは `判定マーカー欠落` タグを付与して保留する（Step 8 完了報告で件数を提示）

> ⚠️ **テーマ単位ページ分離型サブパターン — 複数ページにまたがる解答ブロックの分断**: 同じ「テーマ単位ページ分離型」でも、1つの解答ブロックが複数ページにまたがるバリアントが存在する（解答ヘッダ直後の解説段落が次ページに送られる、または1ページあたり数件の解答のみで占有されるなど）。この場合、`---` 区切りをページ境界としてページ単位で即時に問題/解答をペア化する素朴な実装では、後続ページに続く解答テキストが捨てられ、解答フィールドが欠損・切り詰めになる。
>
> **対策**: `pages` バッファにページを**累積**しながら、(a) 次のテーマ見出し（`**テーマ\d+`・`## 第\d+節` 等）を検出した時、または (b) 次の問題ページ群が始まると判定できた時にまとめて `flush` して Q&A ペアを構築する。パース単位のアンカーは「解答ヘッダが出現した開始ページ」ではなく**「テーマセクション全体」**とすること。

#### 判定7: 混在型

**判定条件:** 1つのEPUB内に異なる構造パターンのセクションが共存する。

**例:** Assessment Test（同一セクション型: `Correct Answer:` + `Explanation:` がQuestion直後）とPractice Exams（前半後半分離型: Questions セクションとSolutions セクションが分離）が共存。

**パース方法:** セクション単位で構造パターンを判定し、各セクションに適したパーサーを適用。最終的に全セクションのQ&Aペアを統合する。

### 判定時の注意事項

- **選択肢との混同回避**: 連番（1., 2.）は問題番号ではなく選択肢の可能性がある。周辺コンテキストで区別
- **解説の扱い**: 「解説」は解答の一部として扱う
- **図表の参照**: テキストはそのまま保持（画像自体は抽出不可）
- **参考資料/URL**: Backフィールドには含めない（除去する）

## フォーマット変換ルール

### 選択肢のリスト化

**形式:**

```html
<ol style="list-style-type: none; padding-left: 0;">
  <li>A. 選択肢テキスト</li>
  <li>B. 選択肢テキスト</li>
</ol>
```

- `list-style-type: none` でデフォルト番号を非表示
- `A.`, `B.` 等のプレフィックスを `<li>` 内に保持
- 全選択肢を**1つの `<ol>` にまとめる**（選択肢ごとに個別の `<ol>` を作らない）
- 選択肢間の空行があっても同一 `<ol>` に含める

**検出パターン:**

| パターン | 例 | 変換 |
|---------|-----|------|
| 半角大文字 + ドット | `A. 選択肢`, `B. 選択肢` | そのまま使用 |
| 半角大文字 + ドット（改行分離） | `A.\n選択肢テキスト` | 1行に結合 |
| バレットポイント（`-`） | `- 選択肢テキスト` | A./B./C./D./E. を自動付与 |

### Frontフィールドの構成

```
[問題文のみ（問題番号ヘッダーは含めない）]
<br><br>
<ol style="list-style-type: none; padding-left: 0;">
  <li>A. ...</li>
  ...
</ol>
```

- **問題番号（「問 N」「Question N」等）はFrontに含めない**
- 問題の内容テキストと選択肢のみ

### Backフィールドの構成

**単一正解の場合:**

```
<b>正解:</b>
A. [正解テキスト]
```

**複数正解の場合（「二つ選択」等）:**

```
<b>正解:</b>
A. [正解テキスト1]
B. [正解テキスト2]
```

**解説セクション:**

```
<b>解説:</b>（太字ラベル）
[解説テキスト]

<b>不正解の選択肢の解説:</b>（太字ラベル）
<b>C:</b> [不正解解説]（各選択肢を太字ラベル付きで）
```

### ○×問題（True/False型）の構成

ソースに選択肢（A./B./C./D.）が存在せず、解答が `○`/`✕`/`×` の判定マーカーで示される場合は以下のフォーマットを使用する。

**Frontフィールド:**

```html
[問題文（平叙文）]<br><br>
<span style="color:#c00;font-weight:bold;">⭐ 重要</span><br><br>   <!-- 重要!マーカー付きのみ -->
<i>（○か✕で答えよ）</i>
```

- 問題番号ヘッダー（`**問題N**`）と難易度マーカー（`★+`）はFrontには**含めない**（タグとして保持）
- `**重要!**` バッジ付きの問題のみ赤色強調表示を挿入
- 末尾の判定指示 `（○か✕で答えよ）` で受験者の解答行動を一意化

**Backフィールド:**

```html
<div style="font-size:2em;font-weight:bold;text-align:center;">○</div>
<br>
<b>解説:</b><br>
[解説テキスト（複数段落は <br><br> で分離）]
```

- 判定結果（`○` or `✕`）を大きく中央表示
- ソースの `×` (U+00D7), `〇` (U+3007) を `✕` (U+2715), `○` (U+25CB) に正規化
- 解説に条文参照（`民法90条`、`憲法29条` 等）がある場合はそのまま保持

> ⚠️ **判定マーカー欠落カードの処理**: 紙面で別カラム表示される `○`/`✕` 判定マーカーがOCR/pandoc変換時に脱落し、解説本文だけが残るケースが資格試験書籍では一定割合発生する。verdict 推論は誤判定リスクが高いため、欠落カードは空欄推論せず、Back先頭に注記を表示する:
>
> ```html
> <div style="font-size:1.2em;color:#888;text-align:center;">（原本に○×マーカー欠落 — 解説で判定）</div>
> <br>
> <b>解説:</b><br>
> [解説テキスト]
> ```
>
> タグに `判定マーカー欠落` を付与しておくと、ユーザーが後から手動補完しやすい。

**タグ生成:**

| メタ情報 | タグ形式 |
|---------|---------|
| 章 | `章::第N章-{章タイトル}` |
| 節 | `節::第N節-{節タイトル}` |
| テーマ | `テーマ::{テーマタイトル}` |
| 難易度 | `難易度::★`, `難易度::★★`, `難易度::★★★` |
| 重要マーカー | `重要` |

> ⚠️ Ankiの階層タグは `::` 区切り。空白は `-` に置換し、`:` `/` 等は除去する。

### 比較テーブルのHTML変換

pandocが出力するスペース区切りのプレーンテキストテーブルを、HTMLテーブルに変換する。

**検出方法:** テキスト行の直後にダッシュ区切り行（`-------- ----------`）がある場合、テーブルとして認識。

**カラム分割:** 2つ以上の連続スペースをカラム区切りとして使用（セパレータのダッシュ位置ではなく、スペース区切りで分割する。日本語文字幅の違いによるズレを回避）。

**変換例:**

元テキスト:
```
  モニタリングツール   主な用途                 最適か
  -------------------- ------------------------ --------
  Cloud Monitoring     メトリクス収集           はい
  Cloud Trace          分散トレーシング         いいえ
```

変換後:
```html
<table style="border-collapse: collapse; width: 100%; font-size: 0.85em; margin-top: 8px;">
<tr><th style="border: 1px solid #ccc; padding: 4px 6px; background: #f0f0f0;">モニタリングツール</th><th ...>主な用途</th><th ...>最適か</th></tr>
<tr><td ...>Cloud Monitoring</td><td ...>メトリクス収集</td><td ...>はい</td></tr>
<tr><td ...>Cloud Trace</td><td ...>分散トレーシング</td><td ...>いいえ</td></tr>
</table>
```

### 改行とテキスト整形

| ルール | 変換 |
|-------|------|
| 連続する空行（3行以上） | 1つの `<br><br>` に圧縮 |
| 段落間の区切り | `<br><br>` |
| 行頭・行末の空白 | トリム |
| Markdownの見出し記号（`#`） | `<b>` タグに変換 |

### 特殊文字の処理

| 文字 | 処理 |
|------|------|
| `<` / `>` | HTMLエンティティに変換（`&lt;` / `&gt;`）※HTMLタグ以外 |
| `&` | `&amp;` に変換 |
| 画像参照 `![alt](url)` | 除去（テキストのみ抽出） |

## 一括投入の効率化

### AnkiConnect バッチAPI

大量のカード（50件以上）を作成する場合、MCP `add_note` の個別呼び出しではなく、AnkiConnect の `addNotes`（複数形）バッチAPIを直接使用する。

| 方法 | 200件作成時の呼び出し回数 | 推奨場面 |
|------|------------------------|---------|
| MCP `add_note` 個別 | 200回 | 少数（10件以下） |
| AnkiConnect `addNotes` バッチ | 4回（50件×4バッチ） | 大量（50件以上） |

**バッチサイズ**: 50件を推奨。

**⚠️ `createDeck` 必須**: `addNotes` 実行前に必ず `createDeck` でデッキの存在を保証すること。デッキが存在しない場合、`addNotes` は `"deck was not found"` エラーで全件失敗する。`createDeck` は既存デッキに対しても安全に呼び出せる（冪等）。

**エラーハンドリング**: `addNotes` の戻り値は配列。各要素が `null` の場合は重複またはエラーでスキップされたことを意味する。バッチ全体が失敗した場合は `error` フィールドにエラーメッセージが返る（`result` は `null`）。両方をチェックすること。

### 再作成時のワークフロー

フォーマット修正等でカードを再作成する場合:

1. `findNotes` でタグ検索（例: `tag:DevOps-Engineer-模擬問題集-200問`）
2. `deleteNotes` で旧カードを一括削除
3. **`createDeck` でデッキ存在を保証**（⚠️ 全カード削除後に空デッキが自動削除されるため必須）
4. `addNotes` で新カードを一括作成

### 翻訳時の処理

100問以上の外国語コンテンツを翻訳する場合も、LM Studioスクリプトを逐次呼び出しで処理する（ローカルLLMのため高速）:

1. 全Q&AペアをJSON（原文のみ）にパース
2. 各Q&AペアについてLM Studioスクリプトを呼び出して翻訳する
   - 問題文、各選択肢、解答、解説を個別に翻訳する
   ```bash
   python3 ${CLAUDE_PLUGIN_ROOT}/skills/translating-with-lmstudio/scripts/lmstudio-translate.py translate --model <選択済みモデル> --text "翻訳対象テキスト"
   ```
3. 翻訳済みJSONからAnkiConnect `addNotes` で一括投入

## `{style="..."}` 形式EPUBの生テキスト境界検出

一部のEPUB（特にIPSpecialist等の英語問題集）では、pandocが `{.class_sXXX}` ではなく `{style="..."}` 形式のインライン属性を生成する。この形式では質問の先頭に特徴的なフォントサイズ指定パターンが付与される。

### 問題: 質問番号と質問内容中のリストが混同される

通常のパーサーは `N. ` パターンで質問番号を検出するが、質問文中に番号付きリスト（例: 要件リスト `1. 高可用性を確保 2. レイテンシを最小化`）が含まれる場合、これらを新しい質問と誤認する。

### 解決策: 生テキストでpandoc固有パターンを検出

pandoc変換後のクリーニング前テキストで、質問開始位置を固有のフォーマットパターンで特定し、ブロック単位で切り出す:

```python
import re

def find_question_boundaries(raw: str) -> list[tuple[int, int, int]]:
    """生テキストでpandoc固有パターンを使い質問境界を検出。

    Returns: list of (char_position, question_number, line_number)
    """
    # EPUBソース固有のフォーマットパターン（例: font:7.0pt は質問番号のスペーサー）
    pattern = re.compile(
        r'^(\d+)\.\s+\[\s+\]\{style="font:7\.0pt',
        re.MULTILINE
    )
    results = []
    for m in pattern.finditer(raw):
        q_num = int(m.group(1))
        if 1 <= q_num <= 200:
            line_num = raw[:m.start()].count('\n') + 1
            results.append((m.start(), q_num, line_num))
    return results
```

**処理フロー:**

1. **生テキスト**で `find_question_boundaries()` を実行して質問開始位置を特定
2. 各質問のテキストブロックを切り出す（次の質問開始位置まで）
3. ブロック単位で `clean_pandoc()` を適用してアーティファクトを除去
4. クリーニング済みブロックから問題文・選択肢・Answer・解説をパース

> ⚠️ パターンはEPUBソースごとに異なる。上記は一例であり、実際のEPUBの生テキストを確認して適切なパターンを特定すること。

### `{style="..."}` のクリーニング

```python
# {style="..."} 付きスパン: テキストのみ抽出
text = re.sub(r'\[([^\]]*)\]\{style="[^"]*"\}', r'\1', text)
# 単独の {style="..."}: 除去
text = re.sub(r'\{style="[^"]*"\}', '', text)
```

## Answer推論（空回答フィールドからの正解推定）

一部のEPUBでは `Answer:` フィールドが空のまま出版されている問題がある。この場合、解説文（Explanation）から正解を推論する。

### 推論パターン

```python
def infer_answer_from_explanation(explanation: str) -> str | None:
    """解説文から正解レターを推論する。"""
    patterns = [
        # "Option X is correct"
        r'Option\s+([A-I])\s+is\s+(?:correct|the\s+(?:correct|right)\s+answer)',
        # "Options X and Y are correct"
        r'Options?\s+([A-I])\s+and\s+([A-I])\s+are\s+correct',
        # "X is correct" at start
        r'^([A-I])\s+is\s+correct',
        # Broader: "Option X ... correct"
        r'Option\s+([A-I])\b.*?correct',
    ]
    for pat in patterns:
        m = re.search(pat, explanation, re.IGNORECASE)
        if m:
            if m.lastindex == 2:
                return f"{m.group(1)} and {m.group(2)}"
            return m.group(1)
    return None
```

**使用タイミング**: `Answer:` が空文字列または `?` の場合にのみ適用。推論できなかった場合は `?` を設定し、WARNINGログを出力する。

### 複数Answer形式への対応

英語EPUBでは以下のAnswer形式が混在する:

| 形式 | 例 | 検出パターン |
|------|-----|------------|
| コロン付き | `Answer: B` | `^Answer:\s*(.+)` |
| コロンなし | `Answer B` | `^Answer\s+([A-I])` |
| 空欄 | `Answer:` | `^Answer:\s*$` → 推論実行 |
| 複数回答 | `Answer: A and B` | `([A-I])\s+and\s+([A-I])` |
