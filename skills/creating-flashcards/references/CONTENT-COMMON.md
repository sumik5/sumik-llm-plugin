## pandocアーティファクトのクリーニング

pandocはEPUBをMarkdown変換する際に以下のアーティファクトを生成する。すべて除去すること。

| アーティファクト | 例 | 除去パターン |
|----------------|-----|------------|
| CSSクラスマーカー | `{.class_s74W}`, `{.heading_s3M-0}` | `\{[.#][^}]*\}` |
| インラインスパン | `[text]{.class_sXXX}` | テキストのみ抽出 |
| ページマーカー | `[]{#c3W.xhtml}` | 完全除去 |
| divマーカー | `::: heading_s6MV` | 行ごと除去 |
| ゼロ幅スペース付きリストマーカー | `[​- ​]`, `[​1. ​]` | 通常のリストマーカーに変換 |
| Google URL リダイレクト | `[text](https://www.google.com/url?q=...)` | テキストのみ抽出 |
| コードブロックdiv | `::: class_sBD ... :::` | base64退避→パース後復元（後述） |
| エスケープ文字 | `\``, `\*`, `\#`, `\'`, `\~`, `\<`, `\>`, `\--\>` | エスケープ除去（順序注意） |
| 空ブラケット | `[]` | 除去 |
| 誤記ヘッダー | `### 解説 {.heading_s6S}`（正解用クラスなのにテキストが解説） | CSSクラスベースで`正解`に修正 |
| 残存bare bracket | `[テキスト]` | テキストのみ抽出（最終クリーンアップ） |
| **inline style属性** | `{style="font:7.0pt 'Times New Roman'"}` | `\{style="[^"]*"\}` で除去 |
| **inline style付きスパン** | `[text]{style="font-size:11.0pt"}` | テキストのみ抽出 |
| **ネストブラケットリンク** | `[[url]](url)` | `\[\[([^\]]*)\]\]\([^)]*\)` でテキスト抽出 |

> ⚠️ **絵文字付き矢印 `➡️` (U+27A1 + Variation Selector-16) のクリーニング**: 資格試験書籍の解説文では `➡️` (RIGHT BLACK ARROW + U+FE0F) が論理関係を示すマーカーとして使われることがある。pandoc EPUB変換後に `➡️` として残留し、通常の矢印除去パターン (`->`, `→`) では捕捉できない。**対策**: クリーニング段階で `text = re.sub(r'[➡]️?', '→', text)` を適用し、絵文字バリアント付き矢印を標準矢印に正規化する（Ankiカードでの表示崩れ防止）。

> ⚠️ **区切り行のエスケープ揺れ**: pandoc は水平ルールを `\-\-\-...\-\-` 形式（バックスラッシュ-ダッシュ繰り返し）に変換するが、行末の数文字は `\\--` のようにアンエスケープな `-` が混在する場合がある。`\\-+` のような厳密な繰り返し regex はマッチに失敗する。**対策**: `\n+(?:\\?-)+\n+` のように **バックスラッシュ optional + 改行前後 1文字以上** で許容範囲を広げる。観測ケース: `\-\-\-\-\-\-...\-\-\-\-\--`（200文字以上 + 末尾 `\\--`）。問題ブロックと回答ブロックの境界として使われているため、検出失敗すると複数ブロックが merge され大量のカードが欠落する（実観測: 法令穴埋め問題集の split_blocks 実装で `\\-+` 厳密 regex を使用し、58 個の問題ブロックすべて検出失敗 = 124→2 cards のリグレッション）。

> ⚠️ **`回答` 後の境界が区切り行を持たないケース**: 章を跨ぐ問題ブロック (`回答` → 次章見出し → 次問題ブロック) では `\\-+` 区切り行が省略されることがある。`回答` 行を boundary とした単純な ans_position ベース split は問題テキストを 0 長に潰してしまう。**対策**: answer text の終端を「内容パターン」で判定する。`_ANSWER_PARA_PAT = re.compile(r'^\s*(?:第[一二三四五六七八九十百\d]+条|[①-⑮]|【[★■●▲☆□◇◆]】|\[第\]|\[\d|施行令|施行規則|※施行)')` のように answer らしい段落を継続条件にし、それ以外の段落（`# 見出し`・条文標題・条文本文）が現れた地点で answer 終了とみなす。実観測: 法令穴埋めで第二章・第三章・第五章・第十章のカードが完全欠落するケースを解消。

### 🔴 クリーニング処理順序の罠

**inline spans (`[text]{.class_sXXX}`) は CSS class markers (`{.class_s...}`) より先に除去すること。**

順序を誤ると、CSS class markers の除去 (`\{[.#][^}]*\}`) が先に `{.class_sXXX}` 部分を消し、inline span regex (`\[text]\{\.class_s...\}`) がマッチしなくなる。結果として bare bracket `[text]` がテキスト中に残留する。

**正しい処理順序:**

```python
# 1. ページマーカー除去（空ブラケット + アンカーID）
text = re.sub(r"\[\]\{#[^}]*\}", "", text)
# 2. inline spans 除去（テキスト抽出）★ CSS class markers より先
text = re.sub(r"\[([^\]]*)\]\{\.class_s[^}]*\}", r"\1", text)
# 3. CSS class markers 除去（残りの単独マーカー）
text = re.sub(r"\{[.#][^}]*\}", "", text)
# 4. ... 他のクリーニング処理 ...
# 5. 最終クリーンアップ: 残存bare bracket除去
text = re.sub(r"\[([^\]\[]*)\]", r"\1", text)
```

**失敗例**: CSS class markers を先に除去 → `[拒否]{.class_s6U0}` が `[拒否]` になる → inline span regex がマッチせず `[拒否]` が残る。

> ⚠️ **`:::` クラスブロック除去 regex の `\s*` greedy 罠**: pandoc は画像挿入の周辺に `::: class_sXX-Y` ブロックを生成する。これを `re.sub(r':::\s*[^\n]*', '', text)` のように削除すると、Python の `\s` がデフォルトで `\n` を含むため、`:::` 直後の `\s*` が **複数行をまたいで貪欲に空行を消費**し、その後の `[^\n]*` が次の意味のある行（例: `【解答】 Ｂ　二つ`）を消し飛ばす。**対策**: `\s*` を `[ \t]*` (改行を含まない horizontal whitespace のみ) に置き換える。または `:::[^\n]*` のように `\s*` 自体を省く。観測ケース: 過去問題集の `:::` ブロック直後の `【解答】` 行が完全に消失し、択一式パーサーが answer 検出に失敗（全 13 個の `【解答】` が clean_inline 出力から消失、70 件のカードが 0 件として検出された）。

### `<pre>`ブロック保護

`\{[^}]*\}`パターンでpandocクラスを除去する際、**`<pre><code>`ブロック内の`{`や`}`も誤ってマッチする**危険がある（`[^}]`は改行にもマッチするため、遠くの`}`まで横断マッチする）。

**対策**: クリーニング処理の前に`<pre>...</pre>`ブロックをプレースホルダーに退避し、処理後に復元する:

```python
pre_blocks = {}
counter = [0]
def save_pre(m):
    key = f'\x00PRE{counter[0]}\x00'
    pre_blocks[key] = m.group(0)
    counter[0] += 1
    return key
text = re.sub(r'<pre[^>]*>.*?</pre>', save_pre, text, flags=re.DOTALL)
# ... クリーニング処理 ...
for key, value in pre_blocks.items():
    text = text.replace(key, value)
```

### 誤記ヘッダーの自動修正

EPUBソースでセクションヘッダーのテキストが誤っていることがある（例: 「正解」セクションのヘッダーテキストが「解説」になっている）。pandocのCSSクラスで本来の意図を推定できる:

```python
# heading_s6S = 正解用クラス、heading_s6V = 解説用クラス
block = re.sub(
    r'^(#{2,4})\s*解説\s*(\{\.heading_s6S\})',
    r'\1 正解 \2',
    block,
    flags=re.MULTILINE,
)
```

### エスケープ文字の除去順序

pandocのエスケープ文字を除去する際、**複合エスケープを先に処理する**こと。順序を誤ると部分置換が発生する:

```python
# ⚠️ 順序が重要: 複合エスケープ → 単純エスケープ
text = text.replace(r'\--\>', '-->')   # 複合を先に
text = text.replace(r'\>', '>')         # 単純を後に
text = text.replace(r'\<', '<')
text = text.replace(r'\~', '~')
text = text.replace(r'\#', '#')
text = text.replace(r'\*', '*')
text = text.replace("\\'", "'")         # アポストロフィ
text = text.replace(r'\$', '$')         # 金額の $（$10,000 → \$10,000）
text = text.replace('\\-->', '-->')     # フォールバック（部分置換の残骸対策）
```

> ⚠️ **金額の `\$` エスケープと「全 `\X` 棚卸し」**: pandoc は本文中の `$`（金額表記 `$10,000` 等）を `\$` にエスケープする。除去テーブルに `\$`→`$` が無いと、上級問題・ビジネス文書で `\$10,000` が残存する。金額表現を含まない書籍では露見しないため、**ソースが変わるたびに `re.findall(r'\\(.)', text)` で全 `\X` パターンを頻度カウントして棚卸し**し、除去対象の漏れ（`\$` `\|` `\^` 等）を機械的に確認すること。

> ⚠️ **`&nbsp;` / ` ` の行内フィラー混入**: 紙面の段組み調整で挿入された non-breaking space が、Markdown変換後も `&nbsp;` HTMLエンティティまたは生の ` ` 文字として行内に残留する。例: `**第 1 問** &nbsp;&nbsp;&nbsp; 難易度 ★★★`。これを残したまま正規表現マッチを行うと、`第\s*\d+\s*問` の `\s` は ` ` にマッチするが `&nbsp;` 文字列にはマッチせず検出漏れを起こす。**クリーニング段階の早期で** `text = text.replace('&nbsp;', ' ').replace(' ', ' ')` を実行し、両形式を半角空白に正規化すること。

**失敗例**: `\>` → `>` を先に実行すると、`\--\>` が `\-->` になり、その後の `\--\>` → `-->` 置換がマッチしなくなる。

> ⚠️ **全角・半角英字混在の Front フィールド正規化**: 資格試験書籍のEPUBでは問題文中に全角英字（`Ａ` U+FF21〜`Ｚ` U+FF3A / `ａ` U+FF41〜`ｚ` U+FF5A）が使われることがある（紙面デザインの都合）。同一問題が年度版間で半角英字に改訂されると、Front フィールドの差分判定で「別問題」と誤判定される。**対策**: Front フィールド生成前に全角英字を半角に正規化する: `import unicodedata; text = unicodedata.normalize('NFKC', text)`。`NFKC` 正規化は全角英数字・全角記号・半角カナも同時に正規化するため、副作用（意図的な全角表記の消失）に注意し、タグ・デッキ名には適用しないこと。

> ⚠️ **同一書籍内で `::: class_sXXX` ラッパクラスが複数混在**: ○×型一問一答書籍では、問題文ラッパが基本 `::: class_s10` だが、ごく一部の問題で `::: class_s36S` などの別クラスに変異する(実観測: 275問中、別クラスラッパは1問、ラッパ完全欠落のプレーンテキスト問題は4問)。同じシリーズでも書籍ごとに基本ラッパが異なる(`class_s10` / `class_s14` 等)。**対策**: 問題文抽出 regex は `:::\s*class_s\w+\s*\n(.*?)\n:::` のように `\w+` ワイルドカードで全クラスに対応する + さらにラッパ完全欠落の問題に対するプレーンテキスト fallback を実装する: 問題マーカー直後から「■ 解答」(または `[■]{.class_sXXX} 解答`)出現位置までの非空行をプレーン問題文として扱う。検出の兆候: パーサ実装で「総問題数N、パース成功M、failed M-N 件」が発生し、failed 問の番号が散発的に分布する。

> ⚠️ **複数行にまたがる `[text]{.class}` スパンの角括弧残骸**: pandoc の inline span `[本文]{.class_sXX}` が長文で2行に分断される（1行目 `[本文前半`／2行目 `本文後半]{.class}`）と、**行単位のクリーニングでは開き `[` と閉じ `]` が別行に残る**。`\[([^\]]*)\]\{...\}` の1行 regex はマッチせず、結合後に `[本文]` の角括弧が残留する。1行に収まったスパンは正常に除去されるため、長い選択肢・設問だけで散発的に発生する。**対策**: 行単位クリーニング後、各フィールド（front／選択肢／解説）を結合した文字列へ `re.sub(r"[\[\]]", "", t)` を適用して残骸角括弧を除去する（選択肢・設問・解説は通常角括弧を含まないため安全）。

## pandoc div形式コードブロック（`::: class_sBD`）

pandocはEPUBの一部のコードブロックを `::: class_sBD ... :::` のdiv形式で出力する。この形式はスパン注釈形式とは異なり、コード全体がdivブロックで囲まれる。

### 構造

```
::: class_sBD
resource "aws_instance" "example" {
  ami           = "ami-12345"
  instance_type = "t2.micro"
}
:::
```

### base64退避パターン

divブロック内のコードは問題テキスト・選択肢テキストの一部として解析する必要があるが、行ベースのパーサーを混乱させる。**パース前にbase64エンコードでプレースホルダーに退避**し、パース後に復元する:

```python
import base64, re

def preprocess_code_blocks(text):
    def replace_block(m):
        code = m.group(1).strip()
        code = re.sub(r'\\\s*$', '', code, flags=re.MULTILINE)  # pandoc行継続除去
        encoded = base64.b64encode(code.encode('utf-8')).decode('ascii')
        return f'{{CODE:{encoded}}}'
    return re.sub(r'::: class_sBD\n(.*?)\n:::', replace_block, text, flags=re.DOTALL)

def decode_code_block(marker):
    m = re.match(r'\{CODE:([A-Za-z0-9+/=]+)\}', marker)
    return base64.b64decode(m.group(1)).decode('utf-8') if m else marker
```

**重要**: この退避処理は`clean_pandoc()`よりも**前に**実行すること。`clean_pandoc`の`:::`行除去がコードブロックの内容を破壊するのを防ぐ。

## pandocスパン注釈コードブロック

pandocはEPUBのコード部分を`[text]{.class_sXXX}`形式のスパン注釈に変換する。このパターンはプレーンテキストのコードブロックとは異なる処理が必要。

### スパン注釈の構造

```
[variable]{.class_s3CW}[ ]{.class_s3C5}["instance_type"]{.class_s3CX}[
{]{.class_s3C5}
```

| クラス種別 | 役割 | 例 |
|-----------|------|-----|
| `class_s3CW` | キーワード | `variable`, `resource` |
| `class_s3C5` | 通常コード | `{`, `}`, スペース |
| `class_s3CX` | 文字列リテラル | `"instance_type"` |
| `class_s3E0` | 空白 | スペース、改行 |
| `class_s3CF` | 演算子 | `=` |
| `class_s3C9-0` | ハイパーリンク（コードではない） | URL |

### 処理上の3つの罠

#### 罠1: スパンの行分割

pandocはスパンを行をまたいで分割する。ブラケット数を数えてバランスが取れるまで行を結合する:

```python
def join_split_spans(text):
    lines = text.split('\n')
    result = []
    i = 0
    while i < len(lines):
        line = lines[i]
        temp = line.replace('\\[', '').replace('\\]', '')
        while temp.count('[') > temp.count(']') and i + 1 < len(lines):
            i += 1
            line = line + ' ' + lines[i].strip()
            temp = line.replace('\\[', '').replace('\\]', '')
        result.append(line)
        i += 1
    return '\n'.join(result)
```

#### 罠2: エスケープされたブラケット

コード内容に`\[`、`\]`、`\"`が含まれる場合、スパンのブラケットと混同される。プレースホルダーで保護する:

```python
def extract_code_text(text):
    text = re.sub(r'\{\.class_s[\w-]+\}', '', text)  # クラス除去
    text = text.replace('\\[', '\x00LB\x00')          # 保護
    text = text.replace('\\]', '\x00RB\x00')
    text = text.replace('\\"', '\x00DQ\x00')
    text = text.replace('[', '').replace(']', '')      # スパンブラケット除去
    text = text.replace('\x00LB\x00', '[')             # 復元
    text = text.replace('\x00RB\x00', ']')
    text = text.replace('\x00DQ\x00', '"')
    return text
```

#### 罠3: コード行間の空行

pandocはコード行間に空行を挿入する。次の非空行もコード行か先読みして、コードブロックの範囲を正しく判定する:

```python
elif stripped == '' and in_code:
    # 先読み: 次の非空行がコード行なら継続
    next_is_code = False
    for j in range(i + 1, min(i + 4, len(lines))):
        if lines[j].strip() == '':
            continue
        if re.search(r'\{\.class_s\w+\}', lines[j].strip()):
            next_is_code = True
        break
    if next_is_code:
        continue  # コードブロック継続
```

### ハイパーリンクとの区別

コードスパンのクラス名は`\w+`パターン（例: `class_s3C5`）、ハイパーリンクは`class_s3C9-0`のようにハイフンを含む。検出時に`\{\.class_s\w+\}`（ハイフンなし）を使うことでコードのみを対象にできる。

> ⚠️ **条文ヘッダーと本文中参照の混同**: `第N条` パターンを単純に検索すると `第42条、第43条及び附則第2条において同じ` のような本文中参照まで article header として誤検出する。**対策**: article header regex は `(?:^|\n)第(\d+)条(?:の(\d+))?(?=[ 　])` のように **行頭限定 + 全角空白フォロー必須** にする。本文中参照は `第N条第一項の規定` のように `第` の後ろが漢数字続きで全角空白ではないため自然に除外される。観測ケース: 法令穴埋め問題集で施行令第41条の article body が「第42条、第43条及び附則...」の参照記述で article_pattern にマッチし、本来の article body が分断されて 0 holes 認識となった。

## pandoc行折り返しの結合

pandocは長い行を任意の位置で改行する。以下のルールで結合する。

### 結合すべきケース

| ケース | 元の形式 | 結合後 |
|-------|---------|--------|
| 選択肢（文字のみ行） | `A.\n選択肢テキスト` | `A. 選択肢テキスト` |
| 選択肢（テキスト折り返し） | `D. テキストが\n次の行に続く` | `D. テキストが次の行に続く` |
| 不正解解説（文字のみ行） | `B:\nテキスト` | `B: テキスト` |
| 不正解解説（折り返し） | `D: テキストが\n次の行に続く` | `D: テキストが次の行に続く` |
| 段落テキスト | `Cloud\nRunサービス` | `Cloud Runサービス` |
| バレット選択肢（インデント継続） | `- テキスト\n  続きの行` | `- テキスト 続きの行` |

### 結合しないケース

以下のマーカーで始まる行は新しいブロックの開始と見なし、前の行とは結合しない:

- 選択肢マーカー: `A.` 〜 `I.`（最大9選択肢まで対応）
- セクションマーカー: `解説:`, `不正解の選択肢の解説:`, `正解の選択肢:`, `参考資料:`
- 不正解解説マーカー: `A:` 〜 `I:`
- 見出し: `#`, `##`, `###`
- 空行

> ⚠️ **日本語ソフトラップ結合での英語専門用語グルー化（無条件 no-space 結合の罠）**: 日本語コンテンツのソフトラップ行を「無条件で区切りなし結合」すると、pandoc が**英語の複合語・専門用語をフレーズ途中で改行**していた場合に `Random Access\nMemory` → `Random AccessMemory`、`Single\nInstruction` → `SingleInstruction`、`hash\nfunction` → `hashfunction` のように**英単語同士がスペースなしで癒着**する。日本語本文は区切りなし結合が正しいが、英文混在箇所だけ破損する。**対策**: ソフトラップ結合を「結合境界の左右文字が両方とも ASCII 英数字 `[A-Za-z0-9]` のときのみ半角スペースを挿入、それ以外は区切りなし」のスマート結合に変更する。

```python
def smart_join(lines):
    out = ""
    for ln in (l.strip() for l in lines):
        if not ln:
            continue
        if out and re.match(r"[A-Za-z0-9]", out[-1]) and re.match(r"[A-Za-z0-9]", ln[0]):
            out += " "   # 英単語境界のみスペース
        out += ln        # 日本語境界は区切りなし
    return out
```

> ⚠️ **句読点直後の英字も空白を補う（任意の改良）**: 上記スマート結合は左右が両方 ASCII 英数字のときだけ空白を挿入するため、`survey.` + `Your`（左が `.`=非英数字）のような soft-wrap で `survey.Your` と癒着する。`l in ".,:;!?\"')" and re.match(r"[A-Za-z0-9(\"']", r)` の条件を OR で足すと、文末句読点直後の英文も自然な空白で結合できる（日本語=非ASCIIには影響しない）。長文パッセージの可読性が上がる。

> ⚠️ **中黒「・」と語彙バレット「・」の区別（行構造を保持した結合）**: 解説に語彙リスト（`・word: 意味` が各行）が含まれる場合、smart_join で1行に連結してから `・`→`<br>・` の正規表現で改行を復元すると、**複合語内の中黒（`【方法・手段】` 等）まで改行して壊す**（`【方法<br>・手段】`）。語彙バレットの `・` は元々行頭、複合語の中黒 `・` は行中なので、join 後の文字列正規表現では両者を区別できない。**対策**: 結合後の正規表現でなく、**元の行構造を保持**して「行頭が `・` または `(A-C)`（選択肢別分析）の行だけ `<br>` を前置、それ以外はソフトラップ結合」する。行中の中黒は行頭でないため触れず、複合語が保たれる。リスニング対策書籍の選択肢別解説（`(A) 適切 / (B) 不適切…` + 語彙）で特に有効。

> 検出の兆候: 品質検証で `[a-z][A-Z][a-z]` パターン（小文字+大文字+小文字）の出現数を数えると、正常な camelCase（`JavaScript` `HttpOnly` `OpenFlow` `DomainKeys` や ASN.1/MIB の `atIndex` `atPhysAddress` 等）と区別がつく。実観測: スマート結合導入前 512 件 → 導入後 64 件に減り、残 64 件は全て正規の識別子だった。`Random Access Memory` 等の自然言語フレーズが癒着していたら本罠を疑う。

## EPUB 埋め込み画像（図・数式・選択肢画像）の Anki メディア取り込み

問題文・解説・**選択肢そのものが画像**として埋め込まれる書籍（IT資格の計算問題の数式選択肢、ネットワーク構成図、SQL結果表など）では、pandoc 変換後の Markdown に `![](image_rsrcXXX.jpg){.class_sYY}` の画像参照が残る。テキストだけ抽出すると問題が成立しないため、画像を Anki メディアコレクションに取り込んで `<img>` で表示する。

**手順**:
1. **画像参照の抽出と `<img>` 化**: `!\[[^\]]*\]\((image_rsrc\w+\.(?:jpg|png|gif))\)(?:\{[^}]*\})?` を `<img src="<prefix>\1" style="max-width:100%;">` に置換。`<prefix>` は他書籍との衝突を避けるためソース固有の接頭辞（例: `joho2026_`）を付ける。
2. **EPUB から画像を取り出す**: EPUB は zip なので `unzip -o "<book>.epub" "*.jpg" -d <tmpdir>` で画像のみ展開（`OEBPS/` 配下に出る）。
3. **AnkiConnect `storeMediaFile` で投入**: 各画像を base64 化し `storeMediaFile {"filename":"<prefix>image_rsrcXXX.jpg","data":"<base64>"}` で登録。MCP / AnkiConnect どちらでも可。
4. **エスケープ順序の罠**: 画像を `<img>` に変換した後で本文を `html.escape` すると `<img>` タグまでエスケープされる。**対策**: 画像参照を一旦 ASCII プレースホルダ（例: `@@IMG@@<filename>@@ENDIMG@@`）に退避 → 本文を escape → 最後にプレースホルダを `<img>` 実タグへ復元、の3段階にする。`\x00` 等の制御文字を `re.sub` の置換文字列に使うと `bad escape \x` で落ちるので ASCII トークンを使うこと。

> ⚠️ **`::: class_sXX` div フェンスに包まれた画像**: 画像はしばしば `::: class_sXX-0\n![](image_rsrcXXX.jpg){.class_sXX-1}\n:::` の div フェンスで囲まれる。選択肢が画像の場合 `<li>ア. ::: class_sN9-0![](...)...` のように div マーカーと画像が同一行に潰れて出ることがある。**対策**: クリーニングで (1) `^::+\s*class_s[A-Za-z0-9_-]+\s*$` の単独行フェンス除去、(2) 行頭以外に潰れた `::+\s*class_s[A-Za-z0-9_-]+\s*` のインライン除去、(3) 閉じフェンス `^::+\s*$` 除去、の3パターンを適用してから画像を `<img>` 化する。

## `{style="..."}` 形式EPUBの生テキスト境界検出

一部のEPUB（特にIPSpecialist等の英語問題集）では、pandocが `{.class_sXXX}` ではなく `{style="..."}` 形式のインライン属性を生成する。この形式では質問の先頭に特徴的なフォントサイズ指定パターンが付与される。

### 問題: 質問番号と質問内容中のリストが混同される

通常のパーサーは `N. ` パターンで質問番号を検出するが、質問文中に番号付きリスト（例: 要件リスト `1. 高可用性を確保 2. レイテンシを最小化`）が含まれる場合、これらを新しい質問と誤認する。

### 解決策: 生テキストでpandoc固有パターンを検出

pandoc変換後のクリーニング前テキストで、質問開始位置を固有のフォーマットパターンで特定し、ブロック単位で切り出す:

```python
import re

def find_question_boundaries(raw: str) -> list[tuple[int, int, int]]:
    """生テキストでpandoc固有パターンを使い質問境界を検出。

    Returns: list of (char_position, question_number, line_number)
    """
    pattern = re.compile(
        r'^(\d+)\.\s+\[\s+\]\{style="font:7\.0pt',
        re.MULTILINE
    )
    results = []
    for m in pattern.finditer(raw):
        q_num = int(m.group(1))
        if 1 <= q_num <= 200:
            line_num = raw[:m.start()].count('\n') + 1
            results.append((m.start(), q_num, line_num))
    return results
```

**処理フロー:**

1. **生テキスト**で `find_question_boundaries()` を実行して質問開始位置を特定
2. 各質問のテキストブロックを切り出す（次の質問開始位置まで）
3. ブロック単位で `clean_pandoc()` を適用してアーティファクトを除去
4. クリーニング済みブロックから問題文・選択肢・Answer・解説をパース

> ⚠️ パターンはEPUBソースごとに異なる。上記は一例であり、実際のEPUBの生テキストを確認して適切なパターンを特定すること。

### `{style="..."}` のクリーニング

```python
# {style="..."} 付きスパン: テキストのみ抽出
text = re.sub(r'\[([^\]]*)\]\{style="[^"]*"\}', r'\1', text)
# 単独の {style="..."}: 除去
text = re.sub(r'\{style="[^"]*"\}', '', text)
```

> ⚠️ **漢数字→アラビア変換時の「百」桁対応**: 法令穴埋め問題集など条文番号を扱う場合、`第百五十二条` のような3桁漢数字を `第12条` に誤変換する単純な kanji_to_arabic 実装が頻発する。「十」分割だけのナイーブ実装は `百五|十|二` を `1*10+2=12` と誤計算する。**対策**: `if '百' in s` 分岐を最優先で実装し、`hundreds*100 + tens*10 + ones` 構成にする。`百`単独=100、`千`単独=1000のエッジケースも明示処理する。実観測: 健康保険法の費用負担章で `第百五十二条` 関連カードが 法第12条 のカードと混同され、解答マッチに失敗した。

> ⚠️ **本文に同マーカーが複数登場するケースのパネル誤判定**: 条文穴埋めで `毎年【★】における...その年の【★】において` のように同一マーカーが本文に2回登場し、その後 `【★】\n\nA：...` 選択肢パネルが続く構造で、「マーカー2回目=パネル位置」というナイーブ実装は本文の2回目を panel_start と誤認する。**対策**: panel 検出は「マーカー単独行 (`\n\n【M】\n\n[A-E]:`)」を正規表現 `re.escape('【M】') + r'(?=\s*\n\s*\n?\s*[A-E][：:])'` で先行検索し、それが無ければマーカー最終出現位置をフォールバック採用する。実観測: 定時改定の最高等級超過時の改定条文（★が body に2回、panel に1回）で発生し、★の選択肢パネルが取得できず word 欠落カードとなった。

## Kindle KFX形式EPUBの直接抽出と前処理（pandoc回避）

画像主体のKindle KFX由来EPUB（各ページに画像/PDFが冗長に埋め込まれ数百MB〜GB級に肥大化し、本文テキストは xhtml 内に存在する）では、pandoc変換すると各文字が `[字]{style="..."}` のように原子化・断片化して使い物にならない。`zipfile` で xhtml を直接読み、タグ除去 + `html.unescape` + NFKC 正規化する**直接抽出**が確実（巨大zipでも `zipfile` は中央ディレクトリ参照で本文 xhtml だけを軽量に個別読み出しできる）。以下はその際に固有の罠と対策。

> ⚠️ **spine順での再抽出（ファイル名順は読み順を壊す）**: KFXのxhtmlは `c0` `c101` `c108` `c10F` … のような内部IDで命名され、**ファイル名の文字列ソート順は本文の読み順と一致しない**。ファイル名順に連結すると問題ブロックの順序がスクランブルされ、特にチャンク（xhtmlファイル）境界が「問題本文と解答の間」に落ちた問題で、**解答が別チャンクへ飛んで誤った問題に紐づく**（straddle 誤紐付け）。各問が `問N` のような明示番号を持っていても、位置ベースのブロック分割では straddle 問題の解答欠落が残る。**対策**: `META-INF/container.xml` → OPF（`<rootfile full-path>`）→ OPFの `<spine><itemref idref>` → `<manifest>` の `href` を辿り、**spine順（正しい読み順）でxhtmlを連結**してから抽出する。抽出後は (1) 問題番号の連番性（`1..N` 欠番ゼロ）、(2) 解答/解説マーカーの欠落件数 を必ず機械検証する（実観測: ファイル名順連結では先頭が問359/問題385から始まり解答欠落が出たが、spine順で欠番0・straddle解消）。

> ⚠️ **直接抽出テキストの「マーカー内部分断」と clean-first 前処理**: 直接抽出したテキストには、(1) xhtmlファイルの内部ID（`c108` `cK4` 等）が**素の独立行**として、(2) 紙面のページ番号が**数字単独行**として混入する。これらが問題境界だけでなく**マーカー文字列の内部に割り込む**ことがある（実観測: `解 \n\ncK4\n\n75\n説:` のように「解説」が、`解答: \n\ncXXX\n\n71\n◯` のように verdict が marker から分断される）。`解\s*説` のような `\s*` 許容 regex は間に挟まる `cK4`/`75` という**文字列**を越えられず検出失敗する。**対策（clean-first）**: マーカー検索の**前に**、行単位で `^c[0-9A-Za-z]+$`（xhtml内部ID行）と `^\d{1,4}$`（ページ番号行）を除去し、ソフトラップ結合（CJK境界は無区切り・ASCII英数境界のみ半角空白）してから `解答:`/`解説:` 等のマーカーを検索する。分断されたマーカーが物理的に再結合する。OCR/組版由来テキスト全般に効く前処理順序（「掃除してから解析」）。

> ⚠️ **解答マーカー内の番号は誤記しうる（番号非依存抽出）**: `問N` 形式の問題で、解答マーカー `【解答N】` の埋め込み番号が組版/OCRミスで**ズレる**ことがある（実観測: `【問題833】…どちらの指定数量が大きいか。【解答834】モーター油…` のように +1 ズレ）。解答抽出時に問題番号との一致を要求すると、この1問の解答を取りこぼし needs_fix に落ちる。**対策**: ブロックは問題マーカー位置で区切ったうえで、解答は `【\s*解答\s*\d*\s*】`（**番号を無視**）で「ブロック内の最初の解答マーカー」として抽出する。番号はブロック分割（問題マーカー）側で既に確定しているため、解答マーカーの番号は照合に使わない。

> ⚠️ **混在型の検出と qtype 振り分け（○×主体の問題集でも選択肢/短答が混在する）**: ○×主体を謳う問題集でも、末尾の総仕上げ・ランダム模試セクションに `解答:(3)` 形式の選択肢型や、`解答・解説:` 結合マーカー（解答と解説が一体化）が少数混入することがある。また○×主体の隣で短答/計算型（`指定数量の倍数はいくらか` → `0.5倍`）が混在する書籍もある。**対策**: 解答マーカー直後を判定し、(a) 先頭が ○×系文字 → `truefalse`、(b) `(\d)` → 選択肢型（`basic` で front に選択肢保持・back に正解＋解説）、(c) それ以外（数値・記述）→ 短答 `basic`（back に解答＋解説）に振り分ける。`解答:` が無く `解答・解説:` がある場合は結合マーカーとして basic 救済する。判定不能のみ needs_fix とし、握り潰さない。

## フォーマット変換ルール

### 選択肢のリスト化

**形式:**

```html
<ol style="list-style-type: none; padding-left: 0;">
  <li>A. 選択肢テキスト</li>
  <li>B. 選択肢テキスト</li>
</ol>
```

- `list-style-type: none` でデフォルト番号を非表示
- `A.`, `B.` 等のプレフィックスを `<li>` 内に保持
- 全選択肢を**1つの `<ol>` にまとめる**（選択肢ごとに個別の `<ol>` を作らない）
- 選択肢間の空行があっても同一 `<ol>` に含める

**検出パターン:**

| パターン | 例 | 変換 |
|---------|-----|------|
| 半角大文字 + ドット | `A. 選択肢`, `B. 選択肢` | そのまま使用 |
| 半角大文字 + ドット（改行分離） | `A.\n選択肢テキスト` | 1行に結合 |
| バレットポイント（`-`） | `- 選択肢テキスト` | A./B./C./D./E. を自動付与 |

### Frontフィールドの構成

```
[問題文のみ（問題番号ヘッダーは含めない）]
<br><br>
<ol style="list-style-type: none; padding-left: 0;">
  <li>A. ...</li>
  ...
</ol>
```

- **問題番号（「問 N」「Question N」等）はFrontに含めない**
- 問題の内容テキストと選択肢のみ

### Backフィールドの構成

**単一正解の場合:**

```
<b>正解:</b>
A. [正解テキスト]
```

**複数正解の場合（「二つ選択」等）:**

```
<b>正解:</b>
A. [正解テキスト1]
B. [正解テキスト2]
```

**解説セクション:**

```
<b>解説:</b>（太字ラベル）
[解説テキスト]

<b>不正解の選択肢の解説:</b>（太字ラベル）
<b>C:</b> [不正解解説]（各選択肢を太字ラベル付きで）
```

> ⚠️ **解説本文冒頭の選択肢テキスト重複除去**: 解答セクションから各選択肢の解説を抽出する際、`ウ × 刑法\n刑法とは、犯罪と刑罰について規定するすべての法令の総称です。〜` のように、**解答行に書かれた選択肢テキスト（「刑法」）が解説本文の先頭1行目にも繰り返し書かれている**ケースが頻発する（OCR/pandoc 由来の重複）。これをそのまま Back HTML 化すると `正解: ウ. 刑法\n\n解説:\n刑法\n刑法とは...` のように冗長な表示になる。**対策**: 解説本文の最初の行が選択肢テキストと完全一致する場合、その行を削除する後処理を実装する:
>
> ```python
> def trim_leading_choice_text(rest_text: str, choice_text: str) -> str:
>     if not choice_text:
>         return rest_text
>     lines_local = rest_text.split("\n", 1)
>     if len(lines_local) >= 1 and lines_local[0].strip() == choice_text.strip():
>         return lines_local[1].strip() if len(lines_local) > 1 else ""
>     return rest_text
> ```
>
> 検出の兆候: Back の解説部分が「[選択肢テキスト]\n[選択肢テキスト]とは、〜」のような重複表示になる。資格試験対策書のように選択肢が短い名詞句（例: 「刑法」「実用新案法」）の場合に顕著。

### ○×問題（True/False型）の構成

ソースに選択肢（A./B./C./D.）が存在せず、解答が `○`/`✕`/`×` の判定マーカーで示される場合は以下のフォーマットを使用する。

**Frontフィールド:**

```html
[問題文（平叙文）]<br><br>
<span style="color:#c00;font-weight:bold;">⭐ 重要</span><br><br>   <!-- 重要!マーカー付きのみ -->
<i>（○か✕で答えよ）</i>
```

- 問題番号ヘッダー（`**問題N**`）と難易度マーカー（`★+`）はFrontには**含めない**（タグとして保持）
- `**重要!**` バッジ付きの問題のみ赤色強調表示を挿入
- 末尾の判定指示 `（○か✕で答えよ）` で受験者の解答行動を一意化

**Backフィールド:**

```html
<div style="font-size:2em;font-weight:bold;text-align:center;">○</div>
<br>
<b>解説:</b><br>
[解説テキスト（複数段落は <br><br> で分離）]
```

- 判定結果（`○` or `✕`）を大きく中央表示
- ソースの `×` (U+00D7), `〇` (U+3007) を `✕` (U+2715), `○` (U+25CB) に正規化
- 解説に条文参照（`民法90条`、`憲法29条` 等）がある場合はそのまま保持

> ⚠️ **判定マーカー欠落カードの処理**: 紙面で別カラム表示される `○`/`✕` 判定マーカーがOCR/pandoc変換時に脱落し、解説本文だけが残るケースが資格試験書籍では一定割合発生する。verdict 推論は誤判定リスクが高いため、欠落カードは空欄推論せず、Back先頭に注記を表示する:
>
> ```html
> <div style="font-size:1.2em;color:#888;text-align:center;">（原本に○×マーカー欠落 — 解説で判定）</div>
> <br>
> <b>解説:</b><br>
> [解説テキスト]
> ```
>
> タグに `判定マーカー欠落` を付与しておくと、ユーザーが後から手動補完しやすい。

> ⚠️ **選択肢判定行 `ア ☓` の Back HTML 色分け化**: 解説本文中の選択肢判定行は `[ア]{.class_s1GV}[　☓]{.class_s1GZ}` のように **2つの inline span に分割**されてpandoc出力される。クリーニング後は `ア ☓` のような短い独立段落として残るが、これを Back の HTML 上で `<b><span style="color:#c00;">ア ✕:</span></b>`（不正解=赤）/ `<b><span style="color:#080;">ウ ○:</span></b>`（正解=緑）のような視覚的に強調された見出しに変換すると、選択肢ごとの判定理由が一目で把握できる Back UI が実現する。**実装**: クリーニング後の段落単位 split で `^([アイウエオ])\s*[　 ]*([○✕×☓〇◯])\s*$` パターンを判定見出しとして検出し、HTML テンプレートで色分けする。Unicode 正規化（`☓` U+2613 / `×` U+00D7 → `✕` U+2715、`〇` U+3007 / `◯` U+25EF → `○` U+25CB）も併用すること。

> ⚠️ **VLM OCR の LaTeX 風判定マーカー（`$\times$` 等）は Unicode 変種ではない**: 画像主体ソースを VLM でOCRすると、判定マーカー `×` が **LaTeX 数式表記 `$\times$`**（稀に `$\circ$`）として出力されることがある。これは `× ☓ 〇 ◯` のような Unicode 変種ではなく `$`+バックスラッシュ+コマンド名の**文字列**なので、Unicode 正規化マップ（`☓`→`✕` 等）では拾えない。**対策**: verdict 抽出の**前に** `text.replace(r"$\times$", "×")`（必要なら `$\\times$` も）で素の `×` に潰してから `[○×✕〇◯]` で抽出する。verdict 正規化マップに足すより前処理で潰す方が、解説本文中の同記法も同時に救える。資格試験の○×問題集（画像主体EPUBをVLM逐次OCR）で複数問が `$\times$` 表記となり観測。

**タグ生成:**

| メタ情報 | タグ形式 |
|---------|---------|
| 章 | `章::第N章-{章タイトル}` |
| 節 | `節::第N節-{節タイトル}` |
| テーマ | `テーマ::{テーマタイトル}` |
| 難易度 | `難易度::★`, `難易度::★★`, `難易度::★★★` |
| 重要マーカー | `重要` |

> ⚠️ Ankiの階層タグは `::` 区切り。空白は `-` に置換し、`:` `/` 等は除去する。

> ⚠️ **過去問出典コード(`RX-Y-Z` / `HX-Y-Z`)のタグ化と Knowledge Area 埋め込み**: 国家資格対策の「過去問一問一答」シリーズでは、問題マーカーに `R5-5-A`(令和5年問5A)・`H26-1-D`(平成26年問1D)のような出典コードが付与されており、これを Anki タグ + Knowledge Area に保存することで「直近5年だけ復習」「平成の過去問のみ抽出」のような学習動線が作れる。**実装**: (1) タグに `出典:R5-5-A`・`年度:令和5年`・`年度:平成26年` を付与(`R` → 令和、`H` → 平成、`S` → 昭和に正規化)、(2) Knowledge Area に `過去問一問一答 [科目名] 問1 (R5-5-A)` のように出典コードを括弧書きで含める、(3) `findNotes "tag:年度:令和5年"` で年度フィルタが直接効くようになる。出典コードがない書籍では空文字でフォールバックする。

> ⚠️ **「資格試験(ランダム)」ノートタイプの Knowledge Area フィールド章メタデータ慣習**: 既存の `資格試験(ランダム)` カスタムノートタイプ（Question / Answer / Knowledge Area の3フィールド構成）では、`Knowledge Area` フィールドに `第N章 [章タイトル]` 形式の短文を埋め込む慣習が既存カード群で確立されており、Anki カードテンプレート側で**非表示の検索用メタフィールド**として機能している。これにより `"deck:検定試験::XXX" "Knowledge Area:第3章*"` のような Anki ブラウザ検索が可能になり、章単位の復習動線が成立する。**実装**: (1) コンテンツ構造分析（Step 3）で章見出し `# 第N章 [タイトル]` を抽出して章番号→タイトル辞書を構築、(2) 各問題が属する章を直近の章見出しから決定、(3) `Knowledge Area = "第{N}章 {title}"` をフィールドに設定。タグ `章::第N章-{title}` との二重管理になるが、Anki ブラウザの検索 UI で章別フィルタが使えるメリットが大きい。問題自体の知識分類（著作権・特許・商標等）も入れたい場合は `Knowledge Area = "第{N}章 {title} | {topic}"` のように `|` 区切りで併記する。新規カード投入時は **既存デッキの先頭1〜2件をプレビューして Knowledge Area フォーマットを継承**すること（同一デッキ内でフォーマット不一致が起こると検索動線が破綻する）。

> ⚠️ **章タグは「検証可能なときだけ」付与する（健全性ゲート）**: 章見出しが本文にインライン出現せず**巻頭目次（TOC）に集約**されている書籍では、位置ベースのフォワードフィル（直近の章マーカーを各問へ割り当てる）が機能せず、全問が最後のTOCエントリ（多くは部レベル）を拾ってしまう。代替として目次の問題数 `(N問)` の累積和から番号レンジ（章k = 問[cum+1 .. cum+count]）を組めるが、**目次のカウントは内部矛盾しうる**（実観測: ある部が「計350問」表記なのに配下章の合計が500問）。さらに章タイトル自体が括弧を含むと `第N章:タイトル(N問)` の `(N問)` 抽出 regex が**タイトル内括弧で早期マッチ**して章を取りこぼす（例: `第9章:酸と塩基(アルカリ)・pH(100問)` で `(アルカリ)` を問数と誤認）。**対策（健全性ゲート）**: (1) 章タイトル抽出は括弧を許容し `[(（]\s*\d+\s*問` を末尾アンカーにする（タイトルは `[^\n【]+?`）、(2) `Σ章問数 == 総問数` を検証し、**一致しなければ章タグを採用せず破棄**する（誤タグは無タグより有害）。章境界が本文インラインに出る書籍では、各章マーカー直後の問題番号から連続レンジ（例: 1 / 401 / 701 始まり）を導出し、`1..総数` を**連続被覆**するか検証してから採用する。基本タグ（資格名・問題集識別子）と問題種別タグ（`○×型`/`選択肢型`/`短答型`）は検証不要で常に付与してよい。

## 一括投入の効率化

### AnkiConnect バッチAPI

大量のカード（50件以上）を作成する場合、MCP `add_note` の個別呼び出しではなく、AnkiConnect の `addNotes`（複数形）バッチAPIを直接使用する。

| 方法 | 200件作成時の呼び出し回数 | 推奨場面 |
|------|------------------------|---------|
| MCP `add_note` 個別 | 200回 | 少数（10件以下） |
| AnkiConnect `addNotes` バッチ | 4回（50件×4バッチ） | 大量（50件以上） |

**バッチサイズ**: 50件を推奨。

**⚠️ `createDeck` 必須**: `addNotes` 実行前に必ず `createDeck` でデッキの存在を保証すること。デッキが存在しない場合、`addNotes` は `"deck was not found"` エラーで全件失敗する。`createDeck` は既存デッキに対しても安全に呼び出せる（冪等）。

**エラーハンドリング**: `addNotes` の戻り値は配列。各要素が `null` の場合は重複またはエラーでスキップされたことを意味する。バッチ全体が失敗した場合は `error` フィールドにエラーメッセージが返る（`result` は `null`）。両方をチェックすること。

### 再作成時のワークフロー

フォーマット修正等でカードを再作成する場合:

1. `findNotes` でタグ検索（例: `tag:DevOps-Engineer-模擬問題集-200問`）
2. `deleteNotes` で旧カードを一括削除
3. **`createDeck` でデッキ存在を保証**（⚠️ 全カード削除後に空デッキが自動削除されるため必須）
4. `addNotes` で新カードを一括作成

### 翻訳時の処理

100問以上の外国語コンテンツを翻訳する場合も、LM Studioスクリプトを逐次呼び出しで処理する（ローカルLLMのため高速）:

1. 全Q&AペアをJSON（原文のみ）にパース
2. 各Q&AペアについてLM Studioスクリプトを呼び出して翻訳する（問題文、各選択肢、解答、解説を個別に翻訳）

   ```bash
   python3 ${CLAUDE_PLUGIN_ROOT}/skills/converting-content/scripts/lmstudio-translate.py translate --model <選択済みモデル> --text "翻訳対象テキスト"
   ```

3. 翻訳済みJSONからAnkiConnect `addNotes` で一括投入

### 折りたたみ原文（`<details>` タグ）

英語等の外国語ソースの場合、原文を `<details>` タグで折りたたみ表示する。

> ⚠️ **`<details>` 内では `<ol><li>` を使用しないこと。** Ankiのカスタムノートタイプ（例: `資格試験(ランダム)`）はjQueryで `$('.question ol').find("li")` のように全`<li>`を取得してランダム化するテンプレートを持つことがある。`<details>` 内に `<ol><li>` があると、メインの選択肢と原文の選択肢が統合され、選択肢が倍増して表示される。原文の選択肢は `<br>` 区切りのプレーンテキストで記述する。

```html
A. [Original choice A]<br>
B. [Original choice B]<br>
C. [Original choice C]<br>
D. [Original choice D]
</details>
```

- デフォルトで閉じた状態（collapsed）
- タップ/クリックで展開
- `<summary>` に言語名を表示（`📄 原文（English）`）
- `<br>` を `<summary>` 直後に入れて余白を確保

## Answer推論（空回答フィールドからの正解推定）

一部のEPUBでは `Answer:` フィールドが空のまま出版されている問題がある。この場合、解説文（Explanation）から正解を推論する。

### 推論パターン

```python
def infer_answer_from_explanation(explanation: str) -> str | None:
    """解説文から正解レターを推論する。"""
    patterns = [
        r'Option\s+([A-I])\s+is\s+(?:correct|the\s+(?:correct|right)\s+answer)',
        r'Options?\s+([A-I])\s+and\s+([A-I])\s+are\s+correct',
        r'^([A-I])\s+is\s+correct',
        r'Option\s+([A-I])\b.*?correct',
    ]
    for pat in patterns:
        m = re.search(pat, explanation, re.IGNORECASE)
        if m:
            if m.lastindex == 2:
                return f"{m.group(1)} and {m.group(2)}"
            return m.group(1)
    return None
```

**使用タイミング**: `Answer:` が空文字列または `?` の場合にのみ適用。推論できなかった場合は `?` を設定し、WARNINGログを出力する。

### 複数Answer形式への対応

英語EPUBでは以下のAnswer形式が混在する:

| 形式 | 例 | 検出パターン |
|------|-----|------------|
| コロン付き | `Answer: B` | `^Answer:\s*(.+)` |
| コロンなし | `Answer B` | `^Answer\s+([A-I])` |
| 空欄 | `Answer:` | `^Answer:\s*$` → 推論実行 |
| 複数回答 | `Answer: A and B` | `([A-I])\s+and\s+([A-I])` |

> ⚠️ **不完全カードの可視化投入戦略**: パーサーで解答 letter または word が完全取得できないカード（ソース原本での解答省略・タイトル位置の特殊穴・サブ法令未対応など）を**完全に捨てる**と再投入時に当該箇所が永久に空白となる。**対策**: 不完全カードにも `_要手修正` タグを付け、Back に `<div style="background:#fff8e1;border:1px solid #ffb300;padding:0.5em;...">⚠️ 要手修正: ソース原本で解答省略 or パーサー未対応の特殊形式</div>` の警告 div を埋め込み、letter/word の欠落フィールドは `?` / `（要手入力）` に置換して Anki 上で目視・手修正できるようにする。タグで `tag:_要手修正` 絞り込みも可能。実観測: 法令穴埋め 155枚中 8枚 + 別法令 228枚中 16枚を `_要手修正` 付きで混在投入し、Anki 上で手修正できる UX を確立した。

> ⚠️ **問題本文そのものが OCR で完全消失した場合の「プレースホルダー投入」vs「欠番」使い分け**: 上記の letter/word 単位欠落（`_要手修正` 戦略）と異なり、**問題本文ごと OCR で消えた**ケースは別軸の判断が必要。判別基準: (1) **解説テキストが残っている** → プレースホルダー（`※ 問題文は OCR 欠落のため再現できません`）+ 解説のみで投入 + `_要手修正` タグ付与。Anki 上で解説から問題文を逆引きして手修正可能。(2) **解説も含めて連続欠落** → 該当問題番号は**欠番のまま投入せず**、欠落報告にリスト化（書籍を別途参照して手投入する経路を残す）。理由: 解説さえあれば「設問の意図」が推定可能なため学習資産として残せる一方、解説まで消えた問題は手修正のヒントがゼロなので投入する価値が極端に下がる。実観測: 一書籍内の一問一答セクションで「解説含め4問連続消失」は欠番、「分野別1問だけ問題文消失（解説残存）」はプレースホルダー投入、と使い分けて対応した。

> ⚠️ **数字整合性監査の初動手順**: ユーザーが「件数の違和感」を訴えた場合の初動チェック手順:
> 1. 既存デッキのサブデッキ別件数確認: `findNotes deck:"デッキ名" -deck:"デッキ名::*"` で「その他」直下カテゴリの有無を検出
> 2. ソースMDの全問題マーカー網羅カウント: `grep -cE "^(\*\*)?第\s*[0-9]+\s*問|^(\*\*)?問\s*[0-9]+" <file>` でbrute-force grep
> 3. 数学的整合性確認: `前年版件数 - 既存判定件数 = 削減件数` が改訂版として妥当な範囲か検証
> 4. サブデッキ構造の誤配置確認: 全件が期待サブデッキに入っているか `findNotes deck:"対象サブデッキ名"` で確認

> ⚠️ **改訂版問題の類似度分類と新規/既存判定の目安**: `difflib.SequenceMatcher` 等で前年版との差分を判定する際、以下の類似度帯域に注意する:
>
> | 類似度 | 改訂パターン | 推奨判定 |
> |--------|------------|---------|
> | 0.90以上 | OCR誤字差・前置きヘッダ差のみ（本質同一） | **既存扱い** |
> | 0.75〜0.90 | 出題回情報差・空白差（本質同一が大半） | **既存扱い** |
> | 0.55〜0.75 | 小問1個差し替え・問題形式変更（別問題が混在） | **新規扱いが安全** |
> | 0.55未満 | 別問題 | **新規** |
>
> **分水嶺は0.90**: 0.90以上を既存、未満を新規とするとOCR誤字を別問題と誤判定しにくい。

> ⚠️ **改訂版書籍の章単位問題数変動**: 同一資格試験対策書籍でも前年版と当年版で章単位の問題数が増減することがある（例: 前年版 第3章 12問 → 当年版 10問 で -2問削減）。差分判定スクリプトで大幅に「既存扱い」件数が増えた場合、OCR抽出漏れと混同しやすい。**判別手順**: `grep -c "^第\s*\d\+\s*問\|^\*\*問" <markdown_file>` でチャプターエリアの問題マーカー数を実数カウントし、「期待件数 - 実カウント = 削減件数」が書籍改訂として妥当な範囲（数問程度）か確認する。削減件数が多すぎる場合はパーサーの検出漏れを疑う。

> ⚠️ **同章内で「質問側問題番号」と「解答側問題番号」がズレる書籍誤植**: 紙面校正漏れで、ある問題の質問見出しが `第9問` と書かれているのに、解答マーカー側は `問8 正解：①` を指している（書籍内の問題番号が誤植）という罠が観測される。質問側番号を真として join すると、解答が 1 問ズレで全問対応する致命的なリグレッションになる。**対策**: 質問・解答どちらが「より構造化されているか」を判定し、**構造化されている方（通常は解答マーカー側）を正規化アンカー**として採用する。具体的には: (1) 解答マーカーの番号系列が `1, 2, 3, ..., N` と連続している場合 → 解答側を真とする、(2) 質問見出しの番号系列に飛び（例: `1, 2, 3, 4, 5, 6, 7, 9, 10`）がある場合 → ギャップ位置の質問番号を「直前番号 + 1」で rewrite する。検出手段: 質問側番号集合と解答側番号集合の差集合 (`set(q_nums) - set(a_nums)` / `set(a_nums) - set(q_nums)`) を比較し、両方が空でない場合に誤植アラートを出す。実観測: 一書籍の特定章で質問見出し `第9問` と解答マーカー `問8 正解` の不整合を発見し、解答側アンカーで正常化した。

> ⚠️ **事例形式問題の「以上を前提として」リード文境界検出**: 実技試験等の事例形式問題では、「事例N (リード文)」+「以上を前提として、問M〜問Lに答えなさい。」+「問M Check! ...」+「問L Check! ...」の構造で複数問が**同一事例リード文を共有**する。各問題ブロックでリード文を更新する単純実装ではブロック2個目以降でリード文が消失する。**対策**: 状態変数 `current_lead` を保持し、Qブロック内の `問N` 開始前テキストに `"以上を前提として"` が含まれている場合のみ更新、そうでなければ前回値を継承する。これにより事例1（問1-4）は同一リード文、事例2（問5-8）で新リード文に切り替わる動作が成立する。

> ⚠️ **試験書籍「(はじめに)」ボイラープレート除去**: 各試験回の冒頭に「すべての問題文の条件設定において、特に断りのない限り、他に特殊な事情がないものとします。」「特に日時の指定のない限り、YYYY年MM月DD日現在で施行されている法律等に基づいて解答しなさい。」「解答は、選択肢ア〜エ（or ア〜ウ）の中から1つ選びなさい。」の3パターンが定型文として挿入される。これらは事例リード文に混入すると認知ノイズになる。**対策**: 事例リード文抽出時に以下のキーワードを含む行を除外する: `"すべての問題文の条件設定"`, `"(はじめに)"`, `"また、各問題の選択肢における条件設定"`, `"特に日時の指定のない限り"`, `"解答は、選択肢"`, `"解答は，選択肢"`。同種のボイラープレートは各章末問題でも繰り返されるので、ファイル全体スイープで除去する設計が安全。

> ⚠️ **A→Q ブロック遷移バグ（解答→次問題への遷移漏れ）**: 「Q ブロック → A ブロック → Q ブロック → A ブロック」と交互に出現する構造を「ブロックタイプ切り替わりタイミング」で分割するパーサーで、**A ブロックから次の Q ブロックへの遷移条件を入れ忘れる**バグが頻発する。`if cur["type"] == "A" and 問題開始パターン.match(line)` の遷移ロジックが無いと、2問目の Q 内容が「先頭の A ブロック」に永続的に追記され続け、結果として **2問目以降の問題ステートメントが全て欠落**する。**対策**: ブロック分割ロジックに必ず以下の遷移条件を入れる:
> ```python
> elif cur["type"] == "A" and q_start_re.match(line.strip()):
>     blocks.append(cur)
>     cur = {"type": "Q", "content": [line], ...}
> ```
> 実観測: 修正前は30問中1-2問しか抽出できず、修正後120問完全取得。デバッグの目印は「ブロック数が解答数の半分以下」状況。

> ⚠️ **事例形式の解答ヘッダーbold混在3パターン**: 同一書籍内・同一試験回内でも以下の3形式が混在する:
>
> | パターン | 例 |
> |---------|-----|
> | プレーン | `問1 問2 解答・解説 問1 正解：× 問2 正解：ウ` |
> | 全bold | `**問3 問4 解答・解説 問3正解:× 問4正解:ア**` |
> | 分散bold | `**問5 問6 解答・解説**  **問5正解：〇**  **問6正解：ウ**` |
>
> **対策**: 解答ヘッダーパース前に `s.replace("**", "")` で `**` を一括除去してから問番号と正解レターを `re.findall(r"問\s*(\d+)", s)` / `re.findall(r"正解\s*[：:]\s*([アイウエ〇○×Ｘ])", s)` で抽出する。問番号と正解レターは出現順で対応付け（最初の問N → 最初の正解、2番目の問N → 2番目の正解）し、`zip(seen, corrects)` で辞書化する。実技1事例で問1問2/問3問4/問5問6/問7問8の4ペアあり、3バリアントが混在するケースが多い。

> ⚠️ **問題ヘッダーのMarkdown装飾混在パターン**: 問題開始マーカーも書籍内で複数の装飾が混在する:
>
> | 形式 | 例 |
> |------|-----|
> | プレーン | `問1 Check! □□□` または `問1\nCheck! □ □ □` |
> | Bold | `**問1** Check! □ □ □` または `**問1**` |
> | Heading | `### 問1` または `### 問1 Check! □□□` |
> | 混在 | `**問19** Check!` または `## 問21解答・解説 正解：ウ` |
>
> **対策**: 問題マーカー regex は `^[*#\s]*問(\d+)[*]*\s*(Check!.*)?\s*$` のように行頭の `*`/`#`/空白、数字後の `*` 終了タグ、`Check!` の有無すべてを許容する。解答マーカー regex も同様に `^[*#\s]*問(\d+)\s*解答・解説[*\s]*正解\s*[：:]\s*([アイウエ])[*]*\s*$` で全装飾混在に対応する。

> ⚠️ **2行分離した解答ヘッダーの結合前処理**: 同一書籍内で `**問N解答・解説**\n\n正解：X`（解答・解説と正解が別行に分離）のパターンが混在することがある。問題が長い章で印刷レイアウト調整のため行分割されている。**対策**: パース前に「`解答・解説` を含む行で `正解` を含まない場合、次の4行以内に `正解：X` 単独行があれば結合」の preprocessing を実装する:
> ```python
> if "解答・解説" in cur and "正解" not in cur:
>     for offset in [1, 2, 3, 4]:
>         m = re.match(r"^[*]*\s*問?\s*\d*\s*正解\s*[：:]\s*([アイウエ〇○×Ｘ])[*]*\s*$", lines[i+offset].strip())
>         if m:
>             merged = cur.rstrip("*").strip() + " 正解：" + m.group(1)
>             # 結合して次のループへ
> ```

> ⚠️ **同シリーズ書籍でも構造変異する前提でパーサーを書く**: 同じ出版社・同じシリーズ・同じ著者の書籍でも、3級用と2級用、または旧版と新版で「pandoc/PDF変換後のMarkdown構造が大きく異なる」ことが頻繁にある。例: (a) 3級は `ア ×\n[解説]` のプレーン、2級は `*   **ア ×**\n    [解説]` のmarkdown list / (b) 同シリーズの旧版は `問N` プレーンマーカー、新版は `**問N**` bold装飾 / (c) 1冊目は問題と解答が交互配置、2冊目は前半後半分離型。**対策**: ファイルごとに必ず最初の Step 3（コンテンツ構造分析）を実施し、過去のパーサーをそのまま再利用しない前提で**サンプル目視確認**を行う。同名スクリプトに「ファイルA用」「ファイルB用」とコピーを作る方が、ジェネリックパーサーで全対応を試みるより堅牢。実観測: 同一執筆者の3級と2級で解説formatが全く異なり、3級パーサーで2級が48問中5問欠落した。

> ⚠️ **解答ブロック内最終フィールドの終端を `\Z` でなく次マーカー手前にする**: 同一セクション型で「①解答→②ポイント→③解説→④プラスアルファ→⑤根拠条文等」のような5段フィールド構造を持つ場合、最終フィールド(`⑤ 根拠条文等：[本文]`)の終端を `re.search(r"⑤\s*根拠条文等?：?\s*\n+(.+?)\Z", section, re.DOTALL)` のように `\Z` (ブロック末尾)で取ると、**次の問の見出しブロック開始マーカー `::: {#cR.xhtml... .heading_sX}` まで吸い込んでしまう**(各問の `split_problems` で次マーカーの「前」までを1ブロックとして切っているが、その先頭の見出しブロック残骸が末尾に残るため)。**対策**: 終端を次マーカーの手前で明示停止する: `r"⑤\s*根拠条文等?：?\s*\n+(.+?)(?=\n+:::|\n+〔問|\Z)"`。検出の兆候: Back カードの末尾(出典 div の直前)に `::: {#cXXX.xhtml_id__XXX_X_ .heading_sX}` のような pandoc 残骸が混入する。
