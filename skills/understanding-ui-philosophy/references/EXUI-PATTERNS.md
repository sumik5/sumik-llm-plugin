# exUI パターン詳細リファレンス

## exUIとは

渡邊恵太が提唱する概念。「**ex**ternal **UI**（外在化されたUI）」。

従来の製品: ハードウェアにボタン・LCD等のUIが組み込まれている
exUI: UIをハードウェアから分離し、スマートフォン等の汎用デバイスで操作する

**核心**: UIはハードウェアに属する必要はない。UIを「外に出す」ことで、製品の可能性が広がる。

---

## exUIの5つのメリット

### メリット1: スタイリングの自由度向上

ハードウェアからボタン・LCD・ダイヤル等の物理UIを排除すると、プロダクトの形状設計が自由になる。

**例**: ボタンが一つもない白いキューブ型デバイス。全操作はスマートフォンアプリで行う。
→ プロダクトがインテリアオブジェクトとして機能できる。機能とデザインを完全に分離。

### メリット2: パーソナライズの容易化

アプリのUIを変えるだけでユーザーごとに異なる体験を提供できる。

**パーソナライズの範囲**:
- 表示言語（多言語対応がソフトウェアだけで完結）
- フォントサイズ（アクセシビリティ設定）
- 機能フィルタリング（初心者モード・上級者モード）
- テーマ・カラースキーム
- 操作フロー・ショートカットのカスタマイズ

### メリット3: A/Bテストの適用

Web・アプリでは当たり前のA/Bテストを、ハードウェア製品にも適用できる。

**ハードウェア製品へのA/Bテスト適用例**:
- UIレイアウトAとBを異なるユーザーグループに配信
- 機能の提示順序・デフォルト設定のテスト
- オンボーディングフローの最適化

### メリット4: 継続的UIアップデート

ハードウェアを買い替えなくても、UIの改善を継続的に届けられる。

**従来の問題**: 家電を購入すると、UIは固定。5年後に「使いにくい」と感じても変えられない。
**exUI の解決**: アプリのアップデートでUIを改善。ユーザーの要望に素早く対応。

### メリット5: スマートデバイスとの連携

既存のスマートフォン・タブレットのセンサー・カメラ・音声認識・AR等を活用できる。

## メタハードウェア

同じハードウェアがUIを変えることで「別の製品」として機能する。

**思考実験**: 自動販売機に外部UIデバイス（タブレット等）を接続する。
- UIセット1: 飲料選択インターフェイス → 自動販売機
- UIセット2: 金融取引インターフェイス → ATM
- UIセット3: ゲームインターフェイス → スロットマシン
- UIセット4: 在庫管理インターフェイス → 倉庫管理端末

**メタハードウェア的発想**: 「このハードウェアで何ができるか」ではなく「このハードウェアに何をUI化するか」で製品を定義する。

### メタハードウェア設計の応用

| 業界 | 従来設計 | メタハードウェア的設計 |
|------|---------|------------------|
| 家電 | 機能別に専用機器 | 汎用ハードウェア + 機能別アプリ |
| 医療機器 | 診断機能が固定 | 診断プロトコルをソフトウェア更新で追加 |
| 工場設備 | 設備ごとに専用パネル | 統一ハードウェア + 設備別UI |
| 小売POS | レジ専用端末 | 汎用タブレット + 用途別アプリ |

## βプロセス志向

完璧な製品を作ってからリリースするのではなく、β版（未完成）でリリースして改善を続ける思想。

### ウォーターフォール vs βプロセス

| 観点 | ウォーターフォール | βプロセス |
|------|-----------------|---------|
| リリース基準 | 完成 | 最小限機能（MVP） |
| フィードバック | リリース後に収集 | 継続的に収集 |
| 改善サイクル | バージョンアップ（年次等） | 継続的デプロイ |
| リスク | 「外したら終わり」 | 小さく外して素早く修正 |
| 制作コスト | 完成まで高コスト | 段階的コスト |

**ハードウェア製品へのβプロセス適用**:
exUIにより、ハードウェアの物理的更新なしにUIを改善できる。Webサービスと同様のイテレーション速度で製品を改善できる。

## IoT設計原則

Internet of Things（IoT）時代のインターフェイス設計で考慮すべき原則。

### 原則1: UIの外在化を検討する

物理デバイスにUIを組み込む前に、exUI設計を検討する。

**判断基準**:
- ユーザーが常にスマートフォンを持ち歩くか？
- デバイスのフォームファクターにUIが必要か？
- パーソナライズが重要な製品か？

### 原則2: 対話型UIで複雑性を除去する

高度な機能を持つIoTデバイスのUIは、選択肢ベースの「会話」形式で操作コストを下げる。

**Dojo（セキュリティ管理デバイス）の事例から抽出したパターン**:
- 機能の直接操作ではなく「どうしたいか」を問う
- 選択肢を絞ることで次のアクションを明確にする
- 状態の視覚化をシンプルなインジケーターで行う

### 原則3: フィジカルとデジタルの統合

物理デバイスとアプリのインタラクションを設計する。

**Play Impossible（物理ボール + スマートフォンアプリ）事例から抽出**:
- 物理動作（ボールを投げる）がアプリに反映される
- アプリの状態が物理デバイスのLED等にフィードバックされる
- 「物理とデジタルの境界を感じさせない」シームレスな統合

### 原則4: UIをAPIとして設計する

将来のUI外在化（スマートグラス・HMD・音声デバイス等）に備えてUIをAPI化する。

**設計方針**:
- UI層とビジネスロジック層を徹底分離
- 表示制御をAPIで提供（任意のデバイスが接続可能）
- 将来のプラットフォーム変化に対応できる抽象化

## exUI設計のIf X then Y ルール

1. **If** IoT製品のUIを設計する **then** UIをハードウェアから分離（外在化）できないか検討する
2. **If** exUI設計を選択した **then** パーソナライズ（言語・フォントサイズ・機能フィルタ）を前提に設計する
3. **If** ハードウェア製品をリリースする **then** βプロセス志向でUIを継続改善できるアーキテクチャにする
4. **If** IoTハードウェアの機能を定義する **then** UIを変えることで製品の定義を変える（メタハードウェア）可能性を考慮する
5. **If** 将来的なUI拡張を設計する **then** スマートグラス・HMD・音声等への外在化に備えてUIをAPIとして設計する
6. **If** 物理デバイスとアプリを連携する **then** 物理操作とデジタル表示のシームレスな統合を設計する
7. **If** 同一ハードウェアで複数ユースケースを想定する **then** UIセットの切り替えによるメタハードウェア化を設計する
8. **If** IoTデバイスのオンボーディングを設計する **then** 対話型UI（会話型選択肢）で複雑な設定を簡素化する
