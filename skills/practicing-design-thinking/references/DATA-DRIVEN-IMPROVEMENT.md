## 1. 定量データ vs 定性データ

| 比較軸 | 定量データ | 定性データ |
|--------|-----------|-----------|
| 主な形式 | 数値・割合・トレンド | 言語・感情・価値観 |
| わかること | **何が・誰に** 使われているか（結果） | **なぜ** 使われているか（理由） |
| 主な収集手法 | アンケート・アクセスログ・属性分析 | インタビュー・行動観察・エスノグラフィ |
| 限界 | 過去/現在の把握のみ。改善方向は示せない | スケール困難。分析者の解釈依存 |
| 最も活きるシーン | 課題の規模感・優先順位付け | 課題の原因・ユーザー心理の深掘り |

### 分析手法のマッピング

```
顕在的  ─────────────────────────────────
         アンケート
         ユーザーインタビュー
         行動観察（エスノグラフィ）
         アクセスログ分析
潜在的  ─────────────────────────────────
```

**実践原則**: 定量データで「課題の何」を発見し、定性データで「理由のなぜ」を深掘りする。両者は線引きでなく、目的によってどちらを主体とするかを選ぶ。

## 2. データドリブン改善プロセス

### 改善の3ステップ

```
課題発見（データ分析）
    ↓
仮説設定（原因推論）
    ↓
改善施策（デザイン実装）＋定性的な検証
```

### データ活用の失敗パターン

| アンチパターン | 症状 | 対策 |
|-------------|------|------|
| データを見ていない | 根拠のない施策を実施 | 定量/定性データを両輪で継続確認 |
| データの誤解釈 | 別の問題を引き起こす | 文脈込みで複数指標を組み合わせて読む |
| 分析が目的化 | 施策が出てこない | 「何のために」を常に先に確認する |

### デザイン改善に参照すべきデータ

```
デザイン視点
├── ユーザー視点
│   ├── アクセスログ分析
│   ├── ユーザーアンケート分析
│   └── SNS投稿分析
└── セールス視点
    ├── 販売実績・属性分析
    └── 流入/流出にもとづく競合比較分析
```

> データは課題発見には役立つが、改善施策は教えてくれない。施策はデザイナーの知見が発揮される領域。

## 3. ビジネス理解とSWOT分析

### SWOT分析フレームワーク

| | プラス要因 | マイナス要因 |
|--|-----------|-------------|
| **内部環境** | 強み（Strengths） | 弱み（Weaknesses） |
| **外部環境** | 機会（Opportunities） | 脅威（Threats） |

**ポイント**: サービスの本当の強みは自社だけではわからない。ユーザーからの評価・競合との比較で初めて明確になることが多い。

### 価値認識ギャップの発見と解消

```
サービス提供者が「価値」と思っているもの
        ↕ ← 認識ギャップ
ユーザーが「価値」と感じているもの
```

**認識ギャップの解消サイクル**:
1. 「強みが本当にユーザーに届いているか」を定量・定性の両面で確認
2. ギャップが見つかれば仮説を立てる
3. 改善施策を実施して検証
4. 仮説が誤っていれば見直し → 繰り返し

### バリュープロポジション検討の問い

- 「なぜユーザーは自社サービスを選んでいるか？」
- 「その要因となっている強みは何か？」
- 「その強みは現在のデザイン・機能として表現されているか？」

## 4. アイデアとロジックの両立

**アイデアの役割**: まだ世の中にない価値を形にする原動力。ただし「アイデア単体」に価値があるのではなく「継続して実現すること」に価値がある。

**ロジックの役割**: アイデアに取り組む理由を作り、「それで何が変わるのか」を説明する。行動を起こすための根拠。

### 両者の関係

```
アイデア ←──────────────────→ ロジック
  └─ 何を作るか               └─ なぜ作るか
  └─ どんな体験か             └─ データ・分析による裏付け
  └─ 独自性・新規性           └─ 実現可能性・投資対効果
```

> アイデアとロジックは分けて考えられるものではなく、両立が求められる。ロジックがあることでアイデアがより引き立つ。

**シンプルさの原則**: どれだけ深く考えても、最終アウトプットは「誰もが初見で理解できるシンプルなもの」であること。数字・イメージで視覚的に伝える。

## 5. プロトタイピングの効果と活用シーン

### プロトタイピングが解決すること

- 構成や機能が理解しにくい → 可視化で共通理解を形成
- 情報共有がしにくい → チームの共通言語を生み出す
- 実現の可能性が事前にわからない → 早期の機会・限界の表面化

### 4つの効果

1. **コミュニケーションがスムーズになる**
2. **機会や限界を早い段階で表面化できる**
3. **アイデアを具体化し、誰もが評価できる**
4. **ユーザー体験を共有できる**

### シーン別の目的

| シーン | 目的 |
|--------|------|
| 探索と実験 | 問題・アイデアの探索。変更の影響テスト |
| 学習と理解 | 問題や仕組みの理解。課題の切り分け |
| テストと体験 | ユーザー・ステークホルダーの深いインサイト発見 |
| 販売・説得 | 新アイデアをステークホルダーに伝え購買を促す |

### プロトタイピングプロセス

```
[調べる・考える]          [可視化する]          [試す・改善する]
リサーチ、アイデア    →    スケッチ、デザイン   →   ユーザーテスト
コンセプト設計             プログラミング             バリエーション作成
                    ↑──── 必要に応じて再検討 ────┘
```

## 6. プロトタイプの種類と使い分け

### フェーズ別プロトタイプ分類

| フェーズ | プロトタイプ種別 | 主目的 |
|---------|---------------|--------|
| アイデア | デザインプロトタイプ | 見た目の検証 |
| アイデア | コンテクシュアルプロトタイプ | 使用状況・世界観の検証 |
| プランニング | ファンクショナルプロトタイプ | 機能の動作検証 |
| 開発 | マスタープロトタイプ | 実際のサービスに近い総合検証 |

### 3種類のプロトタイプ詳細

**デザインプロトタイプ**
- ツール: Figma / Sketch などデザインツール
- 目的: ビジュアル・見た目の確認。重要な画面を数画面作り込む
- 対象: チームメンバー・ステークホルダー

**ファンクショナルプロトタイプ**
- ツール: ペーパープロトタイピング / Adobe XD / Prott
- 目的: 機能の動作・インタラクションの検証
- 対象: ユーザーテスト・開発チーム

**コンテクシュアルプロトタイプ**
- 形式: サービスカタログ・利用シーン動画
- 目的: 使用状況の再現。ユーザーに疑似体験させる
- 適したシーン: サービスの世界観・ストーリーを伝えたい場面

### 用語の整理（スケッチ → プロトタイプの段階）

| 段階 | 説明 |
|------|------|
| スケッチ | 紙にフリーハンドで描いたアイデアの可視化。最もスピーディー |
| ワイヤーフレーム | 画面の骨格・構造を形にしたもの。配置決定フェーズ |
| モックアップ | 色・フォント・テキスト・画像を加えたUI表現 |
| プロトタイプ | インタラクション・アニメーションを含む動作する試作品 |

## 7. KGI/KPIによるチーム目標管理

### 指標の定義

| 指標 | 定義 | 用途 |
|------|------|------|
| **KGI** (Key Goal Indicator) | 最終的に達成すべき定量的な目標値 | 組織・プロジェクト全体の戦略目標 |
| **KPI** (Key Performance Indicator) | KGI達成プロセスの進捗を把握する指標 | 日常的な進捗管理・行動評価 |

### チームの発展段階

| 段階 | 状態 |
|------|------|
| 形成期 | メンバー同士を知らない。目標も定まっていない |
| 混乱期 | 目的・目標について意見が食い違い、対立が生まれる |
| 統一期 | チームの目的・進め方・役割が統一・共有されている |
| 機能期 | 相互サポート・問題解決し、チームとして成果を出す |

### チームの意識統一チェックリスト

行動する前に問うべき問い:
- [ ] これはやる価値があることか？
- [ ] これはユーザーにどのような価値をもたらすのか？
- [ ] これについてチーム全員が理解しているか？

### グループ vs チームの意思決定

```
グループ: リーダーが意思決定 → メンバーが実行
チーム:   メンバー間でプロセスを経て意思決定 → 全員で実行
```

### 対話 vs 会話 vs 議論

| 種別 | 目的 |
|------|------|
| 会話 | 互いの親密度を高める・情報収集 |
| 議論 | 議題に対する最適解を探し合意形成する |
| 対話 | 互いを相互理解し、新たな知識やアイデアを生み出す |

> チームでサービス開発を進めるために必要なのは「対話」。Howを考える前にWhyを考える。
