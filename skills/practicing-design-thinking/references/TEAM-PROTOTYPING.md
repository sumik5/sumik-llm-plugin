# チームで行うプロトタイピング実践ガイド

チームが協業してUXデザインを進める際の実践手法。パラレルトラック方式による開発統合から、戦術的・体験的プロトタイピングの使い分け、軽量ユーザーテストの運用まで。

> **DESIGN-PROCESS.mdとの分担:** 本ファイルは**チーム協業・ワークショップ手法・実践Tips**を扱う。HCDプロセス理論・7段階構造・忠実度選択基準は [DESIGN-PROCESS.md](DESIGN-PROCESS.md) を参照。

## パラレルトラック方式：開発への組み込み

### 概念

UXデザイン/UIデザインが**開発より少し先行する**進め方。従来のHCDプロセス（調査だけで4週間など）を軽量化し、開発のワークサイクルに合わせたデザイン主導の進行を実現する。

```
開発側:  [ プランニング ] [ 実装 ] [ 実装 ] [ 実装 ]
                  ↑
UX側:   [ 調査 ] [ 低コスト設計 ] [ テスト/調査/設計 ]
                  ↓
        データを設計へインサイト → [ Design | Code | Design | Code ]
```

### パラレルトラックが解決する課題

| 従来の問題 | パラレルトラックによる解決 |
|-----------|------------------------|
| 調査フェーズに多大な時間消費 | 手法の工夫で調査を軽量化 |
| 完成後に致命的欠陥を発見 | 早期の漸進的プロトタイピングで早期発見 |
| ステークホルダー間の認識齟齬 | デザイン工程の可視化で共通認識構築 |
| 開発サイクルとのミスマッチ | UXパートをアジャイルサイクルに適応 |

## UX評価フレームワーク

### UXハニカム（6軸評価モデル）

ユーザーにとって「価値ある体験」を6軸で評価する構造モデル。チームがUX品質を語り合う**共通言語**として活用する。

| 評価軸 | 英語 | チェック観点 |
|--------|------|-------------|
| **使いやすい** | Usable | 目的達成できるか・操作方法がわかるか |
| **見つけやすい** | Findable | 目的の情報が発見しやすいか・現在位置が把握できるか |
| **信頼できる** | Credible | コンテンツ・技術・組織が信頼できるか |
| **アクセスしやすい** | Accessible | 多様な端末・障がいのある人も利用できるか |
| **好ましい** | Desirable | イメージが伝わるか・印象がよいか |
| **役に立つ** | Useful | コンテンツが充実しているか・新発見があるか |
| **価値がある** | Valuable | 上記6軸が統合された中心的価値 |

### UXピラミッド（3レベル分類）

UXハニカムの7要素を3レベルに階層化したモデル。**上位ほど高い満足をもたらす**が、下位が未達だと上位は成立しない。

```
┌─────────────────────────────┐
│  満足できる   嬉しい | 価値がある   │  ← 最高レベル
├─────────────────────────────┤
│  安心できる   役に立つ | 信頼できる │  ← 中間レベル
├─────────────────────────────┤
│  利用できる   使いやすい | アクセスしやすい | 見つけやすい │  ← 最低条件
└─────────────────────────────┘
```

**実践ポイント:** 評価で問題が出た時、どのレベルの問題かを特定することで改善の優先度が決まる。

## 2種類のプロトタイピング

### 基本分類

| 種類 | 視点 | 目的 | 代表手法 |
|------|------|------|---------|
| **戦術的プロトタイピング** | プロダクト視点 | 誰が・何を・どのように提供するか前提を揃える | ペルソナ・6upスケッチ・カスタマージャーニー |
| **体験的プロトタイピング** | ストーリー視点 | 実際に触れて視覚的に動作を確認し精度の高いFBを得る | ペーパープロトタイプ・ツールプロトタイプ |

**重要:** どちらもシナリオが必要。「誰が・何を・どのように」という前提をチームで揃えることが先決。

## 戦術的プロトタイピングの実践

### プラグマティックペルソナ

通常ペルソナ（1〜4週間）を軽量化した**1時間で作るペルソナ**。テストサイクルの迅速化とアップデートを前提とした手法。

**作成フォーマット:**

| 項目 | 内容 |
|------|------|
| イラスト/名前/年齢/職業 | 具体的なユーザー像を可視化 |
| 行動につながる情報 | 家族構成・職業・趣味など行動に影響する属性 |
| ニーズとペイン | 現在の不満・困っていること |
| ニーズや潜在的欲求 | 本人が言語化していない深層ニーズ |

⚠️ **注意:** 都合のよいペルソナを捏造しない。身近なターゲットユーザーを参照しつつ、後のテストでアップデートする前提で作成する。

### 6upスケッチング

1つのシナリオに対して**6つの視点・パターン**を一枚のシートに描くワークショップ手法。チームで複数のシナリオ仮説を素早く可視化する。

**進行手順:**
1. ペルソナと利用シーンを前提共有（5分）
2. 個人で6コマのシナリオスケッチを描く（15〜20分）
3. チームでスケッチを見せ合い議論（15分）
4. 共通パターンや優先シナリオを特定する

**ポイント:** 絵の上手さは不要。「誰が・どんな状況で・どう使うか」が伝われば十分。

### テキストベースのシナリオ作成

カスタマージャーニーマップの前工程として、**テキストでシナリオの骨組みを準備**する。

- 1つの体験軸に複数パターンのシナリオを書き起こす
- 完璧・理想は求めず「仮説ファースト」で
- 事実の羅列ではなく**登場人物の感情を表現**する
- 複数のシナリオバリエーションを用意する（異なるユーザー属性・利用状況）

### カスタマージャーニーマップ（ASIS）

ユーザーがゴールを達成するまでの「旅の体験」を視覚化するツール。ワークショップ形式で作成する場合は**当事者に近いメンバーを参加させる**と効果的。

**基本フォーマット:**

| 要素 | 記述内容 |
|------|---------|
| ペルソナ | 主人公・目的・ニーズ |
| ステージ | 時系列のフェーズ（認知→検索→利用中→利用後など） |
| 行動シナリオ | 各ステージでの行動を記述 |
| ツール | 使用デバイス・アプリ・紙媒体等 |
| 場所 | 利用環境 |
| 感情 | その場面での感情・心理状態 |

**活用Tips:**
- サービスによっては利用者が複数属性を持つ → 属性ごとに作成を検討
- 現状ユーザーと将来増やしたいユーザーは別マップで
- 仮説検証前提で複数フォーマットを試す

### ストーリーボーディング

シナリオを機能要件・UI設計に落とし込むための**計画的な協業ツール**。5W1Hを明確にして各ステップを構造化する。

**フォーマット:**

| ステップ(5W1H) | シナリオスケッチ | ワイヤーフレーム | 機能・仕様要件 |
|--------------|--------------|--------------|-------------|
| いつ・どこで・誰が・何を・なぜ・どのように | ユーザーの行動と感情を含むシナリオ | UIの構造スケッチ | 対応する機能要件 |

**効果:** シナリオからユーザーのアクション・タッチポイントを導き出し、機能要件につなぎ込む。ステップの粒度は体験に応じて調整可。

## 体験的プロトタイピングの実践

### 抽象度・忠実度と評価ポイントの関係

**核心:** プロトタイプの完成度が高いほど、評価者が表面的な情報（フォント・色・レイアウト）に引っ張られ、本質的なフィードバックを得にくくなる。

| 種類 | 忠実度 | 得やすいフィードバック |
|------|--------|----------------------|
| スケッチプロトタイピング | 低 | 根本的な課題・方向性・大きな問題点 |
| ペーパープロトタイピング | やや低 | 情報構造・ナビゲーション・主要フロー |
| プレゼンテーションプロトタイピング | やや高 | レイアウト・視覚表現・ブランド印象 |
| インタラクティブプロトタイピング | 高 | 詳細な操作性・インタラクション品質 |

**原則:** 開発序盤はグラフィカルなデザインデータを避け、意図的に抽象度を下げることで本質的な機能面の評価を促す。

### ペーパープロトタイプの価値

紙とペンだけで作るプロトタイプの利点:
- 専門知識不要で誰でも参加できる
- 手軽に作成・破棄できる → 心理的コストが低い
- チーム全員がデザインを考えるきっかけになる
- ウェアラブル・DAppsなど複雑なUIでも効果的

### アンチパターン：体験的プロトタイピングに偏った開発

```
❌ 体験的プロトタイピングのみのパターン:
初期設計 → プロトタイピング（体験的のみ） → テスト → 開発

問題:
- 忠実度が高いため承認は得やすいが「未完成物」に凝りすぎる
- 戦術的欠陥（シナリオの破綻）を見落とす
- 負荷テストなどテクニカルなテストが後回しになる

✅ 両輪を回すパターン:
初期設計 → 戦術的/体験的プロトタイピング（並行） → テスト → 開発

効果:
- シナリオが破綻してもまだやり直せる段階で気づける
- UXテスト・負荷テスト等を余裕を持って実施できる
```

### ユーザーの声を聞きすぎないこと

体験的プロトタイプへのフィードバックを重ねると「夢のプロダクト」になりがち。

**本質:** ユーザーの声をすべてくみ取るのではなく、**体験をテストして本質的に求められる価値を提案すること**が目的。MVPの考え方と連動させる。

## 軽量ユーザーテストの設計

### 「必要十分」で小さくスタート

専用設備・専門機器は必須ではない。重要なのは**小さな単位でテストを繰り返すこと**。

| 軽量UTの原則 | 説明 |
|------------|------|
| 分散して実施 | チームのメンバーでテストを分担・繰り返す |
| 早期に実施 | ペーパープロトタイプ段階から始める |
| 複数回実施 | 1回の大規模テストより複数回の小規模テスト |
| チームで協業 | テスト実施自体をチームの学習機会として活用 |

### 被験者選定のポイント

| 選定方針 | 説明 |
|---------|------|
| チームの人脈を活用 | 信頼関係構築が容易。コンプライアンス手続きも簡略化可能 |
| 対象属性に近い人を選ぶ | ペルソナ属性が合致しないと検証結果が無意味になる |
| プロジェクト未参加者が理想 | 先入観がないため率直なフィードバックが得られる |
| 「例外を多く再現してくれる人」 | 開発者の想定外の行動を見せてくれる被験者が貴重 |

### ユーザビリティテストの評価ポイント（観察の順序）

1. **① タスク完了できたか** — 独力で目的を達成できたか（できなければ根本的な問題あり）
2. **② 迷いや無駄な操作はないか** — 想定動線から逸れていないか
3. **③ 不満を感じていないか** — 言葉に出ていなくても表情・態度に注目

**注意:** ②以降の観察に集中しすぎると、①の重大な問題を見落とす危険がある。

### ユーザーテストのアンチパターン

| アンチパターン | 問題点 | 対策 |
|-------------|--------|------|
| 完成してからテストする | 修正コストが跳ね上がる。環境再現なしでは本来の意味がない | ペーパープロトタイプ段階から開始 |
| ターゲット属性から外れた被験者 | リテラシーが同じでも目的・思考がずれると無意味 | ペルソナ属性と利用シーンを事前確認 |
| 被験者に質問する | プレゼンの場に変わり先入観を植え付ける | 観察に徹する。「どう思いますか？」禁止 |

### エキスパートレビューとの組み合わせ

ユーザーテスト単体では全問題を特定できない。エキスパートレビューと組み合わせることで補完される。

| 手法 | 強み | 推奨タイミング |
|------|------|-------------|
| ユーザービリティテスト | 実際のユーザー行動の観察 | プロトタイプ段階〜完成品 |
| エキスパートレビュー | 専門知識による課題洗い出し + 改善提案 | 完成品の最終確認・補完として |

## ツール選択の原則

プロトタイピングツールの多様化（Figma・InVision・Adobe XDなど）は生産性を上げる一方、依存しすぎると本質を見失う。

**原則:**

- 紙とペンで十分できることを確認してからツールを検討する
- 開発フェーズに見合ったフィードバックを得られるツールを選ぶ
- 高解像度ツールは「承認コスト」を下げる一方で「凝りすぎリスク」を高める
- コミュニケーションの質 > ツールの高機能性

**判断基準:**

| 状況 | 推奨アプローチ |
|------|-------------|
| 初期設計・方向性確認 | 紙・ペン・付箋（抽象度を意図的に下げる） |
| インタラクション確認 | プロトタイピングツール（Figma等） |
| ステークホルダー承認 | ビジュアルが整った高忠実度プロトタイプ |
| 最終品質確認 | 実装コードベースのα/βテスト |
