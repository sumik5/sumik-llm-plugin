# Webサービスのチームカイゼンプロセス

チームでWebサービスのUIを継続的に改善するための考え方・プロセス・体制づくり。
サービス成長のためのOODAループ活用と、デザインをチーム全員で担う組織設計を扱う。

---

## 改善の出発点：「誰のための改善か」を明確にする

改善の価値を届ける先はユーザーである。思い込みや「何となく」でUIを変えるのではなく、
**「どのような価値を、誰に対して提供するための改善なのか」** を最初に明確にする。

### ユーザー理解の2軸

| 概念 | 定義 | 特性 |
|------|------|------|
| **インサイト** | 行動や態度の奥底にある、本人も意識していない本音 | 直接質問では得られない |
| **コンテキスト** | ユーザーが行動する際に影響する周囲の状況・背景 | 時間帯・場所・天候・利用環境・前後の行動・心理状態など |

どちらも固定されたものではなく、設計時だけでなく**運用時にも継続して意識する**。

### ペルソナとカスタマージャーニーマップの使い方

- **ペルソナ**: チーム内で対象ユーザー像を統一するための登場人物設定
- **カスタマージャーニーマップ**: ペルソナがサービスを利用する中での行動・感情・課題を時系列で可視化

> 定量リサーチ（アクセスログ等）と定性リサーチ（インタビュー等）を組み合わせ、現状を正しく把握することが改善のスタートとなる。

## デザインが解決できることとできないこと

サービスはデザイン・テクノロジー・ビジネスの3領域で構成される。

```
      デザイン
       /   \
      /     \
テクノロジー   ビジネス
```

| 領域 | 解決できる課題の例 |
|------|-----------------|
| デザイン | 操作感・情報構造・UIのわかりやすさ |
| テクノロジー | レコメンド精度・セキュリティ・パフォーマンス |
| ビジネス | 物流・販売経路・価格設計 |

取り組もうとしている課題がどの領域に属するかを正確に判断することで、デザインで解決可能な「デザイン要件」を特定できる。

### デザインプロセスの選択

| 状況 | 推奨プロセス |
|------|------------|
| デザイン要件が明確 | 視覚化 → 改良 → 実行 |
| デザイン要件が不明確 | 理解 → 観察 → 視覚化 → 改良 → 実行 |

普遍的なデザインプロセスは存在しない。プロジェクトごとに最適なプロセスを設計・共有することが重要。

## 課題の見つけ方と優先順位づけ

### 問題と課題の区別

- **問題**: 理想の状態と現状とのギャップ（原因そのもの）
- **課題**: 問題を解決するために起こす具体的なアクション

仮説を立てる前に、ユーザーの現状について情報収集・整理を行い、ユーザーが望むことと現状のギャップを明確にする。

### 施策の優先順位づけ

**緊急性・重要度の2軸は避ける。** 緊急性が高いものは重要に見えてしまい、優先順位が歪む。

代わりに **「実現可能性」と「有効性」の2軸** で判断する:

| 実現可能性 ↓ \ 有効性 → | 高 | 低 |
|------------------------|----|----|
| **高** | 🔴 最優先で取り組む | 保留（効果が薄い） |
| **低** | 中期検討（効果大だが難しい） | 取り組む必要なし |

## 改善プロセスの選択：OODAループ vs PDCAサイクル

### PDCAサイクルの問題点

PDCAは「計画が前提」のフレームワーク。実行後の評価でようやく現場フィードバックを得るため、計画と現場の乖離があると修正に多くの時間が必要になる。

> 想定外の事態に対する対応が手薄になりやすく、サイクルを回している間に致命的な状況に至るリスクがある。

### OODAループの特徴

```
Observe（観察）→ Orient（状況判断）→ Decide（意思決定）→ Act（実行）→ Observe...
  みる              わかる               きめる            うごく
```

- PDCAが「自分の計画」から始まるのに対し、OODAは「相手（ユーザー）の観察」から始まる
- 不明確で変化し続ける状況の中で、最善の判断を下し即座に行動することを目的とする

### 使い分け判断基準

| 特性 | PDCA | OODA |
|------|------|------|
| 適した課題 | 業務改善・工程の明確な改善 | 新規サービス開発・サービス成長 |
| サイクル速度 | 月〜年単位 | 日〜週単位 |
| 出発点 | 計画策定 | 現状観察 |
| 強み | 生産効率・工程の最適化 | 変化への即応・意思決定の高速化 |

## ダブルダイヤモンド：正しい問題を見つけてから解く

英国デザイン協議会が提唱する課題解決フレームワーク。2つの「発散→収束」サイクルで構成される。

```
[発散1: 問題の探索]    [収束1: 正しい問題]    [発散2: 解決策の探索]    [収束2: 正しい解決策]
   課題や状況を          本当に解くべき          多様なアプローチを          実行する解決策
   幅広く調査 ●→──→●  問題を定義する ●→──→●  生成・検討する ●→──→●  を選定する
```

**重要な原則**: いくら解決策が優れていても、問題が正しく設定されていなければ課題解決にならない。
ダブルダイヤモンドは「解くべき問題の正しい設定」を最初のダイヤモンドで保証する。

### デザインプロセスの可視化

デザインの思考プロセスはブラックボックスになりがち。言語化・可視化することで:

- チームメンバーが認識のずれなく共通理解を持てる
- デザインと各自の業務との関係性が見えてくる
- 一体感のある開発チームを構築できる

可視化する対象: **情報の可視化 / 思考の可視化 / 対話の可視化**

## サービスを成長させるチーム体制

### 小さなチームを複数並列で動かす

| アプローチ | 問題点 |
|-----------|--------|
| 大人数での意思決定 | コミュニケーションコストが高い |
| 単一の改善チーム | 施策の数とスピードが制限される |
| **小チーム × 複数並列** | ✅ スピード感を保ちながら多くの施策を実行 |

意思決定は必要最小限の人数で行い、施策実施〜検証のサイクルを速くすることで、サービス全体の成長確率が上がる。

### 全員がデザインに向き合う

デザイナーだけがデザインを担うのではなく、チーム全員がユーザー体験に影響を与えていることを認識する。

| 役職 | ユーザー体験への影響例 |
|------|----------------------|
| エンジニア | パフォーマンス調整・実装品質 |
| 営業 | ユーザーへのメッセージ・コミュニケーション |
| 企画 | 機能の優先度・ユーザーストーリー |
| デザイナー | 情報設計・ビジュアル・インタラクション |

> 「デザイナーのスキルが必要」ということではなく、自分の決断がどのようにユーザー体験に影響を与えるかを理解することが重要。

### チームビルディングの原則

- **初期フェーズから**: デザイナーとエンジニアが共に参加し「体験の大事なポイント」の共通理解を作る
- **ペルソナ・コンセプトはチームで定義**: UXデザイナーだけが考えるのではなく全員でワーク
- **問題は複数領域にまたがる**: 単一職種では解決できない。相互の専門知識を活用する

## プロトタイピングの導入

### プロトタイピングが解決する問題

各メンバーのイメージには認知バイアスがかかっており、デザイナーが思い描くものとの差が生じやすい。プロトタイピングの早期導入で:

- **可視化のスピードが速くなる**: ツールの進化により作業負荷が低下
- **イメージの共有がしやすくなる**: 動くプロトタイプで確度の高い確認ができる
- **決断しやすくなる**: 曖昧な状態での議論を減らす

### プロトタイピングの注意点

- プロトタイプを **完成物として扱わない**
- 「どの部分について確認してほしいか」を事前に明確に伝える
- 確認範囲の合意が曖昧なままにすると、後から大規模な変更が発生する

## デザインの運用

### デザインシステムの役割

チームが変わっても、デザインの一貫性を保つために**デザインシステム（デザインガイドライン）**が必要になる。

| 側面 | 効果 |
|------|------|
| 一貫性の維持 | さまざまな製品・チーム間でビジュアルと体験を統一 |
| 認知的負荷の軽減 | ゼロから判断する回数を減らし開発スピードが向上 |
| 共通言語の提供 | チームの誰もが同じ基準でデザインに向き合える |

デザインシステムはスタイルガイドやパターンライブラリであると同時に、製品開発の設計図でもある。

## チーム内コミュニケーションツールの導入

### ツールとルールは1セット

どんなに優れたツールも、導入するだけでは機能しない。ツール導入時には必ずルールも合わせて設定する。

**導入時のチェックリスト**:
- [ ] 何のために導入するか（目的）を明確に規定する
- [ ] 用途ごとに単一のツールを使用する（複数ツールの併用禁止）
- [ ] 最低限の使用ルール（どういった場合にどう使うか）を事前に設定する

### 情報共有とチームの信頼関係

「自分に必要な情報が共有されない」と感じると、コミュニケーションコストが増大しモチベーションが下がる。

適切な情報共有の効果:
- 業務の円滑化
- 「伝えていない・聞いていない」の行き違いが減少
- メンバー間の信頼関係の構築

> 情報共有とは、蓄積された情報を整理し、共有し、活用することで生産性向上につなげること。
