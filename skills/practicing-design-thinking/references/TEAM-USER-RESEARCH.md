# チーム参加型ユーザーリサーチ 実践ガイド

チームでリサーチに参加することで、報告資料を読むだけでは得られない当事者意識と改善アイデアが生まれる。
長い報告書より、同じユーザーをチームで一緒に見る体験の方が課題感を共有しやすい。

## リサーチ手法の使い分け

| 手法 | 目的 | コスト | 向いている場面 |
|------|------|--------|--------------|
| ユーザーインタビュー | 行動・価値観・課題の深掘り | 中〜高 | 仮説がある・深いインサイトが欲しい |
| ゲリラインタビュー | 素早い方向性確認 | 低 | 時間・予算が少ない・潜在層に触れたい |
| ユーザビリティテスト | UIの使いやすさ評価 | 中 | 特定画面・機能の操作性検証 |
| シャドーイング | 生活文脈の全体理解 | 高 | ユーザーの日常実態を丸ごと把握したい |
| エキスパートレビュー | 専門知識の補完・コスト削減 | 低〜中 | ユーザーにアクセスしにくい・スピード重視 |
| ミートアップイベント | コミュニティ形成 + 本音収集 | 中 | SaaS・BtoB・ファン形成したい |

## ユーザーインタビュー実践

### ① 計画フェーズ

**目的の明確化**:
- 何を知りたいのか（課題・行動・価値観）を具体的に言語化する
- ユーザーインタビューが最適かを検討（特定UIがある → ユーザビリティテスト、生活文脈が知りたい → シャドーイング）
- プロジェクト・スケジュール・予算の制約を確認する

**リクルーティング方法**:
- 企業サイト・SNSで直接ユーザーに呼びかけて募集
- 社内部署（営業・CSなどユーザーとの接点が多い部署）に相談
- 外部の専門サービスを活用

### ② 準備フェーズ

**仮説の言語化**（インタビュー前に必ず行う）:

| 項目 | 内容 |
|------|------|
| サービス体験 | 機能が提供できる価値・体験 |
| 課題仮説 | 現状どういった課題があるか・その原因 |
| 改善案 | 課題への解決アプローチ |

仮説を言語化しておくと、インタビュー結果と対比しながら分析できる。

**質問設計のコツ**:
- 知りたいことをリストアップし、グループ分けして優先度をつける
- 当日は文脈に応じて質問するため、予定通りには進まない → 多めに用意しておく
- ポストイットやスプレッドシートで整理するとチームで共有しやすい

**インタビューガイドの構成例**:

| カテゴリ | 所要時間 | トピック | スクリプト例 | 検証したい仮説 |
|---------|---------|---------|------------|-------------|
| 準入 | 10分 | 挨拶・主旨説明・同意書 | 「今回はサービスについてお話を聞かせてください…」 | — |
| タッチポイント | 5分 | サービスを知ったきっかけ | 「〇〇を知ったきっかけは？」 | 流入経路 |
| 利用シーン | 10分 | 使い方・習慣 | 「どの端末で・いつ使っているか？」 | 利用文脈 |
| 深掘り | 20分 | 課題・感情・価値観 | 「そのとき、どう感じましたか？」 | 課題仮説 |
| まとめ | 5分 | 総合感想・要望 | 「最後に伝えておきたいことは？」 | — |

**リハーサル**: ロールプレイング形式で練習して伝え方・タイミングを確認する。

**役割分担**（1名のユーザーに対してスタッフ2名が基本）:
- **質問者（モデレーター）**: ユーザーとの対話・深掘りに集中
- **記録者**: ユーザーの発言を議事録に記録
- **オブザーバー**: 表情・しぐさの観察、話が脱線した際のサポート

複数スタッフが参加したい場合はビデオ中継が有効（多人数がユーザーを取り囲まない）。

### ③ 質問フェーズ — インサイトを引き出すテクニック

**心理的安全性の確保**:
- アイスブレイクで緊張を解きほぐす（ユーモア・世間話）
- 信頼関係（ラポール）を築いてから本題に入る

**やってはいけないこと**:
- 自分の意見を主張する
- 誘導質問をする
- 「はい/いいえ」で終わる二者択一型の質問に偏る

**深掘りのコツ**:
- 「なぜ？」と直接聞くと、論理化・省略・ウソが起きやすい
- 代わりに **「いつ・どこで・どのくらい・誰が・何を・どうやって」** で具体的なエピソードを引き出す
- **バックトラッキング（オウム返し）**: キーワードをそのまま繰り返すことで信頼関係を築き、「もっと話してよい」と感じてもらう
- ユーザーの発言を鵜呑みにしない: 建前と本音を分けて観察する

### ④ 記録フェーズ

**観察ポイント**:
- 頻出する単語・言葉使い
- 発言時の表情・態度の変化
- よく登場する人物・状況
- 発言した「事実」と「記録者の解釈」を常に区別する

**議事録の取り方** — 事実と解釈を分けて記録する:

スプレッドシート方式（列で分ける）:

| モデレーターの発言 | ユーザーの発言 | 気づき（記録者の解釈） |
|-----------------|-------------|---------------------|
| 昨日のランチは何を食べましたか？ | コンビニで買って机で食べた | 食べる時間・場所がない？ |
| 何を買いましたか？ | サンドウィッチとサラダ | — |

メモ方式（記号で分ける）:
```
@モデレーターの発言
→ ユーザーの発言
/ 気づき（解釈）
```

録音・録画は後の分析と共有に活用。必ず事前に本人の許可を得る。

### ⑤ チーム振り返りフェーズ

インタビュー直後、**記憶が鮮明なうちに**振り返りを実施する。

**分析手順**:
1. **議事録作成**: 録音データを確認しながら記録を補足する
2. **要点抽出**: 仮説に関連する重要発言・頻出トピックをピックアップ
3. **法則性の発見**: 共通点・相違点でグループ分けし、見出しをつける（KJ法）
4. **図解**: グループ間の因果関係・時系列を可視化（カスタマージャーニーマップも有効）
5. **考察**: 仮説検証結果・改善課題・新発見をまとめ、次のアクションを提言

**インタビュー報告の構成**:

| 実施概要 | インタビュー内容 | 考察 |
|---------|---------------|------|
| 調査目的・日時・場所・参加メンバー | ユーザープロフィール・議事録・撮影物 | 仮説検証・図解まとめ・改善案 |

**共有のポイント**:
- 主観的解釈に流れず、事実から課題の本質を見極める
- チームが閲覧しやすいフォーマット（チャット・ホワイトボード・ポストイット）を使う

## ユーザビリティテスト — 軽量版アプローチ

> **軽量UTのすすめ**: まず近くにいる社内関係者を相手に小さく始めてみる。大切なのは、実際に使ってもらいながら盲点や改善点を洗い出すこと。

### 準備

**タスク設計のポイント**:
- **シンプルに**: 優先度の高い機能に絞る
- **ユーザー視点で**: 担当者の主観ではなくユーザーの行動・ニーズを起点にする
- **シナリオ化**: ユーザーがリアリティを感じながら操作できる状況設定を添える
- タスクに製品用語・操作手順を含めない（ゴールだけを示す）

### 当日フロー

| フェーズ | 内容 |
|---------|------|
| ① イントロ | 環境準備・あいさつ・同意書記入 |
| ② 事前インタビュー | アイスブレイク・プロフィール・タスク関連の会話 |
| ③ タスク実施 | タスク提示・操作・思考発話 |
| ④ 事後インタビュー | 感想・主観的評価・要望 |
| ⑤ エンディング | 謝礼・見送り |

**思考発話法**: タスク中に「今何を考えているか・どう感じているか」を口に出してもらう。認知的な問題箇所の把握に有効。

**注意事項の伝え方**: 「あなたの能力を測るのではなく、画面の良し悪しを調べています。うまく操作できなくても気にしないでください。」

**タスク実施中は観察に徹する**: 積極的に話しかけたり、誘導したりしない。

### 結果分析

**課題の優先度マトリクス**:

| 発生頻度 | 効果（ゴール未達成） | 効率（無駄・戸惑い） | 満足度（不満感） |
|---------|-----------------|-----------------|--------------|
| 高 | 優先度1 | 優先度2 | 優先度3 |
| 中 | 優先度2 | 優先度3 | 優先度4 |
| 低 | 優先度3 | 優先度4 | 優先度5 |

**仮説検証の観点**:
- 仮説と結果のギャップが少ない → 確度の高いアイデア
- 課題・改善点が存在 → 解決案を考案する
- 想定外の発見 → 新しい知見として学習する

## ゲリラインタビュー

街頭で通りかかった人に話しかける**インターセプトインタビュー**。少ないリソースで素早く方向性を確認できる。

**ユーザーインタビューとの比較**:

| | ゲリラインタビュー | ユーザーインタビュー |
|--|---------------|----------------|
| 対象者 | 偶然居合わせた人（潜在層含む） | サービス利用者 |
| リクルーティング | 不要 | 必要 |
| コスト | 低 | 中〜高 |
| インサイトの深さ | 浅め（短時間） | 深い（1〜2時間） |

**活用シーン**:
- 本格的なインタビュー前に大まかな方向性を確認したい
- デプスインタビューで得た知見の妥当性を確かめたい
- 時間・予算のリソースが少ない

**実施のコツ**:
- 実際の利用シーンに当てはまる場所・時間帯で実施する
- 「商品アンケートに答えてほしい」などオフィシャル感を出して声をかける
- 2名で連携（質問者 + 記録・観察担当）

## シャドーイング

ユーザーに長時間密着して生活・行動を観察する手法（参与観察）。

**目的**: デジタルサービスの枠にとらわれず、ユーザーの日常文脈・感情・行動の背景を深く理解する。

**実施のポイント**:
- 事前仮説を過剰に持ち込まない: 仮説立証に思考が誘導され、見えるものが限定される
- 「なぜこのユーザーは今こう行動しているのか？」と問いかけながら観察する
- **記録を最優先**: シャドーイング中に分析すると目前の事象を見逃す
- 写真・動画など視覚情報を優先（参加していないメンバーにも情景が伝わりやすい）
- 分析はデータが揃ってから全体を俯瞰して行う

**ユーザーインタビューとの組み合わせ**:
インタビューで聞いた内容を実際の生活で観察すると、会話では省略されていた事実やステークホルダーとの関係性が見えてくる。複数の視点から課題感を探ることが重要。

## ミートアップイベント

ユーザー・ステークホルダーとの交流イベント。広報目的に加え、本音のインサイト収集・コミュニティ形成に活用できる。

**BtoB・SaaSで特に有効**: サービス開発側とユーザーの距離が遠くなりがちな領域でユーザー理解を深める機会になる。

**本音収集のコツ**:
- 懇親会の雰囲気はリラックスした対話に適している
- ユーザー同士の会話を傾聴・観察することで、サービス提供者がいると出にくい本音が見える
- **ケーススタディ（優良ユーザーの活用事例）を発表させる**: 共通属性・課題を持つユーザー同士の情報交換は共感性が高く、リアルな知見が得られる

## エキスパートレビュー

有識者からフィードバックをもらい、仮説検証・改善点を素早く発見する手法。

**2種類のエキスパート**:

| 種類 | 具体例 | 得られる情報 |
|------|--------|------------|
| ドメイン知識観点 | 業界有識者・営業・CS担当・ユーザー周辺の活動者 | マーケット・ユーザー行動の体系的知見・複数事例の比較 |
| デザイン観点 | UIデザイン経験者・他のデザイナー | ユーザビリティ・情報設計の問題点・考慮漏れ |

**ユーザーテストとの使い分け**:
- 時間・コストを削減したい → エキスパートレビュー（リクルーティング・謝礼・会場が不要）
- 実際のユーザーの使い方を観察したい → ユーザビリティテスト
- 複数ユーザー事例を体系化して理解したい → 有識者にヒアリング

**社内活用**: 専門領域（教育・金融・医療等）では有識者による社内勉強会がチームのドメイン理解向上に有益。日頃の企画・設計・テストで「もしユーザーならどう使うか」を具体的に想像しやすくなる。

## AskUserQuestion: リサーチ手法の判断

```
1. 何を調べたいか？
   - 未知の課題を探索したい → ゲリラインタビュー or シャドーイング（探索的）
   - 仮説を検証したい → ユーザーインタビュー or ユーザビリティテスト（検証的）
   - 特定UIの操作性を評価したい → ユーザビリティテスト（特定画面・機能がある場合）
   - コストを抑えてスピーディに → ゲリラインタビュー or エキスパートレビュー

2. チームで参加できるか？
   - できる → チーム全員で同じユーザーを見る機会にする（課題感の共有に効果大）
   - できない → ビデオ中継・録画で共有できる環境を整える

3. 今すぐ始められるか？
   - 準備に時間をかけられない → 社内関係者を相手に軽量UTから始める
   - 時間・リソースがある → 正式なインタビュー計画を立てる
```
