# practicing-design-thinking — 詳細ガイド

> 出典: 「エンジニアのためのデザイン思考入門」（東京工業大学 EDP）
> d.school（スタンフォード大学）・IDEO を源流とする方法論

---

## 概要

デザイン思考（Design Thinking）は、d.school（スタンフォード大学）とIDEO が源流の、ユーザー体験を中心に置いた問題解決・価値創造の方法論。本スキルは東京工業大学 EDP（エンジニアリングデザインプロジェクト）の実践知識を基に構成。

## 5ステップの概要

```
共感 → 問題定義 → 発想 → プロトタイプ → テスト
(Empathize)  (Define)  (Ideate)  (Prototype)  (Test)
```

各ステップは線形ではなく、フィードバックによって何度も行き来する反復プロセス。

## 使用タイミング

### 適している場面

- 新製品・サービスのアイデア創出
- ユーザーリサーチの設計と実施
- 問題の本質を発見したい場面
- デザインスプリント・ワークショップのファシリテーション
- エンジニアが「何を作るべきか」を探索する段階

### 適していない場面

- 要件が既に確定している実装タスク
- 大規模な市場データ分析（→ 市場調査の手法を使用）
- 10年以上先の未来予測（→ 未来洞察の手法を使用）
- UI の視覚的品質改善（→ `applying-design-guidelines` を使用）
- 行動変容を促すデザイン（→ `applying-behavior-design` を使用）

## Step 1: 共感（Empathize）— ユーザーリサーチ手法

### 共感 vs 同情

| 共感（Empathize）| 同情（Sympathize）|
|----------------|-----------------|
| 相手を自分と**同一視しない** | 相手を自分と同一視する |
| 相手の視点から感じ、自分の中でも経験する | 相手の感情を自分のこととして感じる |
| 多様なユーザーそれぞれに異なる共感を持てる | 相手を集団としてひとまとめに扱いがち |
| 新しい発想の出発点になる | 独りよがりになりやすい |

**重要**: ユーザーリサーチの目的は「事実確認」ではなく「インスピレーションを得ること」。

---

### インタビュー手法

#### 誰に聞くか？

- **既存ユーザー**: 改良・改善のニーズを探る場合
- **エクストリームユーザー法**: 新しい発想が欲しい場合

  > 平均的なユーザーは現状を「当たり前」と思うため、発想に刺激を与えにくい。
  > 極端に使用量が少ない/多い人、特殊な嗜好を持つ人に話を聞いてインスピレーションを得る。
  > ⚠️ 注意: 極端なニーズに**そのまま応える**のではなく、連想ゲームのように発想を広げることが目的。

#### 何を聞くか？

- 予定のインタビュー項目より、**予想外の発言**に素早く反応してつっこむことが重要
- 「Whyを5回重ねる」は**インタビュアーの問題意識を深める**ためのもの。相手に直接5回「なぜ？」と聞くのは NG
- 答えに詰まったら周囲の状況・経緯・感情を聞いて背景を浮かび上がらせる
- 相手が**自分でも気づかなかったこと**に行き着いたら成功

#### どこで聞くか？

- 相手の**ホームグラウンド**（職場・自宅など）に出向く基本
- 現場で**身体で感じる**（ロールプレイ・疑似体験）
- 複数人で行く（質問・メモ・観察を分担）
- 会話の**箇条書き記録は避ける**（大事な情報が落ちる）。話したとおりに記録する

---

### 観察・参与

- **気づかれない観察**: 自然なふるまいを見るのに適する（公共の場に限定）
- **了承を得た近距離観察**: 疑問を適宜質問できる。インタビューと組み合わせる
- **参与観察**: 作業を手伝う・習うことで外部観察者では得られない情報を得る

**身体で感じる**: ユーザーの疑似体験をすることで、言葉・視覚だけではわからない気づきを得る
（例: 高齢者体験キットを使って農家の三脚作業を体験 → 恐怖感を実感）

**ロールプレイ**: チームでユーザー役・関係者役を割り振り、即興演劇のように演じる
（例: 駅の階段でベビーカーを使う母親のロールプレイ → 精神的・肉体的負担を体験）

---

### リサーチ結果の表現方法

#### 共感マップ（Say / Do / Think / Feel）

```
┌────────────────┬────────────────┐
│   Say（言っていること） │   Do（やっていること）  │
├────────────────┼────────────────┤
│  Think（考えていること）│  Feel（感じていること）│
└────────────────┴────────────────┘
```

- Say/Do は表面に見える情報（埋めやすい）
- Think/Feel はユーザーが言語化していない深層を**チームでイメージ**することがポイント

#### ペルソナ

- 実在の人物 or 複数人物の合成で「対象ユーザー像」を作る
- 名前・年齢・経歴・目標・生活スタイル・写真/似顔絵を1枚にまとめる
- ⚠️ 凝りすぎに注意: カスタマージャーニーマップや物語づくりの**登場人物設定**として使う

#### カスタマージャーニーマップ

- ユーザーの体験を時系列で表現（前・最中・後 など）
- 行動・感情・使用ツール・人間関係を列挙
- 「動詞」で考える（名詞ではなく行動と感情を中心に）

#### ユーザーの物語づくり（ストーリーボード・寸劇・動画）

物語の基本型:
1. 初期状態（主人公の紹介と日常）
2. 問題・障害の発生
3. 解決の示唆（← デザイン思考では「解決」より「可能性の示唆」）
4. 解決後の状態（オプション）

表現形式: 6コマ漫画・絵コンテ・デジタル紙芝居・寸劇・ショートムービー

---

### ユーザーリサーチが独りよがりにならないために

- **チームの多様性**: 多様なバックグラウンドのメンバーが共感できる範囲を広げる
- **コンテクストを全体として理解**: 要素分解しすぎると文脈が失われる
- **物語として語る**: 他者の反応を通じてチームの共感の範囲を広げる
- **プロトタイピング**: 実物の力を借りて幅広い人の共感を得る

## Step 2: 問題定義（Define）— POV・HMW

### ざわざわ感の発見

「問題点」（当事者も認識し、解決可能なもの）ではなく、**「違和感」**（当事者が気にしていない些細な点）を探す。

EDPではこれを**「ざわざわ感」**と呼ぶ（福本伸行作品の心理描写から）。

違和感を見つける4つの型（『デザインコンサルタントの仕事術』より）:
1. **応急処置**: 根本原因が未解決なまま
2. **価値観**: 製品とユーザーの価値観が合致していない
3. **惰性**: 単に惰性で同じものを使い続けている
4. **長期的利益と目先の欲求の対立**

---

### POV（Point of View）— タテマエメソッド

d.school の POV Madlib を東工大 EDP が改良した手法。

**オリジナルの POV**:
```
[ユーザー] は [ニーズ] をする必要がある。なぜなら [驚くべきインサイト] があるからだ。
```
→ 問題: 「驚くべきインサイト」を埋めることが難しく、ありきたりな視点になりがち

**タテマエメソッド（EDP 独自改良版）**:
```
[状況] にある [ユーザー] は、
[行動] をしている/する必要がある。
なぜなら [タテマエ] だからだ。
とはいえ、本当は [インサイト] である。
```

使い方:
1. まず [タテマエ]（「まあそうだろうね」な理由）を素早く見つける
2. 「とはいえ」で視点を切り替えてユーザーの深層心理を探る
3. [タテマエ] と [インサイト] の間に **TCS（Tensions/Contradictions/Surprisings）** が生まれれば成功

**例**:
```
（悪い例）
10代の女の子は、栄養のある食事をとる必要がある。
なぜならビタミンは健康に不可欠だからだ。
→「まあそうだろうね」で終わる

（タテマエメソッド適用後）
10代の女の子は、栄養のある食事をとる必要がある。
なぜならビタミンは健康に不可欠だからだ。
とはいえ、毎日は食べられない。
→「不可欠」なのに「食べられない」という対立（Contradiction）が生まれ、課題が浮かぶ
```

---

### HMW（How Might We）— どうすれば私たちは〜できそうか？

POV からアイデアを引き出す問い。

- **How**: 疑問形で解決策を探しやすくする
- **Might**: 「かもしれない」でさまざまなアイデアを受け入れる
- **We**: チームで取り組んでいることを示す

HMW の方向性バリエーション（d.school より）:
- よい面を伸ばす
- 悪い面を除去する
- 反対を考えてみる
- 前提を疑う
- 形容詞で考える（「慌てる」→「心地よい」）
- 他のリソースを活用する
- ニーズやコンテクストから連想する
- 課題を分割する

**HMW の粒度調整**:
- 解空間が広すぎてアイデアがまとまらない → 「形容詞で考える」「課題を分割する」で絞る
- アイデアが小粒 → 「反対を考えてみる」「前提を疑う」で広げる

## Step 3: 発想（Ideate）— ブレインストーミング・発想法

### アイデアを生み出すガイドライン（EDP）

#### 1. タンジブル（Tangible）なものを考える
「ヒト」と「モノ」がつながり、「コト」を引き起こすものを考える。
名詞（モノ）だけでなく、動詞（行動・体験）でデザインする。
> 「我々がデザインしているのは名詞ではなく動詞である」— Bill Moggridge（IDEO 共同創業者）

#### 2. 素人発想
知識があると「それは難しい」と判断してしまう。
「どうしてやってるの？」「やめちゃえば？」「こうすれば簡単じゃない？」という新鮮な視点を持つ。
（参考: 金出武雄「素人発想、玄人実行」）

#### 3. 名詞ではなく動詞で考える
UX・ジョブ理論の考え方。カスタマージャーニーマップを使って時系列で考える。

#### 4. 組み合わせで考える（枯れた技術の水平思考）
新しいアイデアはゼロから生まれるのではなく、既存のものの組み合わせから生まれる。
（参考: 横井軍平「枯れた技術の水平思考」→ゲーム＆ウオッチ誕生）

---

### 発想法

| 手法 | 説明 |
|-----|------|
| アイデアしりとり | テーマに対してしりとりを続け、出てきた言葉からアイデアを連想 |
| 常識破りの○○ | 「普通ならこう考える」を列挙し否定・反転させる（シルク・ドゥ・ソレイユ） |
| ワックスオン・ワックスオフ | まったく違う分野・製品を融合する（映画『ベスト・キッド』より） |
| ブレインライティング | 紙に書くブレスト。個人で5分3アイデアを書いて紙を回す |

**ブレインストーミングの注意点**:
- 集団でのブレストは満足度が高いが、実際のアイデア量・質は個人の方が優れることがある（「グループの生産性の幻想」）
- まず**個人**でアイデアを出し、その後「個人→ペア→グループ」と段階的に増やす
- ブレストはアイデアを出す場ではなく「チーム学習とアイデア評価基準の構築」に有効

**「雑談」と「落書き」**:
- 言葉を重視しすぎると議論が停滞する
- 雑談（チームインタビュー）と落書きをアイデア出しの中心に置く
- 抽象的な言葉から複数のアイデアが生まれる可能性を許容する

---

### 共感からアイデアへ

1. カスタマージャーニーマップでデータをモデル化
2. データを眺めて「ざわざわ感」を付箋紙でメモ（「問題点」ではなく「違和感」）
3. タテマエメソッドで POV を作成
4. HMW で課題を問いに変換
5. 雑談・落書きでアイデアを具体化
6. コント（寸劇）でアイデアのコンテクストを評価
7. チラシ（製品名・キャッチコピー・効果）でアイデアを1枚にまとめる

**アイデアの評価**:
WOW（すごい）→ Like（いいね）→ So So（まあまあ）→ Sorry（いまいち）の4段階で評価し、上位に共通する傾向から「デザイン原則」を抽出する。

**名前重要**:
端的にわかりやすい名前をつけることで設計の8割が完成する（小林製薬の命名スタイルを参考に）。

## Step 4: プロトタイプ（Prototype）

### プロトタイプの定義

デザイン思考では、アイデアをカタチにしてユーザーに伝えるものをすべてプロトタイプと呼ぶ。
スケッチ・4コマ漫画・動画・ダンボール模型・動作する検証機、すべてが含まれる。

### プロトタイプの種類（スタンフォード大学 ME310 より）

| 略称 | 名前 | 目的 |
|-----|------|------|
| **CEP** | Critical Experience Prototype | ユーザー体験・価値を伝える（4コマ漫画・寸劇・動画）|
| **CFP** | Critical Function Prototype | 重要な機能を最初に作る（ダンボール模型でOK）|
| **DHP** | Dark Horse Prototype | 常識を外した思い切ったデザインを試す（穴馬）|
| **FKP** | FunKtional Prototype | CEP/CFP/DHPの知見を統合。収束フェーズへ |
| **FCP** | FunCtional Prototype | 最終形をイメージした機能プロトタイプ |
| **XFP** | X-is-Finished Prototype | 中心的機能が動作する（魅力品質を実装）|
| **VFP** | Validated Final Prototype | 量産前の最終試作機 |

EDPの最低ゴール: **XFP**（中心機能完成プロトタイプ）まで作ること。

---

### 雑に作る（雑っぴんぐ）

**「クイック＆ダーティ」= EDPでは「雑に作る」**

雑に作ることの利点:
- 手を動かすことで新たなアイデアが浮かぶ（構築主義 / コンストラクショニズム）
- 完成品でないことが相手に伝わるため、厳しいフィードバックを受けても心理的に平気
- 未練なく捨てられる（丁寧に作りすぎると捨てられなくなる）
- フィードバックサイクルが高速化する

**雑に作るための工夫**:
- 使える素材を限定: 紙・ダンボール・粘土・ガムテープ・ブロック玩具・グルーガン
- 汚してもいい・散らかしてもいい環境を用意する
- 雑に作ることを称賛する雰囲気を作る
- テストに快く応じてくれる人を確保する

**どこまで雑に作るか**:
- 今後の方向性のフィードバックが欲しい → 品質を下げる
- 細部のフィードバックが欲しい → 全体的な品質を高める
- ユーザーに接する部分の完成度が高すぎると期待値が上がりすぎる点に注意

---

### CEP の作り方（ユーザー体験の試作）

#### 4コマ漫画
「起承転結」の最小構成。絵の上手さより**内容が伝わるか**が重要（棒人間でOK）。

構成例:
1. 起: ユーザーの紹介と日常
2. 承: 問題・違和感の発生
3. 転: ソリューションの登場（まだ詳細未定でよい）
4. 結: ユーザーがもたらされる価値（最重要）

#### 寸劇（コント）
- 事前に台本を用意する（アドリブは伝わりにくい）
- コントの「コンテクスト」がおもしろければ、アイデアとコンテクストが「適合（fit）」している証拠
- 「ウケたらOK」ではなく「ユーザー体験が表現できているか」を重視

#### 動画
- 30〜60秒程度のショートムービー
- 起承転結（または三幕構成: 発端・中盤・結末）
- スマホ撮影→つなぎ合わせるだけでOK
- 制作の目的: 価値の整理 + チームが「描く世界への納得感」を得る

## Step 5: テスト（Test）

### ユーザーテストの基本

- **フィードバックの質は成果の完成度に応じて決まる**: 作りかけのものには相応の意見しかもらえない
- 作業範囲を小さく保ち、**2週間以内に完了できるサイズ**に分割
- 完成度が低くても粗くても、まず完成させることが重要（「終わらせることを始めよう」）

### テストの注意点

- **安全面**: 耐久性に注意。何度もテストで脆くなるため定期的に補強。フェイルセーフを考慮
- **倫理面**: 技術者倫理を軽視しない。文脈・利用環境を含めた全体で評価・説明する
- プロトタイプが**雑**であるほど慎重に安全確認を行う

### 反復（Iteration）

- プロトタイプ → テスト → フィードバック → 改良 のサイクルを繰り返す
- 目標物は狩猟の獲物のように移動するため、定期的に立ち止まって方向を確認する（**狩猟採集モデル**）
- 1つのアイデアが失敗しても「アイデアの泉（デザイン原則）」があれば何度でも挑戦できる

## チームの多様性とファシリテーション

### T型人材

- 深い専門性 + 幅広い関心を持つ人材
- チームメンバーが多様であるほど、共感できる範囲が広がる（認知的多様性が重要）

### HRT（Humility / Respect / Trust）

- 謙虚・尊敬・信頼がチームの基本姿勢

### 役割分担をやめる

役割分担の失敗パターン:
- 特定のスキル持ちに作業が集中 → 「指示する顧客」と「制作する会社」の関係になる
- 多様性が損なわれ、チームの利点が失われる

代替アプローチ:
- 全員が全てに関わる（「いつも一緒に作業する」）
- 統合を先に設計する（進捗確認を先に・作業を後に）
- WIP（同時着手数）を制限して「協業」を促す

### カンバン

1. **見える化**: すべての作業を一覧にしてチームで共有
2. **流れの管理**: カードを素早く「Done」に移動させる
3. **WIP制限**: 同時着手数をメンバー数より少なく設定（例: 6人なら2〜4個）

### KPT（ふりかえり）

```
Keep   → 続けたいこと（よかったこと）
Problem → 問題点
Try    → 次に改善したいこと
```

付箋に書いてホワイトボードに貼り付けるだけでOK。

### 圧倒的当事者意識

- 客観視しすぎると「確かに有用だけど何かおもしろくない」ものになる
- 「自分をなくす」ほど夢中になって取り組んでいるかどうかでアイデアのよし悪しが決まる（**熱量**）
- 作り手が本気で楽しそうにしていることが、プロジェクトを前進させる最大のエンジン

## ソフトウェア開発への応用

| デザイン思考の活動 | ソフトウェア開発での対応 |
|-----------------|----------------------|
| ユーザーリサーチ | ユーザーインタビュー・ユーザビリティテスト |
| エクストリームユーザー法 | パワーユーザー・ライトユーザーへのインタビュー |
| カスタマージャーニーマップ | ユーザーフロー・サービスブループリント |
| POV（タテマエメソッド） | 課題の本質定義・ユーザーストーリー |
| HMW | 設計の問いかけ・スプリントゴール設定 |
| 雑っぴんぐ | MVP（Minimum Viable Product）開発 |
| プロトタイプ → テスト | Sprint → Review → Retrospective |
| カンバン | スクラムボード・カンバンボード（Jira / Trello）|

**デザイン思考 × アジャイル開発の組み合わせ**:
- 「共感→問題定義→発想」= Discovery フェーズ
- 「プロトタイプ→テスト」= Delivery フェーズ
- 反復サイクルを短く（2週間スプリント）して学習速度を最大化

**ソフトウェアエンジニアへの注意点**:
> 「大学で学ぶ理論の多くは『どのように作るか』に関するものであり、『何を作るべきか』に知見を与えるものではない。そのようなエンジニアは、新製品を生み出すときですら、『自分がやれそうなこと』から発想し、『おもしろそうなこと』を後回しにする傾向がある。」（EDP まえがき）

デザイン思考は「**何を作るべきか**」を探索するための方法論。技術的実現可能性ではなく、ユーザー価値を起点に考える習慣を持つことが重要。

## よくある落とし穴

| 落とし穴 | 対策 |
|---------|------|
| 役割分担で多様性が失われる | 「いつも一緒に作業する」 |
| ブレストでアイデアが出ない | 個人で考えてからグループへ。雑談・落書きを先に |
| プロトタイプを丁寧に作りすぎる | 「雑っぴんぐ」の精神を徹底する |
| インサイトが「まあそうだろうね」で終わる | タテマエメソッドで「とはいえ」の視点で切り替える |
| 未来予測で思考が停滞する | 近い未来のアーリーアダプターを対象にする / TTFTFカードを使う |
| 多様性だけに頼ってアイデアが出ない | 認知的ツールボックス（専門知識・経験）を増やす |
| 議論で時間を使いすぎる | 「終わらせることを始めよう」「まず作る」 |
| ユーザーの言葉を表面だけで受け取る | 行動の観察とコンテクストの理解を組み合わせる |
